# 1. uuid 类型与公开资源 ID

本节目标：理解 `uuid` 类型的作用，掌握如何生成和查询 UUID，并知道为什么很多后端项目会用 UUID 作为公开资源 ID。

`uuid` 是 Universally Unique Identifier，通用唯一标识符。

一个 UUID 长这样：

```text
7b9f4d8c-9f6f-4e64-9b68-3f1a22a7f7c1
```

它比自增整数长很多，但有一个很重要的优点：

```text
不容易被猜到。
```

---

## 一、为什么不直接暴露自增 ID

假设文章详情接口是：

```text
GET /articles/1
GET /articles/2
GET /articles/3
```

用户很容易猜到：

```text
系统里大概有多少文章。
下一篇文章 ID 是多少。
能不能访问别人的资源。
```

自增 ID 当然不是安全漏洞本身。真正的权限仍然必须由后端校验。

但把自增 ID 直接暴露给外部，会带来几个问题：

- 资源数量容易被枚举。
- 用户可以轻易遍历 ID。
- 多系统合并数据时 ID 容易冲突。
- URL 看起来像内部数据库主键。

因此很多项目会采用：

```text
内部主键：bigint 自增 ID，方便关联和索引。
外部 ID：uuid，暴露给接口和用户。
```

---

## 二、启用 UUID 生成函数

学习阶段建议先启用扩展：

```sql
CREATE EXTENSION IF NOT EXISTS pgcrypto;
```

然后可以生成 UUID：

```sql
SELECT gen_random_uuid();
```

如果你的 PostgreSQL 环境中 `gen_random_uuid()` 已经可用，也可以直接使用。为了教程统一，先保留 `CREATE EXTENSION IF NOT EXISTS pgcrypto;`。

---

## 三、创建带 UUID 的表

推荐实践是：

```text
id：内部主键，用 bigint。
public_id：外部公开 ID，用 uuid。
```

示例：

```sql
CREATE TABLE demo.uuid_articles (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    public_id uuid NOT NULL DEFAULT gen_random_uuid(),
    title text NOT NULL,
    content text NOT NULL,
    created_at timestamptz NOT NULL DEFAULT now(),
    CONSTRAINT uuid_articles_public_id_unique UNIQUE (public_id)
);
```

这里有两个关键点：

```sql
public_id uuid NOT NULL DEFAULT gen_random_uuid()
```

表示不传 `public_id` 时自动生成。

```sql
UNIQUE (public_id)
```

表示公开 ID 不能重复。

---

## 四、插入数据

```sql
INSERT INTO demo.uuid_articles (title, content)
VALUES
    ('PostgreSQL UUID 入门', '学习如何使用 uuid 作为公开 ID。'),
    ('后端接口 ID 设计', '内部 ID 和公开 ID 可以分开设计。')
RETURNING id, public_id, title;
```

返回结果中会同时看到：

```text
id：1、2 这种内部主键。
public_id：随机 UUID。
```

在后端接口中，可以返回 `public_id`：

```json
{
  "id": "7b9f4d8c-9f6f-4e64-9b68-3f1a22a7f7c1",
  "title": "PostgreSQL UUID 入门"
}
```

不要把内部自增 `id` 当成对外字段。

---

## 五、根据 UUID 查询

```sql
SELECT id, public_id, title, created_at
FROM demo.uuid_articles
WHERE public_id = '7b9f4d8c-9f6f-4e64-9b68-3f1a22a7f7c1';
```

因为 `public_id` 有唯一约束，PostgreSQL 会自动为它创建唯一索引。

你可以查看执行计划：

```sql
EXPLAIN
SELECT id, public_id, title
FROM demo.uuid_articles
WHERE public_id = '7b9f4d8c-9f6f-4e64-9b68-3f1a22a7f7c1';
```

通常会看到索引扫描。

---

## 六、URL 中使用 UUID

接口可以设计成：

```text
GET /articles/7b9f4d8c-9f6f-4e64-9b68-3f1a22a7f7c1
```

后端处理流程：

```text
1. 从 URL 中拿到 public_id。
2. 用 public_id 查询文章。
3. 查询到内部 id。
4. 继续用内部 id 做关联查询、权限判断、统计等操作。
```

内部关联仍然可以用 bigint：

```sql
SELECT c.id, c.content
FROM demo.comments AS c
JOIN demo.uuid_articles AS a ON a.id = c.article_id
WHERE a.public_id = $1;
```

这种做法兼顾：

- 外部 ID 不易枚举。
- 内部关联仍然高效清晰。

---

## 七、只用 UUID 做主键可以吗

可以。

例如：

```sql
CREATE TABLE demo.uuid_only_articles (
    id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    title text NOT NULL,
    content text NOT NULL,
    created_at timestamptz NOT NULL DEFAULT now()
);
```

这种设计也很常见，尤其在分布式系统、离线生成 ID、多服务写入等场景。

但学习阶段我更推荐先掌握“双 ID”模式：

```text
bigint id：内部主键。
uuid public_id：外部公开 ID。
```

原因是：

- 和前面阶段的自增主键知识衔接更自然。
- 外键关联更直观。
- URL 又不会暴露内部主键。

---

## 八、UUID 的优点

### 1. 不容易枚举

用户很难猜到另一个 UUID。

注意：这不能替代权限校验。即使 ID 很难猜，后端仍然必须判断当前用户是否有权限访问资源。

### 2. 可以在应用侧生成

有些系统可以在应用代码中生成 UUID，再插入数据库。

这在分布式场景中很方便。

### 3. 跨系统合并更安全

不同系统各自生成 UUID，冲突概率极低。

---

## 九、UUID 的成本

### 1. 更长

UUID 比 bigint 占用更多存储空间。

### 2. 可读性差

调试时：

```text
id = 123
```

比：

```text
id = 7b9f4d8c-9f6f-4e64-9b68-3f1a22a7f7c1
```

更容易看。

### 3. 随机 UUID 对索引局部性不如递增 ID

随机 UUID 插入时不会像自增 ID 那样天然按顺序增长，可能带来更多索引页分裂和缓存压力。

学习阶段不用过度纠结这点，但要知道 UUID 不是完全免费的。

---

## 十、什么时候适合用 UUID

适合：

- 对外暴露的用户 ID、文章 ID、订单 ID。
- 文件、图片、公开资源标识。
- 多系统生成 ID。
- 不希望被枚举的资源。

不一定适合：

- 纯内部、小型、单机学习表。
- 高频写入且极度追求索引局部性的核心表。
- 需要人工频繁输入和核对的编号。

例如订单通常可以同时有：

```text
id：内部主键。
public_id：UUID，对外 API 使用。
order_no：业务订单号，给用户和客服使用。
```

三者解决的问题不同。

---

## 十一、常见误区

### 1. UUID 能替代权限校验吗？

不能。

UUID 只是难猜，不是权限系统。所有资源访问仍然要做身份和权限检查。

### 2. 有 UUID 就不需要自增 ID 吗？

不一定。

可以只用 UUID 做主键，也可以使用 bigint 内部主键 + UUID 公开 ID。两种都可以，取决于项目需要。

### 3. UUID 适合给用户手动输入吗？

通常不适合。

如果用户需要手动输入、客服需要口头沟通，业务编号通常应该更短、更可读。

### 4. UUID 一定不会重复吗？

理论上存在极低概率冲突。实践中仍然应该加唯一约束，让数据库兜底。

- 创建 `uuid` 字段。
- 使用 `gen_random_uuid()` 生成 UUID。
- 使用 UUID 作为公开资源 ID。
- 理解内部主键和公开 ID 的区别。
- 为 UUID 字段添加唯一约束。
- 根据 UUID 查询资源。
- 知道 UUID 不能替代权限校验。
- 知道 UUID 的优点和成本。
