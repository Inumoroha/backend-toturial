# 0. PostgreSQL 特色能力总览

本阶段目标：理解 PostgreSQL 相比普通关系型数据库更强的能力，并知道什么时候该用、什么时候不该用。

前面几个阶段主要学习了关系型数据库的基础能力：

```text
表结构。
SQL 查询。
事务和并发。
索引和执行计划。
```

第七阶段开始接触 PostgreSQL 更有特色的能力：

```text
uuid
jsonb
Array
GIN 索引
枚举类型
范围类型
全文搜索
```

这些能力会让 PostgreSQL 不只是一个“普通 SQL 数据库”。它可以更自然地处理公开 ID、半结构化配置、数组标签、文档搜索、时间范围、状态枚举等业务问题。

但这一阶段最重要的不是“看到高级能力就全部用上”，而是学会判断：

```text
这个场景用 PostgreSQL 特色能力是否更清晰？
还是应该继续使用普通关系表、约束和关联表？
```

---

## 一、本阶段为什么重要

真实后端项目中经常遇到这些问题：

```text
不想把自增 ID 暴露给用户，应该用什么公开 ID？
用户偏好设置字段很多，而且以后会变，应该怎么存？
文章标签能不能直接用数组？
订单状态用 text、CHECK 还是 enum？
会议室预约时间段如何判断冲突？
站内搜索能不能先不用 Elasticsearch？
JSONB 很强，能不能把所有业务表都变成一个 JSON 字段？
```

这些问题没有一个永远正确的答案。

PostgreSQL 提供了很多工具，但工具越强，越需要边界感。

---

## 二、本阶段核心内容

本阶段建议拆成以下主题学习：

1. `uuid` 类型：适合作为公开资源 ID，避免暴露自增主键。
2. `jsonb` 类型：存储扩展配置、用户偏好、第三方回调原始数据等半结构化内容。
3. JSONB 与关系表对比：什么时候用 JSONB，什么时候用普通表。
4. Array 类型：存储小规模、边界清晰的数组值。
5. GIN 索引：支持 JSONB、Array、全文搜索等包含类查询。
6. 枚举类型：表达稳定、有限的状态集合。
7. 范围类型：表达时间段、数字区间，并做重叠判断。
8. 全文搜索基础：使用 `tsvector`、`tsquery` 和 GIN 索引做基础站内搜索。
9. 特色能力选择指南：把本阶段内容放回后端业务设计中。

---

## 三、推荐学习顺序

建议按下面顺序学习：

```text
uuid
-> jsonb
-> JSONB 与关系表对比
-> Array
-> GIN 索引
-> 枚举类型
-> 范围类型
-> 全文搜索
-> 选择指南和综合实践
```

原因是：

- `uuid` 最容易落地，很多接口都需要公开 ID。
- `jsonb` 很常用，也最容易被滥用，所以要单独讲清边界。
- Array 和 JSONB 都容易被拿来省表，需要重点对比关系建模。
- GIN 索引是 JSONB、Array、全文搜索性能优化的关键。
- 枚举和范围类型更适合特定业务场景。
- 全文搜索可以作为轻量站内搜索入门。

## 四、本阶段统一练习模型

为了让示例连贯，本阶段使用一个内容平台模型。

包含这些表：

- `demo.feature_users`：用户表，使用 `uuid` 作为公开 ID。
- `demo.feature_user_preferences`：用户偏好设置，使用 `jsonb`。
- `demo.feature_articles`：文章表，使用状态、标签、搜索字段。
- `demo.feature_article_categories`：文章分类关系表，用来和 Array/JSONB 对比。
- `demo.feature_events`：活动表，使用范围类型表达活动时间。

## 五、初始化练习环境

创建 schema：

```sql
CREATE SCHEMA IF NOT EXISTS demo;
```

启用常用扩展：

```sql
CREATE EXTENSION IF NOT EXISTS pgcrypto;
```

`pgcrypto` 常用于生成随机 UUID：

```sql
SELECT gen_random_uuid();
```

如果你的 PostgreSQL 环境中 `gen_random_uuid()` 已经可直接使用，执行扩展语句也通常没有坏处。学习阶段保持这个写法最省心。

## 六、初始化练习表

```sql
DROP TABLE IF EXISTS demo.feature_article_categories;
DROP TABLE IF EXISTS demo.feature_events;
DROP TABLE IF EXISTS demo.feature_articles;
DROP TABLE IF EXISTS demo.feature_user_preferences;
DROP TABLE IF EXISTS demo.feature_users;
DROP TYPE IF EXISTS demo.article_status;

CREATE TYPE demo.article_status AS ENUM ('draft', 'published', 'archived');

CREATE TABLE demo.feature_users (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    public_id uuid NOT NULL DEFAULT gen_random_uuid(),
    username text NOT NULL,
    email text NOT NULL,
    created_at timestamptz NOT NULL DEFAULT now(),
    CONSTRAINT feature_users_public_id_unique UNIQUE (public_id),
    CONSTRAINT feature_users_email_unique UNIQUE (email)
);

CREATE TABLE demo.feature_user_preferences (
    user_id bigint PRIMARY KEY,
    preferences jsonb NOT NULL DEFAULT '{}'::jsonb,
    updated_at timestamptz NOT NULL DEFAULT now(),
    CONSTRAINT feature_user_preferences_user_id_fk
        FOREIGN KEY (user_id)
        REFERENCES demo.feature_users (id)
        ON DELETE CASCADE,
    CONSTRAINT feature_user_preferences_is_object_check
        CHECK (jsonb_typeof(preferences) = 'object')
);

CREATE TABLE demo.feature_articles (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    public_id uuid NOT NULL DEFAULT gen_random_uuid(),
    author_id bigint NOT NULL,
    title text NOT NULL,
    content text NOT NULL,
    status demo.article_status NOT NULL DEFAULT 'draft',
    tags text[] NOT NULL DEFAULT ARRAY[]::text[],
    metadata jsonb NOT NULL DEFAULT '{}'::jsonb,
    search_document tsvector,
    created_at timestamptz NOT NULL DEFAULT now(),
    published_at timestamptz,
    CONSTRAINT feature_articles_public_id_unique UNIQUE (public_id),
    CONSTRAINT feature_articles_author_id_fk
        FOREIGN KEY (author_id)
        REFERENCES demo.feature_users (id)
        ON DELETE RESTRICT,
    CONSTRAINT feature_articles_metadata_is_object_check
        CHECK (jsonb_typeof(metadata) = 'object')
);

CREATE TABLE demo.feature_article_categories (
    article_id bigint NOT NULL,
    category text NOT NULL,
    created_at timestamptz NOT NULL DEFAULT now(),
    PRIMARY KEY (article_id, category),
    CONSTRAINT feature_article_categories_article_id_fk
        FOREIGN KEY (article_id)
        REFERENCES demo.feature_articles (id)
        ON DELETE CASCADE
);

CREATE TABLE demo.feature_events (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name text NOT NULL,
    room_name text NOT NULL,
    active_period tstzrange NOT NULL,
    created_at timestamptz NOT NULL DEFAULT now(),
    CONSTRAINT feature_events_period_not_empty_check
        CHECK (NOT isempty(active_period))
);
```

## 八、本阶段最重要的边界感

### 1. JSONB 很强，但不能替代表设计

适合 JSONB 的数据通常是：

- 扩展配置。
- 用户偏好。
- 第三方回调原始内容。
- 字段不稳定、查询要求不高的数据。
- 少量需要灵活扩展的属性。

不适合随便塞进 JSONB 的数据通常是：

- 核心业务实体。
- 关系清晰的数据。
- 需要强外键约束的数据。
- 高频统计和关联查询的数据。
- 需要严格字段类型和完整约束的数据。

### 2. Array 不是为了省掉关联表

Array 适合：

- 小规模标签。
- 简单列表。
- 不需要复杂属性的值集合。

如果数组里的每个元素本身也有属性、权限、排序、统计、外键关系，通常应该使用关联表。

### 3. GIN 索引适合包含类查询

GIN 索引常用于：

- JSONB 包含查询。
- Array 包含和重叠查询。
- 全文搜索。

但 GIN 索引也有成本，不能乱建。

### 4. PostgreSQL 特色能力要服务业务表达

特色能力的目的不是让 SQL 看起来高级，而是让表结构更准确、查询更清晰、约束更可靠。

## 九、本阶段达标标准

学完第七阶段后，你应该能够做到：

- 使用 `uuid` 作为公开资源 ID。
- 使用 `jsonb` 存储用户偏好和扩展配置。
- 查询 JSONB 字段中的嵌套属性。
- 使用 GIN 索引优化 JSONB、Array 和全文搜索。
- 区分 JSONB 和普通关系表的适用场景。
- 使用 Array 存储简单数组，并知道什么时候该改用关联表。
- 创建和使用枚举类型。
- 使用范围类型表达时间段，并判断区间是否重叠。
- 使用 `tsvector`、`tsquery` 做基础全文搜索。
- 知道 PostgreSQL 特色能力的边界，不把它们当成逃避建模的捷径。
