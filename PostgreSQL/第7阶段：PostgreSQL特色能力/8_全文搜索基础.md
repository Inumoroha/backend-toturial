# 8. 全文搜索基础

本节目标：理解 PostgreSQL 全文搜索的基本概念，掌握 `tsvector`、`tsquery`、`@@`、`ts_rank` 和 GIN 索引的基础用法。

全文搜索用来解决这类问题：

```text
用户输入关键词，搜索标题和正文中相关的文章。
```

最简单的写法可能是：

```sql
WHERE title ILIKE '%postgresql%'
   OR content ILIKE '%postgresql%'
```

这种写法容易理解，但数据量大时性能有限，也不具备词法分析、相关度排序等能力。

PostgreSQL 提供了内置全文搜索能力，可以作为轻量站内搜索的起点。

---

## 一、全文搜索的核心概念

### 1. tsvector

`tsvector` 是把文档处理成可搜索词条后的结果。

例如：

```sql
SELECT to_tsvector('english', 'PostgreSQL supports full text search');
```

可能得到类似：

```text
'full':3 'postgresql':1 'search':5 'support':2 'text':4
```

它会把文本拆成词条，并记录位置。

### 2. tsquery

`tsquery` 表示搜索条件。

```sql
SELECT plainto_tsquery('english', 'full text search');
```

可能得到：

```text
'full' & 'text' & 'search'
```

### 3. @@

`@@` 表示：

```text
文档是否匹配查询。
```

```sql
SELECT to_tsvector('english', 'PostgreSQL supports full text search')
       @@ plainto_tsquery('english', 'text search') AS matched;
```

## 二、创建文章搜索表

```sql
CREATE TABLE demo.search_articles (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    title text NOT NULL,
    content text NOT NULL,
    search_document tsvector,
    created_at timestamptz NOT NULL DEFAULT now()
);
```

插入数据：

```sql
INSERT INTO demo.search_articles (title, content)
VALUES
    ('PostgreSQL JSONB Guide', 'JSONB is useful for flexible configuration and user preferences.'),
    ('PostgreSQL Full Text Search', 'Full text search uses tsvector and tsquery.'),
    ('Go Backend Development', 'Go is often used to build backend services with PostgreSQL.');
```

---

## 三、生成搜索文档

```sql
UPDATE demo.search_articles
SET search_document =
    to_tsvector(
        'english',
        coalesce(title, '') || ' ' || coalesce(content, '')
    );
```

这里：

```sql
coalesce(title, '')
```

是为了避免字段为 `NULL` 时拼接结果变成 `NULL`。

虽然当前表里 `title` 和 `content` 都是 `NOT NULL`，但真实项目中保留这种写法更稳。

---

## 四、基础搜索

```sql
SELECT id, title
FROM demo.search_articles
WHERE search_document @@ plainto_tsquery('english', 'postgresql search');
```

这会查找同时匹配 `postgresql` 和 `search` 的文章。

使用 `plainto_tsquery` 的好处是：

```text
用户输入普通文本，PostgreSQL 帮你转换成 tsquery。
```

不要直接把用户输入拼接成 `to_tsquery` 字符串，容易出错，也更难处理特殊字符。

---

## 五、相关度排序

使用 `ts_rank`：

```sql
SELECT
    id,
    title,
    ts_rank(search_document, plainto_tsquery('english', 'postgresql search')) AS rank
FROM demo.search_articles
WHERE search_document @@ plainto_tsquery('english', 'postgresql search')
ORDER BY rank DESC;
```

`rank` 越高，表示越相关。

基础站内搜索通常会：

```text
先匹配。
再按相关度排序。
再按发布时间作为辅助排序。
```

例如：

```sql
SELECT
    id,
    title,
    ts_rank(search_document, plainto_tsquery('english', 'postgresql search')) AS rank,
    created_at
FROM demo.search_articles
WHERE search_document @@ plainto_tsquery('english', 'postgresql search')
ORDER BY rank DESC, created_at DESC
LIMIT 20;
```

## 六、标题权重更高

标题通常比正文更重要。

可以给标题和正文不同权重：

```sql
UPDATE demo.search_articles
SET search_document =
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(content, '')), 'B');
```

权重从高到低：

```text
A
B
C
D
```

搜索时：

```sql
SELECT
    id,
    title,
    ts_rank(search_document, plainto_tsquery('english', 'postgresql')) AS rank
FROM demo.search_articles
WHERE search_document @@ plainto_tsquery('english', 'postgresql')
ORDER BY rank DESC;
```

标题中出现关键词的文章通常排名更靠前。

## 七、创建 GIN 索引

全文搜索通常需要 GIN 索引：

```sql
CREATE INDEX search_articles_document_gin_idx
ON demo.search_articles
USING gin (search_document);
```

查看执行计划：

```sql
EXPLAIN
SELECT id, title
FROM demo.search_articles
WHERE search_document @@ plainto_tsquery('english', 'postgresql search');
```

数据量大时，GIN 索引能明显减少扫描成本。

## 八、自动维护 search_document

如果每次插入或更新文章后，都要手动更新 `search_document`，很容易忘。

一种简单做法是使用生成列。

```sql
CREATE TABLE demo.search_articles_generated (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    title text NOT NULL,
    content text NOT NULL,
    search_document tsvector GENERATED ALWAYS AS (
        setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
        setweight(to_tsvector('english', coalesce(content, '')), 'B')
    ) STORED,
    created_at timestamptz NOT NULL DEFAULT now()
);
```

然后建立索引：

```sql
CREATE INDEX search_articles_generated_document_gin_idx
ON demo.search_articles_generated
USING gin (search_document);
```

这样 `title` 或 `content` 更新后，`search_document` 会自动重新计算。

## 九、中文搜索注意点

PostgreSQL 内置全文搜索对英文这类以空格分词的语言更友好。

中文文本没有天然空格分词，例如：

```text
PostgreSQL全文搜索基础
```

内置 `simple` 配置可以做非常基础的处理，但效果通常不如专门中文分词。

中文搜索如果要求较高，常见选择包括：

- 使用 PostgreSQL 扩展做中文分词。
- 在应用侧分词后写入搜索字段。
- 使用专门搜索引擎，例如 Elasticsearch、OpenSearch、Meilisearch 等。

学习阶段先掌握 PostgreSQL 全文搜索的基本概念即可，不要期待内置功能直接解决所有中文搜索需求。

## 十、全文搜索适合什么场景

适合：

- 中小规模站内搜索。
- 后台管理系统搜索。
- 简单文章标题和正文搜索。
- 不想一开始引入独立搜索引擎的项目。

不适合：

- 复杂中文分词和高质量搜索体验。
- 海量数据搜索。
- 复杂相关度调优。
- 拼写纠错、同义词、搜索推荐等高级功能。

这些场景可能需要专业搜索系统。

---

## 十一、全文搜索和 ILIKE 对比

`ILIKE '%keyword%'`：

- 简单。
- 容易理解。
- 适合小数据量。
- 不具备词法分析。
- 前导 `%` 对普通 B-Tree 不友好。

全文搜索：

- 支持词条化。
- 支持相关度排序。
- 可使用 GIN 索引。
- 配置和理解成本更高。
- 中文场景需要额外处理。

---

## 十二、常见误区

### 1. PostgreSQL 全文搜索能完全替代 Elasticsearch 吗？

不一定。

简单站内搜索可以先用 PostgreSQL。复杂搜索体验仍然可能需要专业搜索引擎。

### 2. 全文搜索对中文开箱即用吗？

效果有限。

中文通常需要额外分词方案。

### 3. 只要建了 GIN 索引，搜索就一定快？

不一定。

还要看数据量、查询条件、排序、返回行数和相关度计算成本。

### 4. 用户输入应该直接拼接到 to_tsquery 吗？

不建议。

普通搜索框更适合 `plainto_tsquery` 或 `websearch_to_tsquery` 这类函数。

学完本节后，你应该能够做到：

- 解释 `tsvector` 和 `tsquery` 的作用。
- 使用 `to_tsvector` 生成搜索文档。
- 使用 `plainto_tsquery` 处理用户搜索词。
- 使用 `@@` 做全文搜索匹配。
- 使用 `ts_rank` 做相关度排序。
- 使用 `setweight` 提高标题权重。
- 为 `tsvector` 创建 GIN 索引。
- 使用生成列自动维护搜索文档。
- 知道 PostgreSQL 内置全文搜索对中文有局限。
