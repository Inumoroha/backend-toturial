# 5. GIN 索引

本节目标：理解 GIN 索引适合什么场景，掌握给 JSONB、Array、全文搜索字段建立 GIN 索引，并知道它的成本和边界。

GIN 是 Generalized Inverted Index，通用倒排索引。

倒排索引可以粗略理解为：

```text
不是从一行数据找它包含哪些元素。
而是从某个元素反查哪些行包含它。
```

例如文章标签：

```text
PostgreSQL -> 第 1 篇、第 2 篇、第 5 篇
JSONB      -> 第 2 篇、第 8 篇
Search     -> 第 3 篇、第 9 篇
```

这类“包含关系”的查询，很适合 GIN。

---

## 一、GIN 索引适合什么

GIN 常用于：

- JSONB 包含查询。
- Array 包含和重叠查询。
- 全文搜索 `tsvector`。

典型操作符：

```text
jsonb @>
jsonb ?
array @>
array &&
tsvector @@ tsquery
```

普通 B-Tree 更适合：

- 等值查询。
- 范围查询。
- 排序。

GIN 更适合：

```text
某个复杂值里面是否包含某个元素。
```

---

## 二、给 JSONB 建 GIN 索引

准备表：

```sql
CREATE TABLE demo.gin_jsonb_articles (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    title text NOT NULL,
    metadata jsonb NOT NULL DEFAULT '{}'::jsonb
);
```

插入数据：

```sql
INSERT INTO demo.gin_jsonb_articles (title, metadata)
VALUES
    ('JSONB 入门', '{"source": "original", "tags": ["PostgreSQL", "JSONB"], "views": 120}'),
    ('全文搜索', '{"source": "original", "tags": ["PostgreSQL", "Search"], "views": 80}'),
    ('转载文章', '{"source": "imported", "tags": ["Database"], "views": 30}');
```

创建 GIN 索引：

```sql
CREATE INDEX gin_jsonb_articles_metadata_gin_idx
ON demo.gin_jsonb_articles
USING gin (metadata);
```

查询包含某个 JSON 片段的数据：

```sql
EXPLAIN
SELECT id, title
FROM demo.gin_jsonb_articles
WHERE metadata @> '{"source": "original"}'::jsonb;
```

查询包含数组元素：

```sql
SELECT id, title
FROM demo.gin_jsonb_articles
WHERE metadata @> '{"tags": ["JSONB"]}'::jsonb;
```

---

## 三、jsonb_ops 和 jsonb_path_ops

JSONB GIN 索引有不同操作类。

默认：

```sql
CREATE INDEX gin_jsonb_articles_metadata_gin_idx
ON demo.gin_jsonb_articles
USING gin (metadata);
```

使用的是默认操作类，支持更多操作符，例如 key 是否存在。

也可以使用：

```sql
CREATE INDEX gin_jsonb_articles_metadata_path_gin_idx
ON demo.gin_jsonb_articles
USING gin (metadata jsonb_path_ops);
```

`jsonb_path_ops` 通常更适合 `@>` 包含查询，索引可能更小，但支持的操作更少。

学习阶段建议：

```text
先用默认 GIN 索引。
当确认主要是 @> 包含查询，并且数据量较大时，再了解 jsonb_path_ops。
```

## 四、给 Array 建 GIN 索引

准备表：

```sql
CREATE TABLE demo.gin_array_articles (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    title text NOT NULL,
    tags text[] NOT NULL DEFAULT ARRAY[]::text[]
);
```

插入数据：

```sql
INSERT INTO demo.gin_array_articles (title, tags)
VALUES
    ('Array 入门', ARRAY['PostgreSQL', 'Array']),
    ('JSONB 入门', ARRAY['PostgreSQL', 'JSONB']),
    ('Go 后端', ARRAY['Go', 'Backend']);
```

创建 GIN 索引：

```sql
CREATE INDEX gin_array_articles_tags_gin_idx
ON demo.gin_array_articles
USING gin (tags);
```

查询包含标签：

```sql
SELECT id, title
FROM demo.gin_array_articles
WHERE tags @> ARRAY['PostgreSQL'];
```

查询标签有交集：

```sql
SELECT id, title
FROM demo.gin_array_articles
WHERE tags && ARRAY['JSONB', 'Search'];
```

## 五、给全文搜索字段建 GIN 索引

准备表：

```sql
CREATE TABLE demo.gin_search_articles (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    title text NOT NULL,
    content text NOT NULL,
    search_document tsvector
);
```

插入数据：

```sql
INSERT INTO demo.gin_search_articles (title, content)
VALUES
    ('PostgreSQL JSONB Guide', 'JSONB and GIN indexes are useful for flexible metadata search.'),
    ('PostgreSQL Full Text Search', 'Full text search uses tsvector and tsquery.'),
    ('Go Backend Development', 'Go services often use PostgreSQL as the primary database.');
```

更新搜索字段：

```sql
UPDATE demo.gin_search_articles
SET search_document =
    to_tsvector('simple', coalesce(title, '') || ' ' || coalesce(content, ''));
```

创建 GIN 索引：

```sql
CREATE INDEX gin_search_articles_document_gin_idx
ON demo.gin_search_articles
USING gin (search_document);
```

搜索：

```sql
SELECT id, title
FROM demo.gin_search_articles
WHERE search_document @@ plainto_tsquery('simple', 'postgresql jsonb');
```

全文搜索会在后面的文档中详细讲。

## 六、GIN 索引的优点

### 1. 很适合包含查询

例如：

```sql
metadata @> '{"source": "original"}'
tags @> ARRAY['PostgreSQL']
search_document @@ plainto_tsquery('simple', 'postgresql')
```

这些都可以从 GIN 中受益。

### 2. 能处理复杂字段内部元素

B-Tree 索引擅长对整个字段排序和比较。

GIN 擅长拆开字段内部元素做倒排。

---

## 七、GIN 索引的成本

### 1. 索引可能较大

JSONB、Array、全文搜索字段内部元素越多，GIN 索引可能越大。

### 2. 写入维护成本更高

插入或更新 JSONB、Array、tsvector 时，PostgreSQL 需要维护倒排结构。

如果字段更新很频繁，GIN 索引成本要认真评估。

### 3. 不适合所有查询

GIN 不适合替代 B-Tree。

例如：

```sql
WHERE created_at >= ...
ORDER BY created_at DESC
```

仍然更适合 B-Tree。

---

## 八、GIN 和表达式索引怎么选

JSONB 查询：

```sql
WHERE metadata ->> 'source' = 'original'
```

可以用表达式 B-Tree 索引：

```sql
CREATE INDEX gin_jsonb_articles_source_expr_idx
ON demo.gin_jsonb_articles ((metadata ->> 'source'));
```

如果查询是：

```sql
WHERE metadata @> '{"source": "original"}'::jsonb
```

可以用 GIN 索引：

```sql
CREATE INDEX gin_jsonb_articles_metadata_gin_idx
ON demo.gin_jsonb_articles
USING gin (metadata);
```

怎么选？

```text
固定路径、等值查询：表达式 B-Tree 索引可能更直接。
灵活包含、多 key、多结构查询：GIN 更合适。
```

---

## 九、使用 EXPLAIN 验证

创建索引后，用执行计划验证：

```sql
EXPLAIN
SELECT id, title
FROM demo.gin_jsonb_articles
WHERE metadata @> '{"source": "original"}'::jsonb;
```

可能看到：

```text
Bitmap Index Scan
Bitmap Heap Scan
```

这很正常。GIN 索引常常配合 Bitmap 扫描。

不要期待所有索引查询都显示 `Index Scan`。

---

## 十、常见误区

### 1. GIN 索引能替代 B-Tree 吗？

不能。

GIN 和 B-Tree 适合的查询不同。

### 2. JSONB 字段只要建一个 GIN 索引就万事大吉？

不是。

如果高频查询固定取某个字段做等值比较，表达式索引可能更合适。

### 3. GIN 索引没有成本吗？

有。

它可能占用更多空间，也会增加写入和更新成本。

### 4. 看到 Bitmap Index Scan 是不是索引没生效？

不是。

Bitmap Index Scan 是使用索引的一种方式，GIN 查询中很常见。

## 十一、本节达标标准

学完本节后，你应该能够做到：

- 解释 GIN 是倒排索引。
- 给 JSONB 字段创建 GIN 索引。
- 给 Array 字段创建 GIN 索引。
- 给 `tsvector` 字段创建 GIN 索引。
- 知道 GIN 适合包含类查询。
- 知道 GIN 不能替代 B-Tree。
- 初步区分 JSONB GIN 索引和表达式索引的适用场景。
- 使用 `EXPLAIN` 验证 GIN 查询是否使用索引。
