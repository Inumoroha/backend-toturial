# 9. 特色能力选择指南与综合实践

本节目标：把第七阶段的特色能力放回真实后端设计中，形成判断标准：什么时候用 `uuid`、JSONB、Array、GIN、枚举、范围类型和全文搜索，什么时候继续使用普通关系表。

PostgreSQL 的特色能力很多，但好的数据库设计不是“用最多功能”，而是：

```text
用最合适的结构表达业务。
用足够清晰的约束保护数据。
用合适的索引支撑查询。
```

---

## 一、能力选择总表

| 能力 | 适合场景 | 不适合场景 |
| --- | --- | --- |
| `uuid` | 公开资源 ID、不希望被枚举、多系统生成 ID | 用户手动输入的短编号 |
| `jsonb` | 用户偏好、扩展配置、第三方原始数据 | 核心关系数据、强约束数据 |
| Array | 小规模同类型值列表、简单标签 | 多对多关系、需要统计和外键的元素 |
| GIN 索引 | JSONB 包含、Array 包含、全文搜索 | 普通范围查询、排序查询 |
| enum | 稳定有限状态 | 经常变化、需要后台配置的字典 |
| range | 时间段、数字区间、重叠判断 | 单个时间点、普通数值字段 |
| 全文搜索 | 基础站内搜索、后台搜索 | 复杂搜索体验、强中文分词需求 |

---

## 二、综合实践：内容平台设计

假设要设计一个文章平台：

需求：

- 文章有内部 ID 和公开 ID。
- 用户可以设置偏好。
- 文章有标题、正文、状态、标签。
- 文章有扩展配置。
- 支持按标签筛选。
- 支持基础全文搜索。
- 支持活动时间段查询。

可以设计：

```sql
CREATE EXTENSION IF NOT EXISTS pgcrypto;

CREATE TYPE demo.practice_article_status AS ENUM (
    'draft',
    'published',
    'archived'
);

CREATE TABLE demo.practice_users (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    public_id uuid NOT NULL DEFAULT gen_random_uuid(),
    username text NOT NULL,
    email text NOT NULL,
    preferences jsonb NOT NULL DEFAULT '{}'::jsonb,
    created_at timestamptz NOT NULL DEFAULT now(),
    CONSTRAINT practice_users_public_id_unique UNIQUE (public_id),
    CONSTRAINT practice_users_email_unique UNIQUE (email),
    CONSTRAINT practice_users_preferences_is_object_check
        CHECK (jsonb_typeof(preferences) = 'object')
);

CREATE TABLE demo.practice_articles (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    public_id uuid NOT NULL DEFAULT gen_random_uuid(),
    author_id bigint NOT NULL,
    title text NOT NULL,
    content text NOT NULL,
    status demo.practice_article_status NOT NULL DEFAULT 'draft',
    tags text[] NOT NULL DEFAULT ARRAY[]::text[],
    metadata jsonb NOT NULL DEFAULT '{}'::jsonb,
    search_document tsvector GENERATED ALWAYS AS (
        setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
        setweight(to_tsvector('english', coalesce(content, '')), 'B')
    ) STORED,
    created_at timestamptz NOT NULL DEFAULT now(),
    published_at timestamptz,
    CONSTRAINT practice_articles_public_id_unique UNIQUE (public_id),
    CONSTRAINT practice_articles_author_id_fk
        FOREIGN KEY (author_id)
        REFERENCES demo.practice_users (id)
        ON DELETE RESTRICT,
    CONSTRAINT practice_articles_metadata_is_object_check
        CHECK (jsonb_typeof(metadata) = 'object')
);

CREATE TABLE demo.practice_events (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name text NOT NULL,
    active_period tstzrange NOT NULL,
    created_at timestamptz NOT NULL DEFAULT now(),
    CONSTRAINT practice_events_period_not_empty_check
        CHECK (NOT isempty(active_period))
);
```

## 三、为综合实践添加索引

公开 ID 查询：

```sql
-- public_id 上的 UNIQUE 约束已经自动创建唯一索引
```

文章列表：

```sql
CREATE INDEX practice_articles_status_published_at_idx
ON demo.practice_articles (status, published_at DESC);
```

标签筛选：

```sql
CREATE INDEX practice_articles_tags_gin_idx
ON demo.practice_articles
USING gin (tags);
```

扩展配置包含查询：

```sql
CREATE INDEX practice_articles_metadata_gin_idx
ON demo.practice_articles
USING gin (metadata);
```

全文搜索：

```sql
CREATE INDEX practice_articles_search_document_gin_idx
ON demo.practice_articles
USING gin (search_document);
```

活动时间范围查询：

```sql
CREATE INDEX practice_events_active_period_gist_idx
ON demo.practice_events
USING gist (active_period);
```

这里出现了 GiST 索引。范围类型的重叠查询常用 GiST，而不是 GIN。当前阶段只需要知道：

```text
JSONB、Array、全文搜索常用 GIN。
范围类型常用 GiST。
```

## 五、常见查询示例

### 1. 根据公开 ID 查询文章

```sql
SELECT id, public_id, title
FROM demo.practice_articles
WHERE public_id = '00000000-0000-0000-0000-000000000000';
```

实际使用时替换成真实 UUID。

### 2. 查询深色主题用户

```sql
SELECT id, username, preferences
FROM demo.practice_users
WHERE preferences @> '{"theme": "dark"}'::jsonb;
```

### 3. 查询包含某个标签的文章

```sql
SELECT id, title, tags
FROM demo.practice_articles
WHERE tags @> ARRAY['PostgreSQL'];
```

### 4. 查询已发布文章列表

```sql
SELECT id, title, published_at
FROM demo.practice_articles
WHERE status = 'published'
ORDER BY published_at DESC
LIMIT 20;
```

### 5. 全文搜索

```sql
SELECT
    id,
    title,
    ts_rank(search_document, plainto_tsquery('english', 'postgresql gin')) AS rank
FROM demo.practice_articles
WHERE search_document @@ plainto_tsquery('english', 'postgresql gin')
ORDER BY rank DESC;
```

### 6. 查询某个时间正在进行的活动

```sql
SELECT id, name, active_period
FROM demo.practice_events
WHERE active_period @> '2026-07-10 10:30:00+08'::timestamptz;
```

## 六、JSONB 还是普通列：判断清单

考虑使用 JSONB 前，问自己这些问题：

```text
这个字段是否是核心业务字段？
是否需要 NOT NULL、UNIQUE、FOREIGN KEY？
是否经常用于 WHERE、ORDER BY、GROUP BY？
是否需要被其他表引用？
是否需要做高频统计？
字段结构是否稳定？
团队是否容易理解 JSONB 查询和索引？
```

如果大多数答案是“是”，优先普通列或普通表。

如果大多数答案是“否”，并且它更像扩展配置，JSONB 是好选择。

---

## 七、Array 还是关联表：判断清单

考虑使用 Array 前，问自己：

```text
数组元素是否只是简单值？
元素是否需要自己的属性？
是否需要外键约束？
是否需要统计每个元素的使用次数？
是否需要重命名、合并、禁用某个元素？
是否需要排序和权限？
```

如果只是简单小列表，可以用 Array。

如果元素是业务对象，用关联表。

---

## 八、enum 还是字典表：判断清单

考虑 enum 前，问自己：

```text
这个集合是否非常稳定？
是否需要后台配置？
是否需要显示名称、颜色、排序？
是否需要多语言？
是否经常新增或下线某个值？
```

稳定、简单、有限：enum 或 `CHECK`。

可配置、有属性、经常变化：字典表。

---

## 九、全文搜索还是搜索引擎：判断清单

PostgreSQL 全文搜索适合：

- 数据量不太大。
- 搜索需求简单。
- 可以接受基础相关度。
- 搜索不是核心产品体验。

专门搜索引擎适合：

- 搜索是核心功能。
- 中文分词要求高。
- 需要拼写纠错、同义词、推荐。
- 数据量大、搜索 QPS 高。
- 需要复杂排序和聚合。

## 十、一个重要原则：先建模，再炫技

遇到新需求时，先问：

```text
这个数据是什么？
它和其他数据有什么关系？
它需要什么约束？
它会怎么查询？
它会怎么变化？
```

然后再决定：

- 普通列。
- 关联表。
- JSONB。
- Array。
- enum。
- range。
- 全文搜索。

不要为了用 PostgreSQL 特色能力而用。

## 十一、常见误区

### 1. PostgreSQL 很强，所以可以不用 Redis、搜索引擎、文档数据库？

不一定。

PostgreSQL 能覆盖很多场景，但不是所有场景的最佳工具。先用 PostgreSQL 做简单可靠的方案很好，但也要知道什么时候引入专门系统。

### 2. JSONB 和 Array 可以减少表数量，所以一定更好？

不是。

减少表数量不等于降低复杂度。很多时候只是把复杂度藏进 JSON 和数组里。

### 3. GIN 索引建上就不用关心性能了？

不是。

GIN 索引也有写入成本、存储成本和适用边界。

### 4. 特色类型会让项目更难维护吗？

取决于使用方式。

用在合适场景，它们会让模型更清晰。滥用时，才会让维护变困难。

## 十二、本节达标标准

学完本节后，你应该能够做到：

- 根据业务场景选择 `uuid`、JSONB、Array、enum、range、全文搜索。
- 判断 JSONB 和普通关系表的边界。
- 判断 Array 和关联表的边界。
- 判断 enum 和字典表的边界。
- 为 JSONB、Array、全文搜索建立 GIN 索引。
- 使用范围类型做时间段查询。
- 知道 PostgreSQL 特色能力是建模工具，不是逃避设计的捷径。
