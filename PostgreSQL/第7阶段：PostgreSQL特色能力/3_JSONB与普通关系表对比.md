# 3. JSONB 与普通关系表对比

本节目标：理解 JSONB 和普通关系表在查询、约束、维护上的差异，知道什么时候应该使用 JSONB，什么时候应该坚持普通表结构。

PostgreSQL 的 JSONB 很强，但这句话后面必须接一句：

```text
JSONB 很强，不等于要把 PostgreSQL 当 MongoDB 用。
```

关系型数据库最强的能力仍然是：

- 清晰表结构。
- 类型约束。
- 主键外键。
- JOIN。
- 聚合统计。
- 事务一致性。

JSONB 是补充，不是替代。

---

## 一、两个设计方案

假设要存储文章的扩展信息：

```text
SEO 标题。
SEO 描述。
文章来源。
阅读量。
是否允许评论。
```

方案一：使用普通列。

```sql
CREATE TABLE demo.article_meta_relational (
    article_id bigint PRIMARY KEY,
    seo_title text,
    seo_description text,
    source text NOT NULL DEFAULT 'original',
    view_count bigint NOT NULL DEFAULT 0,
    allow_comments boolean NOT NULL DEFAULT true
);
```

方案二：使用 JSONB。

```sql
CREATE TABLE demo.article_meta_jsonb (
    article_id bigint PRIMARY KEY,
    metadata jsonb NOT NULL DEFAULT '{}'::jsonb,
    CONSTRAINT article_meta_jsonb_is_object_check
        CHECK (jsonb_typeof(metadata) = 'object')
);
```

插入 JSONB：

```sql
INSERT INTO demo.article_meta_jsonb (article_id, metadata)
VALUES
    (
        1,
        '{
            "seo_title": "PostgreSQL JSONB Guide",
            "seo_description": "Learn JSONB in PostgreSQL",
            "source": "original",
            "view_count": 120,
            "allow_comments": true
        }'
    );
```

---

## 二、查询对比

### 普通列查询

```sql
SELECT article_id, seo_title, view_count
FROM demo.article_meta_relational
WHERE source = 'original'
  AND allow_comments = true;
```

写法直接，类型清晰。

### JSONB 查询

```sql
SELECT
    article_id,
    metadata ->> 'seo_title' AS seo_title,
    (metadata ->> 'view_count')::bigint AS view_count
FROM demo.article_meta_jsonb
WHERE metadata ->> 'source' = 'original'
  AND (metadata ->> 'allow_comments')::boolean = true;
```

JSONB 写法更灵活，但要处理：

- 路径取值。
- 文本转换类型。
- 字段不存在时的结果。
- 类型不符合预期时的错误。

---

## 三、约束对比

### 普通列约束

```sql
source text NOT NULL DEFAULT 'original',
view_count bigint NOT NULL DEFAULT 0,
allow_comments boolean NOT NULL DEFAULT true
```

数据库天然知道：

- `source` 是文本。
- `view_count` 是数字。
- `allow_comments` 是布尔值。
- 哪些字段不能为空。

### JSONB 约束

JSONB 需要你额外写约束：

```sql
ALTER TABLE demo.article_meta_jsonb
ADD CONSTRAINT article_meta_jsonb_source_check
CHECK (
    metadata ->> 'source' IS NULL
    OR metadata ->> 'source' IN ('original', 'imported', 'partner')
);
```

限制数字：

```sql
ALTER TABLE demo.article_meta_jsonb
ADD CONSTRAINT article_meta_jsonb_view_count_check
CHECK (
    metadata -> 'view_count' IS NULL
    OR jsonb_typeof(metadata -> 'view_count') = 'number'
);
```

可以做，但约束会越来越复杂。

如果字段非常核心，普通列通常更清晰。

---

## 四、索引对比

### 普通列索引

```sql
CREATE INDEX article_meta_relational_source_idx
ON demo.article_meta_relational (source);
```

查询：

```sql
WHERE source = 'original'
```

简单直观。

### JSONB 表达式索引

```sql
CREATE INDEX article_meta_jsonb_source_idx
ON demo.article_meta_jsonb ((metadata ->> 'source'));
```

查询必须匹配表达式：

```sql
WHERE metadata ->> 'source' = 'original'
```

### JSONB GIN 索引

```sql
CREATE INDEX article_meta_jsonb_metadata_gin_idx
ON demo.article_meta_jsonb
USING gin (metadata);
```

适合包含查询：

```sql
WHERE metadata @> '{"source": "original"}'::jsonb
```

普通列索引更明确，JSONB 索引更灵活但也更需要理解查询模式。

---

## 五、维护对比

### 普通表更适合长期稳定结构

如果字段稳定，并且业务长期依赖它们：

```text
订单金额。
支付状态。
用户邮箱。
文章标题。
库存数量。
外键关系。
```

优先普通列。

原因是：

- 类型清晰。
- 约束清晰。
- 索引清晰。
- 查询简单。
- 迁移可控。
- 团队容易理解。

### JSONB 更适合扩展配置

如果字段变化快、不是核心关系、查询不复杂：

```text
用户界面偏好。
通知开关。
第三方回调原文。
实验性配置。
不同渠道带来的扩展属性。
```

JSONB 很合适。

---

## 六、关系数据不要轻易塞进 JSONB

错误示例：

```json
{
  "author": {
    "id": 1,
    "username": "alice"
  },
  "comments": [
    {"id": 1, "content": "good"},
    {"id": 2, "content": "nice"}
  ]
}
```

如果作者、评论都是系统核心实体，应该使用关系表：

```text
users
articles
comments
```

原因：

- 评论需要分页。
- 评论需要审核。
- 评论需要统计。
- 评论有自己的创建时间、作者、状态。
- 作者信息更新后不应该到处同步 JSON。

这类关系数据用普通表更清晰。

---

## 七、什么时候从 JSONB 拆成普通列

如果 JSONB 里的某个字段开始满足这些条件，就应该考虑拆出来：

- 高频查询。
- 高频排序。
- 高频更新。
- 需要唯一约束。
- 需要外键。
- 需要复杂统计。
- 需要严格类型和非空约束。
- 其他表需要引用它。

例如一开始用户偏好里有：

```json
{"language": "zh-CN"}
```

只是展示界面用，放 JSONB 没问题。

后来业务要求：

```text
按 language 分组统计用户。
根据 language 推送内容。
language 必须引用语言配置表。
```

这时就可以考虑把 `language` 提升为普通列。

---

## 八、JSONB 和普通列可以混合使用

真实项目中经常这样设计：

```sql
CREATE TABLE demo.products_with_extra (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name text NOT NULL,
    price numeric(12, 2) NOT NULL,
    stock integer NOT NULL,
    status text NOT NULL,
    extra jsonb NOT NULL DEFAULT '{}'::jsonb,
    created_at timestamptz NOT NULL DEFAULT now()
);
```

核心字段用普通列：

- `name`
- `price`
- `stock`
- `status`

扩展字段用 JSONB：

- 展示配置。
- 渠道扩展信息。
- 临时实验字段。
- 第三方平台返回字段。

这通常比“全普通列”或“全 JSONB”都更平衡。

---

## 九、常见误区

### 1. JSONB 比普通表更先进，所以都应该用 JSONB？

不是。

JSONB 是灵活工具，不是关系建模的替代品。

### 2. 字段以后可能变化，就一定要 JSONB？

不一定。

如果字段是核心业务概念，即使会变化，也应该通过迁移管理表结构。

### 3. JSONB 查询一定很慢？

不一定。

合适的 GIN 索引或表达式索引可以让 JSONB 查询表现很好。但查询、约束和维护复杂度仍然要考虑。

### 4. 普通列和 JSONB 只能二选一？

不是。

最常见的好设计是核心字段普通列，扩展字段 JSONB。

---

## 十、本节达标标准

学完本节后，你应该能够做到：

- 对比普通列和 JSONB 的查询写法。
- 对比普通列和 JSONB 的约束复杂度。
- 知道 JSONB 可以建立表达式索引和 GIN 索引。
- 解释为什么关系清晰的数据优先使用普通表。
- 判断 JSONB 字段何时应该拆成普通列。
- 设计“核心字段普通列 + 扩展字段 JSONB”的表结构。
