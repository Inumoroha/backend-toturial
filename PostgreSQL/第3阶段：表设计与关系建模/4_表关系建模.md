# 3. 表关系建模：一对一、一对多、多对多

本节目标：理解关系型数据库中最常见的三种表关系：一对一、一对多、多对多，并能使用 PostgreSQL 的主键、外键、唯一约束和关联表设计出清晰的数据模型。

前面已经学习了字段类型和约束。本节开始进入表设计的核心：如何把真实业务中的对象和关系，拆分成结构清晰、约束可靠的数据表。

---

## 一、为什么要学习表关系

真实业务很少只有一张表。

例如一个博客系统中可能有：

- 用户。
- 用户资料。
- 文章。
- 评论。
- 标签。

它们之间存在关系：

```text
一个用户有一份用户资料。
一个用户可以发布多篇文章。
一篇文章可以有多个标签，一个标签也可以属于多篇文章。
```

这些关系分别对应：

- 一对一。
- 一对多。
- 多对多。

关系建模的目标是：

- 减少重复数据。
- 保证数据一致性。
- 让查询和维护更清晰。
- 让业务规则能落到数据库结构里。

## 二、准备练习环境

请先连接到学习数据库 `learn_pg`。

创建 schema：

```sql
CREATE SCHEMA IF NOT EXISTS demo;
```

为了避免旧表影响本节练习，先按依赖关系删除旧表：

```sql
DROP TABLE IF EXISTS demo.post_tags;
DROP TABLE IF EXISTS demo.tags;
DROP TABLE IF EXISTS demo.comments;
DROP TABLE IF EXISTS demo.posts;
DROP TABLE IF EXISTS demo.user_profiles;
DROP TABLE IF EXISTS demo.users;
```

注意：有外键依赖时，通常应该先删除子表，再删除父表。

## 三、一对一关系

一对一表示：一条 A 表记录最多对应一条 B 表记录，一条 B 表记录也只对应一条 A 表记录。

典型例子：

```text
一个用户只有一份用户资料。
一份用户资料只属于一个用户。
```

可以设计为：

- `demo.users`：用户主表，存登录和身份相关信息。
- `demo.user_profiles`：用户资料表，存头像、简介、生日等扩展信息。

### 1. 创建一对一表结构

创建用户表：

```sql
CREATE TABLE demo.users (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    username text NOT NULL UNIQUE,
    email text NOT NULL UNIQUE,
    created_at timestamptz NOT NULL DEFAULT now()
);
```

创建用户资料表：

```sql
CREATE TABLE demo.user_profiles (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    user_id bigint NOT NULL UNIQUE,
    display_name text,
    bio text,
    birthday date,
    created_at timestamptz NOT NULL DEFAULT now(),
    CONSTRAINT user_profiles_user_id_fk
        FOREIGN KEY (user_id)
        REFERENCES demo.users (id)
        ON DELETE CASCADE
);
```

关键点是：

```sql
user_id bigint NOT NULL UNIQUE
```

- `NOT NULL`：每份资料必须属于一个用户。
- `UNIQUE`：一个用户最多只能有一份资料。
- `FOREIGN KEY`：资料中的 `user_id` 必须引用真实存在的用户。

### 2. 插入一对一数据

插入用户：

```sql
INSERT INTO demo.users (username, email)
VALUES ('zhangsan', 'zhangsan@example.com')
RETURNING id, username;
```

假设返回的用户 ID 是 `1`，插入资料：

```sql
INSERT INTO demo.user_profiles (user_id, display_name, bio, birthday)
VALUES (1, '张三', '喜欢 PostgreSQL 和 Go', '2000-01-01');
```

### 3. 查询一对一关系

```sql
SELECT
    u.id,
    u.username,
    u.email,
    p.display_name,
    p.bio,
    p.birthday
FROM demo.users AS u
LEFT JOIN demo.user_profiles AS p ON p.user_id = u.id;
```

这里使用 `LEFT JOIN`，因为有些用户可能还没有填写资料。

### 4. 一对一关系的常见用途

适合拆成一对一的情况：

- 主表字段太多，想把不常用字段拆出去。
- 敏感信息单独管理。
- 用户基础信息和用户扩展资料分开。
- 某些字段不是每个用户都有。

不适合过度拆表。字段很少、访问频率类似时，直接放在同一张表里也可以。

## 四、一对多关系

一对多表示：一条 A 表记录可以对应多条 B 表记录，但一条 B 表记录只属于一条 A 表记录。

典型例子：

```text
一个用户可以发布多篇文章。
一篇文章只属于一个作者。
```

可以设计为：

- `demo.users`：用户表。
- `demo.posts`：文章表。

在多的一方，也就是 `posts` 表中保存外键 `user_id`。

### 1. 创建一对多表结构

创建文章表：

```sql
CREATE TABLE demo.posts (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    user_id bigint NOT NULL,
    title text NOT NULL,
    content text NOT NULL,
    is_published boolean NOT NULL DEFAULT false,
    created_at timestamptz NOT NULL DEFAULT now(),
    CONSTRAINT posts_user_id_fk
        FOREIGN KEY (user_id)
        REFERENCES demo.users (id)
        ON DELETE RESTRICT
);
```

关键点是：

```sql
user_id bigint NOT NULL
```

它表示每篇文章必须属于一个用户。

外键表示：

```sql
FOREIGN KEY (user_id) REFERENCES demo.users (id)
```

也就是 `posts.user_id` 必须引用 `users.id`。

### 2. 插入一对多数据

再插入一个用户：

```sql
INSERT INTO demo.users (username, email)
VALUES ('lisi', 'lisi@example.com')
RETURNING id, username;
```

给用户 1 插入多篇文章：

```sql
INSERT INTO demo.posts (user_id, title, content, is_published)
VALUES
    (1, 'PostgreSQL 入门', '学习数据库基础概念', true),
    (1, 'SQL 查询基础', '学习 SELECT 和 WHERE', true),
    (1, '事务入门', '理解 COMMIT 和 ROLLBACK', false);
```

给用户 2 插入文章：

```sql
INSERT INTO demo.posts (user_id, title, content, is_published)
VALUES (2, 'Go 后端入门', '使用 Go 编写 HTTP 服务', true);
```

### 3. 查询用户和文章

查询每篇文章及其作者：

```sql
SELECT
    p.id AS post_id,
    p.title,
    p.is_published,
    u.id AS user_id,
    u.username
FROM demo.posts AS p
JOIN demo.users AS u ON p.user_id = u.id
ORDER BY p.id;
```

查询某个用户的所有文章：

```sql
SELECT id, title, is_published, created_at
FROM demo.posts
WHERE user_id = 1
ORDER BY created_at DESC;
```

统计每个用户有多少篇文章：

```sql
SELECT
    u.id,
    u.username,
    COUNT(p.id) AS post_count
FROM demo.users AS u
LEFT JOIN demo.posts AS p ON p.user_id = u.id
GROUP BY u.id, u.username
ORDER BY u.id;
```

### 4. 一对多关系的建模规则

记住一个简单规则：

```text
一对多关系中，外键通常放在“多”的一方。
```

例如：

- 一个用户多篇文章：`posts.user_id`。
- 一篇文章多条评论：`comments.post_id`。
- 一个分类多个商品：`products.category_id`。
- 一个订单多条订单项：`order_items.order_id`。

### 5. 再加一个评论表

评论也是一对多关系：

```text
一篇文章可以有多条评论。
一条评论只属于一篇文章。
```

创建评论表：

```sql
CREATE TABLE demo.comments (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    post_id bigint NOT NULL,
    user_id bigint NOT NULL,
    content text NOT NULL,
    created_at timestamptz NOT NULL DEFAULT now(),
    CONSTRAINT comments_post_id_fk
        FOREIGN KEY (post_id)
        REFERENCES demo.posts (id)
        ON DELETE CASCADE,
    CONSTRAINT comments_user_id_fk
        FOREIGN KEY (user_id)
        REFERENCES demo.users (id)
        ON DELETE RESTRICT
);
```

这里有两个外键：

- `post_id`：评论属于哪篇文章。
- `user_id`：评论是谁写的。

插入评论：

```sql
INSERT INTO demo.comments (post_id, user_id, content)
VALUES
    (1, 2, '写得很清楚'),
    (1, 1, '谢谢支持'),
    (2, 2, 'SELECT 很常用');
```

查询评论及文章标题、评论作者：

```sql
SELECT
    c.id AS comment_id,
    p.title AS post_title,
    u.username AS comment_author,
    c.content,
    c.created_at
FROM demo.comments AS c
JOIN demo.posts AS p ON c.post_id = p.id
JOIN demo.users AS u ON c.user_id = u.id
ORDER BY c.id;
```

## 五、多对多关系

多对多表示：一条 A 表记录可以对应多条 B 表记录，一条 B 表记录也可以对应多条 A 表记录。

典型例子：

```text
一篇文章可以有多个标签。
一个标签也可以属于多篇文章。
```

这种关系不能只靠在某一张表里放一个外键解决。通常需要一张中间表，也叫关联表。

可以设计为：

- `demo.posts`：文章表。
- `demo.tags`：标签表。
- `demo.post_tags`：文章和标签的关联表。

### 1. 创建标签表

```sql
CREATE TABLE demo.tags (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name text NOT NULL UNIQUE,
    created_at timestamptz NOT NULL DEFAULT now()
);
```

### 2. 创建关联表

```sql
CREATE TABLE demo.post_tags (
    post_id bigint NOT NULL,
    tag_id bigint NOT NULL,
    created_at timestamptz NOT NULL DEFAULT now(),
    PRIMARY KEY (post_id, tag_id),
    CONSTRAINT post_tags_post_id_fk
        FOREIGN KEY (post_id)
        REFERENCES demo.posts (id)
        ON DELETE CASCADE,
    CONSTRAINT post_tags_tag_id_fk
        FOREIGN KEY (tag_id)
        REFERENCES demo.tags (id)
        ON DELETE CASCADE
);
```

关键点是：

```sql
PRIMARY KEY (post_id, tag_id)
```

它表示同一篇文章和同一个标签的组合不能重复。

也就是说，不能给同一篇文章重复绑定同一个标签。

### 3. 插入标签

```sql
INSERT INTO demo.tags (name)
VALUES
    ('PostgreSQL'),
    ('SQL'),
    ('Go'),
    ('Backend')
RETURNING id, name;
```

### 4. 绑定文章和标签

假设：

- 文章 1 是 PostgreSQL 入门。
- 标签 1 是 PostgreSQL。
- 标签 2 是 SQL。
- 标签 4 是 Backend。

插入关联关系：

```sql
INSERT INTO demo.post_tags (post_id, tag_id)
VALUES
    (1, 1),
    (1, 2),
    (1, 4),
    (2, 2),
    (3, 1),
    (4, 3),
    (4, 4);
```

### 5. 查询文章的所有标签

```sql
SELECT
    p.id AS post_id,
    p.title,
    t.name AS tag_name
FROM demo.posts AS p
JOIN demo.post_tags AS pt ON pt.post_id = p.id
JOIN demo.tags AS t ON t.id = pt.tag_id
WHERE p.id = 1
ORDER BY t.name;
```

### 6. 查询某个标签下的所有文章

```sql
SELECT
    t.name AS tag_name,
    p.id AS post_id,
    p.title
FROM demo.tags AS t
JOIN demo.post_tags AS pt ON pt.tag_id = t.id
JOIN demo.posts AS p ON p.id = pt.post_id
WHERE t.name = 'SQL'
ORDER BY p.id;
```

### 7. 聚合每篇文章的标签

PostgreSQL 可以使用 `array_agg` 把多行标签聚合成数组：

```sql
SELECT
    p.id,
    p.title,
    array_agg(t.name ORDER BY t.name) AS tags
FROM demo.posts AS p
LEFT JOIN demo.post_tags AS pt ON pt.post_id = p.id
LEFT JOIN demo.tags AS t ON t.id = pt.tag_id
GROUP BY p.id, p.title
ORDER BY p.id;
```

这类查询在后端返回文章列表时很常见。

## 六、三种关系对比

| 关系 | 含义 | 典型例子 | 建模方式 |
| --- | --- | --- | --- |
| 一对一 | A 最多对应一个 B，B 也最多对应一个 A | 用户和用户资料 | 外键 + 唯一约束 |
| 一对多 | A 对应多个 B，B 只属于一个 A | 用户和文章 | 外键放在多的一方 |
| 多对多 | A 有多个 B，B 也有多个 A | 文章和标签 | 使用中间关联表 |

简单记忆：

```text
一对一：外键 + UNIQUE。
一对多：外键放在多的一方。
多对多：拆成两组一对多，用中间表连接。
```

## 七、完整练习

可以从头执行下面这组 SQL，建立一个小型博客模型。

```sql
DROP TABLE IF EXISTS demo.post_tags;
DROP TABLE IF EXISTS demo.tags;
DROP TABLE IF EXISTS demo.comments;
DROP TABLE IF EXISTS demo.posts;
DROP TABLE IF EXISTS demo.user_profiles;
DROP TABLE IF EXISTS demo.users;

CREATE TABLE demo.users (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    username text NOT NULL UNIQUE,
    email text NOT NULL UNIQUE,
    created_at timestamptz NOT NULL DEFAULT now()
);

CREATE TABLE demo.user_profiles (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    user_id bigint NOT NULL UNIQUE,
    display_name text,
    bio text,
    birthday date,
    created_at timestamptz NOT NULL DEFAULT now(),
    CONSTRAINT user_profiles_user_id_fk
        FOREIGN KEY (user_id)
        REFERENCES demo.users (id)
        ON DELETE CASCADE
);

CREATE TABLE demo.posts (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    user_id bigint NOT NULL,
    title text NOT NULL,
    content text NOT NULL,
    is_published boolean NOT NULL DEFAULT false,
    created_at timestamptz NOT NULL DEFAULT now(),
    CONSTRAINT posts_user_id_fk
        FOREIGN KEY (user_id)
        REFERENCES demo.users (id)
        ON DELETE RESTRICT
);

CREATE TABLE demo.comments (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    post_id bigint NOT NULL,
    user_id bigint NOT NULL,
    content text NOT NULL,
    created_at timestamptz NOT NULL DEFAULT now(),
    CONSTRAINT comments_post_id_fk
        FOREIGN KEY (post_id)
        REFERENCES demo.posts (id)
        ON DELETE CASCADE,
    CONSTRAINT comments_user_id_fk
        FOREIGN KEY (user_id)
        REFERENCES demo.users (id)
        ON DELETE RESTRICT
);

CREATE TABLE demo.tags (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name text NOT NULL UNIQUE,
    created_at timestamptz NOT NULL DEFAULT now()
);

CREATE TABLE demo.post_tags (
    post_id bigint NOT NULL,
    tag_id bigint NOT NULL,
    created_at timestamptz NOT NULL DEFAULT now(),
    PRIMARY KEY (post_id, tag_id),
    CONSTRAINT post_tags_post_id_fk
        FOREIGN KEY (post_id)
        REFERENCES demo.posts (id)
        ON DELETE CASCADE,
    CONSTRAINT post_tags_tag_id_fk
        FOREIGN KEY (tag_id)
        REFERENCES demo.tags (id)
        ON DELETE CASCADE
);
```

插入练习数据：

```sql
INSERT INTO demo.users (username, email)
VALUES
    ('zhangsan', 'zhangsan@example.com'),
    ('lisi', 'lisi@example.com');

INSERT INTO demo.user_profiles (user_id, display_name, bio)
VALUES
    (1, '张三', '喜欢 PostgreSQL'),
    (2, '李四', 'Go 后端开发者');

INSERT INTO demo.posts (user_id, title, content, is_published)
VALUES
    (1, 'PostgreSQL 入门', '学习数据库基础概念', true),
    (1, 'SQL 查询基础', '学习 SELECT 和 WHERE', true),
    (2, 'Go 后端入门', '使用 Go 编写 HTTP 服务', true);

INSERT INTO demo.comments (post_id, user_id, content)
VALUES
    (1, 2, '写得很清楚'),
    (1, 1, '谢谢支持'),
    (2, 2, 'SELECT 很常用');

INSERT INTO demo.tags (name)
VALUES
    ('PostgreSQL'),
    ('SQL'),
    ('Go'),
    ('Backend');

INSERT INTO demo.post_tags (post_id, tag_id)
VALUES
    (1, 1),
    (1, 2),
    (1, 4),
    (2, 2),
    (3, 3),
    (3, 4);
```

查询文章列表，包括作者和标签：

```sql
SELECT
    p.id,
    p.title,
    u.username AS author,
    array_agg(t.name ORDER BY t.name) AS tags
FROM demo.posts AS p
JOIN demo.users AS u ON u.id = p.user_id
LEFT JOIN demo.post_tags AS pt ON pt.post_id = p.id
LEFT JOIN demo.tags AS t ON t.id = pt.tag_id
GROUP BY p.id, p.title, u.username
ORDER BY p.id;
```

## 八、常见误区

### 1. 一对一是不是必须拆成两张表？

不是。

如果字段少、访问频率相近，放在同一张表里也可以。拆成一对一通常是为了隔离扩展字段、敏感字段或低频字段。

### 2. 一对多的外键应该放在哪边？

通常放在“多”的一方。

例如一个用户多篇文章，外键放在 `posts.user_id`，而不是在 `users` 表里存一堆文章 ID。

### 3. 多对多可以用数组字段吗？

简单场景可以，但多数业务不建议。

如果标签需要独立管理、统计、查询、约束，就应该使用中间表，而不是 `tags text[]`。

### 4. 外键和索引是一回事吗？

不是。

外键保证数据引用关系正确，索引提升查询速度。外键列经常用于查询时，通常还应该单独建索引。

### 5. 所有关联删除都应该用 ON DELETE CASCADE 吗？

不是。

`ON DELETE CASCADE` 会自动删除子表数据，必须确认符合业务规则。比如删除文章时删除评论通常合理；删除用户时自动删除订单通常就很危险。

## 九、本节达标标准

学完本节后，你应该能够做到：

- 解释一对一、一对多、多对多分别是什么。
- 使用外键和唯一约束设计一对一关系。
- 把外键放在“多”的一方设计一对多关系。
- 使用中间表设计多对多关系。
- 理解 `PRIMARY KEY (post_id, tag_id)` 在关联表中的作用。
- 能写出用户、资料、文章、评论、标签之间的基础表结构。
- 能使用 `JOIN` 查询关联数据。
- 知道什么时候该用 `ON DELETE CASCADE`，什么时候要谨慎。

掌握表关系建模之后，就可以继续学习更系统的表设计方法，例如一对一、一对多、多对多在真实业务中的取舍，以及如何避免数据冗余和更新异常。
