# 1. 约束基础：让单表数据更可靠

本节目标：理解数据库约束的作用，掌握 PostgreSQL 中常见的单表约束：`NOT NULL`、`DEFAULT`、`UNIQUE`、`PRIMARY KEY`、`CHECK`，并能用它们设计更可靠的表结构。

字段类型能限制“一列能存什么类型的数据”，约束能进一步限制“哪些数据是合法的”。真实项目中，约束是保证数据质量的第一道防线。

---

## 一、约束是什么

约束是数据库对数据施加的规则。

例如：

- 用户邮箱不能为空。
- 用户邮箱不能重复。
- 年龄不能小于 0。
- 商品价格不能是负数。
- 每张表应该有一个主键。

这些规则不应该只依赖后端代码判断。因为后端代码可能有 bug，也可能存在多个入口写入数据库。把关键规则放到数据库约束里，可以让数据更可靠。

---

## 二、常见单表约束概览

| 约束 | 作用 | 示例 |
| --- | --- | --- |
| `NOT NULL` | 不允许为空 | 用户名不能为空 |
| `DEFAULT` | 默认值 | 创建时间默认当前时间 |
| `UNIQUE` | 不允许重复 | 邮箱不能重复 |
| `PRIMARY KEY` | 主键，唯一标识一行 | 用户 ID |
| `CHECK` | 自定义检查条件 | 年龄必须大于等于 0 |

本节先学习单表约束。跨表关系约束 `FOREIGN KEY` 会单独放在下一篇教程中学习。

## 四、NOT NULL：不能为空

`NOT NULL` 表示这一列不能存空值。

创建用户表：

```sql
CREATE TABLE demo.users (
    id bigint GENERATED BY DEFAULT AS IDENTITY,
    username text NOT NULL,
    email text NOT NULL,
    age integer,
    created_at timestamptz DEFAULT now()
);
```

这里：

- `username text NOT NULL`：用户名不能为空。
- `email text NOT NULL`：邮箱不能为空。
- `age integer`：年龄可以为空。

插入合法数据：

```sql
INSERT INTO demo.users (username, email, age)
VALUES ('zhangsan', 'zhangsan@example.com', 18);
```

尝试插入没有用户名的数据：

```sql
INSERT INTO demo.users (email, age)
VALUES ('no_name@example.com', 20);
```

这条语句会失败，因为 `username` 不允许为空。

### 什么时候用 NOT NULL

适合加 `NOT NULL` 的字段：

- 用户名。
- 邮箱。
- 订单号。
- 商品名称。
- 创建时间。
- 状态字段。

不确定是否一定存在的字段，可以暂时允许为空，例如：

- 用户头像。
- 个人简介。
- 删除时间 `deleted_at`。
- 发布时间 `published_at`。

## 五、DEFAULT：默认值

`DEFAULT` 用来给列设置默认值。

例如：

```sql
created_at timestamptz DEFAULT now()
```

插入数据时不传 `created_at`：

```sql
INSERT INTO demo.users (username, email, age)
VALUES ('lisi', 'lisi@example.com', 20);
```

查询结果：

```sql
SELECT id, username, email, created_at
FROM demo.users
ORDER BY id;
```

`created_at` 会自动使用当前时间。

常见默认值：

```sql
created_at timestamptz DEFAULT now()
is_active boolean DEFAULT true
visit_count bigint DEFAULT 0
status text DEFAULT 'pending'
```

注意：默认值只在插入时没有提供该列值时生效。如果你明确插入 `NULL`，是否允许取决于该列有没有 `NOT NULL`。

## 六、UNIQUE：不能重复

`UNIQUE` 表示某一列或某几列的值不能重复。

邮箱通常应该唯一：

```sql
DROP TABLE IF EXISTS demo.users;

CREATE TABLE demo.users (
    id bigint GENERATED BY DEFAULT AS IDENTITY,
    username text NOT NULL,
    email text NOT NULL UNIQUE,
    age integer,
    created_at timestamptz DEFAULT now()
);
```

插入第一条数据：

```sql
INSERT INTO demo.users (username, email, age)
VALUES ('zhangsan', 'zhangsan@example.com', 18);
```

再次插入相同邮箱：

```sql
INSERT INTO demo.users (username, email, age)
VALUES ('another', 'zhangsan@example.com', 20);
```

这条语句会失败，因为 `email` 违反了唯一约束。

### 多列唯一约束

有时不是单列唯一，而是多个字段组合起来唯一。

例如同一个项目里用户名不能重复：

```sql
CREATE TABLE demo.project_members (
    id bigint GENERATED BY DEFAULT AS IDENTITY,
    project_id bigint NOT NULL,
    username text NOT NULL,
    UNIQUE (project_id, username)
);
```

含义：

- 同一个 `project_id` 下，`username` 不能重复。
- 不同项目中，可以有相同的 `username`。

## 七、PRIMARY KEY：主键

主键用于唯一标识表中的一行数据。

一张表通常应该有一个主键。

推荐写法：

```sql
DROP TABLE IF EXISTS demo.users;

CREATE TABLE demo.users (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    username text NOT NULL,
    email text NOT NULL UNIQUE,
    age integer,
    created_at timestamptz DEFAULT now()
);
```

这里：

```sql
id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY
```

表示：

- `id` 是大整数。
- `id` 由数据库自动生成。
- `id` 是主键。

主键有两个重要特性：

- 不能为 `NULL`。
- 不能重复。

查询数据：

```sql
INSERT INTO demo.users (username, email, age)
VALUES
    ('zhangsan', 'zhangsan@example.com', 18),
    ('lisi', 'lisi@example.com', 20)
RETURNING id, username, email;
```

可以看到 `id` 自动生成。

## 八、CHECK：自定义检查条件

`CHECK` 用来定义自定义规则。

例如年龄不能小于 0：

```sql
DROP TABLE IF EXISTS demo.users;

CREATE TABLE demo.users (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    username text NOT NULL,
    email text NOT NULL UNIQUE,
    age integer CHECK (age >= 0),
    created_at timestamptz DEFAULT now()
);
```

插入合法年龄：

```sql
INSERT INTO demo.users (username, email, age)
VALUES ('zhangsan', 'zhangsan@example.com', 18);
```

插入非法年龄：

```sql
INSERT INTO demo.users (username, email, age)
VALUES ('bad_age', 'bad_age@example.com', -1);
```

这条语句会失败，因为违反了 `CHECK (age >= 0)`。

### 商品价格示例

```sql
DROP TABLE IF EXISTS demo.products;

CREATE TABLE demo.products (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name text NOT NULL,
    price numeric(12, 2) NOT NULL CHECK (price >= 0),
    stock integer NOT NULL DEFAULT 0 CHECK (stock >= 0),
    created_at timestamptz DEFAULT now()
);
```

这个表保证：

- 商品名不能为空。
- 价格不能为空，且不能小于 0。
- 库存不能为空，默认是 0，且不能小于 0。

## 九、给约束命名

PostgreSQL 会自动给约束生成名字，但真实项目中，给重要约束起名字更清晰。

示例：

```sql
DROP TABLE IF EXISTS demo.products;

CREATE TABLE demo.products (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name text NOT NULL,
    price numeric(12, 2) NOT NULL,
    stock integer NOT NULL DEFAULT 0,
    CONSTRAINT products_price_non_negative CHECK (price >= 0),
    CONSTRAINT products_stock_non_negative CHECK (stock >= 0),
    CONSTRAINT products_name_unique UNIQUE (name)
);
```

好处：

- 报错信息更容易看懂。
- 后续修改或删除约束时更方便。
- 团队协作时更清晰。

## 十、ALTER TABLE 添加约束

约束不仅可以在建表时定义，也可以后续添加。

先创建一张简单表：

```sql
DROP TABLE IF EXISTS demo.accounts;

CREATE TABLE demo.accounts (
    id bigint GENERATED BY DEFAULT AS IDENTITY,
    email text,
    balance numeric(12, 2)
);
```

添加非空约束：

```sql
ALTER TABLE demo.accounts
ALTER COLUMN email SET NOT NULL;
```

添加唯一约束：

```sql
ALTER TABLE demo.accounts
ADD CONSTRAINT accounts_email_unique UNIQUE (email);
```

添加检查约束：

```sql
ALTER TABLE demo.accounts
ADD CONSTRAINT accounts_balance_non_negative CHECK (balance >= 0);
```

注意：如果表中已有数据违反约束，添加约束会失败。真实项目中给旧表加约束前，需要先清理已有脏数据。

## 十一、完整练习

创建一张更接近真实项目的用户表：

```sql
DROP TABLE IF EXISTS demo.users;

CREATE TABLE demo.users (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    username text NOT NULL,
    email text NOT NULL UNIQUE,
    age integer CHECK (age >= 0),
    is_active boolean NOT NULL DEFAULT true,
    created_at timestamptz NOT NULL DEFAULT now()
);
```

插入合法数据：

```sql
INSERT INTO demo.users (username, email, age)
VALUES
    ('zhangsan', 'zhangsan@example.com', 18),
    ('lisi', 'lisi@example.com', 20)
RETURNING id, username, email, is_active, created_at;
```

尝试插入重复邮箱：

```sql
INSERT INTO demo.users (username, email, age)
VALUES ('dup', 'zhangsan@example.com', 22);
```

尝试插入空用户名：

```sql
INSERT INTO demo.users (username, email, age)
VALUES (NULL, 'null_name@example.com', 22);
```

尝试插入非法年龄：

```sql
INSERT INTO demo.users (username, email, age)
VALUES ('bad_age', 'bad_age@example.com', -1);
```

查看最终数据：

```sql
SELECT id, username, email, age, is_active, created_at
FROM demo.users
ORDER BY id;
```

## 十二、常见误区

### 1. 只在后端代码里校验，不加数据库约束可以吗？

不建议。

后端校验很重要，但数据库约束也很重要。关键规则应该同时在业务层和数据库层保证。

### 2. PRIMARY KEY 和 UNIQUE 有什么区别？

主键用于唯一标识一行，一张表通常只有一个主键。

唯一约束用于保证某个字段或字段组合不重复，一张表可以有多个唯一约束。

### 3. DEFAULT 会自动覆盖 NULL 吗？

不会。

如果插入时省略该列，默认值会生效。

如果你明确插入 `NULL`，那就是插入空值。是否允许取决于该列有没有 `NOT NULL`。

### 4. CHECK 可以替代所有业务校验吗？

不能。

`CHECK` 适合简单、稳定、数据库能判断的规则，例如价格不能小于 0。复杂业务规则仍然应该放在业务代码中处理。

### 5. 已有表加约束一定能成功吗？

不一定。

如果已有数据违反新约束，添加约束会失败。需要先清理数据，再添加约束。
