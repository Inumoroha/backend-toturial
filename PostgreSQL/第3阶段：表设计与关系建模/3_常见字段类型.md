# 3. PostgreSQL 常见字段类型

本节目标：理解 PostgreSQL 中常见字段类型的作用，能够为用户、文章、订单等常见业务表选择合适的数据类型。

数据库表不是随便放数据的地方。每一列都必须有明确的数据类型。选对字段类型，可以让数据更准确、查询更可靠、后续维护更轻松。

---

## 一、为什么字段类型很重要

字段类型决定了一列能存什么样的数据。

例如：

```sql
CREATE TABLE demo.users (
    id integer,
    username text,
    age integer,
    is_active boolean,
    created_at timestamptz
);
```

这里每一列都有自己的类型：

- `id integer`：只能存整数。
- `username text`：存文本。
- `age integer`：存整数。
- `is_active boolean`：存 true 或 false。
- `created_at timestamptz`：存带时区的时间。

选择类型时要考虑：

- 这个字段表达什么业务含义。
- 它是否需要计算。
- 它是否需要排序。
- 它是否可能为空。
- 它的范围有多大。
- 它未来是否会变化。

---

## 二、整数类型

PostgreSQL 常见整数类型：

| 类型 | 占用空间 | 范围 | 常见用途 |
| --- | --- | --- | --- |
| `smallint` | 2 字节 | 较小整数 | 状态码、小范围数字 |
| `integer` / `int` | 4 字节 | 常规整数 | 年龄、数量、普通计数 |
| `bigint` | 8 字节 | 大整数 | 大 ID、大计数、访问量 |

### 推荐用法

年龄：

```sql
age integer
```

访问次数：

```sql
visit_count bigint
```

库存数量：

```sql
stock integer
```

### 注意点

不要为了节省一点空间过早使用 `smallint`。普通业务中，`integer` 更常用。

如果是可能持续增长的计数，例如访问量、日志数量、订单号，优先考虑 `bigint`。

---

## 三、自增 ID 类型

PostgreSQL 以前常用：

```sql
serial
bigserial
```

现在更推荐使用 SQL 标准写法：

```sql
GENERATED BY DEFAULT AS IDENTITY
```

示例：

```sql
CREATE TABLE demo.users (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    username text NOT NULL
);
```

含义：

- `bigint`：ID 使用大整数。
- `GENERATED BY DEFAULT AS IDENTITY`：由数据库自动生成 ID。
- `PRIMARY KEY`：作为主键。

### serial 和 identity 的关系

`serial` 仍然能用，也很常见：

```sql
id bigserial PRIMARY KEY
```

但新项目中更推荐了解并使用 identity，因为它是更标准的自增写法。

---

## 四、文本类型

PostgreSQL 常见文本类型：

| 类型 | 含义 | 常见用途 |
| --- | --- | --- |
| `text` | 任意长度文本 | 用户名、标题、描述、正文 |
| `varchar(n)` | 最多 n 个字符 | 有明确长度限制的字符串 |
| `char(n)` | 固定长度字符串 | 很少使用 |

### 推荐用法

用户名：

```sql
username text
```

邮箱：

```sql
email text
```

文章标题：

```sql
title text
```

如果业务明确要求最大长度，可以使用：

```sql
username varchar(50)
```

### text 和 varchar 怎么选

在 PostgreSQL 中，`text` 和不带长度限制的 `varchar` 性能上通常没有明显差异。

初学阶段可以记住：

```text
没有明确长度限制时，用 text。
有明确业务长度限制时，用 varchar(n)。
```

例如：

- 用户昵称不一定需要数据库层限制，可以用 `text`。
- 手机验证码固定 6 位，可以用 `char(6)` 或 `text` 加约束。
- 邮箱如果想限制长度，可以用 `varchar(255)`。

## 五、布尔类型

布尔类型是：

```sql
boolean
```

它可以存：

```text
true
false
```

示例：

```sql
is_active boolean
is_deleted boolean
email_verified boolean
```

插入数据：

```sql
INSERT INTO demo.users (username, is_active)
VALUES ('zhangsan', true);
```

查询启用用户：

```sql
SELECT username
FROM demo.users
WHERE is_active = true;
```

也可以简写：

```sql
SELECT username
FROM demo.users
WHERE is_active;
```

### 命名建议

布尔字段建议使用能表达“是否”的名字：

- `is_active`
- `is_deleted`
- `is_published`
- `email_verified`
- `has_avatar`

这样查询时更容易读懂。

## 六、时间类型

PostgreSQL 常见时间类型：

| 类型 | 含义 | 常见用途 |
| --- | --- | --- |
| `date` | 日期 | 生日、日期字段 |
| `time` | 时间 | 较少单独使用 |
| `timestamp` | 不带时区的日期时间 | 特定本地时间 |
| `timestamptz` | 带时区语义的日期时间 | 创建时间、更新时间、登录时间 |

后端项目中，推荐优先使用：

```sql
timestamptz
```

常见字段：

```sql
created_at timestamptz DEFAULT now()
updated_at timestamptz DEFAULT now()
deleted_at timestamptz
```

### 为什么推荐 timestamptz

`timestamptz` 更适合服务端系统，因为它能更好地处理不同时区下的时间点。

例如用户创建时间、本质上是一个确定的时间点，不应该只依赖某个本地时区字符串。

### 插入当前时间

```sql
INSERT INTO demo.users (username, created_at)
VALUES ('lisi', now());
```

查询当前时间：

```sql
SELECT now();
```

## 七、数值类型

PostgreSQL 常见小数类型：

| 类型 | 特点 | 常见用途 |
| --- | --- | --- |
| `numeric` / `decimal` | 精确小数 | 金额、财务数据 |
| `real` | 单精度浮点 | 科学计算，较少用于业务金额 |
| `double precision` | 双精度浮点 | 科学计算、统计计算 |

### 金额应该用什么

金额、余额、价格这类字段，推荐使用：

```sql
numeric(12, 2)
```

示例：

```sql
price numeric(12, 2)
balance numeric(18, 2)
```

不要用浮点类型存金额，例如：

```sql
price double precision
```

浮点数可能有精度误差，不适合财务金额。

## 八、UUID 类型

UUID 是一种全局唯一标识符，PostgreSQL 原生支持：

```sql
uuid
```

示例：

```sql
CREATE TABLE demo.resources (
    id uuid PRIMARY KEY,
    name text NOT NULL
);
```

UUID 常用于：

- 公开暴露的资源 ID。
- 分布式系统中的 ID。
- 不想暴露自增 ID 的场景。

### 生成 UUID

PostgreSQL 可以使用函数生成 UUID，但需要确认相关函数可用。

常见写法之一：

```sql
SELECT gen_random_uuid();
```

如果不可用，可能需要启用扩展：

```sql
CREATE EXTENSION IF NOT EXISTS pgcrypto;
```

然后可以写：

```sql
CREATE TABLE demo.resources (
    id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    name text NOT NULL
);
```

### 自增 ID 和 UUID 怎么选

简单建议：

```text
内部主键、单体项目：bigint identity 很常见。
对外暴露 ID、分布式场景：uuid 更合适。
```

## 九、JSONB 类型

PostgreSQL 支持 JSON 和 JSONB。

更常用的是：

```sql
jsonb
```

示例：

```sql
CREATE TABLE demo.user_profiles (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    username text NOT NULL,
    preferences jsonb
);
```

插入数据：

```sql
INSERT INTO demo.user_profiles (username, preferences)
VALUES (
    'zhangsan',
    '{"theme": "dark", "language": "zh-CN"}'::jsonb
);
```

查询 JSONB 字段：

```sql
SELECT username, preferences ->> 'theme' AS theme
FROM demo.user_profiles;
```

### JSONB 适合什么

适合：

- 用户偏好设置。
- 第三方接口返回的扩展数据。
- 不固定的配置项。
- 少量灵活字段。

不适合：

- 结构稳定的核心业务数据。
- 需要强约束和频繁 JOIN 的关系数据。
- 本来应该拆表建模的数据。

JSONB 很强，但不是用来逃避表设计的。

## 十、数组类型

PostgreSQL 支持数组字段，例如：

```sql
tags text[]
scores integer[]
```

示例：

```sql
CREATE TABLE demo.articles (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    title text NOT NULL,
    tags text[]
);
```

插入数据：

```sql
INSERT INTO demo.articles (title, tags)
VALUES ('PostgreSQL 入门', ARRAY['database', 'postgresql', 'sql']);
```

查询：

```sql
SELECT title, tags
FROM demo.articles;
```

判断数组是否包含某个元素：

```sql
SELECT title
FROM demo.articles
WHERE tags @> ARRAY['sql'];
```

### 数组适合什么

适合：

- 简单标签。
- 简单列表。
- 不需要复杂关系维护的小集合。

不适合：

- 需要统计、约束、关联查询的多对多关系。

例如文章和标签如果需要独立管理标签、统计标签热度，通常更适合使用关联表，而不是数组字段。

## 十一、常见业务字段类型选择

### 用户表

```sql
CREATE TABLE demo.users_example (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    username text NOT NULL,
    email text NOT NULL,
    age integer,
    is_active boolean DEFAULT true,
    created_at timestamptz DEFAULT now()
);
```

### 文章表

```sql
CREATE TABLE demo.posts_example (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    title text NOT NULL,
    content text NOT NULL,
    is_published boolean DEFAULT false,
    published_at timestamptz,
    created_at timestamptz DEFAULT now()
);
```

### 商品表

```sql
CREATE TABLE demo.products_example (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name text NOT NULL,
    price numeric(12, 2) NOT NULL,
    stock integer NOT NULL DEFAULT 0,
    created_at timestamptz DEFAULT now()
);
```

## 十二、完整练习

创建一张综合练习表：

```sql
DROP TABLE IF EXISTS demo.type_examples;

CREATE TABLE demo.type_examples (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    public_id uuid DEFAULT gen_random_uuid(),
    username text NOT NULL,
    age integer,
    balance numeric(12, 2) DEFAULT 0,
    is_active boolean DEFAULT true,
    preferences jsonb,
    tags text[],
    created_at timestamptz DEFAULT now()
);
```

如果 `gen_random_uuid()` 不可用，先执行：

```sql
CREATE EXTENSION IF NOT EXISTS pgcrypto;
```

插入数据：

```sql
INSERT INTO demo.type_examples (
    username,
    age,
    balance,
    preferences,
    tags
)
VALUES (
    'zhangsan',
    18,
    99.50,
    '{"theme": "dark", "language": "zh-CN"}'::jsonb,
    ARRAY['new_user', 'vip']
);
```

查询数据：

```sql
SELECT
    id,
    public_id,
    username,
    age,
    balance,
    is_active,
    preferences ->> 'theme' AS theme,
    tags,
    created_at
FROM demo.type_examples;
```

## 十三、常见误区

### 1. 所有 ID 都用 integer 可以吗？

小项目可以，但更通用的选择是 `bigint`。

如果数据长期增长，自增主键建议优先使用：

```sql
bigint GENERATED BY DEFAULT AS IDENTITY
```

### 2. 金额可以用 double precision 吗？

不建议。

金额应该使用 `numeric` 或 `decimal`，避免浮点误差。

### 3. 时间字段用 timestamp 还是 timestamptz？

后端服务中的创建时间、更新时间、登录时间，通常推荐使用 `timestamptz`。

### 4. text 会不会比 varchar 慢？

在 PostgreSQL 中，`text` 和不带长度限制的 `varchar` 通常没有明显性能差异。

没有明确长度限制时，使用 `text` 很常见。

### 5. JSONB 可以替代表设计吗？

不建议这样理解。

JSONB 适合存部分灵活扩展数据，不适合替代清晰的关系建模。

### 6. 数组字段可以替代关联表吗？

简单场景可以，但复杂关系不建议。

如果需要查询、统计、约束和关联，优先考虑单独建表和关联表。
