# 2. FOREIGN KEY：外键约束

本节目标：理解 `FOREIGN KEY` 的作用，掌握如何在 PostgreSQL 中建立表与表之间的引用关系，并理解 `ON DELETE RESTRICT`、`ON DELETE CASCADE`、`ON DELETE SET NULL` 的区别和风险。

前一篇学习的是单表约束，例如 `NOT NULL`、`UNIQUE`、`PRIMARY KEY`、`CHECK`。本篇学习跨表约束：外键。外键是理解一对多、多对多关系建模的基础。

---

## 一、FOREIGN KEY 是什么

`FOREIGN KEY` 用来建立表与表之间的引用关系。

它最常见的作用是：保证一张表中的某个字段，必须引用另一张表中已经存在的记录。

例如：

```text
一个用户可以有多篇文章。
每篇文章都必须属于一个已经存在的用户。
```

这时可以设计两张表：

- `demo.users`：用户表。
- `demo.posts`：文章表。

其中：

```text
demo.posts.user_id -> demo.users.id
```

这表示文章表中的 `user_id` 必须引用用户表中真实存在的 `id`。

---

## 二、准备练习环境

请先连接到学习数据库 `learn_pg`。

创建 schema：

```sql
CREATE SCHEMA IF NOT EXISTS demo;
```

删除旧表。因为 `posts` 依赖 `users`，删除时应该先删除子表：

```sql
DROP TABLE IF EXISTS demo.posts;
DROP TABLE IF EXISTS demo.users;
```

## 三、创建带外键的表

创建用户表：

```sql
CREATE TABLE demo.users (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    username text NOT NULL UNIQUE,
    created_at timestamptz NOT NULL DEFAULT now()
);
```

创建文章表，并添加外键：

```sql
CREATE TABLE demo.posts (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    user_id bigint NOT NULL,
    title text NOT NULL,
    content text NOT NULL,
    created_at timestamptz NOT NULL DEFAULT now(),
    CONSTRAINT posts_user_id_fk
        FOREIGN KEY (user_id)
        REFERENCES demo.users (id)
);
```

这段外键的含义是：

```sql
CONSTRAINT posts_user_id_fk
    FOREIGN KEY (user_id)
    REFERENCES demo.users (id)
```

- `posts_user_id_fk`：外键约束名称。
- `FOREIGN KEY (user_id)`：当前表中的 `user_id` 是外键列。
- `REFERENCES demo.users (id)`：它引用 `demo.users` 表中的 `id` 列。

---

## 四、插入符合外键约束的数据

先插入用户：

```sql
INSERT INTO demo.users (username)
VALUES ('zhangsan')
RETURNING id, username;
```

假设返回的用户 ID 是 `1`，再插入文章：

```sql
INSERT INTO demo.posts (user_id, title, content)
VALUES (1, 'PostgreSQL 入门', '这是第一篇文章');
```

查询文章和作者：

```sql
SELECT
    p.id AS post_id,
    p.title,
    u.id AS user_id,
    u.username
FROM demo.posts AS p
JOIN demo.users AS u ON p.user_id = u.id;
```

这里的 `JOIN` 负责把两张表的数据查出来；外键负责保证 `posts.user_id` 引用的是一个真实存在的用户。

---

## 五、插入不存在的 user_id 会失败

如果插入一篇文章，但 `user_id` 不存在：

```sql
INSERT INTO demo.posts (user_id, title, content)
VALUES (999, '无效文章', '这个 user_id 不存在');
```

这条语句会失败，因为 `demo.users` 中没有 `id = 999` 的用户。

这就是外键的价值：它可以防止出现“没有作者的文章”这类脏数据。

---

## 六、删除被引用的数据

如果某个用户已经被文章引用，直接删除这个用户通常会失败：

```sql
DELETE FROM demo.users
WHERE id = 1;
```

原因是 `demo.posts.user_id` 还引用着这个用户。

这可以避免误删用户后留下无归属文章。

---

## 七、ON DELETE 行为

外键可以设置删除父表记录时，子表如何处理。

常见行为：

| 行为 | 含义 | 适合场景 |
| --- | --- | --- |
| `ON DELETE RESTRICT` / 默认行为 | 如果子表还引用父表，禁止删除父表记录 | 用户有订单时禁止删除用户 |
| `ON DELETE CASCADE` | 删除父表记录时，自动删除子表相关记录 | 删除文章时自动删除评论 |
| `ON DELETE SET NULL` | 删除父表记录时，把子表外键列设为 NULL | 删除分类后，文章分类变为空 |

### 1. ON DELETE CASCADE

删除用户时自动删除该用户的文章：

```sql
DROP TABLE IF EXISTS demo.posts;

CREATE TABLE demo.posts (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    user_id bigint NOT NULL,
    title text NOT NULL,
    content text NOT NULL,
    created_at timestamptz NOT NULL DEFAULT now(),
    CONSTRAINT posts_user_id_fk
        FOREIGN KEY (user_id)
        REFERENCES demo.users (id)
        ON DELETE CASCADE
);
```

注意：`ON DELETE CASCADE` 很方便，但也很危险。它会自动删除关联数据，生产环境中必须确认这符合业务规则。

### 2. ON DELETE SET NULL

如果希望删除用户后，文章保留，但作者变为空，可以这样设计：

```sql
DROP TABLE IF EXISTS demo.posts;

CREATE TABLE demo.posts (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    user_id bigint,
    title text NOT NULL,
    content text NOT NULL,
    created_at timestamptz NOT NULL DEFAULT now(),
    CONSTRAINT posts_user_id_fk
        FOREIGN KEY (user_id)
        REFERENCES demo.users (id)
        ON DELETE SET NULL
);
```

注意：使用 `ON DELETE SET NULL` 时，外键列不能是 `NOT NULL`，否则删除父表记录时无法把子表外键设为 `NULL`。

## 八、使用 ALTER TABLE 添加外键

外键也可以在建表之后添加。

先创建没有外键的文章表：

```sql
DROP TABLE IF EXISTS demo.posts;

CREATE TABLE demo.posts (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    user_id bigint NOT NULL,
    title text NOT NULL,
    content text NOT NULL,
    created_at timestamptz NOT NULL DEFAULT now()
);
```

后续添加外键：

```sql
ALTER TABLE demo.posts
ADD CONSTRAINT posts_user_id_fk
FOREIGN KEY (user_id)
REFERENCES demo.users (id);
```

注意：如果 `demo.posts` 中已经存在无法匹配 `demo.users.id` 的 `user_id`，添加外键会失败。需要先清理这些脏数据。

---

## 九、外键列通常需要索引

PostgreSQL 会自动为 `PRIMARY KEY` 和 `UNIQUE` 创建索引，但不会总是自动为外键列创建索引。

例如 `demo.posts.user_id` 经常用于查询某个用户的文章：

```sql
SELECT *
FROM demo.posts
WHERE user_id = 1;
```

通常建议为外键列创建索引：

```sql
CREATE INDEX posts_user_id_idx
ON demo.posts (user_id);
```

当前只需要先知道：外键负责保证引用关系，索引负责提升查询性能。索引会在后续阶段详细学习。

## 十、完整练习

```sql
DROP TABLE IF EXISTS demo.posts;
DROP TABLE IF EXISTS demo.users;

CREATE TABLE demo.users (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    username text NOT NULL UNIQUE,
    created_at timestamptz NOT NULL DEFAULT now()
);

CREATE TABLE demo.posts (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    user_id bigint NOT NULL,
    title text NOT NULL,
    content text NOT NULL,
    created_at timestamptz NOT NULL DEFAULT now(),
    CONSTRAINT posts_user_id_fk
        FOREIGN KEY (user_id)
        REFERENCES demo.users (id)
        ON DELETE RESTRICT
);

CREATE INDEX posts_user_id_idx
ON demo.posts (user_id);

INSERT INTO demo.users (username)
VALUES ('zhangsan'), ('lisi')
RETURNING id, username;

INSERT INTO demo.posts (user_id, title, content)
VALUES
    (1, 'PostgreSQL 入门', '这是第一篇文章'),
    (1, '外键约束', '学习 FOREIGN KEY'),
    (2, 'Go 后端', '连接 PostgreSQL');

SELECT
    p.id AS post_id,
    p.title,
    u.username AS author
FROM demo.posts AS p
JOIN demo.users AS u ON u.id = p.user_id
ORDER BY p.id;
```

## 十一、常见误区

### 1. 有了外键就不需要 JOIN 了吗？

不是。

外键负责保证数据引用关系正确，`JOIN` 负责把多张表的数据查询出来。二者解决的问题不同。

### 2. 外键一定要加吗？

不一定，但关系清晰、数据一致性重要的业务建议加。

有些高并发系统会出于性能、分库分表或架构原因不使用数据库外键，而在业务层维护关系。但学习阶段建议先掌握外键，因为它能帮助你建立正确的关系建模思维。

### 3. 所有关联删除都应该用 ON DELETE CASCADE 吗？

不是。

`ON DELETE CASCADE` 会自动删除子表数据，必须确认符合业务规则。比如删除文章时删除评论通常合理；删除用户时自动删除订单通常就很危险。

### 4. 外键和索引是一回事吗？

不是。

外键保证数据引用关系正确，索引提升查询速度。外键列经常用于查询时，通常还应该单独建索引。
