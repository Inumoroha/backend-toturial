# 7. 分页查询：LIMIT/OFFSET 与游标分页

本节目标：掌握 PostgreSQL 中常见分页查询写法，理解 `LIMIT/OFFSET` 的使用方式和局限，并了解基于游标的分页思路。

分页是后端接口中非常常见的能力，例如：

- 文章列表。
- 用户列表。
- 订单列表。
- 评论列表。

---

## 一、LIMIT 和 OFFSET 基础分页

查询第一页，每页 3 条：

```sql
SELECT id, title, created_at
FROM demo.posts
ORDER BY created_at DESC, id DESC
LIMIT 3 OFFSET 0;
```

查询第二页，每页 3 条：

```sql
SELECT id, title, created_at
FROM demo.posts
ORDER BY created_at DESC, id DESC
LIMIT 3 OFFSET 3;
```

分页公式：

```text
OFFSET = (page - 1) * page_size
```

---

## 二、分页必须配合稳定排序

不推荐：

```sql
SELECT id, title
FROM demo.posts
LIMIT 10 OFFSET 0;
```

推荐：

```sql
SELECT id, title, created_at
FROM demo.posts
ORDER BY created_at DESC, id DESC
LIMIT 10 OFFSET 0;
```

原因是：如果没有 `ORDER BY`，数据库不保证返回顺序稳定。

加上 `id DESC` 是为了在 `created_at` 相同的情况下仍然有稳定顺序。

---

## 三、普通分页的后端参数

常见接口参数：

```text
page=1
page_size=20
```

对应 SQL：

```sql
SELECT id, title, created_at
FROM demo.posts
ORDER BY created_at DESC, id DESC
LIMIT 20 OFFSET 0;
```

第 3 页，每页 20 条：

```text
OFFSET = (3 - 1) * 20 = 40
```

```sql
SELECT id, title, created_at
FROM demo.posts
ORDER BY created_at DESC, id DESC
LIMIT 20 OFFSET 40;
```

---

## 四、普通分页的局限

`LIMIT/OFFSET` 简单直观，但有局限：

- 深分页可能变慢。
- 数据变化时，翻页可能重复或漏数据。
- OFFSET 越大，数据库需要跳过的数据越多。

例如：

```sql
LIMIT 20 OFFSET 100000
```

可能需要扫描并跳过大量数据。

---

## 五、游标分页思路

游标分页不使用页码，而是使用上一页最后一条数据的位置继续查。

例如按 `created_at DESC, id DESC` 排序。

第一页：

```sql
SELECT id, title, created_at
FROM demo.posts
ORDER BY created_at DESC, id DESC
LIMIT 3;
```

假设第一页最后一条是：

```text
created_at = 2026-01-03 10:00:00+08
id = 3
```

下一页：

```sql
SELECT id, title, created_at
FROM demo.posts
WHERE (created_at, id) < ('2026-01-03 10:00:00+08'::timestamptz, 3)
ORDER BY created_at DESC, id DESC
LIMIT 3;
```

这表示继续查询排序位置更靠后的数据。

## 六、什么时候用哪种分页

| 场景 | 推荐方式 |
| --- | --- |
| 管理后台页码跳转 | `LIMIT/OFFSET` |
| 数据量小的列表 | `LIMIT/OFFSET` |
| 信息流、滚动加载 | 游标分页 |
| 大数据量深分页 | 游标分页 |
| 要跳到任意页 | `LIMIT/OFFSET` 更方便 |

简单建议：

```text
先掌握 LIMIT/OFFSET。
数据量变大或出现深分页问题时，再使用游标分页。
```

## 七、分页总数统计

很多列表页还需要总数：

```sql
SELECT COUNT(*) AS total_count
FROM demo.posts
WHERE is_published = true;
```

列表数据：

```sql
SELECT id, title, created_at
FROM demo.posts
WHERE is_published = true
ORDER BY created_at DESC, id DESC
LIMIT 10 OFFSET 0;
```

注意：总数统计和列表查询通常是两条 SQL。

## 八、常见误区

### 1. 分页可以不写 ORDER BY 吗？

语法上可以，但不推荐。

分页必须有稳定排序，否则翻页结果不可靠。

### 2. OFFSET 很大也没问题吗？

不一定。

深分页可能变慢，需要关注执行计划和索引。

### 3. 游标分页能跳到第 100 页吗？

不方便。

游标分页更适合“下一页”“加载更多”，不适合任意页码跳转。
