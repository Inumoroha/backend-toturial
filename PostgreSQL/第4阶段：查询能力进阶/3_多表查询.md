# 3. 多表查询：INNER JOIN 与 LEFT JOIN

本节目标：理解多表查询的作用，掌握 PostgreSQL 中常用的 `INNER JOIN`、`LEFT JOIN`，能够基于用户、文章、评论、标签等表写出真实后端常见的关联查询。

前面已经学习了表关系建模，也学习了聚合函数和分组查询。本节重点解决一个后端开发中非常高频的问题：数据分散在多张表里，如何一次查询出业务需要的完整结果。

---

## 一、为什么需要多表查询

关系型数据库通常不会把所有数据都塞进一张大表，而是按业务对象拆成多张表。

例如博客系统中：

- `demo.users` 存用户。
- `demo.user_profiles` 存用户资料。
- `demo.posts` 存文章。
- `demo.comments` 存评论。
- `demo.tags` 存标签。
- `demo.post_tags` 存文章和标签的关联关系。

如果你想查询“文章列表，并显示作者名称”，单查 `demo.posts` 不够，因为作者名称在 `demo.users` 中。

这时就需要 `JOIN`。

```sql
SELECT
    p.id,
    p.title,
    u.username AS author
FROM demo.posts AS p
JOIN demo.users AS u ON u.id = p.user_id;
```

`JOIN` 的核心作用就是：

```text
根据表与表之间的关联条件，把多张表的数据拼到一起查询。
```

## 三、JOIN 的基本结构

常见写法：

```sql
SELECT
    表1.列,
    表2.列
FROM 表1
JOIN 表2 ON 表1.关联列 = 表2.关联列;
```

例如文章表和用户表：

```sql
SELECT
    p.id,
    p.title,
    u.username
FROM demo.posts AS p
JOIN demo.users AS u ON p.user_id = u.id;
```

这里：

- `demo.posts AS p`：给文章表起别名 `p`。
- `demo.users AS u`：给用户表起别名 `u`。
- `ON p.user_id = u.id`：连接条件。

### 为什么要用表别名

如果不使用别名，SQL 会变长：

```sql
SELECT
    demo.posts.id,
    demo.posts.title,
    demo.users.username
FROM demo.posts
JOIN demo.users ON demo.posts.user_id = demo.users.id;
```

使用别名后更清晰：

```sql
SELECT
    p.id,
    p.title,
    u.username
FROM demo.posts AS p
JOIN demo.users AS u ON p.user_id = u.id;
```

真实项目中，多表查询通常都会使用别名。

## 四、INNER JOIN：只返回匹配的数据

`INNER JOIN` 只返回两张表中能匹配上的数据。

在 PostgreSQL 中，下面两种写法等价：

```sql
JOIN demo.users AS u ON p.user_id = u.id
```

```sql
INNER JOIN demo.users AS u ON p.user_id = u.id
```

### 查询文章及作者

```sql
SELECT
    p.id AS post_id,
    p.title,
    u.id AS user_id,
    u.username AS author
FROM demo.posts AS p
INNER JOIN demo.users AS u ON p.user_id = u.id
ORDER BY p.id;
```

结果只会包含有合法作者的文章。

由于 `demo.posts.user_id` 有外键约束，正常情况下每篇文章都能匹配到作者。

### 查询评论、文章和评论作者

```sql
SELECT
    c.id AS comment_id,
    p.title AS post_title,
    u.username AS comment_author,
    c.content,
    c.created_at
FROM demo.comments AS c
INNER JOIN demo.posts AS p ON p.id = c.post_id
INNER JOIN demo.users AS u ON u.id = c.user_id
ORDER BY c.id;
```

这条 SQL 连接了三张表：

- 评论表 `comments`。
- 文章表 `posts`。
- 用户表 `users`。

它适合查询“评论列表，并显示评论所属文章和评论作者”。

## 五、LEFT JOIN：保留左表数据

`LEFT JOIN` 会保留左表的所有行，即使右表没有匹配数据。

它非常适合查询“主体列表”，并附带可选信息。

### 查询所有用户及资料

有些用户可能没有填写资料，如果使用 `INNER JOIN`，这些用户会被过滤掉。

使用 `LEFT JOIN` 可以保留所有用户：

```sql
SELECT
    u.id,
    u.username,
    u.email,
    p.display_name,
    p.bio,
    p.birthday
FROM demo.users AS u
LEFT JOIN demo.user_profiles AS p ON p.user_id = u.id
ORDER BY u.id;
```

如果某个用户没有资料，`display_name`、`bio`、`birthday` 会显示为 `NULL`。

### 查询所有文章及评论数

```sql
SELECT
    p.id,
    p.title,
    COUNT(c.id) AS comment_count
FROM demo.posts AS p
LEFT JOIN demo.comments AS c ON c.post_id = p.id
GROUP BY p.id, p.title
ORDER BY p.id;
```

这里使用 `LEFT JOIN` 是为了保留没有评论的文章。

如果使用 `INNER JOIN`，没有评论的文章会消失。

---

## 六、INNER JOIN 和 LEFT JOIN 的区别

假设左表是文章，右表是评论。

| JOIN 类型 | 是否保留没有评论的文章 | 典型用途 |
| --- | --- | --- |
| `INNER JOIN` | 不保留 | 只查询有评论的文章 |
| `LEFT JOIN` | 保留 | 查询所有文章，并附带评论信息 |

### 只查询有评论的文章

```sql
SELECT DISTINCT
    p.id,
    p.title
FROM demo.posts AS p
INNER JOIN demo.comments AS c ON c.post_id = p.id
ORDER BY p.id;
```

### 查询所有文章，包括没有评论的文章

```sql
SELECT
    p.id,
    p.title,
    c.id AS comment_id,
    c.content
FROM demo.posts AS p
LEFT JOIN demo.comments AS c ON c.post_id = p.id
ORDER BY p.id, c.id;
```

### 查询没有评论的文章

```sql
SELECT
    p.id,
    p.title
FROM demo.posts AS p
LEFT JOIN demo.comments AS c ON c.post_id = p.id
WHERE c.id IS NULL
ORDER BY p.id;
```

这个模式很常见：

```text
LEFT JOIN + WHERE 右表.id IS NULL
```

用于找“没有关联数据”的记录。

## 七、多张表 JOIN

真实业务查询经常需要连接三张或更多表。

### 1. 查询文章详情，包括作者和资料

```sql
SELECT
    p.id AS post_id,
    p.title,
    p.content,
    p.created_at,
    u.id AS author_id,
    u.username AS author_username,
    up.display_name AS author_display_name,
    up.bio AS author_bio
FROM demo.posts AS p
JOIN demo.users AS u ON u.id = p.user_id
LEFT JOIN demo.user_profiles AS up ON up.user_id = u.id
WHERE p.id = 1;
```

这里：

- 文章必须有作者，所以 `posts` 到 `users` 使用 `JOIN`。
- 用户资料可能不存在，所以 `users` 到 `user_profiles` 使用 `LEFT JOIN`。

### 2. 查询评论列表，包括文章标题和评论作者

```sql
SELECT
    c.id AS comment_id,
    c.content,
    c.created_at,
    p.id AS post_id,
    p.title AS post_title,
    u.id AS user_id,
    u.username AS comment_author
FROM demo.comments AS c
JOIN demo.posts AS p ON p.id = c.post_id
JOIN demo.users AS u ON u.id = c.user_id
ORDER BY c.created_at DESC;
```

## 八、多对多查询

多对多关系通常需要通过中间表查询。

文章和标签的关系是：

```text
posts -> post_tags -> tags
```

### 1. 查询文章及其标签

```sql
SELECT
    p.id AS post_id,
    p.title,
    t.id AS tag_id,
    t.name AS tag_name
FROM demo.posts AS p
JOIN demo.post_tags AS pt ON pt.post_id = p.id
JOIN demo.tags AS t ON t.id = pt.tag_id
ORDER BY p.id, t.id;
```

这会返回多行。因为一篇文章有多个标签，所以一篇文章可能出现多次。

### 2. 查询某个标签下的所有文章

```sql
SELECT
    t.name AS tag_name,
    p.id AS post_id,
    p.title
FROM demo.tags AS t
JOIN demo.post_tags AS pt ON pt.tag_id = t.id
JOIN demo.posts AS p ON p.id = pt.post_id
WHERE t.name = 'SQL'
ORDER BY p.id;
```

### 3. 查询文章列表，并把标签聚合成数组

```sql
SELECT
    p.id,
    p.title,
    COALESCE(
        array_agg(t.name ORDER BY t.name) FILTER (WHERE t.name IS NOT NULL),
        ARRAY[]::text[]
    ) AS tags
FROM demo.posts AS p
LEFT JOIN demo.post_tags AS pt ON pt.post_id = p.id
LEFT JOIN demo.tags AS t ON t.id = pt.tag_id
GROUP BY p.id, p.title
ORDER BY p.id;
```

这类写法很适合后端接口返回文章列表。

## 九、JOIN 和 WHERE 的位置

JOIN 的连接条件应该写在 `ON` 中。

过滤条件通常写在 `WHERE` 中。

例如查询已发布文章及作者：

```sql
SELECT
    p.id,
    p.title,
    u.username AS author
FROM demo.posts AS p
JOIN demo.users AS u ON u.id = p.user_id
WHERE p.is_published = true
ORDER BY p.created_at DESC;
```

这里：

- `ON u.id = p.user_id`：说明两张表怎么连接。
- `WHERE p.is_published = true`：过滤只看已发布文章。

### LEFT JOIN 中要特别小心 WHERE

查询所有文章及评论：

```sql
SELECT
    p.id,
    p.title,
    c.content
FROM demo.posts AS p
LEFT JOIN demo.comments AS c ON c.post_id = p.id
ORDER BY p.id;
```

如果你加上：

```sql
WHERE c.content LIKE '%SQL%'
```

就可能把没有评论的文章过滤掉，使 `LEFT JOIN` 看起来像 `INNER JOIN`。

如果你想保留所有文章，但只匹配内容包含 SQL 的评论，可以把条件放进 `ON`：

```sql
SELECT
    p.id,
    p.title,
    c.content
FROM demo.posts AS p
LEFT JOIN demo.comments AS c
    ON c.post_id = p.id
   AND c.content LIKE '%SQL%'
ORDER BY p.id;
```

这个区别在真实业务中非常重要。

## 十、避免列名冲突

多表查询时，不同表可能有同名列。

例如：

- `users.id`
- `posts.id`
- `comments.id`
- `users.created_at`
- `posts.created_at`

所以建议明确写表别名和列别名：

```sql
SELECT
    p.id AS post_id,
    p.created_at AS post_created_at,
    u.id AS user_id,
    u.created_at AS user_created_at
FROM demo.posts AS p
JOIN demo.users AS u ON u.id = p.user_id;
```

不要在复杂多表查询中依赖模糊字段名。

## 十一、常见后端查询示例

### 1. 文章列表页

查询已发布文章，包含作者、评论数、标签数组：

```sql
SELECT
    p.id,
    p.title,
    p.created_at,
    u.username AS author,
    COUNT(DISTINCT c.id) AS comment_count,
    COALESCE(
        array_agg(DISTINCT t.name) FILTER (WHERE t.name IS NOT NULL),
        ARRAY[]::text[]
    ) AS tags
FROM demo.posts AS p
JOIN demo.users AS u ON u.id = p.user_id
LEFT JOIN demo.comments AS c ON c.post_id = p.id
LEFT JOIN demo.post_tags AS pt ON pt.post_id = p.id
LEFT JOIN demo.tags AS t ON t.id = pt.tag_id
WHERE p.is_published = true
GROUP BY p.id, p.title, p.created_at, u.username
ORDER BY p.created_at DESC;
```

注意这里使用了：

```sql
COUNT(DISTINCT c.id)
```

因为文章同时 JOIN 评论和标签时，行数可能被放大。使用 `DISTINCT` 可以避免评论数被重复统计。

### 2. 文章详情页

```sql
SELECT
    p.id,
    p.title,
    p.content,
    p.created_at,
    u.username AS author,
    up.display_name,
    up.bio
FROM demo.posts AS p
JOIN demo.users AS u ON u.id = p.user_id
LEFT JOIN demo.user_profiles AS up ON up.user_id = u.id
WHERE p.id = 1;
```

### 3. 文章评论列表

```sql
SELECT
    c.id,
    c.content,
    c.created_at,
    u.username AS comment_author
FROM demo.comments AS c
JOIN demo.users AS u ON u.id = c.user_id
WHERE c.post_id = 1
ORDER BY c.created_at ASC;
```

## 十二、常见误区

### 1. JOIN 条件可以不写吗？

不建议，也通常是错误的。

如果缺少正确的连接条件，可能产生笛卡尔积，返回大量错误组合数据。

错误示例：

```sql
SELECT *
FROM demo.posts, demo.users;
```

这会把每篇文章和每个用户都组合一遍。

### 2. INNER JOIN 和 LEFT JOIN 可以随便换吗？

不可以。

- `INNER JOIN` 只保留匹配数据。
- `LEFT JOIN` 保留左表所有数据。

是否保留没有关联数据的记录，是选择 JOIN 类型的关键。

### 3. LEFT JOIN 后在 WHERE 中过滤右表字段要小心吗？

要小心。

如果在 `WHERE` 中写右表字段条件，可能会把没有匹配的左表记录过滤掉。

### 4. 多对多为什么需要中间表？

因为一篇文章可以有多个标签，一个标签也可以对应多篇文章。用中间表可以清晰表达这种双向多关系，并且可以添加约束防止重复绑定。

### 5. 为什么多表统计时 COUNT 可能变大？

多个一对多 JOIN 同时出现时，行数可能被放大。

例如文章同时 JOIN 评论和标签，一篇文章有 2 条评论、3 个标签，连接后可能产生 6 行组合。统计时可能需要 `COUNT(DISTINCT ...)`。

## 十三、本节达标标准

学完本节后，你应该能够做到：

- 解释为什么需要多表查询。
- 使用表别名写清晰的 JOIN 查询。
- 使用 `INNER JOIN` 查询两边都匹配的数据。
- 使用 `LEFT JOIN` 保留左表数据。
- 区分 `INNER JOIN` 和 `LEFT JOIN` 的结果差异。
- 查询一对一、一对多、多对多关系中的关联数据。
