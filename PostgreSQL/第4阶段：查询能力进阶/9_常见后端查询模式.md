## 二、列表页查询模式

列表页查询通常有几个特点：

- 返回多条记录。
- 通常需要分页。
- 通常需要排序。
- 通常只返回摘要字段，不返回大段正文。
- 可能附带作者、评论数、标签等关联信息。

### 1. 最基础的文章列表

```sql
SELECT
    id,
    title,
    created_at
FROM demo.posts
WHERE is_published = true
ORDER BY created_at DESC, id DESC
LIMIT 10 OFFSET 0;
```

这个查询适合最简单的文章列表。

### 2. 文章列表带作者

```sql
SELECT
    p.id,
    p.title,
    p.created_at,
    u.username AS author
FROM demo.posts AS p
JOIN demo.users AS u ON u.id = p.user_id
WHERE p.is_published = true
ORDER BY p.created_at DESC, p.id DESC
LIMIT 10 OFFSET 0;
```

### 3. 文章列表带作者、评论数、标签

```sql
SELECT
    p.id,
    p.title,
    p.created_at,
    u.username AS author,
    COUNT(DISTINCT c.id) AS comment_count,
    COALESCE(
        array_agg(DISTINCT t.name) FILTER (WHERE t.name IS NOT NULL),
        ARRAY[]::text[]
    ) AS tags
FROM demo.posts AS p
JOIN demo.users AS u ON u.id = p.user_id
LEFT JOIN demo.comments AS c ON c.post_id = p.id
LEFT JOIN demo.post_tags AS pt ON pt.post_id = p.id
LEFT JOIN demo.tags AS t ON t.id = pt.tag_id
WHERE p.is_published = true
GROUP BY p.id, p.title, p.created_at, u.username
ORDER BY p.created_at DESC, p.id DESC
LIMIT 10 OFFSET 0;
```

注意：这里用了 `COUNT(DISTINCT c.id)`，因为同时 JOIN 评论和标签时，行数可能被放大。

## 三、详情页查询模式

详情页查询通常有几个特点：

- 查询单条主记录。
- 返回字段更完整。
- 可能需要作者信息、标签、评论列表等关联数据。
- 有时会拆成多条 SQL，而不是强行一条 SQL 查完全部内容。

### 1. 查询文章详情和作者信息

```sql
SELECT
    p.id,
    p.title,
    p.content,
    p.created_at,
    u.id AS author_id,
    u.username AS author_username,
    up.display_name AS author_display_name,
    up.bio AS author_bio
FROM demo.posts AS p
JOIN demo.users AS u ON u.id = p.user_id
LEFT JOIN demo.user_profiles AS up ON up.user_id = u.id
WHERE p.id = 1;
```

### 2. 查询文章标签

```sql
SELECT
    t.id,
    t.name
FROM demo.post_tags AS pt
JOIN demo.tags AS t ON t.id = pt.tag_id
WHERE pt.post_id = 1
ORDER BY t.name;
```

### 3. 查询文章评论列表

```sql
SELECT
    c.id,
    c.content,
    c.created_at,
    u.id AS user_id,
    u.username AS comment_author
FROM demo.comments AS c
JOIN demo.users AS u ON u.id = c.user_id
WHERE c.post_id = 1
ORDER BY c.created_at ASC, c.id ASC;
```

### 4. 详情页为什么可以拆成多条 SQL

文章详情接口可以拆成：

1. 查询文章主体和作者。
2. 查询标签。
3. 查询评论列表。

这样通常比一条超复杂 SQL 更容易维护。是否合并成一条 SQL，要看接口需求、性能要求和代码组织方式。

## 四、统计页查询模式

统计页通常用于后台管理、数据看板、运营页面。

常见统计包括：

- 用户总数。
- 文章总数。
- 评论总数。
- 已发布文章数。
- 每个用户文章数。
- 每篇文章评论数。
- 每天新增文章数。

### 1. 全站基础统计

```sql
SELECT
    (SELECT COUNT(*) FROM demo.users) AS user_count,
    (SELECT COUNT(*) FROM demo.posts) AS post_count,
    (SELECT COUNT(*) FROM demo.comments) AS comment_count,
    (SELECT COUNT(*) FROM demo.tags) AS tag_count;
```

这种写法适合统计几个独立总数。

### 2. 已发布和未发布文章数

```sql
SELECT
    COUNT(*) AS total_posts,
    COUNT(*) FILTER (WHERE is_published) AS published_count,
    COUNT(*) FILTER (WHERE NOT is_published) AS draft_count
FROM demo.posts;
```

### 3. 每个用户的文章数

```sql
SELECT
    u.id,
    u.username,
    COUNT(p.id) AS post_count
FROM demo.users AS u
LEFT JOIN demo.posts AS p ON p.user_id = u.id
GROUP BY u.id, u.username
ORDER BY post_count DESC, u.id;
```

### 4. 每天新增文章数

```sql
SELECT
    created_at::date AS post_date,
    COUNT(*) AS post_count
FROM demo.posts
GROUP BY created_at::date
ORDER BY post_date;
```

### 5. 热门文章排行

```sql
SELECT
    p.id,
    p.title,
    s.view_count,
    s.like_count,
    COUNT(c.id) AS comment_count
FROM demo.posts AS p
JOIN demo.post_stats AS s ON s.post_id = p.id
LEFT JOIN demo.comments AS c ON c.post_id = p.id
GROUP BY p.id, p.title, s.view_count, s.like_count
ORDER BY s.view_count DESC, s.like_count DESC
LIMIT 10;
```

## 五、搜索筛选查询模式

搜索筛选通常有几个条件：

- 关键词。
- 状态。
- 作者。
- 标签。
- 时间范围。
- 排序。
- 分页。

### 1. 按关键词搜索文章标题

```sql
SELECT
    p.id,
    p.title,
    p.created_at,
    u.username AS author
FROM demo.posts AS p
JOIN demo.users AS u ON u.id = p.user_id
WHERE p.title ILIKE '%SQL%'
ORDER BY p.created_at DESC, p.id DESC
LIMIT 10 OFFSET 0;
```

`ILIKE` 是 PostgreSQL 中不区分大小写的模糊匹配。

### 2. 按作者筛选

```sql
SELECT
    p.id,
    p.title,
    p.created_at,
    u.username AS author
FROM demo.posts AS p
JOIN demo.users AS u ON u.id = p.user_id
WHERE u.username = 'zhangsan'
ORDER BY p.created_at DESC, p.id DESC;
```

### 3. 按标签筛选

```sql
SELECT
    p.id,
    p.title,
    p.created_at
FROM demo.posts AS p
JOIN demo.post_tags AS pt ON pt.post_id = p.id
JOIN demo.tags AS t ON t.id = pt.tag_id
WHERE t.name = 'SQL'
ORDER BY p.created_at DESC, p.id DESC;
```

### 4. 按时间范围筛选

```sql
SELECT
    id,
    title,
    created_at
FROM demo.posts
WHERE created_at >= '2026-01-01 00:00:00+08'
  AND created_at <  '2026-02-01 00:00:00+08'
ORDER BY created_at DESC, id DESC;
```

时间范围查询通常推荐使用：

```text
>= 开始时间
<  结束时间
```

这样比 `BETWEEN` 更适合处理日期时间边界。

### 5. 多条件组合筛选

查询已发布、标题包含 SQL、作者是 zhangsan 的文章：

```sql
SELECT
    p.id,
    p.title,
    p.created_at,
    u.username AS author
FROM demo.posts AS p
JOIN demo.users AS u ON u.id = p.user_id
WHERE p.is_published = true
  AND p.title ILIKE '%SQL%'
  AND u.username = 'zhangsan'
ORDER BY p.created_at DESC, p.id DESC
LIMIT 10 OFFSET 0;
```

## 六、动态筛选的后端思路

真实后端接口中，筛选条件通常来自用户请求。

例如：

```text
keyword=SQL
author=zhangsan
is_published=true
tag=Backend
page=1
page_size=10
```

SQL 通常不是把所有条件都写死，而是在后端根据参数动态拼接查询条件。

但要注意：动态 SQL 必须使用参数化查询，不能直接拼接用户输入。

危险思路：

```text
WHERE title ILIKE '%用户输入%'
```

安全思路：

```text
WHERE title ILIKE $1
```

然后把参数设置为：

```text
%SQL%
```

参数化查询会在 Go 接入 PostgreSQL 阶段详细学习。

## 七、列表页和详情页的字段选择

列表页不要返回过多字段。

列表页适合返回：

- ID。
- 标题。
- 作者。
- 创建时间。
- 评论数。
- 标签。

详情页适合返回：

- 正文内容。
- 作者详细信息。
- 标签列表。
- 评论列表。

示例：列表页不返回 `content`：

```sql
SELECT
    p.id,
    p.title,
    p.created_at,
    u.username AS author
FROM demo.posts AS p
JOIN demo.users AS u ON u.id = p.user_id
WHERE p.is_published = true
ORDER BY p.created_at DESC, p.id DESC
LIMIT 10 OFFSET 0;
```

详情页再返回 `content`：

```sql
SELECT
    p.id,
    p.title,
    p.content,
    p.created_at
FROM demo.posts AS p
WHERE p.id = 1;
```

这样可以减少列表接口的数据传输和查询开销。

## 八、常见误区

### 1. 列表页可以直接 SELECT * 吗？

不建议。

列表页应该明确选择需要的字段，避免返回大字段和敏感字段。

### 2. 详情页一定要一条 SQL 查完吗？

不一定。

详情页可以拆成多条 SQL，尤其是主体、标签、评论这种不同结构的数据。拆开后代码和 SQL 可能更清晰。

### 3. 搜索都能用 ILIKE 吗？

小数据量可以。

数据量大时，`ILIKE '%keyword%'` 可能很慢。后续可以学习索引、全文搜索或专门搜索引擎。

### 4. 多条件筛选可以直接拼字符串吗？

不建议，也不安全。

用户输入必须参数化，避免 SQL 注入。

### 5. 统计页查询一定要实时查吗？

不一定。

简单统计可以实时查。复杂统计、数据量很大时，可能需要缓存、物化视图、定时任务或专门分析数据库。
