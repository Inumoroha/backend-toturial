# 0_查询能力进阶总览

本阶段目标：在已经掌握基础 `SELECT`、表设计和关系建模之后，进一步学习真实后端业务中高频出现的查询能力，包括聚合统计、分组、JOIN、多表查询、子查询、UPSERT、分页优化和常见查询模式。

如果说第二阶段解决的是“能查出数据”，第三阶段解决的是“能把表设计清楚”，那么第四阶段要解决的是：

```text
能从多张表中查出业务真正需要的数据。
能写统计类 SQL。
能理解 JOIN、GROUP BY、HAVING、子查询这些高频能力。
能把查询结果组织成适合后端接口返回的形状。
```

---

## 一、本阶段适合什么时候学习

在开始本阶段前，建议你已经掌握：

- `SELECT ... FROM ...`
- `WHERE`
- `ORDER BY`
- `LIMIT` / `OFFSET`
- `INSERT` / `UPDATE` / `DELETE`
- 常见字段类型
- 基础约束
- 外键
- 一对一、一对多、多对多关系建模

如果这些内容还不熟，建议先回到前几个阶段练习。查询进阶并不是单纯背 SQL 语法，它依赖你对表结构和业务关系的理解。

---

## 二、本阶段核心内容

本阶段建议拆成以下几个主题学习：

1. 聚合函数：`COUNT`、`SUM`、`AVG`、`MAX`、`MIN`。
2. 分组查询：`GROUP BY`、`HAVING`。
3. 多表查询：`INNER JOIN`、`LEFT JOIN`。
4. 子查询：在 `WHERE`、`FROM`、`SELECT` 中使用子查询。
5. `EXISTS` 与 `IN`：判断关联数据是否存在。
6. UPSERT：`INSERT ... ON CONFLICT ... DO UPDATE`。
7. 分页查询：普通分页与基于游标的分页思路。
8. 常见后端查询模式：列表页、详情页、统计页、搜索筛选。

---

## 三、推荐学习顺序

建议按下面顺序学习：

```text
聚合函数
→ GROUP BY / HAVING
→ INNER JOIN / LEFT JOIN
→ 子查询
→ EXISTS / IN
→ UPSERT
→ 分页与常见查询模式
```

原因是：

- 聚合和分组是统计查询的基础。
- JOIN 是多表查询的基础。
- 子查询和 EXISTS 建立在单表、多表查询之上。
- UPSERT 是真实业务里很常见的写入模式。
- 分页和查询模式更接近后端接口设计，需要前面的能力支撑。

## 六、本阶段会学到的典型查询

### 1. 查询每个用户发布了多少篇文章

```sql
SELECT
    u.id,
    u.username,
    COUNT(p.id) AS post_count
FROM demo.users AS u
LEFT JOIN demo.posts AS p ON p.user_id = u.id
GROUP BY u.id, u.username
ORDER BY u.id;
```

### 2. 查询每篇文章的评论数

```sql
SELECT
    p.id,
    p.title,
    COUNT(c.id) AS comment_count
FROM demo.posts AS p
LEFT JOIN demo.comments AS c ON c.post_id = p.id
GROUP BY p.id, p.title
ORDER BY p.id;
```

### 3. 查询文章列表，带作者和标签

```sql
SELECT
    p.id,
    p.title,
    u.username AS author,
    array_agg(t.name ORDER BY t.name) AS tags
FROM demo.posts AS p
JOIN demo.users AS u ON u.id = p.user_id
LEFT JOIN demo.post_tags AS pt ON pt.post_id = p.id
LEFT JOIN demo.tags AS t ON t.id = pt.tag_id
WHERE p.is_published = true
GROUP BY p.id, p.title, u.username
ORDER BY p.created_at DESC;
```

### 4. 查询没有评论的文章

```sql
SELECT
    p.id,
    p.title
FROM demo.posts AS p
LEFT JOIN demo.comments AS c ON c.post_id = p.id
WHERE c.id IS NULL;
```

### 5. 插入或更新标签

```sql
INSERT INTO demo.tags (name)
VALUES ('PostgreSQL')
ON CONFLICT (name)
DO UPDATE SET name = EXCLUDED.name
RETURNING id, name;
```

这些 SQL 看起来比前几阶段复杂，但它们都可以拆成几个基础能力：

```text
SELECT
+ JOIN
+ WHERE
+ GROUP BY
+ ORDER BY
+ 聚合函数
```

## 七、本阶段常见误区

### 1. JOIN 不是越多越高级

JOIN 是多表查询的工具，不是炫技。能用清晰表结构和明确关系解决问题，才是重点。

### 2. GROUP BY 不是只能配 COUNT

`GROUP BY` 可以配合很多聚合函数，例如：

- `COUNT`
- `SUM`
- `AVG`
- `MAX`
- `MIN`
- `array_agg`
- `json_agg`

### 3. LEFT JOIN 和 INNER JOIN 不能混用着猜

`INNER JOIN` 只返回两边都匹配的数据。

`LEFT JOIN` 会保留左表数据，即使右表没有匹配。

后续教程会专门讲它们的区别。

### 4. 分页不只是 LIMIT 和 OFFSET

`LIMIT` / `OFFSET` 是最容易理解的分页方式，但在大数据量、深分页场景下可能会变慢。后续会介绍基于游标的分页思路。

### 5. 子查询不是越多越好

子查询很有用，但复杂查询要注意可读性和性能。很多子查询也可以改写成 JOIN 或 CTE。

## 八、本阶段达标标准

学完第四阶段后，你应该能够做到：

- 使用聚合函数完成基础统计。
- 使用 `GROUP BY` 按用户、文章、标签等维度分组。
- 使用 `HAVING` 过滤分组后的结果。
- 区分 `INNER JOIN` 和 `LEFT JOIN`。
- 写出用户、文章、评论、标签之间的多表查询。
- 使用子查询和 `EXISTS` 判断关联数据是否存在。
- 使用 `ON CONFLICT` 处理插入冲突。
- 写出常见后端接口需要的列表查询、详情查询和统计查询。
- 理解普通分页的写法和局限。

完成本阶段后，你的 SQL 能力会从“能查单表”提升到“能写真实业务查询”。这也是后端开发中非常关键的一步。
