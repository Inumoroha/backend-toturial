# 2. 分组查询：GROUP BY 与 HAVING

本节目标：理解分组查询的作用，掌握 `GROUP BY` 和 `HAVING` 的用法，能够按用户、文章、标签等维度统计数据，并能区分 `WHERE` 和 `HAVING` 的使用场景。

上一节学习了聚合函数，例如 `COUNT`、`SUM`、`AVG`、`MAX`、`MIN`。如果只使用聚合函数，通常只能得到整张表的总体统计。本节的 `GROUP BY` 可以让你“按某个维度分别统计”。

---

## 一、GROUP BY 是什么

`GROUP BY` 用来把数据按某些列分组，然后对每一组分别执行聚合函数。

例如：

```text
统计所有文章数量：不分组。
统计每个用户发布了多少篇文章：按用户分组。
统计每个标签下有多少篇文章：按标签分组。
统计每天新增了多少评论：按日期分组。
```

没有 `GROUP BY`：

```sql
SELECT COUNT(*) AS post_count
FROM demo.posts;
```

结果是一行，表示所有文章数量。

有 `GROUP BY`：

```sql
SELECT user_id, COUNT(*) AS post_count
FROM demo.posts
GROUP BY user_id;
```

结果是多行，每个用户一行，表示每个用户的文章数量。

## 三、按单列分组

### 1. 按用户统计文章数

```sql
SELECT
    user_id,
    COUNT(*) AS post_count
FROM demo.posts
GROUP BY user_id
ORDER BY user_id;
```

这条 SQL 的含义：

- 按 `user_id` 把文章分组。
- 每组统计文章数量。
- 每个用户返回一行统计结果。

### 2. 只统计已发布文章

```sql
SELECT
    user_id,
    COUNT(*) AS published_post_count
FROM demo.posts
WHERE is_published = true
GROUP BY user_id
ORDER BY user_id;
```

这里的执行逻辑可以简单理解为：

```text
先 WHERE 过滤已发布文章。
再 GROUP BY 按 user_id 分组。
最后 COUNT 统计每组数量。
```

### 3. 按发布状态统计文章数

```sql
SELECT
    is_published,
    COUNT(*) AS post_count
FROM demo.posts
GROUP BY is_published
ORDER BY is_published;
```

这会得到：

- 已发布文章数量。
- 未发布文章数量。

## 四、按多列分组

有时一个维度不够，需要按多个字段组合分组。

例如统计每个用户在每种发布状态下的文章数量：

```sql
SELECT
    user_id,
    is_published,
    COUNT(*) AS post_count
FROM demo.posts
GROUP BY user_id, is_published
ORDER BY user_id, is_published;
```

含义：

```text
同一个 user_id + 同一个 is_published 组成一组。
```

例如：

```text
用户 1 的已发布文章
用户 1 的未发布文章
用户 2 的已发布文章
```

这类查询常见于管理后台统计，例如：

- 每个用户不同状态的订单数量。
- 每个分类不同状态的商品数量。
- 每天每种事件类型的日志数量。

## 五、GROUP BY 配合 JOIN

真实业务中，经常不是只返回 ID，而是要返回更可读的名称。

### 1. 统计每个用户的文章数，并显示用户名

```sql
SELECT
    u.id,
    u.username,
    COUNT(p.id) AS post_count
FROM demo.users AS u
LEFT JOIN demo.posts AS p ON p.user_id = u.id
GROUP BY u.id, u.username
ORDER BY u.id;
```

这里使用 `LEFT JOIN`，是为了保留没有文章的用户。

为什么使用 `COUNT(p.id)`？

因为 `LEFT JOIN` 会保留左表用户，即使没有文章。如果使用 `COUNT(*)`，没有文章的用户也可能被统计成 1。使用 `COUNT(p.id)` 只统计真实存在的文章。

### 2. 统计每篇文章的评论数

```sql
SELECT
    p.id,
    p.title,
    COUNT(c.id) AS comment_count
FROM demo.posts AS p
LEFT JOIN demo.comments AS c ON c.post_id = p.id
GROUP BY p.id, p.title
ORDER BY p.id;
```

### 3. 统计每个标签下的文章数

```sql
SELECT
    t.id,
    t.name,
    COUNT(pt.post_id) AS post_count
FROM demo.tags AS t
LEFT JOIN demo.post_tags AS pt ON pt.tag_id = t.id
GROUP BY t.id, t.name
ORDER BY t.id;
```

这类查询是多对多关系中非常常见的统计查询。

## 六、HAVING：过滤分组后的结果

`WHERE` 用来过滤分组前的行。

`HAVING` 用来过滤分组后的结果。

### 1. 查询文章数大于 1 的用户

```sql
SELECT
    u.id,
    u.username,
    COUNT(p.id) AS post_count
FROM demo.users AS u
LEFT JOIN demo.posts AS p ON p.user_id = u.id
GROUP BY u.id, u.username
HAVING COUNT(p.id) > 1
ORDER BY u.id;
```

这里不能用：

```sql
WHERE COUNT(p.id) > 1
```

因为 `WHERE` 执行时，分组和聚合还没有完成。

### 2. 查询评论数大于等于 2 的文章

```sql
SELECT
    p.id,
    p.title,
    COUNT(c.id) AS comment_count
FROM demo.posts AS p
LEFT JOIN demo.comments AS c ON c.post_id = p.id
GROUP BY p.id, p.title
HAVING COUNT(c.id) >= 2
ORDER BY p.id;
```

### 3. 查询总浏览量大于 100 的作者

```sql
SELECT
    u.id,
    u.username,
    SUM(s.view_count) AS total_views
FROM demo.users AS u
JOIN demo.posts AS p ON p.user_id = u.id
JOIN demo.post_stats AS s ON s.post_id = p.id
GROUP BY u.id, u.username
HAVING SUM(s.view_count) > 100
ORDER BY total_views DESC;
```

## 七、WHERE 和 HAVING 的区别

| 子句 | 执行位置 | 作用 | 能不能使用聚合函数 |
| --- | --- | --- | --- |
| `WHERE` | 分组前 | 过滤原始行 | 通常不能 |
| `HAVING` | 分组后 | 过滤分组结果 | 可以 |

示例：统计已发布文章中，文章数大于 1 的用户。

```sql
SELECT
    u.id,
    u.username,
    COUNT(p.id) AS published_post_count
FROM demo.users AS u
JOIN demo.posts AS p ON p.user_id = u.id
WHERE p.is_published = true
GROUP BY u.id, u.username
HAVING COUNT(p.id) > 1
ORDER BY u.id;
```

这条 SQL 中：

- `WHERE p.is_published = true`：先只保留已发布文章。
- `GROUP BY u.id, u.username`：再按用户分组。
- `HAVING COUNT(p.id) > 1`：最后只保留已发布文章数大于 1 的用户。

## 八、GROUP BY 中普通列的规则

使用聚合函数时，如果 `SELECT` 中出现普通列，那么这些普通列通常必须出现在 `GROUP BY` 中。

错误示例：

```sql
SELECT
    user_id,
    title,
    COUNT(*) AS post_count
FROM demo.posts
GROUP BY user_id;
```

这通常会报错，因为 `title` 既没有被聚合，也没有出现在 `GROUP BY` 中。

正确写法之一：

```sql
SELECT
    user_id,
    COUNT(*) AS post_count
FROM demo.posts
GROUP BY user_id;
```

如果你确实想按 `user_id` 和 `title` 组合分组：

```sql
SELECT
    user_id,
    title,
    COUNT(*) AS post_count
FROM demo.posts
GROUP BY user_id, title;
```

简单记忆：

```text
SELECT 中的普通列，要么放进 GROUP BY，要么放进聚合函数。
```

PostgreSQL 在某些主键函数依赖场景下会更智能，但初学阶段先按这条规则写，最稳。

## 九、按日期分组

统计类查询经常需要按天、按月统计。

### 1. 按日期统计文章数量

```sql
SELECT
    created_at::date AS post_date,
    COUNT(*) AS post_count
FROM demo.posts
GROUP BY created_at::date
ORDER BY post_date;
```

`created_at::date` 表示把时间转换成日期。

### 2. 使用 date_trunc 按天统计

```sql
SELECT
    date_trunc('day', created_at) AS post_day,
    COUNT(*) AS post_count
FROM demo.posts
GROUP BY date_trunc('day', created_at)
ORDER BY post_day;
```

### 3. 按月统计

```sql
SELECT
    date_trunc('month', created_at) AS post_month,
    COUNT(*) AS post_count
FROM demo.posts
GROUP BY date_trunc('month', created_at)
ORDER BY post_month;
```

`date_trunc` 在报表统计中很常用。

## 十、聚合成数组

PostgreSQL 支持 `array_agg`，可以把一组数据聚合成数组。

查询每篇文章的标签数组：

```sql
SELECT
    p.id,
    p.title,
    array_agg(t.name ORDER BY t.name) AS tags
FROM demo.posts AS p
LEFT JOIN demo.post_tags AS pt ON pt.post_id = p.id
LEFT JOIN demo.tags AS t ON t.id = pt.tag_id
GROUP BY p.id, p.title
ORDER BY p.id;
```

如果某篇文章没有标签，`array_agg` 可能得到包含 NULL 的结果。可以使用 `FILTER` 过滤掉 NULL。

```sql
SELECT
    p.id,
    p.title,
    COALESCE(
        array_agg(t.name ORDER BY t.name) FILTER (WHERE t.name IS NOT NULL),
        ARRAY[]::text[]
    ) AS tags
FROM demo.posts AS p
LEFT JOIN demo.post_tags AS pt ON pt.post_id = p.id
LEFT JOIN demo.tags AS t ON t.id = pt.tag_id
GROUP BY p.id, p.title
ORDER BY p.id;
```

这类查询很适合后端接口返回文章列表。

## 十二、常见误区

### 1. WHERE 和 HAVING 可以随便换吗？

不可以。

`WHERE` 过滤分组前的数据，`HAVING` 过滤分组后的结果。

### 2. 为什么 SELECT 中的列必须写进 GROUP BY？

因为分组后，一组里可能有多行数据。数据库需要知道普通列应该如何合并。

简单规则：

```text
SELECT 中的普通列，要么放进 GROUP BY，要么放进聚合函数。
```

### 3. LEFT JOIN 后为什么 COUNT(*) 结果不对？

`LEFT JOIN` 会保留左表行。统计右表数量时，通常应该使用右表的非空主键：

```sql
COUNT(c.id)
```

而不是：

```sql
COUNT(*)
```

### 4. GROUP BY 会自动排序吗？

不会保证。

如果需要稳定顺序，必须显式写 `ORDER BY`。

### 5. HAVING 能不用就不用吗？

不是。

只要你要过滤聚合结果，例如“文章数大于 1 的用户”，就应该使用 `HAVING`。
