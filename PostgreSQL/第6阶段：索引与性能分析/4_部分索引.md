# 4. 部分索引

本节目标：理解部分索引的作用，掌握 `CREATE INDEX ... WHERE ...`，知道它适合哪些场景，以及为什么查询条件必须能匹配索引条件。

部分索引是：

```text
只给表中满足某个条件的一部分行建立索引。
```

普通索引：

```sql
CREATE INDEX perf_orders_status_idx
ON demo.perf_orders (status);
```

会包含整张表的数据。

部分索引：

```sql
CREATE INDEX perf_orders_pending_created_at_idx
ON demo.perf_orders (created_at)
WHERE status = 'pending';
```

只包含 `status = 'pending'` 的订单。

---

## 一、为什么需要部分索引

有些业务只经常查询一小部分数据。

例如订单状态：

```text
paid：大量历史订单。
cancelled：一些取消订单。
refunded：一些退款订单。
pending：少量待处理订单。
```

后台任务经常查待处理订单：

```sql
SELECT id, order_no, user_id, created_at
FROM demo.perf_orders
WHERE status = 'pending'
ORDER BY created_at ASC
LIMIT 100;
```

如果只给 `status` 建普通索引，索引包含所有状态。

如果业务只关心 `pending` 队列，可以创建部分索引：

```sql
CREATE INDEX perf_orders_pending_created_at_idx
ON demo.perf_orders (created_at ASC)
WHERE status = 'pending';
```

它更小、更聚焦，维护成本也可能更低。

---

## 二、创建部分索引

语法：

```sql
CREATE INDEX 索引名
ON 表名 (索引字段)
WHERE 条件;
```

例子：

```sql
CREATE INDEX perf_orders_pending_created_at_idx
ON demo.perf_orders (created_at ASC)
WHERE status = 'pending';
```

查询：

```sql
EXPLAIN
SELECT id, order_no, user_id, created_at
FROM demo.perf_orders
WHERE status = 'pending'
ORDER BY created_at ASC
LIMIT 100;
```

可能使用这个部分索引。

---

## 三、部分索引适合的场景

### 1. 只查询某个状态的数据

例如：

```sql
WHERE status = 'pending'
```

适合：

```sql
CREATE INDEX perf_orders_pending_created_at_idx
ON demo.perf_orders (created_at)
WHERE status = 'pending';
```

### 2. 软删除

很多系统使用软删除：

```text
deleted_at IS NULL 表示未删除。
deleted_at IS NOT NULL 表示已删除。
```

常见查询只查未删除数据：

```sql
WHERE deleted_at IS NULL
```

可以创建：

```sql
CREATE INDEX users_active_email_idx
ON demo.users (email)
WHERE deleted_at IS NULL;
```

这样索引只包含未删除用户。

### 3. 已发布内容列表

如果系统经常查已发布文章：

```sql
WHERE is_published = true
ORDER BY created_at DESC
```

可以：

```sql
CREATE INDEX perf_posts_published_created_at_idx
ON demo.perf_posts (created_at DESC)
WHERE is_published = true;
```

### 4. 稀有条件

如果某个条件只占表中很小比例，例如异常状态、待审核状态、失败任务，部分索引通常很有价值。

---

## 四、查询条件必须匹配部分索引条件

部分索引能不能用，一个关键点是：

```text
查询条件必须能让 PostgreSQL 判断：这条查询只会访问部分索引覆盖的数据。
```

索引：

```sql
CREATE INDEX perf_orders_pending_created_at_idx
ON demo.perf_orders (created_at)
WHERE status = 'pending';
```

能匹配：

```sql
SELECT *
FROM demo.perf_orders
WHERE status = 'pending'
ORDER BY created_at
LIMIT 100;
```

不适合：

```sql
SELECT *
FROM demo.perf_orders
WHERE status IN ('pending', 'paid')
ORDER BY created_at
LIMIT 100;
```

因为这个查询还需要 `paid` 订单，而部分索引里没有 `paid`。

---

## 五、部分唯一索引

部分索引也可以是唯一索引。

这是非常实用的能力。

例如用户表使用软删除，希望：

```text
未删除用户中，email 不能重复。
已经删除的历史用户，不参与唯一性判断。
```

可以：

```sql
CREATE TABLE demo.soft_delete_users (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    email text NOT NULL,
    username text NOT NULL,
    deleted_at timestamptz
);

CREATE UNIQUE INDEX soft_delete_users_email_active_uidx
ON demo.soft_delete_users (email)
WHERE deleted_at IS NULL;
```

这样：

- `deleted_at IS NULL` 的用户，email 必须唯一。
- `deleted_at IS NOT NULL` 的历史数据，不受这个唯一索引约束。

这比把软删除数据物理删除更灵活，也比只靠应用代码判断更可靠。

---

## 六、部分索引和组合索引结合

部分索引可以结合多个索引字段。

例如待处理订单队列：

```sql
CREATE INDEX perf_orders_pending_user_created_at_idx
ON demo.perf_orders (user_id, created_at ASC)
WHERE status = 'pending';
```

适合：

```sql
SELECT id, order_no, created_at
FROM demo.perf_orders
WHERE status = 'pending'
  AND user_id = 12345
ORDER BY created_at ASC
LIMIT 50;
```

部分索引的 `WHERE` 条件负责缩小索引范围，索引字段负责在这部分数据里继续过滤和排序。

---

## 七、部分索引的成本

部分索引通常比全量索引小，但不是没有成本。

当插入或更新数据时，PostgreSQL 需要判断这行是否满足部分索引条件。

例如：

```sql
UPDATE demo.perf_orders
SET status = 'pending'
WHERE id = 1;
```

如果这行从不满足条件变成满足条件，PostgreSQL 需要把它加入部分索引。

反过来：

```sql
UPDATE demo.perf_orders
SET status = 'paid'
WHERE id = 1;
```

如果这行从 `pending` 变成 `paid`，PostgreSQL 需要从部分索引中移除对应条目。

所以部分索引仍然要围绕真实高频查询设计。

---

## 八、常见误区

### 1. 部分索引是不是只适合小表？

不是。

部分索引在大表上往往更有价值，因为它可以只索引高频访问的小部分数据。

### 2. 有了部分索引，所有查询都能用它吗？

不能。

只有查询条件能匹配部分索引 `WHERE` 条件时，才可能使用。

### 3. 部分索引的 WHERE 条件可以随便写吗？

不建议。

条件应该稳定、清晰、和高频查询一致。过于复杂或不稳定的条件会增加理解和维护成本。

### 4. 部分索引一定比组合索引好？

不一定。

部分索引和组合索引解决的问题不同，也可以结合使用。要根据查询模式和执行计划判断。

---

## 九、本节达标标准

学完本节后，你应该能够做到：

- 使用 `CREATE INDEX ... WHERE ...` 创建部分索引。
- 解释部分索引只索引满足条件的一部分数据。
- 知道部分索引适合待处理状态、软删除、已发布内容等场景。
- 知道查询条件必须能匹配部分索引条件。
- 创建部分唯一索引。
- 理解部分索引仍然有写入维护成本。
