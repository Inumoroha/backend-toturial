# 0. 索引与性能分析总览

本阶段目标：能判断查询为什么慢，并知道如何优化。

前面几个阶段已经学习了：

```text
如何建表。
如何写查询。
如何处理事务和并发。
```

第六阶段要解决的是另一个后端开发中非常常见的问题：

```text
为什么这条 SQL 在数据少时很快，数据多了以后变慢？
为什么我建了索引，PostgreSQL 还是不用？
EXPLAIN 输出里 Seq Scan、Index Scan、Bitmap Index Scan 分别是什么意思？
应该加什么索引，而不是乱加一堆索引？
```

索引和性能分析不是背几个语法就结束。真正重要的是建立判断能力：

```text
先看查询条件。
再看数据量和选择性。
再看执行计划。
最后决定是否需要索引、需要什么索引。
```

---

## 一、为什么要学索引

没有索引时，数据库查找数据最朴素的办法是：

```text
从表的第一行开始，一行一行检查，直到找出符合条件的数据。
```

这就是顺序扫描。

如果表只有几十行，顺序扫描很快。

如果表有几百万行，查询某个邮箱：

```sql
SELECT *
FROM demo.users
WHERE email = 'user_100000@example.com';
```

还要一行一行找，就会很慢。

索引的作用可以理解为：

```text
给数据建立一份便于查找的目录。
```

有了合适索引后，数据库可以先在索引里快速定位，再去表里取数据。

---

## 二、本阶段核心内容

本阶段建议拆成以下主题学习：

1. 性能分析总览和练习数据准备。
2. B-Tree 索引：最常用的索引类型。
3. 唯一索引：既能加速查询，也能保证唯一性。
4. 组合索引：多个字段一起建索引，以及最左前缀思想。
5. 部分索引：只给满足条件的一部分数据建索引。
6. 索引没有被使用的常见原因。
7. `EXPLAIN`：查看 PostgreSQL 计划怎么执行 SQL。
8. `EXPLAIN ANALYZE`：查看 SQL 实际怎么执行。
9. 常见扫描方式：`Seq Scan`、`Index Scan`、`Bitmap Index Scan`。
10. 慢查询优化流程：从业务问题到执行计划，再到索引设计。

---

## 三、推荐学习顺序

建议按下面顺序学习：

```text
准备数据和建立性能直觉
-> B-Tree 索引
-> 唯一索引
-> 组合索引
-> 部分索引
-> 索引没有被使用的原因
-> EXPLAIN
-> EXPLAIN ANALYZE
-> Seq Scan / Index Scan / Bitmap Index Scan
-> 慢查询优化流程
```

原因是：

- 先理解索引解决什么问题。
- 再学习几种最常用的索引设计。
- 然后学习为什么索引可能没有被用上。
- 最后学习执行计划，用证据判断 SQL 的执行方式。

## 四、本阶段统一练习模型

本阶段使用一个偏真实后端业务的练习模型：

- `demo.perf_users`：用户表。
- `demo.perf_posts`：文章表。
- `demo.perf_orders`：订单表。

它们覆盖了常见查询：

- 根据邮箱查用户。
- 根据用户名模糊搜索。
- 查询某个用户的文章列表。
- 按发布时间排序。
- 查询某个订单号。
- 查询某个用户的已支付订单。
- 查询待处理订单。
- 做分页和列表筛选。

## 五、初始化练习数据

创建表：

```sql
CREATE SCHEMA IF NOT EXISTS demo;

DROP TABLE IF EXISTS demo.perf_orders;
DROP TABLE IF EXISTS demo.perf_posts;
DROP TABLE IF EXISTS demo.perf_users;

CREATE TABLE demo.perf_users (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    username text NOT NULL,
    email text NOT NULL,
    status text NOT NULL DEFAULT 'active',
    created_at timestamptz NOT NULL DEFAULT now(),
    CONSTRAINT perf_users_status_check
        CHECK (status IN ('active', 'disabled', 'deleted'))
);

CREATE TABLE demo.perf_posts (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    user_id bigint NOT NULL,
    title text NOT NULL,
    content text NOT NULL,
    is_published boolean NOT NULL DEFAULT false,
    created_at timestamptz NOT NULL DEFAULT now(),
    CONSTRAINT perf_posts_user_id_fk
        FOREIGN KEY (user_id)
        REFERENCES demo.perf_users (id)
);

CREATE TABLE demo.perf_orders (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    order_no text NOT NULL,
    user_id bigint NOT NULL,
    status text NOT NULL,
    total_amount numeric(12, 2) NOT NULL,
    created_at timestamptz NOT NULL DEFAULT now(),
    paid_at timestamptz,
    CONSTRAINT perf_orders_user_id_fk
        FOREIGN KEY (user_id)
        REFERENCES demo.perf_users (id),
    CONSTRAINT perf_orders_status_check
        CHECK (status IN ('pending', 'paid', 'cancelled', 'refunded'))
);
```

插入练习数据：

```sql
INSERT INTO demo.perf_users (username, email, status, created_at)
SELECT
    'user_' || gs,
    'user_' || gs || '@example.com',
    CASE
        WHEN gs % 100 = 0 THEN 'deleted'
        WHEN gs % 20 = 0 THEN 'disabled'
        ELSE 'active'
    END,
    now() - (gs || ' minutes')::interval
FROM generate_series(1, 100000) AS gs;

INSERT INTO demo.perf_posts (user_id, title, content, is_published, created_at)
SELECT
    ((gs - 1) % 100000) + 1,
    'Post title ' || gs,
    'Post content ' || gs,
    gs % 3 <> 0,
    now() - (gs || ' minutes')::interval
FROM generate_series(1, 300000) AS gs;

INSERT INTO demo.perf_orders (order_no, user_id, status, total_amount, created_at, paid_at)
SELECT
    'ORD' || lpad(gs::text, 12, '0'),
    ((gs - 1) % 100000) + 1,
    CASE
        WHEN gs % 20 = 0 THEN 'refunded'
        WHEN gs % 10 = 0 THEN 'cancelled'
        WHEN gs % 3 = 0 THEN 'pending'
        ELSE 'paid'
    END,
    (10 + (gs % 500))::numeric(12, 2),
    now() - (gs || ' minutes')::interval,
    CASE
        WHEN gs % 3 <> 0 THEN now() - (gs || ' minutes')::interval + interval '5 minutes'
        ELSE NULL
    END
FROM generate_series(1, 500000) AS gs;
```

更新统计信息：

```sql
ANALYZE demo.perf_users;
ANALYZE demo.perf_posts;
ANALYZE demo.perf_orders;
```

`ANALYZE` 会让 PostgreSQL 收集表数据分布统计信息。执行计划依赖统计信息，所以准备性能练习时建议执行它。

## 六、本阶段会反复使用的 SQL

### 1. 根据邮箱查用户

```sql
SELECT id, username, email
FROM demo.perf_users
WHERE email = 'user_50000@example.com';
```

### 2. 查询某个用户的文章列表

```sql
SELECT id, user_id, title, created_at
FROM demo.perf_posts
WHERE user_id = 100
ORDER BY created_at DESC
LIMIT 20;
```

### 3. 查询某个订单号

```sql
SELECT id, order_no, user_id, status, total_amount
FROM demo.perf_orders
WHERE order_no = 'ORD000000100000';
```

### 4. 查询待处理订单

```sql
SELECT id, order_no, user_id, created_at
FROM demo.perf_orders
WHERE status = 'pending'
ORDER BY created_at ASC
LIMIT 50;
```

### 5. 查询某个用户的已支付订单

```sql
SELECT id, order_no, total_amount, paid_at
FROM demo.perf_orders
WHERE user_id = 12345
  AND status = 'paid'
ORDER BY paid_at DESC
LIMIT 20;
```

## 七、索引不是越多越好

索引能提升查询性能，但也有成本。

### 1. 索引会占用磁盘

索引本身是一份额外的数据结构。表越大，索引也可能越大。

### 2. 索引会增加写入成本

每次插入、更新、删除数据时，PostgreSQL 不仅要修改表数据，还要维护相关索引。

例如 `demo.perf_orders` 上有 5 个索引，那么插入一条订单时，可能要同时维护这 5 个索引。

### 3. 索引会增加规划选择成本

索引太多时，优化器需要考虑更多可能的执行路径。通常这不是最大问题，但也说明索引设计应该有目的。

核心原则是：

```text
为高频、重要、选择性较好的查询建立合适索引。
不要因为“可能会用到”就给所有字段都建索引。
```

## 八、性能优化的基本流程

遇到慢查询时，不要第一反应就是加索引。

更好的流程是：

```text
1. 明确慢的是哪条 SQL。
2. 明确数据量有多大。
3. 使用 EXPLAIN 查看执行计划。
4. 使用 EXPLAIN ANALYZE 查看真实执行情况。
5. 判断慢在哪里：扫描太多行、排序太重、JOIN 太贵、过滤条件不合适。
6. 根据查询模式设计索引。
7. 再次 EXPLAIN ANALYZE 对比优化前后。
```

本阶段所有教程都会围绕这个流程展开。

## 九、本阶段常见误区

### 1. 建了索引，查询一定会变快？

不一定。

如果查询返回表中大部分数据，顺序扫描可能比走索引更快。

### 2. EXPLAIN 看到 Seq Scan 就一定有问题？

不一定。

小表或返回大量数据时，`Seq Scan` 很正常。

### 3. 索引失效是不是索引坏了？

通常不是。

很多时候是优化器判断“不用这个索引更划算”。更准确的说法是“索引没有被使用”。

### 4. 组合索引里字段顺序无所谓？

不是。

组合索引字段顺序非常重要，会影响它能支持哪些查询。

## 十、本阶段达标标准

学完第六阶段后，你应该能够做到：

- 解释 B-Tree 索引适合哪些查询。
- 创建普通索引、唯一索引、组合索引、部分索引。
- 理解组合索引字段顺序的重要性。
- 解释为什么某些 SQL 没有使用索引。
- 使用 `EXPLAIN` 查看执行计划。
- 使用 `EXPLAIN ANALYZE` 查看真实执行时间和行数。
- 看懂 `Seq Scan`、`Index Scan`、`Bitmap Index Scan` 的基本含义。
- 能根据执行计划判断一个查询慢在哪里。
- 能为常见后端查询设计合适索引。

第六阶段学完后，你不只是“知道索引能加速”，而是能开始用证据分析 SQL 性能。这是从会写 SQL 到会维护真实数据库的重要一步。
