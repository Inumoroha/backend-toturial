# 5. 索引没有被使用的常见原因

本节目标：理解“索引失效”的常见原因，知道在 PostgreSQL 中更准确的说法通常是“优化器没有选择使用索引”，并能根据 SQL 写法和数据分布排查问题。

很多教程会说“索引失效”。这个说法容易让人误解为索引坏了。

在 PostgreSQL 中，更准确的理解是：

```text
索引存在，但优化器判断使用它不划算，或者当前 SQL 写法无法有效利用它。
```

---

## 一、返回数据太多

索引最适合快速找到少量数据。

如果查询要返回表中大部分行，顺序扫描可能更快。

例如：

```sql
SELECT id, username, email
FROM demo.perf_users
WHERE status = 'active';
```

如果 95% 用户都是 `active`，即使 `status` 上有索引：

```sql
CREATE INDEX perf_users_status_idx
ON demo.perf_users (status);
```

PostgreSQL 也可能选择 `Seq Scan`。

原因是：

```text
走索引找到大量行，再回表读取大量数据，可能比直接顺序扫描整张表更慢。
```

这不是索引坏了，而是优化器认为不用索引更便宜。

---

## 二、选择性太差

选择性表示一个条件能过滤掉多少数据。

选择性好：

```sql
WHERE email = 'user_50000@example.com'
```

通常只返回 1 行。

选择性差：

```sql
WHERE is_published = true
```

如果大部分文章都已发布，过滤效果就不明显。

选择性差的字段不一定不能建索引，但要更谨慎。

如果查询总是：

```sql
WHERE is_published = true
ORDER BY created_at DESC
LIMIT 20
```

可以考虑组合索引或部分索引：

```sql
CREATE INDEX perf_posts_published_created_at_idx
ON demo.perf_posts (created_at DESC)
WHERE is_published = true;
```

而不是只建：

```sql
CREATE INDEX perf_posts_is_published_idx
ON demo.perf_posts (is_published);
```

---

## 三、在索引列上使用函数

假设有索引：

```sql
CREATE INDEX perf_users_email_idx
ON demo.perf_users (email);
```

查询：

```sql
SELECT id, username, email
FROM demo.perf_users
WHERE lower(email) = 'user_50000@example.com';
```

普通 `email` 索引不一定能直接用于 `lower(email)`。

更好的做法有两种。

### 1. 存储时统一格式

例如邮箱入库前统一转小写：

```text
应用代码保证 email 永远是小写。
数据库建 email 普通索引或唯一索引。
```

查询时直接：

```sql
WHERE email = 'user_50000@example.com'
```

### 2. 创建表达式索引

```sql
CREATE INDEX perf_users_lower_email_idx
ON demo.perf_users (lower(email));
```

这样查询：

```sql
WHERE lower(email) = 'user_50000@example.com'
```

才更可能使用这个表达式索引。

---

## 四、LIKE 前面有通配符

普通 B-Tree 索引适合从左到右定位。

可能可用：

```sql
WHERE username LIKE 'user_100%'
```

通常很难使用普通 B-Tree：

```sql
WHERE username LIKE '%100%'
```

因为 `%100%` 表示任意位置包含 `100`，数据库无法从 B-Tree 的有序结构中快速定位起点。

这种场景后续可以学习全文搜索、三元组索引等能力。当前阶段先记住：

```text
前导通配符会让普通 B-Tree 很难发挥作用。
```

---

## 五、类型不匹配或隐式转换

索引字段类型和查询条件类型不一致时，可能影响索引使用。

例如字段是 `text`：

```sql
order_no text NOT NULL
```

查询时应该写文本：

```sql
WHERE order_no = 'ORD000000100000'
```

不要把字符串编号当数字处理。

又例如字段是 `timestamptz`，就用时间类型比较：

```sql
WHERE created_at >= '2026-01-01 00:00:00+08'::timestamptz
```

保持字段类型和条件类型清晰一致，能减少隐式转换带来的不确定性。

---

## 六、组合索引没有从最左字段开始用

索引：

```sql
CREATE INDEX perf_orders_user_status_paid_idx
ON demo.perf_orders (user_id, status, paid_at DESC);
```

适合：

```sql
WHERE user_id = 12345
```

适合：

```sql
WHERE user_id = 12345
  AND status = 'paid'
```

适合：

```sql
WHERE user_id = 12345
  AND status = 'paid'
ORDER BY paid_at DESC
```

不太适合只写：

```sql
WHERE status = 'paid'
```

因为跳过了最左字段 `user_id`。

这不是索引失效，而是索引设计和查询模式不匹配。

---

## 七、范围条件影响后续字段

索引：

```sql
CREATE INDEX perf_orders_created_status_idx
ON demo.perf_orders (created_at, status);
```

查询：

```sql
WHERE created_at >= '2026-01-01'
  AND created_at < '2026-02-01'
  AND status = 'paid'
```

因为 `created_at` 是范围条件，后面的 `status` 未必能像等值前缀那样充分发挥作用。

如果高频查询是：

```sql
WHERE status = 'paid'
  AND created_at >= ...
  AND created_at < ...
```

可能更适合：

```sql
CREATE INDEX perf_orders_status_created_idx
ON demo.perf_orders (status, created_at);
```

---

## 八、OR 条件不容易匹配单个索引

查询：

```sql
SELECT *
FROM demo.perf_orders
WHERE user_id = 12345
   OR status = 'pending';
```

这个条件包含两个不同字段的 `OR`。

PostgreSQL 有时可以使用 BitmapOr 组合多个索引，但不一定总是理想。

很多时候可以从业务上拆成两个查询，或者重新设计查询入口。

不要机械地认为：

```text
user_id 有索引，status 有索引，所以 OR 一定快。
```

仍然要看 `EXPLAIN ANALYZE`。

---

## 九、统计信息过旧

PostgreSQL 优化器依赖统计信息估算：

- 表有多少行。
- 某个条件大概会返回多少行。
- 某些值是否很常见。

如果大量写入、删除、更新后统计信息过旧，优化器可能做出不理想选择。

可以手动更新：

```sql
ANALYZE demo.perf_orders;
```

或：

```sql
ANALYZE;
```

正常情况下，Autovacuum 会自动做统计信息维护。但学习和压测时，手动 `ANALYZE` 很有帮助。

---

## 十、ORDER BY 和索引顺序不匹配

查询：

```sql
SELECT id, user_id, title, created_at
FROM demo.perf_posts
WHERE user_id = 100
ORDER BY created_at DESC
LIMIT 20;
```

索引：

```sql
CREATE INDEX perf_posts_user_id_idx
ON demo.perf_posts (user_id);
```

能帮助过滤 `user_id`，但不能直接满足 `created_at DESC` 排序。

更合适：

```sql
CREATE INDEX perf_posts_user_id_created_at_idx
ON demo.perf_posts (user_id, created_at DESC);
```

索引不仅要看 `WHERE`，还要看 `ORDER BY` 和 `LIMIT`。

---

## 十一、查询写法和部分索引条件不匹配

部分索引：

```sql
CREATE INDEX perf_orders_pending_created_at_idx
ON demo.perf_orders (created_at)
WHERE status = 'pending';
```

查询：

```sql
WHERE status = 'pending'
```

可能使用。

查询：

```sql
WHERE status IN ('pending', 'paid')
```

不能只用这个部分索引完成，因为索引里没有 `paid`。

所以部分索引要求查询条件能匹配它的 `WHERE` 条件。

---

## 十二、小表顺序扫描很正常

如果表只有几十行，PostgreSQL 选择 `Seq Scan` 通常没问题。

因为：

```text
读完整张小表可能比走索引更快。
```

不要看到 `Seq Scan` 就紧张。

执行计划要结合：

- 表大小。
- 返回行数。
- 执行时间。
- 查询频率。
- 业务场景。

---

## 十三、常见误区

### 1. 没走索引就是数据库有问题？

不是。

通常是 SQL、索引设计、数据分布或统计信息导致优化器选择了其他方案。

### 2. 强制让数据库走索引就一定更快？

不一定。

PostgreSQL 没有像某些数据库那样常用的强制索引提示。更推荐通过正确索引、更新统计信息、改写 SQL 来让优化器自然选择好计划。

### 3. 低选择性字段一定不能建索引？

不绝对。

如果它和排序、部分索引、组合索引结合，仍然可能很有价值。

### 4. EXPLAIN 显示不用索引，就应该马上删索引？

不要急。

一个索引可能服务其他查询。删除前要确认整体使用情况。

---

## 十四、本节达标标准

学完本节后，你应该能够做到：

- 知道“索引失效”通常是“索引没有被优化器选择”。
- 解释返回数据太多时为什么可能不用索引。
- 理解选择性差会影响索引价值。
- 知道在索引列上使用函数可能影响普通索引使用。
- 知道前导 `%` 的 `LIKE` 很难使用普通 B-Tree。
- 理解组合索引最左字段的重要性。
- 知道统计信息过旧会影响执行计划。
- 知道 `ORDER BY` 也需要考虑索引顺序。
- 能根据 SQL 写法初步判断为什么索引没有被使用。
