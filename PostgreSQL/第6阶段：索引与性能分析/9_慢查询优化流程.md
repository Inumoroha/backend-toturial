# 9. 慢查询优化流程

本节目标：把索引和执行计划知识串起来，形成一套分析慢查询的流程，能从业务 SQL 出发，判断是否需要索引、需要什么索引，以及优化是否真的有效。

慢查询优化最怕两件事：

```text
1. 不看执行计划，凭感觉加索引。
2. 加完索引不复测，不知道有没有效果。
```

更稳的做法是：

```text
先定位问题。
再用 EXPLAIN ANALYZE 找证据。
然后设计索引或改写 SQL。
最后对比优化前后。
```

---

## 一、慢查询优化的基本流程

可以按下面步骤：

1. 找到具体慢 SQL。
2. 明确查询目的和业务频率。
3. 查看表数据量和现有索引。
4. 执行 `EXPLAIN` 看计划。
5. 执行 `EXPLAIN (ANALYZE, BUFFERS)` 看真实情况。
6. 判断慢在哪里。
7. 设计索引或改写 SQL。
8. 执行 `ANALYZE` 更新统计信息。
9. 再次执行 `EXPLAIN ANALYZE` 对比。
10. 评估写入成本和其他查询影响。

---

## 二、第一步：明确慢 SQL

不要只说：

```text
订单列表很慢。
```

要拿到具体 SQL：

```sql
SELECT id, order_no, user_id, status, total_amount, created_at
FROM demo.perf_orders
WHERE status = 'pending'
ORDER BY created_at ASC
LIMIT 100;
```

还要知道：

- 这个接口多久调用一次。
- 每次大概返回多少行。
- 用户是否能接受延迟。
- 是否是后台任务。
- 是否影响核心链路。

高频核心查询和低频后台查询，优化优先级不同。

---

## 三、第二步：看表大小和索引

查看估算行数：

```sql
SELECT reltuples::bigint AS estimated_rows
FROM pg_class
WHERE oid = 'demo.perf_orders'::regclass;
```

查看索引：

```sql
SELECT
    indexname,
    indexdef
FROM pg_indexes
WHERE schemaname = 'demo'
  AND tablename = 'perf_orders'
ORDER BY indexname;
```

如果表很小，慢可能不在数据库扫描。

如果表很大，且没有匹配索引，就要重点分析。

---

## 四、第三步：看执行计划

```sql
EXPLAIN
SELECT id, order_no, user_id, status, total_amount, created_at
FROM demo.perf_orders
WHERE status = 'pending'
ORDER BY created_at ASC
LIMIT 100;
```

重点看：

- 是 `Seq Scan` 还是索引扫描。
- `WHERE` 条件在 `Filter` 还是 `Index Cond`。
- 有没有 `Sort`。
- 估算 `rows` 是否很大。
- `LIMIT` 是否能尽早生效。

---

## 五、第四步：看真实执行

```sql
EXPLAIN (ANALYZE, BUFFERS)
SELECT id, order_no, user_id, status, total_amount, created_at
FROM demo.perf_orders
WHERE status = 'pending'
ORDER BY created_at ASC
LIMIT 100;
```

重点看：

- `Execution Time`。
- 实际扫描了多少行。
- `Rows Removed by Filter` 是否很多。
- `Buffers` 是否很高。
- 排序是否耗时。
- 估算行数和真实行数差距是否很大。

如果看到类似：

```text
Seq Scan on perf_orders
  Filter: (status = 'pending'::text)
  Rows Removed by Filter: 很多
Sort
```

说明可能是：

```text
扫描大量订单，过滤出 pending，再排序。
```

---

## 六、第五步：设计索引

针对这个查询：

```sql
WHERE status = 'pending'
ORDER BY created_at ASC
LIMIT 100
```

可以考虑部分索引：

```sql
CREATE INDEX perf_orders_pending_created_at_idx
ON demo.perf_orders (created_at ASC)
WHERE status = 'pending';
```

原因：

- 查询固定只看 `pending`。
- `pending` 通常是一小部分数据。
- 需要按 `created_at ASC` 排序。
- `LIMIT 100` 可以利用索引顺序尽快取到前 100 条。

创建索引后更新统计信息：

```sql
ANALYZE demo.perf_orders;
```

再执行：

```sql
EXPLAIN (ANALYZE, BUFFERS)
SELECT id, order_no, user_id, status, total_amount, created_at
FROM demo.perf_orders
WHERE status = 'pending'
ORDER BY created_at ASC
LIMIT 100;
```

对比优化前后。

## 七、例子：用户已支付订单列表

慢 SQL：

```sql
SELECT id, order_no, total_amount, paid_at
FROM demo.perf_orders
WHERE user_id = 12345
  AND status = 'paid'
ORDER BY paid_at DESC
LIMIT 20;
```

分析查询模式：

```text
user_id 是等值条件。
status 是等值条件。
paid_at 是排序字段。
LIMIT 20 表示只要最新 20 条。
```

适合组合索引：

```sql
CREATE INDEX perf_orders_user_status_paid_at_idx
ON demo.perf_orders (user_id, status, paid_at DESC);
```

如果查询只查已支付订单，也可以考虑部分索引：

```sql
CREATE INDEX perf_orders_paid_user_paid_at_idx
ON demo.perf_orders (user_id, paid_at DESC)
WHERE status = 'paid';
```

怎么选？

- 如果还有很多状态组合查询，组合索引更通用。
- 如果高频查询固定是 `status = 'paid'`，部分索引更小更聚焦。
- 以执行计划和业务频率为准。

---

## 八、例子：根据订单号查订单

慢 SQL：

```sql
SELECT id, order_no, user_id, status, total_amount
FROM demo.perf_orders
WHERE order_no = 'ORD000000100000';
```

业务规则：

```text
订单号应该唯一。
```

适合唯一索引：

```sql
CREATE UNIQUE INDEX perf_orders_order_no_uidx
ON demo.perf_orders (order_no);
```

这个索引同时解决：

- 查询加速。
- 防止订单号重复。
- 支持 `ON CONFLICT`。

---

## 九、例子：文章列表分页

查询：

```sql
SELECT id, user_id, title, created_at
FROM demo.perf_posts
WHERE user_id = 100
ORDER BY created_at DESC
LIMIT 20;
```

适合：

```sql
CREATE INDEX perf_posts_user_id_created_at_idx
ON demo.perf_posts (user_id, created_at DESC);
```

如果使用基于游标的分页：

```sql
SELECT id, user_id, title, created_at
FROM demo.perf_posts
WHERE user_id = 100
  AND created_at < '2026-06-01 00:00:00+08'
ORDER BY created_at DESC
LIMIT 20;
```

这个索引依然适合，因为它先定位用户，再按时间顺序范围扫描。

## 十、优化不一定只靠索引

有些慢查询不是加索引就能解决。

### 1. SELECT * 返回太多列

```sql
SELECT *
FROM demo.perf_orders
WHERE user_id = 12345;
```

如果接口只需要 5 个字段，就不要查全部字段。

### 2. 返回数据太多

没有分页的列表接口很危险：

```sql
SELECT *
FROM demo.perf_orders
WHERE status = 'paid';
```

需要分页、限制时间范围，或者改成异步导出。

### 3. 排序数据太多

如果要对大量数据排序，即使有索引也可能很重。

可以考虑：

- 缩小过滤范围。
- 使用能支持排序的组合索引。
- 使用游标分页。

### 4. 业务需求本身太重

例如实时统计全站数据，可能需要：

- 预聚合表。
- 定时任务。
- 缓存。
- 异步计算。

---

## 十一、索引上线前要考虑写入成本

每个索引都会影响写入。

添加索引前要问：

- 这条查询频率高吗？
- 是否在核心链路？
- 索引能减少多少扫描和排序？
- 表写入频率高吗？
- 索引字段是否经常更新？
- 是否已有相似索引可以复用？

不要为一次低频后台查询给高写入表加很多索引。

---

## 十二、索引维护和清理

随着业务变化，可能出现无用索引。

可以通过数据库统计视图观察索引使用情况，例如：

```sql
SELECT
    schemaname,
    relname AS table_name,
    indexrelname AS index_name,
    idx_scan
FROM pg_stat_user_indexes
WHERE schemaname = 'demo'
ORDER BY idx_scan ASC, indexrelname;
```

`idx_scan` 表示索引被扫描的次数。

注意：

```text
idx_scan 低不代表一定能删。
```

有些索引用于唯一约束，哪怕很少被扫描，也在保证数据正确性。

删除索引前要确认：

- 是否支撑唯一约束。
- 是否支撑低频但重要的查询。
- 是否最近刚重启数据库导致统计清零。
- 是否有定时任务只在特定时间使用。

---

## 十三、常见误区

### 1. 慢查询第一步就是加索引？

不是。

第一步是拿到 SQL 和执行计划。

### 2. 优化只看 Execution Time？

不够。

还要看扫描行数、排序、Buffers、估算和真实行数差距。

### 3. 一个复杂索引能解决所有查询？

不能。

索引要贴合查询模式。不同接口可能需要不同索引，也可能需要改写接口。

### 4. 无用索引可以随便删？

不能。

先确认它不是唯一约束、不是重要低频查询依赖、不是统计数据暂时看不到。

---

## 十四、本节达标标准

学完本节后，你应该能够做到：

- 按步骤分析慢查询。
- 查看表大小和现有索引。
- 使用 `EXPLAIN` 和 `EXPLAIN ANALYZE` 找证据。
- 根据 `WHERE`、`ORDER BY`、`LIMIT` 设计索引。
- 区分普通索引、唯一索引、组合索引、部分索引的适用场景。
- 对比优化前后的执行计划。
- 知道优化不一定只靠索引。
- 考虑索引对写入和维护的成本。
