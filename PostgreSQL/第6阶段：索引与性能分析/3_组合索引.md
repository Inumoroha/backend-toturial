# 3. 组合索引

本节目标：理解组合索引的作用，掌握多字段索引的创建方式，知道字段顺序为什么重要，并能为常见列表查询设计组合索引。

组合索引也叫多列索引。

例如：

```sql
CREATE INDEX perf_orders_user_id_status_paid_at_idx
ON demo.perf_orders (user_id, status, paid_at DESC);
```

它不是三个独立索引，而是一棵按多个字段组织的索引。

可以粗略理解为：

```text
先按 user_id 排。
user_id 相同的，再按 status 排。
user_id 和 status 都相同的，再按 paid_at DESC 排。
```

---

## 一、为什么需要组合索引

后端列表查询经常不止一个条件：

```sql
SELECT id, order_no, total_amount, paid_at
FROM demo.perf_orders
WHERE user_id = 12345
  AND status = 'paid'
ORDER BY paid_at DESC
LIMIT 20;
```

这个查询同时有：

- `user_id = 12345`
- `status = 'paid'`
- `ORDER BY paid_at DESC`
- `LIMIT 20`

如果只给 `user_id` 建索引：

```sql
CREATE INDEX perf_orders_user_id_idx
ON demo.perf_orders (user_id);
```

PostgreSQL 可以快速找到这个用户的订单，但还需要继续过滤 `status`，并按 `paid_at` 排序。

如果建立更贴合查询的组合索引：

```sql
CREATE INDEX perf_orders_user_id_status_paid_at_idx
ON demo.perf_orders (user_id, status, paid_at DESC);
```

数据库更可能直接按索引顺序拿到这个用户的已支付订单，并且顺序已经符合 `paid_at DESC`。

---

## 二、创建组合索引

语法：

```sql
CREATE INDEX 索引名
ON 表名 (字段1, 字段2, 字段3);
```

例子：

```sql
CREATE INDEX perf_posts_user_id_created_at_idx
ON demo.perf_posts (user_id, created_at DESC);
```

适合查询：

```sql
SELECT id, user_id, title, created_at
FROM demo.perf_posts
WHERE user_id = 100
ORDER BY created_at DESC
LIMIT 20;
```

这个组合索引同时服务：

```text
WHERE user_id = 100
ORDER BY created_at DESC
LIMIT 20
```

---

## 三、最左前缀思想

组合索引字段顺序非常重要。

索引：

```sql
CREATE INDEX perf_orders_user_status_paid_idx
ON demo.perf_orders (user_id, status, paid_at DESC);
```

可以较好支持：

```sql
WHERE user_id = 12345
```

也可以较好支持：

```sql
WHERE user_id = 12345
  AND status = 'paid'
```

也可以较好支持：

```sql
WHERE user_id = 12345
  AND status = 'paid'
ORDER BY paid_at DESC
```

但通常不能很好支持只按 `status` 查询：

```sql
WHERE status = 'paid'
```

原因是索引首先按 `user_id` 组织。没有 `user_id` 条件时，很难直接跳到所有 `status = 'paid'` 的位置。

这就是最左前缀思想：

```text
组合索引更适合从左到右连续使用。
跳过最左字段时，索引价值会大幅下降。
```

---

## 四、字段顺序怎么选

字段顺序没有一个永远正确的公式，但可以按下面思路判断。

### 1. 等值条件通常放前面

例如：

```sql
WHERE user_id = 12345
  AND status = 'paid'
ORDER BY paid_at DESC
```

可以：

```sql
ON demo.perf_orders (user_id, status, paid_at DESC)
```

因为 `user_id` 和 `status` 都是等值条件。

### 2. 排序字段放在过滤字段后面

如果查询先过滤某个用户，再按时间倒序：

```sql
WHERE user_id = 100
ORDER BY created_at DESC
LIMIT 20
```

适合：

```sql
ON demo.perf_posts (user_id, created_at DESC)
```

### 3. 范围条件后面的字段通常难以继续充分利用

例如索引：

```sql
ON demo.perf_orders (created_at, status)
```

查询：

```sql
WHERE created_at >= '2026-01-01'
  AND created_at < '2026-02-01'
  AND status = 'paid'
```

`created_at` 是范围条件。索引能定位时间范围，但 `status` 的使用效果可能不如把等值字段放前面：

```sql
ON demo.perf_orders (status, created_at)
```

如果你的高频查询是“查某个状态在某段时间内的订单”，后者通常更合适。

---

## 五、组合索引不是多个单列索引的简单替代

三个单列索引：

```sql
CREATE INDEX perf_orders_user_id_idx
ON demo.perf_orders (user_id);

CREATE INDEX perf_orders_status_idx
ON demo.perf_orders (status);

CREATE INDEX perf_orders_paid_at_idx
ON demo.perf_orders (paid_at);
```

不等于一个组合索引：

```sql
CREATE INDEX perf_orders_user_id_status_paid_at_idx
ON demo.perf_orders (user_id, status, paid_at DESC);
```

组合索引中的数据顺序是连续组织好的，尤其适合：

```text
多个条件一起过滤 + 排序 + LIMIT
```

单列索引有时可以通过 Bitmap 扫描组合使用，但不一定能避免排序，也不一定比组合索引更高效。

---

## 六、组合索引支持 ORDER BY

查询：

```sql
SELECT id, user_id, title, created_at
FROM demo.perf_posts
WHERE user_id = 100
ORDER BY created_at DESC
LIMIT 20;
```

索引：

```sql
CREATE INDEX perf_posts_user_id_created_at_desc_idx
ON demo.perf_posts (user_id, created_at DESC);
```

有机会让 PostgreSQL：

```text
先定位 user_id = 100 的索引范围。
再按 created_at DESC 的索引顺序取前 20 条。
```

这对分页列表非常重要。

如果没有这个组合索引，数据库可能要：

```text
找到 user_id = 100 的所有文章。
再排序。
再取前 20 条。
```

数据量大时差距会很明显。

---

## 七、覆盖查询和 INCLUDE

有时查询只需要索引中已有的字段。

例如：

```sql
SELECT id, order_no
FROM demo.perf_orders
WHERE user_id = 12345
  AND status = 'paid'
ORDER BY paid_at DESC
LIMIT 20;
```

可以考虑：

```sql
CREATE INDEX perf_orders_user_status_paid_include_idx
ON demo.perf_orders (user_id, status, paid_at DESC)
INCLUDE (order_no);
```

`INCLUDE` 中的字段不会参与索引排序，但可以存放在索引里，减少回表取数据的可能。

学习阶段先知道：

```text
组合索引的主字段用于过滤和排序。
INCLUDE 字段用于让查询更可能直接从索引拿到所需列。
```

不要过早滥用 `INCLUDE`，它也会增加索引大小和写入成本。

## 八、常见组合索引设计例子

### 1. 用户文章列表

查询：

```sql
WHERE user_id = ?
ORDER BY created_at DESC
LIMIT ?
```

索引：

```sql
CREATE INDEX perf_posts_user_id_created_at_idx
ON demo.perf_posts (user_id, created_at DESC);
```

### 2. 已发布文章列表

查询：

```sql
WHERE is_published = true
ORDER BY created_at DESC
LIMIT ?
```

索引：

```sql
CREATE INDEX perf_posts_published_created_at_idx
ON demo.perf_posts (is_published, created_at DESC);
```

如果大多数文章都是已发布，`is_published` 选择性可能不高。后面学习部分索引时会有更适合的写法。

### 3. 用户已支付订单

查询：

```sql
WHERE user_id = ?
  AND status = 'paid'
ORDER BY paid_at DESC
LIMIT ?
```

索引：

```sql
CREATE INDEX perf_orders_user_status_paid_at_idx
ON demo.perf_orders (user_id, status, paid_at DESC);
```

### 4. 状态和时间范围

查询：

```sql
WHERE status = 'pending'
  AND created_at < now() - interval '10 minutes'
ORDER BY created_at ASC
LIMIT 100
```

索引：

```sql
CREATE INDEX perf_orders_status_created_at_idx
ON demo.perf_orders (status, created_at ASC);
```

## 九、常见误区

### 1. 组合索引字段顺序无所谓？

不是。

字段顺序决定了索引能支持哪些过滤和排序。

### 2. 有了 `(a, b)`，还需要 `(a)` 吗？

不一定。

`(a, b)` 通常可以支持只按 `a` 查询。但如果 `(a)` 查询非常高频，且 `(a, b)` 索引很大，是否保留单列索引要结合执行计划和写入成本判断。

### 3. 有了 `(a, b)`，就能很好支持 `WHERE b = ?` 吗？

通常不能。

因为跳过了最左字段 `a`。

### 4. 单列索引越多，效果越接近组合索引？

不是。

单列索引可能被 Bitmap 扫描组合使用，但不能完全替代贴合查询模式的组合索引。

## 十、本节达标标准

学完本节后，你应该能够做到：

- 使用 `CREATE INDEX ... (col1, col2)` 创建组合索引。
- 解释组合索引不是多个单列索引的简单相加。
- 理解最左前缀思想。
- 知道组合索引字段顺序很重要。
- 能为 `WHERE + ORDER BY + LIMIT` 查询设计组合索引。
- 知道等值条件通常放前面，排序字段放在过滤字段后面。
- 知道范围条件会影响后续字段的充分使用。
- 初步理解 `INCLUDE` 的作用和成本。
