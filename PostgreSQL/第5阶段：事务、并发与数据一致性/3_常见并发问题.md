# 3. 常见并发问题：脏读、不可重复读、幻读、丢失更新

本节目标：理解多个事务同时执行时可能出现的典型问题，包括脏读、不可重复读、幻读和丢失更新。

学习并发问题时，不要一开始就背概念。先记住一句话：

```text
并发问题通常发生在多个事务同时读写同一批数据时。
```

---

## 一、准备练习数据

```sql
CREATE SCHEMA IF NOT EXISTS demo;

DROP TABLE IF EXISTS demo.concurrent_accounts;
DROP TABLE IF EXISTS demo.concurrent_products;

CREATE TABLE demo.concurrent_accounts (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    owner_name text NOT NULL,
    balance numeric(12, 2) NOT NULL,
    CONSTRAINT concurrent_accounts_balance_check CHECK (balance >= 0)
);

CREATE TABLE demo.concurrent_products (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name text NOT NULL,
    stock integer NOT NULL,
    price numeric(12, 2) NOT NULL,
    CONSTRAINT concurrent_products_stock_check CHECK (stock >= 0)
);

INSERT INTO demo.concurrent_accounts (owner_name, balance)
VALUES ('Alice', 1000.00);

INSERT INTO demo.concurrent_products (name, stock, price)
VALUES
    ('Keyboard', 1, 499.00),
    ('Book', 10, 99.00);
```

后续需要打开两个数据库会话：

```text
会话 A：模拟第一个请求。
会话 B：模拟第二个请求。
```

## 二、脏读

脏读表示：

```text
一个事务读到了另一个事务尚未提交的数据。
```

例如：

```text
事务 A 修改 Alice 余额，但还没有提交。
事务 B 读到了事务 A 的未提交余额。
事务 A 最后回滚。
事务 B 刚才读到的数据就是脏数据。
```

### PostgreSQL 会发生脏读吗？

PostgreSQL 不会发生脏读。

即使你设置：

```sql
SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
```

PostgreSQL 实际上也会按 `READ COMMITTED` 处理。

也就是说，在 PostgreSQL 中，普通事务不会读到别的事务尚未提交的数据。

### 验证脏读不会发生

会话 A：

```sql
BEGIN;

UPDATE demo.concurrent_accounts
SET balance = 1
WHERE owner_name = 'Alice';

-- 先不要 COMMIT，也不要 ROLLBACK
```

会话 B：

```sql
SELECT owner_name, balance
FROM demo.concurrent_accounts
WHERE owner_name = 'Alice';
```

会话 B 看到的不是 `1`，而是会话 A 修改前已经提交的余额。

会话 A：

```sql
ROLLBACK;
```

这说明 PostgreSQL 没有让会话 B 读到未提交数据。

## 三、不可重复读

不可重复读表示：

```text
同一个事务中，两次读取同一行数据，结果不一样。
```

原因通常是：

```text
事务 A 第一次读取数据。
事务 B 修改并提交了这行数据。
事务 A 第二次读取同一行，看到新结果。
```

在 PostgreSQL 默认的 `READ COMMITTED` 隔离级别下，不可重复读可能发生。

### 验证不可重复读

先重置数据：

```sql
UPDATE demo.concurrent_accounts
SET balance = 1000
WHERE owner_name = 'Alice';
```

会话 A：

```sql
BEGIN;

SHOW transaction_isolation;

SELECT owner_name, balance
FROM demo.concurrent_accounts
WHERE owner_name = 'Alice';
```

会话 B：

```sql
BEGIN;

UPDATE demo.concurrent_accounts
SET balance = 800
WHERE owner_name = 'Alice';

COMMIT;
```

会话 A 再查一次：

```sql
SELECT owner_name, balance
FROM demo.concurrent_accounts
WHERE owner_name = 'Alice';

COMMIT;
```

在 `READ COMMITTED` 下，会话 A 第二次查询会看到会话 B 已提交的新余额。

这就是不可重复读。

## 四、幻读

幻读表示：

```text
同一个事务中，两次执行同一个范围查询，第二次多出或少了一些行。
```

它关注的不是同一行被修改，而是查询范围内的行集合发生变化。

例如：

```text
事务 A 查询库存大于 0 的商品，有 2 个。
事务 B 插入一个新的有库存商品并提交。
事务 A 再查库存大于 0 的商品，变成 3 个。
```

### 验证幻读

会话 A：

```sql
BEGIN;

SELECT id, name, stock
FROM demo.concurrent_products
WHERE stock > 0
ORDER BY id;
```

会话 B：

```sql
BEGIN;

INSERT INTO demo.concurrent_products (name, stock, price)
VALUES ('Mouse', 3, 59.00);

COMMIT;
```

会话 A 再查一次：

```sql
SELECT id, name, stock
FROM demo.concurrent_products
WHERE stock > 0
ORDER BY id;

COMMIT;
```

在 `READ COMMITTED` 下，会话 A 第二次查询会看到会话 B 新插入的商品。

这就是幻读。

## 五、丢失更新

丢失更新表示：

```text
两个事务都基于旧值计算新值，后提交的更新覆盖了先提交的更新，导致其中一次修改丢失。
```

这是后端项目里非常常见、也非常危险的问题。

### 错误写法：先读到应用层，再计算后写回

假设 Alice 余额是 1000。

会话 A：

```sql
BEGIN;

SELECT balance
FROM demo.concurrent_accounts
WHERE owner_name = 'Alice';
```

应用代码拿到 `1000`，准备加 100，算出 `1100`。

会话 B：

```sql
BEGIN;

SELECT balance
FROM demo.concurrent_accounts
WHERE owner_name = 'Alice';
```

应用代码也拿到 `1000`，准备加 200，算出 `1200`。

会话 A：

```sql
UPDATE demo.concurrent_accounts
SET balance = 1100
WHERE owner_name = 'Alice';

COMMIT;
```

会话 B：

```sql
UPDATE demo.concurrent_accounts
SET balance = 1200
WHERE owner_name = 'Alice';

COMMIT;
```

最终余额是 `1200`。

但正确结果应该是：

```text
1000 + 100 + 200 = 1300
```

会话 A 的加 100 被覆盖了。

这就是丢失更新。

### 更好的写法：让数据库基于当前值更新

对于简单累加，不要在应用层读出旧值再写回新值，而应该写成：

```sql
UPDATE demo.concurrent_accounts
SET balance = balance + 100
WHERE owner_name = 'Alice';
```

这种写法让数据库在行级锁保护下基于当前值更新。

## 六、库存扣减中的并发问题

库存扣减是非常典型的并发场景。

错误思路：

```text
1. SELECT stock FROM products WHERE id = 1;
2. 在应用代码中判断 stock > 0。
3. UPDATE products SET stock = stock - 1 WHERE id = 1;
```

两个请求并发时，都可能在第一步读到库存为 1，然后都继续扣减。

更好的写法是把判断和扣减合在一条 SQL 里：

```sql
UPDATE demo.concurrent_products
SET stock = stock - 1
WHERE id = 1
  AND stock >= 1
RETURNING id, name, stock;
```

如果返回 1 行，说明扣减成功。

如果返回 0 行，说明库存不足。

这条 SQL 的好处是：

```text
检查库存和扣减库存由数据库在一次原子更新中完成。
```

## 七、这些问题和隔离级别的关系

不同隔离级别能避免的问题不同。

大致可以先记住：

| 问题 | PostgreSQL READ COMMITTED | PostgreSQL REPEATABLE READ | PostgreSQL SERIALIZABLE |
| --- | --- | --- | --- |
| 脏读 | 不会发生 | 不会发生 | 不会发生 |
| 不可重复读 | 可能发生 | 不会发生 | 不会发生 |
| 幻读 | 可能发生 | 普通查询不会看到新快照 | 不会发生串行化异常以外的结果 |
| 丢失更新 | 取决于写法 | 可能报错，需要重试 | 可能报错，需要重试 |

表格只是帮助建立直觉，后面会专门讲隔离级别。

## 八、常见误区

### 1. PostgreSQL 支持 READ UNCOMMITTED，所以会脏读吗？

不会。

PostgreSQL 虽然接受 `READ UNCOMMITTED` 这个语法，但实际行为和 `READ COMMITTED` 类似，不会读到未提交数据。

### 2. 不可重复读一定是 bug 吗？

不一定。

很多普通接口只需要每条 SQL 看到最新已提交数据，`READ COMMITTED` 就够了。

如果一个业务事务要求多次读取必须基于同一个快照，就要考虑更高隔离级别或显式锁。

### 3. 幻读只和 INSERT 有关吗？

不只。

只要导致范围查询结果集合变化，都可能表现为幻读。例如插入、删除、更新某些字段后让记录进入或离开查询范围。

### 4. 只要加事务就不会有并发问题吗？

不是。

事务提供了边界，但并发正确性还取决于隔离级别、SQL 写法、锁策略和业务约束。

## 九、本节达标标准

学完本节后，你应该能够做到：

- 解释脏读是什么，并知道 PostgreSQL 不会发生脏读。
- 解释不可重复读是什么。
- 解释幻读是什么。
- 解释丢失更新是什么。
- 知道 `READ COMMITTED` 下同一事务中的两条查询可能看到不同已提交结果。
- 知道库存扣减应尽量把判断和更新合成原子 SQL。
- 知道“加了事务”不等于“没有并发问题”。
