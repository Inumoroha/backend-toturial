# 0. 事务、并发与数据一致性总览

本阶段目标：理解 PostgreSQL 如何保证数据可靠，并能在后端业务代码中正确使用事务。

前面几个阶段主要解决的是：

```text
如何建表。
如何写 SQL。
如何表达表与表之间的关系。
如何写真实业务中的查询。
```

第五阶段要解决的是另一个更接近生产环境的问题：

```text
当多条 SQL 必须一起成功时，数据库如何保证不出错？
当多个请求同时修改同一批数据时，数据库如何避免数据混乱？
当程序报错、连接断开、服务宕机时，已经写入的数据是否可靠？
```

这些问题就是事务、并发控制和数据一致性要处理的内容。

---

## 一、为什么后端开发必须学事务

很多数据库 bug 并不是 SQL 写错，而是事务边界不清楚。

例如转账场景：

```text
A 账户扣 100 元。
B 账户加 100 元。
```

这两个操作必须作为一个整体：

- 如果都成功，才算转账成功。
- 如果任意一步失败，前面已经做的修改必须撤销。

如果没有事务，可能出现：

```text
A 已经扣钱。
B 没有收到钱。
```

这就是非常严重的数据一致性问题。

再看库存场景：

```text
商品库存还剩 1 件。
两个用户同时下单。
```

如果业务代码只是简单地：

```text
先查库存。
判断库存是否大于 0。
再扣减库存。
```

在并发请求下，就可能两个请求都读到库存为 1，然后都扣减成功，最终卖出 2 件。

这类问题只靠“SQL 会写”不够，还必须理解事务、隔离级别和锁。

## 二、本阶段核心内容

本阶段建议拆成以下主题学习：

1. 事务基础：`BEGIN`、`COMMIT`、`ROLLBACK`。
2. ACID：原子性、一致性、隔离性、持久性。
3. 常见并发问题：脏读、不可重复读、幻读、丢失更新。
4. PostgreSQL 的 MVCC：为什么读写可以并发。
5. 事务隔离级别：`READ COMMITTED`、`REPEATABLE READ`、`SERIALIZABLE`。
6. 常见锁问题：行锁、表锁、锁等待、死锁。
7. 长事务风险：版本膨胀、阻塞、连接占用、业务超时。
8. 后端业务代码中的事务实践：事务边界、错误处理、重试、缩短事务时间。

---

## 三、推荐学习顺序

建议按下面顺序学习：

```text
事务语句
-> ACID
-> 并发问题
-> MVCC
-> 隔离级别
-> 锁
-> 长事务风险
-> 业务代码事务实践
```

原因是：

- 先学事务语句，知道怎么手动开启、提交、回滚。
- 再学 ACID，建立“事务到底保证什么”的整体概念。
- 然后看并发问题，理解为什么并发下会出 bug。
- 接着学 MVCC，理解 PostgreSQL 为什么能做到读写并发。
- 再学隔离级别，知道不同隔离级别能解决哪些问题。
- 最后学锁和长事务，把知识落到真实业务排错和代码实践上。

## 四、本阶段统一练习模型

为了让示例连贯，本阶段使用一个账户和订单模型。

包含这些表：

- `demo.accounts`：账户表，用来练习转账、余额扣减、行锁。
- `demo.account_logs`：账户流水表，用来记录账户变更。
- `demo.products`：商品表，用来练习库存扣减。
- `demo.orders`：订单表，用来练习下单事务。

初始化 SQL：

```sql
CREATE SCHEMA IF NOT EXISTS demo;

DROP TABLE IF EXISTS demo.orders;
DROP TABLE IF EXISTS demo.products;
DROP TABLE IF EXISTS demo.account_logs;
DROP TABLE IF EXISTS demo.accounts;

CREATE TABLE demo.accounts (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    owner_name text NOT NULL,
    balance numeric(12, 2) NOT NULL,
    created_at timestamptz NOT NULL DEFAULT now(),
    updated_at timestamptz NOT NULL DEFAULT now(),
    CONSTRAINT accounts_balance_check CHECK (balance >= 0)
);

CREATE TABLE demo.account_logs (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    account_id bigint NOT NULL,
    amount numeric(12, 2) NOT NULL,
    log_type text NOT NULL,
    remark text,
    created_at timestamptz NOT NULL DEFAULT now(),
    CONSTRAINT account_logs_account_id_fk
        FOREIGN KEY (account_id)
        REFERENCES demo.accounts (id),
    CONSTRAINT account_logs_log_type_check
        CHECK (log_type IN ('deposit', 'withdraw', 'transfer_in', 'transfer_out'))
);

CREATE TABLE demo.products (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name text NOT NULL,
    stock integer NOT NULL,
    price numeric(12, 2) NOT NULL,
    created_at timestamptz NOT NULL DEFAULT now(),
    updated_at timestamptz NOT NULL DEFAULT now(),
    CONSTRAINT products_stock_check CHECK (stock >= 0),
    CONSTRAINT products_price_check CHECK (price >= 0)
);

CREATE TABLE demo.orders (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    product_id bigint NOT NULL,
    quantity integer NOT NULL,
    total_amount numeric(12, 2) NOT NULL,
    status text NOT NULL DEFAULT 'created',
    created_at timestamptz NOT NULL DEFAULT now(),
    CONSTRAINT orders_product_id_fk
        FOREIGN KEY (product_id)
        REFERENCES demo.products (id),
    CONSTRAINT orders_quantity_check CHECK (quantity > 0),
    CONSTRAINT orders_status_check CHECK (status IN ('created', 'paid', 'cancelled'))
);

INSERT INTO demo.accounts (owner_name, balance)
VALUES
    ('Alice', 1000.00),
    ('Bob', 500.00),
    ('Carol', 300.00);

INSERT INTO demo.products (name, stock, price)
VALUES
    ('PostgreSQL Book', 10, 99.00),
    ('Go Course', 5, 199.00),
    ('Limited Keyboard', 1, 499.00);
```

## 五、本阶段会反复用到两个会话

学习并发和锁时，经常需要打开两个数据库连接。

你可以理解为：

```text
会话 A：模拟第一个请求。
会话 B：模拟第二个请求。
```

如果使用 `psql`，可以打开两个终端，同时连接同一个数据库。

如果使用 DBeaver 或 DataGrip，可以打开两个 SQL 控制台。注意要确认两个控制台使用的是不同连接或不同会话，否则并发现象可能看不出来。

后续教程会用下面格式表示：

```text
会话 A 执行：
...

会话 B 执行：
...
```

这不是两段 SQL 的先后顺序，而是两个并发连接中的操作步骤。

## 六、本阶段最重要的直觉

### 1. 事务是业务动作的边界

事务不是为了把 SQL 包起来好看，而是为了表达：

```text
这些数据库操作属于同一个业务动作。
它们必须一起成功或一起失败。
```

例如：

- 转账。
- 创建订单并扣减库存。
- 支付成功后更新订单状态并记录支付流水。
- 注册用户并初始化用户资料。

### 2. 并发问题通常发生在“先读再写”

很多并发 bug 长这样：

```text
先 SELECT 查当前状态。
在应用代码里判断。
再 UPDATE 修改数据。
```

如果中间没有合适的事务和锁，另一个请求可能在你读完之后、写入之前修改了同一行数据。

### 3. PostgreSQL 默认不是串行执行所有请求

数据库为了性能，不会把所有请求排成一条队伍慢慢执行。PostgreSQL 允许多个事务同时读写，并通过 MVCC、锁和隔离级别来保证正确性。

这意味着：

- 并发是默认存在的。
- 事务之间可能互相等待。
- 锁使用不当可能导致死锁。
- 隔离级别不合适可能产生业务层面的并发 bug。

### 4. 事务越短越好

事务期间会占用连接、持有锁、保留旧版本数据。事务越长，对系统影响越大。

一个很重要的实践原则是：

```text
事务里只放必须放在同一个原子操作里的数据库语句。
不要在事务中做慢网络请求、等待用户输入或执行耗时计算。
```

## 七、本阶段达标标准

学完第五阶段后，你应该能够做到：

- 手动使用 `BEGIN`、`COMMIT`、`ROLLBACK`。
- 解释 ACID 四个特性。
- 理解脏读、不可重复读、幻读、丢失更新的含义。
- 知道 PostgreSQL 使用 MVCC 支持读写并发。
- 知道 PostgreSQL 默认隔离级别是 `READ COMMITTED`。
- 能解释 `READ COMMITTED`、`REPEATABLE READ`、`SERIALIZABLE` 的基本区别。
- 能使用 `SELECT ... FOR UPDATE` 锁定需要修改的行。
- 能识别锁等待和死锁的基本现象。
- 知道长事务为什么危险。
- 能在后端代码中正确划分事务边界。

第五阶段学完后，你会从“会操作数据库”进一步变成“能安全地操作数据库”。这一步非常关键，因为真实业务最怕的不是报错，而是悄悄写错数据。
