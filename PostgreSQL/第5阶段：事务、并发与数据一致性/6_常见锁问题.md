# 6. 常见锁问题

本节目标：理解 PostgreSQL 中常见的锁现象，掌握行锁、锁等待、死锁的基本排查思路，并知道什么时候需要 `SELECT ... FOR UPDATE`。

学习锁时先记住一句话：

```text
MVCC 让读写并发更顺畅，但写写冲突仍然需要锁。
```

---

## 一、锁解决什么问题

锁的核心作用是：

```text
防止多个事务同时修改同一份关键数据，导致结果混乱。
```

例如两个事务同时扣减同一个账户余额：

```sql
UPDATE demo.accounts
SET balance = balance - 100
WHERE id = 1;
```

PostgreSQL 需要保证同一行不会被两个事务同时写坏。

因此，当一个事务正在更新某行时，另一个事务想更新同一行，通常要等待。

---

## 二、行锁：UPDATE 会锁住被修改的行

准备数据：

```sql
UPDATE demo.accounts
SET balance = 1000
WHERE owner_name = 'Alice';
```

会话 A：

```sql
BEGIN;

UPDATE demo.accounts
SET balance = balance - 100
WHERE owner_name = 'Alice';

-- 先不要 COMMIT
```

会话 B：

```sql
BEGIN;

UPDATE demo.accounts
SET balance = balance - 50
WHERE owner_name = 'Alice';
```

会话 B 会等待，因为会话 A 已经锁住了 Alice 这一行。

会话 A：

```sql
COMMIT;
```

会话 B 才会继续执行。

这就是典型的行锁等待。

---

## 三、普通 SELECT 通常不会被行锁阻塞

在会话 A 还没提交时，会话 B 执行普通查询：

```sql
SELECT owner_name, balance
FROM demo.accounts
WHERE owner_name = 'Alice';
```

通常不会被阻塞。

原因是 PostgreSQL 可以通过 MVCC 读取旧版本。

可以先建立这个直觉：

```text
UPDATE 会阻塞 UPDATE 同一行。
普通 SELECT 通常不会被 UPDATE 阻塞。
```

---

## 四、SELECT ... FOR UPDATE

有些业务是“先查，再根据查询结果更新”。

例如：

```text
先查账户余额。
如果余额足够，再扣款。
```

如果只是普通 `SELECT`，并不会锁住这行。其他事务仍然可能在你查询后修改它。

这时可以使用：

```sql
SELECT ...
FOR UPDATE;
```

它的含义是：

```text
查询这行，并对这行加更新锁。
在当前事务结束前，其他事务不能修改这行。
```

示例：

```sql
BEGIN;

SELECT id, owner_name, balance
FROM demo.accounts
WHERE owner_name = 'Alice'
FOR UPDATE;

-- 应用代码判断余额是否足够

UPDATE demo.accounts
SET balance = balance - 100
WHERE owner_name = 'Alice';

COMMIT;
```

当另一个事务也想 `UPDATE` Alice 这行时，就会等待当前事务结束。

---

## 五、FOR UPDATE 的典型使用场景

### 1. 账户扣款

```sql
BEGIN;

SELECT id, balance
FROM demo.accounts
WHERE id = 1
FOR UPDATE;

-- 判断余额是否足够

UPDATE demo.accounts
SET balance = balance - 100
WHERE id = 1;

COMMIT;
```

### 2. 领取任务

多个 worker 从任务表中领取任务时，可以使用行锁避免多个 worker 领取同一任务。

```sql
BEGIN;

SELECT id
FROM demo.jobs
WHERE status = 'pending'
ORDER BY id
LIMIT 1
FOR UPDATE SKIP LOCKED;

-- 更新任务状态为 processing

COMMIT;
```

`SKIP LOCKED` 表示跳过已经被其他事务锁住的行，常用于任务队列。

### 3. 防止并发修改同一业务对象

例如同一个订单只能被一个事务执行状态流转：

```sql
SELECT id, status
FROM demo.orders
WHERE id = 1
FOR UPDATE;
```

## 六、锁等待

锁等待表示：

```text
一个事务持有锁，另一个事务需要这个锁，只能等待。
```

短暂锁等待很正常。问题出在等待时间过长。

常见原因：

- 某个事务开启后迟迟不提交。
- 事务中包含慢查询。
- 事务中调用外部服务。
- 批量更新太多行。
- 锁顺序不一致导致死锁。

可以设置锁等待超时，避免请求无限等待：

```sql
SET lock_timeout = '3s';
```

在事务中使用：

```sql
BEGIN;

SET LOCAL lock_timeout = '3s';

UPDATE demo.accounts
SET balance = balance - 100
WHERE id = 1;

COMMIT;
```

`SET LOCAL` 只在当前事务内生效。

---

## 七、死锁

死锁表示：

```text
事务 A 等事务 B 释放锁。
事务 B 又等事务 A 释放锁。
双方互相等待，谁也走不下去。
```

### 制造一个死锁

确保有两行账户：

```sql
SELECT id, owner_name
FROM demo.accounts
ORDER BY id;
```

会话 A：

```sql
BEGIN;

UPDATE demo.accounts
SET balance = balance - 10
WHERE id = 1;
```

会话 B：

```sql
BEGIN;

UPDATE demo.accounts
SET balance = balance - 10
WHERE id = 2;
```

会话 A：

```sql
UPDATE demo.accounts
SET balance = balance + 10
WHERE id = 2;
```

会话 A 等会话 B 释放 id = 2 的锁。

会话 B：

```sql
UPDATE demo.accounts
SET balance = balance + 10
WHERE id = 1;
```

会话 B 又等会话 A 释放 id = 1 的锁。

这就是死锁。

PostgreSQL 会检测死锁，并中止其中一个事务。被中止的事务需要回滚，业务代码通常应该重试整个事务。

---

## 八、如何减少死锁

### 1. 保持固定加锁顺序

转账时，如果两个账户都要锁，统一按账户 ID 从小到大锁。

错误思路：

```text
Alice -> Bob 的转账先锁 Alice，再锁 Bob。
Bob -> Alice 的转账先锁 Bob，再锁 Alice。
```

并发时容易死锁。

更好思路：

```text
不管谁给谁转账，都按 account_id 从小到大加锁。
```

示例：

```sql
BEGIN;

SELECT id, balance
FROM demo.accounts
WHERE id IN (1, 2)
ORDER BY id
FOR UPDATE;

-- 再执行扣款和加款

COMMIT;
```

### 2. 缩短事务时间

事务持锁时间越短，死锁概率越低。

### 3. 避免事务中做外部调用

不要在持锁期间调用支付接口、短信接口、HTTP 接口。

### 4. 小批量处理

大批量更新可能锁住大量行。可以考虑分批处理，降低一次事务的锁范围。

## 九、表锁和 DDL

除了行锁，PostgreSQL 还有表级锁。

一些 DDL 操作会获取较强的表锁，例如：

```sql
ALTER TABLE demo.accounts ADD COLUMN remark text;
```

如果表很大、业务正在频繁读写，DDL 可能造成阻塞。

学习阶段先记住：

```text
生产环境改表要谨慎。
DDL 也会涉及锁。
大表变更要评估锁影响。
```

后续学习迁移工具和性能优化时，还会继续遇到这个问题。

---

## 十、常见误区

### 1. 有 MVCC 就不需要锁了吗？

不是。

MVCC 主要减少读写之间的阻塞。写写冲突、显式保护业务对象、任务领取等场景仍然需要锁。

### 2. SELECT ... FOR UPDATE 可以随便加吗？

不建议。

它会增加锁等待风险。只有当你确实需要保护后续更新逻辑时再使用。

### 3. 死锁是不是数据库坏了？

不是。

死锁是并发系统中的正常风险。PostgreSQL 会检测并中止其中一个事务，应用需要正确回滚和重试。

### 4. 锁等待一定是坏事吗？

短暂等待是正常的。长时间等待才需要排查事务是否太长、SQL 是否太慢、加锁顺序是否混乱。

---

## 十一、本节达标标准

学完本节后，你应该能够做到：

- 知道 `UPDATE` 会锁住被修改的行。
- 知道普通 `SELECT` 通常不会被行锁阻塞。
- 使用 `SELECT ... FOR UPDATE` 锁定后续要修改的行。
- 解释锁等待是什么。
- 解释死锁是什么。
- 知道 PostgreSQL 会检测死锁并中止其中一个事务。
- 知道固定加锁顺序可以减少死锁。
- 知道事务越短，持锁时间越短，系统越稳定。
