# 8. 后端业务代码中的事务实践

本节目标：把事务、隔离级别、锁和长事务知识落到业务代码设计中，知道如何划分事务边界、处理错误、处理重试，并避免常见坑。

前面几篇主要讲 PostgreSQL 的行为。这一篇更偏后端工程实践。

事务在业务代码里的核心问题是：

```text
哪些 SQL 必须放在同一个事务里？
事务中出错后如何回滚？
哪些并发错误需要重试？
哪些操作不应该放进事务？
```

---

## 一、事务边界应该跟业务动作对齐

事务边界不是按“函数有多长”划分，也不是按“代码写起来方便”划分，而是按业务动作划分。

一个好的事务边界应该能说清楚：

```text
这个事务提交，表示某个业务动作完成。
这个事务回滚，表示这个业务动作没有发生。
```

例如：

- 转账成功。
- 创建订单成功。
- 支付流水记录成功。
- 用户注册成功。
- 库存扣减成功。

---

## 二、适合放进同一个事务的操作

### 1. 转账

```text
扣减转出账户余额。
增加转入账户余额。
记录两边账户流水。
```

这些必须一起成功。

### 2. 创建订单

```text
检查并扣减库存。
创建订单。
记录订单事件。
```

这些通常应该在一个事务里。

### 3. 注册用户

```text
创建用户。
创建用户资料。
初始化默认设置。
```

这些可以放在一个事务里，避免出现“用户有了但资料没初始化”的半成品数据。

---

## 三、不适合放进事务的操作

### 1. 外部 HTTP 调用

例如：

- 调用支付网关。
- 调用短信服务。
- 调用邮件服务。
- 调用第三方风控接口。

这些操作慢、不稳定，而且数据库回滚无法撤销它们。

### 2. 等待用户操作

不要在事务里等待用户输入验证码、扫码确认、上传文件。

### 3. 大量 CPU 计算

事务中不适合做耗时计算。可以先在事务外计算好，再进入事务做最终校验和写入。

### 4. 大批量长时间处理

例如一次事务更新几百万行，可能长时间持锁和产生大量旧版本。更推荐分批处理。

---

## 四、库存扣减推荐写法

库存扣减常见错误是：

```text
先 SELECT 查库存。
应用代码判断库存够不够。
再 UPDATE 扣库存。
```

并发下容易出问题。

更推荐把判断和扣减合成一条原子 SQL：

```sql
UPDATE demo.products
SET
    stock = stock - $1,
    updated_at = now()
WHERE id = $2
  AND stock >= $1
RETURNING id, name, stock, price;
```

应用代码根据返回行数判断：

```text
返回 1 行：扣减成功。
返回 0 行：商品不存在或库存不足。
```

然后在同一个事务里创建订单：

```sql
INSERT INTO demo.orders (product_id, quantity, total_amount, status)
VALUES ($1, $2, $3, 'created')
RETURNING id;
```

## 五、转账推荐写法

转账需要同时锁定两个账户。

为了减少死锁，建议固定加锁顺序：

```sql
BEGIN;

SELECT id, balance
FROM demo.accounts
WHERE id IN ($1, $2)
ORDER BY id
FOR UPDATE;

-- 应用代码检查转出账户余额

UPDATE demo.accounts
SET
    balance = balance - $3,
    updated_at = now()
WHERE id = $1;

UPDATE demo.accounts
SET
    balance = balance + $3,
    updated_at = now()
WHERE id = $2;

INSERT INTO demo.account_logs (account_id, amount, log_type, remark)
VALUES
    ($1, -$3, 'transfer_out', 'transfer out'),
    ($2,  $3, 'transfer_in', 'transfer in');

COMMIT;
```

重点不是照抄 SQL，而是理解三个原则：

- 需要保护的账户行要加锁。
- 多行加锁顺序要固定。
- 扣款、加款、流水在同一个事务里。

## 六、错误处理原则

后端代码中事务处理必须保证：

```text
BEGIN 成功后，最终一定要 COMMIT 或 ROLLBACK。
```

典型结构：

```text
开启事务
执行 SQL 1
执行 SQL 2
执行 SQL 3
如果全部成功：提交事务
如果任意失败：回滚事务
```

伪代码：

```text
tx = begin()
try:
    execute sql 1
    execute sql 2
    execute sql 3
    commit tx
except:
    rollback tx
    return error
```

注意：如果 `COMMIT` 失败，也要把它当成事务失败处理。业务不能假设提交一定成功。

---

## 七、哪些错误可以重试

并发系统中，有些错误不是业务失败，而是事务冲突。

常见可重试错误包括：

- 死锁。
- 序列化失败。
- 某些锁超时，取决于业务是否允许重试。

例如 `SERIALIZABLE` 或 `REPEATABLE READ` 下可能遇到：

```text
could not serialize access due to concurrent update
```

正确重试方式是：

```text
回滚当前事务。
重新开启一个新事务。
重新执行完整业务逻辑。
```

不要只重试失败 SQL，因为事务前面读到的数据、做出的判断都可能已经过期。

---

## 八、重试要有上限

重试不能无限循环。

推荐策略：

```text
最多重试 2 到 3 次。
每次失败后短暂等待。
如果仍失败，返回明确错误或进入异步补偿。
```

重试还要注意幂等性。

例如创建订单接口，如果客户端超时后重试，不能重复创建多笔订单。通常需要：

- 客户端请求号。
- 业务唯一键。
- 幂等表。
- 唯一约束。
- `INSERT ... ON CONFLICT`。

---

## 九、事务和幂等

事务保证的是：

```text
一次事务内部的多个数据库操作一起成功或失败。
```

幂等保证的是：

```text
同一个请求重复执行，不会产生重复业务结果。
```

它们不是一回事。

例如支付回调可能重复发送。你需要用唯一约束保证同一个支付流水号只处理一次：

```sql
CREATE TABLE demo.payment_events (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    provider_event_id text NOT NULL UNIQUE,
    order_id bigint NOT NULL,
    amount numeric(12, 2) NOT NULL,
    created_at timestamptz NOT NULL DEFAULT now()
);
```

处理回调时：

```sql
INSERT INTO demo.payment_events (provider_event_id, order_id, amount)
VALUES ($1, $2, $3)
ON CONFLICT (provider_event_id) DO NOTHING
RETURNING id;
```

如果没有返回行，说明这个事件已经处理过或至少已经记录过。

## 十、隔离级别选择建议

### 1. 默认使用 READ COMMITTED

大部分业务可以使用默认隔离级别。

配合：

- 数据库约束。
- 原子更新。
- 必要的 `FOR UPDATE`。
- 正确的错误处理。

### 2. 需要稳定快照时使用 REPEATABLE READ

例如事务内多次统计必须基于同一份快照。

### 3. 强一致复杂规则考虑 SERIALIZABLE

但必须实现完整事务重试。

## 十一、事务中的 SQL 顺序

事务中的 SQL 顺序会影响锁持有时间和死锁概率。

建议：

- 先校验明显错误，减少无意义事务。
- 尽量按固定顺序访问表和行。
- 先锁关键行，再做依赖这些行的判断。
- 尽快完成写入并提交。

例如转账时统一按账户 ID 锁定：

```sql
SELECT id, balance
FROM demo.accounts
WHERE id IN ($1, $2)
ORDER BY id
FOR UPDATE;
```

---

## 十二、常见误区

### 1. 事务越大越安全？

不是。

事务越大，锁持有越久，失败回滚成本越高，对系统影响越大。

### 2. 所有并发问题都靠 SERIALIZABLE 解决？

不建议。

很多问题可以通过唯一约束、原子更新、行锁和幂等键更直接地解决。

### 3. ROLLBACK 能撤销外部调用？

不能。

数据库回滚只能撤销数据库事务内部的修改。

### 4. 事务失败后只重试最后一条 SQL 就行？

通常不行。

应该回滚并重试整个事务，因为前面的读取和判断可能已经不再成立。

### 5. 不写事务也没事，数据库会自动提交？

自动提交只适合单条 SQL。多条 SQL 组成一个业务动作时，必须显式事务。

---

## 十三、本节达标标准

学完本节后，你应该能够做到：

- 按业务动作划分事务边界。
- 知道哪些操作应该放进事务，哪些不应该。
- 写出库存扣减的原子更新 SQL。
- 写出转账时固定顺序加锁的思路。
- 知道事务出错必须回滚。
- 知道死锁和序列化失败通常要重试整个事务。
- 知道重试要有次数上限。
- 区分事务和幂等。
- 能在后端项目中更安全地使用 PostgreSQL 事务。
