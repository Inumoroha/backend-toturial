# 1. 事务语句：BEGIN、COMMIT、ROLLBACK

本节目标：理解事务的基本用法，掌握 `BEGIN`、`COMMIT`、`ROLLBACK`，并能用事务保证多条 SQL 要么一起成功，要么一起失败。

事务可以理解为数据库中的一个“操作包”：

```text
包里的 SQL 全部成功，才真正生效。
只要中途出错，就可以撤销整个包里的修改。
```

---

## 一、为什么需要事务

假设有一个转账场景：

```text
Alice 给 Bob 转账 100 元。
```

这个业务至少包含两步：

```sql
UPDATE demo.accounts
SET balance = balance - 100
WHERE owner_name = 'Alice';

UPDATE demo.accounts
SET balance = balance + 100
WHERE owner_name = 'Bob';
```

如果第一条 SQL 成功，第二条 SQL 失败，就会出现：

```text
Alice 的钱少了。
Bob 的钱没增加。
```

这显然是不允许的。

事务的作用就是把这两条 SQL 绑定成一个整体：

```text
要么两条都成功。
要么两条都不生效。
```

## 二、准备练习数据

```sql
CREATE SCHEMA IF NOT EXISTS demo;

DROP TABLE IF EXISTS demo.account_logs;
DROP TABLE IF EXISTS demo.accounts;

CREATE TABLE demo.accounts (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    owner_name text NOT NULL,
    balance numeric(12, 2) NOT NULL,
    created_at timestamptz NOT NULL DEFAULT now(),
    updated_at timestamptz NOT NULL DEFAULT now(),
    CONSTRAINT accounts_balance_check CHECK (balance >= 0)
);

CREATE TABLE demo.account_logs (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    account_id bigint NOT NULL,
    amount numeric(12, 2) NOT NULL,
    log_type text NOT NULL,
    remark text,
    created_at timestamptz NOT NULL DEFAULT now(),
    CONSTRAINT account_logs_account_id_fk
        FOREIGN KEY (account_id)
        REFERENCES demo.accounts (id),
    CONSTRAINT account_logs_log_type_check
        CHECK (log_type IN ('deposit', 'withdraw', 'transfer_in', 'transfer_out'))
);

INSERT INTO demo.accounts (owner_name, balance)
VALUES
    ('Alice', 1000.00),
    ('Bob', 500.00);
```

查询初始余额：

```sql
SELECT id, owner_name, balance
FROM demo.accounts
ORDER BY id;
```

## 三、BEGIN：开启事务

`BEGIN` 用来开启一个事务：

```sql
BEGIN;
```

从 `BEGIN` 开始，后续 SQL 都处在同一个事务里，直到执行：

- `COMMIT`：提交事务。
- `ROLLBACK`：回滚事务。

完整结构如下：

```sql
BEGIN;

-- 多条 SQL

COMMIT;
```

或者：

```sql
BEGIN;

-- 多条 SQL

ROLLBACK;
```

---

## 四、COMMIT：提交事务

执行一次正常转账：

```sql
BEGIN;

UPDATE demo.accounts
SET
    balance = balance - 100,
    updated_at = now()
WHERE owner_name = 'Alice';

UPDATE demo.accounts
SET
    balance = balance + 100,
    updated_at = now()
WHERE owner_name = 'Bob';

COMMIT;
```

查询结果：

```sql
SELECT owner_name, balance
FROM demo.accounts
ORDER BY id;
```

你会看到：

```text
Alice: 900.00
Bob:   600.00
```

`COMMIT` 的含义是：

```text
确认这个事务中的修改，让它们真正生效。
```

---

## 五、ROLLBACK：回滚事务

再执行一次转账，但最后回滚：

```sql
BEGIN;

UPDATE demo.accounts
SET
    balance = balance - 200,
    updated_at = now()
WHERE owner_name = 'Alice';

UPDATE demo.accounts
SET
    balance = balance + 200,
    updated_at = now()
WHERE owner_name = 'Bob';

ROLLBACK;
```

查询结果：

```sql
SELECT owner_name, balance
FROM demo.accounts
ORDER BY id;
```

你会发现余额没有变化。

`ROLLBACK` 的含义是：

```text
撤销当前事务中已经执行过的修改。
```

## 六、事务中插入流水

真实转账通常还需要记录流水：

```sql
BEGIN;

UPDATE demo.accounts
SET
    balance = balance - 100,
    updated_at = now()
WHERE owner_name = 'Alice';

INSERT INTO demo.account_logs (account_id, amount, log_type, remark)
SELECT id, -100, 'transfer_out', 'transfer to Bob'
FROM demo.accounts
WHERE owner_name = 'Alice';

UPDATE demo.accounts
SET
    balance = balance + 100,
    updated_at = now()
WHERE owner_name = 'Bob';

INSERT INTO demo.account_logs (account_id, amount, log_type, remark)
SELECT id, 100, 'transfer_in', 'transfer from Alice'
FROM demo.accounts
WHERE owner_name = 'Bob';

COMMIT;
```

这里有四个数据库操作：

1. Alice 扣款。
2. 记录 Alice 转出流水。
3. Bob 加款。
4. 记录 Bob 转入流水。

它们必须属于同一个事务。

如果扣款成功但流水失败，数据就不完整。

如果流水成功但加款失败，数据也不完整。

## 七、故意制造错误并观察回滚

下面这个例子会故意插入一个不合法的流水类型：

```sql
BEGIN;

UPDATE demo.accounts
SET balance = balance - 50
WHERE owner_name = 'Alice';

INSERT INTO demo.account_logs (account_id, amount, log_type, remark)
SELECT id, -50, 'bad_type', 'invalid log type'
FROM demo.accounts
WHERE owner_name = 'Alice';

COMMIT;
```

因为 `log_type` 有检查约束：

```sql
CHECK (log_type IN ('deposit', 'withdraw', 'transfer_in', 'transfer_out'))
```

所以 `bad_type` 会导致插入失败。

在 PostgreSQL 中，一旦事务中的某条语句出错，当前事务会进入失败状态。之后继续执行 SQL，通常会看到类似含义的错误：

```text
current transaction is aborted
```

这时应该执行：

```sql
ROLLBACK;
```

然后再查询余额：

```sql
SELECT owner_name, balance
FROM demo.accounts
ORDER BY id;
```

你会发现前面的扣款也没有真正提交。

## 八、自动提交是什么

很多客户端默认开启自动提交。

自动提交可以理解为：

```text
你执行的每一条 SQL，都会被数据库自动包装成一个小事务。
执行成功后自动 COMMIT。
```

例如你执行：

```sql
UPDATE demo.accounts
SET balance = balance + 10
WHERE owner_name = 'Alice';
```

在自动提交模式下，它大致等价于：

```sql
BEGIN;

UPDATE demo.accounts
SET balance = balance + 10
WHERE owner_name = 'Alice';

COMMIT;
```

这对单条 SQL 很方便，但当一个业务动作包含多条 SQL 时，就不能依赖自动提交，而应该显式开启事务。

## 九、事务不是越大越好

事务应该只包住同一个业务动作中必须保持一致的数据库操作。

例如创建订单时，事务里可以放：

- 检查库存。
- 扣减库存。
- 创建订单。
- 写订单日志。

不应该放：

- 调用第三方支付接口。
- 发送短信。
- 等待用户输入验证码。
- 生成大型报表。
- 调用很慢的远程服务。

事务越长，占用资源越久，也越容易引发锁等待和性能问题。

## 十、常见误区

### 1. 事务只在出错时才需要吗？

不是。

事务首先是表达业务原子性的工具。只要多条 SQL 必须一起成功或一起失败，就应该考虑事务。

### 2. ROLLBACK 能撤销所有事情吗？

只能撤销当前事务中的数据库修改。

如果你在事务中调用了外部接口、发了短信、写了外部文件，这些不会因为数据库 `ROLLBACK` 自动撤销。

### 3. SELECT 需要事务吗？

普通查询不一定需要显式事务。

但如果你希望多次查询看到一致的数据快照，或者查询后要继续更新同一批数据，就需要理解事务和隔离级别。

### 4. COMMIT 后还能 ROLLBACK 吗？

不能。

一旦 `COMMIT` 成功，事务已经结束，修改已经生效。之后的 `ROLLBACK` 不能撤销已经提交的事务。

## 十一、本节达标标准

学完本节后，你应该能够做到：

- 使用 `BEGIN` 开启事务。
- 使用 `COMMIT` 提交事务。
- 使用 `ROLLBACK` 回滚事务。
- 理解自动提交模式。
- 能写出转账事务。
- 知道事务中的一条语句出错后应该回滚。
- 知道事务应该尽量短，只包住必要的数据库操作。
