# 5. 事务隔离级别

本节目标：理解 PostgreSQL 中常用事务隔离级别的行为，知道默认隔离级别是什么，以及在业务中如何选择。

隔离级别回答的问题是：

```text
多个事务同时执行时，一个事务能看到另一个事务的哪些修改？
```

PostgreSQL 支持这些隔离级别：

- `READ UNCOMMITTED`
- `READ COMMITTED`
- `REPEATABLE READ`
- `SERIALIZABLE`

其中最常用的是：

- `READ COMMITTED`
- `REPEATABLE READ`
- `SERIALIZABLE`

---

## 一、查看当前隔离级别

```sql
SHOW transaction_isolation;
```

PostgreSQL 默认通常是：

```text
read committed
```

也可以在事务中查看：

```sql
BEGIN;

SHOW transaction_isolation;

COMMIT;
```

---

## 二、设置事务隔离级别

开启事务时设置：

```sql
BEGIN ISOLATION LEVEL READ COMMITTED;
```

或者：

```sql
BEGIN ISOLATION LEVEL REPEATABLE READ;
```

或者：

```sql
BEGIN ISOLATION LEVEL SERIALIZABLE;
```

也可以在事务开始后、执行第一条查询或修改前设置：

```sql
BEGIN;

SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;

SELECT ...

COMMIT;
```

注意：事务已经执行查询或修改后，再设置隔离级别通常会失败。

---

## 三、READ UNCOMMITTED

标准 SQL 中，`READ UNCOMMITTED` 允许读取其他事务未提交的数据，也就是可能发生脏读。

但 PostgreSQL 不会脏读。

在 PostgreSQL 中：

```sql
BEGIN ISOLATION LEVEL READ UNCOMMITTED;
```

实际行为接近 `READ COMMITTED`。

学习时知道这个结论即可：

```text
PostgreSQL 接受 READ UNCOMMITTED 语法，但不会让普通查询读到未提交数据。
```

---

## 四、READ COMMITTED

`READ COMMITTED` 是 PostgreSQL 默认隔离级别。

它的核心规则是：

```text
每条 SQL 语句只能看到语句开始前已经提交的数据。
每条语句会使用新的快照。
```

这意味着：

- 不会脏读。
- 同一个事务里，两次查询可能看到不同结果。
- 可能出现不可重复读。
- 可能出现幻读。

### 示例：同一事务中两次查询结果不同

会话 A：

```sql
BEGIN ISOLATION LEVEL READ COMMITTED;

SELECT balance
FROM demo.accounts
WHERE owner_name = 'Alice';
```

会话 B：

```sql
UPDATE demo.accounts
SET balance = balance + 100
WHERE owner_name = 'Alice';
```

会话 A：

```sql
SELECT balance
FROM demo.accounts
WHERE owner_name = 'Alice';

COMMIT;
```

会话 A 第二次查询可以看到会话 B 已提交的新结果。

### 适合场景

`READ COMMITTED` 适合大多数普通业务：

- 列表查询。
- 详情查询。
- 普通增删改。
- 不要求整个事务内多次读取完全一致快照的场景。

但在“先读后写”的关键业务中，要小心并发问题，必要时使用原子更新或显式行锁。

---

## 五、REPEATABLE READ

`REPEATABLE READ` 的核心规则是：

```text
同一个事务中的查询通常看到同一个快照。
```

这意味着：

- 不会脏读。
- 不会出现普通意义上的不可重复读。
- 同一事务内重复查询结果更稳定。

### 示例：同一事务中看到一致结果

会话 A：

```sql
BEGIN ISOLATION LEVEL REPEATABLE READ;

SELECT balance
FROM demo.accounts
WHERE owner_name = 'Alice';
```

会话 B：

```sql
UPDATE demo.accounts
SET balance = balance + 100
WHERE owner_name = 'Alice';
```

会话 A：

```sql
SELECT balance
FROM demo.accounts
WHERE owner_name = 'Alice';

COMMIT;
```

会话 A 第二次查询仍然看到事务开始时快照中的结果。

### 可能出现序列化失败

在 `REPEATABLE READ` 下，如果事务尝试更新一行，而这行已经被其他并发事务更新并提交，PostgreSQL 可能报错：

```text
could not serialize access due to concurrent update
```

这时正确做法通常是：

```text
回滚当前事务。
重新执行整个事务。
```

不要只重试失败的那一条 SQL，因为事务的快照和前面的判断都可能已经失效。

### 适合场景

`REPEATABLE READ` 适合：

- 一个事务内需要多次查询一致快照。
- 报表类读取。
- 需要避免不可重复读的业务。

但如果事务中有写操作，要准备处理并发更新失败和重试。

---

## 六、SERIALIZABLE

`SERIALIZABLE` 是 PostgreSQL 最严格的隔离级别。

它的目标是让并发事务的执行结果等价于某种串行顺序执行。

可以理解为：

```text
虽然事务实际上并发执行，但最终结果必须像一个一个排队执行那样正确。
```

在这个隔离级别下，PostgreSQL 可能为了保证串行化结果而中止某个事务，报错类似：

```text
could not serialize access due to read/write dependencies among transactions
```

遇到这种错误，应用代码需要重试整个事务。

### 适合场景

`SERIALIZABLE` 适合：

- 数据正确性要求极高。
- 业务逻辑复杂，难以通过简单行锁表达。
- 可以接受事务失败后重试。

不适合盲目用于所有请求，因为它可能带来更多事务失败和重试成本。

---

## 七、隔离级别与并发问题对照

在 PostgreSQL 中，可以用下面的表建立直觉：

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 备注 |
| --- | --- | --- | --- | --- |
| `READ UNCOMMITTED` | 不会 | 可能 | 可能 | PostgreSQL 中行为接近 `READ COMMITTED` |
| `READ COMMITTED` | 不会 | 可能 | 可能 | 默认级别，每条语句一个快照 |
| `REPEATABLE READ` | 不会 | 不会 | 普通查询看同一快照 | 并发更新可能失败，需要重试 |
| `SERIALIZABLE` | 不会 | 不会 | 不会产生非串行化结果 | 可能序列化失败，需要重试 |

注意：表格是学习用总结。真实业务中是否正确，还要看 SQL 写法、约束、锁和重试策略。

---

## 八、业务中如何选择

### 1. 默认先用 READ COMMITTED

大多数业务使用 PostgreSQL 默认的 `READ COMMITTED` 就够了。

但要配合正确 SQL：

```sql
UPDATE demo.products
SET stock = stock - 1
WHERE id = 1
  AND stock >= 1
RETURNING id, stock;
```

这种原子更新通常比“提高隔离级别”更直接。

### 2. 需要稳定快照时用 REPEATABLE READ

例如一个事务里要生成一组互相关联的统计结果，希望所有查询基于同一个数据快照，可以考虑 `REPEATABLE READ`。

### 3. 复杂一致性要求考虑 SERIALIZABLE

当业务规则跨多行、多表，且难以用唯一约束、外键、检查约束、原子更新、行锁表达时，可以评估 `SERIALIZABLE`。

但应用层必须支持重试。

---

## 九、常见误区

### 1. 隔离级别越高越好？

不是。

隔离级别越高，能避免的并发异常越多，但也可能带来更多冲突、等待、失败和重试。

### 2. SERIALIZABLE 就不会报错了吗？

恰恰相反。

为了保证结果可串行化，PostgreSQL 可能主动中止某个事务，让应用重试。

### 3. READ COMMITTED 不安全吗？

不能这么说。

`READ COMMITTED` 是很多生产系统的默认选择。关键是要在重要写入场景中使用正确的 SQL、约束和锁。

### 4. 提高隔离级别能替代业务约束吗？

不能。

例如邮箱唯一、余额非负、订单必须引用真实商品，这些仍然应该用数据库约束表达。

## 十、本节达标标准

学完本节后，你应该能够做到：

- 使用 `SHOW transaction_isolation` 查看隔离级别。
- 知道 PostgreSQL 默认隔离级别是 `READ COMMITTED`。
- 知道 PostgreSQL 不会脏读。
- 解释 `READ COMMITTED` 每条语句使用新快照的含义。
- 解释 `REPEATABLE READ` 事务内快照稳定的含义。
- 知道 `SERIALIZABLE` 追求结果像串行执行。
- 知道高隔离级别下可能需要重试整个事务。
- 能根据业务场景初步选择隔离级别。
