# 7. 长事务的风险

本节目标：理解长事务为什么危险，知道长事务会影响锁、连接、MVCC 清理和系统稳定性，并掌握避免长事务的基本实践。

长事务指的是：

```text
事务开启后，长时间不提交，也不回滚。
```

它不一定是执行了很多 SQL。有时只是：

```sql
BEGIN;

SELECT ...

-- 开着不管了
```

这也可能成为长事务。

---

## 一、长事务为什么危险

事务开始后，数据库需要为它维护一系列状态。

如果事务很快结束，问题不大。

如果事务长时间挂着，会带来很多隐性成本：

- 持有锁，阻塞其他事务。
- 占用数据库连接。
- 保留很老的 MVCC 快照。
- 阻碍 `VACUUM` 清理旧版本。
- 导致表和索引膨胀。
- 增加死锁和超时概率。
- 让线上问题更难排查。

---

## 二、长事务会长时间持有锁

会话 A：

```sql
BEGIN;

UPDATE demo.accounts
SET balance = balance - 100
WHERE owner_name = 'Alice';

-- 这里故意不 COMMIT
```

会话 B：

```sql
UPDATE demo.accounts
SET balance = balance - 50
WHERE owner_name = 'Alice';
```

会话 B 会等待。

如果会话 A 一直不提交，会话 B 就一直卡住。

这在后端接口里表现为：

```text
请求迟迟不返回。
连接池被占满。
后续请求继续排队。
系统雪崩。
```

---

## 三、长事务会占用连接

后端通常通过连接池访问数据库。

每个正在执行事务的请求，都会占用一个数据库连接。

如果事务中做了慢操作：

```text
开启事务。
更新订单。
调用第三方支付接口，等待 5 秒。
更新支付流水。
提交事务。
```

这 5 秒里连接一直被占用，锁也可能一直持有。

高并发下，连接池很容易被耗尽。

正确思路通常是：

```text
事务只处理本地数据库一致性。
外部调用放在事务外，或用状态机、消息队列、补偿任务处理。
```

---

## 四、长事务会阻碍 VACUUM

PostgreSQL 使用 MVCC。更新或删除数据时，旧版本不会立刻消失，而是等待 `VACUUM` 清理。

但如果有一个很老的事务还没结束，PostgreSQL 必须假设它可能还需要看到旧版本。

于是旧版本不能被清理。

这会导致：

```text
表越来越膨胀。
索引越来越膨胀。
查询扫描更多无效版本。
性能逐渐变差。
```

这类问题很隐蔽，因为长事务本身可能只是一条很普通的查询，但它让整个库背上了历史包袱。

---

## 五、常见导致长事务的写法

### 1. 开启事务后等待用户操作

错误思路：

```text
BEGIN
创建订单草稿
等待用户输入验证码
扣库存
COMMIT
```

用户可能几十秒甚至几分钟才输入验证码。

事务不能这么开着等人。

### 2. 事务中调用外部接口

错误思路：

```text
BEGIN
扣减库存
调用支付接口
创建订单
COMMIT
```

支付接口可能慢、失败、超时，而且数据库回滚无法撤销外部系统的动作。

### 3. 事务中执行大量计算

错误思路：

```text
BEGIN
查询大量数据
在应用内循环计算几分钟
更新结果
COMMIT
```

可以考虑把计算放到事务外，事务内只读取必要数据、校验版本并写入结果。

### 4. 手动打开事务后忘记提交

学习或排查时，在 SQL 客户端里执行：

```sql
BEGIN;

SELECT ...
```

然后去做别的事，忘了 `COMMIT` 或 `ROLLBACK`。

这在开发环境和生产排查中都很常见。

## 六、如何发现长事务

可以查询 `pg_stat_activity`：

```sql
SELECT
    pid,
    usename,
    state,
    xact_start,
    now() - xact_start AS xact_age,
    query
FROM pg_stat_activity
WHERE xact_start IS NOT NULL
ORDER BY xact_start;
```

字段含义：

- `pid`：数据库后端进程 ID。
- `usename`：数据库用户。
- `state`：连接状态。
- `xact_start`：事务开始时间。
- `xact_age`：事务已经持续多久。
- `query`：当前或最后执行的 SQL。

如果看到某个事务持续了很久，就要排查它是否正常。

注意：生产环境中不要随便终止事务。先确认来源和影响。

## 七、事务超时设置

可以通过超时配置减少长事务风险。

### 1. statement_timeout

限制单条 SQL 执行时间：

```sql
SET statement_timeout = '5s';
```

### 2. lock_timeout

限制等待锁的时间：

```sql
SET lock_timeout = '3s';
```

### 3. idle_in_transaction_session_timeout

限制事务开启后空闲太久：

```sql
SET idle_in_transaction_session_timeout = '30s';
```

这个配置对防止“BEGIN 后忘记提交”很有帮助。

在业务代码里，也可以在每个事务中使用：

```sql
SET LOCAL lock_timeout = '3s';
SET LOCAL statement_timeout = '5s';
```

`SET LOCAL` 只在当前事务内生效。

## 八、避免长事务的实践原则

### 1. 事务要短

事务里只放必须原子执行的数据库操作。

### 2. 先准备数据，再开启事务

例如请求参数校验、权限初步检查、外部数据准备，可以尽量放在事务前。

### 3. 事务中不要做外部网络调用

尤其是支付、短信、HTTP API、对象存储等。

### 4. 事务中不要等待用户

用户操作和数据库事务不是一个时间尺度。

### 5. 出错一定要回滚

后端代码中，开启事务后必须保证：

```text
成功就 COMMIT。
失败就 ROLLBACK。
```

不能让事务悬挂。

### 6. 大批量任务分批提交

如果要更新百万行，不要一个事务全部做完。可以按主键范围或分页分批处理。

## 九、常见误区

### 1. 只要没有 UPDATE，就不是长事务问题？

不是。

只读事务也可能持有旧快照，影响 MVCC 旧版本清理。

### 2. 长事务只影响自己吗？

不是。

长事务可能阻塞别人、占用连接、拖住 `VACUUM`，影响整个数据库。

### 3. 把所有业务都包进一个大事务更安全吗？

不是。

事务越大，持锁越久，失败回滚成本越高，系统越不稳定。正确做法是把事务边界划在真正需要原子性的地方。

### 4. 超时设置能替代正确代码吗？

不能。

超时是保护措施，不是事务设计。代码仍然要正确提交、回滚，并控制事务范围。

---

## 十、本节达标标准

学完本节后，你应该能够做到：

- 解释什么是长事务。
- 知道长事务会持有锁、占用连接、影响 `VACUUM`。
- 知道只读长事务也可能有风险。
- 使用 `pg_stat_activity` 查看长事务。
- 知道 `statement_timeout`、`lock_timeout`、`idle_in_transaction_session_timeout` 的作用。
- 知道事务中不要调用外部服务或等待用户。
- 能把事务边界控制在必要的数据库操作范围内。
