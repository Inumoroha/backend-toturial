# 1. 数据库迁移是什么

本节目标：理解数据库迁移的意义，知道为什么表结构变化应该通过迁移文件管理，而不是依赖手动 SQL。

数据库迁移可以理解为：

```text
用一组有顺序的文件，描述数据库结构如何一步一步变化。
```

例如：

```text
第 1 步：创建 users 表。
第 2 步：给 users 表增加 avatar_url 字段。
第 3 步：创建 posts 表。
第 4 步：给 posts.user_id 增加外键。
```

每一步都写成迁移文件，并提交到 Git。

---

## 一、没有迁移会发生什么

假设项目一开始有一张用户表：

```sql
CREATE TABLE users (
    id bigserial PRIMARY KEY,
    email text NOT NULL UNIQUE
);
```

后来你需要增加用户名：

```sql
ALTER TABLE users ADD COLUMN username text;
```

如果你只在本地手动执行，那么：

- 其他同事的本地数据库没有这个字段。
- 测试环境没有这个字段。
- 生产环境什么时候加、谁加、是否成功，没有统一记录。
- 回滚时不知道该执行什么。

代码里可能已经写了：

```sql
SELECT id, email, username FROM users;
```

但测试环境表里没有 `username`，程序就会报错。

---

## 二、迁移文件解决什么问题

迁移文件让数据库变化变成可追踪的历史。

例如：

```text
migrations/
  00001_create_users.sql
  00002_add_users_username.sql
  00003_create_posts.sql
```

每个文件都表示一次变更。

好处：

- 所有人按同一套文件创建数据库。
- 新同事可以一键初始化本地环境。
- CI 可以自动创建测试数据库。
- 测试环境和生产环境可以按版本升级。
- 代码评审时能看到数据库变化。
- 回滚时有明确脚本。

---

## 三、迁移和普通 SQL 脚本的区别

普通 SQL 脚本：

```text
执行过没有？
执行到哪一步？
执行失败怎么办？
能不能重复执行？
谁来记录版本？
```

这些通常要靠人记。

迁移工具会记录版本。

例如数据库中有版本表：

```text
version | applied_at
00001   | 2026-07-05 10:00:00
00002   | 2026-07-05 10:05:00
```

下次执行迁移时，工具会知道：

```text
00001 已执行。
00002 已执行。
00003 还没执行。
```

然后只执行未执行的迁移。

---

## 四、迁移不是只用来建表

迁移可以做很多结构变化：

- 创建 schema。
- 创建表。
- 删除表。
- 新增字段。
- 删除字段。
- 修改字段类型。
- 添加索引。
- 删除索引。
- 添加约束。
- 删除约束。
- 创建枚举类型。
- 创建扩展。
- 创建触发器和函数。

例如：

```sql
ALTER TABLE app.users
ADD COLUMN avatar_url text;
```

也应该写进迁移，而不是手动执行完就结束。

---

## 五、迁移和业务代码的关系

数据库结构和业务代码必须同步。

例如代码里新增了用户头像字段：

```go
type User struct {
    ID        int64
    Email     string
    AvatarURL *string
}
```

对应数据库也要有：

```sql
avatar_url text
```

这类变更通常应该在同一个 PR 里出现：

```text
业务代码变更。
迁移文件变更。
测试变更。
文档或接口说明变更。
```

这样代码合并后，数据库也有明确升级路径。

---

## 六、迁移应该进入 Git

迁移文件是代码的一部分。

应该提交：

```text
migrations/00001_create_users.sql
migrations/00002_add_users_avatar_url.sql
```

不应该只保存在：

- 本地电脑。
- 数据库客户端历史记录。
- 聊天记录。
- 个人笔记。

数据库结构变化必须可追踪、可审查、可复现。

---

## 七、迁移文件应该小而清晰

不推荐一个迁移里同时做很多无关事情：

```text
创建用户表。
修改订单表。
删除评论字段。
创建索引。
插入测试数据。
```

更推荐按业务变更拆分：

```text
00001_create_users
00002_add_users_avatar_url
00003_create_posts
00004_add_posts_user_id_index
```

每个迁移尽量表达一个清晰意图。

---

## 八、什么时候创建迁移

当你要改变数据库结构时，就应该创建迁移。

例如：

- 新功能需要新表。
- 新接口需要新字段。
- 查询变慢，需要新增索引。
- 字段约束需要调整。
- 删除废弃字段。
- 添加枚举值。

不要等到功能写完才想起数据库改动。

更好的习惯是：

```text
先设计表结构。
写迁移。
执行迁移。
再写 Go 代码。
再写测试。
```

## 九、常见误区

### 1. 小项目不需要迁移吗？

不建议这么想。

越早使用迁移，越早形成稳定习惯。小项目也会长大。

### 2. 我记得执行过哪些 SQL，就不用迁移？

人的记忆不可靠。

迁移文件和版本表才可靠。

### 3. 迁移只管生产环境？

不是。

本地、测试、CI、生产都应该使用同一套迁移文件。

### 4. 迁移文件可以随便改旧文件吗？

已执行过、已合并、已上线的迁移不应该随便改。需要新变更时，新增迁移文件。

---

## 十、本节达标标准

学完本节后，你应该能够做到：

- 解释数据库迁移是什么。
- 解释为什么手动改表不可靠。
- 知道迁移文件应该进入 Git。
- 知道迁移工具会记录已执行版本。
- 知道迁移不只是建表，也包括索引、约束、字段变化。
- 知道数据库结构变化应该和业务代码一起提交。
- 知道已执行的旧迁移不应该随便修改。
