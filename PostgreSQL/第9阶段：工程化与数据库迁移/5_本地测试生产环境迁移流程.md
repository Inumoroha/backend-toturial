# 5. 本地、测试、生产环境迁移流程

本节目标：理解同一套迁移文件在本地、测试、生产环境中的使用方式，知道不同环境执行迁移时应该注意什么。

迁移文件应该只有一套：

```text
migrations/
```

但它会被执行到不同环境：

```text
本地开发数据库。
CI 测试数据库。
测试环境数据库。
预发环境数据库。
生产环境数据库。
```

不同环境的目标不同，流程也不同。

---

## 一、本地环境

本地环境目标：

```text
方便开发。
方便重建。
方便回滚和试错。
```

常见操作：

```powershell
goose -dir migrations postgres $env:DATABASE_URL up
goose -dir migrations postgres $env:DATABASE_URL down
goose -dir migrations postgres $env:DATABASE_URL status
```

或者：

```powershell
migrate -path migrations -database $env:DATABASE_URL up
migrate -path migrations -database $env:DATABASE_URL down 1
migrate -path migrations -database $env:DATABASE_URL version
```

本地可以比较自由地：

- 重建数据库。
- 回滚迁移。
- 修改尚未提交的迁移。
- 插入本地种子数据。

但要注意：

```text
只要迁移已经提交、合并、被别人使用，就不要随便改旧文件。
```

---

## 二、CI 测试环境

CI 环境目标：

```text
每次测试从干净数据库开始，自动执行所有迁移，再运行测试。
```

流程通常是：

```text
1. 创建临时测试数据库。
2. 执行 migrations up。
3. 运行 Go 测试。
4. 删除测试数据库。
```

CI 中不应该依赖手动建表。

如果 CI 通过迁移创建数据库，说明：

```text
新同事和新环境也能通过迁移得到正确表结构。
```

---

## 三、测试环境

测试环境目标：

```text
让 QA、后端、前端在接近真实部署的环境中验证功能。
```

测试环境迁移通常由部署流程自动执行：

```text
部署应用前或部署过程中执行 migrations up。
```

注意：

- 测试环境也不要手动乱改表。
- 如果测试环境出现结构异常，优先查迁移历史。
- 测试环境可以重建，但也要尽量模拟生产流程。

---

## 四、预发环境

预发环境目标：

```text
在接近生产的数据量和配置下验证迁移是否安全。
```

预发环境尤其适合检查：

- 迁移是否耗时过长。
- 是否锁表。
- 是否影响旧版本应用。
- 是否需要拆成多步上线。
- 是否需要提前创建索引。

如果项目有预发环境，生产迁移前最好先在预发跑一遍。

---

## 五、生产环境

生产环境目标：

```text
安全、可控、可观察地升级数据库结构。
```

生产迁移最重要的不是“能不能执行成功”，而是：

```text
执行过程中是否阻塞业务？
失败后如何处理？
应用代码和数据库结构是否兼容？
是否有备份和回滚方案？
```

生产执行前应该确认：

- 迁移文件已在测试环境执行过。
- 涉及大表时评估锁和耗时。
- 是否有数据备份。
- 应用代码是否兼容迁移前后的结构。
- 是否有监控和日志。
- 是否知道失败后如何处理。

---

## 六、应用部署和迁移的顺序

有些变更可以直接：

```text
先迁移数据库。
再部署应用。
```

例如新增一个 nullable 字段：

```sql
ALTER TABLE app.users
ADD COLUMN avatar_url text;
```

旧代码不会使用它，新代码上线后再使用。

但有些变更不能简单一次完成。

例如把字段 `name` 改名为 `username`，如果直接：

```sql
ALTER TABLE users RENAME COLUMN name TO username;
```

旧代码还在读 `name`，就会报错。

更安全的流程通常是多步：

```text
1. 新增 username 字段。
2. 让应用同时写 name 和 username。
3. 回填历史数据。
4. 让应用改读 username。
5. 确认无旧代码依赖 name。
6. 删除 name 字段。
```

这叫兼容式迁移。

---

## 七、向前兼容和向后兼容

生产迁移要考虑两个方向：

### 向前兼容

旧代码能在新数据库结构上运行。

例如新增 nullable 字段通常向前兼容。

### 向后兼容

新代码能在旧数据库结构上运行。

例如新代码立刻查询新字段，而数据库还没迁移，就不兼容。

部署系统中，应用实例可能不是同时切换。短时间内可能新旧代码共存，所以兼容性很重要。

---

## 八、迁移执行权限

生产环境中，迁移通常不应该由开发者随手在本地连接生产执行。

更推荐：

- CI/CD 流程执行。
- 运维平台执行。
- 发布系统执行。
- 有审批和日志记录。

这样可以知道：

- 谁执行了迁移。
- 什么时候执行。
- 执行结果如何。
- 失败时谁处理。

---

## 九、不同环境的配置

不要把生产连接字符串写进迁移文件。

迁移命令应该从环境变量读取：

```powershell
$env:DATABASE_URL="postgres://..."
```

或在 CI/CD 中通过 secret 注入。

迁移文件只描述结构变化，不包含环境密码。

---

## 十、常见误区

### 1. 本地能跑成功，生产就一定没问题？

不是。

生产数据量、并发、锁等待和权限都可能不同。

### 2. 迁移一定要和应用同时部署吗？

不一定。

很多安全变更会先迁移数据库，再部署应用，或拆成多阶段发布。

### 3. 生产迁移失败就直接 down？

不一定。

要看失败发生在哪一步、是否已写入数据、应用是否已部署。有时向前修复比回滚更安全。

### 4. 测试环境可以手动改表，生产再用迁移？

不推荐。

测试环境也应该验证迁移流程，否则生产第一次执行风险更高。

---

## 十一、本节达标标准

学完本节后，你应该能够做到：

- 说明本地、CI、测试、预发、生产环境迁移目标的差异。
- 知道 CI 应该从空数据库执行完整迁移。
- 知道生产迁移要关注锁、耗时、备份和兼容性。
- 理解兼容式迁移的多步发布思路。
- 知道迁移命令应通过环境变量或 secret 获取连接信息。
- 知道生产迁移应该有审计和发布流程。
