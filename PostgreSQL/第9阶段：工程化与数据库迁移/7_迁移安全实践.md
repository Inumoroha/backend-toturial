# 7. 迁移安全实践

本节目标：理解生产数据库迁移中的风险，掌握大表变更、锁、回滚、不可逆操作和索引创建的基本安全原则。

本地迁移通常很顺利：

```text
表很小。
没有并发请求。
失败了可以重建数据库。
```

生产环境完全不同：

```text
表可能有几千万行。
业务请求正在读写。
迁移可能持锁。
失败会影响用户。
数据不能随便丢。
```

所以迁移必须考虑安全。

---

## 一、生产迁移的主要风险

常见风险：

- 锁表导致请求阻塞。
- 大表变更耗时很久。
- 迁移失败导致版本处于中间状态。
- 删除字段或数据后无法恢复。
- 新旧应用代码不兼容。
- 索引创建占用大量资源。
- 回滚方案不完整。

迁移评审时应该逐项考虑这些问题。

---

## 二、新增 nullable 字段通常比较安全

例如：

```sql
ALTER TABLE app.users
ADD COLUMN avatar_url text;
```

这是比较安全的变更：

- 旧代码不受影响。
- 新字段允许 NULL。
- 通常不需要回填所有历史数据。

推荐流程：

```text
1. 先加 nullable 字段。
2. 部署新代码开始写入新字段。
3. 如果需要，再异步回填历史数据。
4. 最后考虑加 NOT NULL 或约束。
```

---

## 三、谨慎添加 NOT NULL

危险写法：

```sql
ALTER TABLE app.users
ADD COLUMN avatar_url text NOT NULL;
```

如果表里已有数据，新增非空字段没有默认值会失败。

如果加默认值，旧版本 PostgreSQL 或某些复杂场景可能导致大表重写或长时间锁。

更稳的多步流程：

```sql
ALTER TABLE app.users
ADD COLUMN avatar_url text;
```

然后应用代码开始写入。

再分批回填：

```sql
UPDATE app.users
SET avatar_url = ''
WHERE avatar_url IS NULL
LIMIT ...
```

PostgreSQL 不支持直接在 `UPDATE` 中写 `LIMIT`，真实分批通常按主键范围做：

```sql
UPDATE app.users
SET avatar_url = ''
WHERE id >= 1
  AND id < 10000
  AND avatar_url IS NULL;
```

确认没有 NULL：

```sql
SELECT COUNT(*)
FROM app.users
WHERE avatar_url IS NULL;
```

最后加约束：

```sql
ALTER TABLE app.users
ALTER COLUMN avatar_url SET NOT NULL;
```

---

## 四、谨慎删除字段

删除字段：

```sql
ALTER TABLE app.users
DROP COLUMN bio;
```

风险：

- 数据丢失。
- 旧代码如果还读这个字段会报错。
- 回滚无法恢复原始数据。

更安全流程：

```text
1. 先让应用停止读写 bio。
2. 观察一段时间。
3. 确认没有代码依赖。
4. 备份或导出数据。
5. 再删除字段。
```

很多团队会采用“废弃字段保留一段时间”的策略。

---

## 五、创建索引要注意锁和耗时

普通创建索引：

```sql
CREATE INDEX users_email_idx
ON app.users (email);
```

在大表上可能影响写入。

PostgreSQL 支持并发创建索引：

```sql
CREATE INDEX CONCURRENTLY users_email_idx
ON app.users (email);
```

`CONCURRENTLY` 可以降低对写入的阻塞，但也有注意点：

- 不能在普通事务块中执行。
- 执行时间可能更长。
- 失败后可能留下无效索引，需要清理。

很多迁移工具默认把迁移放在事务中执行。遇到 `CREATE INDEX CONCURRENTLY` 时，要确认工具是否支持关闭事务或特殊标记。

goose 可以通过注释控制某些迁移不在事务中执行，具体要看工具版本和配置。生产使用前一定要在预发环境验证。

---

## 六、修改字段类型要谨慎

例如：

```sql
ALTER TABLE app.users
ALTER COLUMN id TYPE text;
```

字段类型修改可能：

- 重写整张表。
- 持有较强锁。
- 导致应用代码不兼容。
- 转换失败。

更安全的流程通常是：

```text
1. 新增新类型字段。
2. 双写新旧字段。
3. 分批回填。
4. 应用改读新字段。
5. 删除旧字段。
```

这比直接改类型更麻烦，但生产风险更可控。

---

## 七、数据回填不要放进一个巨大事务

危险做法：

```sql
UPDATE app.users
SET avatar_url = ''
WHERE avatar_url IS NULL;
```

如果表非常大，这条 SQL 可能：

- 执行很久。
- 产生大量 WAL。
- 持有大量行锁。
- 影响 autovacuum。
- 造成复制延迟。

更好的做法：

```text
按主键范围分批。
每批提交。
控制速率。
监控影响。
```

例如：

```sql
UPDATE app.users
SET avatar_url = ''
WHERE id >= 1
  AND id < 10000
  AND avatar_url IS NULL;
```

然后下一批：

```sql
WHERE id >= 10000
  AND id < 20000
```

---

## 八、使用 lock_timeout 和 statement_timeout

生产迁移可以设置超时，避免无限等待。

```sql
SET lock_timeout = '5s';
SET statement_timeout = '5min';
```

含义：

- `lock_timeout`：等待锁超过时间就失败。
- `statement_timeout`：语句执行超过时间就失败。

这可以避免迁移长时间卡住业务。

但超时不是万能的。迁移失败后仍然需要知道如何处理。

---

## 九、不可逆迁移要明确标注

例如：

```sql
ALTER TABLE app.users
DROP COLUMN bio;
```

down：

```sql
-- This down migration restores the column structure only.
-- It cannot restore dropped bio data.
ALTER TABLE app.users
ADD COLUMN bio text;
```

这样评审者能看到风险。

不要写一个看似完整、实际无法恢复数据的 down，给团队造成错觉。

---

## 十、迁移前备份

高风险迁移前应该有备份。

尤其是：

- 删除字段。
- 删除表。
- 大规模更新数据。
- 修改字段类型。
- 拆表合表。

备份策略取决于部署环境。

学习阶段先记住：

```text
生产高风险迁移前，必须能回答：如果失败，数据如何恢复？
```

---

## 十一、常见误区

### 1. ALTER TABLE 都很快吗？

不是。

不同 ALTER 操作成本差别很大，大表上尤其要谨慎。

### 2. CREATE INDEX 一定不影响业务吗？

不是。

大表创建索引会消耗 IO、CPU，并可能影响写入。需要评估是否使用 `CONCURRENTLY`。

### 3. down 写了就一定安全？

不是。

down 可能只能恢复结构，不能恢复数据。

### 4. 本地执行 1 秒，生产也差不多？

不是。

生产数据量和并发完全不同。

---

## 十二、本节达标标准

学完本节后，你应该能够做到：

- 说出生产迁移的常见风险。
- 知道新增 nullable 字段通常更安全。
- 知道加 NOT NULL 可以拆成多步。
- 知道删除字段通常不可完整回滚。
- 知道大表索引可以考虑 `CREATE INDEX CONCURRENTLY`。
- 知道大批量回填应该分批执行。
- 使用 `lock_timeout` 和 `statement_timeout` 控制迁移风险。
- 能在迁移文件中明确标注不可逆风险。
