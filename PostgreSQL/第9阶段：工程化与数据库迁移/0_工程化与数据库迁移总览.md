# 0. 工程化与数据库迁移总览

本阶段目标：用工程化方式管理数据库结构变化，而不是手动改表。

前面几个阶段已经学习了：

```text
如何设计表。
如何写查询。
如何处理事务。
如何分析性能。
如何在 Go 中操作 PostgreSQL。
```

但真实项目还有一个非常关键的问题：

```text
数据库表结构会不断变化。
这些变化应该如何被记录、审查、执行、回滚和同步到不同环境？
```

这就是数据库迁移要解决的问题。

---

## 一、为什么不能只靠手动改表

学习阶段，你可能经常这样操作：

```sql
ALTER TABLE users ADD COLUMN avatar_url text;
```

在本地数据库里手动执行，确实能生效。

但真实团队里会遇到这些问题：

```text
张三本地执行了 ALTER TABLE，李四不知道。
测试环境改了，生产环境忘了改。
生产环境手动改错了，没人知道执行过什么。
某个字段上线后发现有问题，不知道怎么回滚。
新同事拉代码后，不知道数据库应该是什么结构。
CI 测试需要自动创建表，却没有统一脚本。
```

所以数据库结构变化必须像代码一样管理。

迁移文件的作用就是：

```text
把数据库结构变化写成文件。
提交到 Git。
按顺序执行。
记录执行状态。
必要时可以回滚。
```

## 二、本阶段核心内容

本阶段建议拆成以下主题学习：

1. 数据库迁移是什么，以及它解决什么问题。
2. 迁移文件的 `up/down` 思想。
3. 使用 goose 管理 PostgreSQL 迁移。
4. 使用 golang-migrate 管理 PostgreSQL 迁移。
5. 本地、测试、生产环境的迁移流程。
6. 种子数据和测试数据的区别。
7. 迁移安全实践：大表变更、锁、回滚、不可逆操作。
8. 在 Go 项目中组织 migrations、README、Makefile。
9. 综合实践：创建 `users` 表、添加 `avatar_url`、回滚并重新执行。

---

## 三、推荐学习顺序

建议按下面顺序学习：

```text
迁移概念
-> up/down
-> goose
-> golang-migrate
-> 多环境流程
-> seed 和测试数据
-> 安全实践
-> 项目落地模板
-> 综合练习
```

原因是：

- 先理解为什么需要迁移。
- 再理解迁移文件如何表达变化。
- 然后学习两个常见工具。
- 接着把工具放进本地、测试、生产流程。
- 最后形成项目模板，能直接落地。

## 四、本阶段会用到的示例项目结构

建议从这个结构开始：

```text
go-pg-demo/
  go.mod
  main.go
  migrations/
    00001_create_users.sql
    00002_add_users_avatar_url.sql
  seeds/
    local_seed.sql
  Makefile
  README.md
```

如果使用 golang-migrate，它通常生成成对文件：

```text
migrations/
  000001_create_users.up.sql
  000001_create_users.down.sql
  000002_add_users_avatar_url.up.sql
  000002_add_users_avatar_url.down.sql
```

如果使用 goose，SQL 文件通常把 up/down 写在同一个文件里：

```text
migrations/
  00001_create_users.sql
  00002_add_users_avatar_url.sql
```

两种方式都可以。重点不是工具名字，而是：

```text
迁移文件必须进入版本控制。
迁移必须可重复执行。
迁移状态必须由工具记录。
不同环境必须按同一套文件升级。
```

## 五、统一练习表

本阶段继续使用用户模块。

最终希望得到的表结构：

```sql
CREATE TABLE app.users (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    email text NOT NULL,
    username text NOT NULL,
    display_name text,
    bio text,
    avatar_url text,
    created_at timestamptz NOT NULL DEFAULT now(),
    updated_at timestamptz NOT NULL DEFAULT now(),
    deleted_at timestamptz,
    CONSTRAINT users_email_unique UNIQUE (email),
    CONSTRAINT users_email_not_empty_check CHECK (length(trim(email)) > 0),
    CONSTRAINT users_username_not_empty_check CHECK (length(trim(username)) > 0)
);
```

但我们不会一次手动建成，而是拆成迁移：

```text
00001_create_users
00002_add_users_avatar_url
```

这样就能练习：

- 创建表。
- 新增字段。
- 回滚新增字段。
- 重新执行迁移。

## 六、迁移工具会记录什么

迁移工具通常会在数据库里创建一张版本记录表。

例如 goose 默认使用类似：

```text
goose_db_version
```

golang-migrate 默认使用类似：

```text
schema_migrations
```

这张表记录：

```text
哪些迁移已经执行。
当前数据库处在哪个版本。
某次迁移是否失败或脏状态。
```

所以不要手动乱改迁移版本表，除非你非常清楚自己在修复什么。

## 七、本阶段最重要的直觉

### 1. 迁移文件是数据库结构的历史记录

它不是临时脚本，而是项目历史的一部分。

### 2. 已经合并或上线的迁移不要随便改

如果要继续变更，新增迁移文件，而不是偷偷修改旧迁移。

### 3. 迁移要小步、清晰、可审查

不要一个迁移里塞几十个无关变更。

### 4. 生产迁移要考虑锁和数据量

本地执行很快的 `ALTER TABLE`，生产大表上可能造成阻塞。

### 5. 回滚不是魔法

有些操作可以回滚，例如新增字段。

有些操作很难安全回滚，例如删除字段、删除数据、修改字段类型。

## 八、本阶段达标标准

学完第九阶段后，你应该能够做到：

- 解释数据库迁移解决什么问题。
- 理解 `up/down` 的含义。
- 使用 goose 创建、执行、回滚迁移。
- 使用 golang-migrate 创建、执行、回滚迁移。
- 知道本地、测试、生产环境的迁移流程差异。
- 区分种子数据和测试数据。
- 知道已上线迁移不要随便修改。
- 能写出创建 `users` 表和新增 `avatar_url` 字段的迁移。
- 能把迁移命令写进 README 或 Makefile。
- 知道生产迁移要关注锁、长事务、大表和不可逆操作。
