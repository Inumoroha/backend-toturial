# 9. 综合实践：users 表迁移

本节目标：完整练习一次数据库迁移流程：创建 `users` 表、新增 `avatar_url` 字段、回滚、重新执行，并把命令写进项目文档。

本节以 goose 为主展示，最后给出 golang-migrate 版本。

---

## 一、目标结构

最终表结构：

```sql
CREATE TABLE app.users (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    email text NOT NULL,
    username text NOT NULL,
    display_name text,
    bio text,
    avatar_url text,
    created_at timestamptz NOT NULL DEFAULT now(),
    updated_at timestamptz NOT NULL DEFAULT now(),
    deleted_at timestamptz,
    CONSTRAINT users_email_unique UNIQUE (email),
    CONSTRAINT users_email_not_empty_check CHECK (length(trim(email)) > 0),
    CONSTRAINT users_username_not_empty_check CHECK (length(trim(username)) > 0)
);
```

拆成两步：

```text
00001_create_users
00002_add_users_avatar_url
```

## 二、goose 版本：创建目录

```powershell
mkdir migrations
```

设置连接字符串：

```powershell
$env:DATABASE_URL="postgres://postgres:postgres@localhost:5432/learn_pg?sslmode=disable"
```

---

## 三、goose 版本：创建第一个迁移

```powershell
goose -dir migrations -s create create_users sql
```

生成：

```text
migrations/00001_create_users.sql
```

内容：

```sql
-- +goose Up
-- +goose StatementBegin
CREATE SCHEMA IF NOT EXISTS app;

CREATE TABLE app.users (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    email text NOT NULL,
    username text NOT NULL,
    display_name text,
    bio text,
    created_at timestamptz NOT NULL DEFAULT now(),
    updated_at timestamptz NOT NULL DEFAULT now(),
    deleted_at timestamptz,
    CONSTRAINT users_email_unique UNIQUE (email),
    CONSTRAINT users_email_not_empty_check CHECK (length(trim(email)) > 0),
    CONSTRAINT users_username_not_empty_check CHECK (length(trim(username)) > 0)
);

CREATE INDEX users_deleted_at_idx
ON app.users (deleted_at);
-- +goose StatementEnd

-- +goose Down
-- +goose StatementBegin
DROP TABLE app.users;
-- +goose StatementEnd
```

## 四、goose 版本：执行第一个迁移

```powershell
goose -dir migrations postgres $env:DATABASE_URL up
```

检查表：

```sql
\d app.users
```

或者：

```sql
SELECT column_name, data_type
FROM information_schema.columns
WHERE table_schema = 'app'
  AND table_name = 'users'
ORDER BY ordinal_position;
```

查看迁移状态：

```powershell
goose -dir migrations postgres $env:DATABASE_URL status
```

## 五、goose 版本：新增 avatar_url 字段

创建第二个迁移：

```powershell
goose -dir migrations -s create add_users_avatar_url sql
```

生成：

```text
migrations/00002_add_users_avatar_url.sql
```

内容：

```sql
-- +goose Up
-- +goose StatementBegin
ALTER TABLE app.users
ADD COLUMN avatar_url text;
-- +goose StatementEnd

-- +goose Down
-- +goose StatementBegin
ALTER TABLE app.users
DROP COLUMN avatar_url;
-- +goose StatementEnd
```

执行：

```powershell
goose -dir migrations postgres $env:DATABASE_URL up
```

检查字段：

```sql
SELECT column_name, data_type
FROM information_schema.columns
WHERE table_schema = 'app'
  AND table_name = 'users'
  AND column_name = 'avatar_url';
```

---

## 六、goose 版本：回滚 avatar_url

回滚最近一条迁移：

```powershell
goose -dir migrations postgres $env:DATABASE_URL down
```

确认字段已删除：

```sql
SELECT column_name
FROM information_schema.columns
WHERE table_schema = 'app'
  AND table_name = 'users'
  AND column_name = 'avatar_url';
```

如果没有返回行，说明字段已删除。

---

## 七、goose 版本：重新执行

```powershell
goose -dir migrations postgres $env:DATABASE_URL up
```

再次确认：

```sql
SELECT column_name
FROM information_schema.columns
WHERE table_schema = 'app'
  AND table_name = 'users'
  AND column_name = 'avatar_url';
```

这一步验证：


## 十、golang-migrate 版本文件

如果使用 golang-migrate，则创建：

```powershell
migrate create -ext sql -dir migrations -seq create_users
migrate create -ext sql -dir migrations -seq add_users_avatar_url
```

文件：

```text
000001_create_users.up.sql
000001_create_users.down.sql
000002_add_users_avatar_url.up.sql
000002_add_users_avatar_url.down.sql
```

`000001_create_users.up.sql`：

```sql
CREATE SCHEMA IF NOT EXISTS app;

CREATE TABLE app.users (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    email text NOT NULL,
    username text NOT NULL,
    display_name text,
    bio text,
    created_at timestamptz NOT NULL DEFAULT now(),
    updated_at timestamptz NOT NULL DEFAULT now(),
    deleted_at timestamptz,
    CONSTRAINT users_email_unique UNIQUE (email),
    CONSTRAINT users_email_not_empty_check CHECK (length(trim(email)) > 0),
    CONSTRAINT users_username_not_empty_check CHECK (length(trim(username)) > 0)
);

CREATE INDEX users_deleted_at_idx
ON app.users (deleted_at);
```

`000001_create_users.down.sql`：

```sql
DROP TABLE app.users;
```

`000002_add_users_avatar_url.up.sql`：

```sql
ALTER TABLE app.users
ADD COLUMN avatar_url text;
```

`000002_add_users_avatar_url.down.sql`：

```sql
ALTER TABLE app.users
DROP COLUMN avatar_url;
```

执行：

```powershell
migrate -path migrations -database $env:DATABASE_URL up
```

回滚一步：

```powershell
migrate -path migrations -database $env:DATABASE_URL down 1
```

重新执行：

```powershell
migrate -path migrations -database $env:DATABASE_URL up
```

## 十二、常见误区

### 1. 练习时能手动 ALTER TABLE，项目里也可以？

不建议。

练习可以帮助理解 SQL，但项目结构变化应该进入迁移。

### 2. 回滚 avatar_url 没问题，所以生产也随便 down？

不能这么想。

如果生产已经写入头像数据，回滚删除字段会丢数据。

### 3. goose 和 migrate 文件可以混在一个 migrations 目录？

不建议。

一个项目选择一种工具。

### 4. README 命令不重要？

很重要。

工程化的目标是让团队稳定复现，而不是只让一个人会操作。
