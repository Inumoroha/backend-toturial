# 2. up/down 迁移思想

本节目标：理解迁移文件中的 `up` 和 `down`，掌握如何为常见结构变化编写正向迁移和回滚迁移。

迁移通常有两个方向：

```text
up：把数据库从旧版本升级到新版本。
down：把数据库从新版本回滚到旧版本。
```

例如：

```text
up：给 users 表增加 avatar_url 字段。
down：删除 users 表的 avatar_url 字段。
```

---

## 一、up 是什么

`up` 表示正向执行的数据库变化。

例如创建用户表：

```sql
CREATE TABLE app.users (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    email text NOT NULL,
    username text NOT NULL,
    created_at timestamptz NOT NULL DEFAULT now()
);
```

例如新增字段：

```sql
ALTER TABLE app.users
ADD COLUMN avatar_url text;
```

例如新增索引：

```sql
CREATE INDEX users_created_at_idx
ON app.users (created_at DESC);
```

---

## 二、down 是什么

`down` 表示撤销这次迁移。

创建表的回滚：

```sql
DROP TABLE app.users;
```

新增字段的回滚：

```sql
ALTER TABLE app.users
DROP COLUMN avatar_url;
```

新增索引的回滚：

```sql
DROP INDEX users_created_at_idx;
```

`down` 不是随便写反向 SQL，而是要认真思考：

```text
回滚后数据库是否能回到上一个版本能运行的状态？
是否会丢数据？
是否会影响已经上线的代码？
```

---

## 三、goose 的 up/down 写法

goose 通常把 up/down 写在同一个 SQL 文件里：

```sql
-- +goose Up
-- +goose StatementBegin
CREATE TABLE app.users (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    email text NOT NULL,
    username text NOT NULL,
    created_at timestamptz NOT NULL DEFAULT now()
);
-- +goose StatementEnd

-- +goose Down
-- +goose StatementBegin
DROP TABLE app.users;
-- +goose StatementEnd
```

`-- +goose Up` 后面是正向迁移。

`-- +goose Down` 后面是回滚迁移。

---

## 四、golang-migrate 的 up/down 写法

golang-migrate 通常把 up/down 分成两个文件：

```text
000001_create_users.up.sql
000001_create_users.down.sql
```

`up.sql`：

```sql
CREATE TABLE app.users (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    email text NOT NULL,
    username text NOT NULL,
    created_at timestamptz NOT NULL DEFAULT now()
);
```

`down.sql`：

```sql
DROP TABLE app.users;
```

本质上和 goose 一样，都是表达正向和反向变化。

---

## 五、常见变更的 down 怎么写

### 1. 创建表

up：

```sql
CREATE TABLE app.users (...);
```

down：

```sql
DROP TABLE app.users;
```

如果有其他表引用它，回滚时要先删除依赖表或外键。

### 2. 新增字段

up：

```sql
ALTER TABLE app.users
ADD COLUMN avatar_url text;
```

down：

```sql
ALTER TABLE app.users
DROP COLUMN avatar_url;
```

注意：删除字段会丢失该字段数据。

### 3. 新增索引

up：

```sql
CREATE INDEX users_email_idx
ON app.users (email);
```

down：

```sql
DROP INDEX users_email_idx;
```

### 4. 新增约束

up：

```sql
ALTER TABLE app.users
ADD CONSTRAINT users_email_not_empty_check
CHECK (length(trim(email)) > 0);
```

down：

```sql
ALTER TABLE app.users
DROP CONSTRAINT users_email_not_empty_check;
```

### 5. 创建枚举类型

up：

```sql
CREATE TYPE app.user_status AS ENUM ('active', 'disabled');
```

down：

```sql
DROP TYPE app.user_status;
```

如果有表字段正在使用这个枚举类型，必须先删除字段或表。

---

## 六、有些 down 是危险的

例如 up 是：

```sql
ALTER TABLE app.users
DROP COLUMN bio;
```

理论上的 down 可能是：

```sql
ALTER TABLE app.users
ADD COLUMN bio text;
```

但这只能恢复字段结构，无法恢复原来的 `bio` 数据。

这类迁移叫不可逆或部分不可逆。

遇到不可逆迁移时，应该在文件里明确说明：

```sql
-- Down migration cannot restore dropped bio data.
ALTER TABLE app.users
ADD COLUMN bio text;
```

不要假装它能完整回滚。

---

## 七、回滚要和代码版本匹配

数据库回滚不是单独动作。

假设新代码依赖：

```sql
avatar_url
```

如果你执行 down 删除了 `avatar_url`，但应用代码还是新版本，就会报错。

正确回滚通常包括：

```text
回滚应用代码。
回滚数据库迁移。
确认旧代码能兼容旧表结构。
```

生产环境回滚要按发布流程处理，不能只对数据库执行 `down`。

---

## 八、什么时候可以不写完整 down

学习阶段建议都写 down。

但真实生产中，有些团队对不可逆迁移会选择：

```text
down 中只报错或写注释。
生产不依赖自动 down 回滚，而采用向前修复。
```

例如删除大量数据、复杂字段类型转换、拆表迁移等，自动回滚可能比向前修复更危险。

不过即使不能完整回滚，也要在迁移评审中说明风险。

---

## 九、常见误区

### 1. down 一定能恢复所有数据吗？

不是。

结构可以恢复，数据不一定能恢复。

### 2. 新增字段的 down 很简单，所以没风险？

也要看字段是否已经写入重要数据。删除字段会丢数据。

### 3. 回滚数据库就等于回滚系统？

不是。

系统回滚通常还包括应用代码、配置、任务、缓存等。

### 4. 没写 down 也没关系？

学习阶段不建议这样。写 down 能训练你理解每次结构变化的反向影响。

## 十、本节达标标准

学完本节后，你应该能够做到：

- 解释 `up` 和 `down` 的含义。
- 为创建表、新增字段、新增索引、新增约束编写 down。
- 理解 goose 和 golang-migrate 的 up/down 文件组织差异。
- 知道删除字段和删除数据通常不可完整恢复。
- 知道数据库回滚要和应用代码版本匹配。
- 能在迁移中标注不可逆风险。
