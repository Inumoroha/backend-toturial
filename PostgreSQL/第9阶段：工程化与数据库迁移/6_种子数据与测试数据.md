# 6. 种子数据与测试数据

本节目标：区分种子数据和测试数据，知道哪些数据应该随环境初始化，哪些数据只应该在测试中使用。

数据库工程化不只包括表结构，还包括数据准备。

但并不是所有数据都应该放进迁移。

常见数据可以分成：

```text
迁移数据。
种子数据。
测试数据。
业务数据。
```

---

## 一、什么是种子数据

种子数据是应用运行所需的基础数据。

例如：

- 默认角色。
- 系统配置。
- 固定字典。
- 默认管理员账号，视项目安全策略而定。
- 国家地区列表，视业务而定。
- 支付渠道配置，视业务而定。

这些数据通常不是测试专用，而是环境启动后应用就需要。

---

## 二、什么是测试数据

测试数据是为了运行测试或演示而准备的数据。

例如：

- 测试用户。
- 测试订单。
- 测试文章。
- 压测数据。
- 本地开发假数据。

这些数据不应该进入生产环境。

---

## 三、什么数据应该放进迁移

迁移主要负责结构变化。

但某些和结构强绑定的固定数据，也可以放在迁移里。

例如创建角色表：

```sql
CREATE TABLE app.roles (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    code text NOT NULL UNIQUE,
    name text NOT NULL
);
```

如果系统必须有这些角色：

```text
admin
member
guest
```

可以在同一个迁移或单独迁移里插入：

```sql
INSERT INTO app.roles (code, name)
VALUES
    ('admin', '管理员'),
    ('member', '成员'),
    ('guest', '访客');
```

为了可重复执行，可以使用 UPSERT：

```sql
INSERT INTO app.roles (code, name)
VALUES
    ('admin', '管理员'),
    ('member', '成员'),
    ('guest', '访客')
ON CONFLICT (code)
DO UPDATE SET name = EXCLUDED.name;
```

---

## 四、什么数据不应该放进迁移

不应该放进迁移：

- 本地测试用户。
- 演示文章。
- 压测订单。
- 临时调试数据。
- 只为某个测试用例服务的数据。

这些应该放在：

```text
seeds/
tests/
fixtures/
```

而不是 migrations。

---

## 五、推荐目录结构

```text
go-pg-demo/
  migrations/
    00001_create_users.sql
  seeds/
    local_seed.sql
    dev_seed.sql
  testdata/
    users_fixture.sql
```

含义：

- `migrations/`：结构变化和必要基础数据。
- `seeds/`：本地或开发环境初始化数据。
- `testdata/`：测试用例使用的数据。

---

## 六、本地 seed 示例

`seeds/local_seed.sql`：

```sql
INSERT INTO app.users (email, username, display_name, bio)
VALUES
    ('alice@example.com', 'alice', 'Alice', '本地测试用户'),
    ('bob@example.com', 'bob', 'Bob', '本地测试用户')
ON CONFLICT (email)
DO UPDATE SET
    username = EXCLUDED.username,
    display_name = EXCLUDED.display_name,
    bio = EXCLUDED.bio,
    updated_at = now();
```

本地执行：

```powershell
psql $env:DATABASE_URL -f seeds/local_seed.sql
```

如果使用 Windows PowerShell，`psql` 命令是否可用取决于 PostgreSQL 客户端是否已加入 PATH。

---

## 七、seed 要可重复执行

本地 seed 经常会执行多次。

所以尽量写成幂等：

```sql
INSERT ...
ON CONFLICT ...
DO UPDATE ...
```

不要写成只能执行一次的脚本：

```sql
INSERT INTO app.users (email, username)
VALUES ('alice@example.com', 'alice');
```

第二次执行就会因为邮箱重复失败。

---

## 八、测试数据应该由测试控制

Go 测试中通常希望：

```text
每个测试准备自己需要的数据。
测试结束后清理。
测试之间互不影响。
```

不要让测试依赖你本地已经手动插入的数据。

测试数据准备可以用：

- SQL fixture。
- Go helper 函数。
- 每个测试事务回滚。
- 每次测试创建临时数据库。

例如：

```sql
INSERT INTO app.users (email, username)
VALUES ('test@example.com', 'test_user');
```

这类数据只服务测试，不应该进生产迁移。

---

## 九、生产 seed 要谨慎

生产环境需要的基础数据可以通过迁移或专门 seed 流程执行。

但要注意：

- 不要插入测试账号。
- 不要插入弱密码默认管理员。
- 不要覆盖生产已有配置。
- 需要审查 seed 的影响。

如果必须创建默认管理员，通常应该：

- 通过安全流程生成密码。
- 首次登录强制修改。
- 或通过运维后台创建。

---

## 十、种子数据和字典表

字典表很适合用 seed 初始化。

例如角色：

```sql
CREATE TABLE app.roles (
    code text PRIMARY KEY,
    name text NOT NULL,
    sort_order integer NOT NULL
);
```

seed：

```sql
INSERT INTO app.roles (code, name, sort_order)
VALUES
    ('admin', '管理员', 10),
    ('member', '成员', 20)
ON CONFLICT (code)
DO UPDATE SET
    name = EXCLUDED.name,
    sort_order = EXCLUDED.sort_order;
```

如果这些角色是系统运行必需，放进迁移也可以。

如果角色由后台配置，可能只需要表结构迁移，不需要固定 seed。

---

## 十一、常见误区

### 1. 所有初始数据都应该放进 migrations？

不是。

迁移主要管理结构。测试数据和本地假数据不要放进迁移。

### 2. seed 可以随便写 INSERT？

不建议。

seed 最好可重复执行，避免第二次执行失败。

### 3. 测试数据可以依赖本地数据库已有数据吗？

不应该。

测试应该自己准备数据，确保可重复。

### 4. 生产环境可以用本地 seed 吗？

不应该。

生产 seed 必须经过审查，不能混入测试数据。

---

## 十二、本节达标标准

学完本节后，你应该能够做到：

- 区分种子数据和测试数据。
- 知道 migrations 主要管理结构变化。
- 知道哪些基础数据可以随迁移或 seed 初始化。
- 编写可重复执行的 seed SQL。
- 使用 `ON CONFLICT` 保证 seed 幂等。
- 知道测试数据应该由测试用例控制。
- 知道生产 seed 要谨慎审查。
