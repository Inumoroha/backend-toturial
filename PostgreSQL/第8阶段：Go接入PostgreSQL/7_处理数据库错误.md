# 7. 处理数据库错误

本节目标：掌握如何识别 PostgreSQL 错误，尤其是唯一约束冲突、外键约束失败、非空约束失败和检查约束失败，并把数据库错误转换成清晰的业务错误。

数据库错误不能全部原样返回给用户。

例如 PostgreSQL 返回：

```text
duplicate key value violates unique constraint "users_email_unique"
```

用户更应该看到：

```text
邮箱已被注册
```

---

## 一、定义业务错误

```go
package user

import "errors"

var (
	ErrUserNotFound      = errors.New("user not found")
	ErrEmailAlreadyUsed  = errors.New("email already used")
	ErrInvalidUserInput  = errors.New("invalid user input")
	ErrReferencedMissing = errors.New("referenced data does not exist")
)
```

业务层和 handler 层可以根据这些错误返回不同 HTTP 状态码：

- `ErrUserNotFound` -> 404
- `ErrEmailAlreadyUsed` -> 409
- `ErrInvalidUserInput` -> 400
- 其他未知错误 -> 500

---

## 二、pgconn.PgError

pgx 中 PostgreSQL 错误通常可以转换为：

```go
*pgconn.PgError
```

示例：

```go
var pgErr *pgconn.PgError
if errors.As(err, &pgErr) {
	fmt.Println(pgErr.Code)
	fmt.Println(pgErr.ConstraintName)
	fmt.Println(pgErr.Message)
}
```

常用字段：

- `Code`：SQLSTATE 错误码。
- `ConstraintName`：违反的约束名。
- `Message`：错误信息。
- `Detail`：详细信息。
- `TableName`：相关表名。
- `ColumnName`：相关列名。

---

## 三、常见 SQLSTATE 错误码

常见错误：

| 错误 | SQLSTATE | 场景 |
| --- | --- | --- |
| unique_violation | `23505` | 唯一约束冲突 |
| foreign_key_violation | `23503` | 外键约束失败 |
| not_null_violation | `23502` | 非空约束失败 |
| check_violation | `23514` | 检查约束失败 |
| serialization_failure | `40001` | 序列化失败 |
| deadlock_detected | `40P01` | 死锁 |

这些错误码比错误文本更稳定。

不要通过字符串包含判断：

```go
strings.Contains(err.Error(), "duplicate key")
```

应该判断 SQLSTATE 和约束名。

## 四、封装错误转换函数

```go
package user

import (
	"errors"

	"github.com/jackc/pgx/v5/pgconn"
)

const (
	pgErrUniqueViolation     = "23505"
	pgErrForeignKeyViolation = "23503"
	pgErrNotNullViolation    = "23502"
	pgErrCheckViolation      = "23514"
)

func mapDBError(err error) error {
	if err == nil {
		return nil
	}

	var pgErr *pgconn.PgError
	if !errors.As(err, &pgErr) {
		return err
	}

	switch pgErr.Code {
	case pgErrUniqueViolation:
		if pgErr.ConstraintName == "users_email_unique" {
			return ErrEmailAlreadyUsed
		}
		return ErrInvalidUserInput
	case pgErrForeignKeyViolation:
		return ErrReferencedMissing
	case pgErrNotNullViolation, pgErrCheckViolation:
		return ErrInvalidUserInput
	default:
		return err
	}
}
```

然后在 Repository 中使用：

```go
if err != nil {
	return nil, mapDBError(err)
}
```

## 五、处理唯一约束冲突

创建用户时邮箱重复：

```go
func (r *Repository) Create(ctx context.Context, p CreateUserParams) (*User, error) {
	var u User

	err := r.pool.QueryRow(ctx, `
		INSERT INTO app.users (email, username, display_name, bio)
		VALUES ($1, $2, $3, $4)
		RETURNING id, email, username, display_name, bio, created_at, updated_at
	`, p.Email, p.Username, p.DisplayName, p.Bio).Scan(
		&u.ID,
		&u.Email,
		&u.Username,
		&u.DisplayName,
		&u.Bio,
		&u.CreatedAt,
		&u.UpdatedAt,
	)
	if err != nil {
		return nil, mapDBError(err)
	}

	return &u, nil
}
```

如果违反：

```sql
CONSTRAINT users_email_unique UNIQUE (email)
```

就返回：

```go
ErrEmailAlreadyUsed
```

## 六、处理外键约束失败

假设有文章表：

```sql
CREATE TABLE app.posts (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    user_id bigint NOT NULL,
    title text NOT NULL,
    CONSTRAINT posts_user_id_fk
        FOREIGN KEY (user_id)
        REFERENCES app.users (id)
);
```

如果插入不存在的 `user_id`：

```sql
INSERT INTO app.posts (user_id, title)
VALUES (999999, 'invalid');
```

会触发：

```text
23503 foreign_key_violation
```

Go 中可以转换成：

```go
ErrReferencedMissing
```

业务层再决定返回：

```text
作者不存在
```

## 七、处理 ErrNoRows

`pgx.ErrNoRows` 不是 PostgreSQL 服务器错误，因此不是 `PgError`。

要单独处理：

```go
if errors.Is(err, pgx.ErrNoRows) {
	return nil, ErrUserNotFound
}
```

常见于：

- 根据 ID 查询。
- `UPDATE ... RETURNING` 没更新到行。
- `DELETE ... RETURNING` 没删除到行。

---

## 八、哪些错误不要暴露给用户

不要直接返回：

- SQL 语句。
- 数据库连接字符串。
- 表名和内部结构。
- 原始数据库错误详情。
- 堆栈信息。

可以记录到日志里，但用户响应应该是业务友好的信息。

例如：

```json
{
  "error": "email already used"
}
```

而不是：

```json
{
  "error": "ERROR: duplicate key value violates unique constraint \"users_email_unique\" (SQLSTATE 23505)"
}
```

## 九、错误处理和日志

Repository 可以返回业务错误。

Service 或 Handler 可以记录日志：

```go
u, err := repo.Create(ctx, params)
if err != nil {
	if errors.Is(err, user.ErrEmailAlreadyUsed) {
		http.Error(w, "email already used", http.StatusConflict)
		return
	}

	log.Printf("create user: %v", err)
	http.Error(w, "internal server error", http.StatusInternalServerError)
	return
}
```

注意不要把所有错误都吞掉，也不要把所有错误都当成 500。

## 十、可重试错误

事务中可能遇到：

- `40001`：序列化失败。
- `40P01`：死锁。

可以识别：

```go
const (
	pgErrSerializationFailure = "40001"
	pgErrDeadlockDetected     = "40P01"
)

func isRetryableDBError(err error) bool {
	var pgErr *pgconn.PgError
	if !errors.As(err, &pgErr) {
		return false
	}

	return pgErr.Code == pgErrSerializationFailure ||
		pgErr.Code == pgErrDeadlockDetected
}
```

遇到可重试错误时，应该重试整个事务，而不是只重试最后一条 SQL。

## 十一、常见误区

### 1. 通过 err.Error() 字符串判断错误可以吗？

不推荐。

应该使用 `errors.As` 转成 `*pgconn.PgError`，再判断 `Code` 和 `ConstraintName`。

### 2. 所有数据库错误都返回 500 吗？

不是。

唯一冲突、外键失败、校验失败通常可以转换成 409 或 400。

### 3. PgError 能处理 ErrNoRows 吗？

不能。

`pgx.ErrNoRows` 要单独判断。

### 4. 数据库约束错误可以直接返回给用户吗？

不建议。

应该转换成业务语言。

---

## 十二、本节达标标准

学完本节后，你应该能够做到：

- 使用 `errors.As` 提取 `*pgconn.PgError`。
- 判断 SQLSTATE 错误码。
- 识别 `23505` 唯一约束冲突。
- 识别 `23503` 外键约束失败。
- 识别 `23502` 非空约束失败。
- 识别 `23514` 检查约束失败。
- 单独处理 `pgx.ErrNoRows`。
- 把数据库错误转换成业务错误。
- 知道死锁和序列化失败通常可以重试整个事务。
