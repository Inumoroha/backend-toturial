# 0. Go 接入 PostgreSQL 总览

本阶段目标：能在 Go 服务中稳定、清晰地操作 PostgreSQL。

前面几个阶段已经学习了 PostgreSQL 自身的能力：

```text
SQL 基础。
表设计。
查询进阶。
事务和并发。
索引和性能分析。
PostgreSQL 特色类型。
```

第八阶段要把这些能力接入 Go 后端服务。

也就是说，不只是能在 `psql` 或 DBeaver 里写 SQL，而是能在真实 Go 项目中完成：

```text
连接数据库。
使用连接池。
执行参数化查询。
查询单行和多行数据。
插入、更新、删除。
使用事务。
识别数据库错误。
封装清晰的 Repository 或 Store。
```

---

## 一、本阶段为什么重要

很多后端项目的数据库问题不是 SQL 不会写，而是代码层面处理不稳：

- 每次请求都新建连接，导致数据库连接爆炸。
- 拼接 SQL 字符串，带来 SQL 注入风险。
- 查询结果没有正确关闭，连接迟迟不释放。
- 事务出错后没有回滚。
- 唯一约束冲突直接返回 500，用户看不懂。
- 数据库超时没有使用 `context.Context` 控制。
- Repository 里到处散落 SQL，后期难维护。

这一阶段就是把 PostgreSQL 能力变成 Go 项目里的稳定代码。

---

## 二、本阶段核心内容

本阶段建议拆成以下主题学习：

1. `pgx` 与 `pgxpool` 入门。
2. 连接池配置、生命周期和 `context.Context`。
3. 参数化查询，避免 SQL 注入。
4. 查询单行和多行数据。
5. 插入、更新、删除。
6. 在 Go 中使用事务。
7. 处理数据库错误，例如唯一约束冲突、外键约束失败。
8. 用户模块综合实践。
9. `database/sql` 与 `pgx` 的区别。

---

## 三、推荐学习顺序

建议按下面顺序学习：

```text
pgx / pgxpool 基础
-> 连接池和 context
-> 参数化查询
-> 查询单行和多行
-> INSERT / UPDATE / DELETE
-> 事务
-> 数据库错误处理
-> 用户模块实践
-> database/sql 与 pgx 对比
```

原因是：

- 先能连上数据库。
- 再理解连接池为什么是 Web 服务必需品。
- 然后掌握安全执行 SQL 的方式。
- 接着完成基础 CRUD。
- 最后处理事务、错误和项目结构。

## 四、为什么选择 pgx

`pgx` 是 Go 中非常常用的 PostgreSQL 驱动和工具包。

如果新项目明确只使用 PostgreSQL，可以优先学习：

```go
github.com/jackc/pgx/v5
github.com/jackc/pgx/v5/pgxpool
```

其中：

- `pgx`：PostgreSQL 原生驱动和低层接口。
- `pgxpool`：基于 pgx 的连接池，适合 Web 服务。
- `pgx/stdlib`：让 pgx 作为 `database/sql` 驱动使用。

学习阶段可以先把重点放在：

```go
pgxpool.Pool
QueryRow
Query
Exec
Begin
pgconn.PgError
```

这些就足够完成大多数后端 CRUD。

## 五、本阶段统一练习表

本阶段用一个用户模块作为练习：

```sql
CREATE SCHEMA IF NOT EXISTS app;

DROP TABLE IF EXISTS app.users;

CREATE TABLE app.users (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    email text NOT NULL,
    username text NOT NULL,
    display_name text,
    bio text,
    created_at timestamptz NOT NULL DEFAULT now(),
    updated_at timestamptz NOT NULL DEFAULT now(),
    deleted_at timestamptz,
    CONSTRAINT users_email_unique UNIQUE (email),
    CONSTRAINT users_email_not_empty_check CHECK (length(trim(email)) > 0),
    CONSTRAINT users_username_not_empty_check CHECK (length(trim(username)) > 0)
);

CREATE INDEX users_deleted_at_idx
ON app.users (deleted_at);
```

这个表覆盖：

- 创建用户。
- 根据 ID 查询用户。
- 根据邮箱查询用户。
- 更新用户信息。
- 软删除用户。
- 处理邮箱唯一约束冲突。

## 六、推荐项目结构

学习阶段可以从简单结构开始：

```text
go-pg-demo/
  go.mod
  main.go
  internal/
    db/
      db.go
    user/
      model.go
      repository.go
      errors.go
```

如果只是练习，也可以先放在一个 `main.go` 里跑通。

但真实项目里建议尽早形成边界：

- `db`：负责连接池初始化和关闭。
- `user`：负责用户相关 SQL 和业务错误。
- `main`：负责启动程序、注入依赖。

## 八、本阶段最重要的实践原则

### 1. Web 服务使用连接池

不要每次请求都新建数据库连接。

应该在程序启动时创建一个 `*pgxpool.Pool`，整个服务复用它，程序退出时关闭。

### 2. 所有外部输入都使用参数化查询

不要拼接：

```go
sql := "SELECT * FROM users WHERE email = '" + email + "'"
```

应该写：

```go
row := pool.QueryRow(ctx, `
    SELECT id, email
    FROM app.users
    WHERE email = $1
`, email)
```

### 3. 查询多行必须关闭 rows

```go
rows, err := pool.Query(ctx, sql)
if err != nil {
    return err
}
defer rows.Close()
```

### 4. 事务必须提交或回滚

开启事务后：

```text
成功：Commit
失败：Rollback
```

常见写法是：

```go
tx, err := pool.Begin(ctx)
if err != nil {
    return err
}
defer tx.Rollback(ctx)

// 执行 SQL

return tx.Commit(ctx)
```

`Commit` 成功后，前面的 `defer Rollback` 通常会返回事务已关闭错误，可以忽略。

### 5. 数据库错误要转换成业务错误

例如唯一约束冲突：

```text
users_email_unique
```

不应该直接把 PostgreSQL 原始错误返回给用户，而应该转换成：

```text
邮箱已被注册
```

## 九、本阶段达标标准

学完第八阶段后，你应该能够做到：

- 使用 `pgxpool.New` 创建连接池。
- 使用 `Ping` 检查数据库连接。
- 使用参数化查询避免 SQL 注入。
- 使用 `QueryRow` 查询单行。
- 使用 `Query` 查询多行，并正确关闭 `rows`。
- 使用 `Exec` 执行更新和删除。
- 使用 `RETURNING` 获取插入后的数据。
- 在 Go 中正确开启、提交、回滚事务。
- 识别唯一约束冲突和外键约束失败。
- 写出一个可维护的用户 Repository。
- 理解 `database/sql` 与 `pgx` 的基本区别。
