# 9. 事务、并发与一致性实践

本节目标：把第五阶段学过的事务、并发和一致性放回短链接项目中，理解哪些地方需要事务，哪些地方不需要过度使用事务。

短链接服务第一版不复杂，但仍然会遇到并发问题：

```text
短码重复。
访问次数并发更新。
读取与更新是否一致。
热门短链行锁竞争。
```

---

## 一、创建短链接是否需要显式事务

第一版创建短链接只做一件事：

```sql
INSERT INTO app.short_links ...
```

单条 SQL 在 PostgreSQL 中天然是一个事务。

也就是说，即使你没有显式写：

```sql
BEGIN;
COMMIT;
```

PostgreSQL 也会保证这条 `INSERT` 要么成功，要么失败。

所以第一版创建短链接不需要显式事务。

---

## 二、跳转计数是否需要显式事务

第一版跳转使用：

```sql
UPDATE app.short_links
SET visit_count = visit_count + 1,
    updated_at = now()
WHERE code = $1
RETURNING original_url;
```

这也是单条 SQL。

它同时完成：

```text
增加访问次数。
返回原始链接。
```

因此也不需要显式事务。

这条 SQL 本身就是原子的。

---

## 三、什么时候需要事务

如果后续增加访问日志表，就可能需要事务。

例如跳转时要同时做两件事：

```text
short_links.visit_count 加 1。
short_link_visits 插入一条访问日志。
```

这时就要考虑：

```text
计数增加成功，但日志插入失败怎么办？
日志插入成功，但计数增加失败怎么办？
```

如果业务要求它们必须同时成功，就需要事务。

---

## 四、增加访问日志表

后续可以增加：

```sql
CREATE TABLE app.short_link_visits (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    short_link_id uuid NOT NULL REFERENCES app.short_links(id),
    visited_at timestamptz NOT NULL DEFAULT now(),
    ip text,
    user_agent text
);

CREATE INDEX short_link_visits_short_link_id_visited_at_idx
ON app.short_link_visits (short_link_id, visited_at DESC);
```

这样每次跳转都可以记录一条访问明细。

## 五、事务版 Resolve

如果跳转需要同时更新计数和插入日志，可以这样写：

```go
type VisitInput struct {
    Code      string
    IP        string
    UserAgent string
}

func (r *Repository) ResolveWithVisitLog(ctx context.Context, input VisitInput) (string, error) {
    tx, err := r.pool.Begin(ctx)
    if err != nil {
        return "", err
    }
    defer tx.Rollback(ctx)

    const updateQuery = `
        UPDATE app.short_links
        SET visit_count = visit_count + 1,
            updated_at = now()
        WHERE code = $1
        RETURNING id, original_url
    `

    var linkID uuid.UUID
    var originalURL string
    err = tx.QueryRow(ctx, updateQuery, input.Code).Scan(&linkID, &originalURL)
    if err != nil {
        if errors.Is(err, pgx.ErrNoRows) {
            return "", ErrNotFound
        }
        return "", err
    }

    const insertVisitQuery = `
        INSERT INTO app.short_link_visits (short_link_id, ip, user_agent)
        VALUES ($1, $2, $3)
    `

    if _, err := tx.Exec(ctx, insertVisitQuery, linkID, input.IP, input.UserAgent); err != nil {
        return "", err
    }

    if err := tx.Commit(ctx); err != nil {
        return "", err
    }

    return originalURL, nil
}
```

注意：

```go
defer tx.Rollback(ctx)
```

即使后面 `Commit` 成功，再执行 `Rollback` 通常也只会返回事务已结束错误。这里忽略即可。

这种写法保证：

```text
计数更新和访问日志插入要么都成功，要么都失败。
```

## 六、事务边界要小

事务不是越大越好。

错误思路：

```text
收到 HTTP 请求。
开启事务。
做 URL 校验。
调用外部 API。
写数据库。
生成响应。
提交事务。
```

这会让事务持有时间过长。

更好的顺序：

```text
先做纯内存校验。
需要数据库一致性时再开启事务。
事务里只放必须一起成功的 SQL。
尽快提交或回滚。
```

长事务会带来：

- 锁持有时间变长。
- 阻塞其他请求。
- 影响 autovacuum 清理旧版本。
- 增加数据库膨胀风险。

## 七、短码冲突和事务的关系

短码冲突不需要用显式事务解决。

因为唯一约束已经保证：

```sql
UNIQUE (code)
```

并发插入同一个 code 时：

```text
一个成功。
另一个失败，返回 23505。
```

Go 代码捕获 `23505` 后重试即可。

不要为了避免冲突而在应用层加大事务范围。

---

## 八、并发更新 visit_count 会不会丢失

使用这个 SQL 不会丢失更新：

```sql
UPDATE app.short_links
SET visit_count = visit_count + 1
WHERE code = $1;
```

因为 PostgreSQL 更新同一行时会加行级锁。

多个请求更新同一行，会排队执行。

假设初始值是 `10`，两个请求同时更新：

```text
请求 A 更新为 11。
请求 B 等待 A 完成，再基于最新值更新为 12。
```

最终结果是 `12`。

---

## 九、热门短链的锁竞争

虽然不会丢失更新，但热门短链可能出现行锁竞争。

如果某个短链每秒很多访问，每次访问都更新同一行：

```sql
UPDATE app.short_links
SET visit_count = visit_count + 1
WHERE code = $1;
```

这行数据会成为热点。

现象可能是：

- 请求延迟升高。
- 数据库 CPU 和锁等待增加。
- 连接池被占满。

第一版不用解决这个问题，但要知道它存在。

后续优化方向：

- Redis 计数，定期批量回写。
- 访问日志异步写入。
- 按小时或按天聚合到统计表。
- 对访问计数做延迟一致性。

---

## 十、事务隔离级别怎么选

短链接第一版通常使用 PostgreSQL 默认隔离级别：

```text
READ COMMITTED
```

已经足够。

原因：

- 创建短链依赖唯一约束。
- 跳转计数使用单条原子 UPDATE。
- 查询详情不需要跨多次读取保持完全一致快照。

不要一上来就改成：

```sql
SERIALIZABLE
```

更高隔离级别会增加冲突和重试复杂度。

只有当业务真的需要严格串行化语义时再考虑。

## 十一、如何观察锁等待

如果怀疑锁等待，可以查询：

```sql
SELECT
    pid,
    state,
    wait_event_type,
    wait_event,
    query
FROM pg_stat_activity
WHERE datname = current_database();
```

如果看到：

```text
wait_event_type = Lock
```

说明有会话正在等锁。

也可以查阻塞关系，但第一版先会看 `pg_stat_activity` 就够。
