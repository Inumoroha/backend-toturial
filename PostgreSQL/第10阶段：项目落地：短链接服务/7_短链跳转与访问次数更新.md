# 7. 短链跳转与访问次数更新

本节目标：实现 `GET /{code}`，根据短码查询原始链接，并增加访问次数。

短链接服务最核心、最频繁的接口就是跳转接口。

它的流程是：

```text
读取 code
-> 查询原始链接
-> 增加访问次数
-> 返回 HTTP 302
```

---

## 一、接口行为

请求：

```http
GET /aB39xQ
```

成功响应：

```http
HTTP/1.1 302 Found
Location: https://example.com/articles/postgresql
```

如果短码不存在：

```http
HTTP/1.1 404 Not Found
```

---

## 二、两种实现方式

跳转接口有两种常见写法。

### 写法一：先查再更新

```text
SELECT original_url FROM short_links WHERE code = $1
UPDATE short_links SET visit_count = visit_count + 1 WHERE code = $1
```

优点：

- 逻辑直观。
- 查询和更新可以分别处理。

缺点：

- 两次 SQL 往返。
- 查询成功后，更新失败时需要考虑怎么处理。

### 写法二：UPDATE RETURNING

```sql
UPDATE app.short_links
SET visit_count = visit_count + 1,
    updated_at = now()
WHERE code = $1
RETURNING original_url;
```

优点：

- 一条 SQL 完成查询和计数更新。
- `visit_count = visit_count + 1` 是原子更新。
- 如果没有匹配行，直接返回没有记录。

第一版推荐使用第二种。

## 三、repository 实现 Resolve

在 `repository.go` 增加：

```go
import (
    "context"
    "errors"

    "github.com/jackc/pgx/v5"
)
```

方法：

```go
func (r *Repository) Resolve(ctx context.Context, code string) (string, error) {
    const query = `
        UPDATE app.short_links
        SET visit_count = visit_count + 1,
            updated_at = now()
        WHERE code = $1
        RETURNING original_url
    `

    var originalURL string
    err := r.pool.QueryRow(ctx, query, code).Scan(&originalURL)
    if err != nil {
        if errors.Is(err, pgx.ErrNoRows) {
            return "", ErrNotFound
        }
        return "", err
    }

    return originalURL, nil
}
```

这条 SQL 同时完成：

```text
找到 code。
visit_count 加 1。
返回 original_url。
```

## 四、为什么 `visit_count = visit_count + 1` 是安全的

不要在 Go 里这样做：

```text
SELECT visit_count
Go 里 +1
UPDATE visit_count = 新值
```

并发下可能丢失更新。

例如两个请求同时读到 `visit_count = 10`：

```text
请求 A 计算 11。
请求 B 计算 11。
请求 A 写入 11。
请求 B 写入 11。
```

最终应该是 12，却变成 11。

正确做法：

```sql
UPDATE app.short_links
SET visit_count = visit_count + 1
WHERE code = $1;
```

这个加法在数据库内部执行。

PostgreSQL 会对被更新的行加行级锁，确保并发更新排队执行，不会简单覆盖。

## 五、service 实现 Resolve

在 `service.go` 增加：

```go
func (s *Service) Resolve(ctx context.Context, code string) (string, error) {
    code = strings.TrimSpace(code)
    if code == "" {
        return "", ErrNotFound
    }

    return s.repo.Resolve(ctx, code)
}
```

第一版短码只允许字母数字，可以加更严格的校验：

```go
func isValidCode(code string) bool {
    if code == "" || len(code) > 32 {
        return false
    }

    for _, ch := range code {
        switch {
        case ch >= '0' && ch <= '9':
        case ch >= 'a' && ch <= 'z':
        case ch >= 'A' && ch <= 'Z':
        default:
            return false
        }
    }

    return true
}
```

然后：

```go
if !isValidCode(code) {
    return "", ErrNotFound
}
```

对非法短码返回 `404`，可以避免泄露太多规则。

## 六、handler 实现 redirect

在 `handler.go` 注册：

```go
func (h *Handler) RegisterRoutes(mux *http.ServeMux) {
    mux.HandleFunc("POST /links", h.createLink)
    mux.HandleFunc("GET /{code}", h.redirect)
}
```

实现：

```go
func (h *Handler) redirect(w http.ResponseWriter, r *http.Request) {
    code := r.PathValue("code")

    ctx, cancel := context.WithTimeout(r.Context(), 3*time.Second)
    defer cancel()

    originalURL, err := h.service.Resolve(ctx, code)
    if err != nil {
        switch {
        case errors.Is(err, ErrNotFound):
            writeJSON(w, http.StatusNotFound, map[string]string{
                "error": "link not found",
            })
        default:
            writeJSON(w, http.StatusInternalServerError, map[string]string{
                "error": "internal server error",
            })
        }
        return
    }

    http.Redirect(w, r, originalURL, http.StatusFound)
}
```

`http.StatusFound` 就是 302。

## 七、路由冲突问题

如果同时有：

```text
GET /links/{code}
GET /{code}
```

一定要确认路由匹配符合预期。

在 Go 新版 `ServeMux` 中，更具体的路径通常优先。

推荐注册：

```go
mux.HandleFunc("GET /links/{code}", h.getLink)
mux.HandleFunc("GET /links", h.listLinks)
mux.HandleFunc("GET /{code}", h.redirect)
```

这样 `/links/aB39xQ` 会进入详情接口，而不是被当作短码。

如果使用第三方框架，也要注意固定路径和通配路径的顺序。

## 八、手动测试跳转

先创建短链接：

```powershell
$resp = Invoke-RestMethod `
  -Method Post `
  -Uri "http://localhost:8080/links" `
  -ContentType "application/json" `
  -Body '{"original_url":"https://example.com/articles/postgresql"}'

$resp
```

访问短链接：

```powershell
Invoke-WebRequest -Uri $resp.short_url -MaximumRedirection 0
```

PowerShell 可能会把 302 当作异常输出。

也可以直接用浏览器访问：

```text
http://localhost:8080/aB39xQ
```

然后查数据库：

```sql
SELECT code, visit_count, updated_at
FROM app.short_links
WHERE code = 'aB39xQ';
```

多访问几次，`visit_count` 应该增加。

## 九、访问计数的业务取舍

第一版每次跳转都直接更新 `visit_count`。

优点：

- 简单。
- 查询详情时能立刻看到最新计数。

缺点：

- 热门短链会频繁更新同一行。
- 高并发下这行可能成为热点。

后续可以优化为：

- 写访问日志，异步聚合。
- Redis 计数，定期回写 PostgreSQL。
- 按天统计到单独表。

但第一版不要过早复杂化。

先用数据库原子更新理解并发和行锁。

## 十、常见错误

### 1. 返回 200 而不是 302

跳转接口应该使用：

```go
http.Redirect(w, r, originalURL, http.StatusFound)
```

不要只返回 JSON。

### 2. 访问次数没有增加

检查 SQL 是否执行的是：

```sql
UPDATE ... RETURNING original_url
```

而不是只做了 `SELECT`。

### 3. code 为空也查询数据库

空短码应该在 service 层提前处理。

### 4. 用 Go 读出计数再加一

这容易产生并发丢失更新。

用数据库原子更新。

## 十一、本节达标标准

学完本节后，你应该能够做到：

- 实现 `GET /{code}` 跳转接口。
- 使用 `UPDATE ... RETURNING` 同时更新访问次数并返回原始链接。
- 解释为什么数据库内 `visit_count = visit_count + 1` 可以避免丢失更新。
- 正确处理短码不存在的情况。
- 返回 HTTP 302 跳转。
- 知道热门短链计数在高并发下可能成为热点。
