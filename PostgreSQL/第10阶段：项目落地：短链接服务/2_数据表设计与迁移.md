# 2. 数据表设计与迁移

本节目标：为短链接服务设计 PostgreSQL 表结构，并用迁移文件管理它。

短链接项目的第一张核心表是：

```text
app.short_links
```

它负责保存：

```text
短码 code
原始链接 original_url
访问次数 visit_count
创建时间 created_at
更新时间 updated_at
```

---

## 一、第一版表结构

推荐结构：

```sql
CREATE SCHEMA IF NOT EXISTS app;

CREATE TABLE app.short_links (
    id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    code text NOT NULL,
    original_url text NOT NULL,
    visit_count bigint NOT NULL DEFAULT 0,
    created_at timestamptz NOT NULL DEFAULT now(),
    updated_at timestamptz NOT NULL DEFAULT now(),
    CONSTRAINT short_links_code_unique UNIQUE (code),
    CONSTRAINT short_links_code_not_empty_check CHECK (length(trim(code)) > 0),
    CONSTRAINT short_links_original_url_not_empty_check CHECK (length(trim(original_url)) > 0),
    CONSTRAINT short_links_visit_count_non_negative_check CHECK (visit_count >= 0)
);
```

这里使用 `uuid` 作为主键。

原因是：

- 不暴露数据库自增序列。
- 适合作为对外资源 ID。
- 后续如果有多服务写入，比纯自增 ID 更灵活。

短码 `code` 单独加唯一约束：

```sql
CONSTRAINT short_links_code_unique UNIQUE (code)
```

因为业务上真正用来跳转的是 `code`。

---

## 二、为什么 `code` 不直接作为主键

可以把 `code` 作为主键吗？

可以，但不建议第一版这么做。

推荐：

```text
id   作为数据库主键和资源唯一标识。
code 作为业务唯一键。
```

这样做的好处是：

- 后续如果支持修改短码，主键不用变。
- 其他表引用短链接时可以引用稳定的 `id`。
- `code` 的业务规则可以单独调整。

例如将来增加访问日志表：

```sql
CREATE TABLE app.short_link_visits (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    short_link_id uuid NOT NULL REFERENCES app.short_links(id),
    visited_at timestamptz NOT NULL DEFAULT now()
);
```

如果用 `short_link_id` 引用 UUID，会比引用可变的 `code` 更稳。

## 三、字段逐个解释

### 1. `id uuid`

短链接记录的主键。

默认值：

```sql
DEFAULT gen_random_uuid()
```

在较新的 PostgreSQL 中，`gen_random_uuid()` 可以直接使用。如果你的环境较老，可能需要：

```sql
CREATE EXTENSION IF NOT EXISTS pgcrypto;
```

为了兼容性，迁移文件里可以加上这句。

### 2. `code text`

短码，例如：

```text
aB39xQ
```

它必须：

- 非空。
- 唯一。
- 适合放在 URL 路径里。

建议只使用：

```text
0-9
a-z
A-Z
```

### 3. `original_url text`

原始长链接。

PostgreSQL 没有必要强行使用 `varchar(255)`。

很多 URL 可能超过 255 字符，直接用 `text` 更自然。

### 4. `visit_count bigint`

访问次数。

使用 `bigint` 是为了避免访问量大时 `integer` 不够。

默认：

```sql
DEFAULT 0
```

约束：

```sql
CHECK (visit_count >= 0)
```

### 5. `created_at` 和 `updated_at`

使用 `timestamptz`，不要使用没有时区语义的 `timestamp`。

```sql
created_at timestamptz NOT NULL DEFAULT now()
```

## 四、索引设计

第一版最重要的查询是：

```sql
SELECT original_url
FROM app.short_links
WHERE code = $1;
```

由于 `code` 有唯一约束，PostgreSQL 会自动创建唯一 B-Tree 索引。

所以不需要再手动写：

```sql
CREATE INDEX short_links_code_idx ON app.short_links (code);
```

否则会重复。

列表查询通常是：

```sql
SELECT id, code, original_url, visit_count, created_at, updated_at
FROM app.short_links
ORDER BY created_at DESC
LIMIT $1 OFFSET $2;
```

可以增加：

```sql
CREATE INDEX short_links_created_at_desc_idx
ON app.short_links (created_at DESC);
```

如果后续经常按创建时间倒序分页，这个索引有价值。

## 五、goose 迁移文件

创建迁移：

```powershell
goose -dir migrations -s create create_short_links sql
```

生成：

```text
migrations/00001_create_short_links.sql
```

内容：

```sql
-- +goose Up
-- +goose StatementBegin
CREATE SCHEMA IF NOT EXISTS app;

CREATE EXTENSION IF NOT EXISTS pgcrypto;

CREATE TABLE app.short_links (
    id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    code text NOT NULL,
    original_url text NOT NULL,
    visit_count bigint NOT NULL DEFAULT 0,
    created_at timestamptz NOT NULL DEFAULT now(),
    updated_at timestamptz NOT NULL DEFAULT now(),
    CONSTRAINT short_links_code_unique UNIQUE (code),
    CONSTRAINT short_links_code_not_empty_check CHECK (length(trim(code)) > 0),
    CONSTRAINT short_links_original_url_not_empty_check CHECK (length(trim(original_url)) > 0),
    CONSTRAINT short_links_visit_count_non_negative_check CHECK (visit_count >= 0)
);

CREATE INDEX short_links_created_at_desc_idx
ON app.short_links (created_at DESC);
-- +goose StatementEnd

-- +goose Down
-- +goose StatementBegin
DROP TABLE app.short_links;
-- +goose StatementEnd
```

执行：

```powershell
$env:DATABASE_URL="postgres://postgres:postgres@localhost:5432/learn_pg?sslmode=disable"
goose -dir migrations postgres $env:DATABASE_URL up
```

查看状态：

```powershell
goose -dir migrations postgres $env:DATABASE_URL status
```

## 六、golang-migrate 迁移文件

如果使用 golang-migrate：

```powershell
migrate create -ext sql -dir migrations -seq create_short_links
```

生成：

```text
000001_create_short_links.up.sql
000001_create_short_links.down.sql
```

`000001_create_short_links.up.sql`：

```sql
CREATE SCHEMA IF NOT EXISTS app;

CREATE EXTENSION IF NOT EXISTS pgcrypto;

CREATE TABLE app.short_links (
    id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    code text NOT NULL,
    original_url text NOT NULL,
    visit_count bigint NOT NULL DEFAULT 0,
    created_at timestamptz NOT NULL DEFAULT now(),
    updated_at timestamptz NOT NULL DEFAULT now(),
    CONSTRAINT short_links_code_unique UNIQUE (code),
    CONSTRAINT short_links_code_not_empty_check CHECK (length(trim(code)) > 0),
    CONSTRAINT short_links_original_url_not_empty_check CHECK (length(trim(original_url)) > 0),
    CONSTRAINT short_links_visit_count_non_negative_check CHECK (visit_count >= 0)
);

CREATE INDEX short_links_created_at_desc_idx
ON app.short_links (created_at DESC);
```

`000001_create_short_links.down.sql`：

```sql
DROP TABLE app.short_links;
```

执行：

```powershell
migrate -path migrations -database $env:DATABASE_URL up
```

## 七、插入一条测试数据

执行迁移后，可以手动插入一条数据：

```sql
INSERT INTO app.short_links (code, original_url)
VALUES ('aB39xQ', 'https://example.com/articles/postgresql')
RETURNING id, code, original_url, visit_count, created_at;
```

查询：

```sql
SELECT id, code, original_url, visit_count
FROM app.short_links
WHERE code = 'aB39xQ';
```

访问次数加一：

```sql
UPDATE app.short_links
SET visit_count = visit_count + 1,
    updated_at = now()
WHERE code = 'aB39xQ'
RETURNING code, original_url, visit_count;
```

这里的：

```sql
visit_count = visit_count + 1
```

是数据库内部的原子更新。并发请求同时更新同一行时，PostgreSQL 会通过行锁保证不会把计数简单覆盖掉。

## 八、检查索引

查看表结构：

```sql
\d app.short_links
```

查看索引：

```sql
SELECT indexname, indexdef
FROM pg_indexes
WHERE schemaname = 'app'
  AND tablename = 'short_links';
```

你应该能看到：

- 主键索引。
- `code` 的唯一索引。
- `created_at DESC` 索引。

## 九、常见设计误区

### 1. `original_url varchar(255)` 更专业？

不一定。

URL 可能很长，PostgreSQL 的 `text` 很适合保存这种不定长字符串。

### 2. 只在代码里保证 code 唯一就够了？

不够。

并发场景下必须由数据库唯一约束兜底。

### 3. visit_count 用 integer 就行？

小项目可以，但 `bigint` 成本不高，未来更稳。

### 4. 所有字段都要建索引？

不需要。

索引是为查询服务的，不是越多越好。

第一版重点索引是：

```text
code 唯一索引。
created_at 倒序索引。
```
