# 11. 错误处理、日志与项目收尾

本节目标：把短链接服务从“能跑”整理到“更像一个项目”，补齐错误处理、日志、README 和 Makefile。

一个后端项目不只是 SQL 和接口。

还需要考虑：

```text
错误怎么返回。
日志怎么记录。
命令怎么复现。
新同事怎么启动。
线上排查时能看到什么。
```

---

## 一、错误处理分层

推荐分层：

```text
repository 层：
    识别数据库错误。
    转换成业务错误。

service 层：
    处理业务规则。
    决定是否重试。
    返回稳定的业务错误。

handler 层：
    把业务错误转换成 HTTP 状态码和 JSON。
```

不要让 PostgreSQL 的底层错误直接穿透到用户响应里。

错误响应应该稳定、简洁。

---

## 二、统一错误响应

可以定义：

```go
type ErrorResponse struct {
    Error string `json:"error"`
}
```

写辅助函数：

```go
func writeError(w http.ResponseWriter, status int, message string) {
    writeJSON(w, status, ErrorResponse{Error: message})
}
```

handler 中就可以写：

```go
writeError(w, http.StatusBadRequest, "invalid original_url")
```

不要每次都手写：

```go
map[string]string{"error": "..."}
```

统一结构后，前端和调用方更容易处理。

---

## 三、错误到状态码映射

短链接项目第一版可以这样映射：

| 错误 | HTTP 状态码 | 响应 |
| --- | --- | --- |
| `ErrInvalidURL` | `400` | `invalid original_url` |
| `ErrNotFound` | `404` | `link not found` |
| `ErrRetryExceeded` | `503` | `please retry later` |
| 未知错误 | `500` | `internal server error` |

示例：

```go
func handleServiceError(w http.ResponseWriter, err error) {
    switch {
    case errors.Is(err, ErrInvalidURL):
        writeError(w, http.StatusBadRequest, "invalid original_url")
    case errors.Is(err, ErrNotFound):
        writeError(w, http.StatusNotFound, "link not found")
    case errors.Is(err, ErrRetryExceeded):
        writeError(w, http.StatusServiceUnavailable, "please retry later")
    default:
        writeError(w, http.StatusInternalServerError, "internal server error")
    }
}
```

这样每个 handler 就不用重复写一大段 `switch`。

## 四、日志应该记什么

第一版可以使用 Go 标准库 `log`。

不要一开始就引入复杂日志系统。

建议记录：

- 服务启动地址。
- 数据库连接失败。
- HTTP 服务启动失败。
- 迁移执行失败，如果你的程序启动时自动迁移。
- handler 中的未知错误。

例如：

```go
if err != nil {
    log.Printf("create link: %v", err)
    handleServiceError(w, err)
    return
}
```

但要注意：

```text
日志里可以记录内部错误。
响应里不要暴露内部错误。
```

---

## 五、不要记录敏感信息

短链接项目里可能涉及：

- 用户访问的 URL。
- IP 地址。
- User-Agent。
- 数据库连接字符串。

学习阶段可以直接打印少量信息。

真实项目要注意：

- 不要打印数据库密码。
- 不要把完整 `DATABASE_URL` 打到日志里。
- 如果 URL 可能包含 token，不要随意记录完整 URL。
- IP 和 User-Agent 可能属于隐私数据，要按业务和合规要求处理。

## 六、README 示例

项目根目录建议写 `README.md`：

````md
# Shortener

A small Go + PostgreSQL URL shortener for learning backend development.

## Requirements

- Go 1.22+
- PostgreSQL
- goose

## Environment

```powershell
$env:HTTP_ADDR=":8080"
$env:BASE_URL="http://localhost:8080"
$env:DATABASE_URL="postgres://postgres:postgres@localhost:5432/learn_pg?sslmode=disable"
```

## Migrations

```powershell
goose -dir migrations postgres $env:DATABASE_URL up
```

Rollback one migration:

```powershell
goose -dir migrations postgres $env:DATABASE_URL down
```

Migration status:

```powershell
goose -dir migrations postgres $env:DATABASE_URL status
```

## Run

```powershell
go run ./cmd/api
```

## Create link

```powershell
Invoke-RestMethod `
  -Method Post `
  -Uri "http://localhost:8080/links" `
  -ContentType "application/json" `
  -Body '{"original_url":"https://example.com/articles/postgresql"}'
```

## List links

```powershell
Invoke-RestMethod -Method Get -Uri "http://localhost:8080/links?limit=20&offset=0"
```
````

注意外层使用四个反引号，是为了 README 示例里还能包含普通 Markdown 代码块。

## 七、Makefile 示例

如果你的环境可以使用 `make`，可以写：

```makefile
DATABASE_URL ?= postgres://postgres:postgres@localhost:5432/learn_pg?sslmode=disable

.PHONY: run migrate-up migrate-down migrate-status

run:
	go run ./cmd/api

migrate-up:
	goose -dir migrations postgres "$(DATABASE_URL)" up

migrate-down:
	goose -dir migrations postgres "$(DATABASE_URL)" down

migrate-status:
	goose -dir migrations postgres "$(DATABASE_URL)" status
```

Windows PowerShell 没有内置 `make`。

如果不想安装 make，可以在 README 里写 PowerShell 命令，或者写一个 `scripts` 目录放 `.ps1` 脚本。

## 八、最小验收清单

项目完成后，至少手动验证：

```text
数据库迁移能执行。
服务能启动。
POST /links 能创建短链接。
GET /{code} 能跳转。
跳转后 visit_count 增加。
GET /links/{code} 能查询详情。
GET /links 能分页查询。
非法 URL 返回 400。
不存在的 code 返回 404。
重复 code 能被唯一约束拦住。
```

这些比写很多花哨功能更重要。

## 九、可以补的单元测试

短链接项目里，最适合先写单元测试的是纯逻辑：

- `GenerateCode` 长度是否正确。
- `GenerateCode` 字符是否来自合法字符集。
- URL 校验是否正确。
- `isValidCode` 是否正确。
- 分页参数修正是否正确。

例如 URL 校验测试：

```go
func TestValidateOriginalURL(t *testing.T) {
    tests := []struct {
        name    string
        raw     string
        wantErr bool
    }{
        {name: "https url", raw: "https://example.com", wantErr: false},
        {name: "http url", raw: "http://example.com", wantErr: false},
        {name: "missing scheme", raw: "example.com", wantErr: true},
        {name: "javascript scheme", raw: "javascript:alert(1)", wantErr: true},
        {name: "empty", raw: "", wantErr: true},
    }

    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            err := validateOriginalURL(tt.raw)
            if tt.wantErr && err == nil {
                t.Fatalf("expected error")
            }
            if !tt.wantErr && err != nil {
                t.Fatalf("unexpected error: %v", err)
            }
        })
    }
}
```

数据库集成测试可以后续再做。

## 十、项目可以如何继续扩展

完成第一版后，可以逐步扩展：

### 1. 过期时间

给表增加：

```sql
expires_at timestamptz
```

跳转时判断：

```sql
WHERE code = $1
  AND (expires_at IS NULL OR expires_at > now())
```

过期时返回：

```http
410 Gone
```

### 2. 访问日志

增加：

```text
app.short_link_visits
```

记录 IP、User-Agent、访问时间。

### 3. 每日统计

聚合访问日志：

```sql
SELECT date_trunc('day', visited_at) AS day, count(*)
FROM app.short_link_visits
WHERE short_link_id = $1
GROUP BY day
ORDER BY day DESC;
```

### 4. 用户系统

增加：

```text
users
short_links.user_id
```

这样每个用户只能管理自己的短链接。

### 5. Redis 缓存

热门短链可以缓存：

```text
code -> original_url
```

但缓存会引入一致性问题，要等 PostgreSQL 基础稳定后再做。

## 十一、本节达标标准

学完本节后，你应该能够做到：

- 统一 JSON 错误响应格式。
- 把业务错误映射为合适的 HTTP 状态码。
- 区分日志里可记录的信息和响应里可返回的信息。
- 写出项目 README 的启动和迁移命令。
- 写出常用 Makefile 命令。
- 列出项目最小验收清单。
- 知道短链接服务后续可以如何扩展。
