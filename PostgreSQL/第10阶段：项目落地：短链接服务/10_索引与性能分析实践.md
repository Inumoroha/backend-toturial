# 10. 索引与性能分析实践

本节目标：把第六阶段学过的索引和 `EXPLAIN ANALYZE` 放到短链接项目中，判断核心查询是否高效。

短链接服务最重要的性能目标是：

```text
根据 code 查询必须快。
根据 code 跳转必须快。
列表按 created_at 倒序分页不能太慢。
```

---

## 一、核心查询有哪些

第一版核心 SQL：

### 1. 创建短链接

```sql
INSERT INTO app.short_links (code, original_url)
VALUES ($1, $2)
RETURNING id, code, original_url, visit_count, created_at, updated_at;
```

重点：

```text
code 唯一约束检查。
```

### 2. 跳转

```sql
UPDATE app.short_links
SET visit_count = visit_count + 1,
    updated_at = now()
WHERE code = $1
RETURNING original_url;
```

重点：

```text
WHERE code = $1 是否走索引。
```

### 3. 查询详情

```sql
SELECT id, code, original_url, visit_count, created_at, updated_at
FROM app.short_links
WHERE code = $1;
```

重点：

```text
WHERE code = $1 是否走唯一索引。
```

### 4. 分页列表

```sql
SELECT id, code, original_url, visit_count, created_at, updated_at
FROM app.short_links
ORDER BY created_at DESC
LIMIT $1 OFFSET $2;
```

重点：

```text
ORDER BY created_at DESC 是否能利用索引。
```

---

## 二、准备测试数据

如果表里只有几条数据，PostgreSQL 可能选择顺序扫描。

这不一定是坏事。

因为小表顺序扫描很便宜。

为了更容易观察索引效果，可以插入一些测试数据：

```sql
INSERT INTO app.short_links (code, original_url, created_at, updated_at)
SELECT
    'code' || gs::text,
    'https://example.com/articles/' || gs::text,
    now() - (gs || ' seconds')::interval,
    now() - (gs || ' seconds')::interval
FROM generate_series(1, 100000) AS gs;
```

注意：

```text
这只是本地测试数据。
不要在生产环境随意执行。
```

## 三、分析详情查询

执行：

```sql
EXPLAIN ANALYZE
SELECT id, code, original_url, visit_count, created_at, updated_at
FROM app.short_links
WHERE code = 'code99999';
```

理想情况下，你会看到类似：

```text
Index Scan using short_links_code_unique on short_links
```

或者：

```text
Index Scan using short_links_code_unique
```

这说明 PostgreSQL 使用了 `code` 唯一索引。

如果看到：

```text
Seq Scan
```

可能原因：

- 表太小，顺序扫描更便宜。
- 查询条件写法导致索引不能使用。
- 统计信息不准确。
- 索引不存在。

---

## 四、检查索引是否存在

执行：

```sql
SELECT indexname, indexdef
FROM pg_indexes
WHERE schemaname = 'app'
  AND tablename = 'short_links';
```

应该看到类似：

```text
short_links_pkey
short_links_code_unique
short_links_created_at_desc_idx
```

如果没有 `short_links_code_unique`，说明唯一约束没有创建成功。

---

## 五、分析跳转 UPDATE

执行：

```sql
EXPLAIN ANALYZE
UPDATE app.short_links
SET visit_count = visit_count + 1,
    updated_at = now()
WHERE code = 'code99999'
RETURNING original_url;
```

你可能看到：

```text
Update on short_links
  -> Index Scan using short_links_code_unique ...
```

重点看里面是否有：

```text
Index Scan
```

因为 `UPDATE` 也需要先根据 `WHERE code = ...` 找到目标行。

---

## 六、分析列表查询

执行：

```sql
EXPLAIN ANALYZE
SELECT id, code, original_url, visit_count, created_at, updated_at
FROM app.short_links
ORDER BY created_at DESC
LIMIT 20 OFFSET 0;
```

理想情况下可能看到：

```text
Index Scan using short_links_created_at_desc_idx
```

如果没有 `created_at` 索引，可能会看到：

```text
Seq Scan
Sort
```

含义是：

```text
先扫描表。
再排序。
最后取前 20 条。
```

数据量大时会变慢。

---

## 七、大 OFFSET 的问题

执行：

```sql
EXPLAIN ANALYZE
SELECT id, code, original_url, visit_count, created_at, updated_at
FROM app.short_links
ORDER BY created_at DESC
LIMIT 20 OFFSET 90000;
```

即使有索引，`OFFSET 90000` 仍然可能比较慢。

原因是：

```text
数据库要跳过前 90000 条，再返回 20 条。
```

这就是为什么大数据分页更推荐游标分页。

---

## 八、游标分页索引

如果使用：

```sql
WHERE (created_at, id) < ($1, $2)
ORDER BY created_at DESC, id DESC
LIMIT 20;
```

建议索引改成：

```sql
CREATE INDEX short_links_created_at_id_desc_idx
ON app.short_links (created_at DESC, id DESC);
```

这样排序和过滤可以更好配合。

但第一版如果只用：

```sql
ORDER BY created_at DESC
```

单列索引已经够练习。

---

## 九、不要盲目加索引

短链接表可能有人想加很多索引：

```text
original_url
visit_count
updated_at
created_at
code
```

但索引不是越多越好。

每个索引都会带来：

- 更多磁盘空间。
- 插入时维护索引的成本。
- 更新字段时维护索引的成本。
- 查询规划成本。

第一版只需要：

```text
code 唯一索引。
created_at 倒序索引。
```

如果以后有这些需求，再补索引：

```text
按 original_url 反查是否已经创建过短链。
按 visit_count 查询热门链接。
按 updated_at 查最近访问过的链接。
```

## 十、索引失效示例

### 1. 对字段做函数

不推荐：

```sql
SELECT *
FROM app.short_links
WHERE lower(code) = lower($1);
```

普通 `code` 索引不能直接高效支持这个条件。

更好的做法是：

```text
短码生成时统一大小写规则。
查询时保持一致。
```

### 2. 类型不匹配

如果字段是 `text`，参数也应该按字符串传入。

不要让数据库做奇怪的隐式转换。

### 3. 低选择性字段

如果以后增加：

```sql
is_active boolean
```

单独给它建索引不一定有效。

因为 `true/false` 选择性太低。

可以考虑部分索引：

```sql
CREATE INDEX short_links_active_created_at_idx
ON app.short_links (created_at DESC)
WHERE is_active = true;
```

## 十一、EXPLAIN ANALYZE 使用注意

`EXPLAIN` 只看计划，不真正执行。

```sql
EXPLAIN
SELECT ...
```

`EXPLAIN ANALYZE` 会真正执行 SQL。

```sql
EXPLAIN ANALYZE
SELECT ...
```

对 `UPDATE`、`DELETE` 使用 `EXPLAIN ANALYZE` 时要小心，因为它真的会修改数据。

如果只是学习，可以放在事务里回滚：

```sql
BEGIN;

EXPLAIN ANALYZE
UPDATE app.short_links
SET visit_count = visit_count + 1
WHERE code = 'code99999'
RETURNING original_url;

ROLLBACK;
```

## 十二、本节达标标准

学完本节后，你应该能够做到：

- 找出短链接服务的核心查询。
- 使用 `EXPLAIN ANALYZE` 分析按 `code` 查询。
- 判断查询是否出现 `Index Scan`。
- 解释 `UNIQUE (code)` 为什么已经提供索引。
- 分析 `ORDER BY created_at DESC LIMIT` 是否能使用索引。
- 解释大 `OFFSET` 为什么慢。
- 知道什么时候考虑游标分页。
- 避免给所有字段盲目建索引。
