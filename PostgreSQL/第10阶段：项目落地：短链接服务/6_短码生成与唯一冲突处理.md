# 6. 短码生成与唯一冲突处理

本节目标：实现短码生成，并正确处理数据库唯一约束冲突。

短链接服务里，短码是核心业务标识。

例如：

```text
http://localhost:8080/aB39xQ
```

其中 `aB39xQ` 就是短码。

---

## 一、短码应该满足什么要求

第一版短码建议满足：

- 长度固定，例如 6 位。
- 字符集可放入 URL 路径。
- 不容易猜到连续规律。
- 数据库中唯一。

字符集可以用：

```text
0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ
```

共 62 个字符。

如果长度是 6，理论组合数是：

```text
62^6 = 56800235584
```

约 568 亿个。

对于学习项目非常够用。

---

## 二、不要用自增 ID 直接转短码作为第一版

有些短链接系统会把自增 ID 转成 Base62。

例如：

```text
1 -> b
2 -> c
10086 -> C8q
```

这也可以，但它有几个问题：

- 容易暴露增长趋势。
- 需要先插入记录拿到 ID，再生成 code，再更新 code。
- 如果想让短码不可预测，还要额外加扰动。

第一版为了简单，可以直接生成随机短码。

但随机短码一定要配合数据库唯一约束和冲突重试。

---

## 三、实现 GenerateCode

创建：

```text
internal/link/code.go
```

代码：

```go
package link

import (
    "crypto/rand"
    "math/big"
)

const codeAlphabet = "0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ"

func GenerateCode(length int) (string, error) {
    if length <= 0 {
        length = 6
    }

    result := make([]byte, length)
    max := big.NewInt(int64(len(codeAlphabet)))

    for i := range result {
        n, err := rand.Int(rand.Reader, max)
        if err != nil {
            return "", err
        }
        result[i] = codeAlphabet[n.Int64()]
    }

    return string(result), nil
}
```

这里使用 `crypto/rand`。

不要用 `math/rand` 生成对外可见的短码。

`math/rand` 更适合模拟、抽样、游戏类场景，不适合生成公开资源标识。

## 四、数据库唯一约束才是最后防线

即使随机空间很大，也不能假设永远不会重复。

表结构里已经有：

```sql
CONSTRAINT short_links_code_unique UNIQUE (code)
```

当插入重复 `code` 时，PostgreSQL 会报唯一约束错误。

Go 代码应该识别这个错误，然后重新生成短码并重试。

---

## 五、识别 PostgreSQL 唯一约束错误

PostgreSQL 唯一约束冲突的 SQLSTATE 是：

```text
23505
```

在 `repository.go` 中可以写一个判断函数：

```go
import (
    "errors"

    "github.com/jackc/pgx/v5/pgconn"
)

func isUniqueViolation(err error) bool {
    var pgErr *pgconn.PgError
    if errors.As(err, &pgErr) {
        return pgErr.Code == "23505"
    }
    return false
}
```

然后在 `Create` 中处理：

```go
if err != nil {
    if isUniqueViolation(err) {
        return Link{}, ErrCodeConflict
    }
    return Link{}, err
}
```

完整片段：

```go
func (r *Repository) Create(ctx context.Context, input CreateLinkInput) (Link, error) {
    const query = `
        INSERT INTO app.short_links (code, original_url)
        VALUES ($1, $2)
        RETURNING id, code, original_url, visit_count, created_at, updated_at
    `

    var item Link
    err := r.pool.QueryRow(ctx, query, input.Code, input.OriginalURL).Scan(
        &item.ID,
        &item.Code,
        &item.OriginalURL,
        &item.VisitCount,
        &item.CreatedAt,
        &item.UpdatedAt,
    )
    if err != nil {
        if isUniqueViolation(err) {
            return Link{}, ErrCodeConflict
        }
        return Link{}, err
    }

    return item, nil
}
```

## 六、service 层重试

短码冲突属于业务可恢复错误。

service 可以重试几次：

```go
const (
    defaultCodeLength = 6
    maxCreateRetries  = 5
)

func (s *Service) Create(ctx context.Context, originalURL string) (Link, error) {
    originalURL = strings.TrimSpace(originalURL)

    if err := validateOriginalURL(originalURL); err != nil {
        return Link{}, err
    }

    for i := 0; i < maxCreateRetries; i++ {
        code, err := GenerateCode(defaultCodeLength)
        if err != nil {
            return Link{}, err
        }

        item, err := s.repo.Create(ctx, CreateLinkInput{
            Code:        code,
            OriginalURL: originalURL,
        })
        if err == nil {
            item.ShortURL = s.baseURL + "/" + item.Code
            return item, nil
        }

        if errors.Is(err, ErrCodeConflict) {
            continue
        }

        return Link{}, err
    }

    return Link{}, ErrRetryExceeded
}
```

需要 import：

```go
import (
    "context"
    "errors"
    "net/url"
    "strings"
)
```

## 七、为什么不先查再插入

有些人会写：

```text
生成 code。
SELECT 看 code 是否存在。
如果不存在，INSERT。
```

这个写法在单请求下没问题，但并发下有竞态。

两个请求可能同时执行：

```text
请求 A 查询 code 不存在。
请求 B 查询 code 不存在。
请求 A 插入成功。
请求 B 插入重复。
```

所以最终仍然需要数据库唯一约束。

既然如此，第一版可以直接：

```text
生成 code。
尝试 INSERT。
如果唯一冲突，就重试。
```

更简单，也更可靠。

## 八、如何验证冲突处理

为了测试，可以临时让 `GenerateCode` 总是返回同一个值：

```go
func GenerateCode(length int) (string, error) {
    return "fixed1", nil
}
```

然后连续请求两次：

```powershell
Invoke-RestMethod -Method Post -Uri "http://localhost:8080/links" -ContentType "application/json" -Body '{"original_url":"https://example.com/1"}'

Invoke-RestMethod -Method Post -Uri "http://localhost:8080/links" -ContentType "application/json" -Body '{"original_url":"https://example.com/2"}'
```

如果 `maxCreateRetries` 后仍然冲突，应该返回内部错误或明确的重试失败错误。

测试后记得恢复随机实现。

## 九、错误应该在哪里转换

推荐分工：

```text
repository:
    PostgreSQL 23505 -> ErrCodeConflict

service:
    ErrCodeConflict -> 重新生成短码并重试
    超过重试次数 -> ErrRetryExceeded

handler:
    ErrInvalidURL -> 400
    ErrRetryExceeded -> 500 或 503
    其他错误 -> 500
```

不要让 handler 直接判断 PostgreSQL SQLSTATE。

HTTP 层不应该知道太多数据库细节。

## 十、短码长度如何选择

长度越短：

- 用户更容易记。
- URL 更短。
- 冲突概率更高。

长度越长：

- 冲突概率更低。
- URL 稍长。

学习项目可以使用 6 位。

真实项目可以从 7 位或 8 位开始。

也可以在冲突频率升高后逐步增加长度。

## 十一、本节达标标准

学完本节后，你应该能够做到：

- 使用 `crypto/rand` 生成短码。
- 解释为什么需要数据库唯一约束兜底。
- 识别 PostgreSQL 唯一约束错误 `23505`。
- 在 repository 层把唯一冲突转换成业务错误。
- 在 service 层对短码冲突进行有限重试。
- 解释为什么“先查再插入”不能解决并发冲突。
