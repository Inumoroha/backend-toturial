# 9. 监控指标与告警

本节目标：知道 PostgreSQL 生产环境应该观察哪些指标，以及这些指标异常时可能意味着什么。

监控的目的不是让仪表盘好看。

它的作用是：

```text
问题发生前能预警。
问题发生时能定位。
问题恢复后能复盘。
```

---

## 一、监控要覆盖哪些方面

PostgreSQL 监控至少关注：

```text
可用性。
连接数。
查询性能。
锁等待。
事务状态。
磁盘空间。
表和索引膨胀。
备份状态。
复制状态，如果有主从。
```

后端开发者不一定负责搭建完整监控系统，但要知道这些指标代表什么。

---

## 二、可用性

最基础检查：

```sql
SELECT 1;
```

应用启动时也会：

```go
pool.Ping(ctx)
```

监控层面要知道：

- 数据库是否能连接。
- 查询是否能执行。
- 响应时间是否异常。

如果 `SELECT 1` 都失败，说明问题已经很严重。

---

## 三、连接数指标

当前连接数：

```sql
SELECT count(*)
FROM pg_stat_activity;
```

按状态分组：

```sql
SELECT state, count(*)
FROM pg_stat_activity
GROUP BY state
ORDER BY count(*) DESC;
```

需要告警的情况：

- 总连接数接近 `max_connections`。
- `active` 连接异常增多。
- `idle in transaction` 长时间存在。

---

## 四、慢查询和查询耗时

从 PostgreSQL 角度：

- 开启慢查询日志。
- 使用 `pg_stat_activity` 看正在运行的慢 SQL。
- 可选使用 `pg_stat_statements` 统计历史 SQL。

`pg_stat_statements` 是 PostgreSQL 常用扩展，用于统计 SQL 执行次数、总耗时、平均耗时等。

如果环境支持，可以启用后查询：

```sql
SELECT
    query,
    calls,
    total_exec_time,
    mean_exec_time,
    rows
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 20;
```

是否能使用取决于数据库配置和扩展启用情况。

---

## 五、锁等待

当前锁等待：

```sql
SELECT
    pid,
    wait_event_type,
    wait_event,
    now() - query_start AS wait_time,
    query
FROM pg_stat_activity
WHERE wait_event_type = 'Lock'
ORDER BY wait_time DESC;
```

需要告警的情况：

- 锁等待持续时间过长。
- 多个业务 SQL 等同一个锁。
- 迁移期间大量请求等待。

锁等待通常和这些有关：

- 长事务。
- DDL 迁移。
- 大批量更新。
- 热点行更新。

---

## 六、事务年龄

长事务会影响 VACUUM。

查看长事务：

```sql
SELECT
    pid,
    state,
    now() - xact_start AS transaction_age,
    query
FROM pg_stat_activity
WHERE xact_start IS NOT NULL
ORDER BY transaction_age DESC
LIMIT 20;
```

需要关注：

```text
transaction_age 很长。
state = idle in transaction。
```

---

## 七、磁盘空间

数据库磁盘空间非常关键。

一旦磁盘满，数据库可能无法写入。

查看数据库大小：

```sql
SELECT pg_size_pretty(pg_database_size(current_database()));
```

查看表大小：

```sql
SELECT
    schemaname,
    relname,
    pg_size_pretty(pg_total_relation_size(relid)) AS total_size
FROM pg_catalog.pg_statio_user_tables
ORDER BY pg_total_relation_size(relid) DESC
LIMIT 20;
```

需要告警：

- 磁盘使用率超过阈值。
- 数据库增长速度异常。
- 某张表突然快速变大。

---

## 八、死元组和 autovacuum

查看：

```sql
SELECT
    schemaname,
    relname,
    n_live_tup,
    n_dead_tup,
    last_autovacuum,
    last_autoanalyze
FROM pg_stat_user_tables
ORDER BY n_dead_tup DESC
LIMIT 20;
```

需要关注：

- `n_dead_tup` 很高。
- 很久没有 `last_autovacuum`。
- 表持续膨胀。

这可能说明：

- 写入更新很频繁。
- autovacuum 跟不上。
- 长事务阻止清理。

---

## 九、备份监控

备份也要监控。

至少要知道：

- 最近一次备份是否成功。
- 备份文件大小是否异常。
- 备份耗时是否异常。
- 是否做过恢复演练。

如果有自动备份任务，失败必须告警。

不要等到需要恢复时才发现备份已经失败一周。

---

## 十、应用侧指标也要结合

数据库指标要和应用指标一起看：

- HTTP 请求耗时。
- HTTP 错误率。
- 数据库查询耗时。
- 连接池等待时间。
- 连接池当前使用数。
- 接口 QPS。

例如：

```text
数据库 active 连接变多。
应用接口耗时升高。
慢查询日志出现同一条 SQL。
```

这几个信号放在一起，定位会更快。

---

## 十一、短链接项目建议告警

短链接服务可以先设置这些告警：

- 数据库无法连接。
- 总连接数超过阈值。
- `idle in transaction` 超过阈值。
- 慢查询超过阈值。
- 磁盘空间不足。
- 备份失败。
- `GET /{code}` 接口错误率升高。
- `GET /{code}` 接口延迟升高。

第一版不需要复杂，但这些方向要有。

---

## 十二、本节达标标准

学完本节后，你应该能够做到：

- 列出 PostgreSQL 生产监控的关键方向。
- 查询当前连接数和连接状态。
- 查询锁等待。
- 查询长事务。
- 查询数据库和表大小。
- 查询死元组和 autovacuum 时间。
- 理解备份任务也需要监控。
- 能为短链接服务列出基本告警项。
