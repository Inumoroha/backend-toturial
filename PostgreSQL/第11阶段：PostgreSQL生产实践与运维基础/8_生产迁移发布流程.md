# 8. 生产迁移发布流程

本节目标：掌握生产环境执行数据库迁移的基本流程，避免表结构变更影响线上业务。

第九阶段已经学习了迁移工具。

这一节关注生产环境：

```text
迁移什么时候执行？
谁执行？
执行前检查什么？
失败后怎么办？
怎样减少锁表风险？
```

---

## 一、生产迁移和本地迁移的区别

本地迁移：

```text
数据少。
可以重建。
失败成本低。
没人正在使用。
```

生产迁移：

```text
数据多。
不能随便重建。
失败可能影响用户。
业务请求持续写入。
锁表会造成接口卡住。
```

所以生产迁移必须走流程。

---

## 二、迁移发布前检查清单

每次生产迁移前，至少检查：

- 迁移文件是否已经 code review。
- 是否在本地和测试环境执行成功。
- down 迁移是否合理。
- 是否涉及大表。
- 是否涉及长时间锁。
- 是否涉及数据删除。
- 是否涉及字段类型修改。
- 是否需要数据回填。
- 是否和应用代码兼容。
- 是否有备份或可恢复方案。
- 是否选择低峰期执行。

尤其要警惕：

```sql
DROP COLUMN
ALTER COLUMN TYPE
ADD COLUMN NOT NULL
CREATE INDEX
UPDATE 大量历史数据
```

---

## 三、向后兼容迁移

生产发布最好使用向后兼容策略。

不要一次同时做：

```text
删除旧字段。
上线只读新字段的代码。
回填历史数据。
```

推荐分步骤：

### 第一步：添加新字段

```sql
ALTER TABLE users ADD COLUMN nickname text;
```

旧代码不受影响。

### 第二步：上线双写代码

新代码同时写旧字段和新字段。

### 第三步：回填历史数据

分批更新旧数据。

### 第四步：切读新字段

确认数据完整后，代码开始读新字段。

### 第五步：删除旧字段

观察稳定后，再单独迁移删除旧字段。

这种方式麻烦一点，但更安全。

---

## 四、避免长事务和大批量回填

不要在一个迁移里做：

```sql
UPDATE huge_table
SET new_col = expensive_function(old_col);
```

如果表很大，这可能：

- 执行很久。
- 产生大量 WAL。
- 持有大量锁。
- 造成表膨胀。
- 影响正常业务。

更推荐：

```text
先加字段。
应用兼容空值。
后台任务分批回填。
每批几百或几千行。
观察数据库压力。
```

示例批量回填思路：

```sql
UPDATE app.short_links
SET updated_at = now()
WHERE id IN (
    SELECT id
    FROM app.short_links
    WHERE updated_at IS NULL
    LIMIT 1000
);
```

实际项目要根据主键和条件设计批次。

---

## 五、创建索引要小心

普通创建索引：

```sql
CREATE INDEX short_links_created_at_idx
ON app.short_links (created_at);
```

在生产大表上可能影响写入。

PostgreSQL 支持并发创建索引：

```sql
CREATE INDEX CONCURRENTLY short_links_created_at_idx
ON app.short_links (created_at);
```

它对写入更友好，但也有注意点：

```text
不能放在普通事务块里执行。
执行时间可能更长。
失败后可能留下无效索引，需要清理。
```

很多迁移工具默认把迁移放在事务里，所以要确认工具如何支持 `CONCURRENTLY`。

---

## 六、设置 lock_timeout

生产 DDL 可以先设置：

```sql
SET lock_timeout = '3s';
```

然后执行：

```sql
ALTER TABLE app.short_links
ADD COLUMN description text;
```

如果 3 秒内拿不到锁，迁移失败。

这比一直等待锁、阻塞发布流程更可控。

也可以设置：

```sql
SET statement_timeout = '30s';
```

防止语句执行过久。

具体时间要结合业务。

---

## 七、迁移和应用代码的发布顺序

一般有两种情况。

### 1. 新增字段

推荐顺序：

```text
先执行迁移，新增字段。
再上线使用新字段的代码。
```

因为旧代码不会受新增 nullable 字段影响。

### 2. 删除字段

推荐顺序：

```text
先上线不再使用旧字段的代码。
观察稳定。
再执行迁移删除旧字段。
```

不要代码还在读字段，就先把字段删了。

---

## 八、生产迁移失败怎么办

迁移失败后，不要立刻乱改数据库。

先判断：

- 迁移执行到哪一步。
- 是否有部分变更已生效。
- 迁移工具状态是否 dirty。
- 是否需要手动修复版本表。
- 是否需要回滚代码。
- 是否需要恢复备份。

如果是 golang-migrate，可能出现 dirty version。

这时要先确认数据库实际状态，再决定是否 `force` 到某个版本。

不要在不了解状态时强行修改版本。

## 九、危险迁移示例

### 1. 直接删除字段

```sql
ALTER TABLE users DROP COLUMN phone;
```

危险点：

- 数据不可恢复，除非有备份。
- 旧代码可能还在读取。

### 2. 大表全量更新

```sql
UPDATE orders SET status = 'paid';
```

危险点：

- 影响大量行。
- 产生大量 WAL。
- 锁和膨胀风险高。

### 3. 修改字段类型

```sql
ALTER TABLE events ALTER COLUMN payload TYPE jsonb USING payload::jsonb;
```

危险点：

- 可能重写表。
- 可能失败。
- 可能长时间锁表。

## 十、短链接项目迁移实践

假设要给短链接增加过期时间：

```sql
ALTER TABLE app.short_links
ADD COLUMN expires_at timestamptz;
```

这是相对安全的新增 nullable 字段。

发布顺序：

1. 执行迁移，新增 `expires_at`。
2. 上线创建短链时可选过期时间的代码。
3. 跳转接口增加过期判断。
4. 观察日志和错误率。

如果要新增索引：

```sql
CREATE INDEX CONCURRENTLY short_links_expires_at_idx
ON app.short_links (expires_at)
WHERE expires_at IS NOT NULL;
```

要确认迁移工具支持非事务迁移。

---

## 十一、本节达标标准

学完本节后，你应该能够做到：

- 区分本地迁移和生产迁移的风险。
- 写出生产迁移前检查清单。
- 理解向后兼容迁移策略。
- 知道大批量数据回填应该分批做。
- 知道生产大表创建索引可以考虑 `CONCURRENTLY`。
- 使用 `lock_timeout` 降低锁等待风险。
- 判断新增字段和删除字段的发布顺序。
- 迁移失败后先确认状态，而不是盲目修复。
