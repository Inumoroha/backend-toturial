# 3. 角色、用户与权限控制

本节目标：理解 PostgreSQL 的角色和权限，学会为应用创建最小权限账号。

生产项目里，应用连接数据库时不应该使用超级用户。

错误示例：

```text
应用直接使用 postgres 超级用户连接生产库。
```

这样一旦应用配置泄露，攻击者可能拥有极高权限。

正确思路：

```text
为应用单独创建账号。
只授予它访问业务 schema 和表的必要权限。
```

---

## 一、角色和用户是什么关系

PostgreSQL 里使用 `ROLE` 表示权限主体。

角色可以：

- 登录。
- 不登录。
- 拥有对象。
- 被授予权限。
- 作为权限组。

常见写法：

```sql
CREATE ROLE app_user LOGIN PASSWORD 'strong_password';
```

这里 `LOGIN` 表示这个角色可以作为用户登录。

所以在 PostgreSQL 里：

```text
用户本质上是带 LOGIN 权限的角色。
```

---

## 二、不要用 postgres 作为应用账号

`postgres` 通常是超级用户。

超级用户可以做很多危险操作：

```sql
DROP DATABASE learn_pg;
CREATE ROLE hacker SUPERUSER;
ALTER SYSTEM SET ...
```

应用服务没有必要拥有这些权限。

应用一般只需要：

- 连接数据库。
- 使用某个 schema。
- 查询、插入、更新、删除业务表。
- 使用序列。

---

## 三、创建应用账号

示例：

```sql
CREATE ROLE shortener_app LOGIN PASSWORD 'replace_with_strong_password';
```

给它连接数据库的权限：

```sql
GRANT CONNECT ON DATABASE learn_pg TO shortener_app;
```

给它使用 schema 的权限：

```sql
GRANT USAGE ON SCHEMA app TO shortener_app;
```

给它业务表权限：

```sql
GRANT SELECT, INSERT, UPDATE, DELETE
ON ALL TABLES IN SCHEMA app
TO shortener_app;
```

如果有序列：

```sql
GRANT USAGE, SELECT
ON ALL SEQUENCES IN SCHEMA app
TO shortener_app;
```

短链接表使用 UUID 主键，不依赖序列，但其他表可能会用 identity 或 serial。

---

## 四、未来新表的默认权限

上面的授权只对当前已有表生效。

以后新建表，可能还要重新授权。

可以设置默认权限：

```sql
ALTER DEFAULT PRIVILEGES IN SCHEMA app
GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO shortener_app;
```

序列：

```sql
ALTER DEFAULT PRIVILEGES IN SCHEMA app
GRANT USAGE, SELECT ON SEQUENCES TO shortener_app;
```

注意：

```text
默认权限和“谁创建对象”有关。
```

如果迁移由 `migration_user` 执行，那么默认权限要对那个创建者设置才有意义。

---

## 五、迁移账号和应用账号分开

推荐生产环境至少区分：

```text
migration_user
shortener_app
readonly_user
```

### 1. `migration_user`

用于执行迁移。

它需要：

- 创建表。
- 修改表。
- 创建索引。
- 创建约束。

权限相对更大。

### 2. `shortener_app`

应用运行时账号。

它只需要日常业务读写。

不应该拥有随意 `DROP TABLE`、`ALTER TABLE` 的权限。

### 3. `readonly_user`

只读账号。

用于：

- 报表。
- 临时查询。
- 数据分析。
- 排查问题。

只授予：

```sql
SELECT
```

---

## 六、创建只读账号

```sql
CREATE ROLE shortener_readonly LOGIN PASSWORD 'replace_with_strong_password';

GRANT CONNECT ON DATABASE learn_pg TO shortener_readonly;
GRANT USAGE ON SCHEMA app TO shortener_readonly;
GRANT SELECT ON ALL TABLES IN SCHEMA app TO shortener_readonly;

ALTER DEFAULT PRIVILEGES IN SCHEMA app
GRANT SELECT ON TABLES TO shortener_readonly;
```

只读账号不能：

```sql
INSERT
UPDATE
DELETE
ALTER TABLE
DROP TABLE
```

这能降低误操作风险。

---

## 七、查看当前用户和权限

查看当前用户：

```sql
SELECT current_user;
```

查看角色：

```sql
\du
```

查看表权限：

```sql
\dp app.short_links
```

或者查询系统视图：

```sql
SELECT grantee, privilege_type
FROM information_schema.role_table_grants
WHERE table_schema = 'app'
  AND table_name = 'short_links'
ORDER BY grantee, privilege_type;
```

---

## 八、撤销权限

撤销某个表的写权限：

```sql
REVOKE INSERT, UPDATE, DELETE
ON app.short_links
FROM shortener_readonly;
```

撤销 schema 使用权限：

```sql
REVOKE USAGE ON SCHEMA app
FROM shortener_readonly;
```

撤销连接权限：

```sql
REVOKE CONNECT ON DATABASE learn_pg
FROM shortener_readonly;
```

权限要会授予，也要会撤销。

---

## 九、权限设计常见误区

### 1. 所有服务共用一个数据库账号

不推荐。

不同服务应该尽量使用不同账号。

这样出问题时更容易定位，也能限制影响范围。

### 2. 应用账号拥有 DDL 权限

应用运行时通常不需要：

```sql
ALTER TABLE
DROP TABLE
CREATE TABLE
```

这些应该由迁移流程负责。

### 3. 只读账号其实能写

有些团队创建了 readonly，但忘了撤销默认写权限。

必须实际验证。

### 4. 密码写在代码里

不要把数据库密码写死在 Go 文件中。

使用环境变量或密钥管理系统。

## 十、短链接项目推荐权限

对短链接项目，可以设计：

```sql
CREATE ROLE shortener_migration LOGIN PASSWORD '...';
CREATE ROLE shortener_app LOGIN PASSWORD '...';
CREATE ROLE shortener_readonly LOGIN PASSWORD '...';
```

迁移账号负责创建：

```text
app schema
app.short_links
索引
约束
```

应用账号只负责：

```text
SELECT
INSERT
UPDATE
DELETE
```

只读账号只负责：

```text
SELECT
```

## 十一、本节达标标准

学完本节后，你应该能够做到：

- 解释 PostgreSQL 中角色和用户的关系。
- 创建一个可以登录的应用角色。
- 给应用角色授予数据库、schema 和表权限。
- 创建只读账号。
- 理解迁移账号和应用账号为什么要分开。
- 使用 `\du`、`\dp` 或系统视图查看权限。
- 知道应用不应该使用 `postgres` 超级用户连接生产库。
