# 4. 连接数、连接池与应用配置

本节目标：理解 PostgreSQL 连接数和 Go `pgxpool` 的关系，避免应用连接池配置不当把数据库打满。

连接池是后端服务访问 PostgreSQL 的必备组件。

但连接池不是越大越好。

很多生产问题不是 SQL 写错，而是：

```text
服务实例太多。
每个实例连接池太大。
总连接数超过 PostgreSQL 能承受的范围。
```

---

## 一、PostgreSQL 连接不是免费资源

每个 PostgreSQL 连接都会占用数据库资源。

连接多了会消耗：

- 内存。
- 后端进程或线程资源。
- 上下文切换成本。
- 锁和事务管理成本。

所以不要把连接池理解成：

```text
开得越大，并发越高。
```

连接池的作用是控制和复用连接，而不是无限扩张连接。

---

## 二、查看最大连接数

查看 PostgreSQL 当前配置：

```sql
SHOW max_connections;
```

返回可能是：

```text
100
```

这表示数据库允许的最大连接数是 100。

但这 100 个连接不能全部给一个应用用。

还要留给：

- 数据库管理连接。
- 迁移工具。
- 后台任务。
- 只读查询。
- 监控系统。
- 紧急排查。

---

## 三、查看当前连接

```sql
SELECT
    state,
    count(*)
FROM pg_stat_activity
WHERE datname = current_database()
GROUP BY state
ORDER BY count(*) DESC;
```

常见状态：

| 状态 | 含义 |
| --- | --- |
| `active` | 正在执行 SQL |
| `idle` | 空闲连接 |
| `idle in transaction` | 事务开启但当前空闲 |

最需要警惕的是：

```text
idle in transaction
```

它表示事务没有结束，可能持有锁，也可能阻止 VACUUM 清理旧版本。

---

## 四、应用实例数和连接池总量

假设：

```text
PostgreSQL max_connections = 100
Go 服务实例数 = 5
每个实例 pgxpool MaxConns = 20
```

应用理论最大连接数：

```text
5 * 20 = 100
```

这已经把数据库连接占满了。

如果再加：

- 迁移工具。
- 管理连接。
- 定时任务。
- 监控连接。

就可能出现：

```text
too many connections
```

更合理的方式是先预留一部分连接。

例如：

```text
max_connections = 100
预留 20
应用可用 80
服务实例 5 个
每个实例 MaxConns 不超过 16
```

## 五、pgxpool 推荐配置

示例：

```go
func NewPool(ctx context.Context, databaseURL string) (*pgxpool.Pool, error) {
    cfg, err := pgxpool.ParseConfig(databaseURL)
    if err != nil {
        return nil, err
    }

    cfg.MaxConns = 10
    cfg.MinConns = 1
    cfg.MaxConnLifetime = time.Hour
    cfg.MaxConnIdleTime = 30 * time.Minute
    cfg.HealthCheckPeriod = time.Minute

    pool, err := pgxpool.NewWithConfig(ctx, cfg)
    if err != nil {
        return nil, err
    }

    pingCtx, cancel := context.WithTimeout(ctx, 5*time.Second)
    defer cancel()

    if err := pool.Ping(pingCtx); err != nil {
        pool.Close()
        return nil, err
    }

    return pool, nil
}
```

学习项目可以先用：

```go
MaxConns = 10
```

生产项目要结合实例数量计算。

---

## 六、连接池配置要用环境变量

不要把连接池大小写死。

推荐配置：

```text
DB_MAX_CONNS
DB_MIN_CONNS
DB_MAX_CONN_LIFETIME
DB_MAX_CONN_IDLE_TIME
```

例如：

```go
type Config struct {
    DatabaseURL string
    DBMaxConns  int32
    DBMinConns  int32
}
```

这样不同环境可以不同配置：

```text
本地：MaxConns = 5
测试：MaxConns = 10
生产：按实例数计算
```

---

## 七、请求超时很重要

数据库查询应该有超时。

handler 中：

```go
ctx, cancel := context.WithTimeout(r.Context(), 3*time.Second)
defer cancel()

item, err := h.service.GetByCode(ctx, code)
```

如果没有超时，慢查询可能拖住连接很久。

连接被拖住后，新请求又拿不到连接，应用会进一步堆积。

---

## 八、连接池耗尽的表现

当连接池耗尽时，应用可能出现：

- 请求变慢。
- 大量请求等待获取连接。
- 接口超时。
- 数据库连接数接近上限。
- 日志里出现连接获取超时。

排查时先看：

```sql
SELECT count(*)
FROM pg_stat_activity
WHERE datname = current_database();
```

再看状态分布：

```sql
SELECT state, count(*)
FROM pg_stat_activity
WHERE datname = current_database()
GROUP BY state;
```

如果 `active` 很多，可能 SQL 慢。

如果 `idle` 很多，可能连接池开太大。

如果 `idle in transaction` 很多，可能事务没有正确提交或回滚。

---

## 九、idle in transaction 为什么危险

`idle in transaction` 表示：

```text
事务已经 BEGIN。
当前没有执行 SQL。
但事务还没 COMMIT 或 ROLLBACK。
```

它可能导致：

- 锁不释放。
- VACUUM 无法清理旧版本。
- 表膨胀。
- 其他事务等待。

Go 代码里常见原因：

```go
tx, err := pool.Begin(ctx)
if err != nil {
    return err
}

// 中途 return 了，但忘记 Rollback
```

推荐写法：

```go
tx, err := pool.Begin(ctx)
if err != nil {
    return err
}
defer tx.Rollback(ctx)

// 执行业务 SQL

if err := tx.Commit(ctx); err != nil {
    return err
}
```

## 十、PgBouncer 是什么

当连接数压力很大时，很多生产环境会使用 PgBouncer。

它是 PostgreSQL 前面的连接池代理。

作用是：

```text
让应用连接 PgBouncer。
PgBouncer 复用到 PostgreSQL 的后端连接。
```

本阶段不深入配置 PgBouncer，但要知道：

- 它可以缓解连接数压力。
- 它不是慢 SQL 的根治方案。
- 它会影响某些会话级特性使用。

对于普通 Go 项目，先把 `pgxpool` 配好更重要。

## 十一、短链接项目建议配置

本地：

```text
DB_MAX_CONNS=5
```

小型生产：

```text
PostgreSQL max_connections = 100
服务实例数 = 4
预留连接 = 20
每实例 MaxConns = 15 到 20 之间
```

计算方式：

```text
(max_connections - reserved_connections) / instance_count
```

不要忘记定时任务、迁移工具和后台管理服务也可能连接同一个数据库。

---

## 十二、本节达标标准

学完本节后，你应该能够做到：

- 解释为什么连接不是越多越好。
- 使用 `SHOW max_connections` 查看最大连接数。
- 使用 `pg_stat_activity` 查看当前连接状态。
- 根据服务实例数估算每个实例的连接池大小。
- 给数据库查询设置 context 超时。
- 识别 `idle in transaction` 的风险。
- 知道 PgBouncer 的基本作用。
