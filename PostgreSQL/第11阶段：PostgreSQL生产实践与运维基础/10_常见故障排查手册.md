# 10. 常见故障排查手册

本节目标：整理 PostgreSQL 生产环境常见问题的排查顺序，让你遇到问题时知道先看什么。

排查故障最怕两件事：

```text
没有信息，只能猜。
看到很多信息，但不知道顺序。
```

这一节给出后端开发者常用的排查路径。

---

## 一、接口突然变慢

先确认是不是数据库问题。

应用侧看：

- 哪个接口变慢。
- 是所有接口慢，还是某几个接口慢。
- 错误率是否升高。
- 最近是否上线新版本。

数据库侧看正在执行的 SQL：

```sql
SELECT
    pid,
    state,
    wait_event_type,
    wait_event,
    now() - query_start AS running_time,
    query
FROM pg_stat_activity
WHERE datname = current_database()
  AND state <> 'idle'
ORDER BY running_time DESC
LIMIT 20;
```

如果看到慢 SQL，下一步：

```sql
EXPLAIN ANALYZE
SELECT ...
```

注意生产上不要随便对写操作使用 `EXPLAIN ANALYZE`。

---

## 二、出现 too many connections

错误：

```text
too many connections
```

先看最大连接数：

```sql
SHOW max_connections;
```

再看当前连接：

```sql
SELECT count(*)
FROM pg_stat_activity;
```

按用户和应用分组：

```sql
SELECT
    usename,
    application_name,
    state,
    count(*)
FROM pg_stat_activity
GROUP BY usename, application_name, state
ORDER BY count(*) DESC;
```

常见原因：

- Go 服务实例数增加，但连接池没调小。
- 每个实例 `MaxConns` 太大。
- 连接泄露。
- 慢查询占住连接。
- 事务没有结束。

处理方向：

- 降低应用连接池上限。
- 增加请求和查询超时。
- 排查慢查询。
- 清理异常连接。
- 必要时引入 PgBouncer。

## 三、出现 idle in transaction

查询：

```sql
SELECT
    pid,
    usename,
    client_addr,
    now() - xact_start AS transaction_age,
    query
FROM pg_stat_activity
WHERE state = 'idle in transaction'
ORDER BY transaction_age DESC;
```

常见原因：

- Go 代码开启事务后忘记 `Commit` 或 `Rollback`。
- 中途 `return` 没有执行回滚。
- 事务里做了耗时操作。
- 客户端工具开启事务后忘记提交。

代码修复方向：

```go
tx, err := pool.Begin(ctx)
if err != nil {
    return err
}
defer tx.Rollback(ctx)

// SQL 操作

return tx.Commit(ctx)
```

生产临时处理可以考虑终止异常连接，但要先确认影响：

```sql
SELECT pg_terminate_backend(pid);
```

## 四、迁移卡住

现象：

```text
迁移命令一直不结束。
业务接口也开始变慢。
```

先查锁等待：

```sql
SELECT
    pid,
    state,
    wait_event_type,
    wait_event,
    now() - query_start AS running_time,
    query
FROM pg_stat_activity
WHERE wait_event_type = 'Lock'
ORDER BY running_time DESC;
```

如果迁移在等锁，或者业务在等迁移持有的锁，要立刻评估：

- 是否取消迁移。
- 是否回滚发布。
- 是否进入维护窗口。

生产迁移前最好设置：

```sql
SET lock_timeout = '3s';
```

避免长时间等待锁。

## 五、查询走了 Seq Scan

看到执行计划：

```text
Seq Scan on short_links
```

不要立刻判定错误。

先判断：

- 表是不是很小。
- 查询是否返回大部分数据。
- 条件选择性是否低。
- 有没有合适索引。
- 统计信息是否过期。

检查索引：

```sql
SELECT indexname, indexdef
FROM pg_indexes
WHERE schemaname = 'app'
  AND tablename = 'short_links';
```

更新统计信息：

```sql
ANALYZE app.short_links;
```

再看执行计划：

```sql
EXPLAIN ANALYZE
SELECT ...
```

## 六、磁盘空间增长很快

先看数据库大小：

```sql
SELECT pg_size_pretty(pg_database_size(current_database()));
```

看大表：

```sql
SELECT
    schemaname,
    relname,
    pg_size_pretty(pg_total_relation_size(relid)) AS total_size
FROM pg_catalog.pg_statio_user_tables
ORDER BY pg_total_relation_size(relid) DESC
LIMIT 20;
```

看死元组：

```sql
SELECT
    schemaname,
    relname,
    n_live_tup,
    n_dead_tup,
    last_autovacuum
FROM pg_stat_user_tables
ORDER BY n_dead_tup DESC
LIMIT 20;
```

常见原因：

- 大量 UPDATE 或 DELETE。
- 长事务阻止 VACUUM。
- autovacuum 跟不上。
- 新增索引或大字段。
- 日志文件暴涨。

处理方向：

- 找出增长最快的表。
- 检查长事务。
- 检查 autovacuum。
- 优化高频更新设计。
- 必要时安排维护窗口。

## 七、唯一约束冲突突然增多

短链接项目中可能出现：

```text
duplicate key value violates unique constraint short_links_code_unique
```

如果偶发，正常。

如果突然大量出现，要检查：

- 短码生成是否退化。
- 随机数实现是否被改坏。
- code 长度是否太短。
- 是否有固定 code 测试代码没删除。
- 是否并发重试逻辑有问题。

应用应该捕获 SQLSTATE：

```text
23505
```

然后重试，而不是直接返回数据库错误给用户。

---

## 八、外键约束失败

错误：

```text
violates foreign key constraint
```

常见原因：

- 插入子表时父表记录不存在。
- 删除父表时还有子表引用。
- 迁移顺序不对。
- 测试数据不完整。

排查：

```sql
SELECT *
FROM child_table
WHERE parent_id = '...';
```

以及确认父表：

```sql
SELECT *
FROM parent_table
WHERE id = '...';
```

业务上要决定：

- 禁止删除。
- 级联删除。
- 软删除。
- 先删除子表。

## 九、数据库连接正常但应用报超时

可能原因：

- SQL 慢。
- 连接池耗尽。
- 等锁。
- 网络抖动。
- context timeout 太短。
- 数据库 CPU 或 IO 繁忙。

排查顺序：

1. 看应用日志里的超时位置。
2. 查 `pg_stat_activity`。
3. 查连接数。
4. 查锁等待。
5. 查慢查询日志。
6. 查数据库机器资源。

不要只看到 timeout 就把超时时间调大。

调大 timeout 可能只是让请求堆积更久。

## 十、误删数据怎么办

第一反应不是继续操作。

应该：

```text
停止相关写入。
确认误删范围。
记录时间点。
确认最近备份。
评估是否能从备份恢复。
必要时恢复到临时库，再导回误删数据。
```

如果有完整备份，可以：

1. 恢复备份到临时数据库。
2. 找出误删数据。
3. 导出需要恢复的数据。
4. 谨慎导回生产。

这就是为什么恢复演练重要。

---

## 十一、本节达标标准

学完本节后，你应该能够做到：

- 接口变慢时知道先查 `pg_stat_activity`。
- 出现连接耗尽时知道如何按用户和状态分组查看连接。
- 识别并处理 `idle in transaction`。
- 迁移卡住时能判断是否锁等待。
- 分析 `Seq Scan` 是否真的有问题。
- 磁盘增长时能查看大表和死元组。
- 唯一约束和外键错误出现时知道排查方向。
- 误删数据时知道先停止写入并从备份恢复验证。
