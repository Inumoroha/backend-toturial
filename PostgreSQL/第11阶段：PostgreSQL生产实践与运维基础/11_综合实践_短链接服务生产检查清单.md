# 11. 综合实践：短链接服务生产检查清单

本节目标：把第十一阶段的生产实践应用到第十阶段的短链接服务中，形成一份上线前检查清单。

这不是让你马上搭建复杂生产环境。

而是让你具备这样的能力：

```text
一个 Go + PostgreSQL 项目准备上线前，我知道要检查什么。
```

---

## 一、项目背景

短链接服务包含：

```text
POST /links
GET /{code}
GET /links/{code}
GET /links
```

核心表：

```text
app.short_links
```

核心风险：

- 短码唯一冲突。
- 跳转接口高频查询。
- 访问次数高频更新。
- 列表分页可能变慢。
- 应用连接池可能过大。
- 迁移可能影响线上访问。

---

## 二、数据库结构检查

检查表是否存在：

```sql
SELECT to_regclass('app.short_links');
```

检查字段：

```sql
SELECT column_name, data_type, is_nullable
FROM information_schema.columns
WHERE table_schema = 'app'
  AND table_name = 'short_links'
ORDER BY ordinal_position;
```

检查约束：

```sql
SELECT conname, contype
FROM pg_constraint
WHERE conrelid = 'app.short_links'::regclass;
```

应该有：

```text
主键约束。
code 唯一约束。
code 非空和非空字符串检查。
original_url 非空和非空字符串检查。
visit_count 非负检查。
```

---

## 三、索引检查

```sql
SELECT indexname, indexdef
FROM pg_indexes
WHERE schemaname = 'app'
  AND tablename = 'short_links';
```

应该有：

```text
short_links_pkey
short_links_code_unique
short_links_created_at_desc_idx
```

说明：

```text
code 唯一约束自带唯一索引。
created_at 索引用于列表倒序。
```

不要重复给 `code` 再建普通索引。

---

## 四、核心查询计划检查

详情查询：

```sql
EXPLAIN ANALYZE
SELECT id, code, original_url, visit_count, created_at, updated_at
FROM app.short_links
WHERE code = 'abc123';
```

跳转更新：

```sql
BEGIN;

EXPLAIN ANALYZE
UPDATE app.short_links
SET visit_count = visit_count + 1,
    updated_at = now()
WHERE code = 'abc123'
RETURNING original_url;

ROLLBACK;
```

列表查询：

```sql
EXPLAIN ANALYZE
SELECT id, code, original_url, visit_count, created_at, updated_at
FROM app.short_links
ORDER BY created_at DESC
LIMIT 20 OFFSET 0;
```

关注：

```text
是否出现合理的 Index Scan。
是否出现大范围 Seq Scan + Sort。
实际耗时是否可接受。
```

---

## 五、权限检查

应用不应该使用 `postgres` 超级用户。

检查当前连接用户：

```sql
SELECT current_user;
```

检查角色：

```sql
\du
```

检查表权限：

```sql
SELECT grantee, privilege_type
FROM information_schema.role_table_grants
WHERE table_schema = 'app'
  AND table_name = 'short_links'
ORDER BY grantee, privilege_type;
```

推荐：

```text
shortener_migration：执行迁移。
shortener_app：应用读写。
shortener_readonly：只读排查。
```

---

## 六、连接池检查

查看数据库最大连接数：

```sql
SHOW max_connections;
```

估算应用连接：

```text
服务实例数 * 每实例 DB_MAX_CONNS
```

例如：

```text
4 个实例 * 15 = 60 个应用连接
```

要给这些保留空间：

- 管理连接。
- 迁移工具。
- 监控。
- 后台任务。

检查当前连接：

```sql
SELECT
    usename,
    state,
    count(*)
FROM pg_stat_activity
WHERE datname = current_database()
GROUP BY usename, state
ORDER BY count(*) DESC;
```

---

## 七、超时配置检查

应用层要有：

```text
HTTP server timeout。
数据库查询 context timeout。
pgxpool 连接池上限。
```

数据库层可以考虑：

```sql
SHOW statement_timeout;
SHOW lock_timeout;
SHOW idle_in_transaction_session_timeout;
```

生产迁移时建议设置：

```sql
SET lock_timeout = '3s';
```

防止迁移一直等待锁。

---

## 八、备份恢复检查

上线前至少做一次恢复演练。

备份：

```powershell
pg_dump -Fc -f shortener_backup.dump $env:DATABASE_URL
```

恢复到测试库：

```powershell
pg_restore -d $env:RESTORE_DATABASE_URL shortener_backup.dump
```

验证：

```sql
SELECT count(*)
FROM app.short_links;

SELECT code, original_url
FROM app.short_links
ORDER BY created_at DESC
LIMIT 5;
```

检查点：

- 备份能生成。
- 备份能恢复。
- 恢复后关键表存在。
- 数据量符合预期。
- 恢复耗时已记录。

---

## 九、慢查询和监控检查

建议开启慢查询日志，例如：

```text
log_min_duration_statement = 500ms
```

具体阈值根据业务调整。

监控至少包括：

- 数据库是否可连接。
- 当前连接数。
- 慢查询数量。
- 锁等待。
- 长事务。
- 磁盘空间。
- 备份是否成功。
- 跳转接口延迟。
- 跳转接口错误率。

## 十、迁移发布检查

每次生产迁移前问：

```text
这个迁移是否已经在测试环境执行？
是否涉及大表？
是否会重写表？
是否会创建索引？
是否会删除数据？
是否兼容旧代码？
是否需要低峰期？
失败后怎么办？
```

短链接服务新增字段时，优先：

```sql
ALTER TABLE app.short_links
ADD COLUMN expires_at timestamptz;
```

这种 nullable 字段通常更容易向后兼容。

删除字段要分阶段做，不要和代码发布混在一起。

## 十一、故障演练清单

可以在本地或测试环境模拟：

### 1. 慢查询

执行一个低效查询，用 `pg_stat_activity` 观察。

### 2. 长事务

一个会话：

```sql
BEGIN;
SELECT * FROM app.short_links;
```

不提交。

另一个会话查看：

```sql
SELECT state, now() - xact_start AS age, query
FROM pg_stat_activity
WHERE state = 'idle in transaction';
```

### 3. 唯一冲突

```sql
INSERT INTO app.short_links (code, original_url)
VALUES ('fixed1', 'https://example.com/1');

INSERT INTO app.short_links (code, original_url)
VALUES ('fixed1', 'https://example.com/2');
```

观察错误，确认 Go 代码能转换成业务错误并重试。

## 十二、最终上线前清单

上线前逐项确认：

- 迁移文件已进入版本控制。
- 生产迁移流程明确。
- 应用账号不是超级用户。
- 应用账号权限最小化。
- 连接池大小经过计算。
- 查询有超时。
- 核心查询执行计划已检查。
- 慢查询日志已配置。
- 有备份。
- 做过恢复演练。
- 有基本监控和告警。
- 知道如何查看连接、锁等待和长事务。
- README 写清楚部署和迁移命令。

---

## 十三、第十一阶段最终达标标准

学完第十一阶段后，你应该能够做到：

- 给一个 PostgreSQL 项目写出上线前数据库检查清单。
- 完成一次 `pg_dump` 和 `pg_restore` 恢复演练。
- 为应用创建最小权限账号。
- 根据实例数估算连接池大小。
- 用 `pg_stat_activity` 排查慢查询、锁等待和长事务。
- 理解 VACUUM、ANALYZE、autovacuum 和表膨胀。
- 知道生产迁移要评估锁、数据量和兼容性。
- 能把短链接服务从学习项目推进到具备基本生产意识的后端项目。
