# 2. 备份与恢复基础

本节目标：掌握 PostgreSQL 最基本的备份和恢复方式，理解“备份可用”必须通过恢复演练证明。

生产数据库最重要的底线是：

```text
数据不能轻易丢。
```

但真正可靠的不是“我有备份文件”，而是：

```text
我能把备份恢复出来，并验证数据正确。
```

---

## 一、备份分两类

PostgreSQL 常见备份方式可以粗略分成：

```text
逻辑备份。
物理备份。
```

本阶段重点学习逻辑备份。

### 1. 逻辑备份

使用工具：

```text
pg_dump
pg_dumpall
```

它导出的是数据库对象和数据的逻辑表示。

可以理解为：

```text
把表结构、索引、约束、数据导出来。
```

适合：

- 小中型数据库。
- 开发、测试环境复制数据。
- 单库迁移。
- 学习和普通项目备份。

### 2. 物理备份

物理备份保存的是 PostgreSQL 数据目录和 WAL。

适合：

- 大型数据库。
- 高恢复要求。
- 时间点恢复。
- 主从复制和灾备。

物理备份更复杂，本阶段先知道概念即可。

---

## 二、使用 pg_dump 备份

备份成 SQL 文本：

```powershell
pg_dump $env:DATABASE_URL > backup.sql
```

这个文件里包含 SQL 语句，可以用文本编辑器查看。

恢复时使用：

```powershell
psql $env:DATABASE_URL < backup.sql
```

这种方式直观，但大文件恢复不够灵活。

---

## 三、推荐使用 custom 格式

更推荐使用 PostgreSQL 自定义格式：

```powershell
pg_dump -Fc -f backup.dump $env:DATABASE_URL
```

解释：

```text
-F 指定格式。
c 表示 custom。
-f 指定输出文件。
```

恢复时使用：

```powershell
pg_restore -d $env:DATABASE_URL backup.dump
```

custom 格式的好处：

- 可以并行恢复。
- 可以选择恢复部分对象。
- 更适合 `pg_restore` 管理。

---

## 四、备份单个数据库

如果不用连接字符串，也可以写：

```powershell
pg_dump -h localhost -p 5432 -U postgres -d learn_pg -Fc -f learn_pg.dump
```

参数含义：

| 参数 | 含义 |
| --- | --- |
| `-h` | 主机 |
| `-p` | 端口 |
| `-U` | 用户 |
| `-d` | 数据库名 |
| `-Fc` | custom 格式 |
| `-f` | 输出文件 |

---

## 五、恢复到新数据库

不要一开始就往原库恢复。

先恢复到一个新库验证：

```sql
CREATE DATABASE learn_pg_restore_test;
```

设置恢复连接：

```powershell
$env:RESTORE_DATABASE_URL="postgres://postgres:postgres@localhost:5432/learn_pg_restore_test?sslmode=disable"
```

恢复：

```powershell
pg_restore -d $env:RESTORE_DATABASE_URL backup.dump
```

验证：

```sql
SELECT count(*)
FROM app.short_links;
```

---

## 六、恢复前要注意什么

恢复不是简单执行命令。

你要先确认：

- 恢复到哪里。
- 是否会覆盖现有数据。
- 当前库是否需要先清空。
- 应用是否还在写入。
- 备份文件是否完整。
- 恢复后如何验证。

生产环境恢复通常需要停写或切换流量。

学习阶段先练习恢复到新数据库。

---

## 七、只备份某个 schema

例如只备份 `app` schema：

```powershell
pg_dump -Fc -n app -f app_schema.dump $env:DATABASE_URL
```

恢复：

```powershell
pg_restore -d $env:RESTORE_DATABASE_URL app_schema.dump
```

`-n` 表示 schema。

适合一个数据库里有多个 schema，只想备份其中一部分的情况。

---

## 八、只备份某张表

例如只备份短链接表：

```powershell
pg_dump -Fc -t app.short_links -f short_links.dump $env:DATABASE_URL
```

恢复：

```powershell
pg_restore -d $env:RESTORE_DATABASE_URL short_links.dump
```

注意：

```text
只备份单表可能缺少依赖对象。
```

比如 schema、扩展、外键引用表等。

真实项目里要谨慎使用。

---

## 九、备份不等于高可用

备份解决的是：

```text
出问题后能恢复数据。
```

它不能解决：

```text
数据库宕机时业务不中断。
```

高可用通常需要：

- 主从复制。
- 自动故障切换。
- 连接切换。
- 监控告警。

本阶段先把备份恢复练熟。

---

## 十、恢复演练比备份命令更重要

很多团队以为自己有备份，直到出事才发现：

```text
备份文件损坏。
备份没有包含某些 schema。
恢复命令没人会用。
恢复时间太长。
恢复后数据不完整。
```

所以要定期做恢复演练。

最小演练流程：

```text
从生产或测试库生成备份。
恢复到一个临时数据库。
检查关键表数量。
检查关键业务查询。
记录恢复耗时。
删除临时数据库。
```

---

## 十一、备份文件安全

备份文件通常包含完整业务数据。

所以它本身也需要保护：

- 不要提交到 Git。
- 不要发到公共聊天群。
- 不要放在无权限控制的目录。
- 生产备份要加密存储。
- 备份访问权限要严格控制。

尤其是包含用户邮箱、手机号、订单、日志等数据时。

---

## 十二、短链接项目练习

基于第十阶段短链接服务，做一次完整练习：

1. 创建几条短链接。
2. 使用 `pg_dump -Fc` 备份数据库。
3. 创建 `learn_pg_restore_test`。
4. 使用 `pg_restore` 恢复。
5. 查询 `app.short_links` 行数。
6. 随机查询一条短码。
7. 删除恢复测试数据库。

验证 SQL：

```sql
SELECT count(*)
FROM app.short_links;

SELECT code, original_url, visit_count
FROM app.short_links
ORDER BY created_at DESC
LIMIT 5;
```

---

## 十三、本节达标标准

学完本节后，你应该能够做到：

- 区分逻辑备份和物理备份。
- 使用 `pg_dump` 备份数据库。
- 使用 `pg_restore` 恢复 custom 格式备份。
- 把备份恢复到新数据库中验证。
- 知道备份文件不能随便泄露。
- 明白“没有恢复演练的备份不可靠”。
- 能为短链接项目做一次完整备份恢复演练。
