# 7. 常用生产配置与参数直觉

本节目标：了解 PostgreSQL 常见生产参数的作用，建立配置直觉，而不是机械背参数。

PostgreSQL 有很多配置项。

后端开发者不需要一开始掌握所有参数，但至少要知道哪些配置会影响：

```text
连接数。
内存。
慢查询日志。
锁等待。
事务超时。
自动清理。
```

---

## 一、在哪里配置

PostgreSQL 参数可能来自：

- `postgresql.conf`。
- 启动命令。
- Docker 配置。
- 云数据库参数组。
- `ALTER SYSTEM`。
- 当前会话 `SET`。

查看参数：

```sql
SHOW max_connections;
SHOW shared_buffers;
SHOW log_min_duration_statement;
```

查看所有参数：

```sql
SELECT name, setting, unit, context
FROM pg_settings
ORDER BY name;
```

---

## 二、连接相关参数

### 1. `max_connections`

最大连接数：

```sql
SHOW max_connections;
```

它决定数据库最多允许多少客户端连接。

不要简单把它调得很大。

连接数越大，数据库资源压力越大。

更常见的做法是：

```text
合理配置应用连接池。
必要时使用 PgBouncer。
```

### 2. `superuser_reserved_connections`

为超级用户保留的连接数。

作用是：

```text
即使普通连接打满，也给管理员留少量连接排查问题。
```

不要让应用占满所有连接。

---

## 三、内存相关参数

### 1. `shared_buffers`

PostgreSQL 共享缓冲区。

它影响数据库缓存数据页的能力。

查看：

```sql
SHOW shared_buffers;
```

后端开发者不必一开始精确调它，但要知道：

```text
它和数据库内存使用密切相关。
```

### 2. `work_mem`

单个查询操作可使用的工作内存。

例如排序、哈希聚合可能用到。

查看：

```sql
SHOW work_mem;
```

注意：

```text
work_mem 不是全局总量。
它可能被每个连接、每个排序或哈希操作使用。
```

如果设置太大，高并发时总内存可能爆掉。

### 3. `maintenance_work_mem`

维护操作使用的内存，例如：

- `CREATE INDEX`
- `VACUUM`
- `ALTER TABLE ADD FOREIGN KEY`

查看：

```sql
SHOW maintenance_work_mem;
```

---

## 四、日志相关参数

### 1. `log_min_duration_statement`

慢查询日志阈值。

```sql
SHOW log_min_duration_statement;
```

例如：

```text
500ms
```

表示超过 500ms 的 SQL 会被记录。

如果是：

```text
-1
```

表示关闭。

学习环境可以设低一点，生产要结合日志量和业务延迟目标。

### 2. `log_statement`

控制记录哪些 SQL。

常见值：

```text
none
ddl
mod
all
```

生产不建议长期使用 `all`，日志量可能非常大。

### 3. `log_lock_waits`

是否记录锁等待。

查看：

```sql
SHOW log_lock_waits;
```

配合：

```sql
SHOW deadlock_timeout;
```

可以帮助发现锁等待问题。

---

## 五、超时相关参数

### 1. `statement_timeout`

单条 SQL 最大执行时间。

查看：

```sql
SHOW statement_timeout;
```

如果设置：

```sql
SET statement_timeout = '3s';
```

超过 3 秒的 SQL 会被取消。

应用也可以在连接级或事务级设置。

但注意：

```text
太短会误伤正常查询。
太长又不能有效保护系统。
```

### 2. `lock_timeout`

等待锁的最大时间。

```sql
SET lock_timeout = '2s';
```

适合生产迁移中使用。

例如执行 DDL 前设置：

```sql
SET lock_timeout = '3s';
ALTER TABLE app.short_links ADD COLUMN description text;
```

如果拿不到锁，就快速失败，避免一直卡住业务。

### 3. `idle_in_transaction_session_timeout`

空闲事务超时。

作用是：

```text
自动终止长时间 idle in transaction 的会话。
```

查看：

```sql
SHOW idle_in_transaction_session_timeout;
```

生产环境可以考虑设置，防止应用 bug 导致长事务长期挂着。

---

## 六、autovacuum 相关参数

查看：

```sql
SHOW autovacuum;
SHOW autovacuum_vacuum_scale_factor;
SHOW autovacuum_analyze_scale_factor;
```

通常：

```text
autovacuum 应该开启。
```

不要随便关闭。

高写入表可能需要针对单表调 autovacuum 参数，但这已经偏 DBA 调优。

后端开发者至少要知道：

```text
autovacuum 负责清理旧版本和更新统计信息。
长事务会影响它。
```

---

## 七、查看某个参数是否需要重启

查询：

```sql
SELECT name, setting, context
FROM pg_settings
WHERE name IN (
    'max_connections',
    'shared_buffers',
    'work_mem',
    'log_min_duration_statement',
    'statement_timeout'
);
```

`context` 表示修改生效范围。

有些参数需要重启，有些可以 reload，有些可以会话级设置。

生产修改参数前，一定要知道是否需要重启。

---

## 八、不要迷信参数调优

很多性能问题不是参数造成的，而是：

- 没有索引。
- SQL 写法不合理。
- 返回数据太多。
- 事务太长。
- 连接池太大。
- 磁盘 IO 不够。
- 表膨胀严重。

调参数前，先分析问题。

不要一遇到慢查询就改 `shared_buffers` 或 `work_mem`。

---

## 九、短链接项目推荐关注参数

短链接服务上线前，重点关注：

```text
max_connections
log_min_duration_statement
statement_timeout
lock_timeout
idle_in_transaction_session_timeout
autovacuum
```

应用层同时配置：

```text
pgxpool MaxConns
HTTP 请求超时
数据库查询 context 超时
```

这比盲目调大数据库参数更实用。

---

## 十、本节达标标准

学完本节后，你应该能够做到：

- 使用 `SHOW` 查看 PostgreSQL 参数。
- 使用 `pg_settings` 查看参数详情。
- 理解 `max_connections` 和连接池的关系。
- 知道 `log_min_duration_statement` 用于慢查询日志。
- 知道 `statement_timeout`、`lock_timeout`、`idle_in_transaction_session_timeout` 的作用。
- 知道 autovacuum 不应该随便关闭。
- 明白参数调优不能替代 SQL 和索引优化。
