# 1. 生产环境数据库直觉

本节目标：建立生产环境数据库的基本直觉，理解为什么很多本地可行的操作，到了生产环境就要谨慎。

学习阶段，你可以随时：

```sql
DROP TABLE users;
CREATE TABLE users (...);
```

但生产环境不能这么随意。

因为生产数据库承载的是正在运行的业务。

---

## 一、本地环境的特点

本地环境通常是：

```text
数据量小。
没有真实用户。
可以随时重建。
可以停服务。
错误成本低。
```

所以你在本地执行：

```sql
ALTER TABLE users ADD COLUMN bio text;
```

可能一瞬间完成。

但这不代表同样操作在生产也安全。

---

## 二、生产环境的特点

生产环境通常是：

```text
数据量大。
请求持续进入。
多个服务实例同时访问。
数据有业务价值。
操作出错可能影响用户。
恢复成本高。
```

生产数据库里，一条 SQL 可能影响：

- 接口响应时间。
- 用户下单。
- 支付状态。
- 库存扣减。
- 后台管理。
- 定时任务。
- 报表统计。

所以生产 SQL 不只是“能不能执行”，还要考虑：

```text
执行多久？
会锁什么？
会不会阻塞写入？
会不会扫描太多数据？
能不能回滚？
失败后数据会不会不一致？
```

---

## 三、生产环境最怕什么

### 1. 数据丢失

例如：

```sql
DELETE FROM orders;
```

忘记 `WHERE`，可能删除整张表。

如果没有备份和恢复方案，后果很严重。

### 2. 长时间锁表

某些 DDL 操作会持有较强的锁。

如果生产大表被锁住，业务接口可能全部卡住。

### 3. 慢查询拖垮数据库

一个没有索引的查询：

```sql
SELECT *
FROM orders
WHERE user_id = 123;
```

如果 `orders.user_id` 没有索引，数据量大时可能全表扫描。

多个慢查询并发执行，会占用 CPU、IO 和连接。

### 4. 连接数耗尽

如果应用连接池配置过大，多个实例一起启动，可能打满 PostgreSQL 最大连接数。

之后新请求会出现：

```text
too many connections
```

### 5. 磁盘空间耗尽

PostgreSQL 的 MVCC 会产生旧版本数据。

如果长期不清理、长事务阻塞 autovacuum，表和索引可能膨胀。

磁盘满了之后，数据库可能无法写入。

---

## 四、后端开发者要养成的生产问题意识

写 SQL 前多问几个问题：

### 1. 这条查询会不会扫全表？

尤其是：

```sql
WHERE email = $1
WHERE user_id = $1
ORDER BY created_at DESC
```

这些高频条件通常需要索引。

### 2. 这个事务会不会太长？

事务里不要做：

- 调用第三方 API。
- 等用户输入。
- 处理大文件。
- 长时间计算。

事务应该尽量短。

### 3. 这个迁移会不会锁表？

例如：

```sql
ALTER TABLE large_table ADD COLUMN new_col text NOT NULL DEFAULT 'x';
```

在某些场景下可能造成较重影响。

生产迁移要先评估表大小和锁风险。

### 4. 这个账号权限是不是太大？

应用账号不应该能：

```sql
DROP DATABASE
CREATE ROLE
ALTER SYSTEM
```

它只需要访问业务 schema 和表。

---

## 五、生产环境中的“可回滚”不是一句话

代码回滚通常比较容易：

```text
重新发布上一个版本。
```

数据库回滚更复杂。

比如你上线了一个迁移：

```sql
ALTER TABLE users DROP COLUMN phone;
```

如果生产已经执行，字段和数据都没了。

即使 down 迁移里写：

```sql
ALTER TABLE users ADD COLUMN phone text;
```

也只能把字段加回来，原来的数据回不来。

所以生产迁移要区分：

```text
结构可回滚。
数据是否可恢复。
业务是否兼容。
```

---

## 六、生产变更的基本顺序

一个相对稳妥的变更流程：

```text
本地验证。
测试环境验证。
评估生产数据量。
评估锁和执行时间。
备份或确认备份可用。
低峰期执行迁移。
观察监控和日志。
确认业务接口正常。
```

如果是高风险迁移，还要准备：

- 回滚方案。
- 只读窗口。
- 灰度发布。
- 分批数据回填。
- 紧急联系人。

---

## 七、生产问题排查的第一反应

当接口突然变慢，不要先盲目改代码。

先看：

```sql
SELECT
    pid,
    state,
    wait_event_type,
    wait_event,
    query,
    now() - query_start AS running_time
FROM pg_stat_activity
WHERE datname = current_database()
ORDER BY running_time DESC;
```

你可能会发现：

- 某条 SQL 跑了很久。
- 有事务 idle in transaction。
- 很多连接正在等待锁。
- 大量连接堆积。

这些信息比猜测有用得多。

---

## 八、不要把生产库当测试库

不要在生产随意执行：

```sql
SELECT * FROM huge_table;
UPDATE huge_table SET ...
DELETE FROM huge_table WHERE ...
EXPLAIN ANALYZE UPDATE ...
```

尤其是 `EXPLAIN ANALYZE`：

```text
它会真的执行 SQL。
```

如果要分析写操作，可以放在事务里回滚：

```sql
BEGIN;

EXPLAIN ANALYZE
UPDATE app.short_links
SET visit_count = visit_count + 1
WHERE code = 'abc123';

ROLLBACK;
```

但生产上仍然要谨慎。

---

## 九、本节达标标准

学完本节后，你应该能够做到：

- 说明本地数据库和生产数据库的差异。
- 知道生产环境最怕数据丢失、锁表、慢查询、连接耗尽和磁盘耗尽。
- 理解数据库回滚比代码回滚更复杂。
- 在写 SQL 和迁移时主动考虑锁、数据量和回滚。
- 知道接口变慢时可以先查 `pg_stat_activity`。
- 不把生产库当作随意试验的地方。
