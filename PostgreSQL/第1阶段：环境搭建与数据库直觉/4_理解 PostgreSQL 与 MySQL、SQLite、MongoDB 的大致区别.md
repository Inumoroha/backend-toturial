# 4. 理解 PostgreSQL 与 MySQL、SQLite、MongoDB 的大致区别

本节目标：了解 PostgreSQL、MySQL、SQLite、MongoDB 在数据模型、部署方式、适用场景和限制上的大致区别，建立选择数据库时的基本判断力。

学习 PostgreSQL 时，顺带理解它和其他常见数据库的差异，可以帮助你知道：为什么当前路线选择 PostgreSQL、什么时候 MySQL 更合适、SQLite 适合解决什么问题、MongoDB 又为什么不是“随便存 JSON”的代名词。

---

## 一、先区分两类数据库

常见数据库可以先粗略分成两类：

```text
关系型数据库：PostgreSQL、MySQL、SQLite
文档型数据库：MongoDB
```

关系型数据库通常使用：

- database
- schema 或类似概念
- table
- column
- row
- SQL
- 事务
- 约束
- JOIN

文档型数据库通常使用：

- database
- collection
- document
- 类 JSON 的数据结构

这不是说关系型数据库只能存表格数据，也不是说文档型数据库不需要设计数据结构。它们只是建模方式和适用场景不同。

---

## 二、PostgreSQL

PostgreSQL 是功能丰富、标准化程度较高、扩展能力很强的开源关系型数据库。

### 主要特点

- 支持标准 SQL。
- 支持事务和 ACID。
- 支持复杂查询、JOIN、窗口函数、CTE 等能力。
- 支持丰富的数据类型，例如 `jsonb`、`uuid`、数组、范围类型等。
- 支持强大的扩展生态，例如 PostGIS。
- 对数据类型、约束和一致性要求比较严格。

### 适合场景

- 需要复杂查询的业务系统。
- 对数据一致性要求较高的系统。
- Go、Java、Python 等后端服务的主数据库。
- 需要同时处理关系型数据和部分 JSON 数据的系统。
- 地理位置、统计分析、复杂报表等 PostgreSQL 擅长的场景。

### 不适合简单理解为

- PostgreSQL 不是所有场景都一定优于 MySQL。
- PostgreSQL 的 JSONB 很强，但不等于可以完全替代 MongoDB。
- PostgreSQL 功能丰富，但也需要学习成本和运维经验。

---

## 三、MySQL

MySQL 是使用非常广泛的开源关系型数据库，在 Web 应用和互联网业务中有很成熟的生态。

### 主要特点

- 使用广泛，资料、工具和经验非常多。
- 适合常规业务系统。
- InnoDB 存储引擎支持事务、行级锁、外键等能力。
- 在电商、博客、论坛、后台管理系统等场景中非常常见。
- 与 PHP、Java、Go 等技术栈都有成熟配套。

### 适合场景

- 常规 Web 后端。
- 电商、内容系统、管理后台。
- 团队已有 MySQL 经验的项目。
- 对生态、招聘、维护便利性要求较高的项目。

### 需要注意

- 现代 MySQL 并不是“不严谨”的数据库。
- 使用时应开启或理解严格模式。
- 表设计、索引、事务同样重要。
- 和 PostgreSQL 相比，某些高级 SQL 能力、扩展能力和类型系统侧重点不同。

---

## 四、SQLite

SQLite 是一个嵌入式关系型数据库。它没有独立的数据库服务器进程，数据库通常就是一个本地文件。

### 主要特点

- 零配置，不需要单独部署数据库服务。
- 数据库保存在本地文件中。
- 支持 SQL 和事务。
- 体积小，稳定可靠。
- 常见于移动端、桌面软件、浏览器、本地工具和测试环境。

### 适合场景

- 手机 App 本地存储。
- 桌面软件本地数据。
- 小型命令行工具。
- 单机应用。
- 自动化测试。
- 原型验证。

### 需要注意

- SQLite 不是玩具数据库，它非常可靠。
- 但它不适合当作高并发服务端主数据库。
- 它尤其不适合大量并发写入的场景。
- 如果你的系统需要多个后端服务共同访问同一个数据库，通常应该选择 PostgreSQL 或 MySQL。

---

## 五、MongoDB

MongoDB 是文档型数据库，使用类似 JSON 的文档来组织数据。它不使用关系型数据库中的 table、row、column 模型，而是使用 collection 和 document。

### 主要特点

- 数据以文档形式存储。
- 文档结构可以比较灵活。
- 适合嵌套数据和变化较快的数据结构。
- 查询方式不同于 SQL。
- 可以做索引、聚合和副本集。
- 支持 schema validation，但不是默认像关系型数据库那样强约束建模。

### 适合场景

- 数据结构变化较快的业务。
- 文档型数据，例如配置、事件、部分内容管理场景。
- 嵌套结构比较自然的数据。
- 不需要大量复杂 JOIN 的场景。

### 需要注意

- MongoDB 不是“不用设计表结构”的借口。
- 文档数据库同样需要数据模型设计。
- 过度嵌套、无限增长数组、缺少索引都会带来问题。
- 如果业务天然是强关系模型，关系型数据库通常更合适。

## 六、核心对比表

| 数据库 | 类型 | 是否独立服务 | 数据模型 | 查询语言 | 典型场景 | 不太适合的场景 |
| --- | --- | --- | --- | --- | --- | --- |
| PostgreSQL | 关系型数据库 | 是 | table / row / column | SQL | 复杂业务系统、强一致性、复杂查询、Go 后端主库 | 极简单本地小工具、完全无结构文档存储 |
| MySQL | 关系型数据库 | 是 | table / row / column | SQL | 常规 Web 后端、电商、博客、管理后台 | 需要 PostgreSQL 特定扩展能力的场景 |
| SQLite | 嵌入式关系型数据库 | 否 | table / row / column | SQL | 本地应用、移动端、测试、小型工具 | 高并发服务端、多实例共同写入 |
| MongoDB | 文档型数据库 | 是 | collection / document | MongoDB Query API | 文档数据、灵活结构、嵌套数据 | 强关系、多 JOIN、强约束关系建模 |

## 七、如何选择

### 选择 PostgreSQL，如果你需要：

- 一个功能丰富的主数据库。
- 标准 SQL 和复杂查询能力。
- 强事务和数据一致性。
- 丰富的数据类型和扩展能力。
- 在 Go 后端项目中长期使用。

### 选择 MySQL，如果你需要：

- 成熟、普及、资料多的关系型数据库。
- 常规 Web 业务主数据库。
- 团队已有 MySQL 经验。
- 项目生态或公司基础设施已经围绕 MySQL 建设。

### 选择 SQLite，如果你需要：

- 本地文件数据库。
- 零配置数据库。
- 移动端或桌面端本地存储。
- 测试环境或原型验证。

### 选择 MongoDB，如果你需要：

- 文档型数据模型。
- 灵活的文档结构。
- 嵌套数据存储。
- 不以复杂 JOIN 为核心的数据访问方式。

## 八、为什么本路线选择 PostgreSQL

这条学习路线选择 PostgreSQL，不是因为它在所有场景中都绝对最好，而是因为它适合培养后端开发者的数据库基本功。

原因包括：

- 它是关系型数据库，能训练 SQL、表设计、约束、事务和索引能力。
- 它对数据类型和约束比较严格，能帮助你养成更好的数据建模习惯。
- 它功能丰富，学完后能覆盖大量后端项目需求。
- 它和 Go 后端生态结合很好，`pgx` 是非常常用的 PostgreSQL 驱动。
- 它既能处理传统关系型数据，也能在合适场景下使用 JSONB 等特色能力。

先学 PostgreSQL，并不会妨碍以后学习 MySQL、SQLite 或 MongoDB。相反，关系型数据库基础扎实之后，再看其他数据库会更容易分清它们解决的问题。

## 九、常见误区

### 1. PostgreSQL 一定比 MySQL 好吗？

不是。

PostgreSQL 和 MySQL 都是成熟的关系型数据库。选择哪个，取决于业务场景、团队经验、已有基础设施和具体需求。

### 2. SQLite 是玩具数据库吗？

不是。

SQLite 非常稳定，也被广泛用于移动端、浏览器、桌面软件和本地工具。它只是不适合高并发服务端主数据库场景。

### 3. MongoDB 不需要设计数据结构吗？

不是。

MongoDB 的文档结构更灵活，但灵活不等于随意。真实项目中仍然需要设计 collection、document 结构、索引和数据增长方式。

### 4. PostgreSQL 的 JSONB 可以完全替代 MongoDB 吗？

不一定。

PostgreSQL 的 JSONB 很强，适合在关系型数据中保存部分灵活结构。但如果系统核心就是文档模型，MongoDB 仍然可能更自然。

### 5. 学 PostgreSQL 之后还需要学 MySQL 吗？

看目标。

如果你的目标是通用后端开发，未来了解 MySQL 很有价值。但学习顺序上，可以先把 PostgreSQL 的 SQL、表设计、索引和事务学扎实，再迁移到 MySQL。

## 十、本节达标标准

学完本节后，你应该能够做到：

- 区分关系型数据库和文档型数据库。
- 说出 PostgreSQL、MySQL、SQLite、MongoDB 的基本定位。
- 知道 PostgreSQL 和 MySQL 都是主流关系型数据库。
- 知道 SQLite 是嵌入式本地文件数据库，不适合高并发服务端主库。
- 知道 MongoDB 是文档型数据库，但仍然需要数据模型设计。
- 能根据简单场景判断大致应该选择哪类数据库。

掌握这些区别之后，就可以进入下一阶段：学习 SQL 基础与单表操作。
