# PostgreSQL 后续进阶学习方向

这份文档用于回答一个问题：

```text
学完当前 PostgreSQL + Go 后端路线后，后面还可以继续学什么？
```

前面 1 到 11 阶段已经覆盖了后端开发者使用 PostgreSQL 的主线能力：

- SQL 基础。
- 表设计与关系建模。
- JOIN、聚合、子查询。
- 事务、并发、锁、MVCC。
- 索引与执行计划。
- PostgreSQL 特色类型。
- Go + pgxpool 接入。
- 数据库迁移。
- 短链接项目落地。
- 生产实践、备份恢复、权限、监控、慢查询排查。

到这里，你已经具备了在普通 Go 后端项目中稳定使用 PostgreSQL 的能力。

后续进阶不建议继续“横向堆知识点”，而是按照真实项目需要，分方向深入。

---

## 一、进阶学习的总体思路

PostgreSQL 后续进阶可以分成五条线：

```text
性能优化线。
数据建模线。
生产运维线。
高可用与扩展线。
项目实战线。
```

建议不要五条线同时学。

更好的方式是：

```text
先补性能优化。
再做一个更复杂的业务项目。
项目中遇到问题，再学习对应专题。
```

因为数据库知识很容易变成“看过很多概念，但不知道什么时候用”。

进阶阶段最重要的是：

```text
带着业务问题学习。
用真实查询和真实数据验证。
```

---

## 二、性能优化进阶

适合目标：

```text
能独立排查慢 SQL。
能判断索引是否合理。
能读懂更复杂的执行计划。
```

### 需要继续学习

- 更深入理解 B-Tree 索引。
- 覆盖索引与 Index Only Scan。
- 表达式索引。
- 部分索引的生产使用。
- 多列索引顺序选择。
- 统计信息与查询优化器。
- `EXPLAIN (ANALYZE, BUFFERS)`。
- `pg_stat_statements`。
- 排序、Hash Join、Nested Loop、Merge Join。
- 大分页优化。
- 批量写入优化。

### 推荐练习

用一张订单表练习：

```text
orders
order_items
users
products
```

然后模拟：

- 按用户查询订单列表。
- 按状态查询待处理订单。
- 按时间范围统计销售额。
- 查询某个商品的订单明细。
- 分页查询历史订单。

对每条 SQL 使用：

```sql
EXPLAIN (ANALYZE, BUFFERS)
```

观察：

- 是否走索引。
- 是否发生排序。
- 是否读了大量 shared buffers。
- 是否返回行数和预估行数差距很大。

### 学到什么程度算够

能做到：

- 不靠猜测优化 SQL。
- 能解释为什么某个索引有效或无效。
- 能发现多列索引顺序问题。
- 能判断 Seq Scan 是否真的有问题。
- 能用 `pg_stat_statements` 找出最消耗数据库资源的 SQL。

## 三、复杂数据建模进阶

适合目标：

```text
能设计更复杂、更长期可维护的业务数据结构。
```

### 需要继续学习

- 订单系统建模。
- 库存系统建模。
- 支付流水建模。
- 审计日志表设计。
- 软删除与归档。
- 状态机字段设计。
- 多租户数据隔离。
- 历史版本表。
- 幂等表设计。
- Outbox Pattern。
- JSONB 与关系表的边界。

### 推荐项目

做一个小型电商后端：

```text
用户。
商品。
SKU。
库存。
购物车。
订单。
订单明细。
支付记录。
库存流水。
```

这个项目比短链接复杂得多。

它会逼你处理：

- 一对多。
- 多对多。
- 金额字段。
- 库存扣减。
- 事务边界。
- 幂等创建订单。
- 状态流转。
- 唯一约束。
- 外键是否使用。
- 历史记录和审计。

### 学到什么程度算够

能做到：

- 根据业务规则设计表，而不是只会 CRUD。
- 知道什么时候用约束保护数据。
- 知道什么时候用事务保护一致性。
- 能设计流水表而不是只覆盖当前值。
- 能解释为什么某些数据不能随便软删除。

---

## 四、事务与并发进阶

适合目标：

```text
能处理库存、余额、订单、幂等等高一致性场景。
```

### 需要继续学习

- 行级锁。
- `SELECT ... FOR UPDATE`。
- 乐观锁。
- 悲观锁。
- 幂等键。
- 唯一约束防重复提交。
- 死锁产生原因。
- 死锁重试策略。
- 隔离级别选择。
- `READ COMMITTED` 与 `SERIALIZABLE` 的差异。
- 事务重试。

### 推荐练习

实现库存扣减：

```text
创建商品库存。
多个并发请求同时下单。
库存不能扣成负数。
重复请求不能重复扣库存。
订单创建和库存扣减必须一致。
```

可以对比两种写法：

```sql
UPDATE products
SET stock = stock - 1
WHERE id = $1
  AND stock > 0;
```

以及：

```sql
SELECT stock
FROM products
WHERE id = $1
FOR UPDATE;
```

观察它们在并发下的表现。

### 学到什么程度算够

能做到：

- 知道什么时候需要显式事务。
- 能避免典型超卖问题。
- 能设计幂等请求。
- 知道死锁不是只能靠“避免”，也要能重试。
- 能解释为什么应用层先查再写不可靠。

## 五、生产运维进阶

适合目标：

```text
能参与生产数据库维护和故障排查。
```

### 需要继续学习

- 更系统的备份策略。
- 时间点恢复。
- WAL 基础。
- 复制槽。
- 慢查询长期分析。
- 表膨胀治理。
- autovacuum 参数调优。
- 大表归档。
- 分区表维护。
- 磁盘空间规划。
- 数据库升级流程。

### 推荐练习

做一次完整演练：

```text
创建测试库。
写入一批数据。
做备份。
模拟误删。
恢复到临时库。
把误删数据导回。
记录恢复步骤和耗时。
```

再做一次慢查询演练：

```text
构造百万级数据。
执行慢 SQL。
使用 pg_stat_activity 观察。
使用 EXPLAIN ANALYZE 分析。
增加索引。
对比优化前后。
```

### 学到什么程度算够

能做到：

- 不只是“有备份”，而是真的恢复过。
- 能判断数据库慢是 SQL、锁、连接数还是磁盘问题。
- 能看懂基本监控指标。
- 能在生产迁移前评估风险。
- 能写出数据库故障排查记录。

## 六、高可用与扩展进阶

适合目标：

```text
理解数据库在更大规模系统中的部署方式。
```

### 需要继续学习

- 主从复制。
- 流复制。
- 读写分离。
- 复制延迟。
- 故障切换。
- 连接路由。
- PgBouncer。
- 分区表。
- 水平分库分表的取舍。
- Citus 等扩展方案的基本概念。

### 注意点

不要太早学习分库分表。

很多项目在遇到真正的数据规模问题前，更应该先做好：

- 索引。
- SQL 优化。
- 连接池。
- 归档。
- 分区表。
- 缓存。
- 读写分离。

分库分表会显著增加复杂度。

它会影响：

- JOIN。
- 事务。
- 唯一约束。
- 分页。
- 数据迁移。
- 运维。
- 故障排查。

### 学到什么程度算够

初期只需要做到：

- 知道主从复制解决什么问题。
- 知道读写分离会有复制延迟。
- 知道 PgBouncer 解决连接数压力。
- 知道分区表适合时间序列或大表归档场景。
- 知道分库分表不是普通项目的默认选择。

## 七、PostgreSQL 高级特性进阶

适合目标：

```text
更充分发挥 PostgreSQL 的特色能力。
```

### 可以继续学习

- 分区表。
- 物化视图。
- 窗口函数。
- 递归 CTE。
- 更深入的全文搜索。
- PostGIS 地理空间扩展。
- `LISTEN/NOTIFY`。
- 触发器。
- 存储过程。
- 行级安全策略 RLS。
- 外部数据包装器 FDW。

### 使用建议

这些能力很强，但不要为了炫技而使用。

例如：

- 报表查询很重，可以考虑物化视图。
- 层级结构查询复杂，可以学习递归 CTE。
- 地理位置查询明显需要空间能力，可以学习 PostGIS。
- 多租户强隔离要求高，可以研究 RLS。
- 简单业务 CRUD 不需要上来就写触发器和存储过程。

### 学到什么程度算够

能做到：

- 知道每个高级特性适合什么场景。
- 知道它会带来什么维护成本。
- 能判断是放在数据库里做，还是放在应用层做。

## 九、推荐学习顺序

如果你已经学完当前 1 到 11 阶段，建议这样继续：

```text
第一步：把短链接服务真正编码跑通。
第二步：给短链接服务加访问日志和统计。
第三步：系统学习 EXPLAIN (ANALYZE, BUFFERS)。
第四步：做一个电商订单系统。
第五步：练习库存扣减、事务和幂等。
第六步：做一次备份恢复和故障演练。
第七步：学习分区表、物化视图等高级特性。
第八步：再看高可用、复制、读写分离。
```

不要跳过第一步。

教程看完和项目跑通之间，还有一段非常重要的距离。

## 十、进阶阶段的学习原则

### 1. 不要只看概念

每学一个概念，都要配一个 SQL 或项目场景。

### 2. 不要盲目加索引

索引必须服务于查询。

加索引前后都要看执行计划。

### 3. 不要过早复杂化

能用普通表解决的问题，不要一开始就分库分表。

能用清晰事务解决的问题，不要一开始就引入复杂中间件。

### 4. 不要忽视恢复演练

备份不恢复，等于只完成了一半。

### 5. 不要把数据库当纯存储

PostgreSQL 不只是保存数据。

它还提供：

- 约束。
- 事务。
- 索引。
- 查询优化。
- 类型系统。
- 并发控制。

用好这些能力，后端系统会更稳。

---

## 十一、最终目标

PostgreSQL 进阶学习的最终目标不是记住所有语法。

而是做到：

```text
能设计可靠的数据结构。
能写清晰且高效的查询。
能处理事务和并发。
能分析慢 SQL。
能安全发布数据库变更。
能在生产出问题时有排查方向。
能知道什么时候该用 PostgreSQL 的高级能力，什么时候不该用。
```

当你能做到这些，PostgreSQL 就不再只是一个数据库软件，而是你构建后端系统时非常可靠的工程工具。
