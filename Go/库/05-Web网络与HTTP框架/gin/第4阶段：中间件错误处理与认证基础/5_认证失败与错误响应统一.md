# 5. 认证失败与错误响应统一

本节目标：让认证失败、权限失败、token 格式错误、token 过期等场景都返回统一 JSON 响应，而不是散落不同格式。

认证中间件通常会在 handler 之前拦截请求，所以它也必须遵守项目统一响应规范。

---

## 一、认证失败有哪些场景

常见认证失败：

```text
没有 Authorization Header。
Authorization 格式不是 Bearer token。
token 为空。
token 签名错误。
token 已过期。
token 解析失败。
```

常见授权失败：

```text
普通用户访问管理员接口。
用户访问不属于自己的资源。
账号被禁用。
```

认证失败通常返回：

```text
401 Unauthorized
```

授权失败通常返回：

```text
403 Forbidden
```

---

## 二、统一响应格式

缺少 token：

```json
{
  "code": 40101,
  "message": "missing token"
}
```

token 格式错误：

```json
{
  "code": 40101,
  "message": "invalid token format"
}
```

token 无效：

```json
{
  "code": 40101,
  "message": "invalid token"
}
```

无权限：

```json
{
  "code": 40301,
  "message": "forbidden"
}
```

---

## 三、为什么不要返回 JWT 内部错误

不要这样：

```go
response.Fail(c, http.StatusUnauthorized, response.CodeUnauthorized, err.Error())
```

JWT 库返回的错误可能包含内部细节，例如：

```text
token is expired
signature is invalid
token contains an invalid number of segments
```

学习阶段看这些错误有帮助，但对外 API 不一定要暴露得这么细。

推荐对外统一：

```text
invalid token
```

详细错误可以写入日志。

---

## 四、封装认证失败函数

可以在 middleware 中写小工具函数：

```go
func abortUnauthorized(c *gin.Context, message string) {
    response.Fail(c, http.StatusUnauthorized, response.CodeUnauthorized, message)
    c.Abort()
}

func abortForbidden(c *gin.Context, message string) {
    response.Fail(c, http.StatusForbidden, response.CodeForbidden, message)
    c.Abort()
}
```

使用：

```go
if authHeader == "" {
    abortUnauthorized(c, "missing token")
    return
}
```

这样认证中间件更清爽。

## 五、完整 Auth 中间件示例

```go
func Auth(secret string) gin.HandlerFunc {
    return func(c *gin.Context) {
        authHeader := c.GetHeader("Authorization")
        if authHeader == "" {
            abortUnauthorized(c, "missing token")
            return
        }

        parts := strings.SplitN(authHeader, " ", 2)
        if len(parts) != 2 || parts[0] != "Bearer" || parts[1] == "" {
            abortUnauthorized(c, "invalid token format")
            return
        }

        claims, err := service.ParseToken(parts[1], secret)
        if err != nil {
            log.Printf("parse token failed: %v", err)
            abortUnauthorized(c, "invalid token")
            return
        }

        c.Set("user_id", claims.UserID)
        c.Set("email", claims.Email)
        c.Set("role", claims.Role)

        c.Next()
    }
}
```

注意：

```go
log.Printf("parse token failed: %v", err)
```

详细错误写日志，而不是返回给客户端。

## 六、RequireRole 统一响应

```go
func RequireRole(role string) gin.HandlerFunc {
    return func(c *gin.Context) {
        currentRole := c.GetString("role")
        if currentRole != role {
            abortForbidden(c, "forbidden")
            return
        }

        c.Next()
    }
}
```

访问管理员接口时，如果当前用户不是 admin，返回：

```text
HTTP 403
```

响应：

```json
{
  "code": 40301,
  "message": "forbidden"
}
```

---

## 七、认证中间件中的 return 很重要

错误示例：

```go
if authHeader == "" {
    abortUnauthorized(c, "missing token")
}

claims, err := service.ParseToken(...)
```

即使 `abortUnauthorized` 调用了 `c.Abort()`，当前函数后续代码仍然可能继续执行。

正确：

```go
if authHeader == "" {
    abortUnauthorized(c, "missing token")
    return
}
```

认证中间件里，失败后一定要 `return`。

---

## 八、测试场景

至少测试：

```text
不带 Authorization。
Authorization: token
Authorization: Bearer
Authorization: Bearer wrong-token
Authorization: Bearer correct-token
普通用户访问 admin 接口。
```

这些场景能覆盖大部分认证失败分支。

---

## 九、常见问题

### 1. token 错误返回了 500

认证失败不是服务内部错误，应返回 401。

### 2. 权限不足返回了 401

如果用户已经登录，只是角色不够，应返回 403。

### 3. 响应格式不统一

检查认证中间件是否使用了 `response.Fail`，而不是 `c.JSON` 自己拼格式。
