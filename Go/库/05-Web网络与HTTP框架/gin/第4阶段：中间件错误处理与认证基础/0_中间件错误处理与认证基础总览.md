# 0. 中间件、错误处理与认证基础总览

本阶段目标：理解 Gin 中间件的执行流程，并掌握真实后端项目中最常见的横切能力：请求日志、错误恢复、CORS、Request ID、JWT 认证、认证失败统一响应和基础角色授权。

如果前三个阶段解决的是：

```text
如何设计路由、读取参数、校验请求、返回统一响应。
```

那么第四阶段要解决的是：

```text
如何把每个接口都需要的公共逻辑抽到中间件里。
```

中间件是 Gin 非常核心的能力。你以后写登录认证、权限控制、接口限流、请求日志、panic 恢复、跨域处理，几乎都会和中间件有关。

---

## 一、中间件适合处理什么

中间件适合处理“很多接口都需要”的逻辑。

常见场景：

```text
请求日志
请求耗时统计
Request ID
panic recovery
CORS 跨域
JWT 认证
角色权限
接口限流
统一错误处理
```

这些逻辑如果写在每个 handler 里，会导致大量重复代码。

例如每个私有接口都写：

```go
token := c.GetHeader("Authorization")
if token == "" {
    response.Fail(c, 401, 40101, "missing token")
    return
}
```

会非常啰嗦，也很容易漏掉。

更好的方式是把它放进认证中间件：

```go
private := api.Group("")
private.Use(middleware.Auth(secret))
{
    private.GET("/profile", userHandler.Profile)
    private.GET("/tasks", taskHandler.List)
}
```

## 三、本阶段推荐学习顺序

建议按下面顺序学习：

```text
中间件执行流程
→ CORS、日志与 Recovery
→ JWT 认证与角色授权
→ 请求耗时与 Request ID
→ 认证失败统一响应
→ 综合实践：保护 Profile 接口
→ 接口测试请求集合
```

原因是：

- 先理解执行流程，再写具体中间件。
- 先写简单中间件，再写认证中间件。
- 先理解认证，再讨论授权。
- 最后用 Profile 接口把整条链路跑通。

## 四、本阶段统一练习目标

本阶段最终完成一个最小认证闭环：

```text
POST /api/v1/auth/register
POST /api/v1/auth/login
GET  /api/v1/profile
GET  /api/v1/admin/ping
```

要求：

- 注册时密码使用 bcrypt 加密。
- 登录成功返回 JWT。
- `/profile` 必须携带 token。
- token 无效或过期返回统一 401。
- `/admin/ping` 只有 admin 角色可以访问。
- 每个请求都带 `X-Request-ID` 响应头。
- 请求日志能看到 method、path、status、latency、request_id。

完成本阶段后，你应该能够做到：

如果这些都能做到，你就已经掌握 Gin 项目里最常用的一类能力。
