# 7. 接口测试请求集合

本节目标：提供第4阶段认证、中间件和权限相关接口的测试请求。你可以复制到 `requests.http` 中逐个验证。

---

## 一、基础地址

```text
http://localhost:8080
```

API 前缀：

```text
/api/v1
```

---

## 二、公开接口测试

### 注册用户

```http
### register user
POST http://localhost:8080/api/v1/auth/register
Content-Type: application/json

{
  "username": "alice",
  "email": "alice@example.com",
  "password": "12345678"
}
```

期望：

```text
HTTP 201
code = 0
响应中没有 password_hash
```

### 重复注册

```http
### register duplicate email
POST http://localhost:8080/api/v1/auth/register
Content-Type: application/json

{
  "username": "alice2",
  "email": "alice@example.com",
  "password": "12345678"
}
```

期望：

```text
HTTP 409
code = 40901
```

### 注册参数错误

```http
### register invalid password
POST http://localhost:8080/api/v1/auth/register
Content-Type: application/json

{
  "username": "a",
  "email": "not-email",
  "password": "123"
}
```

期望：

```text
HTTP 400
code = 40001
```

---

## 三、登录接口测试

### 登录成功

```http
### login success
POST http://localhost:8080/api/v1/auth/login
Content-Type: application/json

{
  "email": "alice@example.com",
  "password": "12345678"
}
```

期望：

```text
HTTP 200
code = 0
data.token 存在
```

复制响应里的 token，替换后面请求中的 `{{token}}`。

### 登录失败

```http
### login wrong password
POST http://localhost:8080/api/v1/auth/login
Content-Type: application/json

{
  "email": "alice@example.com",
  "password": "wrong-password"
}
```

期望：

```text
HTTP 401
code = 40101
message = invalid email or password
```

## 四、Profile 认证测试

### 不带 token

```http
### profile missing token
GET http://localhost:8080/api/v1/profile
```

期望：

```text
HTTP 401
code = 40101
message = missing token
```

### token 格式错误

```http
### profile invalid token format
GET http://localhost:8080/api/v1/profile
Authorization: token
```

期望：

```text
HTTP 401
code = 40101
message = invalid token format
```

### token 错误

```http
### profile invalid token
GET http://localhost:8080/api/v1/profile
Authorization: Bearer wrong-token
```

期望：

```text
HTTP 401
code = 40101
message = invalid token
```

### token 正确

```http
### profile success
GET http://localhost:8080/api/v1/profile
Authorization: Bearer {{token}}
```

期望：

```text
HTTP 200
code = 0
返回当前用户信息
```

## 五、管理员接口测试

### 普通用户访问 admin

```http
### admin ping forbidden
GET http://localhost:8080/api/v1/admin/ping
Authorization: Bearer {{user_token}}
```

期望：

```text
HTTP 403
code = 40301
message = forbidden
```

### admin 用户访问 admin

```http
### admin ping success
GET http://localhost:8080/api/v1/admin/ping
Authorization: Bearer {{admin_token}}
```

期望：

```text
HTTP 200
code = 0
message = admin pong
```

学习阶段如果还没有后台修改角色接口，可以临时在代码里创建一个 admin 用户用于测试。

## 六、Request ID 测试

### 服务端生成 request id

```http
### request id generated
GET http://localhost:8080/api/v1/profile
Authorization: Bearer {{token}}
```

检查响应头：

```text
X-Request-ID: ...
```

### 客户端传入 request id

```http
### request id from client
GET http://localhost:8080/api/v1/profile
Authorization: Bearer {{token}}
X-Request-ID: demo-request-id
```

期望响应头：

```text
X-Request-ID: demo-request-id
```

---

## 七、推荐测试顺序

建议按下面顺序：

1. 不带 token 访问 profile。
2. 注册用户。
3. 重复注册。
4. 错误密码登录。
5. 正确密码登录并复制 token。
6. token 格式错误访问 profile。
7. token 错误访问 profile。
8. token 正确访问 profile。
9. 普通用户访问 admin。
10. 验证 `X-Request-ID` 响应头。

这个顺序能覆盖本阶段大部分关键路径。

## 九、本节验收标准

完成本请求集合后，请确认：

- [ ] 不带 token 访问受保护接口返回 `401`。
- [ ] token 格式错误返回 `401`。
- [ ] token 无效返回 `401`。
- [ ] 正确 token 可以访问 profile。
- [ ] 普通用户访问 admin 返回 `403`。
- [ ] admin 用户可以访问 admin。
- [ ] 响应头包含 `X-Request-ID`。
- [ ] 客户端传入 `X-Request-ID` 时服务端能原样返回。
- [ ] 所有认证失败响应都使用统一 JSON。

这些请求全部通过后，第4阶段的 JWT、中间件执行顺序、认证失败响应和 RequestID 才算闭环。
