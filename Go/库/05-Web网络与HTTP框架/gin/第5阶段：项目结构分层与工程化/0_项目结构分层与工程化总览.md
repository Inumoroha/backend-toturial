# 0. 项目结构、分层与工程化总览

本阶段目标：把前面写在 `main.go` 里的 Gin 代码，逐步重构成更接近真实后端项目的结构。完成本阶段后，你应该能说清楚 handler、service、repository、model、router、response、config 各自负责什么，也能独立把一个简单 CRUD 模块拆成分层代码。

前三个阶段里，为了把 Gin 的核心概念讲清楚，很多代码可以先写在一个文件里：

```text
main.go
```

这对入门很友好，因为你能一眼看到完整流程。但真实项目不能一直这样写。

当接口变多后，单文件代码会出现这些问题：

- handler 函数越来越长。
- 参数处理、业务逻辑、数据访问混在一起。
- 重复响应代码越来越多。
- 想写测试时很难隔离依赖。
- 想换数据库或存储方式时影响范围很大。
- 新人接手项目时找不到代码边界。

本阶段要解决的问题就是：

```text
让代码按职责分层，而不是按“写着方便”堆在一起。
```

---

## 一、什么叫分层

分层不是为了让目录看起来复杂，而是为了让每一部分代码职责清楚。

推荐理解：

```text
handler
负责 HTTP 世界：读取请求，返回响应。

service
负责业务世界：业务规则、默认值、状态变化。

repository
负责数据世界：保存、查询、更新、删除数据。

model
负责数据结构：用户、任务、文章等核心对象。

router
负责路由注册：哪个路径交给哪个 handler。

response
负责响应格式：成功、失败、分页、错误码。

config
负责配置加载：端口、数据库地址、JWT secret。
```

一个请求进入系统后，大致流向是：

```text
router -> handler -> service -> repository -> database
```

响应返回时再反向回来：

```text
database -> repository -> service -> handler -> client
```

---

## 二、分层后每层不该做什么

理解“不该做什么”比背目录更重要。

### handler 不该做什么

handler 不应该：

- 直接写复杂业务规则。
- 直接写数据库查询。
- 直接生成密码 hash。
- 直接拼很多重复响应结构。
- 直接处理事务。

handler 应该保持薄一点：

```text
取参数 -> 调 service -> 根据结果返回响应
```

### service 不该做什么

service 不应该：

- 依赖 `*gin.Context`。
- 直接返回 HTTP 状态码。
- 直接调用 `c.JSON`。
- 关心请求路径和 Header。

service 应该和 HTTP 框架解耦。这样以后即使你不用 Gin，业务逻辑也能复用。

### repository 不该做什么

repository 不应该：

- 处理 HTTP。
- 返回 `400`、`404` 这种状态码。
- 决定业务提示文案。
- 做复杂业务决策。

repository 只负责数据访问，并把数据访问错误转换成上层能理解的错误，例如：

```go
var ErrUserNotFound = errors.New("user not found")
```

## 三、本阶段适合什么时候学习

建议在你完成下面内容后学习本阶段：

- 能写 Gin 路由。
- 能读取路径参数、查询参数、JSON 请求体。
- 能做 binding 校验。
- 能使用统一响应。
- 能完成内存版用户 CRUD。

如果你还不能独立写出下面接口：

```text
GET    /api/v1/users
POST   /api/v1/users
GET    /api/v1/users/:id
PUT    /api/v1/users/:id
DELETE /api/v1/users/:id
```

建议先回到第2、第3阶段继续练习。

## 四、本阶段统一练习目标

本阶段继续使用用户模块，把它从单文件重构成分层项目。

最终结构类似：

```text
gin-layer-demo/
  cmd/
    server/
      main.go
  internal/
    handler/
      user_handler.go
    service/
      user_service.go
    repository/
      user_repository.go
    model/
      user.go
    response/
      response.go
      code.go
    router/
      router.go
```

本阶段仍然可以使用内存 map 作为数据存储。数据库会在第6阶段展开。

这样做的好处是：

```text
先把分层思想学清楚，再替换 repository 的内部实现。
```

也就是说，repository 现在可以用 map，后面可以改为 GORM，但 handler 和 service 的结构不需要大改。

## 六、本阶段完成标准

完成本阶段后，你应该能够做到：

- 搭建一个清晰的 Gin 项目目录。
- 解释 handler、service、repository 的职责边界。
- 使用构造函数注入依赖。
- 在 `main.go` 中完成对象组装。
- 把请求 DTO 和数据库/业务 model 分开。
- 把重复响应逻辑放到 response 包。
- 把路由注册放到 router 包。
- 把内存 map 操作封装到 repository。
- 让 handler 不直接操作 map 或数据库。
- 让 service 不依赖 Gin。

如果你能做到这些，后续接入 GORM、JWT、Redis、测试时就会自然很多。
