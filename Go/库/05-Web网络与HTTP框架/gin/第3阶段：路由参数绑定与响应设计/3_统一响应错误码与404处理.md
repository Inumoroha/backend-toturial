# 3. 统一响应、错误码与 404 处理

本节目标：为 Gin 项目建立统一响应格式，设计稳定的业务错误码，并让 404、405 等框架级错误也返回一致的 JSON 结构。

上一节解决的是“请求进来时如何校验”。本节解决的是“请求处理完后如何返回”。

---

## 一、为什么要统一响应

如果每个接口随意返回不同结构，调用方会很难处理。

例如接口 A 返回：

```json
{
  "message": "ok",
  "user": {}
}
```

接口 B 返回：

```json
{
  "data": {},
  "success": true
}
```

接口 C 返回：

```json
{
  "error": "user not found"
}
```

前端每接一个接口都要写特殊判断。后端自己维护起来也会越来越乱。

统一响应的目标是：

```text
成功响应长得一样。
失败响应长得一样。
分页响应长得一样。
错误码稳定可判断。
```

---

## 二、推荐响应结构

创建：

```text
internal/response/response.go
```

定义：

```go
package response

type Body struct {
    Code    int         `json:"code"`
    Message string      `json:"message"`
    Data    interface{} `json:"data,omitempty"`
}
```

成功响应：

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

失败响应：

```json
{
  "code": 40401,
  "message": "user not found"
}
```

`omitempty` 的作用是：当 `Data` 为空时，不输出这个字段。

---

## 三、封装成功响应

```go
package response

import (
    "net/http"

    "github.com/gin-gonic/gin"
)

func OK(c *gin.Context, data interface{}) {
    c.JSON(http.StatusOK, Body{
        Code:    CodeOK,
        Message: "ok",
        Data:    data,
    })
}

func Created(c *gin.Context, data interface{}) {
    c.JSON(http.StatusCreated, Body{
        Code:    CodeOK,
        Message: "ok",
        Data:    data,
    })
}
```

查询成功使用：

```go
response.OK(c, user)
```

创建成功使用：

```go
response.Created(c, user)
```

---

## 四、封装失败响应

```go
func Fail(c *gin.Context, httpStatus int, code int, message string) {
    c.JSON(httpStatus, Body{
        Code:    code,
        Message: message,
    })
}

func FailWithData(c *gin.Context, httpStatus int, code int, message string, data interface{}) {
    c.JSON(httpStatus, Body{
        Code:    code,
        Message: message,
        Data:    data,
    })
}
```

参数错误：

```go
response.Fail(c, http.StatusBadRequest, response.CodeInvalidRequest, "invalid request")
```

资源不存在：

```go
response.Fail(c, http.StatusNotFound, response.CodeNotFound, "user not found")
```

带字段级错误：

```go
response.FailWithData(c, http.StatusBadRequest, response.CodeInvalidRequest, "invalid request", fields)
```

## 五、错误码常量

创建：

```text
internal/response/code.go
```

内容：

```go
package response

const (
    CodeOK              = 0
    CodeInvalidRequest  = 40001
    CodeUnauthorized    = 40101
    CodeForbidden       = 40301
    CodeNotFound        = 40401
    CodeConflict        = 40901
    CodeTooManyRequests = 42901
    CodeInternalError   = 50001
)
```

错误码设计不必一开始非常复杂，但必须稳定。

稳定的意思是：

```text
同一种错误，每次都返回同一个 code。
```

不要今天用户不存在返回 `40401`，明天又返回 `10003`。

## 六、HTTP 状态码和业务错误码的关系

HTTP 状态码表示协议层结果。

业务错误码表示业务层原因。

例如邮箱已存在：

```text
HTTP 状态码：409 Conflict
业务错误码：40901
```

响应：

```json
{
  "code": 40901,
  "message": "email already exists"
}
```

为什么不只用业务码？

因为很多通用工具、网关、日志系统、监控系统会根据 HTTP 状态码判断请求结果。

为什么不只用 HTTP 状态码？

因为一个 `400` 可能代表很多业务错误，前端需要更细的 code 做提示和处理。

## 七、分页响应结构

列表接口建议统一分页格式：

```go
type PageResult struct {
    Items    interface{} `json:"items"`
    Total    int64       `json:"total"`
    Page     int         `json:"page"`
    PageSize int         `json:"page_size"`
}
```

响应示例：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "items": [],
    "total": 100,
    "page": 1,
    "page_size": 10
  }
}
```

字段解释：

- `items`：当前页数据。
- `total`：总数据量。
- `page`：当前页。
- `page_size`：每页数量。

## 八、统一 404

如果访问不存在的路由，Gin 默认会返回自己的 404 文本。为了保持统一，需要自定义：

```go
r.NoRoute(func(c *gin.Context) {
    response.Fail(c, http.StatusNotFound, response.CodeNotFound, "route not found")
})
```

测试：

```http
GET http://localhost:8080/api/v1/not-exists
```

期望：

```json
{
  "code": 40401,
  "message": "route not found"
}
```

---

## 九、统一 405

405 表示路径存在，但 HTTP 方法不允许。

例如只注册了：

```go
api.GET("/users", listUsers)
```

但客户端请求：

```http
POST /api/v1/users
```

如果没有注册 POST，就应该返回 405。

Gin 中启用：

```go
r.HandleMethodNotAllowed = true
```

自定义：

```go
r.NoMethod(func(c *gin.Context) {
    response.Fail(c, http.StatusMethodNotAllowed, response.CodeInvalidRequest, "method not allowed")
})
```

## 十、内部错误不要直接暴露

不要这样写：

```go
if err != nil {
    c.JSON(500, gin.H{"message": err.Error()})
    return
}
```

原因：

- 可能暴露数据库表名。
- 可能暴露 SQL。
- 可能暴露文件路径。
- 可能暴露第三方服务错误细节。

对外建议：

```go
response.Fail(c, http.StatusInternalServerError, response.CodeInternalError, "internal server error")
```

详细错误写日志。日志阶段会进一步展开。

## 十一、handler 中的使用示例

```go
func getUser(c *gin.Context) {
    id, err := strconv.ParseUint(c.Param("id"), 10, 64)
    if err != nil {
        response.Fail(c, http.StatusBadRequest, response.CodeInvalidRequest, "invalid user id")
        return
    }

    user, ok := users[uint(id)]
    if !ok {
        response.Fail(c, http.StatusNotFound, response.CodeNotFound, "user not found")
        return
    }

    response.OK(c, user)
}
```

这个 handler 的结构很清楚：

```text
参数错误 -> 400
资源不存在 -> 404
成功 -> 200
```

## 十二、常见问题

### 1. 所有业务错误都返回 200 可以吗

不推荐。

虽然有些公司内部系统会这么做，但对新手学习和通用 API 设计来说，不利于日志、监控、网关和客户端判断。

### 2. 业务错误码要不要非常细

入门阶段不用太细。

先按大类稳定下来：

```text
40001 参数错误
40101 未登录
40401 不存在
40901 冲突
50001 内部错误
```

以后项目变大，再拆更细。

### 3. 错误 message 用中文还是英文

教程中使用英文，因为 API 通常面向程序调用。真实项目可以根据团队约定决定。
