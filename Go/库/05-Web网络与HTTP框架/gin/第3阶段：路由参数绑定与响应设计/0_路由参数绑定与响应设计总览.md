# 0. 路由、参数绑定与响应设计总览

本阶段目标：在已经能用 Gin 写出基础 CRUD 的前提下，把接口入口做规范。你要学会设计清晰的 RESTful 路由，明确每一种参数应该从哪里读取，使用 binding 做请求校验，并让所有接口返回稳定、统一的 JSON 响应。

如果第二阶段解决的是：

```text
我能用 Gin 写出接口。
```

那么第三阶段要解决的是：

```text
我能写出更像真实项目的接口。
```

真实项目里的接口不能只追求“能跑”。它还应该具备：

- 路径含义清楚。
- 请求参数来源明确。
- 参数错误能被稳定识别。
- 响应格式统一。
- HTTP 状态码使用合理。
- 业务错误码稳定。
- 前端和调用方容易处理。

---

## 一、本阶段适合什么时候学习

开始本阶段前，建议你已经完成第二阶段的内存版用户 CRUD，并且能独立写出下面这些接口：

```text
GET    /api/v1/users
GET    /api/v1/users/:id
POST   /api/v1/users
PUT    /api/v1/users/:id
DELETE /api/v1/users/:id
```

你还应该知道：

- `c.Param` 用来读取路径参数。
- `c.Query` 用来读取查询参数。
- `c.ShouldBindJSON` 用来读取 JSON 请求体。
- `c.JSON` 用来返回 JSON 响应。

如果这些还不熟，建议先回到第二阶段继续练习。

---

## 二、本阶段核心问题

本阶段围绕 6 个问题展开。

### 1. 路由怎么设计

例如用户资源应该写成：

```text
GET /api/v1/users
POST /api/v1/users
GET /api/v1/users/:id
```

而不是：

```text
GET /getUsers
POST /createUser
POST /deleteUser
```

路由设计会影响后续项目结构、接口文档和前后端协作。

### 2. 参数从哪里来

常见参数来源：

```text
路径参数：/users/:id
查询参数：?page=1&page_size=10
JSON 请求体：POST /users 的 body
Header：Authorization、Content-Type
```

你需要明确每类参数适合什么场景。

### 3. 参数怎么校验

创建用户时，不能允许这种请求进入业务层：

```json
{
  "username": "",
  "email": "abc",
  "age": 200
}
```

Gin 的 binding 能帮你在 handler 入口处拒绝明显非法的请求。

### 4. 错误怎么返回

不要一会儿返回：

```json
{"error":"bad request"}
```

一会儿返回：

```json
{"message":"user not found"}
```

应该统一成稳定格式。

### 5. 状态码怎么选

不同错误应该有不同 HTTP 状态码：

```text
400 参数错误
401 未登录
403 无权限
404 资源不存在
409 资源冲突
500 服务内部错误
```

如果所有错误都返回 200 或 500，调用方会非常痛苦。

### 6. 本阶段如何验收

你最终要把用户 API 改造成：

- RESTful 路由。
- 参数校验完整。
- 响应格式统一。
- 错误码稳定。
- 列表支持分页筛选。
- 404 和 405 也返回统一格式。

## 四、本阶段推荐学习顺序

建议按下面顺序学习：

```text
RESTful 路由设计
→ binding 校验与字段级错误
→ 统一响应、错误码与 404 处理
→ ShouldBindQuery 分页筛选实践
→ 状态码选择实践
→ 综合实践：规范用户 API
```

这个顺序是有原因的：

- 路由是接口的外形。
- 参数绑定是接口入口。
- 响应格式是接口出口。
- 分页筛选是列表接口高频需求。
- 状态码是调用方判断结果的基础。
- 综合实践把前面内容整合成一个可运行模块。

## 六、本阶段完成标准

完成本阶段后，你应该能够做到：

- 看到业务需求后，能设计出合理的 RESTful 路由。
- 能解释路径参数、查询参数、JSON 请求体和 Header 的边界。
- 能使用 `ShouldBindJSON` 校验创建和更新请求。
- 能使用 `ShouldBindQuery` 校验分页和筛选参数。
- 能把校验错误转换成统一响应。
- 能为 400、401、403、404、409、500 选择合适场景。
- 能让所有成功和失败响应保持统一结构。

如果这些都能做到，你写出来的 Gin 接口就已经开始接近真实团队项目的风格。
