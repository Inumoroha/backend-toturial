# 7. 本阶段综合实践：容器化部署 Gin 项目

本节目标：把 Gin 项目完整容器化，使用 Docker Compose 启动 app、MySQL、Redis、Nginx，并完成上线前检查。

这一节是第9阶段收口。你要把前面写过的 Dockerfile、Compose、Nginx、环境变量、健康检查和排错流程整合起来。

---

## 一、最终交付物

项目根目录应包含：

```text
Dockerfile
.dockerignore
docker-compose.yml
.env.example
nginx.conf
README.md
```

应用中应包含：

```text
/healthz
/readyz
/ping
/swagger/index.html
```

如果你已经实现任务管理系统或用户系统，则业务接口也应能通过 Nginx 访问。

---

## 二、第一步：本地测试

在容器化之前先运行：

```bash
go mod tidy
go test ./...
go build -o bin/server ./cmd/server
```

这一步的目标是确认代码本身没有问题。

如果这里失败，不要进入 Docker 阶段。先修代码。

---

## 三、第二步：准备环境变量

复制：

```bash
cp .env.example .env
```

Windows PowerShell：

```powershell
Copy-Item .env.example .env
```

修改 `.env`：

```dotenv
MYSQL_ROOT_PASSWORD=change-me
MYSQL_DATABASE=gin_user_api
APP_DATABASE_DSN=root:change-me@tcp(mysql:3306)/gin_user_api?charset=utf8mb4&parseTime=True&loc=Local
APP_JWT_SECRET=please-change-this-secret
```

确认 `.gitignore` 包含：

```gitignore
.env
.env.*
!.env.example
```

---

## 四、第三步：构建并启动

执行：

```bash
docker compose up -d --build
```

查看：

```bash
docker compose ps
```

期望看到：

```text
mysql   running / healthy
redis   running / healthy
app     running
nginx   running
```

如果 app 没起来，马上看日志：

```bash
docker compose logs app
```

---

## 五、第四步：验证基础接口

通过 app 端口：

```bash
curl http://localhost:8080/healthz
curl http://localhost:8080/readyz
```

通过 Nginx：

```bash
curl http://localhost/healthz
curl http://localhost/readyz
curl http://localhost/ping
```

预期：

- `/healthz` 返回 200。
- `/readyz` 在数据库和 Redis 正常时返回 200。
- Nginx 能代理到 app。

---

## 六、第五步：验证业务接口

以登录为例：

```bash
curl -X POST http://localhost/api/v1/login \
  -H "Content-Type: application/json" \
  -d '{"email":"a@example.com","password":"123456"}'
```

如果需要先注册：

```bash
curl -X POST http://localhost/api/v1/register \
  -H "Content-Type: application/json" \
  -d '{"email":"a@example.com","password":"123456","nickname":"alice"}'
```

继续验证：

- 用户详情缓存。
- 登录失败限流。
- 头像上传。
- Swagger 页面。

---

## 七、第六步：验证数据持久化

停止服务：

```bash
docker compose down
```

重新启动：

```bash
docker compose up -d
```

检查：

- MySQL 数据是否还在。
- Redis 是否按预期保留或清空。
- 上传文件是否还在。

如果使用 named volume，数据通常会保留。如果执行：

```bash
docker compose down -v
```

volume 会被删除，数据会丢失。这个命令要谨慎使用。

---

## 八、第七步：生产上线前检查

上线前逐项确认：

- [ ] Gin mode 为 release。
- [ ] JWT Secret 已更换为强随机值。
- [ ] 数据库密码不是默认值。
- [ ] `.env` 未提交 Git。
- [ ] app 不直接暴露公网，只通过 Nginx 或网关访问。
- [ ] `/readyz` 能检查数据库和 Redis。
- [ ] 上传目录已持久化。
- [ ] 日志输出到 stdout/stderr。
- [ ] Swagger 是否需要关闭或加保护已明确。
- [ ] README 有部署、启动、停止、查看日志命令。

## 十、常见最终问题

### 1. 本地可以，Docker 不行

优先检查配置和网络。Docker 中依赖地址通常要用服务名。

### 2. Docker 可以，Nginx 不行

优先检查 Nginx upstream、app 是否运行、端口是否正确。

### 3. 重启后数据没了

检查是否使用 volume，以及是否执行过 `docker compose down -v`。

### 4. 上传大文件失败

同时检查 Nginx `client_max_body_size` 和 Gin handler 的大小限制。

## 十一、本阶段最终验收

请按顺序完成：

```bash
go test ./...
docker compose up -d --build
docker compose ps
curl http://localhost/healthz
curl http://localhost/readyz
curl http://localhost/ping
docker compose logs app
```

确认：

- 测试通过。
- 容器全部启动。
- 健康检查正常。
- Nginx 能代理请求。
- 业务接口可访问。
- 日志可查看。
- 数据和上传文件有持久化策略。

完成第9阶段后，你已经走完一个 Gin 后端项目从本地开发到容器化部署的完整路径。后续项目实战阶段就应该把这些能力自然地用到真实业务里，而不是只停留在单点知识。
