# 0. 部署上线与生产检查总览

第9阶段的目标是把 Gin 项目从“本地能跑”推进到“可以部署、可以检查、可以排错”的状态。

很多后端学习项目停在本地运行：

```bash
go run ./cmd/server
```

这还不够。真正的后端工程师需要知道服务如何容器化、如何配置环境变量、如何和数据库 Redis 一起启动、如何接入 Nginx、如何做健康检查、出现问题时如何定位。

---

## 一、本阶段学习目标

完成本阶段后，你应该能做到：

- 编写适合 Go 程序的多阶段 Dockerfile。
- 使用 Docker Compose 启动 Gin、MySQL、Redis。
- 使用环境变量覆盖生产配置。
- 添加健康检查接口。
- 使用 Nginx 反向代理 Gin 服务。
- 理解容器中 `localhost` 的含义。
- 排查常见部署问题。
- 写出生产上线前检查清单。

---

## 二、为什么部署阶段必须单独学习

代码写得好，不代表能顺利部署。部署阶段会遇到完全不同的问题：

- 本地数据库地址写的是 `127.0.0.1`，容器里却连不上。
- Docker 镜像太大。
- 配置文件带了真实密码。
- 服务启动了，但健康检查失败。
- Nginx 返回 502。
- 静态上传文件容器重启后丢失。
- 日志只在容器里，看不到请求错误。

这些问题不是 Gin 语法问题，而是工程交付问题。

建议顺序：

1. 先把单个 Gin 程序打成镜像。
2. 再用 Compose 启动依赖。
3. 然后处理环境变量和健康检查。
4. 最后接入 Nginx 并做完整排错。

## 四、本阶段推荐最终结构

项目根目录：

```text
gin-user-api/
├── cmd/
│   └── server/
├── config/
├── internal/
├── docs/
├── uploads/
├── Dockerfile
├── docker-compose.yml
├── .dockerignore
├── .env.example
├── nginx.conf
├── Makefile
└── README.md
```

这些文件各自职责：

- `Dockerfile`：描述如何构建应用镜像。
- `.dockerignore`：排除不需要进入镜像构建上下文的文件。
- `docker-compose.yml`：编排应用、数据库、Redis、Nginx。
- `.env.example`：说明需要哪些环境变量。
- `nginx.conf`：反向代理配置。
- `README.md`：说明部署和排错步骤。

## 五、部署阶段的基本原则

### 1. 配置和代码分离

不要把生产密码写进代码或镜像。

### 2. 容器只负责运行一个主进程

Gin 应用容器只运行 Gin 服务，不要顺便启动 MySQL 或 Redis。

### 3. 镜像尽量小

Go 项目适合多阶段构建，把编译环境和运行环境分开。

### 4. 健康检查必须明确

服务能启动不代表服务健康。健康检查要能反映依赖状态。

### 5. 日志输出到 stdout/stderr

容器环境下，日志优先输出到标准输出，由容器平台采集。

---

## 六、本阶段最终验收

完成本阶段后，请确认：

- `docker build` 可以成功构建应用镜像。
- `docker compose up` 可以启动 app、mysql、redis。
- app 可以通过服务名连接 mysql 和 redis。
- `/healthz` 返回服务基础状态。
- `/readyz` 能检查数据库和 Redis。
- Nginx 可以代理访问 Gin 接口。
- 上传目录使用 volume 持久化。
- `.env.example` 存在，但真实 `.env` 不提交。
- README 中有部署和排错命令。

第9阶段完成后，你的 Gin 学习项目就具备了完整后端工程闭环：开发、配置、日志、测试、缓存、限流、上传、部署和排错。
