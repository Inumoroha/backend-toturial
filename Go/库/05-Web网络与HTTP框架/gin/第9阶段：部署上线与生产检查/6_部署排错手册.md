# 6. 部署排错手册

本节目标：整理 Gin 项目容器化部署中最常见的问题，并给出稳定的排查顺序。

部署排错最怕没有顺序。看到接口访问失败就乱改配置，往往越改越乱。正确做法是先判断问题在哪一层，再查看对应日志和配置。

---

## 一、排错总顺序

推荐按这个顺序：

```text
1. 容器是否运行
2. 应用日志是否有启动错误
3. 配置是否正确注入
4. 服务之间网络是否可达
5. 数据库/Redis 是否健康
6. Nginx 是否能代理到 app
7. 应用接口自身是否报错
```

常用命令：

```bash
docker compose ps
docker compose logs app
docker compose logs mysql
docker compose logs redis
docker compose logs nginx
docker compose config
```

---

## 二、问题一：容器启动失败

现象：

```bash
docker compose ps
```

app 状态是 `Exited`。

排查：

```bash
docker compose logs app
```

常见原因：

- 配置文件路径错误。
- 数据库连接失败。
- Redis 连接失败。
- JWT Secret 缺失。
- 端口监听失败。

处理：

- 看清第一条 fatal/error 日志。
- 不要只看最后一行。
- 确认配置路径和环境变量。

---

## 三、问题二：app 连不上 MySQL

典型日志：

```text
dial tcp 127.0.0.1:3306: connect: connection refused
```

原因：容器里写了 `127.0.0.1:3306`。

正确 DSN：

```text
root:password@tcp(mysql:3306)/gin_user_api?charset=utf8mb4&parseTime=True&loc=Local
```

排查：

```bash
docker compose ps mysql
docker compose logs mysql
docker compose exec mysql mysqladmin ping -h localhost -uroot -p
docker compose config
```

重点检查：

- MySQL 是否 healthy。
- `.env` 中 DSN 是否使用 `mysql:3306`。
- 数据库名是否存在。
- 密码是否一致。

---

## 四、问题三：app 连不上 Redis

典型日志：

```text
dial tcp 127.0.0.1:6379: connect: connection refused
```

正确配置：

```text
APP_REDIS_ADDR=redis:6379
```

排查：

```bash
docker compose ps redis
docker compose exec redis redis-cli ping
docker compose logs redis
```

如果 Redis 返回：

```text
PONG
```

说明 Redis 容器本身正常，继续检查 app 环境变量。

---

## 五、问题四：Nginx 返回 502

502 表示 Nginx 找不到或连不上上游 app。

排查：

```bash
docker compose ps app nginx
docker compose logs nginx
docker compose logs app
```

检查 `nginx.conf`：

```nginx
upstream gin_app {
    server app:8080;
}
```

检查 app 是否监听：

```text
:8080
```

不要只监听：

```text
127.0.0.1:8080
```

---

## 六、问题五：接口返回 404

先判断 404 来自 Nginx 还是 Gin。

查看响应头或日志：

```bash
curl -i http://localhost/api/v1/users
docker compose logs nginx
docker compose logs app
```

如果 app 没有收到请求，问题在 Nginx 路由。

如果 app 收到请求但返回 404，问题在 Gin 路由注册：

- BasePath 是否是 `/api/v1`。
- HTTP 方法是否正确。
- 路径是否多写或少写斜杠。

---

## 七、问题六：环境变量没生效

排查：

```bash
docker compose config
docker compose exec app env | grep APP_
```

检查：

- `.env` 是否在 Compose 文件同级目录。
- Compose 中是否引用变量。
- 变量名是否和 Viper 映射一致。
- 修改 `.env` 后是否重启容器。

重启：

```bash
docker compose up -d --force-recreate app
```

---

## 八、问题七：上传文件重启后丢失

原因：文件保存在容器内部，没有挂载 volume。

Compose 中应有：

```yaml
volumes:
  - uploads_data:/app/uploads
```

检查：

```bash
docker compose exec app ls -R /app/uploads
docker volume ls
```

生产中更推荐对象存储，例如 S3、OSS、COS 等。

---

## 九、问题八：Swagger 页面访问不到

排查：

- 是否生成 docs。
- 是否匿名导入 docs。
- 是否注册 `/swagger/*any`。
- Nginx 是否代理了该路径。

命令：

```bash
swag init -g cmd/server/main.go -o docs
curl http://localhost/swagger/index.html
```

如果生产环境不希望暴露 Swagger，可以通过配置关闭或加认证。

---

## 十、问题九：健康检查失败

先访问：

```bash
curl http://localhost/healthz
curl http://localhost/readyz
```

如果 `/healthz` 正常、`/readyz` 失败，说明进程活着，但依赖异常。

继续查：

```bash
docker compose logs mysql
docker compose logs redis
docker compose logs app
```

## 十一、排错记录模板

建议每次排错写下来：

```md
## 问题现象

访问 /api/v1/login 返回 502。

## 影响范围

所有经 Nginx 的接口不可用。

## 排查过程

1. docker compose ps 发现 app exited。
2. docker compose logs app 发现数据库连接失败。
3. docker compose config 发现 DSN 写成 127.0.0.1。

## 根因

容器内错误使用 localhost 连接 MySQL。

## 修复

改为 mysql:3306 并重启 app。

## 复盘

README 增加容器网络说明。
```

这会训练你的工程排错能力。

---

## 十二、验收标准

完成本节后，你应该能：

- 看懂 `docker compose ps`。
- 用 logs 定位 app 启动失败原因。
- 解释容器中的 localhost 问题。
- 排查 Nginx 502。
- 排查环境变量未生效。
- 排查上传文件丢失。
