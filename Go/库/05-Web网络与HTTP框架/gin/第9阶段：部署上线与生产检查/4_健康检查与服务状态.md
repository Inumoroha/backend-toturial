# 4. 健康检查与服务状态

本节目标：为 Gin 项目添加健康检查接口，让部署平台和开发者能判断服务是否存活、是否准备好接收流量。

服务能启动，不等于服务健康。比如 Gin 进程还在，但数据库断了、Redis 断了，这时接口可能已经无法正常工作。

---

## 一、两类常见健康检查

推荐区分：

```text
/healthz：进程是否存活
/readyz：服务是否准备好处理请求
```

区别：

- `/healthz`：只要应用进程正常，就返回成功。
- `/readyz`：需要检查数据库、Redis 等依赖。

这样部署系统可以做不同判断：

- 进程挂了，重启容器。
- 依赖没准备好，暂时不转发流量。

---

## 二、实现 healthz

```go
r.GET("/healthz", func(c *gin.Context) {
    c.JSON(http.StatusOK, gin.H{
        "status": "ok",
    })
})
```

验证：

```bash
curl http://localhost:8080/healthz
```

预期：

```json
{"status":"ok"}
```

这个接口不要做复杂检查。它越简单，越适合作为进程存活判断。

## 三、实现 readyz

依赖检查函数：

```go
type HealthHandler struct {
    db  *gorm.DB
    rdb *redis.Client
}

func NewHealthHandler(db *gorm.DB, rdb *redis.Client) *HealthHandler {
    return &HealthHandler{db: db, rdb: rdb}
}

func (h *HealthHandler) Readyz(c *gin.Context) {
    ctx, cancel := context.WithTimeout(c.Request.Context(), 2*time.Second)
    defer cancel()

    checks := make(map[string]string)
    status := http.StatusOK

    sqlDB, err := h.db.DB()
    if err != nil || sqlDB.PingContext(ctx) != nil {
        checks["database"] = "down"
        status = http.StatusServiceUnavailable
    } else {
        checks["database"] = "up"
    }

    if err := h.rdb.Ping(ctx).Err(); err != nil {
        checks["redis"] = "down"
        status = http.StatusServiceUnavailable
    } else {
        checks["redis"] = "up"
    }

    c.JSON(status, gin.H{
        "status": checks,
    })
}
```

注册：

```go
healthHandler := handler.NewHealthHandler(db, redisClient)
r.GET("/readyz", healthHandler.Readyz)
```

## 四、readyz 响应示例

全部正常：

```json
{
  "status": {
    "database": "up",
    "redis": "up"
  }
}
```

Redis 异常：

```json
{
  "status": {
    "database": "up",
    "redis": "down"
  }
}
```

状态码应为：

```text
503 Service Unavailable
```

## 五、Compose 中配置 healthcheck

app 服务中增加：

```yaml
healthcheck:
  test: ["CMD", "wget", "-q", "-O", "-", "http://localhost:8080/healthz"]
  interval: 10s
  timeout: 3s
  retries: 5
```

如果运行镜像中没有 `wget`，可以：

- 在运行镜像中安装 curl 或 wget。
- 使用更适合的基础镜像。
- 由外部平台做 HTTP 健康检查。

学习阶段先理解 healthcheck 的意义，不必纠结某一个命令。

## 六、Nginx 代理健康检查

通过 Nginx 访问：

```bash
curl http://localhost/healthz
curl http://localhost/readyz
```

如果 Nginx 能访问，而直接 app 端口不能访问，说明外部入口是 Nginx。

如果 Nginx 返回 502，优先查 app 是否存活：

```bash
docker compose ps app
docker compose logs app
```

---

## 七、健康检查不要做什么

不要在健康检查里做这些事：

- 写数据库。
- 执行业务复杂查询。
- 调用慢外部 API。
- 做需要认证的逻辑。
- 返回大量敏感配置。

健康检查应该快、稳定、信息适度。

---

## 八、常见问题

### 1. `/healthz` 正常但业务接口失败

这可能是数据库或 Redis 不可用。继续检查 `/readyz`。

### 2. `/readyz` 太慢

给检查设置短超时，例如 2 秒。健康检查不能拖垮服务。

### 3. 要不要把详细错误返回给外部

生产环境不建议直接暴露完整错误。可以返回 up/down，详细错误写日志。

## 十、验收标准

完成后确认：

- `/healthz` 始终能快速返回。
- `/readyz` 能反映数据库和 Redis 状态。
- 依赖异常时 `/readyz` 返回 `503`。
- 健康检查不泄露敏感配置。
- Compose 能显示 app 健康状态。
