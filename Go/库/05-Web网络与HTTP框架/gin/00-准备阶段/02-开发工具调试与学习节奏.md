# 00-02 开发工具、调试流程与学习节奏

本节目标：把学习 Gin 时每天会用到的工具流程固定下来。很多初学者不是卡在框架本身，而是卡在“服务怎么启动、接口怎么测、错误怎么看、代码怎么回滚”这些基础流程上。

## 一、推荐工作流

每天学习时按这个顺序来：

1. 打开当前阶段教程。
2. 创建或切换到对应代码目录。
3. 运行服务，确认上一节代码仍然可用。
4. 写一个小功能。
5. 用接口工具测试。
6. 记录错误和解决方式。
7. 提交一次 Git 记录。

不要等一大章全部写完才运行。后端学习最怕“攒一堆代码一起报错”，那时你很难判断问题从哪里开始。

## 二、Git 基础建议

初始化：

```bash
git init
git status
```

每完成一个小阶段提交一次：

```bash
git add .
git commit -m "learn basic gin routing"
```

推荐提交粒度：

- 完成标准库 HTTP 服务。
- 完成 Gin CRUD。
- 完成统一响应。
- 完成项目分层。
- 接入数据库。
- 完成 JWT。
- 完成 Docker 部署。

这样后面代码坏掉时，你可以对比差异，而不是靠记忆找错。

## 三、接口测试文件

如果你使用 VS Code REST Client，可以在项目里创建：

```text
requests.http
```

示例：

```http
### ping
GET http://localhost:8080/ping

### create user
POST http://localhost:8080/api/v1/users
Content-Type: application/json

{
  "username": "alice",
  "email": "alice@example.com",
  "age": 20
}
```

这样接口请求可以和代码一起保存，复习时非常方便。

## 四、如何看错误

遇到错误时不要只看最后一行。后端错误通常要按顺序看：

1. HTTP 状态码是多少。
2. 响应体里业务错误码是什么。
3. 控制台日志有没有报错。
4. 报错发生在哪个文件和行号。
5. 请求参数是否真的按预期发出。
6. 数据库或 Redis 是否正常启动。

常见例子：

```text
400 Bad Request
```

通常先检查：

- JSON 是否合法。
- `Content-Type` 是否是 `application/json`。
- binding 校验是否失败。

```text
500 Internal Server Error
```

通常检查：

- 数据库连接。
- 空指针。
- SQL 错误。
- 依赖没有初始化。

## 五、日志观察方法

开发阶段建议每次启动服务都盯住终端。

Gin 默认日志类似：

```text
[GIN] 2026/07/05 - 18:00:00 | 200 | 1.2ms | 127.0.0.1 | GET "/ping"
```

你至少要能看懂：

- 请求方法：GET、POST。
- 请求路径。
- 状态码。
- 耗时。
- 客户端地址。

如果你请求了 `/api/v1/users`，但日志里没有任何输出，说明请求根本没打到服务，先检查端口和地址。

## 六、调试顺序

接口不通时按这个顺序查：

1. 服务是否启动。
2. 端口是否正确。
3. 路由路径是否正确。
4. HTTP 方法是否正确。
5. 请求头是否正确。
6. 请求体是否正确。
7. handler 是否进入。
8. service 是否返回错误。
9. repository 是否查到数据。

不要一上来就怀疑框架。大部分问题都是路径、方法、JSON、配置或数据库连接。

## 七、学习节奏建议

每一节都按“三遍法”学习：

第一遍：照着敲，跑通即可。

第二遍：不看教程，重新写一遍核心代码。

第三遍：加一个小需求，例如新增字段、增加筛选、调整错误码。

能完成第三遍，才说明你不是只复制了代码。

## 八、本节练习

完成：

- 初始化 Git。
- 创建 `requests.http`。
- 写 3 个接口请求。
- 故意写错一个 JSON，观察 400 错误。
- 故意访问不存在的路由，观察 404 错误。
- 把错误现象和解决方式写进学习笔记。

## 九、验收清单

你应该能够回答：

- 为什么每完成一个阶段要提交 Git？
- 接口测试文件有什么好处？
- 400、404、500 分别常见于什么问题？
- Gin 默认日志里能看到哪些信息？
- 一个接口不通时应该按什么顺序排查？
