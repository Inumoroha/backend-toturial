# 6. 数据库排错清单

本节目标：建立 GORM 接入数据库后的排错顺序。数据库问题很常见，但大多数问题都可以按固定步骤排查，不需要盲目猜。

---

## 一、先判断是哪一类问题

数据库相关问题大致分为：

```text
连接失败
认证失败
数据库不存在
表不存在
字段不匹配
查询不到数据
唯一索引冲突
软删除导致数据不可见
Docker 网络地址错误
```

排错时不要一上来就改代码。先看错误信息属于哪一类。

---

## 二、连接失败

常见错误：

```text
connection refused
```

排查：

1. MySQL 是否启动。
2. 端口是否正确。
3. Docker 端口是否映射。
4. DSN 中 host 是否正确。
5. 防火墙或安全软件是否拦截。

Docker 检查：

```bash
docker ps
docker logs gin-mysql
```

---

## 三、认证失败

错误：

```text
Access denied for user 'root'
```

排查：

- 用户名是否正确。
- 密码是否正确。
- Docker 启动时设置的 `MYSQL_ROOT_PASSWORD` 是否和 DSN 一致。
- 是否连接到了另一个 MySQL 实例。

---

## 四、数据库不存在

错误：

```text
Unknown database 'gin_learning'
```

解决：

```sql
CREATE DATABASE gin_learning DEFAULT CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
```

或者 Docker 启动时设置：

```text
MYSQL_DATABASE=gin_learning
```

---

## 五、表不存在

错误：

```text
Table 'gin_learning.users' doesn't exist
```

排查：

- 是否执行 `AutoMigrate`。
- 是否连接到正确数据库。
- 模型是否传给了 `AutoMigrate`。

代码：

```go
if err := db.AutoMigrate(&model.User{}); err != nil {
    log.Fatal(err)
}
```

---

## 六、字段不匹配

常见原因：

- 修改了模型，但没有迁移数据库。
- 手动建表字段名和 GORM 默认字段名不一致。
- 字段类型不兼容。

学习阶段可以重新建表或使用 AutoMigrate。

生产环境不要随意删表，要使用迁移工具。

---

## 七、查询不到数据

排查：

1. 是否连接错数据库。
2. 表里是否真的有数据。
3. 查询条件是否正确。
4. 数据是否被软删除。
5. 是否有租户/用户 ID 条件。

软删除检查：

```go
db.Unscoped().First(&user, id)
```

如果 `Unscoped` 能查到，说明数据被软删除了。

---

## 八、唯一索引冲突

错误：

```text
Duplicate entry
```

说明唯一字段重复，例如 email。

正确处理：

```text
repository 转换为 ErrEmailAlreadyExists。
handler 返回 409。
```

不要把它作为 500 暴露给用户。

---

## 九、Docker Compose 中地址错误

本地运行 Go：

```text
127.0.0.1:3306
```

Go 也在容器里运行：

```text
mysql:3306
```

如果 app 容器中使用 `127.0.0.1:3306`，它会连接 app 容器自己，而不是 MySQL 容器。

---

## 十、时间字段问题

DSN 建议：

```text
parseTime=True&loc=Local
```

如果没有 `parseTime=True`，`time.Time` 字段可能解析异常。

---

## 十一、GORM 日志排错

遇到 SQL 行为不符合预期时，先打开 GORM 日志。

开发环境可以这样配置：

```go
db, err := gorm.Open(mysql.Open(dsn), &gorm.Config{
    Logger: logger.Default.LogMode(logger.Info),
})
```

这样你能看到实际执行的 SQL：

```sql
SELECT * FROM `users` WHERE `users`.`id` = 1 AND `users`.`deleted_at` IS NULL ORDER BY `users`.`id` LIMIT 1
```

通过 SQL 可以判断：

- 查询的是不是正确表。
- WHERE 条件是否正确。
- 是否带了软删除条件。
- LIMIT、OFFSET 是否符合分页预期。
- 更新语句有没有真的更新字段。

生产环境不要长期开启过于详细的 SQL 日志，否则可能影响性能并暴露敏感数据。

---

## 十二、零值更新没生效

GORM 使用 struct 更新时，默认不会更新零值字段。

例如：

```go
db.Model(&user).Updates(model.User{
    Age:    0,
    Status: "",
})
```

`0` 和空字符串可能不会被更新。

如果你确实要更新零值，可以使用 map：

```go
db.Model(&model.User{}).
    Where("id = ?", userID).
    Updates(map[string]interface{}{
        "age":    0,
        "status": "",
    })
```

或者明确选择字段：

```go
db.Model(&user).Select("age", "status").Updates(model.User{
    Age:    0,
    Status: "",
})
```

自查问题：

- 是否使用 struct 更新零值。
- 是否应该改用 map。
- 是否需要 `Select` 指定字段。
- handler DTO 是否能区分“不传字段”和“传了零值”。

---

## 十三、First、Find、Take 的区别

常见查询：

```go
db.First(&user, id)
db.Take(&user, "email = ?", email)
db.Find(&users)
```

区别：

```text
First：按主键或排序取第一条，查不到返回 ErrRecordNotFound。
Take：不指定排序取一条，查不到返回 ErrRecordNotFound。
Find：查询多条，查不到通常不返回 ErrRecordNotFound，而是空切片。
```

列表接口不要把空列表当错误：

```go
var users []model.User
err := db.Where("status = ?", status).Find(&users).Error
```

即使没有数据，也应该返回：

```json
{
  "items": [],
  "total": 0
}
```

而不是 `404`。

---

## 十四、事务没有回滚

事务适合这些场景：

```text
创建订单 + 扣库存
创建用户 + 创建用户资料
转账扣款 + 入账
写主表 + 写明细表
```

推荐写法：

```go
err := db.Transaction(func(tx *gorm.DB) error {
    if err := tx.Create(&user).Error; err != nil {
        return err
    }

    if err := tx.Create(&profile).Error; err != nil {
        return err
    }

    return nil
})
```

只要函数返回非 nil 错误，GORM 会回滚。

常见错误：

```go
db.Transaction(func(tx *gorm.DB) error {
    tx.Create(&user)
    tx.Create(&profile)
    return nil
})
```

这里没有检查 `Error`，即使创建失败，也可能返回 nil，事务不会按预期回滚。

自查问题：

- 事务中的每一步是否检查错误。
- 是否把 `tx` 传给 repository，而不是继续用外层 `db`。
- 是否在事务中吞掉错误。
- 是否把业务错误返回给 Transaction 函数。

---

## 十五、唯一索引错误转换

数据库唯一索引是最后一道防线。即使 service 先查了邮箱是否存在，也可能在并发情况下发生重复插入。

repository 应该把数据库错误转换为业务可理解的错误：

```go
var ErrEmailExists = errors.New("email already exists")
```

MySQL 重复键常见错误码是 `1062`。可以判断：

```go
var mysqlErr *mysql.MySQLError
if errors.As(err, &mysqlErr) && mysqlErr.Number == 1062 {
    return ErrEmailExists
}
```

handler 最终返回：

```text
HTTP 409
code 40901
```

不要把重复邮箱当成 `500`。

---

## 十六、分页 total 不准确

常见错误：

```go
db.Offset(offset).Limit(pageSize).Find(&users)
db.Model(&model.User{}).Count(&total)
```

如果列表有筛选条件，`Count` 也必须带同样条件。

推荐：

```go
query := db.Model(&model.User{})

if status != "" {
    query = query.Where("status = ?", status)
}

if keyword != "" {
    query = query.Where("username LIKE ?", "%"+keyword+"%")
}

if err := query.Count(&total).Error; err != nil {
    return nil, 0, err
}

if err := query.Offset(offset).Limit(pageSize).Find(&users).Error; err != nil {
    return nil, 0, err
}
```

自查问题：

- `Count` 和 `Find` 是否使用同一组筛选条件。
- page 是否从 1 开始。
- offset 是否计算为 `(page - 1) * pageSize`。
- page_size 是否限制最大值。

---

## 十七、AutoMigrate 的边界

学习阶段可以使用：

```go
db.AutoMigrate(&model.User{})
```

它适合快速创建表和新增字段。

但你要知道它的边界：

- 不适合管理复杂生产变更。
- 不会安全处理所有字段删除。
- 不等价于完整迁移系统。
- 多人协作时不如迁移文件可追踪。

生产项目更推荐使用迁移工具，例如 golang-migrate、goose、atlas 等。

学习阶段验收重点是：

```text
模型能创建表。
字段和 tag 符合预期。
索引和唯一约束生效。
不要误以为 AutoMigrate 能解决所有数据库演进问题。
```

---

## 十八、排错顺序模板

遇到数据库问题时，按这个顺序：

```text
1. 看完整错误信息。
2. 确认 MySQL 是否启动。
3. 确认 DSN。
4. 确认数据库是否存在。
5. 确认表是否存在。
6. 确认数据是否存在。
7. 确认查询条件。
8. 确认软删除。
9. 确认唯一索引或其他约束。
10. 打开 GORM SQL 日志确认实际语句。
11. 检查零值更新、事务和分页 total。
```

## 十九、本节练习

故意制造并记录这些错误：

1. 写错密码。
2. 写错数据库名。
3. 注释掉 AutoMigrate。
4. 重复创建同一邮箱。
5. 删除用户后再查询。
6. Docker Compose 中把 host 写成 `127.0.0.1`。
7. 使用 struct 把年龄更新为 0，观察是否生效。
8. 在事务中故意让第二步失败，确认第一步是否回滚。
9. 给列表接口加 status 筛选，检查 total 是否准确。

每个错误记录：

```text
现象
错误信息
原因
解决方法
```
