# 5. 分页、排序、筛选实践

本节目标：使用 GORM 实现真实列表接口常见的分页、排序和筛选。完成后，你应该能把第3阶段的内存分页改成数据库分页。

---

## 一、目标接口

```http
GET /api/v1/users?page=1&page_size=10&status=active&keyword=alice
```

响应：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "items": [],
    "total": 100,
    "page": 1,
    "page_size": 10
  }
}
```

---

## 二、分页参数默认值

默认值建议在 service 中处理：

```go
func normalizePage(page, pageSize int) (int, int) {
    if page <= 0 {
        page = 1
    }
    if pageSize <= 0 {
        pageSize = 10
    }
    if pageSize > 100 {
        pageSize = 100
    }
    return page, pageSize
}
```

handler 负责校验请求，service 负责业务默认值。

---

## 三、offset 计算

```go
offset := (page - 1) * pageSize
```

示例：

```text
page=1,page_size=10 -> offset=0
page=2,page_size=10 -> offset=10
page=3,page_size=10 -> offset=20
```

---

## 四、基础分页

```go
var users []model.User
err := db.Order("id DESC").
    Offset(offset).
    Limit(pageSize).
    Find(&users).Error
```

一定要有稳定排序。

如果没有 `Order`，数据库返回顺序不应该被依赖。

---

## 五、查 total

分页接口通常要返回总数：

```go
var total int64
err := db.Model(&model.User{}).Count(&total).Error
```

为什么需要 total？

- 前端计算总页数。
- 前端显示“共 N 条”。
- 调用方判断是否还有下一页。

---

## 六、筛选条件

```go
query := db.Model(&model.User{})

if status != "" {
    query = query.Where("status = ?", status)
}

if keyword != "" {
    like := "%" + keyword + "%"
    query = query.Where("username LIKE ? OR email LIKE ?", like, like)
}
```

注意使用参数化查询，不要拼接 SQL：

不推荐：

```go
query = query.Where("username LIKE '%" + keyword + "%'")
```

推荐：

```go
query = query.Where("username LIKE ?", like)
```

## 七、完整 repository List

```go
func (r *UserRepository) List(params ListUsersParams) ([]model.User, int64, error) {
    var users []model.User
    var total int64

    query := r.db.Model(&model.User{})

    if params.Status != "" {
        query = query.Where("status = ?", params.Status)
    }

    if params.Keyword != "" {
        like := "%" + params.Keyword + "%"
        query = query.Where("username LIKE ? OR email LIKE ?", like, like)
    }

    if err := query.Count(&total).Error; err != nil {
        return nil, 0, err
    }

    offset := (params.Page - 1) * params.PageSize
    if err := query.Order("id DESC").Offset(offset).Limit(params.PageSize).Find(&users).Error; err != nil {
        return nil, 0, err
    }

    return users, total, nil
}
```

## 八、service 调用

```go
func (s *UserService) List(input ListUsersInput) ([]model.User, int64, int, int, error) {
    page, pageSize := normalizePage(input.Page, input.PageSize)

    users, total, err := s.repo.List(repository.ListUsersParams{
        Page:     page,
        PageSize: pageSize,
        Status:   input.Status,
        Keyword:  input.Keyword,
    })
    if err != nil {
        return nil, 0, page, pageSize, err
    }

    return users, total, page, pageSize, nil
}
```

## 九、handler 返回

```go
users, total, page, pageSize, err := h.service.List(service.ListUsersInput{

    Total:    total,
    Page:     page,
    PageSize: pageSize,
```

## 十、索引意识

如果经常按 `status` 查询，可以给 status 加索引：

```go
Status string `gorm:"index"`
```

如果经常按 email 精确查询，email 应该有唯一索引：

```go
Email string `gorm:"uniqueIndex"`
```

`LIKE "%keyword%"` 这类查询普通索引不一定有效。后续可以学习全文搜索或专门搜索引擎。

## 十一、常见问题

### 1. total 不受分页影响吗

应该不受分页影响。

先 Count，再 Offset/Limit。

### 2. page 很大时返回空数组正常吗

正常。

例如总共 20 条，请求第 999 页，返回空 items。

### 3. 查询越来越慢怎么办

先检查：

- 是否有合适索引。
- page_size 是否过大。
- 是否有不必要的大字段。
- 是否需要游标分页。
