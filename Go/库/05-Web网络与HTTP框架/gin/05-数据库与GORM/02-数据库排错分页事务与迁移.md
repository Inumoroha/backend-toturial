# 05-02 数据库排错、分页、事务与迁移

本节目标：补充数据库实战中最常见的问题。接入 GORM 只是开始，真正写项目时你会频繁遇到连接、分页、唯一索引、事务和迁移问题。

## 一、数据库连接排错

连接失败时先看 DSN：

```text
user:password@tcp(host:port)/dbname?charset=utf8mb4&parseTime=True&loc=Local
```

逐项检查：

- 用户名是否正确。
- 密码是否正确。
- host 是否正确。
- port 是否正确。
- 数据库是否已经创建。
- MySQL 是否允许当前用户连接。

本地运行通常是：

```text
127.0.0.1:3306
```

Docker Compose 内部通常是：

```text
mysql:3306
```

这两个不要混淆。

## 二、唯一索引错误处理

邮箱唯一时，重复插入会报错。不要直接返回 500。

更好的方式是在 service 中先查：

```go
_, err := s.userRepo.GetByEmail(input.Email)
if err == nil {
    return nil, ErrEmailAlreadyExists
}
if err != nil && !errors.Is(err, repository.ErrUserNotFound) {
    return nil, err
}
```

然后再创建。

注意：先查再插仍然可能有并发问题。生产环境还要兜底处理数据库唯一索引错误。学习阶段先理解这个风险即可。

## 三、分页为什么要限制 page_size

如果不限制，用户可以请求：

```text
GET /api/v1/users?page=1&page_size=1000000
```

这会让数据库和服务承受很大压力。

建议：

```go
if page <= 0 {
    page = 1
}
if pageSize <= 0 {
    pageSize = 10
}
if pageSize > 100 {
    pageSize = 100
}
```

分页返回要包含：

- 当前页数据。
- 总数量。
- 当前页码。
- 每页数量。

## 四、事务使用场景

需要事务的典型场景：

- 创建订单并扣减库存。
- 注册用户并创建用户资料。
- 转账：A 扣钱，B 加钱。
- 创建任务并写操作日志。

事务示例：

```go
err := db.Transaction(func(tx *gorm.DB) error {
    if err := tx.Create(&task).Error; err != nil {
        return err
    }

    log := model.OperationLog{
        UserID: userID,
        Action: "create_task",
    }
    if err := tx.Create(&log).Error; err != nil {
        return err
    }

    return nil
})
```

只要返回错误，事务会自动回滚。

## 五、不要滥用事务

不是所有操作都要事务。

单表单行更新通常不需要手动事务，因为数据库本身会保证单条 SQL 的原子性。

事务时间越长，占用连接和锁的时间越长。事务中不要做：

- 调用第三方 HTTP 接口。
- 发送邮件。
- 长时间计算。
- 等待用户输入。

## 六、数据库迁移

`AutoMigrate` 适合学习，但生产环境更推荐迁移工具。

迁移文件示例：

```sql
CREATE TABLE users (
  id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
  username VARCHAR(32) NOT NULL,
  email VARCHAR(128) NOT NULL UNIQUE,
  password_hash VARCHAR(255) NOT NULL,
  role VARCHAR(20) NOT NULL DEFAULT 'user',
  created_at DATETIME,
  updated_at DATETIME,
  deleted_at DATETIME,
  INDEX idx_users_deleted_at (deleted_at)
);
```

迁移的价值：

- 数据库结构变更可追踪。
- 多人协作更安全。
- 生产环境可控执行。

## 七、慢查询意识

学习阶段先记住几个原则：

- 查询条件字段尽量建索引。
- 列表接口不要一次查太多。
- 不要在循环里频繁查数据库。
- 不要返回不需要的大字段。
- 先让功能正确，再学会用 explain 分析 SQL。
