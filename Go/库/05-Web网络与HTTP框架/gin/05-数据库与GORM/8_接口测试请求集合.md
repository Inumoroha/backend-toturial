# 8. 接口测试请求集合

本节目标：提供第6阶段数据库版用户模块的测试请求。你可以复制到 `requests.http` 中验证 GORM 持久化、分页、筛选、唯一索引和软删除。

---

## 一、健康检查数据库连接

如果你实现了 `/healthz`：

```http
### health
GET http://localhost:8080/healthz
```

期望：

```text
HTTP 200
```

---

## 二、创建用户

```http
### create db user
POST http://localhost:8080/api/v1/users
Content-Type: application/json

{
  "username": "db-user",
  "email": "db-user@example.com",
  "age": 20,
  "status": "active"
}
```

期望：

```text
HTTP 201
code = 0
```

---

## 三、邮箱重复

再次发送同样请求：

```http
### create duplicate db user
POST http://localhost:8080/api/v1/users
Content-Type: application/json

{
  "username": "db-user-2",
  "email": "db-user@example.com",
  "age": 22,
  "status": "active"
}
```

期望：

```text
HTTP 409
code = 40901
```

---

## 四、查询详情

```http
### get db user
GET http://localhost:8080/api/v1/users/1
```

不存在：

```http
### get db user not found
GET http://localhost:8080/api/v1/users/999999
```

期望不存在返回：

```text
HTTP 404
code = 40401
```

## 五、分页列表



```http

```

---

## 八、删除用户

```http
### delete db user
DELETE http://localhost:8080/api/v1/users/1
```

再次查询：

```http
### get deleted db user
GET http://localhost:8080/api/v1/users/1
```

期望：

```text
HTTP 404
```

数据库中可以查看 `deleted_at` 是否有值。

---

## 九、重启服务验证持久化

步骤：

1. 创建一个用户。
2. 停止 Go 服务。
3. 重新启动 Go 服务。
4. 查询该用户。

如果还能查到，说明数据库持久化生效。

---

## 十、数据库侧验证 SQL

除了 HTTP 请求，还应该进入 MySQL 验证数据：

```sql
USE gin_learning;

SELECT id, username, email, status, created_at, updated_at, deleted_at
FROM users
ORDER BY id DESC
LIMIT 10;
```

验证邮箱唯一索引：

```sql
SHOW INDEX FROM users;
```

验证软删除：

```sql
SELECT id, username, deleted_at
FROM users
WHERE deleted_at IS NOT NULL;
```

HTTP 响应和数据库状态要能互相对应。比如删除接口返回成功后，普通详情接口应该查不到，但数据库中 `deleted_at` 应该有值。

## 十二、本节验收标准

完成本请求集合后，请确认：

- [ ] 创建用户后 MySQL 中能查到记录。
- [ ] 服务重启后用户数据仍然存在。
- [ ] 重复邮箱返回 `409`。
- [ ] 查询不存在用户返回 `404`。
- [ ] 删除用户后普通查询返回 `404`。
- [ ] 删除用户后数据库 `deleted_at` 有值。
- [ ] 状态筛选结果正确。
- [ ] 分页返回 `items`、`total`、`page`、`page_size`。
- [ ] HTTP 行为和数据库状态一致。

这些请求全部通过后，第6阶段的 GORM 持久化实践才算真正验收完成。
