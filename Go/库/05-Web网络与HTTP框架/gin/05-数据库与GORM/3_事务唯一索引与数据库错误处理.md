# 3. 事务、唯一索引与数据库错误处理

本节目标：掌握 GORM 项目中最常见的数据库错误处理方式，包括记录不存在、唯一索引冲突和事务。完成本节后，你应该能把数据库错误转换成 service/handler 能理解的业务错误。

---

## 一、为什么数据库错误不能直接返回给前端

数据库错误可能长这样：

```text
Error 1062 (23000): Duplicate entry 'alice@example.com' for key 'users.email'
```

或者：

```text
record not found
```

如果直接返回给前端：

```json
{
  "message": "Error 1062 (23000): Duplicate entry..."
}
```

会有几个问题：

- 暴露数据库细节。
- 前端不好稳定判断。
- 用户体验差。
- 不同数据库错误格式不同。

更好的做法是：

```text
repository 识别数据库错误。
service 识别业务错误。
handler 转换为 HTTP 响应。
```

---

## 二、处理记录不存在

GORM 查询单条记录：

```go
var user model.User
err := db.First(&user, id).Error
```

如果不存在，返回：

```go
gorm.ErrRecordNotFound
```

repository 中应该转换：

```go
var ErrUserNotFound = errors.New("user not found")

func (r *UserRepository) GetByID(id uint) (*model.User, error) {
    var user model.User
    err := r.db.First(&user, id).Error
    if errors.Is(err, gorm.ErrRecordNotFound) {
        return nil, ErrUserNotFound
    }
    if err != nil {
        return nil, err
    }
    return &user, nil
}
```

handler 最终返回：

```text
HTTP 404
code 40401
```

## 三、唯一索引

模型：

```go
Email string `json:"email" gorm:"size:128;not null;uniqueIndex"`
```

这会在数据库中创建唯一索引，保证同一个邮箱不能重复插入。

为什么不能只靠 service 先查？

因为并发下可能出现：

```text
请求 A 查询邮箱不存在。
请求 B 查询邮箱不存在。
请求 A 插入成功。
请求 B 插入时才发现冲突。
```

所以唯一性最终必须由数据库保证。

---

## 四、邮箱重复的两层处理

第一层：service 先查，提供友好错误。

```go
_, err := s.repo.GetByEmail(input.Email)
if err == nil {
    return nil, ErrEmailAlreadyExists
}
if err != nil && !errors.Is(err, repository.ErrUserNotFound) {
    return nil, err
}
```

第二层：repository 创建时兜底处理唯一索引错误。

MySQL 中可以识别错误码 1062：

```go
func isDuplicateEntryError(err error) bool {
    var mysqlErr *mysql.MySQLError
    if errors.As(err, &mysqlErr) {
        return mysqlErr.Number == 1062
    }
    return false
}
```

需要导入：

```go
import "github.com/go-sql-driver/mysql"
```

## 五、Create 中处理唯一索引冲突

```go
func (r *UserRepository) Create(user *model.User) error {
    err := r.db.Create(user).Error
    if isDuplicateEntryError(err) {
        return ErrEmailAlreadyExists
    }
    return err
}
```

这样上层就不需要知道 MySQL 错误码。

repository 把底层错误转换为：

```go
ErrEmailAlreadyExists
```

service/handler 再处理业务响应。

## 六、事务是什么

事务用于保证多步数据库操作要么全部成功，要么全部失败。

典型场景：

```text
创建订单 + 扣减库存。
创建用户 + 创建用户资料。
创建任务 + 写操作日志。
转账：A 扣钱 + B 加钱。
```

如果只成功一半，数据就会不一致。

---

## 七、GORM 事务写法

```go
err := db.Transaction(func(tx *gorm.DB) error {
    if err := tx.Create(&user).Error; err != nil {
        return err
    }

    if err := tx.Create(&profile).Error; err != nil {
        return err
    }

    return nil
})
```

规则：

```text
函数返回 nil，事务提交。
函数返回 error，事务回滚。
```

---

## 八、任务和操作日志示例

假设有任务表和操作日志表。

```go
type OperationLog struct {
    ID        uint      `gorm:"primaryKey"`
    UserID    uint      `gorm:"not null;index"`
    Action    string    `gorm:"size:64;not null"`
    CreatedAt time.Time
}
```

事务：

```go
func (r *TaskRepository) CreateWithLog(task *model.Task, logItem *model.OperationLog) error {
    return r.db.Transaction(func(tx *gorm.DB) error {
        if err := tx.Create(task).Error; err != nil {
            return err
        }

        logItem.UserID = task.UserID
        logItem.Action = "create_task"

        if err := tx.Create(logItem).Error; err != nil {
            return err
        }

        return nil
    })
}
```

## 九、不要滥用事务

事务中不要做：

```text
调用第三方 HTTP API。
发送邮件。
等待用户输入。
长时间计算。
上传文件。
```

原因：

- 事务会占用数据库连接。
- 事务可能持有锁。
- 事务时间越长，影响越大。

事务里应该只放必须一起成功或失败的数据库操作。

## 十、错误应该在哪一层转换

推荐：

```text
GORM 错误 -> repository 错误
repository 错误 -> service 错误
service 错误 -> handler HTTP 响应
```

示例：

```go
if errors.Is(err, repository.ErrUserNotFound) {
    return nil, ErrUserNotFound
}
```

handler：

```go
if errors.Is(err, service.ErrUserNotFound) {
    response.Fail(c, http.StatusNotFound, response.CodeNotFound, "user not found")
    return
}
```

这样每层都只处理自己该关心的东西。

## 十一、常见错误处理表

| 数据库/Repository 错误 | Service 错误 | HTTP |
| --- | --- | --- |
| `gorm.ErrRecordNotFound` | `ErrUserNotFound` | 404 |
| MySQL 1062 | `ErrEmailAlreadyExists` | 409 |
| 其他数据库错误 | 原样或包装 | 500 |
| 事务中任一步失败 | 业务失败 | 500 或业务错误 |
