# 04-02 依赖注入、接口设计与重构步骤

本节目标：把项目分层真正讲透。很多人照着目录拆了文件，但不知道为什么这么拆，也不知道重构时应该先动哪一层。

## 一、什么是依赖注入

依赖注入就是：一个对象需要什么依赖，不在内部自己创建，而是从外部传进来。

不推荐：

```go
func NewUserHandler() *UserHandler {
    repo := repository.NewUserRepository()
    service := service.NewUserService(repo)
    return &UserHandler{service: service}
}
```

推荐：

```go
func NewUserHandler(service *service.UserService) *UserHandler {
    return &UserHandler{service: service}
}
```

原因：

- 依赖关系更清楚。
- 测试时可以替换 service。
- main 函数统一负责组装对象。
- 不容易出现隐藏依赖。

## 二、main 函数是组装层

main 可以稍微“啰嗦”一点，因为它负责把项目拼起来：

```go
db := database.NewMySQL(cfg.Database.DSN)

userRepo := repository.NewUserRepository(db)
taskRepo := repository.NewTaskRepository(db)

userService := service.NewUserService(userRepo)
taskService := service.NewTaskService(taskRepo)

userHandler := handler.NewUserHandler(userService)
taskHandler := handler.NewTaskHandler(taskService)

r := router.New(userHandler, taskHandler)
```

业务逻辑不要写在 main，但依赖组装写在 main 很正常。

## 三、什么时候定义接口

初学者容易过度抽象。不是所有地方都要先写接口。

适合定义接口的场景：

- service 需要方便测试，想替换 repository。
- 某个依赖有多种实现，例如本地存储和云存储。
- 你确实需要隔离外部系统，例如支付、短信、邮件。

不必一开始就给每个结构体都写接口。

示例：

```go
type UserRepo interface {
    GetByID(id uint) (*model.User, error)
    Create(user *model.User) error
}

type UserService struct {
    repo UserRepo
}
```

接口可以定义在使用方，也就是 service 包里。这样 service 只声明自己需要什么能力，而不是被 repository 包牵着走。

## 四、重构顺序

如果你已经把代码都写在 `main.go`，不要一次性全拆。

推荐顺序：

1. 先抽 model。
2. 再抽 response。
3. 再抽 handler。
4. 再抽 repository。
5. 再抽 service。
6. 最后抽 router。

每抽一步都运行一次。

为什么先抽 model 和 response？因为它们依赖少，不容易引发循环依赖。

## 五、避免循环依赖

错误依赖例子：

```text
handler -> service -> repository -> handler
```

这会导致 Go 编译失败。

正常方向：

```text
handler -> service -> repository
```

如果 repository 需要返回错误，错误应该定义在 repository 或更公共的 domain 包，而不是去导入 handler。

## 六、DTO 和 model 的区别

请求体结构不一定等于数据库模型。

请求 DTO：

```go
type CreateUserRequest struct {
    Username string `json:"username" binding:"required"`
    Email    string `json:"email" binding:"required,email"`
    Password string `json:"password" binding:"required,min=8"`
}
```

数据库 model：

```go
type User struct {
    ID           uint
    Username     string
    Email        string
    PasswordHash string
}
```

不要让前端直接传 `PasswordHash`。handler 接收 password，service 负责加密后生成 model。

## 七、本节练习

把现有用户模块检查一遍：

- handler 是否只处理 HTTP？
- service 是否包含业务规则？
- repository 是否只处理数据访问？
- request DTO 是否和 model 分开？
- main 是否负责依赖组装？
- 是否存在循环依赖？
