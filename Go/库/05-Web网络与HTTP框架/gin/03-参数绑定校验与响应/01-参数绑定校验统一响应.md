# 03-01 参数绑定、校验与统一响应

本阶段目标：让接口从“能用”变成“好用”。真实项目中，参数校验、统一响应和统一错误处理非常重要。

## 一、为什么需要参数校验

上一阶段的创建用户接口有一个问题：即使传空用户名也能创建成功。

例如：

```json
{
  "username": "",
  "email": "not-email"
}
```

这会把脏数据写入系统。后端必须在入口处校验请求。

## 二、使用 binding tag

定义请求结构体：

```go
type CreateUserRequest struct {
    Username string `json:"username" binding:"required,min=2,max=32"`
    Email    string `json:"email" binding:"required,email"`
    Age      int    `json:"age" binding:"gte=1,lte=120"`
}
```

常用规则：

- `required`：必填。
- `min`：字符串最小长度或数字最小值。
- `max`：字符串最大长度或数字最大值。
- `email`：邮箱格式。
- `gte`：大于等于。
- `lte`：小于等于。
- `oneof`：限定枚举值。

状态字段示例：

```go
Status string `json:"status" binding:"omitempty,oneof=active disabled"`
```

`omitempty` 表示如果字段为空就不校验后面的规则。

## 三、绑定 JSON 参数

```go
func createUser(c *gin.Context) {
    var req CreateUserRequest
    if err := c.ShouldBindJSON(&req); err != nil {
        c.JSON(http.StatusBadRequest, gin.H{
            "code":    40001,
            "message": "invalid request",
        })
        return
    }

    c.JSON(http.StatusCreated, gin.H{
        "code":    0,
        "message": "ok",
        "data":    req,
    })
}
```

`ShouldBindJSON` 失败时会返回错误，但不会自动写响应。这样你可以完全控制错误格式。

## 四、绑定查询参数

列表接口通常有分页：

```go
type ListUsersRequest struct {
    Page     int    `form:"page" binding:"omitempty,gte=1"`
    PageSize int    `form:"page_size" binding:"omitempty,gte=1,lte=100"`
    Status   string `form:"status" binding:"omitempty,oneof=active disabled"`
}
```

handler：

```go
func listUsers(c *gin.Context) {
    var req ListUsersRequest
    if err := c.ShouldBindQuery(&req); err != nil {
        c.JSON(http.StatusBadRequest, gin.H{
            "code":    40001,
            "message": "invalid query",
        })
        return
    }

    if req.Page == 0 {
        req.Page = 1
    }
    if req.PageSize == 0 {
        req.PageSize = 10
    }

    c.JSON(http.StatusOK, gin.H{
        "code":    0,
        "message": "ok",
        "data":    req,
    })
}
```

注意：查询参数使用 `form` tag，不是 `json` tag。

## 五、统一响应结构

创建 `response` 包，例如：

```text
internal/response/response.go
```

内容：

```go
package response

import (
    "net/http"

    "github.com/gin-gonic/gin"
)

type Body struct {
    Code    int         `json:"code"`
    Message string      `json:"message"`
    Data    interface{} `json:"data,omitempty"`
}

func OK(c *gin.Context, data interface{}) {
    c.JSON(http.StatusOK, Body{
        Code:    0,
        Message: "ok",
        Data:    data,
    })
}

func Created(c *gin.Context, data interface{}) {
    c.JSON(http.StatusCreated, Body{
        Code:    0,
        Message: "ok",
        Data:    data,
    })
}

func Fail(c *gin.Context, httpStatus int, code int, message string) {
    c.JSON(httpStatus, Body{
        Code:    code,
        Message: message,
    })
}
```

使用方式：

```go
response.OK(c, user)
response.Fail(c, http.StatusBadRequest, 40001, "invalid request")
```

这样所有接口返回格式一致，前端处理会轻松很多。

## 六、业务错误设计

不要所有错误都返回 `500`。常见分类：

```text
0      成功
40001  请求参数错误
40101  未登录或 token 无效
40301  无权限
40401  资源不存在
40901  资源冲突，例如邮箱已存在
50001  系统内部错误
```

HTTP 状态码和业务错误码可以同时使用：

- HTTP 状态码表示协议层结果。
- 业务错误码表示业务层原因。

例如邮箱已存在：

```json
{
  "code": 40901,
  "message": "email already exists"
}
```

HTTP 状态码可以是 `409 Conflict`。

## 七、把错误处理写清楚

初学者常见写法：

```go
if err != nil {
    c.JSON(500, gin.H{"message": err.Error()})
    return
}
```

问题是：

- 可能把内部错误暴露给用户。
- 所有错误都变成 500。
- 前端无法稳定判断错误类型。

更推荐：

```go
if errors.Is(err, ErrUserNotFound) {
    response.Fail(c, http.StatusNotFound, 40401, "user not found")
    return
}

response.Fail(c, http.StatusInternalServerError, 50001, "internal server error")
```

## 八、阶段练习

改造上一阶段用户 API：

- 创建用户时校验用户名、邮箱、年龄。
- 查询列表时支持分页参数校验。
- 所有成功响应使用统一格式。
- 所有失败响应使用统一格式。
- 邮箱重复时返回 `40901`。
- 用户不存在时返回 `40401`。
