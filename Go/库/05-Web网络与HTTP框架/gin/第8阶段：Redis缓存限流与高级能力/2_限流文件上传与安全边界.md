# 2. 限流、文件上传与安全边界

本节目标：在正式写代码前，先理解限流和文件上传为什么属于安全边界问题。

很多初学者会把限流和文件上传当成普通功能：能限制次数、能上传成功就行。但在真实后端项目中，这两类能力都直接影响系统安全和稳定性。

---

## 一、为什么需要限流

限流是为了保护系统和用户。

典型场景：

- 登录接口被暴力破解。
- 验证码接口被频繁调用。
- 搜索接口被高频刷请求。
- 上传接口被反复上传大文件。
- 某个用户或 IP 请求过于频繁。

没有限流时，攻击者或异常客户端可能让数据库、Redis、短信服务、文件存储被迅速打满。

---

## 二、登录失败限流策略

学习项目可以实现一个简单策略：

```text
同一个邮箱连续登录失败 5 次
接下来 15 分钟禁止继续尝试
登录成功后清除失败计数
```

Redis key：

```text
login:fail:email:a@example.com
```

失败时：

```text
INCR login:fail:email:a@example.com
第一次失败时设置 EXPIRE 15 分钟
```

检查时：

```text
GET login:fail:email:a@example.com
如果次数 >= 5，拒绝登录
```

成功时：

```text
DEL login:fail:email:a@example.com
```

---

## 三、为什么不用内存变量做限流

你可能会想到：

```go
var loginFailCount = map[string]int{}
```

学习时可以理解，但真实项目不适合：

- 服务重启后计数丢失。
- 多实例部署时每个实例计数不同。
- 并发读写 map 有风险。
- 无法自动过期。

Redis 适合做这类带过期时间的计数器。

---

## 四、返回什么状态码

限流通常返回：

```text
429 Too Many Requests
```

统一响应示例：

```json
{
  "code": "TOO_MANY_REQUESTS",
  "message": "尝试次数过多，请稍后再试"
}
```

不要返回 `500`。限流不是服务器异常，而是请求被策略拒绝。

---

## 五、文件上传的风险

文件上传如果没有限制，风险很大：

- 上传超大文件，耗尽磁盘或内存。
- 上传脚本文件，被错误执行。
- 使用路径穿越覆盖系统文件。
- 文件名重复覆盖已有文件。
- 伪造扩展名，例如 `a.jpg.exe`。

所以文件上传至少要限制：

- 文件大小。
- 文件扩展名。
- MIME 类型。
- 保存目录。
- 保存文件名。

## 六、头像上传推荐规则

学习项目中，头像上传可以这样设计：

```text
接口：POST /api/v1/profile/avatar
认证：需要登录
字段名：file
最大大小：2MB
允许类型：jpg、jpeg、png、webp
保存目录：uploads/avatars
返回 URL：/static/avatars/{filename}
```

保存文件名不要直接使用用户上传的原始文件名。推荐：

```text
用户ID_时间戳_随机字符串.ext
```

例如：

```text
1_20260705103000_f3a9c2.png
```

---

## 七、路径安全

危险写法：

```go
savePath := "uploads/avatars/" + file.Filename
```

如果文件名被构造成特殊路径，可能产生安全问题。更稳妥的做法：

```go
filename := filepath.Base(file.Filename)
ext := strings.ToLower(filepath.Ext(filename))
```

再自己生成新文件名：

```go
newName := fmt.Sprintf("%d_%d%s", userID, time.Now().UnixNano(), ext)
savePath := filepath.Join(uploadDir, newName)
```

这样不会直接信任用户传来的文件名。

---

## 八、上传大小限制放在哪里

可以在 Gin 中设置：

```go
r.MaxMultipartMemory = 8 << 20
```

但这不是唯一限制。handler 中还应该检查文件大小：

```go
if file.Size > maxSize {
    response.Fail(c, http.StatusBadRequest, "FILE_TOO_LARGE", "文件过大")
    return
}
```

配置中可以写：

```yaml
upload:
  max_avatar_size_mb: 2
```

## 九、常见问题

### 1. 限流 key 应该按 IP 还是用户

看场景。登录失败适合按邮箱或账号限流，也可以叠加 IP 限流。验证码接口常按手机号、IP、设备做组合限制。

### 2. 上传文件只检查扩展名够不够

不够。扩展名可以伪造。学习阶段至少检查扩展名和 MIME 类型；生产环境还要考虑文件内容检测、对象存储安全策略等。

### 3. 上传目录能不能放在项目代码目录

学习阶段可以。生产中更推荐对象存储或独立挂载目录，不要依赖容器内部临时文件系统。
