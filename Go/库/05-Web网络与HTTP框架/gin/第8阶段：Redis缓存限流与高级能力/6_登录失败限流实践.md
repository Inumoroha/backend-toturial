# 6. 登录失败限流实践

本节目标：使用 Redis 实现登录失败次数限制，防止账号被暴力尝试。

登录接口是后端项目中最需要保护的接口之一。如果没有限制，攻击者可以不断尝试密码。即使密码最终没有被猜中，也会给数据库和服务造成压力。

---

## 一、限流规则设计

本节实现一个学习项目中足够清晰的规则：

```text
同一个邮箱 15 分钟内登录失败达到 5 次
后续登录请求返回 429
登录成功后清除失败次数
```

Redis key：

```text
login:fail:email:{email}
```

例如：

```text
login:fail:email:a@example.com
```

---

## 二、定义 Limiter

创建 `internal/cache/login_limiter.go`：

```go
package cache

import (
    "context"
    "fmt"
    "time"

    "github.com/redis/go-redis/v9"
)

type LoginLimiter struct {
    rdb        *redis.Client
    maxFails   int
    window     time.Duration
}

func NewLoginLimiter(rdb *redis.Client, maxFails int, window time.Duration) *LoginLimiter {
    return &LoginLimiter{
        rdb:      rdb,
        maxFails: maxFails,
        window:   window,
    }
}

func loginFailKey(email string) string {
    return fmt.Sprintf("login:fail:email:%s", email)
}
```

`maxFails` 和 `window` 不要写死在函数内部，方便通过配置调整。

---

## 三、检查是否被限制

```go
func (l *LoginLimiter) IsBlocked(ctx context.Context, email string) (bool, error) {
    count, err := l.rdb.Get(ctx, loginFailKey(email)).Int()
    if err == redis.Nil {
        return false, nil
    }
    if err != nil {
        return false, fmt.Errorf("get login fail count: %w", err)
    }

    return count >= l.maxFails, nil
}
```

注意：Redis 不存在 key 时，说明当前没有失败记录，不能当作错误处理。

---

## 四、记录登录失败

```go
func (l *LoginLimiter) RecordFail(ctx context.Context, email string) error {
    key := loginFailKey(email)

    count, err := l.rdb.Incr(ctx, key).Result()
    if err != nil {
        return fmt.Errorf("incr login fail count: %w", err)
    }

    if count == 1 {
        if err := l.rdb.Expire(ctx, key, l.window).Err(); err != nil {
            return fmt.Errorf("expire login fail count: %w", err)
        }
    }

    return nil
}
```

这里使用：

```text
INCR + EXPIRE
```

第一次失败时设置过期时间。后续失败只增加次数，不刷新过期时间。

---

## 五、登录成功后清除计数

```go
func (l *LoginLimiter) Clear(ctx context.Context, email string) error {
    if err := l.rdb.Del(ctx, loginFailKey(email)).Err(); err != nil {
        return fmt.Errorf("clear login fail count: %w", err)
    }
    return nil
}
```

只要用户登录成功，就应该清除失败次数。

## 六、在 service 登录流程中使用

定义接口：

```go
type LoginLimiter interface {
    IsBlocked(ctx context.Context, email string) (bool, error)
    RecordFail(ctx context.Context, email string) error
    Clear(ctx context.Context, email string) error
}
```

登录流程：

```go
func (s *UserService) Login(ctx context.Context, req LoginRequest) (*LoginResult, error) {
    blocked, err := s.loginLimiter.IsBlocked(ctx, req.Email)
    if err != nil {
        s.log.Warn("check login limiter failed", zap.String("email", req.Email), zap.Error(err))
    }
    if blocked {
        return nil, ErrTooManyLoginAttempts
    }

    user, err := s.repo.FindByEmail(ctx, req.Email)
    if err != nil {
        _ = s.loginLimiter.RecordFail(ctx, req.Email)
        return nil, ErrInvalidCredentials
    }

    if !checkPassword(user.PasswordHash, req.Password) {
        _ = s.loginLimiter.RecordFail(ctx, req.Email)
        return nil, ErrInvalidCredentials
    }

    if err := s.loginLimiter.Clear(ctx, req.Email); err != nil {
        s.log.Warn("clear login fail count failed", zap.String("email", req.Email), zap.Error(err))
    }

    token, err := s.jwt.Generate(user.ID)
    if err != nil {
        return nil, err
    }

    return &LoginResult{Token: token}, nil
}
```

为了安全，邮箱不存在和密码错误都返回统一错误：

```text
邮箱或密码错误
```

不要告诉攻击者“邮箱存在但密码错”。

## 七、handler 返回 429

service 定义错误：

```go
var ErrTooManyLoginAttempts = errors.New("too many login attempts")
```

handler 中判断：

```go
if errors.Is(err, service.ErrTooManyLoginAttempts) {
    response.Fail(c, http.StatusTooManyRequests, "TOO_MANY_LOGIN_ATTEMPTS", "尝试次数过多，请稍后再试")
    return
}
```

HTTP 状态码：

```text
429 Too Many Requests
```

## 八、配置化限制

配置文件：

```yaml
security:
  login_max_fails: 5
  login_fail_window_minutes: 15
```

结构体：

```go
type SecurityConfig struct {
    LoginMaxFails          int `mapstructure:"login_max_fails"`
    LoginFailWindowMinutes int `mapstructure:"login_fail_window_minutes"`
}
```

初始化：

```go
loginLimiter := cache.NewLoginLimiter(
    redisClient,
    cfg.Security.LoginMaxFails,
    time.Duration(cfg.Security.LoginFailWindowMinutes)*time.Minute,
)
```

## 九、验证

连续输入错误密码：

```bash
curl -X POST http://localhost:8080/api/v1/login \
  -H "Content-Type: application/json" \
  -d '{"email":"a@example.com","password":"wrong"}'
```

执行 5 次后，再请求一次，应该返回：

```text
HTTP/1.1 429 Too Many Requests
```

查看 Redis：

```bash
docker exec -it redis-dev redis-cli
GET login:fail:email:a@example.com
TTL login:fail:email:a@example.com
```

登录成功后：

```redis
GET login:fail:email:a@example.com
```

应该返回空。

## 十、常见问题

### 1. Redis 出错时是否允许登录

这取决于安全策略。学习项目可以记录 warn 日志后继续验证账号密码。安全要求高的系统可能选择拒绝登录。

### 2. key 中直接放邮箱安全吗

学习阶段可以。生产中可以对邮箱做 hash，避免敏感信息直接暴露在 Redis key 中。

### 3. 为什么失败次数不随每次失败刷新 TTL

固定窗口更容易理解。生产中还可以使用滑动窗口、令牌桶等策略。

## 十二、验收标准

完成后确认：

- 同一邮箱连续失败会增加 Redis 计数。
- 第一次失败时 key 有 TTL。
- 达到上限后接口返回 `429`。
- 登录成功后失败计数被清除。
- 错误响应不泄露“邮箱是否存在”。
