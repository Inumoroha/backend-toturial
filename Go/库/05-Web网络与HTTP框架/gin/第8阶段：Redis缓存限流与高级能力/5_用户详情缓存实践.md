# 5. 用户详情缓存实践

本节目标：为用户详情接口添加 Redis 缓存，实现“先读缓存，未命中再查数据库”的流程。

用户详情是很适合练习缓存的场景。它通常读取频繁，修改不算频繁，而且接口响应结构相对稳定。

---

## 一、目标接口

假设已有接口：

```text
GET /api/v1/users/:id
```

响应：

```json
{
  "code": "OK",
  "message": "success",
  "data": {
    "id": 1,
    "email": "a@example.com",
    "nickname": "alice"
  }
}
```

本节要实现：

```text
第一次请求：Redis 未命中 -> 查询数据库 -> 写入 Redis -> 返回
第二次请求：Redis 命中 -> 直接返回
更新用户资料：更新数据库 -> 删除缓存
```

---

## 二、定义 UserCache 接口

创建 `internal/cache/user_cache.go`：

```go
package cache

import (
    "context"
    "encoding/json"
    "fmt"
    "time"

    "github.com/redis/go-redis/v9"
)

type UserDTO struct {
    ID       uint   `json:"id"`
    Email    string `json:"email"`
    Nickname string `json:"nickname"`
}

type UserCache struct {
    rdb *redis.Client
    ttl time.Duration
}

func NewUserCache(rdb *redis.Client, ttl time.Duration) *UserCache {
    return &UserCache{
        rdb: rdb,
        ttl: ttl,
    }
}

func userDetailKey(id uint) string {
    return fmt.Sprintf("user:detail:%d", id)
}
```

实际项目中 DTO 可以放在 service 层或单独 dto 包。这里为了讲缓存流程，示例写在 cache 中，项目落地时可以按你的分层调整。

---

## 三、读取缓存

```go
func (c *UserCache) GetUserDetail(ctx context.Context, id uint) (*UserDTO, error) {
    value, err := c.rdb.Get(ctx, userDetailKey(id)).Result()
    if err == redis.Nil {
        return nil, nil
    }
    if err != nil {
        return nil, fmt.Errorf("get user detail cache: %w", err)
    }

    var user UserDTO
    if err := json.Unmarshal([]byte(value), &user); err != nil {
        return nil, fmt.Errorf("unmarshal user detail cache: %w", err)
    }

    return &user, nil
}
```

注意：

- `redis.Nil` 表示 key 不存在，不是系统错误。
- JSON 解析失败要返回错误，并记录日志。
- 读取缓存失败时，业务上可以降级查数据库。

---

## 四、写入缓存

```go
func (c *UserCache) SetUserDetail(ctx context.Context, user *UserDTO) error {
    data, err := json.Marshal(user)
    if err != nil {
        return fmt.Errorf("marshal user detail cache: %w", err)
    }

    if err := c.rdb.Set(ctx, userDetailKey(user.ID), data, c.ttl).Err(); err != nil {
        return fmt.Errorf("set user detail cache: %w", err)
    }

    return nil
}
```

写入缓存时必须设置 TTL：

```go
cache.NewUserCache(redisClient, 10*time.Minute)
```

---

## 五、删除缓存

```go
func (c *UserCache) DeleteUserDetail(ctx context.Context, id uint) error {
    if err := c.rdb.Del(ctx, userDetailKey(id)).Err(); err != nil {
        return fmt.Errorf("delete user detail cache: %w", err)
    }
    return nil
}
```

更新用户资料后调用：

```go
_ = userCache.DeleteUserDetail(ctx, userID)
```

如果删除失败，至少要记录日志：

```go
log.Warn("delete user detail cache failed",
    zap.Uint("user_id", userID),
    zap.Error(err),
)
```

## 六、在 service 中使用缓存

定义 service 依赖：

```go
type UserCache interface {
    GetUserDetail(ctx context.Context, id uint) (*cache.UserDTO, error)
    SetUserDetail(ctx context.Context, user *cache.UserDTO) error
    DeleteUserDetail(ctx context.Context, id uint) error
}

type UserService struct {
    repo  UserRepository
    cache UserCache
    log   *zap.Logger
}
```

读取用户详情：

```go
func (s *UserService) GetByID(ctx context.Context, id uint) (*UserDTO, error) {
    cached, err := s.cache.GetUserDetail(ctx, id)
    if err == nil && cached != nil {
        return &UserDTO{
            ID:       cached.ID,
            Email:    cached.Email,
            Nickname: cached.Nickname,
        }, nil
    }

    if err != nil {
        s.log.Warn("read user cache failed", zap.Uint("user_id", id), zap.Error(err))
    }

    user, err := s.repo.FindByID(ctx, id)
    if err != nil {
        return nil, err
    }

    dto := &UserDTO{
        ID:       user.ID,
        Email:    user.Email,
        Nickname: user.Nickname,
    }

    if err := s.cache.SetUserDetail(ctx, &cache.UserDTO{
        ID:       dto.ID,
        Email:    dto.Email,
        Nickname: dto.Nickname,
    }); err != nil {
        s.log.Warn("write user cache failed", zap.Uint("user_id", id), zap.Error(err))
    }

    return dto, nil
}
```

缓存失败时没有直接让接口失败，这是常见降级策略：缓存是加速层，不应该轻易影响核心读取。

---

## 七、更新用户后删除缓存

```go
func (s *UserService) UpdateProfile(ctx context.Context, userID uint, req UpdateProfileRequest) error {
    if err := s.repo.UpdateProfile(ctx, userID, req.Nickname); err != nil {
        return err
    }

    if err := s.cache.DeleteUserDetail(ctx, userID); err != nil {
        s.log.Warn("delete user cache failed",
            zap.Uint("user_id", userID),
            zap.Error(err),
        )
    }

    return nil
}
```

流程是：

```text
先更新数据库
再删除缓存
```

不要先删缓存再更新数据库。否则数据库更新失败时，缓存已经被删，虽然不一定错误，但会制造额外不确定性。

## 八、如何验证缓存命中

启动 Redis 和服务后：

```bash
curl http://localhost:8080/api/v1/users/1
curl http://localhost:8080/api/v1/users/1
```

查看 Redis：

```bash
docker exec -it redis-dev redis-cli
```

执行：

```redis
GET user:detail:1
TTL user:detail:1
```

你应该能看到 JSON 和剩余过期时间。

也可以在 service 中临时加日志：

```go
s.log.Info("user cache hit", zap.Uint("user_id", id))
s.log.Info("user cache miss", zap.Uint("user_id", id))
```

## 九、常见问题

### 1. 缓存一直不命中

检查：

- 读写 key 是否一致。
- 写入缓存是否报错。
- TTL 是否太短。
- Redis DB 编号是否一致。

### 2. 更新资料后还是旧数据

检查更新成功后是否调用了 `DeleteUserDetail`，以及删除的 key 是否和读取 key 一致。

### 3. Redis 出错导致接口失败

用户详情这类场景建议降级查数据库。缓存错误记录 warn 日志即可。

## 十一、验收标准

完成后检查：

- 第一次请求用户详情能写入 Redis。
- 第二次请求同一用户能命中缓存。
- Redis 中 key 有 TTL。
- 更新用户资料后 Redis key 被删除。
- Redis 暂时不可用时，用户详情接口仍能查数据库返回。
