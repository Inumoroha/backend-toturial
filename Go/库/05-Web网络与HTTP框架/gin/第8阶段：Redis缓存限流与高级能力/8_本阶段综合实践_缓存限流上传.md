# 8. 本阶段综合实践：缓存、限流与上传

本节目标：把 Redis 缓存、登录失败限流、文件上传、context 和优雅关闭整合到同一个 Gin 项目中。

这一节的重点不是新增知识点，而是检查你的项目是否真的具备了第8阶段要求的高级能力。

---

## 一、综合目标

完成后，项目应支持：

```text
Redis 初始化和连接检查
用户详情缓存
更新资料后删除缓存
登录失败限流
头像上传
静态文件访问
context 贯穿 handler/service/repository/cache
HTTP Server 超时配置
优雅关闭
```

---

## 二、配置最终形态

配置文件至少包含：

```yaml
server:
  port: 8080
  mode: "debug"
  read_timeout_seconds: 10
  write_timeout_seconds: 10
  idle_timeout_seconds: 60
  shutdown_timeout_seconds: 10

redis:
  addr: "127.0.0.1:6379"
  password: ""
  db: 0
  dial_timeout_seconds: 5
  read_timeout_seconds: 3
  write_timeout_seconds: 3

security:
  login_max_fails: 5
  login_fail_window_minutes: 15

upload:
  avatar_dir: "uploads/avatars"
  max_avatar_size_mb: 2
```

敏感配置依然应支持环境变量覆盖。

---

## 三、main 组装顺序

推荐顺序：

```text
1. 加载配置
2. 初始化 logger
3. 初始化数据库
4. 初始化 Redis
5. 创建 cache / limiter
6. 创建 repository
7. 创建 service
8. 创建 handler
9. 注册中间件和路由
10. 创建 http.Server
11. 启动服务并监听退出信号
```

不要在 handler 中临时初始化 Redis，也不要在每个请求中创建数据库连接。

## 四、接口验收清单

### 1. Redis 健康检查

```bash
curl http://localhost:8080/health/redis
```

预期：

```json
{"status":"up"}
```

### 2. 用户详情缓存

```bash
curl http://localhost:8080/api/v1/users/1
curl http://localhost:8080/api/v1/users/1
```

Redis CLI：

```redis
GET user:detail:1
TTL user:detail:1
```

预期：

- Redis 中存在 key。
- TTL 大于 0。
- 第二次请求日志显示 cache hit。

### 3. 更新资料删除缓存

```bash
curl -X PUT http://localhost:8080/api/v1/profile \
  -H "Authorization: Bearer your-token" \
  -H "Content-Type: application/json" \
  -d '{"nickname":"new-name"}'
```

Redis CLI：

```redis
GET user:detail:1
```

预期：缓存被删除或下一次请求重新写入新值。

### 4. 登录失败限流

连续 5 次错误登录：

```bash
curl -X POST http://localhost:8080/api/v1/login \
  -H "Content-Type: application/json" \
  -d '{"email":"a@example.com","password":"wrong"}'
```

预期：达到次数后返回 `429`。

### 5. 头像上传

```bash
curl -X POST http://localhost:8080/api/v1/profile/avatar \
  -H "Authorization: Bearer your-token" \
  -F "file=@avatar.png"
```

预期：

- 返回 `avatar_url`。
- 静态 URL 可访问。
- 用户详情中头像地址更新。

## 五、代码质量检查

搜索是否存在不合理写法：

```bash
rg "context.Background\\(\\)" internal
```

请求链路内不应该大量出现 `context.Background()`。

搜索 Redis 客户端创建：

```bash
rg "redis.NewClient" .
```

正常情况下应该只在初始化位置出现一次。

搜索文件上传原始文件名：

```bash
rg "Filename" internal
```

确认没有直接把 `file.Filename` 拼接成保存路径。

## 六、测试建议

本阶段可以补充这些测试：

- `login limiter` 失败计数。
- `login limiter` 达到上限后阻止登录。
- `user cache` key 生成函数。
- 上传扩展名校验函数。
- service 更新头像后调用删除缓存。

如果 Redis 相关测试暂时不好写，可以先把 key 生成、扩展名判断、service 分支测起来。

## 七、常见问题

### 1. Redis 挂了整个项目启动失败

如果你的设计中 Redis 是强依赖，这是合理的。如果只做缓存，可以改成 Redis 不可用时降级，但要记录清楚日志。

### 2. 缓存命中后返回字段少了

检查缓存 DTO 是否和接口响应 DTO 字段一致。不要漏掉 `avatar_url` 等字段。

### 3. 上传到容器后文件丢失

容器重启后内部文件可能丢失。生产部署要使用挂载卷或对象存储。第9阶段会继续讲部署问题。

## 八、本阶段最终验收

请逐项确认：

- [ ] Redis 配置可读取。
- [ ] Redis 启动时 ping 成功。
- [ ] 用户详情缓存有 TTL。
- [ ] 更新用户资料会删除缓存。
- [ ] 登录失败次数记录在 Redis。
- [ ] 达到限制后返回 `429`。
- [ ] 上传头像限制大小和类型。
- [ ] 静态文件 URL 可访问。
- [ ] context 从 handler 传到 repository/cache。
- [ ] HTTP 服务支持优雅关闭。
