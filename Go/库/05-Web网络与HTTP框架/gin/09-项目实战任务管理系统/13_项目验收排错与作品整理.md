# 13. 项目验收、排错与作品整理

本节目标：把任务管理系统从“功能写完”整理成一个可以展示、可以复盘、可以继续迭代的后端项目。

项目实战阶段不能只停在“接口能跑”。你还需要证明：

- 功能链路完整。
- 权限边界正确。
- 数据库状态正确。
- 错误响应稳定。
- 测试和部署能跑。
- README 能让别人启动项目。

这份清单就是最终交付前的验收手册。

---

## 一、功能链路验收

按下面顺序完整跑一遍：

```text
1. 注册用户 A
2. 用户 A 登录
3. 用户 A 获取 profile
4. 用户 A 创建任务
5. 用户 A 查询任务列表
6. 用户 A 查询任务详情
7. 用户 A 更新任务标题和描述
8. 用户 A 修改任务状态
9. 用户 A 删除任务
10. 删除后再次查询任务详情
```

每一步都要记录：

```text
HTTP 状态码
业务 code
响应 data
数据库状态
```

如果某一步失败，不要跳过。项目实战的价值就在于把链路打通。

---

## 二、权限边界验收

任务系统最重要的权限规则是：

```text
用户只能管理自己的任务。
```

测试步骤：

```text
1. 注册用户 A。
2. 注册用户 B。
3. 用户 A 创建任务。
4. 用户 B 登录。
5. 用户 B 查询用户 A 的任务详情。
6. 用户 B 更新用户 A 的任务。
7. 用户 B 删除用户 A 的任务。
```

期望：

```text
查询别人的任务 -> 404 或 403，按你的设计保持一致。
更新别人的任务 -> 404 或 403。
删除别人的任务 -> 404 或 403。
```

推荐学习项目返回 `404`，避免暴露“这个任务存在但不属于你”的信息。

repository 查询时应该带上 `user_id`：

```go
db.Where("id = ? AND user_id = ?", taskID, userID).First(&task)
```

不要先只按 `id` 查出任务，再在 handler 中判断归属。

## 三、错误响应验收

至少验证这些错误：

```text
注册邮箱重复
登录密码错误
未登录访问 profile
未登录创建任务
创建任务标题为空
任务状态非法
查询不存在任务
普通用户访问别人的任务
```

期望响应：

```json
{
  "code": 40001,
  "message": "invalid request"
}
```

或：

```json
{
  "code": 40101,
  "message": "unauthorized"
}
```

自查：

- [ ] 没有直接返回 Gin 默认错误。
- [ ] 没有把数据库错误原文返回给用户。
- [ ] HTTP 状态码和业务 code 对应稳定。
- [ ] 参数错误有字段级信息或明确提示。

## 四、数据库验收

进入数据库检查：

```sql
SELECT id, email, created_at FROM users ORDER BY id DESC LIMIT 5;

SELECT id, user_id, title, status, deleted_at
FROM tasks
ORDER BY id DESC
LIMIT 10;
```

检查点：

- [ ] users 表有注册用户。
- [ ] tasks 表有创建任务。
- [ ] task.user_id 指向创建者。
- [ ] 删除任务后 `deleted_at` 有值。
- [ ] 普通查询不返回软删除任务。
- [ ] email 唯一索引生效。
- [ ] task status 只出现允许值。

如果 HTTP 响应和数据库状态不一致，优先查 repository 查询条件和事务逻辑。

## 五、Redis 验收

如果实现了 Redis：

登录失败限制：

```text
连续输错密码 -> Redis 计数增加
达到上限 -> 返回 429
登录成功 -> 清除失败计数
```

检查：

```bash
docker compose exec redis redis-cli
GET login:fail:email:a@example.com
TTL login:fail:email:a@example.com
```

任务缓存如果实现了，也要检查：

- 缓存 key 是否有 TTL。
- 更新任务后是否删除缓存。
- 删除任务后是否删除缓存。

```text

- 登录成功
```

运行：

```bash

```

验收：

- [ ] 测试能通过。

- [ ] 权限边界有测试。
- [ ] 测试不依赖手动操作顺序。

## 七、Docker 部署验收

执行：

```bash

docker compose ps
curl http://localhost/healthz

```

检查点：

- [ ] app 容器正常运行。
- [ ] mysql 容器正常运行。
- [ ] redis 容器正常运行。
- [ ] nginx 能代理到 app。
- [ ] `/healthz` 返回 200。
- [ ] `/readyz` 能检查数据库和 Redis。
- [ ] 上传目录或必要数据使用 volume。

## 九、排错清单

### 1. 登录后访问 profile 仍然 401

检查：

- Authorization 是否是 `Bearer token`。
- JWT secret 是否一致。
- token 是否过期。
- Auth 中间件是否正确设置 `user_id`。

### 2. 用户能看到别人的任务

检查 repository 查询是否带 `user_id` 条件。

### 3. 删除后任务还在列表里

检查是否使用软删除，列表查询是否默认过滤 `deleted_at`。

### 4. Docker 中连不上数据库

检查 DSN 是否使用 `mysql:3306`，而不是 `127.0.0.1:3306`。

### 5. Swagger 接口和实际不一致

重新生成：

```bash
swag init -g cmd/server/main.go -o docs
```
