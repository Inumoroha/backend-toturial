# 1. 需求分析与接口设计

本节目标：在写代码之前，把任务管理系统的业务规则、接口路径、请求响应和错误码设计清楚。

真实后端开发不要一上来就写 handler。先把接口设计清楚，后面代码会顺很多。

---

## 一、用户故事

这个系统有一个最小业务闭环：

```text
用户注册账号。
用户登录获得 token。
用户携带 token 创建任务。
用户查看自己的任务列表。
用户修改任务状态。
用户删除已经完成或不需要的任务。
```

这里最重要的安全规则是：

```text
用户只能访问自己的任务。
```

这条规则后面必须在数据库查询条件中体现，不能只依赖前端。

---

## 二、用户规则

用户模块规则：

- 邮箱唯一。
- 用户名不能为空。
- 密码至少 8 位。
- 密码不能明文存储。
- 登录成功返回 JWT。
- JWT 过期后需要重新登录。

注册请求：

```json
{
  "username": "alice",
  "email": "alice@example.com",
  "password": "12345678"
}
```

注册响应：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "id": 1,
    "username": "alice",
    "email": "alice@example.com"
  }
}
```

注意：响应中不能出现 `password` 或 `password_hash`。

---

## 三、任务规则

任务字段：

```text
id
user_id
title
description
status
created_at
updated_at
deleted_at
```

任务状态：

```text
todo
doing
done
```

规则：

- 标题必填。
- 描述可选。
- 状态为空时默认为 `todo`。
- 状态只能是 `todo`、`doing`、`done`。
- 查询列表按创建时间倒序。
- 删除任务使用软删除。

---

## 四、接口设计

创建任务：

```http
POST /api/v1/tasks
Authorization: Bearer token
Content-Type: application/json

{
  "title": "学习 Gin 参数绑定",
  "description": "完成 ShouldBindJSON 和 ShouldBindQuery",
  "status": "todo"
}
```

查询列表：

```http
GET /api/v1/tasks?page=1&page_size=10&status=todo
Authorization: Bearer token
```

查询详情：

```http
GET /api/v1/tasks/1
Authorization: Bearer token
```

修改状态：

```http
PATCH /api/v1/tasks/1/status
Authorization: Bearer token
Content-Type: application/json

{
  "status": "done"
}
```

## 五、统一响应格式

成功：

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

失败：

```json
{
  "code": 40001,
  "message": "invalid request"
}
```

分页：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "items": [],
    "total": 0,
    "page": 1,
    "page_size": 10
  }
}
```

## 六、错误码设计

```text
0      成功
40001  参数错误
40101  未登录或 token 无效
40301  无权限
40401  资源不存在
40901  资源冲突
42901  请求过于频繁
50001  系统内部错误
```

例子：

- 邮箱已存在：HTTP 409 + code 40901。
- token 无效：HTTP 401 + code 40101。
- 任务不存在：HTTP 404 + code 40401。

## 九、用户故事拆解

把需求写成用户故事，有助于你判断接口是否完整。

```text
作为一个新用户，我希望注册账号，这样我可以保存自己的任务。
作为一个已注册用户，我希望登录后获得 token，这样我可以访问私有接口。
作为一个已登录用户，我希望创建任务，这样我可以记录待办事项。
作为一个已登录用户，我希望查看任务列表，这样我可以知道还有哪些事情没做。
作为一个已登录用户，我希望修改任务状态，这样我可以标记任务进度。
作为一个已登录用户，我希望删除任务，这样我可以清理不需要的任务。
```

每个用户故事都应该能落到一个或多个接口。

## 十、字段设计

用户字段：

```text
id
email
password_hash
nickname
created_at
updated_at
deleted_at
```

任务字段：

```text
id
user_id
title
description
status
due_date
created_at
updated_at
deleted_at
```

任务状态建议：

```text
todo
doing
done
cancelled
```

字段边界：

- `password_hash` 只在数据库模型中出现，不出现在响应中。
- `user_id` 由 JWT 当前用户决定，不允许前端传入。
- `status` 必须限制枚举值。
- `due_date` 可以为空。
- `deleted_at` 用于软删除，不直接暴露给普通响应。

## 十一、接口契约表

| 功能 | 方法 | 路径 | 是否登录 | 成功状态码 |
| --- | --- | --- | --- | --- |
| 注册 | POST | `/api/v1/auth/register` | 否 | 201 |
| 登录 | POST | `/api/v1/auth/login` | 否 | 200 |
| 获取个人信息 | GET | `/api/v1/profile` | 是 | 200 |
| 创建任务 | POST | `/api/v1/tasks` | 是 | 201 |
| 任务列表 | GET | `/api/v1/tasks` | 是 | 200 |
| 任务详情 | GET | `/api/v1/tasks/:id` | 是 | 200 |
| 更新任务 | PUT | `/api/v1/tasks/:id` | 是 | 200 |
| 修改状态 | PATCH | `/api/v1/tasks/:id/status` | 是 | 200 |
| 删除任务 | DELETE | `/api/v1/tasks/:id` | 是 | 200 或 204 |

如果你采用 `204 No Content` 删除成功，就不要返回响应体。学习阶段为了统一响应，也可以用 `200`。

## 十二、错误场景表

| 场景 | HTTP 状态码 | 业务 code | 说明 |
| --- | --- | --- | --- |
| 请求参数错误 | 400 | 40001 | JSON 格式错、字段缺失、枚举非法 |
| 未登录 | 401 | 40101 | 没有 token 或 token 无效 |
| 无权限 | 403 | 40301 | 已登录但无权访问 |
| 资源不存在 | 404 | 40401 | 用户或任务不存在 |
| 邮箱重复 | 409 | 40901 | 注册邮箱已存在 |
| 登录失败过多 | 429 | 42901 | Redis 限制登录尝试 |
| 系统错误 | 500 | 50001 | 数据库异常等内部错误 |

设计接口时先列错误场景，后面写 handler 会更稳。

## 十三、权限规则

任务接口都必须满足：

```text
当前登录用户 ID == task.user_id
```

查询任务详情：

```sql
SELECT * FROM tasks WHERE id = ? AND user_id = ? AND deleted_at IS NULL;
```

更新任务：

```sql
UPDATE tasks SET title = ?, description = ?
WHERE id = ? AND user_id = ? AND deleted_at IS NULL;
```

删除任务：

```sql
UPDATE tasks SET deleted_at = now()
WHERE id = ? AND user_id = ? AND deleted_at IS NULL;
```

不要只按任务 ID 操作。否则用户 B 可能修改用户 A 的任务。
