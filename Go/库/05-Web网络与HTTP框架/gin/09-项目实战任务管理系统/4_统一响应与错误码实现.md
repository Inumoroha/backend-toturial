# 4. 统一响应与错误码实现

本节目标：在项目中实现统一响应包，后续所有 handler 都使用同一套成功和失败格式。

---

## 一、为什么要统一响应

如果每个接口都随意返回：

```json
{"message":"ok"}
```

或：

```json
{"error":"user not found"}
```

前端和调用方就很难统一处理。

统一格式建议：

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

失败：

```json
{
  "code": 40401,
  "message": "resource not found"
}
```

---

## 二、创建 response 包

`internal/response/response.go`：

```go
package response

import (
    "net/http"

    "github.com/gin-gonic/gin"
)

type Body struct {
    Code    int         `json:"code"`
    Message string      `json:"message"`
    Data    interface{} `json:"data,omitempty"`
}

type PageResult struct {
    Items    interface{} `json:"items"`
    Total    int64       `json:"total"`
    Page     int         `json:"page"`
    PageSize int         `json:"page_size"`
}

func OK(c *gin.Context, data interface{}) {
    c.JSON(http.StatusOK, Body{
        Code:    0,
        Message: "ok",
        Data:    data,
    })
}

func Created(c *gin.Context, data interface{}) {
    c.JSON(http.StatusCreated, Body{
        Code:    0,
        Message: "ok",
        Data:    data,
    })
}

func Fail(c *gin.Context, httpStatus int, code int, message string) {
    c.JSON(httpStatus, Body{
        Code:    code,
        Message: message,
    })
}
```

## 三、错误码常量

可以增加：

```go
const (
    CodeOK              = 0
    CodeInvalidRequest  = 40001
    CodeUnauthorized    = 40101
    CodeForbidden       = 40301
    CodeNotFound        = 40401
    CodeConflict        = 40901
    CodeTooManyRequests = 42901
    CodeInternalError   = 50001
)
```

然后使用：

```go
response.Fail(c, http.StatusBadRequest, response.CodeInvalidRequest, "invalid request")
```

这样比到处写数字更清楚。

---

## 四、404 和 405 统一处理

路由中：

```go
r.NoRoute(func(c *gin.Context) {
    response.Fail(c, http.StatusNotFound, response.CodeNotFound, "route not found")
})

r.NoMethod(func(c *gin.Context) {
    response.Fail(c, http.StatusMethodNotAllowed, response.CodeInvalidRequest, "method not allowed")
})
```

启用 405：

```go
r.HandleMethodNotAllowed = true
```

---

## 五、内部错误不要直接暴露

不要这样：

```go
c.JSON(500, gin.H{"message": err.Error()})
```

原因：

- 可能泄露数据库结构。
- 可能泄露文件路径。
- 前端也不需要这些内部细节。

对外返回：

```go
response.Fail(c, http.StatusInternalServerError, response.CodeInternalError, "internal server error")
```

详细错误写日志。

## 八、任务系统错误映射表

在任务管理系统里，建议先把常见错误映射固定下来。

| 场景 | HTTP 状态码 | 业务 code | message |
| --- | --- | --- | --- |
| JSON 格式错误 | 400 | 40001 | invalid request |
| 参数校验失败 | 400 | 40001 | invalid request |
| 未登录 | 401 | 40101 | unauthorized |
| token 过期 | 401 | 40101 | unauthorized |
| 无权访问任务 | 404 或 403 | 40401 或 40301 | task not found |
| 邮箱已存在 | 409 | 40901 | email already exists |
| 登录失败过多 | 429 | 42901 | too many requests |
| 数据库异常 | 500 | 50001 | internal server error |

学习项目推荐对“访问别人任务”返回 `404`，避免告诉调用方这个任务是否存在。

## 九、handler 中的错误处理模板

handler 不应该每个接口随便写错误响应。可以形成统一模板：

```go
func (h *TaskHandler) GetTask(c *gin.Context) {
    taskID, err := strconv.ParseUint(c.Param("id"), 10, 64)
    if err != nil {
        response.Fail(c, http.StatusBadRequest, response.CodeInvalidRequest, "invalid task id")
        return
    }

    userID := middleware.MustGetUserID(c)

    task, err := h.taskService.GetByID(c.Request.Context(), userID, uint(taskID))
    if err != nil {
        switch {
        case errors.Is(err, service.ErrTaskNotFound):
            response.Fail(c, http.StatusNotFound, response.CodeNotFound, "task not found")
        default:
            response.Fail(c, http.StatusInternalServerError, response.CodeInternalError, "internal server error")
        }
        return
    }

    response.OK(c, task)
}
```

这个模板清楚分成：

```text
解析参数
获取当前用户
调用 service
错误转换
成功响应
```

## 十、分页响应在任务列表中的使用

任务列表返回：

```go
response.OK(c, response.PageResult{
    Items:    tasks,
    Total:    total,
    Page:     req.Page,
    PageSize: req.PageSize,
})
```

响应：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "items": [],
    "total": 0,
    "page": 1,
    "page_size": 10
  }
}
```

不要让任务列表一会儿返回数组，一会儿返回分页对象。前端会很难处理。

## 十一、统一响应自查命令

搜索项目：

```bash
rg "c.JSON|gin.H" internal
```

不是所有结果都要删除，但你要逐个判断：

- handler 是否应该改用 `response.OK`。
- 错误是否应该改用 `response.Fail`。
- 分页是否应该改用 `PageResult`。
- 健康检查是否允许简单返回。

第10阶段完成时，业务接口不应该散落很多临时响应结构。

## 十三、最终检查清单

- [ ] 所有成功响应都有 `code=0`。
- [ ] 所有失败响应都有业务 code。
- [ ] 不直接返回 `err.Error()`。
- [ ] 404 路由不存在是统一 JSON。
- [ ] 405 方法不允许是统一 JSON。
- [ ] 任务列表使用分页响应。
- [ ] 登录认证失败返回 `40101`。
- [ ] 访问不存在或无权访问任务返回稳定错误。

这一步做好，后续写注册、登录、任务接口时才不会每个接口各写一套响应。
