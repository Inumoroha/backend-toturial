# 7. 本阶段综合实践：工程化收口

本节目标：把本阶段学到的配置、日志、测试、Swagger 和 Makefile 整合到一个可交付的 Gin 项目中。

这一节不是学习新知识，而是做收口。你要把前面零散完成的内容整理成一个别人能拉下来、能启动、能测试、能看文档的项目。

---

## 一、最终目录检查

你的项目至少应该包含：

```text
gin-user-api/
├── cmd/
│   └── server/
│       └── main.go
├── config/
│   ├── config.yaml
│   ├── config.local.yaml
│   └── config.test.yaml
├── docs/
│   ├── docs.go
│   ├── swagger.json
│   └── swagger.yaml
├── internal/
│   ├── config/
│   ├── handler/
│   ├── middleware/
│   ├── repository/
│   ├── response/
│   └── service/
├── pkg/
│   └── logger/
├── Makefile
├── README.md
├── go.mod
└── go.sum
```

如果某个目录暂时没有内容，不要为了凑结构强行创建。目录应该服务于实际代码。

---

## 二、任务一：配置收口

检查代码中是否还存在硬编码配置。

重点搜索：

```bash
rg "8080|root:|password|secret|localhost" .
```

这些字符串不一定全部错误，但需要逐个判断：

- 示例配置中可以出现。
- 测试文件中可以出现。
- README 中可以出现。
- 业务代码中不应该写死生产敏感信息。

main 中应该类似：

```go
cfg, err := config.Load("config/config.local.yaml")
if err != nil {
    log.Fatalf("load config failed: %v", err)
}
```

更进一步，可以用启动参数指定配置：

```go
configPath := flag.String("config", "config/config.local.yaml", "config file path")
flag.Parse()

cfg, err := config.Load(*configPath)
```

这样启动时可以写：

```bash
go run ./cmd/server -config=config/config.test.yaml
```

## 三、任务二：日志收口

确认项目启动时输出关键日志：

```go
logger.Info("server starting",
    zap.String("env", cfg.App.Env),
    zap.Int("port", cfg.Server.Port),
)
```

确认请求中间件已经注册：

```go
r.Use(middleware.RequestID())
r.Use(middleware.AccessLog(logger))
r.Use(gin.Recovery())
```

发起请求：

```bash
curl -i http://localhost:8080/ping
```

检查：

- 响应头有 `X-Request-ID`。
- 控制台有请求日志。
- 日志中有 method、path、status、latency。
- 请求不存在接口时也有日志。

## 四、任务三：测试收口

至少补齐：

```text
handler:
- ping 成功
- 注册成功
- 注册参数错误
- profile 未登录

service:
- 注册成功
- 邮箱已存在
- repository 创建失败
- 登录密码错误
```

运行：

```bash
go test ./...
```

再运行覆盖率：

```bash
go test ./... -cover
```

学习阶段不必盲目追求 90% 覆盖率，但核心业务分支要有测试。

---

## 五、任务四：Swagger 收口

执行：

```bash
swag init -g cmd/server/main.go -o docs
```

启动服务后访问：

```text
http://localhost:8080/swagger/index.html
```

检查：

- 项目标题正确。
- BasePath 正确。
- 用户相关接口出现在文档中。
- 请求体字段有说明或示例。
- 错误响应有说明。
- 需要登录的接口带认证标记。

## 六、任务五：Makefile 收口

推荐 Makefile：

```makefile
.PHONY: run test test-cover swagger tidy

run:
	go run ./cmd/server -config=config/config.local.yaml

test:
	go test ./...

test-cover:
	go test ./... -cover

swagger:
	swag init -g cmd/server/main.go -o docs

tidy:
	go mod tidy
```

验证：

```bash
make test
make swagger
```

如果你在 Windows 上暂时没有 make，也要把这些命令写进 README。

## 八、完整验收流程

按顺序执行：

```bash
go mod tidy
go test ./...
swag init -g cmd/server/main.go -o docs
go run ./cmd/server -config=config/config.local.yaml
```

然后打开另一个终端：

```bash
curl -i http://localhost:8080/ping
curl -i http://localhost:8080/swagger/index.html
```

如果你使用浏览器访问 Swagger，也可以直接打开：

```text
http://localhost:8080/swagger/index.html
```

## 九、常见问题

### 1. 测试通过但启动失败

检查配置文件路径、数据库是否启动、DSN 是否正确。

### 2. Swagger 页面 404

检查是否注册了：

```go
r.GET("/swagger/*any", ginSwagger.WrapHandler(swaggerFiles.Handler))
```

也要检查是否匿名导入 docs：

```go
_ "gin-user-api/docs"
```

### 3. Makefile 执行失败

检查命令前是否是 Tab。Makefile 对缩进非常敏感。

### 4. 日志里没有请求记录

检查请求日志中间件是否注册在业务路由之前。
