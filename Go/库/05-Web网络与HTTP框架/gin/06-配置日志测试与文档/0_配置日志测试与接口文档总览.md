# 0. 配置、日志、测试与接口文档总览

第7阶段开始，你的 Gin 项目不再只是“能跑起来”的练习代码，而要逐步变成一个别人也能启动、测试、排错和阅读的工程项目。

前面几个阶段我们已经做过路由、参数绑定、统一响应、中间件、JWT、分层结构和数据库持久化。本阶段要解决的是工程化收口问题：配置从哪里来，日志怎么看，测试怎么写，接口文档怎么生成，项目命令怎么统一。

---

## 一、本阶段学习目标

学完本阶段后，你应该能做到：

- 使用 Viper 加载配置文件，并支持环境变量覆盖。
- 区分本地环境、测试环境和生产环境的配置差异。
- 使用 zap 输出结构化日志。
- 编写请求日志中间件，记录 method、path、status、latency、request_id。
- 使用 `httptest` 测试 Gin handler。
- 使用表格驱动测试 service 业务逻辑。
- 使用 Swagger 注释生成接口文档。
- 使用 Makefile 统一启动、测试、文档生成等常用命令。
- 为项目补充一个能让新同事快速启动的 README。

---

## 二、为什么要在这个阶段学工程化

很多初学者写 Gin 项目时，会把精力放在接口能不能返回正确 JSON 上。这当然重要，但后端工程师真正进入团队协作后，会遇到更多问题：

- 项目换一台电脑启动不了。
- 数据库地址、JWT 密钥写死在代码里。
- 线上出错时，只看到一串 `500`，不知道是哪一个请求失败。
- 接口改了，但文档没有同步。
- 测试全靠手动点 Postman。
- 每个人启动项目的命令都不一样。

这些问题不解决，项目越写越乱。工程化的作用，就是把“靠记忆和手动操作”的部分沉淀成规范、配置、命令和测试。

---

## 三、本阶段推荐项目结构

建议在第5、第6阶段分层项目基础上继续扩展：

```text
gin-user-api/
├── cmd/
│   └── server/
│       └── main.go
├── config/
│   ├── config.yaml
│   ├── config.local.yaml
│   └── config.test.yaml
├── docs/
│   └── swagger 生成文件
├── internal/
│   ├── app/
│   ├── config/
│   ├── handler/
│   ├── middleware/
│   ├── repository/
│   ├── response/
│   └── service/
├── pkg/
│   └── logger/
├── Makefile
├── README.md
├── go.mod
└── go.sum
```

其中：

- `config/`：放 YAML 配置文件。
- `internal/config/`：负责读取配置并转换为 Go 结构体。
- `pkg/logger/`：封装 zap 初始化。
- `internal/middleware/`：放请求日志、Recovery、RequestID 等中间件。
- `docs/`：Swagger 生成物，不手写。
- `Makefile`：统一项目常用命令。

## 七、本阶段验收标准

完成本阶段后，请逐项检查：

- 项目中不存在硬编码的数据库 DSN、JWT Secret、服务端口。
- `config.local.yaml` 可以用于本地开发。
- `config.test.yaml` 可以用于测试环境。
- zap 日志能输出 JSON 格式日志。
- 请求日志包含 `request_id`。
- handler 测试能覆盖成功、参数错误、未登录三类情况。
- service 测试能覆盖正常分支和异常分支。
- Swagger 文档可以通过浏览器访问。
- README 中的新手启动步骤不超过 5 分钟能跑通。

只要这些检查都能通过，你的 Gin 项目就已经从“课堂练习”向“可维护后端工程”迈出了一大步。

## 融合补充

+### 06-01 配置、环境、日志与 Swagger

#### 强类型配置

配置优先级推荐为：默认值 < YAML 文件 < 环境变量 < 命令行。示例文件只提交安全占位值：

```yaml
# configs/config.example.yaml
server:
  addr: ":8080"
database:
  dsn: "user:password@tcp(mysql:3306)/task_api?parseTime=true&charset=utf8mb4&loc=Local"
jwt:
  secret: "replace-in-production"
redis:
  addr: "redis:6379"
```

Viper 可以读取文件与环境变量，但不要让任意包直接 `viper.GetString`。启动时将配置解析、校验成 `Config` 结构体，再通过构造函数注入。生产密钥、`.env`、真实 DSN 和私钥绝不能提交到仓库。

#### 结构化日志

使用 zap 或 slog 记录 `request_id`、路径、方法、状态、耗时、用户 ID 和错误类别。日志服务的是排错与审计，所以字段应可查询、可聚合：

```go
logger.Info("task created",
    zap.String("request_id", requestID),
    zap.Uint("user_id", userID),
    zap.Uint("task_id", task.ID),
)
```

不要记录密码、Authorization、完整 body、token、DSN 或敏感个人数据。错误日志保留内部 cause，响应只返回稳定的通用信息。

#### 接口文档

选择 Swagger/OpenAPI 注释生成或直接维护 OpenAPI YAML 均可，关键是文档与代码同时变化。至少写清：认证方式、端点、参数、成功响应、错误码、分页格式和可复制的请求示例。发布前用真实请求检查文档，不要只相信生成命令成功。

**验收：**无需改代码即可用环境变量切换配置；每个错误响应可用 request ID 找到日志；API 文档覆盖注册、登录、任务 CRUD 和通用错误格式。

+### 06-02 Handler、Service 测试与 Makefile

#### 测试分层

- Handler：用 `httptest` 验证路由、绑定、状态码和响应格式。
- Service：表格驱动 + fake repository，验证业务规则、权限和状态转换。
- Repository：连接隔离测试库，验证 SQL、索引、软删除和事务。
- 集成测试：启动真实应用与依赖，验证注册、登录、创建任务、查询任务完整链路。

Handler 示例：

```go
func TestCreateUser_InvalidBody(t *testing.T) {
    gin.SetMode(gin.TestMode)
    r := gin.New()
    r.POST("/users", handler.Create)
    req := httptest.NewRequest(http.MethodPost, "/users", strings.NewReader(`{"email":"bad"}`))
    req.Header.Set("Content-Type", "application/json")
    rec := httptest.NewRecorder()
    r.ServeHTTP(rec, req)
    if rec.Code != http.StatusBadRequest { t.Fatalf("got %d", rec.Code) }
}
```

Service 测试重点覆盖：重复邮箱、错误密码、访问他人任务、非法状态转换、分页边界和数据库错误映射。测试名称表达行为，而不是实现细节；每个用例独立初始化数据，避免执行顺序依赖。

#### 常用命令

```makefile
run:
	go run ./cmd/api
test:
	go test ./...
test-race:
	go test -race ./...
lint:
	golangci-lint run
build:
	go build ./cmd/api
```

`-race` 可发现 map、缓存或全局状态的并发访问问题。它不是生产压力测试，也不能代替集成测试。把 `test`、`lint`、构建和接口文档校验接入 CI，阻止明显回归进入主分支。

**验收：**`go test ./...` 与 `go test -race ./...` 通过；失败测试能明确指出行为；Makefile 在新机器上可复现常用操作。
