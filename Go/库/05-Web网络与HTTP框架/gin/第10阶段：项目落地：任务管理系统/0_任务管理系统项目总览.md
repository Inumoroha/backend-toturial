# 0. 任务管理系统项目总览

本阶段目标：完成一个可以作为 Go 后端入门作品的 Gin 任务管理系统。这个项目会把前面学到的路由、参数校验、项目分层、GORM、JWT、Redis、测试和 Docker 部署串起来。

项目不追求复杂页面，也不做前端。我们只做后端 API。

---

## 一、项目要解决什么问题

用户可以注册账号，登录后管理自己的任务。

核心功能：

```text
注册
登录
获取个人信息
创建任务
查询任务列表
查询任务详情
更新任务
修改任务状态
删除任务
```

后端重点不只是“功能能跑”，还要训练：

- API 设计。
- 分层结构。
- 参数校验。
- 统一响应。
- 错误处理。
- 数据库操作。
- 登录认证。
- 权限边界。
- 测试和部署。

---

## 二、项目技术栈

建议使用：

```text
Go
Gin
GORM
MySQL 或 PostgreSQL
Redis
JWT
bcrypt
Viper
zap
Docker Compose
```

数据库可以二选一。本教程示例更偏向 MySQL + GORM，因为 Go/Gin 学习者常见组合是 Gin + GORM + MySQL。

如果你更想练 PostgreSQL，也可以替换 GORM driver。

## 三、接口清单

认证接口：

```text
POST /api/v1/auth/register
POST /api/v1/auth/login
```

用户接口：

```text
GET /api/v1/profile
PUT /api/v1/profile
```

任务接口：

```text
POST   /api/v1/tasks
GET    /api/v1/tasks
GET    /api/v1/tasks/:id
PUT    /api/v1/tasks/:id
PATCH  /api/v1/tasks/:id/status
DELETE /api/v1/tasks/:id
```

健康检查：

```text
GET /healthz
```

## 四、本阶段拆分

本阶段建议按以下小节完成：

1. 需求分析与接口设计。
2. 项目目录与配置文件。
3. 数据表设计与 GORM 模型。
4. 统一响应与错误码。
5. 注册接口。
6. 登录接口与 JWT。
7. 认证中间件与 profile。
8. 任务创建接口。
9. 任务列表和详情。
10. 任务更新、状态修改和删除。
11. Redis 缓存与登录失败限制。
12. 测试、Docker 部署与项目收尾。

不要一次写完整项目。每完成一节，都要能运行和验证。

---

## 五、项目完成标准

项目完成后应该具备：

- `docker compose up -d` 可以启动依赖。
- `go run ./cmd/server` 可以启动服务。
- 注册登录流程正常。
- 私有接口必须带 JWT。
- 用户只能访问自己的任务。
- 参数错误返回统一格式。
- 数据库中能看到 users 和 tasks 数据。
- 至少有关键 service 和 handler 测试。
- README 写清楚启动方式和接口示例。

## 六、项目里程碑

不要一次性把所有功能都写完。建议按里程碑推进：

```text
M1：项目能启动
M2：用户注册能写入数据库
M3：登录能返回 JWT
M4：认证中间件能保护 profile
M5：任务 CRUD 能完成
M6：用户只能访问自己的任务
M7：测试、Redis、Docker 收口
```

每个里程碑都要能运行和验证。

例如 M2 完成标准：

```text
POST /api/v1/auth/register 返回成功。
users 表中能看到用户。
重复邮箱返回 409。
响应中不包含 password_hash。
```

M5 完成标准：

```text
登录后能创建任务。
能查询自己的任务列表。
能查询任务详情。
能更新任务。
能删除任务。
```

这样拆分的好处是：你随时知道项目当前卡在哪一步，而不是写了很多代码却不知道哪里坏了。

## 七、项目边界

本项目做：

```text
后端 API。
JWT 登录认证。
用户自己的任务管理。
数据库持久化。
基础 Redis 能力。
基础测试和 Docker 部署。
```

本项目暂时不做：

```text
前端页面。
多人协作任务。
任务评论。
任务附件。
复杂权限系统。
消息通知。
支付或外部开放平台。
```

明确边界很重要。初学项目最怕一边学一边不断加需求，最后主线散掉。

## 八、推荐目录结构

最终项目可以组织为：

```text
task-api/
├── cmd/
│   └── server/
│       └── main.go
├── config/
│   └── config.yaml
├── internal/
│   ├── config/
│   ├── database/
│   ├── handler/
│   ├── middleware/
│   ├── model/
│   ├── repository/
│   ├── response/
│   ├── router/
│   └── service/
├── docs/
├── Dockerfile
├── docker-compose.yml
├── Makefile
├── README.md
├── go.mod
└── go.sum
```

职责边界：

- `handler`：处理 HTTP 输入输出。
- `service`：处理业务规则。
- `repository`：处理数据库访问。
- `middleware`：认证、日志、CORS、Recovery。
- `response`：统一响应和错误码。
- `router`：集中注册路由。
- `cmd/server`：程序入口和依赖组装。

## 九、主线数据流

创建任务的数据流：

```text
HTTP POST /api/v1/tasks
  ↓
JWT 中间件解析当前用户
  ↓
handler 绑定 JSON 并校验参数
  ↓
service 设置默认状态并检查业务规则
  ↓
repository 写入 tasks 表
  ↓
handler 返回统一响应
```

查询任务详情的数据流：

```text
HTTP GET /api/v1/tasks/:id
  ↓
JWT 中间件解析当前用户
  ↓
handler 解析路径参数
  ↓
service 调用 repository
  ↓
repository 使用 id + user_id 查询
  ↓
返回任务或 404
```

注意这里的 `user_id`。它是权限边界的核心。

## 十、学习方法建议

每一小节都按这个节奏：

```text
1. 先读需求。
2. 写请求和响应示例。
3. 写 DTO。
4. 写 service 规则。
5. 写 repository。
6. 写 handler。
7. 用 requests.http 测。
8. 记录问题。
```

不要从数据库表直接开始，也不要从 handler 乱写。真实项目里，接口契约和业务规则应该先想清楚。
