# 09-02 部署排错、环境变量与生产清单

本节目标：补足部署时最容易出问题的地方。很多项目本地能跑，放进容器就不行，通常是配置、网络、文件路径和启动顺序出了问题。

## 一、配置优先级

推荐配置优先级：

```text
环境变量 > 配置文件默认值 > 代码默认值
```

例如：

```bash
APP_SERVER_PORT=8080
APP_DATABASE_DSN=...
APP_JWT_SECRET=...
```

敏感信息优先用环境变量：

- 数据库密码。
- JWT secret。
- 第三方服务密钥。

不要提交到 Git。

## 二、Docker Compose 环境变量

示例：

```yaml
services:
  app:
    build: .
    environment:
      APP_SERVER_PORT: "8080"
      APP_DATABASE_DSN: "root:password@tcp(mysql:3306)/gin_learning?charset=utf8mb4&parseTime=True&loc=Local"
      APP_REDIS_ADDR: "redis:6379"
      APP_JWT_SECRET: "change-me"
```

如果配置文件里还是 `127.0.0.1:3306`，容器里的 app 会连自己容器的 3306，而不是 mysql 容器。

## 三、启动顺序和重试

`depends_on` 不代表 MySQL 已经准备好接收连接。

应用启动时可以做简单重试：

```go
for i := 0; i < 10; i++ {
    db, err := gorm.Open(mysql.Open(dsn), &gorm.Config{})
    if err == nil {
        return db
    }
    time.Sleep(2 * time.Second)
}
log.Fatal("database not ready")
```

学习阶段这样足够。生产环境可以使用更完善的健康检查和编排系统。

## 四、容器日志

查看 app 日志：

```bash
docker compose logs -f app
```

查看 MySQL 日志：

```bash
docker compose logs -f mysql
```

进入容器：

```bash
docker compose exec app sh
```

检查环境变量：

```bash
env
```

检查配置文件：

```bash
ls -R /app
```

## 五、生产模式

Gin 生产模式：

```bash
GIN_MODE=release
```

Docker Compose：

```yaml
environment:
  GIN_MODE: "release"
```

开发模式会输出更详细的调试信息，生产环境不建议使用。

## 六、上传文件持久化

如果头像上传到容器内目录，容器删除后文件也可能丢失。

开发阶段可以挂载 volume：

```yaml
services:
  app:
    volumes:
      - ./uploads:/app/uploads
```

生产环境更推荐对象存储，例如 S3、OSS、COS。

## 七、上线前手动验证流程

上线前按顺序验证：

1. `GET /healthz` 是否成功。
2. 注册是否成功。
3. 登录是否返回 token。
4. 携带 token 访问私有接口是否成功。
5. 不带 token 是否返回 401。
6. 创建任务是否成功。
7. 查询任务列表是否成功。
8. 重启容器后数据是否还在。
9. 日志是否能看到错误。
10. 配置是否没有泄露敏感信息。

## 八、常见部署问题表

```text
现象：app 容器启动失败
排查：docker compose logs -f app

现象：数据库连接失败
排查：DSN、mysql 服务名、端口、密码、数据库名

现象：接口 404
排查：路径、API 前缀、Nginx 转发路径

现象：上传后文件访问不到
排查：静态路由、容器路径、volume 挂载

现象：登录后 token 无效
排查：JWT secret 是否和签发时一致、token 是否过期
```

## 九、本节练习

完成：
