# 8. 生产上线检查清单

本节目标：在 Gin 项目真正部署前，提供一份可以逐项核对的生产上线检查清单。

前面几节已经讲了 Dockerfile、Docker Compose、Nginx、环境变量、健康检查和部署排错。本节把它们整理成上线前最后一道闸门。

上线检查的意义不是让流程变复杂，而是避免这些低级事故：

- 生产 JWT Secret 仍然是 `change-me`。
- app 容器里还连着本地 `127.0.0.1:3306`。
- `.env` 被提交到仓库。
- 上传文件保存在容器内部，重启后丢失。
- Nginx 返回 502，但没人知道从哪里查。
- `/readyz` 没检查数据库和 Redis。

---

## 一、代码与测试检查

上线前先确认代码本身没有明显问题：

```bash
go mod tidy
go test ./...
go build -o bin/server ./cmd/server
```

检查点：

- [ ] `go test ./...` 通过。
- [ ] `go mod tidy` 后没有异常依赖变更。
- [ ] 本地可以编译。
- [ ] 核心接口手动请求通过。
- [ ] Swagger 文档和真实接口路径一致。

如果测试失败，不要进入部署阶段。部署不会修复代码问题，只会让问题更难排查。

---

## 二、Dockerfile 检查

Dockerfile 应该使用多阶段构建：

```dockerfile
FROM golang:1.22-alpine AS builder
...
RUN CGO_ENABLED=0 GOOS=linux GOARCH=amd64 go build -o /app/server ./cmd/server

FROM alpine:3.20
...
COPY --from=builder /app/server /app/server
```

检查点：

- [ ] 使用多阶段构建。
- [ ] 运行镜像不包含完整 Go 编译环境。
- [ ] `go.mod`、`go.sum` 先复制，能利用缓存。
- [ ] 镜像中没有 `.env`。
- [ ] 应用监听 `:8080`，不是 `127.0.0.1:8080`。
- [ ] 日志输出到 stdout/stderr。

构建验证：

```bash
docker build -t gin-user-api:prod .
docker run --rm -p 8080:8080 gin-user-api:prod
```

---

## 三、环境变量检查

真实 `.env` 不提交，`.env.example` 提供示例。

`.gitignore` 应包含：

```gitignore
.env
.env.*
!.env.example
```

检查点：

- [ ] `.env` 没有提交。
- [ ] `.env.example` 包含必要变量。
- [ ] `APP_JWT_SECRET` 已更换为强随机值。
- [ ] 数据库密码不是默认值。
- [ ] `APP_SERVER_MODE=release`。
- [ ] `APP_DATABASE_DSN` 使用 `mysql:3306`。
- [ ] `APP_REDIS_ADDR=redis:6379`。
- [ ] 没有真实密钥写在 `config.yaml` 中。

验证：

```bash
docker compose config
docker compose exec app env | grep APP_
```

---

## 四、Docker Compose 检查

Compose 至少包含：

```text
app
mysql
redis
nginx
```

检查点：

- [ ] app 通过服务名连接 mysql。
- [ ] app 通过服务名连接 redis。
- [ ] mysql 使用 volume。
- [ ] redis 根据需要使用 volume。
- [ ] uploads 使用 volume。
- [ ] app 依赖 mysql、redis 健康状态。
- [ ] Nginx 依赖 app。
- [ ] 对外只暴露必要端口。

启动验证：

```bash
docker compose up -d --build
docker compose ps
docker compose logs app
```

---

## 五、数据库检查

检查点：

- [ ] 数据库名正确。
- [ ] 用户表已创建。
- [ ] 唯一索引生效。
- [ ] 软删除字段存在。
- [ ] 时间字段正常。
- [ ] 生产环境不要随意依赖 AutoMigrate 改表。

命令：

```bash
docker compose exec mysql mysql -uroot -p
```

SQL：

```sql
USE gin_user_api;
SHOW TABLES;
SHOW INDEX FROM users;
SELECT COUNT(*) FROM users;
```

生产环境建议使用迁移文件管理数据库结构，而不是每次启动都自动迁移。

---

## 六、Redis 检查

检查点：

- [ ] Redis 可以 ping 通。
- [ ] app 使用 `redis:6379`。
- [ ] 登录失败限流 key 有 TTL。
- [ ] 用户详情缓存 key 有 TTL。
- [ ] Redis 不可用时策略明确。

命令：

```bash
docker compose exec redis redis-cli ping
docker compose exec redis redis-cli keys "*"
```

生产中慎用 `KEYS *`，数据多时可能阻塞 Redis。学习环境可以用来观察。

---

## 七、Nginx 检查

检查 `nginx.conf`：

```nginx
upstream gin_app {
    server app:8080;
}
```

检查点：

- [ ] upstream 使用 `app:8080`。
- [ ] 不是 `localhost:8080`。
- [ ] 设置了代理头。
- [ ] 设置了合适的 `client_max_body_size`。
- [ ] 502 时能查看 Nginx 和 app 日志。
- [ ] 如果部署公网，HTTPS 策略明确。

验证：

```bash
curl http://localhost/healthz
curl http://localhost/readyz
curl http://localhost/ping
```

---

## 八、健康检查

检查点：

- [ ] `/healthz` 只检查进程存活。
- [ ] `/readyz` 检查数据库。
- [ ] `/readyz` 检查 Redis。
- [ ] 依赖异常时 `/readyz` 返回 `503`。
- [ ] 健康检查不返回敏感配置。

验证：

```bash
curl -i http://localhost/healthz
curl -i http://localhost/readyz
```

临时停止 Redis：

```bash
docker compose stop redis
curl -i http://localhost/readyz
docker compose start redis
```

观察 `/readyz` 是否正确变化。

---

## 九、上传文件检查

检查点：

- [ ] 上传目录使用 volume。
- [ ] Nginx 请求体大小限制合理。
- [ ] Gin handler 限制文件大小。
- [ ] Gin handler 限制扩展名。
- [ ] 保存文件名由服务端生成。
- [ ] 静态访问路径可用。
- [ ] 容器重启后上传文件不丢。

验证：

```bash
curl -X POST http://localhost/api/v1/profile/avatar \
  -H "Authorization: Bearer your-token" \
  -F "file=@avatar.png"
```

重启：

```bash
docker compose restart app
```

再次访问返回的头像 URL。

---

## 十、日志与排错检查

检查点：

- [ ] app 日志能看到启动配置摘要。
- [ ] 请求日志包含 request_id。
- [ ] 4xx、5xx 有日志。
- [ ] panic 被 Recovery 捕获。
- [ ] Nginx 日志可查看。
- [ ] 数据库连接失败有清晰错误。
- [ ] Redis 连接失败有清晰错误。

命令：

```bash
docker compose logs -f app
docker compose logs -f nginx
```

不要把明文密码、完整 token、身份证、银行卡等敏感信息写进日志。

---

## 十一、回滚与备份检查

上线前要想清楚失败后怎么办。

检查点：

- [ ] 当前镜像有明确 tag。
- [ ] 上一个稳定镜像还在。
- [ ] 数据库变更有备份。
- [ ] 上传文件有持久化或备份。
- [ ] 回滚命令已写进 README 或部署文档。

示例：

```bash
docker compose pull
docker compose up -d
```

如果要回滚到旧镜像：

```yaml
app:
  image: gin-user-api:previous
```

然后：

```bash
docker compose up -d
```

数据库变更比应用回滚更危险，生产中必须谨慎处理迁移脚本。

---

## 十二、最终上线流程

推荐顺序：

```text
1. 本地测试通过
2. 构建镜像
3. 准备环境变量
4. 启动 Compose
5. 检查容器状态
6. 检查 healthz / readyz
7. 检查 Nginx 代理
8. 检查核心业务接口
9. 检查日志
10. 记录版本和回滚方案
```

对应命令：

```bash
go test ./...
docker compose up -d --build
docker compose ps
curl http://localhost/healthz
curl http://localhost/readyz
curl http://localhost/ping
docker compose logs --tail=100 app
```

## 十三、本节验收

请确认你能回答：

- 为什么生产环境不能把 `.env` 提交到仓库？
- Docker Compose 中为什么要用 `mysql:3306` 而不是 `127.0.0.1:3306`？
- `/healthz` 和 `/readyz` 的区别是什么？
- Nginx 502 应该从哪些日志开始查？
- 上传文件为什么要用 volume？
- Redis 缓存 key 为什么要设置 TTL？
- 生产环境为什么需要回滚方案？

如果这些问题都能回答清楚，第9阶段的部署上线学习才算真正闭环。
