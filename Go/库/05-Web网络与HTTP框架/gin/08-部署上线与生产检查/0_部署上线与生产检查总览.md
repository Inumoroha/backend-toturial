# 0. 部署上线与生产检查总览

第9阶段的目标是把 Gin 项目从“本地能跑”推进到“可以部署、可以检查、可以排错”的状态。

很多后端学习项目停在本地运行：

```bash
go run ./cmd/server
```

这还不够。真正的后端工程师需要知道服务如何容器化、如何配置环境变量、如何和数据库 Redis 一起启动、如何接入 Nginx、如何做健康检查、出现问题时如何定位。

---

## 一、本阶段学习目标

完成本阶段后，你应该能做到：

- 编写适合 Go 程序的多阶段 Dockerfile。
- 使用 Docker Compose 启动 Gin、MySQL、Redis。
- 使用环境变量覆盖生产配置。
- 添加健康检查接口。
- 使用 Nginx 反向代理 Gin 服务。
- 理解容器中 `localhost` 的含义。
- 排查常见部署问题。
- 写出生产上线前检查清单。

---

## 二、为什么部署阶段必须单独学习

代码写得好，不代表能顺利部署。部署阶段会遇到完全不同的问题：

- 本地数据库地址写的是 `127.0.0.1`，容器里却连不上。
- Docker 镜像太大。
- 配置文件带了真实密码。
- 服务启动了，但健康检查失败。
- Nginx 返回 502。
- 静态上传文件容器重启后丢失。
- 日志只在容器里，看不到请求错误。

这些问题不是 Gin 语法问题，而是工程交付问题。

- `nginx.conf`：反向代理配置。
- `README.md`：说明部署和排错步骤。

## 五、部署阶段的基本原则

### 1. 配置和代码分离

不要把生产密码写进代码或镜像。

### 2. 容器只负责运行一个主进程

Gin 应用容器只运行 Gin 服务，不要顺便启动 MySQL 或 Redis。

### 3. 镜像尽量小

Go 项目适合多阶段构建，把编译环境和运行环境分开。

### 4. 健康检查必须明确

服务能启动不代表服务健康。健康检查要能反映依赖状态。

### 5. 日志输出到 stdout/stderr

容器环境下，日志优先输出到标准输出，由容器平台采集。

## 六、本阶段最终验收

完成本阶段后，请确认：

- `docker build` 可以成功构建应用镜像。
- `docker compose up` 可以启动 app、mysql、redis。
- app 可以通过服务名连接 mysql 和 redis。
- `/healthz` 返回服务基础状态。
- `/readyz` 能检查数据库和 Redis。
- Nginx 可以代理访问 Gin 接口。
- 上传目录使用 volume 持久化。
- `.env.example` 存在，但真实 `.env` 不提交。
- README 中有部署和排错命令。

第9阶段完成后，你的 Gin 学习项目就具备了完整后端工程闭环：开发、配置、日志、测试、缓存、限流、上传、部署和排错。

## 融合补充

+### 08-01 Dockerfile 与 Docker Compose 部署

#### 多阶段构建

```dockerfile
FROM golang:1.24-alpine AS build
WORKDIR /src
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -trimpath -ldflags="-s -w" -o /app ./cmd/api

FROM gcr.io/distroless/static-debian12:nonroot
WORKDIR /app
COPY --from=build /app /app/api
USER nonroot:nonroot
EXPOSE 8080
ENTRYPOINT ["/app/api"]
```

使用明确镜像版本，配合 `.dockerignore` 排除 `.git`、本地二进制、`.env`、测试产物和上传文件。若项目需要 CGO、时区数据库或 shell 调试，按实际依赖调整运行时镜像，不要盲目套用最小镜像。

#### Compose 编排

```yaml
services:
  app:
    build: .
    environment:
      APP_DATABASE_DSN: ${APP_DATABASE_DSN}
      APP_JWT_SECRET: ${APP_JWT_SECRET}
      APP_REDIS_ADDR: redis:6379
    depends_on:
      mysql:
        condition: service_healthy
      redis:
        condition: service_healthy
  mysql:
    image: mysql:8.4
    environment:
      MYSQL_DATABASE: task_api
      MYSQL_ROOT_PASSWORD: ${MYSQL_ROOT_PASSWORD}
    healthcheck:
      test: ["CMD-SHELL", "mysqladmin ping -h localhost -p$$MYSQL_ROOT_PASSWORD"]
      interval: 5s
      timeout: 3s
      retries: 20
  redis:
    image: redis:7-alpine
```

在容器内部用 `mysql:3306`、`redis:6379` 等服务名访问依赖，不能使用宿主机 `localhost`。`depends_on` 的健康检查只解决启动先后，不取代应用自身重试、超时与错误处理。

使用 `docker compose up --build` 启动，查看 `docker compose logs -f app`，再在宿主机或 Nginx 容器中发起健康检查和完整 API 验收。

**验收：**空环境能构建和启动；应用容器可以连接 MySQL 与 Redis；所有密钥从环境文件或部署平台注入，未写入镜像和仓库。

+### 08-02 Nginx、健康检查、环境覆盖与生产清单

Nginx 通常负责 TLS、反向代理、上传大小限制和基础访问控制，Gin 监听内部端口。代理配置需要转发 `Host`、`X-Request-ID` 和经过验证的真实客户端信息；不要无条件信任任意来源伪造的 `X-Forwarded-For`。

区分两个探针：

- `/healthz`：进程是否存活，尽量轻量。
- `/readyz`：是否可以接收流量，可检查关键依赖但要设置短超时。

不要用登录、任务列表等业务接口作健康检查。应用应在 release 模式运行，日志中不输出调试数据与敏感值。

#### 上线前清单

- 已备份数据库；迁移已在副本数据演练并有回滚计划。
- TLS、代理超时、请求体限制、可信代理和 CORS 均按实际流量配置。
- JWT 密钥、数据库密码、Redis 凭据由部署环境注入，已轮换默认示例值。
- 注册、登录、鉴权、越权、分页、缓存失效、限流、上传、关闭分别经过真实请求验证。
- 具备错误告警、日志保留、磁盘和连接池监控；容器重启不会丢失数据库数据。

#### 部署排错顺序

1. `docker compose ps` 确认容器和健康状态。
2. 查看应用、MySQL、Redis、Nginx 各自日志的首个错误。
3. 在应用容器内检查环境变量、DNS 和依赖端口。
4. `curl` 应用内部健康检查，再验证 Nginx 的外部路由与 TLS。
5. 最后检查数据库迁移、权限、连接池和慢查询。

**验收：**服务重启时可优雅下线；外部经 Nginx 能访问健康检查和 API；依赖故障时日志和探针能给出足够的定位信息。
