# 5. 环境变量与配置覆盖

本节目标：建立生产部署中的配置管理规则，确保敏感信息不写死、不进镜像、不提交仓库。

第7阶段已经学习过 Viper 配置。本节站在部署角度重新整理：哪些配置可以放 YAML，哪些必须用环境变量，Docker Compose 中如何注入。

---

## 一、配置分层原则

推荐分层：

```text
默认配置：config/config.yaml
本地配置：config/config.local.yaml
测试配置：config/config.test.yaml
生产敏感配置：环境变量
```

代码中只读取配置，不写死具体值。

---

## 二、哪些配置适合环境变量

适合环境变量：

- 数据库 DSN。
- Redis 地址和密码。
- JWT Secret。
- 第三方 API Key。
- 日志级别。
- 运行环境。

不适合直接提交真实值：

```text
root:real-password@tcp(...)
super-secret-jwt-key
cloud-access-key
```

---

## 三、Viper 环境变量映射规则

第7阶段配置过：

```go
v.SetEnvPrefix("APP")
v.SetEnvKeyReplacer(strings.NewReplacer(".", "_"))
v.AutomaticEnv()
```

映射关系：

```text
server.port       -> APP_SERVER_PORT
database.dsn      -> APP_DATABASE_DSN
redis.addr        -> APP_REDIS_ADDR
jwt.secret        -> APP_JWT_SECRET
log.level         -> APP_LOG_LEVEL
security.login_max_fails -> APP_SECURITY_LOGIN_MAX_FAILS
```

命名要稳定，不要一会儿 `GIN_PORT`，一会儿 `APP_SERVER_PORT`。

---

## 四、Compose 中注入环境变量

示例：

```yaml
app:
  environment:
    APP_APP_ENV: production
    APP_SERVER_MODE: release
    APP_SERVER_PORT: 8080
    APP_DATABASE_DSN: ${APP_DATABASE_DSN}
    APP_REDIS_ADDR: redis:6379
    APP_JWT_SECRET: ${APP_JWT_SECRET}
    APP_LOG_LEVEL: info
```

`.env`：

```dotenv
APP_DATABASE_DSN=root:change-me@tcp(mysql:3306)/gin_user_api?charset=utf8mb4&parseTime=True&loc=Local
APP_JWT_SECRET=please-change-this-secret
```

`.env.example` 只放示例值，不放真实密钥。

---

## 五、确认环境变量是否进入容器

执行：

```bash
docker compose exec app sh
```

容器内：

```sh
env | grep APP_
```

你应该能看到：

```text
APP_DATABASE_DSN=...
APP_REDIS_ADDR=redis:6379
APP_JWT_SECRET=...
```

如果没有，检查 Compose 文件和 `.env` 文件是否在同一目录。

---

## 六、不要把 .env 提交到 Git

`.gitignore` 中应包含：

```gitignore
.env
.env.*
!.env.example
```

含义：

- 忽略真实 `.env`。
- 忽略环境专用 `.env.production`。
- 保留 `.env.example`。

如果不小心提交了真实密钥，仅仅删除提交不一定安全。应立即更换密钥。

---

## 七、生产配置检查清单

上线前检查：

```text
JWT Secret 是否足够复杂
数据库密码是否不是默认值
Gin mode 是否为 release
日志级别是否合适
数据库 DSN 是否指向生产库
Redis 地址是否为生产 Redis
Swagger 是否需要关闭或加保护
上传目录是否持久化
```

特别注意 Swagger。内部系统可以保留，公网系统要谨慎暴露。

## 八、常见问题

### 1. 环境变量设置了但程序没读到

检查：

- 前缀是否是 `APP_`。
- key 是否用下划线。
- Viper 是否调用 `AutomaticEnv()`。
- 是否在同一个容器或进程中设置。

### 2. Compose 变量为空

检查 `.env` 是否和 `docker-compose.yml` 在同一目录。执行：

```bash
docker compose config
```

可以查看变量渲染后的结果。

### 3. config.yaml 和环境变量谁优先

在 Viper 中，如果启用 `AutomaticEnv()`，环境变量可以覆盖对应配置。实际行为也取决于读取和绑定方式，建议为关键配置写启动日志确认。

## 九、练习

请完成：

1. 创建 `.env.example`。
2. 确认 `.env` 被 Git 忽略。
3. 用环境变量覆盖数据库 DSN。
4. 用环境变量覆盖 JWT Secret。
5. 进入 app 容器查看 `APP_` 变量。
6. 执行 `docker compose config` 检查最终配置。

## 十、验收标准

完成后确认：

- 真实 `.env` 没有提交。
- `.env.example` 说明了必要配置。
- app 容器内能看到 `APP_` 环境变量。
- Gin 以 release mode 运行。
- 生产敏感信息没有写死在代码或镜像里。
