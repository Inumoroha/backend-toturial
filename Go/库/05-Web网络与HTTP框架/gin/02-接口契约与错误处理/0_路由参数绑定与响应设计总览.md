# 0. 路由、参数绑定与响应设计总览

本阶段目标：在已经能用 Gin 写出基础 CRUD 的前提下，把接口入口做规范。你要学会设计清晰的 RESTful 路由，明确每一种参数应该从哪里读取，使用 binding 做请求校验，并让所有接口返回稳定、统一的 JSON 响应。

如果第二阶段解决的是：

```text
我能用 Gin 写出接口。
```

那么第三阶段要解决的是：

```text
我能写出更像真实项目的接口。
```

真实项目里的接口不能只追求“能跑”。它还应该具备：

- 路径含义清楚。
- 请求参数来源明确。
- 参数错误能被稳定识别。
- 响应格式统一。
- HTTP 状态码使用合理。
- 业务错误码稳定。
- 前端和调用方容易处理。

## 一、本阶段适合什么时候学习

开始本阶段前，建议你已经完成第二阶段的内存版用户 CRUD，并且能独立写出下面这些接口：

```text
GET    /api/v1/users
GET    /api/v1/users/:id
POST   /api/v1/users
PUT    /api/v1/users/:id
DELETE /api/v1/users/:id
```

你还应该知道：

- `c.Param` 用来读取路径参数。
- `c.Query` 用来读取查询参数。
- `c.ShouldBindJSON` 用来读取 JSON 请求体。
- `c.JSON` 用来返回 JSON 响应。

如果这些还不熟，建议先回到第二阶段继续练习。

## 二、本阶段核心问题

本阶段围绕 6 个问题展开。

### 1. 路由怎么设计

例如用户资源应该写成：

```text
GET /api/v1/users
POST /api/v1/users
GET /api/v1/users/:id
```

而不是：

```text
GET /getUsers
POST /createUser
POST /deleteUser
```

路由设计会影响后续项目结构、接口文档和前后端协作。

### 2. 参数从哪里来

常见参数来源：

```text
路径参数：/users/:id
查询参数：?page=1&page_size=10
JSON 请求体：POST /users 的 body
Header：Authorization、Content-Type
```

你需要明确每类参数适合什么场景。

### 3. 参数怎么校验

创建用户时，不能允许这种请求进入业务层：

```json
{
  "username": "",
  "email": "abc",
  "age": 200
}
```

Gin 的 binding 能帮你在 handler 入口处拒绝明显非法的请求。

### 4. 错误怎么返回

不要一会儿返回：

```json
{"error":"bad request"}
```

一会儿返回：

```json
{"message":"user not found"}
```

应该统一成稳定格式。

### 5. 状态码怎么选

不同错误应该有不同 HTTP 状态码：

```text
400 参数错误
401 未登录
403 无权限
404 资源不存在
409 资源冲突
500 服务内部错误
```

如果所有错误都返回 200 或 500，调用方会非常痛苦。

### 6. 本阶段如何验收

你最终要把用户 API 改造成：

- RESTful 路由。
- 参数校验完整。
- 响应格式统一。
- 错误码稳定。
- 列表支持分页筛选。
- 404 和 405 也返回统一格式。

建议按下面顺序学习：


- 综合实践把前面内容整合成一个可运行模块。

## 融合补充

+### 02-01 绑定、校验与统一响应

#### DTO 不等于数据库模型

请求 DTO 定义客户端可以提交什么，数据库 Model 定义存储结构，响应 DTO 定义可以暴露什么。不要把带有密码哈希、删除标记等字段的 Model 直接序列化给客户端。

```go
type CreateTaskInput struct {
    Title string `json:"title" binding:"required,min=1,max=120"`
    Description string `json:"description" binding:"max=2000"`
}

type ListQuery struct {
    Page int `form:"page,default=1" binding:"min=1"`
    PageSize int `form:"page_size,default=20" binding:"min=1,max=100"`
    Status string `form:"status" binding:"omitempty,oneof=todo doing done"`
}
```

使用 `ShouldBindJSON`、`ShouldBindQuery`、`ShouldBindUri`，由 Handler 决定错误格式。它们返回错误但不会自动写响应，便于全局保持一致。请求 body 是一次性流，不能先读取再重复绑定。

#### 统一响应

```go
type Body struct {
    Code int `json:"code"`
    Message string `json:"message"`
    Data interface{} `json:"data,omitempty"`
    RequestID string `json:"request_id,omitempty"`
}

func OK(c *gin.Context, data interface{}) {
    c.JSON(http.StatusOK, Body{Code: 0, Message: "ok", Data: data, RequestID: c.GetString("request_id")})
}

func Error(c *gin.Context, status, code int, message string) {
    c.JSON(status, Body{Code: code, Message: message, RequestID: c.GetString("request_id")})
}
```

约定示例：`10001` 参数错误、`20001` 未认证、`20002` 禁止访问、`30001` 资源不存在、`30002` 资源冲突、`50000` 内部错误。HTTP 状态码负责通用协议语义，业务码方便客户端按类别处理；发布后的业务码应保持兼容。

```go
func listTasks(c *gin.Context) {
    var q ListQuery
    if err := c.ShouldBindQuery(&q); err != nil {
        Error(c, http.StatusBadRequest, 10001, "page, page_size or status is invalid")
        return
    }
    // tasks, err := service.List(c.Request.Context(), q)
}
```

字段级校验错误应翻译为可读消息，例如“email 格式不正确”，而不是直接把 validator 的内部错误、结构体名或 tag 返回。每次写入响应后立即 `return`，避免同一请求被多次写入。

**验收：**无效 JSON、字段缺失、错误邮箱、越界分页的格式一致；错误响应中没有内部实现细节。

+### 02-02 REST、分页、状态码与排错

#### 列表接口约定

列表接口不能无限制返回全部数据。限制 `page_size`，保持排序稳定，并携带分页元数据：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "items": [],
    "page": 1,
    "page_size": 20,
    "total": 0
  }
}
```

排序字段必须用白名单，不能把客户端参数直接拼接进 SQL：

```go
orders := map[string]string{"created_at": "created_at desc", "due_at": "due_at asc"}
order := orders[c.DefaultQuery("sort", "created_at")]
if order == "" { order = "created_at desc" }
```

筛选条件用查询参数，如 `?status=todo`。空筛选、非法筛选、无结果是不同概念：非法值返回 400；合法筛选但无结果返回带空数组的 200。

#### 状态码决策表

| 情况 | 正确处理 |
| --- | --- |
| 创建任务成功 | `201` 与新资源 |
| 删除任务成功 | `204`，不再写 JSON body |
| 缺少必要字段 | `400` + 参数业务码 |
| 登录失败 | `401`，不要说明“账号不存在”还是“密码错误” |
| 访问他人任务 | `403` 或基于业务策略返回 404 |
| 任务不存在 | `404` |
| 重复邮箱 | `409` |
| 数据库超时或未知错误 | 记录日志，返回通用 `500` |

#### 接口测试请求集

每新增一个端点，至少保存以下请求：成功、空 body、字段错误、未认证、无权限、不存在资源、重复创建和分页边界。可在 REST Client 文件中维护：

```http
### 创建任务
POST http://localhost:8080/api/v1/tasks
Content-Type: application/json
Authorization: Bearer {{token}}

{"title":"finish tutorial"}

### 非法分页
GET http://localhost:8080/api/v1/tasks?page=0&page_size=999
Authorization: Bearer {{token}}
```

接口返回异常时，依次确认请求实际发送的 URL 和方法、Handler 是否被命中、绑定和校验结果、Service 错误映射、最终日志中的 request ID。这样能迅速区分客户端、路由、业务和依赖问题。

**验收：**对列表、创建、更新、删除各写一组成功和失败请求；分页、状态码与业务码均符合约定。
