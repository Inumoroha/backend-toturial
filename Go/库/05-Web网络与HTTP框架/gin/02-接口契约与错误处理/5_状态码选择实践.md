# 5. 状态码选择实践

本节目标：建立常见接口场景和 HTTP 状态码之间的对应关系。完成本节后，你应该能在写 handler 时自然判断一个错误应该返回 400、401、403、404、409 还是 500。

状态码不是装饰。它会影响：

- 前端如何处理响应。
- 接口测试工具如何显示结果。
- 日志系统如何统计错误。
- 网关和监控如何判断服务健康。
- 其他开发者如何理解接口行为。

---

## 一、状态码的基本分类

常见 HTTP 状态码大致分为：

```text
2xx 成功
4xx 客户端请求问题
5xx 服务端内部问题
```

在后端 API 中最常见的是：

```text
200 OK
201 Created
400 Bad Request
401 Unauthorized
403 Forbidden
404 Not Found
409 Conflict
429 Too Many Requests
500 Internal Server Error
```

---

## 二、成功状态码

### 1. 200 OK

最常见的成功响应。

适合：

```text
查询成功。
更新成功。
删除成功并返回提示。
普通操作成功。
```

示例：

```go
response.OK(c, user)
```

### 2. 201 Created

表示资源创建成功。

适合：

```text
POST /api/v1/users
POST /api/v1/tasks
POST /api/v1/articles
```

示例：

```go
response.Created(c, user)
```

### 3. 204 No Content

表示成功，但没有响应体。

适合：

```text
DELETE /api/v1/users/:id
```

不过本教程为了保持统一响应，删除成功仍然可以返回 200：

```json
{
  "code": 0,
  "message": "ok"
}
```

入门阶段优先保持响应结构一致。

---

## 三、400 Bad Request

400 表示请求参数有问题。

适合：

```text
JSON 格式错误。
字段类型错误。
必填字段缺失。
邮箱格式错误。
分页参数非法。
路径 ID 不是数字。
枚举值不合法。
```

示例：

```go
id, err := strconv.ParseUint(c.Param("id"), 10, 64)
if err != nil {
    response.Fail(c, http.StatusBadRequest, response.CodeInvalidRequest, "invalid user id")
    return
}
```

请求：

```http
GET /api/v1/users/abc
```

响应：

```json
{
  "code": 40001,
  "message": "invalid user id"
}
```

---

## 四、401 Unauthorized

401 表示没有通过认证。

适合：

```text
没有携带 token。
token 格式错误。
token 过期。
token 签名无效。
登录邮箱或密码错误。
```

示例：

```go
response.Fail(c, http.StatusUnauthorized, response.CodeUnauthorized, "missing token")
c.Abort()
return
```

注意：401 的名字叫 Unauthorized，但语义上更接近“未认证”。

---

## 五、403 Forbidden

403 表示已经知道你是谁，但你没有权限。

适合：

```text
普通用户访问管理员接口。
用户访问不属于自己的资源。
账号被禁用后访问受限接口。
```

401 和 403 的区别：

```text
401：你还没证明你是谁。
403：你已经证明了你是谁，但你不能做这件事。
```

示例：

```go
if role != "admin" {
    response.Fail(c, http.StatusForbidden, response.CodeForbidden, "forbidden")
    return
}
```

---

## 六、404 Not Found

404 表示资源不存在。

适合：

```text
用户 ID 不存在。
任务 ID 不存在。
文章 ID 不存在。
路由不存在。
```

示例：

```go
user, ok := users[id]
if !ok {
    response.Fail(c, http.StatusNotFound, response.CodeNotFound, "user not found")
    return
}
```

注意：对于“用户访问别人的任务”，有些系统会返回 403，有些系统为了隐藏资源存在性会返回 404。两种都可以，但项目内要保持一致。

---

## 七、409 Conflict

409 表示资源冲突。

适合：

```text
邮箱已存在。
用户名已存在。
重复提交导致资源已创建。
状态冲突，例如已完成订单不能再次取消。
```

示例：

```go
if emailExists {
    response.Fail(c, http.StatusConflict, response.CodeConflict, "email already exists")
    return
}
```

为什么不是 400？

因为请求格式本身没错，只是和当前系统状态冲突。

---

## 八、429 Too Many Requests

429 表示请求过于频繁。

适合：

```text
登录失败次数过多。
短信验证码发送过快。
接口限流。
```

示例：

```go
response.Fail(c, http.StatusTooManyRequests, response.CodeTooManyRequests, "too many requests")
```

这个状态码会在 Redis 限流阶段继续使用。

---

## 九、500 Internal Server Error

500 表示服务内部错误。

适合：

```text
数据库连接失败。
Redis 异常但业务必须依赖 Redis。
程序出现未预期错误。
外部服务调用失败且无法降级。
```

不要把参数错误返回 500。

错误示例：

```go
if err := c.ShouldBindJSON(&req); err != nil {
    response.Fail(c, http.StatusInternalServerError, response.CodeInternalError, "internal server error")
    return
}
```

这里应该返回 400，而不是 500。

## 十一、handler 中的状态码选择

示例：

```go
func createUser(c *gin.Context) {
    var req CreateUserRequest
    if err := c.ShouldBindJSON(&req); err != nil {
        response.FailWithData(c, http.StatusBadRequest, response.CodeInvalidRequest, "invalid request", response.FormatValidationError(err))
        return
    }

    user, err := userService.Create(req)
    if errors.Is(err, ErrEmailAlreadyExists) {
        response.Fail(c, http.StatusConflict, response.CodeConflict, "email already exists")
        return
    }
    if err != nil {
        response.Fail(c, http.StatusInternalServerError, response.CodeInternalError, "internal server error")
        return
    }

    response.Created(c, user)
}
```

这个结构很典型：

```text
参数错误 -> 400
业务冲突 -> 409
未知内部错误 -> 500
成功创建 -> 201
```

## 十二、常见错误选择

### 1. 把所有错误都返回 500

不推荐。

参数错误、资源不存在、邮箱重复都不是服务内部崩了。

### 2. 把所有错误都返回 200

不推荐。

这会让监控和调用方难以判断接口是否真的成功。

### 3. 401 和 403 混用

记住：

```text
没登录或 token 无效 -> 401
登录了但没权限 -> 403
```

## 十三、本节练习

为下面场景选择 HTTP 状态码和业务 code：

```text
1. 创建用户时邮箱格式错误。
2. 创建用户时邮箱已存在。
3. 查询用户时 id=abc。
4. 查询用户时 id=999 不存在。
5. 登录时密码错误。
6. 普通用户访问管理员接口。
7. 数据库查询超时。
8. 登录失败次数超过限制。
```

参考：

```text
1. 400 + 40001
2. 409 + 40901
3. 400 + 40001
4. 404 + 40401
5. 401 + 40101
6. 403 + 40301
7. 500 + 50001
8. 429 + 42901
```
