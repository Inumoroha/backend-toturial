# 8. 参数校验、统一响应排错清单

本节目标：把第3阶段最容易出错的地方集中整理成一份排错清单。你可以在完成本阶段综合实践后，按这份清单逐项检查自己的代码。

前面几节分别讲了 RESTful 路由、binding 校验、统一响应、分页筛选、状态码和接口测试请求。真正写项目时，问题往往不是“完全不会”，而是某个细节漏了：

- 查询参数写了 `json` tag，导致绑定不到。
- 错误响应有的用统一结构，有的直接 `gin.H`。
- 参数错误返回了 `500`。
- 404 仍然是 Gin 默认文本。
- 字段级错误返回 Go 字段名，前端看不懂。
- 创建成功返回了 `200`，但团队约定应该是 `201`。

这份清单就是用来发现这些问题的。

---

## 一、路由设计排错

先检查路由是否像资源，而不是像动作。

推荐：

```text
GET    /api/v1/users
POST   /api/v1/users
GET    /api/v1/users/:id
PUT    /api/v1/users/:id
PATCH  /api/v1/users/:id/status
DELETE /api/v1/users/:id
```

不推荐：

```text
GET  /api/v1/getUsers
POST /api/v1/createUser
POST /api/v1/deleteUser
POST /api/v1/updateUserStatus
```

自查问题：

- 是否把动词写进了路径。
- 是否所有接口都有统一 `/api/v1` 前缀。
- 是否用 HTTP 方法表达动作。
- 是否使用复数资源名 `users`。
- 是否把资源 ID 放在路径参数中。

如果你看到很多 `getXXX`、`createXXX`、`deleteXXX`，说明路由还停留在 RPC 风格，需要回到第1节重新整理。

## 二、参数来源排错

不同参数应该来自不同位置。

```text
资源 ID：路径参数
分页筛选：查询参数
创建/更新内容：JSON body
认证信息：Header
```

示例：

```text
GET /api/v1/users/10
```

`10` 是路径参数：

```go
id := c.Param("id")
```

示例：

```text
GET /api/v1/users?page=1&page_size=10
```

`page` 和 `page_size` 是查询参数：

```go
var req ListUsersRequest
_ = c.ShouldBindQuery(&req)
```

示例：

```http
POST /api/v1/users
Content-Type: application/json

{
  "username": "alice",
  "email": "alice@example.com"
}
```

请求体使用：

```go
var req CreateUserRequest
_ = c.ShouldBindJSON(&req)
```

常见错误：

```go
type ListUsersRequest struct {
    Page int `json:"page"`
}
```

列表查询参数应该写：

```go
type ListUsersRequest struct {
    Page int `form:"page" binding:"omitempty,gte=1"`
}
```

---

## 三、binding 校验排错

检查请求结构体是否有明确校验规则。

创建用户：

```go
type CreateUserRequest struct {
    Username string `json:"username" binding:"required,min=2,max=32"`
    Email    string `json:"email" binding:"required,email"`
    Age      int    `json:"age" binding:"gte=1,lte=120"`
    Status   string `json:"status" binding:"omitempty,oneof=active disabled"`
}
```

列表查询：

```go
type ListUsersRequest struct {
    Page     int    `form:"page" binding:"omitempty,gte=1"`
    PageSize int    `form:"page_size" binding:"omitempty,gte=1,lte=100"`
    Status   string `form:"status" binding:"omitempty,oneof=active disabled"`
    Keyword  string `form:"keyword" binding:"omitempty,max=32"`
}
```

自查问题：

- 必填字段是否写了 `required`。
- 邮箱是否写了 `email`。
- 数值范围是否写了 `gte`、`lte`。
- 枚举字段是否写了 `oneof`。
- 可选字段是否配合 `omitempty`。
- 查询参数是否使用 `form` tag。
- JSON 请求是否设置 `Content-Type: application/json`。

## 四、字段级错误排错

最低要求：

```json
{
  "code": 40001,
  "message": "invalid request",
  "data": {
    "Username": "min",
    "Email": "email"
  }
}
```

更友好的目标：

```json
{
  "code": 40001,
  "message": "invalid request",
  "data": {
    "username": "min",
    "email": "email"
  }
}
```

如果你希望把 Go 字段名转换为 JSON 字段名，可以写一个小工具。示例：

```go
func jsonFieldName(field reflect.StructField) string {
    tag := field.Tag.Get("json")
    if tag == "" {
        return field.Name
    }

    name := strings.Split(tag, ",")[0]
    if name == "" || name == "-" {
        return field.Name
    }

    return name
}
```

然后在格式化 validation error 时，结合结构体类型读取字段 tag。学习阶段不强制实现，但你至少要知道为什么错误响应里可能出现 `Username` 而不是 `username`。

自查问题：

- JSON 格式错误是否能返回 `body: invalid json`。
- binding 校验错误是否能返回字段名。
- 字段错误是否放在统一响应的 `data` 中。
- 是否没有把原始内部错误直接暴露给调用方。

## 五、统一响应排错

成功响应应该稳定：

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

失败响应应该稳定：

```json
{
  "code": 40001,
  "message": "invalid request"
}
```

分页响应应该稳定：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "items": [],
    "total": 0,
    "page": 1,
    "page_size": 10
  }
}
```

搜索项目中是否还有散落响应：

```bash
rg "gin.H|c.JSON" internal
```

不是所有 `c.JSON` 都错，但第3阶段以后，业务 handler 中应该尽量使用：

```go
response.OK(...)
response.Created(...)
response.Fail(...)
response.FailWithData(...)
```

自查问题：

- 成功响应是否都有 `code` 和 `message`。
- 失败响应是否都有稳定业务错误码。
- 分页列表是否统一放在 `items`、`total`、`page`、`page_size` 中。
- handler 是否还在直接返回临时结构。

## 六、HTTP 状态码排错

常见映射：

```text
200 查询、更新成功
201 创建成功
204 删除成功且不返回 body
400 参数错误
401 未登录
403 无权限
404 资源不存在或路由不存在
405 方法不允许
409 资源冲突
429 请求过于频繁
500 内部错误
```

本阶段重点检查：

- 参数错误不要返回 `500`。
- 资源不存在不要返回 `200`。
- 邮箱重复这类冲突不要返回 `400` 也可以，优先考虑 `409`。
- 创建成功可以用 `201`。
- 不存在路由要统一 JSON。
- 方法不允许要统一 JSON。

测试命令：

```http
### route not found
GET http://localhost:8080/api/v1/not-exists

### method not allowed
POST http://localhost:8080/api/v1/users/1
```

## 七、NoRoute 和 NoMethod 排错

Gin 默认 404 不是你的统一响应。需要显式配置：

```go
r.NoRoute(func(c *gin.Context) {
    response.Fail(c, http.StatusNotFound, response.CodeNotFound, "route not found")
})
```

405 需要先开启：

```go
r.HandleMethodNotAllowed = true
```

再配置：

```go
r.NoMethod(func(c *gin.Context) {
    response.Fail(c, http.StatusMethodNotAllowed, response.CodeInvalidRequest, "method not allowed")
})
```

自查问题：

- 是否设置了 `r.NoRoute`。
- 是否设置了 `r.HandleMethodNotAllowed = true`。
- 是否设置了 `r.NoMethod`。
- 404 和 405 是否都返回统一 JSON。

## 八、接口测试请求排错

第3阶段至少保留这些测试请求：

```text
列表成功
分页成功
分页参数错误
状态筛选成功
状态筛选错误
创建成功
创建参数错误
邮箱重复
详情成功
详情 ID 非法
详情不存在
更新成功
状态更新成功
删除成功
路由不存在
方法不允许
```

如果你只保留成功请求，说明测试资产不完整。真实项目里，错误请求同样重要。

---

## 九、最终验收清单

请逐项确认：

- [ ] 路由使用 RESTful 风格。
- [ ] 所有接口有统一 `/api/v1` 前缀。
- [ ] 路径参数、查询参数、JSON body 分工清晰。
- [ ] 创建用户请求有 binding 校验。
- [ ] 更新用户请求有 binding 校验。
- [ ] 列表查询有 `ShouldBindQuery` 校验。
- [ ] 参数错误返回 `400`。
- [ ] 字段级错误放在统一响应 `data` 中。
- [ ] 成功响应统一使用 `response.OK` 或 `response.Created`。
- [ ] 失败响应统一使用 `response.Fail` 或 `response.FailWithData`。
- [ ] 分页响应包含 `items`、`total`、`page`、`page_size`。
- [ ] 404 路由不存在返回统一 JSON。
- [ ] 405 方法不允许返回统一 JSON。
- [ ] 邮箱重复等冲突类错误返回稳定业务码。
- [ ] `requests.http` 覆盖成功和失败请求。

这些全部通过后，第3阶段才算真正完成。下一阶段会在这个基础上加入中间件、JWT、CORS、Recovery、RequestID 和认证失败响应统一。
