# 0. 配置、日志、测试与接口文档总览

第7阶段开始，你的 Gin 项目不再只是“能跑起来”的练习代码，而要逐步变成一个别人也能启动、测试、排错和阅读的工程项目。

前面几个阶段我们已经做过路由、参数绑定、统一响应、中间件、JWT、分层结构和数据库持久化。本阶段要解决的是工程化收口问题：配置从哪里来，日志怎么看，测试怎么写，接口文档怎么生成，项目命令怎么统一。

---

## 一、本阶段学习目标

学完本阶段后，你应该能做到：

- 使用 Viper 加载配置文件，并支持环境变量覆盖。
- 区分本地环境、测试环境和生产环境的配置差异。
- 使用 zap 输出结构化日志。
- 编写请求日志中间件，记录 method、path、status、latency、request_id。
- 使用 `httptest` 测试 Gin handler。
- 使用表格驱动测试 service 业务逻辑。
- 使用 Swagger 注释生成接口文档。
- 使用 Makefile 统一启动、测试、文档生成等常用命令。
- 为项目补充一个能让新同事快速启动的 README。

---

## 二、为什么要在这个阶段学工程化

很多初学者写 Gin 项目时，会把精力放在接口能不能返回正确 JSON 上。这当然重要，但后端工程师真正进入团队协作后，会遇到更多问题：

- 项目换一台电脑启动不了。
- 数据库地址、JWT 密钥写死在代码里。
- 线上出错时，只看到一串 `500`，不知道是哪一个请求失败。
- 接口改了，但文档没有同步。
- 测试全靠手动点 Postman。
- 每个人启动项目的命令都不一样。

这些问题不解决，项目越写越乱。工程化的作用，就是把“靠记忆和手动操作”的部分沉淀成规范、配置、命令和测试。

---

## 三、本阶段推荐项目结构

建议在第5、第6阶段分层项目基础上继续扩展：

```text
gin-user-api/
├── cmd/
│   └── server/
│       └── main.go
├── config/
│   ├── config.yaml
│   ├── config.local.yaml
│   └── config.test.yaml
├── docs/
│   └── swagger 生成文件
├── internal/
│   ├── app/
│   ├── config/
│   ├── handler/
│   ├── middleware/
│   ├── repository/
│   ├── response/
│   └── service/
├── pkg/
│   └── logger/
├── Makefile
├── README.md
├── go.mod
└── go.sum
```

其中：

- `config/`：放 YAML 配置文件。
- `internal/config/`：负责读取配置并转换为 Go 结构体。
- `pkg/logger/`：封装 zap 初始化。
- `internal/middleware/`：放请求日志、Recovery、RequestID 等中间件。
- `docs/`：Swagger 生成物，不手写。
- `Makefile`：统一项目常用命令。

## 四、本阶段文件安排

本阶段按以下顺序学习：

```text
0_配置日志测试与接口文档总览.md
1_Viper配置与环境区分.md
2_zap日志与请求日志中间件.md
3_测试策略Swagger与Makefile.md
4_handler测试实践.md
5_service测试与表格驱动.md
6_Swagger注释与接口文档.md
7_本阶段综合实践_工程化收口.md
```

建议不要跳着看。配置、日志、测试、文档之间是有关联的：

- 配置决定程序如何启动。
- 日志帮助你定位启动和请求过程中的问题。
- 测试保证重构后功能没有坏。
- Swagger 让接口行为可读、可沟通。
- Makefile 把上面这些动作变成固定命令。

## 五、本阶段最终效果

完成后，你的项目应该能支持这些命令：

```bash
make run
make test
make test-cover
make swagger
make lint
```

并且能做到：

- 修改 `config/config.local.yaml` 后，服务端口、数据库地址等配置生效。
- 设置环境变量后，可以覆盖 YAML 配置。
- 每个请求都会输出结构化访问日志。
- handler 层至少有核心接口测试。
- service 层至少有表格驱动测试。
- Swagger 页面可以查看主要接口说明。
- README 写清楚项目启动、测试、接口文档生成方式。

---

## 六、学习前准备

你需要确认已经掌握：

- Gin 路由注册和 handler 写法。
- `ShouldBindJSON`、`ShouldBindQuery` 等参数绑定方法。
- 统一响应结构。
- JWT 中间件的基本写法。
- service、repository、handler 的分层职责。
- GORM 数据库连接和基础 CRUD。

如果这些内容还不熟，建议先回看第3到第6阶段。第7阶段不会重新解释每一个 Gin 基础概念，而是把它们组织成工程实践。

---

## 七、本阶段验收标准

完成本阶段后，请逐项检查：

- 项目中不存在硬编码的数据库 DSN、JWT Secret、服务端口。
- `config.local.yaml` 可以用于本地开发。
- `config.test.yaml` 可以用于测试环境。
- zap 日志能输出 JSON 格式日志。
- 请求日志包含 `request_id`。
- handler 测试能覆盖成功、参数错误、未登录三类情况。
- service 测试能覆盖正常分支和异常分支。
- Swagger 文档可以通过浏览器访问。
- README 中的新手启动步骤不超过 5 分钟能跑通。

只要这些检查都能通过，你的 Gin 项目就已经从“课堂练习”向“可维护后端工程”迈出了一大步。
