# 0. 中间件、错误处理与认证基础总览

本阶段目标：理解 Gin 中间件的执行流程，并掌握真实后端项目中最常见的横切能力：请求日志、错误恢复、CORS、Request ID、JWT 认证、认证失败统一响应和基础角色授权。

如果前三个阶段解决的是：

```text
如何设计路由、读取参数、校验请求、返回统一响应。
```

那么第四阶段要解决的是：

```text
如何把每个接口都需要的公共逻辑抽到中间件里。
```

中间件是 Gin 非常核心的能力。你以后写登录认证、权限控制、接口限流、请求日志、panic 恢复、跨域处理，几乎都会和中间件有关。

## 一、中间件适合处理什么

中间件适合处理“很多接口都需要”的逻辑。

常见场景：

```text
请求日志
请求耗时统计
Request ID
panic recovery
CORS 跨域
JWT 认证
角色权限
接口限流
统一错误处理
```

这些逻辑如果写在每个 handler 里，会导致大量重复代码。

例如每个私有接口都写：

```go
token := c.GetHeader("Authorization")
if token == "" {
    response.Fail(c, 401, 40101, "missing token")
    return
}
```

会非常啰嗦，也很容易漏掉。

更好的方式是把它放进认证中间件：

```go
private := api.Group("")
private.Use(middleware.Auth(secret))
{
    private.GET("/profile", userHandler.Profile)
    private.GET("/tasks", taskHandler.List)
}
```

```text
POST /api/v1/auth/register
POST /api/v1/auth/login
GET  /api/v1/profile
GET  /api/v1/admin/ping
```

要求：

- 注册时密码使用 bcrypt 加密。
- 登录成功返回 JWT。
- `/profile` 必须携带 token。
- token 无效或过期返回统一 401。
- `/admin/ping` 只有 admin 角色可以访问。
- 每个请求都带 `X-Request-ID` 响应头。
- 请求日志能看到 method、path、status、latency、request_id。

## 融合补充

+### 03-01 中间件、CORS、日志与 Recovery

中间件在 Handler 前后执行，适合承载横切能力。典型顺序是：Recovery -> Request ID -> 访问日志 -> CORS -> 限流 -> 认证 -> 授权 -> 业务 Handler。`c.Next()` 后的代码会在下游完成后继续执行；拒绝请求时必须 `c.Abort()` 并 `return`。

#### Request ID 与访问日志

```go
func RequestID() gin.HandlerFunc {
    return func(c *gin.Context) {
        id := c.GetHeader("X-Request-ID")
        if id == "" { id = uuid.NewString() }
        c.Set("request_id", id)
        c.Header("X-Request-ID", id)
        c.Next()
    }
}

func AccessLog(logger *zap.Logger) gin.HandlerFunc {
    return func(c *gin.Context) {
        start := time.Now()
        c.Next()
        logger.Info("http request",
            zap.String("request_id", c.GetString("request_id")),
            zap.String("method", c.Request.Method),
            zap.String("path", c.Request.URL.Path),
            zap.Int("status", c.Writer.Status()),
            zap.Duration("latency", time.Since(start)),
        )
    }
}
```

日志应记录结构化字段而非拼接长文本。禁止记录密码、Authorization、完整请求体、DSN 和其他敏感数据。把 request ID 同时写进日志和响应，用户报告错误时才有可追踪线索。

#### Recovery 与 CORS

`gin.Default()` 含 Recovery；用 `gin.New()` 时必须显式 `r.Use(gin.Recovery())` 或使用自己的恢复中间件。恢复中间件可记录 panic 详情，但客户端只能收到通用 500，不能暴露堆栈。

CORS 只允许实际可信的前端来源、方法和 Header。若允许携带 Cookie 或 Authorization，`Access-Control-Allow-Origin` 不能为 `*`。预检 `OPTIONS` 请求也要能正确返回，不要在所有服务上直接复制“全开放”配置。

#### 本节验收

- 每个响应带 `X-Request-ID`，日志包含同一 ID、路径、状态和耗时。
- Handler panic 后进程不退出，客户端得到通用 500。
- 前端跨域请求仅来自显式允许的来源和 Header。

+### 03-02 注册、登录、JWT 与授权

#### 密码规则

只保存 bcrypt 哈希，绝不保存明文、可逆加密文本或客户端传来的“已加密密码”：

```go
hash, err := bcrypt.GenerateFromPassword([]byte(password), bcrypt.DefaultCost)
err = bcrypt.CompareHashAndPassword([]byte(user.PasswordHash), []byte(password))
```

注册时验证邮箱、密码长度和唯一性；登录失败时统一返回“邮箱或密码错误”，不要泄露账号是否存在。生产环境还需要配合第 7 阶段的登录失败限流。

#### 签发与验证 JWT

```bash
go get github.com/golang-jwt/jwt/v5 golang.org/x/crypto/bcrypt
```

```go
type Claims struct {
    Role string `json:"role"`
    jwt.RegisteredClaims
}

func makeToken(userID uint, role, secret string) (string, error) {
    claims := Claims{Role: role, RegisteredClaims: jwt.RegisteredClaims{
        Subject: strconv.FormatUint(uint64(userID), 10),
        IssuedAt: jwt.NewNumericDate(time.Now()),
        ExpiresAt: jwt.NewNumericDate(time.Now().Add(2 * time.Hour)),
    }}
    return jwt.NewWithClaims(jwt.SigningMethodHS256, claims).SignedString([]byte(secret))
}
```

中间件严格验证 `Authorization: Bearer <token>`、签名算法、签名和过期时间，成功后只把必要身份写入 Context：

```go
func Auth(secret string) gin.HandlerFunc {
    return func(c *gin.Context) {
        parts := strings.Fields(c.GetHeader("Authorization"))
        if len(parts) != 2 || parts[0] != "Bearer" { response.Error(c, 401, 20001, "authentication required"); c.Abort(); return }
        token, err := jwt.ParseWithClaims(parts[1], &Claims{}, func(t *jwt.Token) (interface{}, error) {
            if _, ok := t.Method.(*jwt.SigningMethodHMAC); !ok { return nil, errors.New("unexpected signing method") }
            return []byte(secret), nil
        })
        claims, ok := token.Claims.(*Claims)
        if err != nil || !ok || !token.Valid { response.Error(c, 401, 20001, "invalid token"); c.Abort(); return }
        id, err := strconv.ParseUint(claims.Subject, 10, 64)
        if err != nil { response.Error(c, 401, 20001, "invalid token"); c.Abort(); return }
        c.Set("user_id", uint(id)); c.Set("role", claims.Role)
        c.Next()
    }
}
```

密钥从环境变量或密钥系统读取，不写进仓库；token 设置短过期时间并规划密钥轮换。角色检查可做成 `RequireRole("admin")` 中间件。资源归属判断放在 Service：用户能否修改任务取决于数据库中的 `task.OwnerID == currentUserID`，绝不信任客户端上传的 owner 或 role。

**验收：**无 token、过期 token、伪造 token 都返回 401；已登录用户访问他人资源返回 403；数据库中找不到明文密码。
