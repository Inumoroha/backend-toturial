# 7. 综合实践：数据库版用户模块

本节目标：把第5阶段的内存版用户模块改造成 GORM 数据库版。接口路径保持不变，但数据真正保存到 MySQL 中。

---

## 一、最终目标

接口仍然是：

```text
GET    /api/v1/users
POST   /api/v1/users
GET    /api/v1/users/:id
PUT    /api/v1/users/:id
PATCH  /api/v1/users/:id/status
DELETE /api/v1/users/:id
```

变化是：

```text
repository 从 map 改成 GORM。
服务重启后数据不丢失。
邮箱唯一由数据库保证。
删除变成软删除。
分页由数据库完成。
```

---

## 二、改造步骤

推荐顺序：

```text
1. 启动 MySQL。
2. 创建 database.NewMySQL。
3. 定义 model.User。
4. 执行 AutoMigrate。
5. 改造 UserRepository。
6. 调整 UserService。
7. 调整 main 依赖注入。
8. 用 requests.http 回归测试。
```

每一步都运行，不要等全部写完再调试。

---

## 三、第一步：启动 MySQL 并创建数据库

如果你使用 Docker：

```bash
  -p 3306:3306 \

```

验证容器：

```bash
docker ps
```

进入 MySQL：

```bash
docker exec -it gin-mysql mysql -uroot -ppassword
```

检查数据库：

```sql

SHOW TABLES;
```

如果能进入 `gin_learning`，说明数据库准备好了。

## 四、第二步：定义数据库连接函数

创建：

```text
internal/database/mysql.go
```

示例：

```go
package database

import (
    "fmt"
    "time"

    "gorm.io/driver/mysql"
    "gorm.io/gorm"
    "gorm.io/gorm/logger"
)

func NewMySQL(dsn string) (*gorm.DB, error) {
    db, err := gorm.Open(mysql.Open(dsn), &gorm.Config{
        Logger: logger.Default.LogMode(logger.Info),
    })
    if err != nil {
        return nil, fmt.Errorf("open mysql: %w", err)
    }

    sqlDB, err := db.DB()
    if err != nil {
        return nil, fmt.Errorf("get sql db: %w", err)
    }

    sqlDB.SetMaxIdleConns(10)
    sqlDB.SetMaxOpenConns(100)
    sqlDB.SetConnMaxLifetime(time.Hour)

    if err := sqlDB.Ping(); err != nil {
        return nil, fmt.Errorf("ping mysql: %w", err)
    }

    return db, nil
}
```

这一步要确认：

- DSN 正确。
- MySQL 能 ping 通。
- 连接池参数已设置。
- 错误信息带上下文，方便排查。

## 五、第三步：定义 User 模型

创建：

```text
internal/model/user.go
```

示例：

```go
package model

import "gorm.io/gorm"

type User struct {
    gorm.Model
    Username string `gorm:"type:varchar(32);not null"`
    Email    string `gorm:"type:varchar(255);not null;uniqueIndex"`
    Age      int    `gorm:"not null;default:1"`
    Status   string `gorm:"type:varchar(20);not null;default:active;index"`
}
```

字段说明：

- `gorm.Model` 包含 `ID`、`CreatedAt`、`UpdatedAt`、`DeletedAt`。
- `Email` 使用唯一索引，防止并发重复注册。
- `Status` 增加普通索引，方便按状态筛选。
- `DeletedAt` 支持软删除。

注意：不要把请求 DTO 直接当数据库模型。请求 DTO 面向 HTTP 输入，model 面向数据库结构。

你应该看到：

## 七、第五步：UserRepository 改造

内存版：

```go
type UserRepository struct {
    users map[uint]model.User
}
```

数据库版：

```go
type UserRepository struct {
    db *gorm.DB
}
```

构造函数：

```go
func NewUserRepository(db *gorm.DB) *UserRepository {
    return &UserRepository{db: db}
}
```

对 service 暴露的方法尽量保持一致。

示例方法：

```go
func (r *UserRepository) Create(user *model.User) error {
    if err := r.db.Create(user).Error; err != nil {
        if isDuplicateKey(err) {
            return ErrEmailAlreadyExists
        }
        return err
    }
    return nil
}

func (r *UserRepository) FindByID(id uint) (*model.User, error) {
    var user model.User
    if err := r.db.First(&user, id).Error; err != nil {
        if errors.Is(err, gorm.ErrRecordNotFound) {
            return nil, ErrUserNotFound
        }
        return nil, err
    }
    return &user, nil
}

func (r *UserRepository) FindByEmail(email string) (*model.User, error) {
    var user model.User
    if err := r.db.Where("email = ?", email).Take(&user).Error; err != nil {
        if errors.Is(err, gorm.ErrRecordNotFound) {
            return nil, ErrUserNotFound
        }
        return nil, err
    }
    return &user, nil
}
```

repository 要负责把 GORM 错误转换成项目内部错误。不要让 handler 直接判断 `gorm.ErrRecordNotFound`。

## 八、第六步：Service 保持业务规则

service 仍然负责：

```text
默认状态 active。
状态合法性。
邮箱重复业务错误。
分页默认值和上限。
```

不要因为接入数据库，就把业务规则全部塞进 repository。

service 示例：

```go
func (s *UserService) Create(input CreateUserInput) (*model.User, error) {
    status := input.Status
    if status == "" {
        status = "active"
    }
    if !isValidStatus(status) {
        return nil, ErrInvalidUserStatus
    }

    _, err := s.repo.GetByEmail(input.Email)
    if err == nil {
        return nil, ErrEmailAlreadyExists
    }
    if err != nil && !errors.Is(err, repository.ErrUserNotFound) {
        return nil, err
    }

    user := &model.User{
        Username: input.Username,
        Email:    input.Email,
        Age:      input.Age,
        Status:   status,
    }

    if err := s.repo.Create(user); err != nil {
        if errors.Is(err, repository.ErrEmailAlreadyExists) {
            return nil, ErrEmailAlreadyExists
        }
        return nil, err
    }

    return user, nil
}
```

这里有两个重点：

```text
service 先查邮箱，给用户更友好的错误。
repository 仍然依赖唯一索引兜底，防止并发重复。
```

不要因为数据库有唯一索引，就完全不在 service 中做业务判断。也不要只靠 service 查询，而忘记数据库唯一索引。

## 九、第七步：分页列表改造

repository 可以提供：

```go
type ListUsersFilter struct {
    Page     int
    PageSize int
    Status   string
    Keyword  string
}
```

实现：

```go
func (r *UserRepository) List(filter ListUsersFilter) ([]model.User, int64, error) {
    var users []model.User
    var total int64

    query := r.db.Model(&model.User{})

    if filter.Status != "" {
        query = query.Where("status = ?", filter.Status)
    }

    if filter.Keyword != "" {
        query = query.Where("username LIKE ? OR email LIKE ?", "%"+filter.Keyword+"%", "%"+filter.Keyword+"%")
    }

    if err := query.Count(&total).Error; err != nil {
        return nil, 0, err
    }

    offset := (filter.Page - 1) * filter.PageSize
    if err := query.Order("id DESC").Offset(offset).Limit(filter.PageSize).Find(&users).Error; err != nil {
        return nil, 0, err
    }

    return users, total, nil
}
```

分页验收重点：

- `Count` 和 `Find` 使用同一组条件。
- `page` 从 1 开始。
- `page_size` 有最大限制。
- 空列表返回 `items: []`，不是错误。

## 十、第八步：Handler 基本不变

handler 仍然：

```text
读取参数。
调用 service。
把 service 错误转成 HTTP 响应。
```

如果第5阶段分层做得好，接入数据库时 handler 改动应该很小。

这是分层的价值。

如果接入 GORM 后 handler 大量修改，通常说明第5阶段分层还不够干净。

## 十三、常见问题

### 1. 重启后数据还是丢失

检查你是不是还在使用内存 map，或者 Docker MySQL 没有挂载 volume。

### 2. 邮箱重复返回 500

说明 repository 没有把唯一索引错误转换成业务错误。应转换为 `ErrEmailAlreadyExists`，handler 返回 `409`。

### 3. 删除后还能查到

检查是否用了 `Unscoped()`。普通查询会自动过滤软删除数据，`Unscoped()` 会查出软删除记录。

### 4. 分页 total 不对

检查 `Count` 是否漏掉 status、keyword 等筛选条件。

## 十四、最终验收清单

完成后检查：

- 用户表已创建。
- 创建用户写入数据库。
- 重启服务数据不丢。
- 邮箱重复返回 409。
- 查询不存在用户返回 404。
- 删除用户后普通列表不显示。
- `Unscoped` 可以查到软删除数据。
- 分页返回 `items`、`total`、`page`、`page_size`。
- GORM SQL 日志能看到实际查询。
- handler 没有直接导入 GORM。
- service 没有返回 HTTP 状态码。
- repository 负责转换 GORM 错误。
