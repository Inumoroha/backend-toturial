# 7. 分层重构检查清单

本节目标：提供一份重构完成后的检查清单。你可以用它判断自己的 Gin 项目是否真的完成了分层，而不是只是创建了几个目录。

---

## 一、目录检查

项目中至少应该有：

```text
cmd/server/main.go
internal/model
internal/repository
internal/service
internal/handler
internal/response
internal/router
```

如果某个目录为空，要思考：

```text
它是后续阶段才会使用，还是我只是为了目录而目录？
```

---

## 二、main.go 检查

main 可以做：

```text
加载配置。
初始化依赖。
组装 repository/service/handler。
启动服务。
```

main 不应该做：

```text
解析请求参数。
处理业务规则。
直接操作数据库。
注册大量具体业务逻辑。
```

理想 main：

```go
userRepo := repository.NewUserRepository()
userService := service.NewUserService(userRepo)
userHandler := handler.NewUserHandler(userService)
r := router.New(userHandler)
r.Run(":8080")
```

---

## 三、handler 检查

handler 应该做：

- 读取路径参数。
- 读取查询参数。
- 读取 JSON 请求体。
- 调用 service。
- 把 service 返回结果转换为 HTTP 响应。

handler 不应该做：

- 直接访问 map。
- 直接写 SQL。
- 直接处理事务。
- 保存复杂业务规则。

检查命令：

```bash
rg "users map|gorm|db\\.|SELECT|INSERT" internal/handler
```

如果 handler 中出现大量数据库或存储细节，就说明分层不干净。

---

## 四、service 检查

service 应该做：

- 业务默认值。
- 状态校验。
- 业务错误判断。
- 调用 repository。
- 组合多个 repository。

service 不应该做：

- 调用 `c.JSON`。
- 接收 `*gin.Context`。
- 返回 HTTP 状态码。

检查命令：

```bash
rg "gin.Context|c.JSON|http.Status" internal/service
```

如果 service 中出现这些内容，要重新考虑边界。

---

## 五、repository 检查

repository 应该做：

- 查询数据。
- 创建数据。
- 更新数据。
- 删除数据。
- 把底层 not found 转换成 repository 错误。

repository 不应该做：

- 返回 HTTP 响应。
- 处理 Gin context。
- 决定前端提示文案。

检查命令：

```bash
rg "gin|http.Status|response\\." internal/repository
```

---

## 六、DTO 和 model 检查

检查这些问题：

- 请求 DTO 是否包含不该由前端控制的字段？
- 注册请求是否允许传 `role`？
- 创建任务请求是否允许传 `user_id`？
- 响应是否可能返回 `password_hash`？

危险字段：

```text
id
user_id
role
password_hash
created_at
updated_at
deleted_at
```

这些字段要么由服务端生成，要么由认证上下文决定，不应该随便从请求体读取。

---

## 七、依赖方向检查

推荐方向：

```text
main -> router -> handler -> service -> repository -> model
```

如果出现循环依赖，Go 会直接编译失败。

常见错误：

```text
service 导入 handler
repository 导入 service
model 导入 handler
```

解决办法通常是：

- 把公共类型移动到 model。
- 把业务错误定义在更合适的层。
- 让上层依赖下层，不要反过来。

---

## 八、接口抽象检查

接口抽象不是越多越好。第5阶段引入接口，主要是为了让上层不依赖具体实现，并方便测试替换。

推荐：

```go
type UserRepository interface {
    Create(ctx context.Context, user *model.User) error
    FindByID(ctx context.Context, id uint) (*model.User, error)
}

type UserService struct {
    repo UserRepository
}
```

这样 service 只知道自己需要一个能创建和查询用户的 repository，不关心它背后是内存 map、GORM、还是测试 fake。

不推荐：

```go
type UserRepositoryInterface interface {
    Create(...)
    FindByID(...)
    FindByEmail(...)
    Update(...)
    Delete(...)
    List(...)
    Count(...)
    BeginTransaction(...)
    Commit(...)
    Rollback(...)
}
```

一开始就把接口做得巨大，会让测试和实现都变得笨重。

自查问题：

- 接口是否由使用方定义，而不是实现方强行定义。
- 接口方法是否刚好满足当前业务需要。
- 是否为了“看起来专业”创建了很多没有实际替换场景的接口。
- 测试中是否能用 fake repository 替换真实 repository。

---

## 九、错误定义检查

错误应该定义在合适的层。

repository 层可以定义：

```go
var ErrNotFound = errors.New("record not found")
```

service 层可以定义：

```go
var ErrEmailExists = errors.New("email already exists")
var ErrInvalidPassword = errors.New("invalid password")
```

handler 层负责把错误翻译成 HTTP 响应：

```go
switch {
case errors.Is(err, service.ErrEmailExists):
    response.Fail(c, http.StatusConflict, response.CodeConflict, "email already exists")
case errors.Is(err, repository.ErrNotFound):
    response.Fail(c, http.StatusNotFound, response.CodeNotFound, "user not found")
default:
    response.Fail(c, http.StatusInternalServerError, response.CodeInternalError, "internal server error")
}
```

常见错误：

```go
return http.StatusNotFound, errors.New("user not found")
```

service 不应该返回 HTTP 状态码。HTTP 是 handler 层的概念。

自查问题：

- repository 是否把底层 GORM not found 转换为自己的错误。
- service 是否定义业务错误。
- handler 是否负责错误到 HTTP 的转换。
- service 是否没有导入 `net/http`。
- repository 是否没有导入 `internal/response`。

---

## 十、依赖创建位置检查

依赖应该在 main 或 app 组装层集中创建。

推荐：

```go
db := database.New(cfg.Database)
userRepo := repository.NewUserRepository(db)
userService := service.NewUserService(userRepo)
userHandler := handler.NewUserHandler(userService)
router := router.New(userHandler)
```

不推荐：

```go
func (h *UserHandler) Create(c *gin.Context) {
    db := database.New(...)
    repo := repository.NewUserRepository(db)
    service := service.NewUserService(repo)
    service.Create(...)
}
```

这样每次请求都创建依赖，性能差，测试难，生命周期也混乱。

检查命令：

```bash
rg "NewUserRepository|NewUserService|database.New|gorm.Open" internal/handler internal/service
```

如果这些初始化函数出现在 handler 中，要重点检查。

---

## 十一、包导入边界检查

你可以用 `rg` 快速检查错误依赖。

handler 不应该导入 GORM：

```bash
rg "gorm.io|database/sql" internal/handler
```

service 不应该导入 Gin：

```bash
rg "github.com/gin-gonic/gin" internal/service
```

repository 不应该导入响应包：

```bash
rg "internal/response|gin.H|http.Status" internal/repository
```

model 不应该导入上层包：

```bash
rg "internal/handler|internal/service|internal/repository" internal/model
```

这些命令不是为了机械清零，而是帮你发现职责混乱的味道。发现后要回到代码语义判断。

---

## 十二、重构前后行为对齐检查

分层重构不应该改变接口行为。重构前后至少要对比：

```text
成功创建用户的状态码和响应。
创建用户参数错误的状态码和响应。
用户不存在的状态码和响应。
列表分页字段。
删除用户后的再次查询结果。
```

建议保留一组 `requests.http`，重构前跑一遍，重构后再跑一遍。

示例：

```http
### create user
POST http://localhost:8080/api/v1/users
Content-Type: application/json

{
  "username": "alice",
  "email": "alice@example.com",
  "age": 20
}

### get user
GET http://localhost:8080/api/v1/users/1

### list users
GET http://localhost:8080/api/v1/users?page=1&page_size=10
```

如果重构后接口路径、响应字段、错误码突然变了，要确认这是有意修改还是重构引入的回归。

---

## 十三、运行检查

执行：

```bash
go test ./...
```

再执行：

```bash
go run ./cmd/server
```

用 `requests.http` 测试：

```http
GET http://localhost:8080/api/v1/users
POST http://localhost:8080/api/v1/users
GET http://localhost:8080/api/v1/users/1
```

重构前后接口行为应该保持一致。

---

## 十四、提交建议

建议分步骤提交：

```bash
git add .
git commit -m "extract model and response"

git add .
git commit -m "extract user repository"

git add .
git commit -m "extract user service and handler"

git add .
git commit -m "extract router and clean main"
```

不要把所有重构混成一个巨大提交。后面排查会很痛苦。

---

## 十五、最终验收问题

请逐个回答：

- 如果要把内存 map 换成 GORM，主要改哪一层？
- 如果要调整 HTTP 状态码，主要改哪一层？
- 如果要修改用户默认状态，主要改哪一层？
- 如果要新增路由，主要改哪一层？
- 如果要新增请求字段校验，主要改哪一层？

参考：

```text
换 GORM：repository。
调整 HTTP 状态码：handler。
修改默认状态：service。
新增路由：router。
新增请求字段校验：handler DTO。
```

如果你能自然回答这些问题，说明分层边界已经基本建立。

---

## 十六、最终验收清单

请逐项确认：

- [ ] `main.go` 只负责启动、初始化和组装。
- [ ] router 只负责注册路由和路由组。
- [ ] handler 只处理 HTTP 输入输出。
- [ ] service 只处理业务规则。
- [ ] repository 只处理数据访问。
- [ ] model 不依赖 handler、service、repository。
- [ ] DTO 没有暴露 `password_hash` 等敏感字段。
- [ ] 请求 DTO 不允许前端传 `role`、`user_id` 等危险字段。
- [ ] service 不导入 Gin。
- [ ] repository 不导入 response。
- [ ] handler 不直接操作数据库。
- [ ] 依赖在 main 或 app 层集中创建。
- [ ] 接口抽象大小合适，能服务测试替换。
- [ ] 重构前后接口行为保持一致。
- [ ] `go test ./...` 能通过。

这份清单通过后，第5阶段才算真正完成。下一阶段接入 GORM 时，你会明显感受到分层的价值：数据库实现变化主要影响 repository，而不是把 handler 和 service 全部搅乱。
