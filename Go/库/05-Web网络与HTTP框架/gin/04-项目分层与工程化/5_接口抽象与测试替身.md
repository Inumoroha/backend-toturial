# 5. 接口抽象与测试替身

本节目标：理解什么时候需要接口抽象，以及如何用 fake repository 测试 service。Go 后端项目不应该一开始到处定义接口，但在合适的位置使用接口，会让测试和替换实现更容易。

---

## 一、先说结论

初学阶段不要过度抽象。

不需要一开始给所有结构体都写接口：

```go
type UserServiceInterface interface {}
type UserHandlerInterface interface {}
type UserRepositoryInterface interface {}
```

这会让代码变复杂，却没有实际收益。

更推荐的原则是：

```text
谁需要被替换，谁就适合抽象。
```

在 Gin 项目中，最常见的是：

```text
service 测试时，希望替换 repository。
```

所以可以让 service 依赖 repository 接口。

---

## 二、为什么 service 测试想替换 repository

假设 service 直接依赖具体 repository：

```go
type UserService struct {
    repo *repository.UserRepository
}
```

如果 repository 内部依赖数据库，那么测试 service 时就必须准备数据库。

这会让测试变得：

- 慢。
- 麻烦。
- 不稳定。
- 难以覆盖异常分支。

如果 service 依赖接口：

```go
type UserRepo interface {
    GetByEmail(email string) (model.User, error)
    Create(user model.User) (model.User, error)
}

type UserService struct {
    repo UserRepo
}
```

测试时就可以传一个 fake repo。

---

## 三、接口定义在哪里

Go 中常见建议：

```text
接口定义在使用方。
```

也就是说，`UserService` 需要什么能力，就在 service 包中定义接口。

示例：

```go
package service

import "gin-layer-demo/internal/model"

type UserRepo interface {
    GetByEmail(email string) (model.User, error)
    Create(user model.User) (model.User, error)
}

type UserService struct {
    repo UserRepo
}

func NewUserService(repo UserRepo) *UserService {
    return &UserService{repo: repo}
}
```

repository 只要实现了这些方法，就自动满足接口，不需要显式声明。

## 四、fake repository

测试中可以写：

```go
type fakeUserRepo struct {
    users map[string]model.User
}

func newFakeUserRepo() *fakeUserRepo {
    return &fakeUserRepo{
        users: make(map[string]model.User),
    }
}

func (r *fakeUserRepo) GetByEmail(email string) (model.User, error) {
    user, ok := r.users[email]
    if !ok {
        return model.User{}, repository.ErrUserNotFound
    }
    return user, nil
}

func (r *fakeUserRepo) Create(user model.User) (model.User, error) {
    if _, exists := r.users[user.Email]; exists {
        return model.User{}, repository.ErrEmailAlreadyExists
    }
    user.ID = uint(len(r.users) + 1)
    r.users[user.Email] = user
    return user, nil
}
```

这个 fake repo 不连接数据库，只在内存里模拟行为。

## 五、service 测试示例

测试创建用户默认状态：

```go
func TestUserService_Create_DefaultStatus(t *testing.T) {
    repo := newFakeUserRepo()
    svc := service.NewUserService(repo)

    user, err := svc.Create(service.CreateUserInput{
        Username: "alice",
        Email:    "alice@example.com",
        Age:      20,
    })
    if err != nil {
        t.Fatal(err)
    }

    if user.Status != "active" {
        t.Fatalf("expected active, got %s", user.Status)
    }
}
```

测试邮箱重复：

```go
func TestUserService_Create_DuplicateEmail(t *testing.T) {
    repo := newFakeUserRepo()
    svc := service.NewUserService(repo)

    input := service.CreateUserInput{
        Username: "alice",
        Email:    "alice@example.com",
        Age:      20,
    }

    _, err := svc.Create(input)
    if err != nil {
        t.Fatal(err)
    }

    _, err = svc.Create(input)
    if !errors.Is(err, service.ErrEmailAlreadyExists) {
        t.Fatalf("expected ErrEmailAlreadyExists, got %v", err)
    }
}
```

## 六、什么时候不需要接口

这些情况可以先不抽接口：

- 项目很小。
- 暂时没有测试需求。
- 只有一种实现。
- 抽接口后没有任何替换场景。

例如 handler 依赖 service 时，初学阶段可以直接：

```go
type UserHandler struct {
    service *service.UserService
}
```

等测试 handler 时再考虑是否抽 service 接口。

---

## 七、不要为了“看起来高级”抽象

不推荐：

```go
type IUserService interface {
    Create(...)
}

type UserServiceImpl struct {}
```

Go 里通常不需要 Java 风格的 `IXXX`、`Impl` 命名。

更自然的方式是：

```go
type UserService struct {}
```

需要接口时，再按使用方需要定义小接口。

---

## 八、接口越小越好

不要定义巨大接口：

```go
type UserRepo interface {
    Create(...)
    Update(...)
    Delete(...)
    GetByID(...)
    GetByEmail(...)
    List(...)
    Count(...)
    Search(...)
    Export(...)
}
```

如果某个 service 方法只需要 `GetByEmail` 和 `Create`，接口就可以只包含这两个方法。

小接口更容易 fake，也更容易维护。
