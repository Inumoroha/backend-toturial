# 1. 推荐目录结构与职责边界

本节目标：建立一个适合 Gin 入门项目的目录结构，并理解每个目录的职责边界。不要把目录结构当成固定模板死记，而要理解为什么这样拆。

---

## 一、从单文件开始的问题

入门时你可能有一个这样的 `main.go`：

```go
package main

func main() {
    r := gin.Default()

    api := r.Group("/api/v1")
    {
        api.GET("/users", listUsers)
        api.POST("/users", createUser)
        api.GET("/users/:id", getUser)
    }

    r.Run(":8080")
}
```

同时 `main.go` 里还有：

```text
User 结构体
CreateUserRequest 结构体
users map
listUsers
getUser
createUser
updateUser
deleteUser
response.OK
parseID
```

接口少时没问题。接口多了以后，`main.go` 很快会变成几百行，甚至上千行。

你会开始遇到这些问题：

- 找不到某个接口在哪里。
- 响应格式重复。
- 业务逻辑和 HTTP 逻辑混在一起。
- 想换存储方式时影响所有 handler。
- 想测试 service 时发现逻辑都在 handler 里。

所以需要按职责拆目录。

---

## 二、推荐目录结构

本阶段推荐结构：

```text
gin-layer-demo/
  cmd/
    server/
      main.go
  internal/
    handler/
      user_handler.go
    service/
      user_service.go
    repository/
      user_repository.go
    model/
      user.go
    response/
      response.go
      code.go
      validation.go
    router/
      router.go
    middleware/
    config/
  go.mod
  requests.http
```

这个结构不算复杂，但已经足够训练真实项目思维。

---

## 三、cmd/server

`cmd/server/main.go` 是程序入口。

它负责：

```text
加载配置。
初始化依赖。
组装 repository、service、handler。
创建 router。
启动 HTTP 服务。
```

它不应该负责：

```text
写具体业务逻辑。
写 handler 函数。
写数据库查询细节。
```

一个比较理想的 main 看起来像这样：

```go
func main() {
    userRepo := repository.NewUserRepository()
    userService := service.NewUserService(userRepo)
    userHandler := handler.NewUserHandler(userService)

    r := router.New(userHandler)

    if err := r.Run(":8080"); err != nil {
        log.Fatal(err)
    }
}
```

main 可以负责组装，但不要变成业务代码堆放处。

---

## 四、internal/model

model 放核心数据结构。

例如：

```go
package model

import "time"

type User struct {
    ID        uint      `json:"id"`
    Username  string    `json:"username"`
    Email     string    `json:"email"`
    Age       int       `json:"age"`
    Status    string    `json:"status"`
    CreatedAt time.Time `json:"created_at"`
}
```

model 不应该依赖 handler、service、repository。

它应该尽量稳定，因为很多层都会使用它。

---

## 五、internal/repository

repository 负责数据访问。

本阶段可以用内存 map：

```go
type UserRepository struct {
    users  map[uint]model.User
    nextID uint
}
```

后面第6阶段可以改成 GORM：

```go
type UserRepository struct {
    db *gorm.DB
}
```

但对上层暴露的方法可以尽量保持一致：

```go
Create(user model.User) (model.User, error)
GetByID(id uint) (model.User, error)
List(...) ([]model.User, int64, error)
Update(user model.User) (model.User, error)
Delete(id uint) error
```

这就是 repository 的价值：隐藏数据来源。

---

## 六、internal/service

service 负责业务逻辑。

例如：

```text
新用户默认 status=active。
用户状态只能是 active/disabled。
邮箱不能重复。
分页默认 page=1,page_size=10。
```

这些规则不应该放在 handler，也不应该放在 repository。

service 示例：

```go
type UserService struct {
    repo *repository.UserRepository
}
```

service 不应该导入：

```go
github.com/gin-gonic/gin
```

因为 service 不应该关心 HTTP 框架。

---

## 七、internal/handler

handler 负责 HTTP 请求和响应。

它做这些事：

```text
读取路径参数。
读取查询参数。
读取 JSON body。
调用 service。
根据 service 返回结果选择 HTTP 状态码。
调用 response.OK 或 response.Fail。
```

handler 示例：

```go
func (h *UserHandler) Get(c *gin.Context) {
    id, err := strconv.ParseUint(c.Param("id"), 10, 64)
    if err != nil {
        response.Fail(c, http.StatusBadRequest, response.CodeInvalidRequest, "invalid user id")
        return
    }

    user, err := h.service.Get(uint(id))
    if errors.Is(err, service.ErrUserNotFound) {
        response.Fail(c, http.StatusNotFound, response.CodeNotFound, "user not found")
        return
    }

    response.OK(c, user)
}
```

handler 可以依赖 Gin，这是它的职责范围。

---

## 八、internal/response

response 放统一响应格式、错误码、校验错误格式化等。

例如：

```go
func OK(c *gin.Context, data interface{}) {}
func Created(c *gin.Context, data interface{}) {}
func Fail(c *gin.Context, status int, code int, message string) {}
```

这样 handler 中就不会到处写：

```go
c.JSON(...)
```

统一响应对前端、测试和接口文档都很重要。

---

## 九、internal/router

router 负责路由注册。

例如：

```go
func New(userHandler *handler.UserHandler) *gin.Engine {
    r := gin.Default()

    api := r.Group("/api/v1")
    {
        api.GET("/users", userHandler.List)
        api.POST("/users", userHandler.Create)
        api.GET("/users/:id", userHandler.Get)
    }

    return r
}
```

这样 `main.go` 不需要知道每条路由细节。

## 十、internal/config 和 internal/middleware

本阶段可以先不深入，但先知道它们放什么。

config：

```text
加载 YAML。
读取环境变量。
保存 server/database/jwt/redis 配置。
```

middleware：

```text
认证中间件。
CORS。
Request ID。
请求日志。
Recovery。
```

这些会在后续阶段继续展开。

---

## 十一、依赖方向

推荐依赖方向：

```text
main -> router -> handler -> service -> repository -> model
```

response 可以被 handler 使用。

model 可以被 repository、service、handler 使用。

不要反过来：

```text
repository -> handler
service -> handler
model -> service
```

一旦出现循环依赖，通常说明职责边界出了问题。

---

## 十二、常见问题

### 1. 目录是不是越多越好

不是。

目录结构服务于理解和维护。小项目不需要一开始就做复杂架构。

本教程推荐的结构是 Gin 单体项目的基础结构，不是微服务架构。

### 2. handler 能不能直接操作数据库

学习时可以临时写，但真实项目不推荐。

直接操作数据库会让 HTTP 层和数据层耦合，后面测试和重构都困难。

### 3. service 为什么不能返回 HTTP 状态码

因为 service 不应该知道 HTTP。

service 可以返回业务错误：

```go
ErrUserNotFound
ErrEmailAlreadyExists
```

handler 再把这些错误转换为：

```text
404
409
```
