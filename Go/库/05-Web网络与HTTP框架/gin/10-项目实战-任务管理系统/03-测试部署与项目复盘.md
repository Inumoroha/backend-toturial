# 10-03 项目实战：测试、部署与复盘

本节目标：为任务管理系统补上测试、部署和复盘，让它从“功能写完”变成“项目完成”。

## 一、必须测试的核心场景

优先测试业务关键路径，不要一开始追求覆盖所有代码。

### 认证模块

- 注册成功。
- 邮箱重复注册失败。
- 密码太短注册失败。
- 登录成功返回 token。
- 密码错误登录失败。
- 未携带 token 访问私有接口失败。

### 任务模块

- 登录用户创建任务成功。
- 标题为空创建失败。
- 查询自己的任务成功。
- 查询别人的任务失败。
- 状态非法时创建或更新失败。
- 分页查询返回正确结构。
- 删除任务后再次查询返回不存在。

## 二、handler 测试示例

handler 测试关注 HTTP 行为：

- 请求方法。
- 请求路径。
- 请求体。
- 状态码。
- 响应 JSON。

示例：

```go
func TestCreateTaskUnauthorized(t *testing.T) {
    gin.SetMode(gin.TestMode)

    r := gin.New()
    r.POST("/api/v1/tasks", func(c *gin.Context) {
        response.Fail(c, http.StatusUnauthorized, 40101, "missing token")
    })

    body := strings.NewReader(`{"title":"learn gin"}`)
    req := httptest.NewRequest(http.MethodPost, "/api/v1/tasks", body)
    req.Header.Set("Content-Type", "application/json")

    w := httptest.NewRecorder()
    r.ServeHTTP(w, req)

    if w.Code != http.StatusUnauthorized {
        t.Fatalf("expected 401, got %d", w.Code)
    }
}
```

真正项目中，你应该测试完整路由和中间件，而不是只测试一个匿名函数。

## 三、service 测试建议

service 测试重点是业务规则：

- 默认状态是否正确。
- 非法状态是否拒绝。
- 分页默认值是否正确。
- 用户是否只能访问自己的数据。

为了方便测试，可以让 service 依赖接口：

```go
type TaskRepo interface {
    Create(task *model.Task) error
    GetByIDAndUserID(id, userID uint) (*model.Task, error)
    ListByUserID(userID uint, page, pageSize int, status string) ([]model.Task, int64, error)
}
```

测试时写一个 fake：

```go
type fakeTaskRepo struct {
    tasks map[uint]*model.Task
}
```

这样不用真的连接数据库。

## 四、接口文档

项目 README 至少要写：

- 项目介绍。
- 技术栈。
- 本地启动方式。
- 配置说明。
- 接口文档地址。
- 测试命令。
- Docker 启动方式。

Swagger 文档至少覆盖：

- 注册。
- 登录。
- 获取个人信息。
- 创建任务。
- 查询任务列表。
- 查询任务详情。
- 更新任务。
- 删除任务。

## 五、Docker Compose 部署

最终项目应该能通过一条命令启动：

```bash
docker compose up -d
```

启动后检查：

```bash
docker compose ps
docker compose logs -f app
```

验证：

```text
GET http://localhost:8080/healthz
```

然后按顺序测试：

1. 注册用户。
2. 登录获取 token。
3. 携带 token 创建任务。
4. 查询任务列表。
5. 更新任务状态。
6. 删除任务。

## 六、项目验收标准

一个合格的任务管理系统至少满足：

- 服务能一键启动。
- 注册登录流程正常。
- 私有接口必须认证。
- 用户不能访问别人的任务。
- 参数错误有明确响应。
- 数据持久化到数据库。
- 核心业务有测试。
- README 能指导别人启动项目。
- Docker Compose 能启动 app、MySQL、Redis。

## 七、复盘问题

项目完成后，请认真回答这些问题：

- 哪个模块写得最顺？
- 哪个模块最容易出错？
- 有没有把太多逻辑写进 handler？
- 哪些错误处理重复了？
- 哪些代码适合抽成公共函数？
- 测试是否覆盖了最重要的业务规则？
- 如果下次重写，我会怎么设计目录？

这些问题比“项目做完了”更重要。复盘会让你真正变强。

## 八、下一步提升方向

完成任务管理系统后，可以继续扩展：

- 增加团队空间。
- 增加任务成员分配。
- 增加任务截止时间和提醒。
- 增加操作日志。
- 增加管理员后台。
- 增加 OpenAPI 自动生成客户端。
- 引入数据库迁移工具。
- 引入 CI 流水线。

不要急着上微服务。先把单体项目写清楚、测清楚、部署清楚。优秀的单体项目，比混乱的微服务项目更能体现工程能力。

## 九、最终验收清单

你应该能够独立完成：

- 从 0 创建 Gin 项目。
- 设计 RESTful API。
- 使用 GORM 建模和查询。
- 实现 JWT 登录认证。
- 编写认证中间件。
- 使用 Redis 做缓存或限流。
- 编写配置、日志和测试。
- 生成 Swagger 文档。
- 使用 Docker Compose 部署。
- 写出清晰 README。

做到这里，你已经具备 Go 后端工程师的入门工程能力。接下来要靠更多项目和更多真实问题继续打磨。
