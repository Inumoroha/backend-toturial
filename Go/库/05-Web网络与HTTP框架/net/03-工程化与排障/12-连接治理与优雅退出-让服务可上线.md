# 12. 连接治理与优雅退出：让服务可上线

本节目标：学完后你能给 TCP 服务增加连接数限制和优雅退出流程。

简短引入：能跑的服务不一定能上线。上线服务要能限制资源、接收停止信号、关闭监听、等待正在处理的连接完成，并在超时后退出。否则发布、扩容、回滚时都可能中断业务。

## 一、为什么需要它

真实后端服务要面对：

- 客户端连接突然增多。
- 发布时进程收到退出信号。
- 连接处理还没完成。
- 监听器关闭后 `Accept` 返回错误。

如果没有治理，服务可能在流量高峰耗尽资源，或在发布时直接断开正在处理的请求。

```text
生产服务不只要会启动，也要会有序地停止。
```

## 二、基本用法

下面示例支持 Ctrl+C 优雅退出，并限制最多 100 个并发连接：

```go
package main

import (
	"bufio"
	"context"
	"errors"
	"fmt"
	"net"
	"os"
	"os/signal"
	"sync"
	"syscall"
	"time"
)

func main() {
	ctx, stop := signal.NotifyContext(context.Background(), os.Interrupt, syscall.SIGTERM)
	defer stop()

	ln, err := net.Listen("tcp", "127.0.0.1:9005")
	if err != nil {
		fmt.Println("listen failed:", err)
		return
	}

	var wg sync.WaitGroup
	sem := make(chan struct{}, 100)

	go func() {
		<-ctx.Done()
		fmt.Println("shutdown signal received")
		_ = ln.Close()
	}()

	fmt.Println("server listening on", ln.Addr())

	for {
		conn, err := ln.Accept()
		if err != nil {
			if errors.Is(err, net.ErrClosed) {
				break
			}
			fmt.Println("accept failed:", err)
			continue
		}

		select {
		case sem <- struct{}{}:
			wg.Add(1)
			go func() {
				defer wg.Done()
				defer func() { <-sem }()
				handle(conn)
			}()
		default:
			fmt.Fprintln(conn, "ERR server busy")
			_ = conn.Close()
		}
	}

	done := make(chan struct{})
	go func() {
		wg.Wait()
		close(done)
	}()

	select {
	case <-done:
		fmt.Println("all connections finished")
	case <-time.After(5 * time.Second):
		fmt.Println("shutdown timeout, exit anyway")
	}
}

func handle(conn net.Conn) {
	defer conn.Close()
	_ = conn.SetDeadline(time.Now().Add(5 * time.Second))

	line, err := bufio.NewReader(conn).ReadString('\n')
	if err != nil {
		fmt.Fprintln(conn, "ERR bad request")
		return
	}

	fmt.Fprintf(conn, "OK %s", line)
}
```

运行：

```bash
go run .
```

Windows PowerShell 和 Linux/macOS 都可以用 `Ctrl+C` 发送中断信号。

## 三、关键参数/语法/代码结构

`signal.NotifyContext`：

- 把系统退出信号转成 `context`。
- 适合统一管理退出流程。

`ln.Close()`：

- 关闭监听器，让 `Accept` 从阻塞中返回。
- 这一步不会自动关闭已经接入的连接。

`sync.WaitGroup`：

- 等待正在处理的连接完成。
- 避免服务一收到信号就立刻退出。

`sem := make(chan struct{}, 100)`：

- 用带缓冲 channel 做并发连接数限制。
- 超出限制时快速返回 `server busy`。

## 四、真实后端场景示例

订单通知网关可以这样上线：

- 监听端口接收合作方通知。
- 每个连接设置 5 秒 deadline。
- 同时最多处理 100 个连接。
- 发布时先关闭监听，不再接收新连接。
- 等待已接入连接处理完成，最多等待 5 秒。
- 如果发布失败，可以回滚上一版本。

这里的核心不是代码复杂，而是每个状态都可控。

## 五、注意点

- 连接数限制要结合机器规格、下游能力和业务 SLA 设置。
- 拒绝请求要返回明确错误，方便客户端重试或降级。
- 优雅退出要有最大等待时间，避免永远停不下来。
- 如果业务写数据库，退出时不能留下半完成状态，事务要么提交，要么回滚。

## 六、常见误区

- 误区一：收到退出信号后直接 `os.Exit`。  
  这样会跳过 defer，正在处理的连接和事务可能被硬切。

- 误区二：只关闭监听器，以为所有连接都结束了。  
  已接入连接仍在处理，需要单独等待或主动关闭。

- 误区三：没有连接数上限。  
  流量异常时服务可能耗尽文件描述符、内存或下游连接池。

## 七、本节达标标准

- 能使用 `signal.NotifyContext` 接收退出信号。
- 能关闭监听器并识别 `net.ErrClosed`。
- 能用 `WaitGroup` 等待连接处理完成。
- 能用 channel 做简单连接数限制。
