# 11. 网络错误处理与日志：让问题可定位

本节目标：学完后你能识别常见网络错误，并记录对排障有用的日志。

简短引入：网络错误很少只写着“真正原因”。你可能看到 `i/o timeout`、`connection reset by peer`、`EOF`、`use of closed network connection`。后端工程师要做的是：把错误分类、记录上下文、避免日志噪音。

## 一、为什么需要它

真实项目中，网络问题往往跨多个组件：

- 客户端、网关、服务端、数据库之间任何一段都可能失败。
- 错误可能偶发，复现困难。
- 没有地址、耗时、命令、连接 ID 的日志，很难定位。

```text
好的网络日志不是越多越好，而是能回答：谁连谁、等了多久、失败在哪一步。
```

## 二、基本用法

下面示例识别超时错误：

```go
package main

import (
	"errors"
	"fmt"
	"net"
	"time"
)

func main() {
	conn, err := net.DialTimeout("tcp", "10.255.255.1:9000", 500*time.Millisecond)
	if err != nil {
		var netErr net.Error
		if errors.As(err, &netErr) && netErr.Timeout() {
			fmt.Println("network timeout:", err)
			return
		}
		fmt.Println("network error:", err)
		return
	}
	defer conn.Close()
}
```

运行：

```bash
go run .
```

## 三、关键参数/语法/代码结构

`errors.As(err, &netErr)`：

- 判断错误链里是否包含 `net.Error`。
- 比直接字符串匹配更稳。

`netErr.Timeout()`：

- 判断是否是超时。
- 超时通常意味着可以告警或按策略重试，但要先确认业务是否幂等。

`net.ErrClosed`：

- Go 标准库中用于表示网络连接或监听器已关闭的错误。
- 优雅退出时常会遇到，通常不应当当成异常刷屏。

`*net.OpError`：

- 很多网络错误会被包装成 `*net.OpError`。
- 它通常包含操作名，例如 `dial`、`read`、`write`。
- 排障时可以用它判断错误发生在哪一步。

示例：

```go
var opErr *net.OpError
if errors.As(err, &opErr) {
	fmt.Println("op:", opErr.Op)
	fmt.Println("network:", opErr.Net)
	fmt.Println("addr:", opErr.Addr)
}
```

日志级别可以按场景区分：

- `debug`：客户端主动断开、健康检查失败但已自动恢复。
- `info`：连接建立、请求处理完成、服务开始退出。
- `warn`：超时、协议错误、服务繁忙、可恢复的下游失败。
- `error`：监听失败、核心下游不可用、数据写入失败、无法自动恢复的问题。

## 四、真实后端场景示例

给连接处理增加基础日志：

```go
func handle(conn net.Conn) {
	start := time.Now()
	remote := conn.RemoteAddr().String()
	defer conn.Close()

	if err := conn.SetReadDeadline(time.Now().Add(3 * time.Second)); err != nil {
		fmt.Printf("remote=%s set_deadline_error=%v\n", remote, err)
		return
	}

	line, err := bufio.NewReader(conn).ReadString('\n')
	if err != nil {
		var netErr net.Error
		if errors.As(err, &netErr) && netErr.Timeout() {
			fmt.Printf("remote=%s event=read_timeout cost=%s\n", remote, time.Since(start))
			return
		}
		fmt.Printf("remote=%s event=read_error err=%v cost=%s\n", remote, err, time.Since(start))
		return
	}

	fmt.Printf("remote=%s event=request bytes=%d cost=%s\n", remote, len(line), time.Since(start))
}
```

真实项目中建议使用结构化日志，例如 `zap`、`slog`、`zerolog`。学习阶段先用 `fmt.Printf` 理解字段即可。

一个更接近生产的日志字段可以这样设计：

```text
time=... level=warn event=read_timeout conn_id=abc123 remote=10.0.0.12:53320 protocol=tcp stage=read cost_ms=3001
```

字段不需要一开始就很多，但至少要覆盖：

- `event`：发生了什么。
- `remote`：谁连接过来。
- `stage`：失败在哪一步，例如 `accept`、`read`、`parse`、`write`。
- `cost_ms`：耗时多少。
- `err`：错误摘要。

如果服务有链路追踪，还应把 `trace_id` 或 `request_id` 写进日志。这样网络层日志、业务层日志和数据库日志才能串起来。

## 五、注意点

- 日志里不要打印密码、token、完整身份证、完整手机号等敏感信息。
- 网络输入可能很大，不要原样整包写日志。
- 超时、拒绝连接、协议错误、业务错误要分开统计。
- 重试日志要带重试次数，否则很难判断是否放大了故障。
- 对高频错误要限速或采样，避免故障时日志把磁盘打满。
- 记录错误时尽量带阶段，不要只写 `network failed`。
- 客户端错误和服务端错误要分开看，客户端断开不一定是服务端 bug。

```text
日志要帮助回放事实，但不能泄露敏感数据，也不能制造新的稳定性问题。
```

## 六、常见误区

- 误区一：用字符串判断错误类型。  
  错误文本可能变化，优先用 `errors.As`、`errors.Is`。

- 误区二：把所有错误都打成 error 级别。  
  客户端主动断开、服务优雅退出时的关闭错误，可能只是正常现象。

- 误区三：日志只写 `failed`。  
  没有 remote、耗时、阶段和错误类型，排障价值很低。

- 误区四：把完整请求体都打到日志里。  
  这可能泄露敏感信息，也可能在异常流量下拖垮日志系统。

## 七、本节达标标准

- 能使用 `errors.As` 判断 `net.Error`。
- 能识别超时错误并区别普通错误。
- 能设计包含 remote、event、cost、err 的基础日志。
- 能说明网络日志中哪些内容不应该直接打印。
- 能根据日志里的阶段字段判断问题发生在接入、读取、解析、业务还是写回。
