# 6. 超时与取消：不要让慢连接拖垮服务

本节目标：学完后你能给连接、读写和拨号设置超时，避免慢连接长期占用服务资源。

简短引入：网络程序最怕“等”。等连接、等对方发数据、等下游响应，如果没有上限，服务就会被少量慢请求拖住。超时不是优化项，而是后端服务的安全边界。

## 一、为什么需要它

真实项目里常见的慢连接问题：

- 客户端连上后迟迟不发送数据。
- 下游服务网络抖动，拨号一直等待。
- 对方读取很慢，服务端写响应被卡住。
- 攻击者故意制造大量慢连接占资源。

```text
所有跨网络等待都应该有上限；没有超时，就没有可控性。
```

## 二、基本用法

服务端给每个连接设置读写总截止时间：

```go
package main

import (
	"bufio"
	"fmt"
	"net"
	"strings"
	"time"
)

func main() {
	ln, err := net.Listen("tcp", "127.0.0.1:9002")
	if err != nil {
		fmt.Println("listen failed:", err)
		return
	}
	defer ln.Close()

	fmt.Println("server listening on", ln.Addr())
	for {
		conn, err := ln.Accept()
		if err != nil {
			fmt.Println("accept failed:", err)
			continue
		}
		go handle(conn)
	}
}

func handle(conn net.Conn) {
	defer conn.Close()

	_ = conn.SetDeadline(time.Now().Add(5 * time.Second))

	line, err := bufio.NewReader(conn).ReadString('\n')
	if err != nil {
		fmt.Fprintln(conn, "ERR read timeout or bad request")
		return
	}

	orderID := strings.TrimSpace(line)
	if orderID == "" {
		fmt.Fprintln(conn, "ERR empty order_id")
		return
	}

	fmt.Fprintf(conn, "OK accepted order=%s\n", orderID)
}
```

客户端使用 `Dialer.DialContext`：

```go
package main

import (
	"context"
	"fmt"
	"net"
	"time"
)

func main() {
	ctx, cancel := context.WithTimeout(context.Background(), 2*time.Second)
	defer cancel()

	dialer := net.Dialer{
		Timeout:   2 * time.Second,
		KeepAlive: 30 * time.Second,
	}

	conn, err := dialer.DialContext(ctx, "tcp", "127.0.0.1:9002")
	if err != nil {
		fmt.Println("dial failed:", err)
		return
	}
	defer conn.Close()

	fmt.Fprintln(conn, "order-1001")
}
```

## 三、关键参数/语法/代码结构

`SetDeadline(t)`：

- 设置整个连接读写的截止时间。
- 到时间后，后续读写会返回超时错误。

`SetReadDeadline(t)`：

- 只限制读。
- 适合防止客户端连上后不发数据。

`SetWriteDeadline(t)`：

- 只限制写。
- 适合防止响应发送卡住。

`Dialer.DialContext(ctx, network, address)`：

- 用 `context` 控制拨号取消。
- 真实项目中更容易和请求级超时打通。

## 四、真实后端场景示例

订单服务接收外部系统通知时，可以这样设计超时：

- 连接建立：2 秒内完成。
- 请求读取：3 秒内必须发完整一行。
- 业务处理：由业务层 `context.WithTimeout` 控制，例如 1 秒。
- 响应写回：2 秒内完成。

这样做的目的不是让请求一定成功，而是让失败尽快暴露，避免占住连接和 goroutine。

## 五、注意点

- `SetDeadline` 是绝对时间，不是持续时间。每次延长都要重新设置 `time.Now().Add(...)`。
- 长连接场景中，通常每处理一条消息就刷新读写 deadline。
- 超时后连接状态通常不可继续信任，保守做法是关闭连接。
- 重试要谨慎，尤其是订单扣库存、支付回调等可能产生副作用的操作。

```text
涉及状态变更的网络调用，重试前必须确认幂等性和事务边界。
```

## 六、常见误区

- 误区一：只设置连接超时，不设置读写超时。  
  连接成功后仍可能卡在读写上。

- 误区二：把超时时间设置得非常大。  
  大到接近无限等待，就失去了保护意义。

- 误区三：超时后继续复用连接。  
  对简单后端服务而言，超时后关闭连接更保守。

## 七、本节达标标准

- 能使用 `Dialer.DialContext` 设置拨号超时。
- 能使用 `SetReadDeadline` 或 `SetDeadline` 保护服务端读写。
- 能解释连接超时、读超时、写超时的区别。
- 能说明为什么状态变更请求不能随便重试。
