# 4. 错误响应与状态码设计

本节目标：建立一套清晰的错误响应习惯。真实 API 中，错误路径和成功路径同样重要。

---

## 一、为什么不能所有错误都返回 200

错误示例：

```http
HTTP/1.1 200 OK
Content-Type: application/json

{"error":"todo not found"}
```

问题是：HTTP 层说成功，业务层说失败。客户端必须解析 Body 才知道失败，统一处理会很麻烦。

更好的响应：

```http
HTTP/1.1 404 Not Found
Content-Type: application/json

{"error":"todo not found"}
```

状态码应该和请求结果一致。

---

## 二、常见状态码选择

创建成功：

```text
201 Created
```

参数错误：

```text
400 Bad Request
```

未登录或 Token 错误：

```text
401 Unauthorized
```

登录了但没权限：

```text
403 Forbidden
```

资源不存在：

```text
404 Not Found
```

方法不允许：

```text
405 Method Not Allowed
```

资源冲突：

```text
409 Conflict
```

服务端内部错误：

```text
500 Internal Server Error
```

---

## 三、统一错误格式

建议 JSON API 统一错误格式：

```json
{
  "error": {
    "code": "invalid_request",
    "message": "title is required"
  }
}
```

定义：

```go
type errorResponse struct {
	Error errorDetail `json:"error"`
}

type errorDetail struct {
	Code    string `json:"code"`
	Message string `json:"message"`
}
```

封装：

```go
func writeError(w http.ResponseWriter, status int, code, message string) {
	writeJSON(w, status, errorResponse{
		Error: errorDetail{
			Code:    code,
			Message: message,
		},
	})
}
```

---

## 四、成功响应也要统一

小项目可以直接返回资源：

```json
{"id":1,"title":"learn"}
```

列表可以返回：

```json
{
  "items": [
    {"id":1,"title":"learn"}
  ]
}
```

分页可以返回：

```json
{
  "items": [],
  "page": 1,
  "page_size": 20,
  "total": 0
}
```

不要在同一个项目里一会儿返回数组，一会儿返回对象，一会儿把错误塞在 `message`，一会儿塞在 `msg`。

---

## 五、不要暴露内部错误

错误示例：

```json
{
  "error": "pq: duplicate key value violates unique constraint users_email_key"
}
```

这会把数据库实现细节暴露给客户端。

更好的响应：

```json
{
  "error": {
    "code": "email_conflict",
    "message": "email already exists"
  }
}
```

内部错误可以写日志，但响应给客户端的信息要克制。

---

## 六、错误处理顺序

一个 Handler 常见结构：

```go
func createTodo(w http.ResponseWriter, r *http.Request) {
	var req createTodoRequest
	if err := readJSON(w, r, &req); err != nil {
		writeError(w, http.StatusBadRequest, "invalid_json", "invalid json body")
		return
	}

	if strings.TrimSpace(req.Title) == "" {
		writeError(w, http.StatusBadRequest, "invalid_request", "title is required")
		return
	}

	todo, err := service.Create(r.Context(), req.Title)
	if err != nil {
		writeError(w, http.StatusInternalServerError, "internal_error", "internal server error")
		return
	}

	writeJSON(w, http.StatusCreated, todo)
}
```

顺序是：

```text
解析输入
-> 校验输入
-> 调用业务
-> 处理业务错误
-> 返回成功
```

---

## 七、http.Error 什么时候用

`http.Error` 简单方便：

```go
http.Error(w, "bad request", http.StatusBadRequest)
```

但它返回的是纯文本，不适合统一 JSON API。

建议：

- 学习早期可以用。
- 写 JSON API 后，改用自己的 `writeError`。
- 中间件和项目实战中统一错误格式。
