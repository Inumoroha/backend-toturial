# 2. 客户端断开与取消传播

本节目标：通过实验理解客户端断开连接后，请求 context 如何被取消。

---

## 一、为什么要关心客户端断开

假设有一个接口要处理 20 秒：

```text
用户发起请求
-> 服务端开始计算
-> 用户 2 秒后关闭页面
-> 服务端如果不知道，仍然继续计算 18 秒
```

这会浪费资源。

如果 Handler 监听 `r.Context().Done()`，就能及时停止。

---

## 二、错误示例：不关心取消

```go
mux.HandleFunc("GET /slow-bad", func(w http.ResponseWriter, r *http.Request) {
	time.Sleep(10 * time.Second)
	fmt.Fprintln(w, "done")
})
```

客户端断开后，服务端仍会继续 sleep。

---

## 三、正确示例：监听取消

```go
mux.HandleFunc("GET /slow-good", func(w http.ResponseWriter, r *http.Request) {
	select {
	case <-time.After(10 * time.Second):
		fmt.Fprintln(w, "done")
	case <-r.Context().Done():
		log.Printf("client gone: %v", r.Context().Err())
		return
	}
})
```

如果客户端断开，服务端日志会打印取消原因。

---

## 四、用 curl 测试

启动服务：

```bash
go run .
```

发起请求：

```bash
curl -i http://localhost:8080/slow-good
```

在等待过程中按：

```text
Ctrl+C
```

观察服务端日志。

---

## 五、context 取消不等于自动停止所有工作

context 被取消后，只有主动检查它的代码才会停止。

例如：

```go
time.Sleep(10 * time.Second)
```

不会因为 context 取消而自动结束。

你需要写成：

```go
select {
case <-time.After(10 * time.Second):
case <-ctx.Done():
	return ctx.Err()
}
```

数据库、HTTP Client 等标准库或驱动中的 Context 版本方法，也会监听 context。

---

## 六、业务循环中检查取消

如果你有一个循环任务：

```go
for i := 0; i < 100; i++ {
	doWork(i)
}
```

可以改成：

```go
for i := 0; i < 100; i++ {
	select {
	case <-ctx.Done():
		return ctx.Err()
	default:
	}

	doWork(i)
}
```

这样请求取消后，循环能尽快退出。
