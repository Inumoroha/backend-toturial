# 10. 接口验证与并发思考

本节目标：用 curl 验证短链接服务完整闭环，并思考并发下的唯一性和访问计数问题。

---

## 一、创建短链接

```bash
curl -i -X POST http://localhost:8080/links `
  -H "Content-Type: application/json" `
  -d "{\"original_url\":\"https://example.com/posts/go-net-http\"}"
```

预期：

```text
201 Created
```

响应里应该包含：

```json
{
  "code": "...",
  "short_url": "http://localhost:8080/..."
}
```

---

## 二、访问短码

假设 code 是 `abc123`：

```bash
curl -i http://localhost:8080/abc123
```

预期：

```text
302 Found
Location: https://example.com/posts/go-net-http
```

如果想让 curl 跟随跳转：

```bash
curl -L -i http://localhost:8080/abc123
```

---

## 三、查询详情

```bash
curl -i http://localhost:8080/links/abc123
```

多访问几次短码后，再查详情，观察：

```json
"visit_count": 3
```

---

## 四、验证错误场景

URL 为空：

```bash
curl -i -X POST http://localhost:8080/links `
  -H "Content-Type: application/json" `
  -d "{\"original_url\":\"\"}"
```

非法 scheme：

```bash
curl -i -X POST http://localhost:8080/links `
  -H "Content-Type: application/json" `
  -d "{\"original_url\":\"ftp://example.com/file\"}"
```

不存在短码：

```bash
curl -i http://localhost:8080/not-exists
```

---

## 五、并发思考

短链接服务有两个关键并发点：

```text
短码创建：多个请求可能生成相同 code。
访问计数：多个请求可能同时更新 visit_count。
```

内存版本靠 `sync.RWMutex` 保证 map 安全。

数据库版本应该靠：

```text
UNIQUE(code) 保证短码唯一。
UPDATE links SET visit_count = visit_count + 1 原子更新计数。
```

---

## 六、压测跳转接口

```bash
hey -n 1000 -c 50 http://localhost:8080/abc123
```

观察：

- 是否有 500。
- `visit_count` 是否接近请求次数。
- 请求日志中 P95/P99 是否明显升高。

内存版本只是学习实现，不代表真实高并发架构。

---

## 七、本节检查点

你应该能完成：

```text
创建短链
-> 访问短码 302
-> 查询详情
-> 访问次数增加
-> 错误 URL 返回 400
-> 不存在短码返回 404
```
