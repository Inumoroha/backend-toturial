# 2. 超时配置详解

本节目标：理解 `http.Server` 中几个常见超时参数分别解决什么问题。

---

## 一、先看一次请求的大致过程

一次服务端请求可以拆成：

```text
建立连接
-> 读取请求头
-> 读取请求体
-> Handler 处理业务
-> 写响应
-> 连接保持空闲，等待复用
```

不同超时参数作用在不同阶段。

---

## 二、ReadHeaderTimeout

```go
ReadHeaderTimeout: 5 * time.Second
```

它限制读取请求头的时间。

请求头包括：

```text
GET /todos HTTP/1.1
Host: localhost:8080
User-Agent: curl/...
Authorization: Bearer ...
```

如果客户端连接上来后，很慢很慢地发送 Header，就会占住服务端连接。`ReadHeaderTimeout` 可以防止这种慢速攻击或异常连接。

通常建议生产服务都设置。

---

## 三、ReadTimeout

```go
ReadTimeout: 10 * time.Second
```

它限制读取整个请求的时间，包括 Header 和 Body。

对于 JSON API，请求体通常不大，设置 10 秒比较合理。

但如果是大文件上传，`ReadTimeout` 太短可能导致正常上传失败。这时要根据业务单独设计上传接口的超时和大小限制。

---

## 四、WriteTimeout

```go
WriteTimeout: 10 * time.Second
```

它限制服务端写响应的时间。

如果客户端接收很慢，服务端可能一直卡在写响应。`WriteTimeout` 可以避免连接长时间占用。

注意：如果你的接口需要长时间流式响应，比如 SSE、下载大文件，`WriteTimeout` 需要谨慎设置，否则可能中途断开。

---

## 五、IdleTimeout

```go
IdleTimeout: 60 * time.Second
```

HTTP/1.1 默认支持 keep-alive。客户端请求结束后，连接可以暂时保持，后续请求复用同一个连接。

`IdleTimeout` 控制连接空闲多久后关闭。

太短：

- 连接复用效果差。
- 客户端频繁重连。

太长：

- 空闲连接占用资源更久。

60 秒是常见起点，真实项目要结合网关和负载均衡配置。

---

## 六、TimeoutHandler 不是 Server 超时的替代

标准库还有：

```go
http.TimeoutHandler(handler, 2*time.Second, "timeout")
```

它给 Handler 处理设置超时，但它不是 Server 读写超时的替代。

两者关注点不同：

- `http.Server` 超时关注连接读写。
- `TimeoutHandler` 关注 Handler 执行时间。

实际项目中，更常见的做法是在业务调用中使用 `context.WithTimeout`，后面 Context 阶段会讲。

---

## 七、推荐起步配置

普通 JSON API 可以先用：

```go
srv := &http.Server{
	Addr:              ":8080",
	Handler:           handler,
	ReadHeaderTimeout: 5 * time.Second,
	ReadTimeout:       10 * time.Second,
	WriteTimeout:      10 * time.Second,
	IdleTimeout:       60 * time.Second,
	MaxHeaderBytes:    1 << 20,
}
```

这不是永远正确的配置，但比完全不设置超时更稳。
