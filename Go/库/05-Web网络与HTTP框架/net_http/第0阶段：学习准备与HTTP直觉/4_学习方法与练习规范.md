# 4. 学习方法与练习规范

本节目标：建立一套适合学习 `net/http` 的练习方式。后端能力不是靠看完文档获得的，而是靠反复写、反复请求、反复排错建立起来的。

---

## 一、每节课都要形成闭环

学习每个知识点时，建议都完成这个闭环：

```text
读概念
-> 写最小代码
-> 启动服务
-> 发请求
-> 看响应
-> 改一处代码
-> 再发请求验证
```

不要只停留在“我看懂了”。HTTP 服务的很多细节只有运行后才会暴露，比如：

- Header 设置晚了是否生效。
- `WriteHeader` 调多次会怎样。
- Body 读完后能不能再读。
- 客户端断开后 context 是否取消。
- 响应状态码和业务错误是否一致。

---

## 二、推荐的练习目录

前期可以用一个小目录反复修改：

```text
playground/
  go.mod
  main.go
```

到中期开始学习工程组织时，再拆成：

```text
todo-api/
  cmd/
    server/
      main.go
  internal/
    handler/
    middleware/
    model/
    service/
  go.mod
```

不要一开始就设计复杂目录。初学阶段最重要的是看清一次请求如何进入 Handler，又如何写出响应。

---

## 三、每个阶段都要写 README

建议每完成一个小项目，都写一个简单 README，至少包含：

- 项目做了什么。
- 如何启动。
- 有哪些接口。
- 如何用 curl 测试。
- 已知问题。
- 下一步准备补什么。

README 不是形式主义。它能训练你把代码能力转换成工程交付能力。

---

## 四、代码风格约定

本教程默认遵守这些习惯：

- Handler 不堆大量业务逻辑。
- JSON 响应统一通过辅助函数写。
- 错误路径和成功路径都要处理。
- 不忽略 `error`。
- 所有外部调用都带 `context.Context`。
- 服务端配置明确超时时间。
- 测试优先覆盖核心行为。

这些习惯一开始会显得麻烦，但它们会让项目变得可维护。

---

## 五、学习时要主动制造错误

建议你每学一个 API，都故意犯几个错：

- JSON 少一个右括号。
- 不设置 `Content-Type`。
- 访问不存在的路径。
- 用 GET 请求 POST 接口。
- 请求体传空。
- 服务端返回 500。

然后观察：

- curl 输出是什么。
- 服务端日志是什么。
- Go 代码报错在哪里。
- 状态码是否符合预期。

会排错，比只会写成功路径重要得多。

---

## 六、什么时候可以进入下一阶段

不要用“看完文档”作为进入下一阶段的标准。建议用下面的标准：

- 能不看答案写出本阶段最小示例。
- 能用 curl 验证成功和失败场景。
- 能解释关键 API 的作用。
- 能说出至少一个常见坑。

例如学完第 1 阶段后，你应该能解释：

```text
http.HandleFunc 和 http.NewServeMux 有什么关系？
ResponseWriter 是用来写什么的？
Request 里能读到哪些请求信息？
ListenAndServe 为什么会阻塞？
```

---

## 七、记录问题清单

建议在学习目录里维护一个 `QUESTIONS.md`：

```markdown
# 问题记录

## 2026-07-05

- 为什么 WriteHeader 之后再设置 Header 不生效？
- 为什么 r.Body 读完一次就空了？
- 为什么 curl 返回 404，但服务端没报错？
```

后面学到相关内容时再回来补答案。这样问题会变成你的学习线索。

---

## 八、本阶段结束检查

你可以进入第 1 阶段，如果你已经能做到：

- Go 环境可用。
- curl 可用。
- 理解 HTTP 请求和响应的基本结构。
- 知道 Method、Path、Query、Header、Body、Status Code 的作用。
- 知道后续每节都要写代码并用 curl 验证。

准备好了，就可以开始写第一个 `net/http` 服务。
