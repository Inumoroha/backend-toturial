# Go net/http 后续进阶学习方向

这份文档用于回答一个问题：

```text
学完当前 net/http 系统教程后，后面还可以继续学什么？
```

前面 0 到 13 阶段已经覆盖了 Go 标准库 `net/http` 的主线能力：

- HTTP 基础直觉。
- 第一个 HTTP 服务。
- 请求与响应处理。
- Handler、ServeMux、Middleware。
- `http.Server` 配置与优雅关闭。
- `context.Context` 请求生命周期。
- `httptest` 测试。
- HTTP Client。
- 文件上传下载。
- 安全基础与 CORS。
- 日志、pprof、压测。
- `net/http` 源码阅读。
- Todo API 项目。
- 短链接服务项目。

到这里，你已经具备了使用 Go 标准库编写普通 Web API 服务的基础能力。

后续进阶不建议继续“横向堆 API”，而是围绕真实后端工程问题分方向深入。

---

## 一、进阶学习的总体思路

`net/http` 后续进阶可以分成七条线：

```text
工程化线。
协议深入线。
性能优化线。
安全加固线。
可观测线。
框架与生态线。
项目实战线。
```

不建议七条线同时推进。

更好的顺序是：

```text
先补工程化。
再补性能和可观测。
然后结合项目学习安全、协议和框架。
最后做一个更完整的综合项目。
```

进阶阶段最重要的是：

```text
带着线上问题学习。
用真实项目验证。
用压测、日志、pprof、测试证明结论。
```

---

## 二、工程化进阶

适合目标：

```text
能把一个 net/http 项目组织成可维护、可测试、可部署的后端服务。
```

### 需要继续学习

- 配置管理。
- 分环境配置：local、test、prod。
- 日志初始化与注入。
- 依赖组装方式。
- Handler、Service、Repository 分层。
- 错误类型设计。
- 统一响应与错误码设计。
- 项目 Makefile。
- Dockerfile。
- 健康检查与 readiness。
- 灰度发布和优雅关闭配合。

### 推荐练习

把 Todo API 或短链接服务改造成：

```text
cmd/server/main.go
internal/config
internal/httpapi
internal/middleware
internal/service
internal/store
internal/logger
```

并补齐：

- `README.md`
- `Makefile`
- `Dockerfile`
- `.env.example`
- `go test ./...`
- `go test -race ./...`

### 达标标准

你应该能做到：

- 新同事看 README 能启动项目。
- 配置不写死在代码里。
- 服务支持优雅关闭。
- 日志格式统一。
- 错误响应格式统一。
- 核心业务有测试覆盖。

---

## 三、HTTP 协议深入

适合目标：

```text
不只会写接口，还能理解 HTTP 行为背后的协议机制。
```

### 需要继续学习

- HTTP/1.1 keep-alive。
- HTTP/2 基础。
- HTTPS 与 TLS 握手。
- 状态码语义。
- 缓存 Header。
- 条件请求：`ETag`、`If-None-Match`。
- Range 请求。
- gzip 压缩。
- 反向代理中的 Header。
- WebSocket 与普通 HTTP 的关系。
- Server-Sent Events。

### 推荐练习

给文件服务增加：

- `ETag`
- `Cache-Control`
- gzip 响应压缩
- Range 下载

再用 curl 验证：

```bash
curl -I http://localhost:8080/static/app.js
curl -H "If-None-Match: ..." -i http://localhost:8080/static/app.js
curl -H "Range: bytes=0-99" -i http://localhost:8080/files/demo.mp4
```

### 达标标准

你应该能解释：

- keep-alive 为什么能提升性能。
- HTTP/2 相比 HTTP/1.1 解决了什么。
- HTTPS 保护了什么，不保护什么。
- 缓存 Header 如何影响浏览器行为。
- 反向代理为什么会改变真实客户端 IP。

---

## 四、性能优化进阶

适合目标：

```text
能定位接口慢、吞吐低、内存高、goroutine 异常增长等问题。
```

### 需要继续学习

- Go HTTP Server 并发模型。
- goroutine 泄漏排查。
- pprof CPU 分析。
- pprof heap 分析。
- mutex/block profile。
- 压测方法。
- 连接池调优。
- JSON 编解码性能。
- 大响应流式写出。
- 限流与背压。

### 推荐练习

对短链接服务做压测：

```bash
hey -n 10000 -c 100 http://localhost:8080/{code}
```

观察：

- QPS。
- P95/P99。
- 错误率。
- CPU profile。
- goroutine 数量。

然后尝试：

- 对比有无日志输出的性能差异。
- 对比内存 Store 和数据库 Store。
- 对比同步计数和异步计数。

### 达标标准

你应该能做到：

- 用压测复现慢接口。
- 用请求日志定位慢路径。
- 用 pprof 找到 CPU 热点。
- 用 heap profile 观察内存分配。
- 解释连接池参数为什么不能乱调。

---

## 五、安全加固进阶

适合目标：

```text
能把普通 HTTP 服务加固到适合真实业务环境。
```

### 需要继续学习

- 认证与授权。
- Session 与 JWT。
- CSRF。
- XSS 与 Cookie 安全属性。
- CORS 白名单。
- 请求体限制。
- 文件上传安全。
- SSRF。
- 限流。
- 敏感日志脱敏。
- TLS 配置。
- 安全响应头。

### 推荐练习

给短链接服务增加安全策略：

- 创建短链需要登录。
- 管理接口需要管理员权限。
- 限制同一 IP 每分钟创建次数。
- 禁止短链跳转到内网地址。
- 日志中不记录完整敏感 URL。

### 达标标准

你应该能解释：

- 认证和授权的区别。
- `401` 和 `403` 的区别。
- 为什么 CORS 不能替代鉴权。
- 为什么短链接服务要防 SSRF。
- 文件上传为什么不能信任文件名和 Content-Type。

## 六、可观测进阶

适合目标：

```text
服务出问题时，能快速知道发生了什么、影响多大、问题在哪里。
```

### 需要继续学习

- 结构化日志规范。
- Request ID 与 Trace ID。
- OpenTelemetry。
- 指标：QPS、错误率、延迟。
- RED 方法：Rate、Errors、Duration。
- Prometheus。
- Grafana。
- 分布式链路追踪。
- 告警规则设计。

### 推荐练习

给 Todo API 或短链接服务增加：

- `/metrics` 指标。
- 请求总数。
- 状态码计数。
- 请求耗时直方图。
- 当前 goroutine 数量。

再用 Grafana 做一个简单面板：

```text
QPS
错误率
P95 延迟
P99 延迟
```

### 达标标准

你应该能回答：

- 一个接口变慢时先看哪些指标。
- 为什么平均延迟不够。
- Request ID 和 Trace ID 的区别。
- 日志、指标、链路追踪分别解决什么问题。

---

## 七、框架与生态进阶

适合目标：

```text
理解标准库后，再学习 Gin、Echo、Chi 等框架，知道它们到底帮你做了什么。
```

### 推荐学习顺序

先学轻量路由库：

```text
chi
```

再学完整 Web 框架：

```text
Gin
Echo
```

原因是：

- `chi` 很贴近 `net/http` 的 Handler 模型。
- Gin/Echo 提供了更多封装，但也隐藏了更多细节。

### 需要比较的内容

- 路由注册方式。
- 路由组。
- Middleware 写法。
- Context 封装。
- 参数绑定。
- JSON 响应。
- 错误处理。
- 测试方式。

### 推荐练习

把同一个 Todo API 分别用：

```text
标准库 net/http
chi
Gin
```

各写一版。

对比：

- 代码量。
- 可测试性。
- Handler 签名。
- Middleware 组合方式。
- 你是否还能看清 HTTP 请求响应本身。

### 达标标准

你应该能做到：

- 不依赖框架也能写服务。
- 使用框架时知道它包装了什么。
- 能在项目需要时选择合适框架，而不是盲目跟风。

---

## 八、微服务与网关方向

适合目标：

```text
从单体 HTTP 服务进阶到多个服务协作。
```

### 需要继续学习

- 服务拆分边界。
- 服务间 HTTP 调用。
- gRPC 基础。
- API Gateway。
- 反向代理。
- Nginx。
- 服务发现。
- 熔断、重试、超时。
- 幂等性。
- 分布式追踪。

### 推荐练习

把短链接服务拆成两个服务：

```text
api-service：处理 HTTP API。
redirect-service：只处理短码跳转。
```

再增加：

- API Gateway。
- 请求 ID 透传。
- 服务间 HTTP Client。
- 超时控制。
- 错误重试策略。

### 达标标准

你应该能解释：

- 什么情况下不应该拆微服务。
- 服务间调用为什么必须有超时。
- 重试为什么可能放大故障。
- 网关负责什么，业务服务负责什么。

---

## 九、数据层结合进阶

适合目标：

```text
把 net/http 服务和真实数据存储结合起来。
```

### 推荐学习内容

- `database/sql`
- PostgreSQL
- `pgxpool`
- 数据库迁移
- Redis 缓存
- 事务
- 乐观锁
- 分页策略
- 连接池配置

### 推荐练习

把短链接服务从内存 Store 改成：

```text
PostgreSQL Store
```

再增加：

```text
Redis 缓存 code -> original_url
```

重点观察：

- Store 接口是否稳定。
- Handler 是否无需大改。
- Service 是否只关注业务。
- 数据库错误如何映射成业务错误。

### 达标标准

你应该能做到：

- 用 Store 接口隔离 HTTP 层和数据层。
- 使用 context 控制数据库查询超时。
- 配置连接池。
- 使用迁移工具管理表结构。
- 给核心查询加索引并用执行计划验证。

## 十一、面试与表达进阶

适合目标：

```text
把会写代码变成能清楚解释工程取舍。
```

你应该能讲清楚：

- `http.Handler` 和 `HandlerFunc` 的关系。
- Middleware 为什么是 Handler 包 Handler。
- `http.Server` 为什么要配置超时。
- 优雅关闭如何实现。
- `r.Context()` 什么时候取消。
- `http.Client` 为什么要复用。
- `resp.Body.Close()` 为什么必须调用。
- `Transport` 负责什么。
- 如何测试 Handler。
- 如何排查慢接口。
- 如何设计统一错误响应。
- 如何保护文件上传接口。

建议每做完一个项目，都写一篇复盘：

```text
项目做了什么？
核心接口是什么？
目录如何组织？
遇到什么问题？
如何测试？
如何观测？
如果上线还缺什么？
```

这类复盘对面试和真实工作都很有价值。

---

## 十二、推荐最终成长路径

如果你想按一个实际节奏继续学习，可以这样安排：

```text
第 1 阶段：把 Todo API 和短链接服务写完整。
第 2 阶段：给短链接服务接 PostgreSQL。
第 3 阶段：加入 Redis 缓存和限流。
第 4 阶段：接入 Prometheus、Grafana、pprof。
第 5 阶段：补认证、授权、安全加固。
第 6 阶段：用 chi 或 Gin 重写一版，对比标准库。
第 7 阶段：做一个更完整的业务项目。
```

不要急着追求“学很多框架”。真正重要的是：

```text
你能不能把一个 HTTP 服务写清楚。
能不能测试它。
能不能部署它。
能不能排查它。
能不能解释它的设计取舍。
```

做到这些，你的 `net/http` 学习就已经从“会用标准库”进入了“具备 Go 后端工程能力”的阶段。
