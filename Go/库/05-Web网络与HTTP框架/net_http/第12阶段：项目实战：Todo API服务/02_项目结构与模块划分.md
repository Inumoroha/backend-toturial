# 2. 项目结构与模块划分

本节目标：把单文件练习拆成更接近真实项目的结构。

---

## 一、推荐目录

```text
todo-api/
  go.mod
  cmd/
    server/
      main.go
  internal/
    httpapi/
      handler.go
      response.go
      routes.go
    middleware/
      chain.go
      logging.go
      recovery.go
      request_id.go
      auth.go
    todo/
      model.go
      service.go
      store.go
      memory_store.go
  README.md
```

---

## 二、目录职责

`cmd/server`：

```text
程序入口。负责组装依赖、启动 HTTP Server。
```

`internal/httpapi`：

```text
HTTP 层。负责解析请求、写响应、注册路由。
```

`internal/middleware`：

```text
通用中间件。
```

`internal/todo`：

```text
Todo 业务模型、服务和存储。
```

---

## 三、为什么使用 internal

`internal` 是 Go 的特殊目录。放在里面的包不能被项目外部随意导入。

这适合放应用内部代码：

- handler。
- service。
- store。
- middleware。

---

## 四、不要过度拆分

当前项目很小，不需要复杂架构。

拆分目标是：

- Handler 不堆业务逻辑。
- Service 能独立测试。
- Store 将来可替换为数据库。
- Middleware 可复用。

不是为了目录看起来高级。
