# 03. 浮点数与金额精度边界

本节目标：学完后，你能用 `ParseFloat` 和 `FormatFloat` 处理权重、评分等小数字段，并知道金额字段为什么要谨慎。

简短引入：后端业务里会遇到小数，比如商品重量、折扣、评分、地理坐标。`strconv.ParseFloat` 能把字符串转成浮点数，但账务金额不能简单依赖浮点数。

## 一、为什么需要它

可以理解为：浮点数适合“近似值”，不适合“每一分钱都必须精确”的账务值。

常见场景是：

- 商品重量：`weight=1.25`
- 用户评分：`score=4.8`
- 优惠折扣：`discount=0.95`
- 金额输入：`amount=19.99`

其中金额最容易踩坑。

```text
金额通常用整数保存，例如 19.99 元保存为 1999 分。
```

## 二、基本用法

Windows PowerShell：

```powershell
@'
package main

import (
	"fmt"
	"strconv"
)

func main() {
	weight, err := strconv.ParseFloat("1.25", 64)
	if err != nil {
		fmt.Println("重量不合法:", err)
		return
	}

	text := strconv.FormatFloat(weight, 'f', 2, 64)
	fmt.Println("weight =", weight)
	fmt.Println("display =", text+"kg")
}
'@ | Set-Content -Encoding UTF8 main.go
go run main.go
```

## 三、关键参数/语法/代码结构

`strconv.ParseFloat(s, bitSize)`：

- `bitSize=64`：返回值通常用 `float64`，后端项目里最常见。
- `bitSize=32`：结果会按 `float32` 范围和精度处理，但返回类型仍是 `float64`。

`strconv.FormatFloat(f, fmt, prec, bitSize)`：

- `fmt='f'`：普通小数格式，适合展示重量、比例。
- `prec=2`：保留 2 位小数。
- `bitSize=64`：按 `float64` 格式化。

## 四、真实后端场景示例

下面处理商品重量，适合物流运费估算。

```go
package main

import (
	"fmt"
	"strconv"
)

func parseWeight(raw string) (float64, error) {
	if raw == "" {
		return 0, fmt.Errorf("weight 不能为空")
	}

	weight, err := strconv.ParseFloat(raw, 64)
	if err != nil {
		return 0, fmt.Errorf("weight 必须是数字")
	}
	if weight <= 0 {
		return 0, fmt.Errorf("weight 必须大于 0")
	}
	if weight > 1000 {
		return 0, fmt.Errorf("weight 超过单件商品上限")
	}
	return weight, nil
}

func main() {
	weight, err := parseWeight("2.50")
	if err != nil {
		fmt.Println("参数错误:", err)
		return
	}

	fmt.Println("用于运费计算的重量:", strconv.FormatFloat(weight, 'f', 2, 64), "kg")
}
```

如果是订单金额，更保守的方式是解析成“分”。

```go
func amountTextToCents(raw string) (int64, error) {
	amount, err := strconv.ParseFloat(raw, 64)
	if err != nil {
		return 0, fmt.Errorf("amount 必须是数字")
	}
	if amount < 0 {
		return 0, fmt.Errorf("amount 不能为负数")
	}
	return int64(amount*100 + 0.5), nil
}
```

这段代码只适合入门理解。真实支付、账务系统建议使用定点数方案、十进制库，或从输入层严格按字符串拆分元和分，避免浮点误差进入核心账务。

## 五、注意点

`ParseFloat` 可以解析 `NaN`、`+Inf`、`-Inf` 这类特殊值。业务接口通常不应该接受它们。真实项目中需要额外判断：

```go
// if math.IsNaN(v) || math.IsInf(v, 0) { return error }
```

展示用 `FormatFloat`，存储用什么类型要看业务。重量、评分可以用浮点数；金额、库存不建议用浮点数。

## 六、常见误区

- 误区一：金额直接用 `float64` 存数据库。  
  错在浮点数不是精确十进制。账务系统要能对账、回滚、审计，不能靠近似值。

- 误区二：`FormatFloat` 能解决精度问题。  
  它只是控制显示格式，不会改变底层计算误差。

- 误区三：忘记限制范围。  
  `weight=99999999` 可能能解析，但业务上不合理，可能拖垮后续计算或风控逻辑。

## 七、本节达标标准

- 能用 `ParseFloat` 解析小数字段。
- 能用 `FormatFloat` 控制小数展示。
- 能说明为什么金额通常用整数分保存。
- 能为小数字段设置最小值、最大值。
- 知道特殊浮点值需要在业务入口拒绝。
