# 01. 字符串与整数转换

本节目标：学完后，你能把 HTTP 请求里的字符串 ID、分页参数转换成整数，并正确处理转换失败。

简短引入：后端接口收到的 query、form、header、path 参数，最开始几乎都是字符串。业务逻辑需要的是数字，比如用户 ID、文章 ID、页码、库存数量，这时就需要 `strconv.Atoi` 和 `strconv.Itoa`。

## 一、为什么需要它

可以理解为：外部世界给你的都是文本，后端服务要先把文本变成可靠的业务类型。

常见场景是：

- `/users?id=1001` 里的 `id` 要转成整数后查用户。
- `/articles?page=2&size=20` 里的分页参数要转成整数。
- 生成缓存 key 时，可能要把整数 ID 转回字符串。

```text
不要假设请求参数一定是合法数字。用户、脚本、爬虫、旧客户端都可能传入脏数据。
```

## 二、基本用法

新建并运行示例。

Windows PowerShell：

```powershell
@'
package main

import (
	"fmt"
	"strconv"
)

func main() {
	rawUserID := "1001"

	userID, err := strconv.Atoi(rawUserID)
	if err != nil {
		fmt.Println("用户 ID 不合法:", err)
		return
	}

	cacheKey := "user:" + strconv.Itoa(userID)
	fmt.Println("userID =", userID)
	fmt.Println("cacheKey =", cacheKey)
}
'@ | Set-Content -Encoding UTF8 main.go
go run main.go
```

Linux/macOS：

```bash
cat > main.go <<'EOF'
```

## 三、关键参数/语法/代码结构

`strconv.Atoi(s)`：把十进制字符串转成 `int`。它适合页码、数量、普通业务 ID 这类“明确是十进制整数”的场景。

`strconv.Itoa(i)`：把 `int` 转成十进制字符串。它常用于拼接日志、缓存 key、响应字段。

`err != nil`：转换失败时必须处理。比如 `""`、`"abc"`、`"1.2"` 都不能直接转成整数。

真实项目中通常会把解析逻辑放到函数里，而不是在 handler 里到处复制。

## 四、真实后端场景示例

下面模拟一个文章列表接口，只处理分页参数。

```go
package main

import (
	"fmt"
	"strconv"
)

func parsePage(rawPage string) (int, error) {
	if rawPage == "" {
		return 1, nil
	}

	page, err := strconv.Atoi(rawPage)
	if err != nil {
		return 0, fmt.Errorf("page 必须是整数")
	}
	if page < 1 {
		return 0, fmt.Errorf("page 必须大于等于 1")
	}
	if page > 1000 {
		return 0, fmt.Errorf("page 不能超过 1000")
	}
	return page, nil
}

func main() {
	page, err := parsePage("2")
	if err != nil {
		fmt.Println("请求参数错误:", err)
		return
	}
	fmt.Println("查询第", page, "页文章")
}
```

这里的重点不是 `Atoi` 本身，而是转换之后马上做业务校验。真实项目中，`page=-1` 虽然能转换成功，但不能作为合法分页参数。

## 五、注意点

`Atoi` 返回的是 `int`，在不同平台上 `int` 宽度可能不同。后端项目里如果处理数据库自增 ID、雪花 ID、毫秒时间戳，建议直接用 `ParseInt` 转成 `int64`。

缓存 key 可以用 `Itoa` 拼接，但 SQL 不能这样拼。

```text
用户输入不能直接拼进 SQL 字符串。即使已经 Atoi 成功，也应使用参数化查询。
```

替代方案：

- 普通十进制 `int`：用 `Atoi`
- 明确需要 `int64`：用 `ParseInt`
- 生成 JSON 响应：优先交给 `encoding/json`，不要手动拼大段 JSON

## 六、常见误区

- 误区一：只要 `Atoi` 成功，参数就是合法的。  
  错在混淆了“类型合法”和“业务合法”。`0`、`-1` 都是整数，但可能不是合法页码。

- 误区二：忽略错误，直接用返回值。  
  `Atoi("abc")` 会返回 `0` 和错误。如果你忽略错误，就可能把非法输入当成 ID 为 0 的数据处理。

- 误区三：所有数字都用 `Atoi`。  
  数据库 ID、订单号、时间戳可能超过 `int` 的安全范围，应优先考虑 `int64` 或字符串保存。

## 七、本节达标标准

- 能用 `Atoi` 解析普通十进制整数参数。
- 能用 `Itoa` 把整数转成字符串。
- 能区分“转换成功”和“业务合法”。
- 能为分页参数设置默认值、最小值和最大值。
- 不会把用户输入直接拼进 SQL。
