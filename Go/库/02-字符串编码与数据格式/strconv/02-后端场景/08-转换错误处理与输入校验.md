# 08. 转换错误处理与输入校验

本节目标：学完后，你能识别 `strconv` 转换错误，并把底层错误转换成清晰、稳定的接口错误。

简短引入：后端接口不能把底层错误原样抛给用户。`strconv.Atoi: parsing "abc": invalid syntax` 对开发者有用，但对前端或调用方不够友好。我们需要在入口层把错误翻译成稳定的业务错误。

## 一、为什么需要它

可以理解为：`strconv` 告诉你“为什么转不了”，业务代码要决定“应该如何响应”。

常见场景是：

- `page=abc`：应该返回参数错误。
- `limit=1000000`：转换成功，但超过业务上限。
- `user_id=999999999999999999999999`：超出范围。
- `stock=-1`：类型合法，但业务非法。

```text
输入校验包含两层：类型转换校验 + 业务范围校验。
```

## 二、基本用法

Windows PowerShell：

```powershell
@'
package main

import (
	"errors"
	"fmt"
	"strconv"
)

func main() {
	_, err := strconv.ParseInt("abc", 10, 64)
	if err == nil {
		return
	}

	var numErr *strconv.NumError
	if errors.As(err, &numErr) {
		fmt.Println("函数:", numErr.Func)
		fmt.Println("输入:", numErr.Num)
		fmt.Println("原因:", numErr.Err)
	}
}
'@ | Set-Content -Encoding UTF8 main.go
go run main.go
```

Linux/macOS：


## 三、关键参数/语法/代码结构

`strconv.NumError` 里常用字段：

- `Func`：哪个转换函数报错。
- `Num`：出错的原始字符串。
- `Err`：具体原因，比如语法错误或超出范围。

真实项目中不一定要把这些字段返回给用户，但可以写进调试日志。

业务响应应该更稳定，例如：

```json
{"code":"INVALID_ARGUMENT","message":"page 必须是 1 到 1000 之间的整数"}
```

## 四、真实后端场景示例

下面写一个分页参数解析器。

```go
package main

import (
	"fmt"
	"strconv"
)

type PageParam struct {
	Page  int
	Limit int
}

func parsePositiveInt(name, raw string, defaultValue, maxValue int) (int, error) {
	if raw == "" {
		return defaultValue, nil
	}

	value, err := strconv.Atoi(raw)
	if err != nil {
		return 0, fmt.Errorf("%s 必须是整数", name)
	}
	if value < 1 {
		return 0, fmt.Errorf("%s 必须大于等于 1", name)
	}
	if value > maxValue {
		return 0, fmt.Errorf("%s 不能超过 %d", name, maxValue)
	}
	return value, nil
}

func parsePageParam(rawPage, rawLimit string) (PageParam, error) {
	page, err := parsePositiveInt("page", rawPage, 1, 1000)
	if err != nil {
		return PageParam{}, err
	}
	limit, err := parsePositiveInt("limit", rawLimit, 20, 100)
	if err != nil {
		return PageParam{}, err
	}
	return PageParam{Page: page, Limit: limit}, nil
}

func main() {
	param, err := parsePageParam("2", "50")
	if err != nil {
		fmt.Println("参数错误:", err)
		return
	}
	fmt.Printf("offset=%d limit=%d\n", (param.Page-1)*param.Limit, param.Limit)
}
```

真实项目里，`offset` 过大可能导致数据库扫描成本上升。分页上限不是为了为难用户，而是保护服务和数据库。

## 五、注意点

错误处理要分层：

- 入口层：解析字符串、给出用户可理解的错误。
- 业务层：校验业务规则，比如库存不能小于 0。
- 存储层：使用参数化查询、事务和唯一索引保护数据一致性。

```text
不要把 strconv 的底层错误原样暴露成接口契约。
```

日志可以记录原始错误，但要避免敏感信息。比如密码、Token、身份证号不应该直接进入日志。

## 六、常见误区

- 误区一：只处理 `err`，不处理范围。  
  `limit=999999` 能转换成功，但会拖垮查询。

- 误区二：把错误信息写死在很多 handler 里。  
  重复代码会导致接口行为不一致，建议抽成小函数。

- 误区三：把底层错误直接返回给外部调用方。  
  这样会暴露内部实现，也不利于前后端稳定协作。

## 七、本节达标标准

- 能识别 `strconv.NumError`。
- 能把转换错误转换成稳定的业务错误。
- 能同时做类型校验和范围校验。
- 能为分页参数设置默认值和上限。
- 知道大分页、脏参数会影响数据库和服务稳定性。
