# 10. 短链接与迁移脚本中的进制转换

本节目标：学完后，你能用 `FormatInt` 和 `ParseInt` 做简单短码转换，并理解在数据迁移中使用进制转换的风险和回滚要求。

简短引入：短链接系统常把数字 ID 转成更短的字符串，比如把十进制 ID 转成 36 进制。`strconv` 支持 2 到 36 进制转换，适合做入门版短码、迁移校验和内部工具。

## 一、为什么需要它

可以理解为：进制越大，同一个数字表示起来越短。

例如：

- 十进制 `123456789`
- 三十六进制 `21i3v9`

常见场景是：

- 短链接 code：`/s/21i3v9`
- 数据迁移时把历史数字 ID 转成文本 code
- 内部工具把十六进制、十进制 ID 互相转换

```text
短码生成不只是一行进制转换，还要考虑唯一性、可枚举性和回滚。
```

## 二、基本用法

Windows PowerShell：

```powershell
@'
package main

import (
	"fmt"
	"strconv"
)

func main() {
	id := int64(123456789)
	code := strconv.FormatInt(id, 36)

	decoded, err := strconv.ParseInt(code, 36, 64)
	if err != nil {
		fmt.Println("短码解析失败:", err)
		return
	}

	fmt.Println("id =", id)
	fmt.Println("code =", code)
	fmt.Println("decoded =", decoded)
}
'@ | Set-Content -Encoding UTF8 main.go
go run main.go
```

Linux/macOS：

```bash
cat > main.go <<'EOF'
```

## 三、关键参数/语法/代码结构

`strconv.FormatInt(id, 36)`：把数字 ID 转成 36 进制字符串。

`strconv.ParseInt(code, 36, 64)`：把 36 进制字符串还原成 `int64`。

36 进制使用 `0-9` 和 `a-z`。它短，但不是加密。别人可以根据规律猜测相邻 ID。

如果短链接面对公网，真实项目中通常还会加入：

- 随机码或哈希方案
- 唯一索引
- 冲突重试
- 访问权限和风控
- 删除、过期、封禁策略

## 四、真实后端场景示例

下面实现一个最小短链接编码器。

```go
package main

import (
	"fmt"
	"strconv"
)

func encodeShortCode(id int64) (string, error) {
	if id <= 0 {
		return "", fmt.Errorf("id 必须大于 0")
	}
	return strconv.FormatInt(id, 36), nil
}

func decodeShortCode(code string) (int64, error) {
	if code == "" {
		return 0, fmt.Errorf("code 不能为空")
	}
	id, err := strconv.ParseInt(code, 36, 64)
	if err != nil {
		return 0, fmt.Errorf("code 不合法")
	}
	if id <= 0 {
		return 0, fmt.Errorf("code 不合法")
	}
	return id, nil
}

func main() {
	code, err := encodeShortCode(123456789)
	if err != nil {
		fmt.Println(err)
		return
	}

	id, err := decodeShortCode(code)
	if err != nil {
		fmt.Println(err)
		return
	}

	fmt.Println("短码:", code)
	fmt.Println("还原 ID:", id)
}
```

这个版本适合学习，不适合直接作为公网短链接系统。因为递增 ID 转 36 进制后仍然可枚举，攻击者可以猜测其他短链接。

## 五、注意点

数据迁移时使用进制转换，要尤其保守。比如你想给历史文章表新增 `short_code` 字段：

1. 先加 nullable 字段，不影响旧逻辑。
2. 分批回填，记录每批处理范围。
3. 为 `short_code` 建唯一索引前，先检查冲突。
4. 灰度切读逻辑，确认新旧映射一致。
5. 保留回滚方案，不要一口气删除旧字段。

```text
迁移脚本要可重试、可观测、可回滚，不要把一次性脚本写成不可恢复的操作。
```

如果写 SQL 更新，仍然使用参数化执行或迁移工具，不要把外部输入拼进 SQL。索引也不是越多越好，唯一索引会增加写入成本，但能保护短码唯一性。

## 六、常见误区

- 误区一：把 36 进制当成加密。  
  它只是表示方式变化，不能防猜测。

- 误区二：短码生成后不建唯一索引。  
  应用层检查可能漏掉并发冲突，数据库唯一约束是最后防线。

- 误区三：迁移脚本不记录进度。  
  中途失败后不知道处理到哪里，容易重复写、漏写或污染数据。

- 误区四：迁移和线上写入同时改同一字段却没有策略。  
  真实项目中要考虑双写、补偿、灰度读和回滚。

## 七、本节达标标准

- 能用 `FormatInt` 把数字转成 36 进制短码。
- 能用 `ParseInt` 把短码还原成数字 ID。
- 知道进制转换不是加密。
- 能说出短链接需要唯一索引和冲突处理。
- 能描述一次保守的数据迁移流程。
