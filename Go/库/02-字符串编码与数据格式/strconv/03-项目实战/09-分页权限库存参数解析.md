# 09. 分页权限库存参数解析

本节目标：学完后，你能写一个面向后端接口的参数解析层，同时处理分页、权限位、库存数量，并把错误控制在入口处。

简短引入：真实后端项目里，handler 不应该塞满 `strconv.Atoi`。更稳妥的做法是把请求参数解析成结构体，后面的业务层只接收已经校验过的类型。

## 一、为什么需要它

可以理解为：参数解析层是外部输入进入业务系统的第一道门。

本节用一个库存管理接口做例子：

- `page`：第几页
- `limit`：每页数量
- `operator_id`：操作人 ID
- `permission_mask`：权限位
- `stock_delta`：库存变化量

```text
handler 接收字符串，业务层接收明确类型。
```

## 二、基本用法

Windows PowerShell：

```powershell
@'
package main

import (
	"fmt"
	"strconv"
)

type InventoryQuery struct {
	Page           int
	Limit          int
	OperatorID     int64
	PermissionMask uint64
	StockDelta     int
}

func main() {
	q, err := parseInventoryQuery(map[string]string{
		"page":            "1",
		"limit":           "20",
		"operator_id":     "10001",
		"permission_mask": "7",
		"stock_delta":     "-3",
	})
	if err != nil {
		fmt.Println("参数错误:", err)
		return
	}
	fmt.Printf("%+v\n", q)
}

func parseInventoryQuery(values map[string]string) (InventoryQuery, error) {
	page, err := parseIntRange("page", values["page"], 1, 1, 1000)
	if err != nil {
		return InventoryQuery{}, err
	}
	limit, err := parseIntRange("limit", values["limit"], 20, 1, 100)
	if err != nil {
		return InventoryQuery{}, err
	}
	operatorID, err := parseInt64Positive("operator_id", values["operator_id"])
	if err != nil {
		return InventoryQuery{}, err
	}
	mask, err := parseUint64("permission_mask", values["permission_mask"])
	if err != nil {
		return InventoryQuery{}, err
	}
	delta, err := parseIntRange("stock_delta", values["stock_delta"], 0, -10000, 10000)
	if err != nil {
		return InventoryQuery{}, err
	}
	return InventoryQuery{page, limit, operatorID, mask, delta}, nil
}

func parseIntRange(name, raw string, defaultValue, min, max int) (int, error) {
	if raw == "" {
		return defaultValue, nil
	}
	v, err := strconv.Atoi(raw)
	if err != nil {
		return 0, fmt.Errorf("%s 必须是整数", name)
	}
	if v < min || v > max {
		return 0, fmt.Errorf("%s 必须在 %d 到 %d 之间", name, min, max)
	}
	return v, nil
}

func parseInt64Positive(name, raw string) (int64, error) {
	v, err := strconv.ParseInt(raw, 10, 64)
	if err != nil || v <= 0 {
		return 0, fmt.Errorf("%s 必须是正整数", name)
	}
	return v, nil
}

func parseUint64(name, raw string) (uint64, error) {
	v, err := strconv.ParseUint(raw, 10, 64)
	if err != nil {
		return 0, fmt.Errorf("%s 必须是非负整数", name)
	}
	return v, nil
}
'@ | Set-Content -Encoding UTF8 main.go
go run main.go
```

## 三、关键参数/语法/代码结构

这个示例的结构可以直接迁移到 Web handler：

- `InventoryQuery`：承载解析后的类型。
- `parseInventoryQuery`：统一解析入口。
- `parseIntRange`：处理默认值、整数转换、范围限制。
- `parseInt64Positive`：处理数据库 ID。
- `parseUint64`：处理权限位。

这里没有把 `strconv` 散落在业务代码里，是为了让业务层更干净。

## 四、真实后端场景示例

在真实库存服务里，解析完成后通常会进入事务：

```go
// tx, err := db.BeginTx(ctx, nil)
// 1. 查询当前库存并加锁
// 2. 校验权限和库存变化是否合法
// 3. 写库存流水
// 4. 更新库存
// 5. 提交事务
```

`strconv` 只负责把 `stock_delta="-3"` 转成 `-3`。它不负责判断是否会扣成负库存，也不负责事务一致性。

```text
库存变更必须放在清晰的事务边界里，不能只靠入口参数校验。
```

## 五、注意点

权限位解析成 `uint64` 后，还要做权限判断。例如：

```go
const PermissionWriteStock uint64 = 1 << 1

func canWriteStock(mask uint64) bool {
	return mask&PermissionWriteStock != 0
}
```

分页参数要限制上限，因为数据库查询成本会随着 `offset` 增加而上升。大数据量列表可以考虑游标分页，而不是无限放大 `page` 和 `limit`。

库存变更要保守处理：

- 入口限制单次变化范围。
- 业务层检查当前库存。
- 数据库层用事务保护。
- 关键操作写流水，便于回滚和审计。

## 六、常见误区

- 误区一：handler 里到处写 `Atoi`。  
  代码会重复，错误信息不一致，后续维护困难。

- 误区二：入口解析成功就直接更新库存。  
  库存是并发敏感数据，必须考虑事务、锁、幂等和流水。

- 误区三：权限位能解析就代表有权限。  
  解析只是得到数字，权限判断还要看具体 bit 是否存在。

## 七、本节达标标准

- 能把多个字符串参数解析成结构体。
- 能为分页、ID、权限位、库存变化设置不同校验规则。
- 能把解析层和业务层分开。
- 知道库存更新需要事务边界和流水记录。
- 知道大分页可能带来数据库成本。
