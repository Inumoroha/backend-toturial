# 11. 短链接参数清洗与校验小项目

本节目标：你能把前面学过的 `strings` 常用能力串起来，完成一个短链接创建请求的清洗、校验和错误处理流程。

简短引入：短链接服务看起来简单，但它包含很多后端字符串处理问题：目标 URL 要清洗，自定义短码要判断大小写规则，标签要拆分，备注要限制长度，日志要脱敏。

## 一、为什么需要它

这个项目可以理解为一个小型后端输入治理练习。

创建短链接时，用户可能提交：

- 目标 URL：`" https://example.com/a?token=abc "`
- 自定义短码：`" GoAPI "`
- 标签：`" Go, Backend,go "`
- 备注：`"给营销活动用\n"`

后端不能把这些值原样入库。要先清洗、校验、规范化，再交给数据库层。

```text
项目里的字符串处理目标不是“改得好看”，而是“让数据规则稳定、可验证、可回滚”。
```

## 二、基本用法

新建 `main.go`，先放入结构体和入口函数。这个项目示例分几段写，等本节函数都添加完后再运行：

```go
package main

import (
	"errors"
	"fmt"
	"strings"
	"unicode"
	"unicode/utf8"
)

type CreateShortLinkRequest struct {
	TargetURL string
	CustomKey string
	Tags      string
	Note      string
}

type NormalizedShortLink struct {
	TargetURL string
	CustomKey string
	Tags      []string
	Note      string
}

func main() {
	req := CreateShortLinkRequest{
		TargetURL: " https://example.com/a?token=abc ",
		CustomKey: " GoAPI ",
		Tags:      " Go, Backend,go ",
		Note:      " campaign link\n",
	}

	link, err := NormalizeCreateRequest(req)
	if err != nil {
		fmt.Println("创建失败:", err)
		return
	}
	fmt.Printf("%+v\n", link)
}
```

真实项目中，这段逻辑通常会放在 service 或 validator 中，由 HTTP handler 调用。

## 三、关键代码结构

继续在同一个 `main.go` 里添加：

```go
func NormalizeCreateRequest(req CreateShortLinkRequest) (NormalizedShortLink, error) {
	targetURL, err := normalizeTargetURL(req.TargetURL)
	if err != nil {
		return NormalizedShortLink{}, err
	}

	customKey, err := normalizeCustomKey(req.CustomKey)
	if err != nil {
		return NormalizedShortLink{}, err
	}

	tags, err := normalizeTags(req.Tags)
	if err != nil {
		return NormalizedShortLink{}, err
	}

	note, err := normalizeNote(req.Note)
	if err != nil {
		return NormalizedShortLink{}, err
	}

	return NormalizedShortLink{
		TargetURL: targetURL,
		CustomKey: customKey,
		Tags:      tags,
		Note:      note,
	}, nil
}
```

这段代码只做一件事：把请求中的每个字段交给独立函数处理。真实项目中这样写更容易测试，也更容易定位错误。

## 四、真实后端场景示例

继续添加字段处理函数：

```go
func normalizeTargetURL(raw string) (string, error) {
	url := strings.TrimSpace(raw)
	if url == "" {
		return "", errors.New("target url is required")
	}
	if !(strings.HasPrefix(url, "https://") || strings.HasPrefix(url, "http://")) {
		return "", errors.New("target url must start with http:// or https://")
	}
	if len(url) > 2048 {
		return "", errors.New("target url too long")
	}
	return url, nil
}

func normalizeCustomKey(raw string) (string, error) {
	key := strings.ToLower(strings.TrimSpace(raw))
	if key == "" {
		return "", errors.New("custom key is required")
	}
	if utf8.RuneCountInString(key) > 32 {
		return "", errors.New("custom key too long")
	}
	for _, r := range key {
		if !(unicode.IsLetter(r) || unicode.IsDigit(r) || r == '-' || r == '_') {
			return "", errors.New("custom key contains invalid character")
		}
	}
	return key, nil
}

func normalizeTags(raw string) ([]string, error) {
	parts := strings.Split(raw, ",")
	seen := make(map[string]bool)
	tags := make([]string, 0, len(parts))

	for _, part := range parts {
		tag := strings.ToLower(strings.TrimSpace(part))
		if tag == "" {
			continue
		}
		if utf8.RuneCountInString(tag) > 20 {
			return nil, errors.New("tag too long")
		}
		if !seen[tag] {
			seen[tag] = true
			tags = append(tags, tag)
		}
	}

	if len(tags) > 5 {
		return nil, errors.New("too many tags")
	}
	return tags, nil
}

func normalizeNote(raw string) (string, error) {
	note := strings.TrimSpace(strings.ToValidUTF8(raw, ""))
	note = strings.Map(func(r rune) rune {
		if unicode.IsControl(r) {
			return -1
		}
		return r
	}, note)
	if utf8.RuneCountInString(note) > 100 {
		return "", errors.New("note too long")
	}
	return note, nil
}
```

这里用到了前面学过的能力：

- `TrimSpace`：清理输入外层空白。
- `ToLower`：统一短码和标签规则。
- `HasPrefix`：初步判断 URL 协议。
- `Split`：拆分标签。
- `Map`：删除备注里的控制字符。
- `ToValidUTF8`：处理非法编码。

真实项目中，URL 校验还可以使用 `net/url` 做更严格解析。这里先保持字符串处理的练习重点。

代码全部添加完后运行：

```powershell

```

Linux/macOS：

```bash

```

## 五、注意点

短链接创建最终要写数据库。保守做法是：

- `custom_key` 建唯一索引。
- 创建短链接和标签关联表写入放进事务。
- 唯一键冲突要返回清晰错误。
- 数据库写入必须使用参数化查询。
- 如果要做迁移，使用迁移工具管理 schema 变更，并准备回滚脚本。

```text
字符串校验通过，只代表输入格式可接受，不代表数据库写入一定成功。
```

比如两个用户同时创建同一个 `custom_key`，代码层预检查可能都通过，最终仍要靠数据库唯一索引兜底。

## 六、常见误区

误区一：只在前端校验短码。

前端校验是用户体验，后端校验是安全边界。后端必须再校验。

误区二：先查有没有短码，再无条件插入。

并发下会出问题。必须依赖数据库唯一索引，并正确处理冲突。

误区三：把目标 URL 当成完全安全。

真实项目中还要考虑 SSRF 风险、内网地址限制、跳转黑名单、风控策略。

误区四：把标签写入和短链接写入拆成多个无事务操作。

中间失败会产生脏数据。有关联的数据写入通常要放在事务边界里。
