# 02. 大小写归一化与登录标识比较

本节目标：你能处理邮箱、用户名、权限码等大小写问题，避免因为大小写规则不清导致登录或鉴权异常。

简短引入：后端系统里有些字符串应该大小写不敏感，比如很多项目里的邮箱登录；有些字符串必须大小写敏感，比如密码、token、短链接码。`ToLower`、`ToUpper` 和 `EqualFold` 帮你把规则写清楚。

## 一、为什么需要它

可以理解为：大小写处理是在定义业务规则。

常见场景是：

- 邮箱登录：`Alice@Example.com` 和 `alice@example.com` 通常应视为同一个登录标识。
- 权限码：`USER_READ` 可以统一转成大写存储。
- token：不能随便转大小写，否则会破坏安全语义。

```text
不要随手 ToLower，先判断这个字段在业务上是否允许大小写不敏感。
```

## 二、基本用法

```go
package main

import (
	"fmt"
	"strings"
)

func main() {
	email := strings.ToLower(strings.TrimSpace(" Alice@Example.COM "))
	fmt.Println(email)

	fmt.Println(strings.EqualFold("ADMIN", "admin"))
}
```

Windows PowerShell：

```powershell
go run main.go
```

Linux/macOS：

```bash
go run main.go
```

这个命令适合验证大小写归一化规则。真实项目中通常会把归一化逻辑写在注册和登录共用的位置，避免两边规则不一致。

## 三、关键代码结构

`strings.ToLower(s)`：把字符串转成小写。

`strings.ToUpper(s)`：把字符串转成大写。

`strings.EqualFold(a, b)`：按 Unicode 大小写折叠规则比较两个字符串是否相等。

真实项目中常见选择：

- 存储邮箱：先 `TrimSpace`，再 `ToLower`。
- 比较输入命令或权限码：可以用 `EqualFold`。
- 存储权限码：统一 `ToUpper`，方便排查。

## 四、真实后端场景示例

下面模拟登录前规范化邮箱：

```go
package main

import (
	"fmt"
	"strings"
)

func NormalizeEmail(raw string) string {
	return strings.ToLower(strings.TrimSpace(raw))
}

func main() {
	input := " Alice@Example.COM "
	dbEmail := "alice@example.com"

	if NormalizeEmail(input) == dbEmail {
		fmt.Println("允许继续校验密码")
	} else {
		fmt.Println("账号不存在")
	}
}
```

真实项目中，下一步应该使用参数化查询查用户：

```text
SELECT id, password_hash FROM users WHERE email = $1
```

不要把邮箱拼进 SQL。即使邮箱已经被 `TrimSpace` 和 `ToLower` 处理过，仍然必须参数化。

## 五、注意点

大小写归一化要在“写入”和“查询”两边保持一致。注册时如果存了小写邮箱，登录时也要把输入转小写再查。

对下面字段要谨慎：

- 密码：不能转大小写。
- token：不能转大小写。
- 签名字符串：不能转大小写。
- 短链接码：如果设计为大小写敏感，就不能归一化。

## 六、常见误区

误区一：所有字符串都转小写。

这会破坏 token、短链接码、邀请码等字段。

误区二：注册时转小写，登录时忘了转。

这会导致用户注册成功后登录失败。真实项目中应把归一化逻辑封装成函数并复用。

误区三：数据库唯一索引规则和代码规则不一致。

如果代码认为邮箱大小写不敏感，数据库也应该通过规范化字段或合适索引保证唯一性，否则可能出现重复账号。

## 七、本节达标标准

- 能区分哪些字段适合大小写归一化，哪些不适合。
- 能用 `ToLower` 规范化邮箱。
- 能用 `EqualFold` 做大小写不敏感比较。
- 能说明大小写规则要和数据库唯一性约束保持一致。
