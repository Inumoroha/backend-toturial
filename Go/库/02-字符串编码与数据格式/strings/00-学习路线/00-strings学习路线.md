# 00. strings 学习路线

本节目标：你能知道学习 Go `strings` 包应该按什么顺序推进，并把每个函数放到后端业务场景里理解。

简短引入：后端开发离不开字符串处理。用户提交的手机号、邮箱、搜索词、文章标签、权限码、订单号、日志内容、迁移 SQL，最终都会以字符串形式进入系统。`strings` 包就是你处理这些文本的第一套工具。

## 一、为什么需要系统学习 strings

初学者常见的问题不是“不知道有哪个函数”，而是不知道在项目里什么时候该用哪个函数。

比如：

- 用户注册时，要去掉邮箱前后的空格。
- 登录时，用户名大小写是否敏感要先定规则。
- 解析 `Bearer token` 时，要判断前缀。
- 处理文章标签时，要拆分、去空、去重。
- 打日志时，要把手机号、邮箱、身份证号脱敏。
- 拼接 SQL 条件时，要知道字符串拼接的边界，不能把用户输入直接拼进去。

```text
strings 不是背函数名，而是训练你把“脏输入”变成“可控数据”的能力。
```

## 三、建议练习方式

每篇文章建议按下面方式学习：

1. 先读“为什么需要它”，建立业务直觉。
2. 复制“基本用法”的代码运行。
3. 修改输入数据，观察输出变化。
4. 再读“真实后端场景示例”，理解它在项目里的位置。
5. 最后按“本节达标标准”自查。

Windows PowerShell 运行示例：

```powershell
go run main.go
```

Linux/macOS 运行示例：

```bash
go run main.go
```

这些命令在真实项目中的作用是快速验证一个最小示例。风险是：示例代码通常没有完整配置、日志、测试和错误处理，不能直接当生产代码复制进项目。真实项目中通常会把逻辑放进 service、handler、validator 或 util 包，并配合单元测试验证边界。

## 四、常见误区

误区一：把 `strings` 学成函数清单。

只背函数名很难迁移到项目里。更好的方式是按输入清洗、判断解析、拆分拼接、替换转换、性能接口来学习。

误区二：以为字符串处理能解决所有安全问题。

`TrimSpace`、`ReplaceAll`、`Join` 都不是安全边界。SQL 注入、XSS、SSRF、权限绕过都有各自的防护方式。

误区三：忽略数据库和生产约束。

真实后端不是把字符串处理完就结束。还要考虑唯一索引、事务边界、迁移回滚、日志脱敏和错误处理。

## 五、最终达标标准

学完这套路线上，你应该能够做到：

- 能对用户输入做基本清洗，不把空格、大小写、非法字符留到数据库层处理。
- 能判断 URL、权限码、文件名、请求头是否符合业务规则。
- 能正确切分标签、搜索词和配置项，并处理空值。
- 能做基础日志脱敏，避免把敏感信息直接输出。
- 能知道什么时候用 `Builder`，什么时候普通拼接就够。
- 能说清楚字符串处理和 SQL 安全的边界：字符串清洗不能替代参数化查询。
