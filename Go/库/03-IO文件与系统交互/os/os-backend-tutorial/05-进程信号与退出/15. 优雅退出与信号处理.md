# 15. 优雅退出与信号处理

本节目标：学完后你能捕获退出信号，并在退出前完成必要清理。

简短引入：后端服务会被 Ctrl+C、容器停止、部署重启中断。优雅退出的目标不是拖延退出，而是给程序一点时间关闭资源、写完日志、停止接收新请求。

## 一、为什么需要它

可以理解为：信号是操作系统通知进程“该发生某件事了”的方式。后端服务最常见的是中断和终止。

Go 的 `os` 包定义了 `os.Interrupt`，实际监听通常配合 `os/signal`。

在本地终端里，Ctrl+C 通常对应 `os.Interrupt`。在 Docker、Kubernetes、systemd 这类部署环境里，停止服务时更常见的是 `SIGTERM`，Go 代码里通常通过 `syscall.SIGTERM` 监听。

```text
退出时要先停止接收新请求，再等待正在处理的任务完成或超时。
```

## 二、基本用法

```go
package main

import (
	"fmt"
	"os"
	"os/signal"
)

func main() {
	ch := make(chan os.Signal, 1)
	signal.Notify(ch, os.Interrupt)

	fmt.Println("running, press Ctrl+C to stop")
	sig := <-ch
	fmt.Println("received:", sig)
	fmt.Println("cleanup done")
}
```

Windows PowerShell：

```powershell
go run .
```

Linux/macOS：

```bash
go run .
```

运行后按 Ctrl+C。

## 三、关键代码结构

`signal.Notify(ch, os.Interrupt)` 表示把中断信号转发到 channel。

channel 建议有缓冲，例如 `make(chan os.Signal, 1)`，避免信号到来时无人接收导致行为不稳定。

真实 HTTP 服务中通常还会配合 `context.WithTimeout` 和 `server.Shutdown(ctx)`。如果服务运行在容器里，还应该监听 `syscall.SIGTERM`。

## 四、真实后端场景示例

下面模拟一个每秒处理任务的服务，收到 Ctrl+C 或 SIGTERM 后进入清理阶段，并设置 5 秒退出超时：

```go
package main

import (
	"context"
	"fmt"
	"os"
	"os/signal"
	"syscall"
	"time"
)

func main() {
	stop := make(chan os.Signal, 1)
	signal.Notify(stop, os.Interrupt, syscall.SIGTERM)

	ticker := time.NewTicker(time.Second)
	defer ticker.Stop()

	fmt.Println("worker started, press Ctrl+C to stop")

	for {
		select {
		case <-ticker.C:
			fmt.Println("processing background jobs")
		case sig := <-stop:
			fmt.Println("received signal:", sig)
			ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
			defer cancel()
			if err := shutdown(ctx); err != nil {
				fmt.Println("shutdown:", err)
			}
			return
		}
	}
}

func shutdown(ctx context.Context) error {
	done := make(chan struct{})
	go func() {
		defer close(done)
		fmt.Println("stop accepting new jobs")
		time.Sleep(2 * time.Second)
		fmt.Println("flush logs and close resources")
	}()

	select {
	case <-done:
		fmt.Println("cleanup done")
		return nil
	case <-ctx.Done():
		return ctx.Err()
	}
}
```

真实项目中这里会做几件事：停止接收新请求、关闭数据库连接、等待后台任务结束、把最后一条日志写出去。超时时间要小于部署平台的强制停止时间，例如平台给 30 秒，你可以让应用最多清理 25 秒。

## 五、注意点

不要收到信号后无限等待。生产发布和容器平台通常有停止超时时间。超过时间，进程可能被强制杀掉。

不要只监听 Ctrl+C。开发机上 Ctrl+C 足够，但容器停止、滚动发布、systemd stop 通常会给进程发送 SIGTERM。

不要在信号处理里做大量复杂业务。把信号当成退出通知，真正清理逻辑交给正常函数执行。

`os.Exit` 会跳过 `defer`，所以优雅退出时更常见的是让 `main` 正常 return。

## 六、常见误区

误区一：直接 Ctrl+C 结束，不考虑写入中的文件和请求。

误区二：收到信号后继续接收新请求。这样会让退出时间不可控。

误区三：用 `os.Exit` 结束所有流程，导致 `defer file.Close()`、`ticker.Stop()` 没机会执行。

误区四：只在本地测试 Ctrl+C，没有验证容器停止时的 SIGTERM。

## 七、本节达标标准

- 能用 `signal.Notify` 监听 `os.Interrupt` 和 `syscall.SIGTERM`。
- 能用 `select` 同时处理工作和退出信号。
- 能解释优雅退出要做哪些事。
- 知道退出不能无限等待，要设置超时，也不能随意跳过清理。
