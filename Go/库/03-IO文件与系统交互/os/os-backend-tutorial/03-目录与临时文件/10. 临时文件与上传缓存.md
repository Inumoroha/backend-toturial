# 10. 临时文件与上传缓存

本节目标：学完后你能用临时文件保存中间结果，并在失败时清理。

简短引入：上传头像、导入订单、生成报表时，程序常常需要先写一个临时文件。确认成功后再移动到最终位置，可以降低半成品文件暴露给业务的风险。

## 一、为什么需要它

可以理解为：临时文件是“还没确认成功的中间状态”。真实项目中常见场景是上传文件校验、图片压缩、CSV 导入预处理、导出文件生成。

直接写最终文件的问题是：写到一半程序崩了，用户可能看到一个损坏文件。

```text
先写临时文件，成功后再替换或移动，是文件操作里很常见的保守做法。
```

## 二、基本用法

```go
package main

import (
	"fmt"
	"os"
)

func main() {
	file, err := os.CreateTemp("", "upload-*.tmp")
	if err != nil {
		fmt.Println("create temp:", err)
		return
	}
	defer os.Remove(file.Name())
	defer file.Close()

	if _, err := file.WriteString("temporary upload data\n"); err != nil {
		fmt.Println("write temp:", err)
		return
	}

	fmt.Println("temp file:", file.Name())
}
```

Windows PowerShell：

```powershell
go run .
```

Linux/macOS：

```bash
go run .
```

## 三、关键代码结构

`os.CreateTemp(dir, pattern)` 会创建一个唯一临时文件。

`dir` 为空字符串时使用系统临时目录。

`pattern` 中的 `*` 会被替换为随机字符串。

`os.MkdirTemp(dir, pattern)` 用来创建临时目录，适合一次任务需要多个中间文件。

## 四、真实后端场景示例

下面模拟上传文件：先写临时文件，校验通过后移动到最终目录。

```go
package main

import (
	"fmt"
	"os"
	"path/filepath"
)

func main() {
	if err := saveUpload([]byte("avatar image bytes"), "u_1001.txt"); err != nil {
		fmt.Println("save upload:", err)
	}
}

func saveUpload(data []byte, finalName string) error {
	uploadDir := filepath.Join("data", "uploads")
	if err := os.MkdirAll(uploadDir, 0755); err != nil {
		return fmt.Errorf("create upload dir: %w", err)
	}

	tmp, err := os.CreateTemp(uploadDir, "upload-*.tmp")
	if err != nil {
		return fmt.Errorf("create temp upload: %w", err)
	}
	tmpName := tmp.Name()
	defer os.Remove(tmpName)

	if _, err := tmp.Write(data); err != nil {
		tmp.Close()
		return fmt.Errorf("write temp upload: %w", err)
	}
	if err := tmp.Close(); err != nil {
		return fmt.Errorf("close temp upload: %w", err)
	}

	finalPath := filepath.Join(uploadDir, finalName)
	if err := os.Rename(tmpName, finalPath); err != nil {
		return fmt.Errorf("move upload into place: %w", err)
	}

	return nil
}
```

`defer os.Remove(tmpName)` 在成功 `Rename` 后通常会删除失败，因为临时文件已经不存在。这里它作为失败兜底清理。

## 五、注意点

如果临时文件和最终文件不在同一个磁盘或挂载点，`os.Rename` 可能失败。上传目录内创建临时文件，通常比系统临时目录更适合最终移动。

用户上传文件名不要直接作为 `finalName`。真实项目通常生成自己的文件 ID，再把原始文件名作为元数据保存。

如果写入后要立即移动，关闭文件后再 `Rename` 更稳妥，尤其在 Windows 上。

## 六、常见误区

误区一：临时文件写在系统临时目录，最终文件在另一个挂载点，移动时失败。

误区二：失败后不清理临时文件。时间久了会占满磁盘。

误区三：把用户原始文件名直接拼到路径里。这样可能造成路径穿越或覆盖问题。

## 七、本节达标标准

- 能用 `os.CreateTemp` 创建临时文件。
- 能用 `defer os.Remove` 做失败清理。
- 能解释为什么要先写临时文件再移动到最终位置。
- 知道上传文件名要由服务端控制。
