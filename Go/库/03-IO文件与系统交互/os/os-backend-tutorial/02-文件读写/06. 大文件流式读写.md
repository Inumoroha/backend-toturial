# 06. 大文件流式读写

本节目标：学完后你能用 `os.Open`、`bufio.Scanner` 和 `bufio.Writer` 分批处理大日志、CSV 导入和导出文件。

简短引入：`os.ReadFile` 和 `os.WriteFile` 很适合配置文件、小型 JSON、一次性状态文件。但后端项目里经常会遇到大日志、订单导入 CSV、几十万行导出文件，这类文件不能简单一次性读进内存。

## 一、为什么需要它

可以理解为：流式读写就是“边读边处理、边生成边写出”。程序不需要把整个文件都放进内存，而是每次处理一小段。

常见场景是：

- 从访问日志里统计某个接口的错误次数。
- 导入运营上传的订单 CSV。
- 分批导出用户、文章、库存记录。

```text
文件越大，越要避免一次性读完；业务越重要，越要设计失败后的重试和回滚。
```

## 二、基本用法

先准备一个日志文件。

Windows PowerShell：

```powershell
Set-Content app.log "INFO /api/users 200"
Add-Content app.log "ERROR /api/orders 500"
Add-Content app.log "ERROR /api/orders 500"
go run .
```

Linux/macOS：

```bash
printf 'INFO /api/users 200\nERROR /api/orders 500\nERROR /api/orders 500\n' > app.log
go run .
```

`main.go`：

```go
package main

import (
	"bufio"
	"fmt"
	"os"
	"strings"
)

func main() {
	file, err := os.Open("app.log")
	if err != nil {
		fmt.Println("open log:", err)
		return
	}
	defer file.Close()

	scanner := bufio.NewScanner(file)
	errorCount := 0
	for scanner.Scan() {
		line := scanner.Text()
		if strings.Contains(line, "ERROR") {
			errorCount++
		}
	}
	if err := scanner.Err(); err != nil {
		fmt.Println("scan log:", err)
		return
	}

	fmt.Println("error lines:", errorCount)
}
```

这个例子一行一行读日志。即使日志很大，程序也不会把整个文件一次性加载进内存。

## 三、关键代码结构

`os.Open(path)` 只读打开文件，适合读取日志、CSV、配置模板。

`bufio.Scanner` 适合按行读取文本文件，代码简单，适合日志和普通 CSV。

`scanner.Err()` 必须检查。循环结束不一定代表正常读完，也可能是中途读取失败。

`bufio.Writer` 适合分批写出内容。写完后要调用 `Flush()`，否则缓冲区里的数据可能还没落到文件。

```text
看到 for scanner.Scan()，就要想到循环后检查 scanner.Err()。
看到 bufio.NewWriter，最后就要想到 Flush()。
```

## 四、真实后端场景示例

下面模拟导入订单 CSV。为了保持示例简单，这里只打印校验结果，不连接数据库。

先准备 CSV。

Windows PowerShell：

```powershell
Set-Content orders.csv "order_id,user_id,total"
Add-Content orders.csv "o_1001,u_1,59.90"
Add-Content orders.csv "o_1002,u_2,21.00"
go run .
```

Linux/macOS：

```bash
printf 'order_id,user_id,total\no_1001,u_1,59.90\no_1002,u_2,21.00\n' > orders.csv
go run .
```

`main.go`：

```go
package main

import (
	"bufio"
	"encoding/csv"
	"fmt"
	"io"
	"os"
)

func main() {
	if err := importOrders("orders.csv"); err != nil {
		fmt.Println("import orders:", err)
	}
}

func importOrders(path string) error {
	file, err := os.Open(path)
	if err != nil {
		return fmt.Errorf("open csv: %w", err)
	}
	defer file.Close()

	reader := csv.NewReader(bufio.NewReader(file))
	header, err := reader.Read()
	if err != nil {
		return fmt.Errorf("read csv header: %w", err)
	}
	if len(header) != 3 || header[0] != "order_id" || header[1] != "user_id" || header[2] != "total" {
		return fmt.Errorf("unexpected csv header: %v", header)
	}

	line := 1
	for {
		record, err := reader.Read()
		if err == io.EOF {
			break
		}
		if err != nil {
			return fmt.Errorf("read csv line %d: %w", line+1, err)
		}
		line++

		if len(record) != 3 {
			return fmt.Errorf("bad column count at line %d", line)
		}

		fmt.Printf("valid order id=%s user=%s total=%s\n", record[0], record[1], record[2])
	}

	return nil
}
```

真实项目中导入订单时，不会只打印。通常会分批写数据库，例如每 500 行一批，并用事务保护一批数据。失败时要记录失败行号，方便运营修复后重新导入。

```text
导入文件不能只考虑“成功路径”，必须设计校验、失败报告、事务边界和可重试。
```

### 分批写导出文件

下面模拟把用户列表分批写成 CSV：

```go
package main

import (
	"bufio"
	"fmt"
	"os"
)

type User struct {
	ID    string
	Email string
}

func main() {
	users := []User{
		{ID: "u_1", Email: "a@example.com"},
		{ID: "u_2", Email: "b@example.com"},
	}

	if err := exportUsers("users.csv", users); err != nil {
		fmt.Println("export users:", err)
	}
}

func exportUsers(path string, users []User) error {
	file, err := os.Create(path)
	if err != nil {
		return fmt.Errorf("create export file: %w", err)
	}
	defer file.Close()

	writer := bufio.NewWriter(file)
	if _, err := writer.WriteString("user_id,email\n"); err != nil {
		return fmt.Errorf("write csv header: %w", err)
	}

	for _, user := range users {
		line := fmt.Sprintf("%s,%s\n", user.ID, user.Email)
		if _, err := writer.WriteString(line); err != nil {
			return fmt.Errorf("write csv row: %w", err)
		}
	}

	if err := writer.Flush(); err != nil {
		return fmt.Errorf("flush export file: %w", err)
	}

	return nil
}
```

这里为了演示直接传了一个切片。真实项目中通常不会一次性把所有用户查出来，而是分页查询数据库，每查一页就写一页。

## 五、注意点

`bufio.Scanner` 默认单行最大 token 有限制。普通日志行没问题，但如果一行特别长，例如大 JSON，可以调整 buffer，或者改用 `bufio.Reader`。

CSV 不要自己用 `strings.Split(line, ",")` 解析。真实 CSV 可能有引号、逗号、换行，应该用 `encoding/csv`。

导入数据库时要考虑事务边界。全文件一个事务可能太大，失败回滚成本高；每行一个事务又太慢。真实项目中通常按批次处理，并记录批次状态。

导出文件如果很重要，不建议直接写最终路径。更保守的方式是先写临时文件，成功 `Flush` 并关闭后，再用 `os.Rename` 替换到最终位置。

## 六、常见误区

误区一：用 `os.ReadFile` 读取几百 MB 的导入文件。这样容易内存暴涨，甚至拖垮服务。

误区二：忘记检查 `scanner.Err()`。读到一半失败时，程序可能误以为处理完成。

误区三：忘记 `writer.Flush()`。程序运行看似成功，但文件末尾可能缺数据。

误区四：自己按逗号切 CSV。只要字段里出现逗号或引号，解析就会错。

## 七、本节达标标准

- 能用 `os.Open` 和 `bufio.Scanner` 按行读取大日志。
- 能用 `encoding/csv` 流式读取 CSV。
- 能用 `os.Create` 和 `bufio.Writer` 分批写导出文件。
- 知道 `scanner.Err()` 和 `writer.Flush()` 必须处理。
- 能说明导入数据库时为什么要考虑批次、事务和失败重试。
