# 07. OpenFile 与文件打开模式

本节目标：学完后你能用 `os.OpenFile` 追加日志，并理解只读、创建、追加、截断这些模式。

简短引入：`ReadFile` 和 `WriteFile` 适合简单场景。真实后端写日志、追加审计记录、分批导出文件时，更常用打开文件后持续写入。

## 一、为什么需要它

可以理解为：`os.OpenFile` 让你明确告诉操作系统“我要怎样使用这个文件”。比如只读打开、没有就创建、写到文件末尾、打开时清空旧内容。

日志和审计记录通常需要追加，而不是覆盖。

```text
业务审计日志通常只能追加，不能随意覆盖或删除。
```

## 二、基本用法

```go
package main

import (
	"fmt"
	"os"
	"time"
)

func main() {
	file, err := os.OpenFile("app.log", os.O_CREATE|os.O_WRONLY|os.O_APPEND, 0644)
	if err != nil {
		fmt.Println("open log:", err)
		return
	}
	defer file.Close()

	line := time.Now().Format(time.RFC3339) + " user login success\n"
	if _, err := file.WriteString(line); err != nil {
		fmt.Println("write log:", err)
		return
	}
}
```

Windows PowerShell：

```powershell
go run .
Get-Content .\app.log
```

Linux/macOS：

```bash
go run .
cat ./app.log
```

## 三、关键参数

常用 flag：

- `os.O_RDONLY`：只读。
- `os.O_WRONLY`：只写。
- `os.O_RDWR`：读写。
- `os.O_CREATE`：不存在就创建。
- `os.O_APPEND`：写入时追加到末尾。
- `os.O_TRUNC`：打开时清空文件。
- `os.O_EXCL`：配合 `O_CREATE`，如果文件已存在则失败。

这些 flag 可以用 `|` 组合。

## 四、真实后端场景示例

下面模拟把用户操作写入审计日志：

```go
package main

import (
	"fmt"
	"os"
	"path/filepath"
	"time"
)

func main() {
	if err := appendAuditLog("u_1001", "delete article a_2001"); err != nil {
		fmt.Println("audit log error:", err)
	}
}

func appendAuditLog(userID, action string) error {
	dir := filepath.Join("data", "logs")
	if err := os.MkdirAll(dir, 0755); err != nil {
		return fmt.Errorf("create log dir: %w", err)
	}

	path := filepath.Join(dir, "audit.log")
	file, err := os.OpenFile(path, os.O_CREATE|os.O_WRONLY|os.O_APPEND, 0644)
	if err != nil {
		return fmt.Errorf("open audit log: %w", err)
	}
	defer file.Close()

	line := fmt.Sprintf("%s user=%s action=%q\n", time.Now().Format(time.RFC3339), userID, action)
	if _, err := file.WriteString(line); err != nil {
		return fmt.Errorf("write audit log: %w", err)
	}

	return nil
}
```

真实项目中日志通常交给日志库处理，但你要理解它为什么需要文件打开模式、轮转和写入错误处理。

## 五、注意点

每次打开文件后都要关闭。对于短命令可以 `defer file.Close()`，对于长生命周期服务要注意不要频繁打开关闭同一个日志文件。

不要随便使用 `O_TRUNC`。它会在打开成功时清空文件，适合重新生成导出文件，不适合日志和审计。

如果要创建“不能覆盖”的结果文件，可以使用 `O_CREATE|O_EXCL|O_WRONLY`。

## 六、常见误区

误区一：写日志时用 `os.WriteFile`，导致每次覆盖。

误区二：打开文件后忘记 `Close`。请求量大时会耗尽文件句柄。

误区三：以为 `O_APPEND` 可以保证复杂多行记录一定完整。多进程并发写日志时仍建议使用成熟日志方案。

## 七、本节达标标准

- 能用 `OpenFile` 追加写日志。
- 能解释 `O_CREATE`、`O_APPEND`、`O_TRUNC` 的风险。
- 能在写入后处理错误。
- 知道审计日志不能随意覆盖。
