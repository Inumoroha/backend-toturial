# 02. 环境变量与配置读取

本节目标：学完后你能用环境变量为后端服务读取运行环境、端口和数据目录。

简短引入：后端服务在开发、测试、生产环境通常使用同一份代码，但配置不同。环境变量是最常见、最容易被容器和部署平台接管的配置入口。

## 一、为什么需要它

可以理解为：环境变量是操作系统给进程的一组键值配置。真实项目中常见配置包括 `APP_ENV`、`HTTP_PORT`、`DATA_DIR`、`DATABASE_URL`。

环境变量的好处是部署时容易注入，不需要为每个环境重新编译程序。风险是值都是字符串，需要你自己校验。

```text
环境变量只说明“配置从哪里来”，不代表配置一定可信。
```

## 二、基本用法

新建 `main.go`：

```go
package main

import (
	"fmt"
	"os"
)

func main() {
	env := os.Getenv("APP_ENV")
	port, ok := os.LookupEnv("HTTP_PORT")

	fmt.Println("APP_ENV:", env)
	fmt.Println("HTTP_PORT exists:", ok, "value:", port)
}
```

Windows PowerShell：

```powershell
$env:APP_ENV="dev"
$env:HTTP_PORT="8080"
go run .
```

Linux/macOS：

```bash
APP_ENV=dev HTTP_PORT=8080 go run .
```

## 三、关键代码结构

`os.Getenv("KEY")` 只返回字符串。变量不存在时返回空字符串。

`os.LookupEnv("KEY")` 返回两个值：字符串和是否存在。它能区分“没设置”和“设置为空字符串”。

真实项目中读配置更推荐写一个集中函数：

```go
func getEnv(key, fallback string) string {
	value, ok := os.LookupEnv(key)
	if !ok || value == "" {
		return fallback
	}
	return value
}
```

## 四、真实后端场景示例

下面模拟服务启动配置：

```go
package main

import (
	"fmt"
	"os"
	"strconv"
)

type Config struct {
	Env     string
	Port    int
	DataDir string
}

func main() {
	cfg, err := loadConfig()
	if err != nil {
		fmt.Println("config error:", err)
		os.Exit(1)
	}

	fmt.Printf("env=%s port=%d data=%s\n", cfg.Env, cfg.Port, cfg.DataDir)
}

func loadConfig() (Config, error) {
	portText := getEnv("HTTP_PORT", "8080")
	port, err := strconv.Atoi(portText)
	if err != nil {
		return Config{}, fmt.Errorf("HTTP_PORT must be number: %w", err)
	}

	return Config{
		Env:     getEnv("APP_ENV", "dev"),
		Port:    port,
		DataDir: getEnv("DATA_DIR", "./data"),
	}, nil
}

func getEnv(key, fallback string) string {
	value, ok := os.LookupEnv(key)
	if !ok || value == "" {
		return fallback
	}
	return value
}
```

这里把字符串端口转成数字，是为了提前发现配置错误。真实服务不要等到监听端口时才发现端口配置错了。

## 五、注意点

不要在代码里硬编码生产数据库地址、密码、对象存储密钥。它们应该由部署系统注入。

但也不要完全相信环境变量。端口要转数字，路径要限制范围，布尔值要明确支持哪些写法。配置加载失败时，应该让程序启动失败，而不是悄悄使用错误默认值。

```text
生产配置宁可启动失败，也不要带着不确定配置继续运行。
```

## 六、常见误区

误区一：所有配置都用 `os.Getenv`，结果无法区分“没设置”和“设置为空”。

误区二：默认值过于激进。例如生产环境没有配置数据库地址时自动连本地数据库，这会造成难查的问题。

误区三：把环境变量打印到日志里。调试时方便，但可能泄露密钥。

## 七、本节达标标准

- 能使用 `os.Getenv` 和 `os.LookupEnv`。
- 能为环境变量设置合理默认值。
- 能把字符串配置转换成数字并处理错误。
- 知道敏感环境变量不能随意打印。
