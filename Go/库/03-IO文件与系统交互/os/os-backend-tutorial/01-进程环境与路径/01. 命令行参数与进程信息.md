# 01. 命令行参数与进程信息

本节目标：学完后你能用 `os.Args` 和进程信息写一个最小的后端运维入口。

简短引入：后端服务经常会有启动参数，例如指定 `serve`、`migrate`、`seed`。这些参数不是业务接口的一部分，但它们决定程序启动后做什么。

## 一、为什么需要它

可以理解为：命令行参数是程序启动时传进来的“第一份配置”。常见场景是同一个二进制文件既能启动 HTTP 服务，也能执行数据库迁移。

真实项目中通常会把复杂命令交给 cobra、urfave/cli 这类库，但初学阶段先理解 `os.Args` 很重要，因为所有命令行库底层都离不开它。

```text
命令行参数适合表达一次性动作，环境变量适合表达运行环境配置。
```

## 二、基本用法

新建 `main.go`：

```go
package main

import (
	"fmt"
	"os"
)

func main() {
	fmt.Println("args:", os.Args)
	fmt.Println("pid:", os.Getpid())
	fmt.Println("parent pid:", os.Getppid())
}
```

Windows PowerShell：

```powershell
go run . serve --port 8080
```

Linux/macOS：

```bash
go run . serve --port 8080
```

你会看到 `os.Args[0]` 是程序名，后面的才是你传入的参数。

## 三、关键代码结构

`os.Args` 是 `[]string`。

- `os.Args[0]`：当前程序路径或名称。
- `os.Args[1]`：第一个业务参数，常见是子命令。
- `os.Getpid()`：当前进程 ID，日志排查时有用。
- `os.Getppid()`：父进程 ID，在容器、脚本启动时偶尔用于排查。

初学时最容易犯的错是直接访问 `os.Args[1]`，但用户可能没有传参数。

```go
if len(os.Args) < 2 {
	fmt.Println("usage: app serve|migrate")
	return
}
```

## 四、真实后端场景示例

下面写一个简化入口：`serve` 表示启动服务，`migrate` 表示执行迁移。

```go
package main

import (
	"fmt"
	"os"
)

func main() {
	if len(os.Args) < 2 {
		fmt.Println("usage: app serve|migrate")
		return
	}

	switch os.Args[1] {
	case "serve":
		fmt.Printf("starting api server, pid=%d\n", os.Getpid())
	case "migrate":
		fmt.Println("running database migration plan")
	default:
		fmt.Println("unknown command:", os.Args[1])
	}
}
```

这里没有真的连接数据库，因为本节只关注入口。真实项目中 `migrate` 通常会调用迁移工具，并且要保证可回滚。

```text
迁移命令必须明确、可审计、可回滚，不要在服务启动时偷偷修改生产数据库。
```

## 五、注意点

命令行参数不适合放密码、Token 这类敏感信息，因为它们可能出现在终端历史、进程列表或运维平台日志里。

如果参数开始变多，建议使用标准库 `flag` 或成熟 CLI 库，不要自己写大量字符串解析逻辑。自己解析很容易漏掉空值、重复参数和错误提示。

## 六、常见误区

误区一：认为 `os.Args[0]` 一定是绝对路径。不同运行方式下它可能只是临时可执行文件路径或命令名。

误区二：把生产密码写成命令行参数。这样做不利于审计和权限隔离。

误区三：入口命令没有帮助信息。真实项目中运维同学需要知道可用命令和风险。

## 七、本节达标标准

- 能解释 `os.Args[0]` 和 `os.Args[1]` 的区别。
- 能在访问参数前先检查长度。
- 能写出 `serve`、`migrate` 这类简单命令入口。
- 知道敏感配置不要放在命令行参数里。
