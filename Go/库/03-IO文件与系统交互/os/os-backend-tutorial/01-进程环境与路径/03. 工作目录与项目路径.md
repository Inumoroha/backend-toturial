# 03. 工作目录与项目路径

本节目标：学完后你能理解工作目录的风险，并为配置、日志和上传文件选择稳定路径。

简短引入：很多初学者会写 `./config.yaml`、`./logs/app.log`。开发时能跑，部署后却找不到文件。原因通常是当前工作目录变了。

## 一、为什么需要它

可以理解为：工作目录是进程解析相对路径时的起点。你在不同目录执行同一个程序，`./data` 指向的位置可能完全不同。

后端项目中路径问题常出现在配置文件、日志文件、上传文件、迁移脚本。真实项目中通常会用环境变量或启动参数指定数据目录，而不是完全依赖当前工作目录。

```text
相对路径方便开发，但生产路径必须明确。
```

## 二、基本用法

```go
package main

import (
	"fmt"
	"os"
)

func main() {
	wd, err := os.Getwd()
	if err != nil {
		fmt.Println("get working dir:", err)
		return
	}

	exe, err := os.Executable()
	if err != nil {
		fmt.Println("get executable:", err)
		return
	}

	fmt.Println("working dir:", wd)
	fmt.Println("executable:", exe)
}
```

Windows PowerShell：

```powershell
go run .
```

Linux/macOS：

```bash
go run .
```

## 三、关键代码结构

`os.Getwd()` 返回当前工作目录。它回答的是“程序从哪里启动”。

`os.Executable()` 返回当前可执行文件路径。它回答的是“程序二进制在哪里”。

`os.UserConfigDir()`、`os.UserCacheDir()`、`os.UserHomeDir()` 可以找到用户级目录，但服务端程序通常更倾向由部署配置明确指定路径。

## 四、真实后端场景示例

下面用 `DATA_DIR` 决定项目数据目录，避免把重要数据写到不可控位置：

```go
package main

import (
	"fmt"
	"os"
	"path/filepath"
)

func main() {
	dataDir := os.Getenv("DATA_DIR")
	if dataDir == "" {
		dataDir = "./data"
	}

	absDataDir, err := filepath.Abs(dataDir)
	if err != nil {
		fmt.Println("bad DATA_DIR:", err)
		return
	}

	logPath := filepath.Join(absDataDir, "logs", "app.log")
	uploadDir := filepath.Join(absDataDir, "uploads")

	fmt.Println("log path:", logPath)
	fmt.Println("upload dir:", uploadDir)
}
```

这里使用 `filepath.Join` 而不是字符串拼接，是为了兼容 Windows 和 Linux 的路径分隔符。

## 五、注意点

不要假设部署时一定从项目根目录启动。systemd、Docker、Kubernetes、Windows 服务都有自己的启动方式。

如果你必须读取相对路径配置，要在启动日志中打印解析后的绝对路径，方便排查。

对用户上传文件尤其要小心，不能让用户传入 `../../secret` 这类路径后直接拼接到保存目录。

```text
用户输入可以作为文件名的一部分，但不能直接决定最终路径边界。
```

下面是一个更接近真实上传场景的安全路径示例。它不相信用户传来的文件名，而是由服务端生成保存名，并检查最终路径仍然落在上传目录内。

```go
package main

import (
	"fmt"
	"os"
	"path/filepath"
	"strings"
)

func main() {
	path, err := buildUploadPath("./data/uploads", "../../avatar.png", "u_1001")
	if err != nil {
		fmt.Println("build upload path:", err)
		return
	}
	fmt.Println("safe upload path:", path)
}

func buildUploadPath(uploadDir, originalName, userID string) (string, error) {
	absUploadDir, err := filepath.Abs(uploadDir)
	if err != nil {
		return "", fmt.Errorf("abs upload dir: %w", err)
	}
	if err := os.MkdirAll(absUploadDir, 0755); err != nil {
		return "", fmt.Errorf("create upload dir: %w", err)
	}

	ext := strings.ToLower(filepath.Ext(filepath.Base(originalName)))
	if ext == "" {
		ext = ".bin"
	}

	// 真实项目中通常使用 UUID、雪花 ID 或数据库生成的文件 ID。
	serverName := userID + "-avatar" + ext
	candidate := filepath.Join(absUploadDir, serverName)

	cleanPath := filepath.Clean(candidate)
	rel, err := filepath.Rel(absUploadDir, cleanPath)
	if err != nil {
		return "", fmt.Errorf("check upload path: %w", err)
	}
	if rel == ".." || strings.HasPrefix(rel, ".."+string(os.PathSeparator)) {
		return "", fmt.Errorf("upload path escapes upload dir")
	}

	return cleanPath, nil
}
```

这个例子里，`originalName` 只用来提取扩展名，最终文件名由服务端控制。`filepath.Clean` 可以规范化路径，但它不是万能安全工具，所以还要用 `filepath.Rel` 判断最终路径是否仍在上传目录中。

## 六、常见误区

误区一：把 `os.Getwd()` 当成源码目录。编译后的程序可能在任何位置运行。

误区二：用字符串拼接路径，例如 `dir + "/" + name`。这会带来跨平台问题，也容易漏掉边界处理。

误区三：相信用户传来的文件名。上传文件名应该清洗、改名或生成服务端 ID。

误区四：以为调用了 `filepath.Clean` 就一定安全。它只能整理路径形式，不能替你判断路径是否越界。

## 七、本节达标标准

- 能解释工作目录和可执行文件目录的区别。
- 能用 `filepath.Join` 组织路径。
- 能通过环境变量指定数据目录。
- 知道用户输入不能直接控制文件保存路径。
- 能用服务端生成文件名，并检查最终路径没有逃出上传目录。
