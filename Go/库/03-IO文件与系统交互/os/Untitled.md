 **为什么需要 `${comspec}`**?Windows 上 Go 1.20+ 拒绝执行 cwd 同名程序, 所以用 `cmd.exe` 的绝对路径绕开。这句话是什么意思

这句话讲的是 Windows 上调用外部命令时的一个坑,以及为什么要用 cmd.exe(也就是 %COMSPEC%)的绝对路径来执行命令。逐层拆解:

背景:Go 1.19 的安全修复

Go 1.19 修复了一个安全漏洞(CVE-2022-30580 等一系列 os/exec 相关问题)。核心规则是:

▎ Go 不再在 PATH 里隐式包含当前工作目录(cwd),并且当 LookPath/exec.Command 解析出来的程序路径是「当前目录下的相对路径」时,会返回错误(ErrDot)而拒绝执行。

为什么这是个问题

在 Windows 上,历史习惯是「当前目录优先」——你在某个目录里敲 cmd,系统会先看当前目录有没有 cmd.exe。

这就带来攻击面:
- 假设你的程序在某个目录里运行,而那个目录里被人放了一个恶意的 cmd.exe(或同名程序)。
- 如果你的代码写 exec.Command("cmd", ...),Go 在解析时可能解析到 当前目录下那个恶意的 cmd.exe,而不是系统的那个。
- 于是你以为在调用系统命令,实际上执行了攻击者放进来的程序。

所以 Go 1.20+(其实 1.19 起)的行为是:如果解析结果指向 cwd 同名程序,直接拒绝执行,报错而不是冒险运行。这就是「Go 1.20+ 拒绝执行 cwd 同名程序」的意思。

解决办法:用绝对路径绕开

既然问题出在「相对路径 / 隐式从当前目录解析」,那解决方法就是 不让 Go 去解析,直接给它一个明确的绝对路径。

%COMSPEC%(代码里写成 ${comspec} 或 os.Getenv("COMSPEC"))是 Windows 的一个环境变量,它指向命令解释器的绝对路径,通常是:

C:\Windows\System32\cmd.exe

所以代码会这样写,而不是 exec.Command("cmd", ...):

comspec := os.Getenv("COMSPEC")          // C:\Windows\System32\cmd.exe
cmd := exec.Command(comspec, "/c", "your command")

因为给的是绝对路径:
1. Go 不会触发「相对路径 / cwd 解析」那条规则 → 不会报 ErrDot 错误。
2. 你确定执行的是 System32 里那个真正的系统 cmd.exe,而不是当前目录里可能被人塞进来的同名恶意程序。

一句话总结

▎ Go 出于安全考虑会拒绝执行「从当前目录解析出来的同名程序」。为了既绕开这个限制、又确保调用的是真正的系统命令解释器,代码用环境变量 COMSPEC 拿到 cmd.exe 的绝对路径来执行,而不是用裸名字 "cmd" 让 Go 去模糊解析。
