# 5. HTTP 请求体：安全读取 JSON 接口输入

本节目标：学完后，你能在 Go HTTP 接口中安全读取 JSON 请求体，并限制请求体大小。

## 简短引入

后端接口最常见的 IO 来源之一就是 HTTP 请求体。登录、创建文章、下单、修改库存，很多接口都会从 `r.Body` 读取 JSON。

`r.Body` 本质上是一个 `io.ReadCloser`：可以读，也需要关闭。服务器端通常由 `net/http` 管理关闭，但你仍然要控制读取大小和解析错误。

## 一、为什么需要它

如果接口不限制请求体大小，恶意或异常客户端可以传一个很大的 body，占用内存和连接资源。

真实项目里处理 JSON 请求时通常要做几件事：

- 限制 body 大小。
- 解析 JSON。
- 校验字段。
- 返回清晰错误。
- 不把原始错误细节全部暴露给客户端。

```text
用户输入不能直接相信，HTTP 请求体也一样。
```

## 二、基本用法

新建 `main.go`：

```go
package main

import (
	"encoding/json"
	"fmt"
	"net/http"
)

type CreateArticleRequest struct {
	Title   string `json:"title"`
	Content string `json:"content"`
}

func createArticle(w http.ResponseWriter, r *http.Request) {
	if r.Method != http.MethodPost {
		http.Error(w, "method not allowed", http.StatusMethodNotAllowed)
		return
	}

	r.Body = http.MaxBytesReader(w, r.Body, 1<<20)

	var req CreateArticleRequest
	if err := json.NewDecoder(r.Body).Decode(&req); err != nil {
		http.Error(w, "invalid json body", http.StatusBadRequest)
		return
	}

	if req.Title == "" || req.Content == "" {
		http.Error(w, "title and content are required", http.StatusBadRequest)
		return
	}

	w.Header().Set("Content-Type", "application/json")
	if err := json.NewEncoder(w).Encode(map[string]any{"ok": true, "title": req.Title}); err != nil {
		fmt.Println("write response failed:", err)
	}
}

func main() {
	http.HandleFunc("/articles", createArticle)
	fmt.Println("listening on http://localhost:8080")
	if err := http.ListenAndServe(":8080", nil); err != nil {
		panic(err)
	}
}
```

运行：

Windows PowerShell：

```powershell
go run .\main.go
```

另开一个 PowerShell：

```powershell
Invoke-RestMethod -Method Post -Uri http://localhost:8080/articles -ContentType "application/json" -Body '{"title":"Go IO","content":"hello"}'
```

Linux/macOS：

```bash
go run ./main.go
```

另开一个终端：

```bash
curl -X POST http://localhost:8080/articles \
  -H 'Content-Type: application/json' \
  -d '{"title":"Go IO","content":"hello"}'
```

## 三、关键参数/语法/代码结构

`r.Body` 是请求体读取入口。读取一次后就消耗了，不能默认再读第二次。

`http.MaxBytesReader(w, r.Body, 1<<20)` 把最大读取限制设为 1MB。真实项目中要按接口类型设置，例如普通 JSON 通常不需要几十 MB。

`json.NewDecoder(r.Body).Decode(&req)` 从流中解析 JSON，不需要先 `ReadAll`。

字段校验不要只依赖 JSON 解析。JSON 合法不代表业务合法，空标题、负库存、非法状态都要单独校验。

## 四、真实后端场景示例

创建文章接口中，IO 只负责把请求体读出来。真正的业务还要校验权限、写数据库、记录日志。

```go
func validateArticle(req CreateArticleRequest) error {
	if len(req.Title) < 2 {
		return fmt.Errorf("title is too short")
	}
	if len(req.Content) < 10 {
		return fmt.Errorf("content is too short")
	}
	return nil
}
```

真实项目中写数据库时要使用参数化查询或 ORM 的参数绑定。

```text
用户输入不能直接拼进 SQL 字符串。
```

## 五、注意点

不要在接口里随意使用 `io.ReadAll(r.Body)`。小接口可以，但要先限制大小。

不要把后端内部错误原样返回给客户端。可以对用户返回 `invalid json body`，同时在服务端日志里记录具体错误。

对需要幂等或事务的接口，例如下单、扣库存，要把 IO 解析和数据库事务边界分清楚。请求体解析成功不代表业务已经成功。

## 六、生产版改造建议

如果这个接口要进入真实项目，可以继续增强：

- 使用统一响应结构，例如 `{ "code": "...", "message": "..." }`，不要每个接口随意返回。
- 对请求体设置合理上限，普通 JSON 接口通常几百 KB 到 1 MB 就够。
- 配合中间件设置请求超时，避免客户端长时间占用连接。
- JSON 解码后做字段级校验，例如标题长度、库存数量范围、订单状态枚举。
- 写数据库时使用参数化查询，并把事务边界放在业务服务层。

```text
请求体解析只是入口，真正的安全来自大小限制、字段校验、权限校验和事务边界。
```

## 七、常见误区

误区一：不限制请求体大小。这样接口可能被大 body 拖慢甚至打满内存。

误区二：读取 body 后又让其他中间件读取。body 默认只能读一次，日志中间件要谨慎处理。

误区三：JSON 解析成功就直接入库。业务字段仍然需要校验。

误区四：把客户端错误和服务端错误混在一起。JSON 格式错误通常是 400，数据库故障通常是 500。

## 八、小练习

把文章创建接口改成“创建短链接”接口。

要求：

- 请求体字段为 `url` 和 `expire_days`。
- 限制请求体最大 64 KB。
- 校验 `url` 不能为空，`expire_days` 必须在 1 到 365 之间。
- 返回 JSON，包含 `ok` 和 `short_code`。

思考：真实项目里，生成短链接前还需要做哪些校验？例如 URL 协议、黑名单、用户权限和重复提交。

## 九、本节达标标准

- 能从 `r.Body` 解码 JSON。
- 能使用 `http.MaxBytesReader` 限制请求体大小。
- 知道请求体通常只能读取一次。
- 能区分 JSON 解析错误和业务校验错误。
- 知道用户输入入库前必须参数化处理。
