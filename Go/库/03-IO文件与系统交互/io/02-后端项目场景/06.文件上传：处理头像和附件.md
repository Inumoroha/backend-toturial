# 6. 文件上传：处理头像和附件

本节目标：学完后，你能实现一个限制大小、生成服务端文件名、保存上传文件的接口。

## 简短引入

头像、合同附件、文章封面、导入文件，都是后端常见上传场景。上传接口看起来只是“接收文件并保存”，但生产中最容易踩坑的是大小限制、文件名、目录权限和清理策略。

## 一、为什么需要它

上传文件来自用户，不能默认可信。真实项目中通常会：

- 限制请求整体大小。
- 限制单个文件大小。
- 不直接使用原始文件名落盘。
- 按业务分类存储。
- 保存元数据到数据库。
- 对敏感文件做权限校验。

```text
上传文件的原始文件名只适合展示，不适合直接作为服务器存储路径。
```

## 二、基本用法

新建 `main.go`：

```go
package main

import (
	"fmt"
	"io"
	"net/http"
	"os"
	"path/filepath"
	"time"
)

func uploadAvatar(w http.ResponseWriter, r *http.Request) {
	if r.Method != http.MethodPost {
		http.Error(w, "method not allowed", http.StatusMethodNotAllowed)
		return
	}

	r.Body = http.MaxBytesReader(w, r.Body, 2<<20)

	file, header, err := r.FormFile("avatar")
	if err != nil {
		http.Error(w, "avatar is required", http.StatusBadRequest)
		return
	}
	defer file.Close()

	dir := filepath.Join("data", "uploads", "avatar")
	if err := os.MkdirAll(dir, 0755); err != nil {
		http.Error(w, "cannot create upload dir", http.StatusInternalServerError)
		return
	}

	ext := filepath.Ext(header.Filename)
	name := fmt.Sprintf("avatar_%d%s", time.Now().UnixNano(), ext)
	dstPath := filepath.Join(dir, name)

	dst, err := os.Create(dstPath)
	if err != nil {
		http.Error(w, "cannot save file", http.StatusInternalServerError)
		return
	}
	defer dst.Close()

	if _, err := io.Copy(dst, file); err != nil {
		http.Error(w, "cannot write file", http.StatusInternalServerError)
		return
	}

	fmt.Fprintf(w, "saved %s as %s\n", header.Filename, dstPath)
}

func main() {
	http.HandleFunc("/upload/avatar", uploadAvatar)
	fmt.Println("listening on http://localhost:8080")
	if err := http.ListenAndServe(":8080", nil); err != nil {
		panic(err)
	}
}
```

运行：

Windows PowerShell：

```powershell
"fake image content" | Set-Content .\avatar.txt
go run .\main.go
```

另开一个 PowerShell：

```powershell
curl.exe -F "avatar=@avatar.txt" http://localhost:8080/upload/avatar
```

Linux/macOS：

```bash
echo "fake image content" > avatar.txt
go run ./main.go
```

另开一个终端：

```bash
curl -F "avatar=@avatar.txt" http://localhost:8080/upload/avatar
```

## 三、关键参数/语法/代码结构

`r.FormFile("avatar")` 从 multipart 表单里取文件字段。字段名要和前端表单一致。

`header.Filename` 是客户端传来的原始文件名，只适合展示或记录。

`os.Create(dstPath)` 创建目标文件。如果文件已存在会截断，所以服务端文件名要避免冲突。

`io.Copy(dst, file)` 负责把上传内容写入目标文件。这里仍然要检查错误。

## 四、真实后端场景示例

头像上传成功后，真实项目通常不会只返回本地路径，而是会保存一条记录：

```text
user_id: 1001
original_name: avatar.png
storage_path: data/uploads/avatar/avatar_1783317000000000000.png
size: 184023
created_at: 2026-07-06 13:00:00
```

如果使用对象存储，`dst` 可能不是本地文件，而是对象存储 SDK 提供的 Writer。业务函数依赖 `io.Reader` 会更容易迁移。

## 五、注意点

`http.MaxBytesReader` 限制的是读取上限，但你仍然要考虑前置网关限制，例如 Nginx 的上传大小限制。

不要只根据扩展名判断文件安全。头像上传通常还会检查 MIME、图片解码、尺寸和安全扫描。

写入数据库和写入文件之间没有天然事务。真实项目中要设计补偿逻辑，例如文件写成功但数据库失败时删除文件，或者定期清理孤儿文件。

## 六、生产版改造建议

如果上传接口要进入真实项目，可以继续增强：

- 使用 `http.DetectContentType` 或图片解码确认文件类型，不只看扩展名。
- 按用户 ID 或业务 ID 分目录，避免单目录文件过多。
- 保存文件大小、哈希、原始文件名、存储路径和上传人。
- 对图片类文件做尺寸限制，避免超大图片拖慢后续处理。
- 文件写成功但数据库写失败时，执行删除文件的补偿逻辑。
- 文件量增长后，把本地磁盘替换为对象存储，例如 S3、OSS、COS。

```text
上传接口的生产风险通常不在 io.Copy，而在文件可信度、权限、一致性和清理策略。
```

## 七、常见误区

误区一：把 `header.Filename` 直接拼到路径里。这样可能重名，也可能包含危险路径片段。

误区二：不限制上传大小。大文件可能占满磁盘或拖慢服务。

误区三：上传成功后不保存元数据。后续权限校验、下载、清理都会变困难。

误区四：文件写入和数据库写入失败时没有补偿策略。长期运行后容易产生大量无主文件。

## 八、小练习

把头像上传接口改成“文章附件上传”接口。

要求：

- 表单字段名改为 `attachment`。
- 存储目录改为 `data/uploads/article-attachment`。
- 限制请求大小为 5 MB。
- 返回原始文件名和服务端存储路径。
