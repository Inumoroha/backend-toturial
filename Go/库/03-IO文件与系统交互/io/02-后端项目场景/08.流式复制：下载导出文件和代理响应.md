# 8. 流式复制：下载导出文件和代理响应

本节目标：学完后，你能用流式方式把文件写给客户端，避免把大文件一次性读入内存。

## 简短引入

后台系统经常提供导出下载：用户列表、订单明细、库存报表、迁移结果。如果文件很大，接口不应该先 `ReadFile` 再写响应，而应该边读文件边写给客户端。

## 一、为什么需要它

导出文件可能从几 MB 增长到几百 MB。一次性读入内存会占用大量资源，多个用户同时下载时更明显。

流式复制的直觉是：

```text
服务端像水管一样搬运数据，而不是先用桶把所有水装起来。
```

## 二、基本用法

先创建一个导出文件：

Windows PowerShell：

```powershell
@"
id,total
9001,199
9002,299
"@ | Set-Content .\orders.csv
```

Linux/macOS：

```bash
cat > orders.csv <<'EOF'
id,total
9001,199
9002,299
EOF
```

新建 `main.go`：

```go
package main

import (
	"fmt"
	"io"
	"net/http"
	"os"
)

func downloadOrders(w http.ResponseWriter, r *http.Request) {
	file, err := os.Open("orders.csv")
	if err != nil {
		http.Error(w, "file not found", http.StatusNotFound)
		return
	}
	defer file.Close()

	w.Header().Set("Content-Type", "text/csv")
	w.Header().Set("Content-Disposition", `attachment; filename="orders.csv"`)

	if _, err := io.Copy(w, file); err != nil {
		fmt.Println("download interrupted:", err)
		return
	}
}

func main() {
	http.HandleFunc("/downloads/orders", downloadOrders)
	fmt.Println("listening on http://localhost:8080")
	if err := http.ListenAndServe(":8080", nil); err != nil {
		panic(err)
	}
}
```

运行后访问：

```text
http://localhost:8080/downloads/orders
```

## 三、关键参数/语法/代码结构

`os.Open("orders.csv")` 打开文件作为 `Reader`。

`w http.ResponseWriter` 同时也是写响应的目标，可以当作 `Writer` 使用。

`io.Copy(w, file)` 会从文件读，写到 HTTP 响应。

`Content-Disposition` 告诉浏览器这是附件下载。

## 四、真实后端场景示例

真实项目通常不会让用户随便传文件路径下载，而是通过导出任务 ID 查询数据库：

```text
GET /exports/123/download

服务端流程：
1. 校验用户是否有权限下载 export_id=123。
2. 查询导出文件 storage_path。
3. 确认文件存在且状态为 finished。
4. 设置响应头。
5. 使用 io.Copy 流式写出。
```

如果导出文件在对象存储里，读取来源可能是对象存储返回的 `Reader`，但下载处理函数的思路不变。

## 五、注意点

不要把客户端传入的路径直接传给 `os.Open`。这会带来路径穿越风险。

下载前要做权限校验。订单、工资单、合同文件都不能只靠“知道链接”访问。

`io.Copy` 返回错误可能是客户端下载中断。这个错误通常要记录，但不一定代表服务端异常。

大文件下载要关注超时、限速、对象存储签名 URL、断点续传等问题。初学阶段先掌握流式写出。

## 六、生产版改造建议

如果下载接口要进入真实项目，可以继续增强：

- 用导出任务 ID 查询文件，不让客户端传物理路径。
- 下载前校验用户权限和导出任务归属。
- 设置 `Content-Length`，方便前端展示进度。
- 静态文件下载可以考虑 `http.ServeFile`，它会处理一些 HTTP 细节。
- 大文件放在对象存储时，可以返回短期有效的签名 URL，减少业务服务带宽压力。
- 对敏感文件设置较短的下载有效期，并记录下载审计日志。

```text
下载接口真正的边界不是文件读取，而是权限、路径控制和带宽成本。
```

## 七、常见误区

误区一：导出下载都用 `os.ReadFile`。并发下载时内存压力会很大。

误区二：让用户传 `?path=...` 直接下载。真实项目中应该使用业务 ID 和权限校验。

误区三：忘记设置响应头。浏览器可能直接展示内容，而不是下载文件。

误区四：把客户端下载中断都当成严重错误。用户关闭页面、网络断开都可能导致写响应失败。

## 九、本节达标标准

- 能用 `io.Copy` 把文件流式写入 HTTP 响应。
- 知道下载接口要做权限校验。
- 知道不要直接使用用户传入的文件路径。
- 能解释为什么大文件下载不适合 `ReadFile`。
- 能设置基础下载响应头。
