# 7. 日志写入：记录业务行为与排障信息

本节目标：学完后，你能把业务日志追加写入文件，并理解日志写入的生产注意点。

## 简短引入

日志是后端排障的基本工具。用户登录失败、订单状态变化、权限拒绝、库存扣减失败，都应该留下足够的信息帮助排查。

Go 标准库的 `log` 可以完成基础日志写入，真实项目中也常用 zap、zerolog 等结构化日志库。本节先掌握底层文件写入思路。

## 一、为什么需要它

没有日志的系统，就像没有监控的仓库。出了问题只能猜。

但日志不是越多越好。生产中要注意：

- 不记录密码、token、身份证等敏感信息。
- 日志文件要追加写，不要随意覆盖。
- 日志要有时间、业务 ID、错误原因。
- 日志量大时要轮转和归档。

```text
日志用于排障，不应该成为泄露敏感数据的地方。
```

## 二、基本用法

新建 `main.go`：

```go
package main

import (
	"log"
	"os"
)

func main() {
	file, err := os.OpenFile("app.log", os.O_CREATE|os.O_WRONLY|os.O_APPEND, 0644)
	if err != nil {
		panic(err)
	}
	defer file.Close()

	logger := log.New(file, "order-service ", log.LstdFlags|log.Lshortfile)
	logger.Println("order_id=9001 status=created")
	logger.Println("order_id=9001 status=paid")
}
```

运行：

Windows PowerShell：

```powershell
go run .\main.go
Get-Content .\app.log
```

Linux/macOS：

```bash
go run ./main.go
cat ./app.log
```

## 三、关键参数/语法/代码结构

`os.OpenFile` 可以指定打开方式。这里使用：

- `os.O_CREATE`：文件不存在就创建。
- `os.O_WRONLY`：只写。
- `os.O_APPEND`：追加写，不覆盖已有内容。

`log.New(file, prefix, flags)` 创建一个写到文件的 logger。

`log.LstdFlags` 输出日期和时间，`log.Lshortfile` 输出文件名和行号。

## 四、真实后端场景示例

下面模拟订单状态变更日志：

```go
package main

import (
	"fmt"
	"log"
	"os"
)

type OrderEvent struct {
	OrderID int
	From    string
	To      string
	Reason  string
}

func WriteOrderEvent(logger *log.Logger, event OrderEvent) {
	logger.Printf("order_id=%d from=%s to=%s reason=%q", event.OrderID, event.From, event.To, event.Reason)
}

func main() {
	file, err := os.OpenFile("order_audit.log", os.O_CREATE|os.O_WRONLY|os.O_APPEND, 0644)
	if err != nil {
		panic(err)
	}
	defer file.Close()

	logger := log.New(file, "audit ", log.LstdFlags)
	WriteOrderEvent(logger, OrderEvent{OrderID: 9001, From: "created", To: "paid", Reason: "payment callback"})

	fmt.Println("audit log written")
}
```

真实项目中，审计日志可能需要写数据库或消息队列，不能只依赖本地文件。

## 五、注意点

多进程同时写同一个日志文件要谨慎。容器部署中更常见的方式是写标准输出，由日志系统采集。

日志文件会增长。生产环境要有轮转策略，例如按大小或日期切分，避免磁盘被打满。

日志内容要服务排障。只写 `failed` 没意义，应该带上业务 ID、阶段和可控错误原因。

敏感字段要脱敏。比如手机号可以只保留后四位，token 不应该打印。

## 六、生产版改造建议

如果日志功能要进入真实项目，可以继续增强：

- 使用结构化日志库，让日志字段可以被检索，例如 `order_id`、`user_id`、`trace_id`。
- 容器环境优先写标准输出，由日志平台采集。
- 本地文件日志要配置轮转，例如按大小或日期切分。
- 接入请求链路 ID，方便从网关、业务服务、数据库慢查询之间串联排查。
- 对敏感字段做统一脱敏，不靠开发者每次手动记住。

```text
生产日志要能被检索、能控制体积、能保护敏感信息。
```

## 七、常见误区

误区一：用 `os.WriteFile` 写日志。它会覆盖文件，不适合追加日志。

误区二：日志里打印完整请求体。可能包含密码、token、隐私字段，也可能造成日志量暴涨。

误区三：把日志当成可靠消息队列。日志主要用于记录和排查，不能替代事务和业务状态。

误区四：没有清理和轮转。系统长期运行后磁盘可能被日志占满。

## 八、小练习

把订单审计日志改成“权限拒绝日志”。

要求：

- 日志字段包含 `user_id`、`resource`、`action`、`reason`。
- 追加写入 `permission_audit.log`。
- 不要记录 token、密码或完整请求头。

思考：真实项目中，权限拒绝日志应该给安全审计看，还是只给开发排障看？不同答案会影响字段设计。

## 九、本节达标标准

- 能用 `os.OpenFile` 追加写日志。
- 能创建一个写入文件的 `log.Logger`。
- 知道日志不能记录敏感信息。
- 知道生产日志需要轮转、采集和清理策略。
- 能设计一条有业务 ID 的排障日志。
