# 1. Reader 与 Writer：理解 Go 后端 IO 的统一入口

本节目标：学完后，你能用 `io.Reader` 和 `io.Writer` 把数据从一个地方复制到另一个地方。

## 简短引入

后端开发里经常要处理“数据从哪来、写到哪去”的问题。比如从 HTTP 请求体读取 JSON，把导出内容写到响应里，从文件读取配置，把日志写到文件。

Go 把这类问题抽象成两个很常用的接口：`io.Reader` 和 `io.Writer`。初学阶段可以先理解为：

- `Reader`：能读出字节的东西。
- `Writer`：能写入字节的东西。

## 一、为什么需要它

真实项目中，数据来源不固定。今天你从文件读用户数据，明天可能从 HTTP 请求体读，后天可能从压缩包里读。

如果每种来源都写一套逻辑，代码会很散。Go 的习惯是让业务逻辑依赖 `io.Reader`，这样同一段处理代码可以服务不同来源。

```text
业务代码尽量依赖 Reader/Writer 这类小接口，而不是一开始就绑死具体文件或网络连接。
```

## 二、基本用法

新建 `main.go`：

```go
package main

import (
	"fmt"
	"io"
	"strings"
)

func main() {
	source := strings.NewReader("user_id=1001 action=login\n")

	n, err := io.Copy(io.Discard, source)
	if err != nil {
		panic(err)
	}

	fmt.Printf("copied %d bytes\n", n)
}
```

运行：

Windows PowerShell：

```powershell
go run .\main.go
```

Linux/macOS：

```bash
go run ./main.go
```

这里先把数据复制到 `io.Discard`，它可以理解为“黑洞 Writer”，用于测试读取链路是否正常。

## 三、关键参数/语法/代码结构

`strings.NewReader("...")` 创建了一个内存里的读取来源。真实项目里，它的位置可能换成文件、请求体或压缩流。

`io.Copy(dst, src)` 的意思是：从 `src` 读，写到 `dst`。它会循环读取，不需要你手写 `for`。

`io.Discard` 会丢弃写入内容。常见场景是测试、跳过响应体、计算读取字节数。

`n, err := io.Copy(...)` 里的 `n` 是复制的字节数。真实项目中可以用来记录日志或判断导出大小。

## 四、真实后端场景示例

假设你有一个“审计日志写入器”，不关心日志从哪里来，只负责把内容写到目标里：

```go
package main

import (
	"bytes"
	"fmt"
	"io"
	"strings"
)

func SaveAuditLog(dst io.Writer, src io.Reader) error {
	_, err := io.Copy(dst, src)
	return err
}

func main() {
	var storage bytes.Buffer
	requestBody := strings.NewReader(`{"user_id":1001,"action":"delete_article"}` + "\n")

	if err := SaveAuditLog(&storage, requestBody); err != nil {
		panic(err)
	}

	fmt.Print(storage.String())
}
```

这个例子里，`SaveAuditLog` 不知道 `src` 是字符串、文件还是请求体。真实项目中这种写法更容易测试，也更容易替换存储目标。

## 五、注意点

`Reader` 和 `Writer` 只处理字节，不负责理解业务含义。JSON 是否合法、用户是否有权限、字段是否缺失，都要在更上层处理。

`io.Copy` 会一直读到 EOF 或发生错误。用于网络连接时，要配合超时、请求大小限制或上下文取消。

如果来源或目标同时实现了 `io.Closer`，真实项目里通常要关闭它。比如文件和 HTTP 响应体都需要关闭。

## 六、常见误区

误区一：以为 `Reader` 只能读文件。其实字符串、请求体、响应体、压缩流都可以是 `Reader`。

误区二：复制数据后不检查错误。磁盘满、连接断开、权限不足都会导致写入失败。

误区三：把 `Reader` 读了一遍还想继续读。大多数 `Reader` 读过就消耗掉了，不能默认重新开始。

## 七、小练习

把 `SaveAuditLog` 改成 `CopyRequestBody`，模拟把一段订单创建请求体复制到 `bytes.Buffer` 中。

要求：

- 输入内容包含 `order_id` 和 `amount`。
- 函数参数仍然使用 `io.Writer` 和 `io.Reader`。
- 打印复制后的内容和复制字节数。

这个练习的重点不是订单业务本身，而是训练你把“数据来源”和“写入目标”拆开。

## 八、本节达标标准

- 能说清楚 `Reader` 是数据来源，`Writer` 是数据目标。
- 能用 `io.Copy` 完成一次复制。
- 能把业务函数参数写成 `io.Reader` 或 `io.Writer`。
- 知道 `Reader` 通常不能无成本重复读取。
