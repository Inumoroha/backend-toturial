# 2. 文件读写：保存用户资料和业务数据

本节目标：学完后，你能把业务数据写入文件，并能从文件中读取回来。

## 简短引入

真实后端项目不会把重要业务数据随便写到文件里替代数据库，但文件读写依然很常见：配置文件、导出结果、批处理输入、临时缓存、审计日志、迁移报告都离不开它。

## 一、为什么需要它

例如后台系统要导出一批用户列表，接口可能先生成一个文件，再让前端下载。又比如一个订单批处理任务，需要把失败记录写成报告，方便运营重新处理。

文件 IO 看起来简单，真正容易出问题的是：

- 文件路径是否存在。
- 权限是否足够。
- 是否覆盖了旧文件。
- 写入失败时是否留下半个文件。
- 错误有没有返回给调用方。

## 二、基本用法

新建 `main.go`：

```go
package main

import (
	"fmt"
	"os"
)

func main() {
	data := []byte("id,name\n1001,Alice\n1002,Bob\n")

	if err := os.WriteFile("users.csv", data, 0644); err != nil {
		panic(err)
	}

	content, err := os.ReadFile("users.csv")
	if err != nil {
		panic(err)
	}

	fmt.Print(string(content))
}
```

运行：

Windows PowerShell：

```powershell
go run .\main.go
Get-Content .\users.csv
```

Linux/macOS：

```bash
go run ./main.go
cat ./users.csv
```

## 三、关键参数/语法/代码结构

`os.WriteFile("users.csv", data, 0644)` 会把数据写入文件。如果文件不存在会创建，如果文件存在会覆盖。

`0644` 是文件权限。初学阶段可以理解为：文件所有者可读写，其他人可读。Windows 上权限表现和 Linux 不完全一样，但代码仍可以运行。

`os.ReadFile("users.csv")` 会一次性读取整个文件。小配置、小报告可以这么做，大文件不建议这么做。

```text
ReadFile 适合小文件；日志、大 CSV、上传文件不要默认一次性读入内存。
```

## 四、真实后端场景示例

下面模拟“导出用户列表到文件”。真实项目中，用户数据通常来自数据库，这里先用内存切片代替。

```go
package main

import (
	"fmt"
	"os"
	"strings"
)

type User struct {
	ID   int
	Name string
}

func ExportUsers(path string, users []User) error {
	var b strings.Builder
	b.WriteString("id,name\n")

	for _, user := range users {
		b.WriteString(fmt.Sprintf("%d,%s\n", user.ID, user.Name))
	}

	return os.WriteFile(path, []byte(b.String()), 0644)
}

func main() {
	users := []User{
		{ID: 1001, Name: "Alice"},
		{ID: 1002, Name: "Bob"},
	}

	if err := ExportUsers("users_export.csv", users); err != nil {
		panic(err)
	}

	fmt.Println("exported users_export.csv")
}
```

这个版本适合小数据量。真实导出上万行时，应该用 `os.Create` 配合逐行写入，避免 `strings.Builder` 占用过多内存。

## 五、注意点

不要把用户传入的文件名直接用于写入路径。比如上传接口里，用户可能传入 `../../important.txt`。真实项目中通常会重新生成文件名，例如用业务 ID、时间戳或 UUID。

写重要文件时，建议先写临时文件，再替换正式文件。这样可以降低写到一半失败导致文件损坏的风险。

覆盖文件前要确认业务语义。导出文件覆盖通常可以接受，配置文件、迁移结果、审计日志覆盖就要非常谨慎。

## 六、生产版改造建议

如果这个导出功能要进入真实项目，可以做三点增强：

- 输出文件先写到 `users_export.csv.tmp`，成功后再重命名为正式文件。
- 文件名带上任务 ID 或日期，避免多次导出互相覆盖。
- 导出任务状态写入数据库，例如 `pending`、`running`、`finished`、`failed`。

```text
生产里的文件写入，重点不是“能写进去”，而是失败时能知道发生了什么，并且不破坏旧数据。
```

## 七、常见误区

误区一：所有文件都用 `ReadFile`。大文件会占用大量内存，甚至拖垮服务。

误区二：写文件不处理错误。磁盘满、目录不存在、权限不足都很常见。

误区三：直接相信用户传来的文件名。上传和导出场景尤其容易出现路径穿越风险。

误区四：把文件当数据库。文件缺少事务、并发控制和查询能力，不适合承载核心在线业务数据。

## 八、小练习

把示例里的“用户导出”改成“订单导出”。

要求：

- 定义 `Order` 结构体，包含 `ID`、`UserID`、`Total`。
- 导出文件名为 `orders_export.csv`。
- CSV 表头为 `id,user_id,total`。
- 思考：如果订单量有 100 万行，这个版本哪里需要改？

## 九、本节达标标准

- 能用 `os.WriteFile` 写小文件。
- 能用 `os.ReadFile` 读小文件。
- 知道 `WriteFile` 会覆盖已有文件。
- 知道大文件要避免一次性读入内存。
- 能解释为什么不能直接使用用户提供的文件路径。
