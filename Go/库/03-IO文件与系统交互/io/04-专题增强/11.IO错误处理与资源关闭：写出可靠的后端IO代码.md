# 11. IO 错误处理与资源关闭：写出可靠的后端 IO 代码

本节目标：学完后，你能写出会关闭资源、会检查缓冲写入、能处理部分写入和超时取消的 IO 代码。

## 简短引入

很多初学者写 IO 代码时，重点都放在“读到了没有、写进去了没有”。真实后端项目里，更重要的是：失败时有没有被发现，资源有没有释放，写到一半怎么办，请求取消后还要不要继续处理。

本节不讲复杂内核细节，只讲后端开发中最常见、最容易踩坑的工程习惯。

```text
可靠的 IO 代码，不是永远不失败，而是失败时可发现、可停止、可清理、可追踪。
```

## 一、为什么需要它

后端 IO 经常和外部世界交互：

- 文件可能打开失败。
- 磁盘可能写满。
- 网络连接可能中断。
- 客户端可能取消请求。
- 缓冲区可能还没真正写到底层文件。
- 数据可能只写入了一部分。

如果代码只写“快乐路径”，线上问题会很难排查。比如导出接口返回成功，但文件只有一半；上传接口失败了，但临时文件没删除；批处理任务中断了，却留下了一个看起来正常的结果文件。

## 二、基本用法

先看一个更稳的文件写入函数：先写临时文件，检查 Flush 和 Close，最后再替换正式文件。

新建 `main.go`：

```go
package main

import (
	"bufio"
	"fmt"
	"os"
)

func WriteReport(path string, lines []string) error {
	tmpPath := path + ".tmp"

	file, err := os.Create(tmpPath)
	if err != nil {
		return err
	}

	writer := bufio.NewWriter(file)
	for _, line := range lines {
		if _, err := writer.WriteString(line + "\n"); err != nil {
			file.Close()
			return err
		}
	}

	if err := writer.Flush(); err != nil {
		file.Close()
		return err
	}

	if err := file.Close(); err != nil {
		return err
	}

	return os.Rename(tmpPath, path)
}

func main() {
	lines := []string{
		"order_id,status",
		"9001,paid",
		"9002,refunded",
	}

	if err := WriteReport("orders_report.csv", lines); err != nil {
		panic(err)
	}

	fmt.Println("report written")
}
```

运行：

Windows PowerShell：

```powershell
go run .\main.go
Get-Content .\orders_report.csv
```

Linux/macOS：

```bash
go run ./main.go
cat ./orders_report.csv
```

## 三、关键参数/语法/代码结构

`defer file.Close()` 很常见，但不是所有场景都适合只依赖 defer。只读文件时，`defer Close` 通常够用。写重要文件时，最好显式检查 `Close` 返回的错误，因为某些写入错误可能在关闭时才暴露。

`bufio.NewWriter(file)` 会先写入内存缓冲区，不一定立刻写到底层文件。写完后必须调用 `Flush()`。

`writer.WriteString(...)` 返回写入字节数和错误。多数标准库 Writer 会在错误时返回非 nil error，真实项目里要检查。

`os.Rename(tmpPath, path)` 用于把临时文件替换成正式文件。同一文件系统内通常可以降低半文件风险。

```text
用了缓冲 Writer，就必须 Flush；写重要文件，就要检查 Flush 和 Close。
```

部分写入可以理解为：你希望写入 100 个字节，但底层只接受了其中一部分。多数时候，标准库的高级工具会帮你循环处理，例如 `io.Copy`、`csv.Writer`、`bufio.Writer`。但如果你直接调用底层 `Write`，就要看返回的 `n` 和 `err`。

```go
n, err := file.Write(data)
if err != nil {
	return err
}
if n != len(data) {
	return fmt.Errorf("short write: wrote %d bytes, want %d", n, len(data))
}
```

真实项目中更常见的保守写法是：能用 `io.Copy`、`io.CopyBuffer`、`csv.Writer`、`bufio.Writer` 这类封装，就不要自己手写复杂循环。自己手写时，一定要处理 `n < len(data)` 的情况。

## 四、真实后端场景示例

下面模拟一个“导出订单报表”的 HTTP 接口。重点不是报表内容，而是请求取消后不要继续无意义地生成数据。

```go
package main

import (
	"context"
	"encoding/csv"
	"fmt"
	"net/http"
	"strconv"
	"time"
)

type Order struct {
	ID     int
	Status string
	Total  int
}

func LoadOrders(ctx context.Context) ([]Order, error) {
	var orders []Order

	for i := 1; i <= 5; i++ {
		select {
		case <-ctx.Done():
			return nil, ctx.Err()
		case <-time.After(200 * time.Millisecond):
			orders = append(orders, Order{
				ID:     9000 + i,
				Status: "paid",
				Total:  100 * i,
			})
		}
	}

	return orders, nil
}

func exportOrders(w http.ResponseWriter, r *http.Request) {
	ctx := r.Context()

	orders, err := LoadOrders(ctx)
	if err != nil {
		http.Error(w, "request canceled or timeout", http.StatusRequestTimeout)
		return
	}

	w.Header().Set("Content-Type", "text/csv")
	w.Header().Set("Content-Disposition", `attachment; filename="orders.csv"`)

	writer := csv.NewWriter(w)
	if err := writer.Write([]string{"id", "status", "total"}); err != nil {
		fmt.Println("write header failed:", err)
		return
	}

	for _, order := range orders {
		record := []string{
			strconv.Itoa(order.ID),
			order.Status,
			strconv.Itoa(order.Total),
		}
		if err := writer.Write(record); err != nil {
			fmt.Println("write row failed:", err)
			return
		}
	}

	writer.Flush()
	if err := writer.Error(); err != nil {
		fmt.Println("flush csv failed:", err)
		return
	}
}

func main() {
	http.HandleFunc("/exports/orders", exportOrders)
	fmt.Println("listening on http://localhost:8080")
	if err := http.ListenAndServe(":8080", nil); err != nil {
		panic(err)
	}
}
```

运行：

Windows PowerShell：

```powershell
go run .\main.go
```

另开一个 PowerShell：

```powershell
Invoke-WebRequest -Uri http://localhost:8080/exports/orders -OutFile .\orders.csv
```

Linux/macOS：

```bash
go run ./main.go
```

另开一个终端：

```bash
curl -o orders.csv http://localhost:8080/exports/orders
```

真实项目中，`LoadOrders` 通常会访问数据库。数据库查询也应该接收 `context.Context`，这样请求取消或超时时，查询可以尽快停止。

## 五、注意点

只读文件可以常规写法：

```go
file, err := os.Open(path)
if err != nil {
	return err
}
defer file.Close()
```

写文件时要更谨慎。尤其是配置、归档、迁移报告、导出结果这类文件，建议检查 `Flush`、检查 `Close`，并优先写临时文件。

HTTP 响应一旦开始写出，状态码通常就不能再随意改了。因此下载接口里，如果中途写失败，通常只能记录日志，不能再给客户端返回一个漂亮的 JSON 错误。

对网络 IO 和数据库 IO，要尽量传递 `context.Context`。请求取消后还继续处理，既浪费资源，也可能造成无意义的写入。

```text
能在函数参数里传 context 的 IO 操作，真实项目中通常都应该传。
```

## 六、常见误区

误区一：用了 `bufio.Writer` 但忘记 `Flush`。程序看起来没报错，文件却可能缺内容。

误区二：写文件只 `defer Close`，从不检查关闭错误。某些写入错误可能延迟到关闭时才出现。

误区三：认为 `Write` 成功一次就代表完整业务成功。真实项目中要区分写入一行成功、整个文件成功、文件替换成功。

误区四：请求取消后后台仍继续生成导出文件。用户已经断开连接，服务还在占用 CPU、数据库连接和磁盘 IO。

误区五：把临时文件留得到处都是。失败时可以保留用于排查，但要有清理策略和命名规范。

## 七、小练习

给 `WriteReport` 增加失败清理逻辑。

要求：

- 如果写入、Flush 或 Close 失败，删除 `.tmp` 文件。
- 删除失败时不要覆盖原始错误，可以把删除失败记录到日志。
- 把正式文件名改成 `inventory_report.csv`。
- 报表内容改成库存字段：`sku,stock`。

进阶思考：如果这个文件是生产配置文件，替换前是否需要先备份旧文件？

## 八、生产版改造建议

如果这类 IO 代码要进入真实项目，可以继续增强：

- 把临时文件写到和正式文件同一个目录，减少跨文件系统 `Rename` 失败概率。
- 对重要输出计算哈希，便于校验内容完整性。
- 对长时间导出任务设置超时，并把任务状态写入数据库。
- 失败时记录输入参数、输出路径、处理数量和错误原因。
- 对临时文件增加定期清理任务。
- 对涉及数据库的 IO 流程明确事务边界，不要把文件写入误当成数据库事务的一部分。

## 九、本节达标标准

- 能解释为什么 `bufio.Writer` 写完要 `Flush`。
- 能区分只读文件关闭和重要写文件关闭的处理差异。
- 能用临时文件降低半文件风险。
- 能在 HTTP 或数据库 IO 中传递 `context.Context`。
- 知道 HTTP 响应写出后，中途错误通常只能记录日志。
- 能为失败的临时文件设计清理策略。
