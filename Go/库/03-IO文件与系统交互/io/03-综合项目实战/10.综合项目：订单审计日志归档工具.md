# 10. 综合项目：订单审计日志归档工具

本节目标：学完后，你能实现一个命令行工具，读取订单审计日志、过滤有效记录，并安全写入归档文件。

## 简短引入

这一节把前面的 IO 能力串起来：路径、逐行读取、JSON 解析、临时文件、错误报告、原子替换和统计输出。

项目背景：订单服务每天产生审计日志，每行是一条 JSON。我们要把指定状态的订单记录归档到一个新文件里，坏数据写入错误报告。

## 一、为什么需要它

真实后端项目里，批处理和归档工具很常见。例如：

- 把订单审计日志归档。
- 把失败回调整理给运营。
- 把迁移结果生成报告。
- 把历史数据转换成新格式。

这些工具通常不复杂，但要求稳：不能破坏原文件，失败要能定位，输出要可回滚。

```text
批处理工具的第一原则：失败时不要破坏原始数据。
```

## 二、基本用法

准备输入目录和日志：

Windows PowerShell：

```powershell
mkdir .\logs -Force
@"
{"order_id":9001,"status":"paid","amount":199}
{"order_id":9002,"status":"created","amount":299}
bad json line
{"order_id":9003,"status":"paid","amount":399}
"@ | Set-Content .\logs\orders.log
```

Linux/macOS：

```bash
mkdir -p ./logs
cat > ./logs/orders.log <<'EOF'
{"order_id":9001,"status":"paid","amount":199}
{"order_id":9002,"status":"created","amount":299}
bad json line
{"order_id":9003,"status":"paid","amount":399}
EOF
```

新建 `main.go`：

```go
package main

import (
	"bufio"
	"encoding/json"
	"flag"
	"fmt"
	"os"
	"path/filepath"
)

type OrderLog struct {
	OrderID int    `json:"order_id"`
	Status  string `json:"status"`
	Amount  int    `json:"amount"`
}

type Stats struct {
	Total    int
	Archived int
	Invalid  int
}

func main() {
	input := flag.String("input", filepath.Join("logs", "orders.log"), "input log file")
	output := flag.String("output", filepath.Join("archive", "paid_orders.jsonl"), "archive output file")
	errors := flag.String("errors", filepath.Join("archive", "errors.log"), "error report file")
	status := flag.String("status", "paid", "status to archive")
	flag.Parse()

	stats, err := ArchiveOrders(*input, *output, *errors, *status)
	if err != nil {
		fmt.Fprintln(os.Stderr, "archive failed:", err)
		os.Exit(1)
	}

	fmt.Printf("total=%d archived=%d invalid=%d\n", stats.Total, stats.Archived, stats.Invalid)
}

func ArchiveOrders(inputPath, outputPath, errorPath, status string) (Stats, error) {
	var stats Stats

	input, err := os.Open(inputPath)
	if err != nil {
		return stats, err
	}
	defer input.Close()

	if err := os.MkdirAll(filepath.Dir(outputPath), 0755); err != nil {
		return stats, err
	}
	if err := os.MkdirAll(filepath.Dir(errorPath), 0755); err != nil {
		return stats, err
	}

	tmpOutput := outputPath + ".tmp"
	out, err := os.Create(tmpOutput)
	if err != nil {
		return stats, err
	}
	defer out.Close()

	errFile, err := os.Create(errorPath)
	if err != nil {
		return stats, err
	}
	defer errFile.Close()

	scanner := bufio.NewScanner(input)
	writer := bufio.NewWriter(out)
	errorWriter := bufio.NewWriter(errFile)
	defer writer.Flush()
	defer errorWriter.Flush()

	lineNo := 0
	for scanner.Scan() {
		lineNo++
		stats.Total++

		var record OrderLog
		if err := json.Unmarshal(scanner.Bytes(), &record); err != nil {
			stats.Invalid++
			fmt.Fprintf(errorWriter, "line=%d reason=%q raw=%q\n", lineNo, err.Error(), scanner.Text())
			continue
		}

		if record.Status != status {
			continue
		}

		if _, err := writer.Write(scanner.Bytes()); err != nil {
			return stats, err
		}
		if err := writer.WriteByte('\n'); err != nil {
			return stats, err
		}
		stats.Archived++
	}

	if err := scanner.Err(); err != nil {
		return stats, err
	}
	if err := writer.Flush(); err != nil {
		return stats, err
	}
	if err := errorWriter.Flush(); err != nil {
		return stats, err
	}
	if err := out.Close(); err != nil {
		return stats, err
	}

	if err := os.Rename(tmpOutput, outputPath); err != nil {
		return stats, err
	}

	return stats, nil
}
```

运行：

Windows PowerShell：

```powershell
go run .\main.go -input .\logs\orders.log -output .\archive\paid_orders.jsonl -errors .\archive\errors.log -status paid
Get-Content .\archive\paid_orders.jsonl
Get-Content .\archive\errors.log
```

Linux/macOS：

```bash
go run ./main.go -input ./logs/orders.log -output ./archive/paid_orders.jsonl -errors ./archive/errors.log -status paid
cat ./archive/paid_orders.jsonl
cat ./archive/errors.log
```

## 三、关键参数/语法/代码结构

`flag.String` 定义命令行参数。真实工具不要把输入输出路径写死，方便定时任务和运维脚本调用。

`bufio.Scanner` 逐行读取日志，适合 JSON Lines 这类“一行一条记录”的格式。

`json.Unmarshal(scanner.Bytes(), &record)` 把当前行解析成结构体。

`tmpOutput := outputPath + ".tmp"` 先写临时文件。全部成功后再 `os.Rename` 到正式文件。

`Stats` 用于返回处理结果。真实项目中还会把统计结果写入任务表或监控指标。

## 四、真实后端场景示例

这个工具可以放进定时任务：

```text
每天凌晨：
1. 读取昨天的订单审计日志。
2. 过滤 paid 或 refunded 状态。
3. 生成归档文件。
4. 上传对象存储。
5. 记录处理统计。
6. 失败时报警，并保留原始日志。
```

如果归档结果后续要入库，要把文件处理和数据库事务分开设计。常见做法是先生成可校验的归档文件，再由另一个任务批量入库。

## 五、注意点

`os.Rename` 在同一文件系统内通常可以作为原子替换手段。跨磁盘或跨挂载点时可能失败，生产环境要根据部署方式验证。

示例里错误报告会覆盖旧文件。真实项目可以按日期、任务 ID 命名，避免覆盖历史排障信息。

`Scanner` 不适合超长单行。若单条 JSON 很大，要调大 buffer 或改用 `bufio.Reader`。

归档工具要有可观测性。至少输出总行数、成功数、失败数和输出路径。

## 六、生产版改造建议

如果这个归档工具要进入真实项目，可以继续增强：

- 输出文件名包含日期和任务 ID，避免重复运行互相覆盖。
- 归档成功后计算文件哈希，方便校验文件是否被篡改或损坏。
- 把处理统计写入任务表或监控系统，便于报警。
- 支持 `--dry-run`，上线前先只统计不写正式输出。
- 对输入目录做白名单限制，避免命令参数误指向危险路径。
- 如果后续要删除原始日志，先确认归档文件存在、校验通过、备份完成。

```text
归档类工具宁可多一步校验，也不要急着删除原始数据。
```

## 七、常见误区

误区一：直接写正式输出文件。中途失败会留下半个归档文件，后续任务可能误读。

误区二：坏行直接 `panic`。真实批处理通常要记录坏行并继续处理可用数据，除非业务要求全量失败。

误区三：没有统计结果。运行成功但不知道处理了多少数据，排查很困难。

误区四：覆盖错误报告。多次运行后历史错误原因丢失，不利于定位问题。

误区五：把归档和删除原文件放在一个步骤里。生产中要先确认归档成功、备份可用，再考虑清理。

## 八、小练习

给归档工具增加一个 `--min-amount` 参数。

要求：

- 只归档 `amount >= min_amount` 的订单。
- 默认值为 `0`，表示不过滤金额。
- 统计输出里增加 `skipped` 数量。
- 思考：如果参数传错，例如传入负数，工具应该直接失败，还是按 0 处理？

- 能用临时文件和 `os.Rename` 降低半文件风险。
