# 10. 覆盖率与 CI：让测试进入日常交付

本节目标：学完后你能用覆盖率发现测试盲区，并设计一组适合后端项目的 CI 测试命令。

简短引入：测试只有进入日常交付才有持续价值。覆盖率可以帮助你发现哪些代码完全没被测试跑到，CI 可以保证团队每次提交都自动验证关键路径。但覆盖率不是业务质量本身，CI 也不能把慢测试一股脑塞进去。

## 一、为什么需要它

真实项目里，测试经常败在两个地方：

- 写过一次，但后续没人跑。
- 覆盖率看起来不错，但核心业务路径没测。

覆盖率和 CI 的作用是把测试变成习惯：

- 提交时知道有没有破坏现有规则。
- 合并前知道关键包是否通过。
- 通过报告发现明显测试空白。

```text
覆盖率是地图，不是成绩单；它提示哪里没走过，但不保证走过的地方没有问题。
```

## 二、基本用法

运行所有单元测试：

Windows PowerShell：

```powershell
go test ./...
```

Linux/macOS：

```bash
go test ./...
```

生成覆盖率文件：

Windows PowerShell：

```powershell
go test ./... -coverprofile=coverage.out
```

Linux/macOS：

```bash
go test ./... -coverprofile=coverage.out
```

在浏览器查看覆盖率 HTML：

Windows PowerShell：

```powershell
go tool cover -html=coverage.out
```

Linux/macOS：

```bash
go tool cover -html=coverage.out
```

只查看函数级覆盖率：

Windows PowerShell：

```powershell
go tool cover -func=coverage.out
```

Linux/macOS：

```bash
go tool cover -func=coverage.out
```

## 三、关键参数/语法/代码结构

`-coverprofile=coverage.out` 会把覆盖率结果写到文件。这个文件通常不提交到仓库。

`go tool cover -html` 适合本地查看哪些分支没被跑到。

`go tool cover -func` 适合 CI 输出文本报告。

`go test ./...` 适合默认单元测试入口。

集成测试建议单独命令：

Windows PowerShell：

```powershell
go test -tags=integration ./...
```

Linux/macOS：

```bash
go test -tags=integration ./...
```

常见项目还会加上 race 检测：

Windows PowerShell：

```powershell
go test -race ./...
```

Linux/macOS：

```bash
go test -race ./...
```

`-race` 用来发现数据竞争，适合有 goroutine、缓存、后台任务、并发 handler 的项目。它会让测试变慢，所以通常放在 CI 的专门步骤里，而不是每次本地都必须运行。

## 四、真实后端场景示例

一个保守的后端项目 CI 可以分三层：

第一层，每次提交都跑：

```bash
go test ./...
```

第二层，合并前或主分支跑覆盖率：

```bash
go test ./... -coverprofile=coverage.out
go tool cover -func=coverage.out
```

第三层，需要数据库、Redis、消息队列时单独跑：

```bash
go test -tags=integration ./...
```

如果项目已经有并发代码，可以增加第四层：

```bash
go test -race ./...
```

如果项目里有 fuzz 测试，不建议每次提交都长时间 fuzz。可以在 CI 中短跑，或者放到定时任务：

```bash
go test -run=^$ -fuzz=FuzzIsValidSlug -fuzztime=30s ./...
```

这条命令里的 `-run=^$` 表示不运行普通测试，只运行指定 fuzz。真实项目中可以针对输入解析、URL 校验、订单号解析这类函数设置少量 fuzz 任务。

如果使用 GitHub Actions，可以参考：

```yaml
name: Go Test

on:
  push:
    branches: [ main ]
  pull_request:

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-go@v5
        with:
          go-version: '1.26.x'
      - name: Run unit tests
        run: go test ./...
      - name: Generate coverage
        run: go test ./... -coverprofile=coverage.out
      - name: Print coverage
        run: go tool cover -func=coverage.out
```

这里的 Go 版本要和你的项目 `go.mod` 保持一致。真实项目中不要随意升级 CI Go 版本，先本地验证依赖和行为。

可以把 CI 的判断标准写得明确一些。例如：

- 单元测试必须全部通过。
- 覆盖率下降时需要人工确认原因。
- 涉及数据库迁移的 PR 必须跑集成测试。
- 涉及并发代码的 PR 必须跑 `-race`。
- 涉及输入解析的 PR 优先补 fuzz 或边界测试。

这些标准不需要一开始就很复杂，但要让团队知道“什么改动需要什么级别的验证”。

## 五、注意点

覆盖率不能替代断言。如果测试只是调用函数但不检查结果，覆盖率会变高，但 bug 仍然会漏。

不要盲目追求 100% 覆盖率。更重要的是关键业务路径：

- 订单金额和状态流转。
- 权限判断。
- 支付回调。
- 库存扣减。
- 数据迁移。
- 用户输入校验。

CI 命令要控制耗时。如果一次普通提交要等二三十分钟，团队会开始绕开流程。

覆盖率文件一般加入 `.gitignore`：

```text
coverage.out
```

覆盖率低时，优先补这些地方：

- 有金额、库存、权限、状态流转的 service。
- 对外暴露的 HTTP handler。
- 解析用户输入的函数。
- 数据库写入和事务边界。
- 曾经出过 bug 的代码路径。

不建议优先补这些地方：

- 只有字段定义的结构体。
- 简单转发且没有业务逻辑的胶水代码。
- 未来很快会删除的临时代码。
- 需要大量 mock 才能测到一点点收益的代码。

## 六、常见误区

- 把覆盖率当 KPI：数字高不代表核心路径安全。
- 测试没有断言：代码跑过了，但没有验证结果。
- CI 只跑当前包：跨包修改可能破坏其他模块。
- 所有测试都在每次提交跑：集成测试太慢时会拖垮反馈速度。
- 忽略失败日志：CI 失败时要先看第一个真正失败的测试，不要只重跑。
- 只看总覆盖率：某个核心包完全没测，总覆盖率也可能被其他简单代码抬高。

## 七、本节达标标准

- 能运行 `go test ./...`。
- 能生成并查看覆盖率报告。
- 能解释覆盖率的价值和局限。
- 能为单元测试和集成测试设计不同 CI 入口。
- 能判断哪些后端路径应该优先补测试。
