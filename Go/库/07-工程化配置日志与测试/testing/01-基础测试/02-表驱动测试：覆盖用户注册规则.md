# 2. 表驱动测试：覆盖用户注册规则

本节目标：学完后你能用表驱动测试覆盖多个业务分支，而不是为每种输入复制一大段测试代码。

简短引入：后端业务很少只有一个输入。用户注册要校验邮箱、密码、昵称、邀请码；订单创建要校验库存、价格、地址、优惠券。表驱动测试是 Go 项目里非常常见的写法，适合把这些分支整理成一张“用例表”。

## 一、为什么需要它

假设注册校验有 5 个规则。如果每个规则都写一个完整测试函数，你会复制大量准备代码。后面新增字段时，维护成本会很高。

表驱动测试可以理解为：

```text
把测试逻辑写一遍，把不同输入和期望结果放进表里。
```

它适合测试纯业务规则，例如：

- 用户注册参数校验。
- 订单金额计算。
- 权限角色判断。
- 短链接后缀合法性校验。
- 库存扣减前的条件判断。

## 二、基本用法

继续使用上一节的 `ValidateRegister`，可以把测试改成这样：

```go
package userdemo

import "testing"

func TestValidateRegister_Table(t *testing.T) {
	tests := []struct {
		name    string
		req     RegisterRequest
		wantErr bool
	}{
		{
			name: "valid request",
			req: RegisterRequest{
				Email:    "alice@example.com",
				Password: "12345678",
			},
			wantErr: false,
		},
		{
			name: "empty email",
			req: RegisterRequest{
				Email:    " ",
				Password: "12345678",
			},
			wantErr: true,
		},
		{
			name: "short password",
			req: RegisterRequest{
				Email:    "alice@example.com",
				Password: "123",
			},
			wantErr: true,
		},
	}

	for _, tt := range tests {
		err := ValidateRegister(tt.req)
		if (err != nil) != tt.wantErr {
			t.Fatalf("%s: error = %v, wantErr %v", tt.name, err, tt.wantErr)
		}
	}
}
```

运行：

```powershell
go test
```

Linux/macOS：

```bash
go test
```

## 三、关键参数/语法/代码结构

`tests := []struct { ... }{ ... }` 是一组匿名结构体切片。每一项就是一个测试用例。

`name` 是用例名称。即使暂时不用子测试，也建议写上，失败时能快速定位业务场景。

`req` 是输入。

`wantErr` 是期望结果。这里没有比较具体错误字符串，是因为很多场景下我们只关心“是否拒绝”。如果业务需要区分错误码，再进一步比较错误类型。

`(err != nil) != tt.wantErr` 的意思是：实际是否出错和期望是否出错不一致。

## 四、真实后端场景示例

现在增加一个订单金额计算函数。新建 `order.go`：

```go
package userdemo

import "errors"

type OrderItem struct {
	PriceCents int
	Quantity   int
}

func CalcOrderTotal(items []OrderItem) (int, error) {
	if len(items) == 0 {
		return 0, errors.New("items is empty")
	}

	total := 0
	for _, item := range items {
		if item.PriceCents <= 0 {
			return 0, errors.New("price must be positive")
		}
		if item.Quantity <= 0 {
			return 0, errors.New("quantity must be positive")
		}
		total += item.PriceCents * item.Quantity
	}
	return total, nil
}
```

新建 `order_test.go`：

```go
package userdemo

import "testing"

func TestCalcOrderTotal(t *testing.T) {
	tests := []struct {
		name    string
		items   []OrderItem
		want    int
		wantErr bool
	}{
		{
			name: "single item",
			items: []OrderItem{
				{PriceCents: 1999, Quantity: 2},
			},
			want:    3998,
			wantErr: false,
		},
		{
			name: "multiple items",
			items: []OrderItem{
				{PriceCents: 1000, Quantity: 1},
				{PriceCents: 500, Quantity: 3},
			},
			want:    2500,
			wantErr: false,
		},
		{
			name:    "empty cart",
			items:   nil,
			wantErr: true,
		},
		{
			name: "invalid quantity",
			items: []OrderItem{
				{PriceCents: 1000, Quantity: 0},
			},
			wantErr: true,
		},
	}

	for _, tt := range tests {
		got, err := CalcOrderTotal(tt.items)
		if (err != nil) != tt.wantErr {
			t.Fatalf("%s: error = %v, wantErr %v", tt.name, err, tt.wantErr)
		}
		if got != tt.want {
			t.Fatalf("%s: total = %d, want %d", tt.name, got, tt.want)
		}
	}
}
```

这个例子贴近真实订单服务。金额使用 `int` 分为单位，不用 `float64` 表示钱，是后端项目里的保守选择。

## 五、注意点

表里的字段不要过多。如果一个测试表有十几个字段，说明这个函数可能承担了太多职责，或者测试场景需要拆分。

不要为了追求“一个测试测完全部”而把成功、失败、权限、数据库行为都塞进一张表。表驱动适合相似结构的分支。

当失败信息不够清楚时，下一节会用 `t.Run` 把每一行变成独立子测试。

## 六、常见误区

- 表驱动但没有 `name` 字段：失败后只能靠输入猜是哪条用例。
- 只比较 `wantErr`：成功场景还要比较核心返回值，否则函数返回错金额也可能通过。
- 直接比较错误字符串：错误文案经常会调整，除非接口契约要求，否则优先比较是否出错或错误类型。
- 用 `float64` 测金额：金额计算容易出现精度问题，后端项目通常用分、厘或 decimal 类型。

## 七、本节达标标准

- 能用 `[]struct` 写测试用例表。
- 能覆盖至少一个成功场景和多个失败场景。
- 能写出清楚的失败信息。
- 能判断什么时候适合表驱动，什么时候应该拆成多个测试。
- 能解释为什么订单金额示例中不用 `float64`。
