# 3. 子测试与测试夹具：让失败定位更清楚

本节目标：学完后你能用 `t.Run`、`t.Helper`、`t.Cleanup`、`t.TempDir` 和 `t.Setenv` 管理测试结构与临时资源。

简短引入：随着测试用例变多，单个测试函数里一旦失败，你需要快速知道是哪条业务场景出了问题。子测试可以让每条用例都有独立名字，测试夹具可以让准备数据和清理资源更稳定。

## 一、为什么需要它

表驱动测试已经能减少重复代码，但失败时如果全部在一个循环里，定位仍然不够舒服。

子测试可以理解为：

```text
在一个大测试里，为每个业务场景创建一个有名字的小测试。
```

真实项目里常见用途：

- 注册校验的每条规则独立显示。
- 一个 handler 的不同 HTTP 方法分别测试。
- 一个 repository 的插入、重复插入、查询不存在分别测试。
- 同一套权限规则按角色拆开。

## 二、基本用法

把上一节订单金额测试改成子测试：

```go
package userdemo

import "testing"

func TestCalcOrderTotal_Subtests(t *testing.T) {
	tests := []struct {
		name    string
		items   []OrderItem
		want    int
		wantErr bool
	}{
		{
			name: "single item",
			items: []OrderItem{
				{PriceCents: 1999, Quantity: 2},
			},
			want: 3998,
		},
		{
			name:    "empty cart",
			items:   nil,
			wantErr: true,
		},
	}

	for _, tt := range tests {
		t.Run(tt.name, func(t *testing.T) {
			got, err := CalcOrderTotal(tt.items)
			if (err != nil) != tt.wantErr {
				t.Fatalf("error = %v, wantErr %v", err, tt.wantErr)
			}
			if got != tt.want {
				t.Fatalf("total = %d, want %d", got, tt.want)
			}
		})
	}
}
```

运行某一个子测试：

```powershell
go test -run "TestCalcOrderTotal_Subtests/empty_cart"
```

Linux/macOS：

```bash
go test -run 'TestCalcOrderTotal_Subtests/empty_cart'
```

注意：Go 会把子测试名字里的空格处理成下划线，所以命令里写 `empty_cart`。

## 三、关键参数/语法/代码结构

`t.Run(name, func(t *testing.T) { ... })` 会创建子测试。失败输出会带上父测试和子测试名称。

`t.Helper()` 用在测试辅助函数里。它能让失败行号指向调用辅助函数的位置，而不是辅助函数内部。

`t.Cleanup(fn)` 用于注册清理逻辑。测试结束后自动执行，适合关闭临时资源、删除测试数据、恢复状态。

`t.TempDir()` 创建临时目录，测试结束后自动删除。适合测试文件上传、日志落盘、迁移文件读取。

`t.Setenv(key, value)` 设置环境变量，并在测试结束后恢复。适合测试配置读取，不要手工 `os.Setenv` 后忘记恢复。

## 四、真实后端场景示例

假设项目里需要从环境变量读取 JWT 密钥。新建 `config.go`：

```go
package userdemo

import (
	"errors"
	"os"
)

func LoadJWTSecret() (string, error) {
	secret := os.Getenv("JWT_SECRET")
	if len(secret) < 16 {
		return "", errors.New("JWT_SECRET is too short")
	}
	return secret, nil
}
```

新建 `config_test.go`：

```go
package userdemo

import "testing"

func TestLoadJWTSecret(t *testing.T) {
	tests := []struct {
		name    string
		secret  string
		wantErr bool
	}{
		{name: "valid secret", secret: "1234567890abcdef"},
		{name: "too short", secret: "short", wantErr: true},
	}

	for _, tt := range tests {
		t.Run(tt.name, func(t *testing.T) {
			t.Setenv("JWT_SECRET", tt.secret)

			got, err := LoadJWTSecret()
			if (err != nil) != tt.wantErr {
				t.Fatalf("error = %v, wantErr %v", err, tt.wantErr)
			}
			if !tt.wantErr && got != tt.secret {
				t.Fatalf("secret = %q, want %q", got, tt.secret)
			}
		})
	}
}
```

这个例子里，`t.Setenv` 的价值是测试之间互不污染。真实项目中，配置测试如果不隔离，很容易出现“单独跑通过，整包跑失败”。

再看一个辅助函数：

```go
package userdemo

import "testing"

func mustTotal(t *testing.T, items []OrderItem) int {
	t.Helper()

	total, err := CalcOrderTotal(items)
	if err != nil {
		t.Fatalf("CalcOrderTotal() unexpected error: %v", err)
	}
	return total
}
```

以后测试里可以复用：

```go
func TestMustTotalExample(t *testing.T) {
	total := mustTotal(t, []OrderItem{{PriceCents: 100, Quantity: 2}})
	if total != 200 {
		t.Fatalf("total = %d, want 200", total)
	}
}
```

## 五、注意点

子测试不是越多越好。它适合表达同一类行为的多个场景。如果每个子测试准备数据完全不同，拆成多个测试函数反而更清楚。

谨慎使用 `t.Parallel()`。并行测试可以加速，但如果测试共享数据库、全局变量、环境变量、临时文件，就可能互相影响。初学阶段先把隔离做好，再考虑并行。

不要在子测试外复用会被修改的变量。如果每个子测试都会修改同一个切片、map 或结构体指针，测试顺序会影响结果。

## 六、常见误区

- 在循环里写子测试但忘记用清晰名称：失败输出仍然难以定位。
- 手动设置环境变量后不恢复：后面的测试会读到错误配置。
- 辅助函数不写 `t.Helper()`：失败行号指向辅助函数内部，排查效率低。
- 所有测试都加 `t.Parallel()`：数据库和全局状态没有隔离时会产生偶发失败。

## 七、本节达标标准

- 能用 `t.Run` 把表驱动用例变成子测试。
- 能用 `go test -run` 运行指定测试或子测试。
- 能用 `t.Setenv` 测配置读取。
- 能用 `t.Helper` 编写测试辅助函数。
- 能说明 `t.Parallel` 的使用风险。
