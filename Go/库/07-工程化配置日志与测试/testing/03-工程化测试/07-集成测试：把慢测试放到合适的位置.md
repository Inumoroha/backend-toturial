# 7. 集成测试：把慢测试放到合适的位置

本节目标：学完后你能把依赖真实数据库或外部服务的测试与日常单元测试分开运行。

简短引入：有些问题必须靠真实环境发现，例如数据库迁移是否正确、唯一索引是否生效、事务是否回滚。但这类测试通常更慢、更脆弱，所以不能和所有单元测试混在一起无差别运行。

## 一、为什么需要它

单元测试强调快和稳定。集成测试强调真实依赖。

真实后端项目常见做法是：

- 提交前默认跑单元测试。
- CI 每次提交跑单元测试。
- 集成测试在合并前、夜间任务或手动触发时运行。
- 涉及数据库迁移、核心 SQL、消息队列消费时补集成测试。

```text
慢测试不是不能写，而是要让它有清楚的运行入口和隔离边界。
```

## 二、基本用法

Go 可以用 build tags 隔离集成测试。文件顶部写：

```go
//go:build integration
```

这样普通 `go test ./...` 不会运行这个文件。只有带上 tag 才会运行。

运行普通单元测试：

```powershell
go test ./...
```

Linux/macOS：

```bash
go test ./...
```

运行集成测试：

```powershell
go test -tags=integration ./...
```

Linux/macOS：

```bash
go test -tags=integration ./...
```

## 三、关键参数/语法/代码结构

`//go:build integration` 必须放在文件最前面附近，通常第一行，然后空一行再写 `package`。

集成测试文件常命名为 `xxx_integration_test.go`，方便人一眼看出来。

环境变量适合放数据库地址，例如 `TEST_DATABASE_DSN`。不要把账号密码写死在测试文件里。

测试数据要可回滚。常见做法是每个测试开启事务，测试结束后 `Rollback`。如果测试必须提交事务，就要使用唯一的测试数据前缀，并在清理阶段删除。

一个比较稳妥的集成测试流程通常是：

1. 准备一个专用测试数据库。
2. 从空库执行正式迁移文件。
3. 每个测试自己插入需要的数据。
4. 测试结束后回滚事务或清理测试数据。
5. CI 中使用单独账号和单独库名，避免误连生产。

```text
集成测试可以慢一点，但不能不确定；可以依赖真实数据库，但不能依赖生产数据库。
```

## 四、真实后端场景示例

下面是一个结构示例。它需要你提供真实测试数据库地址，所以默认不会参与普通测试。

新建 `user_repo_integration_test.go`：

```go
//go:build integration

package repodemo

import (
	"context"
	"database/sql"
	"os"
	"testing"

	_ "github.com/lib/pq"
)

func TestUserRepo_FindByEmail_Integration(t *testing.T) {
	dsn := os.Getenv("TEST_DATABASE_DSN")
	if dsn == "" {
		t.Skip("TEST_DATABASE_DSN is empty")
	}

	db, err := sql.Open("postgres", dsn)
	if err != nil {
		t.Fatalf("open db: %v", err)
	}
	t.Cleanup(func() {
		_ = db.Close()
	})

	ctx := context.Background()
	tx, err := db.BeginTx(ctx, nil)
	if err != nil {
		t.Fatalf("begin tx: %v", err)
	}
	t.Cleanup(func() {
		_ = tx.Rollback()
	})

	_, err = tx.ExecContext(ctx, `INSERT INTO users(email) VALUES($1)`, "it-alice@example.com")
	if err != nil {
		t.Fatalf("insert user: %v", err)
	}

	var id int64
	err = tx.QueryRowContext(ctx, `SELECT id FROM users WHERE email = $1`, "it-alice@example.com").Scan(&id)
	if err != nil {
		t.Fatalf("select user: %v", err)
	}
	if id <= 0 {
		t.Fatalf("id = %d, want positive", id)
	}
}
```

如果使用 PostgreSQL，还需要安装驱动：

Windows PowerShell：

```powershell
go get github.com/lib/pq
```

Linux/macOS：

```bash
go get github.com/lib/pq
```

这个示例强调结构和边界。真实项目中你应该先运行迁移，再执行集成测试，不要让测试代码临时创建一套和生产不一致的表结构。

如果你的项目使用迁移工具，可以把集成测试命令拆成两步。下面用伪命令表示，因为不同团队可能使用 `goose`、`migrate`、`atlas` 或自研迁移工具：

Windows PowerShell：

```powershell
$env:TEST_DATABASE_DSN="postgres://user:password@localhost:5432/order_test?sslmode=disable"
go run ./cmd/migrate -dsn $env:TEST_DATABASE_DSN up
go test -tags=integration ./...
```

Linux/macOS：

```bash
export TEST_DATABASE_DSN='postgres://user:password@localhost:5432/order_test?sslmode=disable'
go run ./cmd/migrate -dsn "$TEST_DATABASE_DSN" up
go test -tags=integration ./...
```

这里的重点不是迁移工具本身，而是“测试库结构必须来自正式迁移”。这样你测到的才接近真实上线结构。

## 五、注意点

集成测试数据库不能是生产库。测试代码通常会插入、更新、删除数据，一旦连错库后果很严重。

迁移要可回滚或可重建。测试环境最好能通过脚本从空库跑到最新结构。

不要让集成测试依赖固定已有数据。它应该自己准备数据，自己清理数据。

事务回滚是常见清理方式，但也有边界：如果被测代码内部自己开启新连接或新事务，外层事务未必能包住所有变化。

如果集成测试要验证事务行为，可以专门设计失败场景。例如库存扣减成功后订单写入失败，最终库存不应该被永久扣掉。这样的测试数量不用多，但要覆盖最关键的数据一致性风险。

## 六、常见误区

- 把集成测试混进普通 `go test ./...`：项目一大，测试会慢到没人愿意跑。
- 使用生产数据库 DSN：这是严重风险。
- 测试依赖执行顺序：Go 测试不应该要求 A 先跑、B 后跑。
- 测试数据不清理：跑几次后唯一索引冲突，结果变得不稳定。
- 在测试里手写临时表结构：和真实迁移文件脱节后，测试会给人虚假的安全感。
- 用集成测试覆盖所有细节：它应该验证关键链路和真实依赖，不应该替代大量快速单元测试。

## 七、本节达标标准

- 能解释单元测试和集成测试的区别。
- 能用 build tags 隔离集成测试。
- 能通过环境变量读取测试数据库地址。
- 能设计可清理、可回滚的测试数据。
- 能说明为什么集成测试不能连接生产库。
