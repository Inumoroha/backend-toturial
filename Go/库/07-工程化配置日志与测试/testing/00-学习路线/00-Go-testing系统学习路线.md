# Go testing 系统学习路线

适用对象：已经会写基础 Go 代码，想成为 Go 后端工程师，并希望把测试真正用到业务项目里的人。

这条路线的目标不是让你背 `testing` 包的所有 API，而是让你知道：后端项目里哪些代码该测、怎么测、测到什么程度、哪些测试不应该随便放进日常流水线。

```text
测试不是为了证明代码永远没有 bug，而是为了让关键业务变化可以被低成本验证。
```

## 一、学习目标

学完后，你应该能够做到：

- 为 service、handler、repository 等后端分层编写可维护测试。
- 使用 `go test`、`testing.T`、表驱动测试、子测试、`httptest`、benchmark、fuzz、coverage 等常用能力。
- 区分单元测试、集成测试、端到端测试的边界。
- 在测试中隔离外部依赖，避免把数据库、Redis、第三方接口变成日常测试的不稳定因素。
- 在 CI 中运行合适的测试命令，并理解覆盖率的价值和局限。

## 二、推荐学习顺序

### 第 1 阶段：建立 Go 测试直觉

重点是明白 Go 的测试文件、测试函数、失败报告和命令如何配合。

建议学习：

1. `*_test.go` 文件放在哪里。
2. `func TestXxx(t *testing.T)` 的命名规则。
3. `t.Fatal`、`t.Errorf`、`t.Helper` 的实际用途。
4. `go test ./...` 在真实项目中的作用。

对应教程：

- [01-基础测试/01-认识Go测试与testing包.md](../01-基础测试/01-认识Go测试与testing包.md)

### 第 2 阶段：写出可维护的业务测试

重点是后端业务经常有很多输入分支，例如创建用户、计算订单金额、校验权限。一个分支写一个测试函数会很快失控，表驱动测试和子测试是 Go 项目里的常用写法。

建议学习：

1. 表驱动测试。
2. 子测试 `t.Run`。
3. 测试夹具、临时目录、环境变量。
4. 常见断言写法。

对应教程：

- [01-基础测试/02-表驱动测试：覆盖用户注册规则.md](../01-基础测试/02-表驱动测试：覆盖用户注册规则.md)
- [01-基础测试/03-子测试与测试夹具：让失败定位更清楚.md](../01-基础测试/03-子测试与测试夹具：让失败定位更清楚.md)

### 第 3 阶段：面向后端分层写测试

重点是把测试放在真实后端项目结构里，而不是只测试纯函数。常见场景是 service 依赖 repository，handler 依赖 service，代码还会调用外部支付、短信、库存服务。

建议学习：

1. 用 interface 隔离依赖。
2. 为 service 写单元测试。
3. 用 `httptest` 测 HTTP handler。
4. 测错误码、响应体、请求方法、鉴权上下文。

对应教程：

- [02-后端业务测试/04-用接口隔离依赖：测试订单Service.md](../02-后端业务测试/04-用接口隔离依赖：测试订单Service.md)
- [02-后端业务测试/05-使用httptest测试HTTP接口.md](../02-后端业务测试/05-使用httptest测试HTTP接口.md)

### 第 4 阶段：测试数据库和外部资源

重点是不要把数据库测试写成危险脚本。真实项目里，数据库测试要考虑数据隔离、事务回滚、迁移版本、索引成本和连接配置。

建议学习：

1. repository 测试应该测什么。
2. 参数化查询与 SQL 注入风险。
3. 集成测试与单元测试分开跑。
4. 使用 build tags 或环境变量控制慢测试。

### 第 5 阶段：性能、稳定性与质量门禁

重点是知道什么时候用 benchmark，什么时候用 fuzz，什么时候看 coverage，以及这些指标不能替代什么。

建议学习：

1. benchmark 的基本写法和误读风险。
2. fuzz 用来找输入边界问题。
3. coverage 用来发现测试盲区，不是 KPI。
4. CI 中的测试命令组合。

## 三、真实项目中的测试分层建议

一个常见 Go 后端项目可以按下面的比例思考：

- service 单元测试：最多，执行快，覆盖核心业务规则。
- handler 测试：适中，覆盖协议转换、状态码、JSON、鉴权入口。
- repository 集成测试：较少，覆盖关键 SQL、事务、唯一索引、迁移兼容。
- 端到端测试：最少，验证关键链路，不追求覆盖所有分支。

```text
越靠近业务规则，测试越应该多；越依赖外部环境，测试越应该少而关键。
```

## 四、每周学习安排

### 第 1 周：会写并会跑测试

- 完成第 1 到第 3 篇。
- 给至少 3 个业务函数补测试。
- 熟悉 `go test`、`go test ./...`、`go test -run`。

### 第 2 周：测试后端分层代码

- 完成第 4 到第 5 篇。
- 为一个 service 写 fake repository。
- 为一个 HTTP handler 写成功和失败测试。

### 第 3 周：数据库与集成测试

- 完成第 6 到第 7 篇。
- 为 repository 设计可回滚的测试数据。
- 把慢测试从日常单元测试中隔离出来。

### 第 4 周：质量工程化

- 完成第 8 到第 10 篇。
- 为一个热点函数写 benchmark。
- 为一个解析函数写 fuzz。
- 在项目里整理一组 CI 测试命令。

## 五、最终项目建议

建议做一个小型订单服务，功能不用复杂，但要有真实后端结构：

- 用户创建订单。
- 校验库存。
- 计算金额。
- 写入订单。
- 返回订单号。
- 查询订单详情。

测试目标：

- service 测业务规则。
- handler 测 HTTP 请求和响应。
- repository 测 SQL 和事务边界。
- benchmark 测订单号或短链接生成器。
- fuzz 测 ID、URL、JSON 字段解析。
- CI 跑单元测试和覆盖率，集成测试单独触发。

## 六、学习时的保守原则

- 不把用户输入直接拼进 SQL。
- 不在单元测试里访问真实生产服务。
- 不依赖测试执行顺序。
- 不让测试共享可变全局状态。
- 不用覆盖率数字掩盖关键路径没有测试的问题。
- 不在 benchmark 里混入日志、网络和随机外部资源。
- 不把数据库迁移当作“上线后手工修一下”的事情。
