# 11. 最终项目：订单服务测试落地

本节目标：学完后你能为一个小型订单服务设计测试分层，并知道每一层应该测什么。

简短引入：前面每节都讲了一个测试能力。真实后端项目不是把这些能力堆在一起，而是按代码职责分层使用。最终项目建议做一个小型订单服务，用它把 service、handler、repository、集成测试、覆盖率和 CI 串起来。

## 一、为什么需要它

订单服务是后端学习里很好的练习对象，因为它包含真实业务里常见的风险点：

- 用户输入校验。
- 库存是否足够。
- 金额计算是否正确。
- 数据写入是否成功。
- 失败时是否能回滚。
- HTTP 状态码是否符合约定。

可以理解为，最终项目不是为了做很多功能，而是为了练习“怎么把测试放到正确的位置”。

```text
最终项目的目标是可验证、可回滚、可维护，而不是功能堆得越多越好。
```

## 三、关键参数/语法/代码结构

订单 service 可以先抽象成三个依赖：

```go
package order

import "context"

type Product struct {
	SKU        string
	PriceCents int
	Stock      int
}

type Order struct {
	ID         int64
	UserID     int64
	SKU        string
	Quantity   int
	TotalCents int
	Status     string
}

type ProductRepo interface {
	FindBySKU(ctx context.Context, sku string) (Product, error)
}

type OrderRepo interface {
	Create(ctx context.Context, order Order) (int64, error)
}

type StockService interface {
	Reserve(ctx context.Context, sku string, quantity int) error
}
```

这里的重点不是接口越多越好，而是把外部依赖隔离出来。service 测试时用 fake，repository 集成测试时再连接真实数据库。

建议测试分层如下：

- `service_test.go`：测参数校验、库存不足、金额计算、创建订单失败。
- `handler_test.go`：测 JSON 解析、状态码、响应体、错误映射。
- `repo_test.go`：用 `sqlmock` 测 SQL 参数和字段映射。
- `repo_integration_test.go`：用 build tag 测真实迁移、唯一索引、事务回滚。

可以先按下面的顺序落地：

1. 先写 service 单元测试，保护业务规则。
2. 再写 handler 测试，保护 HTTP 契约。
3. 再写 repository 单元测试，保护 SQL 参数和映射。
4. 最后写少量集成测试，保护迁移、约束和事务。

不要反过来一开始就写端到端测试。端到端测试能证明链路能跑通，但定位问题慢，不适合承担所有细节验证。

## 四、真实后端场景示例

先实现一个最小 service。新建 `internal/order/service.go`：

```go
package order

import (
	"context"
	"errors"
)

type Service struct {
	products ProductRepo
	orders   OrderRepo
	stock    StockService
}

func NewService(products ProductRepo, orders OrderRepo, stock StockService) *Service {
	return &Service{products: products, orders: orders, stock: stock}
}

func (s *Service) Create(ctx context.Context, userID int64, sku string, quantity int) (int64, error) {
	if userID <= 0 {
		return 0, errors.New("user id is required")
	}
	if sku == "" {
		return 0, errors.New("sku is required")
	}
	if quantity <= 0 {
		return 0, errors.New("quantity must be positive")
	}

	product, err := s.products.FindBySKU(ctx, sku)
	if err != nil {
		return 0, err
	}
	if product.Stock < quantity {
		return 0, errors.New("stock is not enough")
	}
	if err := s.stock.Reserve(ctx, sku, quantity); err != nil {
		return 0, err
	}

	return s.orders.Create(ctx, Order{
		UserID:     userID,
		SKU:        sku,
		Quantity:   quantity,
		TotalCents: product.PriceCents * quantity,
		Status:     "created",
	})
}
```

再写 service 测试。新建 `internal/order/service_test.go`：

```go
package order

import (
	"context"
	"errors"
	"testing"
)

type fakeProductRepo struct {
	product Product
	err     error
}

func (f fakeProductRepo) FindBySKU(ctx context.Context, sku string) (Product, error) {
	return f.product, f.err
}

type fakeOrderRepo struct {
	order Order
	id    int64
	err   error
}

func (f *fakeOrderRepo) Create(ctx context.Context, order Order) (int64, error) {
	f.order = order
	return f.id, f.err
}

type fakeStockService struct {
	err error
}

func (f fakeStockService) Reserve(ctx context.Context, sku string, quantity int) error {
	return f.err
}

func TestService_Create(t *testing.T) {
	tests := []struct {
		name    string
		product Product
		stockErr error
		orderErr error
		wantID  int64
		wantErr bool
	}{
		{
			name:    "created",
			product: Product{SKU: "book-1", PriceCents: 2000, Stock: 5},
			wantID:  1001,
		},
		{
			name:    "stock not enough",
			product: Product{SKU: "book-1", PriceCents: 2000, Stock: 1},
			wantErr: true,
		},
		{
			name:     "reserve failed",
			product:  Product{SKU: "book-1", PriceCents: 2000, Stock: 5},
			stockErr: errors.New("reserve failed"),
			wantErr:  true,
		},
		{
			name:     "create order failed",
			product:  Product{SKU: "book-1", PriceCents: 2000, Stock: 5},
			orderErr: errors.New("db failed"),
			wantErr:  true,
		},
	}

	for _, tt := range tests {
		t.Run(tt.name, func(t *testing.T) {
			orderRepo := &fakeOrderRepo{id: tt.wantID, err: tt.orderErr}
			svc := NewService(
				fakeProductRepo{product: tt.product},
				orderRepo,
				fakeStockService{err: tt.stockErr},
			)

			gotID, err := svc.Create(context.Background(), 1, "book-1", 2)
			if (err != nil) != tt.wantErr {
				t.Fatalf("error = %v, wantErr %v", err, tt.wantErr)
			}
			if !tt.wantErr && gotID != tt.wantID {
				t.Fatalf("id = %d, want %d", gotID, tt.wantID)
			}
			if !tt.wantErr && orderRepo.order.TotalCents != 4000 {
				t.Fatalf("total = %d, want 4000", orderRepo.order.TotalCents)
			}
		})
	}
}
```

运行：

```powershell
go test ./...
```

Linux/macOS：

```bash
go test ./...
```

这个测试先把订单创建最关键的业务规则保护起来。后面再补 handler 和 repository，而不是一开始就把 HTTP、数据库、库存服务全部混进一个大测试。

继续落地时，可以为 handler 设计这些用例：

- `POST /orders` 请求 JSON 合法，返回 `201` 和订单 ID。
- JSON 格式错误，返回 `400`。
- `quantity <= 0`，返回 `400`。
- service 返回库存不足，返回 `409` 或业务约定状态码。
- service 返回未知错误，返回 `500`，但不要把底层数据库错误直接暴露给用户。

repository 层可以设计这些用例：

- 创建订单时使用参数化 SQL。
- 查询订单时字段正确映射到结构体。
- 查询不存在订单时把 `sql.ErrNoRows` 转成业务错误。
- 唯一订单号冲突时返回可识别错误。
- 事务失败时回滚。

## 五、注意点

库存扣减和订单创建涉及一致性。真实项目中通常要明确事务边界：如果库存和订单在同一个数据库里，可以放在一个事务中；如果库存是独立服务，就要考虑失败补偿、幂等和消息一致性。

测试里不要假装分布式事务不存在。初学项目可以先做简单版本，但文档和代码要清楚标注边界。

数据库迁移要用工具管理，不要上线后手工改表。测试环境应该能从迁移文件重建结构。

索引不是越多越好。订单查询可能需要 `user_id`、`created_at`，但每个索引都会增加写入成本。repository 集成测试可以验证查询字段和唯一约束，但性能还要结合真实数据量评估。

最终项目做到下面这个程度就比较合适：

- 核心 service 有比较完整的单元测试。
- handler 覆盖成功和典型失败响应。
- repository 至少覆盖关键 SQL 和不存在数据的情况。
- 集成测试只保留关键数据库行为，不追求数量。
- CI 默认快速反馈，慢测试有单独入口。

## 六、常见误区

- 一上来写端到端大测试：失败时不知道是 handler、service、repository 还是环境问题。
- service 测试连接真实数据库：业务规则测试变慢，也更难覆盖异常分支。
- 忽略失败分支：库存不足、扣减失败、订单写入失败都是订单服务的关键路径。
- 没有事务边界意识：订单创建成功但库存扣减失败，会造成数据不一致。
- 把覆盖率当最终目标：订单金额、库存、权限这些关键规则更重要。
- 只写教程里的示例，不迁移到自己的项目结构：学习 testing 的目的，是让真实项目改动更稳。

## 七、本节达标标准

- 能为订单服务设计测试分层。
- 能先用 fake 写 service 单元测试。
- 能说明 handler、service、repository 分别应该测什么。
- 能识别订单创建中的事务和一致性风险。
- 能把前面章节的测试能力组合到一个小项目里。
