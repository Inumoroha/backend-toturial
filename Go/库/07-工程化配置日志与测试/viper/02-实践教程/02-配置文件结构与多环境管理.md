# 02. 配置文件结构与多环境管理

本节目标：学完后你能为 Go 后端项目设计清晰的配置文件结构，并按开发、测试、生产环境加载不同配置。

## 简短引入

一个后端服务通常不止在本机运行。它可能有本地开发环境、CI 测试环境、预发环境和生产环境。不同环境的数据库地址、日志级别、迁移开关都可能不同。

Viper 不替你决定项目应该怎么分环境，但它可以帮助你按文件名或路径加载不同环境的配置。

## 一、为什么需要它

以文章服务为例，本地开发可以连接本机 PostgreSQL，测试环境连接测试库，生产环境连接生产库。如果只有一个 `config.yaml`，很容易出现两个问题：

- 本地改配置时误改了生产配置。
- 部署时不知道当前服务实际用了哪套配置。

常见做法是按环境拆文件：

```text
configs/
├── config.dev.yaml
├── config.test.yaml
├── config.prod.yaml
└── config.local.yaml
```

```text
生产配置和开发配置必须有清晰边界，不能靠“记得别改错”来保证安全。
```

## 二、基本用法

创建目录：

Windows PowerShell：

```powershell
mkdir viper-demo-02
cd viper-demo-02
go mod init viper-demo-02
go get github.com/spf13/viper
mkdir configs
```

Linux/macOS：

```bash
mkdir viper-demo-02
cd viper-demo-02
go mod init viper-demo-02
go get github.com/spf13/viper
mkdir -p configs
```

创建 `configs/config.dev.yaml`：

```yaml
app:
  name: article-service
  env: dev

server:
  addr: ":8080"

database:
  dsn: "postgres://dev:dev@localhost:5432/article_dev?sslmode=disable"
```

创建 `configs/config.prod.yaml`：

```yaml
app:
  name: article-service
  env: prod

server:
  addr: ":8080"

database:
  dsn: ""
```

真实项目中还经常准备一个本地覆盖文件，例如 `configs/config.local.yaml`。这个文件只给开发者本机使用，不提交到仓库。

可以创建 `.gitignore`：

```gitignore
configs/config.local.yaml
*.secret.yaml
```

这样团队可以提交 `config.dev.yaml`、`config.test.yaml` 和不含密钥的 `config.prod.yaml`，但不提交个人本地密码和临时调试配置。

创建 `main.go`：

```go
package main

import (
	"flag"
	"log"

	"github.com/spf13/viper"
)

func main() {
	env := flag.String("env", "dev", "runtime environment: dev, test, prod")
	flag.Parse()

	viper.SetConfigName("config." + *env)
	viper.SetConfigType("yaml")
	viper.AddConfigPath("./configs")

	if err := viper.ReadInConfig(); err != nil {
		log.Fatalf("read config failed: %v", err)
	}

	log.Printf("app=%s env=%s addr=%s",
		viper.GetString("app.name"),
		viper.GetString("app.env"),
		viper.GetString("server.addr"),
	)
}
```

运行开发环境：

```powershell
go run . -env dev
```

Linux/macOS 同样使用：

```bash
go run . -env dev
```

运行生产环境配置：

```powershell
go run . -env prod
```

## 三、关键参数/语法/代码结构

`flag.String("env", "dev", "...")` 用来从命令行接收环境名。这里默认是 `dev`，方便本地开发。

`SetConfigName("config." + *env)` 会根据 `env` 拼出配置名，例如：

```text
config.dev
config.test
config.prod
```

`AddConfigPath("./configs")` 将配置目录固定为 `configs`。这比散落在项目根目录更清楚。

这里的生产 `database.dsn` 故意为空，因为生产密码不应该写进仓库。后续会用环境变量覆盖它。

`.gitignore` 的作用不是“隐藏配置”，而是防止本地临时配置和密钥误提交。真实项目中更推荐提供一个 `config.example.yaml`，告诉新人需要哪些字段；每个人再复制成本地文件填写自己的值。

## 四、真实后端场景示例

真实项目里常见启动命令：

Windows PowerShell：

```powershell
go run . -env dev
```

Linux/macOS：

```bash
go run . -env dev
```

容器中可能是：

```bash
./article-service -env prod
```

CI 测试可能是：

```bash
go test ./... -count=1
```

测试代码中通常不直接使用 `config.dev.yaml`，而是创建专门的测试配置，避免本地开发配置污染自动化测试。

## 五、注意点

环境名不要太多。常见保守选择是 `dev`、`test`、`prod`。如果每个开发者都有一套配置，建议用本地未提交文件或环境变量，不要创建大量 `config.zhangsan.yaml`。

生产配置文件中可以保留结构和非敏感默认值，但敏感值应该留空，并在启动时校验。

如果你使用 Docker 或 Kubernetes，配置文件可以作为镜像内默认值，敏感配置由环境变量或 Secret 注入。

本地覆盖文件要谨慎使用。它适合个人开发，不适合成为团队约定的唯一配置来源。团队共享配置仍然应该放在可审查的示例文件或环境配置中。

## 六、常见误区

误区一：把生产数据库密码写进 `config.prod.yaml` 并提交。

这会让密钥进入 Git 历史，即使后面删除也不安全。真实项目中应立即轮换密码。

误区二：让代码根据 `app.env` 决定读取哪个文件。

如果还没读取配置，就不能依赖配置里的 `app.env`。环境选择应该来自命令行参数、环境变量或部署系统。

误区三：开发、测试、生产配置结构不一致。

结构不一致会导致某个环境启动时才暴露问题。建议所有环境配置保留同样的层级，只是值不同。

误区四：把 `.gitignore` 当成密钥管理方案。

`.gitignore` 只能降低误提交概率，不能替代密钥管理。已经提交过的密钥要立即作废并轮换。

## 七、本节达标标准

- 能按环境拆分配置文件。
- 能用 `-env` 参数选择加载哪套配置。
- 知道生产敏感值不应提交到仓库。
- 能解释配置结构为什么要在多环境中保持一致。
- 能说出配置文件、命令行参数、环境变量各自适合承担什么职责。
- 能使用 `.gitignore` 隔离本地临时配置，并知道它不能替代密钥管理。
