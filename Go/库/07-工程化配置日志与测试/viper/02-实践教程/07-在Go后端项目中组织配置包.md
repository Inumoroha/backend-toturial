# 07. 在 Go 后端项目中组织配置包

本节目标：学完后你能把 Viper 封装进项目内部配置包，让业务代码使用清晰的配置结构体。

## 简短引入

真实项目不会把所有配置代码都写在 `main.go`。更常见的做法是创建 `internal/config` 包，专门负责读取、绑定和校验配置。

这样 `main.go` 只负责组织启动流程，业务组件只接收自己需要的配置。

## 一、为什么需要它

以短链接服务为例，它可能需要：

- HTTP 配置。
- PostgreSQL 配置。
- Redis 配置。
- 日志配置。
- JWT 配置。
- 限流配置。

如果所有地方都直接调用 Viper，后续维护会很痛苦。配置包的作用是把外部配置转换成项目内部稳定结构。

```text
Viper 应该停留在配置加载层，业务层不应该到处感知 Viper。
```

## 二、基本用法

创建项目结构：

Windows PowerShell：

```powershell
mkdir viper-demo-07
cd viper-demo-07
go mod init viper-demo-07
go get github.com/spf13/viper
mkdir internal
mkdir internal\config
mkdir configs
```

Linux/macOS：

```bash
mkdir viper-demo-07
cd viper-demo-07
go mod init viper-demo-07
go get github.com/spf13/viper
mkdir -p internal/config configs
```

创建 `configs/config.dev.yaml`：

```yaml
app:
  name: shortlink-service
  env: dev

server:
  addr: ":8080"

database:
  dsn: "postgres://dev:dev@localhost:5432/shortlink_dev?sslmode=disable"
```

创建 `internal/config/config.go`：

```go
package config

import (
	"errors"

	"github.com/spf13/viper"
)

type Config struct {
	App      App      `mapstructure:"app"`
	Server   Server   `mapstructure:"server"`
	Database Database `mapstructure:"database"`
}

type App struct {
	Name string `mapstructure:"name"`
	Env  string `mapstructure:"env"`
}

type Server struct {
	Addr string `mapstructure:"addr"`
}

type Database struct {
	DSN string `mapstructure:"dsn"`
}

func Load(env string, paths ...string) (*Config, error) {
	v := viper.New()
	v.SetConfigName("config." + env)
	v.SetConfigType("yaml")

	for _, path := range paths {
		v.AddConfigPath(path)
	}

	if err := v.ReadInConfig(); err != nil {
		return nil, err
	}

	var cfg Config
	if err := v.Unmarshal(&cfg); err != nil {
		return nil, err
	}

	if err := cfg.Validate(); err != nil {
		return nil, err
	}

	return &cfg, nil
}

func (c Config) Validate() error {
	if c.App.Name == "" {
		return errors.New("app.name is required")
	}
	if c.App.Env == "" {
		return errors.New("app.env is required")
	}
	if c.Server.Addr == "" {
		return errors.New("server.addr is required")
	}
	if c.Database.DSN == "" {
		return errors.New("database.dsn is required")
	}
	return nil
}
```

创建 `main.go`：

```go
package main

import (
	"flag"
	"log"

	"viper-demo-07/internal/config"
)

func main() {
	env := flag.String("env", "dev", "runtime environment")
	flag.Parse()

	cfg, err := config.Load(*env, "./configs")
	if err != nil {
		log.Fatalf("load config failed: %v", err)
	}

	log.Printf("start %s env=%s addr=%s",
		cfg.App.Name,
		cfg.App.Env,
		cfg.Server.Addr,
	)
}
```

运行：

```powershell
go run . -env dev
```

Linux/macOS：

```bash
go run . -env dev
```

## 三、关键参数/语法/代码结构

`internal/config` 表示配置包只给当前项目使用，不暴露给外部模块。

`Load(env string, paths ...string)` 接收环境名和配置目录，方便本地、测试、部署传入不同路径。

`main.go` 不再知道 Viper 的细节，只调用 `config.Load`。

业务组件可以接收局部配置：

```go
func NewRepository(cfg config.Database) {
	// 初始化数据库仓储。
}
```

## 四、真实后端场景示例

短链接服务启动可以整理成：

```text
config.Load -> init logger -> open database -> init redis -> build router -> start server
```

配置包只负责第一步。不要把数据库连接、Redis 连接也塞进配置包，否则职责会混乱。

配置包的输出应该稳定，例如：

```go
type Config struct {
	Server   Server
	Database Database
}
```

后续你可以把底层从 Viper 换成别的工具，但业务代码不需要大改。

## 五、注意点

不要在 `init()` 函数里读取配置。`init()` 执行时机隐式，测试和排查都不方便。

不要在配置包里直接启动服务。配置包只负责加载和校验配置。

配置结构要按组件划分，而不是按“谁写的代码”划分。

## 六、常见误区

误区一：在全局变量里保存配置并到处读取。

这会让依赖关系变隐式。小项目可以暂时这么做，但学习工程化时应尽量显式传参。

误区二：配置包承担太多职责。

配置包不应该初始化数据库连接，也不应该注册路由。它只负责产生配置结构体。

误区三：业务包直接 import Viper。

这样以后想测试业务逻辑时，要准备完整配置环境，成本变高。

## 七、本节达标标准

- 能创建 `internal/config` 包。
- 能把 Viper 读取逻辑封装到 `Load` 函数。
- 能让 `main.go` 只依赖配置包。
- 能让业务组件接收局部配置结构。
- 知道配置包和组件初始化之间的职责边界。
