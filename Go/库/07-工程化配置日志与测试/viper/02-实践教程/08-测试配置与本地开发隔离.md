# 08. 测试配置与本地开发隔离

本节目标：学完后你能为配置加载逻辑编写测试，并避免测试依赖本机真实配置。

## 简短引入

配置读取看起来简单，但它是服务启动链路的第一步。配置读错，后面所有组件都会受影响。因此配置加载和校验也应该有测试。

重点不是测试 Viper 本身，而是测试你的配置结构、默认值、必填项和路径策略。

## 一、为什么需要它

假设你把 `database.dsn` 改名为 `database.url`，但代码结构体仍然使用 `dsn`。如果没有测试，可能直到服务部署时才发现启动失败。

配置测试能帮你提前发现：

- 配置文件结构变化。
- 必填项缺失。
- 默认值没有生效。
- 环境变量污染测试。

```text
测试配置加载逻辑，是为了保护服务启动流程，而不是为了证明 Viper 能工作。
```

## 二、基本用法

创建项目：

Windows PowerShell：

```powershell
mkdir viper-demo-08
cd viper-demo-08
go mod init viper-demo-08
go get github.com/spf13/viper
mkdir config
```

Linux/macOS：

```bash
mkdir viper-demo-08
cd viper-demo-08
go mod init viper-demo-08
go get github.com/spf13/viper
mkdir -p config
```

创建 `config/config.go`：

```go
package config

import (
	"errors"

	"github.com/spf13/viper"
)

type Config struct {
	Server Server `mapstructure:"server"`
	Log    Log    `mapstructure:"log"`
}

type Server struct {
	Addr string `mapstructure:"addr"`
}

type Log struct {
	Level string `mapstructure:"level"`
}

func Load(path string) (*Config, error) {
	v := viper.New()
	v.SetDefault("log.level", "info")
	v.SetConfigName("config")
	v.SetConfigType("yaml")
	v.AddConfigPath(path)

	if err := v.ReadInConfig(); err != nil {
		return nil, err
	}

	var cfg Config
	if err := v.Unmarshal(&cfg); err != nil {
		return nil, err
	}

	if cfg.Server.Addr == "" {
		return nil, errors.New("server.addr is required")
	}

	return &cfg, nil
}
```

创建 `config/config_test.go`：

```go
package config

import (
	"os"
	"path/filepath"
	"testing"
)

func TestLoadConfig(t *testing.T) {
	dir := t.TempDir()
	content := []byte(`
server:
  addr: ":8080"
`)
	if err := os.WriteFile(filepath.Join(dir, "config.yaml"), content, 0600); err != nil {
		t.Fatal(err)
	}

	cfg, err := Load(dir)
	if err != nil {
		t.Fatal(err)
	}

	if cfg.Server.Addr != ":8080" {
		t.Fatalf("server addr = %q", cfg.Server.Addr)
	}
	if cfg.Log.Level != "info" {
		t.Fatalf("log level = %q", cfg.Log.Level)
	}
}

func TestLoadConfigRequiresServerAddr(t *testing.T) {
	dir := t.TempDir()
	content := []byte(`
log:
  level: debug
`)
	if err := os.WriteFile(filepath.Join(dir, "config.yaml"), content, 0600); err != nil {
		t.Fatal(err)
	}

	_, err := Load(dir)
	if err == nil {
		t.Fatal("expected error")
	}
}
```

如果配置支持环境变量覆盖，可以用 `t.Setenv` 写一个专门测试。不要在测试中依赖你电脑上已经存在的环境变量。

```go
func TestLoadConfigWithEnv(t *testing.T) {
	t.Setenv("APP_SERVER_ADDR", ":18080")

	// 这里的前提是 Load 内部设置了：
	// v.SetEnvPrefix("APP")
	// v.SetEnvKeyReplacer(strings.NewReplacer(".", "_"))
	// v.BindEnv("server.addr")
	// v.AutomaticEnv()
}
```

```powershell

```

Linux/macOS：

```bash

```

## 三、关键参数/语法/代码结构

`t.TempDir()` 为每个测试创建临时目录。测试结束后 Go 会自动清理。

`os.WriteFile` 在测试中写入临时配置文件。这里是测试代码，使用它创建输入文件是合理的。

每次 `Load` 内部都用 `viper.New()` 创建新实例，这能避免不同测试之间共享 Viper 状态。

测试不依赖项目根目录的 `config.yaml`，所以在 CI 和本机都更稳定。

`t.Setenv` 会在测试结束后自动恢复环境变量，适合测试环境变量覆盖。不要手动 `os.Setenv` 后忘记恢复，否则后面的测试可能被污染。

## 四、真实后端场景示例

真实项目中可以为这些场景写测试：

- 最小配置能启动。
- 缺少数据库 DSN 会失败。
- 日志级别缺失时默认 `info`。
- 生产环境下 `jwt.secret` 不能为空。
- `server.read_timeout` 能正确解析成时间。

比如生产环境校验可以这样写：

```go
if cfg.App.Env == "prod" && cfg.JWT.Secret == "" {
	return errors.New("jwt.secret is required in prod")
}
```

这类规则很适合用测试固定下来。

## 五、注意点

测试不要读取真实生产配置。生产配置可能不存在，也可能包含敏感内容。

测试不要依赖开发者本机环境变量。需要环境变量时，使用 `t.Setenv`，并且每个测试自己设置。

不要只测成功路径。配置系统最重要的价值之一就是在配置错误时阻止服务启动。

如果你的配置加载逻辑支持环境变量，至少要有一个测试覆盖环境变量优先于配置文件的场景。这样以后重构配置包时，不容易把生产注入能力改坏。

## 六、常见误区

误区一：配置加载没有测试。

服务启动链路的第一步没有测试，后面组件测试再多也可能被配置错误拖垮。

误区二：测试直接读取 `configs/config.dev.yaml`。

开发配置会经常变化，容易让测试变得不稳定。测试应该创建自己的最小配置。

误区三：测试使用全局 Viper。

全局状态会在测试之间互相影响。配置测试应优先使用独立实例。

## 七、本节达标标准

- 能为配置加载函数写单元测试。
- 能使用 `t.TempDir()` 创建隔离测试配置。
- 能测试默认值是否生效。
- 能测试必填配置缺失时是否报错。
- 知道测试不应该依赖本机真实配置和生产配置。
- 能使用 `t.Setenv` 测试环境变量覆盖，并避免污染其他测试。
