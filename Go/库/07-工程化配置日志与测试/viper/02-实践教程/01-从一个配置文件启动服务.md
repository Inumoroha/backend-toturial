# 01. 从一个配置文件启动服务

本节目标：学完后你能用 Viper 读取一个 YAML 配置文件，并用配置启动一个最小 Go 后端服务。

## 简短引入

真实后端项目不会把端口、数据库地址、日志级别写死在代码里。开发环境、测试环境、生产环境的配置不同，如果都写在代码中，每次发布都要改代码，风险很高。

Viper 可以理解为 Go 项目中的配置读取工具。它负责从配置文件、环境变量、命令行参数等来源读取值，然后交给你的服务使用。

## 一、为什么需要它

假设你正在写一个用户服务。开发时监听 `8080`，测试时监听 `18080`，生产时可能监听容器内部的 `8080`。如果端口写死，服务部署时就会很僵硬。

更常见的是这些配置：

- HTTP 服务端口。
- 数据库连接地址。
- Redis 地址。
- 日志级别。
- JWT 密钥。
- 是否开启迁移。

```text
配置应该描述运行环境，代码应该描述业务行为。
```

## 二、基本用法

新建一个练习目录：

Windows PowerShell：

```powershell
mkdir viper-demo-01
cd viper-demo-01
go mod init viper-demo-01
go get github.com/spf13/viper
```

Linux/macOS：

```bash
mkdir viper-demo-01
cd viper-demo-01
go mod init viper-demo-01
go get github.com/spf13/viper
```

创建 `config.yaml`：

```yaml
app:
  name: user-service
  env: dev

server:
  addr: ":8080"
```

创建 `main.go`：

```go
package main

import (
	"fmt"
	"log"
	"net/http"

	"github.com/spf13/viper"
)

func main() {
	viper.SetConfigName("config")
	viper.SetConfigType("yaml")
	viper.AddConfigPath(".")

	if err := viper.ReadInConfig(); err != nil {
		log.Fatalf("read config failed: %v", err)
	}

	appName := viper.GetString("app.name")
	addr := viper.GetString("server.addr")

	http.HandleFunc("/health", func(w http.ResponseWriter, r *http.Request) {
		_, _ = fmt.Fprintf(w, "%s ok", appName)
	})

	log.Printf("%s starting at %s", appName, addr)
	if err := http.ListenAndServe(addr, nil); err != nil {
		log.Fatalf("server stopped: %v", err)
	}
}
```

运行：

Windows PowerShell：

```powershell
go run .
```

Linux/macOS：

```bash
go run .
```

访问：

```text
http://localhost:8080/health
```

## 三、关键参数/语法/代码结构

`SetConfigName("config")` 表示配置文件名是 `config`，不包含扩展名。

`SetConfigType("yaml")` 表示配置格式是 YAML。真实项目中常见的是 YAML，因为层级结构清晰。

`AddConfigPath(".")` 表示从当前目录查找配置。项目中也常见 `./configs`。

`ReadInConfig()` 是真正读取配置的动作。这个错误不能忽略，因为配置读取失败通常意味着服务不能正确启动。

`GetString("server.addr")` 使用点号读取嵌套字段，可以理解为读取 YAML 中的：

```yaml
server:
  addr: ":8080"
```

## 四、真实后端场景示例

用户服务启动时通常会先加载配置，再初始化组件：

```text
读取配置 -> 初始化日志 -> 连接数据库 -> 初始化路由 -> 启动 HTTP 服务
```

这个顺序很重要。比如数据库地址来自配置，如果配置没读到，就不应该继续初始化数据库。否则程序可能在后面报出更难定位的错误。

在真实项目中，读取配置失败一般直接退出：

```go
if err := viper.ReadInConfig(); err != nil {
	log.Fatalf("read config failed: %v", err)
}
```

这不是“不够优雅”，而是启动阶段的合理失败。后端服务启动时发现缺少关键配置，继续运行反而更危险。

## 五、注意点

配置路径要明确。开发时用当前目录很方便，但部署时工作目录可能变化。生产项目中通常会通过命令行参数或环境变量指定配置路径。

配置文件不要放敏感生产密码。可以在本地配置里放示例值，但生产密码更适合通过环境变量或密钥管理平台注入。

配置读取只应该出现在启动流程。不要在每个 handler 里反复调用 `ReadInConfig()`，这会让请求逻辑变慢，也让行为不可控。

## 六、常见误区

误区一：把配置读取错误打印一下就继续启动。

如果端口、数据库地址、JWT 密钥没有读到，服务继续运行只会制造更隐蔽的问题。启动阶段应该尽早失败。

误区二：业务代码到处直接调用 `viper.GetString`。

初学阶段可以这么写，但真实项目中应该逐步把配置绑定到结构体，再传给需要的组件。

误区三：把所有配置都写进一个很大的平铺文件。

平铺配置短期简单，长期难维护。建议按组件分组，例如 `server`、`database`、`redis`、`log`。

## 七、本节达标标准

- 能创建一个 `config.yaml`。
- 能用 Viper 读取 YAML 配置。
- 能用配置中的端口启动 HTTP 服务。
- 知道配置读取失败时为什么应该让服务启动失败。
- 知道配置读取应该放在服务启动流程中。
