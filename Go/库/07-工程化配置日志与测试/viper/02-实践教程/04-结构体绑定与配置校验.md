# 04. 结构体绑定与配置校验

本节目标：学完后你能把 Viper 配置绑定到 Go 结构体，并在启动阶段完成基础校验。

## 简短引入

前几节直接使用 `viper.GetString("database.dsn")`。这适合演示，但真实项目中如果到处写字符串 key，很容易拼错，也不利于测试。

更稳妥的做法是把配置绑定到结构体，业务代码只依赖强类型配置。

## 一、为什么需要它

假设库存服务要用数据库和 Redis。如果每个初始化函数都自己读 Viper：

```go
dsn := viper.GetString("database.dsn")
redisAddr := viper.GetString("redis.addr")
```

问题是：

- key 拼错时编译器发现不了。
- 函数隐藏依赖全局 Viper，测试困难。
- 配置结构变化时影响范围不清晰。

结构体配置更像后端项目中的启动合同。

```text
业务代码应该依赖明确的配置结构体，而不是依赖全局配置读取器。
```

## 二、基本用法

创建项目：

Windows PowerShell：

```powershell
mkdir viper-demo-04
cd viper-demo-04
go mod init viper-demo-04
go get github.com/spf13/viper
```

Linux/macOS：

```bash
mkdir viper-demo-04
cd viper-demo-04
go mod init viper-demo-04
go get github.com/spf13/viper
```

创建 `config.yaml`：

```yaml
app:
  name: inventory-service
  env: dev

server:
  addr: ":8080"

database:
  dsn: "postgres://dev:dev@localhost:5432/inventory_dev?sslmode=disable"
  max_open_conns: 20

redis:
  addr: "localhost:6379"
  db: 0
```

创建 `main.go`：

```go
package main

import (
	"errors"
	"log"

	"github.com/spf13/viper"
)

type Config struct {
	App      AppConfig      `mapstructure:"app"`
	Server   ServerConfig   `mapstructure:"server"`
	Database DatabaseConfig `mapstructure:"database"`
	Redis    RedisConfig    `mapstructure:"redis"`
}

type AppConfig struct {
	Name string `mapstructure:"name"`
	Env  string `mapstructure:"env"`
}

type ServerConfig struct {
	Addr string `mapstructure:"addr"`
}

type DatabaseConfig struct {
	DSN          string `mapstructure:"dsn"`
	MaxOpenConns int    `mapstructure:"max_open_conns"`
}

type RedisConfig struct {
	Addr string `mapstructure:"addr"`
	DB   int    `mapstructure:"db"`
}

func main() {
	cfg, err := LoadConfig(".")
	if err != nil {
		log.Fatal(err)
	}

	log.Printf("app=%s env=%s server=%s db_conns=%d redis=%s/%d",
		cfg.App.Name,
		cfg.App.Env,
		cfg.Server.Addr,
		cfg.Database.MaxOpenConns,
		cfg.Redis.Addr,
		cfg.Redis.DB,
	)
}

func LoadConfig(path string) (*Config, error) {
	v := viper.New()
	v.SetDefault("database.max_open_conns", 10)
	v.SetDefault("redis.db", 0)

	v.SetConfigName("config")
	v.SetConfigType("yaml")
	v.AddConfigPath(path)

	if err := v.ReadInConfig(); err != nil {
		return nil, err
	}

	var cfg Config
	if err := v.Unmarshal(&cfg); err != nil {
		return nil, err
	}

	if err := cfg.Validate(); err != nil {
		return nil, err
	}

	return &cfg, nil
}

func (c Config) Validate() error {
	if c.App.Name == "" {
		return errors.New("app.name is required")
	}
	if c.Server.Addr == "" {
		return errors.New("server.addr is required")
	}
	if c.Database.DSN == "" {
		return errors.New("database.dsn is required")
	}
	if c.Redis.Addr == "" {
		return errors.New("redis.addr is required")
	}
	return nil
}
```

运行：

```powershell
go run .
```

Linux/macOS：

```bash
go run .
```

## 三、关键参数/语法/代码结构

`viper.New()` 创建独立实例。真实项目中建议这么做，避免全局 Viper 被测试或其他包污染。

`mapstructure:"database"` 告诉 Viper 如何把 YAML 字段映射到结构体字段。

`Unmarshal(&cfg)` 把配置整体绑定到结构体。绑定后，后续代码使用 `cfg.Database.DSN`，不再写字符串 key。

`Validate()` 是配置校验。它不需要一开始就很复杂，但关键配置必须检查。

## 四、真实后端场景示例

初始化数据库时，可以这样传参：

```go
func OpenDB(cfg DatabaseConfig) error {
	if cfg.DSN == "" {
		return errors.New("database dsn is empty")
	}
	// 这里可以继续初始化 sql.DB 或 GORM。
	return nil
}
```

这样 `OpenDB` 不依赖 Viper，也不需要知道配置来自 YAML、环境变量还是命令行参数。它只关心数据库配置结构。

这会让单元测试更简单：

```go
err := OpenDB(DatabaseConfig{
	DSN:          "postgres://test:test@localhost:5432/test?sslmode=disable",
	MaxOpenConns: 5,
})
```

## 五、注意点

结构体字段要使用导出字段，也就是首字母大写。否则 `Unmarshal` 无法写入。

YAML 字段和结构体标签要保持一致。`max_open_conns` 对应 `mapstructure:"max_open_conns"`。

校验不要拖到业务请求阶段。配置错误应该在启动时暴露，而不是用户下单时才发现数据库地址为空。

## 六、常见误区

误区一：把 `*viper.Viper` 传到所有业务函数里。

这会让业务代码和配置工具强耦合。更好的方式是传递结构体配置。

误区二：只绑定结构体，不做校验。

绑定成功只代表类型转换大致完成，不代表配置值合理。例如空 DSN 也可以绑定成功。

误区三：把所有配置都放进一个超大的 `Config` 后到处传。

更好的做法是组件需要什么就传什么，例如数据库初始化只接收 `DatabaseConfig`。

## 七、本节达标标准

- 能定义与 YAML 对应的配置结构体。
- 能使用 `Unmarshal` 绑定配置。
- 能使用 `viper.New()` 创建独立配置实例。
- 能为关键配置编写基础校验。
- 知道业务代码应该依赖配置结构体，而不是直接依赖 Viper。
