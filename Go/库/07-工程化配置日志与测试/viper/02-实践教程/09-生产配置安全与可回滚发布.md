# 09. 生产配置安全与可回滚发布

本节目标：学完后你能从生产安全角度设计 Viper 配置，避免密钥泄露、危险默认值和不可回滚变更。

## 简短引入

配置系统进入生产后，重点不再只是“能不能读到值”，而是“这个值是否安全、是否可审计、出问题能不能回滚”。

Viper 是工具，生产安全靠的是配置分层、校验、发布流程和保守默认值。

## 一、为什么需要它

真实事故常常不是代码算法写错，而是配置变更引发：

- 连接到了错误数据库。
- 功能开关默认开启，导致未灰度功能直接上线。
- JWT 密钥为空，认证逻辑失效。
- 自动迁移在生产执行了破坏性变更。
- 日志打印了数据库密码。

```text
生产配置的第一原则是保守：不确定就不启动，不确认就不开启，不可回滚就不变更。
```

## 二、基本用法

下面示例展示生产环境下的保守校验。

创建项目：

Windows PowerShell：

```powershell
mkdir viper-demo-09
cd viper-demo-09
go mod init viper-demo-09
go get github.com/spf13/viper
```

Linux/macOS：

```bash
mkdir viper-demo-09
cd viper-demo-09
go mod init viper-demo-09
go get github.com/spf13/viper
```

创建 `config.prod.yaml`：

```yaml
app:
  env: prod

database:
  dsn: ""
  auto_migrate: false

jwt:
  secret: ""

feature:
  enable_new_payment: false
```

创建 `main.go`：

```go
package main

import (
	"errors"
	"log"
	"strings"

	"github.com/spf13/viper"
)

type Config struct {
	App      App      `mapstructure:"app"`
	Database Database `mapstructure:"database"`
	JWT      JWT      `mapstructure:"jwt"`
	Feature  Feature  `mapstructure:"feature"`
}

type App struct {
	Env string `mapstructure:"env"`
}

type Database struct {
	DSN         string `mapstructure:"dsn"`
	AutoMigrate bool   `mapstructure:"auto_migrate"`
}

type JWT struct {
	Secret string `mapstructure:"secret"`
}

type Feature struct {
	EnableNewPayment bool `mapstructure:"enable_new_payment"`
}

func main() {
	v := viper.New()
	v.SetConfigName("config.prod")
	v.SetConfigType("yaml")
	v.AddConfigPath(".")

	v.SetEnvPrefix("PAYMENT")
	v.SetEnvKeyReplacer(strings.NewReplacer(".", "_"))
	v.AutomaticEnv()
	if err := bindEnvs(v, "database.dsn", "jwt.secret"); err != nil {
		log.Fatalf("bind env failed: %v", err)
	}

	if err := v.ReadInConfig(); err != nil {
		log.Fatalf("read config failed: %v", err)
	}

	var cfg Config
	if err := v.Unmarshal(&cfg); err != nil {
		log.Fatalf("unmarshal config failed: %v", err)
	}

	if err := cfg.Validate(); err != nil {
		log.Fatalf("invalid config: %v", err)
	}

	log.Printf("config ok: env=%s auto_migrate=%v new_payment=%v",
		cfg.App.Env,
		cfg.Database.AutoMigrate,
		cfg.Feature.EnableNewPayment,
	)
}

func bindEnvs(v *viper.Viper, keys ...string) error {
	for _, key := range keys {
		if err := v.BindEnv(key); err != nil {
			return err
		}
	}
	return nil
}

func (c Config) Validate() error {
	if c.App.Env != "prod" {
		return errors.New("app.env must be prod")
	}
	if c.Database.DSN == "" {
		return errors.New("database.dsn is required")
	}
	if c.JWT.Secret == "" {
		return errors.New("jwt.secret is required")
	}
	if c.Database.AutoMigrate {
		return errors.New("database.auto_migrate must be false in prod")
	}
	return nil
}
```

Windows PowerShell 注入配置：

```powershell
$env:PAYMENT_DATABASE_DSN="postgres://payment:secret@db:5432/payment?sslmode=require"
$env:PAYMENT_JWT_SECRET="change-me-to-a-real-secret"
go run .
```

Linux/macOS：

```bash
PAYMENT_DATABASE_DSN="postgres://payment:secret@db:5432/payment?sslmode=require" \
PAYMENT_JWT_SECRET="change-me-to-a-real-secret" \
go run .
```

## 三、关键参数/语法/代码结构

生产配置文件中敏感值留空，依赖环境变量覆盖。

`Validate()` 中加入生产专用规则：

- `database.dsn` 必须存在。
- `jwt.secret` 必须存在。
- `database.auto_migrate` 生产必须关闭。

功能开关 `enable_new_payment` 默认 `false`，这符合保守策略。新功能应该逐步开启，而不是默认全量。

这里显式调用 `BindEnv`，是因为结构体绑定时要确保环境变量覆盖进入 `Unmarshal` 的结果。只用 `AutomaticEnv` 再直接 `GetString` 通常能读到环境变量，但绑定结构体时建议显式绑定关键 key。

## 四、真实后端场景示例

数据库迁移是典型高风险操作。真实项目通常会使用专门迁移工具，例如 `goose`、`migrate` 或平台流水线，而不是让业务服务每次启动都自动迁移生产库。

保守建议：

```text
生产数据库结构变更应走迁移工具和发布流程，业务服务启动时不要默认执行自动迁移。
```

功能开关也要可回滚。比如新支付通道上线：

```yaml
feature:
  enable_new_payment: false
```

先部署代码但不开启功能，再通过受控配置打开。如果异常，可以快速关闭开关，降低回滚成本。

## 五、注意点

不要打印完整配置对象。结构体里可能包含密码、Token、证书。

生产配置变更要有记录。哪怕只是改环境变量，也应该能追踪是谁改的、什么时候改的、为什么改。

配置回滚要提前设计。不能只知道怎么改上去，也要知道怎么退回来。

## 六、常见误区

误区一：生产服务启动时自动迁移数据库。

这可能在多个实例同时启动时重复执行，也可能执行不可逆变更。生产迁移应有明确边界和回滚方案。

误区二：新功能开关默认开启。

默认开启会让未验证功能直接影响用户。保守做法是默认关闭，逐步放量。

误区三：把密钥放进 YAML 后认为私有仓库就安全。

私有仓库也可能泄露，密钥进入 Git 历史后很难彻底清除。生产密钥应通过专门渠道管理。

## 七、本节达标标准

- 能区分配置文件、环境变量、密钥管理平台适合存放的内容。
- 能写出生产环境配置校验规则。
- 知道生产默认值要保守。
- 知道自动迁移、密钥、功能开关的风险。
- 能解释配置变更为什么需要审计和回滚。
