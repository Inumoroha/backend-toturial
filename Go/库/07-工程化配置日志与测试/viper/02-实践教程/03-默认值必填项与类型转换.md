# 03. 默认值必填项与类型转换

本节目标：学完后你能为配置设置合理默认值，读取常见类型，并检查关键配置是否缺失。

## 简短引入

不是所有配置都一样重要。有些配置可以有默认值，例如日志级别默认 `info`；有些配置必须显式提供，例如生产数据库 DSN、JWT 密钥。

Viper 提供了 `SetDefault` 和各种 `Get` 方法，但你仍然需要自己判断哪些配置能默认，哪些必须校验。

## 一、为什么需要它

以订单服务为例：

- `server.read_timeout` 可以默认 3 秒。
- `log.level` 可以默认 `info`。
- `database.dsn` 在生产环境必须提供。
- `jwt.secret` 必须提供，否则令牌签名不可信。

```text
默认值适合非敏感、低风险、可接受的运行参数；关键连接信息和密钥不应该靠默认值糊过去。
```

## 二、基本用法

创建项目：

Windows PowerShell：

```powershell
mkdir viper-demo-03
cd viper-demo-03
go mod init viper-demo-03
go get github.com/spf13/viper
```

Linux/macOS：

```bash
mkdir viper-demo-03
cd viper-demo-03
go mod init viper-demo-03
go get github.com/spf13/viper
```

创建 `config.yaml`：

```yaml
app:
  env: dev

server:
  addr: ":8080"

database:
  dsn: "postgres://dev:dev@localhost:5432/order_dev?sslmode=disable"
```

创建 `main.go`：

```go
package main

import (
	"log"

	"github.com/spf13/viper"
)

func main() {
	viper.SetDefault("log.level", "info")
	viper.SetDefault("server.read_timeout", "3s")
	viper.SetDefault("server.write_timeout", "3s")

	viper.SetConfigName("config")
	viper.SetConfigType("yaml")
	viper.AddConfigPath(".")

	if err := viper.ReadInConfig(); err != nil {
		log.Fatalf("read config failed: %v", err)
	}

	if err := requireKeys("server.addr", "database.dsn"); err != nil {
		log.Fatal(err)
	}

	addr := viper.GetString("server.addr")
	level := viper.GetString("log.level")
	readTimeout := viper.GetDuration("server.read_timeout")

	log.Printf("addr=%s log.level=%s read_timeout=%s", addr, level, readTimeout)
}

func requireKeys(keys ...string) error {
	for _, key := range keys {
		if !viper.IsSet(key) || viper.GetString(key) == "" {
			return &missingConfigError{key: key}
		}
	}
	return nil
}

type missingConfigError struct {
	key string
}

func (e *missingConfigError) Error() string {
	return "missing required config: " + e.key
}
```

运行：

```powershell
go run .
```

Linux/macOS：

```bash
go run .
```

## 三、关键参数/语法/代码结构

`SetDefault("log.level", "info")` 表示配置文件没有写 `log.level` 时使用默认值。

`GetDuration("server.read_timeout")` 可以把 `3s` 解析成 `time.Duration`。这类配置比写毫秒整数更直观。

`IsSet("database.dsn")` 用来判断配置是否存在。注意，存在但为空字符串也通常应该视为无效，所以示例中同时判断了 `GetString(key) == ""`。

`requireKeys` 是启动阶段的配置检查。真实项目中你可以把它放进 `internal/config` 包。

## 四、真实后端场景示例

订单服务启动时，需要数据库 DSN：

```yaml
database:
  dsn: "postgres://order:secret@localhost:5432/order?sslmode=disable"
  max_open_conns: 20
  max_idle_conns: 5
```

连接池参数可以有默认值，但 DSN 不能随便默认。因为连接到错误数据库比启动失败更危险。

对于迁移开关，默认值也要保守：

```go
viper.SetDefault("database.auto_migrate", false)
```

```text
生产环境中的破坏性行为默认应该关闭，只有明确开启才执行。
```

## 五、注意点

`GetInt`、`GetBool`、`GetString` 在配置缺失时会返回零值。零值不一定代表正确配置，所以关键配置要用 `IsSet` 或结构体验证。

时间配置建议使用 `1s`、`500ms` 这种格式，便于阅读。团队内要统一，不要一部分写秒，一部分写毫秒整数。

默认值要写在读取配置之前，这样 Viper 合并配置时可以正确使用默认值。

## 六、常见误区

误区一：认为 `GetString` 返回空字符串就一定是业务想要的值。

空字符串可能是配置缺失，也可能是人为写空。关键配置需要明确校验。

误区二：给数据库 DSN、JWT 密钥设置开发默认值。

这样很容易在生产忘记覆盖，导致服务用错误凭据启动。敏感配置宁可启动失败。

误区三：默认值散落在业务代码各处。

默认值应该集中在配置初始化阶段，方便审查和调整。

## 七、本节达标标准

- 能使用 `SetDefault` 设置默认值。
- 能读取字符串、布尔、整数和时间类型配置。
- 能判断哪些配置必须显式提供。
- 能写出启动阶段的必填配置检查。
- 知道默认值不能替代生产安全校验。
