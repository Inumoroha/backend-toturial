# 06. 配置热更新与运行期风险

本节目标：学完后你能使用 Viper 监听配置文件变化，并判断哪些配置适合热更新。

## 简短引入

热更新听起来很方便：改配置，不重启服务。但后端项目里不是所有配置都适合运行期变更。日志级别可以热更新，数据库连接地址通常不应该直接热更新。

Viper 提供 `WatchConfig` 和 `OnConfigChange`，但是否使用热更新，需要工程判断。

## 一、为什么需要它

常见适合热更新的配置：

- 日志级别。
- 某些非核心功能开关。
- 限流阈值。

常见不适合热更新的配置：

- 数据库 DSN。
- Redis 集群地址。
- JWT 签名密钥。
- 数据库迁移开关。

```text
能热更新不代表应该热更新。运行期变更必须考虑一致性、线程安全和回滚。
```

## 二、基本用法

创建项目：

Windows PowerShell：

```powershell
mkdir viper-demo-06
cd viper-demo-06
go mod init viper-demo-06
go get github.com/spf13/viper
go get github.com/fsnotify/fsnotify
```

Linux/macOS：

```bash
mkdir viper-demo-06
cd viper-demo-06
go mod init viper-demo-06
go get github.com/spf13/viper
go get github.com/fsnotify/fsnotify
```

创建 `config.yaml`：

```yaml
log:
  level: info

feature:
  enable_order_export: false
```

创建 `main.go`：

```go
package main

import (
	"log"
	"sync/atomic"
	"time"

	"github.com/fsnotify/fsnotify"
	"github.com/spf13/viper"
)

type RuntimeConfig struct {
	LogLevel          atomic.Value
	EnableOrderExport atomic.Bool
}

func main() {
	var runtime RuntimeConfig

	v := viper.New()
	v.SetConfigName("config")
	v.SetConfigType("yaml")
	v.AddConfigPath(".")

	if err := v.ReadInConfig(); err != nil {
		log.Fatalf("read config failed: %v", err)
	}

	applyRuntimeConfig(v, &runtime)

	v.OnConfigChange(func(e fsnotify.Event) {
		log.Printf("config changed: %s", e.Name)
		applyRuntimeConfig(v, &runtime)
	})
	v.WatchConfig()

	for {
		log.Printf("log.level=%s enable_order_export=%v",
			runtime.LogLevel.Load(),
			runtime.EnableOrderExport.Load(),
		)
		time.Sleep(3 * time.Second)
	}
}

func applyRuntimeConfig(v *viper.Viper, runtime *RuntimeConfig) {
	level := v.GetString("log.level")
	if level == "" {
		level = "info"
	}
	runtime.LogLevel.Store(level)
	runtime.EnableOrderExport.Store(v.GetBool("feature.enable_order_export"))
}
```

运行后修改 `config.yaml` 中的 `log.level` 或 `feature.enable_order_export`，观察输出变化。

## 三、关键参数/语法/代码结构

`WatchConfig()` 开始监听配置文件变化。

`OnConfigChange(...)` 注册变化后的回调。

示例中使用 `atomic.Value` 和 `atomic.Bool` 保存运行期配置，是为了避免多个 goroutine 同时读写普通变量导致数据竞争。

真实项目中如果配置更复杂，可以使用读写锁保护一份配置快照。

## 四、真实后端场景示例

订单服务出现问题时，可能需要临时把日志级别从 `info` 调成 `debug`：

```yaml
log:
  level: debug
```

这类配置热更新比较合理，因为它不改变核心资源连接，也不影响数据一致性。

再比如订单导出功能开关：

```yaml
feature:
  enable_order_export: false
```

如果导出功能消耗数据库资源过高，可以快速关闭。但这类功能开关也要谨慎设计，默认值应该保守。

## 五、注意点

热更新回调里不要执行耗时操作。比如重建数据库连接池、执行迁移、清空缓存都不适合直接放在回调里。

热更新的配置要有线程安全载体。HTTP handler 在请求中读取配置时，可能与配置更新同时发生。

配置文件保存方式会影响事件触发次数。某些编辑器保存文件时可能触发多次事件，所以回调逻辑要幂等。

## 六、常见误区

误区一：认为所有配置都可以热更新。

数据库连接、密钥、迁移开关这类配置变更会影响系统一致性，通常更适合重启服务并走发布流程。

误区二：热更新时直接修改普通全局变量。

多个 goroutine 并发读写会产生数据竞争。应使用 `atomic`、锁或不可变快照。

误区三：把热更新当作替代发布流程。

生产配置变更仍然需要审计、回滚和监控。热更新只是执行方式，不是治理方式。

## 七、本节达标标准

- 能使用 `WatchConfig` 监听配置变化。
- 能使用 `OnConfigChange` 处理变化事件。
- 能判断日志级别、功能开关等配置是否适合热更新。
- 知道数据库、密钥、迁移开关通常不适合热更新。
- 知道运行期配置读取需要考虑线程安全。
