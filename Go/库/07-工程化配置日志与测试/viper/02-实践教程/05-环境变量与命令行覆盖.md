# 05. 环境变量与命令行覆盖

本节目标：学完后你能用环境变量和命令行参数覆盖配置文件，适配容器部署和 CI/CD。

## 简短引入

配置文件适合保存项目的默认配置，但生产环境中的密码、密钥、连接地址经常由部署系统注入。Viper 支持环境变量和命令行参数，让你不必为每个环境重新打包代码。

## 一、为什么需要它

以权限服务为例，`config.prod.yaml` 可以声明需要数据库，但真实 DSN 不应该提交：

```yaml
database:
  dsn: ""
```

部署时再注入：

```text
AUTH_DATABASE_DSN=postgres://auth:secret@db:5432/auth?sslmode=require
```

这样做的好处是：

- 镜像可以复用。
- 密钥不进入 Git 仓库。
- 配置变更不一定需要重新编译。

```text
配置文件放结构和非敏感默认值，环境变量放环境相关值和敏感值。
```

## 二、基本用法

创建项目：

Windows PowerShell：

```powershell
mkdir viper-demo-05
cd viper-demo-05
go mod init viper-demo-05
go get github.com/spf13/viper
```

Linux/macOS：

```bash
mkdir viper-demo-05
cd viper-demo-05
go mod init viper-demo-05
go get github.com/spf13/viper
```

创建 `config.yaml`：

```yaml
app:
  name: auth-service

server:
  addr: ":8080"

database:
  dsn: ""
```

创建 `main.go`：

```go
package main

import (
	"flag"
	"log"
	"strings"

	"github.com/spf13/viper"
)

func main() {
	configPath := flag.String("config", ".", "config directory")
	flag.Parse()

	v := viper.New()
	v.SetConfigName("config")
	v.SetConfigType("yaml")
	v.AddConfigPath(*configPath)

	v.SetEnvPrefix("AUTH")
	v.SetEnvKeyReplacer(strings.NewReplacer(".", "_"))
	v.AutomaticEnv()

	if err := v.ReadInConfig(); err != nil {
		log.Fatalf("read config failed: %v", err)
	}

	log.Printf("app=%s addr=%s dsn=%s",
		v.GetString("app.name"),
		v.GetString("server.addr"),
		mask(v.GetString("database.dsn")),
	)
}

func mask(s string) string {
	if s == "" {
		return "<empty>"
	}
	return "<set>"
}
```

Windows PowerShell 设置环境变量并运行：

```powershell
$env:AUTH_DATABASE_DSN="postgres://auth:secret@localhost:5432/auth?sslmode=disable"
go run .
```

Linux/macOS：

```bash
AUTH_DATABASE_DSN="postgres://auth:secret@localhost:5432/auth?sslmode=disable" go run .
```

## 三、关键参数/语法/代码结构

`SetEnvPrefix("AUTH")` 表示环境变量前缀是 `AUTH`，可以避免和系统中其他变量冲突。

`SetEnvKeyReplacer(strings.NewReplacer(".", "_"))` 表示把配置 key 里的点号替换成下划线。所以：

```text
database.dsn -> AUTH_DATABASE_DSN
server.addr  -> AUTH_SERVER_ADDR
```

`AutomaticEnv()` 表示读取配置时自动检查环境变量。

命令行参数 `-config` 用来指定配置目录。真实部署中服务工作目录可能不固定，通过参数指定更可靠。

Viper 常见读取优先级可以先这样理解：

```text
显式设置的值 > 命令行参数 > 环境变量 > 配置文件 > 默认值
```

本节示例主要演示环境变量覆盖配置文件。后续如果你把配置 `Unmarshal` 到结构体，建议对关键环境变量显式调用 `BindEnv`：

```go
if err := v.BindEnv("database.dsn"); err != nil {
	return err
}
```

这是 Viper 很容易踩的点：直接 `GetString("database.dsn")` 通常能读到环境变量，但结构体绑定时显式绑定更稳。

## 四、真实后端场景示例

容器启动时常见做法：

```bash
./auth-service -config /app/configs
```

配合环境变量：

```bash
AUTH_DATABASE_DSN="postgres://auth:secret@db:5432/auth?sslmode=require"
AUTH_SERVER_ADDR=":8080"
```

在 Kubernetes 中，这些值可能来自 ConfigMap 和 Secret。应用代码不需要知道它们来自哪里，只需要通过环境变量读取。

对于命令行参数，常见适合放：

- 配置目录。
- 运行环境名。
- 是否执行一次性任务。

不适合放：

- 大量复杂业务配置。
- 数据库密码。
- 多行证书内容。

## 五、注意点

日志中不要打印完整敏感配置。示例里用 `mask` 只输出 `<set>`，这是为了确认配置存在，而不是泄露内容。

环境变量名要统一规范。建议使用服务名前缀，例如 `AUTH_`、`ORDER_`、`SHORTLINK_`。

环境变量覆盖配置文件后，排查问题时要同时检查文件和部署系统。不要只盯着仓库里的 YAML。

命令行参数适合少量启动选项，例如 `-config`、`-env`。如果参数越来越多，说明应该回到配置文件或环境变量中组织，否则启动命令会难以审计。

## 六、常见误区

误区一：生产环境仍然依赖仓库中的明文密码。

这会带来严重泄露风险。至少应使用环境变量，更成熟的团队会接入密钥管理平台。

误区二：环境变量没有前缀。

没有前缀时容易与系统变量或其他服务变量冲突，排查困难。

误区三：把完整 DSN 打进日志。

日志经常会进入集中平台，完整 DSN 可能包含用户名、密码和数据库地址。

误区四：以为 `AutomaticEnv` 能解决所有绑定问题。

当你直接 `GetString` 时它很方便；当你要 `Unmarshal` 到结构体时，关键字段最好显式 `BindEnv`，避免环境变量覆盖没有进入结构体结果。

## 七、本节达标标准

- 能解释 `SetEnvPrefix` 和 `SetEnvKeyReplacer` 的作用。
- 能用命令行参数指定配置目录。
- 知道敏感配置不应该明文提交和完整打印。
- 知道环境变量适合生产注入和容器部署。
- 能说出 Viper 常见配置来源的覆盖优先级。
- 知道结构体绑定场景下关键环境变量建议显式 `BindEnv`。
