# GORM 项目简历话术模板

本节目标：帮助你把“我学过 GORM”转化成简历和面试中可信的项目表达。简历不是堆技术名词，而是要体现你解决了什么业务问题、用了什么技术方案、规避了什么风险、得到什么结果。

核心原则：

```text
不要写：使用 GORM 完成增删改查
要写：围绕具体业务，说明模型设计、关联关系、事务一致性、性能优化和工程分层
```

---

## 一、简历描述的基本公式

### 公式 1：业务模块 + 技术动作 + 工程价值

模板：

```text
负责 XXX 模块数据访问层设计，基于 GORM 建模 XXX 关系，封装 Repository 查询方法，并通过 XXX 方案解决 XXX 问题。
```

示例：

```text
负责博客系统文章模块数据访问层设计，基于 GORM 建模用户、文章、分类、标签和评论关系，封装文章列表、详情、发布和标签绑定等 Repository 方法，并通过 Preload 和字段裁剪优化文章详情与列表查询。
```

---

### 公式 2：复杂写入 + 事务 + 一致性

模板：

```text
使用 GORM Transaction 实现 XXX 多步骤写入，将 XXX、XXX、XXX 放入同一事务，并通过 XXX 保证失败回滚和数据一致性。
```

示例：

```text
使用 GORM Transaction 实现订单创建流程，将订单主表写入、订单明细写入和库存扣减放入同一事务，并通过条件更新和 RowsAffected 判断库存是否充足，避免并发下库存超扣。
```

---

### 公式 3：性能问题 + 排查方法 + 优化结果

模板：

```text
针对 XXX 接口响应慢问题，通过 GORM SQL 日志和 EXPLAIN 定位 XXX，采用 XXX 优化，使 XXX。
```

示例：

```text
针对文章列表接口响应慢问题，通过 GORM SQL 日志定位到列表查询加载了 content 大字段，并使用 EXPLAIN 检查索引命中情况，最终通过 Select 字段裁剪和 category_id/status/created_at 联合索引优化查询。
```

---

### 公式 4：工程分层 + 可维护性

模板：

```text
按照 Handler、Service、Repository、Model 分层组织 GORM 代码，将事务边界放在 Service 层，将查询条件封装在 Repository 层，提升代码复用性和测试便利性。
```

示例：

```text
按照 Handler、Service、Repository、Model 分层组织博客系统后端代码，将文章发布、标签替换等事务边界放在 Service 层，将分页、状态筛选、预加载等查询逻辑封装在 Repository 层，降低 Handler 与数据库细节耦合。
```

## 二、博客系统简历模板

### 简历写法 A：适合初级后端

```text
基于 Go、Gin、GORM、PostgreSQL 实现博客系统，完成用户、文章、分类、标签、评论等核心模型设计，支持文章发布、分页列表、详情查询、评论管理和软删除。使用 GORM Repository 封装常用查询，并通过 DTO 隔离接口响应和数据库模型，避免敏感字段泄露。
```

亮点解释：

- 有完整业务。
- 有模型设计。
- 有查询和软删除。
- 有 Repository 和 DTO 意识。

面试可展开：

- 用户和文章是什么关系？
- 文章和标签为什么是多对多？
- 为什么 Model 和 DTO 要分离？

---

### 简历写法 B：适合有项目优化经历

```text
负责博客系统文章模块的数据建模与查询优化，基于 GORM 建模文章、分类、标签、评论等关联关系，使用 Preload 加载文章详情关联数据，避免循环查询导致的 N+1 问题。针对文章列表接口，通过 Select 裁剪正文大字段，并为 category_id、status、created_at 设计联合索引，提升列表查询稳定性。
```

面试可展开：

- `Preload` 和 `Joins` 区别是什么？
- 如何确认存在 N+1？
- 联合索引顺序怎么考虑？

---

### 简历写法 C：适合强调工程分层

```text
在博客系统中落地 Handler、Service、Repository、Model 分层，Handler 负责参数校验和响应转换，Service 负责文章发布、编辑、标签替换等业务流程，Repository 统一封装 GORM 查询和预加载逻辑。通过该分层减少重复查询代码，并便于对事务流程和查询逻辑进行单独测试。
```

面试可展开：

- 事务为什么不放在 Repository？
- Repository 如何支持事务？
- 你怎么测试 Repository？

## 三、订单系统简历模板

### 简历写法 A：订单创建与库存一致性

```text
基于 Go、GORM 实现订单创建流程，设计订单主表、订单明细表和商品库存模型。使用 GORM Transaction 将订单写入、明细写入和库存扣减纳入同一事务，并通过 stock >= quantity 条件更新和 RowsAffected 判断库存是否充足，避免并发下库存超扣。
```

面试可展开：

- 为什么不能先查库存再扣库存？
- 事务内部为什么必须用 `tx`？
- `RowsAffected` 为 0 说明什么？

---

### 简历写法 B：支付幂等

```text
负责订单支付回调处理，基于支付流水号唯一索引和订单状态机实现幂等控制，通过条件更新限制订单只能从待支付状态变更为已支付，避免重复回调导致重复发放权益或重复更新订单。
```

面试可展开：

- 重复支付回调为什么会发生？
- 状态机如何设计？
- 唯一索引和业务判断分别解决什么问题？

---

### 简历写法 C：订单取消与库存恢复

```text
实现订单超时取消与库存恢复逻辑，通过 GORM 事务保证订单状态变更和库存返还一致完成，并使用条件更新限制只有待支付订单可以取消，避免已支付订单被错误回滚库存。
```

面试可展开：

- 取消订单为什么也需要事务？
- 如何防止已支付订单被取消？
- 定时任务重复执行怎么办？

## 五、简历项目结构模板

### 项目名称

```text

```

### 技术栈

```text
Go, Gin, GORM, PostgreSQL/MySQL, Redis, Docker
```

### 项目描述

```text
该项目是一个面向内容发布/电商交易/后台管理的 Go 后端系统，提供用户、文章、评论、订单、商品、库存等核心业务能力。
```

### 个人职责

```text
负责数据库模型设计、GORM 数据访问层封装、事务一致性流程、列表查询优化和核心接口实现。
```

### 技术亮点

```text

4. 使用 Select 字段裁剪、分页限制、联合索引和 EXPLAIN 排查慢查询。
```

## 六、面试口述模板

### 1 分钟版本

```text
我做过一个 Go + GORM 的博客系统，主要负责数据模型、Repository 封装和文章模块。系统里用户和文章是一对多，文章和标签是多对多，文章详情用 Preload 加载分类、作者和标签。文章发布和标签绑定放在同一个事务里，避免文章创建成功但标签关系失败。列表接口使用 Select 避免查询正文大字段，并配合分页和索引优化查询性能。
```

---

### 3 分钟版本

```text
我做的是一个 Go 后端项目，技术栈是 Gin、GORM 和 PostgreSQL。

数据模型方面，我用 GORM 定义了用户、文章、分类、标签、评论等模型。用户和文章是一对多，分类和文章是一对多，文章和标签是多对多，评论属于用户和文章。

工程结构上，我没有在 Handler 里直接写数据库操作，而是拆成 Handler、Service、Repository、Model。Handler 负责参数和响应，Service 处理业务流程和事务，Repository 封装 GORM 查询。

项目里比较重要的是文章发布流程。创建文章后还要绑定标签，我把文章创建、标签校验、标签关联替换放在一个 GORM Transaction 里，所有操作都使用 tx，任何一步失败都会回滚。

性能方面，文章列表接口最开始查询了 content 正文字段，数据量大后响应变慢。我通过 GORM SQL 日志看到查询字段过多，然后用 Select 只取列表需要的字段，并为 category_id、status、created_at 增加联合索引。文章详情则使用 Preload 加载作者、分类和标签，避免循环查询导致 N+1。
```

---

### 5 分钟版本

结构：

```text
1. 项目背景
2. 数据模型
3. 工程分层
4. 关键业务流程
5. 事务一致性
6. 性能优化
7. 遇到的问题和复盘
```

示例：

```text
这个项目是一个内容管理类后端系统，主要功能包括用户管理、文章发布、分类标签、评论和后台审核。

我负责的重点是 GORM 数据层和文章模块。模型上，用户和文章是一对多，文章和分类是多对一，文章和标签是多对多，文章和评论是一对多。多对多关系通过 GORM 的 many2many 建模，中间表是 article_tags。

代码组织上，我把项目分成 Handler、Service、Repository、Model。Repository 只关心查询和写入，Service 负责业务规则和事务边界。比如创建文章时，Service 开启事务，先创建文章，再校验标签 ID，最后使用 Association("Tags").Replace(tags) 绑定标签。

我遇到过两个比较典型的问题。第一个是结构体 Updates 不更新零值，导致用户无法把昵称改为空字符串。后来改成 map 更新，并补了测试。第二个是文章列表查询变慢，我打开 GORM SQL 日志后发现列表接口查出了 content 字段，后来用 Select 裁剪字段，并通过 EXPLAIN 检查索引。

通过这个项目，我对 GORM 的理解不只是 CRUD，还包括关联加载、事务一致性、SQL 日志、索引优化和工程分层。
```

## 七、常见低质量写法与改法

### 写法 1：太空泛

低质量：

```text
使用 GORM 完成数据库操作。
```

改法：

```text
基于 GORM 封装用户、文章、标签、评论等 Repository 方法，支持分页查询、关联预加载、软删除和事务写入。
```

---

### 写法 2：只有技术，没有业务

低质量：

```text
使用 GORM、Preload、Transaction、Logger。
```

改法：

```text
在文章详情接口中使用 Preload 加载作者、分类和标签，避免循环查询导致的 N+1；在文章发布流程中使用 Transaction 保证文章主体和标签绑定一致提交。
```

---

### 写法 3：声称优化但没有依据

低质量：

```text
优化了数据库查询性能。
```

改法：

```text
通过 GORM SQL 日志定位文章列表查询加载正文大字段，使用 Select 字段裁剪并补充 category_id/status/created_at 联合索引，减少无效数据读取。
```

---

### 写法 4：职责边界不清

低质量：

```text
在 Controller 中使用 GORM 完成业务逻辑。
```

改法：

```text
采用 Handler、Service、Repository 分层组织代码，Handler 负责参数和响应，Service 负责业务规则和事务，Repository 封装 GORM 数据访问。
```

## 八、简历自查清单

提交简历前检查：

- 是否出现了具体业务模块。
- 是否出现了具体 GORM 能力，例如 `Preload`、事务、Scope、软删除、字段裁剪。
- 是否说明了为什么使用这些能力。
- 是否能解释背后的 SQL。
- 是否能讲出一个踩坑和修复过程。
- 是否避免了“完成增删改查”这种低价值表述。
- 是否把敏感字段、事务边界、SQL 注入、索引这些工程问题讲清楚。

达标标准：

```text
面试官看到你的项目描述后，至少能追问出：
1. 关联关系怎么设计？
2. 事务怎么保证一致性？
3. 慢查询怎么排查？
4. 你遇到过什么坑？
5. 这个项目和普通 CRUD 有什么区别？
```
