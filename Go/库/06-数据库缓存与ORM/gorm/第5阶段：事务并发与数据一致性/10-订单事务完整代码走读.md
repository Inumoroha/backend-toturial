# 10 订单事务完整代码走读

本节目标：把订单下单事务从输入结构、模型、Service 到每一步 SQL 行为完整走一遍。前面已经分散讲过事务、库存扣减、回滚测试，本节把它们收束成一份可以照着实现的完整代码走读。

---

## 一、业务输入

```go
type CreateOrderItemInput struct {
    ProductID uint
    Quantity  int
}

type CreateOrderInput struct {
    UserID uint
    Items  []CreateOrderItemInput
}
```

注意：

- `UserID` 来自登录态，不应该来自前端请求体。
- `Quantity` 必须大于 0。
- `Items` 不能为空。

---

## 二、模型

```go
type Product struct {
    ID        uint
    Name      string
    Price     int64
    Stock     int
    CreatedAt time.Time
    UpdatedAt time.Time
}

type Order struct {
    ID          uint
    UserID      uint
    OrderNo     string
    TotalAmount int64
    Status      string
    Items       []OrderItem
    CreatedAt   time.Time
    UpdatedAt   time.Time
}

type OrderItem struct {
    ID        uint
    OrderID   uint
    ProductID uint
    Price     int64
    Quantity  int
    CreatedAt time.Time
}
```

金额用 `int64` 存分，库存用 `int`。

## 三、完整事务代码

```go
func CreateOrder(db *gorm.DB, input CreateOrderInput) (*Order, error) {
    if input.UserID == 0 {
        return nil, fmt.Errorf("user required")
    }
    if len(input.Items) == 0 {
        return nil, fmt.Errorf("items required")
    }

    var created Order

    err := db.Transaction(func(tx *gorm.DB) error {
        var total int64
        orderItems := make([]OrderItem, 0, len(input.Items))

        for _, item := range input.Items {
            if item.Quantity <= 0 {
                return fmt.Errorf("invalid quantity")
            }

            var product Product
            if err := tx.First(&product, item.ProductID).Error; err != nil {
                return err
            }

            if product.Stock < item.Quantity {
                return fmt.Errorf("stock not enough")
            }

            total += product.Price * int64(item.Quantity)
            orderItems = append(orderItems, OrderItem{
                ProductID: product.ID,
                Price:     product.Price,
                Quantity:  item.Quantity,
            })
        }

        created = Order{
            UserID:      input.UserID,
            OrderNo:     fmt.Sprintf("NO%d", time.Now().UnixNano()),
            TotalAmount: total,
            Status:      "pending",
        }

        if err := tx.Create(&created).Error; err != nil {
            return err
        }

        for i := range orderItems {
            orderItems[i].OrderID = created.ID
        }

        if err := tx.Create(&orderItems).Error; err != nil {
            return err
        }

        for _, item := range input.Items {
            result := tx.Model(&Product{}).
                Where("id = ? AND stock >= ?", item.ProductID, item.Quantity).
                Update("stock", gorm.Expr("stock - ?", item.Quantity))
            if result.Error != nil {
                return result.Error
            }
            if result.RowsAffected == 0 {
                return fmt.Errorf("stock not enough")
            }
        }

        return nil
    })
    if err != nil {
        return nil, err
    }

    return &created, nil
}
```

## 四、每一步为什么这样写

### 1. 先校验输入

空订单没有业务意义，数量小于等于 0 也必须拒绝。

### 2. 查询商品快照

下单时要记录商品当时价格，所以 `OrderItem.Price` 使用当前商品价格，而不是以后查询商品表价格。

### 3. 创建订单后再创建明细

订单明细需要 `OrderID`，所以必须先创建订单，拿到主键。

### 4. 扣库存必须条件更新

即使前面查过库存，扣减时也要带：

```sql
stock >= quantity
```

因为并发请求可能在你查询后修改库存。

### 5. 检查 RowsAffected

库存不足时 SQL 可能执行成功但影响 0 行。必须把 0 行当作业务失败。

## 五、失败路径

以下任一步失败都应该回滚：

- 商品不存在。
- 数量非法。
- 库存不足。
- 创建订单失败。
- 创建订单明细失败。
- 扣库存失败。

回滚后应该没有半截数据。
