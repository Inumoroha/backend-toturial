# 05 主键、时间字段与基础模型

本节目标：系统理解 GORM 模型中最基础的字段：主键、创建时间、更新时间、删除时间。很多后续能力，比如软删除、排序、分页、审计，都依赖这些字段。

---

## 一、最小模型

```go
type User struct {
    ID   uint
    Name string
}
```

GORM 默认认为：

- `ID` 是主键。
- 表名是 `users`。
- 字段名是 `id` 和 `name`。

这个模型能跑，但真实项目通常不够。

---

## 二、显式主键

```go
type User struct {
    ID uint `gorm:"primaryKey"`
}
```

显式写 `primaryKey` 的好处是可读性更强。别人看到模型时，不需要记 GORM 默认约定。

---

## 三、自增主键

MySQL 下，`uint` 主键通常会生成自增字段。

你插入：

```go
user := User{Name: "Tom"}
db.Create(&user)
fmt.Println(user.ID)
```

GORM 会把数据库生成的 ID 回填。

---

## 四、CreatedAt

```go
CreatedAt time.Time
```

GORM 在创建记录时自动设置。

适合用于：

- 列表按创建时间排序。
- 展示注册时间。
- 统计新增数据。
- 排查数据来源。

---

## 五、UpdatedAt

```go
UpdatedAt time.Time
```

GORM 在更新记录时自动维护。

适合用于：

- 展示最后编辑时间。
- 判断数据是否近期变更。
- 做简单审计。

---

## 六、DeletedAt

```go
DeletedAt gorm.DeletedAt `gorm:"index"`
```

有这个字段后，普通 `Delete` 会变成软删除。

普通查询会自动排除：

```sql
deleted_at IS NULL
```

建议给 `DeletedAt` 加索引，因为软删除过滤会经常出现在查询中。

---

## 七、gorm.Model

GORM 内置：

```go
type Model struct {
    ID        uint `gorm:"primarykey"`
    CreatedAt time.Time
    UpdatedAt time.Time
    DeletedAt DeletedAt `gorm:"index"`
}
```

使用：

```go
type User struct {
    gorm.Model
    Name string
}
```

优点：少写字段。

缺点：初学时不够直观，真实项目中有时需要自定义 ID 类型、JSON tag 或删除策略。

---

## 八、推荐写法

学习阶段推荐显式字段：

```go
type User struct {
    ID        uint           `gorm:"primaryKey"`
    Username  string         `gorm:"size:64;not null;uniqueIndex"`
    CreatedAt time.Time
    UpdatedAt time.Time
    DeletedAt gorm.DeletedAt `gorm:"index"`
}
```

这样每个字段的数据库含义都清楚。

---

## 九、常见错误

### 1. 忘记 CreatedAt

后面做列表排序时，没有 `created_at` 会很不方便。

### 2. 所有表都软删除

不是所有表都需要软删除。比如中间表、临时表、日志表，要根据业务决定。

### 3. 不给 DeletedAt 加索引

软删除表查询经常带 `deleted_at IS NULL`，建议加索引。

---

## 十、本节达标标准

学完本节后，你应该能够做到：

- 解释 GORM 默认主键约定。
- 说明 `CreatedAt` 和 `UpdatedAt` 的自动维护行为。
- 说明 `DeletedAt` 如何触发软删除。
- 说清 `gorm.Model` 的优缺点。
- 为业务模型选择显式字段还是嵌入 `gorm.Model`。

## 颗粒度补强：本节应该如何真正练会

这一节不要只停留在“看懂概念”。建议你按下面的方式把「主键时间字段与基础模型」练成可以在项目中使用的能力。

### 1. 先写最小可运行示例

在 `code/gorm-study` 中准备一个最小 Go 程序，只保留和本节主题直接相关的模型、数据库连接和 GORM 调用。不要一开始就放进完整项目结构，否则你很难判断问题来自 GORM、分层、路由还是参数绑定。

建议流程：

```text
1. 定义最小模型
2. AutoMigrate 建表
3. 插入 2 到 3 条测试数据
4. 执行本节核心 GORM API
5. 打印 SQL 日志
6. 到数据库客户端里验证结果
```

### 2. 必须观察 SQL

学习 GORM 的关键不是背 API，而是看它生成的 SQL。执行本节代码时，建议临时开启：

```go
db = db.Debug()
```

或者在初始化时使用：

```go
Logger: logger.Default.LogMode(logger.Info)
```

你至少要回答：

```text
这段 GORM 代码生成了几条 SQL？
SQL 中有没有 WHERE 条件？
有没有 LIMIT 或 ORDER BY？
有没有使用软删除条件 deleted_at IS NULL？
更新或删除时 RowsAffected 是多少？
```

### 3. 用数据库客户端验证

不要只相信 Go 程序输出。每次运行后，都在 MySQL 客户端中执行对应查询，例如：

```sql
SHOW TABLES;
SHOW CREATE TABLE users;
SHOW INDEX FROM users;
SELECT * FROM users ORDER BY id DESC LIMIT 10;
```

如果本节涉及文章、评论、订单或标签，就去查对应表：

```sql
SELECT * FROM articles ORDER BY id DESC LIMIT 10;
SELECT * FROM comments ORDER BY id DESC LIMIT 10;
SELECT * FROM orders ORDER BY id DESC LIMIT 10;
SELECT * FROM article_tags ORDER BY article_id DESC LIMIT 10;
```

### 4. 主动制造一个错误

每节都建议故意制造一个小错误，然后观察报错和数据变化。比如：

```text
把数据库密码写错，观察连接错误。
把唯一字段重复插入，观察唯一索引错误。
把查询 ID 改成不存在的值，观察 ErrRecordNotFound。
把更新条件去掉，观察 GORM 是否阻止全表更新。
把事务里的 return err 去掉，观察是否错误提交。
```

你真正排查过错误，才算把这个知识点学稳。

### 5. 放回真实项目场景

最后把本节知识放回博客系统或订单系统中思考：

```text
博客系统里哪个接口会用到这个知识点？
订单系统里哪个流程会因为这个知识点写错而出问题？
这个知识点应该放在 Model、Repository、Service 还是 Handler？
它需要测试吗？需要看 SQL 日志吗？需要事务吗？
```

如果你能回答这些问题，说明你已经从“知道 API”进入到“能做后端项目”的层次。
