# GORM 后续进阶学习方向

本节目标：当你完成当前 GORM 系统教程后，知道下一步应该往哪里深入。GORM 只是 Go 后端数据访问的一部分，真正成为后端工程师，还需要继续补齐数据库设计、工程化、性能、分布式一致性、测试和生产实践能力。

---

## 一、进阶路线总览

建议后续按这条路线继续：

```text
GORM 熟练使用
  -> SQL 与数据库能力加深
  -> Go Web 工程化
  -> 测试与质量保障
  -> 性能优化与可观测性
  -> 缓存、消息队列与异步任务
  -> 分布式事务与一致性
  -> 生产部署与运维
  -> 架构设计与项目沉淀
```

不要把进阶理解成“学更多 API”。后端工程师的进阶，本质是能处理更真实、更复杂、更高风险的业务场景。

---

## 二、方向一：继续加深 SQL 能力

### 为什么要学

GORM 最终生成的是 SQL。你越懂 SQL，就越能写出高质量 GORM 代码，也越能判断什么时候应该绕开 ORM 写原生 SQL。

### 重点内容

- 多表 JOIN 深入
- 子查询
- 聚合统计
- 窗口函数
- CTE
- 执行计划
- 索引设计
- 事务隔离级别
- 锁机制
- 慢查询优化

### 推荐练习

- 用原生 SQL 重写博客系统文章列表。
- 用原生 SQL 统计每个用户的文章数、评论数、总阅读量。
- 对订单表写月度销售额统计。
- 对慢查询执行 `EXPLAIN` 并写优化记录。

### 达标标准

- 能看懂 GORM 生成的 SQL。
- 能独立写复杂统计 SQL。
- 能根据查询设计索引。
- 能解释一次慢查询为什么慢。

---

## 三、方向二：深入数据库迁移

### 为什么要学

学习阶段可以用 `AutoMigrate`，但真实生产环境不能完全依赖它。生产数据库变更需要可审查、可回滚、可追踪。

### 重点内容

- 迁移版本管理
- up/down 迁移
- 字段新增、删除、重命名
- 索引变更
- 数据迁移
- 灰度发布中的数据库变更
- 回滚方案

### 推荐工具

- golang-migrate
- goose
- atlas

### 推荐练习

- 把博客项目从 `AutoMigrate` 改成迁移脚本。
- 写一个新增字段迁移。
- 写一个新增索引迁移。
- 写一个数据修复迁移。
- 模拟迁移失败后的回滚。

### 达标标准

- 不依赖 `AutoMigrate` 管理生产表结构。
- 每次数据库变更都有迁移文件。
- 能说明迁移上线前要检查什么。

---

## 四、方向三：GORM Gen 类型安全查询

### 为什么要学

普通 GORM 查询里大量字段名是字符串：

```go
db.Where("email = ?", email)
```

字段写错时，编译器发现不了。GORM Gen 可以生成类型安全查询代码，适合中大型项目。

### 重点内容

- Gen 代码生成
- query 包使用
- 类型安全 Where
- 类型安全 Select
- Gen 与 Repository 结合
- 普通 GORM 和 Gen 的取舍

### 推荐练习

- 为博客系统模型生成 query 代码。
- 用 Gen 重写用户按邮箱查询。
- 用 Gen 重写文章分页查询。
- 用 Gen 重写订单详情查询。

### 达标标准

- 能完成 Gen 初始化。
- 能用生成代码替代字符串字段查询。
- 能说明什么时候不需要 Gen。

## 五、方向四：测试体系完善

### 为什么要学

后端项目不是“能跑”就够。事务、权限、库存、支付、删除这些场景必须有测试保障。

### 重点内容

- Repository 测试
- Service 测试
- Handler 测试
- 集成测试
- 测试数据库
- 事务回滚测试
- 并发测试
- 测试数据构造

### 推荐练习

- 测试用户重复注册。
- 测试文章标签不存在时回滚。
- 测试库存不足时订单不创建。
- 测试重复支付只能成功一次。
- 测试文章详情没有 N+1。

### 达标标准

- 关键业务都有测试。
- 失败路径有测试。
- 测试库和开发库分离。
- 能在修改代码后用测试防止回归。

---

## 六、方向五：性能优化与可观测性

### 为什么要学

真实项目上线后，问题不再只是“代码对不对”，还包括“接口慢不慢、SQL 多不多、连接池够不够、错误能不能追踪”。

### 重点内容

- GORM SQL 日志
- 慢查询日志
- EXPLAIN
- 连接池指标
- 接口耗时统计
- Prometheus
- Grafana
- OpenTelemetry
- 链路追踪
- 结构化日志

### 推荐练习

- 给博客 API 加请求耗时日志。
- 给每个接口记录 SQL 条数。
- 用 Prometheus 暴露基础指标。
- 用 Grafana 看接口耗时。
- 模拟慢查询并定位原因。

### 达标标准

- 能定位一个慢接口。
- 能区分 SQL 慢、SQL 多、返回数据过大。
- 能通过日志和指标找到问题证据。

---

## 七、方向六：缓存与 Redis

### 为什么要学

数据库不是所有读请求的唯一答案。热门文章、分类列表、标签列表、验证码、登录态、限流等场景都经常使用 Redis。

### 重点内容

- Redis 基础数据结构
- 缓存热点数据
- 缓存穿透
- 缓存击穿
- 缓存雪崩
- 分布式锁
- 限流
- 排行榜
- 延迟队列基础

### 推荐练习

- 缓存文章详情。
- 缓存分类列表。
- 用 Redis 做热门文章榜单。
- 给登录接口加限流。
- 实现订单防重复提交 key。

### 达标标准

- 能判断哪些数据适合缓存。
- 能设计缓存 key。
- 能处理缓存过期。
- 知道缓存一致性风险。

---

## 八、方向七：消息队列与异步任务

### 为什么要学

很多后端任务不应该放在 HTTP 请求里同步完成，比如发送邮件、写操作日志、统计阅读量、订单超时关闭。

### 重点内容

- RabbitMQ / Kafka 基础
- 生产者和消费者
- 消息确认
- 重试
- 死信队列
- 幂等消费
- 延迟任务
- 事务消息基本思想

### 推荐练习

- 注册后异步发送欢迎邮件。
- 创建文章后异步写操作日志。
- 阅读文章后异步增加统计。
- 订单 30 分钟未支付自动关闭。
- 支付成功后异步通知库存或积分模块。

### 达标标准

- 能把耗时任务从请求链路中拆出去。
- 能处理消息重复消费。
- 能设计失败重试和死信处理。

---

## 九、方向八：分布式一致性与幂等

### 为什么要学

订单、支付、库存、优惠券、积分等业务都绕不开一致性问题。GORM 事务只能解决单数据库事务，跨服务、跨资源时需要更完整的设计。

### 重点内容

- 幂等 key
- 状态机
- 唯一业务单号
- 乐观锁
- 悲观锁
- 本地消息表
- Saga
- TCC 基本思想
- Outbox Pattern
- 最终一致性

### 推荐练习

- 防止重复下单。
- 防止重复支付。
- 支付回调幂等处理。
- 订单状态机设计。
- 使用本地消息表记录待发送事件。

### 达标标准

- 能解释幂等和事务的区别。
- 能设计支付回调幂等。
- 能说清单库事务解决不了什么问题。

---

## 十、方向九：权限系统与后台管理

### 为什么要学

多数 Go 后端岗位都会遇到后台管理、用户权限、角色菜单、操作日志等需求。

### 重点内容

- RBAC
- 用户、角色、权限
- 菜单权限
- 接口权限
- 数据权限
- 操作日志
- 管理员审计

### 推荐练习

- 给博客系统增加管理员。
- 管理员可以删除任意评论。
- 普通用户只能删除自己的评论。
- 给后台接口加角色校验。
- 记录管理员操作日志。

### 达标标准

- 能设计基础 RBAC 表结构。
- 能在 Service 或 Middleware 中做权限校验。
- 能区分接口权限和数据权限。

## 十一、方向十：生产部署与运维

### 为什么要学

后端工程师最终要让服务稳定运行，而不是只在本地跑通。

### 重点内容

- Dockerfile
- Docker Compose
- Nginx 反向代理
- Linux 部署
- CI/CD
- 数据库备份
- 日志收集
- 健康检查
- 优雅关闭
- 配置管理

### 推荐练习

- 给博客 API 写 Dockerfile。
- 使用 Docker Compose 启动 Go 服务、MySQL、Redis。
- 使用 Nginx 反向代理。
- 写 GitHub Actions 自动测试。
- 模拟服务重启和数据库断连。

### 达标标准

- 项目能容器化运行。
- 能写清楚部署步骤。
- 能做基础健康检查。
- 能处理配置和日志。

## 十三、进阶学习节奏

建议按 8 到 12 周继续：

```text
第 1-2 周：SQL、索引、迁移
第 3-4 周：测试、工程化、GORM Gen
第 5-6 周：Redis、缓存、限流
第 7-8 周：消息队列、异步任务
第 9-10 周：幂等、一致性、订单支付
第 11-12 周：部署、监控、项目复盘
```

每个方向都要有代码产出，不要只看资料。

---

## 十四、最终进阶目标

后续进阶完成后，你应该具备：

- 能设计可维护的数据库模型。
- 能写复杂 SQL 并优化慢查询。
- 能用 GORM 做工程化封装。
- 能写关键业务测试。
- 能处理缓存、消息队列和异步任务。
- 能设计订单、支付、库存等一致性方案。
- 能把 Go 后端项目部署到服务器。
- 能在面试中讲清楚项目设计和技术取舍。
