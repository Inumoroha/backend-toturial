# Redis-Go 后端工程师学习路线

本路线面向想成为 Go 后端工程师的学习者，目标不是背 Redis 命令，而是学会在真实项目中判断：什么时候该用 Redis，怎么用 Go 安全接入，如何处理缓存一致性、并发、错误和生产风险。

```text
Redis 不是数据库的替代品，它通常是后端系统里的加速层、协作层和削峰层。
```

## 一、学习前置要求

在开始 Redis-Go 之前，建议已经具备：

- Go 基础：结构体、接口、context、错误处理、goroutine 的基本理解。
- HTTP 服务基础：能写一个简单的 Gin、Echo 或 net/http 接口。
- 数据库基础：知道主键、索引、事务、唯一约束和迁移工具的作用。
- 命令行基础：能使用 PowerShell 或 Linux/macOS 终端运行 Go、Docker 命令。

不需要先掌握 Redis 内核、集群源码、网络协议。这些属于后期能力，初学阶段先学会把 Redis 用对。

## 二、推荐技术栈

教程默认使用：

- Go 1.22+ 或当前稳定版本。
- Redis 7.x。
- Go 客户端：`github.com/redis/go-redis/v9`。
- 本地运行方式：Docker Compose。
- 项目结构：普通 Go module，不强绑定某个 Web 框架。

常见替代方案：

- `redigo`：老牌客户端，适合维护旧项目。
- `rueidis`：偏高性能和高级特性，初学阶段不建议一开始就上。
- 云 Redis：生产环境常见，但本地学习先用 Docker 降低环境成本。

## 三、阶段路线

### 第 1 阶段：环境和连接

目标：能启动 Redis，能在 Go 项目中连接 Redis，并知道连接池、超时、context 的作用。

对应文章：

- `01-环境与项目初始化/01-用Docker启动Redis并创建Go项目.md`
- `01-环境与项目初始化/02-连接Redis与配置连接池.md`

学习重点：

- Redis 本地环境搭建。
- Go module 初始化。
- `go-redis/v9` 基本使用。
- `context.Context`、连接池、超时配置。

### 第 2 阶段：核心数据结构

目标：知道 Redis 的 String、Hash、List、Stream、Set、ZSet 在后端项目中分别适合解决什么问题。

对应文章：

- `02-核心数据结构/03-String缓存用户资料.md`
- `02-核心数据结构/04-Hash保存对象字段.md`
- `02-核心数据结构/05-List与Stream处理日志和消息.md`
- `02-核心数据结构/06-Set与ZSet实现权限和排行榜.md`

学习重点：

- 不是按命令背诵，而是按业务建模。
- 用 TTL 控制缓存生命周期。
- 区分缓存、队列、集合、排行榜的边界。
- 理解 Redis 数据结构的成本，不把所有数据都塞进 Redis。

### 第 3 阶段：真实后端缓存模式

目标：能把 Redis 放进用户、文章、订单、短链接、库存等业务场景里，并处理常见风险。

对应文章：

- `03-项目实践/07-缓存文章详情与防穿透.md`
- `03-项目实践/08-短链接服务设计.md`
- `03-项目实践/09-库存扣减与Lua原子操作.md`

学习重点：

- Cache Aside 模式。
- 缓存穿透、击穿、雪崩的实用处理方式。
- 短链接映射和过期策略。
- 库存扣减为什么需要原子性。
- Lua 在 Redis 中的保守使用方式。

### 第 4 阶段：工程化和生产安全

目标：学会把 Redis 代码写得可维护、可观测、可回滚，而不是只会调用命令。

对应文章：

- `04-工程化与生产/10-Pipeline事务与错误处理.md`
- `04-工程化与生产/11-过期策略一致性与可观测性.md`
- `04-工程化与生产/12-最终项目用户文章订单缓存服务.md`

学习重点：

- Pipeline 和事务的区别。
- 错误处理、超时、重试边界。
- Key 命名规范。
- 指标、日志和慢查询。
- 缓存上线、降级和回滚。

## 四、建议学习节奏

### 第 1 周：能跑起来

每天完成 1 篇文章，重点是复制命令、运行代码、看见结果。

达标标准：

- 能启动 Redis。
- 能用 Go 连接 Redis。
- 能完成 `Set/Get/Del/Expire`。
- 能解释为什么 Redis 操作要带 `context`。

### 第 2 周：能建模

围绕用户、文章、权限、日志、排行榜练习数据结构。

达标标准：

- 知道 String、Hash、Set、ZSet 适合什么场景。
- 能为业务 key 命名。
- 能设置合理 TTL。
- 能避免把 Redis 当主数据库。

### 第 3 周：能处理真实问题

练习缓存穿透、短链接、库存扣减。

达标标准：

- 能写 Cache Aside。
- 能处理缓存未命中。
- 能用 `SetNX` 或 Lua 解决简单并发问题。
- 能说明 Redis 和数据库之间的一致性边界。

### 第 4 周：能接近生产

整理项目结构、日志、错误处理和可观测性。

达标标准：

- 能封装 Redis 客户端。
- 能统一 key 规范。
- 能区分 Pipeline、事务和 Lua。
- 能写出上线前检查清单。

## 五、最终项目建议

最终做一个“小型内容服务”：

- 用户资料缓存。
- 文章详情缓存。
- 文章点赞集合。
- 热门文章排行榜。
- 短链接跳转。
- 订单库存扣减。
- 操作日志写入 Stream。

这个项目规模不大，但覆盖了 Go 后端使用 Redis 的主要能力。完成后，你可以继续接入 HTTP 框架、PostgreSQL、MySQL、消息队列和部署环境。

## 六、学习原则

```text
先让业务正确，再让缓存变快；先保证可回滚，再追求复杂优化。
```

保守建议：

- 用户输入不要直接拼到 SQL 字符串里，数据库访问要参数化。
- 涉及数据库结构变更时，使用迁移工具，不手工改生产库。
- 缓存 key 要可读、可查、可删除。
- Redis 里的数据要有过期策略，除非你明确知道为什么不过期。
- 重要业务状态仍以数据库为准。
- 生产环境不要随意执行 `KEYS *`、`FLUSHALL` 这类高风险命令。
- 出错时要允许系统绕过 Redis 继续提供核心功能。
