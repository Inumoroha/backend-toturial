# 13. 阶段练习与最终项目验收

本节目标：你将能按阶段完成 Redis-Go 练习，并用一套标准判断自己是否真的具备后端项目落地能力。

简短引入：前面的文章讲“怎么做”和“为什么这么做”，这一节负责把学习变成项目训练。真实后端学习不能只看懂代码，还要能独立搭环境、定位错误、写出可回滚的方案。

## 一、为什么需要它

很多初学者学 Redis 时会卡在两个地方：

- 看命令时都懂，自己写接口时不知道该选哪种数据结构。
- 示例能跑，换成真实业务后不知道怎么处理错误、一致性和上线风险。

可以理解为，这一节是你的练习地图。每完成一个阶段，都要留下可运行代码、可解释的设计和可检查的结果。

```text
学 Redis-Go 的达标标准不是会背命令，而是能把 Redis 放进一个可靠的 Go 后端流程里。
```

## 二、基本用法

建议你在单独目录中创建最终练习项目：

Windows PowerShell：

```powershell
mkdir redis-go-final
cd redis-go-final
go mod init redis-go-final
go get github.com/redis/go-redis/v9
```

Linux/macOS：

```bash
mkdir redis-go-final
cd redis-go-final
go mod init redis-go-final
go get github.com/redis/go-redis/v9
```

准备 Redis：

```yaml
services:
  redis:
    image: redis:7
    container_name: redis-go-final
    ports:
      - "6379:6379"
    command: ["redis-server", "--appendonly", "yes"]
```

启动：

Windows PowerShell：

```powershell
docker compose up -d
docker exec -it redis-go-final redis-cli ping
```

Linux/macOS：

```bash
docker compose up -d
docker exec -it redis-go-final redis-cli ping
```

预期输出：

```text
PONG
```

## 三、关键参数/语法/代码结构

最终项目建议保持这个结构：

```text
redis-go-final/
  docker-compose.yml
  go.mod
  cmd/
    api/
      main.go
  internal/
    cache/
      redis.go
      keys.go
    user/
      service.go
    article/
      service.go
    shortlink/
      service.go
    order/
      service.go
```

各目录职责：

- `cmd/api`：程序入口，负责加载配置、创建 Redis 客户端、组装 service。
- `internal/cache/redis.go`：封装 Redis 连接、连接池、超时和启动检查。
- `internal/cache/keys.go`：集中管理所有 key 命名。
- `internal/user`：用户资料缓存。
- `internal/article`：文章详情、点赞、热榜。
- `internal/shortlink`：短链接缓存和回源。
- `internal/order`：库存扣减和订单事件。

真实项目中，数据库访问应该放在 repository 层。为了 Redis 学习不跑偏，练习阶段可以先用函数模拟数据库，但你必须在设计说明里写清楚哪些数据最终要落库。

## 四、真实后端场景示例

你要完成一个“小型内容服务”，至少包含这些能力。

### 1. 用户资料缓存

要求：

- 使用 String + JSON 缓存用户资料。
- key 格式为 `user:profile:{userID}`。
- 设置 10 到 30 分钟 TTL。
- 反序列化失败时删除缓存并回源。

验收方式：

Windows PowerShell：

```powershell
docker exec -it redis-go-final redis-cli GET user:profile:1001
docker exec -it redis-go-final redis-cli TTL user:profile:1001
```

Linux/macOS：

```bash
docker exec -it redis-go-final redis-cli GET user:profile:1001
docker exec -it redis-go-final redis-cli TTL user:profile:1001
```

你应该能解释：

- 为什么缓存用户资料要有 TTL。
- 为什么 Redis 不是用户资料的事实来源。
- 用户修改昵称后应该删除哪个 key。

### 2. 文章详情缓存与防穿透

要求：

- 使用 `article:detail:{articleID}` 缓存文章详情。
- 不存在的文章写入短 TTL 空值缓存。
- 文章更新后删除缓存。
- Redis 读取失败时允许回源数据库。

验收方式：

```text
第一次请求：查数据库并写 Redis。
第二次请求：直接命中 Redis。
请求不存在文章：写入 30 秒到 2 分钟的空值缓存。
```

你应该能解释：

- 缓存穿透是什么。
- 空值缓存 TTL 为什么不能太长。
- 为什么更新数据库后通常删除缓存，而不是直接更新缓存。

### 3. 权限和点赞

要求：

- 使用 Set 保存用户权限：`user:permissions:{userID}`。
- 使用 Set 保存文章点赞用户：`article:likes:{articleID}`。
- 重复点赞不能重复增加热度。
- 权限变更后能删除或刷新权限缓存。

验收方式：

```powershell
docker exec -it redis-go-final redis-cli SISMEMBER user:permissions:1001 article:write
docker exec -it redis-go-final redis-cli SCARD article:likes:9001
```

Linux/macOS：

```bash
docker exec -it redis-go-final redis-cli SISMEMBER user:permissions:1001 article:write
docker exec -it redis-go-final redis-cli SCARD article:likes:9001
```

你应该能解释：

- 为什么不能在生产中随意对大 Set 执行 `SMEMBERS`。
- 点赞记录为什么通常还要落数据库。

### 4. 热门文章排行榜

要求：

- 使用 ZSet 保存热度：`rank:article:hot`。
- 点赞、阅读、评论可以给不同分值。
- 只读取 Top N，不一次性读取全部。
- 定期清理旧成员或按周期拆分排行榜。

验收方式：

```powershell
docker exec -it redis-go-final redis-cli ZREVRANGE rank:article:hot 0 9 WITHSCORES
```

Linux/macOS：

```bash
docker exec -it redis-go-final redis-cli ZREVRANGE rank:article:hot 0 9 WITHSCORES
```

你应该能解释：

- ZSet 为什么适合排行榜。
- 排行榜为什么需要清理策略。
- 热榜分数是否等于数据库里的最终事实。

### 5. 短链接跳转缓存

要求：

- 使用 `shortlink:{code}` 缓存短码到长 URL 的映射。
- 短码必须校验字符集和长度。
- 短码唯一性以数据库唯一约束为准。
- 访问日志不要同步阻塞跳转主链路。

验收方式：

```text
创建短链接：先写数据库，再写 Redis。
访问短链接：先查 Redis，未命中查数据库并回填。
删除短链接：先处理数据库状态，再删除 Redis key。
```

你应该能解释：

- 为什么不能只把短链接存在 Redis。
- 为什么 `SetNX` 不能替代数据库唯一约束。
- 什么是开放跳转风险。

### 6. 库存扣减

要求：

- 使用 Lua 脚本完成库存检查和扣减。
- 库存不足不能扣成负数。
- 订单创建失败时要有回滚或补偿设计。
- 回滚库存必须幂等。

验收方式：

```powershell
docker exec -it redis-go-final redis-cli GET stock:sku:1001
```

Linux/macOS：

```bash
docker exec -it redis-go-final redis-cli GET stock:sku:1001
```

你应该能解释：

- 为什么 `GET` 后在 Go 里判断，再 `DECR` 会超卖。
- Redis 扣减成功但数据库订单失败时怎么办。
- 为什么数据库仍然需要事务边界和唯一约束。

### 7. 订单事件日志

要求：

- 使用 Stream 写入订单事件：`stream:order-events`。
- 限制 Stream 长度。
- 事件字段至少包含 `type`、`order_id`、`user_id`。
- 说明事件重复消费时如何保证幂等。

验收方式：

```powershell
docker exec -it redis-go-final redis-cli XRANGE stream:order-events - + COUNT 5
```

Linux/macOS：

```bash
docker exec -it redis-go-final redis-cli XRANGE stream:order-events - + COUNT 5
```

你应该能解释：

- Stream 和 List 的区别。
- 为什么事件写入 Redis 不等于永久审计日志。
- 什么场景应该改用 Kafka 或 RabbitMQ。

## 五、注意点

### 1. 错误处理检查

你至少要区分四类错误：

- 缓存未命中：`redis.Nil`，通常回源数据库。
- Redis 网络错误：记录日志，可降级或失败。
- 数据格式错误：删除坏缓存，回源数据库。
- 业务错误：库存不足、权限不足、短码不存在。

```text
不要把 redis.Nil 当成系统异常，也不要把所有 Redis 错误都静默忽略。
```

### 2. 一致性检查

你要能说清楚每个功能的事实来源：

- 用户资料：数据库。
- 文章详情：数据库。
- 点赞记录：数据库或可靠事件，Redis 用于快速判断。
- 热门榜：Redis 可作为实时榜，但长期统计要有落库或离线计算。
- 短链接映射：数据库。
- 库存和订单：数据库订单是最终事实，Redis 用于并发扣减。

### 3. 生产安全检查

上线前至少检查：

- Redis 是否配置认证、网络隔离和最小权限。
- 连接池大小是否和服务实例数量匹配。
- 所有缓存 key 是否有 TTL 或明确的清理策略。
- 是否有大 key、热 key 排查办法。
- 是否有 Redis 错误日志和指标。
- 是否准备了降级策略。
- 是否避免在生产执行 `KEYS *`、`FLUSHALL`、`FLUSHDB`。

### 4. 数据库工程检查

Redis 练习不能绕开数据库工程原则：

- 用户输入不能拼接 SQL，要使用参数化查询。
- 表结构变更要使用迁移工具。
- 唯一性要靠数据库唯一约束兜底。
- 索引只服务真实查询，不为了“完整”乱加。
- 订单、库存、支付等关键状态要有事务边界。
- 失败补偿要可重试、可追踪、可回滚。

## 六、常见误区

- 误区：示例代码能跑，就等于项目能上线。  
  原因：上线还需要错误处理、日志、指标、降级、回滚和数据一致性设计。

- 误区：所有功能都用 Redis 做最终存储。  
  原因：Redis 通常承担加速和协作，重要业务事实仍要落数据库。

- 误区：只验证成功路径。  
  原因：真实项目更多问题出在 Redis 不可用、缓存脏数据、数据库写入失败、重复请求这些异常路径。

- 误区：练习时不写 key 规范。  
  原因：项目一复杂，缓存删除、问题排查和数据迁移都会变难。

- 误区：把复杂业务逻辑塞进 Lua。  
  原因：Lua 会阻塞 Redis 执行，复杂逻辑应该留在 Go 服务和数据库事务里。

## 七、本节达标标准

- 能独立搭建 Redis-Go 练习项目。
- 能完成用户、文章、短链接、权限、库存、日志等核心场景。
- 能为每类 Redis key 设计命名和 TTL。
- 能区分缓存未命中、Redis 故障、业务错误和数据格式错误。
- 能说明每个功能的数据库事实来源。
- 能给出 Redis 故障时的降级或失败策略。
- 能列出上线前的生产安全检查项。
- 能把最终项目讲清楚，而不只是让代码跑起来。
