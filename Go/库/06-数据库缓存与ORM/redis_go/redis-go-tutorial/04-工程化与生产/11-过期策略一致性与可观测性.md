# 11. 过期策略一致性与可观测性

本节目标：你将能为 Redis key 设计过期策略，并知道上线后如何观察 Redis 是否健康。

简短引入：Redis 用得久了，问题往往不是某条命令不会写，而是 key 乱、TTL 乱、错误没日志、慢查询没人看。工程化的重点是让系统可控、可查、可回滚。

## 一、为什么需要它

真实项目里，Redis 常见故障不是突然出现的，而是慢慢积累：

- 没有 TTL 的缓存越来越多。
- key 命名混乱，无法定位业务。
- 大量 key 同时过期，数据库瞬间被打满。
- Redis 超时了，但应用日志里没有关键信息。

```text
缓存不是写进去就结束，必须考虑过期、删除、监控和回滚。
```

## 二、基本用法

设置带抖动的 TTL：

```go
package main

import (
	"context"
	"fmt"
	"math/rand"
	"time"

	"github.com/redis/go-redis/v9"
)

func main() {
	ctx := context.Background()
	rdb := redis.NewClient(&redis.Options{Addr: "localhost:6379"})
	defer rdb.Close()

	key := "article:detail:1001"
	ttl := 10*time.Minute + time.Duration(rand.Intn(120))*time.Second

	if err := rdb.Set(ctx, key, "cached article json", ttl).Err(); err != nil {
		fmt.Println("set cache failed:", err)
		return
	}

	left, err := rdb.TTL(ctx, key).Result()
	if err != nil {
		fmt.Println("read ttl failed:", err)
		return
	}

	fmt.Println("ttl:", left)
}
```

运行：

Windows PowerShell：

```powershell
go run .
```

Linux/macOS：

```bash
go run .
```

## 三、关键参数/语法/代码结构

常见 TTL 策略：

- 用户资料缓存：10 到 30 分钟。
- 文章详情缓存：5 到 30 分钟。
- 验证码：3 到 5 分钟。
- 登录 session：按安全策略设置，例如 2 小时或更短。
- 空值缓存：30 秒到 2 分钟。
- 分布式锁：必须有短 TTL，避免死锁。

Key 命名建议统一封装：

```go
func ArticleDetailKey(articleID int64) string {
	return fmt.Sprintf("article:detail:%d", articleID)
}

func UserProfileKey(userID int64) string {
	return fmt.Sprintf("user:profile:%d", userID)
}
```

不要在项目里到处手写 key 字符串。统一封装后，删除缓存、排查问题、迁移命名都更容易。

## 四、真实后端场景示例

文章更新后的保守处理：

```go
func UpdateArticle(ctx context.Context, rdb *redis.Client, articleID int64, title string) error {
	if err := updateArticleInDB(ctx, articleID, title); err != nil {
		return err
	}

	key := ArticleDetailKey(articleID)
	if err := rdb.Del(ctx, key).Err(); err != nil {
		logCacheError("delete article cache failed", key, err)
	}

	return nil
}

func updateArticleInDB(ctx context.Context, articleID int64, title string) error {
	return nil
}

func logCacheError(message string, key string, err error) {
	fmt.Printf("%s key=%s err=%v\n", message, key, err)
}
```

这里先更新数据库，再删除缓存。删除失败要记录日志，必要时进入补偿任务。

## 五、注意点

上线前建议至少观察：

- Redis 命令耗时。
- Redis 错误数量。
- 缓存命中率。
- 连接池使用情况。
- Redis 内存使用量。
- 慢查询。
- 大 key 和热 key。

生产环境不要随意运行：

```text
KEYS *
FLUSHALL
FLUSHDB
```

排查 key 时优先用 `SCAN` 分批扫描，并限制范围。

## 六、常见误区

- 误区：所有缓存 TTL 设置成一样。  
  原因：容易同一时间大量过期，引发数据库压力。

- 误区：key 到处手写。  
  原因：删除缓存时容易漏，排查也困难。

- 误区：Redis 报错不打日志。  
  原因：线上只看到数据库变慢，却不知道缓存已经失效。

- 误区：生产用 `KEYS *` 查问题。  
  原因：可能阻塞 Redis，影响线上请求。

## 七、本节达标标准

- 能为不同业务 key 设置合理 TTL。
- 能使用随机抖动降低同时过期风险。
- 能统一封装 key 命名。
- 能说出 Redis 上线后要观察哪些指标。
- 能识别生产高风险命令。
