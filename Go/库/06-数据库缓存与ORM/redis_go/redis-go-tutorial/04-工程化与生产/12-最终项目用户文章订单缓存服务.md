# 12. 最终项目：用户文章订单缓存服务

本节目标：你将能把前面学到的 Redis-Go 能力整合成一个小型后端项目设计。

简短引入：学 Redis-Go 的最终目标不是会几条命令，而是能把它放进一个真实后端系统里，知道哪些数据放 Redis，哪些数据必须落数据库，哪里需要降级和补偿。

## 一、为什么需要它

最终项目建议做一个“用户文章订单缓存服务”，覆盖常见后端能力：

- 用户资料缓存。
- 文章详情缓存。
- 文章点赞去重。
- 热门文章排行榜。
- 短链接跳转缓存。
- 库存扣减。
- 订单事件日志。

这不是大项目，但足够训练 Redis 在后端里的主要用法。

```text
项目练习的目标是形成判断力：哪些地方该用 Redis，哪些地方不能只靠 Redis。
```

## 三、关键参数/语法/代码结构

`internal/cache/redis.go`：

```go
package cache

import (
	"context"
	"time"

	"github.com/redis/go-redis/v9"
)

func NewRedis(addr string) (*redis.Client, error) {
	rdb := redis.NewClient(&redis.Options{
		Addr:         addr,
		PoolSize:     20,
		MinIdleConns: 5,
		DialTimeout:  3 * time.Second,
		ReadTimeout:  2 * time.Second,
		WriteTimeout: 2 * time.Second,
	})

	ctx, cancel := context.WithTimeout(context.Background(), 3*time.Second)
	defer cancel()

	if err := rdb.Ping(ctx).Err(); err != nil {
		_ = rdb.Close()
		return nil, err
	}

	return rdb, nil
}
```

## 四、真实后端场景示例

文章点赞服务：

```go
package article

import (
	"context"
	"fmt"

	"redis-go-app/internal/cache"

	"github.com/redis/go-redis/v9"
)

type Service struct {
	rdb *redis.Client
}

func NewService(rdb *redis.Client) *Service {
	return &Service{rdb: rdb}
}

func (s *Service) Like(ctx context.Context, articleID, userID int64) (bool, error) {
	likeKey := cache.ArticleLikesKey(articleID)
	rankKey := cache.HotArticleRankKey()

	added, err := s.rdb.SAdd(ctx, likeKey, fmt.Sprintf("%d", userID)).Result()
	if err != nil {
		return false, err
	}
	if added == 0 {
		return false, nil
	}

	if err := s.rdb.ZIncrBy(ctx, rankKey, 10, fmt.Sprintf("article:%d", articleID)).Err(); err != nil {
		return false, err
	}

	return true, nil
}
```

订单事件写入 Stream：

```go
func PublishOrderEvent(ctx context.Context, rdb *redis.Client, orderID string, userID int64) error {
	_, err := rdb.XAdd(ctx, &redis.XAddArgs{
		Stream: "stream:order-events",
		MaxLen: 10000,
		Approx: true,
		Values: map[string]any{
			"type":     "order_created",
			"order_id": orderID,
			"user_id":  userID,
		},
	}).Result()
	return err
}
```

真实项目中，点赞和订单事件都应考虑数据库落库。Redis 用于加速和异步协作，不能替代所有持久化。

## 五、注意点

项目落地时建议按这个顺序做：

1. 先写数据库表和迁移。
2. 写不依赖 Redis 的核心业务。
3. 再加入 Redis 缓存。
4. 加入日志和指标。
5. 压测热点接口。
6. 准备 Redis 故障时的降级策略。

迁移工具可以选择 `golang-migrate`、`goose` 或团队已有工具。不要靠手工改生产库。

数据库访问必须使用参数化查询或 ORM 的安全 API。索引要为真实查询服务，不要为了“看起来完整”乱加索引。

## 六、常见误区

- 误区：项目一开始就把所有数据放 Redis。  
  原因：会模糊事实来源，后期一致性很难处理。

- 误区：Redis 出错就整个服务不可用。  
  原因：缓存场景应尽量降级，但关键 Redis 依赖要明确失败策略。

- 误区：没有 key 规范。  
  原因：项目变大后，删除缓存和排查线上问题会非常困难。

- 误区：不写迁移，直接手改数据库。  
  原因：无法回滚、无法复现、多人协作容易出事故。

- 误区：不做错误分类。  
  原因：缓存未命中、网络失败、业务失败会被混在一起。

## 七、本节达标标准

- 能设计一个包含 Redis 的小型 Go 后端项目结构。
- 能统一封装 Redis 客户端和 key。
- 能把 String、Hash、Set、ZSet、Stream 用在合适业务里。
- 能说明 Redis 与数据库的职责边界。
- 能列出上线前的缓存、迁移、错误处理和回滚检查项。
