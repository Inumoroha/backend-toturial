# 8. 短链接服务设计

本节目标：你将能用 Redis 实现短链接跳转缓存，并理解短码唯一性、过期和回源数据库的边界。

简短引入：短链接是 Redis 很适合练习的场景。请求路径短、读多写少、对延迟敏感，而且非常贴近真实后端项目。

## 一、为什么需要它

短链接服务通常有两个动作：

- 创建短链接：长 URL 生成短码，保存映射。
- 访问短链接：通过短码找到长 URL，然后跳转。

访问短链接是高频读操作，适合用 Redis 缓存。

```text
短链接的唯一映射要落数据库，Redis 负责加速跳转。
```

## 二、基本用法

```go
package main

import (
	"context"
	"errors"
	"fmt"
	"log"
	"time"

	"github.com/redis/go-redis/v9"
)

var ErrShortCodeNotFound = errors.New("short code not found")

func CacheShortLink(ctx context.Context, rdb *redis.Client, code string, longURL string) error {
	key := fmt.Sprintf("shortlink:%s", code)
	return rdb.Set(ctx, key, longURL, 24*time.Hour).Err()
}

func ResolveShortLink(ctx context.Context, rdb *redis.Client, code string) (string, error) {
	key := fmt.Sprintf("shortlink:%s", code)

	longURL, err := rdb.Get(ctx, key).Result()
	if err == nil {
		return longURL, nil
	}
	if err != nil && !errors.Is(err, redis.Nil) {
		log.Printf("read short link cache failed: %v", err)
	}

	longURL, err = queryShortLinkFromDB(ctx, code)
	if err != nil {
		return "", err
	}

	_ = rdb.Set(ctx, key, longURL, 24*time.Hour).Err()
	return longURL, nil
}

func queryShortLinkFromDB(ctx context.Context, code string) (string, error) {
	if code != "goredis1" {
		return "", ErrShortCodeNotFound
	}
	return "https://example.com/articles/1001", nil
}

func main() {
	ctx := context.Background()
	rdb := redis.NewClient(&redis.Options{Addr: "localhost:6379"})
	defer rdb.Close()

	code := "goredis1"
	longURL := "https://example.com/articles/1001"

	if err := CacheShortLink(ctx, rdb, code, longURL); err != nil {
		log.Fatal(err)
	}

	got, err := ResolveShortLink(ctx, rdb, code)
	if err != nil {
		log.Fatal(err)
	}

	fmt.Println("redirect to:", got)
}
```

运行：

Windows PowerShell：

```powershell
go run .
```

Linux/macOS：

```bash
go run .
```

## 三、关键参数/语法/代码结构

`CacheShortLink` 负责把短码和长 URL 写入 Redis，适合在创建短链接成功后调用。

`ResolveShortLink` 是访问短链接时的查询入口。它先查 Redis，未命中再查数据库，最后把数据库结果回填到 Redis。

`queryShortLinkFromDB` 在示例里只是模拟数据库查询。真实项目中应该使用参数化查询，并依赖数据库唯一约束保存短码。

创建短码时，数据库层应该对 `code` 加唯一约束。Redis 的 `SetNX` 可以辅助防并发，但不能替代数据库唯一约束。

## 四、真实后端场景示例

创建短链接的保守流程：

1. 校验长 URL 格式，只允许 `http` 或 `https`。
2. 生成短码。
3. 写数据库，依赖唯一约束处理冲突。
4. 数据库写入成功后写 Redis 缓存。
5. 返回短链接。

访问短链接的保守流程：

1. 校验短码字符集和长度。
2. 查 Redis。
3. 未命中查数据库。
4. 命中数据库后回填 Redis。
5. 记录访问日志，跳转。

```text
短码来自用户请求路径，也要校验，不要直接拼到 SQL 或日志格式里。
```

## 五、注意点

短链接缓存可以设置较长 TTL，但数据库仍是最终来源。

如果短链接支持过期，Redis TTL 要和数据库里的过期时间保持一致。不要只让 Redis 过期，数据库还显示有效。

如果短链接支持删除，删除数据库记录后要删除 Redis key。

访问日志不建议每次同步写数据库，可以写入 Stream 或消息队列，再异步落库。

## 六、常见误区

- 误区：短链接只存 Redis。  
  原因：Redis 数据可能被淘汰或丢失，链接映射需要长期保存。

- 误区：用 `SetNX` 就不用数据库唯一约束。  
  原因：Redis 与数据库之间仍可能出现不一致。

- 误区：不校验长 URL。  
  原因：可能变成开放跳转风险。

- 误区：访问日志同步写入主链路。  
  原因：日志慢会拖慢跳转接口。

## 七、本节达标标准

- 能用 Redis 缓存短码到长 URL 的映射。
- 能设计短链接回源数据库流程。
- 能说明为什么短码唯一性要靠数据库约束兜底。
- 能处理短链接过期和删除缓存。
- 能识别开放跳转和日志拖慢主链路的风险。
