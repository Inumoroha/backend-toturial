# 7. 缓存文章详情与防穿透

本节目标：你将能实现文章详情缓存，并处理不存在文章被反复请求造成的缓存穿透。

简短引入：文章详情页是典型读多写少场景。缓存能明显减轻数据库压力，但如果只会简单 `Get/Set`，遇到不存在的数据、热点 key 和批量过期时就容易出问题。

## 一、为什么需要它

假设接口是：

```text
GET /articles/{id}
```

正常情况下，文章存在，第一次查数据库，后续走 Redis。

但如果有人反复请求不存在的文章 ID，比如 `999999999`，Redis 永远查不到，每次都会打到数据库。这就是缓存穿透。

```text
缓存未命中不一定是异常，可能是正常回源，也可能是攻击入口。
```

## 二、基本用法

下面用一个可运行示例模拟文章查询：

```go
package main

import (
	"context"
	"encoding/json"
	"errors"
	"fmt"
	"log"
	"time"

	"github.com/redis/go-redis/v9"
)

type Article struct {
	ID    int64  `json:"id"`
	Title string `json:"title"`
	Body  string `json:"body"`
}

var ErrNotFound = errors.New("article not found")

func GetArticle(ctx context.Context, rdb *redis.Client, articleID int64) (Article, error) {
	key := fmt.Sprintf("article:detail:%d", articleID)

	raw, err := rdb.Get(ctx, key).Bytes()
	if err == nil {
		if string(raw) == "{}" {
			return Article{}, ErrNotFound
		}
		var article Article
		if err := json.Unmarshal(raw, &article); err == nil {
			return article, nil
		}
		_ = rdb.Del(ctx, key).Err()
	}

	if err != nil && !errors.Is(err, redis.Nil) {
		log.Printf("read article cache failed: %v", err)
	}

	article, err := queryArticleFromDB(ctx, articleID)
	if errors.Is(err, ErrNotFound) {
		_ = rdb.Set(ctx, key, "{}", 1*time.Minute).Err()
		return Article{}, ErrNotFound
	}
	if err != nil {
		return Article{}, err
	}

	raw, err = json.Marshal(article)
	if err == nil {
		_ = rdb.Set(ctx, key, raw, 10*time.Minute).Err()
	}

	return article, nil
}

func queryArticleFromDB(ctx context.Context, articleID int64) (Article, error) {
	if articleID != 1001 {
		return Article{}, ErrNotFound
	}
	return Article{ID: 1001, Title: "Redis-Go 实战", Body: "文章内容..."}, nil
}

func main() {
	ctx := context.Background()
	rdb := redis.NewClient(&redis.Options{Addr: "localhost:6379"})
	defer rdb.Close()

	article, err := GetArticle(ctx, rdb, 1001)
	if err != nil {
		log.Fatal(err)
	}
	fmt.Printf("%+v\n", article)
}
```

运行：

Windows PowerShell：

```powershell
go run .
```

Linux/macOS：

```bash
go run .
```

## 三、关键参数/语法/代码结构

`GetArticle` 是这一节的核心函数。它先查 Redis，命中后直接反序列化返回；未命中时再调用 `queryArticleFromDB` 模拟查数据库。

`redis.Nil` 表示缓存里没有这个 key。它不是系统异常，真实项目中通常继续回源数据库。

`json.Unmarshal` 失败时删除缓存，是为了处理旧字段、坏数据或版本变更留下的脏缓存。

这里把不存在的文章短暂缓存为 `"{}"`，叫空值缓存。它能挡住同一个不存在 ID 的重复请求。

## 四、真实后端场景示例

文章详情接口中常见流程：

1. 校验 articleID 格式和范围。
2. 查 Redis。
3. 命中则返回。
4. 未命中则查数据库。
5. 数据库不存在时写短 TTL 空值缓存。
6. 数据库存在时写正常 TTL 缓存。
7. 文章更新或删除时删除缓存。

```text
更新数据库后优先删除缓存，不要试图在所有场景里手动同步缓存内容。
```

## 五、注意点

空值缓存 TTL 要短。否则刚创建的新文章可能因为旧的空值缓存而短时间不可见。

文章 ID 要先做基础校验。比如负数、过长字符串、明显非法参数，不应该进入数据库查询。

缓存击穿是另一个问题：某个热点文章过期后，很多请求同时回源数据库。常见处理方式是互斥锁、热点 key 延长 TTL、后台刷新。初学阶段先理解风险，不急着堆复杂方案。

## 六、常见误区

- 误区：不存在的数据不需要缓存。  
  原因：不存在的 ID 被重复请求时会持续打数据库。

- 误区：空值缓存 TTL 很长。  
  原因：会影响新数据可见性。

- 误区：数据库更新后只更新缓存。  
  原因：更新缓存失败会留下脏数据；通常先更新数据库，再删除缓存。

- 误区：所有缓存错误都返回 500。  
  原因：文章详情缓存失败时，很多项目可以降级查数据库。
