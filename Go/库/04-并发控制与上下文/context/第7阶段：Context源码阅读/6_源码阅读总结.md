# 6. 源码阅读总结

本节目标：把本阶段源码概念串起来，形成一张完整的心智图。

---

## 一、四种核心结构

可以这样记：

```text
emptyCtx：根 context。
cancelCtx：支持取消。
timerCtx：支持 deadline / timeout。
valueCtx：支持 key-value。
```

它们都实现 `Context` 接口。

---

## 二、API 和结构的对应关系

```text
Background / TODO -> emptyCtx
WithCancel -> cancelCtx
WithTimeout / WithDeadline -> timerCtx
WithValue -> valueCtx
```

这不是要求你背源码，而是帮助你理解每个 API 背后创建了什么能力。

---

## 三、取消流程

可以粗略理解为：

```text
调用 cancel
  -> 记录 err
  -> 关闭 Done channel
  -> 取消所有 children
  -> 从父 context 移除自己
```

父取消会向下传播。

子取消不会向上传播。

---

## 四、超时流程

```text
WithTimeout
  -> 计算 deadline
  -> 创建 timerCtx
  -> timer 到期
  -> 调用 cancel
  -> Err 返回 DeadlineExceeded
```

手动 `cancel()` 可以提前释放 timer。

---

## 五、取值流程

```text
ctx.Value(key)
  -> 当前 valueCtx 是否匹配
  -> 不匹配就找 parent
  -> 一直找到根 context
```

所以 value 适合少量请求级元数据，不适合大量业务数据。

## 七、把源码知识用于代码 Review

读源码不是为了炫技，而是为了在工程中做出更好的判断。

看到下面代码：

```go
ctx, cancel := context.WithTimeout(context.Background(), time.Second)
```

如果它出现在 HTTP handler 的下游函数里，你应该能指出问题：

```text
它切断了上游请求 context。
客户端断开连接时，这个子任务不会自动取消。
```

看到下面代码：

```go
ctx = context.WithValue(ctx, "user_id", userID)
```

你应该想到 `valueCtx` 的链式结构和隐式依赖问题：

```text
如果 userID 是业务必需参数，应该写进函数签名。
```

看到下面代码：

```go
ctx, cancel := context.WithTimeout(ctx, time.Second)
do(ctx)
```

你应该提醒：

```text
缺少 defer cancel()。
```

## 八、源码知识和性能直觉

读完源码后，你会更容易理解这些经验：

### 1. 不要无意义派生 context

每次 `WithCancel`、`WithTimeout`、`WithValue` 都会创建新的包装对象。

不是说它很重，而是没有必要时不要乱建。

### 2. 不要塞太多 Value

`valueCtx` 是链式查找。

少量 request id、trace id 没问题。

大量业务参数会让性能和可维护性都变差。

### 3. 及时 cancel 是资源管理

`timerCtx` 持有 timer。

及时 cancel 可以释放 timer，并从父 context 中解除子节点关系。

## 九、源码复盘表

可以用这张表复习：

| API | 内部结构 | 增加的能力 | 常见错误 |
| --- | --- | --- | --- |
| `Background` | `emptyCtx` | 根 context | 在请求链路中乱用 |
| `TODO` | `emptyCtx` | 临时占位 | 长期不处理 |
| `WithCancel` | `cancelCtx` | 主动取消 | 创建后不调用 cancel |
| `WithTimeout` | `timerCtx` | 持续时间超时 | 超时过短或忘记 cancel |
| `WithDeadline` | `timerCtx` | 具体截止时间 | 不考虑父 deadline |
| `WithValue` | `valueCtx` | 请求级元数据 | 滥用成参数容器 |
