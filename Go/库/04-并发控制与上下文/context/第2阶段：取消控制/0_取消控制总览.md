# 0. 取消控制总览

本阶段目标：掌握 `context.WithCancel`，理解父子 context 的取消传播，并能用它避免 goroutine 泄漏。

取消控制是 `context` 最核心的能力之一。

它解决的问题是：

```text
当一个任务已经不需要继续时，如何通知正在工作的 goroutine 停下来？
```

## 二、取消不是强制终止

调用 `cancel()` 不会杀死 goroutine。

它只是关闭 `ctx.Done()`：

```go
ctx, cancel := context.WithCancel(context.Background())
cancel()
```

如果 goroutine 内部监听了：

```go
case <-ctx.Done():
	return
```

它就会退出。

如果它完全不检查 `ctx.Done()`，那就不会退出。

所以 context 是协作式取消。

---

## 三、取消的典型场景

后端项目中常见取消场景：

- HTTP 客户端断开连接。
- 一个并发任务失败，取消其他任务。
- 用户主动取消操作。
- 服务关闭，通知 worker 退出。
- 上游请求已经超时，下游任务没有必要继续。

---

## 四、你需要形成的习惯

只要函数可能耗时、阻塞、循环或启动 goroutine，就应该考虑：

```text
它是否需要接收 context？
它是否需要监听 ctx.Done()？
它取消后应该返回什么错误？
```

这不是形式主义，而是服务稳定性的基础。

## 六、本阶段核心图

可以把取消控制理解为：

```text
创建可取消 ctx
  -> 把 ctx 传给任务
  -> 任务监听 ctx.Done
  -> 调用 cancel
  -> 任务退出
```

缺任何一步都不完整。

例如只调用 cancel，但任务不监听 Done，任务不会退出。

---

## 七、取消控制的典型代码形状

```go
ctx, cancel := context.WithCancel(parent)
defer cancel()

go func() {
	for {
		select {
		case <-ctx.Done():
			return
		default:
			doWork()
		}
	}
}()
```

后续你会在 worker、HTTP 聚合、服务关闭中反复看到这种形状。

---

## 八、取消和超时的关系

超时是自动取消。

主动取消是手动调用 cancel。

两者都会关闭 `Done()`，但错误原因不同：

```text
主动取消：context.Canceled
超时：context.DeadlineExceeded
```

本阶段先学主动取消，下一阶段再学超时。

## 九、本阶段完成标准

你应该能独立写出：

```text
启动多个 worker。
运行一段时间。
调用 cancel。
所有 worker 收到信号后退出。
main 等待 worker 退出。
```

如果 main 不等待 worker，只是 Sleep 一下，也能演示，但不够工程化。

## 十、取消控制和资源释放

取消不只是为了让日志好看。

它经常意味着释放资源：

```text
停止 goroutine。
停止等待 channel。
停止下游 HTTP 请求。
停止数据库查询。
关闭临时任务。
释放 timer。
```

如果任务不响应取消，资源就会继续占用。

在高并发服务中，这类小问题会逐渐累积，最后表现为：

- goroutine 数量上涨。
- 请求越来越慢。
- 连接池耗尽。
- CPU 或内存异常。
