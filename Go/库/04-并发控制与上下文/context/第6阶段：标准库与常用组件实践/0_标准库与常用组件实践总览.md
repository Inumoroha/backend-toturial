# 0. 标准库与常用组件实践总览

本阶段目标：把 `context` 放到真实 Go 后端组件中理解，而不是停留在 demo。

你会重点学习：

- `net/http` 服务端和客户端。
- `database/sql` 和 `pgx`。
- `os/signal` 的优雅关闭。
- `errgroup` 的并发协同取消。

---

## 一、为什么要学组件实践

`context` 的价值不在于自己创建一个 `ctx`，而在于整个生态都认它。

很多 Go 标准库和第三方库都会接收 context：

```go
http.NewRequestWithContext(ctx, ...)
db.QueryContext(ctx, ...)
signal.NotifyContext(...)
errgroup.WithContext(ctx)
```

这意味着你可以用同一套机制，把请求生命周期传到很多地方。

## 五、组件实践的共同原则

无论是 HTTP、数据库、signal 还是 errgroup，核心原则都是一样的：

```text
上游 ctx 向下传。
下游调用设置边界。
耗时任务监听 Done。
取消时返回 ctx.Err。
派生 ctx 后调用 cancel。
```

如果某个库提供了带 context 的方法，优先使用带 context 的版本。

例如：

```text
QueryContext 优于 Query。
NewRequestWithContext 优于 NewRequest。
BeginTx 优于 Begin。
```

这不是形式上的差异，而是决定你的服务能不能在取消和超时时释放资源。

## 六、本阶段常见错误

### 1. 只在 handler 里拿 ctx，但下游不用

```go
ctx := r.Context()
_ = ctx
repo.FindByID(id)
```

这样 context 没有真正进入调用链。

### 2. 数据库用了不带 context 的方法

```go
db.Query("SELECT ...")
```

更推荐：

```go
db.QueryContext(ctx, "SELECT ...")
```

### 3. errgroup 的任务不监听 ctx

`errgroup.WithContext` 会取消 ctx，但任务不监听就不会及时退出。

### 4. Shutdown 使用已取消的 signal ctx

收到退出信号后，signal ctx 已经取消。`server.Shutdown` 应该使用新的 timeout context。

## 七、本阶段学习检查

每学完一个组件，都检查：

```text
这个组件如何接收 ctx？
ctx 被取消时它会发生什么？
是否需要额外调用 Close、Rollback、Release？
错误如何判断是超时还是普通失败？
```

能回答这四个问题，才算真正会用 context 接这个组件。

---

## 八、组件实践的学习顺序

建议不要同时学所有组件。

推荐顺序：

```text
先 HTTP 服务端：理解 r.Context。
再 HTTP 客户端：理解 NewRequestWithContext。
再数据库：理解 QueryContext。
再 signal：理解进程级取消。
最后 errgroup：理解多个 goroutine 协同取消。
```

这个顺序从单请求到多任务，再到整个进程，范围逐步扩大。

---

## 九、组件接入时的统一检查

每接一个新库，都检查文档里有没有：

```text
Context 参数。
WithContext 方法。
带 Context 后缀的方法。
取消后的错误表现。
资源关闭方法。
```

例如数据库有 `QueryContext`，HTTP 有 `NewRequestWithContext`，gRPC 有 deadline 和 metadata。

不同库名字不一样，但思想一样。
