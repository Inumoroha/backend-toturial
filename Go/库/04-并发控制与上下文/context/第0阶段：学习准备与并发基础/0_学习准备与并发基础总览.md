# 0. 学习准备与并发基础总览

本阶段目标：在正式学习 `context.Context` 之前，先补齐必要的 Go 后端基础和并发直觉。

`context` 不是一个孤立知识点。它经常和下面这些东西一起出现：

```text
goroutine
channel
select
HTTP 请求
数据库查询
RPC 调用
服务优雅关闭
```

如果只背 API，很容易知道 `WithTimeout` 怎么写，却不知道什么时候该用、为什么要 `defer cancel()`、为什么 goroutine 要监听 `ctx.Done()`。

所以这一阶段先做准备。

---

## 一、本阶段要解决什么问题

你需要先建立三个直觉：

### 1. 后端请求有生命周期

一次请求不是无限存在的。

它可能因为下面这些原因结束：

- 客户端正常收到响应。
- 客户端主动断开连接。
- 服务端超时。
- 上游网关超时。
- 服务正在关闭。
- 某个下游依赖失败。

`context` 的核心价值，就是把这种“请求已经不需要继续了”的信息传递给下游代码。

### 2. goroutine 不会自动退出

启动一个 goroutine 很容易：

```go
go doSomething()
```

但它什么时候退出？

如果里面一直阻塞、一直循环、一直等待 channel，它可能永远不退出。这就是 goroutine 泄漏。

`context` 常用来通知 goroutine：

```text
任务取消了，你可以停了。
```

### 3. 外部调用必须有边界

后端服务会调用很多外部资源：

- 数据库。
- Redis。
- 其他 HTTP 服务。
- gRPC 服务。
- 消息队列。

这些调用都可能慢、卡住或失败。

没有超时控制，请求就可能一直挂住，占着 goroutine、连接、内存和数据库资源。

## 四、本阶段学习方法

每篇教程建议这样学：

1. 先读概念。
2. 手敲代码。
3. 运行代码。
4. 改几个参数观察结果。
5. 写下你自己的理解。

比如超时时间从 1 秒改成 5 秒，任务耗时从 2 秒改成 500 毫秒，你会立刻看到结果不同。

## 六、为什么把并发基础放在 Context 前面

`context` 的 API 很少，但它背后的模型并不浅。

例如：

```go
select {
case <-ctx.Done():
	return ctx.Err()
case result := <-resultCh:
	return result
}
```

如果你不理解 channel 和 select，这段代码就只是固定模板。

但如果你理解了 channel，马上就能看出：

```text
ctx.Done() 是一个取消信号。
resultCh 是一个业务结果。
select 表示谁先来就处理谁。
```

这就是为什么先学 goroutine、channel、select。

## 九、常见学习误区

### 1. 先背源码

不用一开始读源码。先会用，再读源码。

### 2. 只看文章不敲代码

并发代码必须运行观察。

### 3. 把 context 当成全局 map

这会在后面的 `Value` 阶段重点纠正。

### 4. 认为 cancel 会杀死 goroutine

不会。goroutine 必须自己监听取消信号。

## 十、本阶段额外练习

请写三个小程序：

1. 一个 goroutine 通过 channel 返回结果。
2. 一个 goroutine 通过 done channel 被通知停止。

这三个程序不用复杂，但必须能独立写出来。

如果这些还不熟，不要急着进入后面的阶段。`context` 不是难在 API，而是难在它背后的工程场景。
