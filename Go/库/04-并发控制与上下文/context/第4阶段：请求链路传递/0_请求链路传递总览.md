# 0. 请求链路传递总览

本阶段目标：掌握在真实 Go 后端分层代码中传递 `context` 的方式。

前面你已经学过取消和超时。

但在真实项目中，context 不是只在一个函数里使用，而是要从请求入口一路传到下游。

典型链路：

```text
HTTP Handler
  -> Middleware
  -> Service
  -> Repository
  -> Database / Redis / RPC
```

## 二、核心原则

一句话：

```text
从入口拿 ctx，向下游传 ctx。
```

HTTP 服务中入口通常是：

```go
ctx := r.Context()
```

然后传给 service：

```go
user, err := h.userService.GetUser(ctx, userID)
```

service 继续传给 repository：

```go
return s.userRepo.FindByID(ctx, userID)
```

repository 传给数据库：

```go
row := db.QueryRowContext(ctx, sql, userID)
```

---

## 三、不要在中间层重新创建根 Context

不推荐：

```go
func (s *UserService) GetUser(id int64) (*User, error) {
	ctx := context.Background()
	return s.repo.FindByID(ctx, id)
}
```

这样会切断请求的取消、超时和元数据。

推荐：

```go
func (s *UserService) GetUser(ctx context.Context, id int64) (*User, error) {
	return s.repo.FindByID(ctx, id)
}
```

## 四、本阶段达标标准

学完本阶段后，你应该能够做到：

- 在 HTTP handler 中获取 `r.Context()`。
- 在 service、repository 中把 `ctx context.Context` 作为第一个参数。
- 避免在底层使用 `context.Background()`。
- 使用中间件向 request context 添加 request id。
- 理解为什么 context 不应该放进结构体字段。

## 五、本阶段统一项目结构

建议用一个用户查询接口贯穿本阶段：

```text
GET /users/1001
```

调用链：

```text
UserHandler.GetUser
  -> UserService.GetUser
  -> UserRepo.FindByID
```

context 流向：

```text
r.Context()
  -> service
  -> repository
  -> QueryContext 或模拟查询
```

这个小例子足够覆盖请求链路的核心问题。

## 六、为什么分层传递很重要

如果只有 handler 使用 context，底层不用，那么请求取消时：

```text
handler 知道取消了。
service 不知道。
repository 不知道。
数据库查询还在跑。
```

这就失去了 context 的意义。

真正有效的链路应该是：

```text
每一层都接收 ctx。
每一层都把 ctx 传给可能耗时的下游。
必要时为具体下游派生更短 timeout。
```

---

## 七、常见错误预告

这一阶段你会重点避免：

- 把 `*http.Request` 传进 service。
- 在 service 里创建 `context.Background()`。
- 把 context 存进 service 结构体。
- 用 context 传业务参数。
- 中间件创建新 ctx 后忘记 `r.WithContext(ctx)`。

---

## 八、本阶段完成标准

你应该能不看教程写出：

```go
func (h *Handler) ServeHTTP(w http.ResponseWriter, r *http.Request) {
	ctx := r.Context()
	_ = h.service.Do(ctx)
}

func (s *Service) Do(ctx context.Context) error {
	return s.repo.Save(ctx)
}
```

并解释为什么不能在 repo 里重新创建 `Background()`。

---

## 九、请求链路里的两类数据

一条请求链路里通常有两类数据：

```text
业务数据：userID、input、page、orderID。
控制和元数据：ctx、request id、trace id、deadline。
```

业务数据应该通过普通参数传递。

控制和请求级元数据通过 context 传递。

这两类数据不要混在一起。

---

## 十、链路中每一层的责任

Handler：

```text
从 r.Context 获取 ctx。
解析 HTTP 参数。
调用 service。
把错误转换成 HTTP 响应。
```

Service：

```text
接收 ctx。
处理业务规则。
调用 repository 或 client。
不依赖 HTTP 类型。
```

Repository：

```text
接收 ctx。
设置必要的数据库子 timeout。
调用 QueryContext / ExecContext。
转换数据访问错误。
```

这样分层后，context 的流向也会自然清晰。
