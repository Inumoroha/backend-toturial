# 0. 工程规范与故障排查总览

本阶段目标：把 context 的知识转化成工程规范，并学会排查常见问题。

会写 API 不等于会在项目里用好。

真实项目更看重：

- 约定统一。
- 超时合理。
- 错误清晰。
- 日志可追踪。
- 不泄漏 goroutine。
- 不隐藏业务依赖。

## 二、本阶段达标标准

学完本阶段后，你应该能够做到：

- 排查常见 goroutine 泄漏和超时问题。

## 三、为什么需要单独学习工程规范

很多人学完 context API 后，代码仍然会出现这些问题：

```text
ctx 参数位置不统一。
下游函数偷偷使用 context.Background。
WithTimeout 后忘记 cancel。
日志里只有 context deadline exceeded，看不出哪个下游慢。
把 userID、db、repo 都塞进 context.Value。
goroutine 启动后没有退出条件。
```

这些问题不是语法问题，而是工程习惯问题。

所以这一阶段要把知识变成约定：

```text
怎么写。
怎么审。
怎么测。
怎么查。
```

## 五、本阶段统一示例场景

后续几节会反复使用这个场景：

```text
GET /profile/{user_id}
  -> 查询用户数据库
  -> 调用订单服务
  -> 调用钱包服务
```

我们关心：

- 整个接口最多等多久。
- 每个下游最多等多久。
- 超时时日志怎么打。
- 错误怎么返回。
- 如何测试超时和取消。
- 如果线上超时变多，怎么排查。

## 六、工程化学习方法

这一阶段不要只看代码片段。

建议你每读一节，都问自己：

```text
这个规范能不能写进团队代码规范？
这个问题在线上怎么发现？
这个错误日志能不能支持排查？
这个测试能不能防止回归？
```

当你开始用这些问题看 context，说明你已经在从“会用 API”转向“会做后端工程”。

---

## 七、本阶段和前面阶段的关系

前面阶段解决的是“怎么写”：

```text
怎么 WithCancel。
怎么 WithTimeout。
怎么传 ctx。
怎么用 Value。
怎么用 errgroup。
```

本阶段解决的是“怎么长期维护”：

```text
团队里大家是否写法一致。
上线后超时能不能查。
错误日志能不能定位下游。
测试能不能覆盖取消和超时。
代码 review 能不能发现坏味道。
```

这两类能力都重要。

只会写 demo，不会排查线上问题，还不能算合格后端工程师。

---

## 八、建议建立自己的检查表

你可以为每个项目建立一份 context 检查表：

```text
所有外部调用是否传 ctx？
所有派生 ctx 是否 cancel？
所有 goroutine 是否有退出路径？
所有 timeout 是否有依据？
所有 context 错误是否保留根因？
所有日志是否能通过 request id 串联？
```

每次提交涉及并发、HTTP、数据库、RPC 的代码时，都过一遍。

这个习惯会减少很多隐藏问题。

---

## 九、本阶段实践任务

请选一个你前面写过的 demo，做一次“工程化改造”：

1. 统一所有函数的 ctx 参数位置。
2. 为外部调用补 timeout。
3. 为取消和超时补错误分类。
4. 为日志补 request id。
5. 为关键函数写超时测试。
6. 写一份排查清单。

这个任务比再写一个新 demo 更有价值，因为真实工作经常是改造旧代码。
