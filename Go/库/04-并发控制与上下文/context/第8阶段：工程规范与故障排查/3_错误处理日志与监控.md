# 3. 错误处理、日志与监控

本节目标：学会把 context 相关错误变成可观察、可排查的信息。

---

## 一、错误分类

常见分类：

```text
context.Canceled：请求取消。
context.DeadlineExceeded：超时。
业务错误：参数错误、资源不存在、状态冲突。
依赖错误：数据库、Redis、下游服务失败。
系统错误：panic、未知错误。
```

不同错误应该有不同日志级别和响应策略。

---

## 二、日志中带 request id

```go
func logError(ctx context.Context, msg string, err error) {
	requestID, _ := RequestIDFromContext(ctx)
	fmt.Printf("level=error request_id=%s msg=%s err=%v\n", requestID, msg, err)
}
```

真实项目中建议使用结构化日志。

request id 能帮助你串联一条请求的所有日志。

---

## 三、超时要记录下游名称和耗时

不够好的日志：

```text
timeout
```

更好的日志：

```text
request_id=req-1 dependency=user-service timeout=300ms err=context deadline exceeded
```

你需要知道到底哪个下游慢。

---

## 四、监控指标

建议关注：

- 请求总耗时。
- 超时错误数量。
- 取消错误数量。
- 下游调用耗时。
- 数据库查询耗时。
- goroutine 数量。
- 连接池占用。

如果超时增多，要结合日志和指标判断：

```text
是流量上涨？
是数据库慢？
是下游故障？
是超时设置太短？
```

---

## 五、本节达标标准

学完本节后，你应该能够做到：

- 区分取消、超时和普通错误。
- 在日志中带上 request id。
- 记录下游名称和 timeout 配置。
- 知道哪些指标能帮助排查 context 相关问题。

---

## 六、推荐错误处理函数

可以把 context 错误转换成统一响应：

```go
func writeError(ctx context.Context, w http.ResponseWriter, err error) {
	switch {
	case errors.Is(err, context.Canceled):
		logInfo(ctx, "request canceled", err)
		http.Error(w, "request canceled", http.StatusRequestTimeout)
	case errors.Is(err, context.DeadlineExceeded):
		logWarn(ctx, "request timeout", err)
		http.Error(w, "request timeout", http.StatusGatewayTimeout)
	default:
		logError(ctx, "internal error", err)
		http.Error(w, "internal error", http.StatusInternalServerError)
	}
}
```

实际项目可以结合业务错误类型进一步细分。

重点是不要把所有错误都当成 500，也不要把超时静默吞掉。

---

## 七、日志字段建议

context 相关日志建议至少包含：

```text
request_id
trace_id
path
method
dependency
timeout
cost
error
```

示例：

```text
level=warn request_id=req-1 path=/profile/1001 dependency=order-service timeout=800ms cost=801ms err="context deadline exceeded"
```

这条日志能回答：

```text
哪个请求？
哪个接口？
哪个下游？
配置超时多少？
实际耗时多少？
错误是什么？
```

---

## 八、context.Canceled 要小心解读

`context.Canceled` 不一定是 bug。

常见原因：

- 用户关闭浏览器页面。
- 前端切换路由取消请求。
- 移动网络中断。
- 上游网关先超时。
- errgroup 中某个任务失败后取消其他任务。

所以日志级别不一定是 error。

如果 canceled 数量突然暴增，需要结合路径、客户端、网关日志一起看。

---

## 九、DeadlineExceeded 更需要关注

`DeadlineExceeded` 通常说明某个环节超过预算。

排查顺序：

```text
看接口总耗时。
看是哪一个下游超时。
看下游自身是否变慢。
看数据库慢查询。
看连接池是否耗尽。
看最近是否发布变更。
看流量是否上涨。
```

不要只把 timeout 调大。

调大 timeout 可能只是把问题延后，并让资源占用更久。

---

## 十、监控指标设计

建议至少有这些指标：

```text
http_request_duration_seconds
http_request_total
dependency_request_duration_seconds
dependency_timeout_total
db_query_duration_seconds
db_query_timeout_total
goroutine_count
```

如果使用 Prometheus，通常可以按 label 区分：

```text
path
method
dependency
status
error_type
```

注意 label 不要放 userID、requestID 这种高基数字段。
