# 5. 常见问题排查清单

本节目标：整理 context 相关问题的排查思路。

---

## 一、请求明明取消了，下游还在跑

检查：

- 下游函数是否接收 ctx。
- 是否用了 `context.Background()` 切断链路。
- 是否调用了不带 context 的库方法。
- goroutine 内部是否监听 `ctx.Done()`。

---

## 二、接口经常超时

检查：

- 超时设置是否过短。
- 数据库是否慢查询。
- 下游服务是否变慢。
- 是否串行调用过多。
- 是否连接池耗尽。
- 是否有锁等待。

---

## 三、goroutine 数量不断上涨

检查：

- 后台循环是否有退出条件。
- channel 发送是否可能无人接收。
- worker 是否监听 ctx。
- 请求内启动的 goroutine 是否随请求结束。
- 是否有外部调用没有超时。

---

## 四、context canceled 日志很多

可能原因：

- 客户端主动断开。
- 网关超时先发生。
- 前端重复取消请求。
- 服务端主动取消并发任务。

不一定都是错误。

需要结合状态码、耗时、请求路径判断。

---

## 五、DeadlineExceeded 很多

可能原因：

- 下游服务慢。
- 数据库慢。
- 网络抖动。
- 超时预算过短。
- 重试导致总耗时增加。

建议同时看：

- request id 日志。
- 下游调用耗时。
- 数据库慢查询。
- 连接池状态。
- 服务 CPU 和内存。

---

## 六、本阶段总结

你现在应该具备工程化使用 context 的能力：

- 有统一代码规范。
- 有超时预算意识。
- 有错误分类和日志意识。
- 会写 context 相关测试。
- 有常见问题排查清单。

下一阶段进入项目实战，把所有内容串起来。

---

## 七、排查流程：接口超时

当线上出现接口超时时，不要马上改 timeout。

建议按这个顺序：

```text
1. 确认是哪个接口超时。
2. 查看 request id 对应日志。
3. 找到最慢的下游依赖。
4. 查看该依赖最近是否有错误或发布。
5. 查看数据库慢查询和连接池。
6. 查看服务 goroutine 数、CPU、内存。
7. 判断是依赖变慢、流量上涨、锁等待，还是 timeout 配置过短。
8. 最后再决定优化 SQL、扩容、降级、限流或调整 timeout。
```

调大 timeout 是最后选项之一，不是第一反应。

---

## 八、排查流程：goroutine 泄漏

如果 goroutine 数量持续上涨，可以检查：

```text
是否每个请求都启动 goroutine？
goroutine 是否监听 ctx.Done？
channel 发送是否可能阻塞？
是否有 time.After 在循环中大量创建？
是否有外部调用没有 timeout？
worker 是否能收到关闭信号？
```

如果服务支持 pprof，可以查看 goroutine dump：

```text
/debug/pprof/goroutine?debug=2
```

看大量 goroutine 卡在哪里。

---

## 九、排查流程：连接池耗尽

context 使用不当也可能导致连接池问题。

例如：

- 查询没有超时。
- `rows.Close()` 漏掉。
- 事务太长。
- 请求取消后数据库查询仍在跑。

检查点：

```text
数据库查询是否使用 ctx。
多行查询是否 defer rows.Close。
事务是否及时 Commit 或 Rollback。
连接池 acquired 是否长期接近 max。
慢查询是否增加。
```

---

## 十、排查流程：日志里只有 context canceled

如果日志只有：

```text
context canceled
```

信息太少。

应该补充：

```text
request_id
path
dependency
operation
cost
timeout
```

否则你无法判断这是用户取消、网关取消，还是内部 errgroup 取消。

```text

```
