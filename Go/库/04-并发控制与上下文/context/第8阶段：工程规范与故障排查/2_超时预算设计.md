# 2. 超时预算设计

本节目标：学习如何为一个接口设计合理的超时预算。

超时不是随便写一个数字。

好的超时设计要考虑：

- 用户能接受多久。
- 上游网关超时。
- 下游服务 SLA。
- 数据库查询复杂度。
- 是否串行或并发调用。
- 是否需要重试。

---

## 一、从入口总超时开始

假设网关给你的服务 2 秒。

你的接口最好不要也写 2 秒满额。

可以设置：

```text
接口总预算：1.5 秒
预留给网关和网络：0.5 秒
```

这样服务有机会及时返回错误，而不是让网关先断。

---

## 二、串行调用预算

如果接口串行调用：

```text
查询用户 -> 查询订单 -> 查询钱包
```

总预算 1 秒，可以分配：

```text
用户：200ms
订单：400ms
钱包：200ms
组装和预留：200ms
```

不能每个都写 1 秒。

---

## 三、并发调用预算

如果三个下游并发执行：

```text
fetchUser
fetchOrders
fetchWallet
```

总预算 1 秒时，每个下游可以有更接近总预算的时间，但仍建议设置自己的子超时。

例如：

```text
接口总超时：1s
用户服务：300ms
订单服务：800ms
钱包服务：500ms
```

这样某个慢下游不会拖住所有资源。

---

## 四、重试会吃掉预算

如果一次调用 timeout 是 500ms，重试 3 次，最坏就是 1.5 秒。

所以重试必须纳入总预算。

常见策略：

```text
总预算内重试。
每次重试前检查 ctx 是否已经快超时。
对非幂等操作谨慎重试。
```

---

## 五、本节达标标准

学完本节后，你应该能够做到：

- 从上游网关超时反推服务超时。
- 区分串行和并发调用预算。
- 为数据库、HTTP、RPC 设置不同 timeout。
- 知道重试会消耗总预算。

---

## 六、一个完整预算示例

假设网关超时是 3 秒。

你的 Go 服务应该给自己留余量：

```text
网关超时：3000ms
服务接口总预算：2200ms
预留网络和响应写出：800ms
```

接口内部：

```text
解析参数：20ms
认证鉴权：100ms
查询用户数据库：300ms
订单服务：800ms
钱包服务：500ms
组装响应：50ms
预留抖动：430ms
```

这不是固定公式，而是一种思考方式。

你要知道每一段大概能花多久，不能让所有下游都无限制等待。

---

## 七、串行调用的代码结构

```go
func (s *Service) GetProfile(ctx context.Context, userID int64) (*Profile, error) {
	ctx, cancel := context.WithTimeout(ctx, 1500*time.Millisecond)
	defer cancel()

	user, err := s.userRepo.FindByID(ctx, userID)
	if err != nil {
		return nil, err
	}

	orders, err := s.orderClient.ListRecent(ctx, userID)
	if err != nil {
		return nil, err
	}

	wallet, err := s.walletClient.Get(ctx, userID)
	if err != nil {
		return nil, err
	}

	return &Profile{User: user, Orders: orders, Wallet: wallet}, nil
}
```

串行调用时，总耗时是每一步累加。

所以每个下游方法内部最好再设置自己的子 timeout。

---

## 八、并发调用的预算

并发调用不是不用预算。

```go
ctx, cancel := context.WithTimeout(ctx, 1500*time.Millisecond)
defer cancel()

g, ctx := errgroup.WithContext(ctx)
```

总 ctx 控制整个接口。

每个下游可以再设置子 timeout：

```go
func (c *OrderClient) ListRecent(ctx context.Context, userID int64) ([]Order, error) {
	ctx, cancel := context.WithTimeout(ctx, 800*time.Millisecond)
	defer cancel()
	// call downstream
}
```

这样既有总预算，也有单个依赖的保护边界。

---

## 九、预算表模板

可以为重要接口建立预算表：

| 环节 | 预算 | 说明 |
| --- | --- | --- |
| 接口总耗时 | 1500ms | 小于网关 2s |
| 用户数据库 | 300ms | 主键查询 |
| 订单服务 | 800ms | 下游较慢 |
| 钱包服务 | 500ms | 需要稳定返回 |
| 响应组装 | 100ms | 本地计算 |
| 预留 | 300ms | 抖动和调度 |

预算表不是一次写完就不变。

上线后要根据监控数据调整。

---

## 十、常见错误

### 1. 每个下游都写同一个超时

```text
接口总超时 1s。
每个下游也都写 1s。
```

串行调用时很容易超过总预算。

### 2. timeout 写得过短

过短会导致正常请求大量失败。

超时应该基于真实耗时分布，而不是拍脑袋。

### 3. 只给 HTTP client 设置全局 Timeout

全局 timeout 是兜底，不等于请求级预算。

每次请求仍然应该传 ctx。
