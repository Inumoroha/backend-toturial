# 6. 综合检查清单

本节目标：用一份清单检查你是否真正掌握 Go `context`。

---

## 一、基础能力

- [ ] 能解释 `context.Context` 四个方法。
- [ ] 能区分 `Background()` 和 `TODO()`。
- [ ] 能使用 `WithCancel`。
- [ ] 能使用 `WithTimeout`。
- [ ] 能使用 `WithDeadline`。
- [ ] 能使用 `WithValue`，并知道边界。

---

## 二、后端分层能力

- [ ] Handler 使用 `r.Context()`。
- [ ] Service 方法第一个参数是 `ctx context.Context`。
- [ ] Repository 继续传 ctx。
- [ ] 数据库查询使用带 context 的方法。
- [ ] 下游 HTTP 请求使用 `NewRequestWithContext`。
- [ ] 不在下游使用 `context.Background()`。

---

## 三、并发能力

- [ ] goroutine 监听 `ctx.Done()`。
- [ ] channel 发送时考虑取消。
- [ ] 能使用 `errgroup.WithContext`。
- [ ] 任一并发任务失败时能取消其他任务。
- [ ] 能避免简单 goroutine 泄漏。

---

## 四、工程能力

- [ ] 派生 context 后调用 `cancel()`。
- [ ] 不把 context 存进结构体。
- [ ] 不把业务参数塞进 context。
- [ ] 能设计接口总超时和下游超时。
- [ ] 能区分 `Canceled` 和 `DeadlineExceeded`。
- [ ] 日志中能带 request id。
- [ ] 能写 context 超时和取消测试。

---

## 五、项目达标

你应该至少完成：

- [ ] HTTP 聚合服务。
- [ ] Worker 优雅关闭服务。
- [ ] 数据库查询超时保护接口。

如果这三个项目都能独立写出来，说明你已经具备在 Go 后端项目中正确使用 `context` 的能力。

---

## 六、后续进阶

学完 context 后，建议继续学习：

- Go 并发模式。
- `errgroup` 深入使用。
- gRPC deadline 和 metadata。
- OpenTelemetry 链路追踪。
- 服务治理：超时、重试、熔断、限流。
- 数据库连接池和慢查询治理。

这些内容会继续强化你作为 Go 后端工程师的工程能力。

---

## 七、项目验收标准

三个项目都完成后，请按下面标准验收。

### HTTP 聚合服务

- [ ] 使用 `r.Context()` 作为入口。
- [ ] service 中设置接口总 timeout。
- [ ] 使用 `errgroup.WithContext` 并发调用。
- [ ] 下游函数监听 `ctx.Done()`。
- [ ] 任一下游失败会取消其他下游。
- [ ] 日志包含 request id。
- [ ] 超时返回 504。
- [ ] 普通下游错误返回 502。

### Worker 优雅关闭

- [ ] 使用 `signal.NotifyContext`。
- [ ] worker 循环监听 ctx。
- [ ] 单个任务处理也监听 ctx。
- [ ] 收到 Ctrl+C 后停止接收新任务。
- [ ] 使用 WaitGroup 等待 worker 退出。
- [ ] 整体关闭有 timeout。

### 数据库查询超时保护

- [ ] handler、service、repository 都传递 ctx。
- [ ] repository 不使用 `context.Background()`。
- [ ] 数据库查询使用带 context 的方法。
- [ ] 单次查询设置子 timeout。
- [ ] 超时错误保留 `DeadlineExceeded` 根因。
- [ ] 用户不存在和超时能区分。

## 八、最终自测问题

请不看教程，回答：

1. 为什么 `ctx` 一般作为第一个参数？
2. 为什么不能把 context 存进 service 结构体？
3. `WithTimeout` 后为什么还要 `defer cancel()`？
4. 父 context 取消后子 context 会怎样？
5. 子 context 超时后父 context 会怎样？
6. 为什么 `context.Value` 不适合传业务参数？
7. errgroup 如何实现一个任务失败后取消其他任务？
8. HTTP 请求中 `r.Context()` 什么时候会取消？
9. 数据库查询为什么要使用 `QueryContext`？
10. 服务优雅关闭为什么要单独创建 shutdown context？

如果这 10 个问题能答清楚，说明 Context 主线已经比较扎实。

## 九、学习成果沉淀

建议最后整理三份材料：

```text
1. Go Context 使用规范。
2. 常见 context 错误排查清单。
3. 三个项目的 README。
```

README 至少写：

- 项目目标。
- 如何运行。
- context 在哪里创建。
- timeout 如何设置。
- 如何测试取消。
- 常见错误说明。

这一步很重要。能写清楚，才说明你真的理解了。
