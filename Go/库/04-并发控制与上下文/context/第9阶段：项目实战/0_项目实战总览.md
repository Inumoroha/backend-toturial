# 0. 项目实战总览

本阶段目标：通过三个小项目，把 `context` 的核心能力落到真实后端场景。

三个项目分别是：

```text
项目一：HTTP 聚合服务。
项目二：支持优雅关闭的 Worker 服务。
项目三：数据库查询超时保护。
```

---

## 一、为什么选择这三个项目

它们覆盖了 context 最常见的工程场景：

- 请求总超时。
- 下游调用超时。
- 并发任务取消。
- request id 日志。
- 服务退出信号。
- worker 生命周期。
- repository 中数据库查询超时。

学完后，你就不只是会写 `WithTimeout`，而是知道它在项目里怎么放。

---

建议按下面顺序学习：

3. `3_项目一_超时错误与RequestID日志.md`

5. `5_项目三_数据库查询超时保护.md`

---

## 三、建议学习方式

每个项目都先做最小闭环。

不要一开始就追求框架、配置、日志库、数据库全部完善。

建议顺序：

```text
先写 main.go 跑通。
再拆 handler / service / repository。
再补超时、取消、日志和错误处理。
最后整理 README。
```

---

## 四、本阶段达标标准

学完本阶段后，你应该能够做到：

- 独立写一个带总超时的聚合接口。
- 使用 errgroup 并发调用多个下游。
- 任一任务失败时取消其他任务。
- 写一个能优雅关闭的 worker 服务。
- 在 repository 中设计数据库查询 timeout。
- 总结自己的 context 使用规范。

---

## 五、项目学习方式

这一阶段不要只复制代码。

每个项目都建议按下面节奏做：

```text
1. 先读需求，画出调用链。
2. 写最小可运行版本。
3. 加入 context 传递。
4. 加入 timeout。
5. 加入取消和错误处理。
6. 加入日志和 request id。
7. 写一份检查清单。
```

如果一开始就上框架、配置文件、日志库、数据库迁移，容易把 context 主线淹没。

本阶段重点是：

```text
生命周期控制。
资源释放。
错误可观察。
```

---

## 六、统一技术选择

为了突出 context，本阶段尽量使用：

```text
Go 标准库 net/http。
context 标准库。
golang.org/x/sync/errgroup。
database/sql 或模拟 repository。
```

你当然可以用 Gin、Echo、Fiber、zap、pgx 等工具，但学习阶段先用标准库能看得更清楚。

---

## 七、每个项目的核心问题

### 项目一：HTTP 聚合服务

核心问题：

```text
一个请求并发调用多个下游，如何统一超时和取消？
```

### 项目二：Worker 优雅关闭

核心问题：

```text
服务收到退出信号时，如何通知后台任务停止？
```

### 项目三：数据库查询超时保护

核心问题：

```text
handler、service、repository 如何传递 ctx，并保护数据库查询？
```

---

## 八、项目完成后的复盘

每完成一个项目，都写一段复盘：

```text
这个项目的根 context 来自哪里？
哪些地方派生了子 context？
哪些地方调用了 cancel？
哪些函数监听了 ctx.Done？
哪些错误会返回 Canceled？
哪些错误会返回 DeadlineExceeded？
如果出现超时，日志够不够排查？
```

如果这些问题答不上来，说明代码虽然跑了，但 context 主线还没真正掌握。

---

## 九、项目实战的验收方式

每个项目不要只看“能不能返回结果”。

还要验证异常路径：

```text
下游变慢时会不会超时？
一个任务失败时其他任务会不会取消？
客户端断开时服务端会不会停止下游调用？
Ctrl+C 时 worker 会不会退出？
数据库查询超时时错误是否清楚？
```

后端工程的质量，往往体现在异常路径。

---

## 十、建议提交物

每个项目最终都建议有：

```text
main.go 或完整项目结构。
README.md。
运行命令。
测试或手动验证步骤。
常见错误说明。
```

README 不需要很长，但必须说明 context 在项目中的位置。

例如：

```text
本项目中，HTTP 请求 ctx 来自 r.Context。
ProfileService 基于它派生 800ms 总超时。
errgroup 使用派生 ctx 并发调用三个下游。
```

能写出这些说明，说明你真正看懂了自己的代码。

---

## 十一、完成后的进阶方向

项目完成后，可以继续扩展：

- 把模拟下游改成真实 HTTP 服务。
- 把模拟 repository 改成 PostgreSQL。
- 引入结构化日志。
- 引入 OpenTelemetry。
- 增加单元测试和集成测试。
- 增加 Makefile 和 README。

这些扩展不是 context 的必修内容，但很接近真实 Go 后端项目。
