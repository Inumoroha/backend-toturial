# 5. 项目三：数据库查询超时保护

本节目标：完成一个带数据库查询超时保护的用户查询接口设计。

接口：

```text
GET /users/{id}
```

要求：

- handler 使用 `r.Context()`。
- service 继续传 ctx。
- repository 使用数据库带 context 的查询方法。
- 单次数据库查询设置 300ms 超时。
- 超时返回明确错误。

---

## 一、Handler

```go
func (h *UserHandler) GetUser(w http.ResponseWriter, r *http.Request) {
	ctx := r.Context()

	user, err := h.service.GetUser(ctx, 1001)
	if err != nil {
		if errors.Is(err, context.DeadlineExceeded) {
			http.Error(w, "database timeout", http.StatusGatewayTimeout)
			return
		}
		http.Error(w, err.Error(), http.StatusInternalServerError)
		return
	}

	_ = json.NewEncoder(w).Encode(user)
}
```

---

## 二、Service

```go
func (s *UserService) GetUser(ctx context.Context, id int64) (*User, error) {
	if id <= 0 {
		return nil, fmt.Errorf("invalid user id")
	}

	return s.repo.FindByID(ctx, id)
}
```

Service 不重新创建根 context。

---

## 三、Repository

```go
func (r *UserRepo) FindByID(ctx context.Context, id int64) (*User, error) {
	ctx, cancel := context.WithTimeout(ctx, 300*time.Millisecond)
	defer cancel()

	row := r.db.QueryRowContext(ctx, `
		SELECT id, name
		FROM users
		WHERE id = ?
	`, id)

	var user User
	if err := row.Scan(&user.ID, &user.Name); err != nil {
		if errors.Is(ctx.Err(), context.DeadlineExceeded) {
			return nil, fmt.Errorf("find user query timeout: %w", ctx.Err())
		}
		return nil, err
	}

	return &user, nil
}
```

---

## 四、注意 Scan 后检查 ctx.Err

有些数据库错误不会直接等于 `context.DeadlineExceeded`。

所以在查询失败时，可以结合：

```go
ctx.Err()
```

判断是否是 context 超时或取消导致的。

## 六、本节达标标准

学完本节后，你应该能够做到：

- 设计 handler、service、repository 的 ctx 传递。
- 在 repository 中给数据库查询设置子超时。
- 超时后保留 `DeadlineExceeded` 根因。
- 不在 repository 中使用 `context.Background()`。

---

## 七、项目结构建议

```text
context-user-api/
  go.mod
  cmd/
    api/
      main.go
  internal/
    user/
      model.go
      handler.go
      service.go
      repository.go
```

职责：

- `handler.go`：解析请求、写响应。
- `service.go`：校验参数和业务逻辑。
- `repository.go`：数据库查询和超时保护。
- `model.go`：用户模型。

---

## 八、错误处理建议

可以定义业务错误：

```go
var ErrUserNotFound = errors.New("user not found")
```

repository 中：

```go
if errors.Is(err, sql.ErrNoRows) {
	return nil, ErrUserNotFound
}
```

handler 中：

```go
switch {
case errors.Is(err, ErrUserNotFound):
	http.Error(w, "user not found", http.StatusNotFound)
case errors.Is(err, context.DeadlineExceeded):
	http.Error(w, "database timeout", http.StatusGatewayTimeout)
default:
	http.Error(w, "internal error", http.StatusInternalServerError)
}
```

这样超时、未找到、内部错误不会混在一起。

## 九、模拟数据库慢查询

如果暂时没有数据库，可以先用 mock repo：

```go
type UserRepo struct{}

func (r *UserRepo) FindByID(ctx context.Context, id int64) (*User, error) {
	ctx, cancel := context.WithTimeout(ctx, 300*time.Millisecond)
	defer cancel()

	select {
	case <-time.After(500 * time.Millisecond):
		return &User{ID: id, Name: "Tom"}, nil
	case <-ctx.Done():
		return nil, fmt.Errorf("find user timeout: %w", ctx.Err())
	}
}
```

这样可以先练 context 流程，再接真实数据库。

---

## 十、真实数据库版本注意点

真实数据库中，超时可能发生在：

- 等待连接池连接。
- SQL 执行中。
- 扫描结果时。
- 遍历 rows 时。

所以要在错误发生后结合：

```go
ctx.Err()
```

判断是否是 context 引起的。

---

## 十一、测试建议

至少写三个测试：

```text
查询成功。
查询超时。
用户不存在。
```

超时测试可以用 mock repo 控制耗时，不要依赖真实慢 SQL。
