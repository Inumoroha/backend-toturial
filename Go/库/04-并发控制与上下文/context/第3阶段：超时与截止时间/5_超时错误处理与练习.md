# 5. 超时错误处理与练习

本节目标：学会区分超时、主动取消和普通错误，并完成本阶段综合练习。

超时在后端系统中很常见，但不能把所有错误都简单返回 500。

你至少要知道：

```text
是自己超时了？
是上游取消了？
是下游业务错误？
是网络错误？
```

---

## 一、区分两类 context 错误

主动取消：

```go
context.Canceled
```

超时或 deadline 到达：

```go
context.DeadlineExceeded
```

判断方式：

```go
if errors.Is(err, context.Canceled) {
	fmt.Println("canceled")
}

if errors.Is(err, context.DeadlineExceeded) {
	fmt.Println("timeout")
}
```

---

## 二、包装错误时保留原始原因

推荐：

```go
return fmt.Errorf("query user timeout: %w", ctx.Err())
```

不推荐：

```go
return errors.New("query user timeout")
```

前者可以继续用 `errors.Is` 判断根因。

---

## 三、HTTP 接口中如何响应

常见策略：

```text
context.Canceled：客户端取消，很多时候不用返回完整响应。
context.DeadlineExceeded：可以返回 504 Gateway Timeout 或 503。
业务参数错误：400。
资源不存在：404。
内部错误：500。
```

如果你写的是普通后端服务，超时通常不应该伪装成业务成功。

---

## 四、综合练习：三个下游调用

模拟一个接口，需要调用三个下游：

```text
fetchUser：耗时 100ms
fetchOrders：耗时 700ms
fetchWallet：耗时 300ms
```

接口总超时 500ms。

参考结构：

```go
func call(ctx context.Context, name string, cost time.Duration) error {
	select {
	case <-time.After(cost):
		fmt.Println(name, "done")
		return nil
	case <-ctx.Done():
		return fmt.Errorf("%s stopped: %w", name, ctx.Err())
	}
}
```

main：

```go
func main() {
	ctx, cancel := context.WithTimeout(context.Background(), 500*time.Millisecond)
	defer cancel()

	if err := call(ctx, "fetchOrders", 700*time.Millisecond); err != nil {
		if errors.Is(err, context.DeadlineExceeded) {
			fmt.Println("request timeout")
			return
		}
		fmt.Println("error:", err)
	}
}
```

先写串行版本，再思考并发版本应该怎么做。

后面的项目阶段会使用 `errgroup` 完成并发调用。

---

## 五、本阶段总结

你现在应该掌握：

- `WithTimeout` 用于持续时间超时。
- `WithDeadline` 用于具体截止时间。
- 超时会关闭 `ctx.Done()`。
- 超时错误是 `context.DeadlineExceeded`。
- 外部 HTTP 请求要使用 `NewRequestWithContext`。
- 数据库查询要使用带 context 的方法。
- 下游调用要有自己的超时预算。

下一阶段学习请求链路传递：如何从 HTTP 入口一路把 context 传到 service 和 repository。

## 六、错误包装示例

推荐写法：

```go
func fetchOrders(ctx context.Context) error {
	select {
	case <-time.After(time.Second):
		return nil
	case <-ctx.Done():
		return fmt.Errorf("fetch orders stopped: %w", ctx.Err())
	}
}
```

调用方：

```go
err := fetchOrders(ctx)
switch {
case errors.Is(err, context.DeadlineExceeded):
	fmt.Println("orders timeout")
case errors.Is(err, context.Canceled):
	fmt.Println("orders canceled")
case err != nil:
	fmt.Println("orders error:", err)
}
```

这样既保留业务上下文，又能判断根因。

## 七、HTTP 响应示例

```go
func writeContextError(w http.ResponseWriter, err error) {
	if errors.Is(err, context.DeadlineExceeded) {
		http.Error(w, "timeout", http.StatusGatewayTimeout)
		return
	}
	if errors.Is(err, context.Canceled) {
		http.Error(w, "request canceled", http.StatusRequestTimeout)
		return
	}
	http.Error(w, "internal error", http.StatusInternalServerError)
}
```

实际项目中还要区分业务错误，例如参数错误、资源不存在、权限不足。

## 八、常见错误

### 1. 用字符串判断错误

不推荐：

```go
if err.Error() == "context deadline exceeded" {
}
```

推荐：

```go
errors.Is(err, context.DeadlineExceeded)
```

### 2. 包装错误时不用 %w

不推荐：

```go
fmt.Errorf("timeout: %v", ctx.Err())
```

推荐：

```go
fmt.Errorf("timeout: %w", ctx.Err())
```

### 3. 所有 timeout 都返回 500

超时通常更适合返回 504 或业务约定的超时错误码。

## 九、综合练习加强版

把前面的三个下游调用扩展成：

```text
fetchUser：100ms，必须成功。
fetchOrders：700ms，可以失败。
fetchWallet：300ms，必须成功。
总 timeout：500ms。
```

要求：

1. 使用 context 控制总 timeout。
2. 每个函数取消时包装 `ctx.Err()`。
3. main 中区分 timeout 和普通错误。
4. 打印哪个下游超时。
5. 思考如果 orders 可以降级，应该如何处理。
