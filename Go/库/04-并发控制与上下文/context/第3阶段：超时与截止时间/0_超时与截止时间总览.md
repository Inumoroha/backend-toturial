# 0. 超时与截止时间总览

本阶段目标：掌握 `context.WithTimeout` 和 `context.WithDeadline`，并能在后端外部调用中设计合理的时间边界。

取消控制解决的是：

```text
我主动告诉任务停止。
```

超时控制解决的是：

```text
最多只能等这么久，时间到了就停止。
```

在后端系统里，超时控制非常重要。没有超时的外部调用，就像没有刹车的车。

## 二、为什么后端必须设置超时

后端服务经常调用：

- 数据库。
- Redis。
- HTTP 下游服务。
- gRPC 服务。
- 第三方 API。
- 文件存储。

这些调用都可能因为网络、锁等待、下游故障而变慢。

如果没有超时：

```text
请求一直挂住。
goroutine 一直占用。
连接池连接不释放。
上游请求堆积。
服务越来越慢。
```

超时不是为了让系统更快，而是为了让系统在异常情况下有边界。

## 三、Timeout 和 Deadline 的区别

`WithTimeout` 表示一段持续时间：

```go
ctx, cancel := context.WithTimeout(parent, 2*time.Second)
```

含义是：

```text
从现在开始最多等 2 秒。
```

`WithDeadline` 表示一个具体时间点：

```go
deadline := time.Now().Add(2 * time.Second)
ctx, cancel := context.WithDeadline(parent, deadline)
```

含义是：

```text
最晚到这个时间点结束。
```

日常项目里 `WithTimeout` 更常用。

---

## 四、一定要调用 cancel

常见写法：

```go
ctx, cancel := context.WithTimeout(parent, time.Second)
defer cancel()
```

即使超时到了 context 会自动取消，也要调用 `cancel()`。

原因是它可以及时释放内部 timer 等资源。

## 六、本阶段统一练习思路

这一阶段建议所有练习都围绕一个问题：

```text
一个任务如果超过预期时间，调用方应该如何停止等待，被调用方应该如何退出？
```

你可以准备三个模拟函数：

```go
func fast(ctx context.Context) error
func slow(ctx context.Context) error
func blocked(ctx context.Context) error
```

分别模拟：

- 很快完成。
- 比 timeout 慢。
- 一直等待直到 ctx 取消。

这样你能清楚观察 `WithTimeout` 和 `WithDeadline` 的行为。

## 七、后端里的超时分层

真实服务里，超时通常不是一个值，而是一层层的：

```text
网关超时。
HTTP server 超时。
接口业务总超时。
下游 HTTP 超时。
数据库查询超时。
Redis 操作超时。
```

context 主要负责业务链路里的超时传播。

例如：

```text
r.Context()
  -> 接口总 timeout 1s
  -> 数据库子 timeout 300ms
  -> 下游 HTTP 子 timeout 500ms
```

越往下游，通常 timeout 越具体。

## 八、常见错误预告

这一阶段你要特别避免：

### 1. 只给调用方超时，不让任务退出

调用方不等了，但 goroutine 还在跑。

### 2. 用 `context.Background()` 创建下游 timeout

这样会切断上游取消信号。

### 3. 忘记 `defer cancel()`

timer 资源不能及时释放。

### 4. 超时后丢失错误根因

不要把 `DeadlineExceeded` 包装成普通字符串错误后丢掉。

## 九、本阶段完成标准

学完后，你应该能独立写出：

```go
ctx, cancel := context.WithTimeout(parent, 500*time.Millisecond)
defer cancel()
```

并能解释：

- parent 是谁。
- 500ms 是怎么定的。
- cancel 为什么必须调用。
- 超时后调用方看到什么错误。
- 被调用方如何监听 `ctx.Done()`。
