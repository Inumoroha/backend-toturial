# 4. 数据库查询超时

本节目标：掌握数据库查询中的 context 使用方式，避免慢查询、锁等待导致请求无限挂起。

Go 标准库 `database/sql` 提供了带 context 的方法：

```go
QueryContext
QueryRowContext
ExecContext
BeginTx
```

很多第三方数据库库也有类似接口，例如 `pgx`：

```go
pool.Query(ctx, ...)
pool.QueryRow(ctx, ...)
pool.Exec(ctx, ...)
```

---

## 一、database/sql 示例

```go
func findUser(ctx context.Context, db *sql.DB, id int64) (*User, error) {
	ctx, cancel := context.WithTimeout(ctx, 300*time.Millisecond)
	defer cancel()

	row := db.QueryRowContext(ctx, `
		SELECT id, name
		FROM users
		WHERE id = ?
	`, id)

	var user User
	if err := row.Scan(&user.ID, &user.Name); err != nil {
		return nil, err
	}

	return &user, nil
}
```

这里给单次数据库查询设置了 300ms 子超时。

---

## 二、pgx 示例

```go
func findUser(ctx context.Context, pool *pgxpool.Pool, id int64) (*User, error) {
	ctx, cancel := context.WithTimeout(ctx, 300*time.Millisecond)
	defer cancel()

	row := pool.QueryRow(ctx, `
		SELECT id, name
		FROM app.users
		WHERE id = $1
	`, id)

	var user User
	if err := row.Scan(&user.ID, &user.Name); err != nil {
		return nil, err
	}

	return &user, nil
}
```

不同库的方法名略有不同，但思想一样：把上游传来的 context 继续传给数据库调用。

---

## 三、查询多行要关闭 rows

```go
rows, err := db.QueryContext(ctx, `
	SELECT id, name
	FROM users
`)
if err != nil {
	return nil, err
}
defer rows.Close()

for rows.Next() {
	// scan
}

if err := rows.Err(); err != nil {
	return nil, err
}
```

超时或取消时，`rows.Err()` 也可能返回错误。

不要只检查 `QueryContext` 的错误。

---

## 四、数据库驱动要支持取消

context 能否真正中断数据库查询，还和数据库驱动有关。

一般成熟驱动都支持得比较好，但你要知道：

```text
context 是 Go 侧的取消信号。
驱动需要把这个信号转换成数据库协议层面的取消。
```

即使底层取消不够及时，context 仍然能让调用方停止等待。

---

## 五、不要在 repository 中使用 Background

不推荐：

```go
func (r *UserRepo) FindByID(id int64) (*User, error) {
	ctx := context.Background()
	return r.query(ctx, id)
}
```

推荐：

```go
func (r *UserRepo) FindByID(ctx context.Context, id int64) (*User, error) {
	return r.query(ctx, id)
}
```

Repository 是请求链路的下游，应该接收并使用上游 context。

## 七、本节达标标准

学完本节后，你应该能够做到：

- 使用数据库带 context 的方法。
- 在 repository 中接收上游 ctx。
- 为单次数据库调用派生更短 timeout。
- 查询多行时关闭 rows 并检查 `rows.Err()`。
- 知道 context 是否真正取消 SQL 还取决于驱动支持。

## 八、完整分层示例

```go
type UserService struct {
	repo *UserRepo
}

func (s *UserService) GetUser(ctx context.Context, id int64) (*User, error) {
	if id <= 0 {
		return nil, fmt.Errorf("invalid id")
	}
	return s.repo.FindByID(ctx, id)
}

type UserRepo struct {
	db *sql.DB
}

func (r *UserRepo) FindByID(ctx context.Context, id int64) (*User, error) {
	ctx, cancel := context.WithTimeout(ctx, 300*time.Millisecond)
	defer cancel()

	row := r.db.QueryRowContext(ctx, `
		SELECT id, name
		FROM users
		WHERE id = ?
	`, id)

	var user User
	if err := row.Scan(&user.ID, &user.Name); err != nil {
		if ctxErr := ctx.Err(); ctxErr != nil {
			return nil, fmt.Errorf("find user stopped: %w", ctxErr)
		}
		return nil, err
	}

	return &user, nil
}
```

这个例子体现了：

```text
service 不关心数据库 timeout 细节。
repository 为具体 SQL 设置子 timeout。
错误保留 context 根因。
```

---

## 九、数据库 timeout 不是 SQL 优化

timeout 的作用是止损。

如果查询经常超时，应该继续排查：

- 是否缺索引。
- 是否返回数据太多。
- 是否有锁等待。
- 是否连接池太小。
- SQL 是否写得不合理。

不要只把 300ms 改成 3s。

---

## 十、事务中的超时策略

事务不宜太长。

可以给整个事务设置 timeout：

```go
ctx, cancel := context.WithTimeout(ctx, time.Second)
defer cancel()
```

事务内所有 SQL 使用同一个 ctx。

如果某一步超时，应该回滚：

```go
defer tx.Rollback()
```

不要在事务里为每条 SQL 随意创建互相独立的根 context。

---

## 十一、常见错误

### 1. 查询单行用了 Query 而不是 QueryRowContext

不是 context 错误，但会让代码更繁琐。

### 2. 多行查询不关闭 rows

连接池很容易被拖垮。

### 3. 超时后直接返回数据库驱动错误

最好结合 `ctx.Err()` 保留明确语义。

### 4. repository 不接收 ctx

这样 handler 的取消信号传不到数据库层。
