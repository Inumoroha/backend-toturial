# 5. 项目收尾：测试、压测与复盘

本节目标：为所有项目补齐验证手段，并形成自己的并发组件复盘模板。

项目能运行只是第一步。并发组件必须能被验证。

---

## 一、统一测试命令

在每个项目目录运行：

```bash
go test ./...
go test -race ./...
go test -bench=. -benchmem ./...
```

如果测试可能卡住，增加超时：

```bash
go test -timeout=5s ./...
```

---

## 二、必须覆盖的测试类型

每个项目都建议包含：

- 单线程功能测试。
- 多 goroutine 并发测试。
- 关闭流程测试。
- 错误路径测试。
- race detector 验证。
- benchmark。

---

## 三、并发组件 README 模板

```markdown
# 组件名称

## 解决什么问题

## 核心 API

## 共享状态

## 同步设计

## 关闭流程

## 测试说明

## benchmark 结果

## 已知限制

## 后续优化
```

写 README 的过程会逼你把并发边界讲清楚。

---

## 四、复盘问题

每完成一个项目，回答：

- 哪些字段是共享状态？
- 哪些方法会并发调用？
- 用了哪些同步原语？
- 为什么选择这些同步原语？
- 有没有持锁做慢操作？
- 关闭时是否会泄漏 goroutine？
- 是否通过 `go test -race`？
- benchmark 说明了什么？

---

## 五、面试表达方式

不要只说：

```text
我用了 Mutex。
```

更好的表达是：

```text
缓存内部用 map 保存数据，HTTP 请求和后台清理 goroutine 都会访问它，所以我用 RWMutex 保护 map。
Get 发现过期 key 时需要删除，因此简单版本直接使用写锁，避免读锁升级复杂度。
后台清理通过 stop channel 退出，Close 中关闭 stop 后用 WaitGroup 等待 goroutine 结束。
最后用 go test -race 验证并发读写路径。
```

这就是后端工程师应该具备的表达。

---

## 六、最终达标标准

完成整个 `sync` 教程后，你应该能做到：

- 根据共享状态选择合适同步原语。
- 写出并发安全组件。
- 用测试和 race detector 验证。
- 用 benchmark 和 pprof 支持性能判断。
- 能解释每个同步设计背后的取舍。
- 能在真实 Go 后端项目中安全使用 `sync` 包。
