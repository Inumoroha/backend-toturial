# 3. Cond 与 channel 对比

本节目标：理解 `Cond` 和 channel 的适用边界，避免在业务代码中过度使用条件变量。

---

## 一、channel 更适合传递数据

如果你的场景是：

```text
生产者产生数据；
消费者接收数据；
数据是一条一条流动的。
```

优先使用 channel：

```go
ch := make(chan string)

go func() {
    ch <- "task"
    close(ch)
}()

for item := range ch {
    handle(item)
}
```

channel 同时表达了数据传递和同步。

---

## 二、Cond 更适合等待复杂条件

如果你的条件不是简单“有数据可读”，而是：

```text
队列非空并且未关闭；
连接池有可用连接或者已关闭；
缓存版本号达到某个值；
多个字段共同决定是否继续。
```

`Cond` 可能更合适。

---

## 三、可读性对比

channel 版本通常更容易读：

```go
task := <-tasks
```

`Cond` 版本需要维护：

```text
共享状态；
锁；
等待条件；
唤醒时机；
关闭语义。
```

所以业务代码中，如果 channel 能清晰表达，就优先 channel。

---

## 四、什么时候必须了解 Cond

即使你平时少写 `Cond`，也应该了解它，因为：

- 一些底层库会使用条件变量。
- 连接池、队列、调度器类组件会遇到类似模型。
- 面试可能考察 `Wait` 为什么要用 `for`。
- 理解它能帮助你更深入掌握同步原语。

---

## 五、本节练习

1. 用 channel 实现生产者消费者。
2. 用 `Cond` 实现同样功能。
3. 对比两版关闭逻辑。
4. 总结哪一版更适合业务代码。

---

## 六、本节达标标准

学完本节后，你应该能做到：

- 优先用 channel 表达数据流。
- 在复杂条件等待场景中理解 `Cond` 的价值。
- 能读懂基于 `Cond` 的队列或连接池实现。
