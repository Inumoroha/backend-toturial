# 0. 学习准备与并发直觉总览

本阶段目标：在正式学习 `sync` 包之前，先建立 Go 并发编程的基本直觉，知道为什么需要同步原语，以及如何用测试工具发现并发问题。

`sync` 包不是孤立知识。它解决的是下面这类问题：

```text
多个 goroutine 同时访问同一份数据时，如何保证数据正确？
多个后台任务同时运行时，主 goroutine 如何等待它们结束？
某些资源只能初始化一次时，如何避免重复初始化？
对象创建成本较高时，如何减少 GC 压力？
```

如果一开始只背 `Mutex`、`WaitGroup`、`Once` 的 API，很容易形成“看起来会用，实际项目里容易写错”的状态。本阶段会先从并发问题本身出发。

---

## 一、本阶段要解决的问题

学习完本阶段后，你应该能解释：

- 为什么 `count++` 在并发下不是安全操作。
- 为什么普通 `map` 不能随便并发读写。
- `go test -race` 能发现什么，不能证明什么。
- 什么是共享状态、临界区、数据竞争。
- 为什么同步原语保护的是“一段访问规则”，而不只是某个变量。
- 为什么 Go 后端工程中并发安全比“代码能跑”更重要。

---

## 二、推荐学习顺序

本阶段建议按下面顺序学习：

```text
环境与工具
-> goroutine 与共享状态
-> 数据竞争与 race detector
-> Go 内存模型与 happens-before 直觉
-> 练习目录和测试规范
```

原因是：

- 先准备好 Go 环境和常用命令，后面每个小节都能直接运行。
- 再理解 goroutine 并发执行带来的不确定性。
- 然后用 race detector 亲眼看到并发读写问题。
- 最后建立 happens-before 直觉，理解为什么加锁、channel、atomic 能建立可见性保证。

## 三、本阶段统一练习目录

建议在当前阶段目录下建立代码练习：

```text
examples/
  race-counter/
    counter.go
    counter_test.go
  map-race/
    main.go
  visibility/
    main.go
```

每个练习都建议至少运行三类命令：

```bash
go test ./...
go test -race ./...
go test -bench=. -benchmem ./...
```

其中：

- `go test` 检查功能正确性。
- `go test -race` 检查运行过程中是否发现数据竞争。
- `go test -bench=. -benchmem` 用于观察性能和内存分配。

## 四、学习 sync 的核心心法

学习 `sync` 时，请始终带着这几个问题看代码：

```text
共享状态是什么？
谁会读它？
谁会写它？
哪些操作必须作为一个整体完成？
并发正确性靠什么保证？
```

例如一个本地缓存：

- 共享状态是 `map[string]Item`。
- HTTP 请求处理 goroutine 会读写它。
- 后台清理 goroutine 也会删除过期 key。
- `Get` 时读取 value 和判断过期时间最好在同一个保护规则内完成。
- 正确性可以由 `Mutex` 或 `RWMutex` 保证。

这比简单说“给 map 加锁”更接近工程思维。

---

## 五、本阶段达标标准

学完本阶段后，你应该能做到：

- 能创建一个 Go module 并运行测试。
- 能写出一个故意存在数据竞争的例子。
- 能使用 `go test -race` 观察并解释 race 报告。
- 能说清楚 `count++` 为什么不是原子操作。
- 能解释普通 map 并发读写为什么危险。
- 能初步理解 happens-before 不是玄学，而是“谁的写入对谁可见”的规则。

完成这些后，再进入 `Mutex` 阶段会轻松很多。
