# 5. 锁粒度与工程实践

本节目标：理解粗锁、细锁的取舍，掌握后端工程中设计锁边界的基本方法。

锁粒度指的是一把锁保护的范围有多大。

---

## 一、粗锁

粗锁是指一把锁保护较大的状态或较长的操作。

优点：

- 实现简单。
- 不容易漏保护。
- 适合初始版本。

缺点：

- 并发度低。
- 临界区太长时容易阻塞。
- 可能把不相关操作串行化。

示例：

```go
type Store struct {
    mu     sync.Mutex
    users  map[int64]User
    orders map[int64]Order
}
```

所有 users 和 orders 操作都用同一把锁，这就是粗锁。

---

## 二、细锁

细锁是指把不同状态拆成不同锁。

示例：

```go
type Store struct {
    userMu  sync.Mutex
    users   map[int64]User
    orderMu sync.Mutex
    orders  map[int64]Order
}
```

优点：

- 并发度更高。
- 不相关操作可以同时执行。

缺点：

- 设计复杂。
- 容易出现锁顺序问题。
- 更难维护不变量。

---

## 三、如何选择

初学和大多数业务代码建议：

```text
先用简单正确的粗锁；
当 benchmark 或线上指标证明锁竞争明显时，再考虑拆细。
```

不要一开始就设计很多锁。锁越多，正确性越难证明。

---

## 四、按 key 分片锁

缓存或计数器有时可以使用分片锁：

```go
type shard struct {
    mu    sync.Mutex
    items map[string]string
}

type ShardedCache struct {
    shards []shard
}
```

根据 key hash 到不同 shard，不同 shard 可以并发访问。

适合：

- key 很多。
- 并发访问量大。
- 单把锁竞争明显。

不适合：

- 数据量小。
- 业务规则复杂。
- 需要跨 shard 保持强一致不变量。

---

## 五、临界区应该放什么

临界区内应该只放必须受保护的操作：

```text
读共享状态
修改共享状态
维护多个字段之间的不变量
```

临界区外尽量放：

```text
参数校验
网络请求
数据库请求
日志格式化
复杂计算
JSON 编解码
```

但不要机械地把所有计算移到锁外。关键是判断这段计算是否依赖共享状态的一致快照。

---

## 六、工程代码中的命名习惯

推荐：

```go
func (c *Cache) deleteLocked(key string) {
    delete(c.items, key)
}
```

`Locked` 后缀表示：

```text
调用这个方法时，调用方必须已经持有锁。
```

这能减少重复加锁和死锁风险。

## 七、本节练习

1. 用一把锁实现缓存。
2. 用分片锁实现缓存。
3. 写 benchmark 比较低并发和高并发场景。
4. 分析哪种实现更容易维护。
5. 找一个持锁期间做慢操作的例子，把慢操作移到锁外。

## 八、本节达标标准

学完本节后，你应该能做到：

- 解释粗锁和细锁的取舍。
- 不盲目优化锁粒度。
- 能用 benchmark 判断锁竞争是否值得优化。
- 会用 `xxxLocked` 命名表达内部锁约定。
