# 3. 并发安全缓存

本节目标：实现一个基于 `map + Mutex` 的并发安全本地缓存，理解锁在真实后端组件中的使用方式。

本地缓存是后端工程里最常见的共享状态之一。普通 map 不支持并发读写，所以必须加同步保护。

---

## 一、缓存的最小功能

我们先实现三个方法：

```text
Set(key, value)
Get(key) (value, ok)
Delete(key)
```

数据结构：

```go
type Cache struct {
    mu    sync.Mutex
    items map[string]string
}
```

初始化：

```go
func NewCache() *Cache {
    return &Cache{
        items: make(map[string]string),
    }
}
```

---

## 二、Set 方法

```go
func (c *Cache) Set(key, value string) {
    c.mu.Lock()
    defer c.mu.Unlock()

    c.items[key] = value
}
```

注意：

- `items` 是共享 map。
- 写 map 必须在锁内。
- 不要返回内部 map 给外部使用。

---

## 三、Get 方法

```go
func (c *Cache) Get(key string) (string, bool) {
    c.mu.Lock()
    defer c.mu.Unlock()

    value, ok := c.items[key]
    return value, ok
}
```

虽然是读取，也要加锁。因为可能有其他 goroutine 同时 `Set` 或 `Delete`。

---

## 四、Delete 方法

```go
func (c *Cache) Delete(key string) {
    c.mu.Lock()
    defer c.mu.Unlock()

    delete(c.items, key)
}
```

删除也是写操作。

---

## 五、不要暴露内部 map

危险写法：

```go
func (c *Cache) Items() map[string]string {
    return c.items
}
```

这会让外部绕过锁直接读写 map，破坏并发安全。

如果需要快照，可以复制一份：

```go
func (c *Cache) Snapshot() map[string]string {
    c.mu.Lock()
    defer c.mu.Unlock()

    snapshot := make(map[string]string, len(c.items))
    for k, v := range c.items {
        snapshot[k] = v
    }
    return snapshot
}
```

这里返回的是副本，外部修改不会影响内部状态。

---

## 六、加入 TTL 的思考

如果缓存支持过期时间，item 可能变成：

```go
type Item struct {
    Value     string
    ExpiresAt time.Time
}
```

`Get` 时需要：

```text
读取 item
判断是否过期
如果过期则删除
返回结果
```

这一组操作应该在同一把锁保护下完成。否则可能出现读取和删除交错导致逻辑混乱。

## 八、本节达标标准

学完本节后，你应该能做到：

- 用 `map + Mutex` 实现并发安全缓存。
- 知道为什么不能暴露内部 map。
- 知道读取 map 也需要同步保护。
- 能识别一个业务操作中哪些步骤应该放在同一个临界区。
