# 1. 临界区与最小互斥模型

本节目标：理解 `Mutex` 真正保护的是临界区，掌握互斥锁最小正确模型。

---

## 一、什么是临界区

临界区是指访问共享状态、并且不能被多个 goroutine 同时执行的一段代码。

例如：

```go
type Counter struct {
    value int
}

func (c *Counter) Add() {
    c.value++
}
```

如果多个 goroutine 同时调用 `Add`，`c.value++` 就是临界区。

修复方式：

```go
type Counter struct {
    mu    sync.Mutex
    value int
}

func (c *Counter) Add() {
    c.mu.Lock()
    defer c.mu.Unlock()

    c.value++
}
```

---

## 二、Lock 和 Unlock 必须成对出现

`Lock` 成功后，如果不执行 `Unlock`，其他 goroutine 会一直阻塞。

错误示例：

```go
func (c *Counter) Add(n int) {
    c.mu.Lock()
    if n <= 0 {
        return
    }
    c.value += n
    c.mu.Unlock()
}
```

当 `n <= 0` 时函数直接返回，锁不会释放。

正确写法：

```go
func (c *Counter) Add(n int) {
    c.mu.Lock()
    defer c.mu.Unlock()

    if n <= 0 {
        return
    }
    c.value += n
}
```

`defer` 能保证函数返回前释放锁，适合绝大多数业务代码。

---

## 三、锁保护的是访问规则

看下面例子：

```go
func (c *Counter) Value() int {
    return c.value
}
```

即使写入时加了锁，如果读取时不加锁，仍然是不安全的。

正确写法：

```go
func (c *Counter) Value() int {
    c.mu.Lock()
    defer c.mu.Unlock()

    return c.value
}
```

只要某个共享字段被锁保护，那么所有读写路径都应该遵守同一把锁。

---

## 四、不要复制 Mutex

错误示例：

```go
func CopyCounter(c Counter) {
    c.Add(1)
}
```

如果 `Counter` 内部包含 `sync.Mutex`，复制 `Counter` 会复制锁的状态，可能导致不可预期行为。

工程建议：

- 包含 `Mutex` 的结构体通常用指针传递。
- 不要给包含 `Mutex` 的结构体实现值接收者方法。
- 可以在结构体中加入注释提醒不能复制。

推荐：

```go
func (c *Counter) Add(n int) {}
```

不要写：

```go
func (c Counter) Add(n int) {}
```

---

## 五、持锁代码要尽量短

不要在持锁期间做这些事情：

- 网络请求。
- 数据库查询。
- 文件 IO。
- 长时间 CPU 计算。
- 调用外部未知函数。
- 等待用户输入。

原因是锁会阻塞其他 goroutine。临界区越长，系统吞吐越差，死锁风险也越高。

推荐思路：

```text
加锁读取必要状态
解锁
执行慢操作
必要时再次加锁更新状态
```

当然，如果慢操作的结果必须和状态变更保持严格一致，就要重新设计边界，不能简单拆锁。

## 六、本节练习

1. 实现一个 `Counter`，包含 `Add`、`Value`、`Reset`。
2. 故意把 `Value` 写成不加锁版本，用 `go test -race` 检测。
3. 把方法接收者从指针改成值，观察代码审查时会产生什么风险。
4. 写一个提前 `return` 忘记解锁的例子，理解为什么 `defer` 很重要。

---

## 七、本节达标标准

学完本节后，你应该能做到：

- 解释临界区的含义。
- 正确写出 `Lock` / `defer Unlock`。
- 知道所有共享状态读写路径都要遵守同一把锁。
- 知道包含 `Mutex` 的结构体不能随意复制。
- 知道持锁期间不应该做慢操作。
