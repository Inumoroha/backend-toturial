# 2. 并发安全计数器

本节目标：通过实现 `SafeCounter` 掌握 `Mutex` 的基本工程用法。

计数器是学习锁的最好入门案例，因为它足够简单，又能完整暴露数据竞争问题。

---

## 一、不安全版本

```go
type Counter struct {
    value int64
}

func (c *Counter) Add(n int64) {
    c.value += n
}

func (c *Counter) Value() int64 {
    return c.value
}
```

问题：

- `c.value += n` 不是原子操作。
- `Value` 和 `Add` 并发执行时也可能发生数据竞争。
- 多个 goroutine 同时写会丢失更新。

---

## 二、Mutex 版本

```go
type SafeCounter struct {
    mu    sync.Mutex
    value int64
}

func (c *SafeCounter) Add(n int64) {
    c.mu.Lock()
    defer c.mu.Unlock()

    c.value += n
}

func (c *SafeCounter) Value() int64 {
    c.mu.Lock()
    defer c.mu.Unlock()

    return c.value
}

func (c *SafeCounter) Reset() {
    c.mu.Lock()
    defer c.mu.Unlock()

    c.value = 0
}
```

这里有三个关键点：

- `value` 是共享状态。
- 所有访问 `value` 的方法都加同一把锁。
- 方法使用指针接收者，避免复制锁。

## 三、并发测试

```go
func TestSafeCounterConcurrentAdd(t *testing.T) {
    var c SafeCounter
    var wg sync.WaitGroup

    const workers = 1000

    for i := 0; i < workers; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            c.Add(1)
        }()
    }

    wg.Wait()

    if got := c.Value(); got != workers {
        t.Fatalf("got %d, want %d", got, workers)
    }
}
```

运行：

```bash
go test -race ./...
```

通过这个测试，至少能说明当前测试路径没有被 race detector 发现数据竞争。

---

## 四、为什么 Value 也要加锁

很多初学者会认为：

```text
只有写入需要加锁，读取不需要。
```

这是错误的。

只要一个 goroutine 写，另一个 goroutine 同时读，并且没有同步保护，就是数据竞争。

所以 `Value` 必须加锁。

---

## 五、什么时候不用 Mutex

如果计数器只做简单加减和读取，可以使用 `atomic.Int64`：

```go
type AtomicCounter struct {
    value atomic.Int64
}

func (c *AtomicCounter) Add(n int64) {
    c.value.Add(n)
}

func (c *AtomicCounter) Value() int64 {
    return c.value.Load()
}
```

但是如果计数器未来变复杂，例如同时维护：

- 当前值。
- 最大值。
- 更新时间。
- 变更原因。

那么 `Mutex` 更容易维护不变量。
