# 4. 死锁与锁顺序

本节目标：理解死锁的常见产生原因，学会通过固定锁顺序、缩短持锁时间和避免重入来降低风险。

死锁的表现通常是：

```text
程序没有 panic；
CPU 也不高；
请求一直卡住；
goroutine dump 里大量 goroutine 阻塞在 Lock 上。
```

---

## 一、忘记 Unlock

最简单的死锁来自忘记释放锁：

```go
func (c *Cache) Set(key, value string) {
    c.mu.Lock()
    if key == "" {
        return
    }
    c.items[key] = value
    c.mu.Unlock()
}
```

修复：

```go
func (c *Cache) Set(key, value string) {
    c.mu.Lock()
    defer c.mu.Unlock()

    if key == "" {
        return
    }
    c.items[key] = value
}
```

---

## 二、重复加同一把 Mutex

Go 的 `sync.Mutex` 不是可重入锁。

错误示例：

```go
func (c *Cache) Set(key, value string) {
    c.mu.Lock()
    defer c.mu.Unlock()

    c.setLocked(key, value)
}

func (c *Cache) setLocked(key, value string) {
    c.mu.Lock()
    defer c.mu.Unlock()

    c.items[key] = value
}
```

同一个 goroutine 已经持有锁，又尝试再次 `Lock`，会把自己卡住。

常见解决方式是明确命名内部方法：

```go
func (c *Cache) setLocked(key, value string) {
    c.items[key] = value
}
```

`setLocked` 的命名表示调用方必须已经持有锁。

---

## 三、多把锁顺序不一致

假设有两个账户：

```go
type Account struct {
    mu      sync.Mutex
    balance int64
}
```

转账时如果这样：

```text
goroutine A: 锁账户 1 -> 锁账户 2
goroutine B: 锁账户 2 -> 锁账户 1
```

就可能互相等待。

解决思路：固定锁顺序。

例如按账户 ID 从小到大加锁：

```text
永远先锁 ID 小的账户，再锁 ID 大的账户。
```

只要所有路径遵守同一顺序，就能避免这类死锁。

---

## 四、持锁期间调用外部函数

危险写法：

```go
func (s *Service) Update(fn func()) {
    s.mu.Lock()
    defer s.mu.Unlock()

    fn()
}
```

问题是 `fn` 由外部传入，你不知道它会不会：

- 再次调用 `s` 的加锁方法。
- 执行慢 IO。
- 阻塞等待其他锁。

工程上应尽量避免在持锁期间调用未知外部代码。

---

## 五、如何排查死锁

常用方法：

- 观察请求是否卡住。
- 打印 goroutine dump。
- 搜索阻塞在 `sync.(*Mutex).Lock` 的 goroutine。
- 检查锁顺序。
- 检查是否存在未释放锁的提前返回。
- 检查是否在持锁期间调用外部函数。

测试阶段可以用超时避免测试永久卡住：

```bash
go test -timeout=5s ./...
```
