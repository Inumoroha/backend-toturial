# Go sync 系统教程总目录

本目录是一套面向 Go 后端工程师的 `sync` 系统学习教程。

学习目标不是背 API，而是形成并发工程判断：

```text
共享状态是什么？
谁会读？
谁会写？
并发边界在哪里？
正确性靠什么保证？
性能判断是否被测试验证？
```


## 掌握 sync 的验收标准

读完教程只能说明你建立了知识框架。真正掌握 `sync`，至少要通过下面这些验收标准。

### 一、概念验收

你应该能清楚解释：

- 什么是共享状态、临界区、数据竞争。
- 为什么 `count++` 不是并发安全操作。
- 为什么普通 `map` 不能随意并发读写。
- happens-before 解决的是互斥问题还是可见性问题。
- 为什么锁保护的是访问规则，而不只是某个变量。

### 二、工具选择验收

给出一个后端并发场景时，你应该能判断：

- 什么时候用 `Mutex`。
- 什么时候用 `RWMutex`，以及为什么它不一定比 `Mutex` 快。
- 什么时候用 `WaitGroup`，什么时候应该用 `errgroup`。
- 什么时候用 `Once`，什么时候初始化失败重试不适合 `Once`。
- 什么时候用 `Cond`，什么时候 channel 更清晰。
- 什么时候用 `sync.Map`，什么时候 `map + RWMutex` 更合适。
- 什么时候用 `sync.Pool`，以及为什么它不能当业务缓存。
- 什么时候用 `atomic`，什么时候应该回到 `Mutex`。

### 三、代码实现验收

你应该能独立写出并解释：

- 并发安全计数器。
- 并发安全本地缓存。
- 支持 TTL 和后台清理的缓存。
- 限制最大并发数的任务执行器。
- 可关闭、可等待退出的后台 worker。
- 简易连接池或阻塞队列。
- 使用 `sync.Pool` 复用 buffer 的日志缓冲器。

这些代码不只要能跑，还要能通过并发测试和 race detector。

### 四、测试验证验收

你应该能做到：

- 为每个并发组件写普通单元测试。
- 为关键路径写多 goroutine 并发测试。
- 使用 `go test -race ./...` 检查数据竞争。
- 使用 `go test -timeout=5s ./...` 避免死锁测试永久卡住。
- 使用 `go test -bench=. -benchmem ./...` 比较锁、atomic、Pool 等方案。
- 能解释 race detector 没报错不等于代码绝对没有并发 bug。

### 五、工程表达验收

你应该能像后端工程师一样描述设计，而不是只说“我加了锁”：

```text
这个组件的共享状态是 map 和 closed 标志。
HTTP 请求 goroutine 会读写 map，后台清理 goroutine 也会删除过期 key。
所以我用 RWMutex 保护这些字段。
Get 发现 key 过期时需要删除，因此简单版本直接使用写锁，避免读锁升级复杂度。
Close 通过关闭 stop channel 通知后台 goroutine 退出，再用 WaitGroup 等待收尾。
最后用 go test -race 验证并发读写路径。
```

能讲到这个层次，说明你已经从“会用 API”进入“能设计并发组件”。

### 六、源码与性能验收

你不需要背源码，但应该能做到：

- 找到 `$GOROOT/src/sync` 下的核心源码文件。
- 读懂 `Mutex`、`RWMutex`、`WaitGroup`、`Once`、`Map`、`Pool` 的官方注释。
- 知道哪些类型不能复制。
- 知道哪些误用会 panic。
- 能用 benchmark 或 pprof 支撑性能判断。
- 不凭感觉说某个方案“更快”。

### 七、最终通过标准

如果你能做到下面这些，就可以认为自己已经比较扎实地掌握了 Go 后端开发中常用的 `sync` 能力：

```text
能识别共享状态；
能选择合适同步原语；
能写出并发安全实现；
能设计关闭和取消流程；
能写 race 测试和 benchmark；
能解释每个同步选择背后的取舍；
能在项目中定位并修复常见并发 bug。
```

如果只是读完文档，可以算“理解 sync”。如果把练习和项目都做完，并能按上面的标准解释和验证，就可以算“掌握 sync”。

## 最重要的学习提醒

`sync` 包真正难的不是 API，而是边界。

写任何并发组件前，先问：

```text
这份状态是不是共享的？
是否存在并发读写？
读写是否都遵守同一套同步规则？
关闭时 goroutine 如何退出？
错误和取消如何传播？
性能优化是否有 benchmark 支撑？
```

能稳定回答这些问题，你就已经在向可靠的 Go 后端工程师靠近。
