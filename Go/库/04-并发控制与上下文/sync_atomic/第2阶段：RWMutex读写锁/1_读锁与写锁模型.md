# 1. 读锁与写锁模型

本节目标：理解 `RWMutex` 的读锁、写锁模型，以及它为什么适合读多写少场景。

---

## 一、Mutex 的限制

使用普通 `Mutex` 时，不管读还是写，都只能一个 goroutine 进入临界区。

```text
读 A 阻塞读 B
读 A 阻塞写 C
写 C 阻塞读 A
```

如果共享状态读很多、写很少，这会浪费并发能力。

---

## 二、RWMutex 的规则

`RWMutex` 允许多个读者并发：

```text
读 A 和读 B 可以同时执行；
写 C 必须独占；
写 C 执行时，其他读写都要等待。
```

这适合配置缓存、路由表、本地只读快照这类场景。

---

## 三、读锁写法

```go
func (c *Cache) Get(key string) (string, bool) {
    c.mu.RLock()
    defer c.mu.RUnlock()

    value, ok := c.items[key]
    return value, ok
}
```

读锁期间只能读共享状态，不能写。

错误示例：

```go
func (c *Cache) GetOrSet(key string, value string) string {
    c.mu.RLock()
    defer c.mu.RUnlock()

    if old, ok := c.items[key]; ok {
        return old
    }
    c.items[key] = value
    return value
}
```

这里持有读锁时写 map，是错误的。

---

## 四、写锁写法

```go
func (c *Cache) Set(key, value string) {
    c.mu.Lock()
    defer c.mu.Unlock()

    c.items[key] = value
}
```

写锁是独占的。写锁期间：

- 其他写者不能进入。
- 读者也不能进入。

---

## 五、读锁不能升级成写锁

常见误用：

```go
c.mu.RLock()
if _, ok := c.items[key]; !ok {
    c.mu.Lock()
    c.items[key] = value
    c.mu.Unlock()
}
c.mu.RUnlock()
```

这可能导致死锁。不要在持有读锁时直接申请写锁。

正确思路：

```text
先释放读锁；
再申请写锁；
申请写锁后重新检查条件。
```

示例：

```go
func (c *Cache) LoadOrStore(key, value string) string {
    c.mu.RLock()
    old, ok := c.items[key]
    c.mu.RUnlock()
    if ok {
        return old
    }

    c.mu.Lock()
    defer c.mu.Unlock()

    if old, ok := c.items[key]; ok {
        return old
    }
    c.items[key] = value
    return value
}
```

第二次检查很重要，因为释放读锁到获取写锁之间，其他 goroutine 可能已经写入了。
