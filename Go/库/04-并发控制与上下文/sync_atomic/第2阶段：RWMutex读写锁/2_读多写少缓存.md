# 2. 读多写少缓存

本节目标：用 `RWMutex` 实现一个适合读多写少场景的本地缓存，并理解它与 `Mutex` 版本的差异。

---

## 一、数据结构

```go
type Cache struct {
    mu    sync.RWMutex
    items map[string]string
}

func NewCache() *Cache {
    return &Cache{
        items: make(map[string]string),
    }
}
```

`items` 仍然是普通 map，只是访问它的规则变成：

```text
读操作使用 RLock；
写操作使用 Lock。
```

---

## 二、Get 方法

```go
func (c *Cache) Get(key string) (string, bool) {
    c.mu.RLock()
    defer c.mu.RUnlock()

    value, ok := c.items[key]
    return value, ok
}
```

多个 goroutine 可以同时执行 `Get`。

---

## 三、Set 方法

```go
func (c *Cache) Set(key, value string) {
    c.mu.Lock()
    defer c.mu.Unlock()

    c.items[key] = value
}
```

写入必须独占，因为 map 写入会修改内部结构。

---

## 四、Delete 方法

```go
func (c *Cache) Delete(key string) {
    c.mu.Lock()
    defer c.mu.Unlock()

    delete(c.items, key)
}
```

删除也是写操作，不能使用读锁。

---

## 五、Snapshot 方法

```go
func (c *Cache) Snapshot() map[string]string {
    c.mu.RLock()
    defer c.mu.RUnlock()

    snapshot := make(map[string]string, len(c.items))
    for k, v := range c.items {
        snapshot[k] = v
    }
    return snapshot
}
```

虽然只是读取，但复制 map 时需要遍历内部 map，仍然应该持有读锁。

---

## 六、TTL 场景中的注意点

如果 `Get` 发现 key 已过期，并且要删除它，那么 `Get` 就不再是纯读操作。

错误思路：

```text
RLock 读取
发现过期
在 RLock 里 delete
```

正确思路：

```text
RLock 读取并判断
释放 RLock
如果需要删除，再 Lock
Lock 后重新检查是否仍然过期
删除
```

或者简单一点，直接让 `Get` 使用写锁，先保证正确。

工程上不要为了使用 `RWMutex` 而把逻辑写得复杂难懂。
