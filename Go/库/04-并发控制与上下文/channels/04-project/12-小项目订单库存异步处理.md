# 12. 小项目订单库存异步处理

本节目标：学完后你能用 channels 串起一个订单库存异步处理流程，并知道上线前还缺哪些生产能力。

简短引入：这一节把前面的知识合起来：任务队列、worker pool、错误返回、限流、关闭流程。为了方便复制运行，我们用内存模拟订单和库存，不接真实数据库。

## 一、为什么需要它

订单系统里常见流程是：接收订单请求、校验库存、扣减库存、记录结果。真实项目中这些步骤必须非常谨慎，因为库存和订单都属于关键数据。

这里用 channel 练习并发编排，但要先明确边界：

```text
本节代码用于学习并发结构；真实订单扣库存必须依赖数据库事务、唯一约束、幂等键和可回滚方案。
```

把下面代码复制到 `main.go`：

```go
package main

import (
	"context"
	"fmt"
	"sync"
	"time"
)

type OrderJob struct {
	OrderID string
	SkuID   string
	Qty     int
}

type OrderResult struct {
	OrderID string
	OK      bool
	Err     error
}

type Inventory struct {
	mu    sync.Mutex
	stock map[string]int
}

func NewInventory() *Inventory {
	return &Inventory{
		stock: map[string]int{
			"sku-1": 5,
			"sku-2": 2,
		},
	}
}

func (i *Inventory) Deduct(skuID string, qty int) error {
	i.mu.Lock()
	defer i.mu.Unlock()

	if qty <= 0 {
		return fmt.Errorf("invalid qty: %d", qty)
	}
	if i.stock[skuID] < qty {
		return fmt.Errorf("stock not enough: sku=%s left=%d need=%d", skuID, i.stock[skuID], qty)
	}
	i.stock[skuID] -= qty
	return nil
}

func worker(ctx context.Context, id int, inv *Inventory, jobs <-chan OrderJob, results chan<- OrderResult, wg *sync.WaitGroup) {
	defer wg.Done()

	for {
		select {
		case <-ctx.Done():
			return
		case job, ok := <-jobs:
			if !ok {
				return
			}

			time.Sleep(100 * time.Millisecond)
			err := inv.Deduct(job.SkuID, job.Qty)

			select {
			case results <- OrderResult{OrderID: job.OrderID, OK: err == nil, Err: err}:
			case <-ctx.Done():
				return
			}

			fmt.Printf("worker %d handled order %s\n", id, job.OrderID)
		}
	}
}

func main() {
	ctx, cancel := context.WithTimeout(context.Background(), 3*time.Second)
	defer cancel()

	inventory := NewInventory()
	jobs := make(chan OrderJob, 4)
	results := make(chan OrderResult, 8)

	var wg sync.WaitGroup
	for i := 1; i <= 3; i++ {
		wg.Add(1)
		go worker(ctx, i, inventory, jobs, results, &wg)
	}

	go func() {
		wg.Wait()
		close(results)
	}()

	orders := []OrderJob{
		{OrderID: "order-1", SkuID: "sku-1", Qty: 1},
		{OrderID: "order-2", SkuID: "sku-1", Qty: 2},
		{OrderID: "order-3", SkuID: "sku-2", Qty: 1},
		{OrderID: "order-4", SkuID: "sku-1", Qty: 3},
		{OrderID: "order-5", SkuID: "sku-2", Qty: 2},
	}

enqueue:
	for _, order := range orders {
		select {
		case jobs <- order:
		case <-ctx.Done():
			fmt.Println("stop enqueue:", ctx.Err())
			break enqueue
		}
	}
	close(jobs)

	success, failed := 0, 0
	for result := range results {
		if result.Err != nil {
			failed++
			fmt.Printf("order %s failed: %v\n", result.OrderID, result.Err)
			continue
		}
		success++
		fmt.Printf("order %s success\n", result.OrderID)
	}

	fmt.Printf("summary: success=%d failed=%d\n", success, failed)
	fmt.Println("left stock:", inventory.stock)
}
```

Windows PowerShell：

```powershell

```

Linux/macOS：

```bash

```

## 三、关键参数/语法/代码结构

`jobs := make(chan OrderJob, 4)` 是订单任务入口。缓冲为 4，表示允许短暂积压。

`results := make(chan OrderResult, 8)` 收集处理结果。容量略大，避免 worker 因结果无人接收而过早卡住。

`worker` 中同时监听 `ctx.Done()` 和 `jobs`，让它既能处理任务，也能响应取消。

`Inventory` 里用了 `sync.Mutex`，因为 map 不是并发安全的。真实数据库场景下，这里会变成事务和条件更新，而不是进程内锁。

## 四、真实后端场景示例

如果把这个例子迁移到真实订单服务，通常会拆成这些步骤：

1. 接收 HTTP 请求，校验参数。
2. 用幂等键检查请求是否重复。
3. 在数据库事务中创建订单并扣减库存。
4. 提交后发送异步事件，例如通知、积分、搜索索引。
5. 后台 worker 消费异步事件，并记录成功失败。

注意，第三步不应该简单丢给内存 channel 异步做。否则服务重启时订单任务可能丢失，用户看到的状态也不稳定。

更保守的做法是：关键状态先可靠落库，再由后台任务处理非关键副作用。

```text
订单创建和库存扣减属于关键一致性路径；通知、日志、索引刷新更适合作为异步任务。
```

## 五、注意点

真实数据库扣库存建议使用短事务和条件更新。例如：

```sql
update sku_stock
set stock = stock - ?
where sku_id = ?
  and stock >= ?;
```

然后检查受影响行数。为 0 时返回库存不足或并发失败。

订单号要有唯一约束，用来防止用户重试导致重复创建。迁移订单表结构时要使用迁移工具，分步骤上线，保留回滚路径。

索引也要谨慎。给订单表加索引能提升查询，但会增加写入成本。高写入表的索引要围绕真实查询设计，不能看到字段就加。

## 六、常见误区

误区一：把扣库存放进内存 channel 后立即返回成功。  
如果进程崩溃，任务丢失，用户却已经看到成功。

误区二：只用 mutex 模拟库存就以为解决并发扣减。  
mutex 只保护当前进程，多个服务实例之间不共享这把锁。

误区三：没有幂等。  
用户超时重试、网关重试、消息重复投递都可能造成重复处理。

误区四：失败任务只打印。  
真实项目要记录失败状态、错误原因、重试次数和人工处理入口。
