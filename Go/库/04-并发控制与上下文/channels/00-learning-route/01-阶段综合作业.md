# Go channels 阶段综合作业

这份作业用于配合学习路线。每个阶段都做一个小交付，不追求功能大，而是训练你把 channel 用在真实后端问题里：任务流、退出、错误、背压和可恢复边界。

```text
综合作业的目标不是写更多 goroutine，而是证明你能让 goroutine 有边界、有错误、有退出路径。
```

## 一、阶段 1 综合作业：用户注册审计日志通道

适用文章：

- `01-channel直觉与最小用法.md`
- `02-无缓冲channel与同步交接.md`
- `03-有缓冲channel与削峰.md`
- `04-close与range让消费者正常退出.md`

### 业务背景

模拟一个用户注册流程。主流程接收用户注册事件，后台消费者负责写审计日志。这里不接真实数据库，只用打印模拟写入。

### 必做要求

1. 定义 `AuditEvent`，至少包含 `UserID`、`Action`、`IP`、`CreatedAt`。
2. 使用有缓冲 channel 接收审计事件。
3. 启动一个消费者 goroutine，用 `range` 持续读取事件。
4. 主流程发送 5 条注册事件后关闭 channel。
5. 使用 `sync.WaitGroup` 等消费者处理完再退出。

### 加分要求

1. 模拟某条日志写入失败，并返回 error。
2. 统计成功数和失败数。
3. 在注释中说明：为什么关键审计日志不能只靠内存 channel。

### 验收标准

- 程序能稳定退出，不依赖 `time.Sleep` 等待 goroutine。
- 能解释谁负责发送、谁负责接收、谁负责关闭 channel。
- 能说出缓冲满了以后可能发生什么。

## 二、阶段 2 综合作业：用户首页并发聚合接口

适用文章：

- `05-select同时等待多个事件.md`
- `06-context取消让请求链路及时收尾.md`
- `07-worker-pool控制并发处理任务.md`
- `08-fan-out-fan-in并行查询与结果汇总.md`

### 业务背景

模拟一个用户首页接口，需要同时加载用户资料、订单数量、权限信息和未读通知数。它们互不依赖，可以并发查询，但整个接口必须有超时。

### 必做要求

1. 使用 `context.WithTimeout` 设置整体超时。
2. 每个查询函数都接收 `context.Context`。
3. 每个查询函数通过结果 channel 返回 `Name`、`Value`、`Err`。
4. 主流程聚合所有结果。
5. 任意关键查询失败时返回错误，并取消其他任务。

### 加分要求

1. 区分关键数据和可降级数据：权限失败必须失败，通知数量失败可以默认 0。
2. 为每个模拟查询设置不同耗时，观察整体耗时。
3. 把 map 聚合结果改成结构体响应。

### 验收标准

- 超时时程序能退出，不留下卡住的 goroutine。
- 能说明为什么并行查询会增加数据库连接池压力。
- 能说明哪些查询适合并行，哪些不适合。

## 三、阶段 3 综合作业：短链接访问日志批处理管道

适用文章：

- `09-错误传递与结果聚合.md`
- `10-限流与背压保护数据库和下游服务.md`
- `11-定时任务与日志管道.md`

### 业务背景

模拟短链接服务的访问日志写入。在线请求把访问事件放入 channel，后台任务按批次写入。为了保护数据库，需要控制并发和处理队列满的情况。

### 必做要求

1. 定义 `VisitLog`，至少包含 `ShortCode`、`UserID`、`IP`、`CreatedAt`。
2. 使用有缓冲 channel 作为日志入口。
3. 后台消费者按 `batchSize` 或 `ticker` 触发 flush。
4. 使用 context 支持停止后台任务。
5. 记录 flush 成功次数和失败次数。

### 加分要求

1. 当日志 channel 满时，使用非阻塞发送并统计丢弃数量。
2. 模拟数据库写入失败，并限制重试次数。
3. 增加一个信号量，限制同时 flush 的数量。

### 验收标准

- 能解释队列满时选择等待、返回错误或丢弃的业务依据。
- 能说明哪些日志可以丢，哪些日志必须可靠落库。
- 能说明批量大小对吞吐、延迟、事务时间和回滚成本的影响。

## 四、阶段 4 综合作业：订单库存异步处理增强版

适用文章：

- `12-小项目订单库存异步处理.md`

### 业务背景

基于订单库存示例继续增强。目标不是写一个完整电商系统，而是把 channel 的工程边界想清楚：哪些适合异步，哪些必须事务化。

### 必做要求

1. 给 `OrderJob` 增加 `UserID`、`IdempotencyKey`。
2. 使用 worker pool 处理订单任务。
3. 用结果 channel 收集所有成功和失败订单。
4. 最后输出汇总：成功数、失败数、失败原因列表、剩余库存。
5. 使用 context 支持整体超时。

### 加分要求

1. 增加重复订单模拟：相同 `IdempotencyKey` 只能处理一次。
2. 为失败订单增加 `Retryable` 标记。
3. 写出真实数据库版本的伪代码，包括事务、条件扣库存、唯一约束和回滚。

### 验收标准

- 能说清楚示例里的 mutex 只能保护单进程内存，不能保护多实例库存。
- 能说明真实订单创建和库存扣减为什么不能只放进内存 channel。
- 能说明通知、积分、索引刷新为什么更适合放到异步任务。

## 五、提交建议

每个阶段可以单独建一个目录，例如：

```text
assignments/
  stage-01-audit-log/
  stage-02-home-page-aggregation/
  stage-03-short-link-log-pipeline/
  stage-04-order-inventory/
```

每个目录至少包含：

- `main.go`：可运行代码。
- `README.md`：说明怎么运行、实现了哪些要求、还有哪些生产风险。

运行方式保持简单：

Windows PowerShell：

```powershell
go run .
```

Linux/macOS：

```bash
go run .
```

## 六、复盘问题

每完成一个阶段，回答下面几个问题：

1. 这个程序里有哪些 goroutine？
2. 每个 channel 的发送方是谁，接收方是谁，关闭方是谁？
3. 如果消费者变慢，会发生什么？
4. 如果主流程超时，后台任务会不会退出？
5. 如果进程崩溃，哪些数据会丢？
6. 哪些任务应该改用数据库、消息队列或事务 outbox？

```text
能回答这些问题，说明你不只是会写 channel，而是开始具备后端并发设计意识。
```
