# 11. 定时任务与日志管道

本节目标：学完后你能用 ticker、channel 和 context 写一个可停止的后台任务。

简短引入：后端服务里常见后台任务包括刷新缓存、写访问日志、同步统计、扫描超时订单。它们通常需要定时运行，并在服务关闭时干净退出。

## 一、为什么需要它

很多初学代码会写一个无限循环：

```go
for {
	doSomething()
	time.Sleep(time.Second)
}
```

这能跑，但不好停，也不好测试。真实服务收到关闭信号时，后台任务应该停止接收新任务，尽量处理完已有任务，然后退出。

```text
后台任务必须设计退出路径，否则优雅关闭只是一句口号。
```

## 二、基本用法

```go
package main

import (
	"context"
	"fmt"
	"time"
)

func main() {
	ctx, cancel := context.WithCancel(context.Background())
	ticker := time.NewTicker(300 * time.Millisecond)
	defer ticker.Stop()

	go func() {
		time.Sleep(time.Second)
		cancel()
	}()

	for {
		select {
		case <-ticker.C:
			fmt.Println("flush logs")
		case <-ctx.Done():
			fmt.Println("stop background task")
			return
		}
	}
}
```

运行：

Windows PowerShell：

```powershell
go run .
```

Linux/macOS：

```bash
go run .
```

## 三、关键参数/语法/代码结构

`time.NewTicker` 按固定间隔产生事件。用完要 `Stop`，避免资源泄漏。

`select` 同时等待 ticker 和取消信号。

`context.WithCancel` 用于主动停止后台任务。真实服务里，取消通常来自操作系统信号或 HTTP server shutdown。

## 四、真实后端场景示例

下面模拟访问日志管道：请求把日志写入 channel，后台每隔一段时间批量 flush。

```go
package main

import (
	"context"
	"fmt"
	"sync"
	"time"
)

type AccessLog struct {
	Path   string
	UserID int64
}

func logWriter(ctx context.Context, logs <-chan AccessLog, wg *sync.WaitGroup) {
	defer wg.Done()

	ticker := time.NewTicker(300 * time.Millisecond)
	defer ticker.Stop()

	batch := make([]AccessLog, 0, 4)

	flush := func() {
		if len(batch) == 0 {
			return
		}
		fmt.Println("flush logs:", batch)
		batch = batch[:0]
	}

	for {
		select {
		case log, ok := <-logs:
			if !ok {
				flush()
				return
			}
			batch = append(batch, log)
			if len(batch) >= 4 {
				flush()
			}
		case <-ticker.C:
			flush()
		case <-ctx.Done():
			flush()
			return
		}
	}
}

func main() {
	ctx, cancel := context.WithCancel(context.Background())
	defer cancel()
	logs := make(chan AccessLog, 10)
	var wg sync.WaitGroup

	wg.Add(1)
	go logWriter(ctx, logs, &wg)

	for i := 1; i <= 7; i++ {
		logs <- AccessLog{Path: "/articles", UserID: int64(1000 + i)}
	}

	close(logs)
	wg.Wait()
}
```

真实项目中，批量写数据库时要使用参数化批量插入。批量大小不要过大，避免单条 SQL 太长、锁时间太久、回滚成本太高。

## 五、注意点

日志管道要先区分日志类型。访问统计可以批量、可延迟；安全审计、资金流水、权限变更日志通常不能只靠内存 channel。

批量 flush 要考虑失败处理。如果数据库暂时失败，可以重试少量次数；如果持续失败，应记录到可靠介质或降级，不能无限占内存。

如果日志表数据量很大，要提前设计索引。索引能提升查询，但会增加写入成本。写入量高的日志表不要随意加很多索引。

## 六、常见误区

误区一：只用 `time.Sleep` 写定时循环。  
它不方便及时响应取消，也不好做多事件等待。

误区二：收到取消后直接丢掉 batch。  
服务关闭前应尽量 flush 已收集的数据，至少明确哪些数据允许丢。

误区三：批量越大越好。  
批量过大会造成内存上涨、SQL 太大、事务时间过长。

误区四：日志写入失败只打印。  
生产环境要有指标、告警和补偿方案。

## 七、本节练习

练习一：把 batch 大小从 `4` 改成 `2` 和 `10`，观察 flush 次数变化。真实项目中，batch 太小浪费吞吐，太大增加延迟和失败回滚成本。

练习二：在 `flush` 中模拟失败，例如每第三次 flush 返回错误。要求记录失败次数，并思考失败日志应该重试、落盘还是告警。

练习三：把 `close(logs)` 注释掉，只调用 `cancel()`，观察还有哪些数据可能没被正常 drain。这个练习用来区分“停止接收新数据”和“立刻取消后台任务”。

## 八、本节达标标准

- 能用 ticker 写定时任务。
- 能用 context 停止后台任务。
- 能用 channel 接收日志并批量 flush。
- 能说明哪些日志可以内存缓冲，哪些必须可靠落库。
