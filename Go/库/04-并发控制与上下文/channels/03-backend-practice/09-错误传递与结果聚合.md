# 9. 错误传递与结果聚合

本节目标：学完后你能让 goroutine 把结果和错误一起返回，而不是只在后台打印日志。

简短引入：后端代码里，后台任务失败不能只 `fmt.Println`。调用方需要知道哪些任务成功、哪些失败，才能决定重试、回滚、告警或返回错误。

## 一、为什么需要它

初学 channel 时常把 goroutine 写成“自己干自己的”。但真实业务里，订单创建、库存校验、权限检查、数据迁移都需要明确的错误处理。

可以把结果和错误放进同一个结构体，通过 channel 返回给协调者。

```text
goroutine 里的错误如果没有被传出来，就等于业务流程看不见这个错误。
```

## 二、基本用法

```go
package main

import (
	"errors"
	"fmt"
)

type Result struct {
	Value string
	Err   error
}

func main() {
	ch := make(chan Result, 1)

	go func() {
		ch <- Result{Err: errors.New("user not found")}
	}()

	result := <-ch
	if result.Err != nil {
		fmt.Println("failed:", result.Err)
		return
	}
	fmt.Println(result.Value)
}
```

运行：

Windows PowerShell：

```powershell
go run .
```

Linux/macOS：

```bash
go run .
```

## 三、关键代码结构

`Result` 结构体同时包含业务值和错误。

channel 容量为 1，是为了避免接收方因为其他原因提前返回时，发送方卡住。当然，更完整的写法还要加 context 取消。

不要在 goroutine 里直接 `panic` 表达普通业务错误。用户不存在、库存不足、权限不够都应该作为 error 返回。

## 四、真实后端场景示例

下面模拟创建订单前并行校验库存和权限。两个校验都通过才继续创建订单。

```go
package main

import (
	"errors"
	"fmt"
	"sync"
)

type CheckResult struct {
	Name string
	Err  error
}

func checkStock(out chan<- CheckResult) {
	out <- CheckResult{Name: "stock", Err: nil}
}

func checkPermission(out chan<- CheckResult) {
	out <- CheckResult{Name: "permission", Err: errors.New("user has no permission")}
}

func main() {
	results := make(chan CheckResult, 2)
	var wg sync.WaitGroup

	checks := []func(chan<- CheckResult){checkStock, checkPermission}
	for _, check := range checks {
		wg.Add(1)
		go func(fn func(chan<- CheckResult)) {
			defer wg.Done()
			fn(results)
		}(check)
	}

	go func() {
		wg.Wait()
		close(results)
	}()

	for result := range results {
		if result.Err != nil {
			fmt.Printf("check %s failed: %v\n", result.Name, result.Err)
			return
		}
	}

	fmt.Println("all checks passed, create order")
}
```

真实项目中，库存扣减不能只靠这个检查。最终扣减必须放在数据库事务里，常见做法是带条件更新：

```go
// update sku_stock set stock = stock - ? where sku_id = ? and stock >= ?
```

还要检查受影响行数。如果为 0，说明库存不足或并发竞争失败。

## 五、注意点

错误聚合要考虑“第一个错误返回”还是“收集所有错误”。接口请求通常第一个关键错误就可以返回；批量任务通常要收集每条失败原因，便于重试。

如果收到错误后立即返回，要取消其他 goroutine。否则它们可能继续写结果、访问数据库或占用资源。后续可以用 `context` 和 `errgroup` 进一步简化。

涉及数据库写入时，要明确事务边界。channel 可以并发准备数据，但提交关键写入时通常要回到清晰的事务流程中。

## 六、常见误区

误区一：只在 goroutine 里打印错误。  
调用方无法据此决定业务结果，也不方便测试。

误区二：把错误和正常值放在两个 channel。  
初学阶段容易接收顺序混乱。一个结果结构体通常更清楚。

误区三：并行校验通过后就认为库存一定够。  
并发场景下，检查和扣减之间可能已经被别人改了。最终一致性要靠事务条件更新或锁策略。

## 七、本节练习

练习一：把订单校验示例改成收集所有校验结果，而不是遇到第一个错误就返回。最后统一打印哪些检查通过、哪些失败。批量任务通常需要这种视角。

练习二：给 `CheckResult` 增加 `Retryable bool` 字段，区分“库存不足”这种业务失败和“下游暂时不可用”这种可重试失败。真实项目的重试策略不能只看 error 字符串。

练习三：把库存校验和权限校验改成带 `context.Context` 的函数。收到关键错误后取消其他检查，避免它们继续访问数据库或下游服务。

## 八、本节达标标准

- 能设计包含 `Err error` 的结果结构体。
- 能让 goroutine 把错误返回给协调者。
- 能区分接口请求和批量任务的错误处理策略。
- 能说明 channel 不能替代数据库事务和最终扣减校验。
