# 10. 限流与背压保护数据库和下游服务

本节目标：学完后你能用 channel 建立容量边界，保护数据库、Redis 和下游 HTTP 服务。

简短引入：后端系统不是越并发越好。限流和背压的目的，是在压力超过承载能力时，让系统以可预期的方式变慢、拒绝或降级。

## 一、为什么需要它

比如文章发布后要刷新缓存。如果瞬间有大量发布请求，每个请求都直接刷新缓存和搜索索引，下游可能被打满。

channel 可以作为一个有容量的入口。容量满时，生产者会等待，或者我们主动返回“稍后再试”。

```text
没有容量边界的异步任务，最终会把压力转移到内存、数据库连接池或下游服务。
```

## 二、基本用法

用 channel 当信号量，限制同时执行的任务数量。

```go
package main

import (
	"fmt"
	"sync"
	"time"
)

func main() {
	sem := make(chan struct{}, 2)
	var wg sync.WaitGroup

	for i := 1; i <= 5; i++ {
		wg.Add(1)
		go func(id int) {
			defer wg.Done()
			sem <- struct{}{}
			defer func() { <-sem }()

			fmt.Println("call downstream", id)
			time.Sleep(300 * time.Millisecond)
		}(i)
	}

	wg.Wait()
}
```

运行：

Windows PowerShell：

```powershell
go run .
```

Linux/macOS：

```bash
go run .
```

同时最多只有 2 个任务进入关键区域。

## 三、关键代码结构

`sem := make(chan struct{}, 2)` 表示最多允许 2 个并发。

`sem <- struct{}{}` 表示获取名额。如果满了，就等待。

`defer func() { <-sem }()` 表示释放名额。必须放在获取成功之后，并确保异常路径也释放。

## 四、真实后端场景示例

下面模拟短链接访问统计写入。我们限制同时写入数据库的 goroutine 数量，避免连接池被后台任务抢光。

```go
package main

import (
	"fmt"
	"sync"
	"time"
)

func writeVisitStat(shortCode string, userID int64) {
	time.Sleep(100 * time.Millisecond)
	fmt.Printf("write stat code=%s user=%d\n", shortCode, userID)
}

func main() {
	limit := make(chan struct{}, 3)
	var wg sync.WaitGroup

	for i := 1; i <= 10; i++ {
		wg.Add(1)
		go func(userID int64) {
			defer wg.Done()

			limit <- struct{}{}
			defer func() { <-limit }()

			writeVisitStat("go123", userID)
		}(int64(1000 + i))
	}

	wg.Wait()
	fmt.Println("all stats written")
}
```

真实项目中，访问统计通常可以批量写入，减少数据库压力。比如先聚合再每秒 flush 一次。但如果涉及用户行为审计，不能随意丢弃，需要可靠存储。

## 五、注意点

限流不是只在代码里放一个 channel。你还要结合：

- HTTP 层超时和最大请求体限制。
- 数据库连接池上限。
- Redis 连接池上限。
- 下游接口限额。
- 队列满时的业务策略。

队列满时常见选择：

- 等待：适合必须处理且调用方能接受延迟。
- 返回错误：适合在线请求，避免请求堆积。
- 丢弃：只适合低价值、可损失事件，例如部分非关键指标。

```text
关键业务不要静默丢弃；低价值事件也要有指标记录丢弃数量。
```

## 六、常见误区

误区一：只限制 goroutine，不限制数据库连接。  
如果每个 goroutine 又开多个查询，数据库仍然可能超载。

误区二：队列满了就新开 goroutine。  
这相当于取消了背压，问题只会更隐蔽。

误区三：没有监控队列长度和失败数。  
生产环境需要知道当前是否积压、是否丢弃、是否持续超时。

误区四：用 channel 解决所有限流。  
单进程 channel 只能限制当前进程。多实例服务还需要网关、Redis、令牌桶或下游配额控制。

## 七、本节练习

练习一：把信号量容量从 `2` 改成 `1`、`3`、`5`，观察同时进入下游调用的任务数量。这个练习对应真实项目里的数据库连接池和下游接口配额。

练习二：把短链接统计示例改成非阻塞获取名额：如果当前并发已满，就返回 `busy` 或记录丢弃。然后写一句注释说明这种策略只适合什么类型的任务。

练习三：增加两个计数器：`accepted` 和 `rejected`。真实服务里，背压策略必须能被监控，否则你不知道系统是在正常限流还是持续过载。

## 八、本节达标标准

- 能用 channel 实现简单信号量。
- 能说明背压的含义：压力过高时让上游等待或失败。
- 能根据业务重要性选择等待、返回错误或丢弃。
- 能说出单进程 channel 限流和分布式限流的区别。
