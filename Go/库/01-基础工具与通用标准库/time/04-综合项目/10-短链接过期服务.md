# 10. 短链接过期服务

本节目标：学完后，你能把时间创建、过期判断、接口超时、数据库字段设计这些知识串成一个可落地的小项目。

简短引入：短链接服务是后端里很适合练习时间处理的项目。它有创建时间、过期时间、访问时判断、后台扫描、数据库索引和可测试逻辑，刚好覆盖前面的大部分内容。

## 一、为什么需要它

真实短链接系统至少要回答三个问题：

- 创建短链接时，过期时间怎么算？
- 用户访问时，怎么判断是否已经过期？
- 过期数据如何清理或标记？

这些问题看起来简单，但如果时间格式、时区、边界判断不统一，很容易出现“明明没过期却不能访问”或者“已经过期还能访问”的问题。

```text
访问短链接时必须实时判断 expired_at，后台扫描只能作为清理手段。
```

## 二、基本用法

下面先用内存版本跑通核心逻辑：

```go
package main

import (
	"errors"
	"fmt"
	"time"
)

type ShortLink struct {
	Code      string
	TargetURL string
	ExpiredAt time.Time
	CreatedAt time.Time
}

type Store struct {
	data map[string]ShortLink
}

func NewStore() *Store {
	return &Store{data: make(map[string]ShortLink)}
}

func (s *Store) Create(code, targetURL string, ttl time.Duration, now time.Time) ShortLink {
	link := ShortLink{
		Code:      code,
		TargetURL: targetURL,
		ExpiredAt: now.Add(ttl).UTC(),
		CreatedAt: now.UTC(),
	}
	s.data[code] = link
	return link
}

func (s *Store) Resolve(code string, now time.Time) (string, error) {
	link, ok := s.data[code]
	if !ok {
		return "", errors.New("短链接不存在")
	}
	if !now.UTC().Before(link.ExpiredAt) {
		return "", errors.New("短链接已过期")
	}
	return link.TargetURL, nil
}

func main() {
	store := NewStore()
	now := time.Date(2026, 7, 6, 10, 0, 0, 0, time.UTC)

	store.Create("abc123", "https://example.com/articles/1", 24*time.Hour, now)

	url, err := store.Resolve("abc123", now.Add(23*time.Hour))
	fmt.Println(url, err)

	url, err = store.Resolve("abc123", now.Add(24*time.Hour))
	fmt.Println(url, err)
}
```

Windows PowerShell：

```powershell
go run .\main.go
```

Linux/macOS：

```bash
go run ./main.go
```

## 三、关键参数/语法/代码结构

`ttl time.Duration` 表示短链接有效期，例如 1 小时、7 天。

`ExpiredAt: now.Add(ttl).UTC()` 表示过期时间统一转成 UTC。

`!now.UTC().Before(link.ExpiredAt)` 表示当前时间已经到达或超过过期时间。

这里使用“到点即过期”的判断：如果 `now == expiredAt`，链接不可访问。

如果要把这段逻辑接到 HTTP 上，可以先用标准库写一个最小版本：

```go
package main

import (
	"errors"
	"fmt"
	"net/http"
	"strings"
	"time"
)

type ShortLink struct {
	Code      string
	TargetURL string
	ExpiredAt time.Time
	CreatedAt time.Time
}

type Store struct {
	data map[string]ShortLink
}

func (s *Store) Resolve(code string, now time.Time) (string, error) {
	link, ok := s.data[code]
	if !ok {
		return "", errors.New("短链接不存在")
	}
	if !now.UTC().Before(link.ExpiredAt) {
		return "", errors.New("短链接已过期")
	}
	return link.TargetURL, nil
}

func main() {
	now := time.Now().UTC()
	store := &Store{data: map[string]ShortLink{
		"abc123": {
			Code:      "abc123",
			TargetURL: "https://example.com/articles/1",
			ExpiredAt: now.Add(24 * time.Hour),
			CreatedAt: now,
		},
	}}

	http.HandleFunc("/r/", func(w http.ResponseWriter, r *http.Request) {
		code := strings.TrimPrefix(r.URL.Path, "/r/")
		targetURL, err := store.Resolve(code, time.Now())
		if err != nil {
			http.Error(w, err.Error(), http.StatusNotFound)
			return
		}
		http.Redirect(w, r, targetURL, http.StatusFound)
	})

	fmt.Println("listen on http://127.0.0.1:8080/r/abc123")
	if err := http.ListenAndServe(":8080", nil); err != nil {
		panic(err)
	}
}
```

运行后访问：

Windows PowerShell：

```powershell
go run .\main.go
Invoke-WebRequest http://127.0.0.1:8080/r/abc123 -MaximumRedirection 0
```

Linux/macOS：

```bash
go run ./main.go
curl -i http://127.0.0.1:8080/r/abc123
```

这个 HTTP 示例仍然用内存存储，是为了先把“访问时实时判断过期”跑通。真实项目里再把 `Store` 换成数据库实现。

## 四、真实后端场景示例

落到数据库时，可以有这样一张表：

```sql
CREATE TABLE short_links (
    id BIGINT PRIMARY KEY,
    code VARCHAR(32) NOT NULL UNIQUE,
    target_url TEXT NOT NULL,
    status VARCHAR(16) NOT NULL,
    expired_at TIMESTAMP NOT NULL,
    created_at TIMESTAMP NOT NULL,
    updated_at TIMESTAMP NOT NULL
);

CREATE INDEX idx_short_links_code ON short_links (code);
CREATE INDEX idx_short_links_expired_at ON short_links (expired_at);
```

访问短链接时，常见流程是：

1. 根据 `code` 查询短链接。
2. 判断 `status` 是否为 `active`。
3. 判断 `now < expired_at`。
4. 通过后返回 302 跳转。

更接近项目分层时，可以这样安排职责：

- Handler：读取 `code`，处理 HTTP 状态码和跳转。
- Service：判断状态、过期时间、访问策略。
- Repository：只负责参数化 SQL 查询和更新。

这样做的好处是：以后要增加访问日志、黑名单、用户权限时，不会把所有逻辑都塞进一个 handler。

查询时不要把用户输入拼进 SQL：

```text
用户输入不能直接拼进 SQL 字符串。
```

应使用参数化查询：

```sql
SELECT code, target_url, status, expired_at
FROM short_links
WHERE code = ?;
```

如果要记录访问日志，建议不要把日志写入和跳转强绑在同一个不可控流程里。常见做法是先完成核心跳转，再用异步日志或消息队列记录访问事件。初学阶段可以同步写日志，但要给数据库操作设置超时，失败时不要影响主流程太久。

## 五、注意点

后台过期扫描可以把过期链接标记为 `expired`，但访问时仍然要判断 `expired_at`。因为扫描可能延迟，服务也可能重启。

如果要批量更新过期状态，更新语句要带状态条件：

```sql
UPDATE short_links
SET status = 'expired', updated_at = ?
WHERE status = 'active' AND expired_at < ?;
```

这样重复执行也更安全。

如果短链接创建和扣减套餐额度在同一个业务里，要明确事务边界。通常需要在一个事务里完成额度校验、额度扣减、短链接写入，失败时整体回滚。

创建短链接时还要校验 TTL。不要允许用户传负数、零、或者特别大的有效期。常见做法是设置最小值和最大值，例如最短 1 分钟，最长 365 天。

```text
过期时间来自用户输入时，必须校验范围，不能直接信任 ttl。
```

访问接口也要设置超时。短链接跳转接口看起来很简单，但它依赖数据库和日志系统，任何外部依赖都可能变慢。真实项目里 handler 会继承请求的 `context`，repository 查询也要使用这个 `context`。

## 六、常见误区

- 只靠后台任务标记过期：扫描延迟会导致过期链接仍可访问。
- 用本地时区写入数据库：换环境后过期判断可能错。
- 用字符串比较过期时间：格式或时区不同会让结果不可靠。
- 批量过期更新不带状态条件：可能覆盖人工恢复或其他状态。
- 不校验用户传入的有效期：可能创建已经过期或永不过期的异常数据。
- 把跳转、日志、统计都写成一个长事务：外部依赖一慢，核心访问路径就会被拖住。
