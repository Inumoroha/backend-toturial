# 08. 数据库中的时间字段

本节目标：学完后，你能为后端表设计常见时间字段，并知道时间查询、索引和迁移里的保守做法。

简短引入：时间字段不是表里的装饰。`created_at` 用于排序和审计，`updated_at` 用于同步和排查，`expired_at` 用于过期判断。设计不好，后面查询和维护都会很难。

## 一、为什么需要它

真实项目中，几乎每张业务表都会有时间字段：

- `created_at`：记录创建时间。
- `updated_at`：记录最后更新时间。
- `deleted_at`：软删除时间。
- `expired_at`：过期时间。

这些字段会参与查询、排序、归档、审计和数据修复。

```text
时间字段设计要服务于查询、审计和可恢复，而不只是“能存进去”。
```

## 二、基本用法

下面是一份偏通用的 SQL 示例：

```sql
CREATE TABLE short_links (
    id BIGINT PRIMARY KEY,
    code VARCHAR(32) NOT NULL UNIQUE,
    target_url TEXT NOT NULL,
    expired_at TIMESTAMP NOT NULL,
    created_at TIMESTAMP NOT NULL,
    updated_at TIMESTAMP NOT NULL
);

CREATE INDEX idx_short_links_expired_at ON short_links (expired_at);
```

如果项目使用迁移文件，可以把它拆成可执行、可回滚的两段。下面是一个简化示例：

```sql
-- +goose Up
CREATE TABLE short_links (
    id BIGINT PRIMARY KEY,
    code VARCHAR(32) NOT NULL UNIQUE,
    target_url TEXT NOT NULL,
    expired_at TIMESTAMP NOT NULL,
    created_at TIMESTAMP NOT NULL,
    updated_at TIMESTAMP NOT NULL
);

CREATE INDEX idx_short_links_expired_at ON short_links (expired_at);

-- +goose Down
DROP INDEX idx_short_links_expired_at;
DROP TABLE short_links;
```

迁移工具的作用不是让 SQL 变高级，而是让团队知道“哪些变更已经执行过、失败后怎么回退、不同环境是否一致”。

如果使用 Go 写入时间，业务层可以这样准备参数：

```go
now := time.Now().UTC()
expiredAt := now.Add(7 * 24 * time.Hour)
```

这里强调的是“用参数传入”，不要拼 SQL 字符串。

如果你使用 `goose` 这类迁移工具，常见命令如下。命令本身不是重点，重点是把数据库结构变更纳入版本管理。

Windows PowerShell：

```powershell
goose -dir .\migrations postgres $env:DATABASE_URL up
goose -dir .\migrations postgres $env:DATABASE_URL down
```

Linux/macOS：

```bash
goose -dir ./migrations postgres "$DATABASE_URL" up
goose -dir ./migrations postgres "$DATABASE_URL" down
```

`up` 用于执行迁移，`down` 用于回滚最近一次迁移。真实项目中，生产环境执行 `down` 要非常谨慎，尤其是会删表、删字段、丢数据的回滚脚本。

## 三、关键参数/语法/代码结构

`created_at` 通常创建后不变。

`updated_at` 每次业务更新时同步更新。

`expired_at` 表示业务过期点，适合短链接、验证码、库存锁定。

索引要按查询来建。如果经常扫描过期数据，`expired_at` 可以建索引。如果只是展示字段，不一定需要索引。

```text
索引能提升查询，也会增加写入成本，不要因为字段是时间就自动加索引。
```

常见的时间查询有两类：

- 按主键或唯一键查一条记录，再判断时间，例如访问短链接时按 `code` 查询。
- 按时间范围批量扫描，例如后台任务找出已过期链接。

前者优先保证 `code` 有唯一索引，后者才考虑 `expired_at` 索引。不要只看字段名字建索引，要看查询方式。

## 四、真实后端场景示例

下面是短链接创建时的伪代码结构：

```go
package main

import (
	"fmt"
	"time"
)

type ShortLink struct {
	Code      string
	TargetURL string
	ExpiredAt time.Time
	CreatedAt time.Time
	UpdatedAt time.Time
}

func NewShortLink(code, targetURL string, ttl time.Duration, now time.Time) ShortLink {
	return ShortLink{
		Code:      code,
		TargetURL: targetURL,
		ExpiredAt: now.Add(ttl).UTC(),
		CreatedAt: now.UTC(),
		UpdatedAt: now.UTC(),
	}
}

func main() {
	link := NewShortLink("abc123", "https://example.com", 7*24*time.Hour, time.Now())
	fmt.Printf("%+v\n", link)
}
```

真实项目里，这个对象会通过参数化 SQL 或 ORM 写入数据库。关键点是时间字段在进入数据库前已经明确了含义。

## 五、注意点

迁移时间字段时要可回滚。比如给大表新增 `expired_at NOT NULL`，不要直接在线上大表执行高风险语句。更保守的步骤是：

1. 先新增允许为空的字段。
2. 分批回填历史数据。
3. 验证数据完整性。
4. 再加非空约束或应用层校验。

如果字段要参与大量范围查询，建索引前要评估写入压力和查询计划。

回滚脚本也要分清“结构回滚”和“数据回滚”。删除一个新索引通常比较安全，删除一个已经写入业务数据的新字段就很危险。生产上更保守的做法是先停止应用使用这个字段，再观察，再删除。

```text
数据库迁移要先考虑失败时怎么停、怎么退、怎么验证，而不是只考虑怎么加字段。
```

## 六、常见误区

- 使用字符串类型存时间：排序和范围查询都不稳定。
- 所有时间字段都建索引：写入成本变高，索引也占空间。
- 迁移脚本不可回滚：线上出问题时无法快速恢复。
- 用应用服务器本地时区写库：多机器、多环境下容易不一致。
- 在生产库直接手工改表：团队其他环境不知道这次变更，后续迁移容易冲突。
- 回滚脚本直接删字段：如果字段已经有业务数据，回滚可能比上线更危险。

## 七、本节达标标准

- 能设计 `created_at`、`updated_at`、`expired_at` 这类字段。
- 知道数据库时间建议统一 UTC。
- 能说明什么时候给时间字段建索引。
- 知道大表迁移时间字段要分步骤、可回滚。
- 能理解迁移工具在团队协作、环境一致性和回滚中的作用。
- 能区分访问查询和后台扫描对索引的不同需求。
