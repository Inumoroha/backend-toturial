# 09. 可测试的时间代码

本节目标：学完后，你能把依赖当前时间的业务逻辑写成可测试代码，避免测试因为真实时间变化而不稳定。

简短引入：时间相关逻辑如果到处直接调用 `time.Now()`，测试会很难写。订单是否超时、会员是否到期、短链接是否失效，都应该可以用固定时间验证。

## 一、为什么需要它

真实项目里的测试要稳定。今天能过，明天也要能过。依赖真实系统时间的测试，很容易出现偶发失败。

常见做法是：

- 业务判断函数接收 `now time.Time`。
- 服务对象持有一个 `Now func() time.Time`。
- 测试中传入固定时间。

```text
需要测试的时间逻辑，不要把 time.Now() 写死在业务判断深处。
```

## 二、基本用法

```go
package main

import (
	"fmt"
	"time"
)

func IsExpired(expiredAt time.Time, now time.Time) bool {
	return !now.Before(expiredAt)
}

func main() {
	expiredAt := time.Date(2026, 7, 6, 10, 0, 0, 0, time.UTC)
	now := time.Date(2026, 7, 6, 10, 1, 0, 0, time.UTC)

	fmt.Println(IsExpired(expiredAt, now))
}
```

Windows PowerShell：

```powershell
go run .\main.go
```

Linux/macOS：

```bash
go run ./main.go
```

## 三、关键参数/语法/代码结构

把 `now` 作为参数传入，是最简单的可测试方式。它适合纯业务判断函数。

如果某个 service 里很多方法都需要当前时间，可以定义：

```go
type Clock func() time.Time
```

然后在生产环境传 `time.Now`，测试环境传固定函数。

## 四、真实后端场景示例

下面是一个可测试的会员到期判断：

```go
package main

import (
	"fmt"
	"time"
)

type Member struct {
	UserID    int64
	ExpiredAt time.Time
}

type MemberService struct {
	Now func() time.Time
}

func (s MemberService) IsActive(m Member) bool {
	now := s.Now()
	return now.Before(m.ExpiredAt)
}

func main() {
	fixedNow := time.Date(2026, 7, 6, 10, 0, 0, 0, time.UTC)
	service := MemberService{
		Now: func() time.Time { return fixedNow },
	}

	member := Member{
		UserID:    1,
		ExpiredAt: fixedNow.Add(24 * time.Hour),
	}

	fmt.Println(service.IsActive(member))
}
```

真实项目中，测试可以构造多个边界：过期前一秒、刚好过期、过期后一秒。

## 五、注意点

不要为了可测试性引入过重抽象。初学阶段用 `now time.Time` 参数或 `Now func() time.Time` 就够了。

如果有很多服务都需要时间，可以在项目基础设施层统一提供一个 clock，但不要一开始就做复杂框架。

## 六、常见误区

- 测试里 `time.Sleep` 等待过期：测试慢且不稳定。
- 在业务函数深处调用 `time.Now()`：很难覆盖边界条件。
- 只测正常情况，不测刚好等于过期时间：边界错误最容易在线上出现。

## 七、本节达标标准

- 能把当前时间作为参数传入业务判断。
- 能用固定时间测试过期逻辑。
- 能识别哪些代码不适合直接写死 `time.Now()`。
- 知道边界时间必须测试。
