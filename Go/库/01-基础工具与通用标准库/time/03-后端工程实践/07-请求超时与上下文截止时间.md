# 07. 请求超时与上下文截止时间

本节目标：学完后，你能用 `context.WithTimeout` 为后端请求设置超时，避免服务一直等待外部依赖。

简短引入：后端接口常常要访问数据库、Redis、第三方服务。外部依赖一旦变慢，如果没有超时控制，请求会堆积，最后拖垮整个服务。

## 一、为什么需要它

真实项目中，超时不是为了“让代码快”，而是为了保护系统。一个请求超过合理时间，就应该停止继续等待，把资源还给服务。

常见场景是：

- 查询数据库超过 2 秒就失败。
- 调用支付服务超过 3 秒就返回处理中。
- 后台任务整体超过 30 秒就中断。

```text
任何外部调用都应该有超时边界。
```

## 二、基本用法

```go
package main

import (
	"context"
	"fmt"
	"time"
)

func slowCall(ctx context.Context) error {
	select {
	case <-time.After(3 * time.Second):
		fmt.Println("调用完成")
		return nil
	case <-ctx.Done():
		return ctx.Err()
	}
}

func main() {
	ctx, cancel := context.WithTimeout(context.Background(), 1*time.Second)
	defer cancel()

	if err := slowCall(ctx); err != nil {
		fmt.Println("调用失败:", err)
	}
}
```

Windows PowerShell：

```powershell
go run .\main.go
```

Linux/macOS：

```bash
go run ./main.go
```

## 三、关键参数/语法/代码结构

`context.WithTimeout(parent, d)`：创建一个带超时时间的上下文。

`defer cancel()`：释放上下文资源。即使超时会自动触发，也建议显式调用。

`ctx.Done()`：当超时、取消或上游请求结束时，这个 channel 会关闭。

`ctx.Err()`：获取结束原因，常见是 `context deadline exceeded` 或 `context canceled`。

## 四、真实后端场景示例

下面模拟创建订单时调用库存服务锁库存：

```go
package main

import (
	"context"
	"fmt"
	"time"
)

func LockStock(ctx context.Context, skuID int64, quantity int) error {
	select {
	case <-time.After(2 * time.Second):
		return nil
	case <-ctx.Done():
		return fmt.Errorf("锁库存超时: %w", ctx.Err())
	}
}

func CreateOrder(ctx context.Context) error {
	ctx, cancel := context.WithTimeout(ctx, 1*time.Second)
	defer cancel()

	if err := LockStock(ctx, 10001, 1); err != nil {
		return err
	}
	return nil
}

func main() {
	if err := CreateOrder(context.Background()); err != nil {
		fmt.Println(err)
	}
}
```

真实项目中，如果锁库存超时，不代表库存一定失败，也可能是对方成功了但响应没回来。因此要结合幂等号、订单状态和补偿任务处理。

## 五、注意点

超时时间不要随手写。它通常来自接口 SLA、数据库性能、第三方服务约定和用户体验要求。

如果一个 HTTP 请求总超时是 3 秒，里面的数据库查询、Redis 调用、第三方调用要共享这个边界，不应该每一层都重新给一个很长的 timeout。

## 六、常见误区

- 创建了 timeout 但下游函数不接收 `ctx`：超时不会真正传递下去。
- 忘记 `defer cancel()`：资源释放不及时。
- 超时后直接重试支付、扣库存等操作：可能造成重复扣款或重复扣减，必须先考虑幂等。

## 七、本节达标标准

- 能用 `context.WithTimeout` 设置请求超时。
- 能在函数参数中传递 `context.Context`。
- 能区分超时失败和业务失败。
- 知道外部调用超时后要考虑幂等和补偿。
