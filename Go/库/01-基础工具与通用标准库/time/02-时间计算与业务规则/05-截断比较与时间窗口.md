# 05. 截断、比较与时间窗口

本节目标：学完后，你能实现按天查询、活动时间窗判断、订单列表起止时间过滤。

简短引入：后端经常要问“这个订单是不是今天创建的”“活动是否正在进行”“查询最近 7 天数据”。这些问题本质上都是时间比较和时间窗口。

## 一、为什么需要它

时间窗口在业务里非常常见：

- 订单列表按创建时间筛选。
- 活动在开始时间和结束时间之间才可报名。
- 每天统计新增用户。

窗口判断的直觉是：先确定开始边界和结束边界，再判断目标时间是否落在范围内。

```text
时间范围建议使用左闭右开：[start, end)，能减少边界重复统计。
```

## 二、基本用法

```go
package main

import (
	"fmt"
	"time"
)

func InRange(t, start, end time.Time) bool {
	return !t.Before(start) && t.Before(end)
}

func main() {
	start := time.Date(2026, 7, 6, 0, 0, 0, 0, time.Local)
	end := start.Add(24 * time.Hour)
	orderTime := time.Date(2026, 7, 6, 10, 30, 0, 0, time.Local)

	fmt.Println(InRange(orderTime, start, end))
}
```

Windows PowerShell：

```powershell
go run .\main.go
```

Linux/macOS：

```bash
go run ./main.go
```

## 三、关键参数/语法/代码结构

`t.Before(x)`：判断 `t` 是否早于 `x`。

`t.After(x)`：判断 `t` 是否晚于 `x`。

`t.Equal(x)`：判断两个时间点是否相等。

`t.Truncate(d)`：按时间长度截断。它适合处理固定长度的窗口，但对“业务自然日”要谨慎，因为自然日和时区有关。

如果要得到某个业务时区下的一天，推荐显式构造当天零点和下一天零点：

```go
func DayRangeInLocation(day time.Time, loc *time.Location) (time.Time, time.Time) {
	y, m, d := day.In(loc).Date()
	start := time.Date(y, m, d, 0, 0, 0, 0, loc)
	end := start.AddDate(0, 0, 1)
	return start, end
}
```

这个函数在真实项目里常用于“按天查订单”“按天统计新增用户”。注意它返回的是业务时区下的开始和结束时间，查数据库前可以转成 UTC。

## 四、真实后端场景示例

下面模拟活动报名时间判断：

```go
package main

import (
	"fmt"
	"time"
)

type Campaign struct {
	Name      string
	StartsAt  time.Time
	EndsAt    time.Time
}

func CanJoin(c Campaign, now time.Time) bool {
	return !now.Before(c.StartsAt) && now.Before(c.EndsAt)
}

func main() {
	campaign := Campaign{
		Name:     "夏季活动",
		StartsAt: time.Date(2026, 7, 1, 0, 0, 0, 0, time.Local),
		EndsAt:   time.Date(2026, 8, 1, 0, 0, 0, 0, time.Local),
	}

	fmt.Println(CanJoin(campaign, time.Now()))
}
```

真实项目中，`CanJoin` 通常会在 service 层里使用。数据库可以先按时间粗筛，业务层再做一次明确判断。

再看一个更贴近订单查询的例子：用户在管理后台选择 `2026-07-06`，后端要查询上海时间这一天创建的订单。

```go
package main

import (
	"fmt"
	"time"
)

func DayRangeInLocation(day time.Time, loc *time.Location) (time.Time, time.Time) {
	y, m, d := day.In(loc).Date()
	start := time.Date(y, m, d, 0, 0, 0, 0, loc)
	end := start.AddDate(0, 0, 1)
	return start, end
}

func main() {
	shanghai, err := time.LoadLocation("Asia/Shanghai")
	if err != nil {
		panic(err)
	}

	day := time.Date(2026, 7, 6, 12, 0, 0, 0, shanghai)
	start, end := DayRangeInLocation(day, shanghai)

	fmt.Println("query start UTC:", start.UTC().Format(time.RFC3339))
	fmt.Println("query end UTC:", end.UTC().Format(time.RFC3339))
}
```

对应 SQL 通常写成：

```sql
SELECT id, user_id, total_amount, created_at
FROM orders
WHERE created_at >= ? AND created_at < ?
ORDER BY created_at DESC;
```

这里的两个参数传 `start.UTC()` 和 `end.UTC()`。这样做的好处是：数据库里仍然保持 UTC，业务上又能按上海自然日查询。

## 五、注意点

查询数据库时，时间窗口也建议用左闭右开：

```sql
WHERE created_at >= ? AND created_at < ?
```

这样按天分页、按月统计时，不会把一天最后一秒写成 `23:59:59`，也不会漏掉毫秒、微秒级别的数据。

如果你要查“上海时区的今天”，应该先构造上海时区的今日零点和明日零点，再转 UTC 去查数据库。

不要直接用 `time.Now().Truncate(24 * time.Hour)` 得到“今天零点”。它按固定 24 小时截断，参考点不是你的业务时区零点。真实项目里，这类代码常常在本地看起来没问题，上线后按天统计就偏一天。

当时间窗口来自用户输入时，还要限制最大跨度。例如订单列表允许查 31 天，不要让用户一次查 5 年。否则即使 SQL 写对了，也可能拖慢数据库。

```text
时间窗口不仅要判断对，还要限制查询范围，避免一次请求扫太多数据。
```

## 六、常见误区

- 使用 `<= 23:59:59` 作为结束条件：数据库里可能有 `23:59:59.500`，会被漏掉。
- 忽略时区就按天统计：跨时区用户的数据会被分到错误日期。
- 直接比较字符串时间：格式不统一或带时区后容易出错。
- 用 `Truncate(24*time.Hour)` 当业务自然日零点：它不是按 `Asia/Shanghai` 这样的业务时区来截断。
- 不限制查询跨度：用户传很大的时间范围时，可能导致慢查询，影响生产库。

## 七、本节达标标准

- 能写出左闭右开的时间范围判断。
- 能用 `Before`、`After`、`Equal` 做时间比较。
- 知道数据库时间筛选建议使用 `created_at >= start AND created_at < end`。
- 知道按自然日统计必须考虑业务时区。
- 能把业务时区的自然日转换成 UTC 查询边界。
- 能为时间范围查询设置合理跨度限制。
