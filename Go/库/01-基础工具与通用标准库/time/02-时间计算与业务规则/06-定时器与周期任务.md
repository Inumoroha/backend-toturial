# 06. 定时器与周期任务

本节目标：学完后，你能用 `Timer` 和 `Ticker` 实现延迟执行和周期轮询，并知道它们不适合替代可靠任务队列。

简短引入：后端项目里常见“过一会儿再做”和“每隔一段时间做一次”。Go 的 `time.Timer` 和 `time.Ticker` 能处理简单场景，但生产任务还要考虑进程重启、重复执行和可恢复。

## 一、为什么需要它

常见场景包括：

- 服务启动后延迟刷新缓存。
- 每 10 秒上报一次运行状态。
- 开发环境中定期扫描过期订单。

这些场景可以先用 `Timer` 和 `Ticker` 建立直觉。涉及订单关闭、库存释放、消息通知这种不能丢的任务时，通常还要结合数据库状态、消息队列或调度系统。

```text
Timer 和 Ticker 适合进程内临时调度，不适合单独承担关键业务任务。
```

## 二、基本用法

```go
package main

import (
	"fmt"
	"time"
)

func main() {
	timer := time.NewTimer(2 * time.Second)
	<-timer.C
	fmt.Println("延迟任务执行")

	ticker := time.NewTicker(1 * time.Second)
	defer ticker.Stop()

	for i := 0; i < 3; i++ {
		<-ticker.C
		fmt.Println("周期任务执行", i+1)
	}
}
```

Windows PowerShell：

```powershell
go run .\main.go
```

Linux/macOS：

```bash
go run ./main.go
```

## 三、关键参数/语法/代码结构

`time.NewTimer(d)`：创建一个只触发一次的定时器。

`time.NewTicker(d)`：创建一个周期触发的计时器。

`ticker.Stop()`：停止 ticker，避免资源继续占用。

`<-ticker.C`：等待下一次触发。

真实项目中，循环任务通常还需要监听退出信号或 `context`，否则服务关闭时可能无法优雅退出。

## 四、真实后端场景示例

下面模拟每 2 秒扫描一次过期短链接：

```go
package main

import (
	"context"
	"fmt"
	"time"
)

func ScanExpiredLinks(ctx context.Context) {
	ticker := time.NewTicker(2 * time.Second)
	defer ticker.Stop()

	for {
		select {
		case <-ctx.Done():
			fmt.Println("停止扫描")
			return
		case now := <-ticker.C:
			fmt.Println("扫描过期短链接:", now.Format("15:04:05"))
		}
	}
}

func main() {
	ctx, cancel := context.WithTimeout(context.Background(), 7*time.Second)
	defer cancel()

	ScanExpiredLinks(ctx)
}
```

这个例子只做扫描演示。真实项目中，扫描到过期数据后，更新数据库时要有幂等条件，例如只更新仍处于 `active` 状态的数据。

## 五、注意点

如果进程重启，内存里的 `Timer` 和 `Ticker` 都会消失。因此关键任务不能只存在内存里。

更稳妥的方式是：

- 数据库里保存任务状态和截止时间。
- 后台任务按时间窗口扫描。
- 更新时加状态条件，保证重复执行也安全。
- 重要任务使用消息队列或可靠调度系统。

## 六、常见误区

- 忘记 `ticker.Stop()`：任务结束后资源仍可能占用。
- 用 ticker 精确控制关键业务时间：服务暂停或任务执行变慢都会导致漂移。
- 认为定时扫描一次就能保证过期即刻生效：真实项目里通常还会在访问时再次判断过期时间。

## 七、本节达标标准

- 能使用 `Timer` 做一次性延迟执行。
- 能使用 `Ticker` 做周期任务。
- 能用 `context` 停止后台循环。
- 知道关键业务任务不能只依赖进程内定时器。
