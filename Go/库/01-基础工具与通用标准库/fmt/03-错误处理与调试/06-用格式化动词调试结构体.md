# 06. 用格式化动词调试结构体

本节目标：学会用 `%v`、`%+v`、`%#v`、`%T` 调试后端结构体。

简短引入：后端项目里大量数据会放在结构体中，例如用户、订单、权限、配置。选对格式化动词，可以更快看清结构和值。

## 一、为什么需要它

你经常会遇到这种问题：接口返回不对、字段没赋值、配置没读到、权限判断异常。此时直接看结构体比逐个字段打印更快。

不同动词适合不同阶段：

- `%v`：快速看值。
- `%+v`：结构体带字段名，调试常用。
- `%#v`：接近 Go 语法的表示，适合看类型细节。
- `%T`：看变量真实类型。

## 二、基本用法

```go
package main

import "fmt"

type User struct {
	ID   int
	Name string
	Role string
}

func main() {
	u := User{ID: 1001, Name: "alice", Role: "admin"}

	fmt.Printf("v  = %v\n", u)
	fmt.Printf("+v = %+v\n", u)
	fmt.Printf("#v = %#v\n", u)
	fmt.Printf("T  = %T\n", u)
}
```

Windows PowerShell：

```powershell
go run .\main.go
```

Linux/macOS：

```bash
go run ./main.go
```

## 三、关键参数/语法/代码结构

`%+v` 对结构体很实用，因为它会显示字段名。排查接口字段映射时，字段名比字段顺序重要。

`%#v` 更偏开发调试，输出会更像 Go 代码。它不适合直接给业务用户看。

`%T` 在接口类型、泛型、`any`、错误类型判断时很有帮助。

## 四、真实后端场景示例

下面模拟权限判断：

```go
package main

import "fmt"

type Permission struct {
	UserID int
	Action string
	Allowed bool
}

func main() {
	p := Permission{
		UserID: 1001,
		Action: "article:delete",
		Allowed: false,
	}

	fmt.Printf("permission check result: %+v\n", p)
}
```

如果线上排查权限问题，真实项目中通常用结构化日志记录这些字段，而不是直接 `fmt.Printf`。但本地学习和脚本验证时，`%+v` 很方便。

## 五、注意点

调试结构体时要警惕敏感字段。比如 `PasswordHash`、`AccessToken`、`RefreshToken` 不应该直接输出。

`%#v` 不是序列化格式。不要把它当 JSON 存储或传输。

当结构体很大时，一次性打印会让输出难以阅读。真实项目中通常只打印关键字段。

排查接口字段问题时，可以分两步：

1. 用 `%+v` 看服务内部结构体是否已经拿到正确字段。
2. 用 JSON 编码结果看对外响应字段是否符合接口约定。

示例：

```go
package main

import (
	"encoding/json"
	"fmt"
)

type UserResp struct {
	ID   int    `json:"id"`
	Name string `json:"name"`
}

func main() {
	resp := UserResp{ID: 1001, Name: "alice"}

	fmt.Printf("debug struct: %+v\n", resp)

	b, _ := json.Marshal(resp)
	fmt.Println("api json:", string(b))
}
```

可以理解为：`fmt` 帮你看 Go 内部状态，`encoding/json` 帮你看接口真实输出。两者不要混用。

## 六、常见误区

用 `%v` 看结构体后误判字段含义。没有字段名时，字段顺序一变就容易看错。

把 `%#v` 输出保存成业务数据。它是调试表示，不是稳定协议。

打印完整用户对象。用户对象里常常混有敏感字段和内部状态。

## 七、本节达标标准

- 能区分 `%v`、`%+v`、`%#v`、`%T`。
- 能用 `%+v` 调试结构体字段。
- 知道调试输出不能直接暴露敏感字段。
- 知道结构化日志比长期 `fmt.Printf` 更适合生产环境。
