# 11. 订单修复命令行工具

本节目标：用 `fmt` 串起一个小型订单修复工具，练习输入解析、格式化输出、错误包装和生产安全边界。

简短引入：后端工程师经常会写内部工具处理数据问题。这个项目不连接真实数据库，而是用内存数据模拟订单修复流程，重点练习 `fmt` 在真实流程里的位置。

## 一、为什么需要它

一个保守的修复工具至少要做到：

- 能读入明确的操作命令。
- 能输出将要做什么。
- 遇到错误能说明业务上下文。
- 默认先 dry-run，不直接修改生产数据。
- 真正修改前要考虑事务、回滚、审计和权限。

```text
生产修复工具默认 dry-run，确认影响范围后再执行。
```

## 二、基本用法

这个工具接受一条命令：

```text
fix-paid ORD-001 paid
```

含义是：把订单 `ORD-001` 修复为 `paid` 状态。

## 三、关键参数/语法/代码结构

本项目会用到：

- `fmt.Sscan`：解析命令。
- `fmt.Sprintf`：生成审计消息。
- `fmt.Printf`：输出执行计划。
- `fmt.Errorf("%w")`：包装错误。
- `String()`：控制订单展示。

## 四、真实后端场景示例

创建 `main.go`：

```go
package main

import (
	"errors"
	"fmt"
)

var ErrOrderNotFound = errors.New("order not found")

type Order struct {
	ID     string
	UserID int
	Status string
}

func (o Order) String() string {
	return fmt.Sprintf("order{id=%s user_id=%d status=%s}", o.ID, o.UserID, o.Status)
}

func parseCommand(input string) (action string, orderID string, status string, err error) {
	n, scanErr := fmt.Sscan(input, &action, &orderID, &status)
	if scanErr != nil || n != 3 {
		return "", "", "", fmt.Errorf("parse command %q: want 3 fields", input)
	}
	return action, orderID, status, nil
}

func findOrder(orders map[string]Order, id string) (Order, error) {
	order, ok := orders[id]
	if !ok {
		return Order{}, fmt.Errorf("find order %s: %w", id, ErrOrderNotFound)
	}
	return order, nil
}

func main() {
	input := "fix-paid ORD-001 paid"
	dryRun := true

	orders := map[string]Order{
		"ORD-001": {ID: "ORD-001", UserID: 1001, Status: "pending"},
	}

	action, orderID, nextStatus, err := parseCommand(input)
	if err != nil {
		fmt.Println("invalid command:", err)
		return
	}

	order, err := findOrder(orders, orderID)
	if err != nil {
		if errors.Is(err, ErrOrderNotFound) {
			fmt.Println("cannot fix:", err)
			return
		}
		fmt.Println("unexpected error:", err)
		return
	}

	audit := fmt.Sprintf("action=%s order_id=%s from=%s to=%s", action, order.ID, order.Status, nextStatus)
	fmt.Println("plan:", audit)
	fmt.Printf("current: %+v\n", order)

	if dryRun {
		fmt.Println("dry-run enabled, no data changed")
		return
	}

	order.Status = nextStatus
	orders[order.ID] = order
	fmt.Println("updated:", order)
}
```

Windows PowerShell：

```powershell
go run .\main.go
```

Linux/macOS：

```bash
go run ./main.go
```

## 五、注意点

这个示例没有连接数据库，是为了把注意力放在 `fmt` 和流程设计上。真实项目里执行订单修复时，建议至少做到：

- 使用参数化查询，不拼 SQL。
- 修改订单和写审计日志放在明确事务边界内。
- 提交事务失败要返回错误。
- 每次修复记录操作人、原因、时间、影响订单。
- 大批量修复要分批，评估索引和锁竞争。
- 先 dry-run 输出影响范围，再执行。
- 保留回滚方案，例如记录修复前状态。

```text
fmt 负责表达信息，不能替代数据库事务、权限校验和审计系统。
```

如果把这个项目继续推进到真实后端工具，建议按下面顺序扩展：

1. 把输入从写死的字符串改成 `flag` 参数，例如 `-action`、`-order-id`、`-status`、`-dry-run`。
2. 查询数据库时使用参数化查询，并设置 `context` 超时。
3. 在同一个事务中更新订单状态和写审计日志。
4. 提交事务失败时返回错误，不要打印后继续。
5. 输出修复前后的关键字段，保留回滚所需的旧状态。
6. 对批量修复增加分批、限速、失败记录和重试策略。

这时 `fmt` 仍然有用，但它只负责把计划、错误和结果表达清楚。真正保证安全的是参数化查询、事务边界、审计记录和可回滚设计。

## 六、常见误区

工具一运行就直接修改数据。真实项目中这非常危险，尤其是生产库。

错误只打印 “failed”。排查时不知道是解析失败、订单不存在还是更新失败。

把命令输入直接拼进 SQL。后台工具同样会有注入风险，不能因为是内部工具就放松。

没有审计记录。修复数据后，如果没人知道改了什么，就很难复盘和回滚。

## 七、本节达标标准

- 能用 `fmt.Sscan` 解析简单修复命令。
- 能用 `fmt.Sprintf` 生成审计消息。
- 能用 `fmt.Errorf("%w")` 包装并判断业务错误。
- 能说明 dry-run、事务、审计、回滚在修复工具中的作用。
- 能把 `fmt` 放在合适位置，而不是让它承担安全和事务职责。
