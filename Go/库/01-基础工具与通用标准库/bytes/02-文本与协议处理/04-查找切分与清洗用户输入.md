# 04. 查找、切分与清洗用户输入

本节目标：学完后你能用 `bytes.Contains`、`bytes.Cut`、`bytes.Fields`、`bytes.TrimSpace` 处理简单后端文本输入。

## 简短引入

后端服务经常会遇到轻量文本协议：请求头、日志行、配置行、任务消息。它们不一定复杂到需要完整解析器，但也不能随便字符串拼凑。

`bytes` 适合处理这种“结构简单、边界明确、输入量可控”的字节文本。

## 一、为什么需要它

比如解析认证头：

```text
Authorization: Bearer token-abc
```

你关心的是：

- 是否存在 `Bearer ` 前缀
- token 是否为空
- 前后空白是否要清理

这些工作用 `bytes` 很直接。

## 二、基本用法

```go
package main

import (
	"bytes"
	"errors"
	"fmt"
)

func parseBearer(raw []byte) ([]byte, error) {
	value := bytes.TrimSpace(raw)
	prefix := []byte("Bearer ")
	if !bytes.HasPrefix(value, prefix) {
		return nil, errors.New("missing bearer prefix")
	}

	token := bytes.TrimSpace(bytes.TrimPrefix(value, prefix))
	if len(token) == 0 {
		return nil, errors.New("empty token")
	}
	return token, nil
}

func main() {
	token, err := parseBearer([]byte("  Bearer token-abc  "))
	if err != nil {
		panic(err)
	}
	fmt.Println(string(token))
}
```

## 三、关键参数和代码结构

`bytes.TrimSpace`

去掉前后空白。适合清洗输入边界，不适合随意删除中间空格。

`bytes.HasPrefix`

判断固定前缀。常见于 token、协议版本、日志类型。

`bytes.Contains`

判断一段字节里是否包含指定片段。常见场景是做轻量过滤，例如跳过包含 `debug=true` 的内部调试日志。它只能回答“有没有”，不能替代权限判断和完整解析。

`bytes.Cut`

按第一个分隔符切成左右两段，比 `Split` 更适合 `key=value`。

`bytes.Fields`

按连续空白拆字段。适合命令行风格、日志行风格的简单文本。

## 四、真实后端场景示例

解析任务队列中的简单消息：

```text
job=send_email user_id=1001 retry=3
```

```go
package main

import (
	"bytes"
	"fmt"
)

func parseJobLine(line []byte) map[string]string {
	result := make(map[string]string)
	if bytes.Contains(line, []byte("debug=true")) {
		result["skip_reason"] = "debug log"
		return result
	}

	fields := bytes.Fields(bytes.TrimSpace(line))

	for _, field := range fields {
		key, value, ok := bytes.Cut(field, []byte("="))
		if !ok || len(key) == 0 || len(value) == 0 {
			continue
		}
		result[string(key)] = string(value)
	}

	return result
}

func main() {
	msg := []byte(" job=send_email user_id=1001 retry=3 ")
	fmt.Println(parseJobLine(msg))
	fmt.Println(parseJobLine([]byte("job=send_email debug=true user_id=1001")))
}
```

真实项目中，如果消息格式会演进，建议使用 JSON、Protobuf 或明确版本化协议。`bytes` 适合内部简单格式，不适合长期复杂协议。

## 五、注意点

输入清洗不能替代校验。

清理空白后，还要判断字段是否合法。例如 `retry` 应该转成整数并限制范围，`user_id` 应该符合业务 ID 格式。

涉及数据库查询时，不要把解析出来的字段拼进 SQL：

```text
解析输入只是第一步，进入数据库前仍然必须参数化查询。
```

## 六、常见误区

误区一：用 `Split` 处理所有场景。

`key=value=extra` 用 `Split` 可能得到多个片段，初学者容易忽略。只需要切第一刀时用 `bytes.Cut` 更清楚。

误区二：清洗后默认可信。

`TrimSpace` 只处理空白，不会帮你判断权限、范围、类型和业务状态。

误区三：自己写复杂协议。

如果字段有转义、嵌套、数组、版本兼容，应使用成熟格式和解析库。

## 七、本节达标标准

- 能用 `bytes.Cut` 解析 `key=value`
- 能用 `bytes.Fields` 处理简单日志或任务消息
- 能区分输入清洗和业务校验
- 能说明为什么解析后的数据库操作仍要参数化
