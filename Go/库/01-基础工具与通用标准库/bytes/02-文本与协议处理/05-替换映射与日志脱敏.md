# 05. 替换、映射与日志脱敏

本节目标：学完后你能用 `bytes.ReplaceAll` 和 `bytes.Map` 做简单内容替换、规范化和日志脱敏。

## 简短引入

后端系统会产生日志、审计记录、错误上下文。日志有用，但也容易泄露手机号、token、邮箱、身份证号等敏感信息。

`bytes` 可以做轻量替换和字符映射，但要记住：它不是完整的数据脱敏平台，只适合边界明确的场景。

## 一、为什么需要它

假设你要把内部日志中的 `token=` 后面内容脱敏，或者把用户提交的文本中的制表符统一成空格。对这种简单替换，`bytes.ReplaceAll` 很直接。

对单个字符的规范化，比如把不可见控制字符去掉，可以用 `bytes.Map`。

## 二、基本用法

```go
package main

import (
	"bytes"
	"fmt"
)

func main() {
	raw := []byte("user=1001\tpath=/api/orders\tstatus=200")
	normalized := bytes.ReplaceAll(raw, []byte("\t"), []byte(" "))

	fmt.Println(string(normalized))
}
```

## 三、关键参数和代码结构

`bytes.ReplaceAll(src, old, new)`

替换全部匹配内容。适合固定片段替换，比如换行规范化、分隔符替换、简单敏感词替换。

`bytes.Map(func(r rune) rune, src)`

逐个 rune 做转换。返回负数表示删除该字符。适合移除控制字符、统一大小写这类字符级处理。

## 四、真实后端场景示例

清洗访问日志：把制表符转空格，移除控制字符，再脱敏固定 token。

```go
package main

import (
	"bytes"
	"fmt"
	"unicode"
)

func normalizeLogLine(line []byte) []byte {
	line = bytes.ReplaceAll(line, []byte("\t"), []byte(" "))

	line = bytes.Map(func(r rune) rune {
		if unicode.IsControl(r) && r != '\n' {
			return -1
		}
		return r
	}, line)

	line = bytes.ReplaceAll(line, []byte("token=secret-token"), []byte("token=***"))
	return line
}

func main() {
	raw := []byte("user=1001\tpath=/api/orders\ttoken=secret-token")
	fmt.Println(string(normalizeLogLine(raw)))
}
```

真实项目中，token 通常不是固定值。更稳的做法是按结构化字段脱敏，例如先解析成键值对，再对 `token`、`password`、`authorization` 字段统一替换。

```text
日志脱敏最好发生在结构化日志字段层，而不是依赖散乱字符串替换。
```

字段级脱敏可以这样写：

```go
func maskKVLogLine(line []byte) []byte {
	fields := bytes.Fields(line)
	var buf bytes.Buffer
	wrote := false

	for _, field := range fields {
		key, value, ok := bytes.Cut(field, []byte("="))
		if !ok {
			continue
		}

		if bytes.Equal(key, []byte("token")) ||
			bytes.Equal(key, []byte("password")) ||
			bytes.Equal(key, []byte("authorization")) {
			value = []byte("***")
		}

		if wrote {
			buf.WriteByte(' ')
		}
		buf.Write(key)
		buf.WriteByte('=')
		buf.Write(value)
		wrote = true
	}

	return buf.Bytes()
}
```

这个例子仍然假设日志字段是简单的 `key=value`，并且字段值里没有空格。真实项目中如果日志已经是结构化字段，应该在写日志前对字段值脱敏，而不是等到一整行文本生成后再补救。

## 五、注意点

脱敏要尽量靠近日志写入边界。

如果你在业务中间层到处手写脱敏，容易漏。更常见的做法是封装统一日志方法，或者在日志字段进入输出前统一处理。

不要记录完整请求体。真实项目中，记录完整请求体可能泄露密码、身份证、银行卡、token，也可能造成日志量暴涨。

## 六、常见误区

误区一：认为 `ReplaceAll` 能完成所有脱敏。

如果敏感字段值不固定，或者格式很多，固定替换会漏掉大量情况。

误区二：脱敏后不保留排查价值。

全部替换成 `***` 虽然安全，但排查问题时可能看不出同一个用户或同一次请求。可以保留后四位、哈希摘要或请求 ID，但要符合公司安全规范。

误区三：在 SQL、迁移脚本、错误返回中复用日志脱敏逻辑。

日志脱敏是输出安全，SQL 安全靠参数化，迁移安全靠工具、评审和可回滚方案。

## 七、本节达标标准

- 能用 `bytes.ReplaceAll` 做固定片段替换
- 能用 `bytes.Map` 移除不需要的字符
- 能说出字符串替换式脱敏的边界
- 能说明为什么生产日志不能随意打印完整请求体
