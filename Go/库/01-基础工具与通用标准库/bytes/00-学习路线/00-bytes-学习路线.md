# 00. bytes 学习路线

本节目标：建立一条从 `[]byte` 基础到后端实践落地的学习路线，知道每一阶段该学什么、练什么、避开什么。

## 简短引入

Go 后端开发经常和字节打交道：HTTP 请求体是字节流，上传文件是字节流，日志落盘前也是字节内容，数据库导入脚本、CSV、协议消息、缓存值也常常先以 `[]byte` 出现。

`bytes` 包不是孤立知识点，它更像连接业务数据和 I/O 世界的小工具。学习它的目标不是背 API，而是在真实项目中判断：

- 什么时候用 `bytes.Buffer`
- 什么时候用 `bytes.Reader`
- 什么时候只需要 `bytes.Contains`、`bytes.TrimSpace`
- 什么时候不该自己拼字符串或自己解析协议

## 一、学习总路线

### 阶段 1：建立直觉

目标：理解 `string` 和 `[]byte` 的边界。

要掌握：

- `[]byte` 表示可变字节序列
- `string` 表示不可变文本
- HTTP、文件、网络中常见数据会先以字节形式出现
- `bytes` 和 `strings` 很像，但处理对象不同

练习：

- 把请求体 `[]byte` 转成字符串打印
- 清洗用户输入前先用 `bytes.TrimSpace`
- 判断上传内容是否包含某个标记

### 阶段 2：构建输出

目标：用 `bytes.Buffer` 逐步拼出响应、日志、导出内容。

要掌握：

- `Write`
- `WriteString`
- `String`
- `Bytes`
- `Reset`

练习：

- 拼接一段订单导出文本
- 生成日志行
- 生成 SQL 迁移预览，但不要直接拼接用户输入

```text
用户输入不能直接拼进 SQL 字符串。
```

### 阶段 3：模拟输入

目标：用 `bytes.Reader` 把内存里的数据伪装成可读取的数据源。

要掌握：

- `bytes.NewReader`
- `Read`
- `Seek`
- `Len`
- 配合 `io.Reader`

练习：

- 测试一个读取请求体的函数
- 模拟上传 CSV
- 给 JSON 解码器传入内存数据

### 阶段 4：清洗和解析

目标：用 `bytes` 处理后端常见轻量文本。

要掌握：

- `Contains`
- `HasPrefix`
- `HasSuffix`
- `Cut`
- `Split`
- `Fields`
- `TrimSpace`
- `ReplaceAll`
- `Map`

练习：

- 解析 `Authorization: Bearer xxx`
- 清洗短链接访问日志
- 脱敏手机号和 token

### 阶段 5：数据安全与内存边界

目标：知道什么时候需要复制，什么时候可以复用。

要掌握：

- `Equal`
- `Clone`
- `Compare`
- `Buffer.Bytes()` 返回的是底层数据视图
- 复用缓冲区要注意生命周期

练习：

- 缓存请求体副本
- 比较签名摘要
- 复用日志缓冲区并避免数据串行污染

### 阶段 6：综合项目

目标：做一个贴近后端的日志清洗器。

项目能力：

- 从输入中读取多行访问日志
- 清洗空白和非法行
- 切分短链接、用户 ID、IP、耗时
- 脱敏敏感字段
- 输出可导入数据库的结构化结果
- 明确哪些部分可以用 `bytes`，哪些必须交给数据库参数化查询、事务、迁移工具

## 二、推荐学习节奏

如果每天学习 1 小时，可以按下面节奏来：

| 天数 | 内容 | 产出 |
| --- | --- | --- |
| 第 1 天 | `string` 与 `[]byte` | 能解释什么时候发生转换 |
| 第 2 天 | `bytes.Buffer` | 能拼接响应、日志、导出内容 |
| 第 3 天 | `bytes.Reader` | 能写基于 `io.Reader` 的测试 |
| 第 4 天 | 查找与切分 | 能解析简单请求头、日志行 |
| 第 5 天 | 替换与脱敏 | 能做日志脱敏 |
| 第 6 天 | 复制与比较 | 能避免缓存数据被意外修改 |
| 第 7 天 | 流式处理 | 能处理较大上传内容 |
| 第 8 天 | 性能边界 | 能说清楚什么时候不优化 |
| 第 9-10 天 | 综合项目 | 完成日志清洗器 |

## 三、阶段验收任务

只看完教程不算学会。每个阶段最好都留下一个能运行的小产物。

### 验收 1：输入清洗函数

写一个函数：

```go
func CleanUserInput(raw []byte) string
```

要求：

- 去掉前后空白
- 空输入返回空字符串
- 不修改调用方后续还要使用的数据
- 能处理用户名、文章标题、短链接 code 这类普通文本

达标判断：你能说清楚为什么最后才转成 `string`。

### 验收 2：订单导出文本

写一个函数：

```go
func BuildOrderExport(orders []Order) string
```

要求：

- 使用 `bytes.Buffer`
- 每行一个订单
- 不直接拼接来自用户输入的 SQL
- 如果字段可能包含逗号，能说明为什么要换成 `encoding/csv`

达标判断：你能区分“构建导出文本”和“构建 SQL”的安全边界。

### 验收 3：导入读取函数

写一个函数：

```go
func ImportFromReader(r io.Reader) error
```

要求：

- 测试时用 `bytes.NewReader`
- 生产场景能说明为什么要限制大小
- 能按行处理，不一次性读取大文件
- 能描述失败行是跳过、记录，还是中断任务

达标判断：你开始从“代码能跑”转向“任务失败后怎么收场”。

### 验收 4：日志清洗器

写一个函数：

```go
func CleanLogLine(line []byte) (map[string]string, error)
```

要求：

- 使用 `bytes.Fields` 和 `bytes.Cut`
- 跳过 token、password 等敏感字段
- 必填字段缺失时返回错误
- 能说明这个格式什么时候应该升级到 JSON Lines

达标判断：你能把 `bytes` 放在合适位置，而不是用它硬扛所有解析任务。

## 四、学习中要保持的工程判断

`bytes` 能帮你处理字节，但不能替代业务层、数据库层和安全边界。

```text
bytes 适合做轻量字节处理，不适合替代 SQL 参数化、JSON 解析器、CSV 解析器、迁移工具和事务控制。
```

真实项目中通常要遵守：

- 数据库写入使用参数化查询
- 多步写入放进事务
- 表结构变化使用迁移工具
- 索引不是越多越好，要考虑写入成本和磁盘成本
- 生产任务先小批量验证，再可回滚上线
- 对外部输入做长度限制和错误处理

## 五、常见误区

误区一：把 `bytes` 当成性能优化专用包。

`bytes` 首先是字节处理工具。只有当你遇到循环拼接、大量临时对象、I/O 适配时，性能收益才更明显。

误区二：学 API 时脱离后端场景。

只背 `Contains`、`Split`、`Buffer` 没有意义。你要把它们放进请求体、日志、导入、响应构建、测试造数这些场景里理解。

误区三：为了练习 `bytes` 手写复杂解析器。

真实项目中，JSON、CSV、SQL、迁移、事务都有各自成熟工具。`bytes` 适合做轻量前置处理，不适合替代这些工具。

## 六、最终达标标准

学完这套教程，你应该能够：

- 说清楚 `bytes` 和 `strings` 的区别
- 用 `bytes.Buffer` 构建输出，而不是大量 `+` 拼接
- 用 `bytes.Reader` 编写不依赖真实文件和网络的测试
- 用 `bytes.Cut`、`bytes.TrimSpace` 等函数处理简单协议文本
- 知道 `Buffer.Bytes()` 和切片复用可能带来的数据污染
- 处理上传、日志、导入时主动考虑大小限制、错误处理和回滚
- 在涉及 SQL、迁移、事务、索引时保持保守工程习惯
