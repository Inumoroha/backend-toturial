# 07. 流式读取上传内容与导入数据

本节目标：学完后你能区分小数据一次性读取和大数据流式处理，并用 `bytes` 配合 `io`、`bufio` 处理上传导入场景。

## 简短引入

后端导入功能很常见：导入用户、商品、库存、订单。初学者容易把上传文件一次性读进内存，然后再处理。小文件可以，大文件就会带来内存风险。

`bytes` 在这里不是主角，主角是流式读取。`bytes` 常用于测试输入、字段清洗和少量中间结果构建。

## 一、为什么需要它

假设运营上传一个 300 MB 的库存文件。如果服务直接 `io.ReadAll`，并发几个请求就可能把内存打满。

更稳的处理方式是：

- 限制上传大小
- 按行或按记录读取
- 批量校验
- 批量入库
- 明确事务边界
- 失败可回滚或可重试

```text
导入任务要先设计失败策略，再写读取代码。
```

## 二、基本用法

下面示例用 `bytes.NewReader` 模拟上传文件，再用 `bufio.Scanner` 按行读取。

```go
package main

import (
	"bufio"
	"bytes"
	"fmt"
	"io"
)

func ImportLines(r io.Reader) error {
	scanner := bufio.NewScanner(r)
	for scanner.Scan() {
		line := bytes.TrimSpace(scanner.Bytes())
		if len(line) == 0 {
			continue
		}
		fmt.Println("import:", string(line))
	}
	return scanner.Err()
}

func main() {
	data := []byte("sku-001,10\n\nsku-002,20\n")
	if err := ImportLines(bytes.NewReader(data)); err != nil {
		panic(err)
	}
}
```

## 三、关键代码结构

`bufio.NewScanner(r)`

按 token 读取，默认按行。适合简单文本行，但单行太长时需要调整缓冲区，或者使用 `bufio.Reader`。

`scanner.Bytes()`

返回当前 token 的字节内容。这个内容会在下一次扫描后失效或变化，需要长期保存时要复制。

`bytes.TrimSpace(scanner.Bytes())`

清洗当前行前后空白。这里马上使用，不跨循环保存，所以可以不复制。

## 四、真实后端场景示例

库存导入通常不是“读一行写一行”这么简单。更稳的是分批处理：

```go
package main

import (
	"bufio"
	"bytes"
	"fmt"
	"io"
)

func ImportInventory(r io.Reader, batchSize int) error {
	scanner := bufio.NewScanner(r)
	batch := make([]string, 0, batchSize)

	for scanner.Scan() {
		line := bytes.TrimSpace(scanner.Bytes())
		if len(line) == 0 || bytes.HasPrefix(line, []byte("#")) {
			continue
		}

		batch = append(batch, string(bytes.Clone(line)))
		if len(batch) == batchSize {
			if err := saveBatch(batch); err != nil {
				return err
			}
			batch = batch[:0]
		}
	}

	if err := scanner.Err(); err != nil {
		return err
	}
	if len(batch) > 0 {
		return saveBatch(batch)
	}
	return nil
}

func saveBatch(lines []string) error {
	fmt.Println("save batch:", lines)
	return nil
}

func main() {
	data := []byte("# sku,stock\nsku-001,10\nsku-002,20\nsku-003,30\n")
	if err := ImportInventory(bytes.NewReader(data), 2); err != nil {
		panic(err)
	}
}
```

真实入库时，`saveBatch` 里应该使用参数化批量写入。是否把整个文件放在一个事务里，要根据业务判断：

- 小批量、强一致：可以一个事务
- 大批量、耗时长：通常拆批次，记录导入任务状态
- 可回滚要求强：设计临时表、版本号、状态切换

## 五、注意点

`bufio.Scanner` 默认 token 最大长度有限。导入超长行时可能报错。常见替代方案：

- 调整 `scanner.Buffer`
- 使用 `bufio.Reader.ReadBytes('\n')`
- 对标准 CSV 使用 `encoding/csv`

如果你确认仍然按行读取，并且业务允许较长单行，可以这样设置缓冲区：

```go
scanner := bufio.NewScanner(r)
buf := make([]byte, 0, 64*1024)
scanner.Buffer(buf, 2*1024*1024) // 单行最多 2 MiB
```

这个设置不是越大越好。真实项目中要先问清楚：一行为什么会这么长，是否是合法业务输入，是否应该拒绝。盲目放大上限可能让恶意输入更容易消耗内存。

数据库索引也要谨慎。导入前后加索引、更新索引、唯一约束冲突都会影响速度和失败策略。

```text
索引能提升查询，也会增加写入成本。导入任务要同时考虑读性能和写性能。
```

## 六、常见误区

误区一：上传文件全部读进内存。

小工具可以这么做，生产导入不能默认这么做。

误区二：一行失败就留下半批脏数据。

要明确事务边界、错误记录、重试策略。失败后用户应该知道导入到哪里、哪些行失败。

误区三：用 `bytes.Split` 一次切完整文件。

这仍然需要完整文件在内存中。大文件应流式处理。

## 七、本节达标标准

- 能用 `bytes.NewReader` 模拟上传内容测试导入逻辑
- 能按行读取并用 `bytes.TrimSpace` 清洗
- 知道什么时候不该使用 `io.ReadAll`
- 能说出导入任务中的事务、索引和回滚考虑
