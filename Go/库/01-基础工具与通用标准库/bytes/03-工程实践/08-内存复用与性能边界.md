# 08. 内存、复用与性能边界

本节目标：学完后你能判断 `bytes.Buffer` 是否需要复用、何时需要复制，以及什么时候不要过早优化。

## 简短引入

性能优化是后端开发绕不开的话题。`bytes` 经常出现在优化讨论里：减少字符串拼接、减少分配、复用缓冲区。

但初学阶段要先记住一句话：

```text
先写正确、可测试、可回滚的代码，再优化真实热点。
```

## 一、为什么需要它

真实服务的性能问题通常来自：

- 数据库慢查询
- 缺少合适索引或索引过多
- 外部接口慢
- 锁竞争
- 大对象频繁分配
- 日志过量

`bytes.Buffer` 能解决的是其中一小类：构建字节内容时的重复分配。不要把它当成所有性能问题的答案。

## 二、基本用法

使用 `Grow` 可以提前给缓冲区一点容量，减少扩容次数。

```go
package main

import (
	"bytes"
	"fmt"
)

func buildLogLine(userID, action string) string {
	var buf bytes.Buffer
	buf.Grow(64)

	buf.WriteString("user_id=")
	buf.WriteString(userID)
	buf.WriteString(" action=")
	buf.WriteString(action)

	return buf.String()
}

func main() {
	fmt.Println(buildLogLine("1001", "create_order"))
}
```

运行：

Windows PowerShell：

```powershell
mkdir bytes-demo-08
cd bytes-demo-08
go mod init bytes-demo-08
notepad main.go
go run .
```


## 三、关键代码结构

`buf.Grow(n)`

告诉缓冲区预计要写入多少字节。不是必须，只在大概知道大小、且构建频繁时考虑。

`buf.Reset()`

清空内容，保留底层容量，适合局部复用。不要跨请求共享同一个 `Buffer`。

`bytes.Clone(buf.Bytes())`

当输出需要长期保存时复制一份，避免后续复用影响已保存内容。

## 四、真实后端场景示例

批量生成审计日志行：

```go
package main

import (
	"bytes"
	"fmt"
)

type Event struct {
	UserID string
	Action string
}

func BuildAuditLines(events []Event) []string {
	lines := make([]string, 0, len(events))
	var buf bytes.Buffer

	for _, event := range events {
		buf.Reset()
		buf.Grow(80)
		buf.WriteString("user_id=")
		buf.WriteString(event.UserID)
		buf.WriteString(" action=")
		buf.WriteString(event.Action)

		lines = append(lines, buf.String())
	}

	return lines
}

func main() {
	lines := BuildAuditLines([]Event{
		{UserID: "1001", Action: "login"},
		{UserID: "1002", Action: "create_order"},
	})
	fmt.Println(lines)
}
```

这里 `buf` 是函数内部局部变量，循环内复用是安全的。返回的是字符串切片，不持有 `buf.Bytes()`。

如果返回的是 `[][]byte`，就必须复制每一行：

```go
line := bytes.Clone(buf.Bytes())
```

## 五、注意点

不要把优化和并发安全混在一起。

`bytes.Buffer` 本身不适合多个 goroutine 同时写。如果多个请求共用一个全局缓冲区，就可能发生数据串写、竞态和安全问题。

生产优化建议：

- 先用压测或 pprof 确认热点
- 优先优化数据库查询和索引
- 再看分配和 CPU 热点
- 小心引入对象池，池化会增加生命周期复杂度

## 六、常见误区

误区一：所有字符串拼接都要换成 `bytes.Buffer`。

简单拼接保持清晰即可。优化应该服务于真实瓶颈。

误区二：复用缓冲区后返回 `buf.Bytes()`。

下次 `Reset` 或写入会影响之前返回的数据。要长期保存必须 `Clone`。

误区三：看到 `Grow` 就随便给很大容量。

容量过大会增加内存占用。只在有估算依据时使用。

## 七、本节达标标准

- 能用 `Grow` 和 `Reset` 做局部缓冲区复用
- 知道全局共享 `bytes.Buffer` 的并发风险
- 能判断什么时候需要 `bytes.Clone`
- 能说明性能优化前应先定位真实热点
