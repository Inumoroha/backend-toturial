# 09. 综合项目：短链接访问日志清洗器

本节目标：学完后你能用 `bytes` 做一个小型日志清洗器，把短链接访问日志转成更适合后续入库的结构化结果。

## 简短引入

短链接系统通常会记录访问日志，例如：

```text
ts=2026-07-06T10:00:00Z code=go123 user_id=1001 ip=192.168.1.8 cost_ms=23 token=secret
```

真实项目里，日志清洗常用于：

- 过滤空行和注释
- 清理不可见字符
- 拆出关键字段
- 脱敏敏感信息
- 生成后续入库任务

本节用 `bytes` 完成轻量处理，同时说明哪些地方不该继续手写。

## 一、为什么需要它

短链接访问日志可能来自网关、应用日志、消息队列或对象存储。原始日志不一定干净，直接入库会带来问题：

- 字段缺失导致查询困难
- token 等敏感字段泄露
- 脏数据破坏统计结果
- 大批量写入失败后难以回滚

所以在入库前，通常要有一个清洗阶段。

## 二、基本用法

完整示例 `main.go`：

```go
package main

import (
	"bufio"
	"bytes"
	"errors"
	"fmt"
	"io"
	"unicode"
)

type AccessLog struct {
	Time   string
	Code   string
	UserID string
	IP     string
	CostMS string
}

func cleanLine(line []byte) []byte {
	line = bytes.TrimSpace(line)
	line = bytes.ReplaceAll(line, []byte("\t"), []byte(" "))
	line = bytes.Map(func(r rune) rune {
		if unicode.IsControl(r) {
			return -1
		}
		return r
	}, line)
	return line
}

func parseLine(line []byte) (AccessLog, error) {
	fields := bytes.Fields(line)
	values := make(map[string]string, len(fields))

	for _, field := range fields {
		key, value, ok := bytes.Cut(field, []byte("="))
		if !ok || len(key) == 0 || len(value) == 0 {
			continue
		}
		if bytes.Equal(key, []byte("token")) {
			continue
		}
		values[string(key)] = string(value)
	}

	log := AccessLog{
		Time:   values["ts"],
		Code:   values["code"],
		UserID: values["user_id"],
		IP:     values["ip"],
		CostMS: values["cost_ms"],
	}

	if log.Time == "" || log.Code == "" || log.UserID == "" {
		return AccessLog{}, errors.New("missing required field")
	}
	return log, nil
}

func CleanAccessLogs(r io.Reader) ([]AccessLog, error) {
	scanner := bufio.NewScanner(r)
	var logs []AccessLog

	for scanner.Scan() {
		line := cleanLine(scanner.Bytes())
		if len(line) == 0 || bytes.HasPrefix(line, []byte("#")) {
			continue
		}

		log, err := parseLine(line)
		if err != nil {
			fmt.Println("skip line:", err)
			continue
		}
		logs = append(logs, log)
	}

	if err := scanner.Err(); err != nil {
		return nil, err
	}
	return logs, nil
}

func main() {
	input := []byte(`
# short link access logs
ts=2026-07-06T10:00:00Z code=go123 user_id=1001 ip=192.168.1.8 cost_ms=23 token=secret
ts=2026-07-06T10:00:01Z code=go999 user_id=1002 ip=192.168.1.9 cost_ms=41 token=secret
bad_line
`)

	logs, err := CleanAccessLogs(bytes.NewReader(input))
	if err != nil {
		panic(err)
	}

	for _, log := range logs {
		fmt.Printf("%+v\n", log)
	}
}
```

## 三、关键代码结构

`CleanAccessLogs(r io.Reader)`

入口只依赖 `io.Reader`，所以它可以处理文件、HTTP 上传、内存测试数据。

`cleanLine(scanner.Bytes())`

逐行清洗，避免一次性加载完整日志。

`bytes.Fields`

把一行日志拆成字段。

`bytes.Cut`

把字段拆成 `key=value`。

`bytes.Equal(key, []byte("token"))`

识别敏感字段并跳过，避免输出 token。

## 四、真实后端场景延伸

如果要把结果写入数据库，不要拼 SQL：

```go
_, err := db.ExecContext(
	ctx,
	"insert into short_link_access_logs(ts, code, user_id, ip, cost_ms) values (?, ?, ?, ?, ?)",
	log.Time,
	log.Code,
	log.UserID,
	log.IP,
	log.CostMS,
)
```

批量导入时更常见的是：

- 每 500 或 1000 条一批
- 每批一个事务，或使用数据库批量接口
- 记录导入任务 ID、成功数、失败数
- 失败行写入错误表或错误文件
- 表结构变更通过迁移工具上线

```text
清洗器负责把脏输入变成可信结构；数据库层负责参数化、事务、约束和可回滚。
```

落到真实项目里，可以先按这个目录拆分：

```text
internal/shortlinklog/
  cleaner.go        // cleanLine、parseLine、CleanAccessLogs
  cleaner_test.go   // 使用 bytes.NewReader 构造输入
  repository.go     // 参数化批量写入数据库
  job.go            // 导入任务状态、批次、失败行记录
```

这样拆不是为了显得复杂，而是为了把职责分清：

- `cleaner.go` 只做字节清洗和字段解析
- `repository.go` 只处理数据库写入、事务和错误返回
- `job.go` 记录任务状态，方便失败后重试或回滚
- 测试优先覆盖清洗规则，不依赖真实数据库

建议补的测试用例：

- 空输入返回空结果
- 注释行和空行会被跳过
- 缺少 `ts`、`code`、`user_id` 的行会被跳过
- `token` 字段不会进入结果
- 包含制表符的日志能被正常清洗
- 坏行不会影响后续好行

## 五、注意点

当前示例只适合字段值不包含空格的日志。如果字段值可能包含空格、引号或转义字符，应该换成结构化日志格式，例如 JSON Lines。

对于生产日志导入，还要考虑：

- 单行最大长度
- 文件总大小限制
- 重复导入如何幂等
- IP、用户 ID、短链 code 的合法性校验
- 数据库唯一约束和索引成本
- 导入失败后的回滚或补偿

## 六、常见误区

误区一：清洗成功就直接入库。

还要做业务校验，例如短链是否存在、用户是否有效、耗时是否为数字。

误区二：遇到坏行直接中断整个任务。

这取决于业务。如果是强一致配置导入，可以中断。如果是日志导入，通常记录坏行并继续。

误区三：日志表索引越多越好。

短链接访问日志通常写入量大。索引会提升查询，但也会拖慢写入并占用磁盘。生产中要根据查询场景设计索引。

误区四：把清洗逻辑和数据库事务混在一起。

清洗、校验、入库最好有清晰边界。边界越清楚，失败越容易定位和回滚。

## 七、本节达标标准

- 能实现一个基于 `io.Reader` 的日志清洗函数
- 能用 `bytes.TrimSpace`、`bytes.Fields`、`bytes.Cut` 解析简单日志
- 能跳过敏感字段，避免 token 进入输出
- 能说出生产导入时事务、索引、迁移和回滚要如何考虑
- 能判断什么时候应该从轻量 `bytes` 解析升级到 JSON Lines 等结构化格式
