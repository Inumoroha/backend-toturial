# 00. Go 后端 LOG 学习路线

本路线目标：帮助你从“会打印日志”逐步走到“能在后端项目中设计可排查、可维护、可上线的日志方案”。

## 一、学习目标

学完这套内容后，你应该能够做到：

- 在 Go 项目中使用 `log` 和 `log/slog` 记录日志。
- 区分调试日志、业务日志、错误日志和审计日志。
- 在 HTTP 服务中记录请求耗时、状态码、请求 ID 和错误信息。
- 在真实项目中处理日志文件、日志轮转、多环境配置和敏感信息脱敏。
- 根据日志排查线上问题，并知道哪些日志适合接入告警。

```text
日志不是为了“多打印一点”，而是为了让系统出问题时能还原关键事实。
```

## 二、阶段路线

### 阶段 1：建立日志直觉

学习内容：

- 日志在后端项目中的作用。
- 标准库 `log` 的基本用法。
- 日志时间、文件名、行号、前缀这些信息为什么重要。

对应文章：

- 01. 认识日志与标准库 log
- 02. 日志级别与输出格式

阶段目标：

- 能在命令行程序和简单服务中输出清晰日志。
- 知道 `Println`、`Fatal`、`Panic` 的使用边界。
- 不把日志当成业务返回值，也不把日志当成错误处理本身。

### 阶段 2：写出可运行、可保留的日志

学习内容：

- 把日志输出到文件。
- 正确处理文件打开失败。
- 理解标准输出、标准错误和日志文件的区别。

对应文章：

- 03. 日志输出到文件与错误处理

阶段目标：

- 能把后台任务、迁移脚本、定时任务的日志保存下来。
- 知道为什么生产环境通常由容器平台或日志代理采集标准输出。

### 阶段 3：进入结构化日志

学习内容：

- 使用 `log/slog` 输出 JSON 日志。
- 理解字段化日志的价值。
- 使用 `context.Context`、请求 ID、用户 ID 串联一次请求。

对应文章：

- 04. slog 结构化日志入门
- 05. 在 HTTP 服务中记录请求日志

阶段目标：

- 能输出便于检索和聚合的日志。
- 能在 HTTP 中间件中记录方法、路径、状态码、耗时和 request_id。

### 阶段 4：形成业务日志边界

学习内容：

- 什么时候记录 Info、Warn、Error。
- 错误应该在哪里记录。
- 如何避免重复记录同一个错误。

对应文章：

- 06. 业务日志与错误日志边界

阶段目标：

- 能围绕用户、订单、库存、权限等后端业务记录关键动作。
- 能让日志服务于排查，而不是制造噪音。

### 阶段 5：工程化与生产安全

学习内容：

- 日志脱敏。
- 日志轮转。
- 多环境日志配置。
- 生产环境中的日志风险。

对应文章：

- 07. 日志脱敏与敏感信息保护
- 08. 日志轮转与文件管理
- 09. 多环境日志配置

阶段目标：

- 不在日志中泄露密码、Token、手机号、身份证、银行卡等敏感信息。
- 知道文件日志必须考虑大小、保留时间和磁盘风险。
- 能根据开发、测试、生产环境调整日志级别和格式。

### 阶段 6：项目落地与排查

学习内容：

- 在短链接服务中综合使用日志。
- 根据日志排查慢请求、错误请求、参数异常和依赖故障。
- 建立生产日志检查清单。

对应文章：

- 10. 短链接服务日志实践
- 11. 日志排查与告警思路
- 12. 生产日志检查清单

阶段目标：

- 能把日志设计嵌入一个真实后端功能。
- 能基于日志提出排查路径，而不是只看错误堆栈。
- 能上线前检查日志是否安全、可读、可查、可控。

## 三、推荐实践节奏

第一轮学习：

- 每天 2 到 3 节。
- 每节代码都复制运行。
- 重点看输出是否符合预期。

第二轮练习：

- 把示例改造成自己的业务，比如文章发布、订单创建、库存扣减。
- 增加 request_id、user_id、order_id 等字段。
- 故意制造错误，观察日志是否能定位原因。

第三轮项目化：

- 写一个小服务。
- 使用中间件记录请求日志。
- 对敏感字段做脱敏。
- 把开发环境和生产环境配置分开。

## 四、最终项目建议

建议做一个短链接服务，功能不用复杂：

- 创建短链接。
- 访问短链接并跳转。
- 查询短链接访问次数。
- 模拟链接不存在、过期、参数错误和存储失败。

这个项目很适合练日志，因为它有明确的请求入口、业务动作、异常路径和统计需求。

```text
一套好的日志方案，应该能回答：谁在什么时候做了什么，结果如何，失败原因是什么，下一步该查哪里。
```
