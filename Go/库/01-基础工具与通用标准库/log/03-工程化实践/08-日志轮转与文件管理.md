# 08. 日志轮转与文件管理

本节目标：学完后，你能理解日志轮转的必要性，并知道 Go 服务写文件日志时应该如何保守处理。

## 简短引入

日志文件如果一直追加，迟早会变得很大。磁盘写满后，轻则服务无法写日志，重则数据库、消息队列、系统进程都受影响。日志轮转就是定期切分、压缩、删除旧日志。

## 一、为什么需要它

文件日志有三个核心问题：

- 单个文件过大，不方便查看和传输。
- 历史日志无限保留，磁盘会被占满。
- 多进程同时写文件，可能导致竞争和混乱。

真实项目中常见做法是：

- 容器环境：应用写标准输出，由平台采集和轮转。
- 传统服务器：使用系统工具或日志代理做轮转。
- 小型工具：应用内使用轮转库，但要控制复杂度。

```text
生产环境不要让应用无限写一个日志文件。
```

## 二、基本用法

标准库没有内置日志轮转。学习阶段可以先用“按日期写文件”的方式理解思路。

```go
package main

import (
	"fmt"
	"log"
	"os"
	"time"
)

func openDailyLogFile() (*os.File, error) {
	name := fmt.Sprintf("app-%s.log", time.Now().Format("2006-01-02"))
	return os.OpenFile(name, os.O_CREATE|os.O_WRONLY|os.O_APPEND, 0644)
}

func main() {
	file, err := openDailyLogFile()
	if err != nil {
		log.Fatal("open log file failed:", err)
	}
	defer file.Close()

	logger := log.New(file, "INFO ", log.LstdFlags|log.Lshortfile)
	logger.Println("daily log started")
}
```

运行后会生成类似 `app-2026-07-06.log` 的文件。

## 三、关键参数/语法/代码结构

这个例子只是帮助你建立轮转直觉：不同日期写入不同文件。但它不是完整生产方案，因为服务跨天运行时不会自动切换文件。

生产中更常见的选择：

- 由 Linux `logrotate` 管理文件。
- 由容器运行时和日志系统采集标准输出。
- 使用成熟 Go 日志轮转库，比如 lumberjack。

这里不急着引入第三方库，是因为初学阶段更重要的是理解风险：文件会增长，磁盘会满，日志必须有生命周期。

## 四、真实后端场景示例

假设你在传统服务器上跑一个订单补偿程序，可以按天生成日志文件：

```go
package main

import (
	"fmt"
	"log"
	"os"
	"time"
)

func main() {
	logName := fmt.Sprintf("order-compensate-%s.log", time.Now().Format("2006-01-02"))
	file, err := os.OpenFile(logName, os.O_CREATE|os.O_WRONLY|os.O_APPEND, 0644)
	if err != nil {
		log.Fatal("open log file failed:", err)
	}
	defer file.Close()

	logger := log.New(file, "INFO ", log.LstdFlags)
	logger.Println("job started", "job", "order_compensate")
	logger.Println("job finished", "success", 98, "failed", 2)
}
```

这个文件名能帮助运维按日期归档，也方便你查看某一天的执行记录。

## 五、注意点

应用内轮转不是唯一答案。很多公司会要求应用只写标准输出，日志采集由基础设施统一处理。这种方式更适合容器、Kubernetes 和集中式日志平台。

如果使用文件日志，要和运维约定：

- 日志目录在哪里。
- 最大保留多少天。
- 单个文件最大多大。
- 是否压缩旧日志。
- 磁盘使用率超过阈值怎么办。

不要把日志文件放在不可控目录，也不要和数据库数据目录混在一起。

## 六、常见误区

误区一：以为按天命名就是完整轮转。  
长时间运行的服务不会自动切换到新日期文件，还需要重新打开文件或使用工具。

误区二：日志越久越好。  
日志保留时间要平衡排查需要、存储成本和合规要求。

误区三：应用和运维各管各的。  
日志路径、权限、采集、清理必须提前约定，否则上线后很容易出事故。
