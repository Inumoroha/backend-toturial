# 07. 日志脱敏与敏感信息保护

本节目标：学完后，你能识别日志中的敏感信息，并在输出前进行简单脱敏。

## 简短引入

日志经常被多人查看，也可能进入第三方日志平台。如果把密码、Token、手机号、身份证号直接写进去，风险非常高。后端工程师必须从一开始就养成日志脱敏习惯。

## 一、为什么需要它

常见敏感信息包括：

- 密码、验证码、短信码。
- Authorization、Cookie、Refresh Token。
- 手机号、邮箱、身份证号。
- 银行卡、支付流水的敏感部分。
- 详细地址、个人备注。

```text
能不记就不记；必须记时，只记排查所需的最小信息。
```

## 二、基本用法

下面对手机号做简单脱敏。

```go
package main

import (
	"log/slog"
	"os"
)

func maskPhone(phone string) string {
	if len(phone) != 11 {
		return "***"
	}
	return phone[:3] + "****" + phone[7:]
}

func main() {
	logger := slog.New(slog.NewJSONHandler(os.Stdout, nil))

	phone := "13812345678"
	logger.Info("send sms code",
		"phone", maskPhone(phone),
	)
}
```

输出中不会出现完整手机号。

## 三、关键参数/语法/代码结构

脱敏的核心不是技巧，而是边界：

- 密码、验证码、Token：通常完全不记录。
- 手机号：可以记录前三后四。
- 邮箱：可以保留域名或部分用户名。
- 身份证、银行卡：除非业务强需要，否则不要进日志。

对于 `slog`，你可以在记录前对字段脱敏，也可以封装一个专门的日志工具函数。但初学阶段先做到“输出前处理字段”即可。

## 四、真实后端场景示例

下面模拟登录失败日志。

```go
package main

import (
	"errors"
	"log/slog"
	"os"
)

func maskEmail(email string) string {
	if len(email) < 3 {
		return "***"
	}
	return email[:2] + "***"
}

func login(email, password string) error {
	if password != "correct-password" {
		return errors.New("invalid credentials")
	}
	return nil
}

func main() {
	logger := slog.New(slog.NewJSONHandler(os.Stdout, nil))

	email := "alice@example.com"
	password := "123456"

	if err := login(email, password); err != nil {
		logger.Warn("login failed",
			"email", maskEmail(email),
			"error", err,
		)
	}
}
```

注意这里没有记录 password。真实项目中，登录失败也不要把验证码、Token、Cookie 打出来。

## 五、注意点

不要依赖“日志平台会保护好”。日志通常会被采集、转发、备份、查询，流转链路越长，泄露面越大。

不要在错误信息里夹带敏感信息。比如第三方 SDK 返回的错误可能包含请求参数，记录前要确认内容。

日志脱敏要尽量靠近输出位置。因为越早把敏感信息变成安全格式，越不容易被后续误用。

## 六、常见误区

误区一：开发环境可以随便打敏感信息。  
开发日志也可能被截图、提交、共享或上传到协作工具。

误区二：只脱敏业务字段，忽略请求头。  
`Authorization`、`Cookie` 往往比手机号更危险。

误区三：为了排查方便记录完整请求。  
短期方便，长期会带来合规和安全风险。

## 七、本节达标标准

- 能识别常见敏感信息。
- 能对手机号、邮箱等字段做简单脱敏。
- 知道密码、验证码、Token 通常不应该进入日志。
- 能在登录、短信、支付等场景主动控制日志内容。
