# 03. 日志输出到文件与错误处理

本节目标：学完后，你能把 Go 日志写入文件，并正确处理文件打开失败和关闭时机。

## 简短引入

很多后端任务不是一直在终端前运行的。比如数据迁移、订单补偿、定时清理脚本，运行完之后你还需要知道它处理了多少数据、失败了哪些记录。这时把日志写入文件就很有用。

## 一、为什么需要它

可以理解为，文件日志适合“需要留痕”的任务。常见场景是：

- 迁移用户数据，需要记录每批处理结果。
- 补偿失败订单，需要记录失败订单号。
- 本地开发没有日志平台，需要临时保留日志。

但文件日志也有风险：文件会变大，磁盘会被写满，权限可能不足。因此生产环境要结合轮转和采集方案。

```text
只要日志写入文件，就必须考虑文件大小、保留时间、权限和磁盘空间。
```

## 二、基本用法

示例代码：

```go
package main

import (
	"log"
	"os"
)

func main() {
	file, err := os.OpenFile("app.log", os.O_CREATE|os.O_WRONLY|os.O_APPEND, 0644)
	if err != nil {
		log.Fatal("open log file failed:", err)
	}
	defer file.Close()

	logger := log.New(file, "INFO ", log.LstdFlags|log.Lshortfile)
	logger.Println("migration started")
	logger.Println("migration success", "total", 100)
}
```

运行：

```bash
go run .
```

查看日志。

Windows PowerShell：

```powershell
Get-Content .\app.log
```

Linux/macOS：

```bash
cat ./app.log
```

## 三、关键参数/语法/代码结构

`os.OpenFile` 的三个关键部分：

- `os.O_CREATE`：文件不存在就创建。
- `os.O_WRONLY`：只写。
- `os.O_APPEND`：追加写入，不覆盖旧日志。

权限 `0644` 表示文件所有者可读写，其他用户只读。Windows 下权限表现会略有差异，但写法可以保留。

为什么打开文件失败可以用 `log.Fatal`？因为这是程序启动阶段的基础依赖，如果日志文件都无法创建，这类脚本通常不应该继续运行。但在 HTTP 请求处理中，不应该用 `Fatal` 处理普通业务错误。

## 四、真实后端场景示例

下面模拟“订单补偿脚本”，失败订单会继续处理，不会因为一条失败就退出。

```go
package main

import (
	"errors"
	"log"
	"os"
)

func compensateOrder(orderID int64) error {
	if orderID == 1003 {
		return errors.New("payment status not confirmed")
	}
	return nil
}

func main() {
	file, err := os.OpenFile("order-compensate.log", os.O_CREATE|os.O_WRONLY|os.O_APPEND, 0644)
	if err != nil {
		log.Fatal("open log file failed:", err)
	}
	defer file.Close()

	logger := log.New(file, "INFO ", log.LstdFlags|log.Lshortfile)
	orders := []int64{1001, 1002, 1003, 1004}

	for _, orderID := range orders {
		if err := compensateOrder(orderID); err != nil {
			logger.Println("compensate order failed", "order_id", orderID, "error", err)
			continue
		}
		logger.Println("compensate order success", "order_id", orderID)
	}
}
```

这个例子里，一条订单失败不影响后续订单。真实项目中，这种日志可以帮助你再次补偿失败记录。

## 五、注意点

文件日志适合脚本和传统部署。容器化部署中，常见做法是把日志写到标准输出，由平台负责采集。这样应用不用关心日志文件路径、轮转和清理。

如果必须写文件，要确认：

- 日志目录存在。
- 程序有写权限。
- 日志不会无限增长。
- 失败记录可以被再次处理。

文件关闭通常用 `defer file.Close()`。长时间运行的服务不会频繁打开关闭日志文件，而是在启动时打开一次，退出时关闭。

## 六、常见误区

误区一：直接覆盖日志文件。  
如果不用 `os.O_APPEND`，可能导致历史日志被覆盖，排查时丢失证据。

误区二：忽略打开文件的错误。  
日志文件打不开时，程序继续跑会让你误以为有日志，实际什么都没留下。

误区三：生产服务无限写单个文件。  
文件会越来越大，最终可能写满磁盘，影响数据库、缓存或整个主机。

## 七、本节达标标准

- 能用 `os.OpenFile` 创建并追加写入日志文件。
- 能解释 `O_CREATE`、`O_WRONLY`、`O_APPEND` 的作用。
- 知道文件日志必须考虑权限、大小和轮转。
- 能为批处理脚本记录成功和失败记录。
