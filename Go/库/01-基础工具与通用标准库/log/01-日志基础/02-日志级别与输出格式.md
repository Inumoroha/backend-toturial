# 02. 日志级别与输出格式

本节目标：学完后，你能为不同严重程度的事件设计日志级别，并配置基础输出格式。

## 简短引入

真实后端项目一天可能产生大量日志。如果所有日志都长得一样，排查时就很难判断哪些是普通流程，哪些是需要关注的问题。日志级别就是为了帮助我们区分信息的重要程度。

## 一、为什么需要它

可以把日志级别理解为“这条日志需要谁关注”。常见习惯是：

- Debug：开发调试时看，生产通常关闭。
- Info：正常业务流程，比如订单创建成功。
- Warn：业务可继续，但存在风险，比如库存不足、调用第三方超时后重试成功。
- Error：当前操作失败，需要排查，比如数据库写入失败。

标准库 `log` 本身没有内置级别，但我们可以通过前缀建立基础习惯。

```text
日志级别不是装饰，它决定了生产环境里谁会被打扰、哪些问题会被优先处理。
```

## 二、基本用法

示例代码：

```go
package main

import (
	"log"
	"os"
)

func main() {
	logger := log.New(os.Stdout, "INFO ", log.Ldate|log.Ltime|log.Lshortfile)

	logger.Println("order created", "order_id", 9527)
}
```

运行：

```bash
go run .
```

输出类似：

```text
INFO 2026/07/06 10:30:00 main.go:11: order created order_id 9527
```

这里比上一节多了 `INFO` 前缀和文件行号。

## 三、关键参数/语法/代码结构

`log.New` 用来创建一个新的 logger：

```go
logger := log.New(os.Stdout, "INFO ", log.Ldate|log.Ltime|log.Lshortfile)
```

三个参数分别是：

- 输出位置：这里是 `os.Stdout`。
- 前缀：这里是 `INFO `。
- 标志位：控制时间、日期、文件名、行号等。

常用标志位：

- `log.Ldate`：日期。
- `log.Ltime`：时间。
- `log.Lmicroseconds`：微秒，排查请求耗时时有用。
- `log.Lshortfile`：短文件名和行号。
- `log.LstdFlags`：等于日期加时间。

真实项目中，文件行号适合开发和排查，但生产 JSON 日志里通常会更关注 `trace_id`、`request_id`、`user_id` 等字段。

## 四、真实后端场景示例

下面模拟一个订单支付流程，用多个 logger 表示不同级别。

```go
package main

import (
	"errors"
	"log"
	"os"
)

var (
	infoLogger  = log.New(os.Stdout, "INFO ", log.LstdFlags|log.Lshortfile)
	warnLogger  = log.New(os.Stdout, "WARN ", log.LstdFlags|log.Lshortfile)
	errorLogger = log.New(os.Stderr, "ERROR ", log.LstdFlags|log.Lshortfile)
)

func payOrder(orderID int64, amount int64) error {
	infoLogger.Println("pay order started", "order_id", orderID, "amount", amount)

	if amount <= 0 {
		warnLogger.Println("invalid pay amount", "order_id", orderID, "amount", amount)
		return errors.New("invalid amount")
	}

	infoLogger.Println("pay order success", "order_id", orderID)
	return nil
}

func main() {
	if err := payOrder(9527, 0); err != nil {
		errorLogger.Println("pay order failed", "error", err)
	}
}
```

这个例子里，参数不合法是 Warn，因为它常常来自用户输入；最终支付失败被调用方记录为 Error，因为这次业务操作没有完成。

## 五、注意点

不要为了“看起来专业”给所有日志都加 Error。Error 过多会让告警失去意义。

标准库 `log` 用前缀模拟级别可以满足学习和小工具需求。但如果你需要按级别过滤、输出 JSON、动态调整级别，应该使用 `log/slog` 或成熟日志库。

`os.Stdout` 和 `os.Stderr` 的选择也有实际意义。容器平台和进程管理工具通常会区分标准输出和标准错误，Error 日志写到 `os.Stderr` 更容易被采集和识别。

## 六、常见误区

误区一：业务失败都记 Error。  
比如用户输入手机号格式错误，这是可预期情况，通常用 Warn 或 Info 即可。

误区二：Debug 日志长期留在生产环境。  
Debug 可能输出大量细节，增加成本，也可能暴露敏感信息。

误区三：日志级别没有团队约定。  
每个人按自己的理解写，排查和告警都会变混乱。

## 七、本节达标标准

- 能用 `log.New` 创建带前缀的 logger。
- 能解释 Info、Warn、Error 的基本使用场景。
- 能配置日志输出日期、时间、文件名和行号。
- 知道标准库 `log` 的级别能力有限，后续需要结构化日志。
