# 06. 业务日志与错误日志边界

本节目标：学完后，你能判断一段业务代码应该在哪里记录日志、记录什么级别、避免重复记录错误。

## 简短引入

初学者常见问题不是“没有日志”，而是“哪里都在打日志”。一旦出错，同一个错误在 controller、service、repository 层重复打印三遍，排查时反而看不清主线。日志边界就是为了让日志既够用，又不制造噪音。

## 一、为什么需要它

真实项目中，一次请求会经过多层：

- handler：处理 HTTP 参数和响应。
- service：处理业务规则。
- repository：访问数据库或缓存。

每一层都可以遇到错误，但不代表每一层都要打印 Error。常见做法是：

- 底层返回带上下文的错误。
- 边界层统一记录一次错误。
- 关键业务状态变化单独记录 Info。

```text
同一个错误通常只在一个清晰的边界记录一次，其他层负责补充上下文并返回。
```

## 二、基本用法

下面演示“返回错误，不在底层重复打印”。

```go
package main

import (
	"errors"
	"fmt"
	"log/slog"
	"os"
)

func findUser(userID int64) error {
	return errors.New("database timeout")
}

func createOrder(userID int64) error {
	if err := findUser(userID); err != nil {
		return fmt.Errorf("find user %d: %w", userID, err)
	}
	return nil
}

func main() {
	logger := slog.New(slog.NewJSONHandler(os.Stdout, nil))

	if err := createOrder(1001); err != nil {
		logger.Error("create order failed",
			"user_id", 1001,
			"error", err,
		)
	}
}
```

输出里只有一条 Error，但错误信息保留了 `find user 1001` 这个上下文。

## 三、关键参数/语法/代码结构

`fmt.Errorf("find user %d: %w", userID, err)` 的作用是给底层错误补充上下文，同时保留原始错误。

可以理解为：

- repository 层知道“数据库超时”。
- service 层知道“查用户失败”。
- handler 或任务入口知道“创建订单失败”。

最终日志应该能把这条链路讲清楚。

业务成功日志一般记录关键状态变化，比如：

- 用户注册成功。
- 订单创建成功。
- 库存扣减成功。
- 权限变更成功。

普通中间过程不要过度记录，否则日志会很吵。

## 四、真实后端场景示例

下面用“库存扣减”说明日志边界。

```go
package main

import (
	"errors"
	"fmt"
	"log/slog"
	"os"
)

func decreaseStock(skuID int64, quantity int) error {
	if quantity > 10 {
		return errors.New("stock not enough")
	}
	return nil
}

func createOrder(userID, skuID int64, quantity int) error {
	if err := decreaseStock(skuID, quantity); err != nil {
		return fmt.Errorf("decrease stock sku_id=%d quantity=%d: %w", skuID, quantity, err)
	}
	return nil
}

func main() {
	logger := slog.New(slog.NewJSONHandler(os.Stdout, nil))

	userID := int64(1001)
	skuID := int64(2001)
	quantity := 20

	if err := createOrder(userID, skuID, quantity); err != nil {
		logger.Warn("create order rejected",
			"user_id", userID,
			"sku_id", skuID,
			"quantity", quantity,
			"error", err,
		)
		return
	}

	logger.Info("create order success", "user_id", userID, "sku_id", skuID)
}
```

库存不足是业务可预期情况，通常用 Warn，而不是 Error。数据库不可用、消息队列发送失败这类基础设施问题更适合 Error。

## 五、注意点

错误日志要记录在有完整上下文的位置。比如 handler 层知道 request_id、user_id、接口路径；service 层知道业务动作；repository 层知道 SQL 或数据源。实际项目中要选择最能说明问题的位置。

如果一个错误已经在上层记录，底层不要再记录同样的 Error。底层可以补充错误信息并返回。

业务日志要服务于审计和排查。比如权限变更、订单退款、库存人工调整，通常需要记录操作者和目标资源。

## 六、常见误区

误区一：每一层都 `logger.Error`。  
这样会导致一处错误出现多条相似日志，告警数量也会被放大。

误区二：把可预期业务失败都当成系统错误。  
比如库存不足、验证码错误、权限不足，很多时候是 Warn 或 Info。

误区三：只返回原始错误，不补充上下文。  
只有 `database timeout` 不够，排查时还需要知道查什么、为谁查、在哪个业务动作中失败。

## 七、本节达标标准

- 能区分业务日志和错误日志。
- 能用 `%w` 包装错误并保留上下文。
- 知道同一个错误通常只记录一次。
- 能判断库存不足、参数错误、数据库失败分别适合什么日志级别。
