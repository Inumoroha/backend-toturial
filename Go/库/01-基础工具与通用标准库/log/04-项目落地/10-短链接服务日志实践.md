# 10. 短链接服务日志实践

本节目标：学完后，你能在一个短链接 HTTP 服务中综合使用结构化日志、请求日志和业务日志。

## 简短引入

短链接服务很适合练习日志：它有创建链接、访问跳转、链接不存在、参数错误、访问统计等场景。我们不追求完整数据库实现，而是先把日志设计落到一个能运行的小服务里。

## 一、为什么需要它

短链接服务上线后，常见问题包括：

- 用户创建短链接失败。
- 短码不存在或过期。
- 某个短链访问量异常。
- 跳转接口变慢。

日志要帮助我们回答：哪个短码、哪个用户、哪个请求、结果如何、耗时多久。

```text
项目日志要围绕业务关键路径设计，而不是等出问题后临时补打印。
```

## 二、基本用法

下面代码可直接运行，使用内存 map 保存短链接。

```go
package main

import (
	"context"
	"encoding/json"
	"log/slog"
	"net/http"
	"os"
	"strconv"
	"strings"
	"time"
)

type requestIDKey struct{}

type statusRecorder struct {
	http.ResponseWriter
	status int
}

func (r *statusRecorder) WriteHeader(code int) {
	r.status = code
	r.ResponseWriter.WriteHeader(code)
}

func accessLog(logger *slog.Logger, next http.Handler) http.Handler {
	return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		start := time.Now()
		requestID := r.Header.Get("X-Request-Id")
		if requestID == "" {
			requestID = strconv.FormatInt(time.Now().UnixNano(), 10)
		}
		ctx := context.WithValue(r.Context(), requestIDKey{}, requestID)
		rec := &statusRecorder{ResponseWriter: w, status: http.StatusOK}

		next.ServeHTTP(rec, r.WithContext(ctx))

		logger.Info("http request completed",
			"request_id", requestID,
			"method", r.Method,
			"path", r.URL.Path,
			"status", rec.status,
			"duration_ms", time.Since(start).Milliseconds(),
		)
	})
}

func requestIDFromContext(ctx context.Context) string {
	requestID, _ := ctx.Value(requestIDKey{}).(string)
	return requestID
}

func main() {
	logger := slog.New(slog.NewJSONHandler(os.Stdout, nil))
	links := map[string]string{}

	mux := http.NewServeMux()

	mux.HandleFunc("/links", func(w http.ResponseWriter, r *http.Request) {
		if r.Method != http.MethodPost {
			http.Error(w, "method not allowed", http.StatusMethodNotAllowed)
			return
		}

		var req struct {
			Code string `json:"code"`
			URL  string `json:"url"`
		}
		if err := json.NewDecoder(r.Body).Decode(&req); err != nil {
			logger.Warn("create short link invalid json",
				"request_id", requestIDFromContext(r.Context()),
				"error", err,
			)
			http.Error(w, "invalid json", http.StatusBadRequest)
			return
		}

		if req.Code == "" || req.URL == "" {
			logger.Warn("create short link invalid params",
				"request_id", requestIDFromContext(r.Context()),
				"code", req.Code,
			)
			http.Error(w, "code and url required", http.StatusBadRequest)
			return
		}

		links[req.Code] = req.URL
		logger.Info("short link created",
			"request_id", requestIDFromContext(r.Context()),
			"code", req.Code,
		)

		w.WriteHeader(http.StatusCreated)
		_, _ = w.Write([]byte("created"))
	})

	mux.HandleFunc("/r/", func(w http.ResponseWriter, r *http.Request) {
		if r.Method != http.MethodGet {
			http.Error(w, "method not allowed", http.StatusMethodNotAllowed)
			return
		}

		code := strings.TrimPrefix(r.URL.Path, "/r/")
		target, ok := links[code]
		if !ok {
			logger.Warn("short link not found",
				"request_id", requestIDFromContext(r.Context()),
				"code", code,
			)
			http.NotFound(w, r)
			return
		}

		logger.Info("short link redirected",
			"request_id", requestIDFromContext(r.Context()),
			"code", code,
		)
		http.Redirect(w, r, target, http.StatusFound)
	})

	logger.Info("server started", "addr", ":8080")
	if err := http.ListenAndServe(":8080", accessLog(logger, mux)); err != nil {
		logger.Error("server stopped", "error", err)
	}
}
```

运行：

```bash
go run .
```

创建短链接。

Windows PowerShell：

```powershell

```

Linux/macOS：

```bash
curl -X POST http://localhost:8080/links -H "Content-Type: application/json" -d '{"code":"go","url":"https://go.dev"}'
```

```bash

```

## 三、关键参数/语法/代码结构

这段代码里有两类日志：

- 请求日志：由中间件统一记录 method、path、status、duration_ms。
- 业务日志：在创建短链、短链不存在、跳转成功时记录 code。

请求日志回答“接口层发生了什么”，业务日志回答“业务动作发生了什么”。两者结合，排查才完整。

## 四、真实后端场景示例

如果未来接入数据库，创建短链的日志可以继续保留：

```go
logger.Info("short link created",
	"user_id", userID,
	"code", code,
	"domain", domain,
)
```

如果数据库写入失败，应记录 Error：

```go
logger.Error("save short link failed",
	"user_id", userID,
	"code", code,
	"error", err,
)
```

本节示例已经把 `request_id` 放入了 `context.Context`，并在创建短链、参数错误、短码不存在、跳转成功这些业务日志里带上它。真实项目中要继续把 `ctx` 传给 service、repository 和外部依赖调用。

如果未来加数据库，创建短链接的真实链路通常会是：

```text
HTTP 请求 -> 参数校验 -> 开启事务 -> 写入 short_links -> 写入审计日志 -> 提交事务 -> 返回结果
```

日志只记录关键节点，不替代事务。比如写入数据库失败时，应该回滚事务，并记录带 `request_id`、`code`、`error` 的 Error 日志。

## 五、注意点

不要把完整长链接里的敏感查询参数直接记录下来。很多推广链接、支付链接、授权回调链接都可能带 Token。

内存 map 只是教程示例，生产中不能用它保存核心数据。真实服务需要数据库、唯一索引、迁移脚本和回滚方案。

短码不存在通常是 Warn 或 Info，不一定是 Error。除非不存在量突然暴涨，才可能代表爬虫、攻击或配置问题。

## 六、常见误区

误区一：只记录接口状态码，不记录业务 code。  
看到 404 还不够，你需要知道哪个短码不存在。

误区二：把完整 URL 打进日志。  
URL 查询参数可能包含敏感信息，应谨慎处理。

误区三：示例代码直接当生产代码。  
本节重点是日志设计，不是完整短链接系统。生产还需要数据库、限流、鉴权、监控和迁移。

## 七、本节达标标准

- 能运行一个带日志的短链接服务。
- 能区分请求日志和业务日志。
- 能为创建、访问、不存在等路径设计日志字段。
- 知道示例内存存储不能直接用于生产。
