# 11. 日志排查与告警思路

本节目标：学完后，你能根据日志建立线上问题的基本排查路径，并知道哪些日志适合告警。

## 简短引入

日志不是写完就结束。真正的价值在问题发生时体现：接口变慢、错误率升高、用户反馈失败、定时任务异常。排查时要按线索推进，而不是在日志里盲目搜索。

## 一、为什么需要它

真实项目中常见故障包括：

- 某个接口 500 增多。
- 某个用户一直下单失败。
- 数据迁移脚本中途停止。
- 短链接跳转突然变慢。

日志排查通常从一个入口开始：时间范围、接口路径、request_id、user_id、order_id、错误关键字。

```text
排查日志先缩小范围，再寻找共同点。
```

## 二、基本用法

假设你有一批 JSON 日志，排查时常用字段是：

```text
time
level
msg
request_id
method
path
status
duration_ms
user_id
order_id
error
```

本地文件可以先用简单命令过滤。

Windows PowerShell：

```powershell
Select-String -Path .\app.log -Pattern '"level":"ERROR"'
```

Linux/macOS：

```bash
grep '"level":"ERROR"' ./app.log
```

这些命令适合本地小文件。生产环境应优先使用日志平台，不要在服务器上长时间 grep 大文件，避免影响机器负载。

如果你已经知道 `request_id`，排查会更直接。

Windows PowerShell：

```powershell
Select-String -Path .\app.log -Pattern '"request_id":"abc"'
```

Linux/macOS：

```bash
grep '"request_id":"abc"' ./app.log
```

如果你只知道接口路径，可以先缩小范围：

```bash
grep '"/r/' ./app.log | grep '"status":500'
```

这类命令不是为了替代日志平台，而是帮助你理解排查思路：先缩小范围，再看同一批日志有什么共同点。

## 三、关键参数/语法/代码结构

排查慢请求时关注：

- `path`：是否集中在某个接口。
- `duration_ms`：是否突然变大。
- `status`：慢请求是否同时失败。
- `request_id`：能否串联业务日志。

排查业务失败时关注：

- `user_id` 或 `order_id`。
- 失败发生的时间。
- 错误是参数问题、权限问题还是依赖问题。
- 同一错误是否集中出现。

告警应该基于可行动的问题，而不是所有 Error 都报警。

可以按下面顺序问自己：

1. 影响范围有多大，是单个用户还是所有用户。
2. 是否集中在某个接口、某个实例、某个依赖。
3. 失败前有没有发布、配置变更、迁移任务或流量变化。
4. 是否有可回滚动作，比如回滚版本、关闭开关、暂停任务。

```text
日志排查的目标不是证明代码哪里错了，而是尽快缩小影响范围并找到可验证的下一步。
```

## 四、真实后端场景示例

短链接跳转变慢，可以这样查：

1. 先筛选 `path=/r/` 的请求日志。
2. 找出 `duration_ms` 较高的请求。
3. 根据 `request_id` 查同一次请求的业务日志。
4. 看是否集中在某些 code、某个数据库查询或外部依赖。

如果你看到类似日志：

```json
{"level":"INFO","msg":"http request completed","path":"/r/go","status":302,"duration_ms":1200,"request_id":"abc"}
```

下一步不是立刻改代码，而是继续查同一个 `request_id` 下是否有数据库、缓存或第三方调用日志。

如果多条慢请求都集中在 `/r/go`，可能是这个短码访问量大、缓存失效、数据库索引缺失或下游跳转规则异常。日志只能提示方向，下一步通常要结合指标看数据库耗时、缓存命中率和实例负载。

如果慢请求分散在所有短码上，更像是公共依赖问题，比如数据库连接池耗尽、缓存不可用、网络抖动或新版本引入了全局慢逻辑。

## 五、注意点

本地命令只能作为临时手段。生产日志量大时，应该通过集中式日志平台检索，配合指标和链路追踪。

告警要有阈值和持续时间。例如“5 分钟内 500 错误率超过 2%”比“出现一条 Error 就报警”更合理。

日志不是监控的替代品。接口 QPS、错误率、耗时分位数更适合用指标系统统计；日志适合提供上下文。

常见可落地告警：

- 5 分钟内某接口 500 错误率超过阈值。
- 10 分钟内登录失败量突然升高。
- 定时任务连续失败。
- 数据库连接错误持续出现。
- 磁盘使用率过高导致日志写入失败。

不建议直接告警：

- 单条用户参数错误。
- 单次库存不足。
- 少量短码不存在。
- 开发调试级别日志。

这些事件可以记录，但不一定值得把人从工作流里打断。

## 六、常见误区

误区一：看到 Error 就认定是根因。  
Error 可能只是结果，根因可能在更早的 Warn、超时或依赖异常。

误区二：告警太多。  
告警如果不能行动，很快就会被忽略。

误区三：只查失败请求。  
有时对比成功请求和失败请求，才能看出差异。

## 七、本节达标标准

- 能按时间、接口、request_id、业务 ID 缩小日志范围。
- 能区分日志排查和指标监控的作用。
- 知道告警应该基于可行动问题和合理阈值。
- 能为短链接跳转慢设计基本排查步骤。
