# 12. 生产日志检查清单

本节目标：学完后，你能在 Go 后端服务上线前检查日志是否安全、可查、可控。

## 简短引入

日志方案不是写几行代码就完成了。上线前要确认日志能被采集、字段稳定、不泄露敏感信息、不会打爆磁盘或日志平台。检查清单能帮助你减少低级事故。

## 一、为什么需要它

生产日志要同时满足几件事：

- 出问题时能查。
- 正常运行时成本可控。
- 不泄露敏感数据。
- 不影响业务性能。
- 能配合监控和告警。

```text
上线前检查日志，比上线后补救日志事故便宜得多。
```

## 二、基本用法

可以把下面清单放到项目文档或发布流程里。

```text
日志上线检查清单：
1. 是否使用结构化日志。
2. 是否包含 request_id。
3. 是否记录 method、path、status、duration_ms。
4. Error 日志是否包含 error 字段。
5. 是否避免记录密码、Token、验证码、完整手机号。
6. 生产环境是否默认 Info 级别。
7. 日志是否输出到标准输出或受控文件。
8. 文件日志是否配置轮转和保留时间。
9. 关键业务动作是否有 Info 日志。
10. 可预期业务失败是否避免滥用 Error。
```

建议上线前至少人工抽样三类日志：

- 一条成功请求日志。
- 一条业务失败日志，比如参数错误、权限不足、库存不足。
- 一条系统错误日志，比如数据库连接失败、消息发送失败。

抽样时不要只看“有没有输出”，还要看字段是否足够排查、是否泄露敏感信息、是否能通过 request_id 串起来。

## 三、关键参数/语法/代码结构

一个较稳妥的生产 logger 初始化可以这样写：

```go
package main

import (
	"log/slog"
	"os"
)

func newProdLogger() *slog.Logger {
	return slog.New(slog.NewJSONHandler(os.Stdout, &slog.HandlerOptions{
		Level: slog.LevelInfo,
	}))
}

func main() {
	logger := newProdLogger()
	logger.Info("application started", "service", "short-link-api")
}
```

这不是完整框架，但方向是保守的：JSON、标准输出、Info 级别、稳定字段。

建议团队提前约定字段命名：

```text
request_id：一次请求的追踪 ID
user_id：当前操作用户
order_id：订单 ID
code：短链接短码
path：HTTP 请求路径
status：HTTP 响应状态码
duration_ms：请求耗时，单位毫秒
error：错误信息
```

字段名一旦被日志平台、告警规则、排查手册使用，就不要随便改。确实要改时，要给查询和告警留过渡期。

## 四、真实后端场景示例

以上线“订单服务”为例，建议检查这些日志点：

- 创建订单成功：记录 `user_id`、`order_id`、`amount`。
- 库存不足：记录 `user_id`、`sku_id`、`quantity`，级别通常 Warn。
- 数据库写入失败：记录 `user_id`、`order_id`、`error`，级别 Error。
- 请求完成：记录 `request_id`、`method`、`path`、`status`、`duration_ms`。
- 支付回调：记录 `order_id`、`payment_id`、结果，但不要记录完整签名密钥。

数据库、事务、迁移相关日志要特别克制。比如迁移脚本可以记录批次、数量、失败 ID，但不要把整行用户数据打印出来。

再看一个迁移任务的日志设计：

```text
迁移开始：记录 migration_name、batch_size、start_id
每批成功：记录 migration_name、batch_no、success_count、last_id
每批失败：记录 migration_name、batch_no、failed_id、error
迁移结束：记录 migration_name、total_success、total_failed、duration_ms
```

这样设计的好处是，失败后可以知道从哪里继续，也能判断是否需要回滚或重跑。不要把整批 SQL 参数、用户手机号、地址等原始数据都写进日志。

## 五、注意点

日志字段要和团队约定保持一致。比如统一用 `request_id`，不要在不同服务里混用 `traceId`、`rid`、`requestID`。

日志和错误处理要配合。记录 Error 后，仍然要返回合适 HTTP 状态码，必要时回滚事务，不能只打日志不处理。

涉及数据库写操作时，日志不能替代事务边界。比如订单创建失败，要依靠事务回滚保证数据一致，日志只是帮助排查。

上线前还要确认日志失败不会拖垮业务。比如日志平台暂时不可用时，应用不能因为同步写远程日志而阻塞核心接口。保守做法是写标准输出，由采集组件异步处理。

## 六、常见误区

误区一：上线前才临时补日志。  
临时补的日志经常字段不统一，也容易遗漏敏感信息检查。

误区二：日志里记录完整 SQL 和参数。  
排查时方便，但可能泄露用户数据。生产中要谨慎，优先记录 SQL 名称、操作类型、耗时和错误。

误区三：认为有日志就能解决一致性问题。  
事务、幂等、回滚、迁移工具才是数据安全的核心，日志只能帮助观察和排查。

## 七、本节达标标准

- 能使用清单检查一个服务的日志方案。
- 能说明生产日志为什么推荐 JSON 和标准输出。
- 能识别敏感字段、日志爆量和文件增长风险。
- 能把日志与错误处理、事务、回滚、监控告警区分开。
