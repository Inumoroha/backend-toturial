# 05. 多字段排序和排序参数白名单

本节目标：学完后，你能够把用户传入的排序参数转换成受控的排序规则，而不是直接相信用户输入。

简短引入：列表接口常见参数是 `sort=created_at&order=desc`，真正危险的不是排序本身，而是把这些参数不加限制地用于 SQL 或业务逻辑。

## 一、为什么需要它

后台订单列表可能允许按创建时间、金额、状态排序。用户请求里传的是字符串，但后端需要把字符串映射成明确字段。

常见场景：

- `sort=created_at`：按创建时间。
- `sort=amount`：按金额。
- `order=asc` 或 `order=desc`：升序或降序。

```text
用户输入不能直接拼进 SQL 字符串，也不能直接变成不受控的排序表达式。
```

## 二、基本用法

创建 `main.go`：

```go
package main

import (
	"fmt"
	"sort"
)

type Order struct {
	ID     int
	Amount int
	Status string
}

func main() {
	orders := []Order{
		{ID: 1, Amount: 300, Status: "paid"},
		{ID: 2, Amount: 100, Status: "pending"},
		{ID: 3, Amount: 300, Status: "pending"},
	}

	sortOrders(orders, "amount", "desc")
	fmt.Println(orders)
}

func sortOrders(orders []Order, field, direction string) {
	desc := direction == "desc"

	sort.Slice(orders, func(i, j int) bool {
		var less bool
		switch field {
		case "amount":
			less = orders[i].Amount < orders[j].Amount
		case "status":
			less = orders[i].Status < orders[j].Status
		default:
			less = orders[i].ID < orders[j].ID
		}

		if desc {
			return !less && valueNotEqual(orders[i], orders[j], field)
		}
		return less
	})
}

func valueNotEqual(a, b Order, field string) bool {
	switch field {
	case "amount":
		return a.Amount != b.Amount
	case "status":
		return a.Status != b.Status
	default:
		return a.ID != b.ID
	}
}
```

这个示例先展示思路。真实项目中建议把参数校验和排序执行拆开，让代码更清楚。

## 三、关键参数/语法/代码结构

建议把排序分成三步：

1. 校验 `sort` 是否在白名单中。
2. 校验 `order` 是否只能是 `asc` 或 `desc`。
3. 根据已校验参数执行排序。

更清晰的白名单写法：

```go
var allowedSortFields = map[string]bool{
	"created_at": true,
	"amount":     true,
	"status":     true,
}

func validateSort(field string) error {
	if !allowedSortFields[field] {
		return fmt.Errorf("unsupported sort field: %s", field)
	}
	return nil
}
```

## 四、真实后端场景示例

如果排序要下推到数据库，不要这样写：

```go
query := "SELECT * FROM orders ORDER BY " + sortField + " " + direction
```

字段名不能像普通值一样参数化，所以要用白名单映射：

```go
var orderByColumns = map[string]string{
	"created_at": "created_at",
	"amount":     "amount",
	"status":     "status",
}

column, ok := orderByColumns[sortField]
if !ok {
	return fmt.Errorf("unsupported sort field")
}
if direction != "asc" && direction != "desc" {
	return fmt.Errorf("unsupported sort direction")
}

query := fmt.Sprintf(
	"SELECT id, amount, status FROM orders ORDER BY %s %s, id DESC LIMIT $1 OFFSET $2",
	column,
	direction,
)
```

这里 `LIMIT` 和 `OFFSET` 仍然应该用参数化查询传值。字段名和方向只能来自你自己的白名单。

## 五、注意点

在真实项目中，排序字段白名单最好靠近接口层或 repository 层维护。不要散落在多个 handler 里，否则新增字段时容易漏改。

数据库排序还要考虑索引。允许用户按任意字段排序，会让数据库很难稳定使用索引，线上可能出现慢查询。

## 六、常见误区

- 误区一：把 `sort` 参数直接拼到 SQL。字段名注入同样危险。
- 误区二：只校验字段，不校验方向。`direction` 也只能允许 `asc` 或 `desc`。
- 误区三：为了灵活允许所有字段排序。真实项目中排序字段越多，索引和性能成本越高。

## 七、本节达标标准

- 能设计排序字段白名单。
- 能校验排序方向。
- 能说明字段名为什么不能直接从用户输入拼 SQL。
- 能把内存排序和数据库排序都纳入同一套受控参数设计。
