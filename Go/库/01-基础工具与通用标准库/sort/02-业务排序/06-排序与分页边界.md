# 06. 排序与分页边界

本节目标：学完后，你能够判断排序应该放在数据库还是 Go 内存里，并理解分页为什么必须依赖稳定排序。

简短引入：很多后端列表接口同时有排序和分页。排序位置选错，会带来性能问题；排序规则不稳定，会带来重复和漏数据。

## 一、为什么需要它

假设订单表有 100 万行，接口要返回第 2 页，每页 20 条。如果先从数据库取出很多数据，再在 Go 里排序和分页，内存和网络成本都很高。

常见判断：

- 数据在数据库里，且需要分页：优先数据库排序分页。
- 数据已经在内存里，数量较小：可以 Go 排序。
- 数据来自多个服务聚合：可能需要 Go 做最后合并排序，但要限制数量。

## 二、基本用法

内存排序后分页的最小示例：

```go
package main

import (
	"fmt"
	"sort"
)

type Order struct {
	ID     int
	Amount int
}

func main() {
	orders := []Order{
		{ID: 1, Amount: 300},
		{ID: 2, Amount: 100},
		{ID: 3, Amount: 500},
		{ID: 4, Amount: 200},
	}

	sort.Slice(orders, func(i, j int) bool {
		return orders[i].Amount > orders[j].Amount
	})

	page := paginateOrders(orders, 1, 2)
	fmt.Println(page)
}

func paginateOrders(orders []Order, page, pageSize int) []Order {
	start := (page - 1) * pageSize
	if start >= len(orders) {
		return []Order{}
	}
	end := start + pageSize
	if end > len(orders) {
		end = len(orders)
	}
	return orders[start:end]
}
```

## 三、关键参数/语法/代码结构

分页参数至少要校验：

- `page` 必须大于等于 1。
- `page_size` 要有上限，例如 100。
- 排序必须在分页前完成。

数据库分页常见写法：

```sql
SELECT id, amount, created_at
FROM orders
ORDER BY created_at DESC, id DESC
LIMIT $1 OFFSET $2;
```

`id DESC` 是兜底排序，避免 `created_at` 相同的时候顺序不确定。

## 四、真实后端场景示例

真实接口中，用户请求：

```text
GET /orders?sort=created_at&order=desc&page=2&page_size=20
```

后端通常会做：

1. 校验 `sort`、`order`、`page`、`page_size`。
2. 把允许的排序字段映射成数据库列。
3. 使用参数化查询传入 `LIMIT` 和 `OFFSET`。
4. 保留唯一字段作为兜底排序。

```text
分页不是先切一页再排序，而是先确定全局顺序，再取其中一页。
```

## 五、注意点

`OFFSET` 分页在数据量大时可能变慢，尤其是翻到很后面的页。生产项目中可以考虑基于游标的分页，例如使用 `(created_at, id)` 作为游标。

但初学阶段先掌握一个原则：排序规则要稳定，分页参数要有上限，数据库查询要有索引支撑。

涉及数据库迁移时，不要直接在高峰期给大表加索引。真实项目中通常会使用迁移工具、灰度执行、观察慢查询，并准备可回滚方案。

## 六、常见误区

- 误区一：先分页再排序。这样每页内部有序，但全局无序。
- 误区二：只按 `created_at` 排序。时间相同的数据可能导致分页跳动。
- 误区三：不限制 `page_size`。用户可以一次请求大量数据，拖慢服务。
- 误区四：看到慢查询就随便加索引。索引会增加写入成本和存储成本，要结合查询模式评估。
