# 01. 排序在后端列表中的位置

本节目标：学完后，你能够判断一个后端列表需求里排序应该解决什么问题，以及什么时候不应该急着在 Go 内存里排序。

简短引入：后端接口经常返回列表，列表如果没有稳定顺序，用户会觉得数据跳动，分页也可能出现重复或漏数据。

## 一、为什么需要它

可以理解为，排序是把一组业务数据变成“用户能理解的顺序”。后台订单通常按创建时间倒序，文章通常按发布时间倒序，权限菜单通常按权重升序。

真实项目中，排序不只是展示问题。它会影响：

- 用户看到的数据是否稳定。
- 分页结果是否重复或遗漏。
- 数据库是否能使用索引。
- 缓存 key 是否需要包含排序条件。

```text
列表接口只要支持分页，就必须有明确且稳定的排序规则。
```

## 二、基本用法

先用最小代码建立直觉。创建 `main.go`：

```go
package main

import (
	"fmt"
	"sort"
)

func main() {
	scores := []int{80, 95, 70, 95}
	sort.Ints(scores)
	fmt.Println(scores)
}
```

Windows PowerShell：

```powershell
go run .\main.go
```

Linux/macOS：

```bash
go run ./main.go
```

输出：

```text
[70 80 95 95]
```

## 三、关键参数/语法/代码结构

`sort.Ints(scores)` 做了两件事：

- 按从小到大排序。
- 直接修改 `scores` 这个切片背后的数组。

这里没有返回新切片。这个点非常重要，因为真实项目中一个切片可能还会被别的逻辑继续使用。

如果你需要保留原顺序，可以先复制：

```go
sorted := append([]int(nil), scores...)
sort.Ints(sorted)
```

## 四、真实后端场景示例

假设你从缓存里拿到用户积分，想给运营后台展示积分从低到高的异常用户：

```go
package main

import (
	"fmt"
	"sort"
)

func main() {
	cacheScores := []int{120, 30, 450, 80}

	sortedScores := append([]int(nil), cacheScores...)
	sort.Ints(sortedScores)

	fmt.Println("原缓存顺序:", cacheScores)
	fmt.Println("展示顺序:", sortedScores)
}
```

真实项目中通常不会直接改缓存返回的共享数据。先复制再排序，可以减少“后面某段逻辑突然发现顺序变了”的问题。

## 五、注意点

小数据量、临时计算、缓存结果二次整理，适合在 Go 内存里排序。

大数据量、数据库表查询、需要分页的接口，通常优先让数据库排序：

```sql
SELECT id, amount, created_at
FROM orders
ORDER BY created_at DESC, id DESC
LIMIT 20 OFFSET 0;
```

数据库排序也不是免费。真实项目中要关注索引成本、慢查询、排序字段选择，以及线上变更是否可回滚。

## 六、常见误区

- 误区一：以为 `sort.Ints` 会返回新切片。它不会，原切片会被修改。
- 误区二：所有排序都放 Go 里做。数据已经在数据库里且要分页时，通常应优先考虑数据库 `ORDER BY`。
- 误区三：分页不写排序。数据库不保证无排序查询的返回顺序，分页结果可能不稳定。

## 七、本节达标标准

- 能说出排序在列表接口中的作用。
- 能运行 `sort.Ints` 示例。
- 能解释原地排序会修改原切片。
- 能初步判断 Go 内存排序和数据库排序的边界。
