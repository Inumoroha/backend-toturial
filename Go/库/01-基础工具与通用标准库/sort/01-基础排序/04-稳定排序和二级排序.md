# 04. 稳定排序和二级排序

本节目标：学完后，你能够在主排序字段相同的时候保持结果可预期，并判断是否需要 `sort.SliceStable`。

简短引入：很多线上问题不是因为完全没排序，而是因为相同字段的数据顺序不稳定，导致分页和用户观感出问题。

## 一、为什么需要它

比如订单列表按状态排序：待支付、已支付、已取消。状态相同的订单如果顺序乱跳，用户会以为数据变了。

解决方式通常有两种：

- 用稳定排序，保留相同字段的原始顺序。
- 写二级排序，让相同字段继续按创建时间或 ID 排序。

后端项目中，二级排序更常见，因为它让结果更确定。

## 二、基本用法

创建 `main.go`：

```go
package main

import (
	"fmt"
	"sort"
)

type User struct {
	ID    int
	Role  string
	Score int
}

func main() {
	users := []User{
		{ID: 1, Role: "admin", Score: 90},
		{ID: 2, Role: "viewer", Score: 80},
		{ID: 3, Role: "admin", Score: 70},
	}

	sort.SliceStable(users, func(i, j int) bool {
		return users[i].Role < users[j].Role
	})

	fmt.Println(users)
}
```

## 三、关键参数/语法/代码结构

`sort.SliceStable` 的比较函数和 `sort.Slice` 一样。差别是：当两个元素被认为相等时，稳定排序尽量保留它们原来的相对顺序。

二级排序写法：

```go
sort.Slice(users, func(i, j int) bool {
	if users[i].Role != users[j].Role {
		return users[i].Role < users[j].Role
	}
	return users[i].ID < users[j].ID
})
```

可以理解为先看 `Role`，如果一样，再看 `ID`。

## 四、真实后端场景示例

后台订单列表常见需求：先按状态排序，同状态内按创建时间倒序，创建时间相同再按 ID 倒序。

```go
package main

import (
	"fmt"
	"sort"
	"time"
)

type Order struct {
	ID        int
	Status    string
	CreatedAt time.Time
}

func main() {
	orders := []Order{
		{ID: 10, Status: "paid", CreatedAt: time.Date(2026, 1, 1, 10, 0, 0, 0, time.UTC)},
		{ID: 11, Status: "pending", CreatedAt: time.Date(2026, 1, 2, 10, 0, 0, 0, time.UTC)},
		{ID: 12, Status: "paid", CreatedAt: time.Date(2026, 1, 2, 10, 0, 0, 0, time.UTC)},
	}

	sort.Slice(orders, func(i, j int) bool {
		if orders[i].Status != orders[j].Status {
			return orders[i].Status < orders[j].Status
		}
		if !orders[i].CreatedAt.Equal(orders[j].CreatedAt) {
			return orders[i].CreatedAt.After(orders[j].CreatedAt)
		}
		return orders[i].ID > orders[j].ID
	})

	for _, order := range orders {
		fmt.Println(order.ID, order.Status, order.CreatedAt.Format("2006-01-02"))
	}
}
```

## 五、注意点

稳定排序不是万能的。如果原始顺序来自数据库但 SQL 没有 `ORDER BY`，这个“原始顺序”本身就不可靠。

```text
稳定排序只能稳定已有顺序，不能替你创造可靠的业务顺序。
```

真实项目中，分页接口建议始终包含唯一字段作为最后排序条件，例如 `id DESC`，这样同一时间创建的数据也有确定顺序。

## 六、常见误区

- 误区一：以为稳定排序能解决所有乱序问题。前提是输入顺序本身可靠。
- 误区二：只按非唯一字段排序后分页，例如只按 `created_at DESC`。同一秒内多条数据可能出现分页跳动。
- 误区三：二级排序字段和业务语义冲突。比如权限菜单同权重时按名称排序，可能破坏产品配置的展示意图。

## 七、本节达标标准

- 能说明普通排序和稳定排序的差别。
- 能使用 `sort.SliceStable`。
- 能写出至少两个字段的排序规则。
- 能解释为什么分页列表常需要唯一字段兜底排序。
