# 10. 统一错误响应与日志记录

本节目标：学完后，你能够把内部错误转换成稳定的 API 响应，同时保留足够的日志用于排查。

真实后端项目里，错误处理不能散落在每个接口里。否则同一个 `ErrUserNotFound`，有的接口返回 `404`，有的接口返回 `500`，有的接口直接把 `err.Error()` 暴露给用户。统一错误响应的目的，就是让接口行为稳定，让日志足够可查。

## 一、为什么需要它

内部错误和外部响应不是一回事。

内部日志需要看到：

```text
get user profile: query user by id 42: connection refused
```

外部用户只需要看到：

```json
{"code":"INTERNAL_ERROR","message":"internal server error"}
```

前者帮助工程师排查，后者保护系统细节，也给前端稳定的错误码。

```text
错误要分两份处理：一份给内部日志，一份给外部响应。
```

## 二、基本用法

复制下面代码到 `main.go`。

```go
package main

import (
    "encoding/json"
    "errors"
    "fmt"
)

var (
    ErrUserNotFound      = errors.New("user not found")
    ErrPermissionDenied  = errors.New("permission denied")
    ErrInvalidCredential = errors.New("invalid credential")
)

type ErrorResponse struct {
    Code    string `json:"code"`
    Message string `json:"message"`
}

func mapError(err error) (int, ErrorResponse) {
    switch {
    case errors.Is(err, ErrUserNotFound):
        return 404, ErrorResponse{Code: "USER_NOT_FOUND", Message: "user not found"}
    case errors.Is(err, ErrPermissionDenied):
        return 403, ErrorResponse{Code: "PERMISSION_DENIED", Message: "permission denied"}
    case errors.Is(err, ErrInvalidCredential):
        return 401, ErrorResponse{Code: "INVALID_CREDENTIAL", Message: "invalid credential"}
    default:
        return 500, ErrorResponse{Code: "INTERNAL_ERROR", Message: "internal server error"}
    }
}

func main() {
    err := fmt.Errorf("get user profile: %w", ErrUserNotFound)

    status, response := mapError(err)
    body, _ := json.Marshal(response)

    fmt.Println("status:", status)
    fmt.Println("response:", string(body))
    fmt.Println("log:", err)
}
```

运行命令：

Windows PowerShell：

```powershell
go run .\main.go
```

Linux/macOS：

```bash
go run ./main.go
```

## 三、关键代码结构

```go
type ErrorResponse struct {
    Code    string `json:"code"`
    Message string `json:"message"`
}
```

响应结构要稳定。前端更适合依赖 `code`，不应该依赖 `message` 的文本。

```go
func mapError(err error) (int, ErrorResponse)
```

统一转换函数把内部错误变成 HTTP 状态码和响应体。

```go
fmt.Println("log:", err)
```

日志保留完整错误链。真实项目中会使用结构化日志库，并记录 request id、user id、path、method 等字段。

## 四、真实后端场景示例

下面是一个更接近 HTTP handler 的写法。示例仍然不依赖第三方框架。

```go
package main

import (
    "encoding/json"
    "errors"
    "fmt"
    "net/http"
)

var ErrUserNotFound = errors.New("user not found")

type ErrorResponse struct {
    Code    string `json:"code"`
    Message string `json:"message"`
}

func mapError(err error) (int, ErrorResponse) {
    if errors.Is(err, ErrUserNotFound) {
        return http.StatusNotFound, ErrorResponse{
            Code:    "USER_NOT_FOUND",
            Message: "user not found",
        }
    }

    return http.StatusInternalServerError, ErrorResponse{
        Code:    "INTERNAL_ERROR",
        Message: "internal server error",
    }
}

func writeError(w http.ResponseWriter, err error) {
    status, response := mapError(err)

    fmt.Println("log:", err)

    w.Header().Set("Content-Type", "application/json")
    w.WriteHeader(status)
    _ = json.NewEncoder(w).Encode(response)
}

func main() {
    http.HandleFunc("/users", func(w http.ResponseWriter, r *http.Request) {
        err := fmt.Errorf("handle get user: %w", ErrUserNotFound)
        writeError(w, err)
    })

    fmt.Println("listen on :8080")
    _ = http.ListenAndServe(":8080", nil)
}
```

运行：

Windows PowerShell：

```powershell
go run .\main.go
```

Linux/macOS：

```bash
go run ./main.go
```

另开一个终端访问：

Windows PowerShell：

```powershell
Invoke-RestMethod http://localhost:8080/users
```

Linux/macOS：

```bash
curl http://localhost:8080/users
```

这个示例里的日志和响应是分开的。真实项目中，日志可以记录更多内部信息，但响应体要保持克制。

## 五、注意点

统一错误响应不是为了隐藏所有问题，而是为了建立边界。

建议：

- 对业务错误返回明确 code。
- 对系统错误统一返回 `INTERNAL_ERROR`。
- 日志中记录完整错误链和 request id。
- 不在响应里返回 SQL、堆栈、内部服务地址。
- 不把日志当作唯一恢复手段，重要写操作还要有审计和回滚策略。

如果错误码对外公开，就要谨慎变更。前端、移动端、第三方调用方都可能依赖它。

## 六、常见误区

误区一：每个 handler 自己写错误响应。

这样很容易不一致。新增错误类型时也容易漏改。

误区二：响应里直接返回 `err.Error()`。

这会把内部错误链暴露给用户。内部错误越详细，越不能直接外发。

误区三：日志只记录“请求失败”。

没有错误链、没有 request id、没有用户或路径信息，线上排查会非常困难。

## 七、本节达标标准

- 能定义稳定的错误响应结构。
- 能用 `errors.Is` 把业务错误映射成状态码和错误码。
- 能区分内部日志和外部响应。
- 能写出统一 `writeError` 函数。
- 能说明为什么系统错误不应该把内部细节返回给用户。
