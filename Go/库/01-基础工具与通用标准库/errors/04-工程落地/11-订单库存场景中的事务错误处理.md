# 11. 订单库存场景中的事务错误处理

本节目标：学完后，你能够在订单和库存这类多步骤写操作中，理解事务边界和错误回滚策略。

订单创建通常不是单个写操作。它可能包括创建订单、扣减库存、写支付记录、写操作日志。如果中间一步失败，前面已经写入的数据怎么办？这就是事务边界要解决的问题。

## 一、为什么需要它

假设创建订单时先写订单表，再扣库存。如果订单写成功了，扣库存失败了，系统里就会出现“有订单但库存没扣”的不一致。

在真实项目中，你需要明确：

```text
同一个业务动作中的多步关键写操作，要么全部成功，要么全部回滚。
```

错误处理在这里不是简单打印日志，而是决定事务是否提交、是否回滚、如何向上返回业务错误。

## 二、基本用法

下面用普通函数模拟事务，不连接真实数据库，先建立直觉。

```go
package main

import (
    "errors"
    "fmt"
)

var ErrInventoryNotEnough = errors.New("inventory not enough")

type Tx struct {
    committed bool
    rolledBack bool
}

func (tx *Tx) Commit() error {
    tx.committed = true
    fmt.Println("commit transaction")
    return nil
}

func (tx *Tx) Rollback() error {
    tx.rolledBack = true
    fmt.Println("rollback transaction")
    return nil
}

func createOrderRecord(tx *Tx) error {
    fmt.Println("insert order")
    return nil
}

func decreaseInventory(tx *Tx) error {
    return ErrInventoryNotEnough
}

func createOrder() error {
    tx := &Tx{}
    defer tx.Rollback()

    if err := createOrderRecord(tx); err != nil {
        return fmt.Errorf("create order record: %w", err)
    }

    if err := decreaseInventory(tx); err != nil {
        return fmt.Errorf("decrease inventory: %w", err)
    }

    if err := tx.Commit(); err != nil {
        return fmt.Errorf("commit create order transaction: %w", err)
    }

    return nil
}

func main() {
    if err := createOrder(); err != nil {
        fmt.Println("log:", err)
        fmt.Println("is inventory not enough:", errors.Is(err, ErrInventoryNotEnough))
    }
}
```

运行命令：

Windows PowerShell：

```powershell
go run .\main.go
```

Linux/macOS：

```bash
go run ./main.go
```

## 三、关键代码结构

```go
defer tx.Rollback()
```

常见做法是先安排回滚。真实 `database/sql` 中，如果后面 `Commit` 成功，再调用 `Rollback` 通常会返回一个可忽略的错误。项目中可以封装事务 helper，让这段逻辑更稳。

```go
return fmt.Errorf("decrease inventory: %w", err)
```

库存扣减失败时，错误要带上业务步骤。否则你只知道库存不足，不知道是在创建订单流程里失败。

```go
errors.Is(err, ErrInventoryNotEnough)
```

handler 可以把库存不足返回成 409 Conflict 或 400，具体取决于接口约定。

## 四、真实后端场景示例

真实 `database/sql` 代码结构通常像这样：

```go
func CreateOrder(ctx context.Context, db *sql.DB, userID, skuID int64, quantity int) error {
    tx, err := db.BeginTx(ctx, nil)
    if err != nil {
        return fmt.Errorf("begin create order transaction: %w", err)
    }
    defer tx.Rollback()

    if err := insertOrder(ctx, tx, userID, skuID, quantity); err != nil {
        return fmt.Errorf("insert order: %w", err)
    }

    if err := decreaseInventory(ctx, tx, skuID, quantity); err != nil {
        return fmt.Errorf("decrease inventory: %w", err)
    }

    if err := tx.Commit(); err != nil {
        return fmt.Errorf("commit create order transaction: %w", err)
    }

    return nil
}
```

数据库更新库存时，要避免先查再无条件更新。更稳的思路是用条件更新：

```sql
UPDATE inventory
SET stock = stock - ?
WHERE sku_id = ? AND stock >= ?;
```

然后检查受影响行数。如果是 0，通常可以转换成 `ErrInventoryNotEnough`。

```text
库存扣减要让数据库条件更新参与保护，不能只依赖应用层先查再减。
```

## 五、注意点

事务边界要尽量短。

建议：

- 把必须保持一致的数据库写操作放进事务。
- 不要在事务里调用慢 RPC、发短信、发邮件。
- 事务失败后返回带上下文的错误。
- `Commit` 失败也要处理，它同样代表业务没有确认成功。
- 需要外部副作用时，考虑 outbox、消息表或补偿任务。

迁移和索引也会影响订单库存场景。比如给库存表加唯一索引或修改字段类型，要通过迁移工具上线，并准备回滚方案。大表加索引可能锁表或拖慢写入，要评估生产风险。

## 六、常见误区

误区一：创建订单和扣库存不在同一个事务里。

这会导致部分成功，后面只能靠人工或补偿任务修复。

误区二：忽略 `Commit` 错误。

`Commit` 失败时，不能当作成功返回。调用方应该看到错误，系统也要记录日志。

误区三：事务里做外部调用。

事务打开期间占用连接和锁，外部调用慢或失败会放大数据库压力。

误区四：用普通错误吞掉业务错误。

库存不足应该能被 `errors.Is(err, ErrInventoryNotEnough)` 识别，方便 handler 返回稳定状态码。
