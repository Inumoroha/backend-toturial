# 08. 用户模块中的错误分层

本节目标：学完后，你能够在 handler、service、repository 三层中合理传递和转换错误。

很多初学者写后端接口时，会把参数校验、数据库查询、业务判断、HTTP 响应都写在一个函数里。短期能跑，后面一旦接口变多，错误处理会越来越乱。用户模块是最适合练习错误分层的场景，因为它有参数错误、用户不存在、密码错误、权限不足、数据库失败等常见情况。

## 一、为什么需要它

后端项目里，每一层对错误的关注点不同：

```text
repository 关心数据访问是否成功。
service 关心业务规则是否成立。
handler 关心如何返回 HTTP 响应。
```

如果 repository 直接返回 HTTP 404，它就和 Web 协议绑定了。以后这个 service 被 gRPC、定时任务、命令行工具复用时，就会很别扭。

## 二、基本用法

复制下面代码到 `main.go`。它没有启动 HTTP 服务，而是用函数模拟三层调用，方便你直接运行看结果。

```go
package main

import (
    "errors"
    "fmt"
)

var (
    ErrUserNotFound      = errors.New("user not found")
    ErrInvalidCredential = errors.New("invalid credential")
)

func repoFindPasswordHash(username string) (string, error) {
    if username == "missing" {
        return "", ErrUserNotFound
    }

    return "secret-hash", nil
}

func serviceLogin(username, password string) error {
    hash, err := repoFindPasswordHash(username)
    if err != nil {
        return fmt.Errorf("login: find user password hash: %w", err)
    }

    if password != hash {
        return ErrInvalidCredential
    }

    return nil
}

func statusCode(err error) int {
    switch {
    case errors.Is(err, ErrUserNotFound):
        return 404
    case errors.Is(err, ErrInvalidCredential):
        return 401
    default:
        return 500
    }
}

func main() {
    err := serviceLogin("missing", "secret-hash")
    if err != nil {
        fmt.Println("status:", statusCode(err))
        fmt.Println("log:", err)
        return
    }

    fmt.Println("status: 200")
}
```

运行命令：

Windows PowerShell：

```powershell
go run .\main.go
```

Linux/macOS：

```bash
go run ./main.go
```

## 三、关键代码结构

repository 层：

```go
func repoFindPasswordHash(username string) (string, error)
```

它只负责查数据。真实项目中这里可能调用数据库，失败时应该包装成 `query user by username: %w`。

service 层：

```go
return fmt.Errorf("login: find user password hash: %w", err)
```

它补充业务动作上下文。登录失败时，日志能看出是在登录流程里查密码失败。

handler 层：

```go
statusCode(err)
```

它把内部错误转换成 HTTP 状态码。真实项目中还会统一返回 JSON。

## 四、真实后端场景示例

下面给一个更接近项目的响应转换函数。

```go
package main

import (
    "encoding/json"
    "errors"
    "fmt"
)

var (
    ErrUserNotFound      = errors.New("user not found")
    ErrInvalidCredential = errors.New("invalid credential")
)

type ErrorResponse struct {
    Code    string `json:"code"`
    Message string `json:"message"`
}

func errorResponse(err error) (int, ErrorResponse) {
    switch {
    case errors.Is(err, ErrUserNotFound):
        return 404, ErrorResponse{Code: "USER_NOT_FOUND", Message: "user not found"}
    case errors.Is(err, ErrInvalidCredential):
        return 401, ErrorResponse{Code: "INVALID_CREDENTIAL", Message: "invalid username or password"}
    default:
        return 500, ErrorResponse{Code: "INTERNAL_ERROR", Message: "internal server error"}
    }
}

func main() {
    err := fmt.Errorf("login: find user password hash: %w", ErrUserNotFound)

    status, body := errorResponse(err)
    data, _ := json.Marshal(body)

    fmt.Println("status:", status)
    fmt.Println("response:", string(data))
    fmt.Println("log:", err)
}
```

这个例子有两个输出：给用户看的 response，给内部排查看的 log。真实项目中，这两者不要混在一起。

## 五、注意点

用户模块有一个安全细节：登录接口通常不应该区分“用户不存在”和“密码错误”的响应文案，否则可能被用来枚举账号。

内部可以记录：

```text
login: find user password hash: user not found
```

外部可以统一返回：

```text
invalid username or password
```

这就是错误处理里的生产安全。

## 六、常见误区

误区一：repository 层直接返回 HTTP 状态码。

这会让底层代码依赖 Web 框架，不利于复用和测试。

误区二：handler 直接把 `err.Error()` 返回给用户。

这可能泄露数据库表名、内部路径、RPC 地址或敏感业务细节。

误区三：service 层只写 `return err`。

这样日志缺少业务动作。至少要在关键边界补充上下文。

## 七、本节达标标准

- 能说清 handler、service、repository 三层的错误处理职责。
- 能在 service 层用 `%w` 补充业务上下文。
- 能在 handler 层用 `errors.Is` 转换状态码和响应码。
- 能区分内部日志和用户响应。
- 能注意登录等场景中的安全边界。
