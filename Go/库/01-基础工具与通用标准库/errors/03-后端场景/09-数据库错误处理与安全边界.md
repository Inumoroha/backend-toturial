# 09. 数据库错误处理与安全边界

本节目标：学完后，你能够在数据库访问中安全处理错误，避免 SQL 注入、错误泄露和不可控变更。

数据库错误是后端项目里最常见的错误来源之一。它可能来自数据不存在、唯一索引冲突、连接超时、事务失败、迁移脚本问题。处理数据库错误时，要同时考虑业务语义、日志排查和生产安全。

## 一、为什么需要它

真实项目中，数据库错误不能简单地 `return err` 到最外层。你需要回答几个问题：

- 这是业务错误还是系统错误？
- 是否应该转换成 `ErrUserNotFound` 这类业务错误？
- 日志里要保留哪些上下文？
- 用户响应里能不能暴露数据库细节？
- 这次写操作失败后，是否需要回滚？

```text
数据库错误要在内部保留细节，在外部隐藏细节。
```

## 二、基本用法

下面示例使用标准库模拟 `sql.ErrNoRows`，不需要连接真实数据库。

```go
package main

import (
    "database/sql"
    "errors"
    "fmt"
)

var ErrUserNotFound = errors.New("user not found")

func repoFindUser(id int64) error {
    if id == 404 {
        return fmt.Errorf("query user by id %d: %w", id, sql.ErrNoRows)
    }

    return nil
}

func serviceGetUser(id int64) error {
    if err := repoFindUser(id); err != nil {
        if errors.Is(err, sql.ErrNoRows) {
            return ErrUserNotFound
        }

        return fmt.Errorf("get user: %w", err)
    }

    return nil
}

func main() {
    err := serviceGetUser(404)
    if err != nil {
        fmt.Println("is user not found:", errors.Is(err, ErrUserNotFound))
        fmt.Println("log:", err)
    }
}
```

运行命令：

Windows PowerShell：

```powershell
go run .\main.go
```

Linux/macOS：

```bash
go run ./main.go
```

## 三、关键代码结构

repository 层保留数据库上下文：

```go
return fmt.Errorf("query user by id %d: %w", id, sql.ErrNoRows)
```

service 层把数据库语义转换成业务语义：

```go
if errors.Is(err, sql.ErrNoRows) {
    return ErrUserNotFound
}
```

handler 层再把业务错误转换成 HTTP 404。

这样做的好处是：数据库细节不会泄露到接口层，但内部仍然能定位是查询用户时没有数据。

## 四、真实后端场景示例

假设你要按用户名查询用户。真实数据库代码通常要使用参数化查询。

```go
func FindUserByName(db *sql.DB, username string) error {
    const query = `SELECT id FROM users WHERE username = ?`

    var id int64
    if err := db.QueryRow(query, username).Scan(&id); err != nil {
        return fmt.Errorf("query user by username: %w", err)
    }

    return nil
}
```

这里的 `?` 是参数占位符。不同数据库驱动占位符可能不同，例如 PostgreSQL 常用 `$1`。

不要这样写：

```go
query := "SELECT id FROM users WHERE username = '" + username + "'"
```

```text
用户输入不能直接拼进 SQL 字符串。
```

拼接 SQL 会带来 SQL 注入风险，也会让数据库执行计划和审计变得更难控制。

## 五、注意点

数据库相关实践要保守。

参数化查询：

- 用户输入必须通过参数绑定传入。
- 不要拼接 SQL 条件。
- 排序字段、表名这类不能参数化的位置，要做白名单。

事务边界：

- 一个业务动作需要多个写操作保持一致时，使用事务。
- 事务中任何一步失败，都要回滚。
- 不要把耗时的外部 RPC 放进事务里，避免长时间占用连接和锁。

迁移工具：

- 表结构变更不要靠手工登录生产库执行。
- 使用迁移工具记录版本。
- 每次迁移都要考虑回滚方案。

索引成本：

- 索引能提升查询，也会增加写入成本和存储成本。
- 不要为了“可能会查”就随意加索引。
- 上线前要评估查询条件、数据量和慢查询。

生产安全：

- 数据库错误不要直接返回给用户。
- 日志里不要打印完整敏感字段。
- 删除、迁移、批量更新前要有备份或可回滚策略。

## 六、常见误区

误区一：把 `sql.ErrNoRows` 直接传到 handler。

handler 不应该关心数据库库里的具体错误。service 层应该转换成业务错误，例如 `ErrUserNotFound`。

误区二：把数据库错误直接返回给用户。

```go
http.Error(w, err.Error(), 500)
```

这可能暴露表名、字段名、SQL 片段或连接信息。

误区三：用字符串判断数据库错误。

```go
if strings.Contains(err.Error(), "no rows") {
    return ErrUserNotFound
}
```

优先使用数据库驱动提供的错误类型或标准错误，例如 `errors.Is(err, sql.ErrNoRows)`。

误区四：迁移失败没有回滚方案。

生产库变更前要先考虑失败时怎么恢复。尤其是删除字段、批量更新、加唯一索引这类操作。

## 七、本节达标标准

- 能用 `errors.Is(err, sql.ErrNoRows)` 判断数据不存在。
- 能把数据库错误转换成业务错误。
- 能解释为什么用户输入不能拼接 SQL。
- 能说明事务边界、迁移工具、索引成本和回滚方案的重要性。
- 能避免把数据库内部错误直接暴露给用户。
