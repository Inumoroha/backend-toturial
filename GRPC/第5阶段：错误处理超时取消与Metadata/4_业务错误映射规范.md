# 4. 业务错误映射规范

本节目标：设计一套项目内统一的业务错误到 gRPC code 的映射方式。

真实项目里，业务层可能返回 `ErrUserNotFound`、`ErrBalanceNotEnough`、`ErrOrderClosed`。这些错误不能原样丢给 gRPC client，需要统一转换为标准 code 和安全 message。

---

## 一、核心直觉

- domain error 表示业务语义。
- transport error 表示通信层错误。
- handler 边界负责把 domain error 映射成 gRPC status。
- 内部错误要记录日志，对外返回通用信息。
- 错误映射应该集中管理，避免散落在每个方法里。

---

## 二、动手步骤

1. 定义业务错误变量。
2. 写一个 `toStatusError(err error) error`。
3. 在 gRPC handler 返回前统一转换。
4. client 只依赖 gRPC code，不依赖服务端内部错误类型。

---

## 三、参考代码或命令

```go
var ErrUserNotFound = errors.New("user not found")

func toStatusError(err error) error {
    switch {
    case errors.Is(err, ErrUserNotFound):
        return status.Error(codes.NotFound, "user not found")
    default:
        return status.Error(codes.Internal, "internal error")
    }
}
```

---

## 四、常见问题

- 在 service 层直接 import gRPC codes：会让业务层依赖传输协议。
- 对外暴露数据库错误：安全性和可维护性都不好。
- 每个 handler 自己写 switch：规则迟早会不一致。

## 七、为什么需要业务错误映射

在真实项目里，业务层不应该到处 import gRPC 的 `codes` 和 `status`。原因是业务逻辑不应该和传输协议绑死。

例如用户领域可能有这些错误：

```go
var (
    ErrUserNotFound   = errors.New("user not found")
    ErrEmailExists    = errors.New("email already exists")
    ErrInvalidUserID  = errors.New("invalid user id")
    ErrPermissionDeny = errors.New("permission denied")
)
```

这些是业务错误。到了 gRPC handler 边界，再统一转换成：

```text
ErrUserNotFound   -> codes.NotFound
ErrEmailExists    -> codes.AlreadyExists
ErrInvalidUserID  -> codes.InvalidArgument
ErrPermissionDeny -> codes.PermissionDenied
```

这样以后如果你要提供 HTTP API，也可以把同一批业务错误映射成 HTTP status，而不用改业务层。

6. client 验证不同业务错误对应不同 gRPC code。

## 九、完整代码：定义业务错误

可以新建 `internal/user/errors.go`：

```go
package user

import "errors"

var (
    ErrInvalidUserID = errors.New("invalid user id")
    ErrUserNotFound  = errors.New("user not found")
    ErrEmailExists   = errors.New("email already exists")
)
```

业务错误使用普通 Go error，保持领域层简单。

## 十、完整代码：业务 service

```go
package user

type Model struct {
    ID    int64
    Name  string
    Email string
}

type Service struct {
    users map[int64]*Model
}

func NewService() *Service {
    return &Service{
        users: map[int64]*Model{
            1: {ID: 1, Name: "Tom", Email: "tom@example.com"},
        },
    }
}

func (s *Service) GetUser(id int64) (*Model, error) {
    if id <= 0 {
        return nil, ErrInvalidUserID
    }

    u := s.users[id]
    if u == nil {
        return nil, ErrUserNotFound
    }

    return u, nil
}
```

注意：这里没有出现 `codes.NotFound` 或 `status.Error`。业务层只表达业务语义。

## 十一、完整代码：错误转换函数

可以在 gRPC server 文件中写：

```go
func toStatusError(err error) error {
    if err == nil {
        return nil
    }

    switch {
    case errors.Is(err, user.ErrInvalidUserID):
        return status.Error(codes.InvalidArgument, "id must be positive")
    case errors.Is(err, user.ErrUserNotFound):
        return status.Error(codes.NotFound, "user not found")
    case errors.Is(err, user.ErrEmailExists):
        return status.Error(codes.AlreadyExists, "email already exists")
    default:
        log.Printf("unexpected error: %v", err)
        return status.Error(codes.Internal, "internal error")
    }
}
```

需要 import：

```go
import (
    "errors"
    "log"

    "example.com/grpc-learning/internal/user"
    "google.golang.org/grpc/codes"
    "google.golang.org/grpc/status"
)
```

---

## 十二、完整代码：handler 中使用转换函数

```go
type userServer struct {
    userv1.UnimplementedUserServiceServer
    svc *user.Service
}

func (s *userServer) GetUser(ctx context.Context, req *userv1.GetUserRequest) (*userv1.GetUserResponse, error) {
    model, err := s.svc.GetUser(req.GetId())
    if err != nil {
        return nil, toStatusError(err)
    }

    return &userv1.GetUserResponse{
        User: &userv1.User{
            Id:    model.ID,
            Name:  model.Name,
            Email: model.Email,
        },
    }, nil
}
```

这样 gRPC handler 的职责就很清楚：

```text
接收 protobuf request
-> 调用业务 service
-> 把业务 model 转成 protobuf response
-> 把业务 error 转成 gRPC status error
```

## 十五、常见错误排查

### 1. 业务层直接返回 status.Error

不推荐：

```go
return nil, status.Error(codes.NotFound, "user not found")
```

如果写在业务层，未来换 HTTP、CLI、消息消费场景都会带着 gRPC 依赖。

### 2. handler 到处写 switch

如果每个 RPC 方法里都自己写错误映射，规则会越来越不一致。建议集中到一个转换函数或转换包。

### 3. default 直接返回原始错误

不要这样：

```go
return err
```

未知错误对外应该返回 `Internal`，真实错误写日志。

### 4. message 和 code 不一致

比如：

```go
status.Error(codes.NotFound, "internal error")
```

code 和 message 要表达同一类语义。

## 十六、业务错误映射表模板

建议每个项目维护一张表：

| 业务错误 | gRPC code | 对外 message | 备注 |
| --- | --- | --- | --- |
| ErrInvalidUserID | InvalidArgument | id must be positive | 参数错误 |
| ErrUserNotFound | NotFound | user not found | 资源不存在 |
| ErrEmailExists | AlreadyExists | email already exists | 唯一冲突 |
| ErrTokenMissing | Unauthenticated | missing token | 未认证 |
| ErrNoPermission | PermissionDenied | permission denied | 无权限 |
| ErrStockNotEnough | FailedPrecondition | stock not enough | 当前状态不满足操作条件 |
