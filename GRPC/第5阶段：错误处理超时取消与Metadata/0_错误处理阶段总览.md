# 0. 错误处理、超时、取消与 Metadata 总览

本节目标：建立 gRPC 调用中的错误、生命周期和上下文传递规范。

跑通 demo 只是开始。真实项目中，你必须让调用方知道失败原因，必须给请求设置超时，必须能传递鉴权信息和 trace id。本阶段开始从“能跑”进入“可维护”。

---

## 一、核心直觉

- gRPC 使用 `codes` 和 `status` 表达标准错误。
- context 负责超时、取消和请求范围的值传递。
- metadata 类似 HTTP Header，适合传 token、trace id、tenant id。
- trailer 可以在响应结束时返回额外信息。
- 错误模型应该统一，不要每个方法各写一套。

---

## 二、动手步骤

1. 把普通 error 改成 `status.Error`。
2. client 使用 `status.FromError` 解析错误码。
3. 给调用设置 timeout。
4. 使用 metadata 传递 token。
5. 设计统一的业务错误映射规则。

---

## 三、参考代码或命令

```go
return nil, status.Error(codes.NotFound, "user not found")

st, ok := status.FromError(err)
if ok {
    switch st.Code() {
    case codes.NotFound:
        // handle not found
    }
}
```

## 四、常见问题

- 直接把数据库错误暴露给客户端：会泄露内部实现。
- 所有错误都返回 Internal：调用方无法做精确处理。
- metadata 中记录敏感信息到日志：token、密码类信息必须脱敏。

## 七、本阶段为什么重要

前面你已经能写 Unary 和 Streaming，但真实服务不能只“成功时能跑”。后端工程里更常见的问题是：

```text
参数不合法怎么办？
资源不存在怎么办？
用户没登录怎么办？
下游太慢怎么办？
客户端取消请求怎么办？
token、trace id、租户 id 怎么传？
业务错误怎么统一映射成 gRPC code？
```

这些问题如果没有统一规范，服务越多越混乱。第 5 阶段就是要把“调用失败时怎么表达”和“请求上下文怎么传递”建立成习惯。

- 参数非法时返回 `InvalidArgument`。
- 用户不存在时返回 `NotFound`。
- 没带 token 时返回 `Unauthenticated`。
- 模拟慢请求，触发 `DeadlineExceeded`。
- 通过 metadata 传递 `authorization` 和 `x-trace-id`。
- 把业务错误统一映射为 gRPC status error。

## 九、推荐实验顺序

请按下面顺序推进：

1. 先把普通 `error` 改成 `status.Error`。
2. client 使用 `status.FromError` 解析 code。
3. 给 `GetUser` 加参数校验。
4. 给 client 增加 timeout。
5. server 模拟慢请求，并监听 `ctx.Done()`。
6. client 通过 metadata 传 token 和 trace id。
7. server 从 metadata 中读取并校验 token。
8. 抽取业务错误映射函数。

不要一上来就写 interceptor。第 6 阶段会专门把鉴权、日志、错误恢复这些逻辑抽到 interceptor。本阶段先在 handler 中理解机制。

## 十、完整代码：错误处理后的 GetUser 骨架

```go
func (s *userServer) GetUser(ctx context.Context, req *userv1.GetUserRequest) (*userv1.GetUserResponse, error) {
    if req.Id <= 0 {
        return nil, status.Error(codes.InvalidArgument, "id must be positive")
    }

    if err := checkAuth(ctx); err != nil {
        return nil, err
    }

    if req.Id == 100 {
        select {
        case <-time.After(3 * time.Second):
        case <-ctx.Done():
            return nil, ctx.Err()
        }
    }

    user := s.users[req.Id]
    if user == nil {
        return nil, status.Error(codes.NotFound, "user not found")
    }

    return &userv1.GetUserResponse{User: user}, nil
}
```

这里已经包含：参数校验、鉴权、超时感知、NotFound 返回。

## 十一、完整代码：客户端错误解析骨架

```go
resp, err := client.GetUser(ctx, &userv1.GetUserRequest{Id: id})
if err != nil {
    st, ok := status.FromError(err)
    if !ok {
        log.Fatalf("non-grpc error: %v", err)
    }

    switch st.Code() {
    case codes.InvalidArgument:
        log.Fatalf("bad request: %s", st.Message())
    case codes.NotFound:
        log.Fatalf("user not found: %s", st.Message())
    case codes.Unauthenticated:
        log.Fatalf("auth failed: %s", st.Message())
    case codes.DeadlineExceeded:
        log.Fatalf("timeout: %s", st.Message())
    default:
        log.Fatalf("grpc error: code=%s message=%s", st.Code(), st.Message())
    }
}

log.Println(resp.GetUser())
```

重点是：client 不要只打印 `err` 字符串，而要解析 `code`。

## 十四、常见错误排查

### 1. 所有错误都变成 Unknown

如果服务端返回普通 `fmt.Errorf`，client 可能只能看到 `Unknown`。建议服务边界返回 `status.Error(code, message)`。

### 2. client 只打印 err 字符串

这样无法稳定区分错误类型。请使用：

```go
st, ok := status.FromError(err)
```

### 3. metadata 读取不到

检查 client 是否使用：

```go
metadata.NewOutgoingContext(ctx, md)
```

server 是否使用：

```go
metadata.FromIncomingContext(ctx)
```

### 4. 超时后服务端还在工作

检查 server 是否监听了：

```go
ctx.Done()
```

如果只是 `time.Sleep`，服务端无法及时感知取消。
