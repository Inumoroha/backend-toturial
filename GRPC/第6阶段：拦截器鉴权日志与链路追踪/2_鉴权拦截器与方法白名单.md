# 2. 鉴权拦截器与方法白名单

本节目标：使用 metadata 和 interceptor 实现统一鉴权，并支持公开方法白名单。

鉴权逻辑如果写在每个 RPC 方法里，会重复且容易漏。更好的方式是在 interceptor 中统一读取 metadata，验证 token，再决定是否放行。

---

## 一、核心规则

- token 通常放在 metadata 的 `authorization` 中。
- 认证失败返回 `codes.Unauthenticated`。
- 权限不足返回 `codes.PermissionDenied`。
- 健康检查、登录、公开查询可以使用方法白名单跳过鉴权。
- 鉴权结果可以写入 context，供后续业务方法读取。

---

## 二、读取 metadata

```go
md, ok := metadata.FromIncomingContext(ctx)
if !ok {
    return nil, status.Error(codes.Unauthenticated, "missing metadata")
}

authValues := md.Get("authorization")
if len(authValues) == 0 {
    return nil, status.Error(codes.Unauthenticated, "missing token")
}
```

metadata key 建议统一小写，例如 `authorization`、`x-trace-id`、`x-tenant-id`。

---

## 三、加入方法白名单

```go
var publicMethods = map[string]bool{
    "/grpc.health.v1.Health/Check": true,
    "/user.v1.UserService/Login": true,
}

func AuthInterceptor(ctx context.Context, req any, info *grpc.UnaryServerInfo, handler grpc.UnaryHandler) (any, error) {
    if publicMethods[info.FullMethod] {
        return handler(ctx, req)
    }

    md, ok := metadata.FromIncomingContext(ctx)
    if !ok || len(md.Get("authorization")) == 0 {
        return nil, status.Error(codes.Unauthenticated, "missing token")
    }

    token := md.Get("authorization")[0]
    if token != "Bearer dev-token" {
        return nil, status.Error(codes.Unauthenticated, "invalid token")
    }

    return handler(ctx, req)
}
```

---

## 四、客户端传 token

```go
md := metadata.Pairs("authorization", "Bearer dev-token")
ctx := metadata.NewOutgoingContext(context.Background(), md)
resp, err := client.GetUser(ctx, &userv1.GetUserRequest{Id: 1})
```

---

## 五、常见问题

- 鉴权失败返回 Internal：调用方无法知道是认证问题。
- 不做白名单：健康检查可能被拦住，导致服务被误判不健康。
- 在日志里打印完整 token：必须脱敏。
- 在业务方法里重复解析 token：应该在拦截器统一处理。

## 七、完整操作步骤

本节目标是把第 5 阶段写在 handler 里的 `checkAuth(ctx)` 移到 Unary interceptor 中。

操作步骤：

1. 创建 `internal/middleware/auth.go`。
2. 定义公开方法白名单。
3. 在 interceptor 中读取 metadata。
4. 校验 `authorization` token。
5. token 缺失或错误时返回 `Unauthenticated`。
6. token 正确时调用 handler。
7. 在 server 中使用 `grpc.ChainUnaryInterceptor` 组合 logging 和 auth。
8. 用 grpcurl 测试无 token、错误 token、正确 token。

---

## 八、完整代码：auth.go

```go
package middleware

import (
    "context"

    "google.golang.org/grpc"
    "google.golang.org/grpc/codes"
    "google.golang.org/grpc/metadata"
    "google.golang.org/grpc/status"
)

var publicMethods = map[string]bool{
    "/grpc.health.v1.Health/Check": true,
}

func AuthInterceptor(
    ctx context.Context,
    req any,
    info *grpc.UnaryServerInfo,
    handler grpc.UnaryHandler,
) (any, error) {
    if publicMethods[info.FullMethod] {
        return handler(ctx, req)
    }

    md, ok := metadata.FromIncomingContext(ctx)
    if !ok {
        return nil, status.Error(codes.Unauthenticated, "missing metadata")
    }

    values := md.Get("authorization")
    if len(values) == 0 {
        return nil, status.Error(codes.Unauthenticated, "missing token")
    }

    if values[0] != "Bearer dev-token" {
        return nil, status.Error(codes.Unauthenticated, "invalid token")
    }

    return handler(ctx, req)
}
```

这个版本使用固定 token，适合学习。真实项目中应该调用认证服务、解析 JWT、验证 mTLS 身份，或对接 IAM。

## 九、server 链式注册

```go
server := grpc.NewServer(
    grpc.ChainUnaryInterceptor(
        middleware.UnaryLoggingInterceptor,
        middleware.AuthInterceptor,
    ),
)
```

顺序会影响日志效果。

如果 Logging 在 Auth 前面：

```text
无 token 请求也会被记录日志
```

如果 Auth 在 Logging 前面：

```text
Auth 拒绝后，Logging 可能记录不到
```

学习阶段建议 Logging 放在 Auth 前面，方便观察所有请求。

## 十、grpcurl 测试命令

无 token：

```powershell
grpcurl -plaintext -d '{"id":1}' localhost:50051 user.v1.UserService/GetUser
```

预期：

```text
ERROR:
  Code: Unauthenticated
  Message: missing token
```

错误 token：

```powershell
grpcurl -plaintext `
  -H "authorization: Bearer wrong-token" `
  -d '{"id":1}' `
  localhost:50051 user.v1.UserService/GetUser
```

预期：

```text
ERROR:
  Code: Unauthenticated
  Message: invalid token
```

正确 token：

```powershell
grpcurl -plaintext `
  -H "authorization: Bearer dev-token" `
  -d '{"id":1}' `
  localhost:50051 user.v1.UserService/GetUser
```

预期返回用户。

## 十一、客户端发送 token


```go

    "authorization", "Bearer dev-token",
    "x-trace-id", "trace-001",
)
```

## 十二、把用户信息写入 context

真实项目里，鉴权成功后通常会得到 user id、角色、租户等信息。可以写入 context：

```go
type contextKey string

const userIDKey contextKey = "user_id"

func ContextWithUserID(ctx context.Context, userID int64) context.Context {
    return context.WithValue(ctx, userIDKey, userID)
}

func UserIDFromContext(ctx context.Context) (int64, bool) {
    v := ctx.Value(userIDKey)
    id, ok := v.(int64)
    return id, ok
}
```

在 Auth 中：

```go
ctx = ContextWithUserID(ctx, 1001)
return handler(ctx, req)
```

业务方法中：

```go
userID, ok := middleware.UserIDFromContext(ctx)
```

注意：context value 适合放请求范围的小信息，不要放大对象。

## 十三、方法白名单

有些方法不应该强制鉴权，例如：

- 健康检查。
- 登录方法。
- 公开配置查询。
- reflection，是否公开要看环境。

可以维护 map：

```go
var publicMethods = map[string]bool{
    "/grpc.health.v1.Health/Check": true,
    "/user.v1.UserService/Login": true,
}
```

完整方法名可以从日志中的 `info.FullMethod` 看到。

---

## 十四、常见错误排查

### 1. metadata 明明传了，但 server 读不到

检查 key 是否一致：

```text
authorization
```

不要一边写 `Authorization`，一边读 `authorization`。gRPC metadata key 建议统一小写。

### 2. 所有方法都被鉴权拦住

检查白名单里的方法名是否和 `info.FullMethod` 完全一致。

### 3. token 被打印到日志

不要完整打印 token。最多打印前后几位或直接打印“present/missing”。

### 4. 鉴权失败返回 PermissionDenied

token 缺失或无效应该是 `Unauthenticated`。已经认证但权限不足才是 `PermissionDenied`。
