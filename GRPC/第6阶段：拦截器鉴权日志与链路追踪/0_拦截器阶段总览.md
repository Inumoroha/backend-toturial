# 0. 拦截器阶段总览

本阶段目标：理解 interceptor 的作用，并把日志、鉴权、恢复、trace 等横切逻辑从业务代码中抽离。

当服务方法越来越多时，如果每个方法都手写日志、鉴权、panic recover、trace id，代码会很快重复且不一致。gRPC interceptor 就是为这类横切逻辑准备的。

---

## 一、核心直觉

- Unary interceptor 处理普通一问一答 RPC。
- Stream interceptor 处理流式 RPC。
- Server interceptor 运行在服务端方法前后。
- Client interceptor 运行在客户端发起调用前后。
- 多个 interceptor 可以组成链，顺序会影响行为。

你可以把 interceptor 理解成 gRPC 世界里的 middleware。业务方法只关心业务，鉴权、日志、恢复、trace 这些通用逻辑放在外层统一处理。

---

## 二、本阶段要完成什么

1. 实现 Unary 日志拦截器。
2. 实现 token 鉴权拦截器。
3. 实现 panic recovery 拦截器。
4. 实现 trace id 传递。
5. 理解 Stream interceptor 和链式组合。

## 三、最小示例

```go
func UnaryLoggingInterceptor(
    ctx context.Context,
    req any,
    info *grpc.UnaryServerInfo,
    handler grpc.UnaryHandler,
) (any, error) {
    start := time.Now()
    resp, err := handler(ctx, req)
    code := codes.OK
    if err != nil {
        code = status.Code(err)
    }
    log.Printf("method=%s code=%s cost=%s", info.FullMethod, code, time.Since(start))
    return resp, err
}
```

注册：

```go
s := grpc.NewServer(
    grpc.UnaryInterceptor(UnaryLoggingInterceptor),
)
```

## 四、常见问题

- 把业务逻辑写进 interceptor：interceptor 适合通用逻辑，不适合具体业务。
- 鉴权拦截器不区分公开方法：健康检查、登录等方法可能需要白名单。
- recovery 吃掉错误但不记录日志：会让线上问题难以排查。
- 只给 Unary 加拦截器，忘了 Streaming：流式方法可能绕过鉴权和日志。

## 七、本阶段为什么重要

前面第 5 阶段已经把错误码、metadata、deadline 学清楚了。但如果每个 RPC 方法都手写这些逻辑，代码会迅速变成这样：

```go
func (s *userServer) GetUser(ctx context.Context, req *userv1.GetUserRequest) (*userv1.GetUserResponse, error) {
    start := time.Now()
    traceID := readTraceID(ctx)
    log.Printf("start trace_id=%s method=GetUser", traceID)

    if err := checkAuth(ctx); err != nil {
        log.Printf("failed trace_id=%s cost=%s", traceID, time.Since(start))
        return nil, err
    }

    defer func() {
        log.Printf("finish trace_id=%s cost=%s", traceID, time.Since(start))
    }()

    // business logic...
}
```

如果有 50 个 RPC 方法，这段日志、鉴权、trace、panic recover 会重复 50 次。Interceptor 就是为了解决这种横切逻辑。

## 十、完整代码：server 注册拦截器骨架

```go
server := grpc.NewServer(
    grpc.ChainUnaryInterceptor(
        middleware.RecoveryInterceptor,
        middleware.UnaryLoggingInterceptor,
        middleware.AuthInterceptor,
    ),
    grpc.ChainStreamInterceptor(
        middleware.StreamLoggingInterceptor,
    ),
)

userv1.RegisterUserServiceServer(server, newUserServer())
reflection.Register(server)
```

顺序很重要。一般建议：

```text
Recovery 在最外层
Logging 记录整体结果
Auth 在业务执行前拦截非法请求
业务 handler 最后执行
```

## 十一、运行命令

启动服务端：

```powershell
go run ./cmd/server
```

不带 token 调用：

```powershell
grpcurl -plaintext -d '{"id":1}' localhost:50051 user.v1.UserService/GetUser
```

带 token 调用：

```powershell
grpcurl -plaintext `
  -H "authorization: Bearer dev-token" `
  -H "x-trace-id: trace-001" `
  -d '{"id":1}' `
  localhost:50051 user.v1.UserService/GetUser
```

## 十二、预期输出

无 token 时 client 或 grpcurl 输出：

```text
ERROR:
  Code: Unauthenticated
  Message: missing token
```

服务端日志：

```text
method=/user.v1.UserService/GetUser code=Unauthenticated duration=1ms trace_id=trace-xxx
```

带正确 token 时输出：

```json
{
  "user": {
    "id": "1",
    "name": "Tom",
    "email": "tom@example.com"
  }
}
```

服务端日志：

```text
method=/user.v1.UserService/GetUser code=OK duration=2ms trace_id=trace-001
```

## 十三、常见错误排查

### 1. 拦截器没有生效

检查是否真的传入了 `grpc.NewServer`：

```go
grpc.NewServer(grpc.ChainUnaryInterceptor(...))
```

如果你创建了 interceptor 函数但没有注册，它不会自动生效。

### 2. 忘记调用 handler

Unary interceptor 中必须调用：

```go
resp, err := handler(ctx, req)
```

否则业务方法不会执行。

### 3. Auth 拦住 health check

健康检查或 reflection 在某些环境中可能不需要鉴权。需要设计白名单。

### 4. Recovery 放在链条内层

如果 Recovery 不在外层，前面的 Logging/Auth panic 可能捕获不到。
