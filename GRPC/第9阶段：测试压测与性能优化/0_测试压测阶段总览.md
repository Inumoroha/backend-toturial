# 0. 测试、压测与性能优化总览

本阶段目标：为 gRPC 服务建立单元测试、集成测试、压测和性能观察能力。

没有测试的 gRPC 服务很难长期维护。你需要测试 handler 逻辑、错误码、metadata、deadline、streaming；上线前还要知道服务的延迟、吞吐和瓶颈。

---

## 一、测试分层

```text
业务单元测试
  -> gRPC bufconn 集成测试
  -> 少量端到端测试
  -> 压测和性能观察
```

- 单元测试关注业务逻辑。
- bufconn 可以在内存中跑 gRPC server，适合集成测试。
- streaming 测试要覆盖 EOF、取消和错误。
- 压测关注 QPS、P95/P99、错误率和资源占用。
- 性能优化先观察，再行动。

---

## 二、本阶段要覆盖的能力

1. 使用 bufconn 写 gRPC 集成测试。
2. 测试错误码、metadata、deadline。
3. 测试 Server Streaming、Client Streaming、双向流。
4. 使用 ghz 做压测。
5. 理解 P95、P99、QPS、错误率、goroutine、GC 等指标。

---

## 三、常见问题

- 只测成功路径：错误码、超时、鉴权更容易出问题。
- 依赖真实网络端口测试：容易不稳定，bufconn 更适合本地集成测试。
- 压测只看平均延迟：P95/P99 更能反映用户体验。
- 没有固定压测参数：结果无法复现。

- 能用 bufconn 跑 gRPC 集成测试。
- 能读懂压测结果。
- 能根据指标提出优化方向。

## 七、为什么 gRPC 服务必须测试

前面你已经能通过 grpcurl、curl、Go client 手动验证服务。但手动验证不能保证长期稳定。只要 proto、拦截器、错误码、metadata、streaming 任意一个地方改错，线上调用方就可能受到影响。

gRPC 服务测试至少要覆盖：

```text
业务成功路径
参数错误
资源不存在
metadata 鉴权
deadline 超时
客户端取消
Unary interceptor
Stream interceptor
Server Streaming EOF
Client Streaming CloseAndRecv
Bidirectional Streaming 读写顺序
```

第 9 阶段就是把这些行为变成自动化测试和压测手段。

## 八、本阶段测试分层

推荐分层：

```text
业务单元测试
  -> 不启动 gRPC，只测 service/repository 逻辑

bufconn gRPC 集成测试
  -> 内存 listener，真实 client 调用真实 gRPC server

端到端测试
  -> 启动真实进程和端口，少量覆盖主链路

压测
  -> 使用 ghz 观察 QPS、延迟、错误率
```

本阶段重点是 bufconn 和 ghz。

## 十一、运行命令

运行所有测试：

```powershell
go test ./...
```

运行某个测试：

```powershell
go test ./... -run TestGetUserOK -v
```

运行带 race 的测试：

```powershell
go test ./... -race
```

压测：

```powershell
ghz --insecure `
  --proto proto/user/v1/user.proto `
  --call user.v1.UserService.GetUser `
  -d '{"id":1}' `
  -c 20 -n 1000 `
  localhost:50051
```

## 十二、预期输出

测试通过：

```text
PASS
ok      example.com/grpc-learning    0.123s
```

某个错误码测试失败时可能看到：

```text
expected NotFound, got Unknown
```

这说明服务端可能返回了普通 error，而不是 `status.Error(codes.NotFound, ...)`。

压测输出会包含：

```text
Summary:
  Count: 1000
  Total: ...
  Slowest: ...
  Fastest: ...
  Average: ...
  Requests/sec: ...

Response time histogram:
Latency distribution:
  50 % in ...
  95 % in ...
  99 % in ...
Status code distribution:
  [OK] 1000 responses
```

## 十三、常见错误排查

### 1. 测试依赖真实端口

如果测试必须先手动启动 server，就不够稳定。优先使用 bufconn。

### 2. 只测成功路径

错误码、metadata、deadline、streaming 更容易出问题，必须覆盖。

### 3. 测试里使用 Sleep 等待

尽量使用 context、channel、EOF 控制测试，不要靠随机 sleep。

### 4. 压测只看平均值

平均值会掩盖长尾延迟。重点看 P95/P99。
