# 3. 字段编号、兼容性与版本演进

本节目标：理解字段编号的长期价值，学会避免破坏客户端兼容性。

Protobuf 的字段编号不是装饰，它参与二进制编码。字段名可以调整，注释可以调整，但字段编号一旦发布就要非常谨慎。

真实项目里，proto 最大的挑战不是第一次写出来，而是半年后如何不破坏老客户端地继续演进。

---

## 一、核心直觉

- 已发布字段不要改编号。
- 已发布字段不要随便改类型。
- 删除字段后应该使用 `reserved` 保留编号和名称。
- 新字段尽量追加新编号。
- 请求和响应分开设计，比复用一个大 message 更容易演进。
- v1、v2 版本包可以并存，避免一次性破坏升级。

---

## 二、动手步骤

1. 写一个 v1 的 `User`。
2. 模拟删除 `email` 字段。
3. 用 `reserved` 保留原编号和字段名。
4. 新增 `phone` 字段使用新编号。
5. 思考老客户端和新服务端如何共存。

---

## 三、参考代码或命令

```proto
message User {
  reserved 3;
  reserved "email";

  int64 id = 1;
  string name = 2;
  string phone = 4;
}
```

---

## 四、常见问题

- 删除字段后复用编号：老数据可能被新字段错误解析。
- 把 `int64 id = 1` 改成 `string id = 1`：这类变更风险很高。
- 一个接口返回所有字段：字段越多，演进成本越高，客户端也更容易依赖内部细节。

## 七、为什么兼容性这么重要

第一次写 proto 很简单，真正困难的是接口发布之后怎么改。

假设 v1 已经发布：

```proto
message User {
  int64 id = 1;
  string name = 2;
  string email = 3;
}
```

客户端已经根据这个结构生成代码并上线。如果你后来把字段 3 改成：

```proto
string phone = 3;
```

旧数据或旧客户端可能把原来的 email 当成 phone 理解。这就是破坏兼容性。

---

## 八、安全变更和危险变更

### 通常安全的变更

- 新增字段，使用新的编号。
- 新增 RPC 方法。
- 新增 enum 值。
- 给 message 增加注释。
- 给 response 增加可选信息。

示例：

```proto
message User {
  int64 id = 1;
  string name = 2;
  string email = 3;
  string phone = 4;
}
```

### 危险变更

- 修改已有字段编号。
- 修改已有字段类型。
- 删除字段后复用编号。
- 修改 service 或 rpc 名称。
- 修改 package 名称。
- 修改字段语义但不改名字。

---

## 九、reserved 的正确用法

如果删除字段，要保留编号和字段名：

```proto
message User {
  reserved 3;
  reserved "email";

  int64 id = 1;
  string name = 2;
  string phone = 4;
}
```

这样以后有人想复用 `3` 或 `email`，protoc 会直接报错，避免误用。

也可以保留范围：

```proto
reserved 10 to 19;
reserved "old_name", "old_email";
```

---

## 十、版本包策略

如果只是新增字段，可以继续在 v1 演进。

如果出现破坏性变化，建议开新版本：

```text
user.v1.UserService
user.v2.UserService
```

目录可以是：

```text
proto/user/v1/user.proto
proto/user/v2/user.proto
```

v1 和 v2 可以并存一段时间，让客户端逐步迁移。

---

## 十一、接口演进示例

第一版：

```proto
message GetUserResponse {
  int64 id = 1;
  string name = 2;
}
```

第二版，如果只是新增 email：

```proto
message GetUserResponse {
  int64 id = 1;
  string name = 2;
  string email = 3;
}
```

这是安全变更。

但如果想把 `id` 从 int64 改成 string，不应该直接改：

```proto
// 不推荐
string id = 1;
```

更稳妥的方式是新增字段：

```proto
int64 id = 1;
string uid = 3;
```

或者开 v2。

---

## 十二、团队协作建议

真实项目中，proto 变更应该走评审。评审时重点看：

- 是否修改已有字段编号。
- 是否修改已有字段类型。
- 是否删除字段但没有 reserved。
- 是否改变已有字段语义。
- 是否需要开 v2。
- 是否影响下游客户端。

后续可以用 buf 的 breaking change 检查自动发现问题。

## 十四、完成标准

你应该能判断：

```text
新增字段：通常安全
改字段编号：危险
改字段类型：危险
删除字段不 reserved：危险
删除字段并 reserved：推荐
重大语义变化：考虑 v2
```
