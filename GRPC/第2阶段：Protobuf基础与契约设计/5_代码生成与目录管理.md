# 5. 代码生成与目录管理

本节目标：学会使用 protoc 生成 Go 代码，并保持 proto 与生成代码目录清晰。

写完 proto 后，还不能直接在 Go 中使用。你需要通过 `protoc` 和插件生成 `.pb.go`、`_grpc.pb.go`。生成代码是 gRPC 开发的桥梁。

---

## 一、核心直觉

- `.pb.go` 主要包含 message、enum、序列化相关代码。
- `_grpc.pb.go` 主要包含 client、server、stream、注册函数。
- `paths=source_relative` 会让生成文件相对 proto 源路径输出。
- 团队项目建议用 Makefile 或 buf 固化生成命令。
- 生成代码不要手改，修改 proto 后重新生成。

---

## 二、动手步骤

1. 在 `proto/user/v1` 中写 `user.proto`。
2. 执行 `protoc` 命令生成 Go 文件。
3. 检查生成文件位置。
4. 执行 `go mod tidy`。
5. 尝试在 Go 文件中 import 生成包。

---

## 三、参考代码或命令

```bash
protoc --go_out=. --go_opt=paths=source_relative \
  --go-grpc_out=. --go-grpc_opt=paths=source_relative \
  proto/user/v1/user.proto
```

PowerShell：

```powershell
protoc --go_out=. --go_opt=paths=source_relative `
  --go-grpc_out=. --go-grpc_opt=paths=source_relative `
  proto/user/v1/user.proto
```

## 四、常见问题

- 生成到了意料之外的目录：通常是 `go_package`、命令路径或 `paths` 参数没理解。
- 生成代码报 import 错：先检查 module 名和 go_package。
- 每次手敲长命令：后面建议写进 Makefile。

## 七、生成命令逐项解释

完整命令：

```powershell
protoc --go_out=. --go_opt=paths=source_relative `
  --go-grpc_out=. --go-grpc_opt=paths=source_relative `
  proto/user/v1/user.proto
```

逐项解释：

- `protoc`：读取 proto 文件。
- `--go_out=.`：调用 `protoc-gen-go`，输出 message 代码。
- `--go_opt=paths=source_relative`：生成文件位置相对 proto 源文件。
- `--go-grpc_out=.`：调用 `protoc-gen-go-grpc`，输出 gRPC 代码。
- `--go-grpc_opt=paths=source_relative`：gRPC 生成文件也使用相对路径。
- `proto/user/v1/user.proto`：输入文件。

## 九、生成代码阅读顺序

### 先看 user.pb.go

搜索：

```text
type User struct
type GetUserRequest struct
type UserStatus int32
```

你会看到 proto 字段变成 Go struct 字段。

例如：

```proto
int64 id = 1;
string name = 2;
```

会生成类似：

```go
Id int64
Name string
```

### 再看 user_grpc.pb.go

搜索：

```text
UserServiceClient
UserServiceServer
RegisterUserServiceServer
```

这些就是第 3 阶段写 server/client 要用的东西。

## 十、使用 Makefile 固化命令

如果每次都手敲长命令，很容易出错。可以创建 `Makefile`：

```makefile
.PHONY: proto

proto:
	protoc --go_out=. --go_opt=paths=source_relative \
	  --go-grpc_out=. --go-grpc_opt=paths=source_relative \
	  proto/user/v1/user.proto
```

Windows 没有 make 也没关系，可以创建 `scripts/gen-proto.ps1`：

```powershell
protoc --go_out=. --go_opt=paths=source_relative `
  --go-grpc_out=. --go-grpc_opt=paths=source_relative `
  proto/user/v1/user.proto
```

## 十一、常见生成错误

### 错误 1：找不到插件

```text
protoc-gen-go-grpc: program not found
```

解决：

```powershell
go install google.golang.org/grpc/cmd/protoc-gen-go-grpc@latest
$env:Path += ";$(go env GOPATH)\bin"
```

### 错误 2：import 路径混乱

检查：

- go.mod 的 module。
- proto 的 go_package。
- 生成文件所在目录。

这三者必须能对得上。

### 错误 3：生成后 Go 编译失败

执行：

```powershell
go mod tidy
go test ./...
```

如果缺依赖，`go mod tidy` 会拉取。

## 十四、生成代码是否应该提交到 Git

这个问题没有唯一答案，但学习阶段建议提交生成代码，原因是：

- 你可以直接查看生成结果。
- 不需要每次打开项目都先安装完整生成工具。
- 对初学者更容易理解 server/client 接口来源。

真实团队中常见两种做法：

```text
做法 A：提交生成代码
优点：使用方便，构建简单
缺点：代码量增加，proto 改动会带来大量生成文件 diff

做法 B：不提交生成代码，CI 中生成
优点：仓库更清爽
缺点：构建环境必须严格准备生成工具
```

不管采用哪种方式，都要保证生成命令稳定可重复。

---

## 十五、生成目录混乱时怎么排查

如果生成结果不符合预期，按下面顺序查：

1. 当前所在目录是否是项目根目录。
2. `protoc` 输入文件路径是否正确。
3. `--go_out` 输出目录写到哪里。
4. 是否使用了 `paths=source_relative`。
5. `option go_package` 是否和 module 对应。
6. 生成后 import 路径是否能被 Go 找到。

你可以用下面命令确认当前目录：

```powershell
Get-Location
```

确认 module：

```powershell
Get-Content go.mod
```

确认生成文件：

```powershell
Get-ChildItem -Recurse -Filter '*pb.go'
```
