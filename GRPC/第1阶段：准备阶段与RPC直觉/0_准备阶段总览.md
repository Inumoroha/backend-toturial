# 0. 准备阶段总览

本节目标：建立 gRPC 学习环境，理解 RPC 的基本直觉，并知道后续每个阶段要解决什么问题。

gRPC 不应该从代码生成开始学，而应该先理解它解决的后端问题：两个服务之间如何像调用本地函数一样通信，同时还能保留超时、错误、鉴权、可观测性和契约治理。

本阶段不追求写很多代码，重点是把工具链装好，把学习工程建好，并形成一个判断：什么时候该用 gRPC，什么时候继续用 HTTP/JSON 就够了。

---

## 一、核心直觉

- RPC 的核心目标是“远程调用本地化”：调用方不直接拼 HTTP 路由和 JSON，而是调用生成出来的 client 方法。
- gRPC 的契约来自 `.proto` 文件，服务端和客户端都依赖这个契约生成代码。
- gRPC 常用于服务间通信，不是所有对外 API 都要直接暴露 gRPC。
- 学习路线要先小闭环，再工程化：先 Unary，再 Streaming，再错误、超时、metadata，最后进入生产能力和项目。

## 二、动手步骤

1. 确认你已经能运行 `go version`。

5. 准备 `grpcurl` 或至少知道它的作用，后面调试服务会频繁使用。

## 七、按天推进的学习安排

如果你是第一次系统学习 gRPC，不建议一上来就写项目。准备阶段最重要的是把几个概念和工具链摆正。可以按下面的节奏推进：

### 第 1 天：理解 RPC 是什么

先不要写代码，只做三件事：

1. 写下一个服务调用另一个服务的例子。
2. 用 HTTP/JSON 描述这个调用。
3. 用 RPC 的方式重新描述这个调用。

例如订单服务需要查用户：

```text
HTTP 思路：
订单服务拼 URL -> 发 GET /users/1 -> 解析 JSON -> 判断状态码

RPC 思路：
订单服务拿到 UserServiceClient -> 调用 GetUser(ctx, req) -> 得到强类型响应
```

你要形成的直觉是：gRPC 不是为了替代所有 HTTP API，而是为了让服务间调用更像“有契约的函数调用”。

### 第 2 天：准备工具链

确认这些命令都可用：

```bash
go version
protoc --version
protoc-gen-go --version
protoc-gen-go-grpc --version
```

如果其中任何一个命令不可用，不要继续后面的代码。gRPC 初学者最常见的卡点就是工具链不完整。

### 第 3 天：建立学习工程

你应该准备一个专门的学习工程，例如：

```text
grpc-learning/
  proto/
  cmd/
  internal/
  gen/
  README.md
```

后面每一阶段都在这个工程里推进。这样你最终会得到一个完整的学习项目，而不是散落的一堆 demo。

## 九、常见学习误区

### 误区 1：先背概念，再写代码

gRPC 的概念并不难，真正容易卡住的是：

- proto 写法不规范。
- go_package 路径不清楚。
- protoc 生成目录混乱。
- server 实现接口时报错。
- client 连接方式写错。
- context 超时没有设置。

所以准备阶段之后，就应该马上进入“写 proto -> 生成代码 -> 跑 server/client”的闭环。

### 误区 2：一开始就上微服务框架

很多 Go 微服务框架会封装 gRPC，但初学时不建议直接用框架。原因是框架会隐藏这些关键细节：

- gRPC server 如何创建。
- service 如何注册。
- client stub 如何生成。
- interceptor 如何执行。
- status code 如何返回。

你先用官方库写一遍，再看框架时会轻松很多。

### 误区 3：只关注“快不快”

gRPC 的价值不只是性能。对 Go 后端工程师来说，更重要的是：

- 强类型接口契约。
- 代码生成减少手写胶水代码。
- 统一错误模型。
- deadline 和 cancellation。
- streaming 能力。
- 服务治理和可观测性生态。
