# 3. 创建学习工程与目录结构

本节目标：建立一个清晰、可持续扩展的 gRPC 学习工程目录。

学习 gRPC 时，目录混乱会很快变成负担。proto 文件、生成代码、server、client 如果随手放，后续一加 streaming、interceptor、gateway 就会看不清。

本节先建立一个最小但工程化的目录结构，后面所有阶段都可以在此基础上扩展。

---

## 一、核心直觉

- `proto/` 保存人工编写的接口契约。
- `gen/` 保存生成代码，尽量不要手改。
- `cmd/server` 和 `cmd/client` 保存可运行入口。
- `internal/` 保存业务实现、配置、中间件等内部代码。
- Go module 路径会影响 `option go_package`，所以要先确定模块名。

---

## 二、动手步骤

1. 创建工程目录 `grpc-learning`。
2. 初始化 Go module。
3. 创建 `proto`、`gen`、`cmd`、`internal` 目录。
4. 约定 proto 包版本，例如 `user/v1`。
5. 在 README 中记录常用命令。

---

## 三、参考代码或命令

```bash
mkdir grpc-learning
cd grpc-learning
go mod init example.com/grpc-learning

mkdir -p proto/user/v1
mkdir -p gen/user/v1
mkdir -p cmd/server cmd/client
mkdir -p internal/user
```

Windows PowerShell：

```powershell
New-Item -ItemType Directory -Force proto/user/v1, gen/user/v1, cmd/server, cmd/client, internal/user
```

---

## 四、常见问题

- module 名随便改：`go_package` 和 import 路径会跟着受影响。
- 手改生成文件：生成代码应该由 proto 驱动，修改源头应该是 `.proto`。
- 所有代码都放在根目录：短期省事，长期难维护。

---

## 五、练习任务

- 创建工程目录并初始化 module。
- 写出你自己的目录结构说明。
- 思考：为什么生成代码和 proto 源文件要分开？

---

## 六、完成标准

- 工程可以执行 `go mod tidy`。
- 目录结构清晰，后续可以直接新增 proto 和服务实现。
- 你能解释每个目录的职责。


---

## 七、从零创建学习工程

建议你不要直接在零散目录里写 demo，而是建立一个长期使用的学习工程。假设工程名为 `grpc-learning`：

```powershell
mkdir grpc-learning
cd grpc-learning
go mod init example.com/grpc-learning
```

执行后会得到：

```text
go.mod
```

查看内容：

```powershell
Get-Content go.mod
```

预期类似：

```text
module example.com/grpc-learning

go 1.22
```

module 名会影响后续 `go_package` 和 import 路径，所以一开始就要定好。

---

## 八、创建目录结构

PowerShell：

```powershell
New-Item -ItemType Directory -Force proto/user/v1
New-Item -ItemType Directory -Force cmd/server
New-Item -ItemType Directory -Force cmd/client
New-Item -ItemType Directory -Force internal/user
New-Item -ItemType Directory -Force gen
```

目标结构：

```text
grpc-learning/
  go.mod
  proto/
    user/
      v1/
        user.proto
  cmd/
    server/
      main.go
    client/
      main.go
  internal/
    user/
  gen/
  README.md
```

说明：

- `proto/`：人工维护的接口契约。
- `cmd/server/`：服务端入口。
- `cmd/client/`：客户端入口。
- `internal/`：业务实现和内部包。
- `gen/`：可选，用于放生成代码。学习阶段也可以先把生成代码放在 proto 同级。

---

## 九、两种生成代码目录策略

### 策略 1：生成代码跟 proto 放一起

```text
proto/user/v1/user.proto
proto/user/v1/user.pb.go
proto/user/v1/user_grpc.pb.go
```

优点：路径直观，初学更容易理解。

缺点：proto 源文件和生成文件混在一起。

### 策略 2：生成代码放到 gen

```text
api/proto/user/v1/user.proto
gen/user/v1/user.pb.go
gen/user/v1/user_grpc.pb.go
```

优点：源文件和生成文件分离，项目更清爽。

缺点：protoc 参数和 go_package 要更仔细。

本教程前期建议使用策略 1，先降低复杂度。项目实战阶段再使用更工程化的 `api/proto` + `gen` 结构。

## 十、创建 README

在工程根目录创建 README：

```markdown
# grpc-learning

## 常用命令

生成代码：

```powershell
protoc --go_out=. --go_opt=paths=source_relative `
  --go-grpc_out=. --go-grpc_opt=paths=source_relative `
  proto/user/v1/user.proto
```

启动服务端：

```powershell
go run ./cmd/server
```

启动客户端：

```powershell
go run ./cmd/client
```
```

README 不是给别人看的装饰品，而是给未来的自己保存操作入口。

## 十一、常见问题

### module 名和 go_package 不一致

如果 module 是：

```text
example.com/grpc-learning
```

那 `go_package` 可以写：

```proto
option go_package = "example.com/grpc-learning/proto/user/v1;userv1";
```

这样生成代码后，Go import 路径才自然。

### 目录层级过深

初学不要过度设计：

```text
pkg/common/core/base/grpc/proto/generated/v1
```

这种目录会让你还没学 gRPC 就先迷路。先保持简单。

## 十二、完成标准

执行：

```powershell
Get-ChildItem -Recurse
```

你应该能看到清晰的工程骨架。并且你能解释：

- 为什么 proto 要单独放。
- 为什么 server 和 client 入口分开。
- 为什么 module 名会影响 go_package。
