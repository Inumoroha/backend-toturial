# 4. 服务发现、负载均衡与重试直觉

本节目标：理解 gRPC 客户端如何找到服务、选择实例，并处理临时失败。

单机 demo 里 client 直接连 `localhost:50051`。真实环境里，一个服务可能有多个实例，地址会变化，调用也可能短暂失败。这就需要服务发现、负载均衡和重试策略。

---

## 一、三个概念

- Name Resolver：把服务名解析成地址列表。
- Load Balancer：从地址列表中选择一个实例。
- Retry：遇到可恢复错误时重新尝试。

在 Kubernetes 中，最常见的服务名可能是：

```text
user-service.default.svc.cluster.local:50051
```

---

## 二、负载均衡直觉

gRPC 是长连接，不是每次请求都重新建连。客户端负载均衡通常发生在 gRPC client 内部。

常见策略：

- `pick_first`：选择第一个可用地址。
- `round_robin`：多个后端之间轮询。

---

## 三、重试策略

重试只适合可安全重试的请求。查询类接口通常比较安全，创建订单、扣库存这类接口必须谨慎。

```text
适合重试：GetUser、GetOrder、ListProducts
谨慎重试：CreateOrder、DeductStock、PayOrder
```

如果要重试创建类接口，通常需要幂等键。

---

## 四、Service Config 示例

```go
serviceConfig := `{
  "loadBalancingConfig": [{"round_robin":{}}],
  "methodConfig": [{
    "name": [{"service": "user.v1.UserService"}],
    "retryPolicy": {
      "MaxAttempts": 3,
      "InitialBackoff": "0.1s",
      "MaxBackoff": "1s",
      "BackoffMultiplier": 2.0,
      "RetryableStatusCodes": ["UNAVAILABLE"]
    }
  }]
}`
_ = serviceConfig
```

---

## 五、常见问题

- 对创建订单这类非幂等接口无脑重试：可能造成重复订单。
- 重试没有总超时：用户等待时间会被放大。
- 只在客户端做重试，不看服务端是否已处理成功：需要幂等键等设计。
- 以为负载均衡只在网关层做：gRPC client 侧也常做负载均衡。

## 九、服务发现的本质

服务发现要回答：

```text
我要调用 user-service，它当前有哪些可用地址？
```

本地 demo 写死：

```text
localhost:50051
```

真实环境可能是：

```text
DNS: user-service.default.svc.cluster.local:50051
注册中心: user-service -> [10.0.1.1:50051, 10.0.1.2:50051]
配置文件: USER_SERVICE_ADDR=...
Service Mesh: sidecar 接管服务发现
```

gRPC 通过 resolver 把名字解析为地址列表，再由 balancer 选择一个地址。

## 十、负载均衡策略

常见策略：

```text
pick_first：选择第一个可用地址
round_robin：多个地址轮询
```

在 gRPC Go 中可以通过 service config 配置：

```go
serviceConfig := `{
  "loadBalancingConfig": [
    {"round_robin": {}}
  ]
}`

conn, err := grpc.Dial(
    target,
    grpc.WithDefaultServiceConfig(serviceConfig),
    grpc.WithTransportCredentials(insecure.NewCredentials()),
)
```

注意：真实 resolver 是否返回多个地址，取决于你的环境。单纯写 `localhost:50051` 不会自动知道还有 `localhost:50052`。

## 十一、重试策略配置示例

查询类接口可以考虑重试：

```go
serviceConfig := `{
  "methodConfig": [{
    "name": [{"service": "user.v1.UserService", "method": "GetUser"}],
    "retryPolicy": {
      "MaxAttempts": 3,
      "InitialBackoff": "0.1s",
      "MaxBackoff": "1s",
      "BackoffMultiplier": 2.0,
      "RetryableStatusCodes": ["UNAVAILABLE"]
    }
  }]
}`
```

client：

```go
conn, err := grpc.Dial(
    "localhost:50051",
    grpc.WithTransportCredentials(insecure.NewCredentials()),
    grpc.WithDefaultServiceConfig(serviceConfig),
)
```

这表示 `GetUser` 遇到 `UNAVAILABLE` 时最多尝试 3 次。

---

## 十二、哪些接口不能随便重试

不能无脑重试：

```text
CreateOrder
PayOrder
DeductStock
CreatePayment
SendMessage
```

原因是它们可能产生副作用。请求第一次可能已经成功，只是响应丢了；客户端重试可能造成重复订单、重复扣款、重复扣库存。

如果必须重试，需要设计幂等键：

```proto
message CreateOrderRequest {
  int64 user_id = 1;
  int64 product_id = 2;
  int32 quantity = 3;
  string idempotency_key = 4;
}
```

服务端用 `idempotency_key` 去重，保证同一个请求重复提交不会产生多次副作用。

## 十三、运行命令和预期现象

启动 server 后，运行 client：

```powershell
go run ./cmd/client
```

如果 server 没启动，client 可能看到：

```text
code=Unavailable message=connection error
```

如果配置了重试，client 会在失败后尝试多次。你可以在日志里打印开始调用时间和结束时间，观察耗时变长。

预期日志示例：

```text
call GetUser start
call GetUser failed code=Unavailable cost=320ms
```

## 十四、常见错误排查

### 1. 配了 round_robin 但没有轮询

可能 resolver 只返回了一个地址。负载均衡需要多个后端地址。

### 2. 重试没有生效

检查：

- service config JSON 是否正确。
- method 名是否匹配。
- 错误 code 是否在 RetryableStatusCodes 中。
- RPC 是否允许重试。

### 3. 创建类接口重复执行

说明缺少幂等设计。不要只靠 client “不要重试”，服务端也要能去重。

### 4. retry 总耗时过长

重试必须和 deadline 一起设计。总 deadline 到期后，不应该无限重试。
