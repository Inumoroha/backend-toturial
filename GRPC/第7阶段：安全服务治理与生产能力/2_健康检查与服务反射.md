# 2. 健康检查与服务反射

本节目标：为 gRPC 服务增加标准健康检查和调试能力。

调度系统、负载均衡器、Kubernetes 都需要知道服务是否健康。开发人员也需要方便查看服务有哪些方法。Health Check 和 Reflection 正好解决这两个问题。

---

## 一、Health Check

gRPC 有标准健康检查协议，常见状态包括：

- `SERVING`：服务可用。
- `NOT_SERVING`：服务不可用。
- `UNKNOWN`：状态未知。

最小注册方式：

```go
healthServer := health.NewServer()
grpc_health_v1.RegisterHealthServer(s, healthServer)
healthServer.SetServingStatus("", grpc_health_v1.HealthCheckResponse_SERVING)
```

空字符串 `""` 表示整个 server 的健康状态。也可以为某个具体服务名设置状态。

---

## 二、Reflection

Reflection 允许工具在没有 proto 文件的情况下查询服务结构。

```go
reflection.Register(s)
```

开启后可以使用：

```bash
grpcurl -plaintext localhost:50051 list
grpcurl -plaintext localhost:50051 describe user.v1.UserService
grpcurl -plaintext localhost:50051 grpc.health.v1.Health/Check
```

---

## 三、生产环境注意点

- 开发环境建议开启 reflection，调试非常方便。
- 生产环境要看安全要求，不一定对外开启。
- 健康检查不应该只判断进程是否活着，还可以检查数据库、缓存、下游依赖。
- 服务关闭前可以先把状态改成 `NOT_SERVING`，再开始优雅关闭。

## 八、完整代码：注册 health check

```go
healthServer := health.NewServer()
grpc_health_v1.RegisterHealthServer(server, healthServer)
healthServer.SetServingStatus("", grpc_health_v1.HealthCheckResponse_SERVING)
```

需要 import：

```go
import (
    "google.golang.org/grpc/health"
    grpc_health_v1 "google.golang.org/grpc/health/grpc_health_v1"
)
```

空字符串 `""` 表示整个 server 的整体健康状态。

也可以设置某个具体服务：

```go
healthServer.SetServingStatus("user.v1.UserService", grpc_health_v1.HealthCheckResponse_SERVING)
```

---

## 九、完整代码：开启 reflection

```go
reflection.Register(server)
```

需要 import：

```go
import "google.golang.org/grpc/reflection"
```

reflection 的作用是让 grpcurl 这类工具在没有 proto 文件的情况下，也能查询服务列表和方法结构。

---

## 十、完整 server 示例

```go
func main() {
    lis, err := net.Listen("tcp", ":50051")
    if err != nil {
        log.Fatalf("listen: %v", err)
    }

    server := grpc.NewServer()
    userv1.RegisterUserServiceServer(server, newUserServer())

    healthServer := health.NewServer()
    grpc_health_v1.RegisterHealthServer(server, healthServer)
    healthServer.SetServingStatus("", grpc_health_v1.HealthCheckResponse_SERVING)

    reflection.Register(server)

    log.Println("gRPC server listening on :50051")
    if err := server.Serve(lis); err != nil {
        log.Fatalf("serve: %v", err)
    }
}
```

## 十二、模拟 NOT_SERVING

服务准备关闭前，可以先设置：

```go
healthServer.SetServingStatus("", grpc_health_v1.HealthCheckResponse_NOT_SERVING)
```

如果此时再调用 health check，预期：

```json
{
  "status": "NOT_SERVING"
}
```

在 Kubernetes 或负载均衡环境中，这可以帮助流量先摘除实例，再进行优雅关闭。

---

## 十三、生产环境如何判断健康

最简单的健康状态只是“进程活着”。更完整的健康检查可能包括：

- 数据库是否可连接。
- Redis 是否可连接。
- 必要下游服务是否可用。
- 配置是否加载成功。
- 服务是否正在关闭。

但是健康检查不要太重。每次 health check 都查一堆下游，可能反而给系统制造压力。

建议区分：

```text
liveness：进程是否活着
readiness：是否可以接流量
```

## 十四、常见错误排查

### 1. grpcurl list 看不到服务

检查是否开启：

```go
reflection.Register(server)
```

### 2. health check 方法找不到

检查是否注册：

```go
grpc_health_v1.RegisterHealthServer(server, healthServer)
```

### 3. 生产环境暴露 reflection

reflection 会暴露服务结构。开发环境建议开启，生产环境要根据安全要求决定是否开启。

### 4. health 永远 SERVING

如果服务已经准备关闭或依赖不可用，还继续 SERVING，流量会继续进来。关闭前应设置 NOT_SERVING。
