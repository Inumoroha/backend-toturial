# 3. 优雅关闭与信号处理

本节目标：实现服务退出时停止接收新请求，并等待已有请求尽量完成。

服务发布、重启、扩缩容时，进程会收到退出信号。如果直接退出，正在处理的请求会失败。优雅关闭可以降低发布对业务的影响。

---

## 一、核心方法

- `GracefulStop()`：停止接收新连接，等待已有 RPC 完成。
- `Stop()`：立即停止 server，适合超时兜底。
- `signal.NotifyContext`：监听系统信号。

---

## 二、基本写法

```go
ctx, stop := signal.NotifyContext(context.Background(), os.Interrupt, syscall.SIGTERM)
defer stop()

go func() {
    if err := s.Serve(lis); err != nil {
        log.Printf("serve error: %v", err)
    }
}()

<-ctx.Done()
log.Println("shutting down grpc server")
healthServer.SetServingStatus("", grpc_health_v1.HealthCheckResponse_NOT_SERVING)
s.GracefulStop()
```

---

## 三、增加超时兜底

有些 stream 可能长期不结束。可以给优雅关闭设置最大等待时间，超时后调用 `Stop()`。

```go
done := make(chan struct{})
go func() {
    s.GracefulStop()
    close(done)
}()

select {
case <-done:
    log.Println("grpc server stopped gracefully")
case <-time.After(10 * time.Second):
    log.Println("force stop grpc server")
    s.Stop()
}
```

---

## 四、常见问题

- 主 goroutine 直接退出：server goroutine 还没来得及收尾。
- 没有超时兜底：如果长连接一直不结束，退出可能被卡住。
- 不更新 health：流量可能继续打到即将退出的实例。
- 发布平台没有给足 termination grace period：应用想优雅也来不及。

## 七、完整操作步骤

本节目标：让 gRPC server 收到 Ctrl+C 或 SIGTERM 时，先停止接收新流量，再等待已有请求完成，最后退出。

操作步骤：

1. 使用 `signal.NotifyContext` 监听退出信号。
2. server 在 goroutine 中启动。
3. 主 goroutine 等待信号。
4. 收到信号后把 health 设置为 `NOT_SERVING`。
5. 调用 `GracefulStop()`。
6. 设置超时兜底，超时后调用 `Stop()`。
7. 写一个慢请求验证关闭行为。

---

## 八、完整代码：waitForShutdown

```go
func waitForShutdown(server *grpc.Server, healthServer *health.Server) {
    ctx, stop := signal.NotifyContext(context.Background(), os.Interrupt, syscall.SIGTERM)
    defer stop()

    <-ctx.Done()
    log.Println("shutdown signal received")

    healthServer.SetServingStatus("", grpc_health_v1.HealthCheckResponse_NOT_SERVING)

    done := make(chan struct{})
    go func() {
        server.GracefulStop()
        close(done)
    }()

    select {
    case <-done:
        log.Println("grpc server stopped gracefully")
    case <-time.After(10 * time.Second):
        log.Println("graceful stop timeout, force stop")
        server.Stop()
    }
}
```

需要 import：

```go
import (
    "context"
    "log"
    "os"
    "os/signal"
    "syscall"
    "time"

    "google.golang.org/grpc"
    "google.golang.org/grpc/health"
    grpc_health_v1 "google.golang.org/grpc/health/grpc_health_v1"
)
```

## 九、完整 server main

```go
func main() {
    lis, err := net.Listen("tcp", ":50051")
    if err != nil {
        log.Fatalf("listen: %v", err)
    }

    server := grpc.NewServer()
    userv1.RegisterUserServiceServer(server, newUserServer())

    healthServer := health.NewServer()
    grpc_health_v1.RegisterHealthServer(server, healthServer)
    healthServer.SetServingStatus("", grpc_health_v1.HealthCheckResponse_SERVING)
    reflection.Register(server)

    go func() {
        log.Println("gRPC server listening on :50051")
        if err := server.Serve(lis); err != nil {
            log.Printf("serve stopped: %v", err)
        }
    }()

    waitForShutdown(server, healthServer)
}
```

## 十二、GracefulStop 和 Stop 的区别

`GracefulStop()`：

```text
停止接收新连接/新请求
等待已有 RPC 完成
再关闭 server
```

`Stop()`：

```text
立即关闭 server
正在处理的 RPC 会被中断
```

生产环境通常先 `GracefulStop`，超时后再 `Stop` 兜底。

---

## 十三、常见错误排查

### 1. Ctrl+C 后没有任何日志

检查是否调用了 `waitForShutdown`，以及 server 是否在 goroutine 中运行。

### 2. 进程一直不退出

可能有长连接或 streaming 一直不结束。需要超时后 `server.Stop()` 兜底。

### 3. 关闭前没有 NOT_SERVING

负载均衡器可能继续把流量打到即将关闭的实例。关闭前应先更新 health 状态。

### 4. Windows 信号差异

本地 PowerShell 中 Ctrl+C 对应 `os.Interrupt`。Linux/Kubernetes 中常见的是 `SIGTERM`。
