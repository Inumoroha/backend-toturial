# 1. 需求分析与接口边界

本节目标：在写代码前明确系统功能、服务职责和接口边界。

项目实战最容易犯的错误是直接写代码。微服务项目更应该先明确服务职责：每个服务管理什么数据，暴露什么能力，哪些调用是同步 gRPC，哪些可以以后改成异步消息。

---

## 一、核心用户故事

```text
用户选择商品和数量。
系统检查用户是否存在。
系统检查商品是否存在。
系统扣减库存。
系统创建订单。
用户可以查询订单。
用户可以监听订单状态变化。
```

---

## 二、服务边界

### user-service

只负责用户基础信息：

- `CreateUser`
- `GetUser`

### product-service

只负责商品和库存：

- `GetProduct`
- `DeductStock`

### order-service

负责订单主流程：

- `CreateOrder`
- `GetOrder`
- `WatchOrderStatus`

### notification-service

负责通知推送：

- `SubscribeOrderNotifications`

---

## 三、创建订单调用链路

```text
HTTP POST /v1/orders
-> api-gateway
-> order-service.CreateOrder
-> user-service.GetUser
-> product-service.GetProduct
-> product-service.DeductStock
-> order-service 保存订单
-> 返回订单结果
```

---

## 四、超时预算

假设外部请求总超时为 2 秒，可以粗略设计：

```text
api-gateway -> order-service：1800ms
order-service -> user-service：300ms
order-service -> product-service.GetProduct：300ms
order-service -> product-service.DeductStock：500ms
order-service 本地处理：200ms
预留：500ms
```

真实项目中要根据实际 P95/P99 调整。

---

## 五、常见问题

- gateway 直接访问多个服务并拼业务：会让入口层变重。
- order-service 管理商品库存：职责越界。
- 一次创建订单跨太多服务且没有超时预算：故障时容易级联。
- 不考虑幂等：客户端重试可能造成重复订单。

---

3. 为创建订单链路设置总超时和每个下游超时。
4. 画出服务调用图。

- 服务职责边界清楚。
- 接口清单完整。
- 创建订单链路可解释。

## 八、完整操作步骤：先写需求边界，再写代码

这一篇不要打开编辑器写 Go 代码，而是先写项目说明。微服务项目里，边界不清会导致后续代码互相调用、错误码混乱、测试无法隔离。建议在项目根目录创建：

```powershell
New-Item -ItemType File .\README.md
New-Item -ItemType File .\docs\requirements.md -Force
```

在 `docs/requirements.md` 中先写四块内容：

1. 本项目要做什么。
2. 本项目暂时不做什么。
3. 每个服务负责什么。
4. 创建订单的主链路是什么。

推荐写法：

```text
本项目实现一个教学版订单微服务系统。

包含能力：
- 用户创建与查询。
- 商品查询与库存扣减。
- 订单创建、查询与状态流。
- HTTP Gateway 对外暴露创建订单和查询订单接口。

暂不包含：
- 真实支付。
- 真实数据库事务。
- 分布式事务最终一致性。
- 用户登录注册的完整安全体系。
- 生产级服务注册发现。
```

这段“不做什么”非常重要。它能避免你在学习 gRPC 时突然陷入支付、库存、消息队列、分布式事务等更大的系统问题。

---

## 九、完整代码：用结构化文档固定服务边界

可以把服务边界写成表格，后续 proto 和代码都按这个表实现：

```markdown
| 服务 | 职责 | 对外接口 | 不负责 |
| --- | --- | --- | --- |
| user-service | 用户资料 | CreateUser, GetUser | 订单、库存、支付 |
| product-service | 商品资料和库存 | GetProduct, DeductStock | 用户、订单 |
| order-service | 订单创建和查询 | CreateOrder, GetOrder, WatchOrderStatus | 直接管理用户和商品库存 |
| gateway | HTTP/JSON 入口 | HTTP routes | 核心业务判断 |
```

再写创建订单链路：

```text
1. Gateway 接收 HTTP 创建订单请求。
2. Gateway 调用 order-service CreateOrder。
3. order-service 调用 user-service GetUser，确认用户存在。
4. order-service 调用 product-service GetProduct，确认商品存在。
5. order-service 调用 product-service DeductStock，扣减库存。
6. order-service 创建订单并返回订单 ID。
7. 客户端可通过 WatchOrderStatus 监听订单状态变化。
```

接口边界也要写错误归属：

```text
用户不存在：user-service 返回 NotFound，order-service 转换为创建订单失败。
商品不存在：product-service 返回 NotFound，order-service 转换为创建订单失败。
库存不足：product-service 返回 FailedPrecondition。
请求参数错误：接收请求的服务返回 InvalidArgument。
下游超时：order-service 返回 DeadlineExceeded 或 Unavailable。
```

## 十、运行命令与预期输出

需求分析本身不需要运行服务，但需要做文档检查。你可以用下面的命令确认项目说明文件已经存在：

```powershell
Get-ChildItem .\README.md, .\docs\requirements.md
```

预期输出：

```text
Mode                 LastWriteTime         Length Name
----                 -------------         ------ ----
-a----               ...                   ...    README.md
-a----               ...                   ...    requirements.md
```

如果你使用 VS Code，可以直接打开需求文档：

```powershell
code .\docs\requirements.md
```

项目推进过程中，每新增一个接口，都要回到这份文档检查：它属于哪个服务？错误码应该由谁返回？它会不会让某个服务越界？

---

## 十一、常见错误排查

- 把 Gateway 写成业务服务：Gateway 只做协议转换、基础鉴权、请求日志和 metadata 传递，不做库存判断。
- OrderService 直接读 product-service 的 map：这破坏服务边界，项目看似能跑，实则没有练到 gRPC 跨服务调用。
- ProductService 创建订单：商品服务只管理商品和库存，不应该知道订单完整生命周期。
- 所有错误都返回 Internal：调用方无法根据错误类型做处理，应该区分 `InvalidArgument`、`NotFound`、`FailedPrecondition`、`Unavailable`。
- 需求文档只写功能，不写不做什么：学习项目很容易越做越大，最后主线失焦。

---

## 十二、教程闭环检查

本篇的完整操作步骤是：创建需求文档、写项目范围、写服务职责、写调用链路、写错误归属。完整代码在这里体现为结构化 Markdown 表格和链路文本，它们会直接指导后续 proto 和 Go 实现。运行命令包含创建文件、检查文件和打开文档。预期输出是需求文档存在且内容完整。常见错误排查覆盖 Gateway 越界、服务直接读对方数据、错误码滥用和范围失控。练习任务是独立画出创建订单调用图，并为每个失败场景标注错误码。完成标准是：不用看代码，也能说清楚每个服务负责什么、不负责什么、谁调用谁。
