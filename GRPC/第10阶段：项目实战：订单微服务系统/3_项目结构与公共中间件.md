# 3. 项目结构与公共中间件

本节目标：搭建项目骨架，并沉淀公共拦截器、错误转换和配置。

项目越往后，公共能力越重要。日志、鉴权、trace、错误转换不能散落在每个服务里，应该放在公共 middleware 或 pkg 中。

---

## 一、推荐结构

```text
internal/middleware/
  logging.go
  auth.go
  recovery.go
  trace.go
internal/config/
  config.go
internal/xerror/
  grpc.go
```

目录职责：

- `cmd/*`：服务入口。
- `internal/*`：业务实现。
- `internal/middleware`：拦截器。
- `internal/config`：配置读取。
- `internal/xerror`：错误转换。

---

## 二、公共拦截器注册

```go
s := grpc.NewServer(
    grpc.ChainUnaryInterceptor(
        middleware.Recovery(),
        middleware.Logging(),
        middleware.Auth(),
    ),
    grpc.ChainStreamInterceptor(
        middleware.StreamLogging(),
    ),
)
```

## 三、配置建议

学习阶段可以先使用环境变量：

```text
USER_SERVICE_ADDR=localhost:50051
PRODUCT_SERVICE_ADDR=localhost:50052
ORDER_SERVICE_ADDR=localhost:50053
AUTH_TOKEN=dev-token
```

真实项目可以使用配置文件、配置中心或 Kubernetes ConfigMap/Secret。

## 四、常见问题

- 公共代码放到外部 pkg 过早开放：先 internal，稳定后再考虑 pkg。
- 每个服务复制一份拦截器：后续修 bug 会漏。
- 配置写死在代码里：端口、下游地址、token 应该可配置。
- main 函数越来越大：服务初始化要适当拆分。

## 七、完整操作步骤：沉淀公共能力

当项目有多个 gRPC 服务时，日志、panic 恢复、鉴权、trace id、超时日志不能每个服务复制一份。建议先创建公共目录：

```powershell
mkdir .\internal\middleware -Force
mkdir .\internal\server -Force
mkdir .\internal\config -Force
```

公共能力的落点建议如下：

```text
internal/config/config.go       配置结构与默认值
internal/middleware/logging.go  请求日志
internal/middleware/recovery.go panic 恢复
internal/middleware/auth.go     简单 token 鉴权
internal/server/grpc.go         统一创建 grpc.Server
```

这一步的目标不是做复杂框架，而是保证每个服务都以同样的方式启动、打日志和处理 panic。

## 八、完整代码：统一 gRPC Server 工厂

`internal/middleware/logging.go`：

```go
package middleware

import (
	"context"
	"log"
	"time"

	"google.golang.org/grpc"
	"google.golang.org/grpc/status"
)

func UnaryLogging() grpc.UnaryServerInterceptor {
	return func(ctx context.Context, req any, info *grpc.UnaryServerInfo, handler grpc.UnaryHandler) (any, error) {
		start := time.Now()
		resp, err := handler(ctx, req)
		code := status.Code(err)
		log.Printf("grpc method=%s code=%s duration=%s", info.FullMethod, code, time.Since(start))
		return resp, err
	}
}
```

`internal/middleware/recovery.go`：

```go
package middleware

import (
	"context"
	"log"

	"google.golang.org/grpc"
	"google.golang.org/grpc/codes"
	"google.golang.org/grpc/status"
)

func UnaryRecovery() grpc.UnaryServerInterceptor {
	return func(ctx context.Context, req any, info *grpc.UnaryServerInfo, handler grpc.UnaryHandler) (resp any, err error) {
		defer func() {
			if r := recover(); r != nil {
				log.Printf("panic recovered method=%s panic=%v", info.FullMethod, r)
				err = status.Error(codes.Internal, "internal server error")
			}
		}()
		return handler(ctx, req)
	}
}
```

`internal/server/grpc.go`：

```go
package server

import (
	"example.com/grpc-shop/internal/middleware"
	"google.golang.org/grpc"
)

func NewGRPCServer() *grpc.Server {
	return grpc.NewServer(
		grpc.ChainUnaryInterceptor(
			middleware.UnaryRecovery(),
			middleware.UnaryLogging(),
		),
	)
}
```

在每个服务的 `main.go` 中统一使用：

```go
grpcServer := server.NewGRPCServer()
```

这样后续增加 auth、trace、metrics 时，只需要改一个地方。

## 九、运行命令与预期输出

格式化：

```powershell
gofmt -w .\internal\middleware .\internal\server
```

测试编译：

```powershell
go test ./...
```

启动任意服务后，调用一次接口，预期服务端日志类似：

```text
grpc method=/user.v1.UserService/GetUser code=OK duration=1.2ms
```

如果接口返回错误，也应该看到对应 code：

```text
grpc method=/user.v1.UserService/GetUser code=NotFound duration=800us
```

这说明业务错误已经进入统一日志链路。

## 十、常见错误排查

- 拦截器顺序错误：一般 recovery 放最外层，确保后续拦截器和 handler panic 都能被恢复。
- 在拦截器里吞掉错误：日志可以记录错误，但不要把业务错误随意改成 OK。
- 每个服务自己 new grpc.Server：后续公共能力会不一致，排查问题很痛苦。
- logging 打印完整请求体：学习阶段可以少量打印，真实项目要避免泄漏敏感信息。
- 配置散落在 main.go：端口、token、下游地址应集中到 config。

## 十一、教程闭环检查

本篇的完整操作步骤是：创建公共目录、实现 logging/recovery、实现统一 server 工厂，并让各服务使用它。完整代码包含两个 unary interceptor 和 `NewGRPCServer`。运行命令包含 `gofmt`、`go test` 和服务调用后的日志观察。预期输出是每次 RPC 都有 method、code、duration 日志。常见错误排查覆盖拦截器顺序、错误处理、重复创建 server、敏感日志和配置散落。练习任务是再加入一个读取 metadata 中 `x-trace-id` 的拦截器。完成标准是：新增服务时不用复制中间件代码，只需要调用统一工厂。

补充判断原则：公共中间件不是为了“看起来架构更复杂”，而是为了让多个服务在日志格式、错误恢复、鉴权入口和追踪字段上保持一致。只要以后新增一个服务时能少复制一段样板代码，这个抽象就是值得的；如果抽象让启动流程更难懂，就应该回到更直接的函数组织方式。
