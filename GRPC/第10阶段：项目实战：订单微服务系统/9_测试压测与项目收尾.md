# 9. 测试、压测与项目收尾

本节目标：为订单微服务项目补齐测试、调试命令、压测和 README。

一个项目是否完整，不只看功能是否能跑，还要看别人能不能启动、测试、调试和理解。项目收尾阶段要补齐测试、Makefile、README、常用命令和检查清单。

## 五、项目限制说明

学习项目可以先明确这些限制：

- 使用内存存储，重启后数据丢失。
- 没有真实数据库事务。
- 扣库存和创建订单之间没有分布式事务。
- 鉴权使用固定 token。
- 没有真正接入 OpenTelemetry 后端。
- Gateway 只做基础 HTTP 转发。

这些限制不是缺点，而是学习项目的边界。后续进阶时再逐步补数据库、缓存、消息队列、Kubernetes、Service Mesh。

## 六、后续优化方向

- 使用 PostgreSQL 保存用户、商品、订单。
- 使用 Redis 做库存缓存或热点数据缓存。
- 使用 Kafka/RabbitMQ 发送订单事件。
- 使用 Saga 或 Outbox 模式处理跨服务一致性。
- 使用 OpenTelemetry + Prometheus + Grafana 做观测。
- 使用 Docker Compose 启动完整环境。
- 部署到 Kubernetes，并配置 readiness/liveness。

## 八、完整操作步骤：给项目做最终验收

项目收尾不是简单写一句“完成了”。你要确保别人拿到这个项目后，可以按文档启动、测试、调试和复现问题。

建议按下面顺序验收：

1. 清理 README，写清楚环境要求。
2. 写清楚 proto 生成命令。
3. 写清楚四个服务的启动顺序。
4. 写 grpcurl 调试命令。
5. 写 curl 调试命令。
6. 跑 `go test ./...`。
7. 跑一次 ghz 压测。
8. 记录项目限制和后续优化方向。

README 至少包含：

````markdown
# grpc-shop

教学版订单微服务系统，用于练习 Go gRPC、Protobuf、Streaming、Gateway、测试和压测。

## 启动顺序

1. user-service
2. product-service
3. order-service
4. gateway

## 常用命令

生成代码：

```powershell
buf generate
```

运行测试：

```powershell
go test ./...
```
````

README 不需要写成论文，但必须让项目可复现。

## 九、完整代码：最小测试样例

以 UserService 为例，可以写一个表驱动测试：

```go
package user_test

import (
	"context"
	"testing"

	userv1 "example.com/grpc-shop/gen/user/v1"
	"example.com/grpc-shop/internal/app/user"
	"google.golang.org/grpc/codes"
	"google.golang.org/grpc/status"
)

func TestGetUser(t *testing.T) {
	svc := user.NewService()

	tests := []struct {
		name string
		id   int64
		code codes.Code
	}{
		{name: "ok", id: 1, code: codes.OK},
		{name: "invalid", id: 0, code: codes.InvalidArgument},
		{name: "not found", id: 999, code: codes.NotFound},
	}

	for _, tt := range tests {
		t.Run(tt.name, func(t *testing.T) {
			_, err := svc.GetUser(context.Background(), &userv1.GetUserRequest{Id: tt.id})
			if got := status.Code(err); got != tt.code {
				t.Fatalf("code = %s, want %s", got, tt.code)
			}
		})
	}
}
```

ProductService 至少补库存不足和并发扣减测试。OrderService 至少补创建订单成功、用户不存在、商品不存在、库存不足四类测试。

## 十、运行命令与预期输出

运行全部测试：

```powershell
go test ./...
```

预期输出：

```text
ok   example.com/grpc-shop/internal/app/user      0.123s
ok   example.com/grpc-shop/internal/app/product   0.145s
ok   example.com/grpc-shop/internal/app/order     0.210s
```

启动服务后，用 grpcurl 做内部调试：

```powershell
grpcurl -plaintext -d '{\"userId\":1,\"productId\":1,\"quantity\":1,\"requestId\":\"final-001\"}' localhost:50053 order.v1.OrderService/CreateOrder
```

用 curl 做外部调试：

```powershell
curl.exe -X POST http://localhost:8080/v1/orders -H "Content-Type: application/json" -d "{\"userId\":1,\"productId\":1,\"quantity\":1,\"requestId\":\"final-http-001\"}"
```

用 ghz 压测核心接口：

```powershell
ghz --insecure --proto .\proto\order\v1\order.proto --call order.v1.OrderService.GetOrder -d '{\"id\":1}' -c 10 -n 500 localhost:50053
```

预期 ghz 输出包含：

```text
Summary:
  Count: 500
  Requests/sec:

Status code distribution:
  [OK]
```

如果压测 GetOrder 前没有创建订单，可能会得到 NotFound。压测前要准备稳定数据。

## 十一、常见错误排查

- README 命令跑不通：说明文档不是可执行文档，要按新机器视角重测一遍。
- 测试依赖服务端口：单元测试尽量直接实例化 service，集成测试再使用 bufconn 或真实端口。
- ghz 全是 NotFound：压测数据没有提前准备，或者请求体 ID 不存在。
- curl 成功但 grpcurl 失败：可能 Gateway 走的是旧服务，或者直接 gRPC 地址写错。
- go test 有数据竞争：对内存 map 的读写没有加锁，用 `go test -race ./...` 检查。
- 项目没有限制说明：教学项目必须写清楚“未实现真实数据库、幂等、分布式事务、鉴权生产化”。

## 十三、教程闭环检查

本篇的完整操作步骤是：整理 README、生成代码、运行测试、启动服务、grpcurl 调试、curl 调试、ghz 压测、记录限制。完整代码包含 UserService 表驱动测试样例，并要求补齐 ProductService 和 OrderService 测试。运行命令包含 `buf generate`、`go test`、`go test -race`、`grpcurl`、`curl.exe` 和 `ghz`。预期输出覆盖测试通过、订单创建响应和 ghz Summary。常见错误排查覆盖 README 不可复现、测试依赖端口、压测数据、Gateway 地址、数据竞争和项目限制。练习任务是按收尾清单逐项打勾，并把失败项写成 issue。完成标准是：项目不仅能跑，而且能被别人按文档复现、测试和调试。
