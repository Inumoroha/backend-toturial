# 0. 订单微服务项目总览

本阶段目标：把前面学过的 gRPC 能力串起来，完成一个接近真实后端场景的项目。

前面阶段学的是单点能力：proto、server、client、streaming、错误、metadata、interceptor、gateway、测试。项目阶段要把这些能力放到同一个系统里。

本项目是一个小型订单微服务系统，目标不是做复杂业务，而是练习服务拆分、契约设计、跨服务调用和工程化能力。

---

## 一、项目服务拆分

```text
api-gateway
  -> user-service
  -> product-service
  -> order-service
  -> notification-service
```

服务职责：

- user-service：用户注册和查询。
- product-service：商品查询和库存扣减。
- order-service：创建订单、查询订单、订单状态流。
- notification-service：订单通知推送。
- api-gateway：HTTP/JSON 对外，内部调用 gRPC。

---

## 二、项目练习重点

- 多个 proto 包的组织方式。
- 多个 gRPC server 的启动方式。
- gRPC client 连接复用。
- 跨服务调用的 timeout 和错误传播。
- interceptor 的复用。
- health check 和 reflection。
- gRPC-Gateway 对外暴露 HTTP API。
- bufconn 测试和 ghz 压测。

## 四、最小闭环

不要一开始就追求完整功能。先跑通：


## 六、完整操作步骤：把项目从空目录搭起来

项目实战阶段不要先追求“微服务数量很多”，而要先追求链路闭环。建议按下面顺序推进：

1. 创建项目目录和 Go module。
2. 创建 `proto` 目录，先定义 user、product、order 三个核心服务。
3. 生成 Go 代码和 gRPC 代码。
4. 启动 user-service，确认用户查询可用。
5. 启动 product-service，确认商品查询和扣库存可用。
6. 启动 order-service，让它通过 gRPC client 调用 user-service 和 product-service。
7. 补充 API Gateway，把 HTTP/JSON 请求转成内部 gRPC 调用。
8. 补充测试、压测、README 和项目复盘。

推荐从项目根目录执行：

```powershell
mkdir grpc-shop
cd grpc-shop
go mod init example.com/grpc-shop
mkdir proto,gen,cmd,internal
mkdir proto\user\v1,proto\product\v1,proto\order\v1
mkdir cmd\user-service,cmd\product-service,cmd\order-service,cmd\gateway
mkdir internal\middleware,internal\config,internal\app
```

项目初期先不要引入数据库。用户、商品、订单都可以先使用内存 map。这样做不是因为真实项目应该用内存保存数据，而是为了让你把注意力放在 gRPC 服务拆分、proto 契约、跨服务调用、错误码和超时控制上。

## 七、完整代码：最小可运行项目骨架

项目根目录建议先形成下面的结构：

```text
grpc-shop/
  go.mod
  buf.yaml
  buf.gen.yaml
  proto/
    user/v1/user.proto
    product/v1/product.proto
    order/v1/order.proto
  gen/
    user/v1/
    product/v1/
    order/v1/
  cmd/
    user-service/main.go
    product-service/main.go
    order-service/main.go
    gateway/main.go
  internal/
    app/
      user/
      product/
      order/
    config/
      config.go
    middleware/
      logging.go
      recovery.go
      auth.go
```

先准备一个统一配置结构：

```go
package config

type Config struct {
	UserAddr    string
	ProductAddr string
	OrderAddr   string
	GatewayAddr string
}

func Default() Config {
	return Config{
		UserAddr:    ":50051",
		ProductAddr: ":50052",
		OrderAddr:   ":50053",
		GatewayAddr: ":8080",
	}
}
```

再准备一个服务启动约定：每个 `cmd/*/main.go` 只负责装配，不写业务逻辑。业务逻辑放进 `internal/app/*`。这样后面写测试时可以直接实例化 service，而不需要真的监听端口。

```go
package main

import (
	"log"
	"net"

	"example.com/grpc-shop/internal/config"
	"google.golang.org/grpc"
)

func main() {
	cfg := config.Default()

	lis, err := net.Listen("tcp", cfg.UserAddr)
	if err != nil {
		log.Fatalf("listen user-service: %v", err)
	}

	server := grpc.NewServer()
	log.Printf("user-service listening on %s", cfg.UserAddr)

	if err := server.Serve(lis); err != nil {
		log.Fatalf("serve user-service: %v", err)
	}
}
```

上面代码还没有注册具体服务，本篇只要求搭好骨架。后续小节会分别补齐 `RegisterUserServiceServer`、`RegisterProductServiceServer` 和 `RegisterOrderServiceServer`。

## 八、运行命令与预期输出

安装依赖：

```powershell
go get google.golang.org/grpc
go get google.golang.org/protobuf
go mod tidy
```

格式化并检查：

```powershell
gofmt -w .\cmd .\internal
go test ./...
```

预期输出：

```text
?    example.com/grpc-shop/cmd/user-service      [no test files]
?    example.com/grpc-shop/internal/config       [no test files]
```

当 user-service 注册完成后，启动命令应该是：

```powershell
go run .\cmd\user-service
```

预期输出：

```text
user-service listening on :50051
```

项目总览阶段的重点是“目录能解释，命令能运行，后续每个服务都有明确位置”。不要把所有代码都塞到一个 `main.go` 里，否则项目越写越难维护。

## 九、常见错误排查

- `go: cannot find main module`：没有在项目根目录执行命令，或者忘记 `go mod init`。
- `package example.com/grpc-shop/internal/... is not in std`：module 名和 import 路径不一致，检查 `go.mod`。
- `listen tcp :50051: bind: Only one usage of each socket address`：端口已经被占用，关闭旧进程或换端口。
- 三个服务互相 import 业务实现：说明边界不清晰。跨服务依赖应该通过生成的 gRPC client，而不是直接引用对方 internal 包。
- 目录很多但没有 README：项目实战必须写清楚启动顺序，否则过几天自己也会忘。

## 十、教程闭环检查

本篇的完整操作步骤是：创建项目、初始化 module、规划目录、安装依赖、运行测试、启动服务。完整代码包含配置结构、目录骨架和 main 函数骨架。运行命令包含 `go mod init`、`go get`、`go mod tidy`、`gofmt`、`go test`、`go run`。预期输出覆盖测试和服务启动日志。常见错误排查覆盖 module、端口、import 和服务边界。练习任务是把目录真实创建出来，并写一个 README 记录启动顺序。完成标准是：项目骨架清晰，任何一个新读者能从 README 里知道每个服务放在哪里、先启动什么、后启动什么。
