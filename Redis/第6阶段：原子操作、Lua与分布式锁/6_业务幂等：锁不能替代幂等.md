# 6. 业务幂等：锁不能替代幂等

分布式锁常被用来防止重复执行。

但在真实系统里，锁不能替代幂等。锁可能获取失败、过期、续期失败、网络超时，也可能因为服务重试让同一业务再次执行。

这一节讲业务幂等。

学完这一节后，你应该能够：

- 理解幂等的含义。
- 区分互斥和幂等。
- 知道为什么分布式锁不能替代幂等。
- 使用数据库唯一约束、状态机和请求号实现幂等。
- 为 Redis 锁保护的业务增加第二道防线。

---

## 一、什么是幂等

幂等的意思是：

```text
同一个业务请求执行一次和执行多次，最终效果一致。
```

例如创建订单：

```text
用户点击一次：创建订单 1001
用户重复点击：仍然只有订单 1001
```

不能变成：

```text
重复点击 3 次，创建 3 个订单
```

幂等关注的是业务结果。

锁关注的是同一时间是否允许多个执行者进入临界区。

---

## 二、互斥和幂等的区别

互斥：

```text
同一时间只有一个请求执行。
```

幂等：

```text
同一个请求重复执行，结果仍然正确。
```

分布式锁解决互斥问题。

幂等解决重复执行问题。

两者经常一起出现，但不是一回事。

---

## 三、为什么锁不能替代幂等

假设创建订单时加了 Redis 锁：

```text
lock:create_order:user:1001:sku:2001
```

仍然可能出现：

```text
1. 请求 A 获取锁成功。
2. A 创建订单成功。
3. A 返回给客户端时网络超时。
4. 客户端以为失败，发起重试。
5. 请求 B 再次获取锁成功。
6. B 又创建一个订单。
```

锁没有失效。

但业务仍然重复了。

原因是：锁只保护同一时间，不保护“同一业务请求以后不能再执行一次”。

---

## 四、幂等键

幂等通常需要一个业务唯一标识。

常见来源：

- 客户端传入 `Idempotency-Key`。
- 业务请求号 `request_id`。
- 订单号。
- 支付流水号。
- 用户 ID + 商品 ID + 活动 ID。
- 消息 ID。

示例：

```text
idempotency:create_order:request_abc123
```

同一个幂等键代表同一次业务请求。

---

## 五、数据库唯一约束

最可靠的幂等防线通常在数据库。

例如订单表：

```sql
CREATE UNIQUE INDEX uniq_order_request_id
ON orders(request_id);
```

创建订单时：

```sql
INSERT INTO orders(request_id, user_id, sku_id, status)
VALUES (?, ?, ?, ?);
```

如果同一个 `request_id` 重复插入，数据库会拒绝。

这是比 Redis 锁更硬的约束。

---

## 六、状态机条件更新

订单状态流转也要幂等。

例如只能从 `pending` 改到 `paid`：

```sql
UPDATE orders
SET status = 'paid', paid_at = NOW()
WHERE id = ? AND status = 'pending';
```

如果重复支付通知到达：

```text
第一次更新影响 1 行
第二次更新影响 0 行
```

第二次不应该报系统错误，而应该按幂等成功处理。

---

## 七、Redis 幂等记录

有些轻量场景可以用 Redis 保存幂等记录：

```redis
SET idem:create_order:request_abc123 processing NX EX 300
```

含义：

```text
如果设置成功，说明第一次处理。
如果设置失败，说明请求已经在处理或处理过。
```

但要注意：

- Redis 幂等记录过期后可能失效。
- Redis 故障时需要降级策略。
- 重要业务最终仍应有数据库约束。

Redis 幂等适合作为前置削峰，不适合作为唯一防线。

---

## 八、处理中的幂等状态

幂等记录常见状态：

```text
processing
succeeded
failed
```

流程：

```text
1. 写入 processing
2. 执行业务
3. 成功后改为 succeeded，并保存结果摘要
4. 重复请求发现 succeeded，直接返回上次结果
5. 如果 processing 超时，需要允许重试或人工处理
```

真实项目里，幂等表比 Redis key 更容易审计和恢复。

---

## 九、Go 里的幂等思路

伪代码：

```go
func (s *OrderService) CreateOrder(ctx context.Context, req CreateOrderRequest) (*Order, error) {
    if req.RequestID == "" {
        return nil, errors.New("request_id required")
    }

    order, err := s.repo.FindByRequestID(ctx, req.RequestID)
    if err == nil {
        return order, nil
    }
    if !errors.Is(err, ErrNotFound) {
        return nil, err
    }

    order, err = s.repo.InsertOrder(ctx, req)
    if isUniqueConflict(err) {
        return s.repo.FindByRequestID(ctx, req.RequestID)
    }
    if err != nil {
        return nil, err
    }

    return order, nil
}
```

关键是数据库唯一约束。

先查只是优化，不能替代唯一约束。

## 十、锁和幂等如何配合

一个更稳的创建订单流程：

```text
1. 校验 request_id
2. 尝试获取 Redis 锁，降低并发冲突
3. 检查幂等记录或订单是否已存在
4. 执行业务创建
5. 数据库唯一约束兜底
6. 释放锁
```

Redis 锁降低并发压力。

数据库幂等保证最终不重复。

不要把顺序反过来理解：锁是优化和保护，幂等是业务正确性的底线。

---

## 十一、哪些业务必须幂等

必须重点考虑：

- 创建订单。
- 支付回调。
- 退款。
- 发券。
- 扣库存。
- 积分变更。
- 发货。
- 消息消费。
- Webhook 回调。

这些场景常见重试、超时和重复通知。

只靠分布式锁风险很高。

---

## 十二、常见错误

### 1. 以为加锁后不会重复创建

锁过期或请求重试后，仍然可能重复。

### 2. 幂等键不稳定

每次重试都生成新的 request id，就无法识别同一次请求。

### 3. 先查再插但没有唯一约束

并发下两个请求都查不到，然后都插入成功。

### 4. 重复通知直接报错

支付回调、消息消费等重复通知应该按幂等结果处理。

### 5. Redis 幂等记录没有数据库兜底

重要业务不要只依赖短 TTL Redis key。

## 十四、本节小结

这一节你学习了业务幂等。

你需要记住：

- 锁解决互斥，幂等解决重复执行。
- 分布式锁不能替代业务幂等。
- 重要业务应使用数据库唯一约束、状态机条件更新或幂等表。
- Redis 幂等记录适合前置削峰，但不适合作为唯一防线。
- 请求重试、网络超时、消息重复投递都要求业务具备幂等能力。
