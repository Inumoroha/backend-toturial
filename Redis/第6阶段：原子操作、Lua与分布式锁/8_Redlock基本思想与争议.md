# 8. Redlock 基本思想与争议

前面几节讲的是单个 Redis 实例上的分布式锁。

但生产环境中 Redis 可能有主从、哨兵或集群。主节点故障、复制延迟和故障转移都会影响锁语义。Redlock 是 Redis 作者提出的一种多节点锁算法，用来降低单点 Redis 锁的风险。

这一节只讲基本思想和取舍，不把它当成必须掌握的实现细节。

学完这一节后，你应该能够：

- 理解单 Redis 锁在故障转移下的问题。
- 知道 Redlock 的基本流程。
- 理解 Redlock 为什么有争议。
- 判断什么时候不需要 Redlock。
- 明白强一致场景不能只靠 Redis 锁。

---

## 一、单 Redis 锁的故障风险

单 Redis 主节点加锁：

```redis
SET lock:order:1001 request-id NX EX 10
```

如果 Redis 是主从架构，可能出现：

```text
1. 客户端 A 在主节点获取锁成功。
2. 锁还没复制到从节点。
3. 主节点宕机。
4. 从节点提升为新主。
5. 客户端 B 在新主上也获取锁成功。
```

结果：

```text
A 和 B 都认为自己持有锁。
```

这不是 Redis 命令原子性的问题，而是复制和故障转移带来的分布式系统问题。

---

## 二、Redlock 的基本目标

Redlock 试图避免单个 Redis 节点故障导致锁失效。

它要求准备多个相互独立的 Redis master 节点。

例如 5 个：

```text
redis-1
redis-2
redis-3
redis-4
redis-5
```

客户端要在多数节点上获取锁成功，才认为加锁成功。

5 个节点时，多数派是 3 个。

---

## 三、Redlock 基本流程

简化流程：

```text
1. 记录当前时间 start。
2. 依次向 N 个独立 Redis 节点执行 SET key value NX PX ttl。
3. 每个节点使用较短超时，避免卡住。
4. 如果在多数节点获取锁成功，并且总耗时小于 ttl，则认为加锁成功。
5. 锁有效时间 = ttl - 获取锁耗时 - 时钟漂移预留。
6. 如果加锁失败，释放已经成功获取的节点上的锁。
```

释放锁时也要在每个节点上执行安全释放 Lua。

---

## 四、为什么要多数派

多数派的直觉是：

```text
如果两个客户端都想获取同一把锁，它们不应该同时在多数节点上成功。
```

例如 5 个节点：

```text
A 成功拿到 redis-1、redis-2、redis-3
B 想拿多数派时，至少需要 3 个节点
```

理论上 B 很难在 A 的锁有效期内再拿到 3 个未被占用节点。

这比单节点锁更能抵抗单点故障。

---

## 五、Redlock 为什么有争议

争议点主要来自分布式系统里的现实问题：

- 客户端可能长时间暂停，例如 GC 或调度停顿。
- 网络可能分区或延迟。
- 不同机器时钟可能有偏差。
- 锁 TTL 过期后旧客户端可能继续执行。
- Redis 节点并不是强一致复制系统。
- 算法正确性依赖很多时间假设。

换句话说，Redlock 比单 Redis 锁更强，但它仍然不是严格意义上的通用强一致锁服务。

如果业务后果严重，仍然需要数据库、状态机、fencing token 或真正的一致性系统兜底。

---

## 六、Redlock 适合什么场景

比较适合：

- 锁失败后业务可以重试。
- 偶发并发执行后果可控。
- 主要想降低单 Redis 节点故障风险。
- 锁保护的是短小临界区。
- 已经有幂等和补偿机制。

例如：

- 缓存重建。
- 定时任务抢占。
- 低风险后台协调。
- 可重试的资源处理。

---

## 七、Redlock 不适合什么场景

不适合把 Redlock 当作唯一防线的场景：

- 金融余额扣减。
- 支付状态变更。
- 唯一订单创建。
- 库存最终一致性要求高的核心链路。
- 无法容忍重复执行的外部副作用。

这些场景应该把正确性放在数据库事务、唯一约束、状态机、消息幂等或强一致协调系统里。

Redis 锁最多作为辅助保护。

---

## 八、Go 项目里该怎么选择

大多数中小型 Go 后端项目可以先按这个顺序考虑：

```text
1. 能不用锁就不用锁，优先数据库约束和幂等。
2. 低风险互斥场景，用单 Redis 基础锁 + Lua 释放。
3. 业务执行可能较长，考虑续期或改成任务状态机。
4. 需要防旧请求写入，增加 fencing token。
5. 对 Redis 单点故障敏感，再评估 Redlock。
```

不要一开始就上 Redlock。

它会增加部署、代码和排查复杂度。

---

## 九、如果使用 Redlock，要注意什么

建议：

- 使用成熟库，不要手写复杂实现。
- Redis 节点要尽量独立，避免同机同故障域。
- 每个节点加锁超时要短。
- 业务临界区要短。
- 仍然保留幂等和数据库约束。
- 记录加锁耗时和成功节点数。
- 对释放失败、锁过期、业务超时做监控。
- 不要把 Redlock 当作强一致事务。

Redlock 的工程价值在于降低风险，不是消灭所有风险。

---

## 十、和 etcd、ZooKeeper 的区别

如果你真正需要强一致协调，可以考虑：

- etcd。
- ZooKeeper。
- Consul。

这些系统通常基于一致性协议，提供租约、会话、顺序节点等能力。

代价是部署和运维复杂度更高，性能模型也不同。

Redis 更适合高性能缓存和轻量协作。

强一致协调不是 Redis 最核心的定位。

---

## 十一、常见错误

### 1. 以为 Redlock 解决所有锁问题

锁超时、旧请求继续执行、业务幂等仍然存在。

### 2. 多个 Redis 节点不独立

如果都在同一台机器上，多节点意义会下降。

### 3. 临界区太长

Redlock 仍然依赖 TTL 和时间假设。

### 4. 没有释放已获取的部分锁

加锁失败后要清理已经成功获取的节点。

### 5. 强一致业务只靠 Redlock

高风险业务应把正确性放到下游持久化系统中。

## 十三、本节小结

这一节你了解了 Redlock。

你需要记住：

- 单 Redis 锁在主从故障转移下可能失效。
- Redlock 通过多个独立 Redis 节点和多数派获取锁来降低风险。
- Redlock 仍然依赖时间、网络和客户端行为假设。
- 它不是通用强一致锁服务。
- 大多数业务应该优先考虑数据库约束、幂等、状态机和 fencing token。
