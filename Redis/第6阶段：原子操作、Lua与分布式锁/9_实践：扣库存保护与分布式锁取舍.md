# 9. 实践：扣库存保护与分布式锁取舍

这一节把第 6 阶段的内容串起来：原子操作、Lua、分布式锁、锁超时、幂等和 fencing token。

我们用“扣库存”作为例子。但重点不是写一个完美秒杀系统，而是学会判断：哪些问题适合 Redis 原子操作，哪些问题不应该只靠分布式锁。

学完这一节后，你应该能够：

- 用 Lua 实现 Redis 库存原子扣减。
- 理解扣 Redis 库存和创建数据库订单之间的一致性问题。
- 判断什么时候需要分布式锁。
- 给创建订单增加幂等约束。
- 设计失败补偿和库存回滚思路。

---

## 一、业务目标

假设有一个商品：

```text
sku_id = 2001
库存 = 100
```

用户下单时希望：

```text
1. 库存大于 0 才允许下单。
2. 同一个请求不能重复创建订单。
3. 并发请求不能把库存扣成负数。
4. Redis 扣减成功但数据库失败时，要能补偿。
```

我们先从 Redis 库存 key 开始：

```text
stock:sku:2001
```

---

## 二、不要用 GET 后 DECR

错误写法：

```go
stock, err := rdb.Get(ctx, "stock:sku:2001").Int()
if err != nil {
    return err
}
if stock <= 0 {
    return ErrStockNotEnough
}
if err := rdb.Decr(ctx, "stock:sku:2001").Err(); err != nil {
    return err
}
```

并发下会出现：

```text
库存 = 1
两个请求都 GET 到 1
两个请求都执行 DECR
库存变成 -1
```

这里应该用 Lua 把判断和扣减放到 Redis 内部原子执行。

---

## 三、扣库存 Lua 脚本

```lua
local stock = tonumber(redis.call("GET", KEYS[1]))
if stock == nil then
    return -1
end
if stock <= 0 then
    return -2
end
return redis.call("DECR", KEYS[1])
```

返回值设计：

| 返回值 | 含义 |
| --- | --- |
| `-1` | 库存 key 不存在 |
| `-2` | 库存不足 |
| `>= 0` | 扣减后的库存 |

注意：返回 `0` 表示扣减后库存刚好为 0，不是失败。

---

## 四、Go 封装扣库存

```go
var deductStockScript = redis.NewScript(`
local stock = tonumber(redis.call("GET", KEYS[1]))
if stock == nil then
    return -1
end
if stock <= 0 then
    return -2
end
return redis.call("DECR", KEYS[1])
`)
```

```go
func DeductStock(ctx context.Context, rdb *redis.Client, skuID int64) (int64, error) {
    key := fmt.Sprintf("stock:sku:%d", skuID)

    n, err := deductStockScript.Run(ctx, rdb, []string{key}).Int64()
    if err != nil {
        return 0, err
    }

    switch n {
    case -1:
        return 0, ErrStockKeyNotFound
    case -2:
        return 0, ErrStockNotEnough
    default:
        return n, nil
    }
}
```

这一步不需要分布式锁。

因为 Lua 已经保证了 Redis 内部“判断库存 + 扣减”的原子性。

---

## 五、创建订单的幂等约束

扣库存只是第一步。

还要防止重复创建订单。

请求里应该带稳定的幂等号：

```text
request_id = create_order_abc123
```

订单表增加唯一索引：

```sql
CREATE UNIQUE INDEX uniq_orders_request_id
ON orders(request_id);
```

创建订单时，如果唯一键冲突：

```text
查询已有订单并返回
```

这比 Redis 锁更可靠。

---

## 六、基础下单流程

```text
1. 校验 request_id
2. 根据 request_id 查询订单是否已存在
3. Redis Lua 扣减库存
4. 创建数据库订单
5. 如果创建订单失败，回滚 Redis 库存或记录补偿任务
6. 返回订单
```

伪代码：

```go
func (s *OrderService) CreateOrder(ctx context.Context, req CreateOrderRequest) (*Order, error) {
    order, err := s.repo.FindByRequestID(ctx, req.RequestID)
    if err == nil {
        return order, nil
    }
    if !errors.Is(err, ErrNotFound) {
        return nil, err
    }

    if _, err := DeductStock(ctx, s.rdb, req.SkuID); err != nil {
        return nil, err
    }

    order, err = s.repo.CreateOrder(ctx, req)
    if isUniqueConflict(err) {
        _ = s.compensateStock(ctx, req.SkuID)
        return s.repo.FindByRequestID(ctx, req.RequestID)
    }
    if err != nil {
        _ = s.compensateStock(ctx, req.SkuID)
        return nil, err
    }

    return order, nil
}
```

这仍然是简化版。

真实项目里补偿不能只靠 `_ =` 忽略结果。

---

## 七、补偿为什么重要

Redis 扣减成功后，数据库创建订单可能失败：

- 数据库超时。
- 唯一键冲突。
- 服务崩溃。
- 网络失败。
- 事务提交失败。

如果不补偿，Redis 库存会少扣。

补偿方式可以是：

- 立即 `INCR` 回滚库存。
- 写补偿任务表，异步重试。
- 用消息队列处理库存确认。
- 定期对账 Redis 库存和数据库订单。

重要业务不能只靠内存里的临时补偿。

---

## 八、回滚库存也要谨慎

简单回滚：

```redis
INCR stock:sku:2001
```

但要注意幂等。

如果同一个失败请求被补偿两次，库存会多加。

更严谨的方式是记录库存流水：

```text
stock_deduct_log(request_id, sku_id, status)
```

补偿时根据 `request_id` 保证只回滚一次。

这又回到了幂等。

---

## 九、什么时候需要分布式锁

对于“库存大于 0 才扣减”这个动作，Lua 已经够了。

不一定需要分布式锁。

可能需要锁的场景：

- 同一个 SKU 的库存初始化不能并发执行。
- 同一个订单的状态修复任务不能并发执行。
- 同一个热点缓存只能一个请求重建。
- 某个后台任务同一时间只能一个实例运行。

扣库存主链路上盲目加锁可能降低吞吐。

秒杀类场景更常见的是：

```text
Redis 原子扣减 + 队列削峰 + 数据库幂等 + 异步确认
```

## 十、如果要用锁保护同一订单

处理同一订单状态时，可以用：

```text
lock:order:{order_id}
```

流程：

```text
1. 获取锁
2. 读取订单状态
3. 条件更新状态
4. 释放锁
```

但数据库仍然要有条件更新：

```sql
UPDATE orders
SET status = 'paid'
WHERE id = ? AND status = 'pending';
```

锁降低并发冲突。

数据库状态机保证最终正确。

---

## 十一、加入 fencing token 的场景

如果存在锁过期后旧请求继续写订单处理结果的风险，可以增加 token：

```text
lock:{order:1001}
lock_token:{order:1001}
```

写数据库：

```sql
UPDATE order_process
SET result = ?, fencing_token = ?
WHERE order_id = ?
  AND fencing_token < ?;
```

旧 token 的请求影响 0 行，就停止后续动作。

这适合有明确资源版本的写入。

## 十二、一个更完整的实践方案

教学版方案：

```text
1. Redis Lua 原子扣库存。
2. 数据库 request_id 唯一约束防重复下单。
3. 创建订单失败时写补偿任务。
4. 补偿任务按 request_id 幂等回滚库存。
5. 订单状态流转使用条件更新。
6. 热点活动前预热库存 key。
7. 活动结束后对账 Redis 扣减和数据库订单。
```

如果流量更高：

```text
1. Redis 预扣库存。
2. 成功请求进入消息队列。
3. 消费者创建订单。
4. 消费端幂等。
5. 失败补偿库存。
6. 前端轮询订单结果。
```

---

## 十三、常见错误

### 1. 扣库存用 `GET + DECR`

并发下可能扣成负数。

### 2. 为扣库存主链路加一把大锁

吞吐会明显下降，也不一定更正确。

### 3. 只有 Redis 库存，没有数据库订单幂等

重复请求仍然可能创建多笔订单。

### 4. 扣 Redis 成功后数据库失败不补偿

会造成库存少扣。

### 5. 补偿没有幂等

重复补偿可能把库存加多。

### 6. 认为分布式锁能解决库存系统所有问题

库存系统真正难的是一致性、削峰、幂等、补偿和对账。

## 十五、本节小结

这一节完成了第 6 阶段的实践闭环。

你需要记住：

- Redis Lua 适合实现“判断库存 + 扣减”的原子操作。
- 扣库存主链路不一定需要分布式锁。
- Redis 扣减成功不代表数据库订单一定创建成功。
- 创建订单必须做业务幂等，常见方式是 `request_id` 唯一约束。
- 失败补偿也要幂等。
- 分布式锁只能保护临界区，不能替代数据库约束、状态机、补偿和对账。

学完第 6 阶段，你应该已经理解 Redis 原子能力和分布式锁的边界。下一阶段会进入限流、计数器与排行榜，这些也是 Redis 在高频后端场景中的经典用法。
