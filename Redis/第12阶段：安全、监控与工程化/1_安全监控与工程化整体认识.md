学到这里，你已经会用 Redis 做缓存、锁、限流、排行榜、消息、持久化、高可用和性能排查。

## 一、为什么 Redis 安全很重要

Redis 安全事故很常见。

常见原因：

- Redis 暴露到公网。
- 没有密码。
- 弱密码。
- 危险命令未限制。
- 配置文件随意修改。
- 没有备份。
- 没有监控。

Redis 里常常存放：

- 登录态。
- 验证码。
- 缓存数据。
- 限流状态。
- 任务队列。
- 热点业务数据。

一旦被攻击，可能导致数据泄露、缓存清空、业务不可用，甚至服务器被进一步入侵。

---

## 二、生产 Redis 的基本原则

基本原则：

```text
1. 不暴露公网。
2. 必须认证。
3. 最小权限。
4. 慎用危险命令。
5. 配置可审计。
6. 数据可恢复。
7. 指标可观测。
8. 故障可降级。
```

学习环境可以简单。

生产环境不能裸奔。

---

## 三、网络隔离优先

Redis 最重要的安全措施不是密码，而是网络隔离。

应该：

- 只监听内网地址。
- 使用安全组限制来源 IP。
- 禁止公网直接访问。
- 容器部署时不要随意映射端口到公网。
- 应用和 Redis 放在同一 VPC 或内网。

配置示例：

```conf
bind 127.0.0.1 10.0.0.10
protected-mode yes
```

如果 Redis 已经暴露公网，密码也不是万无一失。

---

## 四、认证和 ACL

基础密码：

```conf
requirepass strong-password
```

Redis 6 以后支持 ACL：

```text
不同用户，不同命令权限，不同 key 权限。
```

例如：

```redis
ACL SETUSER app on >app-password ~cache:* +get +set +del
```

ACL 可以让业务用户只访问自己的 key 前缀和必要命令。

这比所有应用共用管理员密码更安全。

---

## 五、危险命令

危险命令包括：

- `FLUSHALL`
- `FLUSHDB`
- `KEYS`
- `CONFIG`
- `SHUTDOWN`
- `SAVE`
- `DEBUG`
- `MODULE`

有些命令会清空数据。

有些命令会阻塞 Redis。

有些命令能修改配置。

生产中应该通过 ACL 或命令重命名限制。

---

## 六、监控不是可选项

Redis 至少要监控：

- QPS。
- 命中率。
- 内存使用率。
- 连接数。
- 慢命令。
- 淘汰 key 数。
- 过期 key 数。
- 主从复制状态。
- 复制延迟。
- AOF/RDB 状态。
- CPU 和网络。

没有监控，就只能等用户报障。

---

## 七、配置文件管理

生产 Redis 配置应该：

- 放入版本管理。
- 区分开发、测试、生产。
- 有变更记录。
- 有回滚方案。
- 禁止手工随意修改。

配置包括：

- 端口和绑定地址。
- 密码和 ACL。
- 持久化。
- maxmemory。
- 淘汰策略。
- 慢日志。
- 复制和 Sentinel。

---

## 八、备份和恢复

持久化不是备份。

备份要考虑：

- 备份文件保存在哪里。
- 备份频率。
- 保留周期。
- 是否加密。
- 是否定期恢复演练。
- 恢复时间目标。
- 允许丢多少数据。

只会备份，不会恢复，等于没有备份。

---

## 九、故障降级

Redis 故障时，不同业务策略不同。

例如：

| 场景 | 降级策略 |
| --- | --- |
| 文章详情缓存 | 查数据库，限流保护 |
| 阅读量计数 | 临时丢弃或本地缓冲 |
| 登录限流 | 加强验证码或保守拒绝 |
| 短信限流 | 拒绝发送，避免成本失控 |
| 排行榜 | 返回旧榜或空榜 |
| 分布式锁 | 谨慎失败，不能盲目放行 |

降级策略要提前写好。

不要等 Redis 挂了再讨论。

## 十一、常见错误

### 1. Redis 暴露公网

这是最危险的错误之一。

### 2. 所有服务共用管理员账号

权限过大，泄露后影响全局。

### 3. 没有限制危险命令

一次误操作可能清空数据。

### 4. 只有监控没有告警

指标没人看，问题仍然发现不了。

### 5. 没有恢复演练

备份文件不一定真的可用。
