# 8. Redis Stream 与 Kafka、RabbitMQ 的区别

Redis Stream 很像消息队列。

但“像”不等于“完全替代”。真实项目里要根据吞吐、可靠性、消息治理和运维能力选择合适工具。

学完这一节后，你应该能够：

- 理解 Redis Stream 的定位。
- 知道它和 Kafka、RabbitMQ 的差异。
- 判断什么时候 Stream 足够。
- 判断什么时候需要专业 MQ。
- 避免把 Redis 用成所有问题的唯一答案。

---

## 一、三者大致定位

简单理解：

| 系统 | 更擅长 |
| --- | --- |
| Redis Stream | 轻量任务队列、Redis 生态内的消息流 |
| Kafka | 高吞吐日志流、事件流、数据管道 |
| RabbitMQ | 复杂路由、传统消息队列、可靠投递 |

这不是绝对分类。

但作为后端选型起点，足够实用。

---

## 二、Redis Stream 的优势

优势：

- 已经使用 Redis 时接入成本低。
- 支持消费者组。
- 支持 ACK。
- 支持 Pending List。
- 命令简单。
- 适合轻量任务和中小规模异步处理。

例如：

- 缓存重建任务。
- 订单轻量事件。
- 发送通知任务。
- 后台处理队列。

如果项目已经依赖 Redis，Stream 是很自然的增强。

---

## 三、Redis Stream 的限制

限制：

- 数据存储在 Redis 内存中，成本较高。
- 超大规模消息堆积会给 Redis 带来压力。
- 分区扩展能力不如 Kafka。
- 消息路由能力不如 RabbitMQ。
- 死信、重试、延迟、优先级等需要自己设计。
- 复杂监控治理生态不如专业 MQ。

Redis 的核心定位仍然是内存数据系统。

不要让消息堆积把缓存系统拖垮。

---

## 四、Kafka 更适合什么

Kafka 更适合：

- 高吞吐日志采集。
- 用户行为事件流。
- 数据管道。
- 多消费者独立订阅。
- 长时间保留消息。
- 大规模分区扩展。
- 数据仓库同步。

例如：

```text
每秒几十万条用户行为日志
需要多个系统分别消费
需要保留 7 天用于回放
```

这种场景更适合 Kafka。

---

## 五、RabbitMQ 更适合什么

RabbitMQ 更适合：

- 传统业务消息。
- 复杂路由。
- 交换机和队列绑定。
- 可靠投递。
- 死信队列。
- 延迟插件。
- 优先级队列。

例如：

```text
订单消息根据类型路由到不同队列
失败后进入死信队列
不同消费者处理不同业务
```

RabbitMQ 的路由模型更成熟。

---

## 六、对比表

| 能力 | Redis Stream | Kafka | RabbitMQ |
| --- | --- | --- | --- |
| 消费者组 | 支持 | 支持 | 支持 |
| ACK | 支持 | 通过 offset 提交 | 支持 |
| 消息持久化 | 支持，但受 Redis 内存模型影响 | 强 | 强 |
| 高吞吐日志流 | 一般 | 强 | 一般 |
| 复杂路由 | 弱 | 一般 | 强 |
| 运维复杂度 | 低到中 | 中到高 | 中 |
| 适合轻量任务 | 强 | 可以但偏重 | 强 |
| 长期消息保留 | 不推荐太长 | 强 | 一般 |

选型不是谁更高级，而是谁更适合当前业务。

---

## 七、什么时候 Stream 足够

Stream 通常足够：

- 项目规模中小。
- 消息量不大。
- Redis 已经是基础组件。
- 任务可以幂等。
- 消息允许保留较短时间。
- 失败重试逻辑简单。
- 没有复杂路由。

例如：

- 异步刷新缓存。
- 异步发送邮件。
- 处理低频订单事件。
- 后台导出任务。

---

## 八、什么时候该上专业 MQ

建议考虑 Kafka 或 RabbitMQ：

- 消息量很大。
- 消息需要保留很久。
- 需要多系统回放。
- 需要复杂路由。
- 需要成熟死信和重试体系。
- Redis 内存成本不可接受。
- 消息堆积不能影响缓存。
- 团队已有 MQ 运维能力。

专业 MQ 的复杂度更高，但边界更清晰。

---

## 九、不要忽视数据库任务表

有些业务不一定需要 MQ。

可以用数据库任务表：

```text
tasks(id, type, status, next_run_at, retry_count)
```

worker 扫描：

```sql
SELECT * FROM tasks
WHERE status = 'pending'
  AND next_run_at <= NOW()
LIMIT 100;
```

优点：

- 可审计。
- 容易人工修复。
- 和业务数据事务更容易放一起。

缺点：

- 吞吐不如 MQ。
- 轮询压力要控制。

很多后台任务，数据库任务表反而更稳。

---

## 十、选型建议

可以按这个顺序想：

```text
1. 只是进程内异步？用 goroutine 或本地队列。
2. 需要可审计任务？用数据库任务表。
3. 已有 Redis，轻量异步？用 Redis Stream。
4. 高吞吐事件流？用 Kafka。
5. 复杂业务路由？用 RabbitMQ。
```

不要为了“架构完整”强行引入 MQ。

也不要在消息量已经很大时还硬撑 Redis。

---

## 十一、常见错误

### 1. 轻量任务一上来就 Kafka

系统复杂度可能远超收益。

### 2. 高吞吐日志还用 Redis Stream 硬扛

会占用大量 Redis 内存，影响缓存主业务。

### 3. 以为 Stream 自带完整死信体系

死信和重试策略需要自己设计。

### 4. 不做消费幂等

任何消息系统都可能重复投递。

### 5. 只看功能，不看运维能力

团队能否稳定运维，也会决定技术选型。

## 十三、本节小结

这一节你比较了 Redis Stream、Kafka 和 RabbitMQ。

你需要记住：

- Redis Stream 适合轻量消息队列和 Redis 生态内任务。
- Kafka 更适合高吞吐事件流和长期日志保留。
- RabbitMQ 更适合复杂路由和传统可靠消息。
- 数据库任务表也是很多业务的好选择。
- 消息系统选型要看吞吐、可靠性、治理能力和团队运维能力。
