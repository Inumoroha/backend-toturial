# 1. 高级数据结构与消息能力整体认识

前面阶段已经学习了 Redis 的核心数据结构、缓存、锁、限流、计数器和排行榜。

第 8 阶段会进入 Redis 的扩展能力：

- Bitmap。
- HyperLogLog。
- GEO。
- Pub/Sub。
- Stream。

这些能力的共同特点是：它们可以用较低成本解决某类特定问题。

学完这一节后，你应该能够：

- 知道这些高级结构分别解决什么问题。
- 理解它们和 String、Set、ZSet 的关系。
- 判断什么时候使用它们可以降低成本。
- 知道它们各自的限制。
- 为后续几节建立整体地图。

---

## 一、为什么要学习高级结构

很多业务如果只用基础结构，也能做。

例如统计用户签到：

```text
user:sign:1001:202607 -> Set 保存日期
```

也可以用 Bitmap：

```text
user:sign:1001:202607 -> 用 bit 表示每天是否签到
```

Bitmap 更省内存。

再比如统计 UV：

```text
Set 保存所有 user_id
```

也可以用 HyperLogLog：

```text
PFADD page:uv:home:20260705 user_id
```

HyperLogLog 不保存完整用户集合，但能用很小内存估算去重数量。

高级结构的价值不是“更高级”，而是在合适场景里更省资源。

---

## 二、Bitmap 解决什么问题

Bitmap 适合保存大量布尔状态。

例如：

- 用户签到。
- 当日活跃用户。
- 某功能是否开启。
- 用户是否完成任务。
- 商品是否被标记。
- 布尔状态压缩存储。

它的核心命令：

```redis
SETBIT key offset 1
GETBIT key offset
BITCOUNT key
```

一个 bit 只占 1 位。

如果有 100 万个用户，每个用户一天是否活跃，用 Bitmap 只需要大约 125KB。

---

## 三、HyperLogLog 解决什么问题

HyperLogLog 适合做去重计数，允许少量误差。

例如：

- 页面 UV。
- 活动独立访问人数。
- 搜索关键词独立用户数。
- 每日独立登录用户估算。

核心命令：

```redis
PFADD page:uv:home:20260705 user:1001
PFCOUNT page:uv:home:20260705
PFMERGE page:uv:home:week page:uv:home:20260701 page:uv:home:20260702
```

它不适合需要知道“具体有哪些用户”的场景。

它只适合估算数量。

---

## 四、GEO 解决什么问题

GEO 适合地理位置查询。

例如：

- 附近门店。
- 附近司机。
- 附近的人。
- 用户和门店距离。
- 城市内地理检索。

核心命令：

```redis
GEOADD shop:geo 116.397128 39.916527 shop:1001
GEOSEARCH shop:geo FROMLONLAT 116.40 39.90 BYRADIUS 3 km WITHDIST
GEODIST shop:geo shop:1001 shop:1002 km
```

Redis GEO 底层基于 ZSet。

适合轻量附近查询，不适合复杂 GIS 分析。

---

## 五、Pub/Sub 解决什么问题

Pub/Sub 是发布订阅。

发布者发送消息：

```redis
PUBLISH notice "hello"
```

订阅者订阅频道：

```redis
SUBSCRIBE notice
```

适合：

- 本地开发通知。
- 简单广播。
- 配置变更通知。
- 多实例缓存失效通知。

但它不可靠：

```text
订阅者不在线时，消息会丢。
```

所以不要用 Pub/Sub 做重要业务消息。

---

## 六、Stream 解决什么问题

Redis Stream 更接近消息队列。

它支持：

- 消息持久化。
- 消费者组。
- ACK 确认。
- Pending List。
- 消息回放。

核心命令：

```redis
XADD order:events * type created order_id 1001
XGROUP CREATE order:events order-workers 0 MKSTREAM
XREADGROUP GROUP order-workers c1 COUNT 10 BLOCK 5000 STREAMS order:events >
XACK order:events order-workers message_id
```

Stream 比 Pub/Sub 可靠很多。

但它仍然不是 Kafka 或 RabbitMQ 的完全替代品。

---

## 七、能力选型地图

| 需求 | 推荐 |
| --- | --- |
| 用户每天是否签到 | Bitmap |
| 统计页面 UV，允许误差 | HyperLogLog |
| 查询 3 公里内门店 | GEO |
| 本地开发广播通知 | Pub/Sub |
| 轻量异步任务队列 | Stream |
| 高吞吐日志流 | Kafka |
| 复杂路由和确认机制 | RabbitMQ |

Redis 适合轻量、近实时、工程简单的场景。

当消息系统要求很高时，应该用专业 MQ。

---

## 八、和前面阶段的关系

这些能力不是孤立的。

例如：

- Bitmap 可以做活跃用户统计。
- HyperLogLog 可以和计数器一起做 PV/UV。
- GEO 可以和缓存结合保存门店详情。
- Pub/Sub 可以通知本地缓存失效。
- Stream 可以承接第 7 阶段的延迟任务可靠化。

学习时不要只背命令。

要问：

```text
这个结构降低了什么成本？
它牺牲了什么能力？
业务能不能接受这种取舍？
```

---

## 九、使用高级结构前先问什么

几个关键问题：

```text
1. 我需要精确结果还是估算结果？
2. 我需要知道明细，还是只需要数量？
3. 数据生命周期多长？
4. Redis 数据丢失后能否恢复？
5. 是否需要消费者确认和重试？
6. 这个能力会不会变成 Big Key？
7. 是否需要专业系统替代 Redis？
```

这些问题会帮你避免把 Redis 用过头。

## 十、常见错误

### 1. 用 HyperLogLog 后又想取出用户列表

HyperLogLog 只能估算去重数量，不能反查成员。

### 2. 用 Pub/Sub 做重要订单消息

订阅者不在线时消息会丢。

### 3. 用 Stream 替代所有 MQ

Stream 很有用，但复杂消息治理仍然建议专业 MQ。

### 4. GEO 存完整门店详情

GEO 里放门店 ID，详情用缓存或数据库查询。

### 5. Bitmap offset 设计混乱

offset 一旦设计错，数据很难解释。

## 十二、本节小结

这一节建立了第 8 阶段的整体地图。

你需要记住：

- Bitmap 适合大量布尔状态。
- HyperLogLog 适合允许误差的去重计数。
- GEO 适合轻量地理位置查询。
- Pub/Sub 适合轻量广播，不保证可靠投递。
- Stream 更接近消息队列，支持消费者组和 ACK。
- 是否使用高级结构，本质上是成本和能力的取舍。
