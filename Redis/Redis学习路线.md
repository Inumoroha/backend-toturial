# Redis + Go 后端学习路线图（综合版）

这份路线图面向“想把 Redis 用到 Go 后端项目里”的学习目标。它保留了原路线中对数据结构、缓存、分布式锁、高可用和性能调优的覆盖，也补充了真实项目里更重要的使用边界、缓存一致性、生产排查、安全配置、监控和 Go 工程实践。

学习顺序建议是：先建立 Redis 的数据结构直觉，再学习缓存与典型业务场景，然后接入 Go 项目，最后理解持久化、高可用、集群和生产排查。

Redis 不是数据库的替代品，也不是所有高并发问题的万能解。它更适合作为缓存、计数器、限流器、排行榜、短期状态存储、轻量消息队列和分布式协作组件。

---

## 第一阶段：环境搭建与 Redis 直觉

目标：能启动 Redis，连接 Redis，并理解 Redis 是一个基于内存、以 key-value 为核心的数据系统。

### 需要掌握

- 使用 Docker 启动 Redis，避免污染本地环境。
- 使用 `redis-cli` 连接 Redis。
- 使用图形化客户端，例如 Another Redis Desktop Manager、RedisInsight。
- 理解 Redis 的 key-value 模型。
- 理解 Redis 单线程主循环、大部分操作在内存中完成的特点。
- 理解 Redis 与 MySQL、PostgreSQL、MongoDB、Kafka 的大致区别。

### 推荐实践

- 启动一个 Redis 容器。
- 使用 `redis-cli` 执行 `PING`、`SET`、`GET`、`DEL`。
- 用图形化客户端查看 key、value、TTL。
- 尝试给 key 设置过期时间，再观察它什么时候消失。

### 注意点

一开始不要急着研究跳表、RDB、AOF、Cluster。先建立直觉：Redis 擅长快速读写临时或热点数据，但它不适合承担所有持久化业务数据。

---

## 第二阶段：通用命令与 key 设计

目标：能熟练管理 key，并设计清晰、可维护、不容易冲突的 key 命名方式。

### 需要掌握

- 基础命令：`SET`、`GET`、`DEL`、`EXISTS`、`TYPE`。
- 过期时间：`EXPIRE`、`TTL`、`PERSIST`。
- 自增自减：`INCR`、`DECR`、`INCRBY`。
- 批量读取：`MGET`、`MSET`。
- 安全遍历：`SCAN`。
- 禁用或慎用：`KEYS *`、`FLUSHALL`、`FLUSHDB`。
- key 命名规范，例如 `user:profile:1001`、`article:detail:9527`。
- value 大小控制和 TTL 规划。

### 推荐实践

- 设计一组用户相关 key，例如用户详情、登录 token、短信验证码。
- 给验证码设置 5 分钟过期时间。
- 用 `SCAN` 分批查找某类 key。
- 对比 `KEYS *` 和 `SCAN` 的区别。

### 注意点

生产环境不要使用 `KEYS *` 扫全库。Redis 是单线程处理命令的，慢命令可能阻塞其他请求。key 命名也不要随意，好的命名能降低排查和迁移成本。

---

## 第三阶段：核心数据结构与典型场景

目标：理解 Redis 最常用的数据结构，并知道每种结构适合解决什么问题。

### 需要掌握

- String：缓存对象、验证码、计数器、分布式锁基础。
- Hash：存储对象字段，例如用户资料、商品属性。
- List：简单队列、最新列表、任务列表。
- Set：去重、抽奖、共同好友、标签集合。
- ZSet：排行榜、延迟队列、按分数排序的业务数据。
- 每种结构的常用命令、适用场景和不适用场景。

### 推荐实践

- 用 String 缓存一篇文章详情。
- 用 Hash 存储用户资料。
- 用 List 实现“最新 10 条动态”。
- 用 Set 实现用户标签去重。
- 用 ZSet 实现文章热榜。

### 注意点

不要只背“五大数据类型”。真正重要的是知道业务数据应该怎么建模。比如对象可以用 String 存 JSON，也可以用 Hash 存字段；前者读写简单，后者适合局部字段更新，但结构维护成本更高。

## 第四阶段：缓存设计与一致性

目标：能在真实后端项目中正确使用 Redis 做缓存，并理解缓存和数据库之间的一致性取舍。

### 需要掌握

- Cache-Aside Pattern：旁路缓存模式。
- 读流程：先查缓存，未命中再查数据库，然后回写缓存。
- 写流程：更新数据库后删除缓存。
- 先更新数据库还是先删缓存。
- 延迟双删的适用场景和局限。
- 缓存穿透、缓存击穿、缓存雪崩。
- 空值缓存。
- Bloom Filter。
- 热点 key 的保护。
- TTL 随机化。
- 逻辑过期。

### 推荐实践

给文章详情接口增加缓存：

- 第一次请求从数据库读取文章，然后写入 Redis。
- 第二次请求直接从 Redis 返回。
- 更新文章时删除缓存。
- 给缓存设置随机 TTL，例如 30 到 40 分钟。
- 对不存在的文章缓存空值，防止反复打到数据库。

### 注意点

缓存一致性没有银弹。大多数业务追求的是“最终一致”，不是强一致。要先明确业务能不能接受短时间旧数据，再决定缓存策略。

## 第五阶段：Go 接入 Redis

目标：能在 Go 服务中稳定、清晰地操作 Redis，并把 Redis 放进真实请求链路中。

### 需要掌握

- `github.com/redis/go-redis/v9`。
- `redis.Client` 初始化。
- 连接池配置：`PoolSize`、`MinIdleConns`。
- `context.Context` 传递超时和取消信号。
- 常用数据结构命令在 Go 中的写法。
- JSON 序列化和反序列化。
- 错误处理：区分 `redis.Nil` 和真实错误。
- Pipeline。
- Transaction Pipeline。
- Lua 脚本：`Eval`、`EvalSha`。

### 推荐实践

用 Go 写一个文章缓存模块：

- `GetArticle(ctx, id)`：优先查 Redis，未命中查数据库。
- `SetArticleCache(ctx, article)`：将文章序列化为 JSON 后写入 Redis。
- `DeleteArticleCache(ctx, id)`：更新文章后删除缓存。
- 对 Redis 操作设置合理超时。
- Redis 故障时允许降级到数据库。

### 注意点

Go 里使用 Redis 时不要忽略 `context`。请求已经取消或超时后，继续等待 Redis 没有意义。也不要把 Redis 错误全部吞掉，至少要打日志并区分缓存未命中和系统异常。

---

## 第六阶段：原子操作、Lua 与分布式锁

目标：理解 Redis 的原子能力，并能谨慎实现分布式协作场景。

### 需要掌握

- Redis 单条命令的原子性。
- `SET key value NX EX seconds` 实现基础锁。
- 锁 value 使用唯一标识，防止误删别人的锁。
- 使用 Lua 脚本释放锁。
- 锁超时问题。
- 自动续期 Watch Dog。
- 业务幂等。
- fencing token。
- Redlock 的基本思想和争议。

### 推荐实践

实现一个简单扣库存保护：

- 抢锁成功后执行业务逻辑。
- 释放锁时校验 value。
- 故意让业务执行时间超过锁过期时间，观察可能出现的问题。
- 给业务增加幂等校验。

### 注意点

分布式锁不是并发问题的万能解。能用数据库唯一约束、乐观锁、事务或消息队列解决的场景，不一定要上 Redis 锁。锁只能保护临界区，不能代替业务幂等。

---

## 第七阶段：限流、计数器与排行榜

目标：掌握 Redis 在高频后端场景中的经典用法。

### 需要掌握

- 固定窗口限流：`INCR` + `EXPIRE`。
- 滑动窗口限流：ZSet。
- 令牌桶和漏桶思想。
- 计数器：浏览量、点赞数、库存预扣。
- 排行榜：ZSet 分数排序。
- 延迟队列：ZSet score 表示执行时间。

### 推荐实践

- 给登录接口增加 IP 限流。
- 给短信验证码接口增加手机号限流。
- 用 ZSet 实现文章阅读排行榜。
- 用 ZSet 实现一个简单延迟任务队列。

### 注意点

限流要考虑维度：用户、IP、接口、租户、设备。不要只实现算法本身，还要考虑误伤、绕过、清理和降级策略。

---

## 第八阶段：高级数据结构与消息能力

目标：理解 Redis 的扩展数据结构，并知道什么时候可以用它们降低成本。

### 需要掌握

- Bitmap：签到、活跃用户、布尔状态压缩。
- HyperLogLog：UV、去重计数，允许少量误差。
- GEO：附近的人、门店距离、地理位置查询。
- Pub/Sub：轻量通知。
- Stream：消费者组、消息持久化、消息确认、Pending List。
- Redis Stream 与 Kafka、RabbitMQ 的区别。

### 推荐实践

- 用 Bitmap 实现用户签到。
- 用 HyperLogLog 统计页面 UV。
- 用 GEO 查询附近门店。
- 用 Pub/Sub 做本地开发中的简单通知。
- 用 Stream 实现一个可靠性更好的异步任务队列。

### 注意点

Pub/Sub 不适合可靠消息投递，订阅者不在线时消息容易丢。Redis Stream 比 Pub/Sub 更接近消息队列，但如果业务对吞吐、回放、分区、跨服务消息治理要求很高，仍然应该考虑专业 MQ。

---

## 第九阶段：持久化、过期与内存管理

目标：理解 Redis 如何保存数据、清理数据，以及内存不足时会发生什么。

### 需要掌握

- RDB：快照、`BGSAVE`、`fork`、COW 写时复制。
- AOF：追加日志、刷盘策略、AOF 重写。
- 混合持久化。
- key 过期删除策略：惰性删除、定期删除。
- 内存淘汰策略：`noeviction`、`allkeys-lru`、`volatile-lru`、`allkeys-lfu` 等。
- LRU 和 LFU 的近似实现。
- Big Key 对内存和阻塞的影响。

### 推荐实践

- 开启 AOF，写入数据后重启 Redis，观察数据恢复。
- 对比 RDB 和 AOF 文件。
- 设置 `maxmemory` 和淘汰策略，观察写入行为。
- 构造一个 Big Key，观察删除耗时，再使用 `UNLINK`。

### 注意点

Redis 可以持久化，但它通常不应该作为核心业务数据的唯一存储。是否允许丢数据、允许丢多少、恢复时间能接受多少，都要结合业务来决定。

---

## 第十阶段：主从复制、哨兵与集群

目标：理解 Redis 在生产环境中的高可用和扩展方式。

### 需要掌握

- 主从复制。
- 全量同步和增量同步。
- 读写分离。
- 复制延迟。
- Sentinel。
- 主观下线和客观下线。
- 故障转移。
- Redis Cluster。
- Hash Slot。
- `MOVED` 和 `ASK` 重定向。
- 多 key 操作在 Cluster 中的限制。

### 推荐实践

- 用 Docker Compose 启动一主一从。
- 手动停止主节点，观察从节点状态。
- 搭建一个简单 Sentinel 环境。
- 了解 Cluster 的槽位分布。
- 尝试使用 hash tag，例如 `{user:1001}:profile`。

### 注意点

高可用不等于没有故障。主从切换期间可能出现短暂不可用、数据丢失或客户端重连问题。Go 客户端配置、超时、重试和降级也要一起考虑。

---

## 第十一阶段：性能分析与生产排查

目标：能在生产环境中定位 Redis 慢、抖动、内存异常和热点问题。

### 需要掌握

- `INFO` 查看整体状态。
- `SLOWLOG` 查看慢命令。
- `MONITOR` 的使用风险。
- `CLIENT LIST` 查看连接。
- `MEMORY USAGE`、`MEMORY STATS`。
- Big Key 发现与治理。
- Hot Key 发现与治理。
- Pipeline 减少 RTT。
- 慢命令和阻塞命令。
- 网络延迟、连接池耗尽、CPU 飙高、内存碎片。
- 本地缓存 + Redis 多级缓存。

### 推荐实践

- 用 `SLOWLOG GET` 查看慢命令。
- 对比单条循环写入和 Pipeline 批量写入的耗时。
- 构造热点 key，然后思考如何分散压力。
- 使用本地缓存减少 Redis 访问。

### 注意点

性能优化不要只盯着 Redis 本身。很多问题来自客户端连接池配置、网络 RTT、序列化成本、Big Key、业务访问模式和数据库回源压力。

---

## 第十二阶段：安全、监控与工程化

目标：把 Redis 当作生产基础设施来管理，而不是只当本地开发工具。

### 需要掌握

- 密码认证。
- ACL。
- 禁止公网裸奔。
- 绑定内网地址。
- TLS 基础。
- 慎用危险命令。
- 配置文件管理。
- Docker Compose 部署。
- Prometheus / Grafana 监控。
- 常见监控指标：QPS、命中率、内存、连接数、慢命令、复制延迟、淘汰 key 数。
- 备份与恢复流程。

### 推荐实践

- 给本地 Redis 配置密码。
- 禁用或重命名危险命令。
- 用 Docker Compose 管理 Redis。
- 整理一份 Redis 运行配置说明。
- 记录 Redis 故障时的降级策略。

### 注意点

Redis 安全事故很常见，尤其是公网暴露、弱密码、危险命令未限制。学习阶段可以简单，生产环境必须认真处理网络隔离、权限、备份和监控。

## 第十三阶段：项目落地：短链接缓存与限流服务

目标：把 Redis 真正接入 Go 后端项目，完成学习闭环。

### 项目功能

基于一个短链接服务，加入 Redis 能力：

- 创建短链接。
- 根据短码跳转到原始链接。
- 使用 PostgreSQL 或 MySQL 保存核心数据。
- 使用 Redis 缓存热门短链接。
- 使用 Redis 记录访问计数。
- 使用 Redis 给创建接口做限流。
- 使用 Redis 防止缓存穿透。
- 更新短链接时删除缓存。

### 推荐 key 设计

```text
shortlink:detail:{code}
shortlink:visit_count:{code}
shortlink:create_limit:{user_id}
shortlink:not_found:{code}
```

### 推荐接口

- `POST /links`：创建短链接。
- `GET /:code`：根据短码跳转。
- `GET /links/:code`：查询短链接详情。
- `PATCH /links/:code`：更新短链接。
- `GET /links/:code/stats`：查询访问统计。

### 需要用到的能力

- Go `net/http` 或 Gin / Chi。
- `github.com/redis/go-redis/v9`。
- PostgreSQL 或 MySQL。
- Cache-Aside Pattern。
- JSON 序列化。
- TTL。
- 计数器。
- 限流。
- 空值缓存。
- Pipeline。
- 日志与错误处理。

### 可选增强

- 用 ZSet 做热门链接排行榜。
- 用 Stream 异步记录访问日志。
- 用 Bitmap 统计每日是否访问。
- 用 Lua 实现更严谨的限流。
- 加入本地缓存，形成多级缓存。

## 推荐学习节奏

如果每天学习 1 到 2 小时，可以按下面节奏推进：

1. 第 1 周：环境搭建、通用命令、key 设计、核心数据结构。
2. 第 2 周：缓存模式、缓存一致性、穿透/击穿/雪崩。
3. 第 3 周：Go + Redis 接入、Pipeline、Lua、错误处理。
4. 第 4 周：分布式锁、限流、计数器、排行榜。
5. 第 5 周：Bitmap、HyperLogLog、GEO、Pub/Sub、Stream。
6. 第 6 周：持久化、内存淘汰、主从、Sentinel、Cluster。
7. 第 7 周：性能排查、安全、监控、短链接项目整合。

如果时间更紧，可以先压缩成一条最小路线：

1. Docker 启动 Redis。
2. 学会 String、Hash、Set、ZSet 和 TTL。
3. 学会 Cache-Aside、缓存穿透、缓存击穿、缓存雪崩。
4. 学会 Go + `go-redis/v9`。
5. 在短链接或文章系统里完成缓存、限流和计数器。

---

## 最终总结

这条综合路线的核心思路是：

1. 先理解 Redis 的 key-value 模型和常用数据结构。
2. 再掌握缓存设计、一致性和典型高并发场景。
3. 然后用 Go 和 `go-redis/v9` 把 Redis 接入真实服务。
4. 接着学习 Lua、分布式锁、限流、排行榜和消息能力。
5. 最后补齐持久化、高可用、集群、性能排查、安全和监控。

学完之后，你的目标不是“知道 Redis 有哪些命令”，而是能够在真实 Go 后端项目中判断什么时候该用 Redis、怎么设计 key、怎么处理缓存一致性、怎么排查性能问题，并在 Redis 出故障时让服务有合理的降级方案。
