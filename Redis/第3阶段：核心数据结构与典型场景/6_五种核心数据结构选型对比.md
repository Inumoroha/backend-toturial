# 6. 五种核心数据结构选型对比

前面几节分别学习了 String、Hash、List、Set、ZSet。

这一节不再讲单个命令，而是把它们放在一起对比：遇到一个业务需求时，应该选择哪种 Redis 数据结构，为什么选它，不选其他结构会有什么问题。

学完这一节后，你应该能够：

- 根据业务访问方式选择 Redis 数据结构。
- 区分缓存对象、对象字段、队列、去重集合、排行榜等场景。
- 知道每种结构的适用和不适用场景。
- 避免用 String 硬做所有事情。
- 为真实 Go 后端项目设计更合理的 Redis key 和 value。

---

## 一、先记住一句话

选择 Redis 数据结构，不是看你会哪个命令，而是看业务数据是什么形态。

你可以先问自己几个问题：

1. 这个数据是不是一个完整对象？
2. 是否经常只读写对象中的某个字段？
3. 是否需要保留顺序？
4. 是否需要自动去重？
5. 是否需要按分数排序？
6. 是否需要过期时间？
7. value 会不会很大？
8. 数据是否会无限增长？

这些问题比背命令更重要。

---

## 二、五种结构总览

| 数据结构 | 核心特点 | 典型场景 |
| --- | --- | --- |
| String | 一个 key 对应一个字符串 value | 缓存对象、验证码、token、计数器、锁基础 |
| Hash | 一个 key 下多个 field-value | 用户资料、商品属性、对象字段、统计字段 |
| List | 有序列表，左右两端操作 | 简单队列、最新列表、任务列表 |
| Set | 无序、不重复集合 | 去重、标签、点赞用户、共同好友、抽奖 |
| ZSet | 不重复成员 + score 排序 | 排行榜、延迟队列、按分数排序 |

这张表是初学阶段最重要的地图。

---

## 三、对象缓存：String 还是 Hash

这是最常见的选择题。

需求：缓存文章详情。

文章对象：

```json
{
  "id": 1001,
  "title": "Redis 数据结构",
  "author_id": 9,
  "content": "..."
}
```

方案一：String + JSON

```redis
SET article:detail:1001 '{"id":1001,"title":"Redis 数据结构","author_id":9}' EX 1800
```

方案二：Hash

```redis
HSET article:detail:1001 id 1001 title "Redis 数据结构" author_id 9
```

怎么选？

如果接口每次都读取完整文章详情，String + JSON 更简单。

如果经常只更新或读取某几个字段，Hash 更方便。

对比：

| 问题 | 更适合 |
| --- | --- |
| 每次读取完整对象 | String + JSON |
| 对象字段扁平，常局部更新 | Hash |
| 对象有复杂嵌套结构 | String + JSON |
| 只想更新某个字段 | Hash |
| 更新时通常删除缓存，下次重建 | String + JSON |

真实项目里，文章详情、商品详情通常优先考虑 String + JSON；用户资料、商品属性、统计字段可以考虑 Hash。

---

## 四、验证码、token、临时状态：优先 String

验证码：

```redis
SET sms:code:phone:13800138000 888888 EX 300
```

登录 token：

```redis
SET login:token:user:1001 token_value EX 604800
```

防重复提交：

```redis
SET submit:once:order:1001 request_id NX EX 30
```

这些场景共同特点：

- 一个 key 对应一个简单值。
- value 很小。
- 通常有 TTL。
- 不需要集合、排序、字段操作。

所以 String 最合适。

不要为了“结构化”强行用 Hash。

---

## 五、计数器：String 或 Hash

单个计数器适合 String：

```redis
INCR article:view_count:1001
```

多个相关计数字段适合 Hash：

```redis
HINCRBY article:stats:1001 view_count 1
HINCRBY article:stats:1001 like_count 1
HINCRBY article:stats:1001 comment_count 1
```

怎么选？

| 场景 | 推荐结构 |
| --- | --- |
| 只有一个计数，如阅读量 | String |
| 一个对象下多个计数，如阅读、点赞、评论 | Hash |
| 需要按计数排序形成排行榜 | ZSet |

比如文章阅读量如果只是累加，用 String 可以。

如果要按阅读量排行，用 ZSet 更合适：

```redis
ZINCRBY rank:article:view 1 article:1001
```

---

## 六、最新列表：List

需求：保存用户最近 10 条动态。

推荐：List。

```redis
LPUSH feed:user:1001 post:9001
LTRIM feed:user:1001 0 9
LRANGE feed:user:1001 0 9
```

为什么不用 Set？

Set 无序，不能表达最新顺序。

为什么不用 ZSet？

ZSet 可以用时间戳排序，但如果只是简单保留最新 N 条，List 更直接。

什么时候用 ZSet？

如果你需要按时间范围查询，比如查询某个时间段的动态，ZSet 更合适。

---

## 七、队列：List、Stream 或专业 MQ

简单队列可以用 List：

```redis
RPUSH task:email task1
LPOP task:email
```

但 List 做队列的可靠性有限。

如果任务很简单，丢失影响不大，List 可以。

如果需要：

- 消息确认。
- 失败重试。
- 消费者组。
- Pending 记录。
- 消息追踪。

应该考虑 Redis Stream。

如果是跨服务大规模消息、日志流、订单事件流，应该考虑 Kafka、RabbitMQ 等专业消息系统。

所以队列选型可以这样记：

| 场景 | 推荐 |
| --- | --- |
| 本地简单异步任务 | List |
| Redis 内可靠性更强的队列 | Stream |
| 大规模跨服务消息 | Kafka / RabbitMQ |

---

## 八、去重：Set

需求：一篇文章只能被同一个用户点赞一次。

推荐：Set。

```redis
SADD article:liked_users:1001 user:2001
SISMEMBER article:liked_users:1001 user:2001
SCARD article:liked_users:1001
```

为什么不用 List？

List 允许重复，需要自己判断是否已经存在，成本更高。

为什么不用 String？

把用户 ID 拼成字符串很难维护，也不好判断是否包含。

Set 的价值就是自动去重和快速判断成员是否存在。

---

## 九、标签集合：Set

用户标签：

```redis
SADD user:tags:1001 go redis mysql
```

判断标签：

```redis
SISMEMBER user:tags:1001 redis
```

删除标签：

```redis
SREM user:tags:1001 mysql
```

如果还要支持“查拥有某个标签的所有用户”，可以加反向集合：

```redis
SADD tag:users:redis 1001
```

这样就能从两个方向查询。

注意：双向维护时，要保证写入和删除逻辑一致，避免正向集合和反向集合不一致。

---

## 十、共同好友：Set 交集

用户好友集合：

```redis
SADD user:friends:1001 2001 2002 2003
SADD user:friends:1002 2002 2003 2004
```

共同好友：

```redis
SINTER user:friends:1001 user:friends:1002
```

这就是 Set 的典型优势。

但如果好友数量非常大，交集操作也可能很重。生产环境要考虑集合大小、调用频率和是否需要缓存结果。

---

## 十一、排行榜：ZSet

文章热榜：

```redis
ZINCRBY rank:article:daily 1 article:1001
ZINCRBY rank:article:daily 5 article:1002
ZREVRANGE rank:article:daily 0 9 WITHSCORES
```

用户积分榜：

```redis
ZADD rank:user:score 100 user:1001 200 user:1002
ZREVRANK rank:user:score user:1001
```

为什么不用 Set？

Set 只能去重，不能按分数排序。

为什么不用 List？

List 只能按插入顺序，不会自动按分数调整位置。

ZSet 的关键是：分数变化后，排名自动变化。

---

## 十二、延迟队列：ZSet 基础方案

延迟队列可以用 ZSet 的 score 表示执行时间。

```redis
ZADD delay:queue 1790000000 task:1001
```

查询到期任务：

```redis
ZRANGEBYSCORE delay:queue -inf 1790000000 LIMIT 0 10
```

到期后删除：

```redis
ZREM delay:queue task:1001
```

这适合理解延迟队列基本思想。

但生产级延迟队列还要处理并发抢占、重试、失败补偿和任务幂等。ZSet 只是基础，不是完整答案。

---

## 十三、不要用一种结构硬做所有事情

常见错误：什么都用 String。

比如：

```text
user:tags:1001 = "go,redis,mysql"
```

这样很难去重、删除、判断成员是否存在。Set 更合适。

再比如：

```text
rank:article = "article:1001:99,article:1002:120"
```

这样很难维护排名。ZSet 更合适。

再比如：

```text
task:queue = "task1,task2,task3"
```

这样很难先进先出。List 更合适。

Redis 提供多种结构，就是为了让数据形态更贴近业务。

## 十五、设计时再加三个约束

选好数据结构后，还要继续考虑三个约束。

### 1. TTL

这个 key 是否应该过期？

验证码、token、缓存、限流计数通常需要 TTL。

排行榜可以按周期设置 TTL，例如日榜保留 7 天。

### 2. 大小

这个 key 会不会变成 Big Key？

List、Set、ZSet、Hash 都可能无限增长。要控制长度、定期清理或拆分 key。

### 3. 一致性

这个数据是否是权威数据？

如果是订单、支付、库存主数据，不能只靠 Redis。Redis 可以做缓存、计数、预处理，但权威数据通常要落数据库。

## 十六、综合案例：文章系统

一个文章系统可以这样用 Redis：

文章详情缓存：

```text
article:detail:{article_id} -> String JSON
```

文章统计：

```text
article:stats:{article_id} -> Hash
  view_count
  like_count
  comment_count
```

文章点赞用户：

```text
article:liked_users:{article_id} -> Set
```

文章最新评论 ID：

```text
article:latest_comments:{article_id} -> List
```

文章日榜：

```text
rank:article:daily:{yyyyMMdd} -> ZSet
```

你会发现，一个业务系统里不会只用一种结构。不同数据形态对应不同结构。

这就是 Redis 建模的核心。

---

## 十七、综合案例：短链接系统

短链接详情缓存：

```text
shortlink:detail:{code} -> String JSON
```

不存在短码的空值缓存：

```text
shortlink:not_found:{code} -> String
```

短链接访问计数：

```text
shortlink:visit_count:{code} -> String counter
```

创建接口限流：

```text
shortlink:create_limit:user:{user_id} -> String counter + TTL
```

热门短链接榜：

```text
rank:shortlink:daily:{yyyyMMdd} -> ZSet
```

访问日志如果只是轻量缓冲，可以用 List；如果需要可靠消费，可以考虑 Stream 或 Kafka。

## 十九、本节小结

这一节你完成了五种核心数据结构的选型总结。

你需要记住：

- String 适合简单值、对象 JSON、验证码、token、计数器、锁基础。
- Hash 适合扁平对象字段和局部更新。
- List 适合简单队列、最新列表、任务列表。
- Set 适合去重、标签、点赞用户、共同好友、抽奖。
- ZSet 适合排行榜、延迟队列、按分数排序的数据。
- 选型时要同时考虑 TTL、value 大小、是否会无限增长、是否需要强一致。

学到这里，你已经不只是会 Redis 命令了，而是开始能用 Redis 的数据结构去表达业务模型。
