# 1. 限流、计数器与排行榜的整体认识

从这一阶段开始，Redis 的角色会从“缓存和锁”扩展到高频业务组件。

很多后端系统都会遇到这些需求：

- 登录接口不能被无限尝试。
- 短信验证码不能频繁发送。
- 文章阅读量要高频增加。
- 点赞要去重并计数。
- 热门文章需要实时排序。
- 延迟任务需要在指定时间后执行。

这些场景都很适合用 Redis 起步。

学完这一节后，你应该能够：

- 理解限流、计数器、排行榜分别解决什么问题。
- 知道 Redis 为什么适合这些高频场景。
- 区分“算法正确”和“工程可用”。
- 设计限流维度和 key。
- 为后续几节建立整体地图。

---

## 一、为什么这些场景适合 Redis

Redis 的优势是：

- 内存读写快。
- 单条命令原子。
- 支持 TTL。
- 支持 Set、ZSet 等结构。
- 支持 Lua 封装多步原子逻辑。
- 客户端生态成熟。

这些特点正好适合高频、小状态、短周期的数据。

例如短信限流：

```text
sms:limit:phone:13800000000:2026070513
```

这个 key 只需要保存一小时内的发送次数，到期自动删除。

没有必要为每次发送都先查数据库。

---

## 二、限流解决什么问题

限流的目标是保护系统。

常见保护对象：

- 登录接口。
- 注册接口。
- 短信验证码。
- 邮件发送。
- 搜索接口。
- 下单接口。
- 第三方 API 调用。
- 后台管理操作。

限流不是为了“拒绝用户”，而是为了避免：

- 暴力破解。
- 恶意刷接口。
- 短信成本失控。
- 下游服务被打垮。
- 数据库被瞬时流量压垮。

---

## 三、限流维度

限流最容易漏掉的是维度。

常见维度：

| 维度 | 示例 |
| --- | --- |
| IP | 同一个 IP 每分钟最多 20 次登录 |
| 用户 | 同一个用户每分钟最多 10 次操作 |
| 手机号 | 同一手机号每分钟最多 1 条短信 |
| 接口 | 某接口全局每秒最多 1000 次 |
| 租户 | 同一租户每分钟最多 5000 次请求 |
| 设备 | 同一设备每小时最多 20 次验证码 |

真实系统经常组合维度：

```text
登录接口：
1. IP 维度
2. 账号维度
3. 设备维度
```

只按 IP 限流，可能误伤公司或学校出口 IP。

只按账号限流，攻击者可以扫很多账号。

---

## 四、限流算法地图

本阶段会讲三类常见限流：

```text
固定窗口：INCR + EXPIRE
滑动窗口：ZSet
令牌桶/漏桶：平滑控制请求速率
```

它们的特点：

| 算法 | 优点 | 缺点 |
| --- | --- | --- |
| 固定窗口 | 简单、成本低 | 窗口边界可能突刺 |
| 滑动窗口 | 更精确 | ZSet 成本更高 |
| 令牌桶 | 允许一定突发 | 实现稍复杂 |
| 漏桶 | 输出更平滑 | 突发流量处理较硬 |

不要一开始就追求最复杂算法。

大多数后台接口，用固定窗口就足够起步。

---

## 五、计数器解决什么问题

计数器用于高频累加。

常见场景：

- 文章阅读量。
- 视频播放量。
- 点赞数。
- 评论数。
- 分享数。
- 下载次数。
- 接口调用次数。
- 库存预扣。

Redis 单条 `INCR` 是原子的：

```redis
INCR article:view_count:1001
```

很多请求同时增加阅读量时，不会丢计数。

但计数器也有工程问题：

- 是否需要实时落库。
- Redis 数据丢失后能否接受。
- 如何批量回写数据库。
- 如何处理刷量。
- 如何做去重计数。

---

## 六、排行榜解决什么问题

排行榜通常需要：

```text
按分数排序
查询 Top N
查询某个对象排名
给对象加分
```

Redis ZSet 天然适合：

```redis
ZINCRBY article:rank:daily 1 1001
ZREVRANGE article:rank:daily 0 9 WITHSCORES
ZREVRANK article:rank:daily 1001
```

典型排行榜：

- 热门文章榜。
- 视频播放榜。
- 用户积分榜。
- 商品销量榜。
- 游戏分数榜。

要注意排行榜通常有时间维度：

```text
daily
weekly
monthly
all_time
```

不同榜单不要混在一个 key 里。

---

## 七、延迟队列解决什么问题

延迟队列表示：

```text
任务不是马上执行，而是在未来某个时间点执行。
```

典型场景：

- 订单 30 分钟未支付自动取消。
- 优惠券到期提醒。
- 任务失败后 5 分钟重试。
- 定时发布文章。
- 延迟清理临时资源。

Redis ZSet 可以用 score 表示执行时间：

```redis
ZADD delay:queue 1790000000000 task-1001
```

worker 定期取出 score 小于当前时间的任务执行。

但要注意，Redis ZSet 延迟队列是轻量方案，不等于专业消息队列。

---

## 八、key 设计先行

本阶段的 key 通常要包含：

- 业务名。
- 维度。
- 目标 ID。
- 时间窗口。

示例：

```text
rate:login:ip:192.168.1.10:202607051305
rate:sms:phone:13800000000:2026070513
counter:article:view:1001
rank:article:daily:20260705
delay:order:cancel
```

命名要能让排查的人一眼看懂。

不要写成：

```text
limit:1
count:user
rank
queue
```

---

## 九、工程上要考虑什么

算法只是第一步。

真实项目还要考虑：

- 限流命中后返回什么。
- 是否记录被限流日志。
- 是否给前端返回重试时间。
- Redis 故障时放行还是拒绝。
- 是否有白名单。
- 是否区分普通用户和管理员。
- 计数器如何落库。
- 榜单如何过期和归档。
- 延迟任务失败如何重试。

这些比单纯写出 Redis 命令更重要。

---

## 十、Redis 故障时的策略

限流和计数器遇到 Redis 故障时，要先明确业务取舍。

例如：

| 场景 | Redis 故障策略 |
| --- | --- |
| 登录防爆破 | 倾向保守，可加强验证码或拒绝高风险请求 |
| 短信发送 | 倾向拒绝或走备用限制，避免成本失控 |
| 阅读量计数 | 可以降级丢弃或本地缓冲 |
| 排行榜 | 可以返回旧榜单 |
| 延迟任务 | 可以暂停消费并告警 |

不要所有 Redis 错误都直接放行。

也不要所有错误都直接拒绝。

不同业务风险不同。

---

## 十一、常见错误

### 1. 只实现算法，不设计维度

没有维度的限流很容易误伤或被绕过。

### 2. 忽略 Redis 故障策略

Redis 抖动时，接口行为会变得不可控。

### 3. 计数器永不落库

如果计数是重要业务数据，要有回写和对账。

### 4. 榜单没有时间维度

所有数据都堆到一个榜单，最后很难清理和解释。

### 5. 把 ZSet 延迟队列当成 Kafka

Redis 可以做轻量延迟队列，但不适合承载复杂消息治理。

## 十三、本节小结

这一节建立了第 7 阶段的整体地图。

你需要记住：

- 限流保护系统和成本，不只是限制用户。
- 限流要先设计维度，再选择算法。
- Redis 计数器适合高频累加，但要考虑落库和对账。
- ZSet 很适合排行榜和轻量延迟队列。
- 工程可用性比算法本身更重要。
