这一节把第 7 阶段的内容串起来。

我们设计一个 Go 后端里常见的 Redis 高频能力组合：

## 一、实践模块结构

推荐把 Redis 能力封装成独立模块：

```text
internal/redisx/
  limiter_fixed.go
  limiter_sliding.go
  counter.go
  rank.go
  delay_queue.go
```

业务模块调用这些能力：

```text
internal/auth/
internal/sms/
internal/article/
internal/order/
```

不要在 HTTP Handler 里散落 Redis 命令。

## 二、登录 IP 限流

需求：

```text
同一 IP 每分钟最多 20 次登录尝试。
```

可以用固定窗口起步。

key：

```go
func LoginIPLimitKey(ip string, now time.Time) string {
    bucket := now.Format("200601021504")
    return fmt.Sprintf("rate:login:ip:%s:%s", ip, bucket)
}
```

调用：

```go
func (s *AuthService) Login(ctx context.Context, ip string, req LoginRequest) error {
    key := LoginIPLimitKey(ip, time.Now())

    allow, current, err := s.fixedLimiter.Allow(ctx, key, 20, 2*time.Minute)
    if err != nil {
        s.logger.Printf("login rate limit failed ip=%s err=%v", ip, err)
        return ErrTryLater
    }
    if !allow {
        s.logger.Printf("login rate limited ip=%s count=%d", ip, current)
        return ErrTooManyRequests
    }

    return s.checkPassword(ctx, req)
}
```

登录是高风险接口。

Redis 限流失败时，示例里选择保守拒绝，也可以改成加强验证码。

## 三、短信验证码限流

需求：

```text
同一手机号每分钟最多 1 条。
同一手机号每小时最多 5 条。
同一手机号每天最多 10 条。
```

key：

```go
func SMSLimitKeys(phone string, now time.Time) []LimitRule {
    return []LimitRule{
        {
            Key:   fmt.Sprintf("rate:sms:phone:%s:min:%s", phone, now.Format("200601021504")),
            Limit: 1,
            TTL:   2 * time.Minute,
        },
        {
            Key:   fmt.Sprintf("rate:sms:phone:%s:hour:%s", phone, now.Format("2006010215")),
            Limit: 5,
            TTL:   2 * time.Hour,
        },
        {
            Key:   fmt.Sprintf("rate:sms:phone:%s:day:%s", phone, now.Format("20060102")),
            Limit: 10,
            TTL:   48 * time.Hour,
        },
    }
}
```

结构体：

```go
type LimitRule struct {
    Key   string
    Limit int64
    TTL   time.Duration
}
```

检查：

```go
func (s *SMSService) CheckSendLimit(ctx context.Context, phone string) error {
    for _, rule := range SMSLimitKeys(phone, time.Now()) {
        allow, current, err := s.fixedLimiter.Allow(ctx, rule.Key, rule.Limit, rule.TTL)
        if err != nil {
            s.logger.Printf("sms rate limit failed phone=%s err=%v", phone, err)
            return ErrTryLater
        }
        if !allow {
            s.logger.Printf("sms rate limited phone=%s key=%s count=%d", phone, rule.Key, current)
            return ErrTooManyRequests
        }
    }

    return nil
}
```

注意：这个简单实现会逐个计数。

如果第二个规则超限，第一个规则已经加了 1。

更严谨的实现可以用 Lua 一次检查多个 key，全部通过才增加计数。

教学阶段先理解多维度限制的思路。

## 四、更严谨的短信多规则脚本思路

生产中可以把多个规则放进一个 Lua 脚本：

```text
1. 先读取所有 key 的当前计数。
2. 只要任意一个达到上限，直接拒绝。
3. 全部未超限，再对所有 key INCR。
4. 新 key 设置 TTL。
```

这样避免部分规则已经计数、后续规则拒绝的问题。

这类脚本要保持短小，参数语义要写清楚。

## 五、文章阅读计数

阅读文章时：

```go
func (s *ArticleService) RecordView(ctx context.Context, articleID int64) error {
    key := fmt.Sprintf("counter:article:view:%d", articleID)
    deltaKey := fmt.Sprintf("counter:article:view_delta:%d", articleID)

    pipe := s.rdb.Pipeline()
    pipe.Incr(ctx, key)
    pipe.Incr(ctx, deltaKey)
    _, err := pipe.Exec(ctx)
    return err
}
```

两个 key 的含义：

```text
counter:article:view:{id}        用于快速展示
counter:article:view_delta:{id}  用于定期落库
```

如果 Redis 计数失败，可以选择：

- 不影响文章详情返回，只记录日志。
- 使用本地缓冲。
- 让前端不展示实时计数。

阅读量通常不是强一致数据。

## 六、文章热榜

阅读时同步更新日榜：

```go
func (s *ArticleService) IncrDailyRank(ctx context.Context, articleID int64) error {
    key := fmt.Sprintf("rank:article:daily:%s", time.Now().Format("20060102"))
    member := strconv.FormatInt(articleID, 10)

    pipe := s.rdb.Pipeline()
    pipe.ZIncrBy(ctx, key, 1, member)
    pipe.Expire(ctx, key, 30*24*time.Hour)
    _, err := pipe.Exec(ctx)
    return err
}
```

查询 Top 10：

```go
func (s *ArticleService) GetDailyHotArticles(ctx context.Context) ([]redis.Z, error) {
    key := fmt.Sprintf("rank:article:daily:%s", time.Now().Format("20060102"))
    return s.rdb.ZRevRangeWithScores(ctx, key, 0, 9).Result()
}
```

实际接口还要根据 articleID 批量查文章详情，并保持 ZSet 顺序。

## 七、阅读计数和热榜合并

可以用 Pipeline：

```go
func (s *ArticleService) RecordViewAndRank(ctx context.Context, articleID int64) error {
    viewKey := fmt.Sprintf("counter:article:view:%d", articleID)
    deltaKey := fmt.Sprintf("counter:article:view_delta:%d", articleID)
    rankKey := fmt.Sprintf("rank:article:daily:%s", time.Now().Format("20060102"))
    member := strconv.FormatInt(articleID, 10)

    pipe := s.rdb.Pipeline()
    pipe.Incr(ctx, viewKey)
    pipe.Incr(ctx, deltaKey)
    pipe.ZIncrBy(ctx, rankKey, 1, member)
    pipe.Expire(ctx, rankKey, 30*24*time.Hour)
    _, err := pipe.Exec(ctx)
    return err
}
```

这里追求的是高效和最终一致。

如果某一步失败，要靠日志、监控和补偿，而不是把阅读接口拖得很重。

## 八、订单取消延迟任务

创建订单后添加延迟任务：

```go
func (s *OrderService) AddCancelTask(ctx context.Context, orderID int64) error {
    queue := "delay:order:cancel"
    member := fmt.Sprintf("order:%d", orderID)
    executeAt := time.Now().Add(30 * time.Minute).UnixMilli()

    return s.rdb.ZAdd(ctx, queue, redis.Z{
        Score:  float64(executeAt),
        Member: member,
    }).Err()
}
```

worker 领取任务时用第 7 节的 Lua 脚本。

处理任务：

```go
func (s *OrderService) HandleCancelTask(ctx context.Context, task string) error {
    orderID, err := parseOrderTask(task)
    if err != nil {
        return err
    }

    affected, err := s.repo.CancelIfPending(ctx, orderID)
    if err != nil {
        return err
    }
    if affected == 0 {
        return nil
    }

    return nil
}
```

数据库层：

```sql
UPDATE orders
SET status = 'cancelled'
WHERE id = ?
  AND status = 'pending';
```

这样订单已支付时不会被取消。

## 九、Redis key 汇总

本实践涉及：

```text
rate:login:ip:{ip}:{yyyyMMddHHmm}
rate:sms:phone:{phone}:min:{yyyyMMddHHmm}
rate:sms:phone:{phone}:hour:{yyyyMMddHH}
rate:sms:phone:{phone}:day:{yyyyMMdd}
counter:article:view:{article_id}
counter:article:view_delta:{article_id}
rank:article:daily:{yyyyMMdd}
delay:order:cancel
```

这些 key 都明确表达了：

- 业务。
- 维度。
- 对象。
- 时间。

排查时会舒服很多。

## 十、监控指标

这些 Redis 能力上线后，至少要关注：

- 限流通过次数。
- 限流拒绝次数。
- Redis 限流错误次数。
- 短信发送成功和失败次数。
- 阅读计数 Redis 错误次数。
- 排行榜更新失败次数。
- 延迟任务积压数量。
- 延迟任务执行失败次数。
- 延迟任务最大延迟。

延迟队列积压可以看：

```redis
ZCARD delay:order:cancel
```

最早任务时间：

```redis
ZRANGE delay:order:cancel 0 0 WITHSCORES
```

## 十一、降级策略

建议：

| 能力 | Redis 故障策略 |
| --- | --- |
| 登录限流 | 保守拒绝或加强验证码 |
| 短信限流 | 拒绝发送，避免成本失控 |
| 阅读计数 | 记录日志，允许文章正常返回 |
| 文章热榜 | 返回旧榜单或空榜单 |
| 延迟任务 | 暂停消费并告警 |

不同业务不能共用一套降级策略。

越涉及安全和成本，越不能随便放行。

## 十二、常见错误

### 1. Handler 里直接写 Redis 命令

逻辑分散，难以测试和复用。

### 2. 短信多规则逐个 INCR 后不考虑部分成功

生产中应该用 Lua 一次检查并计数。

### 3. 文章热榜 member 存 JSON

榜单应该存 ID，详情另查。

### 4. 延迟任务不做幂等

任务重复执行时会改变已支付订单。

### 5. 没有监控限流和任务积压

出问题时不知道系统是在保护自己，还是已经坏了。

## 十四、本节小结

这一节完成了第 7 阶段的实践闭环。

你需要记住：

- 登录和短信限流要先设计维度。
- 固定窗口适合简单限流，滑动窗口和令牌桶适合更精细控制。
- 文章阅读量可以用 Redis 计数器，重要数据要落库。
- 文章热榜适合用 ZSet，member 保存 ID，score 保存分数。
- ZSet 延迟队列适合轻量按时间调度，任务处理必须幂等。
- 每个 Redis 高频能力都要有监控和降级策略。

学完第 7 阶段，你已经掌握了 Redis 在高频后端场景中的经典用法。下一阶段会进入 Bitmap、HyperLogLog、GEO、Pub/Sub 和 Stream。
