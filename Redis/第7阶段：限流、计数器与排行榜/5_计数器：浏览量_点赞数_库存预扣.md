# 5. 计数器：浏览量、点赞数、库存预扣

计数器是 Redis 最经典的用法之一。

它看起来很简单：`INCR` 一下就行。但真实业务里，计数器常常牵涉去重、落库、刷量、补偿和一致性。

学完这一节后，你应该能够：

- 使用 `INCR`、`HINCRBY`、`ZINCRBY` 实现计数。
- 区分普通累加和去重计数。
- 用 Set + Lua 实现点赞去重。
- 理解计数器落库和对账。
- 知道库存预扣和普通计数器的区别。

---

## 一、浏览量计数

最简单的浏览量：

```redis
INCR article:view_count:1001
```

读取：

```redis
GET article:view_count:1001
```

Go 写法：

```go
func IncrArticleView(ctx context.Context, rdb *redis.Client, articleID int64) error {
    key := fmt.Sprintf("counter:article:view:%d", articleID)
    return rdb.Incr(ctx, key).Err()
}
```

这适合对精确性要求不高的展示型数据。

---

## 二、批量计数可以用 Hash

如果一篇文章有多个计数字段：

```text
view_count
like_count
comment_count
share_count
```

可以用 Hash：

```redis
HINCRBY article:stats:1001 view_count 1
HINCRBY article:stats:1001 like_count 1
HGETALL article:stats:1001
```

好处：

- 同一对象多个计数字段集中管理。
- 读取文章统计时可以一次取回。

缺点：

- 单个 Hash 过大时仍然是 Big Key 风险。
- 不适合把全站所有文章都放进一个 Hash。

---

## 三、计数器要不要设置 TTL

看业务。

临时计数可以设置 TTL：

```text
接口调用次数
验证码发送次数
活动临时点击数
```

长期计数通常不设置短 TTL：

```text
文章总阅读量
视频总播放量
用户总积分
```

但长期计数要考虑：

- Redis 持久化是否可靠。
- 是否需要写回数据库。
- Redis 丢失后是否可以重算。

重要计数不要只放 Redis。

---

## 四、计数器落库

常见模式：

```text
1. 请求实时写 Redis 计数。
2. 后台任务定期读取 Redis 增量。
3. 批量更新数据库。
4. 成功后清理或归零增量 key。
```

例如使用增量 key：

```text
counter:article:view_delta:1001
```

每次阅读：

```redis
INCR counter:article:view_delta:1001
```

后台每分钟：

```text
读取 delta
数据库 view_count += delta
清理 delta
```

清理时要注意并发写入，不能简单 `GET` 后 `DEL`。

可以用 `GETDEL`，或者 Lua 原子取出并删除。

---

## 五、原子取出增量

Redis 6.2+ 支持：

```redis
GETDEL counter:article:view_delta:1001
```

含义：

```text
读取 value，然后删除 key。
```

如果使用 Lua：

```lua
local value = redis.call("GET", KEYS[1])
if value then
    redis.call("DEL", KEYS[1])
    return value
end
return 0
```

这样可以避免：

```text
GET 后还没 DEL，新请求又 INCR
DEL 把新增量删掉
```

---

## 六、点赞计数不能只 INCR

点赞不是普通累加。

同一个用户对同一篇文章通常只能点赞一次。

错误写法：

```redis
INCR article:like_count:1001
```

如果用户重复点击，就会重复增加。

应该保存点赞关系：

```text
article:liked_users:1001
```

使用 Set：

```redis
SADD article:liked_users:1001 2001
```

`SADD` 返回 1 表示新点赞，返回 0 表示已经点过赞。

---

## 七、点赞 Set + 计数器

点赞时：

```text
1. SADD liked_users user_id
2. 如果返回 1，点赞数 +1
3. 如果返回 0，不重复增加
```

这两步要用 Lua：

```lua
local added = redis.call("SADD", KEYS[1], ARGV[1])
if added == 1 then
    redis.call("INCR", KEYS[2])
end
return added
```

参数：

```text
KEYS[1]：点赞用户 Set
KEYS[2]：点赞数 key
ARGV[1]：用户 ID
```

---

## 八、取消点赞

取消点赞也要去重：

```lua
local removed = redis.call("SREM", KEYS[1], ARGV[1])
if removed == 1 then
    redis.call("DECR", KEYS[2])
end
return removed
```

只有用户原来点过赞，才减少点赞数。

否则重复取消会把计数减错。

---

## 九、Go 封装点赞

```go
var likeScript = redis.NewScript(`
local added = redis.call("SADD", KEYS[1], ARGV[1])
if added == 1 then
    redis.call("INCR", KEYS[2])
end
return added
`)
```

```go
func LikeArticle(ctx context.Context, rdb *redis.Client, articleID, userID int64) (bool, error) {
    usersKey := fmt.Sprintf("article:liked_users:%d", articleID)
    countKey := fmt.Sprintf("counter:article:like:%d", articleID)

    n, err := likeScript.Run(ctx, rdb, []string{usersKey, countKey}, userID).Int()
    if err != nil {
        return false, err
    }
    return n == 1, nil
}
```

返回 `true` 表示本次确实新增点赞。

## 十、库存预扣不是普通计数器

库存预扣看起来像：

```redis
DECR stock:sku:2001
```

但不能简单当成普通计数。

库存要保证：

- 不能扣成负数。
- 创建订单失败要补偿。
- 重复请求要幂等。
- Redis 和数据库要对账。

第 6 阶段已经讲过，应该用 Lua 判断库存后扣减：

```text
库存大于 0 才 DECR
```

并且配合订单幂等和补偿。

---

## 十一、计数器防刷

阅读量、播放量这类计数容易被刷。

常见处理：

- 按用户去重。
- 按 IP 限制频率。
- 按设备限制。
- 同一用户短时间只计一次。
- 异常流量进入风控。

示例：

```text
article:viewed:user:1001:20260705
```

用 Set 记录用户当天是否看过某文章。

但去重越严格，成本越高。

要根据业务决定是否需要精确去重。

---

## 十二、计数器和排行榜联动

文章阅读量增加后，可能还要更新排行榜：

```redis
INCR counter:article:view:1001
ZINCRBY rank:article:daily:20260705 1 1001
```

如果希望两步原子，可以用 Lua。

但要注意：

```text
Redis 内部原子，不等于数据库已经同步。
```

排行榜通常允许最终一致。

---

## 十三、常见错误

### 1. 点赞只用 INCR

重复点赞会让计数不准。

### 2. 取消点赞直接 DECR

重复取消会把计数减成负数。

### 3. 增量落库用 GET 后 DEL

并发写入时可能丢增量。

### 4. 重要计数只放 Redis

Redis 故障或数据丢失会影响业务。

### 5. 把库存预扣当普通 DECR

库存需要防负数、幂等、补偿和对账。

## 十五、本节小结

这一节你学习了 Redis 计数器。

你需要记住：

- 普通累加可以用 `INCR`、`HINCRBY`。
- 多字段统计可以用 Hash。
- 重要计数要考虑落库、对账和恢复。
- 点赞这类去重计数要用 Set + Lua。
- 库存预扣要防负数、幂等和补偿，不能只靠 `DECR`。
