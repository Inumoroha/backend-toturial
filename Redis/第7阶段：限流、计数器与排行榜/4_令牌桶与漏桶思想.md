# 4. 令牌桶与漏桶思想

固定窗口和滑动窗口都在回答一个问题：

```text
一段时间内请求次数是否超过限制？
```

令牌桶和漏桶更关注请求速率。

它们常用于保护下游服务，让流量更平滑，或允许有限突发。

学完这一节后，你应该能够：

- 理解令牌桶和漏桶的区别。
- 判断什么时候使用令牌桶。
- 用 Redis + Lua 实现简单令牌桶。
- 知道令牌桶实现中的时间和容量问题。
- 理解这些算法的工程取舍。

---

## 一、令牌桶是什么

令牌桶可以想象成一个桶：

```text
桶最多装 capacity 个令牌。
系统按固定速率往桶里放令牌。
每个请求消耗一个令牌。
有令牌就放行，没有令牌就拒绝或等待。
```

例如：

```text
桶容量：10
生成速率：每秒 2 个令牌
```

含义：

- 平均每秒允许 2 个请求。
- 如果一段时间没有请求，桶里最多攒 10 个令牌。
- 突发最多允许 10 个请求快速通过。

---

## 二、令牌桶适合什么场景

适合：

- 允许短暂突发。
- 需要控制平均速率。
- 下游服务能承受一定瞬时流量。
- API 调用配额。
- 用户操作频率控制。

例如：

```text
每个用户每秒平均 5 次请求，最多允许瞬间突发 20 次。
```

这比固定窗口更平滑，也比纯滑动窗口更适合突发友好的业务。

---

## 三、漏桶是什么

漏桶可以想象成：

```text
请求先进入桶。
桶按固定速度漏出请求。
桶满了，新请求被拒绝。
```

它强调输出速率稳定。

例如：

```text
每秒只处理 10 个请求，多余请求排队或丢弃。
```

漏桶适合保护处理能力固定的下游。

缺点是对突发不够友好，请求可能排队等待。

---

## 四、令牌桶和漏桶区别

| 算法 | 特点 |
| --- | --- |
| 令牌桶 | 控制平均速率，允许突发 |
| 漏桶 | 控制输出速率，平滑处理 |

简单理解：

```text
令牌桶：攒许可证，请求拿到许可证就走。
漏桶：请求排队，系统按固定速度处理。
```

在 Redis 接口限流里，令牌桶更常见。

漏桶更像队列调度问题，常会结合消息队列或本地队列实现。

---

## 五、令牌桶需要保存什么

每个限流对象需要保存：

```text
tokens：当前令牌数
last_refill_ms：上次补充令牌时间
```

可以用 Redis Hash：

```text
rate:bucket:user:1001
```

字段：

```text
tokens = 8
last_refill_ms = 1790000000000
```

每次请求时，根据当前时间计算应该补充多少令牌。

---

## 六、令牌补充公式

假设：

```text
rate = 每秒生成 5 个令牌
capacity = 桶容量 20
```

当前时间距离上次补充过去了 2 秒：

```text
new_tokens = 2 * 5 = 10
tokens = min(capacity, old_tokens + new_tokens)
```

然后请求消耗 1 个令牌：

```text
if tokens >= 1:
    tokens = tokens - 1
    allow
else:
    reject
```

## 七、Redis Lua 实现令牌桶

```lua
local key = KEYS[1]
local now = tonumber(ARGV[1])
local rate = tonumber(ARGV[2])
local capacity = tonumber(ARGV[3])
local requested = tonumber(ARGV[4])
local ttl = tonumber(ARGV[5])

local bucket = redis.call("HMGET", key, "tokens", "last_refill_ms")
local tokens = tonumber(bucket[1])
local last_refill = tonumber(bucket[2])

if tokens == nil then
    tokens = capacity
    last_refill = now
end

local delta = math.max(0, now - last_refill)
local refill = delta * rate / 1000
tokens = math.min(capacity, tokens + refill)

local allowed = 0
if tokens >= requested then
    allowed = 1
    tokens = tokens - requested
end

redis.call("HMSET", key, "tokens", tokens, "last_refill_ms", now)
redis.call("EXPIRE", key, ttl)

return {allowed, tokens}
```

参数：

```text
rate：每秒生成令牌数
capacity：桶容量
requested：本次请求消耗令牌数，通常为 1
ttl：key 过期秒数
```

## 八、Go 封装令牌桶

```go
var tokenBucketScript = redis.NewScript(`
local key = KEYS[1]
local now = tonumber(ARGV[1])
local rate = tonumber(ARGV[2])
local capacity = tonumber(ARGV[3])
local requested = tonumber(ARGV[4])
local ttl = tonumber(ARGV[5])

local bucket = redis.call("HMGET", key, "tokens", "last_refill_ms")
local tokens = tonumber(bucket[1])
local last_refill = tonumber(bucket[2])

if tokens == nil then
    tokens = capacity
    last_refill = now
end

local delta = math.max(0, now - last_refill)
local refill = delta * rate / 1000
tokens = math.min(capacity, tokens + refill)

local allowed = 0
if tokens >= requested then
    allowed = 1
    tokens = tokens - requested
end

redis.call("HMSET", key, "tokens", tokens, "last_refill_ms", now)
redis.call("EXPIRE", key, ttl)

return {allowed, tokens}
`)
```

```go
func AllowByTokenBucket(ctx context.Context, rdb *redis.Client, key string, rate, capacity, requested int64) (bool, float64, error) {
    now := time.Now().UnixMilli()
    ttl := int64(3600)

    result, err := tokenBucketScript.Run(ctx, rdb, []string{key}, now, rate, capacity, requested, ttl).Result()
    if err != nil {
        return false, 0, err
    }

    values := result.([]interface{})
    allowed := values[0].(int64) == 1
    tokens, _ := strconv.ParseFloat(fmt.Sprint(values[1]), 64)

    return allowed, tokens, nil
}
```

Lua 中可能返回小数 token，所以 Go 里要注意类型转换。

## 九、是否允许小数令牌

上面的实现允许小数令牌。

例如：

```text
tokens = 3.42
```

这样补充更平滑。

如果你只想用整数令牌，可以对 refill 取整：

```lua
local refill = math.floor(delta * rate / 1000)
```

但取整会让低速率限流不够平滑。

例如每 10 秒 1 个令牌，用小数更自然。

---

## 十、时间来源问题

上面的实现使用客户端时间：

```go
time.Now().UnixMilli()
```

如果多台服务器时间不一致，可能影响令牌补充。

更严谨的方式是用 Redis `TIME` 命令。

但在 Lua 中使用 `TIME` 会让脚本依赖 Redis 服务器时间。

大多数业务只要服务器使用 NTP 同步，客户端时间就够用。

如果是强风控场景，需要更谨慎。

## 十一、令牌桶和固定窗口怎么选

简单选择：

| 场景 | 推荐 |
| --- | --- |
| 短信每天最多 10 次 | 固定窗口 |
| 登录最近 60 秒最多 20 次 | 滑动窗口 |
| API 平均每秒 100 次，允许突发 200 次 | 令牌桶 |
| 下游每秒只能稳定处理 50 个任务 | 漏桶或队列 |

没有绝对最好的算法。

看业务是否需要：

- 简单。
- 精确。
- 平滑。
- 允许突发。

---

## 十二、常见错误

### 1. 桶容量设置太大

会允许过高突发，打垮下游。

### 2. 桶容量设置太小

业务正常抖动也会频繁被限流。

### 3. 忽略时间同步

多实例时间差太大会影响限流准确性。

### 4. Redis Hash 不设置 TTL

冷门用户的桶会长期残留。

### 5. 把漏桶强行塞进 Redis 单脚本

真正排队处理通常需要队列和 worker 配合。
