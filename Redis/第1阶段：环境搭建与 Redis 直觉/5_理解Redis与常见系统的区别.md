# 5. 理解 Redis 与 MySQL、PostgreSQL、MongoDB、Kafka 的区别

学 Redis 时，很容易把它和数据库、消息队列混在一起：Redis 能存数据，MySQL 也能存数据；Redis 有 Stream，Kafka 也能做消息；Redis 可以保存 JSON，MongoDB 也能保存文档。

这一节的目标不是讲每个系统的全部细节，而是建立边界感：它们分别擅长什么，不擅长什么，真实项目里应该如何搭配。

学完这一节后，你应该能够：

- 说清 Redis 和关系型数据库的大致区别。
- 说清 Redis 和 MongoDB 的大致区别。
- 说清 Redis 和 Kafka 的大致区别。
- 知道 Redis 在 Go 后端项目中常见的位置。
- 避免把 Redis 当成所有问题的答案。

---

## 一、先给 Redis 一个定位

Redis 常见定位是：高性能内存数据结构服务。

它经常用来做：

- 缓存。
- 验证码。
- 登录 token。
- 计数器。
- 限流。
- 排行榜。
- 分布式锁基础能力。
- 简单队列或轻量消息。
- 短期状态存储。

你可以把 Redis 放在后端系统的加速层或协作层，而不是默认把它当成核心业务数据库。

典型结构是：

```text
Go 服务 -> Redis
Go 服务 -> MySQL 或 PostgreSQL
```

核心数据进数据库，高频读取和临时状态进 Redis。

---

## 二、Redis 与 MySQL 的区别

MySQL 是关系型数据库，常用于保存核心业务数据。

比如：

- 用户表。
- 订单表。
- 商品表。
- 支付流水表。
- 地址表。

Redis 更常用于保存缓存和临时状态。

比如：

- 商品详情缓存。
- 短信验证码。
- 登录 token。
- 接口限流计数。
- 商品浏览量计数。

简单对比：

| 维度 | Redis | MySQL |
| --- | --- | --- |
| 主要定位 | 内存数据结构、缓存、临时状态 | 关系型数据库、核心业务数据 |
| 数据模型 | key-value，多种 value 结构 | 表、行、列、索引、关系 |
| 查询方式 | 主要通过 key 访问 | SQL 查询能力强 |
| 事务能力 | 支持有限事务和 Lua 原子操作 | 完整事务能力成熟 |
| 持久化 | 支持，但通常不是唯一数据源 | 核心能力之一 |
| 常见用途 | 缓存、计数、限流、排行榜 | 订单、用户、支付、库存主数据 |

一个常见例子：文章详情页。

MySQL 保存文章主数据：

```text
articles 表：id、title、content、author_id、created_at
```

Redis 保存文章缓存：

```text
article:detail:1001
```

请求流程：

```text
先查 Redis
命中则直接返回
未命中再查 MySQL
查到后写回 Redis
```

这就是后面会重点学习的 Cache-Aside Pattern。

---

## 三、Redis 与 PostgreSQL 的区别

PostgreSQL 也是关系型数据库，但功能比传统印象中的关系型数据库更丰富。它支持强大的 SQL、事务、索引、JSONB、全文搜索、地理扩展等能力。

在 Go 后端项目里，PostgreSQL 很适合作为核心数据库。

Redis 和 PostgreSQL 的区别可以这样理解：

| 维度 | Redis | PostgreSQL |
| --- | --- | --- |
| 主要强项 | 高速 key-value 访问、内存数据结构 | 强一致事务、复杂查询、数据完整性 |
| 数据关系 | 不擅长复杂关联查询 | 擅长 JOIN、约束、事务 |
| 数据可靠性 | 可持久化，但常作为缓存层 | 通常作为权威数据源 |
| 查询语言 | Redis 命令 | SQL |
| 使用方式 | 按 key 设计访问路径 | 按表结构、索引和查询设计 |

比如短链接项目：

PostgreSQL 保存短链接主数据：

```text
short_links 表：id、code、original_url、created_at、expired_at
```

Redis 缓存短码到原始链接的映射：

```text
shortlink:detail:abc123
```

当用户访问短链时，Redis 可以减少 PostgreSQL 查询压力。但最终可靠数据仍然应该在 PostgreSQL 里。

---

## 四、Redis 与 MongoDB 的区别

MongoDB 是文档数据库，适合保存 JSON-like 文档。

它常用于：

- 文档结构灵活的数据。
- 字段变化较多的业务对象。
- 内容管理系统。
- 日志或事件类文档。
- 不需要强关系建模的场景。

Redis 也可以保存 JSON 字符串，但这不代表 Redis 就等于 MongoDB。

对比：

| 维度 | Redis | MongoDB |
| --- | --- | --- |
| 主要定位 | 内存数据结构、缓存、临时状态 | 文档数据库 |
| 数据访问 | 以 key 访问为主 | 按字段、条件、索引查询文档 |
| 数据容量 | 受内存成本影响明显 | 更适合保存大量持久化文档 |
| 查询能力 | 不适合复杂条件查询 | 支持丰富文档查询 |
| 常见用途 | 热点缓存、排行榜、限流 | 商品文档、内容文档、灵活结构数据 |

比如商品详情：

MongoDB 可以保存完整商品文档：

```json
{
  "id": 1001,
  "name": "机械键盘",
  "attrs": {
    "switch": "red",
    "layout": "87"
  }
}
```

Redis 可以缓存这个商品详情 JSON：

```text
product:detail:1001
```

MongoDB 是数据源，Redis 是缓存层。二者可以搭配，不是互相完全替代。

---

## 五、Redis 与 Kafka 的区别

Kafka 是分布式消息系统，擅长高吞吐、可持久化、可回放的消息流。

常见用途：

- 日志采集。
- 用户行为事件流。
- 订单事件异步处理。
- 跨系统消息解耦。
- 大数据实时处理链路。

Redis 也有 Pub/Sub 和 Stream，但它们和 Kafka 的定位不同。

对比：

| 维度 | Redis | Kafka |
| --- | --- | --- |
| 主要定位 | 缓存、内存数据结构、轻量消息 | 分布式消息流平台 |
| 消息持久化 | Stream 支持持久化，Pub/Sub 不可靠 | 核心能力之一 |
| 消息回放 | Stream 有一定能力 | 强项，按 offset 回放 |
| 吞吐和扩展 | 适合轻量场景 | 面向高吞吐和大规模分区 |
| 消费模型 | Pub/Sub、Stream 消费者组 | Consumer Group、Partition、Offset |
| 常见用途 | 本地通知、小队列、延迟任务雏形 | 日志、事件流、跨服务消息 |

简单理解：

- Redis Pub/Sub 更像轻量通知，订阅者不在线时消息容易丢。
- Redis Stream 更像轻量消息队列，支持消费者组和确认。
- Kafka 更适合大规模、可回放、跨服务治理的消息流。

如果只是本地开发或小规模异步任务，Redis Stream 可能够用。

如果是订单事件、支付事件、日志流、用户行为流，通常应该认真考虑 Kafka 或其他专业 MQ。

## 六、一个真实后端项目里它们如何搭配

以短链接服务为例。

核心功能：

- 创建短链接。
- 根据短码跳转。
- 统计访问次数。
- 限制用户创建频率。
- 记录访问日志。

可以这样分工：

| 系统 | 负责内容 |
| --- | --- |
| PostgreSQL 或 MySQL | 保存短链接主数据、用户、权限、配置 |
| Redis | 缓存短码映射、访问计数、限流、空值缓存 |
| Kafka | 异步发送访问事件、日志、统计流水 |
| MongoDB | 如果业务需要，可保存灵活结构的分析文档 |

访问短链时：

```text
用户请求 /abc123
Go 服务先查 Redis: shortlink:detail:abc123
命中则直接跳转
未命中则查 PostgreSQL
查到后写回 Redis
同时发送访问事件到 Kafka
```

Redis 在这里的价值是减少数据库压力，提高热点访问速度。

但短链接的权威数据仍然在 PostgreSQL 或 MySQL 中。

## 七、什么时候该用 Redis

你可以用下面几个问题判断。

### 1. 这个数据访问频率高吗

如果某个数据被频繁读取，比如热门文章详情、热门商品详情，适合缓存到 Redis。

### 2. 这个数据是否可以短时间不一致

缓存通常意味着可能读到短时间旧数据。

如果业务能接受最终一致，就适合 Redis 缓存。

如果必须强一致，比如支付状态判断，就要谨慎。

### 3. 这个数据是否天然有过期时间

验证码、登录 token、限流计数都很适合 Redis，因为它们天然需要 TTL。

### 4. 是否需要 Redis 的数据结构

排行榜适合 ZSet。

去重适合 Set。

计数适合 String 的 `INCR`。

队列可以考虑 List 或 Stream。

如果 Redis 的数据结构刚好匹配业务，会比自己在数据库里硬凑更自然。

---

## 八、什么时候不该急着用 Redis

下面情况不要急着上 Redis：

- 数据访问量很低，数据库完全扛得住。
- 数据一致性要求极高。
- 团队还没有缓存失效和故障降级方案。
- 只是为了“看起来高并发”而加缓存。
- 核心业务数据没有可靠数据库兜底。

Redis 能提高性能，也会引入复杂度：

- 缓存和数据库一致性。
- key 设计和过期策略。
- 缓存穿透、击穿、雪崩。
- Redis 故障时的降级。
- 内存成本和淘汰策略。

所以 Redis 是工具，不是装饰品。

## 九、一个简单记忆表

| 场景 | 更常用的选择 |
| --- | --- |
| 保存订单、支付、用户主数据 | MySQL / PostgreSQL |
| 缓存文章详情、商品详情 | Redis |
| 保存灵活文档 | MongoDB |
| 接口限流、验证码、登录 token | Redis |
| 排行榜 | Redis ZSet |
| 大规模事件流、日志流 | Kafka |
| 轻量异步任务或本地通知 | Redis Stream / Pub/Sub |
| 复杂 SQL 查询和事务 | MySQL / PostgreSQL |

这张表不是绝对规则，但足够帮助你在初学阶段避免混乱。

## 十、本节练习

请思考下面几个问题：

1. 用户注册信息应该只放 Redis 吗？为什么？
2. 短信验证码适合放 Redis 吗？为什么？
3. 热门文章详情为什么适合缓存到 Redis？
4. 支付流水为什么更适合放 MySQL 或 PostgreSQL？
5. 用户行为日志流为什么可能适合 Kafka？
6. 商品详情文档为什么可能适合 MongoDB？
7. 如果 Redis 挂了，你的 Go 服务应该直接崩溃，还是降级查数据库？
8. 如果缓存中的文章详情比数据库旧 10 秒，业务能不能接受？

这些问题没有统一答案，关键是你要开始从业务特点出发选择工具。

## 十一、本节小结

这一节你建立了 Redis 和常见系统的边界感。

你需要记住：

- Redis 常见定位是缓存、临时状态、高性能数据结构服务。
- MySQL 和 PostgreSQL 更适合保存核心业务数据。
- MongoDB 更适合保存灵活的文档数据。
- Kafka 更适合大规模、可回放、高吞吐的消息流。
- Redis 可以持久化，也有消息能力，但不代表它应该替代数据库或 Kafka。
- 真实项目里常见做法是组合使用：数据库保存权威数据，Redis 提速，Kafka 解耦事件流。

学到这里，第一阶段的直觉已经初步建立：你能启动 Redis、连接 Redis、观察数据，理解 key-value 模型，也知道 Redis 快在哪里，以及它和其他系统的边界在哪里。
