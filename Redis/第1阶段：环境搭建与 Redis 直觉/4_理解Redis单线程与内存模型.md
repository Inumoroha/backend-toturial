# 4. 理解 Redis 单线程主循环与内存模型

前面我们已经知道 Redis 是一个 key-value 系统。这一节继续理解另一个重要直觉：Redis 大部分操作都在内存中完成，并且核心命令处理流程长期以单线程事件循环为主。

这一节不是讲源码，而是帮你建立使用 Redis 时必须具备的性能直觉。

学完这一节后，你应该能够：

- 理解 Redis 为什么通常很快。
- 理解“单线程”指的是什么。
- 知道单线程不等于 Redis 只能做一件事。
- 知道慢命令为什么会影响其他请求。
- 理解 Big Key 和阻塞操作的风险。

---

## 一、Redis 为什么快

Redis 快，主要有几个原因：

1. 大部分数据在内存中读写。
2. 常用数据结构实现高效。
3. 核心命令处理流程简单。
4. 使用事件循环处理大量连接。
5. 避免了多线程竞争带来的复杂锁开销。

你可以先把 Redis 理解为：一个把常用数据结构放进内存里，并通过网络命令访问的高性能服务。

比如执行：

```redis
GET user:1:name
```

Redis 通常是在内存里的哈希表中找到这个 key，然后返回 value。这个过程不需要像数据库那样频繁访问磁盘页。

---

## 二、内存读写是什么意思

传统数据库的数据通常以磁盘为主要存储，再通过缓存、索引、缓冲池提高性能。

Redis 的常用工作方式是：数据主要放在内存中。

内存访问速度远高于磁盘访问速度，所以 Redis 很适合：

- 缓存热点数据。
- 保存短期临时数据。
- 做高频计数。
- 做限流状态。
- 做排行榜。

但这也带来一个问题：内存比磁盘贵，而且容量更有限。

所以使用 Redis 时要关心：

- key 数量。
- value 大小。
- 是否设置 TTL。
- 是否有 Big Key。
- 内存淘汰策略。
- 持久化是否符合预期。

Redis 快，不代表你可以无限塞数据。

---

## 三、单线程到底是什么意思

很多教程会说 Redis 是单线程的。这个说法容易让初学者误解。

更准确地说：Redis 的核心命令执行长期主要由一个主线程串行处理。

也就是说，当 Redis 执行命令时，大体像这样：

```text
接收请求 -> 解析命令 -> 执行命令 -> 返回结果
```

这些命令会进入一个事件循环，由主线程一个一个处理。

比如同时来了三个命令：

```text
客户端 A: GET user:1
客户端 B: SET count 1
客户端 C: INCR page:view
```

Redis 会很快地依次处理它们，而不是让多个线程同时修改同一份数据。

这带来的好处是：

- 单条命令天然具备原子性。
- 避免大量加锁和线程切换成本。
- 数据结构操作更简单可控。

---

## 四、单线程不等于不能处理并发

单线程主循环不代表 Redis 只能服务一个客户端。

Redis 使用网络事件模型，可以同时维护大量客户端连接。它会不断处理已经就绪的网络事件和命令。

你可以把它想象成一个动作很快的窗口工作人员：

```text
取一个请求 -> 处理 -> 返回
取下一个请求 -> 处理 -> 返回
```

只要每个请求都很短，它就能在单位时间内处理非常多的请求。

问题出在某个请求太慢。

如果一个命令执行很久，后面的命令就要排队等待。

---

## 五、为什么慢命令危险

因为核心命令串行执行，所以一个慢命令可能拖住后面的请求。

比如下面这些操作都要谨慎：

- 对超大 key 执行复杂操作。
- 使用 `KEYS *` 扫描全库。
- 一次删除特别大的 List、Set、ZSet。
- 执行复杂 Lua 脚本。
- 一次读取或写入特别大的 value。

举个例子：

```redis
KEYS *
```

在本地学习时没什么问题，因为 key 很少。

但如果生产环境有几百万个 key，`KEYS *` 可能会让 Redis 忙着扫描全库，其他正常请求也要等。

更推荐使用：

```redis
SCAN 0 MATCH user:* COUNT 100
```

`SCAN` 是渐进式遍历，每次只扫一部分，更适合线上排查。

---

## 六、什么是 Big Key

Big Key 指的是 value 很大的 key。

比如：

- 一个 String 保存了几 MB 甚至几十 MB 的 JSON。
- 一个 List 里有几十万条元素。
- 一个 Set 里有上百万个成员。
- 一个 Hash 里有几十万个字段。

Big Key 的问题不只是占内存，还会影响命令执行时间、网络传输、删除耗时和持久化成本。

比如：

```redis
GET article:huge
```

如果 value 很大，Redis 要读取它、拷贝它、通过网络发给客户端。这个过程会比普通小 key 慢很多。

再比如：

```redis
DEL huge:set
```

删除一个特别大的集合，也可能造成明显阻塞。

后面学习生产排查时，会讲 `MEMORY USAGE`、`SLOWLOG`、Big Key 发现和治理。

当前阶段先记住：Redis 喜欢小而快的操作，不喜欢又大又慢的操作。

---

## 七、单条命令的原子性

因为 Redis 主线程串行执行命令，所以单条命令执行期间不会被其他命令插入。

比如：

```redis
INCR article:view_count:1001
```

即使很多客户端同时执行这个命令，Redis 也会一个一个处理，计数不会出现两个请求读到同一个旧值再覆盖的问题。

这就是为什么 Redis 很适合做计数器。

但要注意：多条命令组合起来不自动具备原子性。

比如：

```redis
GET stock:1001
DECR stock:1001
```

这两条命令之间可能插入其他客户端的命令。后面学习 Lua 和事务时，会专门讲如何处理多步原子操作。

---

## 八、持久化不等于每次都写磁盘

Redis 支持持久化，例如 RDB 和 AOF。

但这不改变一个基本事实：Redis 的高性能主要来自内存操作。

如果开启 AOF，Redis 会把写命令追加到日志文件中，用于重启恢复。但根据配置不同，它不一定每条命令都立刻强制刷盘。

所以你要区分两个概念：

| 概念 | 含义 |
| --- | --- |
| 内存数据 | Redis 正在服务请求时使用的数据 |
| 持久化文件 | 用于重启恢复的数据文件 |

Redis 可以持久化，但它不是“所有写入都像数据库事务日志一样强保证”的万能存储。是否允许丢数据、允许丢多少，要根据业务和配置判断。

---

## 九、从 Go 后端角度理解 Redis

在 Go 服务里访问 Redis，通常是这样的链路：

```text
HTTP 请求 -> Go 服务 -> Redis 客户端 -> 网络 -> Redis -> 返回结果
```

Redis 自己很快，但整个链路还包括：

- 网络开销。
- JSON 序列化和反序列化。
- Go 客户端连接池。
- context 超时设置。
- value 大小。
- Redis 当前负载。

所以不要只说“Redis 很快”。更准确的说法是：Redis 单次小数据结构操作通常很快，但你仍然要控制访问方式和数据大小。

---

## 十、如何写出对 Redis 友好的操作

初学阶段可以先遵守这些原则：

- key 命名清晰。
- value 不要过大。
- 临时数据设置 TTL。
- 不在生产环境使用 `KEYS *`。
- 不随意执行大范围删除。
- 不把复杂计算塞进 Lua 脚本。
- 高频接口设置合理超时。
- 批量操作考虑 Pipeline。

比如缓存文章详情时，不要把一个用户所有文章详情都塞进一个巨大 JSON。

更合理的是：

```text
article:detail:1001
article:detail:1002
article:detail:1003
```

每篇文章一个 key，更容易设置 TTL、删除和更新。

请完成下面练习：

## 十二、本节小结

这一节你建立了 Redis 的性能直觉。

你需要记住：

- Redis 大部分常用操作在内存中完成，所以通常很快。
- Redis 核心命令处理长期以单线程主循环为主。
- 单线程不代表不能处理并发，它依靠事件循环服务大量连接。
- 单条命令具有原子性，多条命令组合不自动原子。
- 慢命令、Big Key、全库扫描可能阻塞其他请求。
- Redis 快，但依然需要控制 key 数量、value 大小、TTL 和访问模式。
