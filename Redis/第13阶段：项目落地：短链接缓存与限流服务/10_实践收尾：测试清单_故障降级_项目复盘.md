# 10. 实践收尾：测试清单、故障降级、项目复盘

这一节收尾第 13 阶段，也收尾整条 Redis 学习路线。

项目不是代码写完就结束。你还需要测试主链路、模拟故障、检查缓存一致性，并复盘自己是否真正掌握了 Redis 在后端项目中的使用边界。

学完这一节后，你应该能够：

- 为短链接项目设计测试清单。
- 模拟 Redis 故障和数据库故障。
- 设计关键接口降级策略。
- 检查缓存一致性和限流效果。
- 复盘第 13 阶段覆盖的 Redis 能力。

## 一、功能测试清单

创建短链接：

```text
正常 URL 可以创建
非法 URL 被拒绝
同一用户超过创建频率返回 429
短码唯一
创建后返回 short_url
```

跳转短链接：

```text
存在的 code 返回 302
不存在的 code 返回 404
禁用的 code 不跳转
过期的 code 返回 410
非法 code 不查数据库
```

查询和更新：

```text
GET /links/:code 返回详情
PATCH /links/:code 可以更新 URL
更新后再次跳转使用新 URL
更新禁用后不再跳转
```

统计：

```text
跳转后访问次数增加
新链接访问次数默认为 0
统计接口能返回 code 和 visit_count
```

---

## 二、Redis key 检查清单

用 `redis-cli` 检查：

```redis
GET shortlink:detail:aB9xK2
GET shortlink:visit_count:aB9xK2
TTL shortlink:create_limit:1001
GET shortlink:not_found:unknownCode
```

检查点：

- 详情缓存是 JSON。
- 详情缓存有 TTL。
- 限流 key 有 TTL。
- 空值缓存有短 TTL。
- 访问计数正确递增。

不要在生产中使用 `KEYS shortlink:*`。

排查可以使用 `SCAN`。

---

## 三、缓存命中测试

测试流程：

```text
1. 创建短链接
2. 第一次访问短码
3. 观察数据库查询日志
4. 第二次访问短码
5. 确认第二次命中 Redis
```

可以在代码里临时记录：

```text
cache_hit=true code=aB9xK2
cache_hit=false code=aB9xK2
```

长期建议用指标记录命中率。

---

## 四、空值缓存测试

测试流程：

```text
1. 请求不存在的 code
2. 数据库查询一次
3. Redis 写入 shortlink:not_found:{code}
4. 再次请求同一个 code
5. 不再查数据库，直接返回 404
6. TTL 到期后再次回源数据库
```

空值缓存不是为了永久记住不存在。

它是为了挡住短时间内的重复穿透。

---

## 五、更新一致性测试

测试流程：

```text
1. 创建短链接，原始 URL 为 A
2. 访问一次，让详情进入 Redis
3. PATCH 更新原始 URL 为 B
4. 检查 shortlink:detail:{code} 已删除
5. 再次访问短码
6. 确认跳转到 B
```

还要测试：

- 更新状态为 disabled。
- 更新过期时间。
- 删除缓存失败时日志可见。
- TTL 到期后旧缓存能自然消失。

---

## 六、限流测试

测试流程：

```text
1. 设置每分钟最多创建 3 个
2. 连续请求 4 次 POST /links
3. 前 3 次成功
4. 第 4 次返回 429
5. 检查 shortlink:create_limit:{user_id} 的 TTL
6. TTL 到期后再次允许创建
```

检查 Redis：

```redis
GET shortlink:create_limit:1001
TTL shortlink:create_limit:1001
```

限流不是只看接口返回，也要看 key 是否正确过期。

## 七、故障降级策略

短链接项目的降级表：

| 场景 | 策略 |
| --- | --- |
| Redis 详情缓存读失败 | 查数据库 |
| Redis 详情缓存写失败 | 记录日志，继续返回 |
| Redis 空值缓存失败 | 记录日志，继续查数据库 |
| Redis 访问计数失败 | 记录日志，继续跳转 |
| Redis 创建限流失败 | 保守拒绝创建 |
| Redis 统计读取失败 | 返回详情，统计标记不可用 |
| 数据库查询失败 | 返回 500 或服务暂不可用 |
| 数据库写入失败 | 创建或更新失败 |

核心原则：

```text
跳转优先可用。
写接口保守保护。
统计可以降级。
缓存失败不等于业务失败。
```

## 八、压测观察指标

可以用简单压测工具请求跳转接口。

观察：

- QPS。
- P95/P99 延迟。
- Redis 命中率。
- 数据库 QPS。
- Redis `cmdstat_get`、`cmdstat_incr`。
- `keyspace_hits` 和 `keyspace_misses`。
- 应用错误率。

压测目标不是追求夸张数字。

目标是验证：

- 缓存是否真的降低数据库压力。
- Redis 超时时接口是否能降级。
- 热点短链接是否造成瓶颈。

## 九、项目复盘问题

完成后回答这些问题：

1. 哪些数据是事实数据，哪些数据是派生数据？
2. 哪些 Redis key 可以丢失并重建？
3. 哪些接口 Redis 失败后可以继续服务？
4. 哪些接口 Redis 失败后应该拒绝？
5. 更新短链接时为什么删除缓存而不是更新缓存？
6. 空值缓存 TTL 为什么要短？
7. Pipeline 解决什么问题？
8. 访问计数是否需要最终落库？
9. 这个项目里最可能出现缓存不一致的地方在哪里？
10. 如果访问量增长 10 倍，先优化哪里？

这些问题比背命令更重要。

## 十一、常见错误

### 1. 只测正常流程

缓存项目最容易在异常和更新场景出问题。

### 2. 没有观察数据库 QPS

缓存是否有效，数据库压力会告诉你。

### 3. Redis 故障时所有接口都 500

很多能力可以降级。

### 4. 只写代码不写运行说明

后面排查时会忘记 key、TTL 和降级策略。

### 5. 访问计数长期只放 Redis

如果统计是业务资产，就要考虑落库或异步聚合。

## 十三、本节小结

这一节完成了第 13 阶段的实践收尾。

- 项目完成不等于代码写完，还要验证、演练和复盘。
