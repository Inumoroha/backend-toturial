# 7. 更新短链接：删除缓存与一致性处理

这一节处理更新短链接后的缓存一致性。

短链接可以修改原始 URL、标题、状态和过期时间。只要这些字段影响跳转或详情展示，就必须考虑 Redis 缓存是否还可信。

学完这一节后，你应该能够：

- 实现 `PATCH /links/:code`。
- 理解为什么更新后通常删除缓存。
- 删除详情缓存和空值缓存。
- 处理删除缓存失败的风险。
- 设计缓存一致性的工程补偿手段。

---

## 一、更新接口目标

接口：

```text
PATCH /links/:code
```

可更新字段：

- 原始 URL。
- 标题。
- 状态。
- 过期时间。

更新成功后必须让后续读取看到新数据。

---

## 二、为什么删除缓存

更新数据有两种常见做法：

```text
1. 更新数据库，再更新缓存
2. 更新数据库，再删除缓存
```

本项目推荐：

```text
更新数据库 -> 删除缓存
```

原因：

- 删除比重写更简单。
- 下一次读取会从数据库加载最新数据。
- 减少缓存对象和数据库对象字段不一致的风险。
- 多个字段更新时更稳。

---

## 三、需要删除哪些 key

更新短链接后删除：

```text
shortlink:detail:{code}
shortlink:not_found:{code}
```

一般不删除：

```text
shortlink:visit_count:{code}
shortlink:create_limit:{user_id}
```

访问计数和创建限流不是详情缓存。

---

## 四、删除缓存方法

示例：

```go
func (c *ShortLinkCache) DeleteDetail(ctx context.Context, code string) error {
    return c.rdb.Del(ctx, detailKey(code)).Err()
}

func (c *ShortLinkCache) DeleteNotFound(ctx context.Context, code string) error {
    return c.rdb.Del(ctx, notFoundKey(code)).Err()
}

func (c *ShortLinkCache) DeleteLinkCache(ctx context.Context, code string) error {
    return c.rdb.Del(ctx, detailKey(code), notFoundKey(code)).Err()
}
```

可以一次 `DEL` 多个 key。

---

## 五、更新服务流程

伪代码：

```go
func (s *ShortLinkService) Update(ctx context.Context, code string, req UpdateLinkRequest) error {
    if !validCode(code) {
        return ErrNotFound
    }

    if err := s.repo.UpdateByCode(ctx, code, req); err != nil {
        return err
    }

    if err := s.cache.DeleteLinkCache(ctx, code); err != nil {
        log.Printf("delete shortlink cache failed: code=%s err=%v", code, err)
    }

    return nil
}
```

数据库更新成功后，即使删除缓存失败，也要记录日志。

是否返回失败，要看业务一致性要求。

学习项目可以返回成功并记录日志。

---

## 六、删除缓存失败怎么办

删除缓存失败会导致旧缓存继续存在，直到 TTL 过期。

降低风险的方法：

- 详情缓存设置有限 TTL。
- 删除失败记录错误日志。
- 删除失败写入重试任务。
- 更新接口返回成功前做短重试。
- 使用消息队列异步重删。

简单短重试示例：

```go
func (s *ShortLinkService) deleteCacheWithRetry(ctx context.Context, code string) {
    for i := 0; i < 2; i++ {
        if err := s.cache.DeleteLinkCache(ctx, code); err == nil {
            return
        }
        time.Sleep(20 * time.Millisecond)
    }
    log.Printf("delete shortlink cache finally failed: code=%s", code)
}
```

不要无限重试。

---

## 七、先删缓存还是先更新数据库

推荐：

```text
先更新数据库
再删除缓存
```

如果先删缓存，再更新数据库，可能发生：

```text
1. 删除缓存
2. 另一个请求读数据库旧数据
3. 另一个请求把旧数据写回缓存
4. 当前请求更新数据库
```

最终缓存里可能仍是旧数据。

先更新数据库再删缓存，也不是绝对完美，但更适合作为默认方案。

---

## 八、延迟双删

更严格一点可以做延迟双删：

```text
更新数据库
删除缓存
等待短时间
再次删除缓存
```

示例：

```go
func (s *ShortLinkService) Update(ctx context.Context, code string, req UpdateLinkRequest) error {
    if err := s.repo.UpdateByCode(ctx, code, req); err != nil {
        return err
    }

    _ = s.cache.DeleteLinkCache(ctx, code)

    go func() {
        time.Sleep(500 * time.Millisecond)
        ctx, cancel := context.WithTimeout(context.Background(), 200*time.Millisecond)
        defer cancel()
        _ = s.cache.DeleteLinkCache(ctx, code)
    }()

    return nil
}
```

学习阶段知道这个思路即可。

真实项目要结合消息队列、任务调度和可观测性。

---

## 九、空值缓存也要删除

为什么更新时还要删空值缓存？

可能出现这种情况：

```text
1. 用户访问 code，数据库不存在
2. 写入 shortlink:not_found:code
3. 后来创建或恢复这个 code
4. 如果不删除空值缓存，短时间内仍然返回 404
```

所以创建和更新成功后，都可以删除空值缓存。

---

## 十、常见错误

### 1. 更新数据库后忘记删缓存

用户继续读旧 URL。

### 2. 只删详情缓存，不删空值缓存

新恢复的短码可能仍然被判定不存在。

### 3. 删除缓存失败没有日志

线上出现旧数据时很难追踪。

### 4. 把访问计数也删掉

更新详情不应该清空统计。

### 5. 缓存永不过期

删除失败时旧数据会长期存在。

## 十二、本节小结

这一节完成了更新短链接后的缓存一致性处理。

你需要记住：

- 更新数据库后通常删除缓存。
- 详情缓存和空值缓存都要处理。
- 删除缓存失败要有日志和补偿思路。
- 缓存 TTL 是一致性的最后兜底。
- 不要因为更新详情而清空访问计数。
