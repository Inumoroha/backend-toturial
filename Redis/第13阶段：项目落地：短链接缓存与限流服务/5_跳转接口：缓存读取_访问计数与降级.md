# 5. 跳转接口：缓存读取、访问计数与降级

跳转接口是短链接服务的核心接口。

它必须尽量快、尽量稳。用户访问短链接时，最重要的结果是跳到原始 URL，而不是精确完成每一个统计动作。

学完这一节后，你应该能够：

- 实现 `GET /:code` 跳转接口。
- 在跳转链路中读取详情缓存。
- 使用 Redis `INCR` 记录访问次数。
- 区分核心流程和非核心流程。
- 设计 Redis 故障时的降级策略。

---

## 一、跳转接口目标

接口：

```text
GET /:code
```

成功：

```text
302 Location: https://example.com/articles/100
```

失败：

| 场景 | 响应 |
| --- | --- |
| 短码不存在 | 404 |
| 短链接禁用 | 404 或 403 |
| 短链接过期 | 410 |
| 数据库异常 | 500 |

---

## 二、完整流程

推荐流程：

```text
1. 从路径中解析 code
2. 校验 code 格式
3. 读取详情缓存
4. 缓存未命中时查数据库
5. 数据库存在则写入详情缓存
6. 判断状态和过期时间
7. Redis 访问计数 +1
8. 返回 302 跳转
```

访问计数失败不应该阻止跳转。

详情读取失败可以回源数据库。

数据库失败才会影响主流程。

## 三、Handler 示例

使用 `net/http` 的伪代码：

```go
func (h *Handler) Redirect(w http.ResponseWriter, r *http.Request) {
    code := chi.URLParam(r, "code")
    if !validCode(code) {
        http.NotFound(w, r)
        return
    }

    ctx, cancel := context.WithTimeout(r.Context(), 800*time.Millisecond)
    defer cancel()

    link, err := h.shortLinkService.GetForRedirect(ctx, code)
    if errors.Is(err, ErrNotFound) {
        http.NotFound(w, r)
        return
    }
    if errors.Is(err, ErrExpired) {
        http.Error(w, "link expired", http.StatusGone)
        return
    }
    if err != nil {
        http.Error(w, "internal error", http.StatusInternalServerError)
        return
    }

    h.shortLinkService.CountVisitAsync(r.Context(), code)
    http.Redirect(w, r, link.OriginalURL, http.StatusFound)
}
```

这里把访问计数放在非核心路径。

如果你不想引入异步，也可以同步执行，但要设置很短超时。

## 四、访问计数

Redis 计数器：

```text
shortlink:visit_count:{code}
```

命令：

```redis
INCR shortlink:visit_count:aB9xK2
```

go-redis 示例：

```go
func (c *ShortLinkCache) IncrVisit(ctx context.Context, code string) error {
    return c.rdb.Incr(ctx, visitCountKey(code)).Err()
}
```

如果是第一次访问，`INCR` 会自动把值从 0 增加到 1。

## 五、异步计数

简单学习项目可以用 goroutine：

```go
func (s *ShortLinkService) CountVisitAsync(parent context.Context, code string) {
    go func() {
        ctx, cancel := context.WithTimeout(context.Background(), 100*time.Millisecond)
        defer cancel()

        if err := s.cache.IncrVisit(ctx, code); err != nil {
            log.Printf("incr visit failed: code=%s err=%v", code, err)
        }
    }()
}
```

注意：

- 不要无限制启动复杂异步任务。
- goroutine 内要设置超时。
- 真实项目可以使用队列、Stream 或批量聚合。

## 六、跳转前状态判断

示例：

```go
func validateRedirect(link *CachedShortLink, now time.Time) error {
    if link.Status != "active" {
        return ErrNotFound
    }
    if link.ExpiresAt != nil && now.After(*link.ExpiresAt) {
        return ErrExpired
    }
    if link.OriginalURL == "" {
        return ErrNotFound
    }
    return nil
}
```

不要只要查到短码就跳转。

禁用和过期都是业务状态。

## 七、缓存降级

Redis 可能出现：

- 超时。
- 连接失败。
- 读写失败。
- 序列化异常。

跳转接口的降级策略：

| Redis 能力 | 失败后处理 |
| --- | --- |
| 详情缓存读取 | 查数据库 |
| 详情缓存写入 | 记录日志，继续返回 |
| 空值缓存读取 | 查数据库 |
| 空值缓存写入 | 记录日志，继续返回 |
| 访问计数 | 记录日志，继续跳转 |

核心原则：

```text
能跳转就先跳转。
统计和缓存失败不阻塞用户访问。
```

## 八、防止恶意短码

在查 Redis 和数据库之前先校验 code。

示例：

```go
var codePattern = regexp.MustCompile(`^[0-9a-zA-Z]{4,32}$`)

func validCode(code string) bool {
    return codePattern.MatchString(code)
}
```

这样可以过滤明显非法请求。

非法 code 不需要写空值缓存。

---

## 九、访问计数是否要落数据库

Redis 里的访问计数是易失的。

如果访问统计是重要数据，需要定期同步到数据库。

常见方案：

- 定时任务读取 Redis 计数并累加到数据库。
- 使用 Stream 记录访问日志后异步消费。
- 使用批量聚合降低数据库写入频率。

本阶段先实现 Redis 计数。

后面可选增强再讨论 Stream。

## 十、常见错误

### 1. 计数失败导致跳转失败

访问计数不是跳转主链路。

### 2. 每次跳转都查数据库

热门短链接会给数据库很大压力。

### 3. 不校验 code 格式

无效请求会污染日志和缓存。

### 4. 过期链接仍然跳转

缓存对象里必须包含过期时间或服务层能判断过期。

### 5. Redis 操作没有超时

跳转接口会被 Redis 慢操作拖住。

## 十二、本节小结

这一节完成了短链接最核心的跳转链路。

你需要记住：

- 跳转接口最重要的是快和稳。
- 详情缓存失败可以回源数据库。
- 访问计数失败不应该影响跳转。
- 状态和过期时间必须在跳转前判断。
- code 格式校验能挡掉很多无效请求。
