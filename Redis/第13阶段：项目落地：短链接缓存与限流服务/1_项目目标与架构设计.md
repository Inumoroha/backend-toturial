# 1. 项目目标与架构设计

第 13 阶段是整条 Redis 学习路线的收尾项目。

前面你已经学习了数据结构、缓存、限流、Lua、持久化、高可用、性能排查、安全和监控。现在要把这些能力放进一个 Go 后端项目里：短链接服务。

学完这一节后，你应该能够：

- 理解短链接服务为什么适合练习 Redis。
- 拆分短链接服务的核心业务流程。
- 设计数据库、Redis 和 Go 服务之间的职责边界。
- 明确本阶段最终要完成的功能。
- 建立项目开发顺序。

---

## 一、项目目标

我们要做一个短链接服务。

核心能力：

- 创建短链接。
- 根据短码跳转到原始链接。
- 查询短链接详情。
- 更新短链接。
- 查询访问统计。

Redis 参与的能力：

- 缓存热门短链接详情。
- 记录访问计数。
- 给创建接口做限流。
- 防止缓存穿透。
- 更新短链接时删除缓存。
- 使用 Pipeline 优化统计读取。

可选增强：

- 用 ZSet 维护热门链接榜。
- 用 Stream 异步记录访问日志。
- 用 Bitmap 统计每日访问。
- 加入本地缓存形成多级缓存。
- 用 Lua 实现更严谨的限流。

---

## 二、为什么用短链接做收尾项目

短链接项目不大，但覆盖了很多真实后端场景。

它有读多写少的特点：

```text
创建链接：低频写
跳转链接：高频读
访问统计：高频写
查询详情：中频读
更新链接：低频写
```

这正好适合练习 Redis。

短链接详情适合缓存。

访问次数适合计数器。

创建接口适合限流。

不存在的短码适合空值缓存。

更新短链接适合练习缓存一致性。

---

## 三、整体架构

建议架构：

```text
Client
  |
  v
Go HTTP API
  |
  +--> PostgreSQL / MySQL
  |
  +--> Redis
```

数据库负责核心事实数据。

Redis 负责加速、计数、限流和防护。

不要把 Redis 当成唯一数据源。

短链接的原始 URL、创建者、状态、过期时间等核心数据应该保存在数据库中。

---

## 四、组件职责

### 1. Go 服务

Go 服务负责：

- 参数校验。
- 生成短码。
- 调用数据库。
- 调用 Redis。
- 控制缓存策略。
- 处理错误和降级。
- 输出日志。

### 2. 数据库

数据库负责：

- 保存短链接主记录。
- 保证短码唯一。
- 支持更新和查询。
- 作为最终可信数据源。

### 3. Redis

Redis 负责：

- 缓存短链接详情。
- 缓存不存在的短码。
- 记录访问次数。
- 做用户维度创建限流。
- 支持统计接口快速读取。

## 五、核心请求链路

### 1. 创建短链接

```text
校验用户限流
校验原始 URL
生成短码
写入数据库
删除可能存在的空值缓存
返回短链接
```

创建接口是写接口，必须先保证数据库写成功。

Redis 限流失败时可以直接拒绝请求。

数据库写入失败时不能写缓存。

### 2. 跳转短链接

```text
读取 Redis 详情缓存
缓存命中 -> 访问计数 + 跳转
缓存未命中 -> 查数据库
数据库存在 -> 写入 Redis 详情缓存 + 访问计数 + 跳转
数据库不存在 -> 写入空值缓存 + 返回 404
```

这是本项目最重要的链路。

它同时涉及 Cache-Aside、空值缓存、访问计数和降级。

### 3. 更新短链接

```text
更新数据库
删除 Redis 详情缓存
删除 Redis 空值缓存
返回更新结果
```

更新后不建议立即写缓存。

更常见的做法是删除缓存，让下一次读请求重新回源数据库。

## 六、接口范围

本阶段推荐实现下面 5 个接口：

| 方法 | 路径 | 作用 |
| --- | --- | --- |
| `POST` | `/links` | 创建短链接 |
| `GET` | `/:code` | 根据短码跳转 |
| `GET` | `/links/:code` | 查询短链接详情 |
| `PATCH` | `/links/:code` | 更新短链接 |
| `GET` | `/links/:code/stats` | 查询访问统计 |

可以使用：

- `net/http`
- Gin
- Chi

如果你还在练基础，推荐先用 `net/http`。

如果你想更贴近工程项目，可以用 Gin 或 Chi。

## 七、开发顺序

推荐顺序：

```text
1. 建立数据库表和领域模型
2. 写短码生成和数据库 CRUD
3. 接入 go-redis
4. 实现短链接详情缓存
5. 实现跳转接口
6. 加入访问计数
7. 加入创建接口限流
8. 加入空值缓存
9. 实现更新后删除缓存
10. 实现统计接口
11. 增加日志、错误处理和测试清单
```

不要一开始就上 Stream、ZSet 和多级缓存。

先把主链路跑通，再增强。

## 八、项目完成标准

完成第 13 阶段时，你应该能做到：

- 可以创建短链接。
- 可以通过短码跳转。
- 热门短链接会命中 Redis 缓存。
- 不存在的短码不会持续打数据库。
- 同一用户创建短链接会被限流。
- 每次跳转都会累计访问次数。
- 更新短链接后缓存会失效。
- 统计接口能读到访问次数。
- Redis 故障时有基本降级策略。

## 九、常见错误

### 1. 把 Redis 当数据库

短链接核心数据要落数据库。

Redis 可以丢，可以重建。

### 2. 创建后只写缓存不写数据库

这样会导致数据不可靠。

创建短链接必须先写数据库。

### 3. 缓存不存在的短码没有 TTL

空值缓存必须设置较短 TTL。

否则错误短码会长期占用 Redis。

### 4. 更新后不处理缓存

更新数据库后如果不删除缓存，用户可能继续读到旧 URL。

### 5. 访问计数强依赖 Redis

计数失败通常不应该影响跳转。

跳转链路的核心目标是把用户带到原始 URL。
