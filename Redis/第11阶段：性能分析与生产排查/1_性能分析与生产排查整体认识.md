# 1. 性能分析与生产排查整体认识

Redis 很快，但生产里仍然会慢、抖动、超时、连接耗尽、内存异常、热点打爆。

第 11 阶段的目标不是再学几个命令，而是建立排查思路：

```text
现象是什么？
影响范围多大？
是 Redis 服务端问题，还是客户端、网络、业务访问模式问题？
证据在哪里？
怎么止血？
怎么复盘？
```

学完这一节后，你应该能够：

- 建立 Redis 性能排查的基本路径。
- 区分服务端、客户端、网络和业务层问题。
- 知道常用排查命令的用途。
- 理解为什么性能优化不能只盯 Redis。
- 为后续几节建立整体地图。

---

## 一、常见生产现象

Redis 相关问题常表现为：

- 接口偶发超时。
- Redis QPS 突然升高。
- Redis CPU 飙高。
- 连接数暴涨。
- 连接池耗尽。
- 慢命令增加。
- 内存持续上涨。
- `evicted_keys` 增长。
- 命中率下降。
- 数据库回源压力暴增。
- 某个 key 被打爆。

不要看到“Redis 超时”就立刻判断 Redis 自身有问题。

很多时候根因在客户端、网络、Big Key、业务流量或数据库回源。

---

## 二、排查先问四个问题

### 1. 是全部接口慢，还是部分接口慢？

如果全部接口都慢，可能是：

- Redis 整体压力。
- 网络问题。
- 客户端连接池。
- 应用整体资源问题。

如果只有某个接口慢，可能是：

- 某类 key。
- 某个命令。
- 某段业务逻辑。
- 某个热点或 Big Key。

### 2. 是读慢，还是写慢？

读慢可能来自：

- Big Key。
- 热点 key。
- 网络传输。
- 反序列化。

写慢可能来自：

- AOF 刷盘。
- 内存淘汰。
- fork/COW。
- 慢 Lua。
- 大集合写入。

### 3. 是 Redis 慢，还是应用等 Redis 慢？

应用等待 Redis，可能是：

- 连接池拿不到连接。
- 网络 RTT 高。
- 客户端超时设置过长。
- goroutine 堆积。
- Redis 命令排队。

### 4. 最近发生了什么变化？

常见变化：

- 新版本上线。
- 新增缓存 key。
- 活动流量。
- 数据量增长。
- Redis 配置调整。
- 主从切换。
- AOF 重写。
- 批任务启动。

排查性能问题时，时间线非常关键。

---

## 三、常用排查命令地图

| 命令 | 用途 |
| --- | --- |
| `INFO` | 查看整体状态 |
| `SLOWLOG GET` | 查看慢命令 |
| `CLIENT LIST` | 查看客户端连接 |
| `MEMORY USAGE` | 查看单个 key 内存 |
| `MEMORY STATS` | 查看内存统计 |
| `TTL` | 查看 key 生命周期 |
| `SCAN` | 安全遍历 key |
| `MONITOR` | 实时观察命令，生产慎用 |
| `LATENCY DOCTOR` | 查看延迟诊断 |

这些命令不是孤立的。

真正排查时要组合使用。

---

## 四、排查路径示例

接口突然 Redis 超时：

```text
1. 看应用日志：哪些命令、哪些 key、错误类型。
2. 看连接池指标：是否等待连接。
3. 看 Redis INFO：CPU、内存、连接、命中率、淘汰。
4. 看 SLOWLOG：是否有慢命令。
5. 看 key 特征：是否 Big Key 或 Hot Key。
6. 看网络：应用到 Redis RTT 是否升高。
7. 看最近变更：是否新上线批量命令或活动流量。
8. 先止血，再复盘优化。
```

止血比完美定位更优先。

例如先限流、降级、切换只读接口、关闭新功能。

---

## 五、服务端常见原因

Redis 服务端问题可能包括：

- 慢命令。
- 大 key 删除。
- Lua 脚本太慢。
- AOF 重写或 RDB fork。
- 内存淘汰压力。
- CPU 被热点打满。
- 连接数过多。
- 主从同步压力。
- 内存碎片。

这些可以通过 `INFO`、`SLOWLOG`、`LATENCY`、日志和监控观察。

---

## 六、客户端常见原因

客户端问题也很常见：

- 连接池太小。
- 连接泄漏。
- 超时时间不合理。
- 没有使用 Pipeline，RTT 太多。
- 大 value 反序列化慢。
- goroutine 堆积。
- 重试风暴。
- 熔断和降级缺失。

很多“Redis 慢”其实是客户端用法不对。

---

## 七、业务访问模式问题

业务访问模式会决定 Redis 压力。

典型问题：

- 热点 key。
- Big Key。
- 大量 key 同时过期。
- 缓存穿透。
- 缓存击穿。
- 缓存雪崩。
- 批量任务扫 Redis。
- 排行榜或大集合无边界增长。

这些问题不能只靠调 Redis 参数解决。

要改 key 设计、缓存策略和业务限流。

---

## 八、网络和部署问题

网络问题也会表现为 Redis 慢：

- 应用和 Redis 跨机房。
- 容器网络抖动。
- DNS 解析问题。
- NAT 连接耗尽。
- TLS 开销。
- 带宽被 Big Key 打满。

Redis 命令本身可能很快，但一次网络往返很慢。

这时 Pipeline、就近部署和连接池配置更关键。

---

## 九、先止血再优化

生产故障时建议：

```text
1. 先降低影响。
2. 再定位根因。
3. 最后做长期优化。
```

止血手段：

- 限流。
- 降级。
- 临时关闭非核心功能。
- 热点 key 本地缓存。
- 扩容 Redis 或应用。
- 避免批任务继续执行。
- 删除或拆分异常 key。

不要在高峰期做大范围危险操作。

## 十一、常见错误

### 1. 只看 Redis，不看客户端

连接池、网络、序列化经常才是瓶颈。

### 2. 一上来就 MONITOR

`MONITOR` 生产风险高，会放大压力。

### 3. 只看平均耗时

P95、P99、最大值更能反映抖动。

### 4. 没有时间线

不知道问题从什么时候开始，很难定位变更。

### 5. 没有止血方案

故障时还在慢慢分析，影响会扩大。
