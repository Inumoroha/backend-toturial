# 6. 缓存雪崩：TTL 随机化、限流降级、多级缓存

缓存雪崩是指大量缓存同时失效，或者 Redis 整体不可用，导致大量请求集中打到数据库。

它和缓存击穿不同：击穿通常是某一个热点 key 失效；雪崩是大面积缓存失效或缓存层整体故障。

学完这一节后，你应该能够：

- 理解缓存雪崩是什么。
- 区分缓存穿透、击穿、雪崩。
- 使用 TTL 随机化降低大量 key 同时过期风险。
- 理解 Redis 故障时限流、降级、熔断的重要性。
- 知道多级缓存和缓存预热如何降低风险。

---

## 一、什么是缓存雪崩

缓存雪崩有两类常见原因。

第一类：大量 key 同时过期。

```text
大量商品缓存都设置 EX 3600
这些 key 在同一时间集中失效
请求同时打到数据库
```

第二类：Redis 整体不可用。

```text
Redis 宕机或网络故障
所有缓存请求失败
请求全部回源数据库
```

结果都类似：数据库突然承受远超平时的压力。

---

## 二、穿透、击穿、雪崩对比

| 问题 | 核心原因 | 典型表现 |
| --- | --- | --- |
| 缓存穿透 | 请求不存在的数据 | Redis 和数据库都没有，反复查数据库 |
| 缓存击穿 | 单个热点 key 过期 | 热点 key 失效瞬间，大量请求查数据库 |
| 缓存雪崩 | 大量 key 同时失效或 Redis 故障 | 大量请求集中打数据库 |

记忆方式：

```text
穿透：查不存在
击穿：一个热点破了
雪崩：一大片塌了
```

---

## 三、为什么相同 TTL 危险

假设系统启动时预热 100 万个商品缓存。

每个 key 都设置：

```text
EX 3600
```

如果这些 key 在几分钟内集中写入，它们也会在一小时后集中失效。

失效后，大量请求未命中 Redis，开始查数据库。

这就是雪崩风险。

缓存不是只要设置 TTL 就万事大吉，TTL 的分布也很重要。

---

## 四、方案一：TTL 随机化

最常见的防雪崩手段是 TTL 随机化。

不要让大量 key 使用完全相同的过期时间。

例如文章详情缓存：

```text
基础 TTL：30 分钟
随机偏移：0 到 10 分钟
最终 TTL：30 到 40 分钟
```

伪代码：

```text
ttl = 1800 + random(0, 600)
```

Redis 命令：

```redis
SET article:detail:1001 '{...}' EX 2147
```

这样不同 key 的过期时间被打散，不会集中在同一秒失效。

---

## 五、TTL 随机化适合什么场景

适合：

- 大量同类缓存。
- 批量预热的数据。
- 商品详情缓存。
- 文章详情缓存。
- 用户资料缓存。
- 列表页缓存。

不太适合：

- 有严格固定过期窗口的业务。
- 验证码。
- 限流计数。
- token 安全策略。

比如验证码就是 5 分钟失效，不应该随便加 10 分钟随机值。

TTL 随机化主要用于普通缓存数据，而不是所有 Redis key。

---

## 六、方案二：缓存预热

缓存预热是指在高流量到来前，提前把热点数据写入 Redis。

适合：

- 首页数据。
- 活动页数据。
- 商品秒杀页。
- 热门文章。
- 排行榜。

预热流程：

```text
定时任务或发布流程
查询数据库热点数据
写入 Redis
设置分散 TTL
```

预热可以降低首次访问数据库压力。

但预热也要控制节奏，不能一次性把数据库打满。

---

## 七、方案三：多级缓存

多级缓存是指不只依赖 Redis，还可以在服务本地保留一层缓存。

结构：

```text
Go 服务本地缓存 -> Redis -> 数据库
```

读取流程：

```text
先查本地缓存
本地未命中查 Redis
Redis 未命中查数据库
```

本地缓存适合：

- 配置数据。
- 字典数据。
- 首页热点数据。
- 极高频读取且变化不频繁的数据。

优点：Redis 故障时，本地缓存还能撑一段时间。

缺点：本地缓存会带来多实例一致性问题。

---

## 八、方案四：限流和降级

如果 Redis 故障，所有请求都回源数据库，很容易把数据库压垮。

这时需要限流和降级。

限流：

```text
限制进入数据库的请求数量
超过阈值直接返回繁忙或降级数据
```

降级：

```text
返回默认值
返回旧数据
返回简化数据
关闭非核心功能
```

比如文章详情接口：

- Redis 失败后允许少量请求查数据库。
- 超过阈值返回“系统繁忙，请稍后再试”。
- 对非核心统计信息返回默认值。

这听起来不完美，但比数据库被打崩更好。

---

## 九、方案五：熔断保护

熔断是指当某个依赖持续失败时，暂时停止请求它。

如果 Redis 持续超时：

```text
短时间内不要每个请求都等 Redis 超时
直接走降级策略
```

如果数据库压力过高：

```text
限制回源查询
保护数据库
```

熔断不是 Redis 命令，而是服务治理策略。

它的目标是保护系统整体，而不是每个请求都硬扛到底。

---

## 十、方案六：高可用 Redis

生产环境 Redis 本身也要高可用。

常见方式：

- 主从复制。
- Sentinel。
- Redis Cluster。
- 云厂商托管 Redis。
- 监控和告警。
- 合理的客户端超时和重试。

这些不属于本阶段重点，但你要知道：防雪崩不只是写代码，还包括基础设施稳定性。

Redis 如果是系统关键依赖，就必须有高可用方案。

---

## 十一、雪崩时不要盲目重试

Redis 超时后，如果所有请求都立刻重试，可能让问题更严重。

错误做法：

```text
Redis 失败 -> 立即重试 3 次 -> 再查数据库
```

高并发下，重试会放大流量。

更稳妥的做法：

- 设置短超时。
- 控制重试次数。
- 加入退避策略。
- 对高流量接口优先降级。
- 保护数据库连接池。

缓存故障时，少做一点，有时比努力做完每件事更安全。

---

## 十二、完整防雪崩思路

一个比较完整的策略：

```text
1. 普通缓存 TTL 加随机偏移
2. 热点数据提前预热
3. 热点 key 使用逻辑过期或互斥重建
4. Redis 设置合理超时
5. Redis 故障时限流回源数据库
6. 非核心功能降级
7. 关键数据使用本地缓存兜底
8. Redis 做高可用和监控告警
```

小项目不需要一次性全部做完，但要知道这些层次。

---

## 十三、常见错误

### 1. 所有缓存 TTL 完全一样

大量 key 可能同时过期。

### 2. Redis 故障后所有请求查数据库

数据库可能被瞬间打垮。

### 3. 没有超时控制

请求长时间等待 Redis，拖慢整个接口。

### 4. 盲目重试

重试会放大故障流量。

### 5. 没有降级预案

等故障发生时再想怎么降级，通常来不及。

## 十五、本节小结

这一节你学习了缓存雪崩。

你需要记住：

- 缓存雪崩是大量 key 同时失效，或 Redis 整体不可用，导致大量请求打到数据库。
- TTL 随机化可以打散过期时间。
- 缓存预热可以降低高峰期首次回源压力。
- 多级缓存可以增强兜底能力，但会带来一致性问题。
- Redis 故障时要有限流、降级、熔断，不能让所有请求直接冲向数据库。
- 防雪崩是代码、架构、运维共同完成的事情。
