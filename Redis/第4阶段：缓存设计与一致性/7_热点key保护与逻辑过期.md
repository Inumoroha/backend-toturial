# 7. 热点 key 保护与逻辑过期

热点 key 是缓存系统中最需要特殊保护的对象。

一个热点 key 可能承载极高流量。一旦它失效、变慢或被删除，数据库和后端服务都可能受到冲击。逻辑过期是一种常见的热点 key 保护方案，它牺牲一部分数据新鲜度，换取系统稳定性。

学完这一节后，你应该能够：

- 理解什么是热点 key。
- 知道热点 key 为什么不能完全按普通缓存处理。
- 理解逻辑过期的设计思想。
- 写出逻辑过期的读流程和异步重建流程。
- 判断逻辑过期适合和不适合的业务场景。
- 综合使用 TTL、互斥锁、预热和降级保护热点数据。

---

## 一、什么是热点 key

热点 key 是被大量请求频繁访问的 Redis key。

例如：

- 首页配置。
- 热门文章详情。
- 秒杀商品信息。
- 活动页数据。
- 热门排行榜。
- 全站公共字典。
- 某个爆款商品详情。

普通 key 每秒访问几次，过期后回源数据库影响不大。

热点 key 每秒可能访问几百、几千甚至更多。一旦失效，影响会被放大。

---

## 二、热点 key 的风险

热点 key 的风险包括：

- 过期瞬间造成缓存击穿。
- 单 key 请求过于集中，Redis 节点压力升高。
- value 太大导致网络传输压力。
- 删除或重建时引发数据库压力。
- Redis 故障时大量请求回源。

所以热点 key 不能只靠普通的：

```text
GET -> 未命中 -> 查数据库 -> SET
```

它需要额外保护。

---

## 三、热点 key 识别方式

常见识别方式：

- 根据业务经验提前判断。
- 根据访问日志统计。
- 根据 Redis key 访问监控。
- 根据接口 QPS 和参数分布分析。
- 根据活动、推荐位、榜单预估。

比如：

```text
首页配置一定是热点
秒杀商品活动开始后一定是热点
排行榜前几名文章可能成为热点
```

热点 key 保护应该尽量提前做，不要等数据库被打爆后再补。

---

## 四、逻辑过期是什么

普通过期是 Redis key 自己过期：

```redis
SET article:detail:1001 '{...}' EX 1800
```

到期后 key 消失。

逻辑过期是 Redis key 不设置物理 TTL，或者设置很长 TTL，而是在 value 里保存一个业务过期时间。

value 示例：

```json
{
  "expire_at": 1790000000,
  "data": {
    "id": 1001,
    "title": "Redis 逻辑过期"
  }
}
```

Redis key 还在，但程序判断 `expire_at` 是否过期。

---

## 五、逻辑过期读流程

逻辑过期的核心读流程：

```text
1. 读取 Redis
2. 如果 key 不存在，走兜底回源或返回降级
3. 如果 key 存在，解析 value
4. 如果 expire_at 未过期，直接返回 data
5. 如果 expire_at 已过期，仍然先返回旧 data
6. 同时尝试获取重建锁
7. 拿到锁的请求异步查询数据库并刷新缓存
8. 没拿到锁的请求直接返回旧 data
```

它的关键点是：逻辑过期后，不让用户请求一起阻塞等待数据库。

用户先拿到旧数据，后台慢慢重建。

---

## 六、为什么逻辑过期能保护热点 key

普通缓存过期后：

```text
key 消失
大量请求未命中
大量请求争抢回源数据库
```

逻辑过期后：

```text
key 还在
大量请求仍然能读到旧数据
只有一个请求或后台任务重建缓存
```

这可以避免热点 key 过期瞬间所有请求打到数据库。

代价是：用户可能在短时间内看到旧数据。

这就是稳定性和数据新鲜度的取舍。

---

## 七、逻辑过期伪代码

```text
func GetHotArticle(id):
    key = HotArticleKey(id)
    cached = redis.Get(key)

    if cached not exists:
        return fallbackOrRebuildSync(id)

    wrapper = JSON.parse(cached)

    if wrapper.expire_at > now:
        return wrapper.data

    lockKey = "lock:rebuild:" + key
    locked = redis.Set(lockKey, requestID, NX, EX 10)

    if locked:
        async rebuild:
            article = db.FindArticle(id)
            newWrapper = {
                expire_at: now + logicalTTL,
                data: article
            }
            redis.Set(key, JSON.stringify(newWrapper))
            releaseLock(lockKey, requestID)

    return wrapper.data
```

注意：逻辑过期时，即使数据过期，也先返回旧数据。

这就是逻辑过期和普通过期最大的区别。

---

## 八、重建缓存要加锁

逻辑过期后，很多请求都会发现：

```text
expire_at 已过期
```

如果每个请求都异步重建缓存，数据库仍然会被打爆。

所以仍然需要互斥锁：

```redis
SET lock:rebuild:article:detail:1001 request_id NX EX 10
```

只有拿到锁的请求负责重建。

其他请求直接返回旧数据。

这就是：

```text
逻辑过期 + 互斥重建
```

---

## 九、逻辑过期适合什么场景

适合：

- 热点数据。
- 读多写少。
- 可以接受短时间旧数据。
- 数据重建成本较高。
- 希望请求稳定返回，不想大量等待数据库。

例如：

- 首页配置。
- 活动页展示信息。
- 热门文章详情。
- 热门商品详情。
- 榜单展示。
- 店铺基础信息。

这些场景通常更看重可用性和稳定性。

---

## 十、逻辑过期不适合什么场景

不适合：

- 支付状态。
- 订单状态。
- 库存可售数量强校验。
- 权限撤销立即生效。
- 风控黑名单立即生效。
- 金融余额。

这些场景不能随便返回旧数据。

如果用户权限已经被撤销，但逻辑过期返回旧权限，后果可能很严重。

所以逻辑过期必须先问：

```text
这个业务能不能接受旧数据？能接受多久？
```

不能接受，就不要用。

---

## 十一、逻辑过期和物理 TTL 可以一起用吗

可以。

有些系统会给 key 设置很长物理 TTL，同时 value 里有较短逻辑过期时间。

例如：

```text
Redis 物理 TTL：24 小时
逻辑过期：30 分钟
```

这样做的好处是：

- 平时按逻辑过期刷新。
- 如果重建逻辑长期失败，key 最终也会物理过期。
- 防止 Redis 里永久保留无效数据。

但要注意：物理 TTL 到期后 key 仍然会消失。对极热 key，可以配合预热或后台刷新。

---

## 十二、逻辑过期和主动更新

热点数据也可以由后台任务主动刷新。

比如首页配置：

```text
每 5 分钟后台查询数据库
刷新 Redis 中的首页配置
```

此时逻辑过期可以作为兜底：

- 正常情况下后台定时刷新。
- 如果后台刷新延迟，读请求发现逻辑过期后触发异步重建。
- 如果重建失败，继续返回旧数据并告警。

这比完全依赖请求触发重建更稳。

---

## 十三、热点 key 拆分

如果某个 key 访问量过高，还可以考虑拆分。

例如一个全站配置 key：

```text
site:config
```

可以按模块拆成：

```text
site:config:nav
site:config:footer
site:config:recommend
```

好处：

- 单个 value 更小。
- 更新影响范围更小。
- 不同模块可以设置不同刷新策略。

但拆分也会增加读取次数和代码复杂度。

不要为了拆而拆，要看访问和更新模式。

---

## 十四、热点 key 本地缓存

极高频、变化少的数据，可以加本地缓存。

结构：

```text
Go 本地缓存 -> Redis -> 数据库
```

适合：

- 配置。
- 字典。
- 首页公共数据。
- 热点但不敏感的数据。

本地缓存可以减少 Redis 压力。

但多实例本地缓存会带来一致性问题：某台机器更新了，其他机器可能还保存旧值。

所以本地缓存也适合能接受短暂旧数据的场景。

---

## 十五、热点 key 保护组合拳

一个比较完整的热点 key 保护方案：

```text
1. 提前识别热点 key
2. 缓存预热
3. 逻辑过期，避免物理过期瞬间击穿
4. 互斥锁控制重建
5. 后台异步刷新
6. 本地缓存兜底
7. 限流和降级保护数据库
8. 监控命中率、QPS、重建失败次数
```

不是所有项目都需要全部做。

但你要知道每一层解决什么问题。

---

## 十六、常见错误

### 1. 逻辑过期后所有请求都重建缓存

必须加互斥锁。

### 2. 逻辑过期用于强一致数据

不能接受旧数据的业务不要用。

### 3. 没有监控重建失败

如果重建一直失败，用户可能长期看到旧数据。

### 4. key 永不过期但没有主动刷新

缓存可能永久陈旧。

### 5. value 太大

热点 key 的 value 太大，会放大网络和序列化成本。

## 十八、本节小结

这一节你学习了热点 key 保护与逻辑过期。

你需要记住：

- 热点 key 的失效会被大量请求放大。
- 逻辑过期是在 value 中保存业务过期时间，key 本身不立即消失。
- 逻辑过期后可以先返回旧数据，再异步重建缓存。
- 异步重建必须加互斥锁，避免大量请求同时查数据库。
- 逻辑过期适合能接受短暂旧数据的热点读场景。
- 强一致业务不要使用逻辑过期返回旧数据。
- 热点保护可以组合预热、逻辑过期、本地缓存、限流、降级和监控。

到这里，第四阶段的核心内容就串起来了：你不仅会写缓存读写流程，也开始理解缓存和数据库之间的一致性取舍，以及面对高并发缓存问题时该如何保护系统。
