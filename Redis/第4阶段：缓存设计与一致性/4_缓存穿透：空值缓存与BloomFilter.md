# 4. 缓存穿透：空值缓存与 Bloom Filter

缓存穿透是 Redis 缓存设计中最常见的问题之一。

它的核心是：请求查询的数据在缓存中不存在，在数据库中也不存在，导致请求每次都穿过缓存打到数据库。

学完这一节后，你应该能够：

- 理解什么是缓存穿透。
- 知道恶意请求不存在 ID 为什么危险。
- 使用空值缓存降低数据库压力。
- 理解 Bloom Filter 的基本思想。
- 知道空值缓存和 Bloom Filter 各自的适用场景与局限。

---

## 一、什么是缓存穿透

正常缓存流程：

```text
查 Redis
未命中 -> 查数据库
查到 -> 写入 Redis
返回
```

如果请求的数据不存在：

```text
查 Redis -> 没有
查数据库 -> 也没有
返回不存在
```

问题是：数据库查不到时，如果不写缓存，下次同样请求还会继续查数据库。

大量不存在请求会持续打到数据库，这就是缓存穿透。

---

## 二、例子：不存在的文章 ID

用户请求：

```text
GET /articles/999999999
```

Redis：

```redis
GET article:detail:999999999
```

返回：

```text
(nil)
```

数据库查询：

```sql
SELECT * FROM articles WHERE id = 999999999;
```

也查不到。

如果有人不断请求大量不存在 ID：

```text
/articles/999999001
/articles/999999002
/articles/999999003
...
```

Redis 每次都未命中，数据库每次都要查。

缓存层没有挡住这些请求。

---

## 三、缓存穿透的危害

缓存穿透可能导致：

- 数据库压力升高。
- 接口响应变慢。
- 恶意请求绕过缓存保护。
- Redis 命中率下降。
- 数据库连接池被打满。

缓存穿透通常发生在：

- 查询不存在 ID。
- 参数随机构造。
- 业务没有做参数校验。
- 爬虫或攻击流量大量请求无效资源。

---

## 四、方案一：参数校验

最简单的防线是参数校验。

比如文章 ID 必须是正整数：

```text
id <= 0 直接拒绝
id 不是数字直接拒绝
id 长度异常直接拒绝
```

如果短链接 code 有固定格式，也可以先校验：

```text
只允许字母数字
长度必须在 6 到 12
```

明显非法的请求，不要查 Redis，更不要查数据库。

参数校验不能解决所有穿透，但它是第一道门。

---

## 五、方案二：空值缓存

空值缓存是最常用方案。

当数据库也查不到时，向 Redis 写一个短 TTL 的空标记。

示例：

```redis
SET article:detail:999999999 "__NULL__" EX 60
```

下次请求：

```redis
GET article:detail:999999999
```

返回：

```text
"__NULL__"
```

服务识别到空值标记后，直接返回 404，不再查数据库。

流程：

```text
Redis 未命中
查数据库
数据库也没有
写空值缓存，TTL 较短
返回不存在
```

---

## 六、空值缓存 TTL 为什么要短

如果空值缓存太久，可能误伤后续新数据。

比如文章 `1001` 原本不存在。

请求第一次查询：

```text
数据库查不到 -> 写空值缓存 1 小时
```

5 分钟后，文章 `1001` 被创建了。

但 Redis 里还有空值缓存，用户仍然会看到不存在。

所以空值缓存 TTL 通常较短：

```text
30 秒到 5 分钟
```

具体取决于业务数据创建频率。

---

## 七、空值缓存的 key 设计

有两种常见方式。

方式一：使用原缓存 key，value 存特殊标记：

```text
article:detail:999999999 -> "__NULL__"
```

优点：读流程简单，只查一个 key。

缺点：业务代码要区分正常 JSON 和空值标记。

方式二：使用单独 not_found key：

```text
article:not_found:999999999 -> 1
```

优点：语义清晰。

缺点：读流程可能要多查一次 key。

两种都可以。小项目用第一种更简单。

---

## 八、空值缓存的局限

空值缓存不能解决所有穿透。

如果攻击者每次请求不同的随机 ID：

```text
999999001
999999002
999999003
...
```

系统会为每个不存在 ID 写一个空值缓存。

问题：

- Redis 会被写入大量无效 key。
- 第一次请求仍然会查数据库。
- 随机空间足够大时，空值缓存命中率仍然不高。

这时可以考虑 Bloom Filter。

---

## 九、Bloom Filter 是什么

Bloom Filter 可以理解为一个高效的“可能存在 / 一定不存在”判断器。

它的判断结果有两个：

```text
一定不存在
可能存在
```

注意：它不能说“一定存在”，只能说“可能存在”。

使用方式：

```text
请求 article_id
先问 Bloom Filter：这个 ID 是否可能存在
如果一定不存在，直接返回 404
如果可能存在，再查 Redis / 数据库
```

这样大量明显不存在的 ID 可以在进入 Redis 和数据库之前被挡掉。

---

## 十、Bloom Filter 的特点

优点：

- 占用空间小。
- 判断速度快。
- 适合拦截大量不存在请求。
- 可以保护数据库。

缺点：

- 有误判率。
- 不能直接删除元素，普通 Bloom Filter 删除困难。
- 数据新增时要同步加入 Bloom Filter。
- 实现和维护比空值缓存复杂。

误判是什么意思？

Bloom Filter 可能把不存在的数据判断为“可能存在”，导致请求继续查 Redis 和数据库。

但它不会把真实存在的数据判断为“一定不存在”。

---

## 十一、Bloom Filter 适合什么场景

适合：

- 数据量较大。
- ID 集合相对明确。
- 不存在请求很多。
- 需要提前挡掉非法 ID。
- 能接受一定误判率。

比如：

- 商品 ID 是否存在。
- 文章 ID 是否存在。
- 短链接 code 是否存在。
- 用户 ID 是否存在。

系统启动或定时任务可以把已有 ID 加入 Bloom Filter。

新增数据时，也把新 ID 加入 Bloom Filter。

---

## 十二、Bloom Filter 不适合什么场景

不适合：

- 数据频繁删除，且删除后必须立即不可见。
- 无法接受误判。
- 数据规模很小，空值缓存足够。
- 团队没有维护 Bloom Filter 的经验。

如果只是一个小项目，先做参数校验和空值缓存通常就够了。

不要为了“高级”强行引入 Bloom Filter。

---

## 十三、完整防穿透流程

比较完整的读流程：

```text
1. 参数校验，不合法直接拒绝
2. Bloom Filter 判断，一定不存在则直接返回 404
3. 查 Redis
4. Redis 命中正常值，返回数据
5. Redis 命中空值，返回 404
6. Redis 未命中，查数据库
7. 数据库查到，写正常缓存
8. 数据库查不到，写空值缓存
```

小项目可以简化为：

```text
参数校验 + 空值缓存
```

大流量项目再考虑 Bloom Filter。

---

## 十四、常见错误

### 1. 数据库查不到时什么都不做

这会导致相同不存在请求反复打到数据库。

### 2. 空值缓存 TTL 太长

新数据创建后，用户可能继续看到不存在。

### 3. 空值缓存没有限制

攻击者随机请求大量 ID，Redis 中可能堆积大量空值 key。

### 4. Bloom Filter 数据不同步

新增数据没有加入 Bloom Filter，可能导致新数据被误挡。

### 5. 忽略参数校验

很多无效请求本来可以在入口直接拒绝。

## 十六、本节小结

这一节你学习了缓存穿透。

你需要记住：

- 缓存穿透是查询的数据在 Redis 和数据库中都不存在，导致请求持续打到数据库。
- 参数校验是第一道防线。
- 空值缓存可以缓存“不存在”结果，但 TTL 要短。
- Bloom Filter 可以提前判断某个 ID 是否一定不存在。
- Bloom Filter 有误判率，维护成本比空值缓存高。
- 小项目优先使用参数校验 + 空值缓存，大流量场景再考虑 Bloom Filter。
