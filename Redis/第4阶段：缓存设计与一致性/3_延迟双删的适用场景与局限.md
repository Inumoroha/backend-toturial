# 3. 延迟双删的适用场景与局限

前面我们学习了 Cache-Aside 的写流程：先更新数据库，再删除缓存。

但在高并发读写场景下，仍然可能出现旧数据被写回缓存的问题。延迟双删就是一种常见补救方案。

这一节重点不是让你迷信延迟双删，而是理解它解决什么问题、解决到什么程度、又有哪些局限。

学完这一节后，你应该能够：

- 理解延迟双删的流程。
- 知道它试图解决哪类并发读写问题。
- 理解延迟时间为什么难以精确设置。
- 知道延迟双删不能保证强一致。
- 判断什么时候可以用，什么时候不该依赖它。

---

## 一、先回顾普通写流程

普通 Cache-Aside 写流程：

```text
1. 更新数据库
2. 删除缓存
```

比如：

```text
UPDATE articles SET title = 新标题 WHERE id = 1001
DEL article:detail:1001
```

这已经是常见做法。

但并发场景里可能发生复杂时序。

---

## 二、旧缓存被写回的并发问题

假设线程 A 更新文章，线程 B 同时读取文章。

一种可能时序：

```text
线程 B：读取 Redis，未命中
线程 B：查询数据库，查到旧值
线程 A：更新数据库为新值
线程 A：删除缓存
线程 B：把刚才查到的旧值写入 Redis
```

最终结果：

```text
数据库：新值
Redis：旧值
```

这个问题出现的前提是：读请求在写请求过程中拿到了旧数据库值，并且在写请求删除缓存之后，把旧值回写到了缓存。

这种概率不一定高，但高并发场景下确实可能发生。

---

## 三、什么是延迟双删

延迟双删的流程是：

```text
1. 删除缓存
2. 更新数据库
3. 等待一小段时间
4. 再删除一次缓存
```

也有一些实现使用：

```text
1. 更新数据库
2. 删除缓存
3. 等待一小段时间
4. 再删除一次缓存
```

实际项目中，更常见的是在“更新数据库后删除缓存”的基础上补一次延迟删除。

也就是：

```text
更新数据库
删除缓存
延迟一段时间
再次删除缓存
```

第二次删除的目的，是清掉并发读可能回写进去的旧缓存。

---

## 四、延迟双删的示例

文章更新：

```text
1. 更新数据库文章标题
2. DEL article:detail:1001
3. 延迟 500ms
4. 再次 DEL article:detail:1001
```

第一次删除：让旧缓存失效。

第二次删除：处理并发读回写旧缓存的情况。

伪代码：

```text
func UpdateArticle(ctx, id, input):
    db.UpdateArticle(id, input)
    redis.Del(ArticleDetailKey(id))

    async after 500ms:
        redis.Del(ArticleDetailKey(id))
```

真实项目里不建议直接在请求线程里 `sleep` 太久。更常见的是投递延迟任务或后台 goroutine，但要注意任务丢失和服务重启问题。

---

## 五、为什么要延迟

延迟是为了等待并发读流程结束。

读流程可能是：

```text
Redis 未命中
查数据库
序列化
写回 Redis
```

如果第二次删除太早，可能发生：

```text
第二次 DEL 执行完
并发读才把旧值写回 Redis
```

这样旧值仍然存在。

所以需要等待一小段时间，让可能的旧读请求先完成回写，再执行第二次删除。

---

## 六、延迟多久合适

这就是延迟双删最麻烦的地方。

延迟时间通常要大于一次读请求的完整耗时：

```text
Redis 查询耗时 + 数据库查询耗时 + 业务处理耗时 + Redis 写入耗时
```

如果接口 P99 延迟是 200ms，延迟可以考虑 500ms 或 1s。

但这不是精确科学。

延迟太短：可能删不到旧缓存。

延迟太长：旧缓存停留更久，而且后台任务堆积时间更长。

所以延迟双删不是强一致方案，只是降低旧缓存残留概率。

---

## 七、延迟双删的适用场景

适合：

- 读多写少。
- 可以接受短暂旧数据。
- 缓存 key 明确，删除成本低。
- 并发读写下偶发旧缓存可以通过二次删除缓解。
- 业务追求最终一致。

比如：

- 文章详情。
- 商品描述。
- 用户昵称。
- 店铺信息。
- 内容页缓存。

这些场景通常能接受几百毫秒到几秒的短暂不一致。

---

## 八、延迟双删不适合什么场景

不适合：

- 支付状态。
- 订单关键状态。
- 库存扣减强一致判断。
- 权限变更立即生效。
- 金融余额。
- 任何不能接受旧数据的场景。

这些场景不能简单依赖缓存最终一致。

更常见做法是：

- 直接查数据库权威数据。
- 使用数据库事务和锁。
- 使用版本号校验。
- 使用消息队列可靠同步。
- 对关键接口绕过缓存。

缓存是性能优化，不应该破坏核心正确性。

---

## 九、延迟双删的实现方式

### 1. 请求线程 sleep

```text
删除缓存
sleep 500ms
再删除缓存
```

不推荐。

因为它会拉长接口响应时间，浪费请求线程资源。

### 2. 后台 goroutine

```text
删除缓存
启动 goroutine，500ms 后再删
```

简单，但服务重启时任务可能丢失。

适合本地学习或低风险场景。

### 3. 延迟队列

把二次删除任务放入延迟队列。

```text
delay:delete_cache -> article:detail:1001, execute_at
```

到时间后由后台消费者删除。

更可靠，但系统复杂度更高。

### 4. 消息队列

数据库更新成功后发送缓存删除消息，消费者执行删除和重试。

适合更复杂的生产系统。

---

## 十、删除失败怎么办

第二次删除也可能失败。

比如 Redis 超时。

所以如果业务比较重要，要考虑：

- 重试。
- 记录失败日志。
- 告警。
- 投递补偿任务。
- 依赖 TTL 兜底。

不要以为加了延迟双删就万事大吉。

真正的系统稳定性来自：

```text
删除 + 重试 + TTL + 监控 + 降级
```

---

## 十一、延迟双删与 TTL 的关系

即使使用延迟双删，缓存也应该设置 TTL。

TTL 是最终兜底。

如果删除失败、任务丢失、服务重启，TTL 仍然能让旧缓存最终过期。

例如：

```redis
SET article:detail:1001 '{...}' EX 1800
```

这表示最坏情况下，旧缓存 30 分钟后也会自动消失。

TTL 越长，最坏不一致时间越长。

TTL 越短，缓存命中率可能越低。

这仍然是业务取舍。

---

## 十二、延迟双删不是银弹

延迟双删不能保证：

- 所有并发旧读都一定被清掉。
- Redis 和数据库强一致。
- 删除操作一定成功。
- 延迟时间一定覆盖所有慢请求。
- 服务崩溃时任务不丢。

它只能降低某类旧缓存残留概率。

所以你应该把它当作一种补偿手段，而不是一致性保证。

---

## 十三、常见错误

### 1. 把延迟双删当强一致方案

它不是。

### 2. 在请求线程里长时间 sleep

会拖慢接口。

### 3. 延迟时间随便写

至少应该参考接口耗时、数据库查询耗时和缓存写入耗时。

### 4. 不设置 TTL

删除失败后旧缓存可能长期存在。

### 5. 所有业务都套延迟双删

有些业务不需要，有些业务不能靠它。

## 十五、本节小结

这一节你学习了延迟双删。

你需要记住：

- 延迟双删用于缓解并发读写导致旧值回写缓存的问题。
- 常见流程是：更新数据库，删除缓存，延迟一段时间，再删一次缓存。
- 延迟时间要参考读请求耗时，但无法精确保证。
- 延迟双删只能降低不一致概率，不能保证强一致。
- 它适合能接受短暂旧数据的读多写少场景。
- 关键强一致业务不能依赖延迟双删。
- TTL、重试、监控、补偿仍然必不可少。
