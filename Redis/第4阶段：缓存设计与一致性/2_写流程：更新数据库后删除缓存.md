# 2. 写流程：更新数据库后删除缓存

上一节学习了 Cache-Aside 的读流程：先查缓存，未命中查数据库，查到后回写缓存。

这一节学习写流程。写流程比读流程更容易出问题，因为它涉及缓存和数据库的一致性。

学完这一节后，你应该能够：

- 理解为什么写操作通常先更新数据库，再删除缓存。
- 理解为什么很多场景不推荐更新数据库后直接更新缓存。
- 分析“先删缓存再更新数据库”和“先更新数据库再删缓存”的差异。
- 知道缓存一致性追求的是最终一致，而不是绝对强一致。
- 写出文章更新后的缓存删除流程。

---

## 一、先明确一个事实

使用 Redis 做缓存时，数据库通常是权威数据源。

Redis 只是缓存层。

也就是说：

```text
数据库里的数据才是准的
Redis 里的数据是为了加速读取
```

因此写操作必须以数据库为中心。

比如更新文章标题：

```text
旧标题：Redis 入门
新标题：Redis 缓存设计
```

真正应该先更新的是数据库中的文章记录。

---

## 二、常见写流程

Cache-Aside 常见写流程是：

```text
1. 更新数据库
2. 删除 Redis 缓存
```

以文章详情为例。

key：

```text
article:detail:1001
```

写流程：

```text
用户提交文章更新
Go 服务更新数据库 articles 表
数据库更新成功后
删除 Redis：DEL article:detail:1001
返回更新成功
```

下次读取文章详情时：

```text
Redis 未命中
查询数据库新数据
写入 Redis
返回新文章
```

这样缓存会在下一次读取时重建。

---

## 三、为什么是删除缓存，而不是更新缓存

很多初学者会想：既然数据库更新了，那我直接把 Redis 也更新成新值不就行了吗？

比如：

```text
更新数据库
SET article:detail:1001 新 JSON
```

这种方式不是完全不能用，但在很多业务里更容易出问题。

原因包括：

- 缓存 value 可能不是数据库单表字段，而是聚合结果。
- 更新缓存时可能遗漏字段。
- 多个并发写入时，旧值可能覆盖新值。
- 缓存构建逻辑可能比较复杂。
- 某些字段更新后，多个缓存 key 都受影响。

删除缓存更简单：让下一次读取按完整读流程重新构建缓存。

所以常见建议是：

```text
更新数据库后，删除缓存
```

而不是：

```text
更新数据库后，更新缓存
```

---

## 四、例子：文章详情是聚合结果

文章详情缓存可能不是只来自文章表。

它可能包含：

- 文章标题。
- 正文。
- 作者昵称。
- 分类名称。
- 标签列表。
- 阅读统计。

缓存 value：

```json
{
  "id": 1001,
  "title": "Redis 缓存设计",
  "author": {
    "id": 9,
    "name": "tom"
  },
  "tags": ["redis", "cache"]
}
```

如果只更新文章标题时顺手更新缓存，很容易漏掉其他关联变化。

删除缓存则简单得多：

```redis
DEL article:detail:1001
```

下次读取时重新查数据库和关联信息，构建完整新缓存。

---

## 五、先删缓存再更新数据库，会怎样

方案 A：

```text
1. 删除缓存
2. 更新数据库
```

这个方案有风险。

并发场景：

```text
线程 A：删除缓存
线程 B：读取文章，发现缓存不存在
线程 B：查询数据库，此时数据库还是旧值
线程 B：把旧值写回 Redis
线程 A：更新数据库为新值
```

结果：

```text
数据库是新值
Redis 是旧值
```

而且这个旧缓存可能会一直存在到 TTL 过期。

所以“先删缓存，再更新数据库”并不理想。

---

## 六、先更新数据库再删除缓存，会怎样

方案 B：

```text
1. 更新数据库
2. 删除缓存
```

并发场景：

```text
线程 A：更新数据库为新值
线程 A：删除缓存
线程 B：读取文章，缓存不存在
线程 B：查询数据库新值
线程 B：写入新缓存
```

这个流程更符合直觉，也更常用。

它仍然不是绝对没有并发问题，但比先删缓存再更新数据库更稳。

因此 Cache-Aside 写流程常见建议是：

```text
先更新数据库，再删除缓存
```

---

## 七、这个方案有没有问题

有。

比如：数据库更新成功了，但删除缓存失败。

```text
更新数据库成功
删除 Redis 失败
```

结果：

```text
数据库是新值
Redis 仍然是旧值
```

这就是缓存和数据库不一致。

怎么降低风险？

常见做法：

- 删除缓存失败时记录日志。
- 重试删除缓存。
- 使用消息队列异步补偿删除。
- 设置合理 TTL，让旧缓存最终过期。
- 对一致性要求高的接口，读时校验版本或直接查数据库。

缓存一致性没有银弹。大多数业务追求的是最终一致。

---

## 八、最终一致是什么意思

最终一致表示：短时间内可能读到旧缓存，但系统会在某个时间点后恢复一致。

比如：

```text
文章标题更新后，有几秒钟用户可能读到旧标题
但缓存删除重试成功或 TTL 过期后，用户会读到新标题
```

这是否能接受，要看业务。

文章详情、商品描述、用户昵称通常可以接受短暂旧数据。

支付状态、订单状态、库存扣减等核心场景就要谨慎，不能简单依赖普通缓存。

---

## 九、写流程伪代码

文章更新流程：

```text
func UpdateArticle(ctx, id, input):
    tx = db.Begin()
    db.UpdateArticle(tx, id, input)
    tx.Commit()

    key = ArticleDetailKey(id)
    err = redis.Del(ctx, key)
    if err != nil:
        log error
        send retry task

    return success
```

注意：数据库事务提交成功后再删缓存。

如果数据库更新失败，不应该删除缓存。否则缓存被删了，下次读取又查数据库旧值再写回缓存，虽然不会造成最终错误，但会制造没必要的缓存抖动。

---

## 十、删除哪些缓存

更新一条数据时，可能影响多个缓存。

比如更新文章标题，可能影响：

```text
article:detail:1001
article:list:page:1:size:20
author:articles:9
rank:article:daily:20260628
```

不是所有缓存都适合立即删除。

常见策略：

- 精确详情缓存：更新后删除。
- 列表缓存：TTL 短一些，或按业务删除相关页。
- 排行榜：如果分数没变，不一定删除。
- 聚合缓存：根据依赖关系设计失效策略。

缓存越多，失效策略越复杂。

所以一开始不要缓存太多东西。先缓存最热点、最明确、最容易失效的对象。

---

## 十一、为什么不能追求绝对强一致

如果要求 Redis 缓存和数据库每一瞬间都完全一致，系统复杂度会很高。

你需要处理：

- 并发读写。
- 数据库事务和 Redis 操作无法天然同事务。
- Redis 删除失败。
- 网络超时。
- 服务崩溃。
- 主从延迟。
- 重试幂等。

Redis 和数据库不是同一个存储系统，普通业务代码无法轻易让它们像一个强一致事务一样工作。

所以缓存设计要先问：

```text
这个业务能接受多久的旧数据？
```

答案决定策略。

---

## 十二、常见错误

### 1. 更新数据库后直接更新缓存

如果缓存构建复杂或并发写入多，容易写入不完整或旧数据。

### 2. 先删缓存再更新数据库

并发读可能把旧数据库值写回缓存。

### 3. 删除缓存失败不处理

会导致旧缓存持续存在。

### 4. 缓存没有 TTL

一旦删除失败，旧缓存可能长期存在。

### 5. 缓存范围过大

更新一个字段要删除大量缓存，维护成本很高。

## 十四、本节小结

这一节你学习了 Cache-Aside 的写流程。

你需要记住：

- 数据库通常是权威数据源。
- 写流程常见做法是：先更新数据库，再删除缓存。
- 删除缓存通常比更新缓存更简单、更稳。
- 先删缓存再更新数据库，可能让并发读把旧值写回缓存。
- 更新数据库后删除缓存也有失败风险，需要重试、补偿和 TTL 兜底。
- 缓存一致性通常追求最终一致，不是绝对强一致。
