# 5. 缓存击穿：热点 key 失效与互斥重建

缓存击穿是指某个热点 key 在过期瞬间，大量请求同时未命中缓存，然后一起打到数据库。

它和缓存穿透不同。穿透查询的是不存在的数据；击穿查询的是存在的数据，只是热点 key 刚好失效了。

学完这一节后，你应该能够：

- 理解缓存击穿是什么。
- 区分缓存击穿和缓存穿透。
- 知道热点 key 过期为什么危险。
- 使用互斥锁控制缓存重建。
- 理解热点 key 永不过期和逻辑过期的思路。
- 知道这些方案的适用场景和局限。

---

## 一、什么是缓存击穿

假设首页有一篇超级热门文章。

缓存 key：

```text
article:detail:1001
```

平时请求都命中 Redis。

某一刻，这个 key 过期了。

此时大量请求同时进来：

```text
请求 1：Redis 未命中 -> 查数据库
请求 2：Redis 未命中 -> 查数据库
请求 3：Redis 未命中 -> 查数据库
...
```

本来只需要一次数据库查询重建缓存，结果变成大量请求一起打数据库。

这就是缓存击穿。

---

## 二、击穿和穿透的区别

| 问题 | 查询数据是否存在 | 核心原因 |
| --- | --- | --- |
| 缓存穿透 | 数据不存在 | Redis 和数据库都没有，反复查数据库 |
| 缓存击穿 | 数据存在 | 热点 key 过期瞬间，大量请求并发重建 |

例子：

```text
穿透：请求 article:999999999，数据库也没有
击穿：请求 article:1001，数据库有，但缓存刚过期
```

解决方式也不同。

穿透常用空值缓存、Bloom Filter。

击穿常用互斥锁、逻辑过期、热点 key 保护。

---

## 三、为什么热点 key 危险

普通 key 过期时，即使有请求回源数据库，压力也不大。

热点 key 不一样。

它可能每秒有成百上千请求。

如果它过期，短时间内所有请求都可能穿过 Redis。

结果：

- 数据库 QPS 突增。
- 接口延迟上升。
- 数据库连接池耗尽。
- 其他正常请求受影响。
- 缓存重建反复竞争。

所以热点 key 需要特殊保护。

---

## 四、方案一：互斥锁重建缓存

核心思路：缓存未命中时，只允许一个请求去查数据库并重建缓存，其他请求等待或稍后重试。

流程：

```text
1. 查 Redis，未命中
2. 尝试获取重建锁
3. 获取锁成功：查数据库，写缓存，释放锁
4. 获取锁失败：等待一小段时间后重试查 Redis
```

锁 key：

```text
lock:rebuild:article:detail:1001
```

加锁命令：

```redis
SET lock:rebuild:article:detail:1001 request_id NX EX 10
```

这里 `NX` 表示只有锁不存在时才能创建。

`EX 10` 防止服务崩溃后锁永远不释放。

---

## 五、互斥重建伪代码

```text
func GetArticle(id):
    key = ArticleDetailKey(id)
    cached = redis.Get(key)
    if cached exists:
        return cached

    lockKey = "lock:rebuild:" + key
    locked = redis.Set(lockKey, requestID, NX, EX 10)

    if locked:
        article = db.FindArticle(id)
        redis.Set(key, articleJSON, EX ttl)
        releaseLock(lockKey, requestID)
        return article

    sleep 50ms
    cached = redis.Get(key)
    if cached exists:
        return cached

    return retry or degrade
```

重点：只有拿到锁的请求负责查数据库和写缓存。

其他请求不要同时查数据库。

---

## 六、释放锁要校验 value

释放锁时不能简单：

```redis
DEL lock:rebuild:article:detail:1001
```

因为锁可能已经过期，又被其他请求拿到了。

如果你直接 `DEL`，可能删掉别人的锁。

更安全的方式是：锁 value 使用唯一 request_id，释放时先判断 value 是否等于自己，再删除。

这个判断和删除要用 Lua 脚本保证原子性。

分布式锁后面会专门讲。当前阶段先记住：锁 value 不能随便写，释放锁不能误删别人的锁。

---

## 七、互斥锁方案的优点

优点：

- 能明显减少数据库并发回源。
- 实现思路清晰。
- 适合热点 key 缓存重建。
- 和 Cache-Aside 兼容。

比如 1000 个请求同时未命中，理论上只有 1 个请求查数据库，其他请求等待缓存重建。

这对数据库保护很有价值。

---

## 八、互斥锁方案的局限

局限：

- 等待请求会增加延迟。
- 锁时间不好设置。
- 重建缓存失败时要处理。
- 锁释放要防误删。
- 高并发下等待和重试也会带来压力。
- 如果热点 key 很多，每个 key 都可能竞争锁。

所以互斥锁不是万能方案。

对于极热 key，还可以考虑逻辑过期或主动预热。

---

## 九、方案二：热点 key 不设置物理过期

一种简单粗暴的方式：热点 key 不设置 Redis TTL。

例如：

```redis
SET article:detail:1001 '{...}'
```

然后由后台任务定时刷新缓存。

这样不会出现 key 过期瞬间大量请求未命中的问题。

适合：

- 首页配置。
- 热门榜单。
- 活动页内容。
- 变化频率可控的热点数据。

但要注意：不设置 TTL 意味着缓存不会自动失效，必须有可靠的主动更新或删除机制。

---

## 十、方案三：逻辑过期

逻辑过期是指 Redis key 本身不过期，但 value 里保存一个业务过期时间。

value 示例：

```json
{
  "expire_at": 1790000000,
  "data": {
    "id": 1001,
    "title": "Redis 缓存击穿"
  }
}
```

读流程：

```text
读取 Redis
如果逻辑未过期，直接返回
如果逻辑已过期，先返回旧数据
同时尝试后台异步重建缓存
```

这样用户请求不会因为缓存过期而全部打到数据库。

逻辑过期牺牲的是数据新鲜度，换来的是稳定性。

后面会有单独一篇详细讲逻辑过期。

---

## 十一、方案四：缓存预热

缓存预热是指在流量到来前，提前把热点数据写入 Redis。

比如：

- 活动开始前预热活动页数据。
- 服务启动后预热首页配置。
- 每天凌晨预热排行榜。
- 根据访问统计预热热门文章。

预热能减少首次访问未命中，也能保护数据库。

但预热要注意：

- 热点数据识别是否准确。
- 预热失败是否告警。
- 预热数据是否过期。
- 预热过程是否打爆数据库。

---

## 十二、如何选择方案

| 场景 | 推荐方案 |
| --- | --- |
| 一般热点缓存 | 互斥锁重建 |
| 极高频热点数据 | 逻辑过期或不设置物理过期 |
| 活动页、首页配置 | 预热 + 主动更新 |
| 可以接受旧数据 | 逻辑过期 |
| 不允许旧数据 | 不应简单依赖缓存，查权威数据 |

如果只是普通文章详情，互斥锁 + TTL 通常够用。

如果是首页超级热点内容，逻辑过期和预热更稳。

---

## 十三、常见错误

### 1. 热点 key 和普通 key 使用同样策略

热点 key 过期影响更大，需要特殊保护。

### 2. 所有请求都去重建缓存

缓存未命中时没有互斥，数据库会被打爆。

### 3. 锁没有过期时间

服务崩溃后锁可能永久存在。

### 4. 释放锁不校验 value

可能误删其他请求持有的锁。

### 5. 逻辑过期没有后台重建

用户会一直读旧数据。

## 十五、本节小结

这一节你学习了缓存击穿。

你需要记住：

- 缓存击穿是热点 key 过期瞬间，大量请求同时打到数据库。
- 它和缓存穿透不同，击穿的数据在数据库中存在。
- 互斥锁可以控制只有一个请求重建缓存。
- 加锁常用 `SET key value NX EX seconds`。
- 释放锁要校验 value，避免误删别人的锁。
- 极热 key 可以考虑逻辑过期、不设置物理过期、缓存预热。
- 击穿保护是在稳定性和数据新鲜度之间做取舍。
