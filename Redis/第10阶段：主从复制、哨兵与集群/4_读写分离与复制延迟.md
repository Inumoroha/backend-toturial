# 4. 读写分离与复制延迟

主从复制常被用来做读写分离。

但读写分离不是把所有读请求丢给 replica 就完事。复制延迟会带来一致性问题。

学完这一节后，你应该能够：

- 理解读写分离的价值。
- 理解复制延迟如何产生。
- 判断哪些读可以走 replica。
- 判断哪些读应该走 master。
- 在 Go 中区分主库读和从库读。

---

## 一、什么是读写分离

读写分离结构：

```text
写请求 -> master
读请求 -> replica
```

目标：

- 降低 master 读压力。
- 提升整体读吞吐。
- 让分析类、低一致性读走从节点。

例如：

```text
文章详情缓存读 replica
统计后台读 replica
用户刚更新后的个人资料读 master
```

---

## 二、复制延迟是什么

Redis 主从复制通常是异步的。

写入 master 后，需要时间传到 replica。

这段时间就是复制延迟。

时间线：

```text
T1 写 master：SET user:name:1001 Alice
T2 立刻读 replica：可能还是旧值
T3 replica 收到复制流：读到 Alice
```

复制延迟通常很短，但不是 0。

在网络抖动、全量同步、从库压力大时，延迟会变大。

---

## 三、读到旧数据的场景

用户更新昵称：

```text
PATCH /profile
```

写入 master 成功后，马上刷新页面：

```text
GET /profile
```

如果 `GET` 走 replica，可能读到旧昵称。

这就是读写分离最常见的问题：

```text
写后读不一致。
```

---

## 四、哪些读可以走 replica

适合走 replica：

- 允许短暂旧数据的缓存。
- 列表页。
- 统计查询。
- 排行榜展示。
- 非关键配置读取。
- 后台报表。

这些场景通常追求吞吐和可用性。

---

## 五、哪些读应该走 master

应该走 master：

- 写入后立即读取。
- 登录态、权限、风控状态。
- 支付状态。
- 订单状态。
- 用户刚更新的资料。
- 强一致要求更高的业务。

如果业务不能接受旧数据，就不要从 replica 读。

---

## 六、如何降低写后读问题

常见策略：

### 1. 写后短时间读 master

用户更新资料后，接下来几秒内该用户读 master。

### 2. 关键接口始终读 master

订单、支付、权限等读 master。

### 3. 使用本地标记

写成功后在请求上下文或用户会话里标记：

```text
force_master_read_until = now + 3s
```

### 4. 业务接受最终一致

列表、统计、推荐位可以读 replica。

---

## 七、Go 中分开 master 和 replica client

简单方式：

```go
master := redis.NewClient(&redis.Options{
    Addr: "redis-master:6379",
})

replica := redis.NewClient(&redis.Options{
    Addr: "redis-replica:6379",
})
```

封装：

```go
type RedisClients struct {
    Master  *redis.Client
    Replica *redis.Client
}
```

业务方法明确选择：

```go
func (c *RedisClients) GetStrong(ctx context.Context, key string) (string, error) {
    return c.Master.Get(ctx, key).Result()
}

func (c *RedisClients) GetEventually(ctx context.Context, key string) (string, error) {
    return c.Replica.Get(ctx, key).Result()
}
```

命名要表达一致性语义。

---

## 八、不要偷偷自动读从

有些封装会自动把所有 `GET` 发到 replica。

这很危险。

因为调用方可能不知道自己读到的是最终一致数据。

更好的做法是显式：

```text
StrongRead
EventuallyRead
```

让业务代码意识到一致性选择。

---

## 九、如何观察复制延迟

```redis
INFO replication
```

关注 offset 差距：

```text
master_repl_offset
slave_repl_offset
```

还可以关注：

```text
master_link_status
master_last_io_seconds_ago
```

业务监控里也可以记录：

- replica 连接状态。
- offset 差距。
- 从库读错误。
- 主从切换次数。

---

## 十、replica 故障时怎么办

如果 replica 不可用：

可选策略：

- 读 master。
- 返回降级数据。
- 返回错误。
- 暂时关闭读写分离。

一般缓存读可以回退 master。

但如果 master 已经压力很大，全部读回 master 可能导致雪崩。

要结合限流和降级。

---

## 十一、读写分离和缓存一致性

如果 Redis 本身是缓存，读写分离还要叠加数据库一致性问题。

例如：

```text
更新数据库
删除 master 上缓存
replica 还没同步删除
读 replica 读到旧缓存
```

这在强一致场景要特别小心。

缓存已经是最终一致，再读 replica 会让旧数据窗口更复杂。

---

## 十二、常见错误

### 1. 所有读都走 replica

写后读可能读到旧数据。

### 2. 重要状态读 replica

订单、支付、权限等容易出事故。

### 3. replica 故障后所有请求打 master

可能压垮 master。

### 4. 不监控复制延迟

从库落后很久仍然被用于读请求。

### 5. 封装隐藏一致性选择

调用方不知道自己在读最终一致数据。

## 十四、本节小结

这一节你学习了读写分离和复制延迟。

你需要记住：

- 读写分离可以提升读能力。
- Redis 复制通常异步，所以 replica 可能读到旧数据。
- 写后读、订单、支付、权限等应读 master。
- 可以用写后短时间强制读 master 降低旧数据问题。
- 读写分离封装要显式表达一致性语义。
