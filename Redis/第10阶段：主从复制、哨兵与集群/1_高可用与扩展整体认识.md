## 一、单 Redis 实例的问题

单实例 Redis 很简单：

```text
应用 -> Redis
```

但生产中会遇到问题：

- Redis 进程挂了，服务不可用。
- 机器宕机，Redis 不可用。
- 单节点内存容量有限。
- 单节点 CPU 和网络能力有限。
- 读流量太高。
- 运维升级或迁移时需要停机。

所以需要复制、高可用和集群。

---

## 二、主从复制解决什么

主从复制结构：

```text
应用写入 -> master
master 复制数据 -> replica
应用可以从 replica 读取
```

作用：

- 复制数据副本。
- 提高读能力。
- 为故障切换准备从节点。
- 备份或分析场景可以读从节点。

但主从复制本身不自动故障转移。

主节点挂了，还需要 Sentinel 或人工切换。

---

## 三、Sentinel 解决什么

Sentinel 是哨兵。

它负责：

- 监控 master 和 replica。
- 判断 master 是否下线。
- 多个 Sentinel 投票确认故障。
- 自动选择一个 replica 提升为新 master。
- 通知客户端新的 master 地址。

结构：

```text
Sentinel 集群
   |
监控 Redis master/replica
```

Sentinel 主要解决：

```text
主从架构中的自动故障转移。
```

---

## 四、Redis Cluster 解决什么

Redis Cluster 用来做分片和高可用。

它把数据分到 16384 个 hash slot：

```text
slot 0-5460      -> master-1
slot 5461-10922  -> master-2
slot 10923-16383 -> master-3
```

每个 master 可以有 replica。

Cluster 解决：

- 单节点内存不够。
- 单节点吞吐不够。
- 数据按槽位分布到多个节点。
- 节点故障时可以通过 replica 接管。

Cluster 比 Sentinel 更复杂。

---

## 五、高可用不等于没有故障

高可用的意思不是永不出错。

它更接近：

```text
故障发生后，系统能在较短时间内自动恢复服务。
```

在故障转移期间可能出现：

- 短暂不可用。
- 客户端连接断开。
- 写请求失败。
- 旧 master 上未复制的数据丢失。
- 读到旧数据。
- 客户端需要重连或刷新拓扑。

所以应用层仍然要有：

- 超时。
- 重试。
- 降级。
- 幂等。
- 日志和监控。

---

## 六、复制不是强一致

Redis 主从复制通常是异步复制。

写入 master 成功后，并不代表 replica 已经立刻收到。

如果 master 刚写入就宕机，而这条写入还没复制到 replica：

```text
故障转移后，这条数据可能丢失。
```

这就是为什么高可用不等于零数据丢失。

核心业务数据仍然应该保存在数据库中。

---

## 七、三种部署形态对比

| 形态 | 解决重点 | 复杂度 |
| --- | --- | --- |
| 单实例 | 本地开发、简单缓存 | 低 |
| 主从 + Sentinel | 自动故障转移、读扩展 | 中 |
| Redis Cluster | 分片扩展、高可用 | 高 |

一般学习顺序：

```text
单实例 -> 主从复制 -> Sentinel -> Cluster
```

不要一上来就 Cluster。

先理解复制和故障转移，再理解分片。

---

## 八、Go 客户端也要配合

不同部署方式，Go 客户端配置不同。

单实例：

```go
redis.NewClient(...)
```

Sentinel：

```go
redis.NewFailoverClient(...)
```

Cluster：

```go
redis.NewClusterClient(...)
```

客户端要能：

- 感知新 master。
- 处理连接重建。
- 处理 MOVED/ASK 重定向。
- 设置合理超时和重试。

## 十、常见错误

### 1. 以为主从复制会自动切主

主从复制只复制数据，自动切主需要 Sentinel 或 Cluster。

### 2. 以为 replica 永远和 master 一致

复制通常是异步的，会有延迟。

### 3. 以为高可用不会丢数据

故障转移时可能丢失未复制的数据。

### 4. Cluster 里随便执行多 key 命令

多 key 必须在同一个 hash slot。

### 5. 客户端不配置超时和重试

故障转移期间请求可能卡住或失败。

## 十二、本节小结

这一节建立了第 10 阶段的整体地图。

你需要记住：

- 主从复制提供数据副本和读扩展。
- Sentinel 提供主从架构下的自动故障转移。
- Redis Cluster 提供分片扩展和高可用。
- Redis 复制通常是异步的，故障转移可能丢数据。
- 应用层要配合超时、重试、幂等和降级。
