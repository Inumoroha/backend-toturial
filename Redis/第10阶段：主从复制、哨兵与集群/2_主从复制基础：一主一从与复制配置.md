# 2. 主从复制基础：一主一从与复制配置

主从复制是 Redis 高可用和读扩展的基础。

它让一个 Redis 节点作为 master，其他节点作为 replica，从 master 同步数据。

学完这一节后，你应该能够：

- 理解 master 和 replica 的角色。
- 使用 `replicaof` 配置从节点。
- 查看复制状态。
- 用 Docker Compose 启动一主一从。
- 知道主从复制的基本限制。

---

## 一、主从复制是什么

结构：

```text
client write -> master
master       -> replica
client read  -> replica
```

master 负责写入。

replica 从 master 同步数据。

一个 master 可以有多个 replica。

---

## 二、为什么需要 replica

replica 的作用：

- 提供数据副本。
- 分担读流量。
- 支持故障转移。
- 可用于备份、统计、只读查询。

但 replica 通常不接收写入。

写入应该发送到 master。

---

## 三、命令配置 replica

在从节点执行：

```redis
REPLICAOF 127.0.0.1 6379
```

旧版本也可能看到：

```redis
SLAVEOF 127.0.0.1 6379
```

现在推荐使用 `REPLICAOF`。

取消复制：

```redis
REPLICAOF NO ONE
```

这会让 replica 变成独立 master。

---

## 四、配置文件方式

从节点配置：

```conf
replicaof redis-master 6379
```

如果 master 有密码：

```conf
masterauth yourpassword
```

生产环境更推荐配置文件或编排系统管理，而不是手工命令。

---

## 五、查看复制状态

```redis
INFO replication
```

master 上关注：

```text
role:master
connected_slaves
master_replid
master_repl_offset
```

replica 上关注：

```text
role:slave
master_host
master_port
master_link_status
slave_repl_offset
```

有些字段仍然使用 `slave` 命名，这是历史兼容。

---

## 六、Docker Compose 一主一从

目录：

```text
redis-replication-lab/
  docker-compose.yml
```

示例：

```yaml
services:
  redis-master:
    image: redis:7
    container_name: redis-master
    ports:
      - "6379:6379"

  redis-replica:
    image: redis:7
    container_name: redis-replica
    command: ["redis-server", "--replicaof", "redis-master", "6379"]
    ports:
      - "6380:6379"
    depends_on:
      - redis-master
```

启动：

```bash
docker compose up -d
```

---

## 七、验证复制

连接 master：

```bash
docker exec -it redis-master redis-cli
```

写入：

```redis
SET name redis
```

连接 replica：

```bash
docker exec -it redis-replica redis-cli
```

读取：

```redis
GET name
```

如果复制正常，能读到：

```text
redis
```

---

## 八、replica 默认只读

在 replica 执行写入：

```redis
SET name other
```

通常会报错：

```text
READONLY You can't write against a read only replica.
```

这是合理的。

如果 replica 允许写，写入的数据不会自动同步回 master，容易造成混乱。

---

## 九、主从复制是异步的

大多数情况下：

```text
master 写成功后，异步复制给 replica。
```

这意味着：

- replica 可能短时间读不到最新数据。
- master 宕机时，未复制的数据可能丢失。

如果业务要求读到最新写入，不应该马上去 replica 读。

---

## 十、复制拓扑

常见：

```text
一主一从
一主多从
```

也可以链式复制：

```text
master -> replica-1 -> replica-2
```

链式复制可以减轻 master 复制压力，但拓扑更复杂，延迟也可能增加。

学习阶段先掌握一主一从和一主多从。

---

## 十一、主从复制不能自动故障转移

如果 master 挂了：

```text
replica 不会自动变成 master。
```

除非使用 Sentinel 或 Cluster。

手动切换可以在 replica 执行：

```redis
REPLICAOF NO ONE
```

但生产环境不建议靠人工临时操作作为主要方案。

---

## 十二、常见错误

### 1. 写请求发到 replica

replica 默认只读，写入会失败。

### 2. 以为 replica 读取一定最新

复制有延迟。

### 3. master 挂了以为 replica 会自动接管

单纯主从复制不会自动故障转移。

### 4. 不查看 INFO replication

复制断开很久可能没人发现。

### 5. 主从机器放同一故障点

同一台物理机故障时主从一起挂。

## 十四、本节小结

这一节你学习了主从复制基础。

你需要记住：

- master 负责写，replica 同步数据并提供副本。
- `REPLICAOF host port` 可以配置从节点。
- `INFO replication` 可以查看复制状态。
- replica 默认只读。
- 主从复制通常是异步的，会有复制延迟。
- 单纯主从复制不会自动故障转移。
