# 1. 持久化、过期与内存管理整体认识

前面阶段我们一直把 Redis 当作缓存、计数器、排行榜、锁和消息组件来使用。

从这一阶段开始，要理解 Redis 作为一个生产基础组件时更底层的问题：

```text
数据怎么保存？
过期 key 怎么删除？
内存不够时会发生什么？
Big Key 为什么危险？
```

学完这一节后，你应该能够：

- 理解 Redis 持久化解决什么问题。
- 区分 RDB、AOF 和混合持久化。
- 知道 key 过期后不一定马上消失。
- 理解内存淘汰策略的作用。
- 知道 Big Key 对内存和阻塞的影响。
- 建立第 9 阶段的整体地图。

---

## 一、Redis 为什么需要持久化

Redis 主要在内存中读写数据。

如果没有持久化，Redis 进程重启后数据就会丢失。

持久化的目标是：

```text
把内存中的数据以某种形式保存到磁盘，
在 Redis 重启时尽量恢复数据。
```

但要注意：

```text
Redis 可以持久化，不等于 Redis 一定适合作为唯一数据库。
```

是否能丢数据、最多能丢多少、恢复要多久，都要结合业务判断。

---

## 二、RDB 是什么

RDB 是快照。

可以理解为：

```text
在某个时间点，把 Redis 内存数据拍一张照片，保存成 dump.rdb。
```

常见命令：

```redis
BGSAVE
```

特点：

- 文件紧凑。
- 恢复速度通常较快。
- 适合备份。
- 两次快照之间的数据可能丢失。

如果 Redis 在两次 RDB 之间崩溃，最近写入的数据可能无法恢复。

---

## 三、AOF 是什么

AOF 是追加日志。

可以理解为：

```text
把每次写命令追加到日志文件里。
重启时重新执行这些命令恢复数据。
```

常见配置：

```conf
appendonly yes
appendfsync everysec
```

特点：

- 数据安全性通常比 RDB 更好。
- 最多可能丢最近 1 秒数据，取决于刷盘策略。
- 文件可能更大。
- 需要 AOF 重写压缩日志。

---

## 四、混合持久化是什么

混合持久化是 Redis 的一种 AOF 重写格式。

它会把：

```text
RDB 快照 + AOF 增量
```

组合到 AOF 文件里。

好处：

- 恢复速度接近 RDB。
- 数据安全性接近 AOF。

生产中如果开启 AOF，通常建议了解并开启混合持久化。

---

## 五、过期 key 怎么删除

设置 TTL：

```redis
SET login:token:abc user:1001 EX 3600
```

很多人以为：

```text
3600 秒一到，key 立刻被删除。
```

实际 Redis 主要结合两种策略：

- 惰性删除。
- 定期删除。

惰性删除：

```text
访问 key 时发现过期，再删除。
```

定期删除：

```text
Redis 周期性抽样检查一部分过期 key。
```

所以过期 key 不一定在过期瞬间马上消失。

---

## 六、内存不够会发生什么

Redis 可以配置最大内存：

```conf
maxmemory 1gb
```

当 Redis 使用内存达到上限时，会根据淘汰策略处理。

例如：

```conf
maxmemory-policy allkeys-lru
```

含义：

```text
从所有 key 中选择最近最少使用的 key 淘汰。
```

如果策略是：

```conf
maxmemory-policy noeviction
```

内存满后，写命令可能直接报错。

---

## 七、常见内存淘汰策略

常见策略：

| 策略 | 含义 |
| --- | --- |
| `noeviction` | 不淘汰，写入报错 |
| `allkeys-lru` | 从所有 key 中按 LRU 近似淘汰 |
| `volatile-lru` | 从设置了 TTL 的 key 中按 LRU 近似淘汰 |
| `allkeys-lfu` | 从所有 key 中按 LFU 近似淘汰 |
| `volatile-lfu` | 从设置了 TTL 的 key 中按 LFU 近似淘汰 |
| `allkeys-random` | 从所有 key 中随机淘汰 |
| `volatile-ttl` | 从设置 TTL 的 key 中优先淘汰快过期的 |

缓存场景常见：

```text
allkeys-lru
allkeys-lfu
```

混合业务要谨慎选择。

---

## 八、Big Key 为什么危险

Big Key 指 value 很大或元素很多的 key。

例如：

- 一个 String 存 20MB JSON。
- 一个 List 有 100 万个元素。
- 一个 Hash 有 50 万个 field。
- 一个 Set 有 100 万个 member。

Big Key 的风险：

- 占用内存大。
- 网络传输慢。
- 删除可能阻塞。
- 迁移和复制成本高。
- 慢命令和延迟抖动。

Big Key 是生产 Redis 排查中非常常见的问题。

## 十、生产思维

Redis 的生产配置要先问：

```text
1. Redis 里保存的是缓存，还是重要业务状态？
2. 允许丢数据吗？允许丢多少？
3. 重启恢复时间能接受多久？
4. 内存超过上限时应该拒绝写入还是淘汰旧数据？
5. key 是否都有合理 TTL？
6. 是否存在 Big Key？
7. 是否有备份和恢复演练？
```

如果这些问题答不出来，Redis 配置就容易变成“默认能跑就行”。

---

## 十一、常见错误

### 1. 以为开启持久化就不会丢数据

RDB 和 AOF 都有丢数据窗口，只是大小不同。

### 2. 把 Redis 当唯一数据库

Redis 可以持久化，但多数后端系统仍应该用 MySQL、PostgreSQL 等保存核心数据。

### 3. 以为 TTL 到期立刻删除

Redis 使用惰性删除和定期删除结合。

### 4. 不配置 maxmemory

Redis 可能持续占用内存，最终影响机器稳定性。

### 5. 忽略 Big Key

Big Key 会导致慢查询、阻塞、复制和迁移问题。

## 十三、本节小结

这一节建立了第 9 阶段的整体地图。

你需要记住：

- Redis 持久化用于重启恢复，但不等于绝对不丢数据。
- RDB 是快照，AOF 是追加日志，混合持久化结合两者。
- key 过期后通过惰性删除和定期删除清理。
- 内存满时由 `maxmemory-policy` 决定淘汰或拒绝写入。
- Big Key 会带来内存、网络、阻塞和运维风险。
