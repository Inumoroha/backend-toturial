# 5. key 过期删除策略：惰性删除与定期删除

Redis 中很多 key 都会设置 TTL。

但 key 到期后，并不一定在那个瞬间立刻从内存中消失。Redis 会结合惰性删除和定期删除来清理过期 key。

学完这一节后，你应该能够：

- 理解 TTL 到期和 key 删除不是同一件事。
- 理解惰性删除。
- 理解定期删除。
- 知道过期 key 对内存的影响。
- 用命令观察 TTL 和过期行为。

---

## 一、设置过期时间

常见命令：

```redis
SET login:token:abc user:1001 EX 3600
EXPIRE article:cache:1001 1800
TTL article:cache:1001
```

TTL 返回：

| 返回值 | 含义 |
| --- | --- |
| 正数 | 剩余秒数 |
| `-1` | key 存在但没有过期时间 |
| `-2` | key 不存在 |

---

## 二、过期不等于立刻删除

假设：

```redis
SET temp:key value EX 10
```

10 秒后，这个 key 逻辑上已经过期。

但 Redis 不一定在第 10 秒瞬间删除它。

删除时机主要来自：

- 惰性删除。
- 定期删除。

这是一种性能和内存之间的取舍。

---

## 三、惰性删除

惰性删除是：

```text
客户端访问某个 key 时，Redis 先检查它是否过期。
如果过期，就删除并当作不存在。
```

例如：

```redis
GET temp:key
```

如果 `temp:key` 已经过期，Redis 会删除它，并返回：

```text
(nil)
```

优点：

- 不主动扫描所有 key，CPU 成本低。

缺点：

- 如果过期 key 一直没人访问，可能暂时占用内存。

---

## 四、定期删除

定期删除是：

```text
Redis 周期性抽样检查一部分设置了过期时间的 key，
发现过期就删除。
```

它不是一次性扫描全库。

原因很简单：

```text
全库扫描会阻塞 Redis。
```

Redis 会控制每次检查的时间和数量，尽量避免影响正常请求。

---

## 五、为什么不精确定时删除

如果每个 key 到期都创建一个定时器：

```text
1000 万个带 TTL 的 key -> 1000 万个定时器
```

管理成本会很高。

Redis 选择惰性删除 + 定期删除，是为了在 CPU 和内存之间取得平衡。

---

## 六、大量 key 同时过期的风险

如果大量 key 设置相同 TTL：

```text
100 万个 key 都在 12:00:00 过期
```

可能导致：

- Redis 短时间集中删除。
- CPU 抖动。
- 缓存大量失效。
- 数据库回源压力暴增。

缓存阶段已经讲过，可以给 TTL 加随机值：

```text
30 分钟 + 0 到 10 分钟随机值
```

这能降低过期集中爆发。

---

## 七、查看过期相关信息

```redis
INFO stats
```

关注：

```text
expired_keys
evicted_keys
```

含义：

- `expired_keys`：因为过期被删除的 key 数量。
- `evicted_keys`：因为内存淘汰被删除的 key 数量。

过期删除和内存淘汰不是一回事。

---

## 八、过期删除和内存淘汰的区别

过期删除：

```text
key 自己设置了 TTL，到期后被删除。
```

内存淘汰：

```text
Redis 达到 maxmemory 后，根据策略主动删除一些 key。
```

即使 key 没到期，也可能因为内存淘汰被删除。

这就是为什么缓存代码必须能处理缓存未命中。

---

## 九、过期 key 和持久化

RDB 生成时：

```text
过期 key 通常不会保存到 RDB。
```

AOF 中：

```text
key 过期删除时，会记录删除相关命令。
```

理解这一点即可。

更重要的是业务层：

```text
不要依赖过期 key 在精确时间点消失来实现严格业务。
```

例如订单超时取消，不建议只靠 key 过期事件做核心逻辑。

---

## 十、key 过期通知要谨慎

Redis 支持 keyspace notification，可以监听过期事件。

但它不适合做关键业务唯一触发器。

原因：

- 配置默认未开启。
- 事件可能受删除时机影响。
- 消费者不在线可能丢事件。
- 可靠性不如任务表或 Stream。

订单自动取消这类业务，更推荐：

- 数据库任务表。
- ZSet 延迟队列。
- Stream。
- 专业 MQ。

## 十一、Go 中 TTL 检查

```go
ttl, err := rdb.TTL(ctx, key).Result()
if err != nil {
    return err
}

switch {
case ttl > 0:
    fmt.Println("has ttl", ttl)
case ttl == -1:
    fmt.Println("exists but no ttl")
case ttl == -2:
    fmt.Println("not exists")
}
```

注意 `TTL` 返回的是 `time.Duration`。

go-redis 会把 `-1` 和 `-2` 表示成负的 duration。

---

## 十二、常见错误

### 1. 以为 TTL 到期瞬间 key 一定消失

Redis 删除过期 key 有惰性和定期策略。

### 2. 大量 key 使用相同 TTL

可能造成过期集中和缓存雪崩。

### 3. 把过期事件当可靠消息

过期通知不适合作为核心业务唯一触发。

### 4. 混淆 expired_keys 和 evicted_keys

前者是过期删除，后者是内存淘汰。

### 5. 重要业务只靠 TTL 表达状态

状态流转最好落到数据库或可靠任务系统。

## 十四、本节小结

这一节你学习了 key 过期删除策略。

你需要记住：

- TTL 到期和物理删除不是同一时刻。
- Redis 使用惰性删除和定期删除结合清理过期 key。
- 大量 key 同时过期会带来 CPU 和回源压力。
- 过期删除和内存淘汰是不同机制。
- key 过期事件不适合作为关键业务的可靠消息。
