# 17. 生产落地检查清单

本节目标：你能把数据结构练习改造成更接近真实后端项目的设计，并知道哪些地方必须靠数据库、缓存、队列和迁移工具兜底。

简短引入：数据结构能让内存里的数据组织得更好，但后端工程不只是在内存里写代码。真实项目还要面对用户输入、数据库事务、索引成本、缓存一致性、消息重试、上线回滚和故障恢复。

## 一、为什么需要它

同一个 map，在练习里可以保存订单；在生产里，如果你只靠 map 去重，服务重启后就会失效。同一个队列，在练习里可以用 channel；在生产里，如果任务不能丢，就需要任务表或可靠消息队列。

可以理解为：数据结构解决局部效率和代码组织，工程兜底解决数据正确性和生产风险。

```text
任何影响钱、库存、权限和用户数据的操作，都不能只靠内存数据结构保证正确。
```

## 二、基本用法

写完一个数据结构练习后，用下面 8 个问题检查：

1. 数据是否来自用户输入？
2. 是否需要持久化？
3. 是否会被多个请求并发访问？
4. 是否要求唯一性？
5. 是否需要事务保证多个步骤一起成功或一起失败？
6. 是否需要索引加速查询？
7. 是否有缓存，缓存失效后怎么办？
8. 上线失败是否能回滚？

如果其中任何一个答案是“是”，就不能只停留在内存示例。

## 三、关键参数/语法/代码结构

### 参数化查询

用户输入不能直接拼进 SQL 字符串。比如短链接 code、用户 ID、订单号都可能来自请求。

```text
用户输入不能直接拼进 SQL 字符串。
```

练习里可以用 map 模拟查询，但真实项目中要用数据库驱动或 ORM 提供的参数绑定能力。这样数据库会把输入当作值，而不是 SQL 语句的一部分。

### 事务边界

事务适合保护同一个数据库里的多步变更，比如创建订单和扣减库存。如果步骤跨数据库、缓存和消息队列，就要额外考虑消息可靠性、重试和补偿。

```text
能放进同一个数据库事务的关键写操作，优先放进同一个事务。
```

### 唯一索引

map 可以提前判断重复，但生产唯一性要靠数据库唯一索引。常见场景是用户邮箱、订单号、短链接 code、幂等 key。

唯一索引会增加写入成本，所以不要随便给所有字段都加索引。只给高频查询、唯一约束和明确排序需求的字段加。

### 迁移工具

表结构变更不要靠手动 SQL 随意执行。真实项目通常使用迁移工具记录版本，保证开发、测试、生产环境一致。

迁移前要问：

- 这个变更是否兼容旧代码？
- 大表加字段或索引是否会锁表？
- 失败后怎么回滚？
- 是否需要先灰度发布代码，再迁移数据？

### 缓存一致性

LRU、Redis、本地 map 都可能过期或丢失。缓存更新要考虑数据库事务是否已经提交。事务没提交就更新缓存，可能把错误状态暴露给用户。

常见保守做法是：先更新数据库，提交成功后删除缓存，让下次读取重新加载。

### 队列可靠性

channel 适合进程内练习和非关键异步任务。订单、支付、库存、发货这类任务必须可追踪、可重试、可补偿。

```text
关键业务消息不能只放内存队列。
```

## 四、真实后端场景示例

以短链接创建为例，练习版可能是：

1. 检查 code 是否为空。
2. 检查 map 中是否存在。
3. 写入 map。

生产版至少要变成：

1. 校验 code 格式和保留前缀。
2. 使用参数化查询或 ORM 写入数据库。
3. 数据库对 code 建唯一索引。
4. 创建操作带幂等 key，避免重复提交。
5. 写入成功后再删除或更新缓存。
6. 访问日志异步写入可靠队列或日志系统。
7. 表结构变更走迁移工具。
8. 上线前准备回滚方案。

可以看到，数据结构仍然有用：Trie 做保留前缀，Bloom Filter 防穿透，LRU 加速热点读取，队列削峰。但最终正确性不能只靠它们。

## 五、注意点

不要把“能跑”误认为“能上线”。练习代码为了突出数据结构，会省略数据库、HTTP、日志、配置、测试和部署。进入真实项目时，要逐步补齐这些工程边界。

判断是否能上线，可以看三个问题：

- 数据错了能不能发现？
- 任务失败能不能重试？
- 发布失败能不能回滚？

如果答案不明确，就还不能算生产就绪。

## 六、常见误区

- 误区一：用 map 保证订单号唯一。服务重启、多实例并发都会破坏这个假设。
- 误区二：先更新缓存再提交事务。事务失败时缓存已经污染。
- 误区三：给所有查询字段都加索引。索引会占空间，也会拖慢写入。
- 误区四：迁移脚本没有回滚说明。生产失败时只能手忙脚乱修数据。
- 误区五：队列消费失败只打印日志。失败任务应该能查询、能重试、能告警。
- 误区六：并发锁范围不清。锁太大影响吞吐，锁太小保护不住业务不变量。

## 七、本节达标标准

- 能说明内存数据结构和生产持久化之间的边界。
- 能知道哪些场景必须使用参数化查询。
- 能为订单、库存、短链接这类业务说明事务边界。
- 能判断唯一索引、普通索引和内存 map 分别解决什么问题。
- 能说明缓存更新、队列重试、数据库迁移的常见风险。
- 能在上线前写出基本回滚和故障处理思路。
