# 16. 阶段练习与验收清单

本节目标：你能按清单检查每一节是否真正学会，并把零散知识推进到一个后端项目里。

简短引入：教程读完不等于能力形成。后端工程师需要能把结构选出来、代码写出来、风险说出来、异常处理补上。本节给你一份可执行的练习清单。

## 一、为什么需要它

很多学习者会在“我好像懂了”和“我能独立写出来”之间卡住。验收清单的作用，就是把模糊感觉变成明确证据。

```text
判断是否学会，不看是否记住定义，而看能否在业务场景里正确使用。
```

## 二、基本用法

每学完一篇文章，至少完成三件事：

1. 复制示例并运行成功。
2. 改一个业务字段或业务规则。
3. 写下一个常见错误以及你会怎么避免。

建议为练习新建一个目录：

```powershell
mkdir go-ds-practice
cd go-ds-practice
go mod init go-ds-practice
```

```bash
mkdir go-ds-practice
cd go-ds-practice
go mod init go-ds-practice
```

## 三、关键参数/语法/代码结构

练习代码建议按后端分层组织，不需要一开始就很复杂：

```text
cmd/        程序入口
internal/   业务代码
pkg/        可复用的小工具
```

第一轮如果只是单文件运行也可以，但进入综合项目时，建议至少拆出 `service` 和 `storage`。这样你会更早理解：数据结构通常不应该散落在 handler 里，而应该被封装在明确的业务模块中。

## 四、真实后端场景示例

下面是按章节安排的练习任务：

| 章节 | 必做练习 | 后端判断标准 |
| --- | --- | --- |

| 5. 队列 | 写异步日志 worker | 能处理队列满和任务失败 |
| 6. 链表与 LRU | 写短链接热点缓存 | 能说明并发加锁和缓存一致性 |

## 五、注意点

每个练习都要补错误路径。后端项目中，真正容易出问题的通常不是正常路径，而是输入为空、重复提交、队列满、缓存未命中、任务重试、服务重启这些情况。

```text
只写成功路径的代码，不能算后端项目代码。
```

## 六、常见误区

- 误区一：练习只打印结果，不返回错误。真实 service 函数要让上层知道失败原因。
- 误区二：没有边界测试。分页、二分、队列、堆都很容易在边界上出错。
- 误区三：把所有逻辑写在 `main` 函数。后续无法测试，也不符合后端分层习惯。
- 误区四：忽略生产替代方案。内存结构要知道将来替换成数据库、Redis 或消息队列的位置。
- 误区五：没有可回滚意识。涉及迁移、索引、缓存更新时，都要知道失败后怎么退。

## 七、本节达标标准

- 能逐章完成练习，并跑通代码。
- 能为每个练习补至少 3 个边界输入。
- 能写出每个数据结构在真实后端中的使用边界。
- 能把练习代码整理成短链接综合项目。
- 能说明哪些地方需要参数化查询、事务、唯一索引、迁移工具和可回滚方案。
