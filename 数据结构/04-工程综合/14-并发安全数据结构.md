# 14. 并发安全数据结构

本节目标：你能判断一个内存数据结构是否需要加锁，并用 `sync.RWMutex` 保护 map。

简短引入：Go 后端服务通常会同时处理多个请求。只要多个 goroutine 会同时读写同一个 map、切片或缓存，就必须考虑并发安全。

## 一、为什么需要它

本地测试只有一个请求时，普通 map 看起来没问题。上线后多个请求同时读写，可能直接报 `concurrent map writes`。并发问题最麻烦的是不稳定，越到高峰越容易出现。

```text
普通 map 不能并发读写；共享数据必须有明确的并发保护策略。
```

`main.go`：

```go
package main

import (
	"fmt"
	"sync"
)

type UserCache struct {
	mu sync.RWMutex
	data map[int]string
}

func NewUserCache() *UserCache {
	return &UserCache{data: make(map[int]string)}
}

func (c *UserCache) Set(id int, name string) {
	c.mu.Lock()
	defer c.mu.Unlock()
	c.data[id] = name
}

func (c *UserCache) Get(id int) (string, bool) {
	c.mu.RLock()
	defer c.mu.RUnlock()
	name, ok := c.data[id]
	return name, ok
}

func main() {
	cache := NewUserCache()
	cache.Set(1, "Alice")
	fmt.Println(cache.Get(1))
}
```

## 三、关键参数/语法/代码结构

- `Lock` 用于写操作。
- `RLock` 用于读操作，允许多个读并发。
- `defer Unlock` 能降低忘记解锁的风险。
- 锁保护的是共享数据，不是某一行代码。

## 四、真实后端场景示例

用户权限缓存会被多个请求读取，也可能在后台配置变更时更新。可以用 `RWMutex` 包住 map。读多写少时，`RWMutex` 比普通 `Mutex` 更合适。

如果只是简单 key-value 且读写模式特殊，也可以考虑 `sync.Map`。但 `sync.Map` 不是所有场景都更快，业务结构清晰时，普通 map 加锁更容易维护。

## 五、注意点

锁的范围要小，但不能小到保护不完整。比如先读库存再扣库存，如果读和写不在同一个锁或数据库事务里，仍然会超卖。

```text
库存扣减这类关键状态，优先使用数据库事务和条件更新兜底。
```

## 六、常见误区

- 误区一：只给写加锁，不给读加锁。读写并发仍然有风险。
- 误区二：锁里做慢操作，比如网络请求，导致其他请求阻塞。
- 误区三：用内存锁保护分布式多实例。不同实例之间的锁互相看不见。
- 误区四：把并发安全和业务正确性混为一谈。加锁不等于事务成功。
