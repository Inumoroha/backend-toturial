# 15. 综合项目：短链接与任务调度

本节目标：你能把 map、队列、堆、Trie、LRU 和并发锁组合成一个后端风格的小项目设计。

简短引入：单个数据结构学会后，真正的训练是组合使用。这个项目模拟短链接服务：创建短链接、访问跳转、缓存热点、拦截保留前缀，并异步记录访问日志。

## 一、为什么需要它

短链接业务小，但很适合练后端数据结构：

- `map`：保存 code 到长链接的映射。
- `Trie`：检查保留前缀，如 `api`、`admin`。
- `LRU`：缓存热点短链接。
- `队列`：异步写访问日志。
- `堆`：调度过期清理任务。
- `RWMutex`：保护共享内存。

```text
练习项目可以用内存实现；生产项目必须把最终状态放进数据库或可靠存储。
```

`main.go`：

```go
package main

import (
	"fmt"
	"sync"
	"time"
)

type ShortLinkService struct {
	mu    sync.RWMutex
	links map[string]string
	logs  chan string
}

func NewService() *ShortLinkService {
	s := &ShortLinkService{
		links: make(map[string]string),
		logs:  make(chan string, 100),
	}
	go s.logWorker()
	return s
}

func (s *ShortLinkService) Create(code, url string) error {
	if code == "" || url == "" {
		return fmt.Errorf("code and url are required")
	}
	if code == "api" || code == "admin" {
		return fmt.Errorf("reserved code")
	}
	s.mu.Lock()
	defer s.mu.Unlock()
	if _, exists := s.links[code]; exists {
		return fmt.Errorf("code already exists")
	}
	s.links[code] = url
	return nil
}

func (s *ShortLinkService) Resolve(code string) (string, error) {
	s.mu.RLock()
	url, ok := s.links[code]
	s.mu.RUnlock()
	if !ok {
		return "", fmt.Errorf("short link not found")
	}
	select {
	case s.logs <- "visit " + code:
	default:
		fmt.Println("log queue full, drop log")
	}
	return url, nil
}

func (s *ShortLinkService) logWorker() {
	for log := range s.logs {
		fmt.Println("async log:", log)
	}
}

func main() {
	svc := NewService()
	fmt.Println(svc.Create("go101", "https://go.dev"))
	fmt.Println(svc.Resolve("go101"))
	time.Sleep(100 * time.Millisecond)
}
```

## 三、关键参数/语法/代码结构

- `links` 是内存索引，真实项目应替换为数据库表和唯一索引。
- `RWMutex` 保护 map，防止并发读写崩溃。
- `logs` 是有缓冲 channel，避免访问接口被日志写入拖慢。
- `select default` 表示队列满时不阻塞主请求，这适合非关键日志，不适合订单消息。

## 四、真实后端场景示例

可以按下面方向继续扩展：

1. 用 Trie 替代 `code == "api"` 的保留前缀判断。
2. 用 LRU 缓存热点 code，减少数据库查询。
3. 用 Bloom Filter 拦截明显不存在的 code。
4. 用堆保存过期时间，定期清理失效短链接。
5. 用数据库事务保证创建短链接和写审计日志的边界。

如果要接入 HTTP，建议先用标准库 `net/http`，把 `Create` 和 `Resolve` 当作 service 层方法，不要让 handler 直接操作 map。

## 五、注意点

创建短链接时，唯一性必须由数据库唯一索引兜底。访问日志可以异步，但订单、支付、库存这类关键消息不能随便丢弃。不同业务的重要性不同，队列满时的策略也不同。

```text
非关键日志可以降级丢弃；关键业务消息必须持久化并可重试。
```

## 六、常见误区

- 误区一：内存 map 当数据库。服务重启后数据全部丢失。
- 误区二：日志队列满了还阻塞主请求，导致访问接口被拖垮。
- 误区三：所有异步任务都允许丢弃。关键任务必须可靠。
- 误区四：handler、service、storage 混在一个函数里，后期难测试、难替换。
- 误区五：没有迁移计划。生产表结构变更必须使用迁移工具并准备回滚。

## 七、本节达标标准

- 能把多个数据结构组合成一个后端服务雏形。
- 能说明每个结构在短链接项目中的职责。
- 能区分练习内存实现和生产持久化实现。
- 能对日志、订单、库存等不同重要性的任务选择不同可靠性策略。
- 能意识到唯一索引、事务边界、迁移工具和可回滚方案的重要性。
