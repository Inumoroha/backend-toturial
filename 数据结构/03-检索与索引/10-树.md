# 10. 树

本节目标：你能用树组织菜单、分类、部门和评论这类层级数据。

简短引入：后端常见的层级数据很多，比如后台权限菜单、商品分类、组织架构、评论回复。树能表达“一个父节点下面有多个子节点”的关系。

## 一、为什么需要它

数据库里通常用 `id` 和 `parent_id` 存层级关系，但接口返回给前端时常常要组装成树。可以理解为：数据库适合存扁平记录，接口适合返回层级结构。

```text
树形接口组装前，要先限制数据范围，避免一次加载过深、过大的层级。
```

`main.go`：

```go
package main

import "fmt"

type Menu struct {
	ID       int
	ParentID int
	Name     string
	Children []*Menu
}

func BuildTree(items []Menu) []*Menu {
	byID := make(map[int]*Menu)
	var roots []*Menu
	for i := range items {
		byID[items[i].ID] = &items[i]
	}
	for i := range items {
		node := &items[i]
		if node.ParentID == 0 {
			roots = append(roots, node)
			continue
		}
		if parent, ok := byID[node.ParentID]; ok {
			parent.Children = append(parent.Children, node)
		}
	}
	return roots
}

func main() {
	menus := []Menu{{1, 0, "系统", nil}, {2, 1, "用户管理", nil}, {3, 1, "角色管理", nil}}
	fmt.Printf("%+v\n", BuildTree(menus))
}
```

## 三、关键参数/语法/代码结构

- `ParentID == 0` 表示根节点，这只是约定，项目里要统一。
- `byID` 用来快速找到父节点。
- `Children []*Menu` 使用指针，避免复制子树。
- 组树前最好校验是否有不存在的父节点。

## 四、真实后端场景示例

权限菜单接口通常先查出当前用户拥有的菜单，再组装树返回。不要先查全量菜单再在内存里过滤权限，否则容易把无权限节点误带出去。

真实项目中还要处理排序字段，比如 `Sort int`。同一层子节点可以用 `sort.SliceStable` 排序，保证前端展示稳定。

## 五、注意点

树结构要警惕环。比如 A 的父节点是 B，B 的父节点又是 A，递归遍历会死循环。菜单、部门这类数据应在写入时校验，数据库也要有外键或应用层约束。

## 六、常见误区

- 误区一：递归不限制深度，异常数据导致栈溢出或死循环。
- 误区二：全量加载组织架构。数据大时接口会很慢。
- 误区三：权限过滤放在组树之后，可能泄露节点信息。
- 误区四：不处理孤儿节点。父节点不存在时数据会悄悄丢失。

## 七、本节达标标准

- 能把扁平 `id,parent_id` 数据组装成树。
- 能说明树在菜单、分类、部门中的用途。
- 能识别环、孤儿节点和深度过大的风险。
- 能理解权限过滤应尽量前置。
