# 7. 排序

本节目标：你能用 Go 排序实现后台列表、排行榜和稳定展示。

简短引入：后端接口经常要按创建时间、价格、优先级、更新时间排序。排序不是为了好看，而是为了让用户每次看到的结果稳定、可解释。

## 一、为什么需要它

如果订单列表每次刷新顺序都变，运营和客服会很难定位问题。真实项目中，排序优先交给数据库；Go 内存排序适合对已经取出的少量数据做二次整理。

```text
大数据量排序优先使用数据库索引，不要把全表拉到内存排序。
```

## 二、基本用法

```powershell

go mod init ds-sort-demo
notepad main.go
go run .
```

```bash
mkdir ds-sort-demo && cd ds-sort-demo

```

`main.go`：

```go
package main

import (
	"fmt"
	"sort"
)

type Order struct {
	ID        int
	Amount    int
	CreatedAt int64
}

func main() {
	orders := []Order{{1, 99, 30}, {2, 188, 10}, {3, 188, 20}}
	sort.SliceStable(orders, func(i, j int) bool {
		if orders[i].Amount == orders[j].Amount {
			return orders[i].CreatedAt < orders[j].CreatedAt
		}
		return orders[i].Amount > orders[j].Amount
	})
	fmt.Println(orders)
}
```

## 三、关键参数/语法/代码结构

- `sort.Slice` 使用比较函数决定顺序。
- `sort.SliceStable` 在比较结果相同时保留原相对顺序，更适合用户可见列表。
- 比较函数要表达严格顺序，不要写成 `<=`。
- 多字段排序要明确主次，比如金额降序，再按创建时间升序。

## 四、真实后端场景示例

后台订单列表常见排序是“高风险订单优先，再按创建时间”。数据库可以先查出一页候选数据，Go 再按业务分数补充排序：

```go
func riskScore(o Order) int {
	score := 0
	if o.Amount > 1000 {
		score += 10
	}
	if o.CreatedAt < 100 {
		score += 1
	}
	return score
}
```

这类排序适合数据量小、规则变化快的场景。如果规则稳定并且数据量大，应考虑把排序字段落库并建索引。

## 五、注意点

索引能提升排序和查询速度，但会增加写入成本。真实项目中新增索引要评估表大小、写入频率、上线方式和回滚方案。

## 六、常见误区

- 误区一：接口层全量查询再排序分页。数据量一大就会慢。
- 误区二：比较函数写错，造成排序结果不稳定。
- 误区三：忽略稳定排序。用户看到同分数据跳来跳去。
- 误区四：随便加索引。索引不是免费午餐，会影响写入和迁移。

## 七、本节达标标准

- 能用 `sort.SliceStable` 对结构体切片排序。
- 能写多字段排序逻辑。
- 能说明数据库排序和内存排序的边界。
- 能理解索引对读写性能和迁移风险的影响。
