# 3. 结构体与组合

本节目标：你能用结构体把后端业务数据组织成清晰、可维护的模型。

简短引入：后端代码不是只处理单个变量，而是处理用户、订单、文章、库存这样的业务对象。结构体能把相关字段放在一起，让函数签名和业务含义更清楚。

## 一、为什么需要它

如果一个函数需要 `userID, userName, email, role, createdAt` 一串参数，调用方很容易传错。结构体可以理解为“业务数据的最小包装”。真实项目中通常会区分数据库模型、接口请求、接口响应，避免一个结构体到处复用。

```text
不要为了省事让数据库模型、请求参数、响应结构完全共用一个结构体。
```

## 二、基本用法

```powershell
mkdir ds-struct-demo; cd ds-struct-demo
go mod init ds-struct-demo
notepad main.go
go run .
```

```bash
mkdir ds-struct-demo && cd ds-struct-demo
go mod init ds-struct-demo
cat > main.go
go run .
```

`main.go`：

```go
package main

import "fmt"

type CreateUserRequest struct {
	Name  string
	Email string
}

type User struct {
	ID    int
	Name  string
	Email string
}

func CreateUser(req CreateUserRequest) (User, error) {
	if req.Name == "" || req.Email == "" {
		return User{}, fmt.Errorf("name and email are required")
	}
	return User{ID: 1, Name: req.Name, Email: req.Email}, nil
}

func main() {
	user, err := CreateUser(CreateUserRequest{Name: "Alice", Email: "a@example.com"})
	fmt.Println(user, err)
}
```

## 三、关键参数/语法/代码结构

- 字段首字母大写表示包外可访问，JSON 序列化时也常需要导出字段。
- 请求结构体只放用户能提交的字段，不放 `ID`、`CreatedAt` 这类服务端生成字段。
- 返回 `(User, error)` 是 Go 后端常见写法，业务失败不要用 panic。
- 结构体零值要小心，比如 `ID == 0` 可能表示未保存。

## 四、真实后端场景示例

创建订单时，可以把输入和内部订单拆开：

```go
type CreateOrderRequest struct {
	UserID    int
	ProductID int
	Quantity  int
}

type Order struct {
	ID        int
	UserID    int
	ProductID int
	Quantity  int
	Status    string
}

func NewOrder(req CreateOrderRequest) (Order, error) {
	if req.Quantity <= 0 {
		return Order{}, fmt.Errorf("quantity must be positive")
	}
	return Order{
		UserID: req.UserID, ProductID: req.ProductID,
		Quantity: req.Quantity, Status: "created",
	}, nil
}
```

## 五、注意点

结构体设计要保守。字段越多，耦合越强。真实项目里，数据库迁移要配合结构体变更，新增字段要考虑默认值、历史数据、回滚方案和接口兼容。

## 六、常见误区

- 误区一：一个结构体走天下。请求、数据库、响应的职责混在一起，后期很难改。
- 误区二：用 panic 表示参数错误。后端服务应返回 error 并让上层决定响应码。
- 误区三：忽略字段零值。`0`、`""`、`false` 在业务里可能有不同含义。
- 误区四：字段随便暴露。用户不能提交的字段不应该出现在请求结构体里。

## 七、本节达标标准

- 能用结构体表达用户、订单等业务对象。
- 能区分请求结构体、领域结构体和响应结构体。
- 能写基本参数校验并返回 error。
- 能理解结构体变更和数据库迁移、接口兼容之间的关系。
