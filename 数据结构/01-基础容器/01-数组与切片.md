# 1. 数组与切片

本节目标：你能用 Go 切片处理后端项目中的列表、分页和批量操作。

简短引入：后端服务经常返回用户列表、文章列表、订单列表。Go 里最常用的列表结构是切片，数组更多是底层固定长度存储，日常业务代码里较少直接使用。

## 一、为什么需要它

可以理解为，切片就是“可增长的业务列表”。常见场景是查询出一批订单，再过滤、分页、组装响应。真实项目中通常会先从数据库拿数据，再在内存里做少量整理。

```text
切片适合处理一批已经取出的数据，不适合替代数据库分页和过滤。
```

## 二、基本用法

新建目录后运行：

```powershell
mkdir ds-slice-demo; cd ds-slice-demo
go mod init ds-slice-demo
notepad main.go
go run .
```

```bash
mkdir ds-slice-demo && cd ds-slice-demo
go mod init ds-slice-demo
cat > main.go
go run .
```

`main.go`：

```go
package main

import "fmt"

type User struct {
	ID   int
	Name string
}

func main() {
	users := []User{
		{ID: 1, Name: "Alice"},
		{ID: 2, Name: "Bob"},
		{ID: 3, Name: "Chen"},
	}

	users = append(users, User{ID: 4, Name: "Dora"})

	page := users[1:3]
	fmt.Println("all:", users)
	fmt.Println("page:", page)
}
```

## 三、关键参数/语法/代码结构

- `[]User{}` 表示创建一个用户切片。
- `append` 返回新的切片，所以要写回 `users = append(users, ...)`。
- `users[1:3]` 是左闭右开，包含下标 1，不包含下标 3。
- 切片背后引用同一段底层数组，截取出来的 `page` 不是深拷贝。

## 四、真实后端场景示例

分页时不要一次把全表读到内存再切片。更合理的方式是数据库先分页，Go 切片只负责承接结果：

```go
func PageUsers(users []User, page, pageSize int) []User {
	if page < 1 || pageSize <= 0 {
		return nil
	}
	start := (page - 1) * pageSize
	if start >= len(users) {
		return []User{}
	}
	end := start + pageSize
	if end > len(users) {
		end = len(users)
	}
	return users[start:end]
}
```

真实项目中，`page` 和 `pageSize` 来自用户输入，必须限制最大页大小，防止一次查询拖垮服务。

## 五、注意点

切片扩容时可能换到底层新数组，所以不要假设 append 后所有旧引用都还指向同一块数据。批处理时也要控制每批大小，比如每 500 条写一次数据库，失败后更容易重试。

```text
用户传入的 pageSize 必须设置上限，例如 100 或 500。
```

## 六、常见误区

- 误区一：把切片分页当数据库分页。数据量大时会占用大量内存。
- 误区二：忘记接收 append 返回值，导致新增数据没有生效。
- 误区三：越界截取，比如 `users[10:20]` 在长度不足时会 panic。
- 误区四：返回内部切片后又修改原切片，造成接口响应数据被意外改变。

## 七、本节达标标准

- 能创建、追加、遍历和截取切片。
- 能写出安全的内存分页函数。
- 能说明为什么生产项目应优先让数据库做分页。
- 能避免 append 返回值、越界和共享底层数组的常见问题。
