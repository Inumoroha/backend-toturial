# 4. 栈

本节目标：你能用栈处理“后进先出”的业务流程，比如撤销、解析和补偿步骤。

简短引入：栈可以理解为一摞盘子，最后放上去的最先拿出来。后端里它常用于表达一组需要反向执行的步骤，例如导入失败后的补偿操作。

## 一、为什么需要它

真实项目中，一个流程可能包含多步：创建订单、扣库存、写日志、发送消息。前面几步成功，后面失败时，常常要按相反顺序补偿。栈能自然记录这些步骤。

```text
补偿不是数据库事务的替代品，能放进同一个事务的操作应优先放进事务。
```

## 二、基本用法

```powershell

go run .
```

`main.go`：

```go
package main

import "fmt"

type Stack[T any] struct {
	items []T
}

func (s *Stack[T]) Push(v T) {
	s.items = append(s.items, v)
}

func (s *Stack[T]) Pop() (T, bool) {
	var zero T
	if len(s.items) == 0 {
		return zero, false
	}
	last := len(s.items) - 1
	v := s.items[last]
	s.items = s.items[:last]
	return v, true
}

func main() {
	var stack Stack[string]
	stack.Push("create order")
	stack.Push("deduct stock")

	for {
		step, ok := stack.Pop()
		if !ok {
			break
		}
		fmt.Println("rollback:", step)
	}
}
```

## 三、关键参数/语法/代码结构

- `Push` 用 `append` 把元素放到末尾。
- `Pop` 取最后一个元素，再缩短切片。
- `ok` 用来表示栈是否为空，避免空栈访问 panic。
- 泛型 `Stack[T any]` 让同一个栈能装不同类型的数据。

## 四、真实后端场景示例

假设订单流程中库存扣减成功，但消息发送失败。可以把可补偿步骤压栈：

```go
type UndoFunc func()

func PlaceOrder() {
	var undo Stack[UndoFunc]

	fmt.Println("create order")
	undo.Push(func() { fmt.Println("cancel order") })

	fmt.Println("deduct stock")
	undo.Push(func() { fmt.Println("restore stock") })

	err := fmt.Errorf("send message failed")
	if err != nil {
		for fn, ok := undo.Pop(); ok; fn, ok = undo.Pop() {
			fn()
		}
	}
}
```

生产中跨数据库、缓存、消息队列的操作不能简单靠内存补偿解决，通常要结合事务消息、状态机和重试表。

## 五、注意点

栈适合进程内的短流程，不适合保存长期任务状态。服务重启后内存会消失，关键业务步骤必须落库。

## 六、常见误区

- 误区一：空栈直接取最后一个元素，导致 panic。
- 误区二：把内存补偿当分布式事务。服务崩溃后补偿函数不会执行。
- 误区三：补偿顺序写反。创建顺序是 A、B、C，回滚通常应是 C、B、A。
- 误区四：补偿函数没有错误处理。真实项目里补偿失败也要记录并重试。

## 七、本节达标标准

- 能用切片实现栈的 Push 和 Pop。
- 能解释后进先出的业务直觉。
- 能用栈记录简单补偿步骤。
- 能说明栈不能替代数据库事务和持久化重试。
