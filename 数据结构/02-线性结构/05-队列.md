# 5. 队列

本节目标：你能用队列表达先进先出的任务处理流程，并理解它在异步后端任务中的作用。

简短引入：队列可以理解为排队办业务，先来的先处理。后端里常见于日志缓冲、邮件发送、订单通知、异步导出等任务。

## 一、为什么需要它

同步接口要尽快返回，不能让用户一直等邮件、短信、报表生成。常见做法是主流程写数据库，然后把后续工作放进队列，由后台 worker 慢慢处理。

```text
队列能削峰，但不能掩盖下游长期处理能力不足的问题。
```

## 二、基本用法

```powershell
mkdir ds-queue-demo; cd ds-queue-demo
go mod init ds-queue-demo
notepad main.go
go run .
```

```bash
mkdir ds-queue-demo && cd ds-queue-demo
go mod init ds-queue-demo
cat > main.go
go run .
```

`main.go`：

```go
package main

import "fmt"

type Queue[T any] struct {
	items []T
}

func (q *Queue[T]) Enqueue(v T) {
	q.items = append(q.items, v)
}

func (q *Queue[T]) Dequeue() (T, bool) {
	var zero T
	if len(q.items) == 0 {
		return zero, false
	}
	v := q.items[0]
	q.items = q.items[1:]
	return v, true
}

func main() {
	var q Queue[string]
	q.Enqueue("send order email")
	q.Enqueue("write audit log")

	for task, ok := q.Dequeue(); ok; task, ok = q.Dequeue() {
		fmt.Println("handle:", task)
	}
}
```

## 三、关键参数/语法/代码结构

- `Enqueue` 入队，放到尾部。
- `Dequeue` 出队，从头部取。
- 这个示例简单清楚，但频繁 `q.items = q.items[1:]` 可能保留底层数组引用。
- 生产项目常用消息队列、channel 或环形队列，而不是无限增长的切片。

## 四、真实后端场景示例

Go 的 channel 很适合做进程内队列：

```go
package main

import "fmt"

type Task struct {
	OrderID int
	Name    string
}

func worker(tasks <-chan Task) {
	for task := range tasks {
		fmt.Println("processing:", task.OrderID, task.Name)
	}
}

func main() {
	tasks := make(chan Task, 10)
	go worker(tasks)

	tasks <- Task{OrderID: 1001, Name: "send email"}
	tasks <- Task{OrderID: 1002, Name: "write log"}
	close(tasks)
}
```

真实项目中，跨进程任务通常使用 Redis Stream、Kafka、RabbitMQ 等外部队列，并配合重试、死信和幂等。

## 五、注意点

接口返回前如果只写入内存队列，服务崩溃会丢任务。关键任务应该先持久化，比如写任务表，再由 worker 扫描执行。

## 六、常见误区

- 误区一：无界队列。请求高峰时内存会被打满。
- 误区二：任务没有幂等。重试可能导致重复发货、重复扣款。
- 误区三：只打印错误不落库。失败任务无法追踪和补偿。
- 误区四：把 channel 当分布式队列。channel 只在当前进程内有效。

## 七、本节达标标准

- 能实现基本先进先出队列。
- 能用 channel 写一个简单 worker。
- 能说明内存队列、channel、外部消息队列的区别。
- 能理解队列使用时必须考虑持久化、重试和幂等。
