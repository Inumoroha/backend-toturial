# 1. Go 后端数据结构学习路线

本节目标：你能知道学习数据结构的顺序，并明白每个阶段和后端开发的关系。

简短引入：后端开发每天都在处理数据：用户列表、订单状态、库存扣减、权限菜单、日志聚合、任务调度。数据结构的作用，就是帮你把这些数据放在合适的位置，用合适的方式查询、修改和保护。

## 一、为什么需要它

很多初学者学数据结构时会先陷入“背定义”。但后端项目更关心三个问题：

- 数据从哪里来，比如 HTTP 请求、数据库、消息队列、缓存。
- 数据要怎么组织，比如列表、字典、树、队列。
- 数据出错时怎么控制风险，比如事务、幂等、回滚、并发安全。

可以理解为：数据结构是业务代码的骨架。骨架搭错了，后面再补日志、缓存、索引都会越来越难。

## 二、学习阶段

### 第一阶段：Go 基础容器

目标：能熟练使用切片、map、结构体表达业务数据。

建议学习：

- 数组与切片：列表、分页、批处理。
- map 与 set：按用户 ID、订单号、权限编码快速查找。
- 结构体组合：把业务对象组织清楚，避免到处传散乱参数。

达标项目：写一个内存版用户列表服务，支持新增、查询、分页、去重。

### 第二阶段：线性结构

目标：理解数据进入和离开系统的顺序。

建议学习：

- 栈：撤销、解析、调用链、事务补偿思路。
- 队列：异步任务、日志缓冲、削峰。
- 链表与 LRU：缓存淘汰。

达标项目：写一个内存任务队列，模拟订单创建后异步发送通知。

### 第三阶段：检索与索引

目标：能为查询、排序、调度选择合适的数据结构。

建议学习：

- 排序：后台列表、排行榜、稳定展示。
- 二分查找：有序配置、时间区间、版本选择。
- 堆：延迟任务、优先级任务。
- 树：部门、菜单、分类、评论。
- Trie：前缀搜索、路由匹配、短链接前缀。
- 图：权限依赖、迁移依赖、服务调用关系。

达标项目：写一个权限菜单生成器，并能检测依赖是否成环。

### 第四阶段：工程综合

目标：把数据结构放进真实后端约束中使用。

建议学习：

- Bloom Filter：缓存穿透防护。
- 并发安全结构：锁、channel、sync.Map 的使用边界。
- 综合项目：短链接或订单任务调度系统。

达标项目：写一个带去重、延迟任务、缓存淘汰、并发保护的内存服务。

## 三、学习方式

每个主题建议按这个顺序练：

1. 用一个业务问题建立直觉。
2. 写最小可运行代码。
3. 解释关键代码。
4. 改造成更贴近真实项目的版本。
5. 总结误区和达标标准。

```text
初学阶段不要追求一次写出“通用框架”，先把一个具体业务场景写清楚。
```

配套阅读顺序建议：

1. 先读本路线，建立全局视角。
2. 再读 [阶段学习计划与项目产出](./02-阶段学习计划与项目产出.md)，确定自己的学习节奏。
3. 每学完一篇教程，就对照 [阶段练习与验收清单](../04-工程综合/16-阶段练习与验收清单.md) 完成练习。
4. 准备把练习项目改得更像真实后端时，再读 [生产落地检查清单](../04-工程综合/17-生产落地检查清单.md)。

每个阶段都建议保留一份小复盘，格式可以很简单：

```text
这个结构解决什么问题：
它不适合什么场景：
生产中要靠什么兜底：
我写代码时踩到的坑：
```

## 四、真实项目中的判断标准

选择数据结构时，优先问：

- 查询多还是写入多？
- 是否要求顺序？
- 是否需要去重？
- 是否会并发访问？
- 数据量是否会增长？
- 出错后能不能回滚？

例如：订单号查订单，优先 map；按创建时间处理订单，优先队列或堆；权限菜单展示，优先树；迁移脚本依赖顺序，优先图。

## 五、注意点

数据结构学习不要脱离数据库、缓存和事务。真实后端项目中，内存结构通常只是加速或编排，最终状态常常还要落到数据库、消息队列或日志系统里。

```text
内存数据结构不能替代数据库事务，也不能替代生产环境的持久化方案。
```

## 六、常见误区

- 误区一：只刷题，不知道项目里怎么用。结果是会写反转链表，却不会设计订单去重。
- 误区二：过早追求复杂结构。很多后台功能用切片、map、排序就够了。
- 误区三：忽略并发安全。map 在并发读写下可能直接崩溃。
- 误区四：把缓存当最终数据源。缓存丢失、过期、击穿都必须有兜底。
- 误区五：只关心性能，不关心可回滚。后端工程更怕数据错，不只是怕慢。

## 七、本节达标标准

- 能说出数据结构和后端业务之间的关系。
- 能按阶段安排自己的学习顺序。
- 能判断列表、字典、队列、树、图分别适合哪些业务。
- 能意识到事务、并发安全、生产回滚对数据结构选择的影响。
