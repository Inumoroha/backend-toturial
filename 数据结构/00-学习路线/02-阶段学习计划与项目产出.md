# 2. 阶段学习计划与项目产出

本节目标：你能把数据结构学习拆成可执行的周计划，并在每个阶段产出一个能运行的小项目。

简短引入：学习路线解决“学什么”，阶段计划解决“怎么学完”。后端学习最怕只看不写，所以每一阶段都要有明确产出：代码能跑、边界能说清、风险能识别。

## 一、为什么需要它

数据结构不是一天学完的。初学时更适合按后端项目的演进顺序学习：先会表示数据，再会组织流程，再会提升查询效率，最后考虑并发、缓存和生产风险。

```text
每个阶段都要有可运行代码；没有代码产出，就很难判断自己是否真的会用。
```

## 二、基本用法

建议用 4 到 6 周完成第一轮。时间少就拉长，时间多也不要跳过练习。

| 阶段 | 建议时间 | 主题 | 阶段产出 |
| --- | --- | --- | --- |
| 第一阶段 | 1 周 | 切片、map、结构体 | 内存版用户与订单查询模块 |
| 第二阶段 | 1 周 | 栈、队列、LRU | 订单异步通知和本地缓存 |
| 第三阶段 | 1 到 2 周 | 排序、二分、堆、树、Trie、图 | 权限菜单、延迟任务、依赖检测 |
| 第四阶段 | 1 到 2 周 | Bloom Filter、并发安全、综合项目 | 短链接服务雏形 |

## 三、关键参数/语法/代码结构

每个阶段都按同一套流程推进：

1. 先复制运行教程中的最小示例。
2. 再把示例改成一个后端业务函数。
3. 给函数补 3 类输入：正常输入、空输入、异常输入。
4. 写一段说明：这个结构为什么适合当前场景。
5. 写一段说明：生产中不能只靠它，还需要什么兜底。

例如学习 map 时，不只是写 `m["a"] = 1`，还要能说明：订单号去重可以用 map 做内存预检查，但最终必须靠数据库唯一索引和幂等记录兜底。

## 四、真实后端场景示例

可以把每个阶段串成同一个短链接项目：

第一阶段：用结构体表示短链接，用 map 保存 `code -> url`，用切片返回创建记录列表。

第二阶段：用队列异步记录访问日志，用 LRU 缓存热点短链接，用栈理解失败补偿顺序。

第三阶段：用排序展示访问榜单，用二分匹配限流档位，用堆处理过期清理，用 Trie 拦截保留前缀，用图检查任务依赖。

第四阶段：用 Bloom Filter 防止随机 code 打穿数据库，用 `RWMutex` 保护内存结构，整理成 service 层、storage 层和 worker 层。

## 五、注意点

阶段项目不用一开始就接数据库，但每次都要写清楚“如果上生产，要怎么替换”。比如内存 map 要替换成数据库表加唯一索引，channel 要替换成可靠消息队列或任务表，日志打印要替换成可查询的日志系统。

```text
练习可以简化实现，但不能简化对生产风险的判断。
```

## 六、常见误区

- 误区一：只看文章不运行代码。看懂和能写之间有很大距离。
- 误区二：每学一个结构都重新换业务。最好用同一个项目持续演进。
- 误区三：过早接复杂框架。第一轮先用标准库，把数据流看清。
- 误区四：练习项目没有异常输入。后端代码的大部分质量来自边界处理。
- 误区五：没有复盘生产替代方案。内存练习和线上系统之间必须有桥。

## 七、本节达标标准

- 能按 4 到 6 周安排自己的数据结构学习节奏。
- 能说出每个阶段应该交付什么代码。
- 能把每个数据结构接到短链接或订单项目里。
- 能区分练习实现和生产实现的边界。
- 能在每个阶段写出风险、兜底和回滚思路。
