# 3. 失败处理、重试与幂等

本节目标：理解 upstream 实例失败时 Nginx 如何处理，以及为什么重试会引出 Go 业务幂等问题。

负载均衡不是“分发请求”这么简单。真实线上更重要的问题是：

```text
某个实例挂了怎么办？
请求超时后能不能重试？
POST 请求重试会不会重复写入？
```

---

## 一、模拟一个实例失败

保持三个实例：

```text
8080
8081
8082
```

停掉 `8081`。

然后请求：

```bash
for i in {1..20}; do curl -s -o /dev/null -w "%{http_code}\n" -H "Host: api.local" http://127.0.0.1/api/ping; done
```

查看错误日志：

```bash
sudo tail -n 50 /var/log/nginx/error.log
```

你可能看到：

```text
connect() failed (111: Connection refused) while connecting to upstream
```

---

## 二、max_fails 与 fail_timeout

配置：

```nginx
upstream go_api {
    server 127.0.0.1:8080 max_fails=3 fail_timeout=10s;
    server 127.0.0.1:8081 max_fails=3 fail_timeout=10s;
    server 127.0.0.1:8082 max_fails=3 fail_timeout=10s;
}
```

含义：

- `max_fails=3`：在指定时间窗口内失败 3 次后，暂时认为该实例不可用。
- `fail_timeout=10s`：失败统计窗口，也是不再选择该实例的时间窗口。

注意：这是被动失败判断，不是主动健康检查。

---

## 三、proxy_next_upstream

配置：

```nginx
location /api/ {
    proxy_pass http://go_api;
    proxy_next_upstream error timeout http_502 http_503 http_504;
    proxy_next_upstream_tries 3;
}
```

含义：

- 当前 upstream 出错时，Nginx 可以尝试下一个 upstream。
- 最多尝试 3 次。

这对 GET 请求通常比较安全，但对 POST 请求要非常谨慎。

---

## 四、为什么重试会带来幂等问题

假设请求：

```http
POST /api/orders
```

流程：

```text
Nginx -> 8080
8080 已经创建订单
8080 返回响应前连接断了
Nginx 认为失败
Nginx 重试到 8081
8081 再创建一次订单
```

结果：订单重复创建。

这就是非幂等接口重试的风险。

---

## 五、什么是幂等

幂等：同一个操作执行一次和执行多次，结果一致。

通常幂等：

- 查询用户详情。
- 查询商品详情。
- 获取配置。

通常非幂等：

- 创建订单。
- 扣库存。
- 支付扣款。
- 发短信。

---

## 六、Go 业务如何做幂等

常见方式：

- 客户端传 `Idempotency-Key`。
- 服务端用请求号去重。
- 数据库唯一索引兜底。
- 订单号、支付流水号唯一。
- Redis `SETNX` 做短期防重复。

示例：

```http
POST /api/orders
Idempotency-Key: order-20260705-0001
```

Go 服务处理前先检查这个 key 是否已经成功处理过。

## 七、不同接口的重试建议

只读接口：

```nginx
location /api/query/ {
    proxy_next_upstream error timeout http_502 http_503 http_504;
    proxy_next_upstream_tries 3;
    proxy_pass http://go_api;
}
```

写接口：

```nginx
location /api/order/ {
    proxy_next_upstream error timeout;
    proxy_next_upstream_tries 1;
    proxy_pass http://go_api;
}
```

关键写接口的可靠性更多依赖业务幂等，而不是 Nginx 重试。
