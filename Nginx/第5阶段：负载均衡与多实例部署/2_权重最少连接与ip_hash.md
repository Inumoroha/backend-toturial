# 2. 权重、最少连接与 ip_hash

本节目标：在默认轮询之外，理解 Nginx upstream 的三种常见分发策略：权重、最少连接、IP Hash，并知道它们分别适合什么场景。

上一节你已经用 `upstream` 把请求分发到三个 Go 实例：

```text
8080
8081
8082
```

默认情况下，Nginx 使用轮询。真实环境中，不同实例的机器配置、接口耗时和会话策略可能不同，这时就需要了解其他策略。

---

## 一、默认轮询

配置：

```nginx
upstream go_api {
    server 127.0.0.1:8080;
    server 127.0.0.1:8081;
    server 127.0.0.1:8082;
}
```

大致分发：

```text
8080 -> 8081 -> 8082 -> 8080 -> 8081 -> 8082
```

适合：

- 每个 Go 实例性能差不多。
- 接口耗时差不多。
- 服务是无状态的。

---

## 二、weight 权重

如果 `8080` 所在机器性能更强，可以给它更高权重：

```nginx
upstream go_api {
    server 127.0.0.1:8080 weight=3;
    server 127.0.0.1:8081 weight=1;
    server 127.0.0.1:8082 weight=1;
}
```

大致效果：

```text
8080 收到更多请求。
8081、8082 收到较少请求。
```

验证：

```bash
for i in {1..30}; do curl -s -H "Host: api.local" http://127.0.0.1/api/ping; echo; done
```

统计返回的端口数量。你应该能看到 `8080` 出现次数更多。

---

## 三、least_conn 最少连接

配置：

```nginx
upstream go_api {
    least_conn;

    server 127.0.0.1:8080;
    server 127.0.0.1:8081;
    server 127.0.0.1:8082;
}
```

含义：优先把请求分给当前活跃连接数更少的实例。

适合：

- 接口耗时差异大。
- 有些请求很慢。
- 单纯轮询可能让某个实例堆积较多慢请求。

练习方式：

1. 用 `/api/slow?seconds=10` 制造慢请求。
2. 同时请求 `/api/ping`。
3. 观察分发是否更倾向连接较少的实例。

---

## 四、ip_hash

配置：

```nginx
upstream go_api {
    ip_hash;

    server 127.0.0.1:8080;
    server 127.0.0.1:8081;
    server 127.0.0.1:8082;
}
```

含义：同一个客户端 IP 会尽量分配到同一个后端实例。

适合：

- 老项目把 session 存在进程内存里。
- 临时需要会话粘滞。

但新项目不推荐依赖它。更好的方式是让 Go 服务无状态：

- 登录态放 JWT。
- session 放 Redis。
- 用户数据放数据库。
- 文件放对象存储或共享存储。

---

## 五、策略选择建议

| 策略 | 适合场景 | 注意点 |
| --- | --- | --- |
| 轮询 | 实例能力接近，服务无状态 | 默认即可 |
| weight | 实例性能不同 | 需要根据实际压测调整 |
| least_conn | 请求耗时差异较大 | 不代表一定更快，要压测 |
| ip_hash | 需要会话粘滞 | 不推荐作为长期架构依赖 |

---

## 六、不要靠 ip_hash 解决设计问题

如果你的 Go 服务这样保存登录态：

```text
map[userID]session
```

那么多实例会出问题：

```text
第一次请求 -> 8080，登录成功。
第二次请求 -> 8081，找不到 session。
```

`ip_hash` 可以缓解，但不是根治：

- 用户换网络后 IP 变了。
- 移动网络 IP 可能变化。
- 公司出口 NAT 可能很多人同一个 IP。
- 实例扩缩容后映射可能变化。

正确方向是无状态或共享状态。

2. 给 `8080` 设置 `weight=3`，请求 30 次统计分布。
3. 配置 `least_conn`，用慢接口观察连接分配。
4. 配置 `ip_hash`，观察同一客户端请求是否倾向同一实例。
5. 写下你的 Go 服务是否依赖进程内存状态。
