# 4. 多实例会话与优雅重启

本节目标：理解多实例部署对会话状态和发布流程的影响，并掌握 Go 服务优雅关闭的基本思路。

---

## 一、多实例下的会话问题

单实例时，很多东西看起来都能工作：

```text
用户登录 -> session 放在 Go 进程内存
后续请求 -> 还是同一个进程
```

多实例后：

```text
用户登录 -> 8080
下一次请求 -> 8081
8081 内存里没有 session
```

这就是多实例会话问题。

---

## 二、不推荐的做法

不要把核心登录态只存在 Go 进程内存里：

```go
var sessions = map[string]Session{}
```

这种方式：

- 实例重启就丢。
- 多实例不共享。
- 扩容后不一致。
- 发布时容易造成用户掉线。

---

## 三、推荐会话策略

常见做法：

- JWT：登录态放在 token 中，服务校验签名。
- Redis session：session 存 Redis，多实例共享。
- 数据库存储：适合强一致但性能要注意。
- 无状态服务：Go 实例不保存用户状态。

生产中常见：

```text
Nginx -> 多个 Go 实例 -> Redis / PostgreSQL
```

---

## 四、发布时的问题

如果你直接 kill Go 进程：

```bash
kill -9 <pid>
```

正在处理的请求会被中断，用户可能看到：

- 502。
- 连接断开。
- 请求执行一半。

Go 服务应该支持优雅关闭。

---

## 五、Go 优雅关闭示例

核心思路：

```text
收到 SIGTERM
-> 停止接收新请求
-> 等待已有请求完成
-> 超时后退出
```

示例：

```go
srv := &http.Server{
    Addr:    ":8080",
    Handler: mux,
}

go func() {
    if err := srv.ListenAndServe(); err != nil && err != http.ErrServerClosed {
        log.Fatal(err)
    }
}()

quit := make(chan os.Signal, 1)
signal.Notify(quit, syscall.SIGINT, syscall.SIGTERM)
<-quit

ctx, cancel := context.WithTimeout(context.Background(), 10*time.Second)
defer cancel()

if err := srv.Shutdown(ctx); err != nil {
    log.Fatal(err)
}
```

需要 import：

```go
import (
    "context"
    "os"
    "os/signal"
    "syscall"
    "time"
)
```

---

## 六、systemd 配合优雅关闭

```ini
[Service]
ExecStart=/opt/myapp/myapp
KillSignal=SIGTERM
TimeoutStopSec=15
Restart=always
```

注意：

```text
TimeoutStopSec 应该大于 Go Shutdown 的等待时间。
```

---

## 七、滚动发布思路

多实例发布时，不要一次停掉全部实例。

推荐：

```text
停止 8080
-> 发布 8080
-> 健康检查通过
-> 发布 8081
-> 健康检查通过
-> 发布 8082
```

如果在 Kubernetes、Docker Swarm 或云平台中，滚动发布通常由平台负责，但你仍然需要理解背后的逻辑。
