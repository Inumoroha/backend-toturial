# 6. 502 与 504 故障演练

本节目标：在项目中主动制造 502 和 504，并用 Nginx 日志和 Go 日志解释故障原因。

线上排障能力不是看书看出来的，要靠可控地复现错误。

---

## 一、演练前准备

确认三个 Go 实例运行：

```bash
curl http://127.0.0.1:8080/api/ping
curl http://127.0.0.1:8081/api/ping
curl http://127.0.0.1:8082/api/ping
```

确认 Nginx 代理正常：

```bash
curl -k https://app.local/api/ping
```

打开日志：

```bash
sudo tail -f /var/log/nginx/app_access.log
```

另一个终端：

```bash
sudo tail -f /var/log/nginx/app_error.log
```

---

## 二、制造 502：停掉所有 Go 实例

停止 `8080`、`8081`、`8082`。

请求：

```bash
curl -k -i https://app.local/api/ping
```

你应该看到：

```text
502 Bad Gateway
```

error log 中可能出现：

```text
connect() failed (111: Connection refused) while connecting to upstream
```

结论：

```text
Nginx 收到了请求，但无法连接任何 Go upstream。
```

---

## 三、制造部分实例失败

只停掉 `8081`，保留 `8080`、`8082`。

连续请求：

```bash
for i in {1..20}; do curl -k -s -o /dev/null -w "%{http_code}\n" https://app.local/api/ping; done
```

观察：

- 是否偶尔出现 502。
- Nginx 是否避开失败实例。
- error log 是否记录连接失败。

这能帮助你理解 `max_fails` 和 `fail_timeout`。

---

## 四、制造 504：慢接口超过超时

确认 Nginx 中：

```nginx
proxy_read_timeout 30s;
```

请求：

```bash
curl -k -i "https://app.local/api/slow?seconds=40"
```

预期：

```text
504 Gateway Timeout
```

error log：

```text
upstream timed out while reading response header from upstream
```

access log 中可能看到：

```text
status=504 rt=30.xxx urt=30.xxx
```

结论：

```text
Go 服务响应时间超过 Nginx 读取 upstream 的超时时间。
```

---

## 五、不要只调大超时

如果接口本来应该 200ms 返回，却出现 504，应该查：

- Go handler 是否阻塞。
- 数据库是否慢查询。
- 外部接口是否超时。
- goroutine 是否堆积。
- 连接池是否耗尽。

调大：

```nginx
proxy_read_timeout 300s;
```

只能让用户等更久，不一定解决问题。

---

## 六、用 request_id 关联日志

access log：

```text
rid=abc123 request="GET /api/slow?seconds=40 HTTP/1.1" status=504 rt=30.001
```

Go 日志：

```text
request_id=abc123 method=GET path=/api/slow?seconds=40 cost=40.001s
```

这样可以证明同一次请求在 Go 中确实执行了很久。

---

## 七、本节练习

1. 停掉所有 Go 实例，复现 502。
2. 只停掉一个实例，观察 Nginx 行为。
3. 请求慢接口，复现 504。
4. 根据 error log 写出故障原因。
5. 根据 access log 的耗时字段判断慢在哪里。
6. 使用 request_id 关联 Go 日志。
