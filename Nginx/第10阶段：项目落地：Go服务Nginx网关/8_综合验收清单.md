# 8. 综合验收清单

本节目标：用一组明确命令验收最终项目，确认你真的掌握了 Go + Nginx 网关项目的核心能力。

---

## 一、启动检查

如果是本机部署，确认 Go 三实例：

```bash
curl http://127.0.0.1:8080/api/ping
curl http://127.0.0.1:8081/api/ping
curl http://127.0.0.1:8082/api/ping
```

确认 Nginx：

```bash
sudo nginx -t
sudo systemctl status nginx
```

如果是 Docker Compose：

```bash
docker compose ps
docker compose logs --tail=50 nginx
```

---

## 二、静态页面验收

```bash
curl -k https://app.local/
```

要求：

- 返回 HTML。
- 页面内容包含 `Nginx Gateway Demo`。

静态资源：

```bash
curl -k -I https://app.local/assets/style.css
```

要求：

- 状态码 200。
- `Content-Type` 是 CSS 相关类型。
- 有缓存响应头。

---

## 三、API 代理验收

```bash
curl -k https://app.local/api/ping
```

要求：

- 返回 `pong`。
- 返回端口字段。

代理头：

```bash
curl -k https://app.local/api/headers
```

要求看到：

- `x_request_id`
- `x_real_ip`
- `x_forwarded_for`
- `x_forwarded_proto`

---

## 四、负载均衡验收

```bash
for i in {1..12}; do curl -k -s https://app.local/api/ping; echo; done
```

要求：

- 能看到不同实例端口。
- access log 中 `upstream=` 显示不同后端。

查看日志：

```bash
sudo tail -n 20 /var/log/nginx/app_access.log
```

---

## 五、HTTPS 跳转验收

```bash
curl -I http://app.local/api/ping
```

要求：

- 返回 301。
- `Location` 指向 `https://app.local/api/ping`。

---

## 六、限流验收

```bash
for i in {1..100}; do curl -k -s -o /dev/null -w "%{http_code}\n" https://app.local/api/ping; done
```

要求：

- 高频请求中能看到 429。

如果没有看到，可能是请求速度不够快，或者限流配置没有加载。

---

## 七、上传验收

```bash
dd if=/dev/zero of=/tmp/20m.bin bs=1M count=20
curl -k -X POST --data-binary @/tmp/20m.bin https://app.local/api/upload
```

要求：

- `/api/upload` 能接收 20MB。
- 普通 API 不应该随便允许大请求体。

---

## 八、502 验收

停掉所有 Go 实例后请求：

```bash
curl -k -i https://app.local/api/ping
```

要求：

- 返回 502。
- error log 能看到 upstream 连接失败。

---

## 九、504 验收

```bash
curl -k -i "https://app.local/api/slow?seconds=40"
```

要求：

- 返回 504。
- error log 能看到 upstream timed out。
- access log 能看到 `rt` 和 `urt` 接近超时时间。

---

## 十、最终复盘问题

请写下答案：

1. 请求 `https://app.local/api/ping` 从客户端到 Go handler 经历了哪些步骤？
2. Nginx 如何选择 HTTPS server？
3. `/api/upload` 为什么要单独 location？
4. `X-Forwarded-Proto` 有什么作用？
5. access log 中 `upstream` 字段有什么价值？
6. 502 和 504 的排查路径分别是什么？
7. Nginx 限流和 Go 业务限流的边界在哪里？
8. 如果要发布新版本，如何避免一次停掉所有实例？

---

## 十一、完成标准

如果你能完成本清单中的所有验证，并能解释每个配置块的作用，说明你已经具备 Go 后端工程师日常工作所需的 Nginx 基础能力。
