# 3. 502、503、504 排查手册

本节目标：系统掌握 Nginx 代理 Go 服务时最常见的三个 5xx 错误，并形成固定排查顺序。

---

## 一、先建立排查顺序

遇到 5xx，不要先改配置。按下面顺序：

```text
看客户端状态码
-> 看 access.log
-> 看 error.log
-> 直连 Go
-> 看 Go 日志
-> 看端口和进程
-> 看数据库或外部依赖
```

---

## 二、502 Bad Gateway

502 通常表示 Nginx 没能从 upstream 得到有效响应。

常见原因：

- Go 服务没启动。
- upstream 端口写错。
- Go 服务崩溃。
- Docker 网络地址写错。
- Go 服务提前断开连接。

排查：

```bash
sudo tail -n 50 /var/log/nginx/error.log
curl http://127.0.0.1:8080/api/ping
ss -lntp | grep 8080
```

典型 error log：

```text
connect() failed (111: Connection refused) while connecting to upstream
```

解释：Nginx 连接 Go 端口失败。

---

## 三、503 Service Temporarily Unavailable

503 表示服务暂时不可用。

常见原因：

- upstream 全部不可用。
- 限流配置返回 503。
- 人为维护页。
- 连接限制触发。

如果你希望限流返回更明确状态码：

```nginx
limit_req_status 429;
```

排查：

```bash
sudo nginx -T | grep limit_req
sudo tail -f /var/log/nginx/access.log
sudo tail -f /var/log/nginx/error.log
```

---

## 四、504 Gateway Timeout

504 通常表示 Nginx 等 Go 响应超时。

常见原因：

- Go 接口慢。
- 数据库慢查询。
- 外部 API 慢。
- Go 服务死锁或阻塞。
- `proxy_read_timeout` 太短。

制造慢请求：

```bash
curl "http://127.0.0.1:8080/api/slow?seconds=40"
```

Nginx 配置：

```nginx
proxy_read_timeout 30s;
```

通过 Nginx 请求：

```bash
curl "http://api.local/api/slow?seconds=40"
```

error log 可能出现：

```text
upstream timed out while reading response header from upstream
```

---

## 五、不要第一反应就调大超时

看到 504，很多人会立刻改：

```nginx
proxy_read_timeout 300s;
```

这不一定正确。

先判断：

- 这个接口本来应该多快？
- 是不是数据库慢？
- 是不是外部服务慢？
- 是不是 Go goroutine 阻塞？
- 是否应该改成异步任务？

只有业务确实需要长连接或长任务，才考虑调大超时。

---

## 六、状态码对照表

| 状态码 | 常见含义 | 优先检查 |
| --- | --- | --- |
| 502 | upstream 连接失败或响应异常 | error.log、Go 是否启动 |
| 503 | 服务不可用或限流 | limit 配置、upstream 状态 |
| 504 | upstream 响应超时 | Go 慢接口、数据库、超时配置 |

---

## 七、本节练习

1. 停掉 Go 服务，制造 502。
2. 把 upstream 端口写错，制造 502。
3. 配置限流并触发 429 或 503。
4. 请求 `/api/slow?seconds=40` 制造 504。
5. 对每个错误写下 access log 和 error log 证据。
