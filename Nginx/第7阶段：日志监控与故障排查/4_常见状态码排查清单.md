# 4. 常见状态码排查清单

本节目标：整理 Nginx 场景下常见 HTTP 状态码的含义和排查方向。

---

## 一、2xx 与 3xx

### 200

请求成功。

检查：

- 响应是否来自预期 location。
- 内容是否正确。

### 301 / 302

重定向。

检查：

```bash
curl -I http://app.local/path
```

关注：

```text
Location
```

---

## 二、4xx

### 400

请求格式错误。可能来自 Nginx，也可能来自 Go。

### 401

未认证。常见于 Basic Auth 或 Go 鉴权。

### 403

无权限。

Nginx 常见原因：

- 文件权限不足。
- `deny all`。
- 目录没有 index。

### 404

资源不存在或 location 不符合预期。

排查：

```bash
sudo nginx -T
curl -v http://app.local/path
```

### 413

请求体过大。

检查：

```nginx
client_max_body_size
```

### 429

请求过多，通常是限流。

检查：

```nginx
limit_req
limit_req_status
```

### 499

Nginx 特有，表示客户端主动断开连接。

常见原因：

- 用户取消请求。
- 客户端超时。
- 网关或上游负载均衡提前断开。
- 后端响应太慢，客户端不等了。

---

## 三、5xx

### 500

服务端内部错误。可能来自 Go。

查 Go 日志。

### 502

Nginx 无法从 upstream 获取有效响应。

查：

```bash
sudo tail -n 50 /var/log/nginx/error.log
curl http://127.0.0.1:8080/api/ping
```

### 503

服务暂时不可用，可能是 upstream 不可用或限流。

### 504

upstream 响应超时。

查：

- `proxy_read_timeout`。
- Go 慢接口。
- 数据库慢查询。
- 外部依赖。

---

## 四、排查顺序模板

```text
1. curl -v 看客户端现象
2. access.log 看 Nginx 是否收到
3. error.log 看 Nginx 错误
4. 直连 Go 看后端是否正常
5. Go 日志看业务错误
6. ss/lsof 看端口
7. 数据库/Redis/外部服务继续排查
```

---

## 五、按现象快速判断

### 1. 浏览器显示打不开，curl 也连接失败

优先判断服务是否监听端口：

```bash
ss -lntp | grep ':80'
ss -lntp | grep ':443'
systemctl status nginx
```

如果 Nginx 没有监听端口，请先看：

```bash
sudo nginx -t
sudo journalctl -u nginx -n 50
```

### 2. Nginx access log 没有记录

说明请求可能根本没有到达 Nginx。

排查方向：

- DNS 是否解析到这台机器。
- hosts 是否配置正确。
- 端口是否开放。
- 云安全组或防火墙是否放行。
- 请求是否打到了另一台 Nginx。

### 3. access log 有记录，但 Go 没日志

说明请求进入了 Nginx，但没有进入 Go。

常见原因：

- 被 Nginx 直接 `return`。
- 命中了静态资源 location。
- 被 `try_files` 返回了 `index.html`。
- 被 `limit_req` 限流。
- 被 `allow/deny` 拒绝。
- location 匹配不是你以为的那个。

这时重点看：

```bash
sudo nginx -T
```

### 4. Go 有日志，但客户端收到 5xx

说明请求进入了 Go，但 Go 处理过程可能出错、超时或连接中断。

排查方向：

- Go handler 是否 panic。
- 数据库是否慢。
- 外部接口是否慢。
- Go 的超时设置是否小于 Nginx。
- 客户端是否提前断开。

## 六、面试回答模板

如果面试官问：

```text
线上接口异常，你怎么根据状态码排查？
```

可以这样回答：

```text
我会先用 curl -v 确认客户端看到的状态码和响应头，然后看 Nginx access log，确认请求是否进入 Nginx、匹配了哪个路径、状态码和耗时是多少。如果是 4xx，会优先看路径、权限、请求体大小、限流等入口层问题；如果是 5xx，会继续看 error log 和 upstream 字段。502 我会重点确认 Go 服务是否存活、端口是否正确；504 我会重点看 upstream_response_time、Go 日志、数据库和外部依赖。整个过程会用 request_id 串联 Nginx 和 Go 日志。
```

这个回答的重点是：

- 先看证据，不靠猜。
- 区分 4xx 和 5xx。
- 区分 Nginx 层和 Go 层。
- 知道 502、504 的核心差异。
