# 3. 上传大小限制与代理缓存

本节目标：掌握 `client_max_body_size` 和 `proxy_cache` 的基础用法，并理解上传限制和缓存带来的风险。

---

## 一、上传大小限制

全局限制：

```nginx
server {
    client_max_body_size 10m;
}
```

表示请求体最大 10MB。

超过后，Nginx 返回：

```text
413 Request Entity Too Large
```

---

## 二、上传接口单独放大

普通 API 限制 10MB，但上传接口允许 100MB：

```nginx
server {
    client_max_body_size 10m;

    location /api/upload {
        client_max_body_size 100m;
        proxy_pass http://127.0.0.1:8080;
    }

    location /api/ {
        proxy_pass http://127.0.0.1:8080;
    }
}
```

这样可以避免所有接口都允许大请求体。

---

## 三、测试上传限制

创建 20MB 文件：

```bash
dd if=/dev/zero of=/tmp/20m.bin bs=1M count=20
```

普通 API：

```bash
curl -X POST --data-binary @/tmp/20m.bin http://api.local/api/ping
```

可能返回 413。

上传 API：

```bash
curl -X POST --data-binary @/tmp/20m.bin http://api.local/api/upload
```

如果 `/api/upload` 允许 100MB，应能进入 Go。

---

## 四、代理缓存基础

在 `http` 上下文定义缓存路径：

```nginx
proxy_cache_path /var/cache/nginx/api_cache
                 levels=1:2
                 keys_zone=api_cache:10m
                 max_size=1g
                 inactive=10m
                 use_temp_path=off;
```

在 location 使用：

```nginx
location /api/public/ {
    proxy_cache api_cache;
    proxy_cache_valid 200 1m;
    proxy_cache_key "$scheme$request_method$host$request_uri";
    add_header X-Cache-Status $upstream_cache_status always;

    proxy_pass http://127.0.0.1:8080;
}
```

---

## 五、验证缓存

第一次请求：

```bash
curl -I http://api.local/api/public/config
```

可能看到：

```text
X-Cache-Status: MISS
```

第二次请求：

```bash
curl -I http://api.local/api/public/config
```

可能看到：

```text
X-Cache-Status: HIT
```

---

## 六、哪些接口适合缓存

适合：

- 公共配置。
- 首页公开数据。
- 商品公开详情。
- 不频繁变化的只读数据。

不适合：

- 用户个人信息。
- 订单。
- 权限相关数据。
- 带 Authorization 的请求。
- POST、PUT、DELETE。

---

## 七、避免缓存认证请求

```nginx
proxy_no_cache $http_authorization;
proxy_cache_bypass $http_authorization;
```

含义：

- 带 Authorization 的请求不写入缓存。
- 带 Authorization 的请求不读取缓存。

这是非常重要的安全边界。
