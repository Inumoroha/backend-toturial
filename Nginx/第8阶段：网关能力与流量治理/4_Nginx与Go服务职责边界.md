# 4. Nginx 与 Go 服务职责边界

本节目标：建立清晰的工程判断：哪些逻辑适合放在 Nginx，哪些逻辑必须放在 Go 服务。

---

## 一、为什么要讨论边界

Nginx 能做很多事：

- 限流。
- 缓存。
- 重写路径。
- 鉴权。
- IP 控制。
- CORS。

但能做不代表都应该做。Nginx 配置适合处理通用入口逻辑，不适合承载复杂业务规则。

---

## 二、适合放 Nginx 的逻辑

### 入口层逻辑

- HTTP 跳 HTTPS。
- 域名路由。
- 路径转发。
- 静态资源。
- TLS 终止。

### 粗粒度保护

- 单 IP 限流。
- 上传大小限制。
- 管理后台 IP 白名单。
- Basic Auth。
- 简单 CORS。

### 通用观测

- access log。
- upstream 耗时。
- request_id。

---

## 三、适合放 Go 的逻辑

### 业务规则

- 用户权限。
- 订单状态。
- 支付验签。
- 库存扣减。
- 审批流程。

### 业务级限流

- 按用户 ID。
- 按租户 ID。
- 按 API Key。
- 按套餐等级。

### 数据一致性

- 幂等。
- 事务。
- 唯一约束。
- 防重复提交。

---

## 四、典型错误边界

### 错误 1：把复杂鉴权写在 Nginx

如果鉴权依赖用户角色、数据库权限、租户配置，就应该放 Go。

Nginx 可以做：

```text
是否来自内网。
是否有 Basic Auth。
是否命中粗粒度路径限制。
```

不适合做复杂 RBAC。

### 错误 2：只靠 Nginx 限流保护登录

Nginx 可以按 IP 限流，但攻击者可以换 IP。Go 服务还应该按用户、手机号、设备、验证码场景做限制。

### 错误 3：把用户数据缓存到 Nginx

如果缓存 key 设计不严谨，可能把 A 用户数据返回给 B 用户。

---

## 五、几个具体场景怎么判断

### 1. 登录接口

Nginx 可以做：

- 单 IP 限流。
- 请求体大小限制。
- 只允许 HTTPS。
- 记录访问日志。

Go 必须做：

- 校验用户名和密码。
- 校验验证码。
- 记录失败次数。
- 按用户、手机号、设备做风控。
- 生成 token 或 session。

结论：

```text
Nginx 做入口保护，Go 做业务认证。
```

### 2. 上传接口

Nginx 可以做：

- 限制最大请求体。
- 单独配置更长超时。
- 限制上传路径访问频率。

Go 必须做：

- 校验文件类型。
- 校验用户权限。
- 计算文件 hash。
- 保存元数据。
- 处理业务归属。

### 3. 管理后台

Nginx 可以做：

- IP 白名单。
- Basic Auth。
- HTTPS。
- 独立日志。

Go 必须做：

- 管理员登录。
- RBAC 权限。
- 操作审计。
- 防止越权访问。

### 4. 公共配置接口

Nginx 可以做：

- 短时间代理缓存。
- 添加缓存命中响应头。
- 限流。

Go 必须做：

- 决定配置内容。
- 控制配置版本。
- 判断是否与用户身份相关。

如果接口返回内容因用户不同而不同，就不要简单放 Nginx 公共缓存。

## 六、职责划分表

| 功能 | Nginx | Go |
| --- | --- | --- |
| HTTPS | 主要负责 | 通常不负责 |
| 静态资源 | 主要负责 | 不推荐 |
| 反向代理 | 主要负责 | 不负责 |
| 单 IP 限流 | 适合 | 可补充 |
| 用户级限流 | 不适合 | 适合 |
| 登录鉴权 | 粗保护 | 主要负责 |
| 订单幂等 | 不适合 | 必须负责 |
| access log | 适合 | 也应有业务日志 |
| 数据缓存 | 仅公共只读 | 业务缓存 |

---

## 七、面试回答模板

如果面试官问：

```text
哪些逻辑适合放 Nginx，哪些适合放 Go？
```

可以这样回答：

```text
我会把 Nginx 理解为入口层，适合做通用、无业务状态、规则相对固定的事情，比如 HTTPS、反向代理、静态资源、单 IP 限流、上传大小限制、Basic Auth、IP 白名单和统一访问日志。Go 服务负责业务规则，比如用户鉴权、RBAC、订单幂等、事务、支付验签、用户级限流和业务缓存。判断标准是：如果逻辑依赖用户、租户、数据库状态或业务语义，通常应该放 Go；如果是入口层通用控制，可以放 Nginx。
```

这个回答能体现：

- 你知道 Nginx 的能力。
- 你不会滥用 Nginx。
- 你理解业务逻辑和网关逻辑的边界。
