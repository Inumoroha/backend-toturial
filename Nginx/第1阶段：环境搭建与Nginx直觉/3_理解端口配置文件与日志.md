# 3. 理解端口、配置文件与日志

本节目标：把 Nginx 的几个基础对象串起来：进程、端口、配置文件、静态文件、访问日志、错误日志。

学习 Nginx 很容易陷入“配置片段记忆”。但后端工程师更需要先建立运行直觉：

```text
Nginx 是进程。
进程监听端口。
端口接收请求。
请求按配置处理。
处理结果写入日志。
```

---

## 一、Nginx 是一个正在运行的进程

如果是系统安装：

```bash
ps -ef | grep nginx
```

你会看到类似：

```text
root      nginx: master process /usr/sbin/nginx
www-data  nginx: worker process
www-data  nginx: worker process
```

如果是 Docker：

```bash
docker ps
```

进入容器后也可以看：

```bash
docker exec -it my-nginx sh
ps
```

这里先记住：

- master 负责管理。
- worker 负责处理请求。

后面性能阶段会详细讲 master/worker。

---

## 二、Nginx 监听端口

系统安装时查看：

```bash
sudo ss -lntp | grep nginx
```

可能看到：

```text
LISTEN 0 511 0.0.0.0:80 0.0.0.0:* users:(("nginx",pid=1234,fd=6))
```

这说明 Nginx 正在监听 `80` 端口。

Docker 方式查看：

```bash
docker ps
```

可能看到：

```text
0.0.0.0:8080->80/tcp
```

含义：

```text
本机 8080 -> 容器内 80
```

访问时要访问本机端口：

```bash
curl http://localhost:8080
```

---

## 三、Nginx 根据配置处理请求

一个最小配置大概是：

```nginx
server {
    listen 80;
    server_name localhost;

    location / {
        return 200 "hello nginx\n";
    }
}
```

含义：

- `listen 80`：监听 80 端口。
- `server_name localhost`：匹配 Host 为 localhost 的请求。
- `location /`：匹配所有路径。
- `return 200`：直接返回响应。

请求：

```bash
curl http://localhost
```

Nginx 会：

```text
收到请求
-> 匹配 server
-> 匹配 location
-> 执行 return
-> 写 access log
```

---

## 四、配置文件不是一个文件

很多人以为 Nginx 只有一个配置文件：

```text
/etc/nginx/nginx.conf
```

实际上它常常通过 `include` 引入其他文件：

```nginx
include /etc/nginx/conf.d/*.conf;
include /etc/nginx/sites-enabled/*;
```

所以你真正需要看的是最终配置：

```bash
sudo nginx -T
```

这条命令会打印所有被加载的配置。

如果你写了 `api.local`，可以查：

```bash
sudo nginx -T | grep api.local
```

查不到，说明配置没有被加载。

## 五、access log 与 error log

### access log

记录请求访问情况。

```bash
sudo tail -f /var/log/nginx/access.log
```

你请求一次：

```bash
curl http://localhost
```

日志可能出现：

```text
127.0.0.1 - - [05/Jul/2026:20:00:00 +0800] "GET / HTTP/1.1" 200 13 "-" "curl/8.0"
```

它告诉你：

- 谁访问了。
- 什么时间访问。
- 请求了什么路径。
- 返回了什么状态码。
- 响应大小是多少。
- 客户端是什么。

### error log

记录 Nginx 运行和转发过程中的错误。

```bash
sudo tail -f /var/log/nginx/error.log
```

常见错误包括：

- 配置加载失败。
- 文件权限不足。
- upstream 连接失败。
- upstream 超时。
- 请求体过大。

---

## 六、一次请求的最小观察方法

打开第一个终端：

```bash
sudo tail -f /var/log/nginx/access.log
```

打开第二个终端：

```bash
curl -v http://localhost
```

观察：

- `curl -v` 看到客户端视角。
- `access.log` 看到 Nginx 服务端视角。

如果请求失败，再看：

```bash
sudo tail -n 50 /var/log/nginx/error.log
```

## 七、常见现象和判断

### 1. curl 连接失败

```text
Connection refused
```

优先看端口：

```bash
sudo ss -lntp | grep ':80'
```

### 2. 返回 404

说明 Nginx 收到了请求，但没有找到资源或匹配结果不是你想的。

看：

```bash
sudo nginx -T
sudo tail -f /var/log/nginx/access.log
```

### 3. 返回 403

常见是权限问题或访问目录没有 index 文件。

看：

```bash
sudo tail -f /var/log/nginx/error.log
```

### 4. 返回 502

通常是代理后端失败。后面反向代理阶段会重点讲。
