# 0. 环境搭建与 Nginx 直觉总览

本阶段目标：把 Nginx 跑起来，并建立最基本的后端工程直觉：Nginx 是一个监听端口、读取配置、处理请求、写入日志的服务进程。

学习 Nginx 不建议一开始就背很多配置指令。第一阶段最重要的是把下面这条链路跑通：

```text
浏览器 / curl
  |
  | HTTP 请求
  v
Nginx 监听 80 端口
  |
  | 读取配置
  v
返回页面或文本响应
  |
  v
写入 access.log / error.log
```

---

## 一、本阶段你要学会什么

完成本阶段后，你应该能够：

- 使用 Docker 启动一个 Nginx。
- 使用 Linux 包管理器安装 Nginx。
- 知道 Nginx 默认监听哪个端口。
- 知道配置文件在哪里。
- 知道访问日志和错误日志在哪里。
- 使用 `curl` 验证 Nginx 是否正常。
- 使用 `nginx -t` 检查配置是否正确。
- 使用 `nginx -T` 查看最终加载的完整配置。
- 修改配置后 reload。
- 遇到端口冲突、配置错误、容器退出时知道先看什么。

## 三、为什么先用 Docker

直接在系统里安装 Nginx 容易遇到这些干扰：

- Windows、macOS、Linux 路径不同。
- 已经有服务占用了 80 端口。
- 配置文件位置和发行版有关。
- 卸载和重装不够干净。

Docker 的好处是：

- 启动快。
- 删除容易。
- 可以明确看到端口映射。
- 可以把配置文件挂载进去反复实验。
- 不容易把系统环境改乱。

但是生产环境未必总是 Docker，所以本阶段也会讲 systemd 方式。

## 四、本阶段最小闭环

本阶段的最小闭环是：

```text
启动 Nginx
-> 访问 localhost
-> 看到响应
-> 修改配置
-> 检查配置
-> reload
-> 再次访问
-> 查看日志
```

只要这个闭环跑通，后面学习静态资源、反向代理、HTTPS、日志排障都会顺很多。

## 五、常见误区

### 误区 1：配置文件改了就一定生效

不一定。你可能改错文件，或者改了文件但没有 reload。

正确流程：

```bash
nginx -t
nginx -s reload
```

如果是 systemd 管理：

```bash
sudo nginx -t
sudo systemctl reload nginx
```

### 误区 2：访问不了就是 Nginx 配置错

不一定。可能是：

- Nginx 没启动。
- 端口没监听。
- 防火墙拦截。
- 域名没有解析。
- 请求匹配到了别的 server。
- 容器端口没有映射出来。

### 误区 3：`nginx -t` 成功就代表业务一定正常

不是。`nginx -t` 只能证明语法基本正确。

例如：

```nginx
proxy_pass http://127.0.0.1:9999;
```

如果 9999 端口没有 Go 服务，`nginx -t` 仍然可能成功，但访问时会返回 502。

## 六、本阶段验收

学完本阶段，你需要能完成：

1. 启动一个 Nginx。
2. 用 `curl` 访问到它。
3. 修改默认响应内容。
4. 检查配置并 reload。
5. 查看 access log。
6. 故意写错配置，能通过错误信息找到问题。

如果这些做不到，不建议急着进入反向代理阶段。
