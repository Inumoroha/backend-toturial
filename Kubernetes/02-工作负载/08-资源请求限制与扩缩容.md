# 8. 资源请求限制与扩缩容

本节目标：学完后你能为后端服务设置 CPU、内存资源，并理解副本扩缩容的判断依据。

## 简短引入

Kubernetes 集群资源是共享的。一个服务如果没有资源约束，可能吃掉节点内存；资源设置太小，又可能被频繁杀掉。后端开发者至少要能给服务一个合理起点。

## 一、为什么需要它

真实项目中常见现象：

- 批量导入接口占满 CPU。
- 日志处理服务内存泄漏。
- 副本很多但每个副本数据库连接池过大。
- 发布后 Pod 一直 OOMKilled。

资源配置的目标不是一次猜准，而是给调度和隔离一个边界，再通过监控调整。

## 二、基本用法

Deployment 容器片段：

```yaml
resources:
  requests:
    cpu: "100m"
    memory: "128Mi"
  limits:
    cpu: "500m"
    memory: "512Mi"
```

查看资源使用需要 metrics-server：

```bash
kubectl top pods
kubectl top nodes
```

如果本地没有安装 metrics-server，`kubectl top` 可能不可用。这不影响你先学习资源字段。

手动扩容：

```bash
kubectl scale deployment short-api --replicas=3
kubectl get pods -l app=short-api
```

## 三、关键参数/语法/代码结构

- `requests.cpu`：调度时申请的 CPU。
- `requests.memory`：调度时申请的内存。
- `limits.cpu`：容器最多可用 CPU。
- `limits.memory`：容器最多可用内存，超过可能 OOMKilled。

CPU 单位：

- `100m` 表示 0.1 核。
- `1` 表示 1 核。

内存单位：

- `128Mi` 表示 128 Mebibytes。
- `512Mi` 是常见小服务起点，但不能代替压测。

## 四、真实后端场景示例

短链接服务主要是 HTTP + 数据库查询，初始配置可以保守一些：

```yaml
resources:
  requests:
    cpu: "100m"
    memory: "128Mi"
  limits:
    cpu: "500m"
    memory: "512Mi"
```

但如果是订单导出服务，可能会把大量数据加载进内存。真实项目中更推荐分页流式处理，而不是简单把内存限制调大。

```text
扩容 Pod 不能修复低效 SQL、无边界查询和内存泄漏。先判断瓶颈，再决定扩容。
```

## 五、注意点

- 内存 limit 过小会导致 OOMKilled。
- CPU limit 过小可能导致延迟升高。
- 副本数增加会增加数据库连接数，要同步检查连接池配置。
- 自动扩缩容 HPA 依赖指标系统，本地学习可以先掌握手动扩容。

HPA 示例：

```bash
kubectl autoscale deployment short-api --cpu-percent=60 --min=2 --max=5
kubectl get hpa
```

生产中 HPA 要结合业务指标和压测，不要只看 CPU。

## 六、常见错误

- 错误：不配置 requests。
  调度器不知道你的服务需要多少资源，容易资源争抢。
- 错误：看到慢就加副本。
  如果瓶颈是数据库锁或慢 SQL，加副本可能更慢。
- 错误：每个副本数据库连接池都设置 100。
  10 个副本就是 1000 个连接，数据库可能先被打垮。

## 七、本节达标标准

- 能配置 requests 和 limits。
- 能手动调整 Deployment 副本数。
- 能说明扩容和数据库连接池之间的关系。
- 知道 HPA 的基本作用和使用前提。

## 八、从现象理解资源字段

调度器主要用 `requests` 判断节点是否放得下 Pod；运行时由系统执行 `limits`。CPU 超限通常表现为被限速，内存超过 limit 则可能出现 `OOMKilled`。

排查资源问题：

```bash
kubectl describe pod <pod-name>
kubectl get pod <pod-name> -o jsonpath='{.status.containerStatuses[0].lastState.terminated.reason}'
kubectl top pod <pod-name>
```

最后一条命令依赖 metrics-server。不可用时先看 `describe` 和应用日志，不需要为了完成本章立刻安装整套监控。

估算数据库连接时要做乘法。例如每个 Pod 最多 20 个连接，最大副本数为 5，仅这个服务就可能建立 100 个连接，还没计算 Job、管理工具和其他服务。

练习：先记下副本数与 Pod 数，再执行 `kubectl scale deployment short-api --replicas=3`。确认出现 3 个 Ready Pod 后缩回 2 个。观察扩容不会自动复制请求数据，也不会改变 Service 地址。
