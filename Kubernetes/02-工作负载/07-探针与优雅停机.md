# 7. 探针与优雅停机

本节目标：学完后你能为 Go 服务配置健康检查，并让服务在滚动更新时尽量不丢请求。

## 简短引入

Pod 处于 Running 不代表服务已经能处理请求。Go 进程启动了，也不代表数据库连接、缓存初始化、路由加载都完成。探针就是让 Kubernetes 判断服务是否健康、是否可以接流量。

## 一、为什么需要它

真实项目常见发布问题：

- 新 Pod 刚启动就被接入流量，但数据库还没连上。
- 服务死锁了，进程还在，Kubernetes 不知道要重启。
- 滚动更新时旧 Pod 被立刻杀掉，请求中断。

Kubernetes 常用探针：

- `readinessProbe`：是否可以接流量。
- `livenessProbe`：是否需要重启。
- `startupProbe`：启动较慢的服务是否还在启动中。

```text
生产后端服务优先配置 readinessProbe。它直接影响发布时流量会不会打到未就绪实例。
```

## 二、基本用法

Deployment 片段：

```yaml
containers:
  - name: short-api
    image: short-api:0.1.0
    ports:
      - containerPort: 8080
    readinessProbe:
      httpGet:
        path: /readyz
        port: 8080
      initialDelaySeconds: 3
      periodSeconds: 5
    livenessProbe:
      httpGet:
        path: /healthz
        port: 8080
      initialDelaySeconds: 10
      periodSeconds: 10
    lifecycle:
      preStop:
        exec:
          command: ["sh", "-c", "sleep 5"]
```

Go 服务示例：

```go
package main

import (
	"context"
	"log"
	"net/http"
	"os"
	"os/signal"
	"syscall"
	"time"
)

func main() {
	mux := http.NewServeMux()
	ready := true

	mux.HandleFunc("/healthz", func(w http.ResponseWriter, r *http.Request) {
		w.WriteHeader(http.StatusOK)
	})
	mux.HandleFunc("/readyz", func(w http.ResponseWriter, r *http.Request) {
		if !ready {
			http.Error(w, "not ready", http.StatusServiceUnavailable)
			return
		}
		w.WriteHeader(http.StatusOK)
	})

	server := &http.Server{Addr: ":8080", Handler: mux}

	go func() {
		log.Println("listening on :8080")
		if err := server.ListenAndServe(); err != nil && err != http.ErrServerClosed {
			log.Fatal(err)
		}
	}()

	stop := make(chan os.Signal, 1)
	signal.Notify(stop, syscall.SIGTERM, syscall.SIGINT)
	<-stop

	ready = false
	time.Sleep(3 * time.Second)

	ctx, cancel := context.WithTimeout(context.Background(), 10*time.Second)
	defer cancel()
	if err := server.Shutdown(ctx); err != nil {
		log.Println("shutdown error:", err)
	}
}
```

## 三、关键参数/语法/代码结构

- `initialDelaySeconds`：容器启动后多久开始探测。
- `periodSeconds`：探测间隔。
- `timeoutSeconds`：单次探测超时时间。
- `failureThreshold`：连续失败多少次才认为失败。
- `preStop`：容器停止前执行的钩子。
- `terminationGracePeriodSeconds`：给进程优雅退出的时间。

探针接口不要做太重的逻辑。`/healthz` 通常只判断进程是否活着，`/readyz` 可以检查关键依赖是否可用，但不要每次都执行慢 SQL。

## 四、真实后端场景示例

订单服务启动时要加载支付渠道配置。加载完成前不应该接收下单请求：

```go
var ready atomic.Bool

func loadDependencies() {
	// 加载配置、初始化连接池
	ready.Store(true)
}
```

`/readyz` 根据 `ready` 返回状态。这样 Kubernetes 在滚动发布时会等新 Pod Ready 后再把流量转过去。

## 五、注意点

- liveness probe 不要依赖外部数据库。数据库短暂抖动时把所有 Pod 重启，可能让故障扩大。
- readiness probe 可以检查关键依赖，但要控制超时和开销。
- Go 服务必须处理 SIGTERM。
- 优雅停机要和网关、Service 摘流量时间配合。

## 六、常见错误

- 错误：只配置 liveness，不配置 readiness。
  进程活着不代表可以接流量。
- 错误：健康检查接口访问数据库慢查询。
  高并发探测会给数据库制造额外压力。
- 错误：收到 SIGTERM 直接退出。
  正在处理的请求可能被中断。

## 七、本节达标标准

- 能解释 readiness 和 liveness 的区别。
- 能为 Deployment 配置 HTTP 探针。
- 能在 Go 服务中实现健康检查接口。
- 能实现基本优雅停机。

## 八、观察探针和终止过程

应用 Deployment 后查看探针结果：

```bash
kubectl get pods -l app=short-api -w
kubectl describe pod <pod-name>
```

正常时 Pod 会从 `0/1` 变为 `1/1`。如果探针失败，Events 中会出现路径、端口、状态码或超时信息。先在容器内确认接口，而不是盲目增加延迟：

```bash
kubectl exec <pod-name> -- wget -qO- http://127.0.0.1:8080/readyz
```

镜像没有 `wget` 时可以使用 `port-forward` 测试。工具缺失与接口故障是两件不同的事。

终止实验：一个终端持续查看日志，另一个终端删除 Pod。正常实现应先把 readiness 设为失败，等待摘流量，再完成正在处理的请求，并在宽限期内退出。`terminationGracePeriodSeconds` 必须大于应用关停所需最长时间。

自测：数据库暂时不可用时为什么不建议让 liveness 失败？因为它可能同时重启所有副本，把外部依赖故障放大为应用重启风暴。
