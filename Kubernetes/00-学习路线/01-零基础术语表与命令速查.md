# 零基础术语表与命令速查

这份手册用于解决阅读教程时最常见的两个问题：看不懂缩写，以及不知道下一条排查命令该用什么。它不是独立课程，建议在学习各章时随时回来查。

## 一、先认识整套系统

### 集群、控制平面和节点

- **集群（Cluster）**：一组由 Kubernetes 管理的计算资源。
- **控制平面（Control Plane）**：接收配置、安排 Pod、持续检查实际状态的管理部分。
- **节点（Node）**：真正运行 Pod 的机器，可以是物理机、虚拟机或本地容器。
- **API Server**：Kubernetes 的统一入口。`kubectl` 本质上是在向它发送请求。

初学时可以把控制平面理解成“调度和管理中心”，把节点理解成“干活的机器”。本地单节点集群会把这些角色放在同一台电脑里。

### 镜像、容器和 Pod

- **镜像（Image）**：只读的应用打包结果，包含程序和运行所需文件。
- **容器（Container）**：镜像的一次运行实例。
- **Pod**：Kubernetes 最小调度单位，通常放一个主要业务容器。

镜像像安装包，容器像正在运行的程序，Pod 则是 Kubernetes 管理该程序时使用的外壳。删除 Pod 不等于删除镜像。

### Deployment、Service 和 Ingress

- **Deployment**：管理 Pod 副本、版本更新和回滚。
- **Service**：给一组会变化的 Pod 提供稳定地址。
- **Ingress**：描述外部 HTTP 请求如何按域名或路径进入 Service。
- **Ingress Controller**：真正读取 Ingress 规则并转发流量的组件。

请求路径通常是：

```text
浏览器 -> Ingress Controller -> Service -> Ready Pod -> Go 进程
```

### 配置和敏感信息

- **ConfigMap**：保存日志级别、端口、环境名等普通配置。
- **Secret**：保存密码、令牌、证书等敏感配置。
- **环境变量**：把配置交给应用进程的常见方式。

Secret 的值经过编码不等于加密。生产环境仍需配合权限、静态加密和外部密钥系统。

### 标签、选择器和注解

- **Label**：资源上的可查询键值，例如 `app: short-api`。
- **Selector**：通过 Label 找到资源。Service 和 Deployment 都依赖它。
- **Annotation**：保存不用于选择资源的附加信息。

Label 对不上是新手最常见的问题之一。Service 不会因为选不到 Pod 就拒绝创建，所以一定要检查 endpoints。

### Namespace 和 Context

- **Namespace**：集群内部的逻辑分组，例如 `backend-dev`、`backend-prod`。
- **Context**：`kubectl` 当前使用的集群、用户身份和默认 Namespace 组合。
- **kubeconfig**：保存集群连接和 Context 的配置文件。

Context 选错可能把本应在本地执行的命令发到其他集群。做变更前先确认：

```bash
kubectl config current-context
kubectl get namespaces
```

## 二、YAML 阅读方法

所有 Kubernetes 资源都先看四个顶层字段：

```yaml
apiVersion: apps/v1       # 使用哪组 API
kind: Deployment          # 创建什么资源
metadata:
  name: short-api         # 资源名称
spec:                     # 你期望它成为什么样
  replicas: 2
```

YAML 依赖缩进表达层级，只使用空格，不使用 Tab。同一层字段必须对齐。字符串是否加引号通常都能解析，但端口、布尔值和类似 `yes` 的内容容易产生歧义，拿不准时按教程写法保留引号。

提交前可以让客户端做基础校验：

```bash
kubectl apply --dry-run=client -f deployment.yaml
```

`dry-run` 通过只说明格式和部分字段可解析，不保证镜像、权限、网络和业务逻辑正确。

## 三、资源状态怎么读

查看 Pod：

```bash
kubectl get pods -o wide
```

常见状态：

| 状态 | 含义 | 第一条排查命令 |
| --- | --- | --- |
| `Pending` | 还未成功调度或准备 | `kubectl describe pod <pod-name>` |
| `ContainerCreating` | 正在拉镜像、挂载存储或创建网络 | `kubectl describe pod <pod-name>` |
| `Running` | 容器已运行，不代表已经 Ready | `kubectl get pod <pod-name>` |
| `CrashLoopBackOff` | 程序反复崩溃并重启 | `kubectl logs <pod-name> --previous` |
| `ImagePullBackOff` | 镜像拉取失败 | `kubectl describe pod <pod-name>` |
| `Completed` | Job 正常执行完成 | `kubectl logs <pod-name>` |
| `Terminating` | 正在停止和清理 | `kubectl describe pod <pod-name>` |

`READY 1/1` 表示 Pod 中一个容器已经通过就绪检查。`STATUS Running` 但 `READY 0/1` 时，Service 通常不会向它发送流量。

## 四、最常用命令

### 查看

```bash
kubectl get pods
kubectl get deploy
kubectl get svc
kubectl get all
kubectl get pods -l app=short-api
kubectl get pods -n backend-dev
```

### 解释和诊断

```bash
kubectl describe pod <pod-name>
kubectl logs <pod-name>
kubectl logs <pod-name> --previous
kubectl logs deployment/short-api --tail=100
kubectl get events --sort-by=.metadata.creationTimestamp
kubectl explain deployment.spec.template.spec.containers
```

### 创建、更新和删除

```bash
kubectl apply -f deployment.yaml
kubectl apply -f k8s/
kubectl delete -f deployment.yaml
kubectl rollout status deployment/short-api
kubectl rollout undo deployment/short-api
```

`apply` 是声明式更新：文件描述期望状态，Kubernetes 计算需要改变什么。`delete` 会改变实际资源，执行前确认 Context、Namespace 和文件内容。

### 临时访问和调试

```bash
kubectl port-forward svc/short-api 8080:8080
kubectl exec -it <pod-name> -- sh
kubectl run curl-test --rm -it --image=curlimages/curl -- sh
```

`port-forward` 命令会持续占用当前终端，按 `Ctrl+C` 停止。另开一个终端再执行请求命令。

## 五、统一排障顺序

遇到“访问不了”时，不要同时乱改多个文件。按下面顺序缩小范围：

1. 确认当前 Context 和 Namespace。
2. 确认 Deployment 存在，期望副本数正确。
3. 确认 Pod 为 `Running` 且 `Ready`。
4. 用 `describe` 查看底部 Events。
5. 用 `logs` 查看当前日志，崩溃重启则加 `--previous`。
6. 确认 Service selector 与 Pod label 一致。
7. 确认 Service 有 EndpointSlice。
8. 先用 `port-forward` 绕过 Ingress 测试 Service。
9. Service 正常后再检查 Ingress、域名和 Controller。

```bash
kubectl get deploy,pods,svc
kubectl get endpointslices -l kubernetes.io/service-name=short-api
kubectl describe pod <pod-name>
kubectl logs <pod-name> --previous
```

一次只改一个因素，改完重新观察。否则服务恢复后你也很难知道真正原因。

## 六、单位和常见缩写

- `m`：CPU 毫核，`100m` 等于 0.1 个 CPU 核心。
- `Mi` / `Gi`：二进制内存单位，例如 `128Mi`。
- `DNS`：把名称解析为地址的系统。
- `RBAC`：基于角色的访问控制。
- `PV` / `PVC`：持久卷和持久卷申领。
- `HPA`：根据指标自动调整 Pod 副本数。
- `CI/CD`：持续集成与持续交付或部署。
- `P95`：95% 的观测值不超过该值，常用于描述接口延迟。
- `OOMKilled`：容器超过内存限制，被系统终止。
