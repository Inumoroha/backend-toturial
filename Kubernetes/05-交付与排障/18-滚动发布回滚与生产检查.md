# 18. 滚动发布、回滚与生产检查

本节目标：学完后你能设计一次保守的 Kubernetes 后端服务发布流程，并在异常时回滚。

## 简短引入

发布不是把新镜像 apply 到集群就结束。真实项目要考虑迁移顺序、健康检查、流量接入、监控观察和回滚方案。

## 一、为什么需要它

后端发布常见风险：

- 新代码依赖未执行的数据库迁移。
- 新 Pod 没准备好就接流量。
- 错误率升高但没人观察。
- 回滚后数据库结构不兼容旧代码。

Kubernetes 提供滚动更新和回滚能力，但工程流程要你自己设计好。

## 二、基本用法

更新镜像：

```bash
kubectl set image deployment/short-api short-api=short-api:0.2.0
kubectl rollout status deployment/short-api
```

查看历史：

```bash
kubectl rollout history deployment/short-api
```

回滚：

```bash
kubectl rollout undo deployment/short-api
kubectl rollout status deployment/short-api
```

Deployment 发布策略示例：

```yaml
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxUnavailable: 0
    maxSurge: 1
```

这个配置表示发布时尽量不减少可用副本，并允许临时多启动一个新副本。

## 三、关键参数/语法/代码结构

- `maxUnavailable`：滚动更新时最多允许多少不可用。
- `maxSurge`：滚动更新时最多额外创建多少 Pod。
- `revisionHistoryLimit`：保留多少历史版本。
- `progressDeadlineSeconds`：发布超过多久没进展判定失败。

示例：

```yaml
revisionHistoryLimit: 5
progressDeadlineSeconds: 300
```

## 四、真实后端场景示例

短链接服务新增 `expires_at` 字段。保守发布流程：

1. 先执行兼容迁移：新增可空字段，不影响旧代码。
2. 发布新代码：新代码开始写入 `expires_at`。
3. 观察日志、错误率、延迟。
4. 确认稳定后，再考虑补数据或添加更严格约束。

```text
数据库变更要优先选择向前兼容。先让新旧代码都能跑，再逐步收紧。
```

发布前检查：

```bash
kubectl get deploy short-api
kubectl get pods -l app=short-api
kubectl rollout history deployment/short-api
```

发布后检查：

```bash
kubectl rollout status deployment/short-api
kubectl logs deployment/short-api --tail=100
kubectl get events --sort-by=.metadata.creationTimestamp
```

## 五、注意点

- 没有 readiness probe 的滚动发布不可靠。
- 回滚代码不一定能回滚数据库。
- 生产镜像标签要唯一，不能反复覆盖同一个标签。
- 发布窗口要考虑业务低峰和团队响应能力。
- 重要发布要有观察指标，不只看 Pod Ready。

## 六、常见错误

- 错误：发布前没有确认当前版本。
  出问题时不知道要回到哪里。
- 错误：数据库迁移和代码发布强耦合。
  一旦失败，很难判断先回滚哪一边。
- 错误：用删除 Pod 当作回滚。
  删除 Pod 只会按当前 Deployment 模板重建，不会恢复旧镜像。

## 七、本节达标标准

- 能更新 Deployment 镜像。
- 能查看发布状态和历史。
- 能执行回滚。
- 能写出一次包含迁移、发布、观察、回滚的保守流程。
