# 17. 日志、事件与故障排查

本节目标：学完后你能用常见 `kubectl` 命令定位后端服务部署失败和运行异常。

## 简短引入

后端服务上 Kubernetes 后，问题不再只看应用日志。还要看 Pod 状态、事件、镜像拉取、探针失败、资源限制等。排障的关键是按层次缩小范围。

## 一、为什么需要它

常见故障：

- Pod 一直 Pending。
- Pod CrashLoopBackOff。
- 镜像 ImagePullBackOff。
- Service 访问超时。
- readiness probe 失败。
- 生产发布后错误率升高。

每种问题背后的原因不同，不能只盯着一条日志。

## 二、基本用法

查看资源：

```bash
kubectl get pods
kubectl get deploy
kubectl get svc
```

看 Pod 详情：

```bash
kubectl describe pod <pod-name>
```

看日志：

```bash
kubectl logs <pod-name>
kubectl logs <pod-name> --previous
kubectl logs deployment/short-api
```

看事件：

```bash
kubectl get events --sort-by=.metadata.creationTimestamp
```

进入容器调试：

```bash
kubectl exec -it <pod-name> -- sh
```

## 三、关键参数/语法/代码结构

- `describe` 重点看 Events。
- `logs --previous` 用来看上一次崩溃前日志。
- `rollout status` 看发布是否完成。
- `get endpoints` 看 Service 是否选到 Pod。
- `top` 看资源使用，但依赖 metrics-server。

排障顺序可以这样：

```text
资源是否存在 -> Pod 是否 Ready -> 事件是否异常 -> 应用日志是否报错 -> Service 是否选到 Pod -> 入口是否转发正确
```

## 四、真实后端场景示例

短链接服务发布后访问 503：

1. 看 Pod：

```bash
kubectl get pods -l app=short-api
```

2. 看 Service 是否有后端：

```bash
kubectl get endpoints short-api
```

3. 如果 endpoints 为空，看 readiness：

```bash
kubectl describe pod <pod-name>
```

4. 看应用日志：

```bash
kubectl logs <pod-name>
```

如果日志显示：

```text
DATABASE_URL is required
```

说明不是 Ingress 问题，而是 Secret 或环境变量注入问题。

## 五、注意点

- 不要一上来重启服务，先保留现场看事件和日志。
- 生产排障要记录操作，特别是删除 Pod、回滚、扩容。
- `exec` 进入容器适合临时确认，不应成为长期运维方式。
- 应用日志要包含请求 ID、关键业务字段和错误原因，但不要打印密码。

## 六、常见错误

- 错误：只看应用日志，不看 Kubernetes 事件。
  调度失败、镜像拉取失败通常在事件里更清楚。
- 错误：Service 访问失败就改 Ingress。
  应先确认 Service endpoints 是否存在。
- 错误：日志里打印完整数据库连接串。
  这会泄露账号密码。

## 七、本节达标标准

- 能使用 `get`、`describe`、`logs`、`events` 排查问题。
- 能定位 ImagePullBackOff、CrashLoopBackOff 的常见原因。
- 能检查 Service endpoints。
- 能按层次排查一次 503 问题。
