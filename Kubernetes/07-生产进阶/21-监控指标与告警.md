# 21. 监控指标与告警

本节目标：学完后你能为 Go 后端服务设计基础监控指标，并知道日志、事件、指标分别怎么用。

## 简短引入

`kubectl logs` 能帮你排查单次问题，但生产环境不能靠人一直盯日志。后端服务需要指标和告警：请求是否变慢、错误率是否升高、Pod 是否频繁重启、数据库连接是否耗尽。

## 一、为什么需要它

真实项目中，服务“还活着”不代表用户体验正常：

- Pod Ready，但接口 P95 延迟已经超过 2 秒。
- 错误率从 0.1% 升到 5%。
- 数据库连接池被打满。
- CronJob 连续失败，但没人发现。
- 发布后 CPU 正常，业务下单成功率下降。

日志适合看具体错误，事件适合看 Kubernetes 调度和状态变化，指标适合看趋势和告警。

```text
生产告警应该围绕用户影响和恢复动作设计，而不是看到任何数字波动就报警。
```

## 二、基本用法

Go 服务可以暴露 `/metrics`，由 Prometheus 抓取。最小示例：

```go
package main

import (
	"net/http"

	"github.com/prometheus/client_golang/prometheus/promhttp"
)

func main() {
	mux := http.NewServeMux()
	mux.Handle("/metrics", promhttp.Handler())
	mux.HandleFunc("/healthz", func(w http.ResponseWriter, r *http.Request) {
		w.WriteHeader(http.StatusOK)
	})
	http.ListenAndServe(":8080", mux)
}
```

安装依赖：

```bash
go get github.com/prometheus/client_golang/prometheus
go get github.com/prometheus/client_golang/prometheus/promhttp
```

Kubernetes Service 暴露端口即可：

```yaml
ports:
  - name: http
    port: 8080
    targetPort: 8080
```

如果团队使用 Prometheus Operator，通常会再配置 ServiceMonitor。不同集群安装方式差异较大，初学阶段先理解“应用暴露指标，监控系统定时抓取”。

## 三、关键参数/语法/代码结构

后端服务优先关注这些指标：

- 请求量：每秒多少请求。
- 错误率：5xx、业务错误码比例。
- 延迟：P50、P95、P99。
- Pod 重启次数。
- CPU、内存使用率。
- 数据库连接池使用量。
- Job/CronJob 成功和失败次数。

Go 业务指标示例：

```go
var createLinkTotal = prometheus.NewCounterVec(
	prometheus.CounterOpts{
		Name: "short_link_create_total",
		Help: "Total short link create requests.",
	},
	[]string{"result"},
)
```

注册并使用：

```go
prometheus.MustRegister(createLinkTotal)
createLinkTotal.WithLabelValues("success").Inc()
createLinkTotal.WithLabelValues("conflict").Inc()
```

## 四、真实后端场景示例

短链接服务可以设计这些告警：

- 5 分钟内 5xx 错误率超过 2%。
- P95 跳转延迟超过 300ms 持续 10 分钟。
- `short-api` Pod 重启次数持续增加。
- `short-link-cleaner` CronJob 连续失败 3 次。
- PostgreSQL 连接池使用率超过 80%。

这些告警背后都有明确动作：回滚、扩容、排查数据库、暂停流量或修复定时任务。

## 五、注意点

- 不要把用户 ID、订单号、短码这类高基数字段放进指标标签。
- 告警要能指导行动，否则会变成噪音。
- 日志里可以有 request_id，指标标签里不要塞 request_id。
- Kubernetes 指标只能说明运行状态，业务是否成功还需要应用指标。
- 监控和告警规则也应该进版本管理。

## 六、常见错误

- 错误：只看 CPU 和内存。
  后端服务更重要的是错误率、延迟和业务成功率。
- 错误：指标标签随便加用户输入。
  高基数会让监控系统压力迅速上升。
- 错误：没有发布后观察窗口。
  新版本上线后应该主动观察关键指标，再宣布发布完成。

## 七、本节达标标准

- 能区分日志、事件和指标的用途。
- 能为 Go 服务暴露 `/metrics`。
- 能列出后端服务的核心监控指标。
- 能设计几条有明确处理动作的告警。

## 八、从一个请求建立可观测性

新手可以用同一个请求串起三种信息：日志记录具体错误并携带 `request_id`；指标累计请求次数、错误率和延迟；Kubernetes 事件说明 Pod 调度、探针或重启等平台变化。

先确认应用指标端点能访问：

```bash
kubectl port-forward svc/short-api 8080:8080
curl http://127.0.0.1:8080/metrics
```

预期看到 Prometheus 文本格式，而不是业务 JSON。调用几次业务接口后再次访问，相关计数器应增加。

设计指标时控制标签集合。例如 `method`、规范化路由 `/links/{code}` 和 `status_class` 通常可控；用户 ID、完整 URL、request_id 会持续产生新值，不能作为指标标签。

为每条告警补齐四项：触发条件、持续时间、影响说明、第一处理动作。例如“5xx 比例超过 2% 持续 5 分钟；影响创建短链接；先检查最近发布并对比数据库错误，必要时回滚”。只有阈值没有动作的规则很难在真实故障中发挥作用。
