# 5. 综合实践：从零完成一次消息收发

本节目标：把第 1 阶段的内容串起来，独立完成一次 Kafka 消息收发实验，并观察 consumer group 的 lag 变化。

这不是新概念课，而是验收课。你要像做一个小型后端实验一样，从环境启动、topic 创建、消息写入、消息消费、lag 查看，一步一步完成。

---

## 一、实验目标

完成下面这条链路：

```text
启动 Kafka
创建 quickstart.orders
发送订单消息
使用 order-service group 消费
查看 offset 和 lag
停止 consumer
继续发送消息
观察 lag 增长
重启 consumer
观察 lag 下降
```

---

## 二、启动 Kafka

进入 `kafka-labs` 目录：

```powershell
docker compose up -d
```

检查容器：

```powershell
docker ps
```

进入容器：

```powershell
docker exec -it kafka bash
```

---

## 三、创建 Topic

```bash
kafka-topics.sh \
  --bootstrap-server localhost:9092 \
  --create \
  --if-not-exists \
  --topic quickstart.orders \
  --partitions 3 \
  --replication-factor 1
```

查看 topic：

```bash
kafka-topics.sh \
  --bootstrap-server localhost:9092 \
  --describe \
  --topic quickstart.orders
```

预期：

```text
PartitionCount: 3
ReplicationFactor: 1
```

---

## 四、发送第一批消息

启动 producer：

```bash
kafka-console-producer.sh \
  --bootstrap-server localhost:9092 \
  --topic quickstart.orders \
  --property parse.key=true \
  --property key.separator=:
```

输入：

```text
order-1:created
order-2:created
order-3:created
order-4:created
order-5:created
```

输入完后按 `Ctrl+C` 退出。

---

## 五、使用 Consumer Group 消费

启动 consumer：

```bash
kafka-console-consumer.sh \
  --bootstrap-server localhost:9092 \
  --topic quickstart.orders \
  --group order-service \
  --from-beginning \
  --property print.key=true \
  --property print.partition=true \
  --property print.offset=true
```

预期能看到刚才的 5 条消息。

消费完成后按 `Ctrl+C` 退出。

---

## 六、查看 Offset 和 Lag

```bash
kafka-consumer-groups.sh \
  --bootstrap-server localhost:9092 \
  --describe \
  --group order-service
```

如果消息已经消费完，预期：

```text
LAG = 0
```

这表示 `order-service` 这个 group 没有积压消息。

---

## 七、停止 Consumer 后继续发送消息

确保 `order-service` consumer 没有运行。

再启动 producer：

```bash
kafka-console-producer.sh \
  --bootstrap-server localhost:9092 \
  --topic quickstart.orders \
  --property parse.key=true \
  --property key.separator=:
```

输入：

```text
order-6:created
order-7:created
order-8:created
```

退出 producer。

此时没有 consumer 在消费，所以 group lag 应该增长。

查看：

```bash
kafka-consumer-groups.sh \
  --bootstrap-server localhost:9092 \
  --describe \
  --group order-service
```

预期：

```text
LAG 大于 0
```

---

## 八、重启 Consumer 观察 Lag 下降

重新启动 consumer：

```bash
kafka-console-consumer.sh \
  --bootstrap-server localhost:9092 \
  --topic quickstart.orders \
  --group order-service
```

它应该消费刚才积压的消息。

再查看 lag：

```bash
kafka-consumer-groups.sh \
  --bootstrap-server localhost:9092 \
  --describe \
  --group order-service
```

预期：

```text
LAG 回到 0
```

## 九、实验现象解释

### 为什么停止 consumer 后 producer 还能继续发送？

Kafka 的 producer 和 consumer 是解耦的。

producer 写入 topic 后，消息保存在 Kafka 中。

consumer 不在线时，消息不会因为没人消费就丢失，而是会根据 topic 的 retention 策略保留。

### 为什么 consumer 重启后还能继续消费？

因为 consumer group 记录了 offset。

Kafka 知道 `order-service` 这个 group 消费到了哪里。

### 为什么 lag 会增长？

consumer 停止时：

```text
LOG-END-OFFSET 增长
CURRENT-OFFSET 不变
```

所以：

```text
LAG = LOG-END-OFFSET - CURRENT-OFFSET
```

会变大。

## 十、Go 后端工程视角

这个实验对应真实项目：

```text
order-service 发布 order.created
inventory-service 消费 order.created
```

如果 `inventory-service` 停止：

- `order-service` 仍然可以继续创建订单并发送事件。
- Kafka 会保存事件。
- `inventory-service` 恢复后继续消费。
- 监控上会看到 consumer lag 先升高后下降。

这就是 Kafka 解耦和削峰的核心价值。

## 十三、本节小结

- Kafka 可以在 consumer 不在线时继续接收 producer 写入。
- consumer group 使用 offset 记录消费进度。
- lag 表示 consumer group 落后多少消息。
- 停止 consumer 后继续写入消息，lag 会增长。
- consumer 恢复后，lag 会下降。
- Kafka 的解耦能力来自 topic 存储和 consumer group offset。
- 第 1 阶段最重要的能力是能亲手完成消息收发并解释现象。

## 十五、排查顺序

如果实验没有按预期工作，按顺序查：

1. Kafka 容器是否运行。
2. `bootstrap-server` 地址是否正确。
3. topic 是否存在。
4. producer 是否真的发送成功。
5. consumer 是否使用了正确 topic。
6. group 是否已经提交过 offset。
7. 是否需要 `--from-beginning`。

不要一上来就删除所有环境。先定位问题，会更有学习价值。
