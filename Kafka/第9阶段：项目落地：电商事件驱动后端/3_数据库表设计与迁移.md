
## 十、processed_events 为什么必须有唯一约束

```sql
event_id VARCHAR(128) PRIMARY KEY
```

这保证同一个事件只能插入一次。

consumer 重复消费时：

```text
INSERT processed_events 冲突
说明事件已处理
直接返回成功
提交 offset
```

## 十四、完整可执行建表脚本

前面为了讲清楚每张表，SQL 是分开展示的。真正落地时，建议把下面内容放进：

```text
migrations/001_init.sql
```

```sql
CREATE TABLE IF NOT EXISTS orders (
  id VARCHAR(64) PRIMARY KEY,
  user_id VARCHAR(64) NOT NULL,
  status VARCHAR(32) NOT NULL,
  total_amount BIGINT NOT NULL CHECK (total_amount >= 0),
  created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE IF NOT EXISTS order_items (
  id BIGSERIAL PRIMARY KEY,
  order_id VARCHAR(64) NOT NULL REFERENCES orders(id),
  sku_id VARCHAR(64) NOT NULL,
  quantity INT NOT NULL CHECK (quantity > 0),
  price BIGINT NOT NULL CHECK (price >= 0)
);

CREATE TABLE IF NOT EXISTS inventory (
  sku_id VARCHAR(64) PRIMARY KEY,
  available INT NOT NULL CHECK (available >= 0),
  updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE IF NOT EXISTS processed_events (
  event_id VARCHAR(128) PRIMARY KEY,
  topic VARCHAR(128) NOT NULL,
  partition_id INT NOT NULL,
  offset_value BIGINT NOT NULL,
  handler VARCHAR(128) NOT NULL,
  processed_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE IF NOT EXISTS outbox_events (
  id VARCHAR(128) PRIMARY KEY,
  aggregate_type VARCHAR(64) NOT NULL,
  aggregate_id VARCHAR(128) NOT NULL,
  event_type VARCHAR(128) NOT NULL,
  topic VARCHAR(128) NOT NULL,
  message_key VARCHAR(128) NOT NULL,
  payload JSONB NOT NULL,
  status VARCHAR(32) NOT NULL DEFAULT 'PENDING',
  retry_count INT NOT NULL DEFAULT 0,
  next_retry_at TIMESTAMP NULL,
  created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  sent_at TIMESTAMP NULL
);

CREATE INDEX IF NOT EXISTS idx_order_items_order_id
ON order_items(order_id);

CREATE INDEX IF NOT EXISTS idx_outbox_events_status_next_retry
ON outbox_events(status, next_retry_at, created_at);

CREATE INDEX IF NOT EXISTS idx_processed_events_topic_offset
ON processed_events(topic, partition_id, offset_value);
```

这里多加了几个 `CHECK` 约束。原因很简单：不要只在 Go 代码里校验数据，数据库也应该守住最后一道门。

## 十五、为什么金额用 BIGINT

订单金额不要用浮点数。

错误示例：

```sql
price NUMERIC(10, 2)
```

这不是完全不能用，但在 Go 项目里初学阶段更推荐用“分”为单位：

```text
9900 表示 99.00 元。
```

优点：

- Go 里可以用 `int64`。
- 不会出现浮点精度问题。
- Kafka 事件里的金额也是整数，跨语言更稳定。

对应 Go 字段：

```go
type OrderItem struct {
    SkuID    string `json:"sku_id"`
    Quantity int    `json:"quantity"`
    Price    int64  `json:"price"`
}
```

## 十六、Outbox 查询语句

`outbox-worker` 每轮要找一批可以发送的事件：

```sql
SELECT id, topic, message_key, payload, retry_count
FROM outbox_events
WHERE status = 'PENDING'
  AND (next_retry_at IS NULL OR next_retry_at <= now())
ORDER BY created_at
LIMIT 100
FOR UPDATE SKIP LOCKED;
```

几个细节要理解：

| 写法 | 作用 |
| --- | --- |
| `ORDER BY created_at` | 尽量按事件产生顺序发送 |
| `LIMIT 100` | 避免一次锁太多数据 |
| `FOR UPDATE` | 锁住本轮要处理的行 |
| `SKIP LOCKED` | 多个 worker 并发时跳过已被锁住的行 |

如果不加 `SKIP LOCKED`，多个 worker 可能互相等待，吞吐会明显下降。

## 十七、发送成功后更新状态

Kafka 发送成功以后，更新 outbox：

```sql
UPDATE outbox_events
SET status = 'SENT',
    sent_at = now()
WHERE id = $1;
```

发送失败时不要删除记录，而是记录重试信息：

```sql
UPDATE outbox_events
SET retry_count = retry_count + 1,
    next_retry_at = now() + interval '30 seconds'
WHERE id = $1;
```

如果超过最大次数，可以标记为 `FAILED`：

```sql
UPDATE outbox_events
SET status = 'FAILED',
    retry_count = retry_count + 1
WHERE id = $1;
```

这样做的好处是：失败不是静默发生的，后面可以通过 SQL 或指标看到。

## 十八、幂等消费的事务顺序

`inventory-service` 处理消息时，推荐事务顺序如下：

```text
begin
  insert processed_events
  if duplicate:
    commit
    return success

  update inventory
  insert inventory result outbox event
commit
commit kafka offset
```

为什么先插入 `processed_events`？

因为它是幂等门禁。只有抢到这个 `event_id` 的处理流程，才允许继续扣库存。

如果先扣库存，再插入 `processed_events`，服务在中间崩溃，就会出现：

```text
库存已经扣了。
processed_events 没写入。
消息重新消费。
库存再扣一次。
```

## 十九、迁移验证脚本

执行 migration 后，用下面 SQL 快速验收：

```sql
INSERT INTO inventory (sku_id, available)
VALUES ('sku_1', 100)
ON CONFLICT (sku_id) DO NOTHING;

INSERT INTO processed_events(event_id, topic, partition_id, offset_value, handler)
VALUES ('evt_test_1', 'order.created', 0, 1, 'inventory-service');

INSERT INTO processed_events(event_id, topic, partition_id, offset_value, handler)
VALUES ('evt_test_1', 'order.created', 0, 1, 'inventory-service');
```

第二次插入应该失败：

```text
duplicate key value violates unique constraint
```

如果使用 `ON CONFLICT DO NOTHING`，第二次影响行数应该是 `0`。

## 二十、常见错误

### 1. outbox_events 没有查询索引

worker 会频繁按 `status` 和 `next_retry_at` 扫描，如果没有索引，数据量稍大就会变慢。

### 2. processed_events 用自增 id 做唯一

自增 id 只能保证每条记录唯一，不能保证每个业务事件只处理一次。幂等必须围绕 `event_id`。

### 3. order_items 没有外键或索引

学习阶段可以保留外键，帮助你尽早发现脏数据。生产里是否使用外键可以根据团队规范决定，但索引不能省。

### 4. JSONB payload 没有版本字段

事件结构一定会演进。payload 里至少要包含 `event_type` 和 `version`，consumer 才知道如何兼容旧消息。
