## 二、项目最终能力

完成后项目应该支持：

- 创建订单。
- 写订单表和 outbox 表。
- outbox worker 发布 `order.created`。
- inventory-service 消费订单事件。
- 使用 `processed_events` 做幂等。
- 消费失败进入 retry topic。
- 多次失败进入 DLQ。
- 日志能按 event_id 串联链路。
- 能观察 consumer lag。
- 有正常链路、重复消费、DLQ 的集成测试。

## 三、项目学习方式

不要一次性写完所有服务。

推荐顺序：

```text
先写数据库 migration。
再写 order-service 创建订单和 outbox。
再写 outbox-worker 发布 Kafka。
再写 inventory-service 消费。
最后补 retry、DLQ、指标和测试。
```

每一步都要有可验证结果。

例如写完 order-service 后，不要求 Kafka 已有消息，但必须能看到：

```text
orders 表有订单。
outbox_events 表有 PENDING 事件。
```

写完 outbox-worker 后，必须能看到：

```text
outbox_events.status = SENT。
Kafka topic order.created 有消息。
```

## 七、项目最小闭环

不要一开始就追求所有异常路径。先完成最小闭环：

```text
POST /orders
-> orders/order_items 写入成功
-> outbox_events 写入 PENDING
-> outbox-worker 发布 order.created
-> outbox_events 变成 SENT
-> inventory-service 消费 order.created
-> processed_events 写入 event_id
-> inventory.available 扣减
-> consumer lag 回到 0
```

每一步都要有命令或 SQL 能验证。

示例 SQL：

```sql
SELECT id, status, total_amount
FROM orders
ORDER BY created_at DESC
LIMIT 1;

SELECT id, event_type, status, retry_count
FROM outbox_events
ORDER BY created_at DESC
LIMIT 1;

SELECT sku_id, available
FROM inventory;

SELECT event_id, handler, processed_at
FROM processed_events
ORDER BY processed_at DESC
LIMIT 5;
```

## 八、项目里必须体现的 Kafka 能力

这个项目不是普通 CRUD，它必须体现 Kafka 工程能力：

| 能力 | 项目体现 |
| --- | --- |
| Producer | outbox-worker 发布 `order.created` |
| Consumer group | inventory-service 消费订单事件 |
| Message key | 使用 `order_id` 保证订单维度分区 |
| Offset | 业务成功或消息转移后提交 |
| 幂等 | `processed_events.event_id` |
| Retry | 数据库超时等可恢复错误 |
| DLQ | JSON 错误等不可恢复错误 |
| Observability | lag、日志、DLQ、outbox pending age |

如果某个能力没有在项目里被验证，就回到对应章节补实验。

## 十、项目常见错误

### 1. order-service 直接发 Kafka

这样会重新引入“数据库成功但 Kafka 失败”的一致性问题。项目主线应该使用 Outbox。

### 2. inventory-service 只靠 offset 防重复

offset 提交失败、rebalance、outbox 重发都可能造成重复消费。必须有业务幂等。

### 3. Retry 和 DLQ 只写概念不验证

项目验收必须能手动制造坏 JSON 和数据库错误，并看到消息进入 DLQ 或 retry topic。

### 4. README 没有排障命令

项目要能展示给别人看。README 必须写清楚如何启动、如何创建订单、如何查 lag、如何看 DLQ。
