# 03 事件驱动架构设计

Kafka 在 Go 后端里最常见的价值，是支撑事件驱动架构。

## 什么是事件

事件表示已经发生的事实。

例如：

- 订单已创建：`order.created`
- 支付已成功：`payment.succeeded`
- 库存已扣减：`inventory.deducted`

事件不是命令。

不推荐：

```text
send_coupon
call_inventory
notify_user
```

推荐：

```text
coupon.granted
inventory.deducted
notification.sent
```

## 服务解耦

传统同步调用：

```text
订单服务 -> 库存服务 -> 优惠券服务 -> 通知服务
```

问题：

- 链路长。
- 任一下游慢都会拖慢上游。
- 下游故障影响核心请求。
- 新增订阅方需要改订单服务。

事件驱动：

```text
order-service -> order.created -> inventory-service
                              -> coupon-service
                              -> notification-service
                              -> analytics-service
```

订单服务只发布事实，下游自己订阅。

## 最终一致性

事件驱动系统通常不是强一致，而是最终一致。

例如：

1. 订单创建成功。
2. 几百毫秒后库存扣减。
3. 再后发送通知。

用户短时间看到中间状态是正常的，但系统最终要收敛到正确状态。

## Saga 思想

复杂业务可以拆成多个本地事务和补偿动作。

例如下单：

1. 创建订单。
2. 扣库存。
3. 支付。
4. 发通知。

如果支付失败，可能需要：

- 取消订单。
- 释放库存。

Kafka 可以承载这些事件，但业务状态机和补偿逻辑仍然要由服务设计。

## 事件边界

不要把数据库表的所有字段无脑塞进事件。

事件应该包含：

- 下游需要的关键信息。
- 事件发生时间。
- 事件版本。
- 业务唯一 ID。

敏感字段要谨慎，例如手机号、身份证、支付信息。
