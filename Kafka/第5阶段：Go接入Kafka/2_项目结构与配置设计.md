# 2. 项目结构与配置设计

本节目标：设计一个适合 Go 后端项目接入 Kafka 的目录结构和配置方式，避免 Kafka 客户端代码散落在业务模块里。

Go 项目接入 Kafka 时，最容易犯的错误是：哪个业务要发消息，就在哪个业务文件里直接创建 Kafka client。短期看很快，长期会导致配置、日志、错误处理、重试、关闭逻辑全部分散。

---

## 一、推荐项目结构

以电商事件项目为例：

```text
ecommerce-events/
  cmd/
    order-service/
      main.go
    inventory-service/
      main.go
  internal/
    kafka/
      config.go
      message.go
      producer.go
      consumer.go
      retry.go
      dlq.go
    order/
      service.go
      handler.go
      event.go
    inventory/
      handler.go
      service.go
    platform/
      logger/
      config/
      shutdown/
```

核心原则：

```text
internal/kafka 只处理 Kafka 通用能力。
业务模块只处理业务逻辑。
```

---

## 二、为什么要封装 internal/kafka

统一封装可以解决：

- broker 配置统一。
- client id 统一。
- producer 关闭逻辑统一。
- consumer 手动提交统一。
- 日志字段统一。
- retry/DLQ 处理统一。
- 后续切换客户端更容易。

业务代码不应该知道底层使用 `franz-go` 还是其他客户端。

---

## 三、config.go

建议：

```go
package kafka

type Config struct {
    Brokers  []string
    ClientID string
    GroupID  string
}
```

从环境变量读取：

```go
func LoadConfigFromEnv() Config {
    brokers := strings.Split(os.Getenv("KAFKA_BROKERS"), ",")

    return Config{
        Brokers:  brokers,
        ClientID: os.Getenv("KAFKA_CLIENT_ID"),
        GroupID:  os.Getenv("KAFKA_GROUP_ID"),
    }
}
```

环境变量示例：

```text
KAFKA_BROKERS=localhost:9092
KAFKA_CLIENT_ID=order-service
KAFKA_GROUP_ID=inventory-service
```

---

## 四、message.go

建议定义通用消息结构：

```go
package kafka

type Message struct {
    Key     []byte
    Value   []byte
    Headers map[string]string
}

type ConsumedMessage struct {
    Topic     string
    Partition int32
    Offset    int64
    Key       []byte
    Value     []byte
    Headers   map[string]string
}
```

业务事件 envelope 可以放到业务模块或 `internal/event`：

```go
type EventEnvelope[T any] struct {
    EventID    string    `json:"event_id"`
    EventType  string    `json:"event_type"`
    Version    int       `json:"version"`
    OccurredAt time.Time `json:"occurred_at"`
    Data       T         `json:"data"`
}
```

---

## 五、producer.go 接口

```go
type Producer interface {
    Publish(ctx context.Context, topic string, msg Message) error
    Close(ctx context.Context) error
}
```

业务层只依赖接口：

```go
type OrderService struct {
    producer kafka.Producer
}
```

这样测试时可以 mock producer。

---

## 六、consumer.go 接口

```go
type Handler interface {
    Handle(ctx context.Context, msg ConsumedMessage) error
}

type Consumer interface {
    Run(ctx context.Context) error
    Close(ctx context.Context) error
}
```

业务 handler 只关心消息内容：

```go
type OrderCreatedHandler struct {
    inventory *InventoryService
}

func (h *OrderCreatedHandler) Handle(ctx context.Context, msg kafka.ConsumedMessage) error {
    // decode and handle
    return nil
}
```

---

## 七、main.go 应该做什么

`cmd/inventory-service/main.go` 负责组装依赖：

```text
加载配置
初始化 logger
初始化 database
初始化 kafka producer
初始化 kafka consumer
注册 handler
监听 SIGTERM
启动 consumer
优雅退出
```

不要把业务处理写进 main。

---

## 八、配置不要写死

错误：

```go
kgo.SeedBrokers("localhost:9092")
```

更好：

```go
kgo.SeedBrokers(cfg.Brokers...)
```

原因：

- 本地是 `localhost:9092`。
- Docker 内部可能是 `kafka:9092`。
- 生产是多个 broker。

---

## 九、日志字段规范

在 `internal/kafka` 里统一日志字段：

```text
topic
partition
offset
key
event_id
event_type
group
duration_ms
error
```

业务模块可以补充：

```text
order_id
user_id
sku_id
```

## 十、常见误区

### 1. Kafka 代码写在业务 handler 里

不推荐。

会让业务逻辑和基础设施耦合。

### 2. 每个服务复制一份 producer 代码

不推荐。

应该抽成统一包。

### 3. 配置硬编码

不推荐。

环境变化时容易出问题。

### 4. 日志字段不统一

线上排查会非常难。

## 十一、本节练习

1. 创建上面的目录结构。
2. 写出 `Config` 结构体。
3. 写出 `Message` 和 `ConsumedMessage`。
4. 写出 `Producer` 和 `Handler` 接口。
5. 思考哪些代码应该放在 `internal/kafka`，哪些应该放在 `internal/order`。

---

## 十二、本节小结

- Kafka 接入代码应该统一封装。
- `internal/kafka` 负责基础能力，不写业务逻辑。
- 业务模块依赖接口，而不是直接依赖第三方客户端。
- 配置通过环境变量传入。
- main.go 负责组装依赖，不承载业务细节。
- 统一日志字段是生产排查的基础。

---

## 十四、配置校验补充

配置加载后要立即校验：

```go
func (c Config) Validate() error {
    if len(c.Brokers) == 0 || c.Brokers[0] == "" {
        return errors.New("KAFKA_BROKERS is required")
    }
    if c.ClientID == "" {
        return errors.New("KAFKA_CLIENT_ID is required")
    }
    return nil
}
```

consumer 服务还要校验 `KAFKA_GROUP_ID`。不要把空 group id 带到运行时才发现问题。

---

## 十五、工程边界

`internal/kafka` 只提供通用能力：

```text
Publish
Subscribe
Commit
Retry
DLQ
```

不要在里面写 `CreateOrder`、`DeductInventory` 这类业务语义。

---

## 十六、目录验收清单

- [ ] `cmd/*/main.go` 只负责启动。
- [ ] `internal/platform/config` 负责配置。
- [ ] `internal/platform/database` 负责数据库连接。
- [ ] `internal/kafka` 负责 Kafka 基础封装。
- [ ] `internal/order` 负责订单业务。
- [ ] `internal/inventory` 负责库存业务。
- [ ] 业务包之间不直接互相调用。

目录结构的目的不是好看，而是让边界清楚。

---

## 十七、配置文件示例

```text
HTTP_ADDR=:8080
DATABASE_URL=postgres://postgres:postgres@localhost:5432/ecommerce?sslmode=disable
KAFKA_BROKERS=localhost:9092
KAFKA_CLIENT_ID=order-service
KAFKA_GROUP_ID=inventory-service
```

真实密码不要提交到 Git，应该使用环境变量或密钥管理。

---

## 十八、配置验收

启动服务时打印非敏感配置：

```text
http_addr
kafka_brokers
kafka_client_id
kafka_group_id
```

不要打印 `DATABASE_URL` 中的密码。配置日志的目的，是让部署问题更容易排查。

---

## 十九、最终检查

如果修改 Kafka broker 地址，只需要改环境变量，而不需要改业务代码。

这说明配置边界是合格的。

---

## 二十一、配置自查清单

配置设计完成后，检查下面几项：

```text
本地、测试、生产是否能使用不同 broker。
topic 名称是否集中配置。
group id 是否集中配置。
超时时间是否可以通过环境变量调整。
日志里是否打印了必要配置，但不会打印密码。
配置缺失时程序是否快速失败，而不是运行到一半才报错。
```

Go 后端服务启动时，配置校验应该尽早发生。Kafka 地址、topic 或 group 配错时，越早失败越容易排查。
