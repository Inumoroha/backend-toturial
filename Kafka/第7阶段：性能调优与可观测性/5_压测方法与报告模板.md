# 5. 压测方法与报告模板

本节目标：学会设计 Kafka producer 和 consumer 压测，并写出可复盘的压测报告。

---

## 一、压测前先定义问题

不要问：

```text
Kafka 能跑多快？
```

要问：

```text
在 2KB 订单事件下，order-service 每秒能发送多少？
inventory-service 每秒能消费多少？
开启 zstd 后 P95 延迟变化多少？
```

---

## 二、记录环境

必须记录：

- Kafka 版本。
- broker 数量。
- topic partition 数。
- replication factor。
- 消息大小。
- producer 参数。
- consumer 参数。
- 机器 CPU/内存。
- Go 版本。

没有环境信息，压测结果不可复用。

---

## 三、Producer 压测维度

调整：

- 消息大小。
- batch size。
- linger。
- compression。
- acks。
- producer 并发数。

记录：

- 吞吐。
- 平均延迟。
- P95。
- P99。
- 错误率。

---

## 四、Consumer 压测维度

调整：

- partition 数。
- consumer 数。
- handler 耗时。
- 单条提交或批量提交。
- 下游数据库延迟。

记录：

- 每秒处理数。
- lag 增长/下降。
- handler P95/P99。
- 错误率。
- rebalance 次数。

---

## 五、压测报告模板

```markdown

| --- | --- | --- | --- | --- | --- |

## 下一步优化
```

---

## 六、常见误区

- 只测 producer，不测 consumer。
- 只看平均延迟。
- 不记录消息大小。
- 不记录错误率。
- 本地单节点结果直接套生产。
- 下游数据库用假实现，却推导真实业务能力。

---

## 七、本节练习

1. 设计 100 万条订单事件 producer 压测。
2. 设计 consumer 慢处理压测。
3. 对比 3 partition 和 6 partition。
4. 写一页压测报告。

---

## 八、本节小结

- 压测必须先定义具体问题。
- 环境和参数必须记录。
- Producer 和 consumer 都要测。
- P95/P99 比平均值更重要。
- 压测结论必须能指导下一步优化。

---

## 九、Producer 压测步骤示例

目标：

```text
测试 10 万条 2KB order.created 消息在不同压缩算法下的吞吐和延迟。
```

步骤：

1. 创建 topic `bench.order.created`，3 个 partition。
2. 准备固定大小消息。
3. 分别使用 `none`、`lz4`、`zstd`。
4. 每组发送 10 万条。
5. 记录吞吐和延迟。

命令准备：

```bash
kafka-topics.sh \
  --bootstrap-server localhost:9092 \
  --create \
  --if-not-exists \
  --topic bench.order.created \
  --partitions 3 \
  --replication-factor 1
```

Go 程序参数可以设计为：

```text
--total=100000
--message-size=2048
--compression=lz4
--topic=bench.order.created
```

---

## 十、Consumer 压测步骤示例

目标：

```text
测试 handler 耗时为 10ms、50ms、100ms 时 consumer lag 的变化。
```

步骤：

1. producer 先写入 10 万条消息。
2. consumer 启动，handler sleep 10ms。
3. 每 10 秒记录一次 lag。
4. 改成 50ms、100ms 重复实验。
5. 对比 lag 下降速度。

记录表：

| 时间 | total lag | partition-0 | partition-1 | partition-2 |
| --- | --- | --- | --- | --- |
| 0s | | | | |
| 10s | | | | |
| 20s | | | | |

---

## 十一、如何避免压测污染结论

注意：

- 每轮测试前清空或换新 topic。
- 每轮使用新的 consumer group。
- 记录机器负载。
- 确认没有其他任务抢资源。
- 重复跑多次取稳定结果。

如果复用同一个 group，offset 会影响消费起点。

如果复用同一个 topic，历史消息会影响统计。

---

## 十二、报告中的结论要有动作

不好的结论：

```text
性能还可以。
```

好的结论：

```text
在 3 partition、3 consumer、handler 50ms 的配置下，最大稳定消费能力约为每秒 60 条。当前峰值写入为每秒 200 条，因此 lag 会持续增长。建议将 partition 增加到 12，并将 inventory-service 扩容到 12 个实例，同时优化数据库扣减 SQL。
```

结论必须能指导下一步。

---

## 十三、压测风险

不要在生产 Kafka 上随意压测。

如果必须在生产环境验证：

- 使用独立 topic。
- 限制流量。
- 避开高峰。
- 提前通知。
- 设置停止条件。
- 监控 broker CPU、磁盘、网络。

---

## 十四、补充练习

1. 设计一个 producer benchmark 的命令行参数。
2. 设计一个 consumer benchmark 的命令行参数。
3. 写出如何重置测试环境。
4. 写一段“结论必须有动作”的压测结论。

## 十五、压测命令参数示例

Producer benchmark 可以设计成：

```bash
go run ./cmd/producer-bench \
  --brokers localhost:9092 \
  --topic bench.order.created \
  --messages 100000 \
  --message-size 1024 \
  --key-cardinality 10000 \
  --batch-size 100 \
  --linger 10ms
```

Consumer benchmark 可以设计成：

```bash
go run ./cmd/consumer-bench \
  --brokers localhost:9092 \
  --topic bench.order.created \
  --group bench-inventory \
  --handler-latency 50ms \
  --commit manual \
  --workers 3
```

参数必须写清楚，否则压测结果无法复现。

报告的最后一定要落到动作，比如扩分区、扩 consumer、优化 SQL、增加 batch、降低消息体大小。

---

## 十七、压测后的清理

压测结束后要清理：

```bash
kafka-consumer-groups.sh \
  --bootstrap-server localhost:9092 \
  --delete \
  --group bench-inventory

kafka-topics.sh \
  --bootstrap-server localhost:9092 \
  --delete \
  --topic bench.order.created
```

如果使用数据库，也要清理测试数据，避免影响后续验证。

---

## 十八、压测前后核对表

压测前：

- [ ] topic 是测试 topic。
- [ ] group 是测试 group。
- [ ] 消息量和消息大小已固定。
- [ ] 有监控窗口。
- [ ] 有停止条件。

压测后：

- [ ] 删除测试 group。
- [ ] 删除或保留测试 topic 的决策明确。
- [ ] 清理数据库测试数据。
- [ ] 保存报告。
- [ ] 写出下一步动作。

---

## 十九、报告结论要求

结论必须包含：

```text
当前最大稳定吞吐。
瓶颈判断。
建议动作。
复测计划。
```

没有复测计划，优化就没有闭环。

---

## 二十、复测示例

```text
第一次测试：6 partition，6 consumer，180 msg/s。
优化动作：库存 SQL 增加索引。
第二次测试：同样参数，260 msg/s。
结论：瓶颈确实在数据库处理。
```

复测要保持其他变量不变。

---

## 二十一、压测报告必须能复现

一份合格的压测报告至少包含：

```text
Kafka 版本：
broker 数量：
topic partition 数：
replication factor：
消息大小：
key 分布方式：
producer 参数：
consumer 参数：
下游依赖是否 mock：
压测持续时间：
预热时间：
采样窗口：
```

如果缺少这些信息，别人很难判断结果是 Kafka 能力、业务代码能力，还是机器环境带来的偶然结果。
