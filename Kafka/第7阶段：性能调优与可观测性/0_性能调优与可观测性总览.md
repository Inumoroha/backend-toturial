# 第 7 阶段：性能调优与可观测性总览

这一阶段学习 Kafka 链路如何跑得快、跑得稳、看得见。性能调优不是背参数，而是用压测和指标证明瓶颈在哪里。

## 二、本阶段学习目标

完成后你应该能做到：

- 区分吞吐优化和低延迟优化。
- 调整 batch、linger、compression。
- 调整 consumer 批量处理和提交策略。
- 监控 lag、处理耗时、错误率、DLQ 数量。
- 写一份 Kafka 压测报告。

## 三、本阶段为什么重要

前面阶段解决的是：

```text
消息能不能正确发送和消费。
```

这一阶段解决的是：

```text
消息链路能不能在高峰期稳定运行。
出问题时能不能快速定位。
```

很多线上 Kafka 问题不是“完全不可用”，而是慢慢变坏：

- lag 一直增长。
- P99 延迟升高。
- 某个 partition 堵住。
- DLQ 突然增加。
- outbox 积压越来越多。
- consumer 频繁 rebalance。

如果没有指标和压测，你只能靠猜。

---

## 四、本阶段实验环境

建议继续使用前面阶段的本地 Kafka。

推荐准备：

```text
topic: perf.order.created
partitions: 3 或 6
replication.factor: 1
consumer group: perf-inventory-service
```

本地单节点实验不能代表生产极限，但可以帮助你理解：

- partition 数如何影响并行度。
- handler 耗时如何影响 lag。
- producer 批量和压缩如何影响吞吐。
- consumer 停止后 lag 如何增长。

## 六、本阶段学习方法

不要只写结论，比如：

```text
增加 partition 可以提升吞吐。
```

要写成实验观察：

```text
3 partition、3 consumer、handler sleep 50ms 时，消费能力约为每秒 60 条。
6 partition、6 consumer、handler sleep 50ms 时，消费能力约为每秒 120 条。
```

Kafka 性能学习必须带数据。

---

## 七、最小压测闭环

本阶段至少要跑通一个小型压测闭环：

```text
准备 perf.order.created topic
-> producer 连续写入固定数量消息
-> consumer 以固定 handler 延迟消费
-> 记录吞吐、P95/P99、lag 曲线
-> 修改一个参数
-> 复测并对比结果
```

一次只改一个变量，例如只改 partition 数，或者只改 handler 延迟。

如果同时改 partition、consumer 数、batch、数据库 SQL，最后很难判断哪个变化真正产生了效果。

---

## 八、关键指标

至少观察这些指标：

| 指标 | 说明 |
| --- | --- |
| producer throughput | producer 每秒写入多少消息 |
| producer latency | 发送到 ack 的耗时 |
| consumer throughput | consumer 每秒处理多少消息 |
| handler duration | 业务处理耗时 |
| consumer lag | 消费积压 |
| error rate | 处理失败比例 |
| DLQ count | 坏消息数量 |
| outbox pending age | outbox 最老待发送事件年龄 |

其中 `lag` 和 `handler duration` 经常要一起看。lag 高不一定是 Kafka 慢，可能是业务 handler 慢。

---

## 九、常见误区

### 1. 只看平均耗时

平均值会掩盖尾部延迟。生产排查更关注 P95、P99。

### 2. lag 一高就加 consumer

如果 partition 数不足，或者热点 key 集中在一个 partition，加 consumer 不一定有用。

### 3. 压测没有固定环境

每次消息大小、partition 数、consumer 数都不同，结果就不可比较。

### 4. 把本地压测当生产容量

本地单 broker 只能帮助理解趋势，不能代表生产集群极限。

## 十二、排查顺序模板

lag 持续升高时，按这个顺序查：

```text
1. producer 写入是否突然变多。
2. consumer 是否还存活。
3. handler 是否变慢。
4. 数据库是否慢查询。
5. 是否有单 partition 热点。
6. 是否频繁 rebalance。
7. DLQ 或 retry 是否增长。
```

不要直接跳到“加机器”。先定位瓶颈，再决定动作。

## 二十、压测前检查

压测前先确认：

- topic 是测试 topic。
- 消息大小固定。
- producer 写入速率可控。
- consumer handler 行为固定。
- 数据库测试数据可清理。
- 有停止条件。
- 有记录模板。

压测不是制造热闹，而是制造可解释的数据。

## 二十四、不要忽略业务指标

Kafka 指标只能说明消息链路状态，不能完全代表业务成功。

还要同时观察：

```text
订单创建成功率。
库存预占成功率。
支付事件处理延迟。
DLQ 中不同业务错误的占比。
用户可感知的接口延迟。
```

性能调优最终服务业务结果。吞吐提升但业务失败率上升，不应该被视为成功优化。
