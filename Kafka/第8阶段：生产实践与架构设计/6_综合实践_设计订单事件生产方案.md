# 6. 综合实践：设计订单事件生产方案

本节目标：为订单事件链路设计一份可上线的 Kafka 生产方案。

---

## 一、业务链路

```text
order-service -> order.created -> inventory-service
order-service -> order.created -> notification-service
order-service -> order.created -> analytics-service
```

---

## 二、Topic 设计

```text
order.created
order.created.retry.1m
order.created.retry.5m
order.created.dlq
```

配置：

```text
partitions=12
replication.factor=3
retention=7d
key=order_id
```

---

```json
{
  "event_id": "evt_order_created_order_1001",
```

## 四、Producer 方案

```text
Outbox Pattern
acks=all
enable.idempotence=true
compression=lz4
```

创建订单事务：

```text
写 orders
写 outbox_events
commit
```

---

## 五、Consumer 方案

库存服务：

```text
group=inventory-service
manual commit
processed_events 幂等
retry/DLQ
```

通知服务：

```text
group=notification-service
允许重复通知时仍需 message_id 去重
```

## 七、失败预案

Kafka 不可用：

```text
outbox 积压，worker 重试
```

库存服务故障：

```text
lag 增长，恢复后继续消费
```

坏消息：

```text
进入 DLQ，告警，修复后重放
```

## 十一、容量估算填写示例

```text
日订单量：500 万
平均消息大小：3KB
保留时间：7 天
副本数：3
预留：30%
```

计算：

```text
3KB * 500万 = 15GB/天
15GB * 7 * 3 = 315GB
315GB * 1.3 = 409.5GB
```

结论：

```text
order.created 至少按 410GB 存储预算。
```

## 十八、方案复盘话术

可以这样总结：

```text
订单服务生产 order.created 时，我不会在业务事务里直接依赖 Kafka。
我会先在同一个数据库事务里写订单和 outbox 事件，再由 outbox-worker 异步发布。
topic 使用 order_id 作为 key 保证同一订单维度的分区顺序。
consumer 必须使用 event_id 做幂等，因为 outbox 可能重复发送。
```

## 十九、设计文档验收标准

最终文档不能只写“使用 Kafka 解耦”。至少要回答：

| 问题 | 必须给出 |
| --- | --- |
| topic 怎么建 | 分区数、副本数、retention |
| key 怎么选 | 为什么用 `order_id` |
| schema 怎么演进 | version 和兼容策略 |
| producer 怎么保证不丢 | Outbox、acks、重试 |
| consumer 怎么防重复 | `processed_events` 或业务唯一约束 |
| 失败怎么处理 | retry、DLQ、告警 |
| 怎么观察 | lag、DLQ、outbox pending age |

如果某一项写不出来，就回到对应阶段复习。

## 二十、可直接放进 README 的摘要

```text
order-service 在创建订单事务中写入 outbox_events，不直接发送 Kafka。
outbox-worker 扫描 PENDING 事件并发布 order.created，发布成功后标记 SENT。
inventory-service 以 consumer group 方式消费 order.created，使用 processed_events.event_id 保证重复事件不会重复扣库存。
可恢复错误进入 retry topic，不可恢复错误进入 DLQ。
链路通过 consumer lag、DLQ 数量和 outbox pending age 观察健康状态。
```

这段摘要可以作为第 9 阶段项目说明的基础。
