## 三、为什么这是分水岭

前面阶段解决的是“会不会用 Kafka”：

```text
能创建 topic。
能发送消息。
能消费消息。
能看到 offset 和 lag。
```

第 6 阶段解决的是“敢不敢把 Kafka 放进关键业务”：

```text
订单创建了，消息不能丢。
库存扣减了，重复消息不能重复扣。
坏消息不能卡住整个 partition。
数据库和 Kafka 不能因为中间失败而长期不一致。
```

很多项目 Kafka 事故不是因为不会调用客户端 API，而是因为没有设计清楚：

- 什么时候提交 offset。
- 什么错误可以 retry。
- 什么错误应该进 DLQ。
- 重复消息如何幂等。
- 数据库成功但 Kafka 失败怎么办。


## 五、最小闭环

本阶段最小闭环不是写完整项目，而是能推演并验证：

```text
创建订单事件
-> consumer 处理成功
-> commit offset
-> 重复消费同一 event_id
-> processed_events 识别重复
-> 库存不重复扣减
```

再验证两个异常：

```text
JSON 格式错误 -> DLQ -> 提交原 offset
数据库临时错误 -> retry topic -> 提交原 offset
```

只要这三个闭环跑通，就说明你已经理解 Kafka 可靠消费的核心。

## 六、关键设计原则

### 1. 宁愿重复，也不要静默丢失

Kafka 业务系统通常追求 at-least-once。重复可以靠幂等解决，静默丢失往往很难补救。

### 2. Offset 不是业务成功证明

offset 只表示 consumer group 的读取进度。业务是否成功，要看数据库事务、幂等表、业务状态。

### 3. Retry 和 DLQ 是主链路保护

可恢复错误进入 retry，不可恢复错误进入 DLQ。坏消息不能永远堵住主 topic。

### 4. Outbox 防丢，幂等防重复

Outbox 能保证业务数据和待发送事件一起落库，但 worker 可能重复发布，所以 consumer 仍然必须幂等。

## 七、Go 后端落地视角

Go 代码里建议形成这样的边界：

```text
handler:
  解析消息
  校验字段
  调用 service
  返回错误分类

service:
  开启事务
  写 processed_events
  执行业务操作
  提交事务

consumer framework:
  根据 handler 返回值写 retry/DLQ
  决定是否提交 offset
```

这样业务逻辑和 Kafka 可靠性逻辑不会混在一起。

## 九、本阶段推荐推演表

学习可靠性时，建议为每个失败点写推演表：

| 失败点 | 结果 | 是否丢消息 | 是否重复 | 解决方案 |
| --- | --- | --- | --- | --- |
| producer 发送失败 | Kafka 没收到 | 可能 | 否 | 返回错误或 Outbox |
| 业务成功但 commit 失败 | 下次重新消费 | 否 | 是 | 业务幂等 |
| 写 DLQ 失败 | 原消息仍在主 topic | 否 | 可能 | 不提交 offset |
| outbox 发送成功但标记失败 | 下次重复发送 | 否 | 是 | consumer 幂等 |

这张表比背概念更重要。可靠性设计就是逐个失败点推演。

## 十、推荐 SQL 资产

本阶段至少写出三张表的 SQL 草案：

```sql
CREATE TABLE processed_events (
  event_id VARCHAR(128) PRIMARY KEY,
  topic VARCHAR(128) NOT NULL,
  partition_id INT NOT NULL,
  offset_value BIGINT NOT NULL,
  handler VARCHAR(128) NOT NULL,
  processed_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

```sql
CREATE TABLE outbox_events (
  id VARCHAR(128) PRIMARY KEY,
  topic VARCHAR(128) NOT NULL,
  message_key VARCHAR(128) NOT NULL,
  payload JSONB NOT NULL,
  status VARCHAR(32) NOT NULL,
  retry_count INT NOT NULL DEFAULT 0,
  created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

这些 SQL 后面会在第 9 阶段项目里真正使用。
