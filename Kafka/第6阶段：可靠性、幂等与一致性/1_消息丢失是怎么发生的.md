# 1. 消息丢失是怎么发生的

本节目标：理解 Kafka 链路中消息丢失可能发生在哪些位置，并能为 Go 后端关键业务设计防丢方案。

很多人说 Kafka 可靠，但“Kafka 可靠”不等于“你的业务消息一定不会丢”。消息从 Go 代码产生，到 Kafka broker，再到 consumer 业务处理，中间有多个环节，每个环节都有可能出问题。

---

## 一、先定义什么叫消息丢失

在业务视角中，消息丢失通常指：

```text
某个业务事件已经发生，但下游最终没有机会处理它。
```

例如：

```text
订单已经创建。
但是库存服务永远没有扣库存。
```

这可能是 Kafka 没收到消息，也可能是 Kafka 收到了但 consumer 提前提交 offset。

---

## 二、Producer 侧丢失

### 1. 发送失败但业务忽略错误

错误代码：

```go
_ = producer.Publish(ctx, "order.created", msg)
```

如果发送失败，业务完全不知道。

正确做法：

```go
if err := producer.Publish(ctx, "order.created", msg); err != nil {
    return fmt.Errorf("publish order.created: %w", err)
}
```

关键业务不能忽略 producer 错误。

---

## 三、Producer Buffer 丢失

异步 producer 可能先把消息放进本地 buffer。

如果服务直接退出：

```text
buffer 中的消息还没发送
进程退出
消息丢失
```

解决：

- 服务退出前 flush 或 close。
- 使用同步发送或等待 delivery report。
- 关键业务使用 outbox。

---

## 四、acks 配置过低

如果：

```text
acks=0
```

producer 不等待 broker 确认，消息可能没到 broker 就丢了。

如果：

```text
acks=1
```

leader 写入成功后返回，但还没复制到 follower 时 leader 宕机，也可能丢。

关键业务推荐：

```text
acks=all
replication.factor=3
min.insync.replicas=2
```

---

## 五、Broker 侧丢失

Broker 层风险包括：

- 副本数太少。
- ISR 不健康。
- leader 宕机时数据未复制。
- 磁盘故障。
- retention 太短。

本地学习环境：

```text
replication.factor=1
```

它没有高可用能力。

生产关键 topic 不应只用 1 个副本。

---

## 六、Retention 导致“看起来丢失”

如果 consumer 停止太久，topic retention 到期，旧消息被清理。

场景：

```text
retention = 1 天
consumer 停止 = 3 天
consumer 恢复
最早两天消息已经被清理
```

这对 consumer 来说就是无法再消费。

解决：

- 根据业务恢复窗口设置 retention。
- 监控 consumer lag。
- 关键事件保留更久。

---

## 七、Consumer 侧丢失

最常见：

```text
拉到消息
提交 offset
业务处理失败
```

错误流程：

```text
poll message
commit offset
handle business
```

如果 handle 失败，Kafka 已经认为这条消息处理过了。

正确流程：

```text
poll message
handle business
commit offset after success
```

---

## 八、写 DLQ 失败但提交 Offset

错误流程：

```text
业务处理失败
尝试写 DLQ
DLQ 写失败
仍然提交原 offset
```

后果：

```text
原消息既没处理成功，也没进入 DLQ
```

这也是消息丢失。

正确：

```text
写 DLQ 成功后才提交原 offset
```

---

## 九、数据库与 Kafka 不一致

典型场景：

```text
订单写数据库成功
发送 Kafka 失败
```

结果：

```text
订单存在，但 order.created 事件没有发布
```

解决：

```text
Outbox Pattern
```

同一个数据库事务：

```text
写 orders
写 outbox_events
提交事务
```

后台 worker 再发送 Kafka。

---

## 十、Go 后端防丢清单

Producer：

- 不忽略发送错误。
- 使用 context timeout。
- 关键业务使用可靠 acks。
- 服务退出前 close/flush。
- 关键链路使用 outbox。

Broker：

- 关键 topic 多副本。
- 合理 min ISR。
- 监控 under replicated partitions。
- 合理 retention。

Consumer：

- 关闭自动提交。
- 业务成功后提交 offset。
- retry/DLQ 成功后才提交。
- 处理失败不吞错误。

## 十二、本节小结

- 消息丢失可能发生在 producer、broker、consumer、数据库与 Kafka 边界。
- producer 发送失败不能忽略。
- 关键 topic 需要多副本和合理 acks。
- retention 太短会让落后 consumer 无法恢复。
- consumer 必须业务成功后提交 offset。
- 写 retry/DLQ 失败不能提交原 offset。
- Outbox 是解决数据库成功但 Kafka 发送失败的重要模式。

## 十三、端到端防丢时间线

以订单事件为例，可靠链路应该这样设计：

```text
T1 order-service 接收创建订单请求
T2 开启数据库事务
T3 写 orders
T4 写 order_items
T5 写 outbox_events(order.created)
T6 提交事务
T7 outbox-worker 扫描 PENDING 事件
T8 producer 发送 Kafka 并等待 ack
T9 发送成功后标记 outbox SENT
T10 inventory-service 消费消息
T11 业务处理成功
T12 提交 offset
```

每一步都要问：

```text
如果这里崩溃，消息还找得到吗？
```

如果答案是否定的，就存在消息丢失风险。

---

## 十四、Outbox 为什么能防丢

不用 Outbox 时：

```text
insert orders 成功
producer send 失败
```

这两个动作不在同一个事务里，无法保证一起成功或一起失败。

使用 Outbox：

```sql
BEGIN;

INSERT INTO orders(...);

INSERT INTO outbox_events(
  id,
  aggregate_type,
  aggregate_id,
  event_type,
  topic,
  message_key,
  payload,
  status
) VALUES (
  'evt_order_created_order_1001',
  'order',
  'order_1001',
  'order.created',
  'order.created',
  'order_1001',
  '{"event_id":"evt_order_created_order_1001"}',
  'PENDING'
);

COMMIT;
```

只要事务提交，订单和待发送事件就同时存在。

Kafka 暂时不可用也没关系，worker 下一轮继续发。

## 十五、Producer 配置示例

不同 Go 客户端配置名称略有差异，但目标一致：

```go
writer := &kafka.Writer{
    Addr:         kafka.TCP("localhost:9092"),
    Topic:        "order.created",
    RequiredAcks: kafka.RequireAll,
    Async:        false,
}
```

关键点：

```text
等待 broker ack。
发送失败返回 error。
调用方不能吞掉 error。
```

如果客户端支持幂等 producer，也应该开启。但要记住：

```text
producer 幂等不能替代 consumer 幂等。
```

## 十六、Consumer 防丢代码顺序

错误顺序：

```go
consumer.Commit(ctx, msg)
err := handler.Handle(ctx, msg)
```

正确顺序：

```go
err := handler.Handle(ctx, msg)
if err != nil {
    return err
}
return consumer.Commit(ctx, msg)
```

如果 handler 失败，不提交 offset。

如果 handler 发现不可重试坏消息：

```go
if err := dlqProducer.Publish(ctx, dlqMsg); err != nil {
    return err
}
return consumer.Commit(ctx, msg)
```

也就是说，消息必须满足下面任一条件，才能提交 offset：

```text
业务处理成功。
已经成功写入 retry topic。
已经成功写入 DLQ。
```

---

## 十七、防丢不等于不重复

很多可靠设计会把系统推向“至少一次”：

```text
宁愿重复，也不要静默丢失。
```

例如：

```text
outbox-worker 发送 Kafka 成功。
更新 outbox status 失败。
下一轮又发送一次。
```

这不是防丢方案失败，而是它选择了“可能重复”的一侧。

所以第 6 阶段的两句话要连在一起理解：

```text
Outbox 防丢。
幂等防重复副作用。
```

---

## 十八、排查消息丢失的顺序

当下游说“没收到消息”时，不要直接怪 Kafka。按顺序查：

1. 业务事件是否真的发生。
2. outbox_events 是否有记录。
3. outbox_events 状态是 `PENDING`、`SENT` 还是 `FAILED`。
4. Kafka topic 是否有消息。
5. consumer group lag 是否异常。
6. consumer 日志是否处理失败。
7. retry topic 或 DLQ 是否有消息。
8. offset 是否被提前提交。

常用 SQL：

```sql
SELECT id, topic, status, retry_count, created_at, sent_at
FROM outbox_events
WHERE aggregate_id = 'order_1001';
```

常用 Kafka 命令：

```bash
kafka-consumer-groups.sh \
  --bootstrap-server localhost:9092 \
  --describe \
  --group inventory-service
```

## 十九、面试表达

可以这样回答“Kafka 消息如何防丢”：

```text
我会分 producer、broker、consumer 和数据库边界来看。
producer 侧不能忽略发送错误，关键 topic 使用 acks=all 和多副本。
数据库与 Kafka 的一致性用 Outbox 模式解决，先在本地事务里写业务表和 outbox 表，再由 worker 异步发布。
consumer 侧关闭自动提交，业务处理成功后再提交 offset；如果失败消息成功写入 retry 或 DLQ，才提交原 offset。
```
