# 3. 业务幂等设计

本节目标：掌握 Go 后端 consumer 常用的幂等设计方法，包括 `processed_events` 去重表、业务唯一约束、状态机和外部接口幂等键。

幂等不是 Kafka 的附属知识，而是消息系统落地的核心。只要 consumer 可能重复处理，你就必须设计幂等。

---

## 一、什么是幂等

幂等是指：

```text
同一个操作执行一次和执行多次，最终业务结果一致
```

例如：

```sql
UPDATE orders SET status = 'PAID' WHERE id = 'order_1';
```

执行多次，状态仍然是 `PAID`。

但：

```sql
UPDATE inventory SET available = available - 1 WHERE sku_id = 'sku_1';
```

执行多次会多扣库存，不幂等。

---

## 二、Event ID 是基础

每个事件必须有稳定唯一的 `event_id`。

例如：

```text
evt_order_created_order_1001
```

不要每次 retry 都生成新的 event_id。

否则 consumer 无法识别这是同一个业务事件。

---

## 三、processed_events 表

通用去重表：

```sql
CREATE TABLE processed_events (
  event_id VARCHAR(128) PRIMARY KEY,
  topic VARCHAR(128) NOT NULL,
  partition_id INT NOT NULL,
  offset_value BIGINT NOT NULL,
  handler VARCHAR(128) NOT NULL,
  processed_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

处理流程：

```text
开启数据库事务
插入 processed_events(event_id)
如果冲突，说明处理过，直接返回成功
执行业务逻辑
提交事务
提交 Kafka offset
```

---

## 四、为什么去重表和业务操作要在同一事务

错误做法：

```text
先插入 processed_events
提交
再扣库存
```

如果插入成功后服务崩溃：

```text
事件被标记处理过
库存却没扣
```

正确：

```text
processed_events 插入和库存扣减在同一个事务
```

事务提交代表这个事件的业务处理完成。

---

## 五、库存扣减示例

伪代码：

```go
tx := db.BeginTx(ctx)
defer tx.Rollback()

inserted := insertProcessedEvent(tx, event.EventID)
if !inserted {
    return tx.Commit()
}

ok := deductInventory(tx, skuID, quantity)
if !ok {
    return ErrInsufficientInventory
}

return tx.Commit()
```

重复消息来时：

```text
insertProcessedEvent 发现 event_id 已存在
直接返回成功
Kafka offset 可以提交
```

---

## 六、业务唯一约束

有些场景可以用业务唯一约束幂等。

例如发券：

```sql
CREATE UNIQUE INDEX uk_coupon_grant
ON user_coupons(user_id, coupon_template_id, source_order_id);
```

重复发券时，第二次插入会冲突。

应用代码把冲突识别为：

```text
已经发过，返回成功
```

---

## 七、状态机幂等

订单状态：

```text
CREATED -> PAID -> SHIPPED -> FINISHED
```

重复收到 `payment.succeeded`：

```text
如果订单已是 PAID 或更后状态，直接返回成功
```

收到非法状态流转：

```text
如果订单已 CANCELLED，却收到 payment.succeeded，需要进入异常处理
```

状态机不仅处理重复，也处理乱序和非法事件。

---

## 八、外部接口幂等键

调用外部系统时也要传幂等键。

例如：

- 支付退款：`refund_id`
- 发券：`grant_id`
- 短信：`message_id`

如果外部接口不支持幂等，你要在本地记录调用结果，避免重复调用。

---

## 九、幂等和 Offset 的关系

业务成功但 commit offset 失败：

```text
消息会重复消费
```

幂等保证重复消费不会重复产生副作用。

所以：

```text
手动提交 offset 仍然需要幂等
```

---

## 十、常见误区

### 1. 有事务就不需要幂等

错误。

事务保证一次处理内部一致，幂等保证重复处理不出错。

### 2. event_id 每次生成新的

错误。

同一业务事件必须使用同一个 event_id。

### 3. 去重表可以先提交

不推荐。

要和业务操作同事务。

### 4. 状态更新都是幂等

不一定。

状态流转要校验当前状态。

## 十二、本节小结

- 幂等是消息系统落地的核心。
- event_id 必须稳定唯一。
- processed_events 是通用去重方案。
- 去重记录和业务操作应在同一数据库事务。
- 业务唯一约束可以实现幂等。
- 状态机可以处理重复和乱序。
- 外部接口也需要幂等键。

## 十三、processed_events 的完整 SQL

建议把去重表设计成这样：

```sql
CREATE TABLE processed_events (
  event_id VARCHAR(128) PRIMARY KEY,
  event_type VARCHAR(128) NOT NULL,
  topic VARCHAR(128) NOT NULL,
  partition_id INT NOT NULL,
  offset_value BIGINT NOT NULL,
  handler VARCHAR(128) NOT NULL,
  processed_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_processed_events_handler_time
ON processed_events(handler, processed_at);
```

字段解释：

| 字段 | 作用 |
| --- | --- |
| `event_id` | 幂等判断的核心 |
| `event_type` | 排查时知道事件类型 |
| `topic` | 原始 topic |
| `partition_id` | 原始分区 |
| `offset_value` | 原始 offset |
| `handler` | 哪个 consumer 处理的 |
| `processed_at` | 处理完成时间 |

`event_id` 用主键，不要只建普通索引。主键可以直接阻止重复插入。

## 十四、Go 实现模板

```go
func (s *Service) HandleOrderCreated(ctx context.Context, msg ConsumedMessage, event OrderCreatedEvent) error {
    tx, err := s.db.Begin(ctx)
    if err != nil {
        return err
    }
    defer tx.Rollback(ctx)

    inserted, err := s.repo.InsertProcessedEvent(ctx, tx, ProcessedEvent{
        EventID:   event.EventID,
        EventType: event.EventType,
        Topic:     msg.Topic,
        Partition: msg.Partition,
        Offset:    msg.Offset,
        Handler:   "inventory-service",
    })
    if err != nil {
        return err
    }

    if !inserted {
        return tx.Commit(ctx)
    }

    if err := s.repo.DeductInventory(ctx, tx, event.Data.Items); err != nil {
        return err
    }

    return tx.Commit(ctx)
}
```

这个模板的关键不是代码长相，而是顺序：

```text
begin
insert processed_events
business side effect
commit
```

Kafka offset 必须在这个函数成功返回以后再提交。

## 十五、插入去重记录的实现

```go
func (r *Repository) InsertProcessedEvent(ctx context.Context, tx pgx.Tx, e ProcessedEvent) (bool, error) {
    tag, err := tx.Exec(ctx, `
        INSERT INTO processed_events(event_id, event_type, topic, partition_id, offset_value, handler)
        VALUES ($1, $2, $3, $4, $5, $6)
        ON CONFLICT (event_id) DO NOTHING
    `, e.EventID, e.EventType, e.Topic, e.Partition, e.Offset, e.Handler)
    if err != nil {
        return false, err
    }

    return tag.RowsAffected() == 1, nil
}
```

为什么不用先查：

```sql
SELECT 1 FROM processed_events WHERE event_id = $1;
```

再插入？

因为并发时可能两个 consumer 同时查不到，然后都继续处理。用唯一约束加 `INSERT` 才是更稳的门禁。

## 十六、幂等不是只能靠去重表

不同业务适合不同方案：

| 业务 | 推荐幂等方式 |
| --- | --- |
| 库存扣减 | `processed_events` + 条件 UPDATE |
| 发券 | 业务唯一约束 |
| 订单支付成功 | 状态机 |
| 发送短信 | 本地发送记录 + message_id |
| 调用退款接口 | 外部幂等键 refund_id |

真实项目经常组合使用：

```text
processed_events 防止同一事件重复进入 handler。
业务唯一约束防止同一业务资源重复创建。
状态机防止非法流转。
```

---

## 十七、幂等和并发安全

幂等还要考虑并发。

库存扣减不要写成：

```go
available := repo.GetAvailable(ctx, skuID)
if available >= quantity {
    repo.UpdateAvailable(ctx, skuID, available-quantity)
}
```

应该写成单条 SQL：

```sql
UPDATE inventory
SET available = available - $1
WHERE sku_id = $2
  AND available >= $1;
```

然后看影响行数：

```text
1 行：扣减成功。
0 行：库存不足。
```

这既是并发控制，也是业务判断。

## 十八、幂等测试用例

你可以为库存服务设计这几组测试：

| 用例 | 输入 | 预期 |
| --- | --- | --- |
| 首次消费 | 新 `event_id` | 库存扣减 |
| 重复消费 | 相同 `event_id` | 库存不变 |
| 库存不足 | 库存小于购买数量 | 不扣减，产生失败事件 |
| 数据库错误 | repository 返回错误 | 不提交 offset，或进入 retry |
| offset 提交失败后重放 | 已有 `processed_events` | 直接成功 |

不要只测 happy path。Kafka 可靠性代码的价值主要体现在异常路径。

## 十九、常见面试追问

### 1. processed_events 会不会无限增长？

会，所以生产里需要保留策略。可以按业务要求保留 30 天、90 天或更久，也可以按事件类型归档。保留时间不能短于可能重放消息的时间窗口。

### 2. 删除 processed_events 后重放旧消息怎么办？

如果旧消息仍可能被重放，就不能删除对应幂等记录。删除策略必须和 Kafka retention、DLQ 重放策略一起设计。

### 3. event_id 谁生成？

一般由事件生产方生成，并在 retry、DLQ、重放时保持不变。consumer 不应该为同一业务事件重新生成 event_id。
