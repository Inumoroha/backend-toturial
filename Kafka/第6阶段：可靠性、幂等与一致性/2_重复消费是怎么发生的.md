# 2. 重复消费是怎么发生的

本节目标：理解 Kafka 中重复消费为什么是正常现象，并知道 Go 后端业务为什么必须做幂等。

很多初学者会把重复消费当成 Kafka 出问题。实际上，只要你追求“不轻易丢消息”，就要接受“可能重复”。在 Kafka 业务系统中，重复消费是必须设计处理的常态。

---

## 一、重复消费的定义

业务视角：

```text
同一个业务事件被 consumer 处理了多次
```

例如：

```text
event_id = evt_order_created_1001
```

库存服务收到两次。

如果没有幂等，可能重复扣库存。

---

## 二、Consumer 成功但提交 Offset 失败

最经典场景：

```text
consumer 拉到消息
业务处理成功
提交 offset 前崩溃
consumer 重启
再次消费同一条消息
```

这不是 Kafka 错误，而是至少一次语义的正常结果。

解决：

```text
业务幂等
```

---

## 三、提交 Offset 请求失败

流程：

```text
业务成功
consumer 提交 offset
网络超时
consumer 不知道提交是否成功
```

如果提交实际没成功，下次还会重复。

即使提交实际成功，客户端也可能因为没收到响应而记录错误。

所以 commit 失败要记录，但不能假设业务没成功。

---

## 四、Rebalance 导致重复

流程：

```text
consumer-A 正在处理 offset 100
还没提交
发生 rebalance
partition 分给 consumer-B
consumer-B 从已提交 offset 继续
offset 100 再次被处理
```

解决：

- handler 不要太慢。
- 优雅退出。
- 业务幂等。

---

## 五、Producer Retry 导致重复消息

流程：

```text
producer 发送成功
ack 丢失
producer 重试
Kafka 中出现两条相同业务事件
```

开启幂等 producer 可以降低这类重复，但不能消灭所有业务重复。

---

## 六、Outbox Worker 重复发布

流程：

```text
outbox worker 发送 Kafka 成功
更新 outbox status=sent 失败
下一轮再次扫描到 PENDING
再次发送同一事件
```

所以：

```text
Outbox 防丢，但可能重复
```

consumer 仍然要幂等。

---

## 七、人工重放 DLQ

生产中修复坏数据后，可能人工重放 DLQ。

如果某些消息其实之前部分处理过，重放会导致重复。

所以 DLQ 重放也要求：

```text
consumer 幂等
```

---

## 八、至少一次语义

Go 后端常见目标：

```text
Kafka 至少一次投递 + 业务幂等 = 用户视角接近只处理一次
```

Kafka 负责尽量不丢。

业务系统负责重复不出错。

---

## 九、重复消费的影响

不同业务影响不同：

| 业务 | 重复后果 |
| --- | --- |
| 扣库存 | 多扣库存 |
| 发优惠券 | 重复发券 |
| 发短信 | 多发短信 |
| 写日志 | 多一条日志，可能可接受 |
| 更新状态为 PAID | 可能天然幂等 |

不是所有重复都同样严重。

---

## 十、Go 后端处理原则

每条事件要有：

```text
event_id
```

consumer 要能判断：

```text
这个 event_id 是否处理过
```

常见方案：

- processed_events 表。
- 业务唯一约束。
- 状态机。
- 外部接口幂等键。

---

## 十一、常见误区

### 1. 重复消费说明 Kafka 不可靠

不是。

重复是至少一次语义的代价。

### 2. 手动提交就不会重复

不是。

commit 失败、rebalance 都可能重复。

### 3. Producer 幂等后 consumer 不需要幂等

错误。

producer 幂等不覆盖 consumer 和业务层。

### 4. 日志重复可以接受，所以所有业务重复都能接受

错误。

库存、支付、发券必须处理重复。

## 十三、本节小结

- 重复消费是 Kafka 业务系统的常态。
- commit 失败、rebalance、producer retry、outbox 重试、DLQ 重放都可能导致重复。
- 关键业务必须幂等。
- event_id 是幂等的基础。
- Kafka 负责尽量不丢，业务负责重复不出错。

## 十四、用时间线理解重复消费

假设库存服务消费 `order.created`：

```text
T1 consumer 拉到 offset=100
T2 consumer 执行业务，库存从 100 扣到 98
T3 consumer 准备提交 offset=101
T4 服务进程崩溃
T5 服务重启
T6 Kafka 从上一次已提交 offset 继续投递 offset=100
```

从 Kafka 角度看，它只是发现：

```text
offset=100 没有被提交。
```

所以它重新投递是合理的。

从业务角度看，如果没有幂等，就会变成：

```text
第一次扣库存：100 -> 98
第二次扣库存：98 -> 96
```

这就是“技术上正常，业务上出错”的典型场景。

## 十五、重复消息和重复消费的区别

这两个概念很像，但来源不同。

```text
重复消息：Kafka topic 里真的有两条业务含义相同的消息。
重复消费：topic 里可能只有一条消息，但 consumer 处理了多次。
```

重复消息可能来自：

- producer retry。
- outbox-worker 重复发布。
- 人工重放。

重复消费可能来自：

- offset 提交失败。
- consumer 崩溃。
- rebalance。

对业务系统来说，不管来源是什么，最终都要靠幂等处理。

## 十六、Go 代码里的危险写法

危险写法：

```go
func (h *Handler) Handle(ctx context.Context, msg Message) error {
    if err := h.repo.DeductInventory(ctx, msg.OrderID); err != nil {
        return err
    }
    return nil
}
```

这段代码的问题是：

```text
它没有判断这个 event_id 是否已经处理过。
```

更合理的结构：

```go
func (h *Handler) Handle(ctx context.Context, msg Message) error {
    tx, err := h.db.Begin(ctx)
    if err != nil {
        return err
    }
    defer tx.Rollback(ctx)

    inserted, err := h.repo.InsertProcessedEvent(ctx, tx, msg.EventID)
    if err != nil {
        return err
    }
    if !inserted {
        return tx.Commit(ctx)
    }

    if err := h.repo.DeductInventory(ctx, tx, msg.OrderID); err != nil {
        return err
    }

    return tx.Commit(ctx)
}
```

核心变化：

```text
先登记 event_id。
登记成功才执行业务。
重复 event_id 直接返回成功。
```

## 十七、如何观察重复消费

可以故意制造一次重复：

```text
1. consumer 处理消息后，在提交 offset 前让进程退出。
2. 重启 consumer。
3. 观察同一个 topic/partition/offset 是否再次出现。
```

日志应该包含：

```text
topic=order.created
partition=0
offset=100
event_id=evt_order_created_1001
```

如果第二次出现同样的 `event_id`，但库存没有重复扣减，说明幂等生效。

## 十八、常见面试追问

### 1. Kafka 能不能做到完全不重复？

可以从 producer、broker 事务等层面降低重复，但业务系统仍然要按可能重复来设计。因为 consumer 处理业务和提交 offset 之间没有一个天然覆盖所有外部系统的原子事务。

### 2. 手动提交 offset 是否就安全？

手动提交只是让你控制提交时机。它不能解决“业务成功但提交失败”。

### 3. 为什么不先提交 offset 再处理业务？

那会变成可能丢消息：

```text
offset 提交成功
业务处理失败
Kafka 不会再投递这条消息
```

所以可靠消费通常选择：

```text
先处理业务，再提交 offset，然后用业务幂等承受重复。
```

## 二十一、重复消费复盘问题

每次遇到重复消费，都按下面问题复盘：

```text
重复的是同一个 event_id，还是业务上等价的两条事件？
第一次处理是否已经成功写入数据库？
offset 是否在业务成功后才提交？
consumer 是否在提交前崩溃或超时？
是否发生了 rebalance？
是否有人手动重放了 DLQ？
幂等表或唯一约束是否生效？
```

这些问题能帮助你判断重复消费是正常的 at-least-once 现象，还是代码边界真的写错了。
