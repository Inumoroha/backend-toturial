# 6. 综合实践：手动提交与失败处理

本节目标：把 consumer group、offset、manual commit、retry、DLQ 的思想串成一条 Go 后端消费链路设计。

这一节以设计和伪代码为主，因为命令行 consumer 不适合完整演示手动提交。真正代码实现会在第 5 阶段展开。

---

## 一、实践场景

业务链路：

```text
order-service 发布 order.created
inventory-service 消费 order.created
扣减库存
成功后提交 offset
失败时进入 retry 或 DLQ
```

## 三、错误分类

### 可重试错误

例如：

- 数据库连接超时。
- Redis 临时不可用。
- 下游 HTTP 超时。

处理：

```text
本地短重试
仍失败则写 retry topic
写 retry 成功后提交原 offset
```

### 不可重试错误

例如：

- JSON 格式错误。
- 缺少 order_id。
- schema version 不支持。

处理：

```text
写 DLQ
写 DLQ 成功后提交原 offset
```

## 四、主流程伪代码

```go
func ConsumeLoop(ctx context.Context) {
    for ctx.Err() == nil {
        msg := consumer.Fetch(ctx)

        err := handler.Handle(ctx, msg)
        if err == nil {
            consumer.Commit(ctx, msg)
            continue
        }

        if IsRetryable(err) {
            retryErr := retryProducer.Publish(ctx, BuildRetryMessage(msg, err))
            if retryErr == nil {
                consumer.Commit(ctx, msg)
            }
            continue
        }

        dlqErr := dlqProducer.Publish(ctx, BuildDLQMessage(msg, err))
        if dlqErr == nil {
            consumer.Commit(ctx, msg)
        }
    }
}
```

重点：

```text
没有处理成功，也没有成功转移到 retry/DLQ，就不要提交 offset
```

## 五、Handler 伪代码

```go
func HandleOrderCreated(ctx context.Context, msg Message) error {
    event, err := DecodeOrderCreated(msg.Value)
    if err != nil {
        return NonRetryable(err)
    }

    if event.EventID == "" || event.Data.OrderID == "" {
        return NonRetryable(errors.New("invalid event"))
    }

    err = inventoryService.Deduct(ctx, event)
    if err != nil {
        return Retryable(err)
    }

    return nil
}
```

## 六、Retry 消息应该包含什么

```json
{
  "event_id": "evt_order_created_1001",
  "original_topic": "order.created",
  "original_partition": 1,
  "original_offset": 203,
  "attempt": 1,
  "last_error": "database timeout",
  "payload": {}
}
```

为什么要保留原始上下文？

因为线上排查时要能知道：

```text
这条 retry 消息来自哪里
原始 offset 是多少
失败原因是什么
已经重试几次
```


DLQ 不是垃圾桶，而是待排查问题列表。

## 八、优雅退出设计

consumer 收到 SIGTERM：

```text
停止拉取新消息
等待当前 handler 完成
成功则提交 offset
关闭 consumer
关闭 retry/DLQ producer
退出
```

设置最大等待时间，例如：

```text
30s
```

超过时间仍未完成，则退出并依赖重复消费和幂等兜底。

## 十、常见误区

### 1. 失败就一直不提交

坏消息会卡住 partition。

不可重试错误应进入 DLQ。

### 2. 写 DLQ 失败也提交

不应该。

否则原消息既没处理，也没被保留。

### 3. retry 不需要 attempt

需要。

否则无法限制最大重试次数。

### 4. consumer 退出不用管当前消息

不建议。

应该尽量优雅退出。

## 十二、本节小结

- Consumer 可靠性核心是 offset 提交时机。
- 业务成功后提交 offset。
- 可重试错误写 retry topic。
- 不可重试错误写 DLQ。
- 写 retry/DLQ 成功后才能提交原 offset。
- commit 失败会导致重复消费，所以业务必须幂等。
- Go consumer 必须支持优雅退出。

## 十四、最终伪代码

```go
err := handler.Handle(ctx, msg)
if err == nil {
    return commit(msg)
}

switch Classify(err) {
case Retryable:
    if publishRetry(msg, err) == nil {
        return commit(msg)
    }
case NonRetryable:
    if publishDLQ(msg, err) == nil {
        return commit(msg)
    }
}

return err
```

这段代码的原则是：

```text
成功处理或成功转移后，才提交 offset。
```

## 十七、常见错误

如果坏 JSON 一直反复消费，说明不可重试错误没有成功进入 DLQ，或者写 DLQ 后没有提交 offset。

如果数据库错误后消息消失，说明 retry 写失败时仍然提交了 offset。

## 十八、最终验收

实践结束后，用表格记录：

| 场景 | retry | DLQ | commit | 结果 |
| --- | --- | --- | --- | --- |
| 正常消息 | 否 | 否 | 是 | 业务成功 |
| 数据库超时 | 是 | 否 | 是 | 原 topic 前进 |
| 坏 JSON | 否 | 是 | 是 | 主链路不卡住 |
| DLQ 写失败 | 否 | 失败 | 否 | 原消息保留 |

这张表是可靠消费的最小验收。

## 二十一、失败路径验收

本节实践必须至少模拟三类失败：

```text
业务处理成功：提交 offset。
业务处理失败但可重试：不立即提交，进入重试策略。
业务处理失败且不可恢复：写入 DLQ 后提交 offset。
```

每种失败都要记录 event_id、topic、partition、offset 和错误类型。

如果没有失败路径日志，后面做可靠性设计时就无法证明系统真的知道自己丢在哪里、重在哪里、卡在哪里。
