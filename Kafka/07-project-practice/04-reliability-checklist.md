# 04 项目可靠性检查清单

本节目标：在项目完成前，用清单检查是否真的具备生产思维。

## Producer 检查

- 是否统一封装 producer。
- 是否复用 producer，而不是每条消息创建一个。
- 是否设置发送超时。
- 关键业务是否使用可靠发送配置。
- 是否记录发送成功和失败日志。
- 服务退出前是否 close 或 flush。
- 消息是否包含 `event_id`、`event_type`、`version`。

## Consumer 检查

- 是否使用 consumer group。
- 是否手动提交 offset。
- 是否业务成功后再提交。
- 是否有错误分类。
- 是否支持 retry topic。
- 是否支持 DLQ。
- 是否优雅退出。
- 是否记录 topic、partition、offset、key、event_id。

## 幂等检查

- 是否有 `processed_events` 表。
- 是否使用业务唯一约束。
- 是否考虑状态机重复事件。
- 外部接口是否传幂等键。
- 重放 DLQ 是否不会造成重复副作用。

## Outbox 检查

- 订单表和 outbox 表是否在同一个数据库事务中写入。
- outbox worker 是否可重试。
- 发布成功但更新状态失败时是否能安全重试。
- outbox 历史数据是否有清理策略。

## Retry 与 DLQ 检查

- retry topic 是否保留原始上下文。
- attempt 是否递增。
- 是否有最大重试次数。
- DLQ 是否包含错误原因。
- DLQ 是否有监控和告警。
- 是否设计了人工修复和重放流程。

## 可观测性检查

- 是否有 consumer lag 监控。
- 是否有处理耗时指标。
- 是否有错误率指标。
- 是否有 DLQ 指标。
- 是否能通过 event_id 查完整链路。

## 本节练习

1. 用这份清单检查你的项目。
2. 每个未完成项都写一个 issue。
3. 选三个最高风险项优先补齐。
4. 思考：如果项目只能保留一个监控指标，你会选哪个？为什么？
