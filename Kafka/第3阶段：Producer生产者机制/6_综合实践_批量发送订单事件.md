# 6. 综合实践：批量发送订单事件

本节目标：综合使用 topic、message key、headers、acks、batch、compression 的知识，设计并执行一次订单事件批量发送实验。

这一节仍然以命令行和设计为主，Go 代码会在第 5 阶段展开。你需要通过实验看见：带 key 的订单消息如何进入 partition，批量发送后如何观察 consumer 输出和 lag。

---

## 一、实践目标

完成以下内容：

```text
创建 order.created topic
发送多条带 order_id key 的消息
打印 key、partition、offset
观察相同 order_id 的分区
设计 Go producer 事件结构
写出生产级 producer 配置建议
```

---

## 二、创建 Topic

进入 Kafka 容器：

```powershell
docker exec -it kafka bash
```

创建 topic：

```bash
kafka-topics.sh \
  --bootstrap-server localhost:9092 \
  --create \
  --if-not-exists \
  --topic practice.order.created \
  --partitions 3 \
  --replication-factor 1
```

查看：

```bash
kafka-topics.sh \
  --bootstrap-server localhost:9092 \
  --describe \
  --topic practice.order.created
```

## 三、准备订单事件

学习阶段可以先用简单 value：

```text
created
paid
shipped
```

生产事件应该是 JSON：

```json
{
  "event_id": "evt_order_created_1001",
  "event_type": "order.created",
  "version": 1,
  "occurred_at": "2026-07-05T12:00:00Z",
  "data": {
    "order_id": "order_1001",
    "user_id": "user_88",
    "total_amount": 19800
  }
}
```

本实验先使用命令行 key/value 观察分区。

## 四、发送带 Key 的消息

启动 producer：

```bash
kafka-console-producer.sh \
  --bootstrap-server localhost:9092 \
  --topic practice.order.created \
  --property parse.key=true \
  --property key.separator=:
```

输入：

```text
order-1001:created
order-1001:paid
order-1001:shipped
order-1002:created
order-1002:paid
order-1003:created
order-1004:created
order-1005:created
```

退出 producer。

## 五、观察 Partition 与 Offset

启动 consumer：

```bash
kafka-console-consumer.sh \
  --bootstrap-server localhost:9092 \
  --topic practice.order.created \
  --from-beginning \
  --property print.key=true \
  --property print.partition=true \
  --property print.offset=true \
  --timeout-ms 10000
```

记录：

| key | value | partition | offset |
| --- | --- | --- | --- |
| order-1001 | created | | |
| order-1001 | paid | | |
| order-1001 | shipped | | |

观察 `order-1001` 是否进入同一个 partition。

## 六、设计 Go Event 结构

建议结构：

```go
type EventEnvelope[T any] struct {
    EventID    string    `json:"event_id"`
    EventType  string    `json:"event_type"`
    Version    int       `json:"version"`
    OccurredAt time.Time `json:"occurred_at"`
    Data       T         `json:"data"`
}

type OrderCreatedData struct {
    OrderID     string `json:"order_id"`
    UserID      string `json:"user_id"`
    TotalAmount int64  `json:"total_amount"`
}
```

发送时：

```text
topic = practice.order.created
key = order_id
value = JSON(event)
headers = event_id, event_type, schema_version
```

## 七、设计 Producer 配置

本地学习：

```text
brokers=localhost:9092
acks=all
compression=lz4
linger.ms=5~20
client.id=order-service
```

生产关键业务：

```text
replication.factor=3
min.insync.replicas=2
acks=all
enable.idempotence=true
retries 开启
delivery timeout 合理
```

注意：

```text
acks=all 在单节点学习环境中不等于生产高可用
```

因为本地 topic 只有一个副本。

## 八、失败处理设计

如果创建订单接口直接发送 Kafka：

```text
写数据库成功
发送 Kafka 失败
```

会很尴尬。

更推荐：

```text
数据库事务写 orders + outbox_events
outbox worker 发送 Kafka
发送失败可重试
```

这个实践会在第 6 阶段和项目阶段详细展开。

## 十、常见问题

### 1. 相同 key 没有进入预期 partition

不同客户端和命令行工具可能显示格式不同，先确认：

- 是否启用了 `parse.key=true`。
- key separator 是否正确。
- 输入格式是否是 `key:value`。

### 2. 只看到 value，看不到 key

consumer 需要加：

```bash
--property print.key=true
```

### 3. 消息顺序看起来不完全按输入顺序

多 partition topic 不保证全局顺序。

只观察同一个 key 在同一个 partition 内的顺序。

## 十二、本节小结

- 订单事件应该使用稳定业务 key。
- 命令行可以观察 key、partition、offset。
- 生产事件应该有 event_id、event_type、version、occurred_at、data。
- 关键业务 producer 配置要兼顾 acks、幂等、重试和副本。
- Kafka 发送失败要有业务补偿方案。
- outbox pattern 是解决数据库和 Kafka 最终一致性的常用方式。

## 十四、批量发送验收表

| 检查项 | 预期 |
| --- | --- |
| key | 使用 `order_id` |
| event_id | 每条事件稳定唯一 |
| version | 明确为 `1` |
| occurred_at | 使用 UTC |
| send error | 不忽略 |
| topic | `order.created` |

发送后用 console consumer 验证：

```bash
kafka-console-consumer.sh \
  --bootstrap-server localhost:9092 \
  --topic order.created \
  --from-beginning \
  --property print.key=true \
  --max-messages 5
```

---

## 十五、常见问题

如果所有消息都落到同一个 partition，先检查 key 是否太集中。

如果发送速度很慢，观察 batch、linger、acks 和消息体大小。

如果偶发发送失败，先不要简单重试到成功，要思考业务是否需要 Outbox 保存未发送事件。

## 十八、验收标准

批量发送实践完成后，应该能证明：

- 消息数量正确。
- key 正确打印。
- event_id 不重复。
- 发送失败有错误日志。
- 同一 order_id 的消息进入同一 partition。

如果只能看到“发送完成”，说明验收还不够细。

## 二十一、验收方式

完成本节后，建议用下面的清单验收：

```text
能批量构造订单事件。
每条事件都有 event_id 和 order_id。
order_id 被用作 message key。
发送成功日志包含 topic、partition、offset。
发送失败日志包含 error 和 event_id。
能通过命令行 consumer 看到消息内容。
能解释失败后是否允许重试。
```

如果只看到了 consumer 输出，却没有记录 key、partition 和失败策略，这个实践还不够工程化。
