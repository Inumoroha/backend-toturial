# 6. 综合排障：从域名到目标 IP 的网络层检查

本节目标：把 DNS 解析后的 IP、路由、网关、ARP、ICMP、端口测试串起来，形成一个后端接口访问失败时的基础排障流程。

---

## 一、排障场景

假设你的 Go 服务要访问：

```text
https://api.example.com
```

现在请求失败或超时。不要一开始就改代码，先确认网络层路径是否成立。

---

## 二、第一步：解析域名

```bash
dig +short api.example.com
```

如果没有结果，问题可能在 DNS。

如果有多个 IP，记录其中一个继续检查：

```text
目标IP = x.x.x.x
```

---

## 三、第二步：查看路由

```bash
ip route get 目标IP
```

看输出中：

- 走哪个网卡。
- 下一跳是谁。
- 源 IP 是什么。

如果提示没有路由，需要检查网络配置、VPN、容器网络或云路由表。

---

## 四、第三步：检查网关或邻居

如果目标在同网段：

```bash
ip neigh | grep 目标IP
```

如果目标不在同网段，检查网关：

```bash
ip neigh | grep 网关IP
```

邻居状态异常时，可能是二层网络或网关问题。

---

## 五、第四步：检查 ICMP

```bash
ping 目标IP
```

注意：ping 失败不一定代表 TCP 不通，但 ping 成功可以提供网络层往返线索。

---

## 六、第五步：检查 TCP 端口

HTTPS：

```bash
nc -vz 目标IP 443
```

HTTP：

```bash
nc -vz 目标IP 80
```

如果端口连接失败，再用：

```bash
curl -v https://api.example.com
```

观察卡在哪一步。

---

## 七、第六步：抓包证明

抓目标 IP：

```bash
sudo tcpdump -i any host 目标IP -nn
```

再发起请求：

```bash
curl -v https://api.example.com
```

观察：

- SYN 是否发出。
- 是否收到 SYN-ACK。
- 是否收到 RST。
- 是否只有发包没有回包。

抓包可以把“我感觉网络有问题”变成证据。

---

## 八、Go 服务中的排障落点

如果 Go 程序请求失败，可以同时加日志记录阶段耗时：

```text
DNS 解析耗时
TCP 连接耗时
TLS 握手耗时
等待响应头耗时
读取响应体耗时
```

后面 HTTP Client 阶段会详细讲如何配置这些超时。

---

## 九、常见判断

| 现象 | 可能原因 |
| --- | --- |
| DNS 无结果 | DNS 配置、域名错误 |
| 无路由 | 路由表、VPN、容器网络 |
| ping 通但端口不通 | 服务未监听、防火墙、安全组 |
| TCP SYN 发出无响应 | 防火墙丢弃、网络路径问题 |
| 收到 RST | 端口关闭或服务主动拒绝 |
| TCP 成功但 HTTPS 失败 | TLS、证书、SNI |

---

## 补充模板：网络层排障记录表

建议每次排障都按表格记录：

| 检查项 | 命令 | 结果 | 判断 |
| --- | --- | --- | --- |
| DNS | `dig 域名` | 解析到哪个 IP | 是否符合预期 |
| 路由 | `ip route get IP` | dev/src/via | 出口是否正确 |
| ARP | `ip neigh` | 网关 MAC | 下一跳是否可达 |
| ICMP | `ping IP` | 成功/失败 | 仅作辅助 |
| 路径 | `traceroute 域名` | 第几跳异常 | 只作线索 |

最终结论不要写成：

```text
网络不通。
```

而要写成：

```text
DNS 解析正常，路由出口为 eth0，默认网关 ARP 正常；ping 被禁但 TCP 443 可连接，因此网络层基本可达，下一步排查 TLS/HTTP。
```

这才是可复核的排障结论。

## 十、常见问题

### 1. 只用 curl -v 能不能完成全部排障？

不能。`curl -v` 很有用，但它无法替代 `ip route get`、`ss` 和 `tcpdump` 提供的网络层证据。

### 2. 服务端没有日志一定是网络问题吗？

不一定，也可能是请求被网关拦截、路由没匹配、日志采集异常。但首先应确认请求是否到达服务端。

### 3. 抓包看到 SYN 发出但没有回包说明什么？

常见是路径中丢包、防火墙丢弃、目标无响应或回程路径异常。

---

## 十一、本节达标标准

学完本节后，你应该能够做到：

- 按 DNS、路由、邻居、ICMP、TCP 端口、抓包的顺序检查网络问题。
- 使用 `ip route get` 和 `tcpdump` 提供证据。
- 区分网络层不可达、端口不可达和应用层失败。
- 形成自己的接口访问失败排障模板。
