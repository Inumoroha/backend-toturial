# 2. 路由表、默认网关与 traceroute

本节目标：学会查看路由表，理解默认网关的作用，并使用 `ip route get` 和 `traceroute` 判断访问目标的路径。

---

## 一、路由表解决什么问题

当主机要访问一个目标 IP 时，需要决定：

```text
这个包应该从哪个网卡发出？
如果目标不在本地网络，下一跳交给谁？
```

这个决策依据就是路由表。

查看路由表：

```bash
ip route
```

典型输出：

```text
default via 192.168.1.1 dev eth0
192.168.1.0/24 dev eth0 proto kernel scope link src 192.168.1.10
```

---

## 二、读懂路由表

这一行：

```text
192.168.1.0/24 dev eth0 proto kernel scope link src 192.168.1.10
```

表示：

- `192.168.1.0/24` 是本地直连网段。
- 访问这个网段走 `eth0`。
- 源地址使用 `192.168.1.10`。

这一行：

```text
default via 192.168.1.1 dev eth0
```

表示：

- 其他不知道怎么走的目标，交给默认网关 `192.168.1.1`。
- 通过 `eth0` 发出。

---

## 三、默认网关是什么

默认网关就是“本机去其他网络的默认出口”。

如果目标不在本地网段，主机会把包交给默认网关。

例如访问：

```bash
curl https://example.com
```

目标 IP 大概率不在你的局域网，本机会把包发给默认网关，再由网关继续转发。

---

## 四、实验：查看访问目标走哪里

执行：

```bash
ip route get 8.8.8.8
```

可能输出：

```text
8.8.8.8 via 192.168.1.1 dev eth0 src 192.168.1.10
```

这表示：

- 访问 `8.8.8.8`。
- 下一跳是 `192.168.1.1`。
- 走 `eth0`。
- 源地址是 `192.168.1.10`。

这是排查网络问题非常实用的命令。

---

## 五、traceroute 是什么

`traceroute` 用来观察到目标的大致路径。

```bash
traceroute example.com
```

Windows：

```powershell
tracert example.com
```

它会显示请求经过的多个中间节点。

注意：

- 某些节点不响应，所以会显示 `*`。
- 路径可能不完全准确。
- 回程路径可能和去程不同。
- 它适合提供线索，不是绝对证据。

## 六、后端排障中的用法

### 1. 服务访问数据库超时

先查路由：

```bash
ip route get 数据库IP
```

再测端口：

```bash
nc -vz 数据库IP 5432
```

如果没有路由，Go 代码再正确也连不上。

### 2. 容器内访问外网失败

进入容器：

```bash
docker exec -it 容器ID sh
ip route
```

看有没有默认路由。

### 3. 跨地域接口变慢

使用：

```bash
traceroute 目标域名
```

观察是否路径绕远或某一跳延迟异常。

## 补充实验：多目标对比路由选择

只看一个目标不够。建议连续观察三个目标：

```bash
ip route get 127.0.0.1
ip route get 192.168.1.1
ip route get 8.8.8.8
```

你会看到三类路径：

```text
127.0.0.1 通常走 lo。
同网段网关通常直接走本地网卡。
公网 IP 通常 via 默认网关。
```

记录时按这个格式写：

```text
目标：
出口网卡：
源 IP：
下一跳：
是否符合预期：
```

如果一台机器有多个网卡、VPN、Docker bridge 或 WSL2，`ip route get` 特别有用。它能告诉你系统真实选择的出口，而不是你脑子里以为的出口。

## 补充排障：traceroute 结果怎么写进结论

不要只截图 `traceroute`。建议写成：

```text
第 1 跳：本机默认网关，延迟 1ms。
第 2-5 跳：运营商网络，延迟逐步升高。
第 6 跳开始出现 *，但 curl 仍然成功。
结论：星号节点不响应探测报文，不能单独证明业务不通。
```

如果 `traceroute` 在第一跳就失败，优先看本机网关、VPN、路由表。

如果最后几跳失败，但 `curl` 业务也失败，再结合 `tcpdump` 和端口测试判断是否中间链路或目标侧过滤。

## 七、常见问题

### 1. 没有 default 路由会怎样？

通常只能访问本地直连网段，无法访问外网或其他未知网段。

### 2. traceroute 有星号是不是网络不通？

不一定。有些路由器不回复 traceroute 使用的探测报文，但仍然正常转发业务流量。

### 3. ping 通是否代表路由没问题？

只能说明 ICMP 层面有响应，不代表 TCP 端口、TLS、HTTP 都正常。

## 八、本节达标标准

学完本节后，你应该能够做到：

- 看懂 `ip route` 中的默认路由。
- 使用 `ip route get` 判断目标下一跳。
- 使用 `traceroute` 或 `tracert` 观察路径。
- 解释默认网关在跨网段通信中的作用。
- 在服务访问失败时先确认路由是否存在。
