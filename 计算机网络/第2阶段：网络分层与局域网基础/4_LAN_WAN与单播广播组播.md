# 4. LAN、WAN 与单播、广播、组播

本节目标：理解局域网与广域网的边界，区分单播、广播、组播的接收范围，并能判断数据应该直接交给目标主机还是先交给默认网关。

---

## 一、LAN 与 WAN 不是两套协议

局域网（LAN）通常覆盖家庭、办公室或一个园区，终端主要通过交换机互联。同一广播域内，主机可以使用 ARP 查询下一跳的 MAC 地址。

广域网（WAN）连接不同地点、不同组织或不同运营商网络，数据需要经过路由器逐跳转发。同一个 IP 包可以先经过源主机所在的 LAN，再穿过多个 WAN 链路，最后进入目标侧 LAN。

两者的核心差异不是使用不同的 TCP/IP 协议，而是覆盖范围、管理边界、链路类型、延迟和故障责任不同。

| 对比项 | LAN | WAN |
| --- | --- | --- |
| 常见范围 | 家庭、办公室、园区 | 城市、跨地域、互联网 |
| 常见设备 | 交换机、无线 AP | 路由器、运营商链路 |
| 常见延迟 | 较低且稳定 | 更高，受距离和路径影响 |
| 故障排查 | VLAN、ARP、交换端口 | 路由、NAT、运营商、跨地域链路 |

## 二、单播、广播与组播

| 发送方式 | 接收范围 | 常见场景 | 工程注意点 |
| --- | --- | --- | --- |
| 单播 | 一个明确目标 | HTTP、数据库、RPC | 后端服务最常见的通信方式 |
| 广播 | 同一二层广播域内所有设备 | ARP 请求、DHCP 发现 | 路由器通常不转发，不能跨网段发现服务 |
| 组播 | 主动加入某个组的设备 | 流媒体、服务发现、路由协议 | 依赖交换机、路由器和云网络支持 |

广播不是“向整个互联网发送”。ARP 请求的目标 MAC 是广播地址，但它的范围被限制在当前二层广播域。VLAN 和路由器都可以划分广播域。

组播介于单播和广播之间。发送方只发送一份数据，网络按需复制给组成员。它适合一对多分发，但云网络、容器网络和公网不一定支持，工程选型前必须验证网络环境。

## 三、同网段与跨网段通信

主机发送数据前，会用本机前缀长度分别计算源 IP 和目标 IP 的网络号：

```text
网络号相同：ARP 查询目标主机的 MAC，直接在本地链路发送。
网络号不同：ARP 查询默认网关的 MAC，先把帧交给网关。
```

跨路由转发时需要区分两层地址：

- 以太网帧的源、目标 MAC 会在每一段链路上改变。
- IP 包的源、目标 IP 通常端到端保持不变；经过 NAT 时才可能被改写。
- TCP/UDP 端口用于标识主机上的进程，不参与二层下一跳选择。

## 四、验证实验

先查看地址、前缀和默认网关：

```bash
ip addr
ip route
ip neigh
```

Windows 可以使用：

```powershell
ipconfig /all
route print
arp -a
```

分别访问一个同网段地址和一个公网地址，然后再次查看邻居表。你应该看到同网段目标本身的 MAC，以及跨网段通信使用的默认网关 MAC，而不会看到公网目标的 MAC。

在 Linux 或 WSL 可抓取 ARP 与 ICMP：

```bash
sudo tcpdump -i any -nn 'arp or icmp'
```

实验时记录：ARP 请求询问谁、响应来自谁、ping 公网地址前是否先解析网关 MAC，以及路由表选择了哪个出口。

## 五、常见问题

### 1. 交换机只工作在二层吗？

传统交换机主要按 MAC 地址转发二层帧；三层交换机还具备路由能力。判断设备作用时看它实际执行的是二层交换还是三层路由，不要只看产品名称。

### 2. 广播多了会怎样？

广播会被广播域内所有设备接收和处理。广播域过大或发生广播风暴时，会浪费链路和终端资源，因此网络常用 VLAN 和路由边界控制范围。

### 3. 为什么访问公网前看不到公网服务器的 MAC？

MAC 地址只在当前链路有效。主机只需要知道下一跳网关的 MAC，后续每台路由器会为下一段链路重新封装帧。
