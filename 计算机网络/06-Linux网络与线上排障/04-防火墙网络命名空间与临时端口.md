# 04-防火墙、网络命名空间与临时端口

## 学习目标

补齐 Linux 网络排障中容易被忽略但非常常见的部分：防火墙、安全组、iptables/nftables、网络命名空间、临时端口、文件描述符。它们经常决定服务“明明启动了为什么访问不到”。

## 一、防火墙和安全组

后端访问链路可能同时经过多层过滤：

```text
客户端
  -> 本机防火墙
  -> 公司网络或运营商
  -> 云安全组
  -> 服务器防火墙
  -> Docker/Kubernetes 转发规则
  -> Go 服务
```

所以端口不通时，不能只看 Go 程序是否监听。

## 二、iptables 与 nftables

iptables/nftables 是 Linux 上常见的包过滤与转发机制。

查看 iptables：

```bash
sudo iptables -L -n -v
sudo iptables -t nat -L -n -v
```

查看 nftables：

```bash
sudo nft list ruleset
```

说明：

- `filter` 表常用于允许或拒绝流量。
- `nat` 表常用于地址转换和端口转发。
- Docker 会自动写入一些 iptables 规则。

## 三、云安全组

云服务器常见访问失败原因：

- Go 服务监听正常。
- 服务器本机 curl 正常。
- 公网访问失败。
- 原因是云安全组没有开放端口。

排查顺序：

1. 本机服务是否监听：

   ```bash
   ss -lntp | grep 8080
   ```

2. 服务器本机访问是否成功：

   ```bash
   curl http://127.0.0.1:8080
   ```

3. 服务器内网 IP 访问是否成功：

   ```bash
   curl http://内网IP:8080
   ```

4. 外部公网访问是否成功。
5. 检查云安全组和系统防火墙。

## 四、网络命名空间

Linux 网络命名空间让不同进程看到不同的网络环境。

Docker 容器就是典型例子：

- 容器有自己的网卡。
- 容器有自己的 IP。
- 容器有自己的路由表。
- 容器内的 `127.0.0.1` 是容器自己，不是宿主机。

这句话非常重要：

```text
容器内的 localhost 不等于宿主机 localhost。
```

## 五、实验：查看容器网络命名空间

启动容器：

```bash
docker run --rm -it --name net-demo alpine sh
```

容器内：

```sh
ip addr
ip route
cat /etc/resolv.conf
```

宿主机：

```bash
docker inspect net-demo
docker network inspect bridge
```

观察：

- 容器 IP。
- 默认网关。
- DNS 配置。
- bridge 网络。

## 六、临时端口

客户端发起 TCP 连接时，会从本机临时端口范围中选择一个源端口。

查看范围：

```bash
cat /proc/sys/net/ipv4/ip_local_port_range
```

典型输出：

```text
32768 60999
```

这意味着可用临时端口数量有限。

## 七、临时端口耗尽

如果程序大量创建短连接，可能导致临时端口耗尽。

表现：

- 请求失败。
- 连接建立异常。
- 大量 TIME_WAIT。
- 到同一个目标的并发连接受限。

排查：

```bash
ss -ant state time-wait | wc -l
ss -ant | awk '{print $1}' | sort | uniq -c
cat /proc/sys/net/ipv4/ip_local_port_range
```

解决方向：

- 复用连接池。
- 减少短连接。
- 调整客户端并发。
- 合理扩大端口范围。
- 避免不必要的主动关闭。

## 八、文件描述符

Linux 中 socket 也是文件描述符。

查看当前 shell 限制：

```bash
ulimit -n
```

查看进程限制：

```bash
cat /proc/进程ID/limits
```

查看进程打开的 fd 数：

```bash
ls /proc/进程ID/fd | wc -l
```

如果连接泄漏，可能出现：

```text
too many open files
```

## 九、Go 后端关联

Go 程序中导致 fd 增长的常见原因：

- TCP 连接未关闭。
- HTTP 响应体未关闭。
- 文件未关闭。
- WebSocket 长连接没有清理。
- goroutine 泄漏导致连接无法释放。

HTTP Client 错误示例：

```go
resp, err := client.Get(url)
if err != nil {
    return err
}
// 忘记关闭 resp.Body
```

修复：

```go
resp, err := client.Get(url)
if err != nil {
    return err
}
defer resp.Body.Close()
io.Copy(io.Discard, resp.Body)
```

## 十、完整排障案例：公网访问不到 Go 服务

现象：

```text
本机启动 Go 服务后，公网 curl 超时。
```

排查路径：

1. 服务是否启动：

   ```bash
   ps aux | grep server
   ```

2. 端口是否监听：

   ```bash
   ss -lntp | grep 8080
   ```

3. 监听地址是否是 `0.0.0.0`：

   ```text
   LISTEN 0 4096 0.0.0.0:8080
   ```

4. 本机访问：

   ```bash
   curl http://127.0.0.1:8080
   ```

5. 内网 IP 访问：

   ```bash
   curl http://服务器内网IP:8080
   ```

6. 防火墙：

   ```bash
   sudo iptables -L -n -v
   ```

7. 云安全组是否放行 `8080`。

8. 外部机器测试：

   ```bash
   curl -v http://公网IP:8080
   ```

## 十一、常见误区

- `EXPOSE 8080` 不等于 Docker 发布端口。
- 容器内 `127.0.0.1` 不是宿主机。
- `ping` 通不代表端口通。
- 服务监听 `127.0.0.1` 时，外部机器无法访问。
- 文件描述符上限不是只影响文件，也影响网络连接。

## 十二、练习题

1. 云服务器本机 curl 成功、公网失败，优先排查什么？
2. 容器内访问 `127.0.0.1` 访问的是哪里？
3. 临时端口耗尽通常和什么连接模式有关？
4. `too many open files` 和网络连接有什么关系？
5. Docker 的 `EXPOSE` 和 `-p` 有什么区别？

## 十三、验收标准

你能解释防火墙、安全组、网络命名空间、临时端口、文件描述符如何影响 Go 服务访问，并能按步骤排查“服务启动但外部访问不到”的问题。
