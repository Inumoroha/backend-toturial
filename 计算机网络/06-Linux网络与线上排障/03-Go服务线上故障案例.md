# 03-Go 服务线上故障案例

## 学习目标

通过典型故障案例，把网络原理、Go 编码习惯和 Linux 排障命令串起来。

## 案例一：接口偶发超时

### 现象

- 用户偶发请求超时。
- 服务端业务日志没有对应请求。
- 重启服务后短暂恢复。

### 可能原因

- DNS 慢或失败。
- 连接池耗尽。
- 下游服务连接建立慢。
- 文件描述符耗尽。
- goroutine 堆积。

### 排查

```bash
dig 下游域名
ss -antp | grep 下游端口
ulimit -n
lsof -p 进程ID | wc -l
```

Go 侧检查：

- 是否复用 `http.Client`。
- 是否关闭 `resp.Body`。
- 是否设置超时。
- 是否记录下游调用耗时。

## 案例二：大量 CLOSE_WAIT

### 现象

```bash
ss -ant state close-wait | wc -l
```

数量持续增长。

### 常见原因

对方已经关闭连接，本地程序没有关闭 socket。

Go 中常见代码问题：

```go
resp, err := http.Get(url)
if err != nil {
    return err
}
// 忘记 resp.Body.Close()
```

### 修复

```go
resp, err := http.Get(url)
if err != nil {
    return err
}
defer resp.Body.Close()
io.Copy(io.Discard, resp.Body)
```

## 案例三：Nginx 504

### 现象

客户端收到 504 Gateway Timeout。

### 含义

网关连接到了上游，但等待上游响应超时。

### 排查

- Nginx access log 中 upstream_response_time。
- Go 服务是否收到请求。
- Go handler 是否耗时过长。
- 数据库或下游 RPC 是否慢。
- 网关超时配置是否过短。

命令：

```bash
curl -v http://网关地址/api
ss -antp | grep 上游端口
```

## 案例四：Connection refused

### 现象

```text
connect: connection refused
```

### 含义

目标机器可达，但目标端口没有服务监听，或主动拒绝连接。

### 排查

```bash
ss -lntp | grep 8080
lsof -i :8080
curl -v http://127.0.0.1:8080
```

如果在容器环境：

```bash
docker ps
docker port 容器ID
docker logs 容器ID
```

## 案例五：服务启动成功但外部访问不到

### 常见原因

- 只监听 `127.0.0.1`。
- 云安全组未开放端口。
- Linux 防火墙拦截。
- Docker 没有端口映射。
- Nginx upstream 地址配置错误。

### 排查顺序

1. 本机 curl 是否成功。
2. 局域网机器是否成功。
3. 公网是否成功。
4. 端口是否监听所有网卡。
5. 防火墙和安全组是否放行。

## 六、练习题

3. 504 和 502 的排查重点有什么不同？
4. 外部访问不到服务时，为什么要先查监听地址？
