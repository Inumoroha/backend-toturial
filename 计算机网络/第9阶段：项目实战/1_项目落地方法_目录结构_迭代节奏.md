# 1. 项目落地方法：目录结构、迭代节奏与交付标准

本节目标：掌握四个网络项目通用的目录组织、五轮迭代、测试方法、复盘方式和交付标准。

把前面四个项目从“知道要做什么”推进到“知道如何一步一步做”。这一篇给出统一项目目录、迭代节奏、提交标准、测试方式和复盘方法。

## 一、为什么项目不能一口吃完

网络项目很容易一开始就想做全：

- 聊天室想做账号、房间、私聊、心跳、历史消息。
- 反向代理想做负载均衡、健康检查、限流、配置热更新。
- API 网关想做鉴权、熔断、指标、灰度。
- gRPC 服务想做 TLS、拦截器、服务发现。

这样容易卡住。正确方式是先做一条最小可用链路，再逐步加能力。

## 二、统一目录结构

建议每个项目都使用类似结构：

```text
project-name/
  README.md
  go.mod
  cmd/
    server/
      main.go
    client/
      main.go
  internal/
    app/
    config/
    transport/
    service/
  testdata/
  scripts/
```

说明：

- `cmd/server`：服务端入口。
- `cmd/client`：客户端或测试入口。
- `internal`：项目内部代码。
- `testdata`：测试数据、配置样例。
- `scripts`：启动、压测、抓包脚本。

小项目不必过度分层，但入口、核心逻辑、配置、测试数据应该清楚。

## 三、每个项目的 README 必须写什么

每个项目至少包含：

```markdown
# 项目名称

## 目标

这个项目用于练习什么网络能力。

## 功能

- 已完成功能
- 未完成功能

## 运行方式

启动服务端：

```bash
go run ./cmd/server
```

启动客户端：

```bash
go run ./cmd/client
```

## 测试方式

```bash
curl ...
nc ...
grpcurl ...
```

## 核心设计

连接如何管理、请求如何转发、超时如何设置。

## 排障记录

开发中遇到的问题和解决方法。
```

## 四、迭代节奏

每个项目按 5 个迭代推进。

### 第 1 轮：跑通最小链路

目标：先让请求从客户端到服务端，再返回。

不要做复杂抽象。只验证：

- 服务能启动。
- 客户端能连上。
- 有最简单响应。

### 第 2 轮：补协议和错误处理

目标：输入输出可控。

补充：

- 命令格式。
- HTTP 状态码。
- gRPC status code。
- 错误日志。

### 第 3 轮：补超时和资源清理

目标：服务不会轻易挂死。

补充：

- ReadTimeout。
- WriteTimeout。
- context timeout。
- 连接关闭。
- goroutine 退出。

### 第 4 轮：补并发和可靠性

目标：多个客户端或多个请求同时工作。

补充：

- 锁。
- channel。
- 连接池。
- 限流。
- 健康检查。

### 第 5 轮：补观测和文档

目标：别人能运行、你能排障。

补充：

- README。
- 请求日志。
- 指标。
- 测试命令。
- 常见问题。

## 五、项目一：TCP 聊天室迭代拆分

### 1. 最小版本

- TCP Server 监听 `:9000`。
- 客户端用 `nc` 连接。
- 服务端收到一行，原样返回。

验收：

```bash
nc 127.0.0.1 9000
```

输入 `hello`，收到 `echo: hello`。

### 2. 多客户端版本

- 每个连接一个 goroutine。
- 服务端维护 `clients` map。
- 用户登录后加入在线列表。

验收：

- 两个终端连接。
- A 发消息，B 能收到。

### 3. 健壮版本

- 读超时。
- 写超时。
- 用户退出清理。
- 异常断开清理。

验收：

```bash
ss -antp | grep 9000
```

用户退出后连接不继续残留。

### 4. 进阶版本

- 私聊。
- 在线用户列表。
- 心跳。
- 慢客户端踢出。

## 六、项目二：HTTP 反向代理迭代拆分

### 1. 最小版本

- 使用 `httputil.NewSingleHostReverseProxy`。
- 所有请求转发到一个后端。

验收：

```bash
curl -v http://127.0.0.1:8080
```

能看到后端响应。

### 2. 日志版本

- 记录 method、path、status、cost。
- 生成 request id。

验收：

请求一次，日志能查到完整链路。

### 3. 超时版本

- 配置 Server timeout。
- 配置 Transport timeout。
- 上游慢响应返回 504。

验收：

后端 sleep 5 秒，代理 1 秒超时并返回 504。

### 4. 多上游版本

- 配置多个 upstream。
- 轮询负载均衡。
- 上游失败返回 502。

## 七、项目三：迷你 API 网关迭代拆分

### 1. 路由版本

- `/users` 转发到用户服务。
- `/orders` 转发到订单服务。
- 未匹配返回 404。

### 2. 配置版本

- 从 YAML/JSON 加载路由。
- 支持多个 upstream。

### 3. 治理版本

- 限流返回 429。
- 上游失败返回 502。
- 上游超时返回 504。

### 4. 观测版本

- 记录 request id。
- 记录 upstream。
- 记录 status 和 cost。
- 暴露 `/health`。

## 八、项目四：gRPC 用户服务迭代拆分

### 1. proto 版本

- 定义 `UserService`。
- 生成 Go 代码。

### 2. Server 版本

- 实现 `GetUser`。
- 使用内存 map 存储用户。
- 找不到返回 `NotFound`。

### 3. Client 版本

- 设置 timeout。
- 处理 status code。

### 4. 工程版本

- 加日志拦截器。
- 加健康检查。
- 用 grpcurl 测试。

## 九、每个项目的测试清单

### 功能测试

- 正常请求成功。
- 错误输入返回合理错误。
- 服务停止时客户端能看到明确错误。

### 超时测试

- 慢请求能超时。
- 超时后资源能释放。

### 并发测试

```bash
hey -n 1000 -c 50 http://127.0.0.1:8080
```

或多个 `nc` 客户端同时连接。

### 连接检查

```bash
ss -antp | grep 端口
```

观察是否有异常堆积。

### 抓包检查

```bash
sudo tcpdump -i any port 端口 -nn
```

观察连接建立和关闭。

## 十、复盘模板

每完成一个项目，写一份复盘：

```markdown
# 项目复盘

## 我实现了什么

## 我遇到的问题

## 我如何定位问题

## 哪些网络知识用上了

## 哪些 Go API 用上了

## 如果上生产还缺什么
```

## 十一、交付标准

一个项目不是“代码能跑”就算完成。至少要满足：

- README 能让别人运行。
- 有明确启动命令。
- 有测试命令。
- 有错误处理。
- 有超时控制。
- 有资源清理。
- 能用 `ss` 或日志观察运行状态。
- 能解释核心设计。

## 十二、本节达标标准

你能把四个项目都拆成可执行迭代，并知道每一轮应该做什么、如何测试、如何证明完成。
