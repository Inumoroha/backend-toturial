# 7. 项目验收、排障与复盘清单

本节目标：为所有项目建立统一验收标准，确保项目不只是“能跑”，而是可测试、可排障、可讲清楚。

---

## 一、每个项目必须有 README

README 至少包含：

```text
项目目标。
功能清单。
运行方式。
测试命令。
核心设计。
已知问题。
排障记录。
```

---

## 二、功能验收

每个项目要验证：

- 正常请求成功。
- 错误输入有合理响应。
- 服务停止时错误可理解。
- 超时能触发。
- 资源能释放。

---

## 三、网络验收

使用：

```bash
ss -lntp
ss -antp
curl -v
tcpdump
```

证明：

- 服务监听正确端口。
- 请求能到达。
- 连接能关闭。
- 错误能定位。

---

## 四、复盘模板

```markdown
# 项目复盘

## 实现了什么

## 用到了哪些网络知识

## 遇到什么问题

## 如何排查

## 如果上线还缺什么
```

---

## 补充验收方式：每个项目都要能从零启动

项目验收不要只看你本机当前进程能跑。应该新开一个目录或清理环境后，按 README 从零执行：

```text
安装依赖。
启动后端。
启动代理或网关。
执行 curl 测试。
观察日志。
关闭服务。
```

如果 README 缺一步，你未来自己也会忘。

---

## 补充复盘问题：把项目变成面试材料

每个项目完成后，写一段复盘：

```text
这个项目解决什么问题？
核心网络知识是什么？
最容易出错的地方是什么？
如何用命令排查？
如果上线，需要补哪些能力？
```

例如 TCP 聊天室可以总结：

```text
TCP 没有消息边界，所以使用换行或长度前缀协议。
每个连接需要 readLoop/writeLoop。
慢客户端不能阻塞广播。
断开连接必须清理在线表。
```

这段复盘会直接变成你的面试表达。

---

## 补充检查：项目提交前的最后一遍自测

提交前按顺序执行：

```text
清理旧进程。
重新启动服务。
执行 README 中所有命令。
故意访问错误端口。
故意传错误参数。
观察日志是否能定位问题。
关闭服务并确认端口释放。
```

如果这些步骤都能通过，项目才算真正可复现。

项目不是“我这里能跑”就结束，而是“别人照着 README 也能跑”。

---

## 补充交付标准：项目必须留下证据

最终提交时建议保留：

```text
README 启动命令。
curl 测试命令。
关键日志样例。
已知限制。
下一步计划。
```

例如：

```text
已知限制：聊天室消息未持久化；反向代理没有主动健康检查；API 网关限流是内存版。
```

能主动写出限制，说明你真正理解项目边界。

---

## 补充要求：复盘要能指导下一次行动

复盘最后要落到：

```text
下一次先改什么。
为什么先改它。
怎么验证改好了。
```

---

## 五、常见问题

### 1. 项目能跑就算完成吗？

不算。还要有 README、测试命令、错误处理、超时控制和排障记录。

### 2. 为什么项目复盘要写“如果上线还缺什么”？

这能训练工程意识，让你区分学习 Demo 和生产服务之间的差距。

### 3. 抓包和 ss 是否每个项目都要用？

建议都用一次。项目实战的目标之一就是把协议和排障工具连起来。
