# 7. TCP、UDP 综合对比与后端选型

本节目标：把 TCP 和 UDP 放回后端工程场景中，理解什么时候用 TCP，什么时候关注 UDP，什么时候选择 HTTP、WebSocket、gRPC 或 QUIC。

---

## 一、后端默认选择

大多数 Go 后端业务接口默认使用：

```text
HTTP/HTTPS
gRPC
数据库连接
Redis 连接
消息队列客户端
```

它们大多基于 TCP。

原因是后端业务通常更重视：

- 可靠性。
- 有序性。
- 生态成熟。
- 易排障。

---

## 二、什么时候关注 UDP

UDP 常见于：

- DNS。
- QUIC / HTTP/3。
- 音视频。
- 游戏。
- 实时指标上报。
- 某些服务发现协议。

如果你不是做这些场景，可能不会直接写 UDP 服务，但理解 UDP 有助于理解 DNS 和 HTTP/3。

---

## 三、协议选择表

| 场景 | 推荐 |
| --- | --- |
| 对外 REST API | HTTP/HTTPS |
| 内部服务强类型调用 | gRPC |
| 浏览器实时聊天 | WebSocket |
| 简单后台管理接口 | HTTP JSON |
| DNS 查询 | UDP/TCP 53 |
| 低延迟音视频 | UDP/QUIC/RTP |
| 自定义长连接协议 | TCP |

---

## 四、Go 后端选型直觉

如果你只是做普通业务系统：

```text
先用 HTTP JSON。
服务间调用复杂后考虑 gRPC。
需要服务端主动推送浏览器，考虑 WebSocket。
不要一开始就手写 TCP 协议。
```

手写 TCP 协议适合学习和少数特定场景，不是普通业务系统的默认选择。

---

## 补充方法：用约束条件做协议选型

协议选型不要从“哪个更高级”开始，而要从约束条件开始。

可以按下面清单问自己：

```text
是否需要浏览器直接访问？
是否需要强可靠和有序？
是否需要服务端主动推送？
是否需要跨语言强契约？
是否能接受连接长期存在？
是否有网关、负载均衡、审计、TLS 终止要求？
是否需要穿过公司代理、防火墙或公网环境？
团队是否有成熟运维经验？
```

把答案映射到常见选择：

```text
浏览器普通 API：HTTP/HTTPS。
浏览器实时双向通信：WebSocket。
内部强契约服务调用：gRPC。
自定义长连接：TCP。
低延迟、可容忍丢包：UDP/QUIC。
```

---

## 补充案例：四个后端场景怎么选

### 1. 用户登录接口

推荐：

```text
HTTPS + JSON
```

原因：

```text
浏览器和移动端支持最好。
容易接入网关、WAF、日志、审计。
请求响应模型简单。
```

### 2. 订单服务调用库存服务

推荐：

```text
HTTP JSON 或 gRPC
```

如果团队已经有 proto 管理和服务治理体系，gRPC 很合适。如果团队还在早期，HTTP JSON 更容易排障。

### 3. 在线客服聊天

推荐：

```text
WebSocket
```

原因：

```text
浏览器支持。
服务端能主动推送。
比轮询更自然。
```

### 4. 游戏位置同步

可能选择：

```text
UDP 或基于 UDP 的协议
```

原因：

```text
旧位置包过期后价值很低。
低延迟比每个包必达更重要。
```

这类场景通常要自己处理序号、丢包、插值、预测和状态同步。

---

## 补充练习：写一份协议选型说明

以后你在项目里选择协议时，不要只写：

```text
我们使用 gRPC，因为性能好。
```

这句话太粗。更好的说明应该包含：

```text
业务场景：订单服务调用库存服务。
调用方向：服务端到服务端。
可靠性要求：必须可靠返回，失败要有明确错误。
数据格式：结构固定，字段可演进。
调用频率：高频内部调用。
客户端类型：Go 服务为主，未来可能有 Java 服务。
网关要求：不直接暴露给浏览器。
排障方式：需要 request id、deadline、status code。
最终选择：gRPC。
备选方案：HTTP JSON。
不选择 UDP 的原因：业务不能接受丢包和乱序。
```

这样的选型说明能让别人看懂你的判断，而不是觉得你在追新技术。

## 五、常见问题

### 1. gRPC 是 TCP 还是 UDP？

常见 gRPC 基于 HTTP/2，HTTP/2 基于 TCP。

### 2. HTTP/3 是 TCP 吗？

不是。HTTP/3 基于 QUIC，QUIC 基于 UDP。

### 3. WebSocket 是 TCP 吗？

WebSocket 通常通过 HTTP Upgrade 建立，底层基于 TCP。

## 六、本节达标标准

学完本节后，你应该能够做到：

- 说清 TCP 和 UDP 的差异。
- 根据业务场景选择 HTTP、WebSocket、gRPC 或自定义 TCP。
- 解释为什么普通业务接口不建议一开始手写 TCP 协议。
- 说明 UDP 在 DNS 和 HTTP/3 中的价值。
