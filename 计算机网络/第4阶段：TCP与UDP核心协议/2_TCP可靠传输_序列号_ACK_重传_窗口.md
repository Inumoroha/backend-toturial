# 2. TCP 可靠传输：序列号、ACK、重传与窗口

本节目标：理解 TCP 为什么能提供可靠、有序的字节流传输，掌握序列号、确认号、重传、滑动窗口、流量控制和拥塞控制的基本直觉。

---

## 一、可靠不是不丢包

网络可能丢包、乱序、重复。TCP 的可靠性不是让网络不出问题，而是：

```text
发现问题。
重传数据。
按顺序交给应用。
控制发送速度。
```

---

## 二、序列号和确认号

TCP 把数据看成字节流，每个字节都有编号。

```text
序列号 seq：当前段中第一个字节的编号。
确认号 ack：我期望收到的下一个字节编号。
```

如果确认号是 `101`，表示：

```text
100 及之前的字节都收到了。
下一个请发 101。
```

---

## 三、重传

如果发送方没有按时收到 ACK，就会重传。

常见重传触发：

- 超时重传。
- 快速重传。

重传能保证数据最终尽量送达，但会增加延迟。网络丢包时，TCP 吞吐和响应时间都会变差。

---

## 四、滑动窗口

如果每发一点数据都等 ACK，效率很低。

滑动窗口允许发送方在未收到 ACK 前连续发送一定数量的数据。

你可以理解为：

```text
当前允许在网络中飞行的未确认数据量。
```

窗口越大，链路利用率越高，但也要考虑接收方和网络承受能力。

---

## 五、流量控制

流量控制保护接收方。

接收方会告诉发送方：

```text
我的接收缓冲区还剩多少空间。
```

如果接收方应用读取很慢，窗口变小，发送方就要慢下来。

Go 后端场景：

- 客户端下载大响应但读取很慢。
- 服务端写响应阻塞。
- WebSocket 慢消费者拖垮服务。

解决方向：

- 设置写超时。
- 限制响应大小。
- 慢客户端断开。

---

## 六、拥塞控制

拥塞控制保护网络。

即使接收方还能接收，中间网络也可能拥塞。TCP 会根据丢包、ACK、延迟等信号调整发送速率。

常见概念：

- 慢启动。
- 拥塞避免。
- 快速重传。
- 快速恢复。

后端直觉：

```text
跨地域 RTT 高，TCP 慢启动影响更明显。
丢包会导致吞吐下降和延迟升高。
连接复用可以减少新连接慢启动和握手成本。
```

---

## 七、实验：观察重传线索

真实重传不容易稳定制造。可以用 Wireshark 抓一段访问公网的流量，过滤：

```text
tcp.analysis.retransmission
```

如果出现重传，说明 Wireshark 判断有 TCP 段被重新发送。

不要为了制造丢包随意修改系统网络规则。初学阶段先能识别即可。

## 补充实验：用大响应观察序列号和 ACK

先写一个会返回较大响应的 Go 服务：

```go
package main

import (
    "bytes"
    "log"
    "net/http"
)

func main() {
    payload := bytes.Repeat([]byte("x"), 256*1024)

    http.HandleFunc("/big", func(w http.ResponseWriter, r *http.Request) {
        w.Header().Set("Content-Type", "text/plain")
        _, _ = w.Write(payload)
    })

    log.Println("listen on :8080")
    log.Fatal(http.ListenAndServe(":8080", nil))
}
```

启动：

```bash
go run .
```

抓包：

```bash
sudo tcpdump -i any -nn -S 'tcp port 8080'
```

请求：

```bash
curl -o /dev/null http://127.0.0.1:8080/big
```

阅读抓包时重点看三列：

```text
seq       当前发送的数据范围。
ack       我已经收到对方哪里之前的数据。
win       我还能接收多少数据。
```

简化示例：

```text
8080 > 53012: seq 1000:2460
53012 > 8080: ack 2460
8080 > 53012: seq 2460:3920
53012 > 8080: ack 3920
```

这表示：

```text
服务端先发 1000 到 2459 这些字节。
客户端确认已经收到 2460 之前的数据。
服务端继续从 2460 发下一段。
```

ACK 号表示“下一个期望收到的字节序号”，不是“我收到了这个编号的包”。

## 补充理解：窗口为什么会影响吞吐

TCP 不能无限制发送。接收方会通过窗口告诉发送方：

```text
我这边还有多少接收缓冲区。
```

如果接收方应用层读得慢，接收缓冲区会逐渐变满，窗口会变小。窗口变成 0 时，发送方就不能继续大量发送，只能等待窗口更新。

这和 Go 后端的关系很直接：

```text
如果客户端读取响应很慢，服务端写响应可能阻塞。
如果服务端读取请求体很慢，客户端发送请求体可能阻塞。
如果连接很多都被慢读慢写拖住，goroutine 和连接资源都会被占用。
```

所以网络排障里看到“服务端 CPU 不高，但请求慢”，不要只看业务代码，也要考虑 TCP 窗口、慢客户端和连接堆积。

## 八、常见问题

### 1. TCP 保证消息边界吗？

不保证。TCP 保证字节流可靠有序，不保证应用层消息边界。

### 2. ACK 是确认包还是确认字节？

确认的是字节序列位置，不是应用消息。

### 3. 拥塞控制和流量控制有什么区别？

流量控制保护接收方，拥塞控制保护网络。
