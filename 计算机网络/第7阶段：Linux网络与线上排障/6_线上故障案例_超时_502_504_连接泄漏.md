# 6. 线上故障案例：超时、502、504、连接泄漏

本节目标：通过典型案例串联网络命令、Go 代码和后端架构。

---

## 一、接口偶发超时

排查：

```bash
dig 下游域名
ip route get 下游IP
curl -v 下游URL
ss -antp | grep 下游端口
```

Go 检查：

- 是否设置超时。
- 是否复用 Client。
- 是否关闭 body。
- 是否有连接池耗尽。

---

## 二、502 Bad Gateway

常见含义：

```text
网关无法正常连接或获取上游响应。
```

排查：

- 上游是否启动。
- upstream 地址是否正确。
- 端口是否监听。
- Nginx 日志。

命令：

```bash
ss -lntp | grep 上游端口
curl -v http://上游地址
```

---

## 三、504 Gateway Timeout

常见含义：

```text
网关连接上游后，等待响应超时。
```

排查：

- 上游 handler 是否慢。
- 下游数据库/RPC 是否慢。
- 网关 timeout 是否合理。
- 是否连接池耗尽。

---

## 四、大量 CLOSE_WAIT

查看：

```bash
ss -ant state close-wait | wc -l
```

优先查：

```go
resp.Body.Close()
conn.Close()
```

---

## 五、too many open files

排查：

```bash
ulimit -n
cat /proc/进程ID/limits
ls /proc/进程ID/fd | wc -l
lsof -p 进程ID | head
```

原因可能是连接泄漏或文件泄漏。

## 补充案例：客户端超时但服务端没有日志

现象：

```text
Go client 报 context deadline exceeded。
目标服务没有 access log。
```

不要立刻查目标服务 handler。先证明请求有没有到达：

```bash
dig 目标域名
ip route get 目标IP
curl -v --connect-timeout 3 https://目标域名/healthz
sudo tcpdump -i any -nn host 目标IP and port 443
```

判断：

```text
DNS 失败：请求没进入 TCP。
只有 SYN 重传：网络或安全组问题。
TLS 报错：证书或 SNI 问题。
HTTP 有状态码：请求至少到了网关或服务。
```

服务端没有日志时，排查重点应先放在：

```text
DNS。
路由。
安全组。
负载均衡。
网关。
TLS。
```

## 补充案例：重启后恢复但过一会又坏

这种现象很像资源泄漏。

观察：

```bash
pid=进程ID
watch -n 2 "ls /proc/$pid/fd | wc -l"
watch -n 2 "ss -antp | grep $pid | wc -l"
```

同时看连接状态：

```bash
ss -ant state close-wait | wc -l
ss -ant state time-wait | wc -l
```

如果 fd、连接数、CLOSE_WAIT 持续增长，优先查：

```text
resp.Body.Close 是否遗漏。
数据库 rows.Close 是否遗漏。
TCP conn.Close 是否遗漏。
goroutine 是否卡在 channel 或锁上。
```

重启只是把资源暂时释放了，不代表根因消失。

## 补充复盘：故障案例要沉淀成检查项

每次线上网络故障结束后，至少补一条检查项：

```text
如果是 DNS：是否需要域名变更前降低 TTL？
如果是安全组：是否需要发布前自动校验端口？
如果是 502：是否需要 upstream 健康检查？
如果是 504：是否需要梳理超时预算？
如果是 CLOSE_WAIT：是否需要代码扫描 resp.Body.Close？
如果是 fd 耗尽：是否需要 fd/goroutine/连接数告警？
```

故障复盘的价值不是写一篇漂亮文档，而是让下一次同类问题更早被发现、更快被定位。

建议最终沉淀：

```text
一条监控。
一条告警。
一条发布检查。
一条代码规范。
一条应急命令。
```

## 六、常见问题

### 1. 504 一定是网关问题吗？

不一定。网关只是报错位置，根因可能是上游或上游的下游慢。

### 2. 服务端无日志但客户端超时怎么办？

先查 DNS、TCP、网关、抓包，确认请求是否到达。

### 3. 重启解决是否代表代码没问题？

不代表。连接泄漏、fd 泄漏重启后会短暂恢复，但根因仍在。
