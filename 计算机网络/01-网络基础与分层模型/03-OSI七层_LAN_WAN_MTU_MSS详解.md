# 03-OSI 七层、LAN/WAN、MTU/MSS 详解

## 学习目标

补齐网络基础中容易“听过但说不清”的概念：OSI 七层模型、TCP/IP 四层模型的对应关系，局域网和广域网的差异，MTU 与 MSS 为什么会影响后端服务性能。

## 一、OSI 七层模型是什么

OSI 七层模型是一种教学和分析模型，把网络通信拆成七层：

| OSI 层级 | 名称 | 解决的问题 | 常见例子 |
| --- | --- | --- | --- |
| 7 | 应用层 | 应用如何表达数据 | HTTP、DNS、SMTP |
| 6 | 表示层 | 数据格式、编码、加密 | JSON、TLS 的部分能力 |
| 5 | 会话层 | 会话建立、保持、恢复 | RPC 会话、登录态概念 |
| 4 | 传输层 | 进程到进程通信 | TCP、UDP |
| 3 | 网络层 | 主机到主机寻址和路由 | IP、ICMP |
| 2 | 数据链路层 | 同一链路内传输 | Ethernet、Wi-Fi、ARP |
| 1 | 物理层 | 比特如何在介质上传输 | 光纤、电缆、无线电 |

实际工程中更常用 TCP/IP 四层模型，但 OSI 七层适合用来描述问题发生在哪一类能力上。

## 二、OSI 与 TCP/IP 如何对应

| TCP/IP 四层 | 大致对应 OSI | 后端工程师关注点 |
| --- | --- | --- |
| 应用层 | OSI 5-7 层 | HTTP、DNS、TLS、gRPC、WebSocket |
| 传输层 | OSI 4 层 | TCP、UDP、端口、连接、超时 |
| 网络层 | OSI 3 层 | IP、路由、ICMP、NAT |
| 网络接口层 | OSI 1-2 层 | MAC、ARP、网卡、交换机、MTU |

不要把模型背成死知识。排障时它的价值是帮你定位：

```text
域名解析失败：应用层 DNS 问题
端口连接超时：可能是传输层、网络层或防火墙问题
ping 不通：可能是网络层或 ICMP 被禁
同网段机器访问异常：可能是 ARP、交换机、网卡问题
```

## 三、LAN 与 WAN

### LAN：局域网

LAN 是较小范围内的网络，例如家庭、办公室、机房内网。

特点：

- 延迟低。
- 带宽较高。
- 通常由交换机连接。
- 常使用私有 IP。

局域网内通信通常依赖 ARP 找到目标 MAC，然后通过交换机转发。

### WAN：广域网

WAN 是跨城市、跨地域、跨运营商的网络，例如公网。

特点：

- 路由跳数更多。
- 延迟更高。
- 更容易出现抖动和丢包。
- 经过多个路由器、运营商链路和安全设备。

后端工程里，跨地域调用比同机房调用更容易出现超时、抖动和尾延迟升高。

## 四、交换机与路由器

### 交换机

交换机主要工作在二层，根据 MAC 地址转发以太网帧。

它关心：

```text
这个 MAC 地址在哪个端口后面？
```

### 路由器

路由器主要工作在三层，根据 IP 地址和路由表转发 IP 包。

它关心：

```text
去这个目标网段，下一跳该走哪里？
```

简单记忆：

- 同网段通信主要经过交换机。
- 跨网段通信必须经过路由器或网关。

## 五、单播、广播、组播

### 单播

一对一通信。大多数后端请求都是单播：

```text
客户端 -> 服务端
```

### 广播

一对同一广播域内所有设备。ARP 请求就是广播：

```text
谁是 192.168.1.1？请告诉我你的 MAC。
```

广播不能无限跨路由器传播，否则网络会被淹没。

### 组播

一对一组感兴趣的接收者。常见于流媒体、服务发现、某些基础设施协议。普通 Go Web 后端较少直接使用。

## 六、MTU 是什么

MTU 是一个链路层帧能承载的最大数据大小。以太网常见 MTU 是 1500 字节。

如果 IP 包大于路径上的 MTU，可能发生：

- IP 分片。
- 被丢弃并依赖路径 MTU 发现。
- 某些网络设备错误处理导致连接卡顿。

后端工程中，MTU 问题常见于：

- VPN。
- 容器网络。
- 跨云网络。
- 隧道网络。
- Kubernetes CNI。

## 七、MSS 是什么

MSS 是 TCP 每个段中能携带的最大应用数据大小。

它和 MTU 的关系大致是：

```text
MSS = MTU - IP 头部 - TCP 头部
```

以 IPv4 + TCP 常见头部为例：

```text
1500 - 20 - 20 = 1460
```

所以常见 TCP MSS 是 1460。

## 八、为什么 MTU/MSS 会影响后端

如果路径 MTU 较小，而发送方还按较大包发送，可能出现：

- 大请求或大响应卡住。
- 小请求正常，大响应超时。
- 某些接口正常，上传文件失败。
- VPN 内访问异常。

排查方向：

```bash
ping -M do -s 1472 example.com
```

说明：

- `-M do` 表示不允许分片。
- `-s 1472` 加上 28 字节 IP/ICMP 头后接近 1500。

如果失败，逐步减小 `-s`，可以估算路径 MTU。

Windows 可以使用：

```powershell
ping example.com -f -l 1472
```

## 九、Go 后端关联

一般 Go 程序不直接设置 MTU/MSS，但它会受到影响：

- HTTP 大响应在某些网络环境下超时。
- gRPC 流式传输在跨网络环境中抖动。
- 容器网络 overlay 增加额外头部，实际可用 MTU 变小。
- 服务间调用小包正常，大包失败。

如果你发现“只有大包失败”，不要只盯业务代码，应该把 MTU 放进排查列表。

## 十、实验任务

1. 查看本机网卡 MTU：

   ```bash
   ip link
   ```

2. 观察默认路由：

   ```bash
   ip route
   ```

3. 测试路径 MTU：

   ```bash
   ping -M do -s 1472 8.8.8.8
   ping -M do -s 1400 8.8.8.8
   ```

4. 用自己的话解释为什么 `1472 + 28 = 1500`。

## 十一、常见误区

- OSI 七层不是实际每个协议栈都严格分七层，它更像分析工具。
- HTTP/HTTPS 不是传输层协议。
- 交换机不根据 IP 路由公网流量，路由器才做三层转发。
- MTU 问题不是只影响网络工程师，后端大响应、大上传、跨云调用都可能遇到。
