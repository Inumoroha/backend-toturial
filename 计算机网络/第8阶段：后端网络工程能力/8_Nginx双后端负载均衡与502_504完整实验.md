# 8. Nginx 双后端负载均衡与 502/504 完整实验

本节目标：从 0 搭建两个 Go 后端服务和一个 Nginx 反向代理，观察负载均衡、请求日志、502、504 和超时配置。学完后你能把 Nginx、Go 服务、上游错误和网关状态码联系起来。

---

## 一、实验目标

最终结构：

```text
客户端 curl
  -> Nginx :8080
    -> backend-1 :8081
    -> backend-2 :8082
```

你会验证：

- Nginx 能转发到 Go 后端。
- 多次请求会轮询到不同后端。
- 后端停止时可能出现 502。
- 后端慢响应时可能出现 504。
- 日志中能看到请求耗时。

---

## 二、准备 Go 后端

创建目录：

```bash
mkdir nginx-lb-lab
cd nginx-lb-lab
go mod init nginx-lb-lab
mkdir backend
```

创建 `backend/main.go`：

```go
package main

import (
    "fmt"
    "log"
    "net/http"
    "os"
    "time"
)

func main() {
    name := os.Getenv("APP_NAME")
    if name == "" {
        name = "backend"
    }

    port := os.Getenv("PORT")
    if port == "" {
        port = "8081"
    }

    mux := http.NewServeMux()

    mux.HandleFunc("/", func(w http.ResponseWriter, r *http.Request) {
        fmt.Fprintf(w, "hello from %s\n", name)
    })

    mux.HandleFunc("/slow", func(w http.ResponseWriter, r *http.Request) {
        time.Sleep(5 * time.Second)
        fmt.Fprintf(w, "slow response from %s\n", name)
    })

    mux.HandleFunc("/health", func(w http.ResponseWriter, r *http.Request) {
        w.WriteHeader(http.StatusNoContent)
    })

    srv := &http.Server{
        Addr:              ":" + port,
        Handler:           mux,
        ReadHeaderTimeout: 2 * time.Second,
        ReadTimeout:       10 * time.Second,
        WriteTimeout:      10 * time.Second,
        IdleTimeout:       60 * time.Second,
    }

    log.Printf("%s listen on :%s", name, port)
    log.Fatal(srv.ListenAndServe())
}
```

## 四、准备 Nginx 配置

创建 `nginx.conf`：

```nginx
events {}

http {
    log_format main '$remote_addr "$request" status=$status '
                    'request_time=$request_time '
                    'upstream_addr=$upstream_addr '
                    'upstream_status=$upstream_status '
                    'upstream_response_time=$upstream_response_time';

    access_log /var/log/nginx/access.log main;

    upstream go_backend {
        server host.docker.internal:8081;
        server host.docker.internal:8082;
    }

    server {
        listen 8080;

        location / {
            proxy_pass http://go_backend;
            proxy_connect_timeout 1s;
            proxy_send_timeout 2s;
            proxy_read_timeout 2s;

            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        }
    }
}
```

说明：

- `upstream go_backend` 定义两个后端。
- `proxy_pass http://go_backend` 转发到 upstream。
- `proxy_read_timeout 2s` 表示等待上游响应最多 2 秒。
- `host.docker.internal` 用于 Docker 容器访问宿主机服务。Linux 上如果不可用，可以改为宿主机网关 IP，或直接在宿主机安装 Nginx。

## 五、启动 Nginx 容器

```bash

  -p 8080:8080 \
  -v "$PWD/nginx.conf:/etc/nginx/nginx.conf:ro" \
  nginx:alpine
```

```powershell
  -p 8080:8080 `
```

## 六、验证负载均衡

另一个终端多次请求：

```bash
for i in {1..6}; do curl -s http://127.0.0.1:8080/; done
```

你可能看到：

```text
hello from backend-1
hello from backend-2
hello from backend-1
hello from backend-2
```

默认情况下，Nginx upstream 使用轮询策略。

## 七、观察 access log

进入 Nginx 容器：

```bash
docker exec -it nginx-lb sh
```

查看日志：

```sh
tail -f /var/log/nginx/access.log
```

你会看到类似：

```text
172.17.0.1 "GET / HTTP/1.1" status=200 request_time=0.001 upstream_addr=172.x.x.x:8081 upstream_status=200 upstream_response_time=0.001
```

重点字段：

- `status`：Nginx 返回给客户端的状态码。
- `request_time`：Nginx 处理整个请求的时间。
- `upstream_addr`：被选中的上游。
- `upstream_status`：上游返回状态码。
- `upstream_response_time`：上游响应耗时。

这些字段是排查 502/504 的核心。

## 八、制造 504

请求慢接口：

```bash
curl -v http://127.0.0.1:8080/slow
```

后端 sleep 5 秒，但 Nginx 配置：

```nginx
proxy_read_timeout 2s;
```

所以 Nginx 会等待上游响应超时，通常返回：

```text
504 Gateway Time-out
```

日志中可能看到：

```text
status=504 upstream_response_time=2.000
```

解释：

```text
504 表示网关连接到上游后，等待上游响应超时。
```

根因可能是上游业务慢、数据库慢、RPC 慢，也可能是 timeout 配置过短。

## 九、制造 502

停止其中一个后端，例如 backend-2。

继续请求：

```bash
for i in {1..10}; do curl -i http://127.0.0.1:8080/; done
```

当 Nginx 轮询到已停止的 backend-2 时，可能出现：

```text
502 Bad Gateway
```

解释：

```text
502 通常表示网关无法连接上游，或上游返回异常响应。
```

排查：

```bash
curl http://127.0.0.1:8082/
ss -lntp | grep 8082
docker logs nginx-lb
```

## 十、修复与改进

### 1. 调整超时

如果 `/slow` 合理需要 5 秒，可以调大：

```nginx
proxy_read_timeout 6s;
```

但更好的做法是：

- 优化上游耗时。
- 对慢接口单独配置。
- 使用异步任务。
- 避免网关无限等待。

### 2. 健康检查

开源 Nginx 对主动健康检查支持有限，生产中可使用：

- Nginx Plus。
- OpenResty。
- API Gateway。
- Kubernetes readinessProbe。
- 服务发现组件。

### 3. 日志增强

建议日志保留：

- request id。
- upstream_addr。
- upstream_status。
- upstream_response_time。
- request_time。

## 十一、常见问题

### 1. 502 一定是 Nginx 问题吗？

不一定。Nginx 报 502，根因通常在上游服务不可达、端口没监听、连接被拒绝或上游异常关闭。

### 2. 504 一定是后端代码慢吗？

不一定。可能是数据库慢、RPC 慢、连接池耗尽、网关 timeout 太短或网络问题。

### 3. request_time 和 upstream_response_time 有什么区别？

`request_time` 是 Nginx 处理客户端请求的总时间。  
`upstream_response_time` 是等待上游响应的时间。

### 4. 为什么 Docker Nginx 访问宿主机要用 host.docker.internal？

容器内的 `127.0.0.1` 是容器自己，不是宿主机。`host.docker.internal` 在 Docker Desktop 中可解析到宿主机。
