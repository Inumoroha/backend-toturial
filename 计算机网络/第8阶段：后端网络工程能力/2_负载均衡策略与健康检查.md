# 2. 负载均衡策略与健康检查

本节目标：理解负载均衡常见策略，并知道健康检查为什么重要。

---

## 一、为什么需要负载均衡

单个服务实例能力有限。多个实例可以：

- 提升吞吐。
- 提高可用性。
- 支持滚动发布。

负载均衡负责把请求分配到不同实例。

---

## 二、常见策略

| 策略 | 说明 |
| --- | --- |
| 轮询 | 依次分配 |
| 加权轮询 | 权重大实例更多请求 |
| 最少连接 | 分配给连接少的实例 |
| 一致性哈希 | 同一 key 尽量到同一实例 |

---

## 三、Nginx upstream

```nginx
upstream go_backend {
    server 127.0.0.1:8081;
    server 127.0.0.1:8082;
}

server {
    listen 8080;
    location / {
        proxy_pass http://go_backend;
    }
}
```

---

## 四、健康检查

如果某个实例已经挂了，负载均衡还继续转发，会产生错误。

健康检查通常访问：

```text
GET /health
```

服务健康返回：

```text
204 No Content 或 200 OK
```

## 补充实验：观察轮询效果

准备两个 Go 后端，分别启动在 `8081` 和 `8082`，返回自己的名字：

```go
package main

import (
    "flag"
    "fmt"
    "log"
    "net/http"
)

func main() {
    addr := flag.String("addr", ":8081", "listen address")
    name := flag.String("name", "backend-a", "backend name")
    flag.Parse()

    http.HandleFunc("/", func(w http.ResponseWriter, r *http.Request) {
        fmt.Fprintln(w, *name)
    })

    log.Fatal(http.ListenAndServe(*addr, nil))
}
```

启动：

```bash
go run . -addr :8081 -name backend-a
go run . -addr :8082 -name backend-b
```

Nginx：

```nginx
upstream go_backend {
    server 127.0.0.1:8081;
    server 127.0.0.1:8082;
}

server {
    listen 8080;
    location / {
        proxy_pass http://go_backend;
    }
}
```

连续请求：

```bash
for i in $(seq 1 6); do curl -s http://127.0.0.1:8080/; done
```

你应该看到两个后端交替出现。

## 补充理解：健康检查分两类

建议区分：

```text
liveness：进程是否活着。
readiness：是否准备好接收流量。
```

例如：

```text
/livez       只检查进程基本存活。
/readyz      检查数据库、配置、依赖是否可用。
```

不要让健康检查太重。如果每秒大量健康检查都打数据库，数据库本身可能被健康检查压垮。

滚动发布时，实例应该先 readiness 通过，再接收真实流量。

---

## 补充取舍：健康检查接口不要只返回 ok

健康检查可以分层：

```text
/livez：进程活着就返回成功。
/readyz：依赖初始化完成、配置加载完成、必要连接可用才成功。
```

错误示例：

```go
http.HandleFunc("/health", func(w http.ResponseWriter, r *http.Request) {
    w.Write([]byte("ok"))
})
```

如果数据库连接池还没初始化、配置还没加载完成，它也返回 ok，负载均衡就会把流量打进来。

更合理的 readiness 至少检查：

```text
服务是否完成启动。
必要配置是否加载。
关键依赖是否在可接受状态。
当前实例是否准备接收流量。
```

但不要在每次健康检查里执行很重的 SQL 或远程调用。

## 五、常见问题

### 1. 负载均衡能解决慢 SQL 吗？

不能根治。它只能分散请求，慢 SQL 仍需优化。

### 2. 健康检查是否要检查数据库？

要谨慎。健康检查太重会反过来压垮依赖。通常分 liveness 和 readiness。

### 3. 一致性哈希适合什么？

适合缓存、会话粘性等希望同一 key 到同一实例的场景。
