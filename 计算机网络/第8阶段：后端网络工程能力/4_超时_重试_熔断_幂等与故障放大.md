# 4. 超时、重试、熔断、幂等与故障放大

本节目标：理解后端网络调用中的可靠性策略，避免重试风暴和故障放大。

---

## 一、超时是基础

没有超时，网络调用可能永久卡住。

```go
ctx, cancel := context.WithTimeout(ctx, 300*time.Millisecond)
defer cancel()
```

每个下游调用都应该有明确边界。

---

## 二、重试

适合重试：

- 短暂网络错误。
- 502、503、504。
- 幂等读请求。

不适合盲目重试：

- 扣款。
- 创建订单。
- 发优惠券。

重试必须有：

- 最大次数。
- 总超时。
- 退避。
- 幂等保障。

---

## 三、幂等

幂等表示执行一次和多次结果一致。

写请求可使用：

```text
Idempotency-Key
唯一业务号
数据库唯一索引
状态机
```

---

## 四、熔断

熔断用于下游持续失败时快速失败。

状态：

- Closed。
- Open。
- Half-Open。

熔断保护调用方，不是修复下游。

---

## 五、故障放大

如果 A 调 B 重试 3 次，B 调 C 也重试 3 次，C 可能承受 9 次请求。

解决：

- 控制重试层级。
- 全链路 deadline。
- 限流。
- 熔断。
- 只对幂等请求重试。

---

## 补充示例：为一次调用设计时间预算

假设用户请求总预算是 `1000ms`：

```text
网关处理：50ms
业务服务处理：200ms
调用库存服务：300ms
调用支付服务：300ms
预留：150ms
```

那么库存服务调用就不能设置成 `3s` 超时。否则用户请求早就超时了，下游调用还在继续浪费资源。

Go 中推荐把上游请求的 context 继续传下去：

```go
func handler(w http.ResponseWriter, r *http.Request) {
    ctx, cancel := context.WithTimeout(r.Context(), time.Second)
    defer cancel()

    if err := callInventory(ctx); err != nil {
        http.Error(w, "inventory failed", http.StatusBadGateway)
        return
    }
}
```

这样客户端断开或网关超时后，下游调用也有机会被取消。

## 补充示例：幂等键如何防止重复创建

创建订单时客户端传：

```http
Idempotency-Key: 20260705-user123-submit001
```

服务端保存：

```text
idempotency_key
request_hash
order_id
status
created_at
```

处理逻辑：

```text
第一次请求：创建订单，并记录幂等键。
重复请求：发现幂等键已存在，直接返回第一次的结果。
请求体不同但幂等键相同：返回冲突错误。
```

这比“POST 一律不重试”更工程化，也比“失败就重试”安全得多。

---

## 补充判断：哪些错误可以重试

可以优先考虑重试：

```text
连接临时失败。
上游 502/503/504。
读请求。
带幂等键的写请求。
明确可重复执行的任务提交。
```

不建议默认重试：

```text
参数错误 400。
未授权 401/403。
资源不存在 404。
唯一冲突 409。
没有幂等保护的扣款、发券、创建订单。
```

重试前必须问：

```text
这次请求重复执行会不会造成副作用？
重试总耗时是否超过用户请求预算？
上游现在是短暂抖动还是已经过载？
有没有退避和最大次数？
```

这几个问题比“重试几次”更重要。

---

## 六、常见问题

### 1. 重试次数越多越可靠吗？

不是。重试过多会放大流量，可能让已经故障的下游更快雪崩。

### 2. 熔断是不是为了让下游恢复？

不是。熔断主要保护调用方和整体系统，下游恢复需要它自己的修复过程。

### 3. 幂等是不是只和 HTTP 方法有关？

不是。HTTP 方法只是语义约定，真正幂等要靠业务设计和数据约束保证。
