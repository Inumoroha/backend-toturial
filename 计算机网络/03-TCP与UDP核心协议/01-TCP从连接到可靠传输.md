# 01-TCP：从连接建立到可靠传输

## 学习目标

理解 TCP 为什么可靠、为什么需要连接、三次握手做了什么、可靠传输靠哪些机制保证。

## 一、TCP 的核心特性

TCP 是传输层协议，提供：

- 面向连接。
- 可靠传输。
- 字节流。
- 有序交付。
- 流量控制。
- 拥塞控制。

这意味着应用层可以像读写文件一样读写连接，但 TCP 不理解应用层消息边界。

## 二、三次握手

三次握手流程：

```text
客户端 -> 服务端：SYN，客户端初始序列号 x
服务端 -> 客户端：SYN + ACK，服务端初始序列号 y，确认 x+1
客户端 -> 服务端：ACK，确认 y+1
```

三次握手的目标：

- 双方确认彼此收发能力。
- 交换初始序列号。
- 建立连接状态。

为什么不是两次？

两次握手时，服务端无法确认客户端是否能收到自己的响应，容易因为旧连接请求造成资源浪费。

## 三、序列号与确认号

TCP 把数据看成字节流，每个字节都有序号。

- 序列号：当前报文段中第一个数据字节的编号。
- 确认号：期望收到的下一个字节编号。

确认号是累计确认。例如确认号是 `101`，表示 `100` 及之前的数据都已经收到。

## 四、重传机制

TCP 通过确认和重传保证可靠：

- 数据发出后等待 ACK。
- 超时未收到 ACK，触发重传。
- 收到重复 ACK，可能触发快速重传。

可靠不是“网络不丢包”，而是“丢了能发现并补救”。

## 五、流量控制

流量控制解决接收方处理不过来的问题。

接收方通过 TCP 窗口告诉发送方：我还能接收多少数据。

如果接收方窗口变小，发送方要降低发送速度。

## 六、拥塞控制

拥塞控制解决网络整体过载的问题。

核心思想：网络拥塞时，发送方主动降低发送速率。

常见概念：

- 慢启动。
- 拥塞避免。
- 快速重传。
- 快速恢复。

作为后端工程师，不一定要记所有算法细节，但要理解网络慢、丢包、抖动会直接影响 TCP 吞吐。

## 七、实验：抓三次握手

启动一个 Go HTTP 服务：

```go
package main

import (
    "fmt"
    "net/http"
)

func main() {
    http.HandleFunc("/", func(w http.ResponseWriter, r *http.Request) {
        fmt.Fprintln(w, "ok")
    })
    http.ListenAndServe(":8080", nil)
}
```

抓包：

```bash
sudo tcpdump -i any tcp port 8080 -nn
```

请求：

```bash
curl http://127.0.0.1:8080
```

观察输出中的 `S`、`S.`、`.`：

- `S` 表示 SYN。
- `S.` 表示 SYN + ACK。
- `.` 表示 ACK。

## 八、Go 后端关联

Go 中建立 TCP 连接：

```go
conn, err := net.DialTimeout("tcp", "127.0.0.1:8080", 3*time.Second)
```

`DialTimeout` 覆盖的是连接建立阶段。HTTP 请求还需要考虑：

- DNS 超时。
- TCP 连接超时。
- TLS 握手超时。
- 请求写入超时。
- 响应头等待超时。
- 响应体读取超时。
