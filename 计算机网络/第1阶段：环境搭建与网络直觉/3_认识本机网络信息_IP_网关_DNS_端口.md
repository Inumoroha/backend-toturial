# 3. 认识本机网络信息：IP、网关、DNS、端口

本节目标：学会查看并理解本机的 IP 地址、网关、DNS、路由和端口监听。后端排障的第一步通常不是看代码，而是确认机器的网络身份和服务入口。

很多初学者会把“能上网”和“服务能被访问”混为一谈。实际上，一台机器能访问外网，不代表别人能访问它的某个端口；服务启动成功，也不代表监听地址对外可达。

---

## 一、先看本机 IP

Linux / WSL2：

```bash
ip addr
```

你会看到多个网络接口，例如：

```text
lo
eth0
docker0
```

常见含义：

- `lo`：本地回环接口，地址通常是 `127.0.0.1`。
- `eth0`：主要网卡。
- `docker0`：Docker bridge 网卡。

你重点关注类似这样的地址：

```text
inet 192.168.1.10/24
```

它表示：

- IP 地址是 `192.168.1.10`。
- `/24` 是网络前缀长度。
- 这台机器位于一个局域网网段中。

Windows：

```powershell
ipconfig
```

重点看：

- IPv4 地址。
- 子网掩码。
- 默认网关。
- DNS 服务器。

---

## 二、理解 127.0.0.1、0.0.0.0 和局域网 IP

### 127.0.0.1

`127.0.0.1` 表示本机回环地址。

如果 Go 服务监听：

```text
127.0.0.1:8080
```

通常只有本机能访问。

### 0.0.0.0

`0.0.0.0` 在监听场景中表示监听本机所有网卡。

如果 Go 服务监听：

```text
0.0.0.0:8080
```

那么本机可以通过：

```text
127.0.0.1:8080
局域网IP:8080
```

访问它。

### 局域网 IP

例如：

```text
192.168.1.10
```

同一局域网内的其他机器可以尝试通过这个地址访问你的服务，前提是：

- 服务监听了对应网卡。
- 防火墙放行。
- 两台机器网络互通。

---

## 三、查看默认网关

Linux：

```bash
ip route
```

典型输出：

```text
default via 192.168.1.1 dev eth0
192.168.1.0/24 dev eth0 proto kernel scope link src 192.168.1.10
```

解释：

- `default via 192.168.1.1`：访问非本地网段时，把包交给 `192.168.1.1`。
- `192.168.1.0/24 dev eth0`：访问本地网段时，直接走 `eth0`。

默认网关通常是你的路由器、云网络网关或容器 bridge。

---

## 四、查看 DNS

Linux：

```bash
cat /etc/resolv.conf
```

你会看到：

```text
nameserver 8.8.8.8
```

或者在 WSL2 / Docker 中看到内部 DNS 地址。

查询域名：

```bash
dig example.com
```

只看结果：

```bash
dig +short example.com
```

如果 DNS 失败，HTTP 请求可能在 TCP 连接之前就失败，所以服务端不会有任何请求日志。

---

## 五、查看端口监听

启动服务后，应该先确认端口是否真的监听。

Linux：

```bash
ss -lntp
```

参数说明：

- `-l`：只看监听状态。
- `-n`：不把端口解析成服务名。
- `-t`：只看 TCP。
- `-p`：显示进程信息。

如果看到：

```text
LISTEN 0 4096 0.0.0.0:8080
```

说明服务监听所有网卡的 `8080` 端口。

如果看到：

```text
LISTEN 0 4096 127.0.0.1:8080
```

说明服务只监听本机回环地址。

---

## 六、测试连通性

测试 IP 层连通性：

```bash
ping 8.8.8.8
```

测试 DNS：

```bash
ping example.com
```

测试 HTTP：

```bash
curl -v https://example.com
```

注意：

```text
ping 通不代表 HTTP 端口通。
HTTP 通不代表 ping 一定通。
```

有些服务器会禁用 ICMP，所以 `ping` 失败不一定表示网站不可访问。

---

## 七、实验：观察访问 example.com 的过程

第一步，查 DNS：

```bash
dig +short example.com
```

第二步，看访问目标走哪条路由：

```bash
ip route get example.com
```

如果 `ip route get example.com` 不支持域名，可以先用 `dig` 查到 IP，再执行：

```bash
ip route get 目标IP
```

第三步，用 curl 访问：

```bash
curl -v https://example.com
```

观察输出中的：

- DNS 解析。
- TCP 连接。
- TLS 握手。
- HTTP 状态码。

---

## 八、Go 后端中的对应关系

当你写：

```go
http.ListenAndServe(":8080", nil)
```

它通常表示监听所有网卡的 `8080`。

当你写：

```go
http.ListenAndServe("127.0.0.1:8080", nil)
```

它通常只允许本机访问。

当你写：

```go
http.Get("https://example.com")
```

背后至少经历：

```text
DNS 解析 -> TCP 连接 -> TLS 握手 -> HTTP 请求响应
```

理解这些步骤，才能在请求慢或失败时分段排查。

## 九、常见问题

### 1. 为什么我本机 curl 成功，别人访问失败？

常见原因：

- 服务只监听 `127.0.0.1`。
- 防火墙拦截。
- 对方和你不在同一网络。
- 路由器没有端口转发。
- 云服务器安全组没有开放端口。

### 2. 为什么 ping 通但 curl 失败？

`ping` 使用 ICMP，`curl` 使用 TCP + HTTP/HTTPS。端口没开、防火墙拦截、服务异常都可能导致 curl 失败。

### 3. 为什么服务启动没有报错，但 ss 看不到端口？

可能服务启动后立即退出，或者监听的是另一个端口。检查程序日志和启动参数。

### 4. 为什么 WSL2 的 IP 和 Windows IP 不一样？

WSL2 运行在一个轻量虚拟网络中，它有自己的 IP 和路由。Windows 与 WSL2 之间有特殊转发规则，不要简单把它们当成同一台网络主机。

## 十、本节达标标准

学完本节后，你应该能够做到：

- 使用 `ip addr` 或 `ipconfig` 找到本机 IP。
- 使用 `ip route` 找到默认网关。
- 使用 `cat /etc/resolv.conf` 和 `dig` 查看 DNS。
- 使用 `ss -lntp` 判断服务是否监听端口。
- 解释 `127.0.0.1`、`0.0.0.0`、局域网 IP 的区别。
- 解释为什么 `ping` 通不代表 HTTP 服务一定可用。
