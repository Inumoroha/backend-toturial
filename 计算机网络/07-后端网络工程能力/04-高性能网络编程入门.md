# 04-高性能网络编程入门

## 学习目标

建立高性能网络编程的基本认识：C10K、I/O 多路复用、Go netpoller、连接池、背压、压测和 pprof。这个阶段不要求你立刻手写高性能框架，但要知道高并发服务的瓶颈在哪里。

## 一、C10K 问题

C10K 指单机同时处理一万个连接的问题。

早期一个连接一个线程的模型在大量连接下会遇到：

- 线程数量过多。
- 上下文切换成本高。
- 内存占用高。
- 调度压力大。

现代服务通常使用 I/O 多路复用或语言运行时调度来解决。

## 二、I/O 多路复用

常见机制：

- `select`
- `poll`
- `epoll`
- `kqueue`
- `IOCP`

Linux 上常见的是 `epoll`。它允许一个线程监听多个 socket 的可读可写事件。

你不一定要直接写 `epoll` 代码，但要理解：高并发网络服务不能简单等于“每个连接一个系统线程”。

## 三、Go 的网络模型

Go 写网络服务时常见写法：

```go
for {
    conn, err := ln.Accept()
    if err != nil {
        continue
    }
    go handleConn(conn)
}
```

看起来是每个连接一个 goroutine，但 goroutine 不是系统线程。

Go runtime 会通过 netpoller 等机制管理网络 I/O。当 goroutine 阻塞在网络读写时，runtime 可以调度其他 goroutine 继续运行。

所以 Go 很适合编写高并发网络服务，但这不代表可以忽略资源限制。

## 四、高并发瓶颈在哪里

常见瓶颈：

- CPU 打满。
- 内存增长。
- goroutine 泄漏。
- 文件描述符耗尽。
- 临时端口耗尽。
- 下游连接池耗尽。
- 锁竞争。
- GC 压力。
- 网络带宽达到上限。
- 下游服务成为瓶颈。

排查时不能只看 QPS，还要看延迟、错误率、资源使用和连接状态。

## 五、连接池

连接池用于复用连接，减少握手成本。

常见于：

- HTTP Client。
- 数据库连接。
- Redis 连接。
- RPC Client。

连接池参数不是越大越好：

- 太小：请求排队，延迟升高。
- 太大：下游压力过大，连接资源浪费。

合理配置要结合：

- 服务并发。
- 下游承载能力。
- 请求耗时。
- 超时策略。

## 六、背压

背压指当下游处理不过来时，上游应该降低发送速度或拒绝请求，而不是无限堆积。

没有背压可能导致：

- 队列无限增长。
- 内存膨胀。
- 请求超时堆积。
- 整个服务雪崩。

后端常见背压方式：

- 限流。
- 有界队列。
- 超时。
- 熔断。
- 拒绝过载请求。

## 七、压测工具

常用工具：

```bash
hey -n 10000 -c 100 http://127.0.0.1:8080/
wrk -t4 -c100 -d30s http://127.0.0.1:8080/
```

指标关注：

- 平均延迟。
- P95 / P99 延迟。
- QPS。
- 错误率。
- CPU。
- 内存。
- goroutine 数。
- 连接数。

不要只看平均延迟。后端系统更关心尾延迟，例如 P99。

## 八、pprof

Go 服务可以开启 pprof：

```go
import _ "net/http/pprof"

go func() {
    http.ListenAndServe("127.0.0.1:6060", nil)
}()
```

常用命令：

```bash
go tool pprof http://127.0.0.1:6060/debug/pprof/profile
go tool pprof http://127.0.0.1:6060/debug/pprof/heap
curl http://127.0.0.1:6060/debug/pprof/goroutine?debug=1
```

线上注意：

- pprof 不要直接暴露到公网。
- 可以只监听本地地址。
- 通过堡垒机、端口转发或内网访问。

## 九、实验：压测一个 Go HTTP 服务

1. 启动一个带 `/health` 接口的 Go HTTP 服务。
2. 使用 `hey` 压测：

   ```bash
   hey -n 5000 -c 100 http://127.0.0.1:8080/health
   ```

3. 同时观察连接：

   ```bash
   ss -antp | grep 8080
   ```

4. 观察 goroutine：

   ```bash
   curl http://127.0.0.1:6060/debug/pprof/goroutine?debug=1
   ```

5. 调整 HTTP Client 或 Server 超时，观察结果变化。

## 十、练习题

1. 为什么 goroutine-per-connection 不等于 thread-per-connection？
2. 连接池太大有什么风险？
3. 什么是背压？
4. 为什么压测要看 P95/P99，而不是只看平均值？
5. pprof 可以帮助分析哪些问题？

## 十一、验收标准

你能压测一个 Go HTTP 服务，能同时观察 CPU、内存、连接数和 goroutine，并能说出高并发服务常见瓶颈。
