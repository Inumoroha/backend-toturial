# 02-重试、限流、熔断与幂等

## 学习目标

理解网络不稳定时后端系统如何保护自己和下游，掌握重试、限流、熔断、幂等的基本设计原则。

## 一、超时是基础

没有超时，重试、熔断、限流都很难正确工作。

每次下游调用都应该有明确超时：

```go
ctx, cancel := context.WithTimeout(ctx, 300*time.Millisecond)
defer cancel()
```

## 二、重试

重试用于处理短暂失败。

适合重试：

- 网络抖动。
- 临时 502、503、504。
- 幂等读请求。

不适合盲目重试：

- 支付扣款。
- 创建订单。
- 发放优惠券。
- 没有幂等键的写操作。

重试要有边界：

- 最多重试次数。
- 总超时限制。
- 指数退避。
- 随机抖动。

## 三、幂等

幂等表示同一个请求执行一次和执行多次的结果一致。

GET 通常应该幂等。POST 不一定幂等。

写操作要支持安全重试，可以引入：

- 幂等键。
- 唯一业务号。
- 数据库唯一索引。
- 状态机。

示例：

```text
POST /orders
Idempotency-Key: user-1-request-123
```

服务端用这个 key 保证重复请求不会创建多笔订单。

## 四、限流

限流用于保护服务不被过量请求压垮。

常见算法：

- 固定窗口。
- 滑动窗口。
- 漏桶。
- 令牌桶。

Go 中可用 `golang.org/x/time/rate` 实现令牌桶：

```go
limiter := rate.NewLimiter(rate.Limit(100), 200)

if !limiter.Allow() {
    http.Error(w, "too many requests", http.StatusTooManyRequests)
    return
}
```

## 五、熔断

熔断用于在下游持续失败时快速失败，避免调用方被拖垮。

状态：

- Closed：正常调用。
- Open：快速失败。
- Half-Open：放少量请求探测恢复。

熔断不是为了“修复下游”，而是为了保护调用方和整体系统。

## 六、重试风暴

如果每层都重试 3 次：

```text
A -> B -> C
```

A 重试 3 次，B 每次又重试 C 3 次，C 可能承受 9 次请求。

层级越多，放大越明显。

解决：

- 控制重试层级。
- 设置全链路 deadline。
- 配合熔断和限流。
- 只对幂等请求重试。

## 七、练习题

1. 为什么重试必须配合超时？
2. 什么是幂等键？
3. 限流和熔断分别保护什么？
4. 为什么重试可能放大故障？

## 八、验收标准

你能设计一个下游调用策略：包含超时、有限重试、幂等保护、限流和熔断边界。
