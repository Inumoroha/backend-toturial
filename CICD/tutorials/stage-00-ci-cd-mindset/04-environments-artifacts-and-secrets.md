# 04：环境、制品与密钥

## 1. 为什么单独讲这三个概念

很多 CI/CD 问题看起来是流水线配置问题，本质却是这三个问题没想清楚：

```text
代码要部署到哪个环境？
部署的到底是哪一个制品？
部署过程使用了哪些密钥？
```

环境、制品、密钥，是后端交付里的三条主线。

## 2. 环境是什么

环境是应用运行的目标上下文。

它包含：

- 服务器或集群。
- 网络。
- 数据库。
- 缓存。
- 配置。
- 域名。
- 日志和监控。
- 权限。
- 数据。

不要把环境理解成一个简单变量，例如：

```text
APP_ENV=prod
```

它只是环境的一小部分。

## 3. 常见环境

| 环境 | 作用 | 特点 |
| --- | --- | --- |
| local | 开发者本地开发 | 速度快，差异大 |
| dev | 联调环境 | 变动频繁，不稳定 |
| test | 测试环境 | 给测试人员验证 |
| staging | 预发布环境 | 尽量接近生产 |
| production | 生产环境 | 用户真实流量 |

初学时，至少建立这三个概念：

```text
local: 我自己的机器
staging: 上线前验证环境
production: 真实用户环境
```

## 4. staging 和 production 的区别

staging 应该尽量模拟 production，但通常不会完全一样。

| 对比项 | staging | production |
| --- | --- | --- |
| 流量 | 内部或测试流量 | 真实用户流量 |
| 数据 | 脱敏数据或测试数据 | 真实业务数据 |
| 权限 | 较宽松 | 严格控制 |
| 发布 | 可自动 | 通常需要更严格门禁 |
| 失败影响 | 较小 | 直接影响用户 |

重要原则：

```text
能在 staging 发现的问题，不要留到 production。
```

## 5. 环境差异来自哪里

常见差异：

- 数据库地址不同。
- Redis 地址不同。
- API key 不同。
- 日志级别不同。
- 副本数不同。
- CPU 和内存限制不同。
- 是否开启调试功能。
- 外部依赖是否使用 mock。

Go 服务一般通过环境变量读取配置：

```text
APP_ENV=staging
HTTP_ADDR=:8080
DATABASE_URL=postgres://...
REDIS_ADDR=redis:6379
LOG_LEVEL=info
```

但要注意：环境变量里可能包含密钥，不能随便打印。

## 6. 制品是什么

制品是一次构建产生的、可以保存、传递、部署或审计的结果。

Go 后端常见制品：

- Go 二进制文件。
- Docker 镜像。
- 测试报告。
- 覆盖率报告。
- Helm Chart。
- SBOM。

最重要的生产制品通常是 Docker 镜像。

## 7. 好制品的标准

一个适合生产部署的制品应该具备：

- 可追溯：能知道来自哪个 commit。
- 可重复：相同源码和依赖能构建出一致结果。
- 可扫描：能检查漏洞和依赖风险。
- 可发布：能推送到制品仓库或镜像仓库。
- 可回滚：旧版本仍然能找到并重新部署。

糟糕的制品管理方式：

```text
scp 一个本地编译的二进制到服务器
```

这种方式很难回答：

- 谁编译的？
- 用哪份代码编译的？
- 编译机器有什么差异？
- 这个文件还能不能重新生成？

## 8. 镜像标签怎么设计

不要只使用：

```text
latest
```

更推荐：

```text
go-cicd-lab:8f3a2c1
go-cicd-lab:v0.3.0
go-cicd-lab:main-8f3a2c1
```

常见策略：

| 标签 | 用途 |
| --- | --- |
| Git SHA | 精确追溯一次提交 |
| 语义化版本 | 正式发布版本 |
| 分支名 + SHA | 测试环境排查方便 |
| latest | 只适合辅助，不适合作为生产唯一依据 |

生产环境推荐使用不可变标签，例如 commit SHA 或版本号。

## 9. 密钥是什么

密钥是能访问敏感资源的信息。

例如：

- 数据库密码。
- Redis 密码。
- JWT 签名密钥。
- 镜像仓库 token。
- 云厂商 Access Key。
- SSH 私钥。
- Kubernetes kubeconfig。

密钥泄漏可能导致：

- 数据库被拖库。
- 镜像被篡改。
- 生产集群被接管。
- 云资源被滥用。
- 用户数据泄露。

所以 CI/CD 安全里，secret 管理非常重要。

## 10. 密钥管理原则

### 原则一：不要进 Git

不要提交：

```text
.env
id_rsa
kubeconfig
service-account.json
```

可以提交模板：

```text
.env.example
```

模板里只写字段名，不写真密钥：

```text
DATABASE_URL=
JWT_SECRET=
```

### 原则二：不要进日志

危险示例：

```bash
echo "$DATABASE_URL"
```

流水线日志通常会被多人查看或长期保存。即使平台有自动脱敏，也不要依赖它解决所有问题。

### 原则三：不要进镜像

危险示例：

```dockerfile
ENV DATABASE_PASSWORD=my-secret-password
```

镜像层可能被缓存、推送、下载、扫描。密钥一旦进入镜像，删除起来很麻烦。

### 原则四：按环境隔离

staging 和 production 应该使用不同密钥。

不要让 staging 的流水线 job 拿到 production 密钥。

### 原则五：最小权限

部署 staging 的 token 只能部署 staging。

部署 production 的 token 只能做生产部署需要的事情。

## 11. 环境、制品、密钥之间的关系

一次部署可以理解成：

```text
把某个制品，使用某组配置和密钥，部署到某个环境。
```

例如：

```text
artifact: registry.example.com/go-cicd-lab:v0.3.0
environment: production
config: LOG_LEVEL=info, REPLICA_COUNT=3
secret: DATABASE_URL, JWT_SECRET
```

如果这三件事不清楚，发布就不可控。

## 13. 本节小结

你现在应该理解：

- 环境不是一个变量，而是一组运行上下文。
- 制品是可保存、可部署、可追溯的构建结果。
- 密钥是高风险资产，不能进代码、日志和镜像。
- 部署的核心句式是：把哪个制品部署到哪个环境，并使用哪些配置和密钥。
