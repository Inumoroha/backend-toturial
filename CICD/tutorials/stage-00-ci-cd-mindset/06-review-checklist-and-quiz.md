## 3. 自测题

### 题目 1：CI 主要回答什么问题？

参考答案：

```text
CI 主要回答“这次代码变更是否具备合并资格”。
它通过自动运行格式检查、静态检查、测试和构建来尽早发现问题。
```

### 题目 2：持续交付和持续部署有什么区别？

参考答案：

```text
持续交付强调代码随时可以发布到生产，但生产发布前通常有人工审批。
持续部署强调自动化检查通过后，代码可以自动发布到生产。
```

### 题目 3：cache 和 artifact 有什么区别？

参考答案：

```text
cache 用来加速后续流水线运行，不代表最终交付结果。
artifact 是本次构建产生的结果，用于保存、传递、部署或审计。
```

### 题目 4：为什么生产环境不建议只使用 latest 镜像标签？

参考答案：

```text
latest 指向不明确，可能随着新构建变化。
生产发布需要可追溯、可审计、可回滚，所以更适合使用 commit SHA 或语义化版本。
```

### 题目 5：PR 阶段和生产发布阶段的关注点有什么不同？

参考答案：

```text
PR 阶段关注代码是否能合并，例如测试、lint、build。
生产发布阶段关注发布是否安全可控，例如制品版本、staging 验证、审批、回滚和监控。
```

### 题目 6：为什么不能把数据库密码写进 Dockerfile？

参考答案：

```text
Docker 镜像层可能被缓存、推送和下载。
密钥进入镜像后很难彻底清理，也可能被无关环境或人员拿到。
```

### 题目 7：staging 环境的价值是什么？

参考答案：

```text
staging 用来在接近生产的环境里验证部署、配置、依赖和核心功能。
它的目标是在生产发布前尽早发现问题。
```

### 题目 8：一次生产发布前，你至少要确认哪些事情？

参考答案：

```text
CI 是否通过。
制品版本是否明确。
staging 是否部署和验证通过。
是否有数据库迁移。
是否有回滚方案。
是否有合适的发布窗口。
生产部署需要的 secret 是否正确且权限最小。
```

## 4. 情景题

### 情景 1

你合并代码后，main 分支流水线构建出了一个 Docker 镜像，并把它推送到了镜像仓库。但部署 staging 时失败。

请回答：

1. 这说明 CI 一定有问题吗？
2. 你会先检查哪些信息？
3. 这次镜像应该删除吗？

参考思路：

```text
不一定。CI 构建成功只能说明代码和镜像构建通过，不代表部署环境一定正常。
可以检查部署日志、镜像 tag、环境变量、secret、数据库连接、服务健康检查、网络权限。
镜像不一定要删除，它仍然是一次可追溯制品。是否保留取决于制品策略。
```

### 情景 2

某同事说：“我们有 CI 了，所以不需要代码评审。”

你怎么回答？

参考思路：

```text
CI 和代码评审解决的问题不同。
CI 擅长自动发现格式、测试、构建等确定性问题。
代码评审擅长发现设计、业务逻辑、可维护性、安全上下文等问题。
两者应该互补。
```

### 情景 3

生产发布后，错误率上升。你准备回滚。

请回答：

1. 你需要知道哪些版本信息？
2. 回滚后要验证什么？
3. 如果这次发布包含数据库字段删除，会有什么风险？

参考思路：

```text
需要知道当前生产镜像 tag、上一个稳定镜像 tag、对应 commit、发布时间和迁移情况。
回滚后要检查 healthz、readyz、核心接口、日志、错误率和延迟。
如果发布删除了数据库字段，旧版本代码可能依赖该字段，回滚后可能无法正常运行。
```
