# 01：CI/CD 基础概念

## 1. 先理解软件交付的问题

后端工程师写完代码之后，真正进入用户手里的过程通常不是这样的：

```text
写完代码 -> 复制到服务器 -> 重启服务 -> 完事
```

真实团队里更像这样：

```text
写代码
-> 提交代码
-> 代码评审
-> 自动测试
-> 静态检查
-> 构建二进制或镜像
-> 安全扫描
-> 发布到测试环境
-> 验证功能
-> 审批
-> 发布到生产环境
-> 观察指标和日志
-> 必要时回滚
```

CI/CD 的目的，就是把这条链路中重复、容易出错、需要标准化的部分自动化。

## 2. CI 是什么

CI 是 Continuous Integration，中文通常叫“持续集成”。

它关注的是：每次代码变更之后，能不能尽快证明这次变更没有破坏项目。

常见 CI 动作包括：

- 拉取代码。
- 安装依赖。
- 检查代码格式。
- 执行静态检查。
- 执行单元测试。
- 执行集成测试。
- 编译项目。
- 生成测试报告。

对 Go 后端项目来说，CI 里经常出现这些命令：

```bash
go mod download
go fmt ./...
go vet ./...
go test ./...
go test -race ./...
go build ./...
```

这些命令你后面会逐个学。现在只需要知道：CI 要回答的是“这次代码能不能合并”。

## 3. CD 是什么

CD 有两个常见含义：

- Continuous Delivery：持续交付。
- Continuous Deployment：持续部署。

它们很像，但有一个关键区别：是否自动发布到生产环境。

## 4. Continuous Delivery：持续交付

持续交付强调：代码随时处于可以发布的状态。

典型流程：

```text
代码合并到 main
-> 自动测试通过
-> 自动构建镜像
-> 自动部署到 staging
-> 人工审批
-> 发布 production
```

注意：生产发布前通常还有人工确认。

这种方式适合：

- 金融、医疗、政企等需要审批的系统。
- 生产发布风险较高的系统。
- 团队还没有足够自动化验证能力的阶段。

## 5. Continuous Deployment：持续部署

持续部署强调：只要自动化检查全部通过，代码就可以自动发布到生产环境。

典型流程：

```text
代码合并到 main
-> 自动测试通过
-> 自动构建镜像
-> 自动部署 production
-> 自动验证
-> 自动监控
```

这种方式适合：

- 自动化测试和监控非常成熟的团队。
- 发布频率很高的互联网服务。
- 可以快速回滚或灰度的系统。

如果你是初学者，建议先理解并实践 Continuous Delivery，再逐步接近 Continuous Deployment。

## 6. CI、Delivery、Deployment 的区别

| 概念 | 关注点 | 典型问题 |
| --- | --- | --- |
| CI | 代码能不能合并 | 测试是否通过？构建是否成功？ |
| Continuous Delivery | 代码能不能随时发布 | 制品是否准备好？是否可部署到环境？ |
| Continuous Deployment | 代码能不能自动上线 | 检查通过后能否自动进入生产？ |

一句话记忆：

```text
CI 解决“能不能合并”。
持续交付解决“能不能发布”。
持续部署解决“能不能自动发布到生产”。
```

## 7. 为什么 Go 后端工程师需要懂 CI/CD

Go 后端工程师不是只负责写 handler、service、repository。

一个合格的后端工程师还要关心：

- 代码怎么测试。
- 服务怎么构建。
- 配置怎么注入。
- 镜像怎么发布。
- 服务怎么部署。
- 失败怎么回滚。
- 生产问题怎么观察。
- 密钥怎么保护。

CI/CD 把这些能力串成一条工程链路。

## 8. 小练习

想象你开发了一个 Go 登录接口。你准备把它合并到主分支。

请在纸上或 Markdown 里回答：

1. 合并前应该自动检查什么？
2. 合并后应该自动生成什么制品？
3. 发布到测试环境后应该验证什么？
4. 发布到生产环境前是否需要人工审批？为什么？
5. 如果发布失败，你希望系统保留哪些信息方便回滚？

## 9. 本节小结

你现在需要记住：

- CI/CD 不是某个工具，而是一套软件交付方法。
- CI 主要关注代码变更质量。
- CD 主要关注制品发布和环境部署。
- 初学阶段先追求“可靠、可理解、可回滚”，不要一开始追求复杂自动化。
