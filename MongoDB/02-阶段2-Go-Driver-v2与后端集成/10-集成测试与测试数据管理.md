# 10. 集成测试与测试数据管理

MongoDB Repository 最好写集成测试。只用 mock 很容易掩盖 BSON tag、ObjectID、唯一索引、查询条件这些真实问题。

## 1. 单元测试和集成测试的区别

单元测试：

- 不依赖真实 MongoDB。
- 适合测试 Service 业务逻辑。
- 可以 mock Repository。

集成测试：

- 连接真实 MongoDB。
- 适合测试 Repository。
- 能发现真实查询、索引、解码问题。

本节重点写 Repository 集成测试。

## 2. 测试数据库

不要直接用开发数据库：

```text
go_mongodb_stage2
```

测试使用：

```text
go_mongodb_stage2_test
```

设置环境变量：

```powershell
$env:MONGO_URI="mongodb://root:example@localhost:27017/?authSource=admin"
$env:MONGO_TEST_DATABASE="go_mongodb_stage2_test"
```

## 3. 测试辅助函数

创建 `internal/user/repository_test.go`：

```go
package user

import (
    "context"
    "os"
    "testing"
    "time"

    "go.mongodb.org/mongo-driver/v2/bson"
    "go.mongodb.org/mongo-driver/v2/mongo"
    "go.mongodb.org/mongo-driver/v2/mongo/options"
)

func setupTestRepository(t *testing.T) (*Repository, func()) {
    t.Helper()

    uri := os.Getenv("MONGO_URI")
    if uri == "" {
        uri = "mongodb://root:example@localhost:27017/?authSource=admin"
    }

    dbName := os.Getenv("MONGO_TEST_DATABASE")
    if dbName == "" {
        dbName = "go_mongodb_stage2_test"
    }

    ctx, cancel := context.WithTimeout(context.Background(), 10*time.Second)
    defer cancel()

    client, err := mongo.Connect(options.Client().ApplyURI(uri))
    if err != nil {
        t.Fatalf("connect mongodb: %v", err)
    }

    db := client.Database(dbName)

    if err := db.Collection("users").Drop(ctx); err != nil {
        t.Fatalf("drop users: %v", err)
    }

    _, err = db.Collection("users").Indexes().CreateOne(ctx, mongo.IndexModel{
        Keys:    bson.D{{"email", 1}},
        Options: options.Index().SetUnique(true),
    })
    if err != nil {
        t.Fatalf("create email index: %v", err)
    }

    cleanup := func() {
        cleanupCtx, cleanupCancel := context.WithTimeout(context.Background(), 10*time.Second)
        defer cleanupCancel()

        _ = db.Collection("users").Drop(cleanupCtx)
        _ = client.Disconnect(cleanupCtx)
    }

    return NewRepository(db), cleanup
}
```

说明：

- 每次测试前清空 `users` 集合。
- 创建唯一索引，保证重复邮箱测试真实有效。
- 返回 cleanup，测试结束后清理。

## 4. 测试 Create 和 FindByEmail

```go
func TestRepository_CreateAndFindByEmail(t *testing.T) {
    repo, cleanup := setupTestRepository(t)
    defer cleanup()

    ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
    defer cancel()

    u := &User{
        Email: "alice@example.com",
        Name:  "Alice",
        Age:   25,
        Roles: []string{"user"},
    }

    if err := repo.Create(ctx, u); err != nil {
        t.Fatalf("create user: %v", err)
    }

    if u.ID.IsZero() {
        t.Fatalf("expected id to be set")
    }

    found, err := repo.FindByEmail(ctx, "alice@example.com")
    if err != nil {
        t.Fatalf("find by email: %v", err)
    }

    if found.Email != u.Email {
        t.Fatalf("expected email %s, got %s", u.Email, found.Email)
    }
}
```

## 5. 测试未找到

```go
func TestRepository_FindByEmail_NotFound(t *testing.T) {
    repo, cleanup := setupTestRepository(t)
    defer cleanup()

    ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
    defer cancel()

    _, err := repo.FindByEmail(ctx, "missing@example.com")
    if err == nil {
        t.Fatalf("expected error")
    }
    if err != ErrNotFound {
        t.Fatalf("expected ErrNotFound, got %v", err)
    }
}
```

如果你的 Repository 还没有把 `mongo.ErrNoDocuments` 转成 `ErrNotFound`，这个测试会失败。失败是好事，它提醒你补错误转换。

## 6. 测试重复邮箱

```go
func TestRepository_Create_DuplicateEmail(t *testing.T) {
    repo, cleanup := setupTestRepository(t)
    defer cleanup()

    ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
    defer cancel()

    u1 := &User{Email: "alice@example.com", Name: "Alice"}
    u2 := &User{Email: "alice@example.com", Name: "Alice 2"}

    if err := repo.Create(ctx, u1); err != nil {
        t.Fatalf("create first user: %v", err)
    }

    err := repo.Create(ctx, u2)
    if err == nil {
        t.Fatalf("expected duplicate email error")
    }
    if err != ErrEmailExists {
        t.Fatalf("expected ErrEmailExists, got %v", err)
    }
}
```

这个测试依赖唯一索引。如果没有唯一索引，重复数据会插入成功。

## 7. 测试更新

```go
func TestRepository_UpdateProfile(t *testing.T) {
    repo, cleanup := setupTestRepository(t)
    defer cleanup()

    ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
    defer cancel()

    u := &User{Email: "alice@example.com", Name: "Alice", Age: 25}
    if err := repo.Create(ctx, u); err != nil {
        t.Fatalf("create user: %v", err)
    }

    name := "Alice New"
    age := 26
    err := repo.UpdateProfile(ctx, u.ID, UpdateProfileInput{
        Name: &name,
        Age:  &age,
    })
    if err != nil {
        t.Fatalf("update profile: %v", err)
    }

    found, err := repo.FindByID(ctx, u.ID)
    if err != nil {
        t.Fatalf("find by id: %v", err)
    }

    if found.Name != name || found.Age != age {
        t.Fatalf("unexpected user after update: %+v", found)
    }
}
```

## 9. 是否需要 testcontainers

真实项目中，可以使用 testcontainers-go 在测试时自动启动 MongoDB 容器。优点是测试更可重复，不依赖本机已有容器。

但第 2 阶段先使用你第 0 阶段搭好的本地 MongoDB，降低学习成本。后面工程化阶段再引入 testcontainers。

## 10. 测试数据管理原则

- 测试数据库和开发数据库分开。
- 每个测试前清理集合。
- 测试需要的索引要在测试中创建。
- 不依赖测试执行顺序。
- 不依赖上一次测试留下的数据。
- 测试完成后清理数据。

## 11. 本节验收

你应该能做到：

- 为 Repository 创建集成测试。
- 测试前清空集合。
- 在测试中创建唯一索引。
- 测试 Create、FindByEmail、NotFound、DuplicateEmail、UpdateProfile。
- 解释为什么 Repository 更适合集成测试。
