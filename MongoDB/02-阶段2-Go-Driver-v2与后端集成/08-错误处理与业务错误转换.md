# 08. 错误处理与业务错误转换

真实后端项目不能把 MongoDB 底层错误原样丢给 Handler。Repository 应该把常见数据库错误转换成业务层能理解的错误。

## 1. 业务错误定义

`internal/user/errors.go`：

```go
package user

import "errors"

var (
    ErrNotFound      = errors.New("user not found")
    ErrEmailExists   = errors.New("email already exists")
    ErrInvalidUpdate = errors.New("invalid update")
)
```

上层 Service 或 Handler 可以根据这些错误返回不同响应。

例如：

```text
ErrNotFound    -> HTTP 404
ErrEmailExists -> HTTP 409
ErrInvalidUpdate -> HTTP 400
```

## 2. 处理 ErrNoDocuments

`FindOne().Decode(&u)` 没查到时，会返回 `mongo.ErrNoDocuments`。

转换：

```go
func (r *Repository) FindByEmail(ctx context.Context, email string) (*User, error) {
    filter := bson.D{
        {"email", email},
        {"deleted_at", bson.D{{"$exists", false}}},
    }

    var u User
    err := r.collection.FindOne(ctx, filter).Decode(&u)
    if err != nil {
        if errors.Is(err, mongo.ErrNoDocuments) {
            return nil, ErrNotFound
        }
        return nil, err
    }

    return &u, nil
}
```

需要导入：

```go
import (
    "context"
    "errors"

    "go.mongodb.org/mongo-driver/v2/bson"
    "go.mongodb.org/mongo-driver/v2/mongo"
)
```

## 3. 处理重复键错误

先创建唯一索引：

```javascript
use go_mongodb_stage2
db.users.createIndex({ email: 1 }, { unique: true })
```

Go Driver v2 提供：

```go
mongo.IsDuplicateKeyError(err)
```

Create 方法中转换：

```go
func (r *Repository) Create(ctx context.Context, u *User) error {
    now := time.Now().UTC()

    if u.ID.IsZero() {
        u.ID = bson.NewObjectID()
    }
    if u.CreatedAt.IsZero() {
        u.CreatedAt = now
    }
    u.UpdatedAt = now
    if u.Status == "" {
        u.Status = "active"
    }

    _, err := r.collection.InsertOne(ctx, u)
    if err != nil {
        if mongo.IsDuplicateKeyError(err) {
            return ErrEmailExists
        }
        return err
    }

    return nil
}
```

官方 API 文档说明，`IsDuplicateKeyError` 会判断错误是否为重复键错误；对于批量写异常，只要其中至少一个是重复键，也会返回 true。

## 4. 处理超时错误

超时可能来自：

- context deadline exceeded。
- driver timeout。
- server selection timeout。
- 网络问题导致的超时。

Go Driver v2 提供：

```go
mongo.IsTimeout(err)
```

示例：

```go
if err != nil {
    if mongo.IsTimeout(err) {
        return fmt.Errorf("mongodb timeout: %w", err)
    }
    return err
}
```

业务项目中可以定义：

```go
var ErrDatabaseTimeout = errors.New("database timeout")
```

再转换：

```go
if mongo.IsTimeout(err) {
    return ErrDatabaseTimeout
}
```

本阶段可以先记录和包装错误，不急着建立完整错误码体系。

## 5. 处理网络错误

Go Driver v2 也提供：

```go
mongo.IsNetworkError(err)
```

网络错误通常不是业务错误，而是系统错误。上层可以返回 500 或 503，并记录日志。

```go
if mongo.IsNetworkError(err) {
    return fmt.Errorf("mongodb network error: %w", err)
}
```

## 6. 错误包装

Repository 可以保留底层错误上下文：

```go
return fmt.Errorf("insert user: %w", err)
```

但如果已经转换成业务错误：

```go
return ErrEmailExists
```

上层就可以直接判断：

```go
if errors.Is(err, user.ErrEmailExists) {
    // 返回 409
}
```

如果你既想保留底层错误，又想让上层判断业务错误，可以设计自定义错误类型。不过学习阶段先保持简单。

## 7. 不要暴露底层错误给前端

不要把这个直接返回给用户：

```text
E11000 duplicate key error collection: go_mongodb_stage2.users index: email_1 dup key...
```

应该转换成：

```json
{
  "code": "EMAIL_ALREADY_EXISTS",
  "message": "邮箱已被使用"
}
```

底层错误适合写入服务日志，不适合直接暴露给客户端。

## 8. 错误处理放在哪一层

推荐：

- Repository：把 MongoDB 错误转换成领域错误。
- Service：根据领域错误决定业务流程。
- Handler：把业务错误转换成 HTTP 状态码和响应体。

例如：

```text
mongo.ErrNoDocuments
  -> user.ErrNotFound
  -> HTTP 404
```

```text
duplicate key
  -> user.ErrEmailExists
  -> HTTP 409
```

## 9. 错误处理检查清单

Repository 中至少处理：

- `mongo.ErrNoDocuments`。
- `mongo.IsDuplicateKeyError(err)`。
- `mongo.IsTimeout(err)`。
- update/delete 的 `MatchedCount == 0` 或 `DeletedCount == 0`。

还要注意：

- Cursor 遍历后检查 `cursor.Err()`。
- Decode 失败可能是模型字段类型不匹配。
- ObjectID 字符串解析错误应该在上层处理。

## 10. 本节验收

你应该能做到：

- 用 `errors.Is(err, mongo.ErrNoDocuments)` 判断未找到。
- 用 `mongo.IsDuplicateKeyError(err)` 判断重复键。
- 用 `mongo.IsTimeout(err)` 判断超时。
- 把 MongoDB 错误转换成 `ErrNotFound`、`ErrEmailExists`。
- 解释为什么底层错误不应该直接返回给前端。
