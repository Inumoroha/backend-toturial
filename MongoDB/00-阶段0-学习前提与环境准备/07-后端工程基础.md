# 07. 后端工程基础

MongoDB 学习最终要落到 Go 后端项目里。本节补齐一些后端工程基础，帮助你把数据库知识放回真实项目。

## 1. 一个后端接口的完整链路

以“创建用户”为例：

```text
客户端发送 HTTP 请求
  -> Handler 解析 JSON
  -> Handler 做基础参数校验
  -> Service 检查业务规则
  -> Repository 写入 MongoDB
  -> Service 处理结果
  -> Handler 返回 HTTP 响应
```

代码层大致是：

```text
POST /users
  handler.CreateUser
    service.CreateUser
      repository.Create
        collection.InsertOne
```

你要避免把所有逻辑都堆到 Handler 里。

## 2. Handler、Service、Repository 各自负责什么

### Handler

负责：

- 读取请求参数。
- 解析 JSON。
- 基础格式校验。
- 调用 Service。
- 返回 HTTP 状态码和响应体。

不负责：

- 拼复杂 MongoDB 查询。
- 处理事务细节。
- 写复杂业务规则。

### Service

负责：

- 业务规则。
- 权限判断。
- 幂等控制。
- 事务编排。
- 调用一个或多个 Repository。

例如：

```text
下单 Service
  -> 检查商品状态
  -> 检查库存
  -> 创建订单
  -> 扣减库存
  -> 写库存流水
```

### Repository

负责：

- MongoDB 查询。
- MongoDB 更新。
- BSON 映射。
- 数据库错误转换。
- 索引相关的查询实现。

Repository 不应该包含太多业务判断。

## 3. 配置管理

MongoDB 连接地址不应该硬编码在代码里。

不推荐：

```go
uri := "mongodb://root:example@localhost:27017/?authSource=admin"
```

推荐从环境变量或配置文件读取：

```text
MONGO_URI=mongodb://root:example@localhost:27017/?authSource=admin
MONGO_DATABASE=mongodb_learning
```

为什么？

因为本地、测试、生产环境的地址不同。

常见配置项：

```text
APP_ENV=local
HTTP_ADDR=:8080
MONGO_URI=...
MONGO_DATABASE=...
MONGO_CONNECT_TIMEOUT=5s
MONGO_QUERY_TIMEOUT=3s
```

## 4. 日志

后端服务必须有日志。

和 MongoDB 相关的日志至少应该能看到：

- 请求 ID。
- 接口路径。
- 数据库操作是否失败。
- 错误类型。
- 耗时。
- 关键业务 ID。

不要在日志里打印：

- 用户密码。
- token。
- 完整身份证号。
- 支付敏感信息。
- 大量完整文档。

学习阶段可以用 Go 标准库 `log/slog`。

## 5. 超时控制

每一次数据库操作都应该有超时。

典型流程：

```text
HTTP 请求进入
  -> 请求 context
  -> Service 继续传递 context
  -> Repository 使用 context 执行 MongoDB 操作
```

错误示例：

```go
collection.Find(context.Background(), filter)
```

如果在请求链路里随手新建 `context.Background()`，就丢失了请求取消和超时控制。

更好的方式：

```go
func (r *UserRepository) FindByEmail(ctx context.Context, email string) (*User, error) {
    ctx, cancel := context.WithTimeout(ctx, 3*time.Second)
    defer cancel()

    // collection.FindOne(ctx, ...)
}
```

## 6. 错误转换

底层数据库错误不应该原样暴露给客户端。

例如：

```text
E11000 duplicate key error collection...
```

用户不需要看到这种错误。

你应该转换成业务错误：

```json
{
  "code": "EMAIL_ALREADY_EXISTS",
  "message": "邮箱已被使用"
}
```

常见转换：

| 底层情况 | 业务错误 |
| --- | --- |
| 没查到文档 | 资源不存在 |
| 唯一索引冲突 | 数据已存在 |
| context timeout | 服务暂时不可用或请求超时 |
| BSON 解码失败 | 系统内部错误 |
| MongoDB 连接失败 | 系统内部错误 |

## 7. API DTO 和数据库 Model 分开

数据库模型：

```go
type User struct {
    ID           ObjectID
    Email        string
    PasswordHash string
    CreatedAt    time.Time
}
```

接口返回 DTO：

```go
type UserResponse struct {
    ID        string    `json:"id"`
    Email     string    `json:"email"`
    CreatedAt time.Time `json:"created_at"`
}
```

为什么要分开？

- 数据库里有些字段不能返回给前端，比如密码哈希。
- API 字段可能需要格式化。
- 数据库模型以后可能调整，不希望直接影响 API。

## 8. 分页

列表接口一定要考虑分页。

简单分页：

```text
page=1&page_size=20
```

数据库查询可能使用：

```javascript
skip + limit
```

但深分页会有性能问题。后面会学习游标分页：

```text
?limit=20&after=last_created_at_or_id
```

第 0 阶段先知道：任何可能返回很多数据的接口，都不能一次性全返回。

## 9. 幂等

幂等是后端非常重要的能力。

例如用户重复点击“提交订单”，服务可能收到两次请求。你要避免创建两笔重复订单。

常见方式：

- 前端传 `idempotency_key`。
- 后端用唯一索引约束。
- 请求处理前先检查业务状态。
- 状态流转要有明确条件。

MongoDB 中唯一索引经常用于辅助幂等。

## 10. 测试

后端项目至少要有：

- 单元测试。
- 集成测试。
- 接口测试。

MongoDB Repository 建议使用集成测试，因为真实问题常常出现在：

- BSON 字段名写错。
- ObjectID 转换错误。
- 查询条件不正确。
- 唯一索引没建。
- 时间字段类型不匹配。

学习阶段可以用 Docker 中的本地 MongoDB 跑集成测试。

## 11. 可观测性

真实系统中，你需要知道：

- 哪个接口慢。
- 哪个 MongoDB 查询慢。
- 错误率是否升高。
- 数据库连接是否耗尽。
- 慢查询是否集中在某个集合。

常见手段：

- 日志。
- 指标。
- 链路追踪。
- MongoDB 慢查询。
- 应用层耗时统计。

第 0 阶段先养成一个习惯：重要操作记录耗时。
