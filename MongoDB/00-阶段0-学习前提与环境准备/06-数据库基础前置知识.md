# 06. 数据库基础前置知识

MongoDB 是文档数据库，但很多数据库通用概念仍然适用。你不需要先精通 MySQL，但必须理解下面这些基础，否则后面学索引、事务和性能时会很吃力。

## 1. 数据库、集合、文档

在 MongoDB 中：

```text
database   数据库
collection 集合
document   文档
```

类比关系型数据库：

```text
MySQL database  ≈ MongoDB database
MySQL table     ≈ MongoDB collection
MySQL row       ≈ MongoDB document
MySQL column    ≈ MongoDB document field
```

但这个类比只适合入门。MongoDB 的文档可以嵌套对象和数组，这一点和表格行很不一样。

## 2. 文档模型

MongoDB 文档类似 JSON：

```json
{
  "_id": "u1001",
  "name": "Alice",
  "email": "alice@example.com",
  "roles": ["user", "admin"],
  "profile": {
    "city": "Shanghai",
    "company": "Example Inc"
  }
}
```

它包含：

- 普通字段：`name`、`email`。
- 数组字段：`roles`。
- 嵌套文档：`profile`。
- 主键字段：`_id`。

MongoDB 内部使用 BSON，不是纯 JSON。BSON 支持更多类型，例如 ObjectId、Date、Decimal128、Binary。

## 3. 主键和唯一性

每个 MongoDB 文档都有 `_id` 字段。

如果插入时不提供 `_id`，MongoDB 会自动生成。

你要理解：

- `_id` 在集合内唯一。
- `_id` 默认有索引。
- `_id` 不应该随意修改。
- 业务唯一性不能只靠代码判断，应该用唯一索引保证。

例如用户邮箱唯一：

```javascript
db.users.createIndex({ email: 1 }, { unique: true })
```

只在代码里写“先查邮箱是否存在，再插入”是不可靠的。并发情况下，两个请求可能同时查到不存在，然后都插入。

## 4. 索引是什么

索引可以理解成数据库为了快速查找数据而维护的额外数据结构。

没有索引时，查询可能扫描整个集合：

```text
collection scan
```

有合适索引时，数据库可以更快定位数据：

```text
index scan
```

示例：

```javascript
db.users.createIndex({ email: 1 })
```

查询：

```javascript
db.users.find({ email: "alice@example.com" })
```

就可以利用 `email` 索引。

## 5. 索引的代价

索引不是越多越好。

索引会带来：

- 更快的读取。
- 更多的磁盘占用。
- 更慢的写入。
- 更复杂的维护成本。

每次插入、更新、删除文档时，相关索引也要更新。

所以真实项目中，索引应该围绕核心查询设计，而不是看到字段就建索引。

## 6. 查询条件、排序和分页

后端常见查询：

```text
按用户查订单
按状态筛选订单
按创建时间倒序排序
分页返回
```

对应 MongoDB 查询可能是：

```javascript
db.orders.find({
  user_id: "u1001",
  status: "paid"
}).sort({ created_at: -1 }).limit(20)
```

这个查询背后会涉及复合索引设计：

```javascript
db.orders.createIndex({ user_id: 1, status: 1, created_at: -1 })
```

这也是后面索引阶段的重点。

## 7. 事务

事务解决的是一组操作要么全部成功，要么全部失败的问题。

例如转账：

```text
A 扣 100
B 加 100
写一条转账流水
```

这三步不能只成功一部分。

MongoDB 有一个重要特点：

- 单文档写入天然是原子的。
- 多文档操作如果需要强一致，可以使用事务。

但事务不是越多越好。事务会增加系统成本，很多场景可以通过合理建模、单文档原子更新、唯一索引和幂等设计解决。

## 8. 一致性

一致性不是一句“数据要正确”这么简单。

后面你会遇到：

- readConcern：读到什么级别的数据。
- writeConcern：写入要确认到什么级别。
- readPreference：从 primary 读还是 secondary 读。

第 0 阶段只需要先记住：

- 写入成功不代表所有副本都已经确认。
- 从副本读可能读到稍旧的数据。
- 重要写入通常需要更强的确认策略。

## 9. 原子更新

MongoDB 很多更新操作是原子的，尤其是单文档更新。

例如库存扣减：

```javascript
db.products.updateOne(
  { _id: "p1001", stock: { $gte: 1 } },
  { $inc: { stock: -1 } }
)
```

这个操作的含义是：

- 只有库存大于等于 1 时才扣减。
- 查询条件和扣减在同一个更新操作中完成。

这比“先查库存，再扣库存”更安全。

## 10. 数据建模

MongoDB 建模不是先画很多表，而是先分析查询。

你要问：

- 这个接口最常读什么数据？
- 哪些数据总是一起返回？
- 哪些数据会频繁更新？
- 哪些数组可能无限增长？
- 哪些字段需要唯一约束？
- 哪些字段需要过期删除？

示例：博客文章和评论。

如果文章评论很少，可以嵌入：

```json
{
  "title": "MongoDB 入门",
  "comments": [
    { "user_id": "u1", "content": "good" }
  ]
}
```

如果文章可能有几万条评论，就不适合全部嵌入文章文档，应该拆成 `comments` 集合。

## 11. 慢查询

慢查询常见原因：

- 没有索引。
- 索引字段顺序不合适。
- 查询条件无法使用索引。
- 返回数据量太大。
- 大量 `skip` 深分页。
- 聚合管道过重。
- `$lookup` 数据量太大。

后面你会学习用：

```javascript
db.collection.find(...).explain("executionStats")
```

来观察查询是否健康。

## 12. 备份恢复

任何真实数据库都要考虑备份恢复。

学习阶段你至少要知道：

- 容器删除不等于数据一定删除，取决于是否使用 volume。
- volume 删除后，本地数据会消失。
- MongoDB 有 `mongodump` 和 `mongorestore`。
- Atlas 提供备份能力。

后面第 7 阶段会系统学习。
