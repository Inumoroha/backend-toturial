# 11. 阶段验收与建模评审清单

这是第 3 阶段的验收文件。建模能力不能只靠“看懂”，必须能独立解释取舍。

## 1. 概念验收

请用自己的话回答：

1. 为什么 MongoDB 建模要先看访问模式？
2. 什么情况下适合嵌入？
3. 什么情况下适合引用？
4. 一对少和一对多有什么建模差异？
5. 多对多什么时候用数组，什么时候用中间集合？
6. 什么是无限增长数组？
7. 为什么 MongoDB 单文档 16MiB 限制会影响建模？
8. 什么是快照型冗余？
9. 什么是展示型冗余？
10. 什么是统计型冗余？
11. 软删除会给唯一索引带来什么影响？
12. Schema Validation 能解决什么问题，不能解决什么问题？

## 2. 操作验收

你应该能写出：

- 用户集合样例文档。
- 商品集合样例文档。
- 购物车集合样例文档。
- 订单集合样例文档。
- 支付记录集合样例文档。
- 库存流水集合样例文档。

并说明每个集合：

- 为什么这样拆。
- 哪些字段嵌入。
- 哪些字段引用。
- 哪些字段冗余。
- 哪些字段可能需要索引。
- 哪些字段涉及生命周期。

## 3. 建模评审清单

每次设计一个集合，都用下面清单检查。

### 业务与访问模式

- 这个集合服务哪些接口？
- 最高频查询是什么？
- 查询条件是什么？
- 是否需要排序？
- 是否需要分页？
- 列表页和详情页字段是否不同？

### 嵌入与引用

- 哪些数据经常一起读取？
- 哪些数据数量有限？
- 哪些数据会无限增长？
- 哪些数据需要独立查询？
- 哪些数据需要独立更新？
- 哪些数据生命周期不同？

### 文档大小

- 文档是否可能接近 16MiB？
- 是否包含无限增长数组？
- 是否需要只保留最近 N 条？
- 是否需要拆集合或分桶？

### 冗余

- 哪些字段是快照？
- 哪些字段是展示冗余？
- 哪些字段是统计冗余？
- 源数据在哪里？
- 源数据变化时是否同步？
- 同步失败怎么处理？

### 生命周期

- 是否软删除？
- 是否物理删除？
- 是否 TTL？
- 是否归档？
- 热数据保留多久？
- 冷数据如何查询？

### 一致性

- 是否需要事务？
- 是否可以用单文档原子更新解决？
- 是否需要幂等键？
- 状态更新是否带条件？
- 是否有并发写入风险？

### Schema 和索引

- 是否需要 Schema Validation？
- 必填字段有哪些？
- 枚举字段有哪些？
- 核心查询是否有索引？
- 唯一约束如何保证？
- 软删除是否影响唯一索引？

## 4. 常见错误复盘

### 错误 1：照搬 MySQL 表结构

表现：

```text
orders
order_items
order_item_snapshots
order_item_prices
```

结果：订单详情需要多次查询或大量 `$lookup`。

修正：订单项通常作为订单快照嵌入。

### 错误 2：所有东西都嵌入

表现：

```javascript
{
  user: "...",
  orders: [
    // 用户全部订单
  ]
}
```

结果：数组无限增长，用户文档巨大。

修正：订单拆集合，通过 `user_id` 查询。

### 错误 3：冗余没有同步策略

表现：

```javascript
author_snapshot: {
  display_name: "Alice"
}
```

但没人知道用户改名后是否要同步文章。

修正：定义同步策略和一致性要求。

### 错误 4：软删除后忘记过滤

表现：

```javascript
db.users.find({ status: "active" })
```

查出了已软删除用户。

修正：

```javascript
db.users.find({
  status: "active",
  deleted_at: { $exists: false }
})
```

### 错误 5：Schema 太自由

表现：

同一集合字段名和类型混乱。

修正：

- Go struct tag。
- Handler 参数校验。
- Schema Validation。
- 数据迁移脚本。

## 5. 进入第 4 阶段的标准

满足下面条件，可以进入第 4 阶段：

- 能根据访问模式设计集合。
- 能解释嵌入和引用的取舍。
- 能识别无限增长数组。
- 能设计快照字段和展示冗余字段。
- 能说明冗余字段同步策略。
- 能设计软删除和归档策略。
- 能写出基础 Schema Validation。
- 能完成电商订单系统建模文档。

第 4 阶段会学习聚合管道。到那时，你会把本阶段设计的数据模型用于统计分析，例如订单金额按天统计、商品销量排行、用户消费排行等。
