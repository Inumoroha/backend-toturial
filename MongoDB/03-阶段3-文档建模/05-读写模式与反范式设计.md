# 05. 读写模式与反范式设计

MongoDB 建模经常会使用反范式，也就是为了读取效率或业务正确性，主动冗余一部分数据。反范式不是偷懒，它是一种需要被设计和维护的工程选择。

## 1. 什么是反范式

关系型范式化设计倾向于减少重复。

反范式则允许重复保存一些字段。

例如文章文档中保存作者快照：

```javascript
{
  _id: ObjectId("..."),
  author_id: ObjectId("..."),
  author_snapshot: {
    display_name: "Alice",
    avatar: "https://example.com/a.png"
  },
  title: "MongoDB 建模"
}
```

这些作者信息在 `users` 集合中也有一份。

## 2. 为什么需要反范式

常见原因：

- 高频列表页减少额外查询。
- 保留历史快照。
- 避免高频 `$lookup`。
- 简化接口响应。
- 降低跨集合事务需求。
- 支持排序和筛选。

例如订单商品快照：

```javascript
{
  product_id: ObjectId("..."),
  product_name: "Go 后端课",
  sku_name: "标准版",
  price: 9900
}
```

商品改价后，历史订单仍然要显示下单时价格。这是业务正确性，不只是性能优化。

## 3. 读多写少适合冗余

如果某些数据：

- 读取非常频繁。
- 更新很少。
- 展示字段少。
- 允许一定同步延迟。

就适合冗余。

例子：文章列表中的作者昵称和头像。

用户改昵称不是高频操作，但文章列表读取很频繁。

可以冗余：

```javascript
author_snapshot: {
  display_name: "Alice",
  avatar: "https://example.com/a.png"
}
```

## 4. 写多读少不适合过度冗余

如果数据写入高频，冗余到多个地方会增加写入成本。

例如商品实时库存：

不要到处冗余：

```text
products.stock
carts.items.stock
orders.items.stock
recommendations.stock
```

库存高频变化，冗余太多会很难保持一致。

订单里可以保存下单时库存无关的商品快照，但不应该依赖订单里的 stock 表示实时库存。

## 5. 快照型冗余

快照保存的是某个时间点的数据。

订单项：

```javascript
{
  product_id: ObjectId("..."),
  product_name: "MongoDB 入门课",
  product_cover: "https://example.com/p.png",
  sku_id: ObjectId("..."),
  sku_name: "标准版",
  price: 9900,
  quantity: 1
}
```

快照一般不随源数据更新。

适合：

- 订单。
- 支付流水。
- 发票。
- 审计日志。
- 历史消息。

## 6. 展示型冗余

展示型冗余用于减少查询。

评论保存用户昵称：

```javascript
{
  post_id: ObjectId("..."),
  user_id: ObjectId("..."),
  user_snapshot: {
    display_name: "Bob",
    avatar: "https://example.com/b.png"
  },
  content: "不错"
}
```

如果用户改头像，是否同步历史评论？

这需要业务决定：

- 评论历史保持当时头像。
- 评论展示最新头像。
- 只同步头像，不同步昵称。

## 7. 统计型冗余

文章保存评论数、点赞数、浏览数：

```javascript
stats: {
  views: 120,
  likes: 15,
  comments: 8
}
```

好处：

- 列表页直接展示。
- 排序更方便。
- 不必每次 count 评论集合。

风险：

- 计数可能不一致。
- 并发更新要用 `$inc`。
- 需要修复脚本或定期校准。

更新评论数：

```javascript
db.posts.updateOne(
  { _id: postId },
  { $inc: { "stats.comments": 1 } }
)
```

## 8. 同步策略

冗余字段必须有同步策略。

### 不同步

快照数据通常不同步。

例如历史订单商品名不随商品改名变化。

### 同步写

用户改头像时，同步更新最近评论中的头像。

优点：展示一致。

缺点：写入成本高，可能需要批量更新。

### 异步同步

用户资料变更后发送事件，由后台任务更新冗余数据。

适合：

- 数据量大。
- 允许短暂不一致。
- 不想阻塞主流程。

### 定期校准

统计字段可以定期重算。

例如每天重算文章评论数，修复异常。

## 9. 一致性边界

每个冗余字段都要定义一致性边界：

```text
字段：author_snapshot.display_name
来源：users.display_name
是否同步：同步
同步方式：异步任务
最大延迟：5 分钟
失败处理：重试和人工修复
```

如果没有这个说明，后面维护的人会不知道数据到底该不该一致。

## 10. 冗余评审问题

设计冗余字段前问：

- 这个字段为什么要冗余？
- 它解决了哪个高频查询？
- 源字段在哪里？
- 源字段变化时是否同步？
- 同步失败怎么办？
- 冗余字段是否参与筛选或排序？
- 冗余字段是否会造成用户看到不一致？
- 是否可以接受最终一致？

## 11. 本节练习

为下面字段判断是否应该冗余：

1. 订单中的商品名称。
2. 订单中的商品价格。
3. 订单中的商品当前库存。
4. 文章列表中的作者昵称。
5. 评论中的用户头像。
6. 商品文档中的评价总数。
7. 用户文档中的订单总数。
8. 购物车中的商品价格。

每个字段写：

```text
是否冗余：
冗余类型：快照/展示/统计
是否同步：
同步策略：
风险：
```

## 12. 本节小结

你需要记住：

- 反范式是为了读取效率或历史正确性主动冗余数据。
- 快照型冗余通常不同步。
- 展示型冗余要决定是否同步。
- 统计型冗余要用原子更新，并准备校准方案。
- 冗余字段必须有来源和同步策略。
