# 02. 嵌入式模型

嵌入式模型是 MongoDB 最重要的建模方式之一。它把相关数据放在同一个文档中，让一次读取就能拿到业务所需的数据。

## 1. 什么是嵌入式模型

嵌入式模型就是在一个文档里放入子文档或数组。

例如用户资料：

```javascript
{
  _id: ObjectId("..."),
  email: "alice@example.com",
  name: "Alice",
  profile: {
    city: "Shanghai",
    company: "Example Inc",
    bio: "Go backend developer"
  },
  settings: {
    language: "zh-CN",
    email_notification: true
  }
}
```

`profile` 和 `settings` 是嵌入在用户文档里的子文档。

## 2. 嵌入的优势

嵌入适合经常一起读取的数据。

优势：

- 一次查询拿到完整数据。
- 减少多集合查询。
- 单文档更新具备原子性。
- 数据局部性好。
- 模型更贴近业务聚合对象。

例如订单和订单项：

```javascript
{
  _id: ObjectId("..."),
  order_no: "O202607050001",
  user_id: ObjectId("..."),
  items: [
    {
      product_id: ObjectId("..."),
      product_name: "MongoDB 入门课",
      price: 9900,
      quantity: 1
    }
  ],
  total_amount: 9900,
  status: "paid"
}
```

查询订单详情时，订单项通常也要一起返回，所以嵌入很自然。

## 3. 适合嵌入的场景

### 一对一，生命周期一致

用户和用户设置：

```javascript
{
  email: "alice@example.com",
  settings: {
    theme: "dark",
    language: "zh-CN"
  }
}
```

用户设置通常跟用户一起读取，生命周期也一致。

### 一对少，数量有上限

用户收货地址：

```javascript
{
  _id: ObjectId("..."),
  name: "Alice",
  addresses: [
    {
      id: ObjectId("..."),
      receiver: "Alice",
      phone: "13800000000",
      province: "Shanghai",
      city: "Shanghai",
      detail: "xxx road",
      is_default: true
    }
  ]
}
```

如果业务限制最多 20 个地址，嵌入可以接受。

### 快照数据

订单中的商品快照：

```javascript
{
  order_no: "O202607050001",
  items: [
    {
      product_id: ObjectId("..."),
      product_name: "Go 后端课",
      sku_name: "标准版",
      price: 9900,
      quantity: 1
    }
  ]
}
```

商品改名后，历史订单不应该变化，所以快照嵌入非常合理。

## 4. 不适合嵌入的场景

### 数量无限增长

文章全部评论：

```javascript
{
  title: "热门文章",
  comments: [
    // 可能几万条
  ]
}
```

不适合全部嵌入。

### 子数据需要独立分页

评论列表通常需要：

```text
GET /posts/{id}/comments?page=1&page_size=20
```

这种场景适合 `comments` 单独集合。

### 子数据更新非常频繁

例如用户的所有登录日志，如果每次登录都更新用户文档，会造成热点写入和文档膨胀。

### 子数据生命周期不同

例如订单和退款申请。退款申请可能有独立状态流转、审核、日志，不一定适合完全嵌入订单。

## 5. 嵌入和单文档原子性

MongoDB 单文档写入是原子的。

例如更新订单状态和订单内某个字段：

```javascript
db.orders.updateOne(
  { _id: orderId },
  {
    $set: {
      status: "paid",
      paid_at: new Date(),
      updated_at: new Date()
    }
  }
)
```

如果相关数据在同一个文档里，经常可以避免事务。

但如果你把不该放一起的数据强行嵌入，只为了避免事务，后期会变成更大的维护问题。

## 6. 嵌入数组的更新

向数组追加地址：

```javascript
db.users.updateOne(
  { _id: userId },
  {
    $push: {
      addresses: {
        id: ObjectId(),
        receiver: "Alice",
        phone: "13800000000",
        city: "Shanghai"
      }
    },
    $set: { updated_at: new Date() }
  }
)
```

更新数组中的某个地址可以用 `$[identifier]`：

```javascript
db.users.updateOne(
  { _id: userId },
  {
    $set: {
      "addresses.$[addr].phone": "13900000000",
      updated_at: new Date()
    }
  },
  {
    arrayFilters: [
      { "addr.id": addressId }
    ]
  }
)
```

数组越复杂，更新语句越复杂。这也是判断是否嵌入的一个信号。

## 7. 嵌入的设计建议

建议：

- 嵌入经常一起读取的数据。
- 嵌入数量有限的数据。
- 嵌入生命周期一致的数据。
- 嵌入业务快照。
- 为数组元素设置内部 `id`，方便更新。
- 控制文档大小。

避免：

- 嵌入无限增长数据。
- 嵌入需要独立分页的数据。
- 嵌入高频独立更新的数据。
- 嵌入生命周期完全不同的数据。
- 嵌套层级过深。

## 8. 示例：用户收货地址

适合嵌入的版本：

```javascript
{
  _id: ObjectId("..."),
  email: "alice@example.com",
  name: "Alice",
  addresses: [
    {
      id: ObjectId("..."),
      receiver: "Alice",
      phone: "13800000000",
      province: "Shanghai",
      city: "Shanghai",
      detail: "xxx road",
      is_default: true,
      created_at: ISODate("2026-07-05T10:00:00Z")
    }
  ]
}
```

理由：

- 地址数量通常有限。
- 地址常和用户一起管理。
- 地址依赖用户存在。
- 用户删除后地址也可以删除。

如果业务变成“企业客户有几万个地址、地址需要复杂审批、地址被订单和发票系统独立引用”，就要考虑拆集合。

## 9. 示例：订单项

订单项适合嵌入：

```javascript
{
  _id: ObjectId("..."),
  order_no: "O202607050001",
  items: [
    {
      product_id: ObjectId("..."),
      sku_id: ObjectId("..."),
      product_name: "MongoDB 入门课",
      sku_name: "标准版",
      price: 9900,
      quantity: 1,
      subtotal: 9900
    }
  ],
  total_amount: 9900
}
```

理由：

- 订单详情必然需要订单项。
- 订单项是下单时的快照。
- 订单项数量通常有限。
- 订单项不需要脱离订单独立存在。

## 11. 本节小结

你需要记住：

- 嵌入适合一起读取、数量有限、生命周期一致的数据。
- 嵌入可以利用单文档原子性。
- 订单项、用户设置、商品规格经常适合嵌入。
- 评论、日志、流水等容易增长的数据要谨慎。
- 嵌入不是越多越好，更新复杂度和文档大小也要考虑。
