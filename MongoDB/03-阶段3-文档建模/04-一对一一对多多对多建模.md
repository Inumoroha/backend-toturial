# 04. 一对一、一对多、多对多建模

本节把常见关系放到 MongoDB 中分析。重点不是套模板，而是根据数量、访问方式、更新频率和生命周期选择嵌入、引用或混合方案。

## 1. 一对一

一对一常见例子：

- 用户和用户设置。
- 用户和个人资料。
- 商品和商品详情。
- 店铺和店铺配置。

### 方案 A：嵌入

```javascript
{
  _id: ObjectId("..."),
  email: "alice@example.com",
  profile: {
    display_name: "Alice",
    avatar: "https://example.com/a.png",
    city: "Shanghai"
  },
  settings: {
    language: "zh-CN",
    theme: "dark"
  }
}
```

适合：

- 总是和主对象一起读取。
- 生命周期一致。
- 数据量小。
- 更新频率不高。

### 方案 B：拆集合

```javascript
// users
{
  _id: ObjectId("..."),
  email: "alice@example.com"
}
```

```javascript
// user_profiles
{
  _id: ObjectId("..."),
  user_id: ObjectId("..."),
  display_name: "Alice",
  avatar: "https://example.com/a.png",
  city: "Shanghai"
}
```

适合：

- 资料很大。
- 资料很少被读取。
- 资料有独立权限或审核流程。
- 资料更新频率和主对象差异很大。

## 2. 一对少

一对少指子数据数量有限。

例子：

- 用户和收货地址。
- 商品和规格参数。
- 文章和精选评论。

嵌入常常适合。

```javascript
{
  _id: ObjectId("..."),
  name: "Alice",
  addresses: [
    {
      id: ObjectId("..."),
      receiver: "Alice",
      phone: "13800000000",
      city: "Shanghai",
      is_default: true
    }
  ]
}
```

判断关键：

- 数量是否有业务上限。
- 是否经常和主对象一起读取。
- 是否需要独立分页。
- 是否需要独立权限。

## 3. 一对多

一对多是最容易误判的关系。

例子：

- 文章和评论。
- 用户和订单。
- 商品和评价。
- 商品和库存流水。

如果子数据数量可能很大，通常用引用。

```javascript
// orders
{
  _id: ObjectId("..."),
  user_id: ObjectId("..."),
  order_no: "O202607050001",
  status: "paid",
  created_at: ISODate("2026-07-05T10:00:00Z")
}
```

按用户查订单：

```javascript
db.orders.find({ user_id: userId }).sort({ created_at: -1 }).limit(20)
```

索引：

```javascript
db.orders.createIndex({ user_id: 1, created_at: -1 })
```

## 4. 一对多的三种常见方案

### 方案 A：全部嵌入

适合数量少、有上限。

```javascript
{
  title: "文章",
  comments: [
    { user_id: ObjectId("..."), content: "good" }
  ]
}
```

### 方案 B：引用

适合数量多、需要分页。

```javascript
// comments
{
  post_id: ObjectId("..."),
  user_id: ObjectId("..."),
  content: "good"
}
```

### 方案 C：混合

文章文档嵌入少量精选评论，全部评论单独集合。

```javascript
{
  title: "文章",
  featured_comments: [
    {
      comment_id: ObjectId("..."),
      user_snapshot: { display_name: "Bob" },
      content: "精选评论"
    }
  ],
  stats: {
    comments: 120
  }
}
```

全部评论：

```javascript
// comments
{
  post_id: ObjectId("..."),
  content: "..."
}
```

这种方案很常见。

## 5. 多对多

多对多例子：

- 用户和角色。
- 文章和标签。
- 学生和课程。
- 商品和促销活动。

### 小规模多对多：数组嵌入 ID 或值

用户角色：

```javascript
{
  _id: ObjectId("..."),
  email: "alice@example.com",
  roles: ["user", "admin"]
}
```

文章标签：

```javascript
{
  _id: ObjectId("..."),
  title: "MongoDB 建模",
  tags: ["mongodb", "backend"]
}
```

适合：

- 数量少。
- 值简单。
- 不需要复杂关系属性。

### 复杂多对多：中间集合

学生选课：

```javascript
// enrollments
{
  _id: ObjectId("..."),
  student_id: ObjectId("..."),
  course_id: ObjectId("..."),
  status: "active",
  enrolled_at: ISODate("2026-07-05T10:00:00Z"),
  progress: 35
}
```

为什么不用数组？

- 选课关系有自己的属性：状态、进度、时间。
- 需要按学生查课程，也需要按课程查学生。
- 数量可能增长。
- 需要独立更新。

索引：

```javascript
db.enrollments.createIndex({ student_id: 1, enrolled_at: -1 })
db.enrollments.createIndex({ course_id: 1, status: 1 })
db.enrollments.createIndex({ student_id: 1, course_id: 1 }, { unique: true })
```

## 6. 父引用和子引用

有两种思路：

### 子文档引用父对象

订单保存 `user_id`：

```javascript
{
  user_id: ObjectId("..."),
  order_no: "O..."
}
```

适合按父对象查询大量子数据。

### 父文档保存子 ID 数组

用户保存最近订单：

```javascript
{
  recent_order_ids: [
    ObjectId("..."),
    ObjectId("...")
  ]
}
```

适合保存少量摘要或缓存，不适合保存全部订单 ID。

## 7. 关系建模决策表

| 关系 | 推荐方案 | 条件 |
| --- | --- | --- |
| 一对一 | 嵌入 | 经常一起读、体积小 |
| 一对一 | 引用 | 体积大、少读、独立权限 |
| 一对少 | 嵌入 | 数量有上限 |
| 一对多 | 引用 | 子数据多、分页、独立查 |
| 一对多 | 混合 | 需要摘要或精选数据 |
| 多对多简单 | 数组 | 数量少、无关系属性 |
| 多对多复杂 | 中间集合 | 有关系属性、双向查询 |

## 9. 本节小结

你需要记住：

- 一对一和一对少经常适合嵌入。
- 一对多要警惕无限增长。
- 多对多简单关系可以数组表达。
- 多对多复杂关系适合中间集合。
- 混合模型很常见：引用负责关系，冗余负责展示。
