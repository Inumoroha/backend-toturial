# 03. 引用式模型

引用式模型把相关数据放在不同集合中，通过 ID 建立关系。它更接近关系型数据库的外键思维，但 MongoDB 中的引用通常由应用层维护，不是数据库自动强制外键约束。

## 1. 什么是引用式模型

示例：文章和评论拆集合。

文章：

```javascript
// posts
{
  _id: ObjectId("..."),
  title: "MongoDB 建模",
  author_id: ObjectId("..."),
  status: "published"
}
```

评论：

```javascript
// comments
{
  _id: ObjectId("..."),
  post_id: ObjectId("..."),
  user_id: ObjectId("..."),
  content: "写得很好",
  status: "visible",
  created_at: ISODate("2026-07-05T10:00:00Z")
}
```

`comments.post_id` 引用 `posts._id`。

## 2. 引用的优势

引用适合：

- 子数据数量很多。
- 子数据需要独立分页。
- 子数据需要独立查询。
- 子数据更新频繁。
- 子数据生命周期不同。
- 多个父对象共享同一份数据。

例如评论：

- 评论可能很多。
- 评论需要分页。
- 评论可能审核、隐藏、删除。
- 评论可能按用户查询。

所以评论通常拆集合。

## 3. 引用的代价

引用会带来：

- 需要多次查询。
- 需要应用层维护关系一致性。
- 可能需要 `$lookup` 或聚合。
- 需要更多索引。
- 事务边界可能变大。

例如查询文章详情和评论：

```javascript
const post = db.posts.findOne({ slug: "mongodb-modeling" })
const comments = db.comments.find({ post_id: post._id }).sort({ created_at: -1 }).limit(20)
```

这是两步查询。

## 4. 两步查询不是坏事

初学者常以为“多查一次数据库就是错的”。这不一定。

如果评论数量巨大，而且需要分页，两步查询比把所有评论嵌入文章更健康。

两步查询的成本可以通过：

- 合理索引。
- 控制返回字段。
- 分页。
- 缓存。
- 冗余摘要字段。

来控制。

## 5. 什么时候用 $lookup

MongoDB 聚合管道支持 `$lookup` 做集合关联。

例如：

```javascript
db.posts.aggregate([
  { $match: { status: "published" } },
  {
    $lookup: {
      from: "users",
      localField: "author_id",
      foreignField: "_id",
      as: "author"
    }
  }
])
```

但是 `$lookup` 不是无脑替代 join。

适合：

- 后台报表。
- 中低频查询。
- 数据量可控的关联。
- 聚合分析。

谨慎用于：

- 高频列表接口。
- 大集合之间的无过滤关联。
- 高并发接口。

很多高频页面会选择冗余作者快照，而不是每次 `$lookup`。

## 6. 引用和冗余可以同时存在

文章可以同时保存：

```javascript
{
  _id: ObjectId("..."),
  author_id: ObjectId("..."),
  author_snapshot: {
    display_name: "Alice",
    avatar: "https://example.com/a.png"
  },
  title: "MongoDB 建模"
}
```

`author_id` 用于关系定位。

`author_snapshot` 用于列表展示。

这样列表不需要额外查用户，详情或后台需要完整用户信息时仍然可以根据 `author_id` 查询。

## 7. 引用关系的一致性

MongoDB 通常不会帮你强制外键。

你需要自己保证：

- 创建评论时，文章存在。
- 删除文章时，评论如何处理。
- 用户删除时，文章和评论如何处理。
- 商品删除时，历史订单是否保留。

常见策略：

### 拒绝删除

如果用户还有订单，不允许删除用户。

### 软删除

用户设置 `deleted_at`，历史订单仍然保留 `user_id`。

### 级联软删除

文章软删除时，评论也标记隐藏。

### 保留历史快照

商品删除后，订单中的商品快照仍然保留。

## 8. 常见引用字段命名

建议：

```text
user_id
author_id
post_id
product_id
order_id
sku_id
```

字段值通常使用 `ObjectId`。

不要混用：

```text
userId
uid
user_id
```

一个项目中要保持统一。

## 9. 引用字段需要索引

如果评论按文章查询：

```javascript
db.comments.find({ post_id: postId }).sort({ created_at: -1 })
```

应该考虑索引：

```javascript
db.comments.createIndex({ post_id: 1, created_at: -1 })
```

如果订单按用户查询：

```javascript
db.orders.find({ user_id: userId }).sort({ created_at: -1 })
```

应该考虑：

```javascript
db.orders.createIndex({ user_id: 1, created_at: -1 })
```

引用字段不建索引，数据量大后查询会很慢。

## 10. 示例：商品和库存流水

商品：

```javascript
{
  _id: ObjectId("..."),
  name: "MongoDB 入门课",
  stock: 100,
  status: "on_sale"
}
```

库存流水：

```javascript
{
  _id: ObjectId("..."),
  product_id: ObjectId("..."),
  type: "order_deduct",
  quantity: -1,
  before_stock: 100,
  after_stock: 99,
  order_id: ObjectId("..."),
  created_at: ISODate("2026-07-05T10:00:00Z")
}
```

流水适合拆集合。

理由：

- 数量持续增长。
- 需要按商品查询。
- 需要审计。
- 不适合塞进商品文档数组。

## 12. 本节小结

你需要记住：

- 引用适合大量、独立查询、独立更新、独立生命周期的数据。
- 引用通常需要应用层维护一致性。
- 两步查询并不一定是坏事。
- `$lookup` 有用，但不是高频接口的默认答案。
- 引用字段通常需要索引。
- 引用和冗余可以一起使用。
