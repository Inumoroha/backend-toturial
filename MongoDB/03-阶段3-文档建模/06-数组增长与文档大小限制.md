# 06. 数组增长与文档大小限制

MongoDB 文档可以包含数组，但数组不是无限容器。数组增长失控是 MongoDB 建模中最常见的坑之一。

## 1. BSON 文档大小限制

MongoDB 单个 BSON 文档最大大小是 16MiB。

这个限制意味着：

- 一个文档不能无限变大。
- 一个数组不能无限追加。
- 嵌入不是万能方案。

16MiB 对普通用户资料、订单、商品规格来说很大；但对评论、日志、消息、流水这种持续增长数据来说，迟早可能不够。

## 2. 无限增长数组是什么

无限增长数组指数组元素数量没有明确业务上限。

典型例子：

- 文章全部评论。
- 商品全部评价。
- 用户全部登录日志。
- 用户全部订单 ID。
- 商品全部库存流水。
- 聊天室全部消息。
- 设备全部上报记录。

这些数据不能简单嵌入主文档。

## 3. 为什么无限增长数组危险

风险：

- 文档越来越大。
- 更新越来越重。
- 网络传输变大。
- 查询可能返回不需要的大字段。
- 单文档热点写入。
- 最终可能触及 16MiB 限制。

例如：

```javascript
{
  _id: ObjectId("..."),
  title: "热门文章",
  comments: [
    // 持续追加
  ]
}
```

每新增一条评论都更新同一篇文章文档。热门文章会成为写热点。

## 4. 解决方案 A：拆集合

文章：

```javascript
{
  _id: ObjectId("..."),
  title: "热门文章",
  stats: {
    comments: 100000
  }
}
```

评论：

```javascript
{
  _id: ObjectId("..."),
  post_id: ObjectId("..."),
  user_id: ObjectId("..."),
  content: "评论内容",
  created_at: ISODate("2026-07-05T10:00:00Z")
}
```

查询评论：

```javascript
db.comments.find({ post_id: postId }).sort({ created_at: -1 }).limit(20)
```

索引：

```javascript
db.comments.createIndex({ post_id: 1, created_at: -1 })
```

这是最常见方案。

## 5. 解决方案 B：只嵌入最近 N 条

文章文档保存最近 3 条评论：

```javascript
{
  _id: ObjectId("..."),
  title: "文章",
  recent_comments: [
    {
      comment_id: ObjectId("..."),
      user_snapshot: { display_name: "Bob" },
      content: "不错",
      created_at: ISODate("2026-07-05T10:00:00Z")
    }
  ],
  stats: {
    comments: 120
  }
}
```

全部评论仍然在 `comments` 集合。

追加时限制数组长度：

```javascript
db.posts.updateOne(
  { _id: postId },
  {
    $push: {
      recent_comments: {
        $each: [newCommentSummary],
        $position: 0,
        $slice: 3
      }
    },
    $inc: { "stats.comments": 1 }
  }
)
```

适合首页卡片、详情页预览。

## 6. 解决方案 C：桶模式

如果数据是时间序列或日志，可以按时间或数量分桶。

例如设备上报：

```javascript
{
  _id: ObjectId("..."),
  device_id: "d1001",
  bucket_start: ISODate("2026-07-05T10:00:00Z"),
  bucket_end: ISODate("2026-07-05T11:00:00Z"),
  count: 3,
  readings: [
    { ts: ISODate("2026-07-05T10:01:00Z"), value: 23.5 },
    { ts: ISODate("2026-07-05T10:02:00Z"), value: 23.7 },
    { ts: ISODate("2026-07-05T10:03:00Z"), value: 23.6 }
  ]
}
```

桶模式适合：

- 传感器数据。
- 日志。
- 事件流。
- 时间序列数据。

注意：MongoDB 也有 time series collection，后面工程化阶段可以单独学习。

## 7. 解决方案 D：Outlier Pattern

大多数用户地址很少，但极少数企业用户地址非常多。

普通用户嵌入地址：

```javascript
{
  _id: ObjectId("..."),
  type: "normal",
  addresses: [
    { id: ObjectId("..."), city: "Shanghai" }
  ]
}
```

超大用户：

```javascript
{
  _id: ObjectId("..."),
  type: "enterprise",
  has_external_addresses: true
}
```

地址拆集合：

```javascript
{
  user_id: ObjectId("..."),
  receiver: "Alice",
  city: "Shanghai"
}
```

这种模式处理少数异常大数据，避免为了极端情况牺牲所有普通情况。

## 8. 数组字段设计建议

建议：

- 明确数组最大数量。
- 为数组元素设置 `id`。
- 对展示型数组使用 `$slice` 限制长度。
- 大数组拆集合。
- 高频追加数据拆集合或分桶。
- 不要把所有历史记录塞进主文档。

## 9. 列表接口避免返回大数组

即使文档里有数组，列表接口也不一定要返回。

投影排除：

```javascript
db.posts.find(
  { status: "published" },
  { content: 0, recent_comments: 0 }
)
```

Go 中：

```go
opts := options.Find().SetProjection(bson.D{
    {"content", 0},
    {"recent_comments", 0},
})
```

接口返回字段越少，网络和解码成本越低。

## 11. 本节小结

你需要记住：

- MongoDB 单文档最大 BSON 大小是 16MiB。
- 无限增长数组是建模高危点。
- 评论、日志、流水、消息通常不要全部嵌入。
- 最近 N 条可以嵌入，全部数据拆集合。
- 分桶适合时间序列和日志类数据。
- Outlier Pattern 适合少数异常大的对象。
