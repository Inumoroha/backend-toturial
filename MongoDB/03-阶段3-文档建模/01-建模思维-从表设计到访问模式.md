# 01. 建模思维：从表设计到访问模式

很多人学 MongoDB 的第一个坑，是把它当成“可以存 JSON 的 MySQL”。这种理解会让你照搬表结构，然后再到处 `$lookup`，最后得到一个不好用的 MongoDB。

MongoDB 建模的核心问题不是“这个实体应该建几张表”，而是：

```text
业务最常怎样读取和写入这些数据？
```

## 1. 关系型数据库的常见思路

以电商订单为例，关系型数据库通常会拆成：

```text
users
products
orders
order_items
payments
inventory_records
```

拆表的主要原因：

- 消除重复。
- 用外键表达关系。
- 保持范式化。
- 通过 join 组合查询结果。

这种思维在关系型数据库里很自然。

## 2. MongoDB 的建模思路

MongoDB 更关心数据如何被使用。

仍然以订单为例。用户查看订单详情时，通常需要：

- 订单基本信息。
- 下单用户 ID。
- 商品名称快照。
- 商品价格快照。
- 商品数量。
- 总金额。
- 支付状态。
- 下单时间。

这些数据经常一起读取，就可以考虑放在一个订单文档里：

```javascript
{
  _id: ObjectId("..."),
  user_id: ObjectId("..."),
  order_no: "O202607050001",
  status: "paid",
  items: [
    {
      product_id: ObjectId("..."),
      product_name: "MongoDB 入门课",
      sku_name: "标准版",
      price: 9900,
      quantity: 1
    }
  ],
  total_amount: 9900,
  payment: {
    method: "alipay",
    paid_at: ISODate("2026-07-05T10:00:00Z")
  },
  created_at: ISODate("2026-07-05T09:59:00Z")
}
```

这里的 `product_name` 和 `price` 是快照，而不是每次都去 `products` 集合 join。

## 3. 为什么访问模式优先

数据库模型最终服务于接口。

假设你有这些接口：

```text
GET /users/{id}
GET /users/{id}/orders
GET /orders/{id}
POST /orders
PATCH /orders/{id}/status
GET /admin/orders?status=paid
```

你应该先写出每个接口需要的数据：

| 接口 | 需要读取什么 | 查询条件 | 排序 |
| --- | --- | --- | --- |
| 用户详情 | 用户基本资料 | `_id` | 无 |
| 用户订单列表 | 订单摘要 | `user_id` | `created_at desc` |
| 订单详情 | 订单和商品快照 | `_id` | 无 |
| 创建订单 | 商品、库存、用户、购物车 | 多个 ID | 无 |
| 修改订单状态 | 订单状态 | `_id` | 无 |
| 后台订单列表 | 订单摘要 | `status` | `created_at desc` |

这些就是访问模式。

MongoDB 建模时，要优先让高频访问路径简单、高效、稳定。

## 4. 先列查询，再画集合

建模前先列问题：

### 读取问题

- 哪些数据经常一起读取？
- 哪些接口是高频接口？
- 列表页需要哪些字段？
- 详情页需要哪些字段？
- 是否需要按时间排序？
- 是否需要按状态筛选？

### 写入问题

- 哪些字段经常更新？
- 是否会并发更新？
- 是否需要原子更新？
- 是否需要事务？
- 写入是否高频？

### 增长问题

- 这个数组会不会无限增长？
- 这类数据每天增长多少？
- 是否需要归档？
- 是否需要 TTL？

### 生命周期问题

- 这类数据能删除吗？
- 是软删除还是物理删除？
- 是否需要审计？
- 是否冷热分层？

## 5. 一个博客系统例子

业务对象：

```text
users
posts
comments
```

常见接口：

```text
GET /posts
GET /posts/{slug}
GET /users/{id}/posts
GET /posts/{id}/comments
POST /posts/{id}/comments
```

可能模型：

```javascript
// posts
{
  _id: ObjectId("..."),
  author_id: ObjectId("..."),
  author_snapshot: {
    display_name: "Alice",
    avatar: "https://example.com/a.png"
  },
  title: "MongoDB 建模入门",
  slug: "mongodb-modeling-intro",
  content: "...",
  tags: ["mongodb", "backend"],
  stats: {
    views: 120,
    likes: 15,
    comments: 8
  },
  status: "published",
  published_at: ISODate("2026-07-05T10:00:00Z")
}
```

评论单独集合：

```javascript
// comments
{
  _id: ObjectId("..."),
  post_id: ObjectId("..."),
  user_id: ObjectId("..."),
  user_snapshot: {
    display_name: "Bob",
    avatar: "https://example.com/b.png"
  },
  content: "写得不错",
  status: "visible",
  created_at: ISODate("2026-07-05T11:00:00Z")
}
```

为什么评论不直接全部嵌入文章？

因为评论可能无限增长，并且评论列表通常独立分页。

为什么文章里冗余 `author_snapshot`？

因为文章列表经常要展示作者昵称和头像，不想每条文章都额外查用户。

## 6. 不要机械追求范式化

MongoDB 中适当冗余是常见设计。

例如订单中的商品名称和价格：

```javascript
items: [
  {
    product_id: ObjectId("..."),
    product_name: "Go 后端课",
    price: 9900
  }
]
```

即使商品名称以后改了，历史订单仍然应该保留下单时的商品名称和价格。这不是脏数据，而是业务快照。

## 7. 也不要无脑反范式

冗余有成本：

- 数据可能不一致。
- 更新逻辑更复杂。
- 需要同步策略。
- 文档可能变大。

例如用户昵称冗余到评论里。如果用户改昵称，历史评论是否要同步？

可能答案：

- 不同步：评论展示历史昵称。
- 同步：所有历史评论都显示新昵称。
- 部分同步：只同步头像，不同步昵称。

这不是数据库语法问题，而是业务规则问题。

## 8. 建模产物应该包括什么

一个合格的 MongoDB 建模文档至少包括：

- 集合列表。
- 每个集合的样例文档。
- 字段说明。
- 核心查询场景。
- 主要索引建议。
- 嵌入/引用/冗余的理由。
- 数据生命周期。
- 一致性和同步策略。
- 潜在风险。

后面的电商练习会按这个格式输出。

## 9. 本节练习

选择一个你熟悉的系统，例如：

- 博客。
- 电商。
- 在线课程。
- 任务管理。
- 聊天系统。

写出：

1. 5 个核心业务对象。
2. 10 个核心接口。
3. 每个接口的查询条件。
4. 每个接口需要返回的字段。
5. 哪些数据经常一起读取。
6. 哪些数据会无限增长。

现在先不要急着画集合。先把访问模式写清楚。

## 10. 本节小结

你需要记住：

- MongoDB 建模先看访问模式。
- 经常一起读取的数据可以考虑嵌入。
- 会无限增长、独立查询、独立更新的数据通常适合拆集合。
- 冗余不是坏事，但必须有同步策略。
- 建模不是一锤子买卖，模型会随着业务和数据规模演进。
