# 10. 阶段练习：电商订单系统建模

本练习是第 3 阶段的核心产出。你要为一个电商订单系统设计 MongoDB 模型，并说明为什么这样设计。

## 1. 业务范围

需要建模：

- 用户。
- 商品。
- SKU。
- 购物车。
- 订单。
- 订单明细。
- 支付记录。
- 库存流水。

暂不深入：

- 优惠券。
- 售后退款。
- 发票。
- 物流轨迹。
- 商家多租户。

这些可以后续扩展。

## 2. 核心查询场景

先列访问模式。

### 用户

1. 根据用户 ID 查询用户详情。
2. 根据邮箱查询用户。
3. 查询用户默认收货地址。
4. 更新用户资料。
5. 软删除用户。

### 商品

1. 查询商品详情。
2. 按分类查询商品列表。
3. 按状态查询上架商品。
4. 查询某个 SKU。
5. 更新商品库存。

### 购物车

1. 查询用户购物车。
2. 添加商品到购物车。
3. 修改购物车商品数量。
4. 删除购物车商品。
5. 勾选部分商品下单。

### 订单

1. 创建订单。
2. 查询订单详情。
3. 查询用户订单列表。
4. 后台按状态查询订单列表。
5. 修改订单状态。

### 支付

1. 创建支付记录。
2. 根据订单查询支付记录。
3. 根据支付流水号查询支付记录。
4. 更新支付状态。
5. 查询支付失败记录。

### 库存流水

1. 下单扣减库存记录。
2. 取消订单回滚库存记录。
3. 按商品查询库存流水。
4. 按订单查询库存流水。
5. 审计某次库存变化。

## 3. 关系型思路

关系型数据库通常拆成：

```text
users
user_addresses
products
skus
carts
cart_items
orders
order_items
payments
inventory_records
```

优点：

- 结构规范。
- 重复少。
- 关系清晰。

但在 MongoDB 中，如果机械照搬，就会导致订单详情、购物车、商品展示等接口需要多次关联。

## 4. MongoDB 集合设计

建议集合：

```text
users
products
carts
orders
payments
inventory_records
```

其中：

- 用户地址嵌入 `users`。
- SKU 嵌入 `products`。
- 购物车商品嵌入 `carts`。
- 订单明细嵌入 `orders`。
- 支付记录拆集合。
- 库存流水拆集合。

## 5. users 集合

```javascript
{
  _id: ObjectId("..."),
  email: "alice@example.com",
  name: "Alice",
  phone: "13800000000",
  status: "active",
  addresses: [
    {
      id: ObjectId("..."),
      receiver: "Alice",
      phone: "13800000000",
      province: "Shanghai",
      city: "Shanghai",
      detail: "xxx road",
      is_default: true
    }
  ],
  created_at: ISODate("2026-07-05T10:00:00Z"),
  updated_at: ISODate("2026-07-05T10:00:00Z")
}
```

理由：

- 地址数量通常有限。
- 地址跟用户生命周期强相关。
- 用户管理地址时经常一起读取。

风险：

- 企业用户地址很多时需要 Outlier Pattern。

核心索引：

```javascript
db.users.createIndex({ email: 1 }, { unique: true })
db.users.createIndex({ phone: 1 }, { unique: true, sparse: true })
```

## 6. products 集合

```javascript
{
  _id: ObjectId("..."),
  name: "MongoDB 入门课",
  category_id: ObjectId("..."),
  status: "on_sale",
  cover: "https://example.com/p.png",
  description: "适合 Go 后端工程师学习 MongoDB",
  tags: ["mongodb", "backend"],
  skus: [
    {
      id: ObjectId("..."),
      name: "标准版",
      price: 9900,
      stock: 100,
      attrs: {
        duration: "12h"
      }
    }
  ],
  stats: {
    sales: 120,
    reviews: 20
  },
  created_at: ISODate("2026-07-05T10:00:00Z"),
  updated_at: ISODate("2026-07-05T10:00:00Z")
}
```

理由：

- SKU 数量通常有限。
- 商品详情需要展示 SKU。
- 商品规格参数适合嵌入。

风险：

- SKU 库存高并发更新时，要仔细设计原子更新条件。
- 如果 SKU 非常多，可能需要拆 `skus` 集合。

核心索引：

```javascript
db.products.createIndex({ category_id: 1, status: 1, created_at: -1 })
db.products.createIndex({ tags: 1 })
```

## 7. carts 集合

一个用户一个购物车：

```javascript
{
  _id: ObjectId("..."),
  user_id: ObjectId("..."),
  items: [
    {
      product_id: ObjectId("..."),
      sku_id: ObjectId("..."),
      product_name: "MongoDB 入门课",
      sku_name: "标准版",
      cover: "https://example.com/p.png",
      price: 9900,
      quantity: 1,
      selected: true,
      added_at: ISODate("2026-07-05T10:00:00Z")
    }
  ],
  updated_at: ISODate("2026-07-05T10:00:00Z")
}
```

理由：

- 购物车商品数量通常有限。
- 查询购物车需要一次返回商品摘要。
- 价格和名称可作为展示快照，但下单时必须重新校验商品状态和价格。

索引：

```javascript
db.carts.createIndex({ user_id: 1 }, { unique: true })
```

重要提醒：

购物车中的 `price` 只是展示，不应作为最终下单价格。创建订单时要重新读取商品和 SKU。

## 8. orders 集合

```javascript
{
  _id: ObjectId("..."),
  order_no: "O202607050001",
  user_id: ObjectId("..."),
  user_snapshot: {
    name: "Alice",
    phone: "13800000000"
  },
  receiver_snapshot: {
    receiver: "Alice",
    phone: "13800000000",
    province: "Shanghai",
    city: "Shanghai",
    detail: "xxx road"
  },
  status: "pending_payment",
  items: [
    {
      product_id: ObjectId("..."),
      sku_id: ObjectId("..."),
      product_name: "MongoDB 入门课",
      sku_name: "标准版",
      cover: "https://example.com/p.png",
      price: 9900,
      quantity: 1,
      subtotal: 9900
    }
  ],
  total_amount: 9900,
  paid_amount: 0,
  created_at: ISODate("2026-07-05T10:00:00Z"),
  updated_at: ISODate("2026-07-05T10:00:00Z")
}
```

理由：

- 订单明细是历史快照。
- 订单详情必须展示下单时商品名、价格、收货地址。
- 订单项数量通常有限。

索引：

```javascript
db.orders.createIndex({ order_no: 1 }, { unique: true })
db.orders.createIndex({ user_id: 1, created_at: -1 })
db.orders.createIndex({ status: 1, created_at: -1 })
```

状态更新必须带条件：

```javascript
db.orders.updateOne(
  { _id: orderId, status: "pending_payment" },
  {
    $set: {
      status: "paid",
      paid_at: new Date(),
      updated_at: new Date()
    }
  }
)
```

## 9. payments 集合

```javascript
{
  _id: ObjectId("..."),
  payment_no: "P202607050001",
  order_id: ObjectId("..."),
  order_no: "O202607050001",
  user_id: ObjectId("..."),
  method: "alipay",
  amount: 9900,
  status: "success",
  third_party_trade_no: "20260705220001",
  paid_at: ISODate("2026-07-05T10:03:00Z"),
  created_at: ISODate("2026-07-05T10:02:00Z"),
  updated_at: ISODate("2026-07-05T10:03:00Z")
}
```

为什么拆集合？

- 支付记录有独立状态。
- 可能有多次支付尝试。
- 支付需要按流水号查询。
- 支付数据有审计价值。

索引：

```javascript
db.payments.createIndex({ payment_no: 1 }, { unique: true })
db.payments.createIndex({ order_id: 1, created_at: -1 })
db.payments.createIndex({ third_party_trade_no: 1 }, { unique: true, sparse: true })
```

## 10. inventory_records 集合

```javascript
{
  _id: ObjectId("..."),
  product_id: ObjectId("..."),
  sku_id: ObjectId("..."),
  order_id: ObjectId("..."),
  type: "order_deduct",
  quantity: -1,
  before_stock: 100,
  after_stock: 99,
  reason: "create_order",
  created_at: ISODate("2026-07-05T10:00:00Z")
}
```

为什么拆集合？

- 库存流水持续增长。
- 需要审计。
- 需要按商品、SKU、订单查询。
- 不适合嵌入商品文档。

索引：

```javascript
db.inventory_records.createIndex({ product_id: 1, created_at: -1 })
db.inventory_records.createIndex({ sku_id: 1, created_at: -1 })
db.inventory_records.createIndex({ order_id: 1 })
```

## 11. 下单流程中的一致性

创建订单通常涉及：

```text
1. 读取用户和地址
2. 读取商品和 SKU
3. 校验商品状态、价格、库存
4. 扣减库存
5. 创建订单
6. 写库存流水
7. 清理购物车选中项
```

这可能需要事务，也可能通过单文档原子更新 + 幂等键设计实现。

第 6 阶段会系统讲事务。本阶段先把模型设计清楚。

库存扣减示例：

```javascript
db.products.updateOne(
  {
    _id: productId,
    "skus.id": skuId,
    "skus.stock": { $gte: quantity }
  },
  {
    $inc: { "skus.$.stock": -quantity },
    $set: { updated_at: new Date() }
  }
)
```

注意：数组中 SKU 的并发扣减需要认真测试。

## 14. 本节小结

这个电商模型不是唯一答案，但它体现了 MongoDB 建模的核心思想：

- 订单项嵌入订单，因为它们是订单快照。
- 支付记录拆集合，因为它有独立状态和审计价值。
- 库存流水拆集合，因为它持续增长。
- 购物车嵌入商品摘要，但下单时必须重新校验价格和库存。
- 冗余字段必须标明是快照、展示还是统计。
