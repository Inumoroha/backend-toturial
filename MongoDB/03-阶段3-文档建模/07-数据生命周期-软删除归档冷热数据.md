# 07. 数据生命周期：软删除、归档、冷热数据

建模不只是字段怎么放，还包括数据从创建到删除的完整生命周期。真实系统中的数据会经历创建、修改、状态流转、软删除、归档、清理、审计等过程。

## 1. 为什么要设计生命周期

如果不提前设计，后面会遇到：

- 删除后数据找不回。
- 列表查到了已删除数据。
- 旧数据越来越多，查询越来越慢。
- 审计要求无法满足。
- 归档逻辑混乱。
- 热数据和冷数据混在一起。

生命周期字段不是可有可无，它会影响查询、索引、归档和权限。

## 2. 常见生命周期字段

```javascript
{
  status: "active",
  created_at: ISODate("2026-07-05T10:00:00Z"),
  updated_at: ISODate("2026-07-05T10:00:00Z"),
  deleted_at: null,
  archived_at: null
}
```

建议：

- `created_at`：创建时间。
- `updated_at`：最后更新时间。
- `deleted_at`：软删除时间。
- `archived_at`：归档时间。
- `status`：业务状态。

未删除时，很多团队会选择不存 `deleted_at` 字段，而不是存 `null`。

这样查询：

```javascript
{ deleted_at: { $exists: false } }
```

## 3. 软删除

软删除不是删除文档，而是标记删除状态。

```javascript
db.users.updateOne(
  { _id: userId },
  {
    $set: {
      deleted_at: new Date(),
      updated_at: new Date()
    }
  }
)
```

查询未删除：

```javascript
db.users.find({
  deleted_at: { $exists: false }
})
```

优点：

- 可恢复。
- 保留审计。
- 避免误删不可逆。

缺点：

- 查询必须加过滤条件。
- 数据量会增长。
- 唯一索引要特殊设计。

## 4. 软删除和唯一索引

假设用户邮箱唯一。

如果用户软删除后，是否允许重新注册同一个邮箱？

### 不允许重新注册

普通唯一索引即可：

```javascript
db.users.createIndex({ email: 1 }, { unique: true })
```

软删除用户仍占用邮箱。

### 允许重新注册

可以考虑部分唯一索引：

```javascript
db.users.createIndex(
  { email: 1 },
  {
    unique: true,
    partialFilterExpression: {
      deleted_at: { $exists: false }
    }
  }
)
```

这表示只对未删除用户要求邮箱唯一。

第 5 阶段索引会详细讲部分索引。

## 5. 状态流转

很多业务不是简单删除，而是状态流转。

订单状态：

```text
pending_payment -> paid -> shipped -> completed
pending_payment -> cancelled
paid -> refunding -> refunded
```

文档：

```javascript
{
  order_no: "O202607050001",
  status: "paid",
  paid_at: ISODate("2026-07-05T10:00:00Z"),
  shipped_at: null,
  completed_at: null,
  cancelled_at: null
}
```

状态更新要带条件：

```javascript
db.orders.updateOne(
  { _id: orderId, status: "pending_payment" },
  {
    $set: {
      status: "paid",
      paid_at: new Date(),
      updated_at: new Date()
    }
  }
)
```

不要无条件改状态。

## 6. 归档

归档是把不常访问的旧数据从热路径中移走。

常见方案：

### 同集合标记归档

```javascript
{
  status: "archived",
  archived_at: ISODate("2026-07-05T10:00:00Z")
}
```

查询热数据时过滤：

```javascript
{ archived_at: { $exists: false } }
```

### 拆归档集合

```text
orders
orders_archive
```

旧订单迁移到 `orders_archive`。

优点：

- 热集合更小。
- 热查询更稳定。

缺点：

- 查询历史数据要查归档集合。
- 迁移和一致性更复杂。

## 7. 冷热数据

热数据：

- 最近订单。
- 活跃用户。
- 最近评论。
- 当前库存。

冷数据：

- 多年前订单。
- 历史日志。
- 已完成很久的工单。
- 过期会话。

设计时要问：

- 热数据保留多久？
- 冷数据是否还需要在线查询？
- 冷数据查询频率多高？
- 是否需要导出到数据仓库？
- 是否需要压缩或归档？

## 8. TTL 自动过期

MongoDB 支持 TTL 索引自动删除过期文档。

适合：

- 验证码。
- 临时 token。
- 过期会话。
- 临时任务锁。

示例：

```javascript
db.verify_codes.createIndex(
  { expired_at: 1 },
  { expireAfterSeconds: 0 }
)
```

文档：

```javascript
{
  email: "alice@example.com",
  code: "123456",
  expired_at: ISODate("2026-07-05T10:05:00Z")
}
```

不适合：

- 订单。
- 支付流水。
- 审计日志。

这些数据通常不能自动删除。

## 9. 审计日志

重要状态变化建议记录审计日志。

```javascript
{
  _id: ObjectId("..."),
  target_type: "order",
  target_id: ObjectId("..."),
  action: "status_changed",
  from_status: "pending_payment",
  to_status: "paid",
  operator_id: ObjectId("..."),
  created_at: ISODate("2026-07-05T10:00:00Z")
}
```

审计日志通常拆集合。

理由：

- 持续增长。
- 不适合嵌入业务文档。
- 查询模式独立。
- 可能需要长期保留。

## 11. 本节小结

你需要记住：

- 生命周期字段会影响查询、索引和归档。
- 软删除适合需要恢复或审计的数据。
- TTL 适合临时数据，不适合订单和支付流水。
- 状态更新要带状态条件，避免非法流转。
- 冷热数据要提前规划，不要等集合巨大后再补。
