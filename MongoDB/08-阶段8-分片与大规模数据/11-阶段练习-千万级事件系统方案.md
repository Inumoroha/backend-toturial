# 11. 阶段练习：千万级事件系统方案

本练习是第 8 阶段的综合产出。你要设计一个每天千万级写入的事件系统，并说明 shard key、索引、TTL 和热点风险。

## 1. 业务背景

系统每天接收：

```text
1000 万到 5000 万事件
```

需要支持：

- 按租户查询。
- 按用户查询。
- 按设备查询。
- 按时间范围查询。
- 最近事件列表。
- 事件类型统计。
- 保留 180 天。
- Go 后端批量写入。

## 3. 数据规模估算

示例：

```text
每天 3000 万事件
平均每条 1KB
每天原始数据约 30GB
保留 180 天约 5.4TB
再加索引空间，实际可能明显更大
```

这只是粗估。真实估算要用样本文档大小：

```javascript
Object.bsonsize(db.events.findOne())
```

## 4. 文档模型要求

至少包含：

```javascript
{
  tenant_id: "t1001",
  user_id: ObjectId("..."),
  device_id: "d1001",
  event_type: "click",
  properties: {},
  created_at: ISODate("..."),
  expired_at: ISODate("...")
}
```

说明哪些字段必填，哪些字段可选。

## 5. shard key 候选要求

至少比较三个方案：

```text
{ tenant_id: 1, user_id: 1 }
{ user_id: "hashed" }
{ tenant_id: 1, created_at: 1 }
```

每个方案写：

- 数据分布。
- 写入热点。
- 用户查询。
- 租户时间范围查询。
- 全局统计。
- 唯一索引影响。

## 6. 索引要求

至少设计：

```javascript
db.events.createIndex({ tenant_id: 1, user_id: 1, created_at: -1 })
db.events.createIndex({ tenant_id: 1, device_id: 1, created_at: -1 })
db.events.createIndex({ tenant_id: 1, event_type: 1, created_at: -1 })
db.events.createIndex({ expired_at: 1 }, { expireAfterSeconds: 0 })
```

然后评估：

- 哪些是必须索引？
- 哪些可能拖慢写入？
- 是否可以删减？

## 7. 查询路由分析

对每个查询判断：

```text
targeted / scatter-gather
```

例如：

```javascript
{ tenant_id: "t1", user_id: userId }
```

如果 shard key 是 `{ tenant_id: 1, user_id: 1 }`，通常更可能 targeted。

如果查询：

```javascript
{ event_type: "click" }
```

可能 scatter。

## 8. 热点风险

必须分析：

- 大租户。
- 超级用户。
- 高频设备。
- 时间递增写入。
- 单事件类型过热。

写出缓解方案：

- 哈希 key。
- 大租户单独拆分。
- 队列削峰。
- 批量写入。
- 离线统计。

## 9. Go 后端要求

设计方法：

```go
InsertEventsBatch(ctx context.Context, events []Event) error
ListUserEvents(ctx context.Context, tenantID string, userID bson.ObjectID, cursor *EventCursor, limit int64) ([]Event, error)
ListDeviceEvents(ctx context.Context, tenantID string, deviceID string, cursor *EventCursor, limit int64) ([]Event, error)
CountEventsByType(ctx context.Context, tenantID string, start, end time.Time) ([]EventTypeCount, error)
```

说明：

- 是否包含 shard key。
- 使用什么索引。
- 是否需要游标分页。
- 是否可能跨 shard。

## 10. 监控指标

至少监控：

- 每秒写入量。
- 批量写入失败率。
- 各 shard 数据量。
- 各 shard QPS。
- chunk 迁移。
- 慢查询。
- TTL 删除滞后。
- 大租户写入占比。
- 磁盘使用率。
- 索引大小。

## 11. 阶段验收

完成后你应该能讲清楚：

- 为什么这个系统可能需要分片。
- 为什么你选择这个 shard key。
- 这个 shard key 的缺点。
- 哪些查询会 targeted。
- 哪些查询会 scatter。
- TTL 如何清理数据。
- Go 后端如何批量写入。
- 如何监控热点。
