# 09. 大规模日志/埋点系统设计

本节设计一个日志/埋点系统：每天千万级写入，支持按用户查询、按时间范围查询，并支持数据过期。

## 1. 业务需求

```text
每天写入 1000 万到 5000 万事件
保留 180 天
按用户查询最近事件
按时间范围统计事件类型
支持 TTL 自动过期
Go 后端批量写入
```

事件类型：

- login
- view_page
- click
- create_order
- pay_order

## 2. 文档模型

```javascript
{
  _id: ObjectId("..."),
  tenant_id: "t1001",
  user_id: ObjectId("..."),
  device_id: "d1001",
  event_type: "click",
  properties: {
    page: "/products/1",
    button: "buy"
  },
  ip: "127.0.0.1",
  user_agent: "Mozilla/5.0",
  created_at: ISODate("2026-07-05T10:00:00Z"),
  expired_at: ISODate("2027-01-01T00:00:00Z")
}
```

字段说明：

- `tenant_id`：租户。
- `user_id`：用户。
- `device_id`：设备。
- `event_type`：事件类型。
- `properties`：事件扩展属性。
- `created_at`：事件时间。
- `expired_at`：过期时间。

## 3. 查询模式

核心查询：

### 用户最近事件

```javascript
db.events.find({
  tenant_id: "t1001",
  user_id: userId
}).sort({ created_at: -1 }).limit(50)
```

### 用户时间范围事件

```javascript
db.events.find({
  tenant_id: "t1001",
  user_id: userId,
  created_at: { $gte: start, $lt: end }
})
```

### 按事件类型统计

```javascript
db.events.aggregate([
  {
    $match: {
      tenant_id: "t1001",
      created_at: { $gte: start, $lt: end }
    }
  },
  {
    $group: {
      _id: "$event_type",
      count: { $sum: 1 }
    }
  }
])
```

### 设备最近事件

```javascript
db.events.find({
  tenant_id: "t1001",
  device_id: "d1001"
}).sort({ created_at: -1 }).limit(50)
```

## 4. shard key 候选

### 方案 A：`{ tenant_id: 1, user_id: 1 }`

优点：

- 租户 + 用户查询定向。
- 适合用户行为查询。
- user_id 基数高。

风险：

- 大租户中超级用户可能热点。
- 按租户全局时间统计可能跨 shard。

### 方案 B：`{ user_id: "hashed" }`

优点：

- 写入分布更均匀。
- 用户等值查询可定向。

风险：

- 多租户隔离弱。
- 按 tenant_id 查询可能 scatter。
- 按时间范围统计跨 shard。

### 方案 C：`{ tenant_id: 1, created_at: 1 }`

优点：

- 租户 + 时间范围查询友好。

风险：

- 大租户写入集中。
- created_at 单调递增，可能热点。

## 5. 推荐初始方案

如果核心查询是“租户内按用户查事件”：

```javascript
{ tenant_id: 1, user_id: 1 }
```

如果写入分布验证后仍有热点，可以考虑：

- 引入更分散维度。
- 对 user_id 做哈希策略。
- 大租户单独拆分。
- 离线统计承担全局时间报表。

## 6. 索引设计

用户最近事件：

```javascript
db.events.createIndex({ tenant_id: 1, user_id: 1, created_at: -1 })
```

设备最近事件：

```javascript
db.events.createIndex({ tenant_id: 1, device_id: 1, created_at: -1 })
```

事件统计：

```javascript
db.events.createIndex({ tenant_id: 1, event_type: 1, created_at: -1 })
```

TTL：

```javascript
db.events.createIndex({ expired_at: 1 }, { expireAfterSeconds: 0 })
```

注意：索引越多，写入越慢。事件系统写入很高，索引必须克制。

## 7. TTL 和归档

如果事件只保留 180 天：

```javascript
expired_at = created_at + 180 days
```

用 TTL 自动删除。

如果需要长期分析：

- MongoDB 保存热数据。
- 定期导出到数据湖/数仓。
- 冷数据走离线查询。

不要把所有历史分析都压在在线 MongoDB 上。

## 8. 批量写入

Go 后端应批量写：

```text
每批 500 到 2000 条，按事件大小和延迟要求压测决定。
```

使用：

- `InsertMany`
- `BulkWrite`
- 异步缓冲
- 队列削峰

## 9. 写入可靠性

埋点通常可以接受少量丢失，业务事件不能随便丢。

区分：

### 普通行为埋点

- 可批量。
- 可异步。
- 可接受少量丢失。

### 关键业务事件

- 下单。
- 支付。
- 风控。

需要更可靠：

- 消息队列。
- 幂等 ID。
- 重试。
- 死信。
- 审计。

## 11. 本节小结

你需要记住：

- 日志系统写入高，索引要克制。
- shard key 要结合查询和写入分布。
- TTL 适合过期事件清理。
- 全局报表可能更适合离线系统。
- 普通埋点和关键业务事件可靠性要求不同。
