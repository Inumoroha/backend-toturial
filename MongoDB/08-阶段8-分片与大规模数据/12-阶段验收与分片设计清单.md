## 3. 分片前评审清单

决定分片前问：

- 数据量是否真的超过单复制集能力？
- 写入吞吐是否真的超过单 primary 能力？
- 是否已经优化索引？
- 是否已经优化模型？
- 是否已经优化分页？
- 是否已经做冷热归档？
- 是否有足够运维能力？
- 是否有备份恢复方案？
- 是否有明确 shard key？
- 是否压测过数据分布和写入分布？

## 4. shard key 评审清单

选择 shard key 前问：

- 基数是否足够高？
- 数据是否分布均匀？
- 高频查询是否包含 shard key？
- 写入是否会集中到单个 chunk？
- 是否存在大租户或超级用户？
- 是否影响唯一索引？
- 是否支持核心范围查询？
- 是否会导致大量 scatter-gather？
- 是否压测验证过？

## 5. 查询评审清单

每个查询问：

- 是否包含 shard key？
- 是 targeted 还是 scatter-gather？
- 是否有 limit？
- 是否有排序？
- 排序是否有索引支持？
- 是否跨 shard 聚合？
- 是否能转成离线统计？
- 是否会被高频调用？

## 6. 常见错误

### 错误 1：慢查询就上分片

正确做法：先 explain、索引、建模、分页优化。

### 错误 2：用 created_at 做 shard key

持续写入容易落到最新 chunk，形成热点。

### 错误 3：用 status 做 shard key

低基数字段，数据分布差。

### 错误 4：高频查询不带 shard key

导致 scatter-gather。

### 错误 5：忽略唯一索引限制

分片集合唯一索引要结合 shard key 设计。

### 错误 6：索引太多

日志系统写入高，索引过多会严重拖慢写入。

## 11. 实战项目验收

建议你实现一个“事件采集系统”作为阶段项目。

### 基础要求

- Go HTTP API 接收事件。
- MongoDB 保存事件。
- 文档包含 `tenant_id`、`user_id`、`event_type`、`created_at`、`bucket_day`。
- Repository 方法必须显式带 `tenant_id`。
- 支持按租户和时间范围查询。
- 支持游标分页。

### 分片设计要求

写出三种 shard key 候选：

```javascript
{ tenant_id: 1, created_at: 1 }
{ tenant_id: "hashed" }
{ tenant_id: 1, bucket_day: 1, user_id: "hashed" }
```

比较：

- 写入分布。
- 查询路由。
- 大租户风险。
- 时间范围查询。
- 归档便利性。

### 性能验证要求

至少记录：

```text
插入 10 万 / 100 万测试数据耗时
核心查询 explain
是否包含 shard key
是否存在全局 skip
是否存在全局聚合
```

### 文档要求

项目 README 必须包含：

- 为什么选择这个 shard key。
- 为什么放弃其他 shard key。
- 索引设计。
- Repository 方法列表。
- 可能的热点风险。
- 后续扩容方案。

## 12. 常见不合格表现

如果出现下面情况，说明阶段 8 没过关：

- 只会说“数据大就分片”。
- shard key 只看数据均匀，不看查询。
- 不知道 mongos 和 shard 的区别。
- 不知道 config server 保存什么。
- 不知道 scatter-gather 的代价。
- Go 代码里到处传任意 `bson.M`。
- 不知道唯一索引和 shard key 的关系。
- 不知道单文档热点不能靠分片解决。
- 没有压测和回滚方案就想上生产分片。
