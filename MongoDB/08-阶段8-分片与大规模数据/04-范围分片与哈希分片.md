# 04. 范围分片与哈希分片

MongoDB 常见分片方式包括范围分片和哈希分片。它们没有绝对好坏，关键看查询模式和写入模式。

## 1. 范围分片

范围分片按照 shard key 的取值范围切分数据。

例如 shard key：

```javascript
{ user_id: 1 }
```

可能范围：

```text
user_id A-H -> shard1
user_id I-P -> shard2
user_id Q-Z -> shard3
```

实际 MongoDB 用 chunk 表示范围。

## 2. 范围分片优点

适合范围查询。

例如：

```javascript
db.events.find({
  tenant_id: "t1",
  created_at: {
    $gte: start,
    $lt: end
  }
})
```

如果 shard key 是：

```javascript
{ tenant_id: 1, created_at: 1 }
```

同一 tenant 的时间范围查询可能更友好。

## 3. 范围分片风险

如果 shard key 单调递增：

```javascript
{ created_at: 1 }
```

新写入都落到最大范围 chunk，容易集中到一个 shard，形成热点。

这就是单调递增 shard key 的典型风险。

## 4. 哈希分片

哈希分片会对 shard key 做哈希，再按哈希值分布。

例如：

```javascript
{ user_id: "hashed" }
```

优点：

- 数据分布更均匀。
- 写入更容易打散。

风险：

- 范围查询能力弱。
- 相邻值被打散到不同 shard。

## 5. 哈希分片适合什么

适合：

- 高基数字段。
- 等值查询。
- 写入需要打散。
- 不依赖 shard key 范围查询。

例如：

```javascript
db.events.find({ user_id: userId })
```

使用：

```javascript
{ user_id: "hashed" }
```

可以定向到对应 shard。

## 6. 范围 vs 哈希

| 维度 | 范围分片 | 哈希分片 |
| --- | --- | --- |
| 数据分布 | 取决于值分布 | 通常更均匀 |
| 范围查询 | 更友好 | 不友好 |
| 写入打散 | 可能热点 | 更好 |
| 单调递增字段 | 风险高 | 哈希后可缓解 |
| 等值查询 | 可定向 | 可定向 |

## 7. 复合 shard key 中的选择

例如日志事件：

```javascript
{
  tenant_id,
  user_id,
  created_at
}
```

候选：

```javascript
{ tenant_id: 1, created_at: 1 }
```

优点：租户 + 时间查询友好。

风险：大租户热点，时间递增导致局部写热点。

候选：

```javascript
{ user_id: "hashed" }
```

优点：写入分散。

风险：按租户和时间统计跨 shard。

候选：

```javascript
{ tenant_id: 1, user_id: 1 }
```

优点：租户内用户查询友好。

风险：大租户仍需验证分布。

## 8. 本节练习

为下面场景选择范围或哈希：

1. 按用户 ID 等值查询为主的事件表。
2. 按时间范围扫描为主的归档表。
3. 写入按自增 ID 递增。
4. 多租户工单按 tenant_id 查询。
5. IoT 设备指标按 device_id 和时间查。
6. 全局订单按创建时间范围统计。

## 9. 本节小结

你需要记住：

- 范围分片适合范围查询，但可能热点。
- 哈希分片更容易均匀分布，但不利于范围查询。
- 单调递增字段不适合直接作为范围 shard key。
- 分片方式要结合查询和写入模式选择。

## 10. 选择范围还是哈希的决策表

| 问题 | 更偏范围分片 | 更偏哈希分片 |
| --- | --- | --- |
| 是否经常按时间范围查询 | 是 | 否 |
| 是否经常按某个 ID 精确查询 | 可以 | 是 |
| 写入字段是否单调递增 | 风险高 | 更安全 |
| 是否需要保留排序局部性 | 是 | 否 |
| 是否希望写入天然打散 | 否 | 是 |
| 是否要按地域/租户做 zone | 是 | 视情况 |

范围分片保留顺序，哈希分片打散顺序。这个差异会影响查询、写入、归档和运维。

## 11. 范围分片的正确用法

范围分片不是不能用时间字段，而是不能无脑只用递增时间字段。

风险设计：

```javascript
{ created_at: 1 }
```

所有新写入都落在时间最大的一端，很容易集中到一个 shard。

更合理的方向：

```javascript
{ tenant_id: 1, created_at: 1 }
```

或者：

```javascript
{ bucket_day: 1, user_id: "hashed" }
```

前者适合租户内查询，后者适合按日期桶管理，同时在日期桶内打散用户。

## 12. 哈希分片的代价

哈希分片常用于打散写入，但它会牺牲范围定位能力。

例如：

```javascript
{ user_id: "hashed" }
```

下面查询通常可以定位：

```javascript
db.events.find({ user_id: 10086 })
```

但下面查询并不能因为 hash 而天然高效：

```javascript
db.events.find({
  created_at: {
    $gte: ISODate("2026-07-01"),
    $lt: ISODate("2026-07-02")
  }
})
```

如果业务主要按时间范围扫描，只有哈希 user_id 可能会让查询扩散到多个 shard。

## 13. 复合 hashed shard key 的思路

很多大规模系统需要复合设计：

```javascript
{ app_id: 1, user_id: "hashed" }
```

或者：

```javascript
{ tenant_id: 1, bucket_day: 1, event_id: "hashed" }
```

设计目标是：

- 前缀字段服务主要查询边界。
- 哈希字段帮助打散写入。
- 时间桶帮助归档和控制范围。

不要为了“均匀”牺牲所有查询模式，也不要为了“范围查询”制造写入热点。

## 14. mongosh 示例

开启数据库分片：

```javascript
sh.enableSharding("appdb")
```

对集合使用范围 shard key：

```javascript
sh.shardCollection(
  "appdb.orders",
  { tenant_id: 1, created_at: 1 }
)
```

对集合使用哈希 shard key：

```javascript
sh.shardCollection(
  "appdb.events",
  { user_id: "hashed" }
)
```

查看状态：

```javascript
sh.status()
```

注意：真实生产集群上执行分片命令前，要先完成索引、压测、备份和回滚评审。

## 15. 设计练习

为下面系统选择范围还是哈希：

```text
系统 A：订单系统
查询：按 tenant_id + created_at 列表，按 order_id 查详情
写入：租户分布不均，大租户明显

系统 B：设备指标系统
查询：按 device_id + 时间范围，按 region 聚合
写入：设备数千万，时间持续递增

系统 C：短链访问日志
查询：按 link_id 查最近访问，离线按天统计
写入：热门 link 高峰明显
```

要求写出：

- 候选 shard key。
- 范围/哈希选择。
- 可能热点。
- 需要的二级索引。
- 不适合的方案及理由。
