# 10. Go 后端接入分片集群注意事项

Go 应用接入分片集群时，通常连接 mongos，而不是直接连接某个 shard。代码层面很多 CRUD 不需要改变，但查询设计、超时、幂等和可观测性更重要。

## 1. 连接 mongos

连接串示例：

```text
mongodb://mongos1:27017,mongos2:27017/?retryWrites=true
```

Go：

```go
clientOpts := options.Client().ApplyURI(uri)
client, err := mongo.Connect(clientOpts)
```

应用不需要知道数据在哪个 shard。

## 2. 不要直连 shard

应用应该通过 mongos 路由。

直连 shard 会绕过路由层，容易造成：

- 查询不完整。
- 写入错误。
- 元数据不一致风险。

## 3. 查询要包含 shard key

Repository 设计时要尽量让高频查询包含 shard key。

例如事件查询：

```go
filter := bson.D{
    {"tenant_id", tenantID},
    {"user_id", userID},
    {"created_at", bson.D{{"$gte", start}, {"$lt", end}}},
}
```

如果漏掉 `tenant_id` 或 `user_id`，可能变成 scatter-gather。

## 4. API 参数设计影响分片效率

用户事件接口：

```text
GET /tenants/{tenant_id}/users/{user_id}/events
```

比：

```text
GET /events?type=click
```

更容易定向查询。

API 设计要服务 shard key。

## 5. 超时和重试

分片集群比单节点更复杂。

Go 后端应：

- 设置 context timeout。
- 设置合理 server selection timeout。
- 对幂等操作做重试。
- 对非幂等写入使用 idempotency key。

不要无脑重试非幂等写入。

## 6. BulkWrite 和批量写入

高吞吐写入要批量：

```go
_, err := collection.InsertMany(ctx, docs)
```

或：

```go
_, err := collection.BulkWrite(ctx, models, options.BulkWrite().SetOrdered(false))
```

分片集群中批量写入也要注意：

- 单批不要过大。
- 错误要记录。
- 幂等 ID 要设计。
- 写入分布要监控。

## 7. 可观测性

应用日志中记录：

- query name。
- filter 关键维度。
- limit。
- duration。
- error。
- request_id。

不要打印完整大文档。

对于分片系统，最好能知道：

- 哪些接口可能 scatter-gather。
- 哪些租户或用户是热点。
- 哪些查询没有 shard key。

## 8. Go 中不要隐藏 shard key

错误抽象：

```go
func ListEvents(ctx context.Context, filter EventFilter) ([]Event, error)
```

但 `EventFilter` 里 tenant_id/user_id 可选，导致调用方漏传。

更明确：

```go
func ListUserEvents(ctx context.Context, tenantID string, userID bson.ObjectID, input EventListInput) ([]Event, error)
```

把 shard key 维度放进方法签名。

## 9. 本节练习

设计 Go Repository 方法：

```go
ListUserEvents(ctx, tenantID, userID, start, end, limit)
ListDeviceEvents(ctx, tenantID, deviceID, start, end, limit)
AggregateTenantEvents(ctx, tenantID, start, end)
```

写出每个方法：

- filter。
- sort。
- limit。
- 是否包含 shard key。
- 可能的索引。
- 是否可能 scatter-gather。

## 10. 本节小结

你需要记住：

- Go 应用连接 mongos。
- 高频查询要包含 shard key。
- API 设计会影响查询路由。
- Repository 方法应显式要求 shard key 维度。
- 批量写入要控制大小和幂等。
- 分片系统更需要日志和监控。

## 11. 连接字符串与部署建议

Go 服务应该连接多个 mongos 地址：

```text
mongodb://mongos-a:27017,mongos-b:27017,mongos-c:27017/app
```

基础连接示例：

```go
client, err := mongo.Connect(options.Client().
    ApplyURI(cfg.MongoURI).
    SetServerSelectionTimeout(5 * time.Second).
    SetConnectTimeout(5 * time.Second).
    SetMaxPoolSize(100).
    SetMinPoolSize(10))
if err != nil {
    return err
}
```

每个请求仍然要有独立 timeout：

```go
ctx, cancel := context.WithTimeout(parent, 2*time.Second)
defer cancel()
```

连接池不是越大越好。池太大可能把数据库压垮，池太小会让应用排队。要结合接口 QPS、请求耗时、mongos 数量和数据库承载能力压测。

## 12. Repository 方法要暴露路由维度

分片后，Repository 不应该只追求“通用”。

不推荐：

```go
func (r *OrderRepo) Find(ctx context.Context, filter bson.M) ([]Order, error)
```

推荐：

```go
func (r *OrderRepo) ListByTenant(
    ctx context.Context,
    tenantID string,
    status string,
    cursor primitive.ObjectID,
    limit int64,
) ([]Order, error)
```

这样方法签名本身就要求调用方提供 shard key 维度。代码评审时也更容易发现路由风险。

## 13. 批量写入要考虑分布

批量写入可以提升吞吐，但要注意：

- 单批不要过大。
- 要设置合理 timeout。
- 要处理部分失败。
- 最好按 shard key 或业务分区组织批次。
- 幂等键和唯一索引要提前设计。

示例：

```go
models := make([]mongo.WriteModel, 0, len(events))
for _, e := range events {
    models = append(models, mongo.NewInsertOneModel().SetDocument(e))
}

res, err := r.events.BulkWrite(ctx, models, options.BulkWrite().SetOrdered(false))
if err != nil {
    return err
}
_ = res
```

`SetOrdered(false)` 可以让部分写入失败时其他写入继续执行，但你必须记录失败数据，并设计重试或死信队列。

## 14. 日志字段建议

分片系统排障时，应用日志必须能还原查询维度：

```go
logger.Info("mongo_query",
    "collection", "orders",
    "operation", "ListByTenant",
    "tenant_id", tenantID,
    "status", status,
    "limit", limit,
    "duration_ms", time.Since(start).Milliseconds(),
)
```

建议至少记录：

- collection。
- operation。
- shard key 相关字段。
- limit。
- sort。
- duration。
- matched/returned 数量。
- 错误类型。

不要记录敏感明文，例如密码、完整 token、身份证号。

## 15. 错误处理与重试边界

分片环境下，网络抖动、mongos 切换、primary 选举都会让错误更多样。重试要谨慎：

```text
读请求：通常更容易安全重试。
幂等写：可以按幂等键重试。
非幂等写：不能盲目重试。
事务：回调必须能安全重试。
```

订单创建这类写入必须有请求幂等键：

```javascript
{ tenant_id: 1, request_id: 1 }
```

否则应用层重试可能创建重复订单。

## 16. 压测时必须验证路由

压测不要只看接口 QPS，还要看：

```text
每个 shard 的 QPS 是否均衡？
mongos CPU 是否过高？
慢查询是否变多？
scatter-gather 查询是否出现？
chunk 迁移期间延迟是否抖动？
大租户是否压垮单 shard？
```

压测数据要尽量模拟真实业务分布。完全均匀的随机数据会掩盖大租户和热点问题。
