# 06. 热点写入与单调递增字段风险

热点写入是分片系统最常见的性能问题之一。分片的目标是分散负载，但错误的 shard key 可能让所有写入仍然集中到一个 shard。

## 1. 什么是热点写入

热点写入指大量写操作集中到：

- 同一个 shard。
- 同一个 chunk。
- 同一个文档。
- 同一个 shard key 值。

结果：

- 某个 shard 压力很高。
- 其他 shard 很空闲。
- 集群总体扩展效果差。

## 2. 单调递增字段风险

常见单调递增字段：

- `created_at`
- 自增 ID
- 递增订单号
- 时间戳前缀 ID

如果 shard key 是：

```javascript
{ created_at: 1 }
```

新写入总是落到最大时间范围的 chunk，可能集中到一个 shard。

这就是时间字段做范围 shard key 的典型风险。

## 3. 哈希缓解热点

哈希 shard key：

```javascript
{ user_id: "hashed" }
```

可以让不同 user_id 打散到不同 shard。

但如果所有写入都来自同一个 user_id，哈希也救不了这个热点。

所以要理解业务写入分布。

## 4. 组合 shard key 缓解热点

日志场景：

```javascript
{ tenant_id: 1, created_at: 1 }
```

可能大租户写入热点。

可以考虑：

```javascript
{ tenant_id: 1, user_id: 1 }
```

或：

```javascript
{ user_id: "hashed" }
```

具体选择要看查询模式和写入分布。

## 5. 热点文档

不是只有 shard key 会热点。

单个热门商品库存：

```javascript
db.products.updateOne(
  { _id: productId },
  { $inc: { stock: -1 } }
)
```

即使分片，所有请求仍然更新同一个商品文档，这就是热点文档。

解决可能需要：

- 预扣库存。
- 分桶库存。
- 队列削峰。
- 秒杀专门方案。
- Redis + 异步落库。

分片不能自动解决单文档热点。

## 6. 热点识别

观察：

- 某个 shard CPU 明显高。
- 某个 shard 写入量明显高。
- 某些 chunk 迁移频繁。
- 某些 shard 延迟高。
- 慢查询集中在某个 shard。

业务侧也要看：

- 大租户。
- 热门用户。
- 热门商品。
- 热门文章。

## 7. 设计时避免热点

建议：

- 不要用纯递增字段做 shard key。
- 避免低基数字段做 shard key。
- 了解大客户/大用户分布。
- 写入维度尽量打散。
- 热点业务单独设计。
- 压测验证写入分布。

## 8. 本节练习

判断热点风险：

1. shard key `{ created_at: 1 }`，日志持续写入。
2. shard key `{ status: 1 }`，订单大量写入 paid。
3. shard key `{ user_id: "hashed" }`，用户均匀写入。
4. 热门商品库存单文档扣减。
5. 多租户系统中一个租户占 80% 写入。
6. IoT 中少数设备每秒上报大量数据。

## 9. 本节小结

你需要记住：

- 热点写入会让分片失去扩展效果。
- 纯时间递增字段容易造成写热点。
- 哈希可以打散不同 key，但不能解决单 key 热点。
- 分片不能解决单文档热点。
- 大租户和热门商品要单独设计。

## 10. 热点的三种层次

热点不是一个问题，而是一类问题。排查时要先判断热点发生在哪一层。

### shard 热点

大量请求集中到某个 shard：

```text
shard1: 80% 写入
shard2: 10% 写入
shard3: 10% 写入
```

常见原因：

- shard key 单调递增。
- shard key 低基数。
- 大租户集中在一个 shard。
- 查询没有带 shard key，导致某些 shard 扫描压力更高。

### chunk 热点

大量写入集中在一个 chunk 的范围内。例如 shard key 是 `{ created_at: 1 }`，所有新数据都写入最大时间范围对应的 chunk。

### 单文档热点

所有请求更新同一个文档：

```javascript
db.products.updateOne(
  { _id: productId },
  { $inc: { stock: -1 } }
)
```

热门商品秒杀时，这种热点不是分片能直接解决的，因为一个文档只能位于一个 shard。

## 11. 单调递增字段为什么危险

假设 shard key 是：

```javascript
{ created_at: 1 }
```

写入顺序是：

```text
10:00:01
10:00:02
10:00:03
10:00:04
```

新数据永远落在 shard key 空间的最右侧。即使集群有很多 shard，最新 chunk 仍可能集中在一个 shard 上。结果是：

- 一个 shard 写入压力非常高。
- balancer 来不及迁移。
- 迁移又会带来额外 IO。
- 分片集群看起来扩容了，实际写入仍被单点限制。

## 12. 哈希能解决什么，不能解决什么

哈希 shard key 可以把不同 key 打散：

```javascript
{ user_id: "hashed" }
```

适合：

- 大量用户均匀写入。
- 按 user_id 精确查询。
- 避免 ObjectId、时间等递增键集中写入。

但它不能解决：

- 单个用户本身写入极高。
- 单个商品库存被高并发更新。
- 单个租户贡献大部分写入。

如果 80% 流量来自一个 `tenant_id`，对 `tenant_id` 做哈希只能把这个租户放到某个 shard，并不能把该租户内部流量拆开。

## 13. 大租户热点设计

多租户系统常见风险：

```text
普通租户：每天 10 万数据
大租户：每天 2 亿数据
```

如果 shard key 是：

```javascript
{ tenant_id: 1 }
```

大租户可能压垮所在 shard。

可选优化方向：

```javascript
{ tenant_id: 1, bucket_day: 1, user_id: "hashed" }
```

或者：

```javascript
{ tenant_id: 1, partition_id: 1 }
```

其中 `partition_id` 可以由业务生成，例如根据用户、设备、订单号 hash 到 0-63。

Go 中可以这样生成简单分区：

```go
func PartitionID(key string, buckets uint32) uint32 {
    h := fnv.New32a()
    _, _ = h.Write([]byte(key))
    return h.Sum32() % buckets
}
```

写入文档时保存：

```go
doc := bson.M{
    "tenant_id":    tenantID,
    "partition_id": PartitionID(userID, 64),
    "user_id":      userID,
    "created_at":   now,
}
```

这样同一个大租户可以被进一步拆散。

## 14. 单文档热点的替代方案

热门计数、库存、点赞数，不一定要每次都更新一个文档。

### 计数器分桶

把一个计数拆成多个桶：

```javascript
{
  product_id: "p1001",
  bucket: 17,
  likes: 1
}
```

更新时随机选择 bucket：

```javascript
db.product_like_counters.updateOne(
  { product_id: "p1001", bucket: 17 },
  { $inc: { likes: 1 } },
  { upsert: true }
)
```

读取时聚合求和。这样写入压力从一个文档拆到多个文档。

### 库存预扣分段

秒杀库存可以按库存段拆分：

```text
product_id + stock_segment
```

每个 segment 有独立库存。扣减时选择一个还有库存的 segment。这样可以降低单文档写冲突，但业务实现更复杂，需要处理最终汇总和售罄判断。

## 15. 监控热点时看什么

不要只看平均 QPS。要看：

- 每个 shard 的读写 QPS。
- 每个 shard 的 CPU、磁盘 IO。
- 慢查询是否集中在某个 shard。
- chunk 迁移是否频繁。
- 大租户、大用户、大商品的写入排名。
- 写冲突、重试次数、超时数量。

Go 服务中建议给核心写入打业务标签：

```go
logger.Info("mongo_write",
    "collection", "events",
    "tenant_id", tenantID,
    "partition_id", partitionID,
    "duration_ms", time.Since(start).Milliseconds(),
)
```

这样排查时能把数据库热点和业务实体对应起来。
