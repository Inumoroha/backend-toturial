# 07. 唯一索引与分片键限制

分片集合中的唯一索引比普通集合更复杂。你必须理解唯一约束和 shard key 的关系，否则可能设计出无法创建或无法保证全局唯一的索引。

## 1. 为什么唯一索引变复杂

在未分片集合中：

```javascript
db.users.createIndex({ email: 1 }, { unique: true })
```

MongoDB 可以在一个集合范围内保证 email 唯一。

分片后，数据分布在多个 shard 上。

如果唯一索引和 shard key 无关，某个 shard 可能不知道另一个 shard 上已经有相同 email。

## 2. 官方核心规则

对分片集合，唯一索引通常要求索引包含 shard key 作为前缀。

例如 shard key：

```javascript
{ tenant_id: 1, user_id: 1 }
```

可接受的唯一索引：

```javascript
{ tenant_id: 1, user_id: 1 }
{ tenant_id: 1, user_id: 1, email: 1 }
```

不适合：

```javascript
{ email: 1 }
```

因为它不包含 shard key 前缀。

## 3. 多租户邮箱唯一

如果业务要求“同一租户内 email 唯一”，可以设计：

shard key：

```javascript
{ tenant_id: 1, user_id: 1 }
```

唯一索引：

```javascript
{ tenant_id: 1, email: 1 }
```

但它是否被允许取决于 shard key 设计和 MongoDB 规则。很多时候需要让 shard key 更贴合唯一约束。

更简单的设计：

shard key：

```javascript
{ tenant_id: 1, email: 1 }
```

唯一索引：

```javascript
{ tenant_id: 1, email: 1 }
```

这样同租户 email 唯一更自然。

## 4. 全局唯一 ID

很多系统使用全局唯一 ID：

```javascript
_id: ObjectId(...)
```

`_id` 在分片集合中的唯一性也和 shard key 有关系。不要简单假设所有唯一索引都和单机一样。

实践中通常：

- 使用业务唯一号做幂等。
- 把唯一约束设计进 shard key。
- 或在应用层/独立集合维护全局唯一。

## 5. 幂等键设计

订单集合按用户分片：

```javascript
{ user_id: 1, created_at: 1 }
```

如果幂等键是全局唯一：

```javascript
idempotency_key
```

唯一索引可能受 shard key 限制。

可以改为：

```javascript
{ user_id: 1, idempotency_key: 1 }
```

业务上保证“同一用户内幂等键唯一”。

## 6. 设计前要明确唯一范围

唯一性有范围：

- 全局唯一。
- 租户内唯一。
- 用户内唯一。
- 订单内唯一。

不要含糊。

例如：

```text
email 是全平台唯一，还是租户内唯一？
order_no 是全局唯一，还是按商户唯一？
idempotency_key 是全局唯一，还是用户内唯一？
```

## 7. 本节练习

判断唯一范围：

1. SaaS 用户邮箱。
2. 电商订单号。
3. 用户提交订单幂等键。
4. 支付平台回调流水号。
5. 设备 ID。
6. 租户内工单编号。

为每个写：

```text
唯一范围：
可能 shard key：
唯一索引：
风险：
```

## 8. 本节小结

你需要记住：

- 分片集合唯一索引通常要求包含 shard key 前缀。
- 唯一约束要先明确范围。
- 幂等键要结合 shard key 设计。
- 全局唯一在分片环境中更复杂。
- 设计 shard key 时必须同时考虑唯一索引需求。

## 9. 为什么唯一索引在分片中更复杂

在单个复制集中，唯一索引只需要在一个索引空间里判断是否重复。分片以后，数据分布在多个 shard 上：

```text
shard1: 一部分用户
shard2: 一部分用户
shard3: 一部分用户
```

如果唯一索引和 shard key 没有关系，MongoDB 很难只在一个目标 shard 上完成唯一性判断。为了保证全局唯一，系统可能需要跨 shard 检查，这会破坏分片的可扩展性。

所以分片集合中的唯一索引设计，必须和 shard key 一起考虑。

## 10. 唯一范围先于唯一索引

业务说“唯一”时，你要追问：在哪个范围内唯一？

```text
邮箱：全平台唯一？
用户名：全平台唯一还是租户内唯一？
订单号：全平台唯一还是商户内唯一？
幂等键：用户内唯一、订单内唯一，还是请求来源内唯一？
设备 ID：全平台唯一还是厂商内唯一？
```

不同答案对应不同索引。

租户内唯一：

```javascript
db.users.createIndex(
  { tenant_id: 1, email: 1 },
  { unique: true }
)
```

全局唯一：

```javascript
db.users.createIndex(
  { email: 1 },
  { unique: true }
)
```

在分片集合中，后者往往更难和 shard key 兼容。

## 11. 推荐把唯一范围放进 shard key

如果用户集合按租户分片：

```javascript
{ tenant_id: 1, user_id: 1 }
```

那么租户内邮箱唯一可以设计为：

```javascript
db.users.createIndex(
  { tenant_id: 1, email: 1 },
  { unique: true }
)
```

这样唯一判断和路由边界一致，更容易扩展。

但如果业务要求 `email` 全平台唯一，就要慎重。可选方案包括：

- 单独维护一个全局唯一注册集合。
- 使用中心化 ID/账号服务。
- 把账号系统和租户业务数据拆开。
- 通过事务或幂等写入保证注册流程一致。

## 12. 幂等键设计

支付回调、订单创建、消息消费都需要幂等。

例如订单创建请求：

```javascript
{
  tenant_id: "t1001",
  request_id: "client-generated-id",
  user_id: "u1001",
  amount: 100
}
```

如果幂等范围是租户内唯一：

```javascript
db.orders.createIndex(
  { tenant_id: 1, request_id: 1 },
  { unique: true }
)
```

Go 中处理重复键：

```go
_, err := orders.InsertOne(ctx, order)
if mongo.IsDuplicateKeyError(err) {
    existing, findErr := findOrderByRequestID(ctx, tenantID, requestID)
    if findErr != nil {
        return nil, findErr
    }
    return existing, nil
}
if err != nil {
    return nil, err
}
```

重复键不是系统异常，而是幂等流程的一部分。

## 13. 全局唯一的常见方案

### 独立唯一集合

先写一个小集合：

```javascript
db.unique_emails.insertOne({
  email: "a@example.com",
  user_id: ObjectId("...")
})
```

给 `email` 建唯一索引。注册成功后再写用户主集合。

优点：

- 全局唯一逻辑清晰。
- 主业务集合 shard key 可以按业务设计。

代价：

- 注册流程多一次写入。
- 需要处理失败补偿。
- 可能需要事务。

### 外部账号中心

大型系统常把账号唯一性放到单独账号服务，业务库只保存账号 ID。这样 MongoDB 分片集合不承担所有全局唯一逻辑。

## 14. 设计检查清单

```text
唯一字段是什么？
唯一范围是什么？
是否包含 shard key？
是否支持主要查询？
是否影响写入吞吐？
重复键错误是否被当作业务分支处理？
迁移历史数据前是否检查重复值？
```
