# 01. 什么时候需要分片

分片是水平扩展方案，但它会显著增加架构复杂度。很多时候系统慢，不是因为没分片，而是索引、建模、查询、硬件或归档策略出了问题。

## 1. 分片解决什么问题

分片主要解决：

- 单个复制集磁盘容量不足。
- 单个复制集写入吞吐不足。
- 工作集太大，单机内存难以承载。
- 数据量持续增长，需要水平扩展。
- 某些业务需要按数据分布扩展。

例如：

```text
每天写入 5000 万事件
保留 180 天
按用户和时间查询
单个复制集存储、写入、索引压力都很大
```

这种场景可能需要分片。

## 2. 什么时候先不要分片

如果问题是下面这些，先不要急着分片：

- 查询没有索引。
- 复合索引字段顺序不合理。
- 大量深分页。
- 文档模型导致大量 `$lookup`。
- 读写都打到热点文档。
- 历史数据不归档。
- 返回大字段导致网络慢。
- 单节点硬件太弱。

分片不能自动修复这些问题。

## 3. 分片带来的复杂度

分片会引入：

- mongos。
- config server。
- 多个 shard。
- shard key 选择。
- chunk 迁移。
- balancer。
- 跨 shard 查询。
- 跨 shard 事务成本。
- 运维和监控复杂度。

如果团队还没掌握索引、复制集和备份，贸然上分片会让问题更难排查。

## 4. 判断是否需要分片的指标

从这些方面判断：

### 容量

- 数据量是否超过单个复制集可承载范围？
- 索引大小是否超出内存能力？
- 历史数据是否可归档？

### 吞吐

- 写入 QPS 是否达到单个 primary 瓶颈？
- 写入是否集中在少数热点文档？
- 批量写入是否已优化？

### 查询

- 查询是否已经有合适索引？
- explain 是否健康？
- 是否存在大量 scatter-gather 风险？

### 运维

- 是否能监控分片集群？
- 是否能备份恢复？
- 是否能处理 chunk 迁移和热点？

## 5. 分片前的优化清单

先做：

- 建模评审。
- 索引评审。
- explain 分析。
- 慢查询优化。
- 深分页改游标分页。
- 冷热数据归档。
- TTL 清理临时数据。
- 批量写入优化。
- 复制集资源扩容。

这些都做过，仍然遇到容量或吞吐瓶颈，再考虑分片。

## 6. 适合分片的场景

常见：

- 日志/埋点事件。
- IoT 设备上报。
- 大规模消息。
- 多租户 SaaS 数据。
- 用户订单超大规模。
- 内容平台行为数据。

这些场景通常有明显的数据增长和查询维度。

## 7. 不适合分片的场景

暂时不适合：

- 数据量小。
- 查询模式不明确。
- 团队缺少复制集和备份经验。
- shard key 无法确定。
- 业务强依赖跨 shard 事务。
- 写入集中在单个热点键。

## 8. 本节练习

判断下面场景是否需要分片：

1. 10 万用户的博客系统，查询慢但没有索引。
2. 每天 3000 万埋点事件，保留 180 天。
3. 订单列表深分页很慢。
4. 商品库存扣减慢，热点商品集中写。
5. IoT 设备每秒上报大量指标。
6. 后台报表聚合慢，但可以离线统计。

格式：

```text
场景：
是否需要分片：
先做哪些优化：
如果分片，可能 shard key：
风险：
```

## 9. 本节小结

你需要记住：

- 分片解决容量和吞吐的水平扩展问题。
- 分片不是慢查询万能药。
- 分片前先优化建模、索引、分页、归档。
- 分片会带来运维和查询复杂度。
- 是否分片要基于数据量、吞吐、查询和团队能力判断。

## 10. 更工程化的判断流程

作为 Go 后端工程师，不要用“数据量大”这一个词决定是否分片。你应该把问题拆成四层：

```text
1. 数据是否真的大到单个复制集承受不了？
2. 查询是否已经被正确建模和正确索引？
3. 写入瓶颈是否来自单个 primary 的吞吐上限？
4. 团队是否已经具备复制集、备份、监控、慢查询治理能力？
```

如果第 2 点还没做好，分片通常会把慢查询扩散到更多节点。  
如果第 4 点还没做好，分片会让故障排查成本明显升高。

推荐的决策顺序：

```text
慢查询
  -> explain
  -> 补索引/改查询
  -> 仍慢
  -> 改模型/减少返回字段/改分页
  -> 仍慢
  -> 评估读写分离、缓存、异步统计
  -> 仍有容量或写入瓶颈
  -> 评估分片
```

容量问题也一样：

```text
磁盘增长快
  -> 看是否有无用数据
  -> TTL/归档/冷热分层
  -> 压缩大字段或移出对象存储
  -> 仍然持续超过单复制集能力
  -> 评估分片
```

## 11. 粗略容量估算

做分片决策前，至少写出一张容量估算表：

```text
日写入文档数：
单文档平均大小：
日新增数据量：
保留天数：
原始数据总量：
索引数量：
索引膨胀系数：
副本数：
预留增长：
```

示例：

```text
每天 5000 万条事件
单条 800 字节
原始数据每天约 40GB
保留 180 天约 7.2TB
索引按 40% 估算约 2.88TB
三副本约 30TB 以上
还要预留压缩、增长、迁移和备份空间
```

这个系统可能就不是简单给单机加磁盘能长期解决的。

## 12. 吞吐估算

写入瓶颈不能只看平均值，要看峰值。

```text
平均 QPS = 每天写入量 / 86400
峰值 QPS = 平均 QPS * 峰值倍率
```

例如：

```text
每天 5000 万写入
平均约 579 writes/s
如果晚高峰是平均 10 倍
峰值约 5790 writes/s
```

你还要继续问：

- 是否每次写入都更新多个索引？
- 是否包含大数组字段？
- 是否有唯一索引检查？
- 是否使用批量写入？
- 是否有热点用户、热点租户、热点设备？

这些因素会决定瓶颈来自哪里。

## 13. Go 后端评估分片时要补充的问题

Go 服务接入分片前，要把下面问题写进技术方案：

```text
连接字符串是否接入 mongos？
是否有多个 mongos 地址？
超时是否合理？
查询是否都能带上 shard key 或 shard key 前缀？
Repository 层是否隐藏了会触发 scatter-gather 的查询？
分页是否避免全局 skip？
批量写入是否按 shard key 分组？
错误重试是否区分可重试错误和业务错误？
监控是否能按 shard 观察慢查询和写入压力？
```

一个容易被忽略的问题是：业务代码里的“通用查询接口”可能会让调用方随意组合条件，最后产生大量跨 shard 查询。进入分片阶段后，Repository 方法应该更明确，例如：

```go
FindEventsByTenantAndTime(ctx, tenantID, start, end, limit)
FindUserOrdersByUserAndCursor(ctx, userID, cursor, limit)
```

而不是：

```go
FindEvents(ctx, filter)
```

通用 filter 看起来灵活，生产中却容易绕过 shard key。
