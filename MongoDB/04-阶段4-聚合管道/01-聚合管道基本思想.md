# 01. 聚合管道基本思想

聚合管道用于对集合中的文档进行多阶段处理。你可以把它理解成一条流水线：每个 stage 接收上一阶段输出的文档，处理后再交给下一阶段。

## 1. 为什么需要聚合管道

CRUD 适合做：

- 查一条用户。
- 查文章列表。
- 更新订单状态。
- 删除测试数据。

但很多后端需求是统计型的：

- 每天订单金额。
- 每个用户消费总额。
- 商品销量排行榜。
- 文章评论数。
- 用户最近 30 天活跃天数。
- 后台看板指标。

这些需求需要过滤、分组、计算、排序、关联和字段转换，聚合管道就是为这类问题准备的。

## 2. Pipeline 和 Stage

基本形式：

```javascript
db.collection.aggregate([
  { stage1 },
  { stage2 },
  { stage3 }
])
```

示例：

```javascript
db.orders.aggregate([
  { $match: { status: "paid" } },
  { $group: { _id: "$user_id", total_amount: { $sum: "$total_amount" } } },
  { $sort: { total_amount: -1 } }
])
```

每个对象是一个 stage。

执行顺序从上到下。

## 3. 文档如何流动

假设 `orders` 有 10000 条文档。

```javascript
[
  { $match: { status: "paid" } },
  { $group: { _id: "$user_id", total_amount: { $sum: "$total_amount" } } },
  { $sort: { total_amount: -1 } }
]
```

执行过程：

```text
10000 条订单
  -> $match 过滤出 paid，比如 6000 条
  -> $group 按 user_id 分组，比如 800 个用户
  -> $sort 对 800 个分组结果排序
```

这就是为什么 `$match` 通常尽量靠前。越早减少数据量，后面的计算越轻。

## 4. 字段引用语法

在聚合表达式里，引用字段通常用 `$字段名`。

例如：

```javascript
{ $sum: "$total_amount" }
```

表示把每条文档的 `total_amount` 字段加起来。

引用嵌套字段：

```javascript
{ $sum: "$stats.views" }
```

普通字符串和字段引用不同：

```javascript
"$status"   // 引用 status 字段
"status"    // 字符串 status
```

## 5. 输出结果不是原文档

聚合的输出可以完全不同于原集合文档。

原订单：

```javascript
{
  user_id: ObjectId("..."),
  status: "paid",
  total_amount: 9900
}
```

聚合结果：

```javascript
{
  _id: ObjectId("..."),
  total_amount: 19900,
  order_count: 2
}
```

聚合结果是计算后的新文档。

## 6. 常用 Stage

| Stage | 作用 |
| --- | --- |
| `$match` | 过滤文档 |
| `$project` | 控制输出字段、计算字段 |
| `$group` | 分组统计 |
| `$sort` | 排序 |
| `$limit` | 限制数量 |
| `$skip` | 跳过数量 |
| `$addFields` | 添加或覆盖字段 |
| `$set` | `$addFields` 的别名 |
| `$unwind` | 展开数组 |
| `$lookup` | 关联其他集合 |
| `$count` | 统计数量 |

## 7. 聚合和 find 的区别

`find`：

```javascript
db.orders.find({ status: "paid" })
```

适合简单查询。

`aggregate`：

```javascript
db.orders.aggregate([
  { $match: { status: "paid" } },
  { $group: { _id: "$user_id", total_amount: { $sum: "$total_amount" } } }
])
```

适合多阶段处理和统计。

不要为了简单查询滥用聚合；也不要用应用代码硬算本该数据库聚合的统计。

## 8. 聚合性能基本原则

先记住这些原则：

- `$match` 尽量前置。
- `$project` 可以尽早裁剪大字段。
- `$sort` 最好能利用索引。
- `$lookup` 前先尽量缩小主集合数据量。
- `$unwind` 会放大文档数量，要谨慎。
- 大数据统计要考虑索引、内存和批处理。

第 5 阶段会系统学习 explain 和性能优化。

## 10. 本节小结

你需要记住：

- 聚合管道由多个 stage 顺序组成。
- 每个 stage 处理上一阶段输出的文档。
- `$match` 通常尽量前置。
- 聚合输出可以是全新的文档结构。
- 字段引用使用 `$字段名`。
