# 01. 复制集基本概念

复制集是 MongoDB 高可用的基础。事务、Change Streams、majority 写入等高级能力，也都和复制集密切相关。

## 1. 什么是复制集

复制集是一组维护同一份数据集的 `mongod` 进程。

典型结构：

```text
replica set
  |
  |-- primary
  |-- secondary
  |-- secondary
```

同一时间只有一个 primary。

客户端默认把写入发送到 primary。

secondary 从 primary 复制数据。

## 2. primary

primary 负责：

- 接收写操作。
- 记录 oplog。
- 向 secondary 复制变更。
- 默认处理读请求。

如果 primary 挂掉，复制集会选举新的 primary。

## 3. secondary

secondary 负责：

- 从 primary 拉取 oplog。
- 应用变更，保持数据副本。
- 在 primary 故障时参与选举。
- 在配置允许时处理读请求。

secondary 读有收益：

- 分担 primary 读压力。
- 支持地理就近读取。
- 支持部分报表读。

但也有风险：

- 可能读到稍旧的数据。
- 不能用于强一致业务判断。

## 4. arbiter

arbiter 只参与投票，不保存数据。

作用：

- 在节点数量不足时帮助形成多数。

风险：

- 不保存数据，不能承担读。
- 不能提升数据冗余。
- 生产环境中通常更推荐使用数据节点，而不是依赖 arbiter。

学习阶段重点掌握 primary 和 secondary，arbiter 了解即可。

## 5. oplog

oplog 是复制集的操作日志。

primary 写入数据后，会把变更记录到 oplog。

secondary 通过读取 oplog 并应用操作来复制数据。

Change Streams 也是基于复制集的变更日志能力。

你可以在复制集里查看：

```javascript
use local
db.oplog.rs.find().sort({ $natural: -1 }).limit(5).pretty()
```

注意：`local` 数据库保存节点本地信息，不会被复制。

## 6. 选举

当 primary 不可用时，符合条件的 secondary 会参与选举。

选举后：

- 新 primary 接收写入。
- 旧 primary 如果恢复，通常会以 secondary 身份重新加入。

客户端使用复制集连接串时，Driver 会自动发现拓扑变化。

## 7. 多数派

复制集依赖多数派判断写入和选举。

三节点复制集：

```text
多数 = 2
```

如果 1 个节点挂掉，剩下 2 个节点还能选 primary。

如果只剩 1 个节点，通常无法维持 primary。

## 8. 为什么生产至少 3 个节点

单节点没有高可用。

两节点容易出现无法多数的问题。

三节点是常见最小高可用形态：

```text
primary + secondary + secondary
```

生产环境还要考虑：

- 不同机器。
- 不同可用区。
- 网络延迟。
- 磁盘性能。
- 备份策略。

## 10. 本节小结

你需要记住：

- 复制集是一组维护同一数据集的 MongoDB 节点。
- primary 接收写入。
- secondary 复制数据并可参与选举。
- oplog 是复制和变更流的基础。
- 高可用依赖多数派。

## 11. 复制集解决的不是性能，而是可用性

很多初学者会误以为“复制集有多个节点，所以写入性能会变成三倍”。这是错误的。

在典型复制集中：

```text
写入 -> primary
primary 写入 oplog
secondary 拉取 oplog 并重放
```

写入入口仍然是 primary。复制集主要解决：

- primary 故障后自动选举。
- 数据多副本保存。
- 支持备份、Change Streams、事务等能力。
- 支持在可接受一致性风险下从 secondary 分担读。

它不是水平写扩展方案。水平写扩展是第 8 阶段分片要解决的问题。

## 12. 多数派为什么重要

三节点复制集中，多数派是 2。

```text
node1 + node2 可形成多数派
node1 + node3 可形成多数派
node2 + node3 可形成多数派
```

如果 primary 所在网络分区只剩 1 个节点，它不能继续安全地对外确认 majority 写入，因为它无法证明自己代表多数派。

这避免了脑裂：

```text
分区 A：node1
分区 B：node2 + node3
```

如果两个分区都能写，就会产生两个不同历史。多数派机制要求只有拥有多数派的一侧能选出 primary。

## 13. oplog 的工程意义

oplog 是一个有序的复制日志。secondary 通过读取 primary 的 oplog 来同步数据。

它影响：

- secondary 复制延迟。
- 节点掉线后能否追上。
- Change Streams 能否继续 resume。
- 备份恢复后的增量追赶能力。

如果某个 secondary 离线太久，primary 的 oplog 已经覆盖了它缺失的那段历史，这个 secondary 就可能无法继续增量追赶，需要重新同步。

因此生产中要监控：

```text
复制延迟
oplog 窗口大小
节点状态
磁盘剩余空间
选举次数
```

## 14. 复制集对 Go 应用的影响

Go Driver 会根据连接字符串和拓扑发现 primary/secondary。

常见连接方式：

```text
mongodb://mongo1:27017,mongo2:27017,mongo3:27017/app?replicaSet=rs0
```

建议：

- 配置多个节点地址。
- 设置 `replicaSet`。
- 每个请求使用 `context.WithTimeout`。
- 不要把写入固定发给某个节点。
- 读偏好要按业务一致性要求设置。

示例：

```go
client, err := mongo.Connect(options.Client().
    ApplyURI("mongodb://mongo1:27017,mongo2:27017,mongo3:27017/app?replicaSet=rs0").
    SetServerSelectionTimeout(5 * time.Second))
if err != nil {
    return err
}
```

primary 切换时，应用可能短时间遇到 server selection 或 network 错误。接口要有合理超时，调用方要能处理短暂失败。

## 15. 生产复制集设计建议

- 至少三投票节点。
- 节点分布在不同故障域。
- 不要把所有节点放在同一台物理机。
- 监控复制延迟和选举频率。
- 为备份节点预留资源。
- 谨慎使用 arbiter。
- 写关键业务时使用 majority writeConcern。
- 定期做 primary 故障切换演练。
