# 08. Atlas Backup 与生产备份策略

生产环境备份不能只靠偶尔手动 `mongodump`。如果使用 MongoDB Atlas，应优先了解 Atlas Backup；如果自建 MongoDB，也要设计自动化备份和恢复演练。

## 1. Atlas Backup 是什么

MongoDB Atlas 提供托管备份能力。

它适合：

- 自动备份。
- 按时间点恢复。
- 降低运维负担。
- 云上生产集群。

学习阶段你可以先了解，不一定马上使用。

## 2. 手动备份的问题

只靠人工 `mongodump`：

- 容易忘。
- 不稳定。
- 缺少保留策略。
- 难以满足 RPO/RTO。
- 没有恢复演练。
- 数据量大时成本高。

生产备份必须自动化。

## 3. 备份策略要回答的问题

```text
备份频率是多少？
保留多久？
恢复到哪里？
谁有权限恢复？
是否加密？
多久演练一次？
RPO 是多少？
RTO 是多少？
```

如果回答不出来，说明备份策略还没有真正建立。

## 4. 常见备份方式

### Atlas Backup

适合 Atlas 集群。

### 文件系统快照

适合自建环境，但要保证一致性。

### mongodump 逻辑备份

适合小中型数据和单库导出。

### 增量或连续备份

适合低 RPO 要求。

### 数据仓库/归档

适合长期历史分析，不一定用于业务恢复。

## 5. 备份不是归档

备份：

```text
用于灾难恢复。
```

归档：

```text
用于把冷数据从热库移走。
```

不要把归档当备份，也不要把备份当查询历史数据的常规方式。

## 6. 恢复演练

每个季度或每月至少做一次恢复演练。

演练内容：

- 从备份恢复到新环境。
- 校验集合数量。
- 校验关键索引。
- 校验核心数据。
- 应用连接恢复库测试。
- 记录恢复耗时。

## 7. 权限和安全

备份通常包含敏感数据。

需要：

- 加密存储。
- 限制下载权限。
- 审计恢复操作。
- 备份脱敏用于测试。
- 不把生产备份随意放到个人电脑。

## 8. 备份监控

必须监控：

- 备份是否成功。
- 备份耗时。
- 备份大小变化。
- 最近一次成功备份时间。
- 恢复演练是否通过。

备份失败应该告警。

## 10. 本节小结

你需要记住：

- 生产备份必须自动化。
- Atlas Backup 是云上托管方案。
- 备份和归档不是一回事。
- RPO/RTO 决定备份策略。
- 恢复演练比备份命令本身更重要。

## 11. RPO 和 RTO 要说人话

备份策略最核心的两个指标：

```text
RPO：最多能丢多少数据。
RTO：多久内必须恢复服务。
```

示例：

```text
RPO = 5 分钟
表示灾难发生时，最多接受丢 5 分钟数据。

RTO = 30 分钟
表示灾难发生后，30 分钟内要恢复可用。
```

如果业务是支付、订单、钱包，RPO 通常要非常低。  
如果业务是埋点分析，RPO 可以结合消息队列重放和离线补偿设计。

## 12. 备份方案选择

| 场景 | 方案 | 说明 |
| --- | --- | --- |
| Atlas 生产集群 | Atlas Backup | 优先使用托管备份和时间点恢复 |
| 小型自建库 | mongodump | 简单，但大库恢复慢 |
| 大型自建库 | 文件系统/云盘快照 | 要保证一致性和演练 |
| 低 RPO | 连续备份/增量能力 | 成本更高 |
| 长期分析 | 归档到数仓/对象存储 | 不是灾备恢复替代品 |

不要只讨论“怎么备份”，还要讨论“怎么恢复”。

## 13. 恢复演练步骤

一次合格的恢复演练至少包括：

```text
1. 选择一个备份点。
2. 恢复到隔离环境。
3. 校验数据库、集合、索引。
4. 校验核心业务数据。
5. 启动 Go 服务连接恢复环境。
6. 跑核心接口 smoke test。
7. 记录恢复耗时。
8. 记录问题和改进项。
```

校验命令示例：

```javascript
show dbs
use appdb
show collections
db.orders.countDocuments()
db.orders.getIndexes()
db.orders.findOne()
```

Go 服务 smoke test 可以包括：

```text
健康检查
用户登录
订单详情
订单列表
核心写入接口在测试环境验证
```

## 14. 备份安全

备份通常比线上数据库更危险，因为它可能被下载、复制、长期保存。

必须考虑：

- 备份加密。
- 最小权限访问。
- 下载审计。
- 恢复操作审计。
- 测试环境恢复时脱敏。
- 离职人员权限回收。
- 不把生产备份放到个人电脑或网盘。

测试环境需要生产数据时，优先脱敏：

```text
手机号 -> 保留前三后四
邮箱 -> 哈希或替换域名
身份证 -> 删除或脱敏
地址 -> 删除详细地址
token -> 全部清空
```

## 15. Go 后端和备份的关系

备份不只是 DBA 的事。Go 后端工程师也要配合：

- 明确哪些集合是核心业务数据。
- 明确哪些集合可以从消息队列或日志重建。
- 给关键写入设计幂等键，方便恢复后补偿。
- 写数据校验脚本。
- 准备恢复环境的配置切换。
- 确认服务启动时不会误连生产依赖。

例如恢复后校验订单：

```go
count, err := orders.CountDocuments(ctx, bson.M{
    "created_at": bson.M{
        "$gte": start,
        "$lt":  end,
    },
})
if err != nil {
    return err
}
fmt.Println("orders:", count)
```

## 16. 生产备份策略示例

```md
# MongoDB 备份策略

## 目标

- RPO：15 分钟
- RTO：60 分钟

## 范围

- appdb.orders：核心
- appdb.payments：核心
- appdb.events：可由消息队列补偿

## 备份方式

- Atlas Continuous Backup
- 每日快照保留 30 天
- 月度快照保留 12 个月

## 恢复演练

- 每月恢复到 staging-restore
- 后端执行 smoke test
- DBA 校验索引和复制状态

## 权限

- 只有 DBA 组可恢复
- 安全组审批后可下载
- 所有操作进入审计

## 告警

- 最近一次备份失败
- 备份延迟超过阈值
- 快照大小异常波动
```
