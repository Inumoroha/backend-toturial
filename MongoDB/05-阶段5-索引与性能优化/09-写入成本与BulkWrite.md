# 09. 写入成本与 BulkWrite

索引会加快读取，但会增加写入成本。本节学习索引对写入的影响，以及如何用批量写入减少网络往返。

## 1. 索引为什么拖慢写入

插入一条文档时，MongoDB 不只写集合数据，还要更新相关索引。

如果集合有 5 个索引：

```text
插入文档
  -> 写入集合
  -> 更新索引 1
  -> 更新索引 2
  -> 更新索引 3
  -> 更新索引 4
  -> 更新索引 5
```

索引越多，写入成本越高。

## 2. 更新索引字段也有成本

如果索引：

```javascript
{ status: 1, created_at: -1 }
```

更新 `status`：

```javascript
db.orders.updateOne(
  { _id: orderId },
  { $set: { status: "paid" } }
)
```

MongoDB 需要维护索引。

更新非索引字段通常成本较低。

## 3. 删除也要维护索引

删除文档时，也要删除对应索引项。

所以批量删除大量文档时会有压力。

真实项目中，清理历史数据要：

- 分批执行。
- 控制速率。
- 避开高峰。
- 观察数据库负载。

## 4. 批量写入 BulkWrite

如果你要执行很多写操作，不要每条都单独请求数据库。

可以使用批量写：

```javascript
db.orders.bulkWrite([
  {
    updateOne: {
      filter: { order_no: "O202607000001" },
      update: { $set: { status: "paid" } }
    }
  },
  {
    updateOne: {
      filter: { order_no: "O202607000002" },
      update: { $set: { status: "cancelled" } }
    }
  }
])
```

好处：

- 减少网络往返。
- 批量执行更高效。
- 适合导入、同步、批量修复。

## 5. 有序和无序批量写

有序：

```javascript
db.orders.bulkWrite(operations, { ordered: true })
```

遇到错误可能停止后续操作。

无序：

```javascript
db.orders.bulkWrite(operations, { ordered: false })
```

遇到某些错误时尽量继续执行其他操作。

适合批量导入和尽量写入成功数据的场景。

## 6. 批量写不是越大越好

一批太大可能导致：

- 内存压力。
- 单次请求太大。
- 错误难定位。
- 锁和资源占用时间变长。

常见做法：

```text
每批 500 / 1000 / 5000 条，根据数据大小和环境测试决定。
```

## 7. 写入优化清单

- 删除不必要索引。
- 避免频繁更新索引字段。
- 批量导入前评估是否先导入再建索引。
- 使用 `BulkWrite` 减少往返。
- 大批量更新分批执行。
- 写入时只写必要字段。
- 热点文档避免高并发集中更新。

## 8. 统计字段更新

浏览量、点赞数这类高频计数：

```javascript
db.posts.updateOne(
  { _id: postId },
  { $inc: { "stats.views": 1 } }
)
```

如果极高频，可能需要：

- Redis 缓冲。
- 批量落库。
- 近似计数。
- 分片计数。

不要让单个热门文档成为写入瓶颈。

## 9. 本节练习

完成：

1. 给 orders 创建多个索引。
2. 批量插入 1000 条订单，观察耗时。
3. 删除非必要索引后再批量插入，对比。
4. 使用 `bulkWrite` 批量更新一批订单状态。
5. 尝试 `ordered: false`。
6. 思考哪些索引是必须的，哪些可以删除。

## 10. 本节小结

你需要记住：

- 索引会增加插入、更新、删除成本。
- 更新索引字段比更新普通字段更重。
- `BulkWrite` 可以减少大量写操作的网络往返。
- 批量写入要控制批大小。
- 读性能和写性能要一起评估。
