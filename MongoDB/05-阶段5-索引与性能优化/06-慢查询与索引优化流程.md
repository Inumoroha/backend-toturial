# 06. 慢查询与索引优化流程

慢查询优化不要靠感觉。你需要一套稳定流程：确定业务查询、复现、explain、设计索引、对比、评估写入成本、上线监控。

## 1. 什么是慢查询

慢查询不是绝对的“超过多少毫秒”，而是相对业务要求。

例如：

- 登录查用户：希望几十毫秒内。
- 订单列表：希望 100ms 级别。
- 后台报表：几秒可以接受。
- 离线统计：分钟级也可能可以接受。

不同场景标准不同。

## 2. 慢查询常见原因

- 没有索引。
- 索引字段顺序不合理。
- 查询条件无法使用索引。
- 排序没有索引支持。
- 返回数据太多。
- 投影返回大字段。
- 深分页 `skip` 太大。
- 正则查询无法高效使用索引。
- `$lookup` 关联数据太大。
- `$unwind` 放大中间结果。

## 3. 优化流程

```text
1. 找到慢接口和慢查询
2. 拿到真实 filter、sort、limit、projection
3. 在测试环境构造接近真实的数据量
4. 跑 explain("executionStats")
5. 判断 COLLSCAN、SORT、扫描量
6. 设计或调整索引
7. 再跑 explain 对比
8. 评估写入成本和索引数量
9. 上线后监控
```

## 4. 例子：订单列表慢

接口：

```text
GET /users/{id}/orders?status=paid&page=1&page_size=20
```

查询：

```javascript
db.orders.find({
  user_id: user1._id,
  status: "paid"
}).sort({ created_at: -1 }).limit(20)
```

没有索引时可能 COLLSCAN。

设计索引：

```javascript
db.orders.createIndex({ user_id: 1, status: 1, created_at: -1 })
```

再 explain。

## 5. 例子：后台订单列表

接口：

```text
GET /admin/orders?status=paid
```

查询：

```javascript
db.orders.find({
  status: "paid"
}).sort({ created_at: -1 }).limit(50)
```

索引：

```javascript
db.orders.createIndex({ status: 1, created_at: -1 })
```

注意：这个索引和用户订单列表索引不同。不要指望一个索引满足所有查询。

## 6. 例子：正则查询慢

```javascript
db.posts.find({
  title: { $regex: "MongoDB" }
})
```

这种无前缀正则可能很难高效利用普通索引。

如果是前缀：

```javascript
db.posts.find({
  title: { $regex: "^MongoDB" }
})
```

可能更有机会利用索引。

复杂搜索建议考虑：

- 文本索引。
- Atlas Search。
- Elasticsearch。

## 7. 例子：返回字段太大

文章列表不需要返回 `content`：

```javascript
db.posts.find(
  { status: "published" },
  { title: 1, tags: 1, "stats.views": 1, created_at: 1 }
).sort({ created_at: -1 }).limit(20)
```

投影可以减少网络传输和 BSON 解码成本。

## 8. 低选择性字段

`status` 只有几个值：

```text
paid / cancelled / pending_payment
```

单独索引：

```javascript
{ status: 1 }
```

不一定总有效，因为每个 status 可能匹配大量文档。

更常见是配合排序或其他字段：

```javascript
{ status: 1, created_at: -1 }
```

或者：

```javascript
{ user_id: 1, status: 1, created_at: -1 }
```

## 9. 索引数量控制

每加一个索引，都要问：

- 支持哪个查询？
- 查询频率多高？
- 是否和已有索引重复？
- 是否能通过复合索引前缀复用？
- 写入成本是否可接受？
- 是否需要唯一约束？

不要为了“可能以后用到”建索引。

## 10. 本节练习

对下面慢查询写优化方案：

2. 用户订单列表慢。

4. 文章列表返回 content 导致响应大。

7. 深分页第 10000 页慢。

每个写：

```text

建议索引或改写：
副作用：
```

## 11. 本节小结

你需要记住：

- 慢查询优化要有流程。
- explain 是证据，不是装饰。
- 索引字段顺序要匹配 filter + sort。
- 投影能减少大字段返回。
- 深分页和模糊搜索通常不是简单加索引能解决。
- 索引数量需要控制。
