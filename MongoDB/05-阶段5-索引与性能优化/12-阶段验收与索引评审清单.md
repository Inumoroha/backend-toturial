# 12. 阶段验收与索引评审清单

这是第 5 阶段的验收文件。你应该能根据查询设计索引，并用 explain 证明索引有效。

## 1. 概念验收

请回答：

1. 什么是 COLLSCAN？
2. 什么是 IXSCAN？
3. 为什么 `_id` 查询通常很快？
4. 唯一索引除了加速查询，还有什么作用？
5. TTL 索引适合哪些数据？
6. 部分索引解决什么问题？
7. 复合索引为什么字段顺序重要？
8. ESR 分别代表什么？
9. `nReturned` 是什么？
10. `totalDocsExamined` 是什么？
11. 为什么 `totalDocsExamined` 远大于 `nReturned` 通常不健康？
12. 覆盖索引为什么可能让 `totalDocsExamined` 为 0？
13. 深分页为什么慢？
14. 游标分页为什么更适合大列表？
15. 索引为什么会拖慢写入？

## 2. 必须掌握的命令

### 创建索引

```javascript
db.users.createIndex({ email: 1 }, { unique: true })
```

### 复合索引

```javascript
db.orders.createIndex({ user_id: 1, status: 1, created_at: -1 })
```

### TTL 索引

```javascript
db.verify_tokens.createIndex({ expired_at: 1 }, { expireAfterSeconds: 0 })
```

### 部分索引

```javascript
db.users.createIndex(
  { email: 1 },
  {
    unique: true,
    partialFilterExpression: {
      deleted_at: { $exists: false }
    }
  }
)
```

### 查看索引

```javascript
db.orders.getIndexes()
```

### 删除索引

```javascript
db.orders.dropIndex("idx_orders_user_status_created_at")
```

### explain

```javascript
db.orders.find({
  user_id: user1._id,
  status: "paid"
}).sort({ created_at: -1 }).explain("executionStats")
```

## 3. 索引评审清单

每个新索引上线前，问：

- 支持哪个具体查询？
- 查询频率多高？
- 是否有真实 explain 证据？
- 原查询 `totalDocsExamined` 是多少？
- 新索引后 `totalDocsExamined` 是多少？
- 是否支持排序？
- 是否和已有索引重复？
- 是否能被已有复合索引前缀覆盖？
- 写入成本是否可接受？
- 索引字段是否经常更新？
- 索引是否需要唯一约束？
- 是否受软删除影响？
- 是否需要自定义索引名？

## 4. explain 评审清单

看 explain 时检查：

- 是否出现 `COLLSCAN`？
- 是否出现 `IXSCAN`？
- 是否出现单独 `SORT`？
- `nReturned` 是多少？
- `totalKeysExamined` 是多少？
- `totalDocsExamined` 是多少？
- 扫描量是否接近返回量？
- 是否用了预期索引？
- 是否因为测试数据太少导致结论不可靠？

## 5. 慢查询优化清单

优化慢查询时：

```text
1. 找到真实查询
2. 固定 filter/sort/limit/projection
3. 准备接近真实的数据量
4. 跑 explain
5. 判断瓶颈
6. 设计索引或改写查询
7. 再跑 explain
8. 记录优化报告
9. 评估写入成本
10. 上线监控
```

## 6. 常见错误

### 看到字段就建索引

错误。索引服务于查询，不服务于字段。

### 只建单字段索引

真实查询往往是 filter + sort + limit，需要复合索引。

### 不看 explain

索引是否有效必须验证。

### 忽略写入成本

索引越多，写入越重。

### 深分页一直 skip

大列表应考虑游标分页。

### 文本搜索用普通 regex 硬撑

复杂搜索应考虑文本索引、Atlas Search 或 Elasticsearch。

## 7. 进入第 6 阶段的标准

满足下面条件，可以进入第 6 阶段：

- 能为用户邮箱设计唯一索引。
- 能为订单列表设计复合索引。
- 能解释 ESR 原则。
- 能使用 explain 对比优化前后。
- 能看懂 COLLSCAN、IXSCAN、nReturned、totalDocsExamined。
- 能设计 TTL 索引。
- 能说明部分索引适合软删除唯一约束。
- 能把深分页改成游标分页。
- 能解释索引对写入的影响。
- 能在 Go 中创建索引并使用 BulkWrite。

第 6 阶段会进入事务、一致性与并发。到那时，你会把唯一索引、原子更新、幂等键和事务结合起来，解决转账、库存扣减、重复提交等高风险问题。
