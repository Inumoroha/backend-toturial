# 01. 索引基本原理与 COLLSCAN

索引的目的很朴素：让数据库少扫描文档，更快找到目标数据。

## 1. 没有索引时发生什么

假设 `orders` 有 100 万条文档，你查询某个用户订单：

```javascript
db.orders.find({ user_id: ObjectId("...") })
```

如果 `user_id` 没有索引，MongoDB 可能需要从头到尾扫描整个集合，这叫：

```text
COLLSCAN
```

也就是 collection scan。

问题：

- 数据越多越慢。
- 扫描大量无关文档。
- 消耗 CPU 和 IO。
- 高并发下会拖垮数据库。

## 2. 有索引时发生什么

创建索引：

```javascript
db.orders.createIndex({ user_id: 1 })
```

再查询：

```javascript
db.orders.find({ user_id: ObjectId("...") })
```

MongoDB 可以通过索引快速定位用户订单，这通常会出现：

```text
IXSCAN
```

也就是 index scan。

## 3. 索引像什么

可以把索引理解成书的目录。

没有目录：

```text
从第一页翻到最后一页
```

有目录：

```text
先查目录，再跳到对应章节
```

索引不是数据本身，而是数据库维护的额外结构。

## 4. _id 默认有索引

每个集合的 `_id` 默认有唯一索引。

所以：

```javascript
db.users.findOne({ _id: ObjectId("...") })
```

通常很快。

但这不代表其他字段也有索引。

例如：

```javascript
db.users.findOne({ email: "alice@example.com" })
```

如果没有 `email` 索引，仍然可能扫描集合。

## 5. 索引的代价

索引会加快读取，但不是免费。

代价：

- 占用磁盘空间。
- 占用内存缓存。
- 插入时要更新索引。
- 更新索引字段时要维护索引。
- 删除文档时要删除索引记录。
- 索引太多会让写入变慢。

所以不要“每个字段都建索引”。

## 6. 索引设计从查询出发

不要这样问：

```text
这个字段要不要建索引？
```

要这样问：

```text
有哪些高频查询？
查询条件是什么？
排序字段是什么？
分页方式是什么？
返回字段是什么？
数据量多大？
写入频率多高？
```

索引服务于查询模式。

## 7. 常见需要索引的字段

通常需要考虑索引：

- 登录邮箱、手机号。
- 业务唯一号，如 `order_no`。
- 外键引用字段，如 `user_id`、`post_id`。
- 高频筛选字段，如 `status`。
- 排序字段，如 `created_at`。
- TTL 过期字段，如 `expired_at`。

但是否建索引，还要结合查询组合。

例如订单列表：

```javascript
db.orders.find({
  user_id: userId,
  status: "paid"
}).sort({ created_at: -1 })
```

比起分别建：

```javascript
{ user_id: 1 }
{ status: 1 }
{ created_at: -1 }
```

更可能需要复合索引：

```javascript
{ user_id: 1, status: 1, created_at: -1 }
```

## 8. 索引不是万能药

索引无法拯救所有问题：

- 查询条件写错。
- 返回数据量太大。
- 大量 `$regex` 模糊查询。
- 大 skip 深分页。
- 聚合中 `$unwind` 后数据爆炸。
- 低选择性字段单独索引效果差。
- 文档模型设计不合理。

例如 `status` 只有三个值：

```text
paid / cancelled / pending
```

单独给 `status` 建索引不一定总有用，因为每个状态可能匹配大量文档。

判断下面查询是否需要索引，应该考虑什么索引：

1. 根据 `_id` 查询用户。

3. 查询某用户订单，按创建时间倒序。


## 10. 本节小结

你需要记住：

- 没有索引可能出现 COLLSCAN。
- 有合适索引通常能出现 IXSCAN。
- `_id` 默认有唯一索引。
- 索引加快读，但拖慢写，并占空间。
- 索引设计必须从查询模式出发。
