# 07. 覆盖索引与投影优化

覆盖索引是指查询所需的字段都能从索引中获得，不需要再读取完整文档。它可以减少文档读取，但需要非常清楚查询字段和索引字段。

## 1. 什么是覆盖索引

假设索引：

```javascript
db.users.createIndex({ email: 1, name: 1, status: 1 })
```

查询：

```javascript
db.users.find(
  { email: "user100@example.com" },
  { _id: 0, email: 1, name: 1, status: 1 }
)
```

过滤字段和返回字段都在索引里，MongoDB 可能不需要读取完整文档。

## 2. 为什么 _id 要注意

MongoDB 默认返回 `_id`。

如果索引不包含 `_id`，查询却返回 `_id`，就可能不是覆盖查询。

所以常见写法：

```javascript
db.users.find(
  { email: "user100@example.com" },
  { _id: 0, email: 1, name: 1, status: 1 }
)
```

显式排除 `_id`。

## 3. 用 explain 验证覆盖

```javascript
db.users.find(
  { email: "user100@example.com" },
  { _id: 0, email: 1, name: 1, status: 1 }
).explain("executionStats")
```

重点看：

```text
totalDocsExamined
```

如果是 0，说明没有读取完整文档，可能是覆盖查询。

## 4. 覆盖索引适合什么

适合：

- 高频小字段查询。
- 登录前根据 email 查少量字段。
- 下拉列表。
- 只返回 ID、名称、状态的列表。
- 报表查询只需索引字段。

不适合：

- 需要返回大文档。
- 字段很多，导致索引过宽。
- 写入非常频繁。
- 低频查询。

## 5. 投影优化

即使不是覆盖索引，投影也有价值。

文章列表不返回 `content`：

```javascript
db.posts.find(
  { status: "published" },
  { title: 1, tags: 1, "stats.views": 1, created_at: 1 }
)
```

好处：

- 减少网络传输。
- 减少客户端解码。
- 避免列表接口返回大字段。

## 6. Go 中使用 Projection

```go
opts := options.Find().
    SetProjection(bson.D{
        {"email", 1},
        {"name", 1},
        {"status", 1},
        {"_id", 0},
    })

cursor, err := collection.Find(ctx, filter, opts)
```

需要导入：

```go
"go.mongodb.org/mongo-driver/v2/mongo/options"
```

## 7. 索引过宽的问题

为了覆盖查询，把很多字段都放进索引：

```javascript
db.users.createIndex({
  email: 1,
  name: 1,
  status: 1,
  city: 1,
  created_at: 1,
  avatar: 1,
  bio: 1
})
```

这可能得不偿失。

问题：

- 索引变大。
- 写入变慢。
- 缓存压力变大。
- 维护成本上升。

覆盖索引只适合高频、字段少、收益明确的查询。

## 8. 本节练习

完成：

1. 给 users 创建 `{ email: 1, name: 1, status: 1 }` 索引。
2. 查询 email，只返回 email、name、status，排除 `_id`。
3. 用 explain 看 `totalDocsExamined`。
4. 去掉 `_id: 0` 再看 explain。
5. 对 posts 列表使用 projection 排除 content。
6. 讨论是否值得为 posts 列表做覆盖索引。

## 9. 本节小结

你需要记住：

- 覆盖索引要求查询和返回字段都在索引中。
- 默认返回 `_id`，会影响覆盖查询。
- 用 explain 验证，不要猜。
- 投影即使不覆盖，也能减少传输和解码。
- 不要为了覆盖而创建过宽索引。
