# 11. 阶段验收与并发安全清单

这是第 6 阶段的验收文件。你应该能判断一个写入流程是否并发安全，而不是只看代码能不能跑。

## 1. 概念验收

请回答：

1. MongoDB 单文档原子性是什么意思？
2. 为什么库存扣减要把库存条件放进 update filter？
3. `matchedCount == 0` 可能代表什么？
4. 什么是幂等？
5. 唯一索引如何帮助幂等？
6. 事务能不能替代幂等？
7. 乐观锁解决什么问题？
8. version 字段如何使用？
9. 什么情况下需要多文档事务？
10. 事务有什么成本？
11. Go Driver v2 中事务内操作为什么要用 `mongo.SessionContext`？
12. 事务回调里为什么不能做外部不可回滚副作用？
13. writeConcern majority 解决什么问题？
14. secondary 读可能有什么风险？

## 2. 必须会写的操作

### 库存扣减

```javascript
db.products.updateOne(
  { _id: productId, stock: { $gte: quantity } },
  {
    $inc: { stock: -quantity, sold: quantity },
    $set: { updated_at: new Date() }
  }
)
```

### 状态流转

```javascript
db.orders.updateOne(
  { _id: orderId, status: "pending_payment" },
  {
    $set: {
      status: "paid",
      paid_at: new Date(),
      updated_at: new Date()
    }
  }
)
```

### 乐观锁

```javascript
db.products.updateOne(
  { _id: productId, version: expectedVersion },
  {
    $set: { name: "new name", updated_at: new Date() },
    $inc: { version: 1 }
  }
)
```

### 幂等索引

```javascript
db.orders.createIndex({ idempotency_key: 1 }, { unique: true })
db.wallet_transactions.createIndex({ transaction_no: 1 }, { unique: true })
```

## 3. 并发安全评审清单

每个写接口上线前检查：

- 是否可能重复提交？
- 是否有幂等键？
- 幂等键是否有唯一索引？
- 是否涉及状态流转？
- 状态更新是否带旧状态条件？
- 是否涉及余额或库存？
- 扣减是否把余额/库存条件放进 filter？
- 是否跨多个文档或集合？
- 是否需要事务？
- 事务回调是否可安全重试？
- 事务里是否做了外部副作用？
- 是否有超时和错误处理？
- 重复键错误如何返回？
- 版本冲突如何返回？
- 哪些错误可以重试？

## 4. 常见错误

### 先查库存再扣库存

并发下不安全。

### 只用事务不用幂等

重复请求仍可能重复执行。

### 状态更新不带旧状态

可能重复支付、重复取消、重复回滚。

### 乐观锁冲突自动覆盖

用户编辑场景应提示刷新，不应盲目重试覆盖。

### 事务中调用外部服务

外部服务不可随事务回滚，容易造成不一致。

## 5. 进入第 7 阶段的标准

满足下面条件，可以进入第 7 阶段：

- 能解释单文档原子性。
- 能写出安全库存扣减。
- 能设计幂等键和唯一索引。
- 能解释事务适用场景和成本。
- 能用 Go Driver v2 写事务骨架。
- 能设计钱包转账事务。
- 能设计库存扣减和流水。
- 能解释 readConcern、writeConcern、readPreference 的基本含义。
- 能完成交易与库存综合练习。

第 7 阶段会进入复制集、高可用与备份恢复。到那时，你会在复制集环境中更深入理解 majority、primary、secondary、故障切换和备份恢复。

## 6. 并发安全设计清单

设计任何写接口前，先回答：

```text
是否可能重复请求？
是否可能并发请求？
是否修改金额、库存、权益？
是否只修改一个文档？
是否需要跨集合一致？
是否需要唯一索引？
是否需要幂等键？
失败后能否安全重试？
是否有外部副作用？
```

如果这些问题没有答案，代码很容易在低并发下正常、高并发下出错。

## 7. 常见方案选择

| 问题 | 首选方案 |
| --- | --- |
| 单文档计数增加 | `$inc` |
| 库存不能扣成负数 | 条件更新 |
| 用户邮箱唯一 | 唯一索引 |
| 重复提交订单 | 幂等键 + 唯一索引 |
| 用户资料编辑冲突 | 乐观锁 |
| 钱包转账 | 事务 + 幂等流水 |
| 创建订单并扣多个库存 | 事务或 Saga/补偿 |
| 外部消息投递 | outbox + 幂等消费 |

## 8. Go 代码验收标准

Repository 层应该体现并发安全：

```go
func (r *InventoryRepo) Deduct(
    ctx context.Context,
    productID primitive.ObjectID,
    qty int,
) (bool, error) {
    res, err := r.coll.UpdateOne(ctx,
        bson.M{
            "_id":   productID,
            "stock": bson.M{"$gte": qty},
        },
        bson.M{"$inc": bson.M{"stock": -qty}},
    )
    if err != nil {
        return false, err
    }
    return res.ModifiedCount == 1, nil
}
```

订单创建要有幂等键：

```javascript
db.orders.createIndex(
  { user_id: 1, request_id: 1 },
  { unique: true }
)
```

状态更新要带旧状态：

```javascript
db.orders.updateOne(
  { _id: orderId, status: "pending" },
  { $set: { status: "paid" } }
)
```

## 9. 压测与并发测试

至少写下面测试：

```text
100 个 goroutine 同时扣同一商品库存
重复提交同一个 request_id
两个请求同时修改同一用户资料版本
钱包转账余额不足
事务中途某一步失败
```

测试断言：

```text
库存不能为负
订单不能重复
余额总和不能变化
版本冲突不能静默覆盖
流水不能重复
```

## 10. 事务设计验收

每个事务都要写明：

```text
事务内有哪些数据库操作？
事务外有哪些副作用？
事务回调是否可重试？
是否有幂等键？
是否有唯一索引？
是否可能长时间运行？
是否有大范围扫描？
```

如果事务里调用短信、支付、HTTP API，通常就是危险信号。

## 11. 最终能力目标

阶段 6 完成后，你应该能：

- 用单文档原子更新解决简单并发。
- 用唯一索引和幂等键解决重复请求。
- 用乐观锁解决编辑冲突。
- 判断什么时候需要事务。
- 写出 Go Driver 事务骨架。
- 设计钱包转账和库存扣减。
- 区分 readConcern、writeConcern、readPreference。
- 说明哪些错误可以重试，哪些不能盲目重试。

这一步学扎实，后面的复制集、分片和大规模系统才不会变成“只会搭环境，不会保数据正确”。
