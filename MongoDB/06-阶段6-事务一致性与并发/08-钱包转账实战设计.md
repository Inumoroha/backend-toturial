# 08. 钱包转账实战设计

钱包转账是学习事务的经典场景。它跨多个文档，金额数据要求强一致，非常适合练习事务、幂等和流水设计。

## 1. 业务要求

转账流程：

```text
1. 校验转账金额 > 0
2. A 钱包余额足够
3. A 扣款
4. B 加款
5. 写交易流水
6. 全部成功才提交
7. 重复请求不能重复转账
```

## 2. 集合设计

钱包：

```javascript
{
  _id: ObjectId("..."),
  user_id: ObjectId("..."),
  balance: 10000,
  status: "active",
  version: 1,
  updated_at: ISODate("2026-07-05T10:00:00Z")
}
```

交易流水：

```javascript
{
  _id: ObjectId("..."),
  transaction_no: "T202607050001",
  from_wallet_id: ObjectId("..."),
  to_wallet_id: ObjectId("..."),
  amount: 1000,
  type: "transfer",
  status: "success",
  created_at: ISODate("2026-07-05T10:00:00Z")
}
```

## 3. 索引

```javascript
db.wallets.createIndex({ user_id: 1 }, { unique: true })
db.wallet_transactions.createIndex({ transaction_no: 1 }, { unique: true })
db.wallet_transactions.createIndex({ from_wallet_id: 1, created_at: -1 })
db.wallet_transactions.createIndex({ to_wallet_id: 1, created_at: -1 })
```

`transaction_no` 唯一用于幂等。

## 4. 扣款条件

扣款必须带余额条件：

```javascript
{
  _id: fromWalletId,
  status: "active",
  balance: { $gte: amount }
}
```

更新：

```javascript
{
  $inc: { balance: -amount, version: 1 },
  $set: { updated_at: now }
}
```

## 5. 加款条件

```javascript
{
  _id: toWalletId,
  status: "active"
}
```

更新：

```javascript
{
  $inc: { balance: amount, version: 1 },
  $set: { updated_at: now }
}
```

## 6. 为什么需要事务

如果没有事务：

```text
A 扣款成功
B 加款失败
流水写入失败
```

系统会出现资金不一致。

事务保证三步作为整体提交。

## 7. 为什么仍然需要幂等

如果同一请求重试：

```text
第一次事务成功，但响应超时
客户端重试
```

没有幂等会再次转账。

唯一 `transaction_no` 可以阻止重复。

重复时：

- 查询已有流水。
- 如果存在且成功，返回成功。
- 如果存在但状态异常，进入人工或补偿流程。

## 8. 错误处理

常见错误：

| 错误 | 处理 |
| --- | --- |
| 金额小于等于 0 | 拒绝请求 |
| 付款钱包不存在 | 返回业务错误 |
| 收款钱包不存在 | 返回业务错误 |
| 余额不足 | 返回业务错误 |
| transaction_no 重复 | 查询已有流水并返回 |
| 事务超时 | 返回可重试错误 |
| 网络错误 | 根据幂等键查询最终结果 |

## 9. 事务后外部通知

如果转账成功后要发通知：

不要在事务回调里直接发短信。

推荐：

```text
事务内写 notification_events
事务提交后异步 worker 发送通知
```

这就是 outbox 思路。

## 11. 本节小结

你需要记住：

- 钱包转账适合事务。
- 扣款必须带余额条件。
- 交易流水必须有唯一交易号。
- 事务解决多写一致性，幂等解决重复请求。
- 事务内不要调用外部不可回滚服务。
