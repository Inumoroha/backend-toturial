# 04. 多文档事务适用场景与成本

事务用于让多个操作作为一个整体提交或回滚。MongoDB 支持多文档事务，但事务有成本，不应该滥用。

## 1. 事务解决什么问题

事务保证：

```text
一组操作要么都成功，要么都失败。
```

例如钱包转账：

```text
1. A 余额扣 100
2. B 余额加 100
3. 写交易流水
```

如果只完成前两步或只完成第一步，数据就错了。

这种场景适合事务。

## 2. 什么时候需要事务

通常需要：

- 多个文档必须强一致。
- 多个集合必须同时成功。
- 失败不能靠简单重试或补偿接受。
- 业务金额、库存、权益等高风险数据。

例子：

- 钱包转账。
- 订单创建 + 多商品库存扣减 + 流水。
- 积分扣减 + 奖励发放 + 流水。
- 支付状态更新 + 订单状态更新 + 支付流水。

## 3. 什么时候不需要事务

不需要或不优先使用：

- 单文档原子更新可以解决。
- 唯一索引可以解决重复写。
- 统计字段允许最终一致。
- 日志写入失败可以异步补偿。
- 读多写少的展示冗余可以异步同步。

例如文章浏览量：

```javascript
db.posts.updateOne(
  { _id: postId },
  { $inc: { "stats.views": 1 } }
)
```

不需要事务。

## 4. 事务的成本

事务会带来：

- 更长的锁和资源占用。
- 更复杂的错误处理。
- 更高的延迟。
- 更高的重试成本。
- 对事务大小和执行时间的限制。
- 对开发者理解要求更高。

事务里不要做：

- 网络调用。
- 用户交互。
- 大批量扫描。
- 长时间计算。
- 并行数据库操作。

Go Driver 官方文档提醒：单个事务中不要并行执行操作。

## 5. 事务不是并发安全的全部答案

事务不能替代：

- 唯一索引。
- 幂等键。
- 状态条件。
- 乐观锁。
- 合理索引。

例如重复下单：

即使用事务，如果同一个请求执行两次，仍然可能创建两笔订单，除非有幂等键或唯一约束。

## 6. 事务要求部署环境

多文档事务需要运行在支持事务的部署上，例如副本集或分片集群。

如果你本地只是单节点 standalone，事务可能不可用。

学习事务时建议后面第 7 阶段搭建复制集后再完整验证。当前阶段先理解代码和设计。

## 7. 事务中的错误处理

事务可能因为：

- 写冲突。
- 网络问题。
- 超时。
- 主节点切换。
- 业务校验失败。

而失败。

Go Driver 的 `WithTransaction` 会处理一部分可重试提交逻辑，但你的回调函数仍要写成可安全重试。

这意味着：

- 回调里不要发送外部不可回滚请求。
- 回调里不要产生不可重复副作用。
- 幂等设计仍然重要。

## 9. 本节小结

你需要记住：

- 事务用于多文档、多集合强一致操作。
- 单文档原子更新优先于事务。
- 事务有成本，不适合长时间和大批量操作。
- 事务不能替代幂等和唯一索引。
- 事务回调要能安全重试。

## 10. 事务决策树

遇到一致性问题时，先不要立刻上事务。按这个顺序判断：

```text
是否只修改一个文档？
  -> 是：优先使用单文档原子更新
  -> 否：继续判断

是否可以通过唯一索引防重复？
  -> 是：使用唯一索引 + 幂等处理
  -> 否：继续判断

是否可以接受最终一致？
  -> 是：使用 outbox、消息队列、异步补偿
  -> 否：继续判断

多个文档是否必须同时成功或失败？
  -> 是：考虑事务
```

事务应该是强一致需求下的工具，而不是所有复杂写入的默认答案。

## 11. 事务适合的典型模型

### 钱包转账

涉及：

- A 账户扣款。
- B 账户加款。
- 写交易流水。

这三个动作必须一起成功或失败，适合事务。

### 订单创建与多商品库存

如果订单包含多个商品：

```text
扣减商品 A
扣减商品 B
扣减商品 C
创建订单
写库存流水
```

其中任何一步失败都应该回滚，否则容易出现“订单创建了但库存没扣完”的问题。

### 状态变更与流水

例如支付成功：

```text
更新 payment 状态
更新 order 状态
写 payment_log
```

如果这三者强绑定，也可以考虑事务。

## 12. 事务不适合的场景

### 大批量导入

一次事务里写几十万文档，事务时间长、锁和资源占用高，失败重试成本也高。更适合分批、幂等、断点续传。

### 外部 HTTP 调用

不要在事务回调里调用：

- 支付网关。
- 短信服务。
- 邮件服务。
- 第三方 API。

MongoDB 回滚不了外部系统的副作用。应该先提交本地事务，再通过 outbox 或任务表异步调用外部服务。

### 用户交互

事务不能等待用户确认、验证码输入、人工审批。这类流程应该拆成状态机。

## 13. outbox 模式简介

如果数据库写入成功后要发消息，可以在同一个事务中写业务数据和 outbox：

```text
事务内：
  1. 创建订单
  2. 扣库存
  3. 写 outbox_events

事务外：
  worker 扫描 outbox_events
  投递消息
  标记已发送
```

outbox 文档示例：

```javascript
{
  _id: ObjectId(),
  event_id: "evt_1001",
  topic: "order.created",
  payload: {
    order_id: "o1001"
  },
  status: "pending",
  created_at: ISODate()
}
```

这样可以避免“订单创建成功但消息没发出去”的问题。

## 14. 事务回调为什么必须可重试

Go Driver 的事务辅助方法可能在遇到可重试错误时重新执行回调。因此回调中不能有不可重复副作用。

不要这样：

```go
session.WithTransaction(ctx, func(sc mongo.SessionContext) (any, error) {
    _, err := orders.InsertOne(sc, order)
    if err != nil {
        return nil, err
    }
    sendSMS(order.UserPhone) // 错误：可能被重复调用
    return nil, nil
})
```

更好的方式：

```go
session.WithTransaction(ctx, func(sc mongo.SessionContext) (any, error) {
    _, err := orders.InsertOne(sc, order)
    if err != nil {
        return nil, err
    }
    _, err = outbox.InsertOne(sc, smsEvent)
    return nil, err
})
```

短信由事务外 worker 读取 outbox 后发送。

## 15. 事务大小控制

生产中要控制：

- 事务持续时间。
- 事务内文档数量。
- 事务内查询是否命中索引。
- 是否存在大范围扫描。
- 是否有不必要的读写。
- 是否可能和高频写入产生冲突。

经验原则：

```text
事务越短越好。
事务内操作越明确越好。
事务内不要做复杂聚合。
事务内不要等待外部系统。
```
