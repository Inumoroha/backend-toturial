# 03. 乐观锁与版本号

乐观锁用于解决并发更新覆盖问题。它不阻塞别人修改，而是在写入时检查版本是否仍然匹配。

## 1. 并发覆盖问题

两个管理员同时编辑商品：

```text
管理员 A 读取商品，version = 1
管理员 B 读取商品，version = 1
A 修改价格，保存成功，version = 2
B 修改标题，保存成功，但可能覆盖 A 的价格
```

如果更新时不检查版本，就可能产生丢失更新。

## 2. version 字段

商品文档：

```javascript
{
  _id: ObjectId("..."),
  name: "MongoDB 入门课",
  price: 9900,
  stock: 100,
  version: 1,
  updated_at: ISODate("2026-07-05T10:00:00Z")
}
```

更新时带版本条件：

```javascript
db.products.updateOne(
  {
    _id: productId,
    version: 1
  },
  {
    $set: {
      name: "MongoDB 入门课新版",
      updated_at: new Date()
    },
    $inc: { version: 1 }
  }
)
```

如果 `matchedCount` 是 0，说明版本已变化，需要提示用户刷新后再提交。

## 3. updated_at 也可以作为版本

更新时带 `updated_at` 条件：

```javascript
db.products.updateOne(
  {
    _id: productId,
    updated_at: oldUpdatedAt
  },
  {
    $set: {
      name: "新名称",
      updated_at: new Date()
    }
  }
)
```

但时间精度、序列化和时区处理容易出问题。工程上更推荐明确的整数 `version`。

## 4. 乐观锁适合什么

适合：

- 管理后台编辑商品。
- 用户资料更新。
- 配置更新。
- 内容编辑。
- 不希望后提交覆盖先提交的场景。

不适合单独解决：

- 钱包转账。
- 多集合强一致写入。
- 库存扣减流水一起成功。

这些仍可能需要条件更新、事务或幂等。

## 5. 和条件更新的关系

乐观锁本质也是条件更新。

库存扣减条件：

```javascript
{ stock: { $gte: quantity } }
```

版本更新条件：

```javascript
{ version: expectedVersion }
```

都是把“当前状态必须符合预期”放进 update filter。

## 6. Go 中错误转换

伪代码：

```go
result, err := collection.UpdateOne(ctx,
    bson.D{{"_id", id}, {"version", expectedVersion}},
    bson.D{
        {"$set", bson.D{
            {"name", input.Name},
            {"updated_at", time.Now().UTC()},
        }},
        {"$inc", bson.D{{"version", 1}}},
    },
)
if err != nil {
    return err
}
if result.MatchedCount == 0 {
    return ErrVersionConflict
}
```

上层可以返回：

```text
409 Conflict
```

提示“数据已被他人修改，请刷新后重试”。

## 7. 乐观锁和重试

是否自动重试，要看业务。

可以自动重试：

- 计数累加。
- 不依赖用户输入的后台任务。

不建议自动重试：

- 用户编辑商品标题和价格。
- 管理员编辑配置。

因为自动重试可能覆盖别人修改。更好的体验是让用户刷新并确认。

## 9. 本节小结

你需要记住：

- 乐观锁解决并发覆盖。
- 常用整数 `version` 字段。
- 更新时 filter 带 `version`。
- 更新成功后 `$inc` version。
- `matchedCount == 0` 表示版本冲突或文档不存在，需要区分处理。
