## 2. 功能自测

### 创建订单

- [ ] API 返回 order_id。
- [ ] orders 表有记录。
- [ ] outbox 有 `order.created`。
- [ ] outbox 有 `order.timeout.check`。

### Outbox 发布

- [ ] Outbox Worker 能发布消息。
- [ ] publisher confirm 成功。
- [ ] outbox 状态变为 sent。
- [ ] RabbitMQ 对应队列收到消息。

### 消费者

- [ ] inventory-worker 能预占库存。
- [ ] points-worker 能加积分。
- [ ] notification-worker 能写通知记录。
- [ ] timeout-worker 能取消超时订单。

## 3. 可靠性自测

- [ ] RabbitMQ 停止时创建订单不会丢 outbox。
- [ ] RabbitMQ 恢复后 outbox 能继续发布。
- [ ] 消费者 ack 前崩溃，消息能重新投递。
- [ ] 重复 `order.paid` 不会重复加积分。
- [ ] 失败消息会进入 retry queue。
- [ ] 超过最大重试次数进入 DLQ。
- [ ] DLQ 有可排查的 message_id 和 error。

## 4. 运维自测

执行：

```powershell
docker exec -it ecommerce-rabbitmq rabbitmqctl list_queues name messages_ready messages_unacknowledged consumers
```

你应该能解释：

- 哪些队列是业务队列。
- 哪些队列是重试队列。
- 哪个是 DLQ。
- ready 为什么增长。
- unacked 为什么增长。
- consumers 为 0 代表什么。

## 5. 架构自测

请不用看文档回答：

1. 为什么订单 API 不直接发布 RabbitMQ？
2. Outbox 解决什么问题？
3. publisher confirm 解决什么问题？
4. mandatory publish 解决什么问题？
5. manual ack 解决什么问题？
6. 为什么消费者必须幂等？
7. retry queue 和 DLQ 的区别是什么？
8. 订单超时取消为什么要查最新状态？
9. 为什么不能把图片或大文件直接放进消息？
