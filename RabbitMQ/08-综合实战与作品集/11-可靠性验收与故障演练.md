# 11. 可靠性验收与故障演练

## 1. 本节目标

验证项目是否真的可靠。

不要只测试正常流程。

你要故意制造：

- RabbitMQ 不可用。
- 消费者崩溃。
- 消息重复。
- 消息处理失败。
- DLQ。
- 队列堆积。

## 2. 验收一：RabbitMQ 停止时创建订单

操作：

```powershell
docker stop ecommerce-rabbitmq
```

调用创建订单 API。

预期：

- orders 写入成功。
- outbox_messages 写入 pending。
- API 不直接依赖 RabbitMQ 发布。

恢复：

```powershell
docker start ecommerce-rabbitmq
```

Outbox Worker 重试后，outbox 状态变为 sent。

## 3. 验收二：重复 order.paid

手动向 RabbitMQ 发布两条相同业务订单的 `order.paid`。

预期：

- points_records 只有一条。
- processed_messages 只有一个幂等键。
- 第二条消息直接 ack。

## 4. 验收三：消费者崩溃

让 points-worker 收到消息后 sleep。

在 ack 前停止进程。

预期：

- 消息重新变为 ready。
- 重启 worker 后重新消费。
- 幂等保证积分不重复。

## 5. 验收四：失败重试

构造一条会失败的消息：

```json
{
  "force_fail": true
}
```

预期：

- 消息进入 retry queue。
- TTL 到期后回到业务队列。
- retry_count 增加。
- 超过次数进入 DLQ。

## 6. 验收五：订单超时取消

学习环境使用 15 秒延迟。

流程：

```text
创建订单
不支付
等待 15 秒
timeout-worker 消费
订单变 cancelled
发布 order.cancelled
```

再测试：

```text
创建订单
立即支付
等待 15 秒
timeout-worker 消费
订单仍 paid
```

## 7. 验收六：队列堆积

停止 inventory-worker。

连续创建多个订单。

观察：

```powershell
docker exec -it ecommerce-rabbitmq rabbitmqctl list_queues name messages_ready messages_unacknowledged consumers
```

预期：

```text
order.created.inventory.queue ready 增长，consumers=0
```

启动 inventory-worker 后，ready 下降。

## 9. 本节小结

可靠性不是口头设计。

你要证明：

- RabbitMQ 短暂不可用不丢事件。
- 重复消息不造成重复业务。
- 消费失败可重试。
- 毒消息进入 DLQ。
- 订单超时逻辑不会误取消。
