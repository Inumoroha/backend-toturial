# 09. 消息拓扑设计与命名规范

## 1. 什么是消息拓扑

消息拓扑是指 RabbitMQ 中这些资源的组合方式：

```text
exchange
queue
binding
routing key
consumer
```

一个好的拓扑设计应该让人一眼看出：

- 谁发布消息。
- 消息表达什么业务事实。
- 哪些队列会收到。
- 哪些服务负责消费。
- 新增消费者时如何扩展。

## 2. 先判断消息类型：任务还是事件

设计前先问：

```text
这条消息是 task，还是 event？
```

### Task

Task 表示要做一件事：

```text
发送邮件
生成报表
压缩图片
导入数据
```

特点：

- 通常只需要一个 worker 处理。
- 适合 Work Queue。
- queue 名称偏任务处理者。

### Event

Event 表示已经发生的事实：

```text
user.registered
order.created
order.paid
payment.succeeded
```

特点：

- 多个系统可能都关心。
- 适合 fanout 或 topic。
- routing key 表达业务事实。

## 3. 任务队列设计示例

需求：

```text
后台异步发送邮件。
```

推荐拓扑：

```text
Exchange: notification.task.direct
Type: direct

Routing key: email.send

Queue: notification.email.worker.queue
Binding: notification.task.direct -- email.send --> notification.email.worker.queue
```

消息：

```json
{
  "message_id": "msg-001",
  "event_type": "email.send",
  "payload": {
    "to": "user@example.com",
    "subject": "welcome"
  }
}
```

## 4. 事件广播设计示例

需求：

```text
用户注册成功后，邮件、积分、分析系统都要处理。
```

可以用 topic：

```text
Exchange: user.events
Type: topic

Routing key: user.registered

Queue:
user.registered.email.queue      binding: user.registered
user.registered.points.queue     binding: user.registered
user.registered.analytics.queue  binding: user.registered
```

也可以用 fanout：

```text
Exchange: user.registered.fanout
Type: fanout
```

但如果用户事件后续很多，例如：

```text
user.registered
user.deleted
user.profile.updated
```

更推荐 topic：

```text
Exchange: user.events
```

## 5. 订单事件总线示例

需求：

```text
订单系统会产生多种事件，多个服务按需订阅。
```

推荐拓扑：

```text
Exchange: order.events
Type: topic

Routing keys:
order.created
order.paid
order.cancelled
order.shipped
```

队列：

```text
order.created.inventory.queue   binding: order.created
order.paid.points.queue         binding: order.paid
order.all.audit.queue           binding: order.#
order.created.notify.queue      binding: order.created
```

这样新增审计服务时，只需要新增队列和绑定：

```text
order.events -- order.# --> order.all.audit.queue
```

订单服务不需要知道审计服务的存在。

## 6. 命名规范

### 6.1 Exchange 命名

建议：

```text
业务域.用途
```

示例：

```text
order.events
user.events
notification.task.direct
payment.events
```

### 6.2 Queue 命名

建议：

```text
业务事件.消费者用途.queue
```

示例：

```text
order.created.inventory.queue
order.paid.points.queue
user.registered.email.queue
notification.email.worker.queue
```

### 6.3 Routing Key 命名

建议：

```text
业务域.动作或状态
```

示例：

```text
order.created
order.paid
user.registered
payment.succeeded
notification.email.send
```

## 7. 避免生产者知道消费者队列

不推荐：

```text
order-service -> routing key: inventory.queue
```

这会让生产者知道消费者内部队列名。

推荐：

```text
order-service -> routing key: order.created
inventory-service 自己绑定 order.created
```

生产者表达业务事实，消费者决定是否订阅。

## 8. 一个队列只服务一个消费语义

不要把多个无关业务都塞进一个队列：

```text
all.messages.queue
```

问题是：

- 消费者逻辑复杂。
- 错误处理混乱。
- 重试策略不同。
- 排查困难。

更好的方式：

```text
order.created.inventory.queue
order.paid.points.queue
notification.email.worker.queue
```

## 9. 队列数量不要极端

队列太少：

- 耦合严重。
- 难以按业务监控。
- 重试和死信策略混乱。

队列太多：

- 管理成本高。
- 资源占用高。
- 拓扑复杂。

建议：

```text
按业务消费语义拆队列，而不是每条消息随便建队列。
```

## 10. 设计检查清单

每设计一个消息拓扑，问自己：

- 这是任务还是事件？
- 生产者是否只表达业务事实？
- exchange 类型是否合适？
- routing key 是否稳定且有业务含义？
- queue 名称是否能看出消费者用途？
- 一个消息进入多个队列是否符合预期？
- 新增消费者是否不需要修改生产者？
- 失败重试策略后续能否单独设计？
- 监控时是否能看懂每个 queue 的意义？

## 11. 本节小结

你要记住：

- 任务通常用 Work Queue。
- 事件通常用 fanout 或 topic。
- 生产者应该表达业务事实，不应该关心消费者队列。
- Queue 名称要体现消费语义。
- 好的拓扑设计会让后续可靠性、监控、排查都更容易。
