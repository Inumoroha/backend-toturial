## 2. 练习一：Exchange 参数解释

请解释下面 exchange 应该如何创建：

```text
Name: order.events
Type: topic
Durable: true
Auto delete: false
Internal: false
```

请回答：

1. 为什么订单事件适合 topic？
2. 为什么业务 exchange 通常 durable？
3. 为什么 auto-delete 通常为 false？
4. internal 为 true 会带来什么限制？

参考思路：

```text
订单事件可能有 order.created、order.paid、order.cancelled 等多种事件，topic 方便按模式订阅。业务 exchange 需要在 RabbitMQ 重启后继续存在，所以 durable。auto-delete 不适合长期业务拓扑。internal 为 true 后，普通客户端不能直接发布消息到该 exchange。
```

## 3. 练习二：Queue 参数选择

为下面两个场景选择 queue 参数。

### 场景 A：订单创建后，库存服务消费扣库存

建议：

```text
durable: true
exclusive: false
auto-delete: false
```

原因：

```text
这是长期业务队列，多个库存服务实例可能共同消费，不能绑定单个连接生命周期。
```

### 场景 B：客户端临时订阅日志广播

建议：

```text
durable: false
exclusive: true
auto-delete: true
name: server-named
```

原因：

```text
这是临时订阅，客户端断开后队列应该自动清理。
```

## 4. 练习三：设计通知任务队列

需求：

```text
系统需要异步发送邮件和短信。
```

请设计 exchange、queue、binding。

参考答案：

```text
Exchange:
notification.task.direct
Type: direct

Queue:
notification.email.worker.queue
notification.sms.worker.queue

Binding:
notification.task.direct -- email.send --> notification.email.worker.queue
notification.task.direct -- sms.send --> notification.sms.worker.queue
```

测试消息：

```text
routing key: email.send
payload: {"task_id":"email-001","to":"a@example.com"}
```

预期：

```text
notification.email.worker.queue 收到消息。
```

## 5. 练习四：设计订单事件系统

需求：

```text
订单系统发布 order.created、order.paid、order.cancelled。
库存服务关心 order.created。
积分服务关心 order.paid。
审计服务关心所有订单事件。
通知服务关心 order.created 和 order.paid。
```

参考答案：

```text
Exchange:
order.events
Type: topic

Routing keys:
order.created
order.paid
order.cancelled

Queues and bindings:
order.created.inventory.queue -- order.created
order.paid.points.queue       -- order.paid
order.all.audit.queue         -- order.#
order.notify.queue            -- order.created
order.notify.queue            -- order.paid
```

说明：

```text
一个 queue 可以有多个 binding。
order.notify.queue 同时绑定 order.created 和 order.paid。
```

进入：

```text
Queue A
```

### routing key: order.payment.succeeded

进入：

```text
Queue A
```

### routing key: payment.created

进入：

```text
Queue C

```

## 8. 练习七：Work Queue 和 Pub/Sub 区分

判断下面场景适合哪种模式：

| 场景 | 推荐模式 |
| --- | --- |
| 10000 张图片等待压缩 | Work Queue |
| 用户注册后邮件、积分、分析系统都要处理 | Publish/Subscribe 或 Topic |
| 邮件和短信任务按类型分发 | Routing |
| 订单各种事件按业务模式订阅 | Topic |
| 多个 worker 共同发送邮件任务 | Work Queue |
