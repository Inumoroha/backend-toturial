# 06. 临时队列、独占队列、自动删除队列

## 1. 为什么需要临时队列

不是所有 queue 都是长期业务队列。

有些场景需要临时队列：

- 临时接收广播日志。
- RPC 请求的 reply queue。
- 某个客户端上线期间接收事件。
- 调试和测试。
- 一次性订阅。

这类队列通常和某个客户端连接的生命周期绑定。

## 2. 长期业务队列 vs 临时队列

### 长期业务队列

例如：

```text
order.created.inventory.queue
notification.email.worker.queue
```

特点：

- 名称固定。
- durable。
- 非 exclusive。
- 非 auto-delete。
- 服务重启后仍然存在。
- 消息需要可靠处理。

### 临时队列

例如：

```text
amq.gen-xxxx
```

特点：

- 名称可以由服务端生成。
- 通常 non-durable。
- 通常 exclusive。
- 通常 auto-delete。
- 客户端断开后自动清理。

## 3. Exclusive Queue

Exclusive queue 的含义：

```text
只允许声明它的连接使用。
声明它的连接关闭后，队列会被删除。
```

适合客户端私有队列。

不适合多个消费者共享。

## 4. Auto Delete Queue

Auto delete queue 的含义：

```text
至少有过一个 consumer 后，当最后一个 consumer 取消订阅，队列会自动删除。
```

注意这个条件：

```text
不是创建后没人用就立刻删除。
```

它通常用于临时订阅场景。

## 5. Server-Named Queue

RabbitMQ 客户端可以声明一个没有名字的队列，让服务端生成名字。

生成的名字通常类似：

```text
amq.gen-abc123...
```

这种队列常和 exclusive、auto-delete 一起使用。

典型用途：

```text
客户端临时订阅一个 fanout exchange
RabbitMQ 生成一个临时队列
客户端断开后队列自动删除
```

Go 客户端阶段会写代码实践这个模式。

## 6. 为什么 UI 不适合深入实验临时队列

临时队列常常依赖客户端连接生命周期。

Management UI 通过 HTTP API 创建资源，和普通 AMQP 客户端连接行为不同。

所以本节重点先理解概念，不强制用 UI 做临时队列实验。

后面 Go 代码阶段更适合实践：

```text
QueueDeclare("", false, true, true, false, nil)
```

含义通常是：

- 名称为空，让服务端生成。
- non-durable。
- exclusive。
- auto-delete。

## 7. 临时订阅模式

假设有一个日志广播 exchange：

```text
logs.fanout
```

每个临时日志观察客户端启动后：

```text
声明一个 server-named exclusive auto-delete queue
绑定到 logs.fanout
开始消费日志
```

客户端退出后：

```text
连接关闭
队列删除
绑定删除
```

这个模式很适合临时监听广播消息。

## 8. RPC Reply Queue

RabbitMQ 可以实现 RPC 模式。

请求方发送请求时，指定：

```text
reply_to: 临时回调队列
correlation_id: 请求 ID
```

服务方处理完成后，把响应发到 `reply_to` 指定的队列。

请求方通过 `correlation_id` 匹配响应。

注意：

```text
RPC 模式要谨慎使用。
```

如果业务本质是同步请求，HTTP/gRPC 可能更简单。RabbitMQ RPC 适合少数需要消息队列承载请求响应的场景。

## 9. 常见误区

### 误区一：auto-delete 等于消息处理完就删除队列

不是。

Auto delete 关注 consumer 生命周期，不是消息数量。

### 误区二：exclusive 队列可以被多个服务共享

不能。

Exclusive 队列属于声明它的连接。

### 误区三：业务队列用 exclusive 更安全

不是。

业务队列通常需要多个实例共享消费，不能 exclusive。

## 10. 参数选择建议

### 业务任务队列

```text
durable: true
exclusive: false
auto-delete: false
```

### 临时订阅队列

```text
durable: false
exclusive: true
auto-delete: true
name: 由服务端生成
```

### 测试队列

可以视情况使用：

```text
durable: false
exclusive: false
auto-delete: true
```

## 11. 本节小结

你要记住：

- 长期业务队列和临时队列的参数不同。
- Exclusive 队列绑定连接生命周期。
- Auto-delete 队列绑定消费者生命周期。
- Server-named queue 适合临时队列。
- 业务队列通常不要 exclusive，也不要 auto-delete。
