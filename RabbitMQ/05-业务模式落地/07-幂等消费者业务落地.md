# 07. 幂等消费者业务落地

## 1. 为什么第 5 阶段还要讲幂等

第 4 阶段你已经知道：

```text
至少一次投递意味着消息可能重复。
```

第 5 阶段要把它落到业务代码里。

幂等不是一句：

```text
消费者要幂等。
```

而是要设计：

- 幂等键。
- 数据库唯一索引。
- 本地事务。
- 重复消息处理结果。
- 日志和告警。

## 2. 业务场景：支付成功加积分

事件：

```text
order.paid
```

积分服务收到后：

```text
给用户增加积分
写积分流水
记录已处理消息
ack
```

风险：

```text
order.paid 重复投递
积分重复增加
```

## 3. 幂等键设计

推荐业务幂等键：

```text
order:{order_id}:points:add
```

例如：

```text
order:1001:points:add
```

不要只依赖 message_id。

原因：

```text
同一个 order.paid 事件可能因为补偿、重发、Outbox 重放而有不同 message_id。
```

业务上仍然只能加一次积分。

## 4. 表结构设计

积分流水表：

```sql
CREATE TABLE points_records (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL,
    order_id BIGINT NOT NULL,
    points INT NOT NULL,
    reason VARCHAR(64) NOT NULL,
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uk_order_reason (order_id, reason)
);
```

消息处理表：

```sql
CREATE TABLE processed_messages (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    idempotency_key VARCHAR(128) NOT NULL,
    message_id VARCHAR(128) NOT NULL,
    event_type VARCHAR(64) NOT NULL,
    processed_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uk_idempotency_key (idempotency_key)
);
```

如果你已经在业务表上有唯一约束，有时可以不单独建 processed_messages。

但单独记录消息处理表更方便排查。

## 5. 正确事务流程

```text
收到 order.paid
  -> 解析消息
  -> 开启数据库事务
  -> 插入 processed_messages
  -> 如果唯一键冲突，说明处理过，提交事务并 ack
  -> 写 points_records
  -> 更新 user_points
  -> 提交事务
  -> ack
```

关键：

```text
幂等记录和业务修改必须在同一个本地事务。
```

## 6. Go 伪代码

```go
func HandleOrderPaid(ctx context.Context, event Event) error {
	var payload OrderPaidPayload
	if err := json.Unmarshal(event.Payload, &payload); err != nil {
		return ErrBadMessage
	}

	key := fmt.Sprintf("order:%d:points:add", payload.OrderID)

	tx, err := db.BeginTx(ctx, nil)
	if err != nil {
		return err
	}
	defer tx.Rollback()

	inserted, err := InsertProcessedMessage(ctx, tx, key, event.MessageID, event.EventType)
	if err != nil {
		return err
	}
	if !inserted {
		return tx.Commit()
	}

	points := CalculatePoints(payload.PaidAmount)

	if err := InsertPointsRecord(ctx, tx, payload.UserID, payload.OrderID, points); err != nil {
		return err
	}

	if err := AddUserPoints(ctx, tx, payload.UserID, points); err != nil {
		return err
	}

	return tx.Commit()
}
```

消费者外层：

```go
if err := HandleOrderPaid(ctx, event); err != nil {
	// 根据错误类型进入 retry 或 DLQ
	return
}

_ = d.Ack(false)
```

## 7. 唯一键冲突如何处理

插入 `processed_messages` 时，如果发生唯一键冲突：

```text
说明该业务事件已经处理过。
```

此时应该：

```text
直接 ack。
```

不要报错重试，否则重复消息会一直重试。

## 8. 先写业务表还是先写 processed_messages

推荐在同一事务里：

```text
先插 processed_messages
再执行业务
```

如果插入失败，说明处理过，直接结束。

如果业务执行失败，事务回滚，processed_messages 也回滚。

下次消息重试时可以重新处理。

## 9. 状态机幂等示例

订单取消：

```sql
UPDATE orders
SET status = 'cancelled'
WHERE id = ? AND status = 'pending_payment';
```

如果影响行数为 1：

```text
取消成功。
```

如果影响行数为 0：

```text
可能已经支付、已经取消或订单不存在。
```

这时要查询当前状态判断是否可以 ack。

## 10. 幂等和 DLQ

重复消息不应该进入 DLQ。

重复消息是正常现象，应该：

```text
识别重复 -> 直接 ack
```

DLQ 应该留给：

- 格式错误。
- 业务无法处理。
- 重试多次仍失败。
- 需要人工介入的问题。

## 11. 本节小结

幂等消费者落地要点：

- 用业务唯一键做幂等。
- 使用数据库唯一索引。
- 幂等记录和业务操作放进同一事务。
- 重复消息直接 ack。
- 不要让重复消息污染 DLQ。
- 状态机也是幂等的重要手段。
