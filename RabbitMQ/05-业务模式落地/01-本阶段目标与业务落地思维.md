# 01. 本阶段目标与业务落地思维

## 1. 从机制到业务

前面阶段你已经学习了：

- exchange、queue、binding、routing key。
- Go producer 和 consumer。
- ack、nack、prefetch。
- publisher confirm。
- retry queue 和 DLQ。

第 5 阶段要回答另一个问题：

```text
这些机制如何放进真实业务系统？
```

例如：

- 用户注册后如何异步发通知？
- 订单创建后如何通知库存和风控？
- 数据库事务成功但消息发送失败怎么办？
- 支付成功事件重复消费怎么办？
- 订单 15 分钟未支付如何自动取消？

## 2. 技术正确不等于业务正确

你可能写出了可靠 producer：

```text
publisher confirm 成功
mandatory return 已处理
```

也可能写出了可靠 consumer：

```text
manual ack
retry queue
DLQ
```

但业务仍然可能错。

例如：

```text
订单创建成功
消息发布失败
库存服务永远不知道这笔订单
```

又例如：

```text
支付成功消息重复消费
积分加了两次
```

所以业务落地需要把 RabbitMQ 和数据库、状态机、幂等、补偿放在一起设计。

## 3. 本阶段核心问题

本阶段你要能回答：

- 这是 task 还是 event？
- 生产者是否应该关心消费者？
- 消息 schema 怎么设计？
- 数据库事务和消息发布如何保持一致？
- 消费者如何幂等？
- 失败后如何重试？
- 超过重试次数后谁处理？
- 用户看到的状态如何表达最终一致性？
- 如何测试这些异步流程？

## 4. task 和 event 的区别

### Task

Task 表示：

```text
请帮我做一件事。
```

例如：

```text
email.send
sms.send
report.generate
image.compress
```

通常特征：

- 一条任务只需要一个 worker 处理。
- 使用 direct exchange 或默认 exchange。
- queue 更偏 worker 所有。

### Event

Event 表示：

```text
某件事已经发生。
```

例如：

```text
user.registered
order.created
order.paid
payment.succeeded
```

通常特征：

- 多个消费者可能感兴趣。
- 使用 topic exchange 较多。
- routing key 表达业务事实。

## 5. 一条业务消息的完整生命周期

```text
业务请求
  -> 数据库事务
  -> 记录 outbox
  -> outbox worker 发布 RabbitMQ
  -> RabbitMQ 路由到队列
  -> consumer 消费
  -> 幂等检查
  -> 执行业务
  -> ack
  -> 失败则 retry / DLQ
```

这比简单的：

```text
Publish -> Consume
```

复杂很多，但也更接近真实工作。

## 7. 本节小结

本阶段重点不是学习更多 RabbitMQ API，而是学会：

- 建模业务事件。
- 处理数据库和消息的一致性。
- 设计幂等消费者。
- 设计可恢复的失败流程。
- 把 RabbitMQ 放进 Go 后端项目。
