# 03. 通知系统实战设计

## 1. 业务目标

设计一个通知系统，支持：

- 发送邮件。
- 发送短信。
- 发送站内信。
- 失败重试。
- 超过最大重试次数进入 DLQ。
- 防止重复发送。

这是 RabbitMQ 最适合入门落地的场景之一。

## 2. 通知系统是 task 还是 event

发送通知通常是 task：

```text
notification.email.send
notification.sms.send
notification.inbox.send
```

含义是：

```text
请通知 worker 发送某种通知。
```

每条通知任务通常只需要一个 worker 处理一次。

## 3. 推荐拓扑

```text
Exchange: notification.task.direct
Type: direct

Queues:
notification.email.worker.queue
notification.sms.worker.queue
notification.inbox.worker.queue

Bindings:
notification.task.direct -- notification.email.send --> notification.email.worker.queue
notification.task.direct -- notification.sms.send --> notification.sms.worker.queue
notification.task.direct -- notification.inbox.send --> notification.inbox.worker.queue
```

失败重试：

```text
notification.retry.exchange
notification.email.retry.30s.queue
notification.sms.retry.30s.queue
notification.dlx
notification.dlq
```

## 5. 数据库表设计

通知任务表：

```sql
CREATE TABLE notifications (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    notification_id VARCHAR(64) NOT NULL,
    user_id BIGINT NOT NULL,
    channel VARCHAR(32) NOT NULL,
    target VARCHAR(255) NOT NULL,
    template VARCHAR(64) NOT NULL,
    status VARCHAR(32) NOT NULL,
    retry_count INT NOT NULL DEFAULT 0,
    last_error TEXT NULL,
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uk_notification_id (notification_id)
);
```

状态建议：

```text
pending
sending
sent
failed
dead
```

## 6. 发送流程

API 或业务服务：

```text
写 notifications 表 status=pending
发布 notification.email.send 消息
```

Worker：

```text
收到消息
幂等检查 notification_id
如果已经 sent，直接 ack
更新 status=sending
调用邮件服务商
成功 -> status=sent -> ack
失败 -> 发布 retry 或 DLQ -> ack 原消息
```

## 7. 幂等设计

幂等键：

```text
notification_id
```

发送前检查：

```sql
SELECT status FROM notifications WHERE notification_id = ?;
```

如果已经是：

```text
sent
```

说明处理过，直接 ack。

也可以通过状态更新防并发：

```sql
UPDATE notifications
SET status = 'sending'
WHERE notification_id = ? AND status IN ('pending', 'failed');
```

如果影响行数为 0，需要判断是否已经处理。

## 8. 失败处理策略

通知失败通常可以重试：

```text
第 1 次失败 -> 30 秒后重试
第 2 次失败 -> 5 分钟后重试
第 3 次失败 -> 30 分钟后重试
仍失败 -> DLQ + status=dead
```

学习阶段可以先实现一个固定 30 秒 retry queue。

生产环境可以做多级 retry queue：

```text
notification.retry.30s.queue
notification.retry.5m.queue
notification.retry.30m.queue
```

## 9. API 示例

```text
POST /api/notifications/email
```

请求：

```json
{
  "user_id": 1001,
  "to": "user@example.com",
  "template": "welcome",
  "variables": {
    "name": "Alice"
  }
}
```

响应：

```json
{
  "notification_id": "notice-001",
  "status": "pending"
}
```

注意：API 返回 pending，不代表邮件已经发送成功。

## 10. 用户体验设计

异步通知系统通常不让用户等待发送完成。

例如：

```text
用户注册成功
  -> 返回注册成功
  -> 后台异步发送欢迎邮件
```

邮件失败不应该影响注册成功。

但后台需要：

- 重试。
- 告警。
- 死信处理。
- 后台管理页查看失败通知。

## 11. 本节小结

通知系统落地重点：

- 通知是 task。
- direct exchange 按渠道路由。
- notification_id 做幂等。
- 失败要 retry。
- 最终失败进入 DLQ。
- API 返回 pending，不等于任务已完成。
