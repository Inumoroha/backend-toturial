# 08. 延迟任务与订单超时取消

## 1. 业务目标

订单创建后，如果 15 分钟未支付，自动取消订单。

流程：

```text
用户创建订单
  -> 订单状态 pending_payment
  -> 发送 order.timeout.check 延迟任务
15 分钟后
  -> worker 检查订单状态
  -> 如果仍未支付，取消订单
  -> 发布 order.cancelled
```

## 2. 延迟任务是 task

`order.timeout.check` 是 task：

```text
请在未来某个时间检查订单是否超时。
```

它不是业务事件。

业务事件是：

```text
order.cancelled
```

## 3. TTL + DLX 实现延迟

RabbitMQ 常见延迟任务方案：

```text
生产者 -> 延迟队列
延迟队列 TTL 到期
  -> dead-letter 到业务 exchange
  -> 进入业务处理队列
```

拓扑：

```text
Exchange: order.delay.exchange
Queue: order.timeout.15m.delay.queue
  x-message-ttl: 900000
  x-dead-letter-exchange: order.task.exchange
  x-dead-letter-routing-key: order.timeout.check

Exchange: order.task.exchange
Queue: order.timeout.check.queue
Binding: order.timeout.check
```

## 4. 声明延迟队列

```go
_, err := ch.QueueDeclare(
	"order.timeout.15m.delay.queue",
	true,
	false,
	false,
	false,
	amqp.Table{
		"x-message-ttl":             int32(15 * 60 * 1000),
		"x-dead-letter-exchange":    "order.task.exchange",
		"x-dead-letter-routing-key": "order.timeout.check",
	},
)
```

注意：

```text
x-message-ttl 单位是毫秒。
```

学习时可以改成 10 秒：

```go
"x-message-ttl": int32(10 * 1000)
```

方便观察。

## 5. 创建订单时发送延迟任务

订单创建事务中：

```text
写 orders 表 status=pending_payment
写 outbox_messages，消息类型 order.timeout.check
```

Outbox Worker 发布到：

```text
exchange: order.delay.exchange
routing key: order.timeout.15m
```

消息 payload：

```json
{
  "message_id": "msg-order-timeout-1001",
  "event_type": "order.timeout.check",
  "version": 1,
  "payload": {
    "order_id": 1001
  }
}
```

## 6. 超时检查 worker

worker 消费：

```text
order.timeout.check.queue
```

处理逻辑：

```text
收到 order.timeout.check
查询订单
如果订单不存在 -> ack
如果订单已 paid -> ack
如果订单已 cancelled -> ack
如果订单仍 pending_payment -> 取消订单并发布 order.cancelled
```

## 7. 状态机保护

取消订单时必须带状态条件：

```sql
UPDATE orders
SET status = 'cancelled',
    cancelled_reason = 'payment_timeout'
WHERE id = ?
  AND status = 'pending_payment';
```

如果影响行数为 1：

```text
取消成功，发布 order.cancelled。
```

如果影响行数为 0：

```text
订单可能已支付或已取消，直接 ack。
```

## 8. 为什么不能只靠内存定时器

不推荐：

```go
time.AfterFunc(15*time.Minute, cancelOrder)
```

问题：

- 服务重启后定时器丢失。
- 多实例难协调。
- 大量订单会占用内存。
- 失败后不易重试。

RabbitMQ 延迟队列更适合跨进程、可恢复的异步任务。

## 9. 延迟队列的局限

TTL + DLX 方案有局限：

- 大量不同延迟时间管理复杂。
- 队列级 TTL 不适合每条消息不同延迟。
- 消息堆积时实际触发时间可能晚于预期。
- 严格定时任务不应完全依赖它。

对于复杂调度，可以考虑：

- 数据库任务表 + 扫描器。
- 专门调度系统。
- RabbitMQ delayed message plugin。
- 时间轮。

学习阶段先掌握 TTL + DLX。

## 10. 订单超时取消的关键点

必须注意：

- 创建订单和写 outbox 在同一事务。
- 超时检查要查询最新订单状态。
- 取消订单要使用状态机条件更新。
- 取消成功后发布 `order.cancelled`。
- 重复的 timeout check 应该安全 ack。

## 11. 本节小结

订单超时取消推荐流程：

```text
创建订单 -> outbox 记录延迟任务
Outbox Worker -> 发布到 delay queue
TTL 到期 -> 进入 timeout check queue
Worker -> 检查状态 -> 条件取消 -> 发布 order.cancelled
```
