# 04. 订单事件系统实战设计

## 1. 业务目标

设计一个订单事件系统，支持：

- 订单创建。
- 订单支付。
- 订单取消。
- 库存服务订阅订单创建。
- 积分服务订阅订单支付。
- 通知服务订阅订单创建和支付。
- 审计服务订阅所有订单事件。

## 2. 订单事件是 event

订单事件表达的是：

```text
某个订单状态已经发生变化。
```

典型事件：

```text
order.created
order.paid
order.cancelled
```

生产者不应该发布：

```text
call_inventory
send_order_sms
add_points
```

这些是消费者自己的职责。

## 3. 推荐拓扑

```text
Exchange: order.events
Type: topic

Routing keys:
order.created
order.paid
order.cancelled
```

队列：

```text
order.created.inventory.queue   binding: order.created
order.paid.points.queue         binding: order.paid
order.notify.queue              binding: order.created, order.paid
order.all.audit.queue           binding: order.#
```

消息流：

```text
order-service -> order.events -- order.created --> inventory queue
                                           \--> notify queue
                                           \--> audit queue
```

## 6. 订单状态机

异步系统一定要设计状态机。

简化订单状态：

```text
pending_payment
paid
cancelled
closed
```

允许流转：

```text
pending_payment -> paid
pending_payment -> cancelled
paid -> closed
```

不允许：

```text
cancelled -> paid
paid -> pending_payment
```

消费者遇到乱序或重复事件时，要靠状态机保护业务。

## 7. 库存服务消费 order.created

库存服务逻辑：

```text
收到 order.created
幂等检查 order_id + inventory.reserve
检查库存
预占库存
记录处理完成
ack
```

如果库存不足：

```text
发布 inventory.reserve_failed
或调用订单服务取消订单
```

更完整的长事务处理会在 Saga 中学习。

## 8. 积分服务消费 order.paid

积分服务逻辑：

```text
收到 order.paid
幂等检查 order_id + points.add
计算积分
写积分流水
更新用户积分
ack
```

幂等键：

```text
order:{order_id}:points:add
```

防止重复加积分。

## 9. 通知服务消费多个事件

通知服务可以使用一个 queue 绑定多个 routing key：

```text
order.notify.queue -- order.created
order.notify.queue -- order.paid
```

消费者根据 `event_type` 分派：

```go
switch event.EventType {
case "order.created":
	handleOrderCreated(event)
case "order.paid":
	handleOrderPaid(event)
}
```

也可以拆成两个队列：

```text
order.created.notify.queue
order.paid.notify.queue
```

如果重试策略和处理逻辑差异很大，拆队列更清晰。

## 10. 审计服务订阅所有订单事件

审计队列：

```text
order.all.audit.queue
binding: order.#
```

这样所有订单事件都会进入审计队列。

审计服务可以写入：

```text
order_event_logs
```

用于排查和数据分析。

## 11. 订单事件系统的坑

### 坑一：生产者知道消费者

不要让 order-service 发布：

```text
inventory.reserve
points.add
notify.sms
```

order-service 应该发布：

```text
order.created
order.paid
```

### 坑二：没有状态机

异步消息可能重复、乱序。

没有状态机就容易出现非法状态流转。

### 坑三：消费者不幂等

积分、库存、优惠券都必须幂等。

### 坑四：没有 outbox

数据库订单创建成功，但消息发布失败，会导致下游永远不知道。

下一节重点解决这个问题。

## 12. 本节小结

订单事件系统推荐：

- 使用 topic exchange。
- routing key 表达业务事实。
- 生产者不关心消费者。
- 消费者队列按业务语义命名。
- 状态机防重复和乱序。
- 消费者必须幂等。
- 订单服务发布事件要结合 Outbox。
