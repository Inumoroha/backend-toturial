# 11. 集成测试与本地演练

## 1. 为什么异步流程需要集成测试

RabbitMQ 业务代码不是单元测试就能覆盖的。

你需要验证：

- 消息是否发布到正确 exchange。
- routing key 是否正确。
- 队列是否收到消息。
- consumer 是否 ack。
- 失败是否进入 retry。
- 超过次数是否进入 DLQ。
- outbox 是否能发布成功。
- 重复消息是否被幂等处理。

这些都需要集成测试或本地演练。

## 2. Docker Compose 测试环境

建议在 demo 项目中创建：

```text
docker-compose.yml
```

内容：

```yaml
services:
  rabbitmq:
    image: rabbitmq:4-management
    container_name: rabbitmq-stage5
    hostname: rabbitmq-stage5
    ports:
      - "5672:5672"
      - "15672:15672"
    environment:
      RABBITMQ_DEFAULT_USER: go_learner
      RABBITMQ_DEFAULT_PASS: go_learner_pwd

  mysql:
    image: mysql:8
    container_name: mysql-stage5
    ports:
      - "3306:3306"
    environment:
      MYSQL_ROOT_PASSWORD: root
      MYSQL_DATABASE: rabbitmq_learning
```

启动：

```powershell
docker compose up -d
```

停止：

```powershell
docker compose down
```

## 3. 测试一：通知任务发布

场景：

```text
调用 SendEmail API
写 notifications 表
写 outbox 表
Outbox Worker 发布 notification.email.send
email worker 收到消息
```

验证：

- `notifications.status = pending`。
- `outbox_messages.status = sent`。
- email worker 处理后 `notifications.status = sent`。
- 队列没有消息堆积。

## 4. 测试二：订单事件发布

场景：

```text
创建订单
写 orders 表
写 outbox order.created
Outbox Worker 发布 order.created
inventory worker 消费
```

验证：

- orders 有记录。
- outbox sent。
- inventory reservation 有记录。
- processed_messages 有幂等记录。

## 5. 测试三：重复消息

手动发布两条相同业务事件：

```text
event_type: order.paid
order_id: 1001
```

验证：

- 积分只增加一次。
- points_records 只有一条。
- processed_messages 只有一个幂等键。
- 第二条消息被 ack。

## 6. 测试四：失败重试

让 consumer 故意失败：

```text
payload: {"force_fail": true}
```

验证：

- 消息进入 retry queue。
- TTL 到期后回到业务 queue。
- retry_count 增加。
- 超过最大次数进入 DLQ。

## 7. 测试五：订单超时取消

学习环境把 15 分钟改成 10 秒。

流程：

```text
创建订单 pending_payment
发送 order.timeout.check 延迟任务
10 秒后 worker 检查
订单仍未支付 -> cancelled
发布 order.cancelled
```

验证：

- 订单状态变成 cancelled。
- outbox 发布 order.cancelled。
- audit queue 收到 order.cancelled。

再测一个已支付订单：

```text
创建订单
立即标记 paid
10 秒后 timeout check 到达
worker 发现 paid，直接 ack
```

验证：

- 订单仍是 paid。
- 不会错误取消。

## 8. 测试六：RabbitMQ 暂时不可用

流程：

```text
停止 RabbitMQ
创建订单
orders 和 outbox 写入成功
Outbox Worker 发布失败
启动 RabbitMQ
Outbox Worker 重试发布成功
```

命令：

```powershell
docker stop rabbitmq-stage5
docker start rabbitmq-stage5
```

验证：

- 业务请求不因 RabbitMQ 短暂不可用而丢失待发消息。
- outbox 最终变成 sent。

## 9. 常用测试命令

查看队列：

```powershell
docker exec -it rabbitmq-stage5 rabbitmqctl list_queues name messages_ready messages_unacknowledged consumers
```

查看 binding：

```powershell
docker exec -it rabbitmq-stage5 rabbitmqctl list_bindings
```

查看 exchange：

```powershell
docker exec -it rabbitmq-stage5 rabbitmqctl list_exchanges name type durable
```

查看日志：

```powershell
docker logs -f rabbitmq-stage5
```

## 10. 自动化测试建议

```text
tests/
  integration/
    notification_test.go

    outbox_test.go
    retry_dlq_test.go
```

- 启动 Docker Compose。
- 清理数据库表。
- 清理 RabbitMQ 队列。
- 声明拓扑。

测试后：

- 清理测试数据。

## 11. 本节小结

异步系统必须测试：

- 正常发布和消费。
- 重复消息。
- 失败重试。
- DLQ。
- Outbox 重试。
- 超时任务。
- RabbitMQ 不可用。

不要只测 happy path。
