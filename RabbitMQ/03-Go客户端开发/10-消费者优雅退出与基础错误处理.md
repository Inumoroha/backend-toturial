# 10. 消费者优雅退出与基础错误处理

## 1. 为什么消费者不能随便退出

消费者正在处理消息时，如果进程直接退出，可能导致：

- 消息处理到一半。
- 业务已经写入数据库，但还没 ack。
- RabbitMQ 重新投递消息。
- 业务重复执行。

所以消费者需要优雅退出：

```text
收到退出信号
停止接收新消息
等待当前处理完成
ack 后退出
```

## 2. 基础优雅退出示例

创建：

```text
cmd/graceful_worker/main.go
```

代码：

```go
package main

import (
	"context"
	"log"
	"os/signal"
	"syscall"
	"time"

	amqp "github.com/rabbitmq/amqp091-go"
)

const rabbitURL = "amqp://go_learner:go_learner_pwd@localhost:5672/"
const queueName = "stage3.hello.queue"

func main() {
	ctx, stop := signal.NotifyContext(context.Background(), syscall.SIGINT, syscall.SIGTERM)
	defer stop()

	conn, err := amqp.Dial(rabbitURL)
	if err != nil {
		log.Fatalf("dial rabbitmq: %v", err)
	}
	defer conn.Close()

	ch, err := conn.Channel()
	if err != nil {
		log.Fatalf("open channel: %v", err)
	}
	defer ch.Close()

	q, err := ch.QueueDeclare(queueName, true, false, false, false, nil)
	if err != nil {
		log.Fatalf("declare queue: %v", err)
	}

	const consumerTag = "stage3-graceful-worker"

	msgs, err := ch.Consume(q.Name, consumerTag, false, false, false, false, nil)
	if err != nil {
		log.Fatalf("register consumer: %v", err)
	}

	go func() {
		<-ctx.Done()
		log.Println("shutdown signal received, cancelling consumer")
		if err := ch.Cancel(consumerTag, false); err != nil {
			log.Printf("cancel consumer failed: %v", err)
		}
	}()

	log.Println("worker started. press Ctrl+C to stop")

	for d := range msgs {
		log.Printf("received: %s", string(d.Body))

		if err := handleMessage(d.Body); err != nil {
			log.Printf("handle message failed: %v", err)
			continue
		}

		if err := d.Ack(false); err != nil {
			log.Printf("ack failed: %v", err)
			continue
		}
	}

	log.Println("consumer stopped")
}

func handleMessage(body []byte) error {
	time.Sleep(2 * time.Second)
	log.Printf("handled message: %s", string(body))
	return nil
}
```

运行：

```powershell
go run .\cmd\graceful_worker
```

再用前面的 producer 发消息：

```powershell
go run .\cmd\hello_producer
```

按 `Ctrl+C` 观察退出日志。

## 3. 这个示例的局限

这个示例只用于学习优雅退出的基本形状。

生产环境还需要：

- 处理正在执行的多个 worker goroutine。
- 设置最大退出等待时间。
- 区分临时错误和永久错误。
- 失败时决定 ack、nack、reject。
- 记录处理结果和耗时。
- 配合 Kubernetes preStop 或 systemd 停止流程。

## 4. 基础错误分类

消费者处理消息时，错误大致分三类。

### 4.1 消息格式错误

例如 JSON 无法解析。

通常不应该无限重试。

学习阶段可以直接 ack 并记录日志。

后续可靠性阶段可以进入死信队列。

### 4.2 临时错误

例如：

- 数据库短暂不可用。
- 外部 API 超时。
- 网络抖动。

通常可以重试。

后续会学习 retry queue 和 DLQ。

### 4.3 业务错误

例如：

- 订单状态不允许流转。
- 用户不存在。
- 优惠券已过期。

要根据业务判断是否重试。

## 5. 不要在 demo 中养成的坏习惯

### 坏习惯一：忽略错误

不要这样：

```go
ch, _ := conn.Channel()
```

应该检查错误：

```go
ch, err := conn.Channel()
if err != nil {
	log.Fatalf("open channel: %v", err)
}
```

### 坏习惯二：处理失败也 ack

学习阶段为了避免消息卡住，有时会 ack 格式错误消息。

但真实业务里要认真设计：

```text
成功 -> ack
可重试失败 -> nack/retry
不可重试失败 -> dead letter 或记录失败
```

### 坏习惯三：无限 requeue

失败后一直重新入队，可能导致毒消息反复打爆消费者。

后续阶段会学习最大重试次数。

## 6. 观察消费者数量

运行 consumer 后查看：

```powershell
docker exec -it rabbitmq-dev rabbitmqctl list_queues name consumers messages_ready messages_unacknowledged
```

你应该看到对应队列的 `consumers` 变为 1。

按 `Ctrl+C` 退出后，`consumers` 会变回 0。

## 7. 本节小结

你要记住：

- 消费者需要优雅退出。
- 退出前要停止接收新消息。
- 当前消息处理成功后再 ack。
- 错误要分类处理。
- 失败重试和死信会在第 4 阶段深入。
