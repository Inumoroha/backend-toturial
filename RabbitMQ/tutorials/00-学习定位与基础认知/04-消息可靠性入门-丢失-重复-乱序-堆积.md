# 04. 消息可靠性入门：丢失、重复、乱序、堆积

## 1. 为什么第 0 阶段就要学可靠性

你还没开始写 RabbitMQ 代码，但必须先理解一个原则：

```text
消息系统的难点不是“发出去”，而是“在各种失败情况下仍然可控”。
```

真实后端系统里，消息链路可能在很多地方出问题：

```text
生产者 -> RabbitMQ -> 队列 -> 消费者 -> 数据库或外部服务
```

任何一个环节失败，都可能影响最终结果。

## 2. 问题一：消息丢失

消息丢失可能发生在多个位置。

### 2.1 生产者发送失败

生产者以为自己发了消息，但消息没有真正到达 RabbitMQ。

可能原因：

- 网络断开。
- RabbitMQ 不可用。
- 发布超时。
- 消息路由失败。
- 代码没有检查发布结果。

后续你会学习 publisher confirm，用来确认 RabbitMQ 是否接收了消息。

### 2.2 RabbitMQ 存储失败

消息到达 RabbitMQ 后，如果队列或消息没有正确持久化，RabbitMQ 重启时可能丢失消息。

后续你会学习：

- durable exchange
- durable queue
- persistent message
- quorum queue

### 2.3 消费者提前确认

如果消费者使用 auto ack，RabbitMQ 把消息交给消费者后就认为处理成功。

如果消费者刚收到消息就崩溃：

```text
RabbitMQ 投递消息
RabbitMQ 自动认为成功
消费者还没处理就崩溃
消息丢失
```

所以重要业务通常使用 manual ack：

```text
消费者收到消息
消费者处理业务成功
消费者手动 ack
RabbitMQ 删除消息
```

## 3. 问题二：消息重复

为了避免消息丢失，系统常常会选择“至少一次投递”。

至少一次投递意味着：

```text
消息一定尽量送到，但可能送多次。
```

例如：

```text
消费者处理成功
消费者准备 ack
网络断开
RabbitMQ 没收到 ack
RabbitMQ 重新投递
```

这时业务已经执行过一次，但消息又来了。

### 3.1 重复消息的风险

如果没有幂等，可能出现：

- 用户积分加两次。
- 优惠券发两张。
- 库存扣两次。
- 订单状态被错误覆盖。
- 通知重复发送。

### 3.2 幂等的基本思路

幂等指的是：同一个操作执行一次和执行多次，最终结果一样。

常见做法：

- 每条消息有唯一 `message_id`。
- 业务操作有唯一键，比如 `order_id + event_type`。
- 消费前先检查是否已处理。
- 数据库使用唯一索引防止重复写入。
- 状态机防止重复流转。

示例：

```text
收到 order.paid
检查 message_id 是否处理过
如果处理过 -> 直接 ack
如果没处理 -> 执行业务 -> 记录已处理 -> ack
```

## 4. 问题三：消息乱序

消息乱序指的是：消费者看到消息的顺序和业务发生顺序不一致。

例如业务真实顺序：

```text
order.created
order.paid
order.shipped
```

消费者看到的顺序可能是：

```text
order.paid
order.created
order.shipped
```

### 4.1 乱序为什么会发生

常见原因：

- 多个队列分别传递不同事件。
- 多个消费者并发处理。
- 某条消息失败后延迟重试。
- 网络延迟不同。
- 下游处理时间不同。

### 4.2 如何降低乱序影响

常见策略：

- 使用业务状态机校验状态流转。
- 消息中携带业务版本号。
- 消息中携带事件发生时间。
- 对强顺序场景使用单队列、单 key 或更严格的消费策略。
- 消费者遇到前置状态不存在时延迟重试。

## 5. 问题四：消息堆积

消息堆积是 RabbitMQ 线上最常见的问题之一。

简单说：

```text
生产速度 > 消费速度
```

队列中的 ready 消息数持续增长，就说明消费跟不上生产。

### 5.1 常见原因

- 消费者数量不够。
- 单条消息处理太慢。
- 数据库慢查询。
- 外部 API 响应慢。
- 消费者频繁失败重试。
- prefetch 设置不合理。
- 消费者进程卡住。

### 5.2 堆积的后果

- 用户感知延迟。
- 队列占用内存和磁盘。
- 死信和重试消息增多。
- 业务状态长时间不一致。
- 故障恢复时间变长。

### 5.3 初步处理思路

- 先确认生产速率和消费速率。
- 查看 ready 和 unacked 数量。
- 检查消费者日志。
- 检查下游数据库和外部服务。
- 临时增加消费者。
- 暂停部分非核心生产流量。
- 对毒消息进入死信队列，而不是无限重试。

## 6. 三种投递语义

### 6.1 至多一次

```text
消息最多处理一次，可能丢失。
```

特点：

- 性能高。
- 可靠性弱。
- 适合不重要的日志或指标类消息。

### 6.2 至少一次

```text
消息尽量不丢，但可能重复。
```

特点：

- 业务系统最常见。
- 需要消费者幂等。
- RabbitMQ 可靠消费通常围绕这个目标设计。

### 6.3 恰好一次

```text
业务结果看起来像只执行了一次。
```

在真实分布式系统中，不要轻易相信“绝对恰好一次”。更常见的工程做法是：

```text
至少一次投递 + 幂等消费者 = 业务上的恰好一次效果
```

## 7. 可靠消息链路的基本图

一个相对可靠的消息链路通常长这样：

```text
生产者
  -> 设置 message_id
  -> 发布 persistent message
  -> 等待 publisher confirm

RabbitMQ
  -> durable exchange
  -> durable queue
  -> 必要时使用 quorum queue

消费者
  -> manual ack
  -> 业务幂等
  -> 失败重试
  -> 超过次数进入 DLQ
```

现在你不需要掌握所有细节，只要先形成整体认识。后续每一项都会单独学习。

## 8. 本节小结

学习 RabbitMQ 时，一定要记住四个关键词：

- 丢失：消息没有到达或没有被正确保存、处理。
- 重复：消息被投递多次，消费者必须幂等。
- 乱序：消息处理顺序不一定等于业务发生顺序。
- 堆积：生产速度长期大于消费速度。

只会发消息还不够。能控制这些风险，才是真正会在后端项目中使用消息队列。
