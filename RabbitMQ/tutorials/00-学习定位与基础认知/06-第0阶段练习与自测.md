# 06. 第 0 阶段练习与自测

## 1. 本阶段练习目标

完成本文件中的练习后，你应该能判断：

- 一个业务场景是否适合使用 RabbitMQ。
- 引入 RabbitMQ 后会带来哪些新问题。
- 业务是否需要幂等、重试、死信和监控。
- 自己是否准备好进入第 1 阶段。

## 2. 练习一：用户注册异步化设计

### 题目

设计一个用户注册后的异步处理流程。

注册成功后，系统需要：

- 发送欢迎邮件。
- 发送新人优惠券。
- 初始化积分账户。
- 通知数据分析系统。

请画出消息流向。

### 参考答案

```text
register-api
  -> 写入用户表
  -> 发布 user.registered 消息
  -> 返回注册成功

RabbitMQ
  -> email-service 消费 user.registered
  -> coupon-service 消费 user.registered
  -> points-service 消费 user.registered
  -> analytics-service 消费 user.registered
```

### 你需要思考

- 邮件服务失败时，注册接口是否应该失败？
- 优惠券服务重复消费时，如何避免重复发券？
- 分析系统挂了几个小时，消息是否可以堆积等待恢复？
- 这些消费者的失败是否都需要告警？

## 3. 练习二：判断是否适合使用 RabbitMQ

请判断下面场景是否适合使用 RabbitMQ，并写出理由。

| 场景 | 是否适合 | 理由 |
| --- | --- | --- |
| 用户登录时校验密码 | 通常不适合 | 必须立即返回登录结果 |
| 用户注册后发送欢迎邮件 | 适合 | 不影响注册主流程 |
| 查询商品详情 | 通常不适合 | 用户需要立即得到查询结果 |
| 支付成功后通知积分系统 | 适合 | 可以通过事件驱动解耦 |
| 后台批量生成报表 | 适合 | 耗时任务可异步处理 |
| 扣减库存并立即告诉用户是否下单成功 | 视情况而定 | 强一致要求高时可能需要同步或状态机设计 |

## 4. 练习三：识别异步系统风险

假设订单服务发布了 `order.paid` 消息，积分服务消费后给用户加积分。

请回答：

1. 如果积分服务处理成功但 ack 前崩溃，会发生什么？
2. 如果 `order.paid` 被重复消费，可能有什么后果？
3. 如何避免重复加积分？
4. 如果积分服务挂了 2 小时，RabbitMQ 中会发生什么？
5. 运维上应该监控什么指标？

### 参考思路

1. RabbitMQ 可能重新投递这条消息。
2. 用户可能被重复加积分。
3. 使用 `order_id + event_type` 做唯一键，或记录已处理的 `message_id`。
4. 消息会堆积，恢复后消费者继续处理。
5. 监控队列 ready 数、unacked 数、消费速率、消费者数量、失败日志和死信队列。

## 5. 练习四：写出你的第一版消息设计

为 `user.registered` 设计一个消息结构。

建议包含：

```json
{
  "message_id": "uuid",
  "event_type": "user.registered",
  "version": 1,
  "occurred_at": "2026-07-04T12:00:00Z",
  "correlation_id": "trace-id",
  "payload": {
    "user_id": 10001,
    "email": "user@example.com"
  }
}
```

请思考：

- `message_id` 用来做什么？
- `event_type` 用来做什么？
- `version` 为什么重要？
- `correlation_id` 对排查问题有什么帮助？
- `payload` 中是否应该放敏感信息？

## 6. 练习五：画出同步和异步的区别

请分别画出这两个流程。

### 同步版本

```text
用户注册
  -> 写用户表
  -> 调邮件服务
  -> 调优惠券服务
  -> 调积分服务
  -> 返回成功
```

### 异步版本

```text
用户注册
  -> 写用户表
  -> 发布 user.registered
  -> 返回成功

消费者异步处理：
  -> 邮件服务
  -> 优惠券服务
  -> 积分服务
```

然后回答：

- 哪个版本响应更快？
- 哪个版本更容易保证所有步骤立刻完成？
- 哪个版本对下游服务故障更不敏感？
- 哪个版本需要更多监控和补偿机制？
