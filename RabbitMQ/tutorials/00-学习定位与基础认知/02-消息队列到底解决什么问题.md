# 02. 消息队列到底解决什么问题

## 1. 消息队列的基本模型

先用最简单的模型理解消息队列：

```text
Producer -> Message Queue -> Consumer
```

含义是：

- Producer：生产者，负责发送消息。
- Message Queue：消息队列，负责接收、存储、转发消息。
- Consumer：消费者，负责处理消息。

在 RabbitMQ 中，实际模型会更丰富：

```text
Producer -> Exchange -> Queue -> Consumer
```

第 0 阶段先不用深究 Exchange 的路由规则，只要记住：

- 生产者不直接把消息交给消费者。
- 生产者把消息交给 RabbitMQ。
- RabbitMQ 根据规则把消息放入队列。
- 消费者从队列里取消息处理。

## 2. 问题一：异步处理

很多操作不需要在用户请求里立刻完成。

例如用户上传头像后：

- 原图保存必须尽快完成。
- 头像压缩可以异步。
- 生成多尺寸缩略图可以异步。
- 内容审核可以异步。

同步流程：

```text
用户上传
  -> 保存原图
  -> 压缩图片
  -> 生成缩略图
  -> 审核
  -> 返回成功
```

异步流程：

```text
用户上传
  -> 保存原图
  -> 发送 image.uploaded 消息
  -> 返回成功

worker
  -> 压缩图片
  -> 生成缩略图
  -> 审核
```

异步处理的关键收益是：主流程更快，用户体验更好。

## 3. 问题二：服务解耦

没有消息队列时，一个服务可能直接调用很多服务。

```text
order-service
  -> inventory-service
  -> coupon-service
  -> points-service
  -> notification-service
```

这样的问题是：

- `order-service` 知道太多下游。
- 新增一个下游就要改订单服务。
- 下游故障会影响订单主流程。
- 调用链越来越长。

引入消息后：

```text
order-service -> 发布 order.created

inventory-service -> 订阅 order.created
coupon-service -> 订阅 order.created
points-service -> 订阅 order.created
notification-service -> 订阅 order.created
```

订单服务只需要表达一个事实：订单已创建。

至于谁关心这个事实，由消费者自己决定。

## 4. 问题三：削峰填谷

假设一个系统平时每秒 100 个请求，活动开始后突然每秒 5000 个请求。

如果所有请求都同步打到数据库和下游系统，很容易雪崩。

消息队列可以把瞬时流量变成可控的后台消费：

```text
高峰请求 -> 快速入队 -> 消费者按固定速度处理
```

例如：

```text
每秒进入 5000 条消息
消费者每秒处理 800 条消息
剩余消息暂存在队列中
活动结束后继续慢慢消费
```

但是要注意：如果生产速度长期大于消费速度，队列会持续堆积。削峰不是消灭压力，而是把压力延后处理。

## 5. 问题四：任务分发

一个队列可以有多个消费者共同处理任务。

```text
task.queue
  -> worker-1
  -> worker-2
  -> worker-3
```

这样可以实现水平扩展：

- 处理慢了，就增加 worker。
- 某个 worker 挂了，其他 worker 可以继续工作。
- 不同 worker 可以部署在不同机器上。

典型场景：

- 批量发邮件
- 批量生成报表
- 图片处理
- 爬虫任务
- 数据导入任务

## 6. 问题五：广播通知

有些消息不是只给一个消费者，而是要通知多个系统。

例如支付成功：

```text
order.paid
  -> 订单服务更新状态
  -> 积分服务增加积分
  -> 优惠券服务标记已使用
  -> 通知服务发送短信
  -> 数据分析服务记录事件
```

RabbitMQ 的 exchange 和 queue 组合可以很好地支持这种模式。

## 7. 消息队列解决问题的本质

消息队列的本质不是“让系统更高级”，而是改变系统之间的通信方式：

```text
同步请求响应
  -> 调用方等待结果
  -> 强依赖下游
  -> 适合必须立即知道结果的场景

异步消息
  -> 调用方只表达事件或任务
  -> 消费者稍后处理
  -> 适合不需要立即完成的场景
```

## 8. 常见误区

### 误区一：用了消息队列，系统就一定更可靠

不一定。

如果没有做好 ack、持久化、重试、死信、监控，消息队列反而会让问题更隐蔽。

### 误区二：用了消息队列，就不会有性能问题

不一定。

消息队列只能缓冲压力，不能凭空提升下游处理能力。

### 误区三：异步就是最终一定成功

不一定。

异步任务也可能失败，而且失败后还要设计重试、人工补偿和告警。

### 误区四：所有服务通信都应该用 MQ

不一定。

查询类请求、强实时请求、必须立即拿到返回值的请求，通常更适合同步调用。

## 9. 本节小结

消息队列主要解决五类问题：

- 异步处理
- 服务解耦
- 削峰填谷
- 任务分发
- 广播通知

但它不会自动解决：

- 数据一致性
- 消息丢失
- 消息重复
- 业务幂等
- 监控告警
- 故障恢复

下一节你会学习：异步架构会带来哪些新问题。
