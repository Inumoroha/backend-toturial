# 08. 吞吐、延迟、顺序、回放、大消息取舍

## 1. 为什么这些维度重要

很多选型争论没有意义，是因为没有先定义非功能需求。

你应该先问：

- 需要多少吞吐？
- 可接受多少延迟？
- 是否要求顺序？
- 是否需要历史回放？
- 消息有多大？
- 消息能否重复？
- 故障恢复时间要求是多少？

## 2. 吞吐

如果是：

```text
每秒几十到几千条业务消息
```

RabbitMQ 通常可以胜任，具体看部署和消息大小。

如果是：

```text
海量日志、埋点、数据流
```

Kafka 通常更适合。

判断重点不是绝对数字，而是：

```text
消息类型 + 保留需求 + 消费模型 + 运维能力
```

## 3. 延迟

RabbitMQ 很适合低延迟业务异步。

例如：

```text
用户下单后快速通知库存服务
支付成功后快速通知积分服务
```

但最终延迟还取决于：

- 队列堆积。
- 消费者处理速度。
- prefetch。
- 下游数据库。
- 重试策略。

中间件低延迟不等于业务低延迟。

## 4. 顺序性

严格顺序是很贵的。

RabbitMQ 中如果：

- 多消费者。
- 多队列。
- 重试队列。
- 并发处理。

都可能影响业务处理顺序。

如果强顺序很重要，要考虑：

- 单队列。
- 单消费者或按 key 分区。
- 状态机保护。
- 版本号。
- 幂等。

不要轻易承诺全局严格顺序。

## 5. 回放

普通 RabbitMQ queue：

```text
ack 后消息删除。
```

不适合历史回放。

如果需要回放：

- Kafka。
- RabbitMQ Streams。
- 事件表。
- 审计日志表。

业务事件如果需要审计，可以同时写：

```text
event_logs 表
```

或进入 Kafka。

## 6. 大消息

不要传大消息。

问题：

- 占用内存。
- 占用磁盘。
- 影响网络。
- 降低吞吐。
- 失败重试成本高。

推荐：

```text
大文件放对象存储
消息只放引用
```

示例：

```json
{
  "file_id": "file-001",
  "object_key": "videos/1001.mp4"
}
```

## 7. 延迟任务

RabbitMQ 可以用 TTL + DLX 做延迟任务。

适合：

- 订单超时取消。
- 稍后重试。
- 简单固定延迟任务。

如果需要：

- 海量定时任务。
- 每条不同精确时间。
- 秒级严格调度。
- 复杂日历规则。

可以考虑：

- 数据库任务表。
- 专门调度系统。
- 时间轮。
- 延迟消息插件。

## 8. 保留和审计

如果业务要求：

```text
每个事件必须可审计
```

不要只依赖 RabbitMQ queue。

可以：

- 写 event_logs 表。
- 使用 Outbox 表留存。
- 同步到 Kafka。
- 保存审计日志。

## 9. 可靠性和重复

可靠消息系统通常会带来重复投递可能。

选型时要问：

```text
业务是否能做幂等？
```

如果业务不能接受重复副作用，必须设计：

- 唯一键。
- 状态机。
- 去重表。
- 幂等接口。

而不是只依赖中间件。

## 10. 本节小结

选型前先写清楚：

```text
吞吐
延迟
顺序
回放
消息大小
延迟任务
审计
重复处理
```

没有非功能要求的架构选型，基本都是拍脑袋。
