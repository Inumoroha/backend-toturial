# 01. 本阶段目标与架构取舍思维

## 1. 为什么最后要学架构取舍

前面 6 个阶段，你已经学习了：

- RabbitMQ 基础模型。
- Go 客户端开发。
- 可靠性机制。
- 业务模式落地。
- 生产运维。

现在你不能只停留在：

```text
我会用 RabbitMQ。
```

更高级的能力是：

```text
我知道什么时候该用 RabbitMQ，什么时候不该用。
```

这就是架构取舍。

## 2. 架构选型不是技术崇拜

很多初学者选型会问：

```text
RabbitMQ、Kafka、Redis Streams 谁更强？
```

这个问题不够好。

更好的问题是：

- 这个场景是任务队列还是事件流？
- 需要复杂路由吗？
- 需要事件长期保留和回放吗？
- 吞吐量有多高？
- 延迟要求多低？
- 消费者失败后怎么恢复？
- 团队运维能力如何？
- 数据一致性要求多强？
- 有没有更简单的数据库任务表方案？

## 3. 选型的六个维度

### 3.1 通信模型

你需要的是：

- 同步请求响应。
- 异步任务。
- 业务事件。
- 日志流。
- 数据管道。
- 延迟任务。

不同模型适合不同技术。

### 3.2 吞吐和延迟

RabbitMQ 适合很多低延迟业务消息。

Kafka 更偏高吞吐事件流和日志管道。

Redis Streams 适合 Redis 生态里的轻量流式消费。

数据库任务表适合低吞吐、强事务、简单任务。

### 3.3 路由能力

RabbitMQ 的 exchange、routing key、binding 很强。

如果你需要：

```text
direct
fanout
topic
复杂业务路由
```

RabbitMQ 很合适。

### 3.4 回放和保留

如果你需要：

```text
事件保留数天或数月
多个消费组反复读取
从历史 offset 重放
```

Kafka 或 RabbitMQ Streams 往往更合适。

普通 RabbitMQ queue 不适合长期日志留存和大规模回放。

### 3.5 一致性和事务边界

如果任务必须和数据库事务强绑定，数据库任务表或 Outbox 可能更简单。

RabbitMQ 不是数据库事务的一部分。

关键业务要结合：

```text
Transactional Outbox
幂等消费者
状态机
补偿
```

### 3.6 运维复杂度

技术越强，运维成本通常越高。

你要考虑：

- 团队是否能维护 RabbitMQ 集群。
- 是否已有 Kafka 平台。
- 是否已有 Redis 高可用。
- 是否能做监控、告警、备份、容量规划。

## 4. 架构取舍的基本公式

可以用这个公式思考：

```text
业务需求
  -> 非功能要求
  -> 团队能力
  -> 技术特性
  -> 风险和成本
  -> 最小可行方案
```

不要一开始就上最复杂的方案。

例如：

```text
每天 1000 个后台任务
强事务一致
只有一个服务消费
```

数据库任务表可能比 RabbitMQ 更好。

如果：

```text
多个服务订阅订单事件
需要低延迟异步处理
需要 routing key 分发
```

RabbitMQ 就很自然。

## 5. 本阶段统一判断框架

后续每种技术都按这些维度比较：

- 核心模型。
- 适合场景。
- 不适合场景。
- 可靠性特点。
- 运维复杂度。
- Go 后端落地建议。
- 面试表达方式。

## 6. 本节小结

架构取舍的目标不是证明 RabbitMQ 最好。

而是能说清楚：

```text
在这个业务场景下，为什么 RabbitMQ 是合适的。
如果不用 RabbitMQ，替代方案是什么。
这个选择带来的成本和风险是什么。
```
