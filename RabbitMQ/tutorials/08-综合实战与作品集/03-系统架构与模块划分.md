# 03. 系统架构与模块划分

## 1. 总体架构

建议使用单仓库多进程结构。

```text
api-service
outbox-worker
inventory-worker
points-worker
notification-worker
timeout-worker
RabbitMQ
MySQL/PostgreSQL
```

消息流：

```text
api-service -> outbox_messages -> outbox-worker -> RabbitMQ -> workers
```

## 2. 为什么单仓库多进程

学习阶段不建议一上来拆多个仓库。

单仓库多进程优点：

- 代码集中。
- 本地运行简单。
- 便于共享 domain、mq、outbox 包。
- 适合学习完整链路。

后续要模拟微服务时，可以再拆分仓库。

## 3. 进程划分

### api-service

职责：

- 提供 HTTP API。
- 写业务表。
- 写 outbox 表。
- 查询订单状态。

### outbox-worker

职责：

- 扫描 outbox。
- 发布 RabbitMQ。
- 等待 publisher confirm。
- 标记 sent 或 retry。

### inventory-worker

职责：

- 消费 `order.created`。
- 预占库存。
- 幂等处理。
- 发布库存结果事件。

### points-worker

职责：

- 消费 `order.paid`。
- 加积分。
- 写积分流水。
- 幂等处理。

### notification-worker

职责：

- 消费订单事件。
- 发送通知或写通知表。
- 失败重试。

### timeout-worker

职责：

- 消费 `order.timeout.check`。
- 检查订单状态。
- 条件取消订单。
- 发布 `order.cancelled`。

## 4. 包结构建议

```text
internal/
  config/
  db/
  mq/
  outbox/
  domain/
    order/
    inventory/
    points/
    notification/
  app/
    api/
    workers/
```

说明：

- `mq` 放 RabbitMQ 通用封装。
- `outbox` 放 Outbox Worker 和 repository。
- `domain` 放业务模型和规则。
- `app` 放应用编排。

## 5. 依赖方向

建议依赖方向：

```text
cmd -> app -> domain
cmd -> app -> mq / db / outbox
domain 不依赖 RabbitMQ
```

领域逻辑不要直接依赖 RabbitMQ 客户端。

这样业务更容易测试。

## 6. 架构图

```text
Client
  -> api-service
      -> orders / outbox_messages
      -> outbox-worker
          -> RabbitMQ
              -> inventory-worker
              -> points-worker
              -> notification-worker
              -> timeout-worker
```

## 7. 本节小结

本项目采用：

```text
单仓库
多 cmd 进程
共享 internal 包
RabbitMQ 作为异步消息层
Outbox 作为数据库和消息之间的桥
```
