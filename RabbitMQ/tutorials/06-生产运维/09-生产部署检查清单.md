# 09. 生产部署检查清单

## 1. 为什么需要部署检查清单

RabbitMQ 上线前，如果只确认：

```text
服务能启动
消息能收发
```

远远不够。

生产部署前必须检查：

- 数据持久化。
- 权限。
- 监控。
- 告警。
- 资源限制。
- 备份。
- 故障恢复。
- 客户端配置。

## 2. 基础环境检查

- [ ] RabbitMQ 版本明确。
- [ ] Erlang/OTP 版本与 RabbitMQ 版本兼容。
- [ ] 使用稳定镜像或安装包，不使用漂移的 `latest`。
- [ ] 数据目录持久化。
- [ ] 日志目录可采集。
- [ ] 系统时间同步。
- [ ] 节点主机名稳定。

Docker 环境要特别确认：

```text
/var/lib/rabbitmq 数据目录已挂载 volume。
```

## 3. 资源检查

- [ ] 内存容量满足峰值。
- [ ] 磁盘容量满足堆积和持久化消息。
- [ ] 文件句柄限制足够。
- [ ] 网络带宽满足 publish/consume。
- [ ] CPU 有余量。
- [ ] 设置合理 resource limits。

## 4. 账号和权限检查

- [ ] 不使用默认 `guest/guest`。
- [ ] 应用使用独立账号。
- [ ] 账号按 vhost 隔离。
- [ ] configure/write/read 权限最小化。
- [ ] 管理员账号数量受控。
- [ ] 密码不硬编码。

## 5. 拓扑检查

- [ ] exchange 命名清晰。
- [ ] queue 命名清晰。
- [ ] routing key 稳定。
- [ ] durable exchange。
- [ ] durable queue。
- [ ] 重要消息 persistent。
- [ ] retry queue 配置正确。
- [ ] DLQ 配置正确。
- [ ] 关键队列有消费者。

## 6. 客户端检查

- [ ] Go 客户端复用 connection。
- [ ] channel 数量受控。
- [ ] publisher confirm 已启用。
- [ ] mandatory publish 已处理。
- [ ] manual ack。
- [ ] prefetch 合理。
- [ ] 消费者幂等。
- [ ] 失败进入 retry/DLQ。
- [ ] 优雅退出。
- [ ] 连接断开后可恢复。

## 7. 监控检查

- [ ] Prometheus 已采集 RabbitMQ。
- [ ] Grafana 看板可用。
- [ ] 队列 ready/unacked 可见。
- [ ] consumer 数量可见。
- [ ] publish/deliver/ack rate 可见。
- [ ] 内存和磁盘可见。
- [ ] connection/channel 可见。
- [ ] DLQ 可见。
- [ ] Outbox 指标可见。

## 8. 告警检查

- [ ] 关键队列 ready 堆积告警。
- [ ] unacked 高告警。
- [ ] consumers 为 0 告警。
- [ ] DLQ 有消息告警。
- [ ] 内存告警。
- [ ] 磁盘告警。
- [ ] 连接数异常告警。
- [ ] Outbox pending 过久告警。
- [ ] 每条告警有处理手册。

## 9. 备份和恢复检查

- [ ] definitions 可导出。
- [ ] exchange/queue/binding/policy 可恢复。
- [ ] 配置文件有版本管理。
- [ ] 用户权限有备份。
- [ ] 数据目录备份策略明确。
- [ ] 做过恢复演练。

## 10. 故障演练检查

上线前至少演练：

- [ ] 消费者全部停止。
- [ ] 消费者处理变慢。
- [ ] RabbitMQ 重启。
- [ ] 消息进入 DLQ。
- [ ] RabbitMQ 暂时不可用，Outbox 恢复发布。
- [ ] 磁盘接近阈值时如何处理。
- [ ] 连接数暴涨时如何定位。

## 11. 本节小结

生产上线不是只看代码。

你需要确认：

```text
拓扑正确
客户端可靠
权限收敛
监控告警完整
容量有余量
故障可恢复
```
