# 06. 容量规划与资源水位

## 1. 为什么要做容量规划

RabbitMQ 可以缓冲流量，但不能无限缓冲。

如果长期：

```text
生产速度 > 消费速度
```

最终一定会出现：

- 队列堆积。
- 磁盘增长。
- 内存压力。
- 发布变慢。
- 业务延迟变长。

容量规划就是提前回答：

```text
当前 RabbitMQ 能承受多少消息、多少连接、多少队列、多少峰值？
```

## 2. 先估算消息量

你需要知道：

- 平均 publish rate。
- 峰值 publish rate。
- 平均消息大小。
- 每条消息平均处理耗时。
- 消费者实例数量。
- 可接受最大延迟。

示例：

```text
订单事件峰值：1000 msg/s
单条消息平均：2 KB
消费者处理能力：每实例 100 msg/s
消费者实例：15 个
```

消费能力：

```text
15 * 100 = 1500 msg/s
```

理论上可以覆盖峰值。

## 3. 队列堆积容量估算

如果峰值期间：

```text
生产 5000 msg/s
消费 2000 msg/s
持续 10 分钟
```

堆积：

```text
(5000 - 2000) * 600 = 1,800,000 条消息
```

如果每条消息 2 KB：

```text
约 3.6 GB 原始消息体
```

还要考虑 RabbitMQ 元数据和磁盘开销。

## 4. 消费者扩容估算

如果目标是：

```text
峰值 5000 msg/s，单实例 250 msg/s
```

需要：

```text
5000 / 250 = 20 个实例
```

再留余量：

```text
建议 25 到 30 个实例
```

但扩容消费者不一定总能解决问题。

瓶颈可能在：

- 数据库。
- Redis。
- 外部 API。
- 下游限流。
- 消费者内部锁。

## 5. 连接和 Channel 规划

原则：

```text
Connection 要复用。
Channel 要控制数量。
```

不推荐：

```text
每条消息创建一个 connection
```

推荐：

```text
每个服务实例维护少量长期 connection
发布和消费使用不同 channel
```

监控：

```powershell
docker exec -it rabbitmq-dev rabbitmqctl list_connections name user vhost state channels
```

## 6. 队列数量规划

队列不是越多越好。

队列过少：

- 业务耦合。
- 监控不清楚。
- 重试策略混乱。

队列过多：

- 管理成本高。
- 指标 cardinality 高。
- 内存和文件资源开销增加。

建议：

```text
按消费语义建队列。
不要为每个用户、每个订单、每个请求创建长期队列。
```

## 7. 内存水位

RabbitMQ 有内存高水位机制。

当内存压力过高时，RabbitMQ 可能阻塞发布端，保护自身稳定。

运维上要关注：

- memory used。
- memory high watermark。
- memory alarm。
- 队列堆积导致的内存变化。

排查：

```powershell
docker exec -it rabbitmq-dev rabbitmq-diagnostics memory_breakdown
```

## 8. 磁盘水位

磁盘不足会严重影响 RabbitMQ。

要关注：

- 磁盘剩余。
- 持久化消息增长。
- DLQ 是否长期堆积。
- retry queue 是否异常堆积。
- 日志是否过大。

队列堆积会消耗磁盘，尤其是持久化消息。

## 9. 文件句柄和 Socket

连接数多时要关注：

- file descriptors。
- socket descriptors。
- 操作系统限制。

在 Management UI Overview 和节点详情中可以看到相关信息。

连接数异常增长时，要优先查应用是否正确复用连接。

## 10. 压测建议

压测不要只测 publish。

至少测：

- 只发布不消费。
- 发布和消费同时进行。
- 消费者处理慢。
- RabbitMQ 重启恢复。
- DLQ 增长。
- Outbox 堆积。
- 消费者扩容。

压测指标：

- 最大稳定 publish rate。
- 最大稳定 ack rate。
- P95/P99 消费延迟。
- 队列堆积恢复时间。
- 内存和磁盘增长曲线。

## 11. 本节小结

容量规划要回答：

- 峰值消息量是多少？
- 消费能力是多少？
- 最大允许堆积是多少？
- 磁盘能撑多久？
- 连接和 channel 是否合理？
- 队列数量是否可治理？
