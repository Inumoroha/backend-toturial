# 02. 从技术消息到业务事件

## 1. 什么是技术消息

技术消息往往只描述“怎么做”：

```text
send_email
call_inventory
run_job
sync_data
```

它偏任务指令。

例如：

```json
{
  "task": "send_email",
  "to": "user@example.com"
}
```

这类消息适合 worker 执行任务，但不适合作为业务事件总线。

## 2. 什么是业务事件

业务事件描述“发生了什么”：

```text
user.registered
order.created
order.paid
payment.succeeded
```

例如：

```json
{
  "message_id": "msg-001",
  "event_type": "order.created",
  "version": 1,
  "occurred_at": "2026-07-04T12:00:00Z",
  "correlation_id": "trace-001",
  "payload": {
    "order_id": 1001,
    "user_id": 2001,
    "amount": 9900
  }
}
```

它表达业务事实，而不是命令某个服务做事。

## 3. 为什么事件更解耦

不推荐：

```text
order-service 发布 call_inventory_service
```

这会让 order-service 知道 inventory-service 的存在。

推荐：

```text
order-service 发布 order.created
inventory-service 自己订阅 order.created
```

这样新增消费者时，不需要修改 order-service。

例如新增：

```text
risk-service
analytics-service
notification-service
```

只需要新增队列和 binding。

## 4. 事件命名规范

推荐格式：

```text
业务域.过去式动作或状态
```

示例：

```text
user.registered
order.created
order.paid
order.cancelled
payment.succeeded
inventory.reserved
inventory.reserve_failed
```

避免：

```text
do_order
call_service
send_to_inventory
msg1
event
```

## 5. 事件 payload 设计原则

payload 应该放消费者完成业务需要的信息。

但不要放太多：

- 不要放敏感数据。
- 不要放巨大字段。
- 不要把整张数据库表都塞进去。
- 不要让消费者依赖不稳定字段。

推荐：

```json
{
  "order_id": 1001,
  "user_id": 2001,
  "amount": 9900,
  "currency": "CNY"
}
```

不推荐：

```json
{
  "order": {
    "all_columns": "..."
  },
  "user_password_hash": "...",
  "large_file_base64": "..."
}
```

## 6. 事件版本

事件结构会变化，所以需要版本：

```json
{
  "event_type": "order.created",
  "version": 1
}
```

升级时尽量使用向后兼容方式：

- 新增字段通常安全。
- 删除字段要谨慎。
- 改字段含义要谨慎。
- 必要时发布新事件类型或新版本。

## 7. Routing Key 和 Event Type

通常可以让：

```text
routing key = event_type
```

例如：

```text
routing key: order.created
event_type: order.created
```

这样最直观。

少数情况下，routing key 可以更粗或更细，但初学和多数业务建议保持一致。

## 8. Topic Exchange 事件总线

推荐订单事件：

```text
Exchange: order.events
Type: topic
```

Routing keys：

```text
order.created
order.paid
order.cancelled
```

消费者队列：

```text
order.created.inventory.queue   binding: order.created
order.paid.points.queue         binding: order.paid
order.all.audit.queue           binding: order.#
```

## 9. 什么时候用 task

有些场景仍然适合 task。

例如：

```text
notification.email.send
notification.sms.send
report.generate
image.compress
```

这些消息表达：

```text
请 worker 执行某个任务。
```

它们通常进入一个 worker queue，由多个 worker 实例分担。

## 10. 本节小结

你要记住：

- task 表示“做一件事”。
- event 表示“某事已发生”。
- 业务事件比技术调用更解耦。
- routing key 通常可以等于 event_type。
- payload 要够用，但不要过度暴露。
- 事件要有版本。
