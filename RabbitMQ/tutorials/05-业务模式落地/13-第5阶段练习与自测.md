## 2. 练习一：判断 task 还是 event

请判断下面消息应该设计成 task 还是 event：

| 场景 | 类型 | 建议名称 |
| --- | --- | --- |
| 用户注册成功 | event | `user.registered` |

## 4. 练习三：设计订单事件系统

需求：

```text
订单服务发布 order.created、order.paid、order.cancelled。
库存服务关心 order.created。
积分服务关心 order.paid。
通知服务关心 order.created、order.paid、order.cancelled。
审计服务关心所有订单事件。
```

请设计 RabbitMQ 拓扑。

参考答案：

```text
Exchange: order.events
Type: topic

Queues:
order.created.inventory.queue   binding: order.created
order.paid.points.queue         binding: order.paid
order.notify.queue              binding: order.created, order.paid, order.cancelled
order.all.audit.queue           binding: order.#
```

## 5. 练习四：解释 Outbox

请回答：

1. 为什么不能在数据库事务提交后直接发布 RabbitMQ？
2. Outbox 表和业务表为什么要放在同一个事务？
3. Outbox Worker 发布成功但标记 sent 失败会怎样？
4. 为什么用了 Outbox 后消费者仍然要幂等？

参考要点：

```text
事务提交后发布可能失败，导致业务数据存在但事件丢失。
Outbox 和业务表同事务，可以保证业务成功时待发布消息也成功写入。
发布成功但标记 sent 失败会导致未来重复发布。
重复发布要求消费者必须幂等。
```

## 7. 练习六：幂等消费者

场景：

```text
积分服务消费 order.paid，给用户加积分。
```

请设计：

- 幂等键。
- 数据库唯一索引。
- 处理流程。

参考：

```text
idempotency_key = order:{order_id}:points:add
```

处理流程：

```text
开启事务
插入 processed_messages
唯一键冲突 -> 已处理 -> 提交并 ack
写 points_records
更新 user_points
提交事务
ack
```

## 10. 练习九：集成测试
