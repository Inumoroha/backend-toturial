# 01. 本阶段目标与环境检查

## 1. 本阶段为什么重要

第 0 阶段你已经知道了消息队列为什么存在。

从第 1 阶段开始，你要真正接触 RabbitMQ 服务本身：

```text
Go 服务或其他客户端
  -> 连接 RabbitMQ
  -> 发布消息
  -> 消费消息
```

在写代码之前，先用 RabbitMQ Management UI 手动操作一遍非常重要。因为后面 Go 代码里写的很多操作，本质上都是在做这些事：

- 创建 exchange。
- 创建 queue。
- 创建 binding。
- 发布消息。
- 消费消息。
- 查看消息堆积。

先在 UI 里看见它们，后面写代码才不会变成背 API。

## 2. 本阶段你要建立的核心图

RabbitMQ 最基础的消息流向是：

```text
Producer -> Exchange -> Queue -> Consumer
```

更完整一点：

```text
生产者发布消息
  -> 指定 exchange
  -> 指定 routing key
  -> RabbitMQ 根据 exchange 类型和 binding 规则路由
  -> 消息进入一个或多个 queue
  -> 消费者从 queue 中取消息处理
```

这一阶段你只要先记住一句话：

```text
生产者通常把消息发给 exchange，消费者通常从 queue 消费消息。
```

## 3. 环境要求

本阶段默认你使用：

- Windows
- PowerShell
- Docker Desktop
- 浏览器

建议版本：

- Docker Desktop：可正常运行 Linux 容器即可。
- RabbitMQ 镜像：`rabbitmq:4-management`。

如果你用的是 macOS 或 Linux，命令大部分也通用，只是终端显示可能略有不同。

## 4. 检查 Docker

在 PowerShell 执行：

```powershell
docker version
```

如果 Docker 正常，会看到 Client 和 Server 的版本信息。

继续检查 Docker Compose：

```powershell
docker compose version
```

再检查当前正在运行的容器：

```powershell
docker ps
```

如果提示 Docker daemon 未运行，先启动 Docker Desktop。

## 5. 检查端口是否被占用

RabbitMQ 本地学习通常会用两个端口：

```text
5672  -> AMQP 客户端连接端口，后面 Go 程序连接这个端口
15672 -> Management UI 管理后台端口，浏览器访问这个端口
```

在 PowerShell 检查端口：

```powershell
netstat -ano | findstr ":5672"
netstat -ano | findstr ":15672"
```

如果没有输出，说明端口大概率没有被占用。

如果有输出，说明端口可能已被其他程序占用。你可以：

- 停掉已有 RabbitMQ。
- 或者启动容器时映射到其他宿主机端口。

例如把管理后台映射到 `15673`：

```powershell
docker run -d --hostname rabbitmq-dev --name rabbitmq-dev -p 5672:5672 -p 15673:15672 -e RABBITMQ_DEFAULT_USER=go_learner -e RABBITMQ_DEFAULT_PASS=go_learner_pwd rabbitmq:4-management
```

然后浏览器访问：

```text
http://localhost:15673
```

## 6. 本阶段学习方式

建议每学一个概念，都按这个顺序：

1. 先看图。
2. 再在 Management UI 里手动操作。
3. 再用 `rabbitmqctl` 命令观察结果。
4. 最后用自己的话解释这个概念。

例如学习 queue：

```text
先理解 queue 是消息存储位置
再在 UI 创建 queue
再发布消息到 queue
再观察 ready 数量变化
再手动 get message
```

## 7. 本节小结

进入下一节前，你应该知道：

- 本阶段重点是安装 RabbitMQ 和理解基础模型。
- `5672` 是程序连接 RabbitMQ 的 AMQP 端口。
- `15672` 是浏览器访问 Management UI 的端口。
- 后续核心图是 `Producer -> Exchange -> Queue -> Consumer`。
