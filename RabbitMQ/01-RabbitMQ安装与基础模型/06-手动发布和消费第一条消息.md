# 06. 手动发布和消费第一条消息

## 1. 本节目标

本节你要完成一次完整消息流：

```text
手动发布消息
  -> learning.direct
  -> routing key hello
  -> learning.hello.queue
  -> 手动取出消息
```

这是 RabbitMQ 最基础的闭环。

## 2. 确认拓扑已经存在

请确认你已经完成上一节：

```text
Exchange: learning.direct
Queue: learning.hello.queue
Binding key: hello
```

可以用命令查看：

```powershell
docker exec -it rabbitmq-dev rabbitmqctl list_exchanges name type
docker exec -it rabbitmq-dev rabbitmqctl list_queues name messages_ready messages_unacknowledged
docker exec -it rabbitmq-dev rabbitmqctl list_bindings
```

## 3. 在 UI 中发布消息

进入：

```text
Exchanges -> learning.direct
```

找到：

```text
Publish message
```

填写：

```text
Routing key: hello
Properties: 留空
Payload:
{
  "message_id": "msg-001",
  "event_type": "learning.hello",
  "payload": {
    "text": "Hello RabbitMQ"
  }
}
Payload encoding: string
```

点击：

```text
Publish message
```

如果 routing key 和 binding 匹配，这条消息会进入 `learning.hello.queue`。

## 4. 查看队列消息数量

进入：

```text
Queues and Streams -> learning.hello.queue
```

你应该看到：

```text
Ready: 1
Unacked: 0
Total: 1
```

含义是：

- 有 1 条消息等待消费。
- 没有消息正在被消费者处理。

也可以用命令查看：

```powershell
docker exec -it rabbitmq-dev rabbitmqctl list_queues name messages_ready messages_unacknowledged consumers
```

预期类似：

```text
learning.hello.queue 1 0 0
```

## 5. 手动取出消息

在 `learning.hello.queue` 页面找到：

```text
Get messages
```

填写：

```text
Messages: 1
Ack mode: Ack message requeue false
Encoding: Auto
```

点击：

```text
Get Message(s)
```

你应该能看到刚才发布的 JSON。

此时因为选择了 `Ack message requeue false`，RabbitMQ 会认为这条消息已经被确认消费，并从队列中删除。

## 6. 再次查看队列

刷新队列页面，应该看到：

```text
Ready: 0
Unacked: 0
Total: 0
```

命令查看：

```powershell
docker exec -it rabbitmq-dev rabbitmqctl list_queues name messages_ready messages_unacknowledged consumers
```

预期：

```text
learning.hello.queue 0 0 0
```

## 7. 故意发错 routing key

现在再试一次，但 routing key 改成：

```text
wrong
```

也就是在 `learning.direct` 发布消息时填写：

```text
Routing key: wrong
```

发布后去 `learning.hello.queue` 查看，你会发现队列没有新增消息。

原因是：

```text
direct exchange 中，routing key wrong 和 binding key hello 不匹配。
```

所以消息没有路由到这个队列。

## 8. 这条消息去哪了

如果消息发布到 exchange 后没有匹配的 queue，默认情况下它可能被 RabbitMQ 丢弃。

这对初学者很重要：

```text
发布成功不等于业务队列一定收到。
```

后续可靠性阶段会学习：

- mandatory publish
- return listener
- publisher confirm
- unroutable message

第 1 阶段先记住：routing key 和 binding 规则必须匹配。

## 9. 观察 Ready 和 Unacked

这一阶段先理解两个指标：

| 指标 | 含义 |
| --- | --- |
| Ready | 还在队列中等待投递的消息 |
| Unacked | 已经投递给消费者，但还没有 ack 的消息 |

在 UI 里手动 `Get messages` 时，如果选择不同 ack mode，你会看到消息状态变化。

初学建议先使用：

```text
Ack message requeue false
```

这样取出后消息会从队列中删除，便于观察。

## 10. 本节小结

你已经完成 RabbitMQ 第一条消息的完整闭环：

```text
Publish -> Route -> Queue -> Get -> Ack
```

请确保你能解释：

- 为什么 routing key 是 `hello` 时消息进队列。
- 为什么 routing key 是 `wrong` 时消息没有进队列。
- Ready 和 Unacked 分别是什么意思。
- 手动 Get message 和真正 consumer 消费有什么相似点。
