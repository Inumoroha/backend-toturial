# 08. 消费者幂等设计

## 1. 为什么必须做幂等

RabbitMQ 常见可靠目标是：

```text
至少一次投递
```

这意味着消息可能重复。

典型场景：

```text
消费者收到 order.paid
给用户加积分成功
准备 ack
消费者进程崩溃
RabbitMQ 没收到 ack
消息重新投递
```

如果没有幂等，用户可能被加两次积分。

## 2. 幂等是什么意思

幂等指：

```text
同一个操作执行一次和执行多次，最终结果一致。
```

例如：

```text
处理 order.paid 加积分
同一个 order_id 只能加一次
重复收到消息时直接认为成功
```

## 3. 消息为什么会重复

常见原因：

- 消费者处理成功但 ack 前崩溃。
- ack 网络包丢失。
- 消费者超时断开。
- 生产者 confirm 超时后重试发布。
- retry queue 重新投递。
- 人工补偿重新发送消息。

所以你要默认：

```text
任何消息都可能重复。
```

## 4. 幂等键怎么选

常见选择：

### 4.1 message_id

每条消息全局唯一：

```text
message_id: msg-20260704-001
```

消费者记录已处理 message_id。

### 4.2 业务唯一键

通常更推荐业务唯一键：

```text
order_id + event_type
payment_id + action
user_id + coupon_id
```

例如积分服务处理支付成功：

```text
idempotency_key = order_id + ":points:order.paid"
```

原因：

```text
同一个业务事件即使被重新包装成不同 message_id，也不应该重复执行业务。
```

## 5. 数据库唯一索引方案

创建一张消费记录表：

```sql
CREATE TABLE processed_messages (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    idempotency_key VARCHAR(128) NOT NULL,
    message_id VARCHAR(128) NOT NULL,
    event_type VARCHAR(64) NOT NULL,
    processed_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uk_idempotency_key (idempotency_key)
);
```

处理流程：

```text
开始数据库事务
  -> 插入 processed_messages(idempotency_key)
  -> 如果唯一键冲突，说明处理过，直接提交并 ack
  -> 执行业务逻辑
提交事务
ack
```

关键是：

```text
幂等记录和业务变更最好在同一个本地事务里。
```

## 6. Go 伪代码

```go
func handleOrderPaid(ctx context.Context, msg Event) error {
	var payload OrderPaidPayload
	if err := json.Unmarshal(msg.Payload, &payload); err != nil {
		return ErrBadMessage
	}

	key := fmt.Sprintf("order:%d:points:order.paid", payload.OrderID)

	tx, err := db.BeginTx(ctx, nil)
	if err != nil {
		return err
	}
	defer tx.Rollback()

	inserted, err := insertProcessedMessage(ctx, tx, key, msg.MessageID, msg.EventType)
	if err != nil {
		return err
	}
	if !inserted {
		return tx.Commit()
	}

	if err := addPoints(ctx, tx, payload.UserID, payload.Points); err != nil {
		return err
	}

	return tx.Commit()
}
```

```go
if err := handleOrderPaid(ctx, event); err != nil {

	return
}

```

## 7. 状态机幂等

有些业务可以通过状态机保证幂等。

例如订单状态：

```text
created -> paid -> shipped -> finished
```

收到 `order.paid` 时：

```sql
UPDATE orders
SET status = 'paid'
WHERE id = ? AND status = 'created';
```

如果影响行数为 0，可能说明：

- 已经处理过。
- 当前状态不允许流转。

这种方案要结合业务判断。

## 8. Redis 去重方案

Redis 可以做短期去重：

```text
SET key value NX EX 86400
```

适合：

- 短时间去重。
- 非金融强一致业务。
- 通知类防重复。

不适合：

- 强一致资金操作。
- 需要永久幂等记录。
- Redis 数据丢失会造成严重问题的场景。

重要业务更推荐数据库唯一索引。

## 9. 幂等和 ack 的关系

正确流程：

```text
收到消息
  -> 幂等检查
  -> 如果已处理，ack
  -> 如果未处理，执行业务并记录幂等
  -> 事务提交成功
  -> ack
```

如果业务成功但 ack 失败，消息可能重投。

重投后幂等记录会挡住重复业务操作。

## 10. 幂等不是去掉所有重复日志

重复消息仍然可能出现。

幂等的目标不是让重复消息消失，而是：

```text
重复消息不会导致重复业务副作用。
```

你仍然应该记录日志：

```text
duplicate message ignored, idempotency_key=...
```

这样方便排查。

## 11. 本节小结

你要记住：

- RabbitMQ 可靠消费通常意味着消息可能重复。
- 消费者必须幂等。
- 幂等键优先考虑业务唯一键。
- 重要业务推荐数据库唯一索引。
- 幂等记录和业务变更最好在同一事务。
- 幂等成功后再 ack。
