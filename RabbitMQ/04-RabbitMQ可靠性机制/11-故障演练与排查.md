# 11. 故障演练与排查

## 1. 为什么要做故障演练

可靠性不是看文档就算掌握。

你需要亲自模拟：

- 消费者崩溃。
- 消息处理失败。
- 消息不可路由。
- 消息堆积。
- unacked 增长。
- RabbitMQ 重启。
- 死信队列增长。

这样线上出问题时，你才知道该看哪里。

## 2. 演练一：消费者 ack 前崩溃

### 操作

写一个 consumer：

```text
收到消息
打印日志
sleep 10 秒
ack
```

发布一条消息。

在 sleep 期间按 `Ctrl+C` 杀掉 consumer。

### 观察

查看队列：

```powershell
docker exec -it rabbitmq-dev rabbitmqctl list_queues name messages_ready messages_unacknowledged consumers
```

你会看到消息从 unacked 回到 ready。

### 结论

manual ack 可以避免消费者崩溃导致消息直接丢失。

但消息可能重复投递，所以消费者要幂等。

## 3. 演练二：忘记 ack

### 操作

消费者收到消息但不调用：

```go
d.Ack(false)
```

### 观察

```powershell
docker exec -it rabbitmq-dev rabbitmqctl list_queues name messages_ready messages_unacknowledged consumers
```

你会看到：

```text
messages_unacknowledged 持续不降
```

### 结论

unacked 高通常意味着：

- 消费者处理慢。
- 消费者卡住。
- 消费者忘记 ack。
- prefetch 太大。

## 4. 演练三：无限 requeue

### 操作

消费者遇到 `bad` 消息就：

```go
d.Nack(false, true)
```

发布：

```text
bad
```

### 观察

消费者日志会反复出现同一条消息。

### 结论

无限 requeue 会制造毒消息循环。

正确方案：

```text
有限重试 + DLQ
```

## 5. 演练四：不可路由消息

### 操作

创建 direct exchange，只绑定：

```text
email.send
```

发布：

```text
routing key: sms.send
```

如果没有 mandatory return，生产者可能感知不到。

### 观察

队列没有消息。

使用 `mandatory=true` 和 `NotifyReturn` 后，可以看到：

```text
NO_ROUTE
```

### 结论

生产端可靠性不仅要 confirm，还要关注不可路由。

## 6. 演练五：RabbitMQ 重启

### 操作

发布 persistent message 到 durable queue。

重启：

```powershell
docker restart rabbitmq-dev
```

查看：

```powershell
docker exec -it rabbitmq-dev rabbitmqctl list_queues name messages_ready messages_unacknowledged
```

### 结论

durable exchange、durable queue、persistent message 能提高重启后的保留能力。

但生产者仍然需要 publisher confirm。

## 7. 演练六：DLQ 增长

### 操作

配置最大重试 3 次。

发布一条永远失败的消息：

```text
fail
```

等待重试结束。

查看 DLQ：

```powershell
docker exec -it rabbitmq-dev rabbitmqctl list_queues name messages_ready messages_unacknowledged
```

### 结论

DLQ 中有消息说明自动处理失败，需要排查。

DLQ 必须告警。

## 8. 排查 ready 高

ready 高表示：

```text
消息还在队列里等待投递或消费。
```

常见原因：

- 没有消费者。
- 消费者数量不足。
- 消费者处理太慢。
- 生产速度过高。
- 下游服务慢。

排查：

```powershell
docker exec -it rabbitmq-dev rabbitmqctl list_queues name messages_ready messages_unacknowledged consumers
```

如果 consumers 为 0，先启动消费者。

如果 consumers 不为 0，但 ready 持续涨，说明消费能力不足。

## 9. 排查 unacked 高

unacked 高表示：

```text
消息已经投递给消费者，但还没 ack。
```

常见原因：

- 消费者卡住。
- 处理耗时长。
- prefetch 太大。
- 下游数据库慢。
- 忘记 ack。

查看 channel：

```powershell
docker exec -it rabbitmq-dev rabbitmqctl list_channels connection number user vhost prefetch_count messages_unacknowledged
```

## 10. 排查重复消费

重复消费常见原因：

- ack 前崩溃。
- ack 超时或连接断开。
- 生产者重试发布。
- retry queue 重投。

处理思路：

- 检查消费者幂等。
- 查看 message id。
- 查看 correlation id。
- 查 processed_messages 表。
- 确认业务唯一键。

不要只从 RabbitMQ 侧找答案。

## 11. 本节小结

线上排查时，先看：

```text
ready
unacked
consumers
bindings
publisher errors
consumer logs
DLQ
```

再结合：

```text
message_id
correlation_id
retry_count
业务状态
```

可靠性不是不出故障，而是故障出现时能定位、恢复和补偿。
