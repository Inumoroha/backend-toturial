# 12. 第 4 阶段练习与自测

## 1. 本阶段练习目标

完成这些练习后，你应该能设计基础可靠消息链路：

- 生产端 confirm。
- 不可路由消息 return。
- 消费端 manual ack。
- 失败 nack/reject。
- prefetch 限流。
- 持久化配置。
- 幂等消费。
- retry queue。
- DLQ。

## 2. 练习一：manual ack

### 任务

写一个 consumer：

- `autoAck=false`
- 收到消息后 sleep 5 秒
- 成功后 `Ack(false)`

发布一条消息，在 sleep 期间观察：

```powershell
docker exec -it rabbitmq-dev rabbitmqctl list_queues name messages_ready messages_unacknowledged consumers
```

### 问题

1. sleep 期间 ready 是多少？
2. sleep 期间 unacked 是多少？
3. ack 后 unacked 如何变化？

## 3. 练习二：消费者崩溃重投

### 任务

消费者收到消息后 sleep 10 秒再 ack。

发布消息后，在 sleep 期间按 `Ctrl+C` 停止消费者。

然后重新启动消费者。

### 预期

消息会重新投递。

### 思考

为什么这会导致重复消费风险？

## 4. 练习三：nack 与 requeue

### 任务

消费者遇到 body 为 `bad` 的消息时：

```go
d.Nack(false, true)
```

观察日志。

再改成：

```go
d.Nack(false, false)
```

观察区别。

### 思考

为什么不建议无限 requeue？

## 5. 练习四：prefetch

### 任务

设置：

```go
ch.Qos(1, 0, false)
```

启动两个 worker，每条消息处理 5 秒。

发布 10 条消息。

观察消息分配。

然后改成：

```go
ch.Qos(5, 0, false)
```

再次观察 `unacked`。

### 思考

prefetch 太大和太小分别有什么问题？

## 6. 练习五：持久化

### 任务

声明：

```text
durable exchange
durable queue
persistent message
```

发布消息后重启 RabbitMQ：

```powershell
docker restart rabbitmq-dev
```

查看队列消息是否还在。

### 思考

为什么 durable queue 不等于消息一定不丢？

## 7. 练习六：publisher confirm

### 任务

写一个 producer：

- `ch.Confirm(false)`
- `NotifyPublish`
- 发布 persistent message
- 等待 confirm ack

### 思考

confirm ack 表示消费者处理成功了吗？

参考答案：

```text
不是。confirm ack 只表示 broker 确认生产者发布的消息。消费者处理成功需要 consumer ack。
```

## 8. 练习七：mandatory return

### 任务

创建 direct exchange，只绑定：

```text
email.send
```

发布：

```text
routing key: sms.send
mandatory: true
```

监听：

```go
ch.NotifyReturn(...)
```

### 预期

收到 NO_ROUTE return。

### 思考

为什么 publish 返回 nil 仍然可能没有任何业务队列收到消息？

## 9. 练习八：幂等设计

### 任务

为 `order.paid` 事件设计幂等方案。

要求：

- 防止重复加积分。
- 使用数据库唯一索引。
- 写出幂等键。

参考：

```text
idempotency_key = order:{order_id}:points:order.paid
```

表：

```sql
CREATE TABLE processed_messages (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    idempotency_key VARCHAR(128) NOT NULL,
    message_id VARCHAR(128) NOT NULL,
    event_type VARCHAR(64) NOT NULL,
    processed_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uk_idempotency_key (idempotency_key)
);
```

## 10. 练习九：retry queue + DLQ

### 任务

设计拓扑：

```text
业务 exchange: stage4.exercise.exchange
业务 queue: stage4.exercise.queue
retry exchange: stage4.exercise.retry.exchange
retry queue: stage4.exercise.retry.5s.queue
DLX: stage4.exercise.dlx
DLQ: stage4.exercise.dlq
```

要求：

- 失败后进入 retry queue。
- 5 秒后回到业务 queue。
- 最多重试 3 次。
- 超过 3 次进入 DLQ。

### 思考

为什么发布到 retry queue 成功后，才能 ack 原消息？
