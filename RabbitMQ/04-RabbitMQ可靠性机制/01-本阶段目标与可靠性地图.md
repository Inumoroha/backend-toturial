# 01. 本阶段目标与可靠性地图

## 1. 为什么可靠性是 RabbitMQ 的核心

第 3 阶段你已经能用 Go 发送和消费消息。

但真实业务不会只问：

```text
消息能不能发出去？
```

它会问：

- 生产者发出的消息 RabbitMQ 真的收到了吗？
- 消息没有路由到队列怎么办？
- RabbitMQ 重启后消息还在吗？
- 消费者处理成功后消息才删除吗？
- 消费者失败了消息怎么办？
- 消息重复投递怎么办？
- 失败消息反复重试会不会拖垮系统？
- 最终失败的消息在哪里看？

这就是第 4 阶段要解决的问题。

## 2. 可靠消息链路总图

一条可靠消息链路可以拆成五段：

```text
生产者
  -> 发布确认
  -> 路由确认
  -> RabbitMQ 存储
  -> 消费者确认
  -> 失败恢复
```

更具体：

```text
Producer
  -> PublishWithContext
  -> publisher confirm
  -> mandatory return
  -> durable exchange
  -> durable queue
  -> persistent message
  -> Consumer
  -> manual ack
  -> retry queue
  -> dead letter queue
  -> idempotent handler
```

## 3. 可靠性不是绝对不重复

RabbitMQ 常见可靠消费目标是：

```text
至少一次投递
```

含义：

```text
尽量不丢，但可能重复。
```

所以可靠性的工程公式通常是：

```text
至少一次投递 + 幂等消费者 = 业务上接近只处理一次
```

不要把可靠性误解为：

```text
RabbitMQ 保证业务逻辑绝对只执行一次。
```

业务逻辑是否重复执行，需要消费者自己设计幂等。

## 4. 四类常见风险

### 4.1 消息丢失

可能发生在：

- 生产者发布失败但没有感知。
- 消息不可路由但生产者不知道。
- queue 或 message 未持久化，RabbitMQ 重启后丢失。
- 消费者 auto ack 后处理失败。

### 4.2 消息重复

可能发生在：

- 消费者处理成功，但 ack 前崩溃。
- 网络断开导致 RabbitMQ 没收到 ack。
- 生产者发布超时后重试。
- 重试队列重新投递。

### 4.3 消息堆积

可能发生在：

- 消费速度低于生产速度。
- prefetch 设置不合理。
- 消费者处理慢。
- 下游数据库或外部 API 慢。
- 毒消息无限重试。

### 4.4 消息不可恢复

可能发生在：

- 没有重试队列。
- 没有死信队列。
- 失败消息没有记录。
- 没有告警。
- 没有 message id 和 correlation id。

## 5. 本阶段统一实验环境

本阶段继续使用前面的 RabbitMQ：

```powershell
docker start rabbitmq-dev
```

连接地址：

```text
amqp://go_learner:go_learner_pwd@localhost:5672/
```

Go 项目仍使用：

```text
demos/stage3-go-client
```

你也可以另建：

```powershell
mkdir demos\stage4-reliability
cd demos\stage4-reliability
go mod init rabbitmq-go-stage4
go get github.com/rabbitmq/amqp091-go
```

本阶段文档中的代码可以放在 `demos/stage4-reliability/cmd/...` 下。

## 6. 常用观察命令

查看队列状态：

```powershell
docker exec -it rabbitmq-dev rabbitmqctl list_queues name messages_ready messages_unacknowledged consumers
```

查看 binding：

```powershell
docker exec -it rabbitmq-dev rabbitmqctl list_bindings
```

查看 exchange：

```powershell
docker exec -it rabbitmq-dev rabbitmqctl list_exchanges name type durable auto_delete
```

查看消费者：

```powershell
docker exec -it rabbitmq-dev rabbitmqctl list_consumers
```

查看 channel：

```powershell
docker exec -it rabbitmq-dev rabbitmqctl list_channels connection number user vhost prefetch_count messages_unacknowledged
```

## 7. 本节小结

本阶段要建立的核心认知：

- 可靠性是一整条链路，不是单个配置。
- RabbitMQ 能帮助实现至少一次投递，但业务要做幂等。
- 生产端要确认消息发到 broker。
- 消费端要成功处理后再 ack。
- 失败消息要有重试和死信出口。
