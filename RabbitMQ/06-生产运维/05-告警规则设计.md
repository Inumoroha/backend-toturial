# 05. 告警规则设计

## 1. 告警不是越多越好

告警的目标是：

```text
在用户受到严重影响前，让人知道并能行动。
```

坏告警：

- 太敏感，天天误报。
- 太笼统，不知道哪个队列有问题。
- 只报节点异常，不报业务队列异常。
- 没有处理手册。

好告警：

- 指向具体队列或节点。
- 有明确阈值和持续时间。
- 有处理建议。
- 能区分警告和严重。

## 2. 队列堆积告警

关注：

```text
messages_ready
```

示例规则：

```text
关键队列 ready > 1000 持续 5 分钟 -> warning
关键队列 ready > 10000 持续 5 分钟 -> critical
```

不同业务阈值不同。

例如：

```text
邮件队列堆积 5000 条可能还能接受。
支付事件队列堆积 100 条可能就很严重。
```

## 3. Unacked 告警

关注：

```text
messages_unacknowledged
```

示例：

```text
unacked > consumers * prefetch 持续异常
unacked 长时间不下降
```

常见原因：

- 消费者卡住。
- 处理耗时异常。
- 忘记 ack。
- 下游服务慢。

## 4. Consumer 消失告警

关键队列：

```text
consumers == 0 持续 1 分钟
```

应该告警。

例如：

```text
order.paid.points.queue consumers == 0
```

意味着支付成功后积分服务没人处理。

## 5. DLQ 告警

DLQ 消息数大于 0 就应该至少 warning。

示例：

```text
dlq messages_ready > 0 持续 1 分钟 -> warning
dlq messages_ready > 100 -> critical
```

DLQ 是需要处理的失败消息，不是可以长期堆放的仓库。

## 6. 发布和消费速率不平衡

如果：

```text
publish rate > ack rate
```

持续较长时间，队列会堆积。

可以设计趋势告警：

```text
ready 持续增长 10 分钟
```

比单纯 ready 阈值更灵敏。

## 7. 内存告警

关注：

- memory used。
- memory high watermark。
- memory alarm。

建议：

```text
内存使用超过 70% -> warning
内存 alarm 触发 -> critical
```

具体阈值要结合部署环境。

## 8. 磁盘告警

关注：

- disk free。
- disk free limit。
- disk alarm。

建议：

```text
磁盘剩余低于 20% -> warning
磁盘 alarm 触发 -> critical
```

磁盘告警很严重，因为 RabbitMQ 可能阻塞发布。

## 9. 连接数告警

关注：

- connection count。
- channel count。
- connection churn。

示例：

```text
connection count 超过历史基线 2 倍持续 5 分钟
channel count 持续增长不下降
```

常见原因：

- 应用连接泄漏。
- 重连风暴。
- 每条消息创建连接。

## 10. Outbox 告警

业务侧必须告警：

```text
outbox pending 数量过高
outbox 最老 pending 时间过长
outbox dead > 0
```

示例：

```text
outbox_oldest_pending_age > 5m -> warning
outbox_oldest_pending_age > 30m -> critical
```

## 11. 告警必须配处理手册

每条告警都应该能回答：

- 这代表什么？
- 影响哪些业务？
- 先看哪个 dashboard？
- 常见原因是什么？
- 如何止血？
- 如何恢复？
- 何时升级给谁？

没有处理手册的告警，很容易变成噪声。

## 12. 本节小结

RabbitMQ 关键告警：

- ready 堆积。
- unacked 高。
- consumers 为 0。
- DLQ 有消息。
- 内存 alarm。
- 磁盘 alarm。
- 连接数异常。
- Outbox 堆积。
