# 12. 线上故障排查手册

## 1. 故障排查总顺序

遇到 RabbitMQ 线上问题，先不要乱改配置。

按顺序看：

1. 影响哪个业务？
2. 哪个 exchange / queue / routing key？
3. ready 是否增长？
4. unacked 是否增长？
5. consumers 是否为 0？
6. publish rate 和 ack rate 是否平衡？
7. DLQ 是否增长？
8. RabbitMQ 节点是否有内存或磁盘告警？
9. 应用消费者日志是否报错？
10. 下游数据库或外部服务是否慢？

## 2. 场景一：ready 持续增长

现象：

```text
messages_ready 持续增长
```

可能原因：

- 没有消费者。
- 消费者数量不足。
- 消费者处理太慢。
- 下游数据库或 API 慢。
- 消费者一直失败并进入重试。

排查命令：

```powershell
docker exec -it rabbitmq-dev rabbitmqctl list_queues name messages_ready messages_unacknowledged consumers
```

处理：

- 确认消费者是否在线。
- 查看消费者日志。
- 临时扩容消费者。
- 检查下游服务。
- 暂停非核心生产者。

## 3. 场景二：unacked 很高

现象：

```text
messages_unacknowledged 很高，ready 不一定高。
```

可能原因：

- 消费者拿到消息但处理慢。
- 消费者卡死。
- 忘记 ack。
- prefetch 太大。
- 下游服务阻塞。

排查：

```powershell
docker exec -it rabbitmq-dev rabbitmqctl list_channels connection number prefetch_count messages_unacknowledged
docker exec -it rabbitmq-dev rabbitmqctl list_consumers
```

处理：

- 检查消费者线程和 goroutine。
- 检查业务处理耗时。
- 调低 prefetch。
- 重启异常消费者。
- 优化下游服务。

## 4. 场景三：consumer 为 0

现象：

```text
关键队列 consumers = 0
```

可能原因：

- 消费者进程挂了。
- 部署失败。
- 连接 RabbitMQ 失败。
- 权限错误。
- queue 名称配置错误。

处理：

- 启动消费者。
- 查看应用日志。
- 检查连接地址和 vhost。
- 检查权限。
- 检查 queue 名称。

## 5. 场景四：消息进入 DLQ

现象：

```text
DLQ messages_ready > 0
```

可能原因：

- 消息格式错误。
- 业务处理失败。
- 重试超过最大次数。
- 消费者主动 nack requeue=false。

处理：

- 抽样查看 DLQ 消息。
- 根据 message_id 查业务日志。
- 查看 retry_count 和 last_error。
- 修复代码或数据。
- 设计补偿或重放流程。

不要直接清空 DLQ。

## 6. 场景五：连接数暴涨

现象：

```text
connection count 快速增加
```

可能原因：

- 应用每条消息创建连接。
- 应用重连风暴。
- 负载均衡健康检查错误。
- 连接泄漏。

排查：

```powershell
docker exec -it rabbitmq-dev rabbitmqctl list_connections name user peer_host state channels
```

处理：

- 找到来源 IP 和用户。
- 检查对应应用版本。
- 限制连接来源。
- 回滚异常应用。
- 修复连接复用。

## 7. 场景六：磁盘告警

现象：

```text
disk alarm
```

影响：

```text
RabbitMQ 可能阻塞发布，保护节点。
```

可能原因：

- 队列堆积。
- DLQ 长期不处理。
- 持久化消息增长。
- 日志过大。
- 磁盘容量不足。

处理：

- 查大队列。
- 处理 DLQ。
- 扩容磁盘。
- 暂停非核心生产者。
- 扩容消费者消化堆积。

## 8. 场景七：内存告警

现象：

```text
memory alarm
```

处理：

```powershell
docker exec -it rabbitmq-dev rabbitmq-diagnostics memory_breakdown
```

可能原因：

- 队列堆积。
- 连接和 channel 太多。
- 消费者不 ack。
- 大消息。

处理：

- 降低生产速率。
- 扩容消费者。
- 检查大消息。
- 检查连接泄漏。
- 必要时扩容节点资源。

## 9. 场景八：发布成功但业务没收到

可能原因：

- routing key 错。
- binding 缺失。
- exchange 错。
- mandatory return 没处理。
- 消费者处理失败进入 DLQ。
- 消费者处理了但业务幂等判断为重复。

排查：

```powershell
docker exec -it rabbitmq-dev rabbitmqctl list_bindings
docker exec -it rabbitmq-dev rabbitmqctl list_queues name messages_ready messages_unacknowledged consumers
```

同时查：

- producer 日志。
- mandatory return 日志。
- publisher confirm 日志。
- consumer 日志。
- DLQ。

## 10. 止血优先级

故障中先止血：

1. 保护核心业务。
2. 防止堆积继续扩大。
3. 恢复消费者。
4. 降低非核心生产流量。
5. 处理 DLQ 和补偿。
6. 复盘根因。

不要一上来就清队列。

清队列前必须确认业务允许丢弃。

## 11. 本节小结

RabbitMQ 故障排查的关键是：

```text
先定位队列，再定位消费者，再定位资源，再定位业务。
```

常用四个数：

```text
ready
unacked
consumers
DLQ
```
