# 10. 配置备份与拓扑治理

## 1. 为什么要备份 definitions

RabbitMQ 中有很多拓扑和权限配置：

- vhost。
- user。
- permission。
- exchange。
- queue。
- binding。
- policy。

这些定义统称为 definitions。

如果节点损坏或误删资源，definitions 可以帮助你恢复拓扑。

## 2. definitions 不等于消息备份

重要区别：

```text
definitions 备份的是配置和拓扑。
不是队列里的消息内容。
```

它可以恢复：

- exchange。
- queue。
- binding。
- policy。
- user。
- permission。

但不能恢复：

```text
已经在队列中的业务消息。
```

业务消息可靠性要靠持久化、镜像/复制队列、Outbox、消费者幂等等机制。

## 3. 导出 definitions

可以通过 Management UI：

```text
Overview -> Export definitions
```

也可以用 HTTP API：

```powershell
curl.exe -u go_learner:go_learner_pwd http://localhost:15672/api/definitions -o rabbitmq-definitions.json
```

导出后建议放入：

```text
ops/rabbitmq/definitions/rabbitmq-definitions.json
```

生产中要注意该文件可能包含敏感信息，不能随意公开。

## 4. 导入 definitions

通过 UI：

```text
Overview -> Import definitions
```

或 HTTP API：

```powershell
curl.exe -u go_learner:go_learner_pwd -H "content-type: application/json" -X POST --data-binary "@rabbitmq-definitions.json" http://localhost:15672/api/definitions
```

导入前要确认：

- 目标环境是否正确。
- 是否会覆盖已有配置。
- 用户和权限是否适合目标环境。

## 5. 拓扑即代码

不要只依赖人工点击 UI 创建拓扑。

推荐：

- definitions 文件版本管理。
- Terraform / Helm / 脚本管理。
- 应用启动时声明必要拓扑。
- CI/CD 中检查拓扑变更。

至少要做到：

```text
生产拓扑可以从代码或配置重建。
```

## 6. 命名治理

随着项目增长，命名会失控。

建议统一：

```text
exchange: 业务域.用途
queue:    事件或任务.消费者.queue
routing:  业务域.事件
```

示例：

```text
order.events
order.created.inventory.queue
order.paid.points.queue
notification.task.direct
notification.email.worker.queue
```

## 7. 清理无用资源

定期检查：

- 没有 binding 的 exchange。
- 没有 consumer 的废弃 queue。
- 长期为空且无人使用的 queue。
- 不再使用的 user。
- 过时 policy。

命令：

```powershell
docker exec -it rabbitmq-dev rabbitmqctl list_queues name consumers messages_ready messages_unacknowledged
docker exec -it rabbitmq-dev rabbitmqctl list_exchanges name type durable auto_delete
docker exec -it rabbitmq-dev rabbitmqctl list_bindings
```

清理前必须确认业务不再使用。

## 8. Policy 治理

Policy 可以统一设置队列参数。

例如：

- TTL。
- DLX。
- queue type。
- max length。

生产中建议用 policy 管理一类队列，而不是每个队列手动设置。

但要小心：

```text
policy 变更可能影响一批队列。
```

变更前要评审和演练。

## 9. 环境隔离

开发、测试、生产不要共用同一个 vhost。

推荐：

```text
/dev
/test
/prod
```

或者不同集群。

不要让测试程序误连生产 RabbitMQ。

连接地址要通过环境配置注入。

## 10. 本节小结

配置备份和治理要点：

- definitions 可以备份拓扑和权限。
- definitions 不是消息备份。
- 拓扑应该可版本化、可审查、可恢复。
- 命名和 policy 要治理。
- 定期清理无用资源。
