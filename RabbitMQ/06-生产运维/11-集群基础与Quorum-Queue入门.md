# 11. 集群基础与 Quorum Queue 入门

## 1. 为什么需要集群

单节点 RabbitMQ 有明显风险：

```text
节点宕机 -> 服务不可用
磁盘损坏 -> 数据风险
维护升级 -> 需要停机
```

生产环境常通过集群和复制队列提升可用性。

但集群不是银弹。

它会带来：

- 运维复杂度。
- 网络要求。
- 数据复制成本。
- 故障恢复复杂度。

## 2. RabbitMQ 集群基础概念

RabbitMQ 集群由多个节点组成。

节点之间共享元数据，例如：

- vhost。
- exchange。
- queue 定义。
- binding。
- user。
- permission。

但消息数据是否复制，取决于队列类型和配置。

重点：

```text
集群不等于所有队列消息自动安全复制。
```

## 3. Classic Queue

Classic Queue 是传统队列。

适合：

- 学习。
- 普通任务队列。
- 对复制要求不高的场景。

如果只使用单副本 classic queue，队列所在节点故障会影响该队列。

## 4. Quorum Queue

Quorum Queue 是面向高可用和数据安全的队列类型。

它基于 Raft 复制日志思想，适合：

- 关键业务消息。
- 需要队列复制。
- 更重视数据安全而不是极限低延迟的场景。

创建队列时指定：

```go
amqp.Table{
	"x-queue-type": "quorum",
}
```

或通过 policy 设置。

## 5. Quorum Queue 的取舍

优点：

- 数据复制。
- leader/follower 模式。
- 更适合关键业务队列。

代价：

- 写入成本更高。
- 资源消耗更高。
- 对网络延迟更敏感。
- 不适合大量临时队列。

选择时要问：

```text
这条队列里的消息是否关键到需要复制？
```

## 6. Stream 简要了解

RabbitMQ Stream 更偏事件流和日志式场景。

适合：

- 高吞吐事件流。
- 可重复读取。
- 较长时间保留。

普通任务队列和业务命令不一定需要 Stream。

第 7 阶段做架构取舍时再深入对比。

## 7. 集群运维基本原则

生产集群要关注：

- 节点数量。
- 网络稳定性。
- 磁盘性能。
- Erlang cookie 一致。
- 节点名称稳定。
- 数据目录持久化。
- 分区处理策略。
- 升级顺序。

不要在不了解集群语义时盲目搭生产集群。

## 8. 客户端连接集群

客户端不要只写死一个节点。

可以：

- 使用负载均衡地址。
- 配置多个 RabbitMQ 节点地址。
- 使用服务发现。

Go 客户端要考虑：

- 连接断开重连。
- channel 重建。
- consumer 重新注册。
- topology 重新声明。

这些属于生产级客户端封装内容。

## 9. 本地学习建议

第 6 阶段不强制你本地搭 RabbitMQ 集群。

建议先掌握：

- 集群共享元数据。
- 队列消息复制取决于队列类型。
- quorum queue 适合关键业务。
- 集群增加运维复杂度。

后续可以单独做集群实验。

## 10. 本节小结

你要记住：

- RabbitMQ 集群提升可用性，但不自动解决所有可靠性问题。
- 元数据共享不等于消息都复制。
- Quorum Queue 更适合关键业务消息。
- Quorum Queue 有资源和性能成本。
- 客户端要能处理节点故障和重连。
