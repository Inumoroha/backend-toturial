# 09. 架构选型决策表

## 1. 为什么需要决策表

架构选型不能只靠感觉。

你需要把需求拆成多个维度，然后对照技术特性做判断。

本节给你一个可复用模板。

## 2. 快速选择表

| 场景 | 推荐方案 |
| --- | --- |
| 后台任务、邮件短信、图片处理 | RabbitMQ / 数据库任务表 |
| 订单、支付、通知等业务事件 | RabbitMQ |
| 海量日志、埋点、数据管道 | Kafka |
| 已有 Redis，轻量消息流 | Redis Streams |
| 强事务、低频任务、简单 worker | 数据库任务表 |
| 需要复杂 routing key | RabbitMQ |
| 需要长期保留和回放 | Kafka / RabbitMQ Streams |
| 关键业务队列需要复制 | RabbitMQ Quorum Queue |
| 查询请求、立即返回结果 | HTTP/gRPC |

## 3. 详细决策维度

| 维度 | RabbitMQ | Kafka | Redis Streams | 数据库任务表 |
| --- | --- | --- | --- | --- |
| 任务队列 | 强 | 一般 | 可用 | 可用 |
| 复杂路由 | 强 | 弱 | 弱 | 弱 |
| 发布订阅 | 强 | 强 | 中 | 弱 |
| 高吞吐日志 | 中 | 强 | 中 | 弱 |
| 历史回放 | 普通 queue 弱，Stream 可用 | 强 | 可用 | 可查但不适合大规模 |
| 事务一致性 | 需 Outbox | 需 Outbox | 需 Outbox | 强 |
| 运维复杂度 | 中 | 高 | 中 | 低 |
| 低延迟业务消息 | 强 | 可用 | 强 | 一般 |
| DLQ/retry 治理 | 强 | 需设计 | 需设计 | 自己实现 |

## 4. 决策问题清单

选型前回答这些问题：

1. 消息是 task 还是 event？
2. 消费者有几个？
3. 是否需要复杂路由？
4. 是否需要历史回放？
5. 吞吐峰值是多少？
6. 可接受延迟是多少？
7. 消息大小是多少？
8. 是否强依赖数据库事务？
9. 是否允许重复投递？
10. 消费者能否幂等？
11. 是否需要 DLQ 和人工补偿？
12. 团队是否具备该中间件运维能力？

## 5. RabbitMQ 选型模板

可以这样写：

```text
我们选择 RabbitMQ，因为该场景是业务消息分发，主要需求是低延迟异步处理、任务队列和复杂路由。消息不需要长期回放，但需要 manual ack、重试、DLQ 和消费者幂等。相比 Kafka，RabbitMQ 的 exchange/routing key 更适合该业务拓扑；相比数据库任务表，RabbitMQ 更适合多消费者和更高实时性。
```

## 6. Kafka 选型模板

```text
我们选择 Kafka，因为该场景是高吞吐事件流，需要事件保留、多个消费组独立读取和历史回放。RabbitMQ 更适合业务任务和复杂路由，但普通 queue 不适合长期日志留存。该场景更像数据管道，因此 Kafka 更合适。
```

## 7. Redis Streams 选型模板

```text
我们选择 Redis Streams，因为系统已有 Redis 高可用基础设施，消息量不大，路由简单，只需要轻量 stream 和 consumer group。相比引入 RabbitMQ，Redis Streams 运维成本更低。但我们会补充 pending 消息处理、失败重试和监控。
```

## 8. 数据库任务表选型模板

```text
我们选择数据库任务表，因为任务量较小，强依赖业务数据库事务，延迟要求不高，只有少量 worker 消费。相比引入 RabbitMQ，数据库任务表更简单，事务一致性更直接。后续如果任务量增长或需要多服务订阅，可以演进到 Outbox + RabbitMQ。
```
