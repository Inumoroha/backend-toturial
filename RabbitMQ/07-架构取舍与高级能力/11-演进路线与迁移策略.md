# 11. 演进路线与迁移策略

## 1. 为什么要考虑演进

架构不是一次性做完的。

一个系统可能经历：

```text
单体应用
数据库任务表
RabbitMQ
Outbox
Kafka
多集群治理
```

好的架构应该允许演进，而不是一开始就过度设计。

## 2. 从同步调用到异步消息

初始状态：

```text
注册接口同步发送邮件
```

问题：

- 注册响应慢。
- 邮件服务失败影响注册。

演进：

```text
注册接口写用户
发布 user.registered
邮件服务异步消费
```

注意：

- API 返回语义要改变。
- 邮件发送变成最终完成。
- 需要重试和 DLQ。

## 3. 从数据库任务表到 RabbitMQ

初始状态：

```text
jobs 表 + worker 轮询
```

适合任务量小。

当出现：

- 轮询延迟不可接受。
- 多业务订阅复杂。
- jobs 表压力大。
- 需要更灵活路由。

可以迁移到：

```text
Outbox + RabbitMQ
```

迁移步骤：

1. 保留 jobs 表。
2. 新增 outbox 表。
3. 新增 RabbitMQ worker。
4. 双写或灰度发布。
5. 对比处理结果。
6. 切流。
7. 下线旧 jobs 消费。

## 4. 从 RabbitMQ 到 Kafka 分工

当业务增长后，你可能发现：

```text
RabbitMQ 负责业务消息很好。
但数据团队需要长期回放和实时数仓。
```

此时不一定要替换 RabbitMQ。

可以分工：

```text
RabbitMQ -> 业务异步处理
Kafka -> 数据流、日志、分析
```

同一业务事件可以：

- 通过 RabbitMQ 通知业务服务。
- 通过 Kafka 进入数据平台。

## 5. 从 Classic Queue 到 Quorum Queue

如果某些队列变成关键业务：

```text
order.paid.points.queue
payment.succeeded.accounting.queue
```

可以评估 Quorum Queue。

迁移注意：

- 队列类型通常不能随意原地修改。
- 需要新建队列。
- 重新绑定。
- 灰度消费者。
- 处理旧队列剩余消息。
- 验证监控和告警。

不要直接在线乱改队列类型。

## 6. 从单 RabbitMQ 到集群

演进触发：

- 单节点不可用风险不可接受。
- 维护需要不停机。
- 关键队列需要复制。

迁移前要准备：

- 集群部署方案。
- 节点命名。
- 网络策略。
- 数据目录。
- quorum queue 策略。
- 客户端连接多个节点或负载均衡。
- 故障演练。

## 7. 灰度迁移原则

迁移消息系统要保守。

原则：

- 先影子消费，不影响业务。
- 再小流量切换。
- 保留回滚路径。
- 监控两边差异。
- 避免重复业务副作用。
- 保证消费者幂等。

## 8. 双写风险

迁移时常见双写：

```text
同时写 RabbitMQ 和 Kafka
```

风险：

- 一边成功一边失败。
- 顺序不一致。
- 重复事件。
- 排查复杂。

关键业务要考虑 Outbox，让多个下游发布由可靠 worker 处理。

## 9. 什么时候不要迁移

如果当前方案：

- 稳定。
- 任务量不大。
- 可观察。
- 失败可恢复。
- 团队熟悉。

就不要为了技术潮流迁移。

迁移必须有业务收益。

## 10. 本节小结

架构演进建议：

```text
同步调用 -> 异步任务
数据库任务表 -> Outbox + RabbitMQ
RabbitMQ -> RabbitMQ + Kafka 分工
Classic -> Quorum
单节点 -> 集群
```

每一步都要有：

- 灰度。
- 回滚。
- 监控。
- 幂等。
- 补偿。
