# 10. 典型业务场景选型案例

## 1. 用户注册后发欢迎邮件

需求：

```text
用户注册成功后异步发送欢迎邮件。
邮件失败可重试。
注册接口不等待邮件发送完成。
```

推荐：

```text
RabbitMQ 或数据库任务表
```

如果业务量小：

```text
数据库任务表足够。
```

如果已有 RabbitMQ，或者通知任务较多：

```text
RabbitMQ 更合适。
```

## 2. 支付成功后通知多个系统

需求：

```text
支付成功后，订单、积分、通知、风控、审计都要处理。
```

推荐：

```text
RabbitMQ topic exchange
```

原因：

- 这是业务事件。
- 多个消费者订阅。
- 需要 routing key。
- 需要 retry/DLQ。
- 低延迟业务处理。

## 3. 用户行为埋点进入数仓

需求：

```text
每秒大量用户行为事件。
需要保留多天。
多个数据处理任务回放。
```

推荐：

```text
Kafka
```

原因：

- 高吞吐。
- 事件日志。
- 多消费组。
- 历史回放。
- 数据管道生态成熟。

RabbitMQ 普通 queue 不适合该主场景。

## 4. 订单 15 分钟未支付自动取消

需求：

```text
订单创建后 15 分钟检查支付状态。
未支付则取消。
```

推荐：

```text
RabbitMQ TTL + DLX
或数据库任务表
```

选择 RabbitMQ：

- 已有 RabbitMQ。
- 延迟任务和业务消息统一。
- 任务量适中。

选择数据库任务表：

- 需要更清晰地查询和修正任务。
- 任务量不大。
- 强依赖订单数据库状态。

## 5. 图片压缩任务

需求：

```text
用户上传图片后异步压缩。
多个 worker 处理。
```

推荐：

```text
RabbitMQ Work Queue
```

消息里不要放图片本体。

应该放：

```text
file_id
object_key
```

## 6. 实时聊天消息

需求：

```text
用户之间实时聊天。
消息要低延迟。
在线推送。
离线消息存储。
```

RabbitMQ 可以作为内部消息分发的一部分，但不能替代完整聊天系统。

你还需要：

- WebSocket 网关。
- 消息存储。
- 在线状态。
- 离线同步。
- 多端一致性。

不要以为引入 RabbitMQ 就完成聊天系统。

## 7. 资金流水同步

需求：

```text
支付系统产生资金流水事件。
账务、风控、审计都要处理。
不能丢。
```

推荐：

```text
Outbox + RabbitMQ Quorum Queue + 幂等消费者
```

也可能同步进入 Kafka 做审计和数据分析。

关键：

- Outbox。
- Quorum Queue。
- publisher confirm。
- 消费者幂等。
- 审计表。
- DLQ 告警。

## 8. 秒杀削峰

需求：

```text
高峰期大量请求进入，下游数据库承受不了。
```

RabbitMQ 可以用于削峰，但不是完整秒杀方案。

还需要：

- 前置限流。
- 库存预扣。
- 防刷。
- 用户资格校验。
- 缓存。
- 幂等。
- 超卖保护。

如果只是把所有请求塞进 RabbitMQ，下游长期消费不过来，仍然会堆积。

## 9. 每天凌晨生成报表

需求：

```text
每天生成一些报表。
任务量不大。
```

推荐：

```text
数据库任务表或定时任务系统
```

如果报表任务很多，需要多 worker 分发，可以用 RabbitMQ。

## 10. 本节小结

不同场景推荐：

```text
业务任务 -> RabbitMQ / DB Job
多服务业务事件 -> RabbitMQ
海量日志流 -> Kafka
轻量 Redis 生态消息 -> Redis Streams
强事务低频任务 -> DB Job
关键业务消息 -> Outbox + RabbitMQ Quorum
```
