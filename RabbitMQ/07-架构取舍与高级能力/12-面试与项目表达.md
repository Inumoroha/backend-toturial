# 12. 面试与项目表达

## 1. 为什么要练表达

学完 RabbitMQ 后，你不仅要会做，还要会讲。

面试官通常不只问：

```text
RabbitMQ 怎么发消息？
```

更会问：

- 为什么用 RabbitMQ？
- 为什么不用 Kafka？
- 消息丢了怎么办？
- 消息重复怎么办？
- 消息堆积怎么办？
- 数据库事务成功但消息发送失败怎么办？
- 如何监控和运维？

## 2. 项目表达结构

推荐按这个顺序讲：

```text
业务背景
问题
方案
可靠性设计
幂等设计
监控运维
取舍
结果
```

不要只说：

```text
我用了 RabbitMQ。
```

要说清楚为什么和怎么用。

## 3. 示例：通知系统

可以这样讲：

```text
我们注册和订单流程中有邮件、短信、站内信等通知。如果同步发送，会增加接口延迟，而且第三方服务失败会影响主流程。所以我把通知设计成异步任务，使用 RabbitMQ direct exchange 按 notification.email.send、notification.sms.send 路由到不同 worker queue。消费者使用 manual ack、prefetch、retry queue 和 DLQ。通过 notification_id 做幂等，避免重复发送。DLQ 和队列堆积接入监控告警。
```

## 4. 示例：订单事件系统

```text
订单创建、支付、取消会影响库存、积分、通知、审计等多个系统。我没有让订单服务直接调用这些服务，而是发布 order.created、order.paid、order.cancelled 业务事件到 topic exchange。各服务通过自己的队列订阅需要的 routing key。这样新增消费者不需要改订单服务。关键事件使用 Outbox 保证数据库事务和待发布消息一致，消费者通过业务唯一键做幂等。
```

## 5. 示例：为什么不用 Kafka

```text
这个场景主要是业务消息和任务分发，需要低延迟、复杂 routing key、重试和死信队列，不需要长期保留和历史回放。因此 RabbitMQ 比 Kafka 更贴合。如果是用户行为日志、实时数仓或需要多个消费组回放历史事件，我会选择 Kafka。
```

## 6. 示例：为什么不用数据库任务表

```text
数据库任务表适合低频、强事务、简单 worker 的场景。但这个业务有多个服务订阅事件，路由关系复杂，并且需要独立扩展消费者和监控队列堆积，所以使用 RabbitMQ。对于创建订单和发布事件的一致性，我用 Outbox 表解决，而不是直接在事务后发布消息。
```

## 7. 消息丢失怎么回答

可以这样答：

```text
消息丢失要分链路看。生产端使用 publisher confirm 确认 RabbitMQ 接收消息，mandatory publish 处理不可路由消息。RabbitMQ 侧使用 durable exchange、durable queue 和 persistent message。消费端使用 manual ack，业务成功后再 ack。数据库事务和消息发布之间使用 Transactional Outbox。失败消息进入 retry 和 DLQ。
```

## 8. 消息重复怎么回答

```text
可靠消息通常是至少一次投递，所以重复是正常风险。消费者必须幂等。我会优先使用业务唯一键，比如 order_id + event_type + action，配合数据库唯一索引。重复消息识别后直接 ack，不重复执行业务副作用。
```

## 9. 消息堆积怎么回答

```text
先看 ready、unacked、consumers、publish rate、ack rate。如果 ready 高，可能是消费者不足或处理慢；如果 unacked 高，可能是消费者卡住、prefetch 太大或忘记 ack；如果 consumers 为 0，说明消费者掉线。处理上可以扩容消费者、排查下游慢服务、暂停非核心生产者、处理毒消息和 DLQ。
```

## 10. 选型总结句

你可以准备这句话：

```text
RabbitMQ 更适合业务消息、任务队列、复杂路由和低延迟异步处理；Kafka 更适合高吞吐事件流、日志保留和回放；Redis Streams 适合已有 Redis 生态下的轻量消息流；数据库任务表适合低频、强事务、简单任务。
```
