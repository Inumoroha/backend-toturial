# 01. 为什么 Go 后端需要学习 RabbitMQ

## 1. 先从一个后端场景开始

假设你正在写一个用户注册接口：

```text
POST /api/register
```

用户注册成功后，系统可能要做这些事情：

- 写入用户表。
- 发送欢迎邮件。
- 发送短信验证码或通知。
- 初始化用户积分账户。
- 通知风控系统。
- 写入运营分析数据。
- 推送站内信。

如果所有事情都放在一个 HTTP 请求里同步完成，流程可能变成：

```text
用户请求
  -> 注册服务写数据库
  -> 邮件服务发送邮件
  -> 短信服务发送短信
  -> 积分服务初始化
  -> 风控服务检查
  -> 分析系统写数据
  -> 返回响应
```

这会带来几个问题：

- 响应慢：用户要等待所有步骤完成。
- 耦合高：注册服务必须知道很多下游服务。
- 容错差：邮件服务慢了或挂了，注册接口可能也失败。
- 扩展难：新增一个“新人优惠券服务”，注册服务又要改代码。
- 流量峰值难扛：活动期间大量注册会直接打到所有下游系统。

RabbitMQ 这类消息队列的核心价值，就是把这种“所有事情都同步做完”的流程，改成“核心流程同步完成，非核心动作异步处理”。

## 2. 引入 RabbitMQ 后会发生什么

使用 RabbitMQ 后，注册流程可以变成：

```text
用户请求
  -> 注册服务写数据库
  -> 注册服务发布 user.registered 消息
  -> 返回注册成功

RabbitMQ
  -> 邮件服务消费 user.registered
  -> 短信服务消费 user.registered
  -> 积分服务消费 user.registered
  -> 风控服务消费 user.registered
  -> 分析服务消费 user.registered
```

也就是：

- 注册服务只负责完成注册和发布事件。
- 下游服务自己订阅并处理感兴趣的消息。
- 邮件服务挂了，不应该影响用户注册成功。
- 新增服务时，可以新增消费者，而不是频繁修改注册服务。

这就是异步解耦。

## 3. Go 后端为什么尤其适合学习 RabbitMQ

Go 常用于编写：

- Web API 服务
- 微服务
- 后台 worker
- 定时任务
- 数据处理服务
- 高并发网络服务

而 RabbitMQ 经常用于连接这些服务：

```text
api-service -> RabbitMQ -> worker-service
order-service -> RabbitMQ -> inventory-service
payment-service -> RabbitMQ -> notification-service
```

Go 的 goroutine、context、并发模型和 RabbitMQ 的消费者模型很契合。你可以用 Go 很自然地写出：

- 多 worker 并发消费。
- 消费超时控制。
- 优雅退出。
- 失败重试。
- 后台任务处理。
- 微服务事件通信。

## 4. RabbitMQ 在后端中的典型角色

RabbitMQ 通常扮演下面几种角色。

### 4.1 异步任务队列

适合处理不需要立刻完成的任务。

例如：

- 发送邮件
- 生成报表
- 图片压缩
- 视频转码
- 数据同步

同步模式：

```text
HTTP 请求 -> 执行耗时任务 -> 返回
```

异步模式：

```text
HTTP 请求 -> 写入任务消息 -> 返回
worker -> 慢慢消费任务
```

### 4.2 服务解耦

服务 A 不直接调用服务 B，而是发送事件。

例如订单支付成功：

```text
payment-service -> 发布 order.paid
order-service -> 消费 order.paid
inventory-service -> 消费 order.paid
coupon-service -> 消费 order.paid
notification-service -> 消费 order.paid
```

支付服务不需要知道后面有多少系统关心支付成功。

### 4.3 削峰填谷

当短时间流量很高时，RabbitMQ 可以暂存消息，让消费者按自己的处理能力慢慢消费。

例如秒杀场景：

```text
大量请求 -> 快速写入队列 -> worker 按固定并发处理
```

注意：RabbitMQ 不是魔法。它可以缓冲压力，但不能让一个完全处理不过来的系统无限承载流量。如果消费者长期处理速度小于生产速度，消息还是会堆积。

### 4.4 广播事件

一条消息可以被多个业务系统接收。

例如用户注册成功：

```text
user.registered
  -> email-service
  -> points-service
  -> risk-service
  -> analytics-service
```

这在事件驱动架构中非常常见。

## 5. 什么时候不该用 RabbitMQ

RabbitMQ 很有用，但不是所有问题都应该用它。

下面这些场景不一定适合 RabbitMQ：

- 业务必须立即拿到下游返回结果。
- 调用链很简单，同步 HTTP/RPC 已经足够。
- 团队还没有能力处理重试、幂等、死信、监控。
- 数据量巨大，并且主要诉求是日志留存和反复回放。
- 只是为了“看起来像微服务”而引入中间件。

一个好的后端工程师不是“什么都上 MQ”，而是知道什么时候用，什么时候不用。

## 6. 本节小结

RabbitMQ 对 Go 后端工程师的价值主要有三点：

- 让耗时任务异步执行。
- 让服务之间降低耦合。
- 让高峰流量可以被队列缓冲。

但它也会引入新的问题：

- 消息可能重复。
- 消息可能失败。
- 消息可能堆积。
- 数据会变成最终一致。
- 系统排查链路会变长。

下一节你会继续学习：消息队列到底解决哪些具体问题。
