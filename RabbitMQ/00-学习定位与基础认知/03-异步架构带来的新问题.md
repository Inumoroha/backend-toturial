# 03. 异步架构带来的新问题

## 1. 先接受一个事实

消息队列能解决同步系统中的一些问题，但它也会制造新的复杂度。

引入 RabbitMQ 后，你的系统从：

```text
请求 -> 处理 -> 返回
```

变成：

```text
请求 -> 发布消息 -> 返回
             |
             v
          异步消费 -> 成功或失败
```

这意味着：用户请求成功时，后面的异步任务可能还没完成。

所以你必须理解异步系统的几个新问题。

## 2. 问题一：最终一致性

同步系统中，调用方通常会等待所有结果。

例如：

```text
创建订单 -> 扣库存 -> 返回成功
```

如果改成异步：

```text
创建订单 -> 发布 order.created -> 返回成功
库存服务稍后消费 order.created -> 扣库存
```

那么在短时间内可能出现：

```text
订单已经创建成功，但库存还没有扣减
```

这就是最终一致性。

最终一致性的意思是：系统不会在每一个瞬间都完全一致，但经过一段时间和重试补偿后，最终会达到一致状态。

## 3. 问题二：调用方拿不到即时结果

同步调用可以立刻知道下游是否成功：

```text
扣库存成功 -> 返回订单创建成功
扣库存失败 -> 返回订单创建失败
```

异步消息则不同：

```text
订单服务发布消息后返回
库存服务稍后处理
```

订单服务通常无法立刻知道库存服务最终处理成功还是失败。

所以你要重新设计业务状态。

例如订单状态可能需要从：

```text
created
```

扩展成：

```text
pending_inventory
confirmed
failed
cancelled
```

异步系统经常需要更细的状态机。

## 4. 问题三：失败处理变复杂

同步调用失败时，调用方可以直接返回错误。

异步消费失败时，问题更多：

- 是立刻重试，还是稍后重试？
- 重试几次？
- 每次重试间隔多久？
- 超过次数后怎么办？
- 失败消息要不要进死信队列？
- 是否需要人工处理？
- 如何告警？

例如发送短信失败：

```text
第 1 次失败 -> 30 秒后重试
第 2 次失败 -> 5 分钟后重试
第 3 次失败 -> 30 分钟后重试
仍然失败 -> 进入死信队列并告警
```

这就是异步失败处理的基本思路。

## 5. 问题四：重复消息

RabbitMQ 常见的可靠消费语义是“至少一次”。

这意味着：消息可能被投递一次，也可能因为重试、网络断开、消费者崩溃等原因被投递多次。

例如消费者处理流程：

```text
收到 order.paid
更新积分成功
准备 ack
消费者进程崩溃
RabbitMQ 没收到 ack
RabbitMQ 再次投递 order.paid
```

如果消费者没有做幂等，用户可能被加两次积分。

所以后端工程师必须记住：

```text
只要使用消息队列，就要默认消息可能重复。
```

## 6. 问题五：消息乱序

假设同一个订单产生两条消息：

```text
order.created
order.paid
```

在复杂系统中，消费者可能先看到 `order.paid`，后看到 `order.created`。

造成乱序的原因可能有：

- 不同队列。
- 多消费者并发。
- 重试消息延迟。
- 网络和处理时间差异。

因此在业务设计中，不要盲目假设消息总是严格有序。

常见处理方式：

- 用业务状态机防御非法状态流转。
- 用版本号或时间戳判断事件新旧。
- 对强顺序场景使用更严格的队列和消费者设计。

## 7. 问题六：消息堆积

消息堆积指的是生产速度大于消费速度，导致队列里未消费消息越来越多。

常见原因：

- 消费者数量不足。
- 消费者处理逻辑变慢。
- 下游数据库变慢。
- 外部 API 变慢。
- 消费者出现错误但没有及时告警。
- 单条消息处理时间太长。

消息堆积不只是“队列里消息多”这么简单，它可能进一步导致：

- 业务延迟越来越高。
- 磁盘压力上升。
- 内存压力上升。
- 旧消息过期。
- 用户看到状态长时间不更新。

## 8. 问题七：链路排查变难

同步调用的排查路径通常比较直接：

```text
请求日志 -> 服务 A -> 服务 B -> 返回错误
```

异步消息系统中，排查链路变长：

```text
请求日志
  -> 生产者是否发出消息
  -> RabbitMQ 是否收到消息
  -> 消息路由到哪个队列
  -> 消费者是否收到
  -> 消费者处理是否成功
  -> 是否 ack
  -> 是否重试
  -> 是否进入死信队列
```

所以生产环境必须重视：

- message id
- correlation id
- 日志
- 指标
- 告警
- 死信队列巡检

## 9. 引入 RabbitMQ 前要问的问题

每次你准备使用 RabbitMQ，都应该先问：

- 这个操作是否真的可以异步？
- 用户是否必须立即知道最终结果？
- 消费失败后怎么办？
- 消息重复消费会不会造成资损或脏数据？
- 消息堆积时业务能接受多长延迟？
- 需要怎样的监控和告警？
- 有没有更简单的方案，比如同步调用或数据库任务表？

## 10. 本节小结

RabbitMQ 带来的价值是异步、解耦和缓冲。

RabbitMQ 带来的代价是：

- 最终一致性
- 失败处理复杂
- 消息可能重复
- 消息可能乱序
- 消息可能堆积
- 排查链路变长

真正成熟的 Go 后端工程师，不是只会写 producer 和 consumer，而是能把这些复杂度纳入设计。
